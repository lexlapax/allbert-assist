# v0.43 Post-Implementation Audit 02

Date: 2026-06-01
Branch reviewed: `main`
Release target: v0.43 Browser/Web Research

## Scope

This audit reviews the implemented v0.43 browser milestone against:

- `docs/plans/v0.43-plan.md`
- `docs/plans/v0.43-request-flow.md`
- `docs/adr/0040-browser-session-and-web-research-policy.md`
- the implemented browser plugin, security policy additions, mix tasks, tests,
  and release gates

The review focuses on implementation readiness, release correctness, vision
fit, safety posture, and document/code drift.

## Validation Baseline

The committed v0.43 implementation has strong automated coverage and the local
release gates passed before this audit:

- `mix allbert.test release.v043` passed:
  - 96 browser action/extractor tests
  - 11 v0.43 security eval tests
  - browser release secret scan
- `mix allbert.test release` passed:
  - Credo strict: no issues
  - core suite: 1242 tests, 0 failures, 2 skipped
  - web suite: 112 tests, 0 failures
  - StockSage/plugin suites: passing
  - Dialyzer: 0 errors
- `git diff --check` passed at release closeout

That said, the implemented release is not yet operator-ready for real browser
research. The main gaps are not around the security floor or deterministic test
coverage; they are around the real browser bridge, the external smoke contract,
session lifetime enforcement, cache retention automation, and CLI/docs drift.

## Executive Summary

v0.43 successfully lands the browser plugin shell, action vocabulary, safety
classes, permission floors, deterministic stub driver, cache metadata shape,
workspace panel contribution, release test group, and security eval inventory.
The implementation is directionally aligned with Allbert's security model:
browser content remains evidence, not authority; browser actions are registered
actions; fill and download are denied by default; and navigation is mediated by
HTTP/resource policy rather than model output.

The release does not yet satisfy the core v0.43 product promise of local
browser-backed web research. The non-test default driver is
`AllbertBrowser.Driver.Playwright`, but that module is a placeholder returning
`:playwright_bridge_unavailable` for every meaningful operation. The documented
external smoke gate also does not currently exercise a real Playwright browser:
the security eval setup pins `AllbertBrowser.Driver.Stub`, and there is no
observed branch that switches to the real driver when
`ALLBERT_BROWSER_EXTERNAL_SMOKE=1` is set.

The safest release posture is therefore:

- v0.43 is acceptable as an internal architecture and security-foundation
  milestone.
- v0.43 is not ready to advertise as working operator browser research.
- A short corrective pass should be treated as release-blocking before tagging
  or announcing the browser feature as usable.

## Findings

### F1 - Critical - Real Playwright browser support is not implemented

Evidence:

- `plugins/allbert.browser/lib/allbert_browser/driver/playwright.ex`
  implements the production default driver module.
- `verify/1`, `start_session/1`, `navigate/3`, `click/3`, `fill/3`,
  `download/3`, `extract/3`, and `screenshot/2` all return
  `{:error, :playwright_bridge_unavailable}`.
- `AllbertBrowser.Driver` selects `AllbertBrowser.Driver.Stub` only in test and
  selects `AllbertBrowser.Driver.Playwright` otherwise.

Why this matters:

The v0.43 M1 locked decision says the release uses Playwright Chromium through a
reviewed local bridge, with macOS/Linux support and no cloud browsing. ADR 0040
also says browser doctor should live-verify the local driver by launching an
ephemeral Chromium context, navigating to `about:blank`, closing it, and
recording status.

The current implementation cannot start a production browser session. In a real
operator environment, `browser_doctor` reports unavailable and
`browser_start_session` is blocked by `Doctor.fresh_ok?/0`.

Impact:

- The central v0.43 web research capability is stub-only.
- Operator validation cannot prove real navigation, extraction, screenshot, or
  download behavior.
- The implementation is safer than a half-bound bridge, but it does not meet the
  release promise.

Recommended fix:

- Implement the reviewed local Playwright bridge or explicitly downgrade the
  release scope in plan/request-flow/changelog to "browser architecture and stub
  harness only."
- Make `browser_doctor` perform the ADR-required live launch/navigation/close
  check against the real bridge.
- Add focused tests for driver error mapping and an external smoke that proves
  the real bridge when available.

### F2 - Critical - The external smoke gate does not exercise the real browser

Evidence:

- The release plan calls for an external smoke path equivalent to
  `mix allbert.test external-smoke -- browser_research`.
- The v0.43 test task maps that smoke to the v0.43 browser research security
  eval.
- `apps/allbert_assist/test/security/v043_browser_research_eval_test.exs`
  setup pins `Application.put_env(:allbert_browser, :driver,
  AllbertBrowser.Driver.Stub)`.
- No use of `ALLBERT_BROWSER_EXTERNAL_SMOKE` was found in that test file.

Why this matters:

The external smoke lane is supposed to be the bridge between deterministic
release tests and real local browser behavior. Today it validates the same
stubbed surface as the normal release gate. That creates false confidence: the
named smoke passes even when production Playwright is entirely unavailable.

Impact:

- Release evidence can claim v0.43 browser smoke coverage without proving the
  browser exists.
- Real OS/browser dependency failures are invisible until an operator tries the
  feature.

Recommended fix:

- Split deterministic security evals from true external smoke.
- When `ALLBERT_BROWSER_EXTERNAL_SMOKE=1`, require the real Playwright driver,
  run `browser_doctor`, start a session, navigate to a local fixture HTTP page,
  extract text, capture a screenshot, and close the session.
- If Playwright is unavailable, report a skipped external smoke only when the
  release plan explicitly allows a skip; otherwise fail.

### F3 - High - Session max lifetime and idle timeout are not enforced

Evidence:

- `plugins/allbert.browser/lib/allbert_browser/session.ex` stores only
  `id`, `driver_state`, `created_at`, and `last_url`.
- The session process has no max-lifetime timer, idle timer, activity
  timestamp, timeout callback, or automatic driver close path.
- `browser.session.max_concurrent` is checked before session start, but
  `browser.session.max_lifetime_ms` and `browser.session.idle_timeout_ms` are
  not enforced by the session process.

Why this matters:

The v0.43 plan and ADR 0040 both require browser sessions to be bounded by
lifetime, idle timeout, and concurrency. Concurrency is only one part of the
resource boundary. Without lifetime and idle enforcement, a browser session can
remain alive indefinitely until manually closed or process termination.

Impact:

- Local browser resources can accumulate.
- The security model's "ephemeral, bounded session" claim is incomplete.
- Cache and grant semantics become harder to reason about because long-lived
  sessions can outlive the intended review window.

Recommended fix:

- Add session state for `last_activity_at`.
- Schedule max-lifetime and idle-timeout timers during `init/1`.
- Refresh the idle timer on navigate/click/fill/download/extract/screenshot.
- On timeout, call `Driver.close/1` and terminate normally.
- Add focused tests that assert automatic close on max lifetime and idle
  timeout using short test settings.

### F4 - High - Cache retention job is implemented but not contributed at startup

Evidence:

- `AllbertBrowser.Cache.ensure_sweep_job/0` creates or updates a paused/active
  local job named "Allbert Browser cache sweep."
- Code search found production references only to the helper and tests; no
  plugin startup, app registration, supervisor, or release setup path calls it.

Why this matters:

The plan says the browser plugin contributes a paused-by-default cache sweep job
through Jobs, and the request-flow says cache retention is enforced by that job.
The helper exists, but the contribution does not appear to happen automatically.

Impact:

- Fresh installs may never get the sweep job.
- Retention is operator/manual rather than plugin-contributed.
- The implementation can pass tests that call `ensure_sweep_job/0` directly
  while missing the actual registration path.

Recommended fix:

- Call `Cache.ensure_sweep_job/0` from a plugin registration/setup path that is
  safe, idempotent, and does not perform unexpected destructive work.
- Keep the default status paused unless
  `browser.cache.sweep.schedule == "operator_approved"`.
- Add a startup/registration test proving the job appears after browser plugin
  contribution.

### F5 - High - Cache `max_bytes` is configured but not enforced

Evidence:

- The settings schema defines `browser.cache.max_bytes`.
- The plan says retention is bounded by `browser.cache.max_bytes` and
  `browser.cache.max_age_ms`.
- `AllbertBrowser.Cache.sweep_expired/1` enforces only age by file mtime.
- `AllbertBrowser.Cache.put/4` writes artifacts without checking current cache
  size or evicting older artifacts.

Why this matters:

Browser extraction, screenshots, and downloads can produce large local
artifacts. The size limit is part of the local-resource safety story, not just a
convenience setting.

Impact:

- Cache growth is unbounded except by filesystem limits and manual cleanup.
- Operator settings can imply a guarantee that the implementation does not
  honor.

Recommended fix:

- Enforce `browser.cache.max_bytes` during sweep and/or after writes.
- Prefer deterministic oldest-first eviction by metadata `created_at` or file
  mtime.
- Add tests for age eviction and size eviction separately.

### F6 - Medium - `browser.navigation.allowed_domains` is defined and documented but unused

Evidence:

- The browser plugin schema includes `browser.navigation.allowed_domains`.
- The plan uses this setting in routing language and says target URLs in the
  operator allowlist should prefer browser research.
- `AllbertBrowser.NavigationPolicy.preflight/1` passes
  `allowed_hosts: ["*"]` to `AllbertAssist.External.HttpPolicy` and uses only
  `browser.navigation.denied_domains` as `blocked_hosts`.

Why this matters:

There are two possible intended semantics:

- `allowed_domains` is a router preference hint, not a security allowlist.
- `allowed_domains` is an operator policy allowlist.

The documents currently read closer to the second meaning in some places and
the first in others. The implementation enforces neither as a navigation
allowlist.

Impact:

- Operators may configure `browser.navigation.allowed_domains` expecting
  enforcement and receive no policy effect.
- App-routing and action-policy language drift apart.

Recommended fix:

- Decide the setting's authority:
  - If it is policy, pass it to `HttpPolicy` as `allowed_hosts` and document
    wildcard/default behavior.
  - If it is only an intent-routing hint, rename or document it as such and keep
    security enforcement in denied domains plus HTTP private-host policy.
- Add tests for both allowed and denied domain behavior.

### F7 - Medium - Planned `mix allbert.browser research` CLI is missing

Evidence:

- The plan and request-flow document describe
  `mix allbert.browser research "<url>" [--extract-format=text|markdown|html|pdf]`.
- `plugins/allbert.browser/lib/mix/tasks/allbert.browser.ex` currently supports:
  - `mix allbert.browser doctor`
  - `mix allbert.browser sessions list`
  - `mix allbert.browser sessions close <session_id>`
- The usage text does not include `research`.

Why this matters:

The CLI was the planned operator smoke surface for the end-to-end browser
research flow. Without it, local validation requires assembling lower-level
actions manually.

Impact:

- Operator ergonomics are weaker than documented.
- Request-flow documentation is stale relative to implementation.
- External smoke has no simple CLI-equivalent path.

Recommended fix:

- Implement the CLI research command as a thin action-runner workflow:
  doctor, start session, navigate, extract, print/save cache reference, close.
- Use existing registered actions rather than direct session calls.
- Require the same permissions and confirmations expected for CLI channel
  operation.
- If the command is intentionally deferred, remove it from v0.43 release docs
  and move it to the next plan.

### F8 - Medium - Workspace panel implementation works through app contribution but drifts from named plan modules

Evidence:

- The plan/request-flow name `AllbertBrowser.SurfaceProvider` and
  `AllbertBrowser.Panels.Results`.
- The implementation contributes a panel through `AllbertBrowser.App`.
- The panel uses `visible_when: :selected_app`, while the request-flow says
  `visible_when: :operator_opened`.
- The panel lists latest global browser cache artifacts rather than rendering a
  specific cache reference produced by the current action flow.

Why this matters:

This is not necessarily a wrong implementation; the app-level provider may be
the preferred current extension point. The drift matters because future agents
or maintainers will look for the named modules and behavior described in the
v0.43 documents.

Impact:

- Documentation misleads implementation work.
- The panel is more of a global recent-artifacts view than the action-specific
  results panel described in the request-flow.

Recommended fix:

- Either update docs to describe `AllbertBrowser.App` as the contribution point,
  or add the named provider/panel modules as thin wrappers if that better
  matches the workspace surface architecture.
- Align `visible_when` terminology.
- Consider passing action/cache references into the panel state when workspace
  flows open browser evidence.

### F9 - Medium - Some settings reads can crash instead of falling back

Evidence:

- `browser_start_session` uses `elem(Settings.get("browser.session.max_concurrent"), 1)`.
- `browser_extract` uses `elem(Settings.get("browser.extraction.max_bytes"), 1)`.
- Other browser code uses safer helper functions that fall back when
  `Settings.get/1` returns an error.

Why this matters:

Settings Central is the right authority, but action code should not assume every
setting read succeeds. Missing or corrupted settings should produce a controlled
action error or fallback, not a process crash.

Impact:

- Rare settings-store failures can turn into action crashes.
- The pattern is inconsistent across the browser plugin.

Recommended fix:

- Replace direct `elem/2` calls with helper functions that pattern match
  `{:ok, value}` and provide a documented fallback or controlled error.
- Add a focused test with the setting absent/unavailable if the settings test
  helpers make that cheap.

### F10 - Low - Session lifecycle actions use an extraction permission label

Evidence:

- Browser session list/close actions use permission semantics tied to
  `:browser_extract` rather than an explicit session lifecycle/read permission.

Why this matters:

This is probably not unsafe in practice because list/close are lower-risk than
navigation, form fill, or download. It is still a taxonomy mismatch: session
management is neither content extraction nor page reading.

Impact:

- Policy audit output is harder to read.
- Future permission changes could accidentally couple session lifecycle to
  extraction behavior.

Recommended fix:

- Either document that `:browser_extract` covers browser read/session lifecycle
  operations, or introduce clearer operation classes for browser session
  lifecycle.

## What Matches The v0.43 Vision

The implementation has several solid foundations:

- Browser actions are registered and route through the action runner.
- Security operation classes exist for browser start, navigate, extract,
  screenshot, click, fill, download, cache sweep, list sessions, and close
  sessions.
- Safety floors are conservative:
  - navigation needs confirmation
  - click needs confirmation
  - fill is denied by default
  - download is denied by default
  - extraction/screenshot are allowed read operations
- Browser content is treated as page evidence, not authority.
- Resource URIs and scopes are browser-aware.
- Confirmation metadata has browser-specific summaries.
- Cache artifacts use content hashes and metadata sidecars.
- The deterministic stub driver gives good local test coverage without network
  or browser dependencies.
- Release tests include a browser-specific security eval inventory and secret
  scan.
- The workspace panel contribution provides an initial operator-visible browser
  artifact surface.

## Milestone Drift

### M1 - Browser Substrate

Implemented:

- Driver behavior abstraction
- Stub driver
- Doctor result persistence
- Session supervisor and registry

Drift:

- Real Playwright bridge is not implemented.
- Doctor cannot perform the ADR-required live browser verification.
- Session lifetime and idle timeout are not enforced.

Status: incomplete for release usability.

### M2 - Safe Browser Actions

Implemented:

- Registered browser actions
- Permission floors and confirmation paths
- Resource URI/scope additions
- Deny-by-default fill/download settings
- Navigation preflight using HTTP policy

Drift:

- `allowed_domains` setting is not enforced or clearly scoped as a hint.
- A few settings reads can crash on error.
- Real driver behavior is untested because production bridge is absent.

Status: mostly aligned for safety, incomplete for real execution.

### M3 - Extraction, Cache, And Workspace Surface

Implemented:

- Extraction and screenshot actions
- Cache write and metadata shape
- Cache sweep action/helper
- Workspace panel contribution

Drift:

- Cache sweep job is not automatically contributed at startup.
- `browser.cache.max_bytes` is not enforced.
- Workspace panel naming/visibility and per-action cache-reference behavior
  differ from request-flow docs.

Status: partial; cache retention needs a corrective pass.

### M4 - CLI, Tests, And Security Evals

Implemented:

- Doctor/session CLI helpers
- Browser action test coverage
- v0.43 security eval file
- Release test task

Drift:

- Planned `mix allbert.browser research` command is missing.
- External smoke path does not use a real browser.

Status: good deterministic gate, insufficient operator smoke.

### M5 - Release Closeout

Implemented:

- Release gates passed.
- Browser release evidence file produced.
- Changelog/docs updated for implemented release.

Drift:

- Release evidence should be interpreted as stub/security foundation evidence,
  not real browser-readiness evidence.
- Docs still describe capabilities that are not implemented.

Status: clean automated release gate, but product readiness is overstated.

## Recommended Corrective Plan

### Release Blocker Pass

1. Implement or intentionally descope the real Playwright bridge.
2. Replace the current external-smoke mapping with a true real-browser smoke.
3. Add session max-lifetime and idle-timeout enforcement.
4. Wire cache sweep job contribution into plugin setup.
5. Enforce `browser.cache.max_bytes`.

### Documentation Reconciliation

1. Update `v0.43-plan.md` and `v0.43-request-flow.md` to clearly distinguish:
   - implemented deterministic stub foundation
   - real Playwright bridge readiness
   - external smoke requirements
2. Clarify `browser.navigation.allowed_domains` as either policy or routing
   hint.
3. Either implement or remove the documented CLI research command.
4. Align workspace panel docs with the implemented app contribution point.

### Focused Test Additions

Recommended focused tests before release-ready declaration:

- Real-driver doctor smoke gated by `ALLBERT_BROWSER_EXTERNAL_SMOKE=1`
- Start-session refusal when doctor is stale/unavailable
- Session automatic close on max lifetime
- Session automatic close on idle timeout
- Cache sweep job contribution during plugin registration/setup
- Cache size eviction by `browser.cache.max_bytes`
- Navigation allowed/denied domain behavior matching the chosen setting
  semantics
- CLI `research` command flow, if retained in scope

## Release Readiness Verdict

v0.43 is a strong browser-security foundation milestone, but it is not ready to
release as a working local browser research feature.

The implementation is safe in the sense that dangerous browser capabilities are
mostly denied, confirmed, or stubbed. The remaining issue is accuracy of release
claims: the browser feature is not actually browser-backed in non-test mode, and
the smoke gate does not prove otherwise.

Recommended label until corrective work lands:

> v0.43 browser architecture/security foundation implemented; real browser
> research bridge and true external smoke pending.
