# v0.39 / v0.39b Post-Implementation Audit 03

Date: 2026-05-27

Scope: third-pass audit of v0.39 first-run onboarding and provider/model
control, plus v0.39b identity namespace and Active Memory retrieval. This pass
reconciles `0.39-impl-audit-01.md` and `0.39-impl-audit-02.md`; the user
referred to `0.39-impl-audit091.md`, but no file with that name exists in the
project root.

## Verdict

The implementation is mostly sound as safe runtime substrate. I found no P0
authority-boundary, secret-handling, or data-loss issue in v0.39/v0.39b.

It is not yet cleanly release-ready against the product intent that v0.39 make
operator onboarding easy. The current onboarding flow is a resumable objective
checklist that points to Settings Central and Mix commands; it does not yet
guide an operator through provider selection, credential setup, doctor
evidence, model enablement, channel setup, and identity/Active Memory readiness
as a coherent first-run experience.

Active Memory is acceptable as a deterministic, inert, body-redacted retrieval
substrate. From a UX perspective it is acceptable only for an advanced/manual
operator path, not yet as a self-discovering memory-retrieval experience. The
operator must manually create identity markdown, review/keep it, enable
direct-answer model mode, and know where to inspect retrieval. That matches the
v0.39b non-goal of "no rich authoring UX", but it does not satisfy an "easy
memory retrieval onboarding" reading without docs and workspace affordance
fixes.

## Reconciliation Of Audit 01 And Audit 02

Audit 01 was directionally right that the implementation has release-readiness
gaps, but it over-rated several findings as P1 safety/correctness problems.
Audit 02 correctly downgraded most of those from an authority-boundary and
runtime-safety perspective.

This third pass separates safety severity from product-readiness severity:

| Topic | Audit 01 | Audit 02 | Audit 03 reconciled rating |
| --- | --- | --- | --- |
| Fresh default points to remote `fast` | P1 | P2 | P2 UX/defaults decision; safe because model paths are off by default, but weak for local-first onboarding. |
| Optional channel registration can complete without setup | P1 | P2 | P1 for "easy onboarding" product readiness; P2 for correctness. It is safe, but the UI records a choice as completion without evidence. |
| Provider doctor diagnostics drift from ADR 0047 | P1 | P2 | P2 now, P1 before v0.40 builds more doctor surfaces on ADR 0047. |
| Active Memory chunking/docs mismatch | P2 | P2 | P2 contract drift. Implementation is deterministic, but docs disagree on scoring/chunking/ellipsis. |
| Active Memory `now` not request-start pinned | P2 | P3 | P3 unless multiple retrievals per request become visible; still worth fixing while in the area. |
| Manual validation pending | P2 | P3/P2 | P2 release-readiness blocker. Current tests are green, but operator manual validation and runbook fixes remain. |
| Future/vision doc drift | P3 | P3 | P3. Not blocking substrate, but it creates roadmap confusion. |
| `mix allbert.onboard` non-exhaustive error case | Not called out | P3 | P3 robustness issue. |

## Findings

### P1: Onboarding is a safe checklist, not yet easy operator onboarding

The v0.39 plan allows the implementation to frame an onboarding objective and
record progress. The vision and v1.0 acceptance language are stronger: a fresh
operator should be guided through a working provider/model setup with visible
doctor evidence and understandable next steps.

Current behavior:

- CLI `mix allbert.onboard` prints step titles and candidate action names, but
  not the exact next command for the current step.
- Workspace onboarding renders a current step card with generic `Complete` and
  `Skip` actions for most steps.
- `Onboarding.complete_step/4` records step transitions; it does not verify that
  the candidate action actually ran or succeeded.
- Channel selection marks Telegram/email as completed immediately.
- Provider/key/doctor/model controls exist in Settings Central, but the
  onboarding panel does not embed or sequence those controls.

Evidence:

- `apps/allbert_assist/lib/allbert_assist/onboarding.ex:16`
- `apps/allbert_assist/lib/allbert_assist/onboarding.ex:110`
- `apps/allbert_assist/lib/mix/tasks/allbert.onboard.ex:87`
- `apps/allbert_assist/lib/mix/tasks/allbert.onboard.ex:73`
- `apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/onboarding.ex:82`
- `apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/onboarding.ex:244`

Fix:

1. Keep the objective tracker, but add evidence-aware step helpers.
2. For provider/model steps, show the relevant Settings Central controls in the
   onboarding destination or add direct step actions that call the same
   registered actions.
3. Store step evidence summaries on the objective step result:
   selected provider/model, doctor status, redacted host, credential status,
   and model-assisted/direct-answer toggle status.
4. For channel setup, record Telegram/email selection as "selected" or
   "deferred" until the corresponding channel settings validate. Only "none"
   should map cleanly to skipped.
5. Update CLI output to print the next command for each current step, not just
   `action=...`.
6. Add tests that completing the doctor/model/channel steps requires, or at
   least displays and records, the relevant evidence.

### P2: Disposable first-run smoke docs are incomplete for CLI onboarding

`docs/plans/v0.39-request-flow.md` says to complete onboarding in a disposable
Allbert Home with only `ALLBERT_HOME=$(mktemp -d ...)`. In development config,
that does place the dev SQLite DB under `ALLBERT_HOME`, but an existing checkout
can still point at a new unmigrated database. `mix allbert.onboard` starts the
app and hits objective/job tables; it does not run migrations.

Evidence:

- `docs/plans/v0.39-request-flow.md:93`
- `config/dev.exs:3`
- `apps/allbert_assist/lib/allbert_assist/paths.ex:197`
- Observed smoke: no-table failure before migrations; successful framing after
  `mix do ecto.create --quiet + ecto.migrate --quiet`.

Fix:

1. Update v0.39 request-flow and operator onboarding docs with a disposable
   smoke prelude:

   ```sh
   export ALLBERT_HOME="$(mktemp -d /tmp/allbert-onboard-smoke.XXXXXX)"
   export DATABASE_PATH="$ALLBERT_HOME/db/allbert.sqlite3"
   mix do ecto.create --quiet + ecto.migrate --quiet
   mix allbert.onboard
   ```

2. Alternatively, add a safe CLI bootstrap check that detects a missing dev DB
   schema and prints the exact migration command instead of a raw table error.

### P2: Provider doctor still lacks an enforced ADR 0047 diagnostic catalog

The doctor is bounded and redacted enough for the existing tests, but the code
still constructs several diagnostic messages dynamically from `inspect(reason)`
or `safe_reason(reason)`. ADR 0047 calls for fixed-catalog diagnostics with
bounded, redacted fields so downstream doctors can reuse the shape without
inventing provider-specific leakage.

Audit 02 is right that this is not an observed secret leak today. Audit 01 is
right that it is contract drift. Because v0.40 MCP doctor planning now depends
on ADR 0047, this should be fixed before implementing more doctor surfaces.

Evidence:

- `apps/allbert_assist/lib/allbert_assist/settings/model_doctor.ex:64`
- `apps/allbert_assist/lib/allbert_assist/settings/model_doctor.ex:111`
- `apps/allbert_assist/lib/allbert_assist/settings/model_doctor.ex:119`
- `apps/allbert_assist/lib/allbert_assist/settings/model_doctor.ex:127`
- `docs/adr/0047-provider-doctor-contract.md:67`
- `docs/plans/v0.40-plan.md:61`
- `docs/plans/v0.40-request-flow.md:47`

Fix:

1. Add a central doctor diagnostic module/catalog for ADR 0047 codes.
2. Keep public messages fixed or templated only over safe enum fields.
3. Put raw reasons, if needed for developer debugging, behind a redacted and
   capped internal metadata field that is never surfaced in CLI/LiveView/traces.
4. Add tests for allowed code set, max message length, no path/query/body echo,
   no raw URL, no token-like substrings, and additive-only envelope fields.

### P2: Local-first defaults are safe but weak for first-run UX

The shipped provider defaults include an enabled local Ollama provider and a
real local model profile. However, `intent.model_profile` and
`intent.direct_answer_model_profile` still default to `fast`, which resolves to
the disabled OpenAI provider. Because model-assisted/direct-answer paths are
default-off, this is safe. From a first-run UX perspective it makes the default
state harder to explain: the operator sees a local provider story in docs but a
remote active profile in settings and CLI output.

Evidence:

- `apps/allbert_assist/lib/allbert_assist/settings/schema.ex:1708`
- `apps/allbert_assist/lib/allbert_assist/settings/schema.ex:1722`
- `apps/allbert_assist/lib/allbert_assist/settings/schema.ex:1749`
- `apps/allbert_assist/test/mix/tasks/allbert_model_test.exs:24`

Fix options:

1. Preferred for easy onboarding: default both active intent profiles to
   `local`, keep model-assisted/direct-answer disabled, and let the local doctor
   explain missing Ollama/model remediation.
2. Alternative: keep `fast`, but have onboarding mark provider/model as
   unconfigured until the operator explicitly chooses `local`, `fast`, or
   `anthropic_fast`.

This needs one product decision before code changes because the tests currently
pin `fast` as the active model profile.

### P2: Active Memory chunking contract has a three-way split

The implementation chunks each `Memory.Entry` body into byte-bounded windows
before scoring. The research note says to score candidate chunks, take top-K,
then trim each selected body to `chunk_max_bytes` and append ellipsis if
truncated. The v0.39b plan says to chunk each entry into byte windows at scoring
time and append ellipsis only if truncated. The implementation does not append
ellipsis because its windows are not represented as truncated excerpts.

The current implementation is deterministic and may retrieve better matches
from long files than the research-note version. The problem is contract drift,
not runtime safety.

Evidence:

- `apps/allbert_assist/lib/allbert_assist/memory/active_memory.ex:94`
- `apps/allbert_assist/lib/allbert_assist/memory/active_memory.ex:178`
- `docs/research/active-memory-retrieval.md:114`
- `docs/plans/v0.39b-plan.md:151`
- `docs/developer/active-memory.md:47`

Fix:

1. Decide the intended contract:
   - If pre-scoring windows are intended, amend the research note and operator
     doc to say entries are split into bounded windows before scoring and remove
     the top-K trim language.
   - If research-note top-K trimming is intended, change implementation to
     score entry-level bodies and trim selected bodies after top-K.
2. Add a fixture that proves long identity files produce the expected chunk ids,
   sizes, body text, and trace metadata.
3. Add an explicit assertion for ellipsis/no-ellipsis behavior.

### P2: Active Memory UX is technically correct but not self-discovering

v0.39b intentionally avoids rich identity authoring UX, so this is not a
straight misimplementation. It is still a UX gap against the user's stated
intent that memory retrieval feel acceptable.

Current operator path:

1. Create markdown under `<ALLBERT_HOME>/memory/identity/`.
2. Use memory review/update to mark entries `:kept`.
3. Enable direct-answer model mode and configure a usable model profile.
4. Ask a direct-answer-routed question.
5. Inspect `## Active Memory` traces or run `mix allbert.memory retrieve`.

That is powerful but not obvious. The workspace onboarding text also still says
"Identity slot and Active Memory arrive in v0.39b", even though v0.39b is now
implemented as `0.39.1`.

Evidence:

- `docs/operator/active-memory.md:34`
- `docs/operator/active-memory.md:50`
- `apps/allbert_assist/lib/allbert_assist/actions/intent/direct_answer.ex:61`
- `apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/onboarding.ex:268`
- `apps/allbert_assist/lib/mix/tasks/allbert.memory.ex`

Fix:

1. Update onboarding copy to present identity/Active Memory as available in
   `0.39.1`, with exact next steps.
2. Add an onboarding "try Active Memory" optional step or sub-panel that shows:
   identity root path, whether any identity entries are `:kept`, whether
   `active_memory.enabled` is true, whether direct-answer model mode is enabled,
   and the current direct-answer model profile health.
3. Add a CLI helper or documented one-shot smoke path that creates a disposable
   identity fixture, marks it kept, runs retrieval, and shows the trace-free
   score breakdown.
4. Keep memory bodies out of traces and normal LiveView diagnostics.

### P3: Active Memory recency `now` is not request-start pinned

The research note says `now` is pinned at request start. Direct-answer currently
sets `now` immediately before invoking `retrieve_active_memory`. For a single
retrieval per turn this is effectively harmless. It becomes observable if later
paths run more than one retrieval during the same request.

Evidence:

- `docs/research/active-memory-retrieval.md:57`
- `docs/developer/active-memory.md:31`
- `apps/allbert_assist/lib/allbert_assist/actions/intent/direct_answer.ex:125`

Fix:

1. Add `request_started_at` or equivalent to the runtime/direct-answer context.
2. Pass it to `retrieve_active_memory` as `now`.
3. Keep the current fallback only when the caller provides no request timestamp.

### P3: Plan, vision, and future stubs still contain stale v0.39/v0.39b wording

Most current docs are good, especially README and the v0.40 plan. Some
developer/vision/future stub docs still describe shipped v0.39/v0.39b work as
future or planned. This creates continuity drift for future agents.

Evidence examples:

- `docs/developer/runtime-boundary-map.md` still describes the v0.39-to-v1.0
  arc as planned future work.
- `docs/plans/allbert-jido-vision.md` still says Active Memory is planned in
  some sections and puts Active Memory under v0.39 in one place.
- `docs/plans/v0.47-plan.md` and `docs/plans/v0.48-plan.md` reference the
  "v0.39 provider doctor path" but should point at ADR 0047 explicitly, as
  v0.40 now does.

Fix:

1. Update runtime boundary and vision docs to state that v0.39 and v0.39b are
   implemented as `0.39.0` and `0.39.1`.
2. Fix the Active Memory milestone wording in `allbert-jido-vision.md`.
3. Update v0.47/v0.48 stubs to depend on ADR 0047, not a vague v0.39 path.

### P3: `mix allbert.onboard` has a non-exhaustive action result case

`completed_action/3` handles `{:ok, %{status: :completed}}` and `{:ok,
response}` but not `{:error, reason}` from `Runner.run/3`. A runner error would
raise a `CaseClauseError` instead of returning the intended onboarding failure
message.

Evidence:

- `apps/allbert_assist/lib/mix/tasks/allbert.onboard.ex:161`

Fix:

Add `{:error, reason} -> {:error, reason}` and a test that stubs or exercises a
runner error path.

## What Looks Correct

- v0.39/v0.39b preserve Security Central boundaries. Runtime-facing behavior is
  behind registered actions and the shared action runner.
- Identity memory is represented as a system namespace, not a fake app id.
- Active Memory retrieval is read-only, deterministic, scoped, and body-free in
  trace metadata.
- Memory bodies are passed only to direct-answer prompt composition, not to
  intent ranking/classification or trace rendering.
- The current focused backend/security and workspace LiveView tests are green.
- v0.40 planning now references ADR 0047 thoroughly, so the future continuity
  problem is mostly localized to the doctor implementation and older plan stubs.
