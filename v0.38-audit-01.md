# v0.38 Implementation Audit 01

Date: 2026-05-27

Scope: audit the pushed v0.38 implementation against the v0.38 vision, plan,
request flow, continuity docs, release claims, and current code/tests. This
report records implementation gaps and drift only; it does not include code
changes.

## Executive Summary

v0.38 is broadly implemented and release-gate healthy. The TemplatePattern
registry, shipped patterns, Mix generators, `workspace:create` destination,
registered template actions, LLM-tool dynamic draft path, security eval rows,
docs, changelog, roadmap, and v0.39 handoff are all present.

The main concerns are contract drift around the live-integration prerequisite
switches and parameter-as-data guarantees. The first is a policy mismatch:
`create_from_template` writes a dynamic draft when `templates.create.enabled`
and `dynamic_codegen.enabled` are true, even though the plan/operator docs say
templated live integration also requires `dynamic_codegen.live_loader_enabled`
and `sandbox.elixir.enabled`. The second is a template-rendering safety gap:
some developer scaffold params are substituted raw into Elixir string literal
positions. That does not grant live authority by itself, but it contradicts
the v0.38 "params are data" invariant and can produce executable or
compile-breaking source if a generated scaffold is compiled after review misses
the injected value.

## Findings

### High: Templated draft creation does not enforce two documented live-integration switches

Authority docs say operator live integration requires three switches:
`dynamic_codegen.enabled`, `dynamic_codegen.live_loader_enabled`, and
`sandbox.elixir.enabled`.

Evidence:

- `docs/plans/v0.38-plan.md:186` says operator live integration additionally
  requires all three switches and the v0.36/v0.37 path.
- `docs/plans/v0.38-request-flow.md:166` lists all three as required live
  integration switches.
- `docs/operator/templated-creation.md:49` through
  `docs/operator/templated-creation.md:51` repeats the same operator-facing
  requirement.
- `apps/allbert_assist/lib/allbert_assist/actions/templates/create_from_template.ex:39`
  through `:42` only checks template creation, dynamic codegen enablement, and
  the permission decision.
- The web test at
  `apps/allbert_assist_web/test/allbert_assist_web/live/workspace_live_test.exs:577`
  through `:599` proves draft creation succeeds after setting only
  `templates.create.enabled` and `dynamic_codegen.enabled`.

Impact:

- This does not register live authority; the action still writes a draft only.
- It does let the operator surface create file-backed dynamic draft state while
  the documented sandbox and live-loader posture remain disabled.
- The security eval wording in `docs/plans/v0.38-request-flow.md:171` through
  `:173` is stronger than the implementation and tests.

Recommended resolution:

- Either add explicit checks for `dynamic_codegen.live_loader_enabled` and
  `sandbox.elixir.enabled` before `CreateFromTemplate` writes a draft, or
  narrow the docs/eval language to say those switches are required for the
  later trial/gate/integration steps, not for draft creation.
- If the implementation remains draft-first, rename UI copy away from "live
  integration" for the draft-write button so the switch semantics are clear.

### High: Some scaffold params are rendered raw into Elixir source literals

v0.38's plan and ADR say template parameters are data and cannot inject code.
The LLM-tool action template uses `inspect/1` and JSON encoders for dangerous
fields, but several developer scaffold fields still render raw inside Elixir
quoted strings.

Evidence:

- `apps/allbert_assist/lib/allbert_assist/templates/patterns/plugin.ex:23`
  through `:32` accepts `version` with only a length bound.
- `apps/allbert_assist/priv/templates/v0_38/plugin/plugin.ex.tmpl:12` through
  `:13` renders `version` as `def version, do: "{{version}}"`.
- `apps/allbert_assist/lib/allbert_assist/templates/patterns/app.ex:23`
  through `:32` has the same `version` shape, and
  `apps/allbert_assist/priv/templates/v0_38/app/app.ex.tmpl:16` through `:17`
  renders it raw.
- `apps/allbert_assist/lib/allbert_assist/templates/patterns/flow.ex:38`
  through `:39` accepts `timezone` with only a length bound, and
  `apps/allbert_assist/priv/templates/v0_38/flow/flow.ex.tmpl:20` renders it
  raw inside a quoted string.
- The security eval at
  `apps/allbert_assist/test/security/template_creation_eval_test.exs:81`
  through `:101` only checks the LLM-tool `instruction` field, which is one of
  the safer fields because it renders through `inspect/1`.

Impact:

- Developer scaffolds are inert until reviewed and compiled, so this is not a
  live authority bypass.
- It is still a violation of the "parameters are data" contract and can
  produce injected or invalid Elixir source in the human-review path.

Recommended resolution:

- Use literal/JSON-normalized fields for every template placeholder that lands
  inside source code, especially `version`, `timezone`, and any future free
  string field.
- Add security eval cases for non-LLM-tool patterns and for fields that render
  into Elixir source, not just Markdown/JSON output.

### Medium: Workspace preview/validation bypass the registered render/validate actions

The Create surface now correctly calls registered actions for effectful create
clicks, but preview and validation still call the scaffold facade directly.

Evidence:

- `docs/plans/v0.38-plan.md:192` through `:194` says template render,
  scaffold, validate, and create-integration route through registered actions
  and emit signals/trace/audit records.
- `docs/plans/v0.38-request-flow.md:92` says the surface owns only
  view/compose state and effectful work uses registered actions.
- `apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/template_create.ex:397`
  through `:405` calls `Scaffold.preview/2` directly.
- `apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/template_create.ex:471`
  through `:479` calls the runner only for `scaffold_template` and
  `create_from_template`.

Impact:

- Preview is read-only, so this is not an effectful authority bypass.
- It does mean the UI preview/validation path lacks the `render_template` /
  `validate_template` action telemetry and permission/audit shape promised by
  the v0.38 docs.

Recommended resolution:

- Route the UI preview state through `render_template` and/or
  `validate_template`, or amend the docs to reserve registered actions for
  writes/draft creation only.

### Medium: Planned generic `mix allbert.gen.<pattern>` dispatcher is absent

The plan lists a generic future-pattern dispatcher as a planned interface, but
the implementation ships only concrete generator tasks plus `gen.flow
--pattern objective`.

Evidence:

- `docs/plans/v0.38-plan.md:135` through `:143` lists
  `mix allbert.gen.<pattern> NAME`.
- Implemented task modules are concrete:
  `apps/allbert_assist/lib/mix/tasks/allbert.gen.plugin.ex:1`,
  `apps/allbert_assist/lib/mix/tasks/allbert.gen.app.ex:1`,
  `apps/allbert_assist/lib/mix/tasks/allbert.gen.tool.ex:1`, and
  `apps/allbert_assist/lib/mix/tasks/allbert.gen.flow.ex:1`.

Impact:

- Current shipped patterns are reachable.
- Future reviewed patterns cannot be reached through the planned generic CLI
  without adding a task or using the internal support module.

Recommended resolution:

- Either add the generic dispatcher before treating the pattern registry as
  externally extensible, or mark the generic task as deferred in the v0.38 plan
  and docs.

### Low: Operator scaffold UI lacks explicit target/force controls

The operator docs describe developer scaffold output in terms of `--target`
and `--force`, but the `workspace:create` component does not expose a target or
force field. It can write the default `./plugins/<slug>` scaffold and it denies
existing roots.

Impact:

- The safe default is preserved.
- The operator-facing docs blur CLI and LiveView capabilities.

Recommended resolution:

- Either add target/force controls with confirmation, or split the docs so
  `--target` and `--force` are clearly CLI-only in v0.38.

Manual verification remains operator-owned. I did not rerun Chrome extension
browser-control verification during this audit; the release changelog records

paths through tests rather than a browser session.

## Recommended Follow-Up Order

1. Resolve the live-integration switch semantics. Decide whether
   `CreateFromTemplate` is a draft-write action or the start of operator live
   integration, then align code, docs, security evals, and UI labels.
2. Harden source-rendered parameters by using literal-safe rendered fields for
   every value that lands in Elixir source.
3. Add broader template injection evals for plugin/app/flow/objective source
   fields.
4. Decide whether the generic `mix allbert.gen.<pattern>` dispatcher is shipped
   or deferred.
5. Clarify or extend `workspace:create` developer scaffold controls around
   target and force.
