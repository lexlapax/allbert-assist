## Scope

Full audit of the v0.38 Templated Creation release as implemented in M1-M6, against the v0.38 plan, the request flow, ADR 0036, the v0.27-v0.37 contract shapes it builds on, the AGENTS.md non-negotiables, the Allbert Jido vision, and the v0.39 Ôćĺ v1.0 continuity arc.

## Verdict

**v0.38 shipped faithfully and safely.** Every planned interface, action, Mix task, Settings Central key, DOM id, security eval row, and documentation deliverable is present. Authority boundaries are preserved ÔÇö templates and parameters grant no authority, developer scaffolds remain inert source, and operator live integration writes a draft only and hands off to the v0.36/v0.37 lifecycle for sandbox trial, gate, confirmation, and rollback. Live integration is correctly bounded to the LLM-tool (action) pattern. Version metadata is bumped to `0.38.0`. CHANGELOG entry is complete. Release is ready for operator manual verification per `docs/operator/templated-creation.md`.

Two **moderate-severity** doc/planning gaps exist that should be closed before they cascade into later milestones, and three **low-severity** strengthening items would harden v0.38 against later drift. None block the release tag.

## Moderate gaps (close before downstream consumers depend on the state)

### G1. v1.0 contract freeze omits v0.38 surfaces

`docs/plans/v1.0-plan.md` Tier 1 lists Runtime, Actions.Registry/Runner, Permissions, Plugin, App, Settings schema **shape**, Allbert Home, Channel, Resource Access. Tier 2 lists SurfaceProvider, Surface DSL catalog, Canvas/ephemeral substrate. **Neither tier lists** `AllbertAssist.Templates`, `AllbertAssist.Templates.Pattern` behaviour, the four registered template actions, the `workspace:create` Canvas destination, or the `templates.*` settings keys.

This is the same class of consumer-stable surface that the freeze is meant to cover. Plugin authors writing patterns against the `TemplatePattern` behaviour after v0.45 marketplace-lite ships will depend on these contracts. v0.46 self-improvement explicitly hands off to v0.38 templates (`docs/plans/v0.46-plan.md:55`). v0.45 surfaces template catalog metadata "through `workspace:create`" (`v0.45-plan.md:49-50`).

**Edit:** add a new Tier 2 group to `docs/plans/v1.0-plan.md` for the v0.38 templated-creation contracts ÔÇö `AllbertAssist.Templates`, `TemplatePattern` behaviour, the four registered template actions by name, the `workspace:create` Canvas destination id, and the two `templates.*` settings keys. Tier 2 is the right tier because they are single-implementation (Allbert core) with one or two known downstream consumers (v0.45, v0.46) and not yet operator-tested for Ôëą 6 months at v1.0; additive-only changes should be allowed post-1.0.

### G2. ADR 0046 schema-migration policy does not list v0.38 template settings as known schema

`docs/adr/0046-settings-schema-migration-policy.md:11-16` enumerates the fragments that have accumulated through v0.37: base, security, skill-write, confirmations, execution, external/package/online-import, jobs, apps, channels, memory, objectives, workspace, settings-fragment substrate, panels/zones, launcher, theming, sandbox, dynamic codegen. **v0.38 templated-creation settings (`templates.create.enabled`, `templates.allowed_patterns`) are not mentioned**, despite landing in `schema.ex` and being a fragment subject to the same migration policy from v0.45 onward.

This is a small documentation drift, not a behavior issue. No live migration is required because v0.38 is the introducing release. But ADR 0046 sets the inventory baseline for the migration tool that ships at v0.50, and the inventory is incomplete.

**Edit:** add v0.38 to the fragment enumeration in `ADR 0046 ┬žContext` ("v0.38 templates") and confirm `schema_version: 1` is the correct starting version for the templates fragment when ADR 0046 is accepted at v0.50.

## Low-severity strengthening items (no blocker)

### S1. Authority-bypass security test trusts the renderer

`test/security/template_creation_eval_test.exs:173-215` (`template-authority-bypass-001`) asserts that a scaffolded action is not registered and no draft is written. It does **not** inspect the generated source file content to confirm the scaffold itself is inert (e.g. that it contains no `use Application`, no compile-time side effects, no `@on_load`, no enabled jobs, no auto-registered routes). The renderer is well-tested and the priv templates are reviewed source, but the eval row would be stronger if it parsed the rendered output and asserted shape invariants.

**Suggested edit:** strengthen the test to read the rendered file from `priv/templates/v0_38/<pattern>/` and assert with `Code.string_to_quoted!/2` that no module attribute mutates application state, no top-level `use` outside the allowlisted set is present, and that any generated `actions/0` list is consistent with the manifest. This is an additive test improvement; the existing eval is not broken.

### S2. Scheduled-flow escalation test asserts on JSON, not on the UI surface

`template_creation_eval_test.exs:314-358` confirms `job_blueprint.json.tmpl` declares `"enabled": false`. It does **not** test the UI behavior when an operator scaffolds a flow and the Settings panel renders the disabled-by-default state, nor whether `mix allbert.dynamic drafts list` would refuse to show a flow draft. Both are correctly out of scope ÔÇö flow drafts are dev-scaffold only ÔÇö but a future regression that flipped the default to `"enabled": true` would not be caught at the UI layer.

**Suggested edit (optional, future milestone):** add a workspace LiveView test that scaffolds a flow and asserts the rendered preview shows `enabled: false` in the diagnostic output. Not a v0.38 blocker.

### S3. Canvas action-boundary test checks action response, not audit/signal

`template_creation_eval_test.exs:275-307` (`template-canvas-authority-001`) sets `permissions.dynamic_codegen_request: "denied"` and confirms `CreateFromTemplate` is denied. It does not confirm the denial produces an audit record or a redacted signal. The action boundary is correct, but the observability path for "user tried something denied" deserves explicit coverage at the eval layer to prevent silent-failure regressions.

**Suggested edit:** add an assertion that an `:action_denied` audit entry was written and `:template_draft_created` signal was not emitted. Optional.

## Closing assessment

v0.38 is a clean release. The pre-implementation plan revisit (PR #3) accurately predicted the live-integration matrix bound and the producer-handoff details, and the implementation matched the predictions verbatim. The two moderate gaps are doc-only artifacts of the v0.38 Ôćĺ v1.0 horizon shifting after v0.38 landed: the v1.0 freeze list and the ADR 0046 migration inventory were drafted before v0.38 templates existed as a concrete surface, and need a small additive update to close the loop.

The release is **ready to tag** and ready for operator manual verification. The proposed doc-only edits above should land before v0.45 marketplace-lite begins, because v0.45 is the first downstream consumer that will materially depend on the v0.38 surface being frozen at known boundaries.
