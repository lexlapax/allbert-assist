## Verdict

**v0.43 ships a complete, well-disciplined substrate. The browser
runtime is not yet operational.** The plan's per-milestone acceptance
criteria are met for M1-M4; M5 closeout discipline is the cleanest
v0.4x release to date. The release gate is green. But the Playwright
driver — the actual mechanism by which v0.43 turns operator prompts
into rendered web pages — is a stub: every operation returns
`{:error, :playwright_bridge_unavailable}`. An operator following the
operator doc cannot research the web today; an external-smoke
invocation against the Playwright driver fails immediately. The
substrate is real, the eval coverage is real, the doc-sync is real;
the runtime is paper.

This is the same pattern v0.42 audit-02 caught: substrate correct,
runtime placeholder. Different shape (v0.42 had bugs in the live
runtime; v0.43 has a deliberately-empty runtime stub), same gap
between operator-facing claim and operator-observable behavior.

The fix is bounded: either (a) ship the Playwright bridge as
v0.43.1, or (b) explicitly park "Playwright bridge implementation" in
operator/developer docs + roadmap + future-features and downgrade the
operator-facing claim to "v0.43 ships the policy substrate; the live
bridge is a v0.43.1 follow-on."

The M5 closeout discipline is otherwise the cleanest of any v0.4x
release — every artifact synchronized in one commit, no D1-D6-style
drift, no need for a sync commit.

## Gaps (prioritized)

### G1 — Playwright bridge is a placeholder (P0)

**Code reality:**

```elixir
# plugins/allbert.browser/lib/allbert_browser/driver/playwright.ex
defmodule AllbertBrowser.Driver.Playwright do
  @moduledoc """
  Playwright driver placeholder.

  The concrete Node/Playwright bridge is exercised in the opt-in
  external-smoke lane. Release tests use AllbertBrowser.Driver.Stub.
  """
  @behaviour AllbertBrowser.Driver

  @impl true
  def verify(_opts), do: {:error, :playwright_bridge_unavailable}
  @impl true
  def start_session(_opts), do: {:error, :playwright_bridge_unavailable}
  @impl true
  def navigate(_state, _url, _opts), do: {:error, :playwright_bridge_unavailable}
  # ...all callbacks return :playwright_bridge_unavailable...
  @impl true
  def close(_state), do: :ok
end
```

Every callback (except `close/1`) returns
`{:error, :playwright_bridge_unavailable}`. No Node sidecar, no CDP
client, no Chromium launch logic exists. The release gate
(`release.v043`) passes only because the Stub driver short-circuits
to `:ok`.

**Operator-observable impact.** An operator following
`docs/operator/browser-and-research.md` and running:

```sh
mix allbert.settings set browser.enabled true
mix allbert.browser doctor
```

…gets a doctor failure (because `Driver.verify(opts)` → Playwright →
`:playwright_bridge_unavailable`). Subsequent `browser_start_session`
calls fail closed (via `Doctor.fresh_ok?`'s `{:error,
{:doctor_not_ok, :unavailable}}` path). The operator cannot research
the web. The `mix allbert.test external-smoke -- browser_research`
task exists as a runnable command (lines 17, 63 of
`apps/allbert_assist/lib/mix/tasks/allbert.test.ex`) but runs against
the same non-functional Playwright placeholder.

**Documentation reality.**

- The operator doc (`browser-and-research.md`) makes the bridge state
  invisible: "Allbert v0.43 ships a policy-bounded browser plugin for
  rendered page research, screenshots, and bounded HTML/text/
  markdown/PDF extraction." No "Playwright bridge implementation
  pending" caveat.
- The developer doc (`browser-plugin.md:22-23`) is closer to honest:
  "The Playwright driver is intentionally exercised only by
  external-smoke work. Release tests stay on the stub driver and
  must not install packages, launch [Chromium]…" — but does not
  explicitly say "the Playwright bridge is a placeholder; rendered
  page research does not function today."
- The Playwright module `@moduledoc` calls itself a "placeholder"
  but no operator-facing surface inherits that signal.
- The roadmap and CHANGELOG describe v0.43 as shipped without flagging
  the placeholder driver.
- The v0.43.x parked-items list (roadmap line 2468-2471) names
  "Windows/WSL2 driver support, persistent profiles + authenticated
  browser operation, headed mode, multi-tab/window orchestration, JS
  evaluation actions, broader document formats" — but does NOT name
  "Playwright bridge implementation" as a deferred item.

**Why this is the v0.42 P0 pattern repeating.** v0.42 audit-02 found
that the trust baseline hash was the wrong quantity (manifest vs.
live tools) — the substrate was correct, the runtime claim was
overstated. v0.43 has the same shape: substrate is correct, runtime
claim ("operator can research the web") is overstated. The
implementation is honest in the dev doc and the module docstring, but
the operator-facing claim is not.

**Fix.** Pick one:

- **(a) Ship the Playwright bridge as v0.43.1.** Treat the placeholder
  as the integration target; v0.43.1 adds the Node sidecar (or BEAM
  CDP client), wires it through `Driver.Playwright`, adds a real
  external-smoke run against macOS + Linux per M0 decision 2,
  CHANGELOG `v0.43.1` section, bump versions to 0.43.1.
- **(b) Explicit park.** Add "Playwright Bridge Implementation" to
  the roadmap v0.43.x deferred list and to
  `docs/plans/future-features.md`; downgrade the operator doc's first
  paragraph from "ships a policy-bounded browser plugin for rendered
  page research" to "ships the policy substrate; the live Chromium
  bridge is a v0.43.1 follow-on." CHANGELOG v0.43.0 §"Changed" or
  §"Known limitations" notes the placeholder. Plan §"Non-Goals" adds
  "live Chromium bridge — placeholder at v0.43; ships in v0.43.1."

Recommendation: **(a) ship in v0.43.1.** The substrate work is done;
adding the bridge is bounded incremental work and unblocks the
operator-facing claim. (b) is a documentation patch that leaves the
feature unshipped.

### G2 — PDF parser scope reduction not in plan body (P2)

**Plan §"Extraction" promises** "bounded HTML, markdown, plain text,
and PDF" without restricting PDF. **Shipped is "text-layer only with
fail-closed on malformed/encrypted/scanned"** (roadmap line 2454-2456;
operator doc line 24-25 calls it "simple PDF text-layer content").

The reduction is sensible (avoids PDF parser RCE surface; avoids OCR
scope creep) and is honestly described in the operator doc and
CHANGELOG. But the plan body (`v0.43-plan.md` §"Extraction") still
reads as if general PDF is shipped.

**Fix.** Plan §"Extraction" + ADR 0040 §"Extraction" updated to read
"text-layer only; encrypted, scanned, and malformed PDFs fail closed."
Already correct in operator doc + roadmap + CHANGELOG; only the plan
+ ADR remain out of sync.

## Closure recommendation

Two paths; pick one.

### Path A — Ship v0.43.1 with the Playwright bridge (recommended)

**Scope:** the Node sidecar (or BEAM CDP client), wire it through
`AllbertBrowser.Driver.Playwright`, real external-smoke run against
macOS + Linux (M0 decision 2), tests, docs.

**Closeout discipline for v0.43.1:** mirror v0.43 M5 — versions ×4 to
0.43.1, CHANGELOG `v0.43.1` section, README + roadmap + vision +
context-map updates. Operator doc updated to "browser research is
operational; doctor verifies live Chromium." External-smoke evidence
recorded.

**Estimated work:** non-trivial (Playwright bridging is real
integration work) but bounded — the entire policy substrate, action
boundary, eval matrix, descriptor, doctor framework, settings shape,
and cache layout are already in place. The bridge slots in.

### Path B — Park the bridge; downgrade the operator claim (faster)

**Doc-only edits:**

1. `docs/operator/browser-and-research.md` first paragraph downgraded
   to "ships the policy substrate, the action surface, and the
   deterministic test gate against the stub driver; the live Chromium
   bridge ships in v0.43.1."
2. `docs/operator/browser-and-research.md` adds a §"Driver Status"
   subsection explaining the placeholder, the
   `:playwright_bridge_unavailable` symptom, and the v0.43.1 ETA.
3. `docs/plans/roadmap.md` §v0.43.x deferred list adds "Playwright
   bridge implementation."
4. `docs/plans/future-features.md` adds a "Playwright Bridge
   Implementation" parked entry (or fold under existing browser
   parking).
5. `CHANGELOG.md` v0.43.0 §"Known limitations" section (or extend
   §"Changed") notes the placeholder.
6. Plan §"Non-Goals" adds "live Chromium bridge — placeholder at
   v0.43; ships in v0.43.1."
7. ADR 0040 §"Ownership" doctor paragraph notes the placeholder.

This is honest disclosure without runtime work. Operators stop being
misled; the substrate ships as documented.

**Recommendation: Path A.** Path B leaves an operator-visible feature
unshipped while claiming the release. Path A ships the feature. The
v0.42 release-discipline lesson is that runtime claims must match
runtime behavior; Path A delivers, Path B explains-away.

If Path A is taken, this audit's G2 (PDF scope reduction in plan body)
and G3 (action count) fold into v0.43.1's closeout pass.
