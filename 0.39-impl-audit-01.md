Control and v0.39b Identity Slot And Active Memory against the active plans,

## Executive Verdict

The v0.39 and v0.39b implementations are substantially present and covered by
focused tests. I did not find evidence of a P0 safety break, missing version
bump, missing action registration, missing identity namespace, missing Active
Memory trace section, or raw Active Memory body leakage into trace metadata.

They are not cleanly release-ready without follow-up. The main gaps are:

- onboarding can mark channel/profile/setup steps complete without verifying
  that the setup actually happened;
- fresh settings still default active intent routing to the remote `fast`
  profile while the plan and onboarding story describe default-local first-run;
- provider doctor diagnostics use dynamic `inspect(reason)` in several paths,
  which drifts from ADR 0047's fixed-catalog and capped-message contract;
- Active Memory chunking and request-time pinning drift from the accepted
  retrieval research note;
- manual acceptance evidence for the real provider/Ollama/cross-OS smoke
  matrix is still pending;
- a few future/vision docs still describe v0.39/v0.39b as planned or name the
  wrong milestone.

## What Is Correct

- v0.39 metadata and docs are present, and v0.39b bumps umbrella/core/web
  versions to `0.39.1`.
- `doctor_model_profile`, `set_active_model_profile`, and
  `onboarding_step_complete` are registered actions and the focused security
  evals pass.
- The provider doctor has separate `credentialed_remote` and `local_endpoint`
  branches, uses `Req`, redacts host to host-only output, and does not auto-pull
  Ollama models.
- `model_profiles.local.model` is `llama3.2:3b`.
- v0.39b adds `:identity` as the fifth memory category, declares the system
  namespace outside the app-id validator, and parses manually edited identity
  files as `origin: system`, `namespace: identity`.
- `retrieve_active_memory` is registered, read-only, deterministic under pinned
  inputs, and bounded by `active_memory.top_k` and
  `active_memory.chunk_max_bytes`.
- Direct-answer model mode retrieves Active Memory after route selection; the
  classifier tests assert raw chunks are not fed to classifier input.
- Trace rendering places `## Active Memory` after `## Intent Candidates` and
  before `## Memory Review`, and drops chunk bodies from trace metadata.
- No embeddings, vector indexes, learned ranking, model-generated identity, or
  permission authority from identity memory landed in v0.39b.

## Findings

### P1 - Fresh first-run default still points at remote `fast`

Evidence:

- v0.39 plan says the operator can accept default `local_ollama` and default
  `local` model profile during onboarding
  (`docs/plans/v0.39-plan.md:111-124`).
- `apps/allbert_assist/lib/allbert_assist/settings/schema.ex:1708-1720`
  still defaults `intent.model_profile` and
  `intent.direct_answer_model_profile` to `fast`.
- Existing tests assert the old default
  (`apps/allbert_assist/test/allbert_assist/settings_test.exs` asserts
  `intent.model_profile == "fast"`).

Risk:

A disposable Allbert Home does not naturally start from the local Ollama path
that v0.39 presents as the first-run default. The user can fix it by running
`mix allbert.model use local`, but the default-local onboarding story and the
actual active profile disagree.

Fix:

- Decide whether `intent.model_profile` should change to `local` for v0.39
  first-run, or whether the docs should explicitly say the active default
  remains `fast` until the operator chooses `local`.
- If changing code, update schema defaults and tests. Consider leaving
  `intent.direct_answer_model_profile` as `fast` if direct-answer model mode is
  intentionally remote-gated and disabled by default; document that split.
- Add a focused assertion that a fresh Allbert Home plus `mix allbert.model
  list` presents the intended default story.

### P1 - Optional channel onboarding can complete without channel setup

Evidence:

- v0.39 plan says existing channel task surfaces are invoked when the operator
  opts into a channel (`docs/plans/v0.39-plan.md:72-77`) and channel credential
  writes are recorded under onboarding (`docs/plans/v0.39-plan.md:129-132`).
- CLI `mix allbert.onboard channel telegram|email` only records the onboarding
  step as completed (`apps/allbert_assist/lib/mix/tasks/allbert.onboard.ex:73-81`).
- Workspace channel buttons do the same
  (`apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/onboarding.ex:31-40`).
- Workspace guidance tells the operator to run `mix allbert.channels ...` and
  then record the choice, but the UI does not verify that happened
  (`apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/onboarding.ex:264-266`).

Risk:

The onboarding objective can claim a channel was completed when no Telegram or
email credential exists. This weakens the "guided first-run" contract and makes
release validation ambiguous. It also leaves a gap between v0.39 onboarding and
the roadmap's remote-channel acceptance item.

Fix:

- Make `channel telegram|email` check existing channel credential/status before
  completing the step, or record a distinct "selected / pending configuration"
  result instead of `completed`.
- In workspace, disable or relabel Telegram/email completion until channel
  status is configured, or route to Settings/channel controls and re-check.
- Add tests for "select telegram without token does not complete as configured"
  and "configured telegram/email can complete the optional channel step".
- Keep `channel none` as a valid skipped outcome.

### P1 - Provider doctor diagnostics drift from ADR 0047 fixed-catalog policy

Evidence:

- ADR 0047 says doctor diagnostics must be capped at 256 bytes and rendered
  from a fixed catalog, with no dynamic raw-input pass-through
  (`docs/adr/0047-provider-doctor-contract.md:87-103`).
- The action fallback returns `inspect(reason)` in the operator-facing message,
  diagnostic, and action metadata
  (`apps/allbert_assist/lib/allbert_assist/actions/settings/doctor_model_profile.ex:86-96`).
- Model doctor branch diagnostics interpolate `inspect(reason)` for host and
  URL failures
  (`apps/allbert_assist/lib/allbert_assist/settings/model_doctor.ex:64-77`,
  `119-132`).

Risk:

The common success/failure fixtures pass, but unusual settings/secret/provider
errors can bypass the fixed-catalog doctor contract. This matters because ADR
0047 is intended to become a v1.0 Tier-1 freeze contract and later doctors
reuse the pattern.

Fix:

- Replace fallback and branch diagnostics with fixed catalog messages keyed by
  diagnostic code.
- Cap every diagnostic message at 256 bytes at construction time.
- Redact and cap internal action metadata errors, or store only a diagnostic
  code plus safe category.
- Add tests for unknown profile, invalid provider base URL with credentials or
  query, secret-store errors, and unexpected `Req` errors.

### P2 - Active Memory chunking does not match the accepted research note

Evidence:

- The research note says top-K selection should take top chunks and then trim
  each body to `active_memory.chunk_max_bytes`, appending an ellipsis only when
  truncated (`docs/research/active-memory-retrieval.md:114-131`).
- Current implementation splits each entry body into multiple grapheme windows
  before scoring (`apps/allbert_assist/lib/allbert_assist/memory/active_memory.ex:94-111`,
  `178-222`).
- Existing tests assert only that returned bodies are under the byte cap; they
  do not assert the truncation/ellipsis contract
  (`apps/allbert_assist/test/allbert_assist/memory/active_memory_test.exs:132-148`).

Risk:

The current behavior is deterministic and bounded, but scoring semantics differ
from the accepted baseline. A long memory entry can contribute multiple
independent candidates, and truncated prompt context has no marker. Operators
replaying retrieval decisions will see a different model than the research doc
describes.

Fix:

- Choose one contract:
  - If one candidate per memory entry is intended, change implementation to
    score the entry body and trim only selected bodies with a deterministic
    truncation marker.
  - If multi-window chunking is intended, update the research note, v0.39b
    plan/request-flow docs, and operator docs to explicitly bless pre-score
    windowing and "no ellipsis" behavior.
- Add tests for long entries, chunk count, tiebreaking, and truncation marker
  behavior.

### P2 - Active Memory `now` is not clearly pinned at request start

Evidence:

- The research note requires `now` to be the request timestamp pinned at request
  start (`docs/research/active-memory-retrieval.md:46-58`).
- Direct-answer model mode sets `now` immediately before calling
  `retrieve_active_memory`
  (`apps/allbert_assist/lib/allbert_assist/actions/intent/direct_answer.ex:125-132`).
- Tests prove determinism by passing a pinned `now` directly to retrieval, but
  do not prove the runtime/direct-answer path uses a request-start timestamp
  (`apps/allbert_assist/test/security/active_memory_eval_test.exs:340-407`).

Risk:

For normal turns this is probably harmless, but it is a spec drift and leaves
same-turn replay sensitive to where the retrieval call happens inside the
direct-answer action.

Fix:

- Prefer a request timestamp from runtime context when present.
- Otherwise pin `now` once at direct-answer action entry and reuse it for the
  retrieval call and trace metadata.
- Add a direct-answer test that injects a request timestamp and asserts
  `retrieve_active_memory` receives that exact value.

### P2 - Manual release validation evidence is still pending

Evidence:

- v0.39 request-flow smoke requires real/disposable Allbert Home onboarding,
  Anthropic credentialed doctor, Ollama missing/present model checks,
  endpoint-kind override, model-assist toggle behavior, workspace resume, and
  macOS/Linux/Windows-WSL2 smoke (`docs/plans/v0.39-request-flow.md:93-133`).
- v0.39b request-flow smoke requires manual identity file review, direct-answer
  trace inspection, disabled-model skip, replay, snapshot, cross-thread/app
  checks, classifier exclusion, and CLI list/retrieve checks
  (`docs/plans/v0.39b-request-flow.md:109-132`).
- The plans currently say "ready for operator manual validation", not "manual
  validation complete".

Risk:

Automated fixtures cover the core behavior, but the release acceptance matrix
cannot be closed without the real environment smoke rows, especially
cross-OS and real provider/Ollama paths.

Fix:

- Add a short checked-in manual validation record or release checklist for
  v0.39/v0.39b with date, OS, commands, profile, and pass/fail evidence.
- Run macOS, Linux, and Windows/WSL2 first-run smoke before tagging.
- Run real Anthropic or another remote profile through ADR 0047 doctor and run
  local Ollama missing/present model checks.
- For any UX/UI changes made during fixes, verify with Chrome extension browser
  control as requested for this release workflow.

### P3 - Documentation drift in future/vision docs

Evidence:

- `docs/developer/runtime-boundary-map.md:69-78` still places v0.39/v0.39b
  under "Post-v0.38 Planned Facades" and says they are expected, not
  implemented.
- `docs/plans/allbert-jido-vision.md:304-308` says v0.39b Active Memory is
  "planned"; `docs/plans/allbert-jido-vision.md:622-623` says "Active Memory
  in v0.39" rather than v0.39b.
- `docs/plans/v0.39-plan.md:129-133` still describes the identity preview doc
  as a v0.39b stub, although v0.39b has now shipped the actual operator doc.

Risk:

This is not a runtime bug, but it violates the roadmap/continuity goal of the
audit and can mislead the next milestone agent.

Fix:

- Move v0.39 and v0.39b rows in the runtime boundary map from "planned" to
  implemented/current boundary notes, or mark only v0.40+ as planned.
- Update `allbert-jido-vision.md` to say v0.39b Active Memory is implemented
  as the deterministic precursor and correct the v0.39/v0.39b split.
- Update the v0.39 identity-preview wording to point at the implemented
  `docs/operator/active-memory.md`.

## Questions For Resolution

   `fast` remains the active default until onboarding changes it?

   before the step can complete, or is "selected for later" an acceptable

   should it be changed to the research note's post-selection trim contract?

## Release Readiness State

Current state after this audit: implementation is mostly complete and focused
tests pass, but release should wait for the P1/P2 fixes above or explicit
operator decisions accepting the current behavior with doc updates. The safest
next step is Milestone A, then Milestone B, then docs/manual validation closeout.
