## Executive Verdict

The v0.39 and v0.39b implementations are present, registered, focused-tested,
warning-clean, and Credo-clean. No P0 and no genuine P1 safety/correctness break
was found:

- All four new actions (`doctor_model_profile`, `set_active_model_profile`,
  `onboarding_step_complete`, `retrieve_active_memory`) are registered and
  route through `Actions.Runner.run/3` + Security Central.
- The provider doctor has solid SSRF protection (metadata-host denial, private-
  host policy per branch, no URL userinfo, no redirects), host-only redaction,
  and never returns raw secret values.
- `:identity` is the fifth `Memory` category; the system namespace is declared
  outside app-id validation (`AllbertAssist.Memory.SystemNamespaces`,
  `origin: :system`, `app_id: nil`); `Memory.upsert_system_entry/1` stamps
  system provenance.
- Active Memory retrieval is read-only, deterministic under pinned inputs,
  `review_status: :kept`-only, bounded by `top_k` + `chunk_max_bytes`, gated by
  `intent.direct_answer_model_enabled`, excluded from the intent classifier
  input, and renders a body-free `## Active Memory` trace section in the correct
  position.
- No embeddings, no learned ranking, no pin-score boost, no model-generated
  identity content, and no permission authority from identity memory.

It is **not cleanly release-ready** without a small remediation pass. The
remaining issues are accuracy/contract-drift class, not safety class. Honest
severity is lower than audit-01 framed two of its P1s (see re-rating notes).

## Findings

Severity legend: P0 safety/data break · P1 correctness break · P2
contract/accuracy drift · P3 cosmetic/forward-continuity.

### P2-1 — Provider doctor still passes raw `inspect(reason)` in several diagnostics (ADR 0047 §3)

ADR 0047 §3 requires diagnostic messages "capped at 256 bytes per diagnostics
entry, rendered from a fixed catalog of strings (no dynamic raw-input
pass-through)". The doctor now uses fixed-catalog messages with codes for the
common paths, and `safe_reason/1` caps transport-error suffixes at 80 bytes —
that is a real improvement over audit-01. But several paths still interpolate
uncapped `inspect(reason)`:

- `apps/allbert_assist/lib/allbert_assist/settings/model_doctor.ex`
  - `:credential_unavailable` → `"… could not be read: #{inspect(reason)}."`
    (line 115-116) — reason comes from the secret store; uncapped.
  - `:invalid_provider_base_url` → `inspect(reason)` (lines 75-77, 130-132).
  - `:provider_host_denied` → `inspect(reason)` (lines 68-70, 122-124).
- `apps/allbert_assist/lib/allbert_assist/actions/settings/doctor_model_profile.ex`
  - action-level `error/3` builds both the operator message and the diagnostic
    from `inspect(reason)` with no cap (lines 88, 91).

Risk: common success/failure fixtures pass, but unusual settings/secret/provider
errors bypass the fixed-catalog + 256-byte contract. ADR 0047 is a v1.0 Tier-1
freeze candidate and v0.40/v0.47/v0.48 doctors are specified to reuse this
shape, so drift here propagates. No secret *value* leaks today (secret-store
errors are refs/atoms, not the secret), but the contract is not met.

Fix:

- Map every diagnostic to a fixed catalog message keyed by code; drop
  `inspect(reason)` from operator-facing strings.
- Enforce a 256-byte cap at `diagnostic/2` construction.
- In the action `error/3`, render a fixed message + safe code instead of
  `inspect(reason)`; keep the raw reason only in non-operator-facing internal
  metadata if needed, capped.
- Add eval rows for: unknown profile, invalid provider base URL (with query),
  secret-store read error, and unexpected `Req` error — each asserting bounded,
  catalog-only output.

### P2-2 — Active Memory chunking/ellipsis contract is a 3-way split (impl vs plan vs research note)

Three artifacts now describe chunking three different ways:

- **Implementation** (`memory/active_memory.ex` `chunks_for_entry/2` +
  `chunk_text/2`): pre-score multi-window — each entry body is split into
  consecutive `chunk_max_bytes` byte-windows *before* scoring, so a long entry
  yields multiple independently-scored candidates. **No ellipsis** is appended.
- **v0.39b plan** (`docs/plans/v0.39b-plan.md:151-153`): "chunks each
  `Memory.Entry` body into `chunk_max_bytes` byte windows at scoring time
  (deterministic; truncate at byte boundary; append `…` only if truncated)."
  Matches impl on windowing, **contradicts impl on the ellipsis**.
- **Research note** (`docs/research/active-memory-retrieval.md:114-131`, Top-K
  Selection step 8): "Take top K. Trim each chunk body to `chunk_max_bytes` …
  append ellipsis only if truncated." Describes **one chunk per entry,
  post-selection trim** with an ellipsis — contradicts impl on both the chunk
  model and the ellipsis.

The research note is cited by the plan as the binding spec ("Spec'd in
`docs/research/active-memory-retrieval.md`"), so an operator replaying retrieval
from the note expects different behavior than ships. The test
(`active_memory_test.exs:147-148`) only asserts `length(chunks) == 1` and
`byte_size <= 128`; it does not assert chunk model or ellipsis, so it passes
under any of the three readings.

Fix (recommended: make docs match shipped behavior; it is deterministic and
tested):

- Update the research note Top-K step to describe pre-score byte-window
  chunking (multiple candidates per entry), and either remove the ellipsis
  language everywhere or implement it.
- Decide the ellipsis: either drop "append `…`" from plan line 153 + research
  note step 8 (doc-only), **or** append a deterministic truncation marker in
  `chunk_text/2` (small code change) and add a test. Pick one; do not leave the
  docs promising a marker the code never writes.
- Add a long-entry test asserting the chosen chunk count + (if implemented)
  the truncation marker.

### P2-3 — Onboarding marks channel/selection steps "completed" without verifying setup

`mix allbert.onboard channel telegram|email`
(`mix/tasks/allbert.onboard.ex:73-81`) and the workspace channel buttons record
`optional_channel_registration` as `completed` with a note, with no check that a
Telegram token or email password exists. `Onboarding.complete_step/4` and
`Actions.Onboarding.StepComplete` only transition the step status; they do not
verify the underlying action happened. The same is true for selection-only
steps (pick provider / pick model): the CLI/workspace can record them complete
independent of whether `set_active_model_profile` / `set_provider_credential`
actually ran.

Re-rated to P2 (audit-01 said P1): marking an objective step complete grants no
authority, writes nothing false to Settings, and the channel adapter still
requires its own credential to function. The defect is that the *guided-setup
progress tracker* can claim completion that did not happen — an accuracy gap in
the v0.39 "guided first-run" contract and a soft tie-in to the roadmap's
remote-channel acceptance item.

Fix:

- For `channel telegram|email`, check the channel credential/status before
  recording `completed`; otherwise record a distinct outcome (e.g.,
  `selected_pending_configuration`) or `skipped`.
- In workspace, disable/relabel channel completion until status is configured.
- Optionally, make `pick_model_profile` completion require that
  `set_active_model_profile` ran in the same objective (e.g., gate on a recorded
  step event), so step completion reflects real writes.
- Tests: "select telegram without token does not complete as configured" and
  "configured telegram/email completes the optional channel step".

### P2-4 — Shipped active intent profile defaults to remote `fast`, not the plan's default-local story

`schema.ex:1709-1711,1719` still defaults `intent.model_profile = "fast"` and
`intent.direct_answer_model_profile = "fast"` (→ `openai`, which ships
`enabled: false`). The v0.39 plan's onboarding steps 2/5 present the default as
`local_ollama` / `local`.

Re-rated to P2 (audit-01 said P1): in a fresh Home this is mostly inert because
`intent.model_assist_enabled` and `intent.direct_answer_model_enabled` both
default `false`, so the `fast`/openai profile is not actually invoked until the
operator turns a model path on — and onboarding step 5 writes `local` when the
operator accepts defaults. The defect is a narrative/consistency mismatch
between the plan ("default local") and the shipped pre-onboarding default
("fast"), plus a latent foot-gun if an operator enables a model path without
running onboarding.

Fix (decision required — see Questions):

- Either change `intent.model_profile` (and possibly
  `intent.direct_answer_model_profile`) default to `local`; update
  `settings_test.exs` which asserts `"fast"`; or
- Keep `fast` and update the v0.39 plan + onboarding copy to state explicitly
  that the active default remains `fast` until the operator selects `local`,
  and that model paths are off by default.
- Add an assertion pinning whichever default story is chosen.

### P3-1 — Active Memory `now` is set at the direct-answer call site, not pinned at request start

`actions/intent/direct_answer.ex:131` sets
`now: DateTime.utc_now() |> truncate(:second)` at the moment it calls
`retrieve_active_memory`, rather than reusing a request-start timestamp from
context. The research note (line 57) says `now` is "the request timestamp pinned
at request start … so concurrent scorers in the same turn see the same `now`."

Re-rated to P3: there is exactly one retrieval call per turn (no concurrent
scorers), and determinism within a turn holds, so this is spec-wording drift
rather than a correctness bug. Same-turn replay is sensitive to call-site timing
only in the theoretical multi-scorer case the note describes.

Fix: prefer a `request_timestamp`/`now` from runtime context when present;
otherwise pin once at direct-answer action entry and reuse for retrieval + trace.
Or soften the research-note wording to "pinned at retrieval invocation (single
scorer per turn)". Add a direct-answer test that injects a request timestamp and
asserts retrieval receives that exact value.

### P3-4 — Non-exhaustive `case` in `mix allbert.onboard`

`mix/tasks/allbert.onboard.ex:161-166` `completed_action/3` matches only
`{:ok, %{status: :completed}}` and `{:ok, response}`. `Runner.run/3` can return
`{:error, …}` (`runner.ex:29,87,90`). Today `onboarding_step_complete` always
returns `{:ok, …}` (StepComplete wraps all branches), so this is not currently
reachable, but a future raise inside the action would surface as a
`CaseClauseError` rather than a clean operator error.

Fix: add an `{:error, reason} -> {:error, reason}` clause (or a catch-all) so
the existing `fail!/2` path renders a clean message.

### P3-5 — Manual release validation evidence still pending

Both request-flow docs say "ready for operator manual validation before release
tagging." The smoke matrices (real Anthropic doctor, Ollama missing/present
model, endpoint-kind override, model-assist toggle, workspace resume,
macOS/Linux/Windows-WSL2, identity-file review, direct-answer trace, classifier
exclusion, replay) are not recorded as executed. Automated fixtures cover the
behavior, but the v1.0 acceptance matrix items 1–2 (cross-OS first-run; choose
any of local/OpenAI/Anthropic/OpenRouter) cannot be signed off from unit tests
alone.

Fix: add a short checked-in manual validation record (date, OS, commands,
profile, pass/fail) and run real-provider + cross-OS smoke before tagging
`v0.39.1`.
