## Feedback items

### 1. Floating tool panel over the chat panel — bad UX, needs a rework
- **Where:** Workspace page, `127.0.0.1:4061/workspace?destination=workspace:settings` (navigated via sidebar TRUST → Settings; also reachable for other tool panels).
- **What was seen (screenshot provided):** The Settings tool panel ("Settings — Workspace tool panel" containing Settings Central) renders as a floating window overlapping the main Chat panel. It covers most of the chat (messages and the "Ask Allbert anything" composer are partially hidden underneath), while the left half of the workspace canvas sits empty.
- **Operator verdict:** "I don't think the floating window over the main chat window/panel looks right — you need to figure out a better UX/UI for this."
- **Ask:** Design a better presentation for workspace tool panels (e.g., docked side-by-side layout / dedicated pane instead of an overlay floating over chat). No specific solution mandated — UX proposal wanted.
- **Screenshot notes:** Panel shows "0/64 tiles" badge, a collapse (⌘-style) button top-right, Settings Central list (acp_server.* keys), Key/Value editor with Save, and a detail readout — all inside the floating panel at ~55% viewport width, right-aligned, overlapping chat.

### 2. Chat message typography hierarchy inverted / wrong sizing
- **Where:** Workspace page, Chat (Runtime conversation) panel — message bubbles ("You" and "Allbert" messages).
- **What was seen (screenshot provided):** Within a message bubble, the visual hierarchy reads wrong. The timestamp ("2m ago") renders larger/heavier than the actual message text, which is smaller monospace. Sender labels ("You", "Allbert") are probably fine per operator.
- **Operator verdict:** "The font size / bold etc in the chat box look inverted or wrong order… '2m ago' looks bigger than the actual text… all of this needs to be right sized."
- **Ask:** Fix the type scale inside chat messages so hierarchy is: message body most prominent, sender label secondary, timestamp smallest/most muted. Timestamp must not out-weigh body text.
- **Screenshot notes:** Body text is monospace (~same size as label), "2m ago" appears in a larger regular-weight sans font below the body; same pattern on both user and assistant messages. Continuity summary rows at top of chat look fine by comparison.

### 3. Left nav and workspace submenu should be collapsible (hamburger)
- **Where:** (a) Main left navigation sidebar (Allbert brand + START/WORK/OPERATE/EXTEND/TRUST sections: Home, Workspace, Objectives, Jobs, Models, Channels, Settings, Trust). (b) The workspace page's own left submenu column (Workspace +, CONVERSATIONS, OUTPUT, APPS: Artifacts/Browser/Notes-files/Research/StockSage, WORKSPACE: Onboard/Create/Plan-Build…).
- **Operator verdict:** "The left menu should be collapsible with a hamburger icon.. same with the submenu on the left of the workspace."
- **Ask:** Both columns get a collapse/expand affordance (hamburger icon) so the operator can reclaim horizontal space — main sidebar collapsible, and independently the workspace-local submenu collapsible.

### 4. Conversation threads should be renamable
- **Where:** Workspace page, CONVERSATIONS list in the workspace-local left submenu (thread entries currently show as e.g. "Workspace s… thr_68b…" — truncated auto-generated id/title).
- **Operator verdict:** "We should be able to rename the threads in the conversation list."
- **Ask:** Add a rename affordance for conversation threads (operator-editable title persisted with the thread), so threads aren't identified only by truncated auto titles/ids.

### 5. "Running" button in chat navigates to Objectives view — intended? Naming inconsistent
- **Where:** Workspace page, chat window header area — the status button (shows agent/run state, e.g. "running"; same slot showed "Neutral" in earlier screenshot). Clicking it navigates to the Objectives view.
- **Operator verdict:** "When I click on 'running' button in the chat window it leads to the objectives view.. is that intended? Then the naming needs to be consistent."
- **Ask:** Two-part: (a) confirm/decide whether the status chip should deep-link to Objectives at all; (b) if yes, make the naming consistent — a control labeled with a run state ("running"/"Neutral") giving no hint it opens Objectives is a mismatch. Either label it as Objectives-related or route it to a run/status view.
- **Note:** Whether this navigation is intended is unresolved — to be researched at triage time (operator deferred investigation during capture).

### 6. Workspace shouldn't need its own nested view — fold its submenu into the main left sidebar
- **Where:** Workspace page — the workspace-local left submenu column (Workspace +, CONVERSATIONS, OUTPUT, APPS, WORKSPACE sections) that renders as a second nested navigation column beside the main product sidebar.
- **Operator verdict:** "I don't see why workspace has to have its own nested view. Why couldn't we think about adding nested submenus for workspace under the main left subpanel, that contextually expands and highlights the right workspace context."
- **Ask:** Rethink the two-column navigation: instead of the workspace carrying its own separate submenu column, explore nesting the workspace sections as expandable submenus under the Workspace entry in the main left sidebar — contextually expanding when in a workspace and highlighting the active workspace context. Questions the premise of the nested view, not just its collapsibility.
- **Note:** Overlaps item 3 (which asked to make both columns collapsible) — if the submenu merges into the main sidebar, item 3's second half may dissolve into this. Also interacts with item 1 (workspace shell/panel presentation). Triage together.

### 7. Shell top bars just take up space — fold their contexts into the left-panel contextual menu
- **Where:** Workspace shell top bar — and shell top bars generally (any other shell with its own top bar).
- **Operator verdict:** "The top bar for the workspace shell, or any other shell, just takes up space. Those contexts should be folded under the correct left panel contextual menu. This gives more space for chat, canvas etc."
- **Ask:** Remove/eliminate the per-shell top bar as a persistent horizontal band; whatever context it carries (titles, status, controls) should relocate into the appropriate left-panel contextual menu instead. Goal is reclaiming vertical space for the primary content — chat, canvas, etc.
- **Note:** Compounds items 3 and 6 into a broader navigation-consolidation theme: one contextual left panel owns navigation + context, top bars go away, and horizontal/vertical space returns to content. Triage together with 1/3/6.

### 8. Dark mode color scheme should be more subtle (light mode is good)
- **Where:** Global — the dark-mode color scheme across the product (Direction C visual language).
- **Operator verdict:** "The light mode color scheme is good. The dark mode scheme maybe needs to be a little more subtle."
- **Ask:** Tone down the dark-mode palette — keep light mode as-is; adjust dark-mode tokens toward something subtler. No specific colors mandated; a refinement pass on the dark token set, preserving contrast/a11y floors.
- **Note:** Token-level change within the v0.60b Direction C language, not a re-choice of direction. A11y axes (contrast, high-contrast mode) must hold through any adjustment.
