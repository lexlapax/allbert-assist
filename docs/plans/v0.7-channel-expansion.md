# v0.7 — Channel expansion

Status: Proposed

## Context

v0.2's daemon (ADR 0018) makes attachable channels a first-class idea (ADR 0013), but in practice only the REPL/CLI channel has existed. v0.6 hardens the v0.5 substrate (session durability, cost cap enforcement, staged-memory review polish, and memory verification). v0.7 turns the channel abstraction into reality by adding one new channel — Telegram as the pilot — while formalising the `Channel` trait so every future channel (Discord, WhatsApp, web, email, SMS) fits the same shape.

Channels differ from a REPL in important ways: they are asynchronous, they have their own presentation idioms, they sometimes lack any confirmation path at all, and each one has its own trust story. The Channel trait must encode those differences as capability flags rather than forcing every channel to pretend it is a terminal.

v0.7 also folds in four retrospective findings that naturally belong alongside the channel work:

- Normalising the **two-tier tool world** (kernel-native vs registry) so hooks observe every tool call uniformly. Channels without direct REPL supervision need uniform policy observability.
- Gating **background web-learning** behind explicit user intent. Channels make this urgent: a Telegram user saying "look that up" should stage research with the same privacy posture as a REPL user.
- Replacing the **one-level sub-agent cap** with a budget-governed model. Channel-driven research workflows need depth; nesting count is a crude proxy.
- **Intent-guided routing that does real work** (not just labels): channels with sparse turns need the intent step to shape prompt construction, agent choice, confirmation style, and tool preference, not just annotate the trace.

## Goal

Deliver the `Channel` trait and its capability model, ship the Telegram pilot, normalize the tool surface so hooks are uniform, wire explicit-intent web-learning, lift sub-agent depth to a budget-governed bound, and upgrade intent routing from label to real routing defaults without turning it into a hard gate.

## Sketch

### Channel trait

The `Channel` trait sits at the daemon-channel boundary. Each channel is:

- a small adapter that speaks the channel's protocol (Telegram bot API, Discord webhook, etc.) and
- a bridge into the kernel's session surface (create/attach/detach/end, deliver input, surface output).

Capability flags on each channel declare what it can and cannot do:

- `supports_inline_confirm` — can present a blocking confirmation prompt.
- `supports_async_confirm` — can ferry confirmation through an out-of-band path.
- `supports_rich_output` — markdown / code blocks.
- `supports_file_attach` — inbound files from the channel.
- `supports_image_input` / `supports_image_output` — images in either direction.
- `supports_voice_input` / `supports_voice_output` — voice notes / TTS output.
- `supports_audio_attach` — audio file attachments beyond voice notes.
- `max_message_size` — for chunking long outputs.
- `latency_class` — synchronous (REPL) vs asynchronous (Telegram, email) vs batch (SMS).

The kernel consults capability flags when constructing a turn:

- `confirm-trust` (ADR 0007) interacts differently: synchronous channels block; async channels emit a pending-approval notification and the agent turn suspends until approval lands. If a channel lacks any confirm capability at all, policy-requiring actions fail closed.
- Scheduled jobs (ADR 0015) already fail closed on interactive actions — the capability model formalises why.
- `memory.surface_staged_on_turn_end` (v0.5, refined in v0.6, ADR 0050) adapts: synchronous channels render inline, async channels append to the next outbound message, batch channels may omit.

### Per-channel trust model

- Each channel has its own allowlist (Telegram chat IDs, Discord user IDs, future email addresses) stored alongside `~/.allbert/config`.
- Unknown senders are ignored with an audit log entry. No channel messages create sessions unless the sender is explicitly allowed.
- Channel tokens / credentials live in `~/.allbert/secrets/` with the same filesystem permissions as v0.2's IPC socket (ADR 0023).

### Telegram pilot

- Outbound: rich output via Telegram MarkdownV2.
- Inbound: text messages become session turns, with file attachments staged for tool read.
- Async confirm: a bot reply with reaction emojis or a `/approve`/`/reject` sub-command.
- Async cost-cap override (v0.6 M3 is REPL-only): a Telegram `/override <reason>` reply to a cap-refusal message applies for one turn.
- Rate limiting: the daemon enforces a minimum interval between agent turns per chat, independent of any Telegram-side rate limit.
- Session model: one long-lived session per allowlisted Telegram sender by default. Messages from the same sender reattach to that sender's session unless the user explicitly resets it. Session durability from v0.6 M2 means Telegram threads survive daemon restart.
- Minimum v0.7 capability contract: text, async confirm, rich output, and file attachments are required. Voice notes and images are explicit stretch goals; if they do not land cleanly in the pilot, they remain deferred rather than being implied by the release name.

### Multimodal I/O

Consistent with ADR 0038's commitment that end users interact via natural language across text and other media:

- Multimodal content (voice notes, images, audio attachments) flows through the channel adapter to the kernel and out to providers that support multimodal input. Providers that do not support a given modality either cause the channel to transcode (e.g. voice → transcript via a transcription skill) or refuse with a clear message.
- Channels declare their modality capabilities through the flags above. The kernel surfaces those flags to skills so a skill can gate behaviour (e.g. "this analysis requires an image" fails fast on a text-only channel).
- Outbound multimodal output is symmetric: channels that declare `supports_voice_output` can receive TTS'd replies; channels without it receive text.
- Media attachments are staged as files under the session's working directory with the same exec-policy-scoped paths used elsewhere in the runtime. No prompt-level embedding of raw binary content.
- Privacy: media received through a channel is treated like any other session content — it does not auto-promote to durable memory. Explicit save flows remain the only path into durable storage.

### Tool-surface normalization

Today the kernel has a two-tier tool world: `ToolRegistry::builtins()` holds six tools (`process_exec`, `read_file`, `write_file`, `request_input`, `web_search`, `fetch_url`); everything else (memory ops, skill ops, sub-agent spawning, reference/script access) is dispatched directly in the agent loop. `BeforeTool`/`AfterTool` hooks observe only the six registry tools.

v0.7 lifts the kernel-native handlers into `Tool` implementations:

- Memory tools: `read_memory`, `write_memory`, `search_memory`, `stage_memory`, `list_staged_memory`, `promote_staged_memory`, `reject_staged_memory`, `forget_memory`.
- Skill tools: `list_skills`, `invoke_skill`, `create_skill`, `read_reference`, `run_skill_script`.
- Agent tools: `spawn_subagent`.

Kernel state (memory indices, skill store) is injected via `ToolCtx` extensions rather than captured in closure state. Hooks then observe every tool call without exception. This matters most for channels: operators supervising a Telegram session expect policy gates to apply to memory writes the same way they apply to process exec.

Covered by ADR 0052.

### Background web-learning under explicit intent

`web_search` and `fetch_url` gain an optional `record_as: "<summary>"` parameter. When the agent populates it (in response to explicit user phrases like "remember this" or "look that up and save it"), the result stages as a memory entry with `kind: research`, `source_url`, `summary`, and `fetched_at`. Without `record_as`, behaviour is unchanged from v0.5.

No tool path auto-stages without explicit parameter. Privacy-by-default: a search the user asks about but not to remember stays ephemeral. `research`-kind entries route through the same 5-per-turn / 90-day / dedup limits as any other staged memory (ADR 0047).

Covered by ADR 0053.

### Sub-agent depth by budget

v0.5 bounds sub-agent spawning to one nested level. v0.7 replaces that with a budget:

- Each sub-agent inherits `remaining_budget_usd` and `remaining_budget_s` from its parent.
- Further spawns subdivide the same budgets.
- Spawn refuses with `budget-exhausted` when either budget reaches zero.
- Root-turn budgets come from `limits.max_turn_usd` (default $0.50) and `limits.max_turn_s` (default 120). `/cost --turn-budget` and `/cost --turn-time` adjust per-turn.

Covered by ADR 0054.

### Intent-guided routing defaults

v0.3 classified intent into `{task, chat, schedule, memory_query, meta}` but used the label only for telemetry and `/status`. v0.7 makes intent operational without turning it into a hard gate:

- `chat`: default conversational prompt, lower side-effect bias, and a stronger preference for staying in natural-language explanation mode unless the user explicitly asks the assistant to act.
- `task`: full problem-solving prompt, normal tool ordering, and default sub-agent budget.
- `schedule`: scheduler-oriented prompt, preferred routing toward job-management and durable-mutation review flows, and confirmation language that foregrounds durable effects.
- `memory_query`: retrieval-first prompt, preferred routing toward memory tools, and concise evidence-oriented response shaping.
- `meta`: operator/meta prompt, strong preference for status/setup/cost surfaces, and side-effect-avoiding guidance.

Intent resolution still falls back to `default_intent` when classification is low-confidence; fallback always behaves as `task` to avoid regressing the v0.3 UX.

No intent removes tools from the runtime. Tools remain available subject to the existing policy envelope, skill fences, and agent judgment. This preserves ADR 0030's accepted "intent is a hint, not a hard gate" rule while still making intent matter in practice.

## Runtime defaults

- **Async confirm timeout**: pending async confirmations expire after a bounded timeout and fail closed by default.
- **Trace attribution**: channel-originated turns are tagged with channel kind and channel-local sender identifier in the trace surface.
- **`HEARTBEAT.md`**: remains deferred. Channel rhythm alone is not enough reason to add another bootstrap artifact — revisit only if async channels produce demonstrable operator pain.
- **Session durability** (v0.6 M2): every channel session is journaled. Telegram threads survive daemon restart.

## Expected ADRs

- **ADR 0052** — Kernel-native operations promote to `Tool` implementations (normalises hook observation).
- **ADR 0053** — Background web-learning stages only under explicit user intent.
- **ADR 0054** — Sub-agent depth is governed by cost/time budget, not nesting count.
- `Channel` trait with capability flags (in-plan for now; promote to ADR if trait shape is contested).
- Async confirm UX and the "suspend until approval" turn pattern.
- Per-channel allowlist as the v0.7 trust model.
- Telegram as a reference channel implementation.
- *Deferred, tracked here for v0.9+*: skill-format adapter that imports legacy / MCP / OpenClaw shapes into AgentSkills format.

## Dependencies

- v0.6 hardening — session durability, cost cap enforcement, and staged-memory notice all become cross-channel concerns here. v0.7 depends on v0.6 landing first.
- v0.5 curated memory — channels with sparse turns need cheap context assembly; ADR 0053 builds on staging schema.
- v0.3 intent routing — channel input goes through the same intent step as REPL input, now with dispatch.
- ADR 0013 (channels) and ADR 0018 (daemon host) as runtime substrate.

## Out of scope for v0.7

- Full web UI (deferred).
- Email and SMS channels (follow Telegram pilot into their own tracking once the trait is stable).
- Cross-channel conversation merging (e.g. starting a turn in Telegram and continuing in REPL).
- Network-addressable daemon (out of scope through v0.9; deferred per [roadmap.md](roadmap.md)).
- Cross-channel continuity and sync (scheduled for v0.8 rather than bundled into v0.7).

## Remaining planning questions

- How does the allowlist interact with bootstrap identity (ADR 0010) when one user spans several channels — one shared persona bundle or channel-scoped overlays?
- Should future channels share the Telegram session model, or is a channel-specific session strategy acceptable as long as the `Channel` trait surfaces it explicitly?
- At what confidence threshold does intent routing route? Below the threshold, fall back to `task`? Or expose the uncertainty to the user?
- Does the Telegram pilot cover voice notes and images in v0.7, or only text + file attachments? Flag for planning once trait shape is settled.
