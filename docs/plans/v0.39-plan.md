This milestone is split from the original "Onboarding + Provider + Identity +

deterministic retrieval algorithm carefully.

## Purpose

Make Allbert usable from a fresh checkout. Turn existing Settings Central,
provider/model profiles, Objective Runtime, and channel setup into one guided
operator setup path. Add a real provider doctor with explicit branches for
credentialed-remote and local-endpoint providers.

## Expected Inputs

- v0.38 templated creation accepted through operator use.
- Existing Settings Central provider/model profile schema with `providers.*`
  defaults for `local_ollama`, `openai`, `anthropic`, `openrouter`.
- Existing encrypted Settings Central secret store.
- Existing Objective Runtime (`AllbertAssist.Objectives.frame/2`,
  `advance/2`, `continue/2`, `cancel/3`) and registered action runner.
- Existing `mix allbert.channels` task for telegram/email registration.

## Operator-Visible Win

An operator can run one guided setup from CLI or `/workspace`, choose a local
Ollama instance or a remote model (OpenAI, Anthropic, OpenRouter), validate
credentials or endpoint reachability, optionally connect a channel, optionally
enable model-assisted intent ranking, and reach a useful first conversation in
roughly fifteen minutes.

## Planned Interfaces

CLI:

- `mix allbert.onboard` ŌĆö guided objective entrypoint; resumable.
- `mix allbert.model list` ŌĆö redacted listing of provider + model profiles.
- `mix allbert.model use PROFILE` ŌĆö set `intent.model_profile`; optional
  `--enable-assist` flag flips `intent.model_assist_enabled` to `true`.
- `mix allbert.model doctor PROFILE` ŌĆö runs the two-branch doctor.

Workspace:

- `workspace:onboard` Canvas destination ŌĆö added to
  `AllbertAssistWeb.Live.WorkspaceLive`'s `@workspace_tools` allowlist
  alongside the existing `create | jobs | objectives | confirmations |
  security | settings` entries. Renders the same onboarding objective state
  the CLI surfaces.
- Workspace provider/model picker ŌĆö view of `list_provider_profiles` +
  `list_model_profiles` with action buttons that call the new doctor and
  set-active actions.

Channels (existing surfaces reused, not new ones):

- `mix allbert.channels telegram set-token TOKEN` and
  `mix allbert.channels email set-password --type {imap,smtp} PASSWORD` are
  the registered surfaces onboarding invokes when the operator opts into a
  channel.

Registered actions added in v0.39:

- `doctor_provider_profile` (read-only; permission `:read_only`;
  confirmation `:not_required`).
- `set_active_model_profile` (permission `:settings_write`; writes
  `intent.model_profile` and optionally `intent.model_assist_enabled` via
  safe-key paths).
- `onboarding_step_complete` (permission `:objective_write`; records step
  progress on the onboarding objective).

## Onboarding Steps

The onboarding objective frames a single durable `Objective` per operator with
the following ordered steps. Each step is a registered action invoked through
`Actions.Runner.run/3`; the operator may skip optional steps.

1. **Welcome + scope** ŌĆö read-only summary of what onboarding will write
   (`intent.model_profile`, `intent.model_assist_enabled`,
   `providers.<name>.enabled`, `providers.<name>.endpoint_kind`, channel keys).
2. **Pick provider profile** ŌĆö operator selects one of the shipped providers
   or accepts the default `local_ollama`.
3. **Resolve credential or endpoint** ŌĆö branches on provider profile's
   `endpoint_kind`:
   - `:credentialed_remote` ŌåÆ set or confirm `api_key_ref` via the existing
     `set_provider_credential` action; secret is stored encrypted, never
     printed.
   - `:local_endpoint` ŌåÆ confirm `base_url` reachability; no credential probe.
4. **Run doctor** ŌĆö `doctor_provider_profile` produces the v0.39 doctor
   summary (see "Provider Doctor ŌĆö Two Branches" below). Failure modes do
   not block onboarding; operator may retry or skip.
5. **Pick model profile** ŌĆö operator selects an existing `model_profiles.*`
   entry (default `local`) or accepts the shipped default. Writes
   `intent.model_profile` via `set_active_model_profile`.
6. **Toggle model-assisted intent** ŌĆö operator opts in (default yes) or out;
   sets `intent.model_assist_enabled`. Explicit prompt because the shipped
   default is `false`.
7. **Optional channel registration** ŌĆö operator selects telegram, email, or
   neither. Calls the existing channel set-token / set-password / map
   actions; failure or skip records `:skipped` on the step.
8. **Identity slot preview** ŌĆö read-only mention that the inert `identity`
   memory namespace and Active Memory retrieval land in v0.39b. No write
   path in v0.39. Points at `docs/operator/active-memory.md` (stub shipped
   with v0.39b plan revision).
9. **Done** ŌĆö onboarding objective transitions to `:completed`; operator
   sees a recap and the workspace home.

The objective is resumable: closing CLI or refreshing `/workspace` and
re-entering `mix allbert.onboard` or `workspace:onboard` returns the operator
to the first incomplete step.

## Provider Doctor ŌĆö Two Branches

The doctor distinguishes provider shape rather than treating all providers as
credentialed remote endpoints. The branch is selected by an explicit
`endpoint_kind` field on the provider profile (see "Settings Central Schema
Changes" below).

- **`:credentialed_remote`** (OpenAI, Anthropic, OpenRouter, hosted Ollama
  cloud, etc.): a bounded probe call against the provider's models endpoint.
  Validates credential format, reachability, and one model availability check.
  Returns redacted summary only.
- **`:local_endpoint`** (Ollama on `http://localhost:11434`, llama.cpp server,
  LM Studio, etc.): a reachability check + `GET /api/tags` (or provider
  equivalent) to confirm the endpoint is up and the requested model is pulled.
  No credential probe.

### Provider Doctor Return Shape

Both branches return the same redacted summary shape per ADR 0047. The shape
is **additive-only** post-v0.39 and is a Tier-1 freeze candidate for v1.0:

```
%{
  endpoint_kind: :credentialed_remote | :local_endpoint,
  credential_ok: boolean() | nil,   # nil for :local_endpoint
  endpoint_ok: boolean(),
  model_available: boolean() | :unknown,
  context_window: pos_integer() | nil,
  deprecation_warning: String.t() | nil,
  last_seen_rate_limit_hint: String.t() | nil,
  redacted_host: String.t(),         # host only; path/query stripped
  diagnostics: [%{code: atom(), message: String.t()}]
}
```

Failure modes return operator-readable messages in `diagnostics` without
leaking endpoint internals, query strings, raw error bodies, secret values,
or full URLs. Downstream doctors (v0.40 MCP, v0.47 voice, v0.48 vision) must
reuse this shape with provider-specific additive fields.

## Settings Central Schema Changes

Additive-only, behind the existing `providers.*.*` fragment. No new
permission classes; all writes use existing `:settings_write` and
`:settings_write_safe_keys` policy.

- **New field**: `providers.*.endpoint_kind` (atom-like string,
  `"credentialed_remote" | "local_endpoint"`).
  - Default derivation: `local_ollama` ŌåÆ `local_endpoint`; everything else
    ŌåÆ `credentialed_remote`. Override is operator-writable through
    `update_setting`.
- **Safe-key additions** (writable through `update_setting` without an
  elevated confirmation): `providers.*.endpoint_kind`,
  `providers.*.enabled`. `providers.*.base_url`, `providers.*.api_key_ref`
  are already safe-keyed.

## Settings Central Writes During Onboarding

Onboarding writes only safe-keys through registered actions. Each write is
audited under the onboarding objective:

- `intent.model_profile` ŌĆö set by `set_active_model_profile`.
- `intent.model_assist_enabled` ŌĆö set by `set_active_model_profile` when the
  operator opts in at step 6.
- `providers.<name>.enabled` ŌĆö set when the operator picks a non-default
  provider.
- `providers.<name>.endpoint_kind` ŌĆö set only when the operator explicitly
  overrides the default derivation.
- `providers.<name>.api_key_ref` ŌĆö set indirectly through
  `set_provider_credential`; secret is encrypted and never printed.
- `channels.telegram.bot_token_ref`, `channels.email.imap.password_ref`,
  `channels.email.smtp.password_ref` ŌĆö set indirectly through the existing
  `mix allbert.channels set-token` / `set-password` paths.

Onboarding never grants new permission classes, never lowers a safety floor,
and never writes to non-safe-key settings.

## Default Profile Hygiene

The shipped `model_profiles.local.model` is currently `"gemma4:26b"`, which
is not a real Ollama model name. v0.39 M2 bumps the shipped default to a
real, commonly-available model (target: `"llama3.2:3b"` or
`"qwen2.5:7b"` ŌĆö pick one during implementation against current Ollama
availability). The fix is a settings schema change only; no action shape or
fragment change.

This is required for the v0.39 acceptance smoke against a fresh Ollama
install to pass without manual model-pulling.

## Cross-OS First-Run

Per the v1.0 acceptance matrix (item 1), v0.39 onboarding must succeed on
macOS, Linux, and Windows/WSL2.

- **Paths**: `<ALLBERT_HOME>` resolution per ADR 0005 already handles
  cross-OS. Onboarding must not assume POSIX-only path separators when
  rendering paths back to the operator.
- **Local endpoint reachability**: WSL2 default localhost `127.0.0.1` reaches
  the Linux side; reaching a Windows-host Ollama (e.g.,
  `host.docker.internal` or the host's IP) is documented but not required.
- **Secret store**: existing Settings Central encrypted secrets work on all
  three OSs (no OS keychain dependency in v0.39; that's parked).
- **Terminal UX**: `mix allbert.onboard` reads from `IO.gets/1` (line-mode)
  for selections and from `IO.gets/1` with echo disabled for secrets;
  Windows/WSL2 + Linux + macOS all support this. Non-interactive runs
  (CI, piped stdin) fail fast with a usable message.
- **Smoke matrix**: `docs/plans/v0.39-request-flow.md` adds explicit
  macOS + Linux + Windows/WSL2 verification rows.

## Security Posture

Onboarding is a registered objective whose steps call registered actions.
Provider checks never print raw secrets or raw error bodies. Every effectful
setup step runs through `Actions.Runner.run/3`. Doctor output is bounded,
redacted, and audited per ADR 0047.

Note on fallback: the post-v0.37 planning pass parked model-provider fallback
policy entirely (see `future-features.md` "Model Fallback / Degradation
Policy"). The doctor reports availability; the runtime does not silently
failover. Operators who need fallback can wait for the parked design or
manually switch profiles.

## Non-Goals

- No identity slot write authority in v0.39 ŌĆö onboarding renders the identity
  slot as a v0.39b preview only.
- No Active Memory in v0.39 (moves to v0.39b).
- No new provider backend.
- No hidden provider failover.
- No automatic channel enrollment without operator input.
- No autonomous skill creation from onboarding traces.
- No OS keychain or remote secrets-manager integration.
- No editing of arbitrary settings keys through onboarding (safe-keys only).

## Definition Of Done

v0.39 is done when:

- A disposable Allbert Home can finish guided onboarding from CLI and from
  `/workspace`, including the optional channel step and the optional
  model-assist toggle.
- The two-branch provider doctor returns the redacted summary shape from
  ADR 0047 for at least one credentialed-remote provider (Anthropic) and
  one local-endpoint provider (Ollama against a fresh install).
- Onboarding writes only safe-keys, only through registered actions, and
  only against the framed onboarding objective.
- The shipped `model_profiles.local.model` default resolves against a
  fresh Ollama install without manual model-pulling.
- Cross-OS smoke passes on macOS, Linux, and Windows/WSL2.
- Security evals from ┬¦M4 pass; doctor-no-leak fixture asserts redacted
  output.
- Version metadata bumped to `0.39.0`; README, CHANGELOG, roadmap,
  agent-context-map, operator onboarding doc, and security-hardening doc
  reflect the shipped state.
- Release gate (`mix compile --warnings-as-errors`, `mix credo --strict`,
  `mix dialyzer`, `mix precommit`, `git diff --check`) passes.
- v0.39 satisfies items 1 (first-run setup on macOS/Linux/Windows-WSL2)
  and 2 (operator can choose local Ollama, OpenAI, Anthropic, or
  OpenRouter) of the v1.0 acceptance matrix.
