# Allbert v0.39 First-Run Onboarding And Provider Control

Request flow: `docs/plans/v0.39-request-flow.md`

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`, post-v0.37 planning pass.

This milestone is split from the original "Onboarding + Provider + Identity +
Active Memory" bundle. **Identity slot and Active Memory move to v0.39b** so
each sub-milestone has focused scope and Active Memory has room to land its
deterministic retrieval algorithm carefully.

## Purpose

Make Allbert usable from a fresh checkout. Turn existing Settings Central,
provider/model profiles, Objective Runtime, and channel setup into one guided
operator setup path. Add a real provider doctor with explicit branches for
credentialed-remote and local-endpoint providers.

## Expected Inputs

- v0.38 templated creation accepted through operator use.
- Existing Settings Central provider/model profile schema.
- Existing encrypted Settings Central secret store.
- Existing Objective Runtime and registered action runner.

## Operator-Visible Win

An operator can run one guided setup from CLI or `/workspace`, choose a local
Ollama instance or a remote model (OpenAI, Anthropic, OpenRouter), validate
credentials or endpoint reachability, optionally connect a channel, and reach
a useful first conversation in roughly fifteen minutes.

## Planned Interfaces

- `mix allbert.onboard`
- `/workspace` onboarding destination.
- `mix allbert.model list`
- `mix allbert.model use PROFILE`
- `mix allbert.model doctor PROFILE`
- Workspace provider/model picker.
- Optional channel command equivalent for model/profile status.

## Provider Doctor — Two Branches

The doctor distinguishes provider shape rather than treating all providers as
credentialed remote endpoints:

- **`:credentialed_remote`** (OpenAI, Anthropic, OpenRouter, hosted Ollama
  cloud, etc.): a bounded probe call against the provider's models endpoint.
  Validates credential format, reachability, and one model availability check.
  Returns redacted summary only.
- **`:local_endpoint`** (Ollama on `http://localhost:11434`, llama.cpp server,
  LM Studio, etc.): a reachability check + `GET /api/tags` (or provider
  equivalent) to confirm the endpoint is up and the requested model is pulled.
  No credential probe.

Both branches return the same redacted summary shape:
`{credential_ok | endpoint_ok, model_available, context_window,
deprecation_warning_if_any, last_seen_rate_limit_hint}`.

Failure modes return operator-readable messages without leaking endpoint
internals or raw error bodies.

## Security Posture

Onboarding is a registered objective whose steps call registered actions.
Provider checks never print raw secrets or raw error bodies. Every effectful
setup step runs through `Actions.Runner.run/3`.

Note on fallback: the post-v0.37 planning pass parked model-provider fallback
policy entirely (see `future-features.md` "Model Fallback / Degradation
Policy"). The doctor reports availability; the runtime does not silently
failover. Operators who need fallback can wait for the parked design or
manually switch profiles.

## Non-Goals

- No identity slot in v0.39 (moves to v0.39b).
- No Active Memory in v0.39 (moves to v0.39b).
- No new provider backend.
- No hidden provider failover.
- No automatic channel enrollment without operator input.
- No autonomous skill creation from onboarding traces.

## Milestones

- M3: Optional channel registration step under the onboarding objective.
- M4: Security evals (credential redaction, doctor-no-leak,
  onboarding-action-boundary), operator docs, closeout.

## Definition Of Done

v0.39 is done when a disposable Allbert Home can finish guided onboarding,
select and validate a model profile through the two-branch doctor, preserve
redacted secret handling, and pass focused docs/security checks plus the
release gate chosen by the milestone.
