# Allbert v0.24 Request Flow — Objective Runtime Foundation

## Purpose

This document describes how a runtime turn flows through the objective
engine. It complements `v0.24-plan.md` (which lists modules, schema,
milestones, and tests).

Three flows are documented end-to-end:

1. Single-step objective (one action; one confirmation).
2. Multi-step objective (two sequential actions).
3. Cancellation.

The Milestone Update Log at the bottom is updated as each milestone
lands.

## Stage Reference

The engine implements seven stages:

| # | Stage | Engine command | What it does |
|---|---|---|---|
| 1 | Receive | `:receive_input` | Accept runtime turn; attach `objective_id` if resuming. |
| 2 | Interpret intent | `:interpret_intent` | Call `Intent.Engine`; receive ranked candidates including `:objective` kind. |
| 3 | Frame/resume objective | `:frame_objective` | Decide: create new, resume existing, or no objective. Deterministic rules. |
| 4 | Propose and evaluate steps | `:propose_steps` + `:evaluate_steps` | Deterministic proposer turns intent candidate into one or more step records. |
| 5 | Authorize selected step | `:authorize_step` | Route through `Actions.Runner` + Security Central + Confirmations. |
| 6 | Execute | `:execute_step` | `Actions.Runner.run/3` |
| 7 | Observe and advance | `:observe_step` | Record observation; evaluate against acceptance; decide continue/block/complete. |

## Flow 1: Single-Step Objective

Example user input: `"analyze AAPL"` (in StockSage app context).

```
┌────────────────────────────────────────────────────────────────────┐
│ T1: User submits "analyze AAPL"                                     │
│   active_app: :stocksage                                            │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 1 — Receive                                                   │
│   AllbertAssist.Runtime.submit_user_input/1                         │
│   → emits allbert.input.received                                    │
│   → no objective_id attached (not a resume)                         │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 2 — Interpret intent                                          │
│   Intent.Engine.decide/1                                            │
│   → candidates: [RunAnalysis(AAPL), …]                              │
│   → selected: RunAnalysis(AAPL)                                     │
│   → emits allbert.runtime.turn.started                              │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 3 — Frame objective                                           │
│   Objectives.Engine.cmd(:frame_objective, %{                        │
│     intent_decision: <selected>,                                    │
│     user_id: "local",                                               │
│     source_thread_id: <thread>,                                     │
│     active_app: :stocksage                                          │
│   })                                                                │
│   → inserts row in `objectives`                                     │
│     title: "Analyze AAPL"                                           │
│     objective: "Produce one completed analysis for AAPL …"          │
│     acceptance_criteria: "one stocksage_analyses row …"             │
│     status: :open                                                   │
│   → inserts row in `objective_events` (kind=:created)               │
│   → emits allbert.objective.created                                 │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 4 — Propose and evaluate steps                                │
│   Proposer.propose/2 → one step:                                    │
│     kind: :action                                                   │
│     candidate_action: "StockSage.Actions.RunAnalysis"               │
│     action_params: %{ticker: "AAPL", analysis_date: "2026-05-16"}  │
│   → inserts row in `objective_steps` (status=:proposed)             │
│   → emits allbert.objective.step.proposed                           │
│   Evaluator.evaluate/2 → :feasible                                  │
│   → step status moves to :selected                                  │
│   → emits allbert.objective.step.selected                           │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 5 — Authorize selected step                                   │
│   Actions.Runner.run("StockSage.Actions.RunAnalysis", params, ctx)  │
│   Security Central → :needs_confirmation (per :stocksage_analyze    │
│     safety floor)                                                   │
│   Confirmations.Store.cmd(:create_pending, %{                       │
│     ...standard v0.07 fields...,                                    │
│     objective_id: <id>,                                             │
│     step_id: <step.id>                                              │
│   })                                                                │
│   → emits allbert.confirmation.created (existing v0.07 signal)      │
│   → step status: :blocked (waiting on confirmation)                 │
│   → engine emits allbert.objective.blocked                          │
└──────────────────────────────┬─────────────────────────────────────┘
                               │  Operator approves via CLI or
                               │  LiveView or Telegram inline button
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 5 (resume) — Authorize re-check on approval                   │
│   approve_confirmation action runs                                  │
│   Security Central re-checks                                        │
│   Confirmation status → :approved                                   │
│   → emits allbert.confirmation.approved (existing v0.07 signal)     │
│   Engine.cmd(:continue_objective, %{id: <obj_id>})                  │
│   → step status: :selected (back from :blocked)                     │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 6 — Execute                                                   │
│   Engine.cmd(:execute_step, %{step_id: <step.id>})                  │
│   → step status: :running                                           │
│   → Actions.Runner.run target (RunAnalysis)                         │
│     StockSage.TraderBridge runs Python; result persists to          │
│     stocksage_analyses with objective_id + step_id                  │
│   → step status: :completed                                         │
│   → emits allbert.objective.step.completed                          │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Stage 7 — Observe and advance                                       │
│   Engine.cmd(:observe_step, %{step_id: <step.id>})                  │
│   → builds observation summary from action result                   │
│   → updates objective.last_observation_summary                      │
│   → increments objective.loop_count                                 │
│   → emits allbert.objective.observed                                │
│   Evaluator.evaluate_acceptance/1 → :met                            │
│   → objective.status: :completed                                    │
│   → objective.completed_at: now                                     │
│   → emits allbert.objective.completed                               │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Response render                                                     │
│   AllbertAssist.Runtime returns rendered response with:             │
│     - analysis summary                                              │
│     - "Objective: Analyze AAPL — completed" footer                  │
│     - trace markdown link                                           │
└────────────────────────────────────────────────────────────────────┘
```

Trace markdown for this flow contains:

- `## Objective` — id, title, status, acceptance, last observation
- `## Objective Steps` — one step row with status, action, result
- `## Confirmation` — standard v0.07 section with `Objective: <id>`
  and `Step: <step_id>` lines
- `## Intent Candidates` — standard v0.19 section with the
  RunAnalysis selection
- `## Selected Skill` — standard
- `## Action Result` — standard
- `## Memory Review` — appears only if reflection step proposed
  candidates (none in this flow)

## Flow 2: Multi-Step Objective

Example user input: `"analyze AAPL and compare to MSFT"`.

The flow is the same as Flow 1 but with these differences:

1. **Stage 3 — Frame objective.** `acceptance_criteria` includes two
   targets and a comparison summary.
2. **Stage 4 — Propose and evaluate steps.** Proposer produces two
   step rows:
   - step 1: action=`RunAnalysis(AAPL)`
   - step 2: action=`RunAnalysis(MSFT)` with `parent_step_id` =
     step 1's id (optional; reserved field)
3. **Stages 5–7 run for step 1.** Same as Flow 1. Confirmation,
   approval, execution, observation. Step 1 completes.
4. **Acceptance evaluator returns `:needs_more_steps`** since only
   one analysis exists; comparison criterion not yet met.
5. **Engine loops back to Stage 4** to advance to step 2.
6. **Stages 5–7 run for step 2.** Step 2 confirmation, approval,
   execution, observation.
7. **Acceptance evaluator returns `:met`.** Objective completes.

Two confirmations are created, one per step. The operator approves
each in turn. The engine does not auto-approve based on prior
approval ("smarter" auto-approve policies are a reserved-vocabulary
concern, not v0.24).

Loop cap: each cycle through Stages 4–7 increments
`objective.loop_count`. Default `max_loop_count = 5` accommodates
the two-step case comfortably. If a hypothetical objective needs
more steps than the cap, the engine records `:impasse` and
transitions to `:blocked`.

## Flow 3: Cancellation

Example: operator runs `mix allbert.objectives cancel <id>` while
an objective is in `:blocked` (awaiting confirmation) or `:running`.

```
┌────────────────────────────────────────────────────────────────────┐
│ Operator: mix allbert.objectives cancel <id> --reason "..."         │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ AllbertAssist.Actions.Objectives.CancelObjective                   │
│   → Security Central: :objective_write (allow by default)           │
│   → Actions.Runner.run/3                                            │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│ Engine.cmd(:cancel_objective, %{objective_id: <id>, reason: ...})   │
│   → objective.status: :cancelled                                    │
│   → any :proposed steps → :cancelled                                │
│   → any :running step is NOT interrupted (cooperative only)         │
│   → in-flight registered action completes normally                  │
│   → its observation is still recorded                               │
│   → but no new steps proposed; engine stops advancing               │
│   → emits allbert.objective.cancelled                               │
│   → records objective_events row (kind=:cancelled)                  │
└────────────────────────────────────────────────────────────────────┘
```

If a confirmation is pending when cancellation runs:

- The confirmation record itself is NOT auto-expired. The operator
  can still approve or deny it. If they approve, the action runs
  (single-shot semantics), but the engine ignores the result since
  the objective is `:cancelled`. The action's own audit trail
  records the run.
- This is a deliberate choice: cancellation does not destroy work
  already in flight. Operators who want to also reject the pending
  confirmation can do so via `mix allbert.confirmations deny`.

Mid-action interruption (killing the in-flight `RunAnalysis` Python
process, for example) is not v0.24 scope. It's deferred to v0.25+
when native trading agents have a clearer interruption story.

## Confirmation Integration Detail

When a step requires confirmation, the confirmation record gets
two new fields (already in the schema from v0.23 migration):

- `objective_id`
- `step_id`

`mix allbert.confirmations show <id>` renders these as:

```
Confirmation: <conf_id>
Action: StockSage.Actions.RunAnalysis
Status: pending
Created: 2026-05-16 14:23:01 UTC
TTL: expires 2026-05-16 15:23:01 UTC

Objective: <objective_id>
  Title: Analyze AAPL
Step: <step_id>
  Kind: action
```

Telegram and email confirmation rendering (M5) inserts the same
`Objective: ...` and `Step: ...` lines into their respective
formats.

## Resume Patterns

Three ways an objective gets resumed:

1. **Confirmation approval.** The most common path. After
   `approve_confirmation` runs, the engine's `:continue_objective`
   command picks up where the step was blocked.
2. **Operator `mix allbert.objectives continue`.** Used when an
   objective is in `:blocked` status for a reason other than
   pending confirmation (e.g., loop-cap-hit impasse with operator
   review).
3. **Scheduled job advance.** A v0.13 scheduled job may carry an
   `objective_id`; when it runs, it advances the objective. This is
   the path used for long-running analyses or follow-ups across
   days.

## Intent Engine Integration

The intent engine (`Intent.Engine.collect_candidates/1`) gains an
`:objective` candidate kind:

- When the user has open objectives matching the current turn's
  topic (deterministic keyword + recency match), an objective-
  resume candidate appears in the ranking.
- Selecting it routes to `:continue_objective` rather than creating
  a new objective.
- ADR 0019 invariants hold: the candidate's score cannot exceed
  permission floors; the candidate does not grant authority.

Candidate fields:

```elixir
%AllbertAssist.Intent.Candidate{
  kind: :objective,
  id: <objective_id>,
  source: :objective,
  score: 0.2..0.5,
  reason: <objective.title>,
  trace_metadata: %{
    objective_id: <id>,
    objective_status: :open | :running | :blocked,
    objective_active_app: <active_app>
  }
}
```

Memory candidates (`:memory`) and objective candidates (`:objective`)
can coexist in the same ranking pass without conflict.

## Settings Behavior

| setting | default | effect |
|---|---|---|
| `objectives.enabled` | `true` | When `false`, runtime turns skip Stages 3–4 and 7. Intent → action runs as in v0.23. |
| `objectives.max_steps_per_turn` | `3` | Engine refuses to propose more than N steps within a single turn. |
| `objectives.max_loop_count` | `5` | After N advance cycles, engine records `:impasse` and transitions to `:blocked`. |
| `objectives.trace_detail` | `:operator` | At `:operator`, trace contains `## Objective` and bounded `## Objective Steps`. At `:debug`, also stage and lifecycle events. |

A `mix allbert.settings set objectives.enabled false` instantly
disables objective creation for new turns. Existing open objectives
remain visible via `mix allbert.objectives list` until they are
cancelled or completed manually.
