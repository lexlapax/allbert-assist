## Purpose

Make the v0.39-v0.58 surface **portable, auditable, freeze-ready, and boring to
operate** before the v1.0 contract freeze. Five outcomes, all on the existing
authority spine:

1. **Portability.** An operator exports an Allbert Home and dry-run validates it
   against a fresh Home with a versioned envelope, a diagnostic, redaction, a
   documented rollback, and the self-improvement import-plan inert invariant.
2. **Settings version contract (ADR 0046, right-sized).** A first-class
   `schema_version` field on every registered Settings fragment, a generated
   fragment inventory backfilled to `1`, additive-only-between-minors enforcement,
   and a fail-closed boot check on unknown/forward versions. **No runtime
   migration runner** — deferred per ADR 0046 §5.
3. **Central param contracts (ADR 0065).** Each action's bypassed
   `validate_params/1` reused at the Runner boundary inside an Allbert strictness
   wrapper, with safe schema-known key normalization, the three context-in-params
   cleanups, an explicit generated `schema: []` / JSON-schema disposition catalog,
   and a release-blocking eval sweep over the ~190-action catalog. Delivered
   freeze-ready (the Registry action schema and the `:invalid_params` shape are
   Tier-1 freeze inputs).
4. **Cross-surface security sweep.** A per-surface PASS sweep over every capability
   class added v0.40-v0.51, with evidence artifacts and the self-improvement-inert
   + voice/vision invariants asserted.
5. **RC-substrate handoff.** Enumerated downstream consumers (v0.61, v0.63, v1.0)
   with a clean, drift-free substrate and a deterministic `release.v059` gate.

The authority boundary is unchanged. No new capability, no new authority surface,
no confirmation-floor change, no contract freeze (that is v1.0). The model still
proposes params only; Security Central remains the sole authority.

## Cross-Cutting Invariants (apply to every milestone)
- **No new authority.** No new capability class, no `:internal` action made
  routable, no confirmation-floor change, no egress grant. Security Central is the
  only authority boundary.
- **Redaction everywhere.** Export envelopes, dry-run diagnostics, the boot-check
  diagnostic, and security-sweep evidence never print raw secret values, secret
  refs in plaintext, endpoint URLs, raw prompts, or provider bodies. Secret VALUES
  stay in the encrypted store; only REFERENCES round-trip.
- **Self-improvement inert in the import plan.** v0.47 suggestion state is
  represented as data in the dry-run diagnostic but never auto-executes or applies
  to the importing machine in v0.59.
- **Voice/vision inert in the import plan.** v0.48-v0.49 media-capture surfaces are
  not auto-enabled by a dry-run import; retention metadata is reported without
  re-arming capture.
- **Settings Central is the only config authority.** Every new key (envelope
  version, schema_version field) is schema-backed; the v0.58 settings-bypass guard
  still holds.
- **One spine.** Export/import and the param seam operate through registered
  actions / documented facades; no surface reaches `Settings.Store`,
  `Confirmations`, or business logic directly (ADR 0073, inherited).

### M1: Allbert Home Export/Import (Portability)
Define a versioned export envelope and a dry-run import that produces a redacted
diagnostic. The envelope carries `envelope_version` (explicit, distinct from
per-fragment `schema_version`), the exporting Allbert version, and a manifest of
what is included. Export covers settings (with `schema_version` per fragment, from
M2), memory, skills, threads, traces, audits, jobs, objectives, confirmations,
plugin/marketplace metadata, self-improvement suggestion state, media retention
metadata, channel pairing metadata, and public-protocol settings. Secret VALUES
are redacted; only secret REFERENCES are exported (M3). Import is **dry-run only**:
it reads the envelope, validates `envelope_version`, runs the M2 boot check against
each fragment's `schema_version`, and emits a diagnostic to stdout or an
operator-supplied evidence path outside the target Home — it does not auto-apply or
write to the target Home.
Rollback is a documented manual procedure (restore the prior Home directory);
the `profile-import-dry-run-rollback-001` eval asserts the dry-run + documented
rollback path, NOT an executable rollback runner.

Portability manifest minimum:

| Domain | Source of truth | Export shape | Redaction / exclude rule |
| --- | --- | --- | --- |
| Settings | `Settings.Fragments`, `Settings.Store`, `Settings.Schema` | user settings + per-fragment `schema_version` from M2 | no raw secret values; structural versions only |
| Secret references | `Settings.Secrets` / `Settings.KeyCustody` status APIs | ref ids, provider namespace, configured/missing status | never export `settings/secrets.yml.enc`, `.settings_key`, or plaintext |
| Memory, traces, audits | `AllbertAssist.Paths` roots and existing memory/trace/audit facades | markdown/data files with manifest hashes and source paths | redact secret-shaped content; exclude temp/cache/build files |
| Runtime DB state | reviewed Repo/facade reads for threads, jobs, objectives, confirmations, channel refs, and public-protocol settings | redacted records in deterministic manifest sections | no provider bodies, raw prompts, tokens, or internal-only report bodies |
| Skills, plugins, marketplace | plugin/marketplace registries and Allbert Home plugin roots | installed/imported metadata and inert files | imported code remains disabled/untrusted; no scripts executed |
| Media/artifacts/self-improvement | artifact/media retention facades and self-improvement state readers | retention metadata, artifact refs, inert suggestion state | import dry-run must not auto-arm capture or auto-apply suggestions |

- **Modules:** `AllbertAssist.Portability.Export` (build envelope),
  `AllbertAssist.Portability.Import` (validate + dry-run diagnostic),
  `AllbertAssist.Portability.Envelope` (envelope struct + `envelope_version`).
- **Files:** `mix/tasks/allbert.home.export.ex`, `mix/tasks/allbert.home.import.ex`
  (or workspace operator actions); the portability modules under
  `lib/allbert_assist/portability/`.
- **Settings:** none new beyond the envelope metadata; reads through documented
  facades, not raw `Settings.Store`.
- **Tests:** export captures every listed surface; the envelope carries
  `envelope_version` + per-fragment `schema_version`; secret values absent and
  refs present; dry-run import produces a diagnostic and applies nothing;
  Home B is byte-identical before/after dry-run import; self-improvement +
  voice/vision stay inert in the diagnostic/import plan. Evals
  `profile-export-redaction-001`, `profile-import-dry-run-rollback-001`,
  `self-improvement-inert-after-import-001`, `media-inert-after-import-001`.
- **Verify:** `mix allbert.home.export` into Home A then dry-run import into Home B
  prints a redacted diagnostic to stdout or the operator evidence path outside Home
  B; `rg` over Home B finds no raw secret material; the diagnostic names
  current/pending/unknown per fragment; Home B's file tree is unchanged.
- **Acceptance:** a disposable Home exports and dry-run validates against a fresh
  Home with a redacted diagnostic, an explicit envelope version, documented
  rollback, and the inert invariants hold.

### M2: Settings Version Contract (right-sized per ADR 0046 — no runner)
Add `schema_version` as a **first-class field on `AllbertAssist.Settings.Fragment`**
(default `1`) and **backfill every currently registered fragment/namespace to
`1`**, superseding the inconsistent schema-row approach. Existing row-style
`*.schema_version` keys remain read-only compatibility reads and are reconciled
into the first-class field. The release gate generates the exact inventory from
`AllbertAssist.Settings.Fragments.registered_fragments/0` so plugin/app fragments
such as `research` are not missed. An un-versioned stored fragment is treated as
version `1`. Enforce **additive-only between minors** with a
schema-diff CI check that fails a non-additive change (rename/remove/type
change/default-behavior change/floor change) that did not bump the version. Add a
**fail-closed boot check**: compare each fragment's stored `schema_version` against
the known max; on unknown/forward (stored > known) surface an operator-visible
diagnostic through a new settings doctor read surface AND `mix allbert.security
status`, and refuse to silently load. A version behind known is recorded as
pending (the runner that would apply it is **deferred per ADR 0046 §5**).
Export/import preserves `schema_version` (M1). The boot check is testable via a
**fixture export carrying a bumped/forward `schema_version`** — no runner is needed
to exercise the fail-closed path.
- **Modules:** `AllbertAssist.Settings.Fragment` (add `schema_version`),
  `AllbertAssist.Settings.VersionContract` (known-max lookup + boot check),
  a schema-diff check module reused by the CI guard; settings doctor read action.
- **Files:** `settings/fragment.ex`, all fragment registrations under
  `settings/schema.ex` and plugin/app fragments (backfill `schema_version: 1`),
  `mix/tasks/allbert.security.ex` (surface the boot-check diagnostic),
  a settings-doctor read surface.
- **Settings:** `schema_version` is now a Fragment field, not a per-namespace key;
  existing row-style versions are reconciled into the field. See the keys table.
- **Tests:** the field exists and defaults to `1`; the generated fragment
  inventory reports every registered fragment at `1`; the schema-diff check fails
  on a seeded non-additive change without a bump and
  passes on an additive one; the boot check fails closed on a fixture forward
  version and surfaces through doctor + `security status`; export preserves the
  field. Evals `settings-schema-version-field-001`,
  `settings-additive-only-enforced-001`, `settings-boot-check-fail-closed-001`.
- **Verify:** `mix allbert.security status` reports per-fragment version; an
  imported fixture with `schema_version: 2` on a fragment whose known max is `1`
  trips the fail-closed diagnostic; the CI schema-diff check is wired into
  `release.v059`.
- **Acceptance:** every registered fragment carries a first-class version at `1`,
  additive-only is enforced, the boot check fails closed on forward versions, and
  the runner remains explicitly deferred (stated, not built).

**Implementation note (2026-06-29):** M2 landed as the structural version

### M7: Central Action Param-Contract Enforcement (ADR 0065)
Reuse each action's already-generated-but-bypassed `validate_params/1` inside an
Allbert strictness wrapper in `Runner.safe_run/3` (`runner.ex:85`) **before**
`run/2`. The exact order is: normalize only schema-known string keys to existing
schema atoms (never create atoms from model/provider input), reject unknown keys
unless the action has an explicit open-key/compatibility disposition, call
`action_module.validate_params/1`, and pass the returned `validated_params` into
`action_module.run/2`. The context/param split **mostly already exists** —

## Non-Goals
- **No runtime migration runner.** `mix allbert.settings.migrate`, the
  `Settings.Migration` DSL, the `migrations/` registry, and the
  deprecated→replacement rewrite are deferred to the first real non-additive
  migration (ADR 0046 §5).
- **No secret-store content migration.** Only secret references round-trip; values
  stay encrypted. v0.61's credential→OS-vault move is a separate operator migration.
- No executable rollback runner; rollback is documented manual restore.
- No new capability surface, no new authority, no confirmation-floor change.
- No guided onboarding (ADR 0069 → v0.62), no packaging/entry points (→ v0.61),
  no web UX polish, no final integrated RC evidence (→ v0.63).
- No public AG-UI/A2UI or MCP Apps iframe evals.
- No contract freeze (→ v1.0).
