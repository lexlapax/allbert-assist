## Settings Schema Migration Substrate (ADR 0046)

v0.59 implements the migration policy drafted in v0.45:

- `mix allbert.settings.migrate` inspects each fragment's current
  `schema_version` against the operator's stored settings; enumerates pending
  migrations; shows redacted previews; requires explicit confirmation per
  migration (or `--all` for batch).
- Migrations are explicit Elixir modules under
  `lib/allbert_assist/settings/migrations/`. No LLM-authored migration steps.
- Pending migrations surface in `mix allbert.security status` and the
  workspace Settings panel on boot.
- Each migration writes a record into
  `<ALLBERT_HOME>/audit/settings_migrations/`.
- v0.59 export/import preserves each fragment's `schema_version` so a v0.43
  export imported on v0.59 is correctly identified as needing migrations.

## Central Action Param Contract Enforcement (M7)

ADR to draft: `docs/adr/0065-central-action-param-contract-enforcement.md`
(next available number at authoring time). Precursor: the v0.54 central
slot/param normalization seam (ADR 0064), which consolidated the scattered
slot→param normalization and gave the Runner correct `:invalid_params`
semantics. M7 is the larger follow-on that v0.54 deliberately deferred.

### Why this is here, and the context that surfaced it

During v0.54 M11 (live punchlist A1, `write_note`), a transient
`FunctionClauseError` in `Enum.reduce/3` was reported. Investigation
(zoom-out over the slot pipeline) established the crash was **not**
reproducible and matched no current code path: every slot **producer**
already guarantees a map (`ReqLLMDisambiguator.parse_slots/1`,
`Descriptor.extract_slots/2`, `Handoff.normalize_map/1`), and the **Runner**
already rejects non-map params at `runner.ex` before any action runs. The
real finding was architectural, not a single bug:

- The Registry centralizes **capability/permission** metadata, but action
  **params** reach `run/2` raw. There is no central param-contract seam.
- Param validation is per-action and inconsistent: at the time of writing,
  **72 of 110 actions have no `is_map(params)` guard**; each action
  hand-rolls its own `field/2` / `required/2` helpers and key normalization
  (atom ↔ string).
- This is the inconsistency with the project's "settings central /
  registration central" principle. v0.54 (option 1) closes the **intent →
  params** seam and the Runner shape/semantics gap; M7 (option 3) closes the
  **schema contract** gap for all actions.

### Scope

Make each action's registered schema enforceable at the Runner — a single,
registration-based param-contract seam, so a malformed or unexpected param
payload yields a uniform, clean `:invalid_params` response instead of relying
on each action's hand-rolled guards.

### Known blast radius (the reason it is a full milestone, not a patch)

- Actions are declared via `use AllbertAssist.Action` → `use Jido.Action`,
  which generates a NimbleOptions-backed schema, but the Runner calls
  `action_module.run/2` directly and **bypasses** that validation.
- Enabling central NimbleOptions validation as-is would **reject** the
  context keys the intent layer injects but actions do not declare in their
  schema (`user_id`, `thread_id`, `session_id`, source text). Every action
  schema would need those keys added, or the Runner must separate
  injected-context params from schema-validated params before validation.
- Some actions intentionally tolerate extra/loose params today; enforcing
  strict schemas is behavior-changing across all 110 actions and needs an
  eval sweep.

### Approach (to refine in the ADR)

1. Decide the param/context split: schema-validated action params vs.
   Runner-injected context (likely a dedicated `context`-carried namespace so
   injected keys never hit NimbleOptions).
2. Add a central `Registry.schema/1` (or `capability.schema`) accessor so the
   schema is queryable centrally, not only on the Jido module.
3. Validate in the Runner before `safe_run`, mapping NimbleOptions errors to a
   clean `:invalid_params` response (no action body runs).
4. Migrate action schemas to declare all accepted params; remove now-redundant
   per-action `is_map`/`required` guards in a follow-up pass.
5. Eval sweep: confirm no valid request regresses to `:invalid_params`.

## Security Posture

No new authority. The milestone audits all newly added capability classes:
MCP client, browser, channels (Discord/Slack and WhatsApp/Signal/Matrix),
Plan/Build, marketplace, self-improvement, voice, vision, and the v0.51 public
protocol surfaces (MCP server, OpenAI-compatible API, and ACP server).

Public AG-UI/A2UI and MCP Apps iframe evals are NOT part of v0.59; those
surfaces remain parked post-1.0.

Secret migration is explicit and redacted. The migration tool never logs raw
secret values.

## Non-Goals

- No broad remote sync service.
- No hosted multi-user auth.
- No new capability surface.
- No expansion of v0.47 self-improvement behavior.
- No contract freeze until v1.0.
- No public AG-UI/A2UI or MCP Apps iframe evals.
- No automatic settings migration on boot. Migrations require operator
  confirmation per ADR 0046.

## Definition Of Done

v0.59 is done when:

- a disposable profile can be exported and re-imported with documented
  behavior;
- pending schema migrations surface on boot and `mix allbert.settings.migrate`
  applies them with operator confirmation;
- all v1.0 acceptance-matrix gates (per `docs/plans/roadmap.md`) have
  evidence;
- ADR 0046 is accepted;
- ADR 0065 is accepted and central action param-contract enforcement (M7) is
  implemented with an eval sweep showing no valid-request regression;
- ADR 0069 is accepted and a new operator can complete first-run onboarding —
  settings, secrets, channel pairing, and doctors — through a guided path on the
  v0.55 TUI, with no new authority introduced;
- no known warning/security/doc drift remains.
