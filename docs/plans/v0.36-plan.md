# Allbert v0.36 Elixir/OTP Sandbox And Gate Runner

Request flow: `docs/plans/v0.36-request-flow.md`

ADRs:

- `docs/adr/0009-local-execution-sandbox-levels.md` (sandbox levels)
- `docs/adr/0037-elixir-otp-sandbox-backend-and-gate-runner.md` (this
  milestone's narrow Level-3 sandbox contract)

Status: implemented as `v0.36.0`; release gate in progress after adding local
sandbox image preparation. The default `allbert-elixir-otp:local` image is now
prepared through explicit v0.36 Mix tasks instead of being an undocumented
operator prerequisite.

v0.36 is the sandbox substrate milestone. It does **not** generate code, run a
code-gen agent committee, hot-load modules, or integrate live capabilities.
Instead, it makes the risky later work implementable by proving one concrete,
default-off OS sandbox for generated **Elixir/OTP** drafts and the explicit
shell commands needed to compile, test, and gate those drafts.

Version ownership is now split deliberately:

- v0.36 owns the sandbox and gate-runner substrate.
- v0.37 owns dynamic code/config generation and gated live integration.
- v0.38 owns templated operator/developer creation.

The later milestones consume the v0.36 sandbox/gate-runner contract instead of
redefining isolation locally.

## Purpose

Allbert cannot safely trial generated Elixir code in the core BEAM node. BEAM
processes, ports, hidden nodes, and distributed Erlang are useful lifecycle
tools, but they are not an OS security boundary (ADR 0009). Before Allbert adds
LLM code generation or live integration, it needs a small, testable sandbox
runner that can:

- copy a bounded Elixir/OTP draft into an isolated work root;
- run explicit `mix` / `elixir` / `erl` argv commands under a container
  backend;
- deny network, secrets, real Allbert Home, Repo credentials, package-manager
  execution, NIFs, ports, traversal, and broad shell strings;
- return redacted, bounded diagnostics and a structured gate report;
- fail closed when no supported backend is configured.

This milestone is intentionally narrow. It covers Elixir/OTP and the shell
commands required by the Elixir gate only. It is not a broad coding-agent
sandbox, package installer, remote builder, or multi-language runtime.

## Expected Inputs

- ADR 0009's sandbox-level vocabulary and v0.08 Level-1 local policy runner.
- v0.31 shared paths, redaction, audit, typed responses, settings fragments,
  and extension registry facades.
- v0.35 Allbert Home discipline: all durable local runtime data derives from
  `ALLBERT_HOME`, tests/smokes use disposable homes, and raw local files remain
  operator-owned.
- Existing project gate commands: `mix compile --warnings-as-errors`,
  `mix credo --strict`, `mix dialyzer`, focused `mix test`, and security evals.
- A local container-capable host. Supported v0.36 backends are:
  - `apple_container` — optional/doctor-gated Apple `container` on supported
    macOS (Apple silicon + macOS 26+), giving VM-per-container host isolation
    when available, but not release-blocking because the tool is pre-1.0 and
    platform-bound;
  - `docker` with explicit hardening flags (cross-platform fallback);
  - `podman_rootless` where available;
  - optional `docker_runsc` / gVisor when the Docker daemon exposes a `runsc`
    runtime; when doctor-available, it is preferred over plain Docker.
  Backend selection defaults to `"auto"`, which resolves an OS-aware fallback
  chain (see Design Decision 3).

Firecracker, remote builders, broader/cross-version Apple Container features,
multi-language targets, and online dependency installs stay parked.

## Safety / Trust Model

1. **Default off.** The sandbox runner is disabled until Settings Central has
   both `sandbox.elixir.enabled=true` and a configured backend that passes
   `mix allbert.sandbox doctor`.
2. **Copy-in/copy-out.** v0.36 never bind-mounts the real Allbert Home,
   Settings, secrets, SQLite files, or the live project tree read-write. It
   creates a disposable sandbox bundle, copies in the minimal project/draft/test
   inputs, and copies out only bounded reports and diagnostics.
3. **Explicit argv only.** Commands are structured as executable plus argv. No
   shell string, `sh -c`, command chaining, glob expansion, redirection,
   background jobs, PTY, or daemon control is accepted.
4. **Elixir-only command allow-list.** Initial allow-list is `mix`, `elixir`,
   and `erl` for compile/test/gate work. Shell commands are supported only as
   explicit allow-listed argv commands needed by the Elixir gate.
5. **No supply-chain authority.** `mix deps.get`, `mix archive.install`, Hex,
   npm, cargo, pip, git clone, curl/wget installers, migrations, NIF builds,
   ports, and unrestricted network are denied.
6. **No trust grant.** A passing sandbox gate produces a report. It does not
   compile or load generated modules in core, register actions, grant
   permissions, alter routes, or enable skills. v0.37 owns gated integration.
7. **Audited, never auto-confirmed.** Settings writes are audited through
   Settings/Security Central. Sandbox actions require no operator confirmation
   because they grant no authority; nothing in v0.36 ever reaches a
   confirmed-to-integrate state. Invoking the container engine is itself a
   bounded Level-1 host execution (ADR 0009/ADR 0012): it runs only through the
   registered sandbox actions, with the enablement, backend, image, and resource
   bounds owned by Settings Central.

## Design Decisions

### 1. Sandbox runner is a context, not a LiveView concern

Add `AllbertAssist.Sandbox` as the public lifecycle facade:

- `doctor/0`
- `build_bundle/1`
- `run_command/2`
- `run_gate/1`
- `cleanup/1`

LiveViews, Mix tasks, objectives, and future code-gen agents call the facade or
registered actions. They do not construct container commands directly.

### 2. Backend contract is pluggable and inspectable

Backends are registered, not hard-coded. Each implements the
`AllbertAssist.Sandbox.Backend` behaviour and registers with
`AllbertAssist.Sandbox.Backend.Registry`, so future engines plug in without
editing the facade, resolver, or gate runner. The behaviour exposes:

- `id/0` — backend atom (`:apple_container`, `:docker`, `:podman_rootless`,
  `:docker_runsc`)
- `platforms/0` — host platforms the backend can run on (`:macos`, `:linux`)
- `available?/1`
- `doctor/1`
- `run/2`
- `cleanup/1`

The backend receives a normalized `Sandbox.CommandSpec` and `Sandbox.Bundle`.
It returns a typed result with exit status, duration, capped stdout/stderr,
resource-limit status, backend metadata, and redacted diagnostics.

Backend implementations must use explicit executable/argv to invoke the
container engine (`container`, `docker`, or `podman`). They must not call a
shell.

`Sandbox.CommandSpec` is a data struct, not a command string. It carries:

- `executable` (`"mix"`, `"elixir"`, or `"erl"`);
- `argv` (validated against the selected gate profile);
- `cwd` (inside the bundle only);
- `env` (bounded allow-list, no host passthrough);
- `profile` (`:compile`, `:focused_tests`, `:credo`, `:dialyzer`,
  `:security_evals`, or `:precommit`);
- `timeout_ms`, `output_bytes`, and resource-limit overrides capped by
  Settings Central.

Any command that needs shell parsing, host environment lookup, glob expansion,
redirection, daemon control, package-manager behavior, image pulling, or
network setup is rejected before a backend sees it.

### 3. Backend selection is OS-aware, inspectable, and non-blocking

`sandbox.elixir.backend` is stored as a Settings Central string and defaults to
`"auto"`; resolver internals may normalize to atoms. An
`AllbertAssist.Sandbox.Backend.Resolver` picks a registered, doctor-available
backend per host:

- macOS (Apple silicon, macOS 26+): `apple_container` when doctor-green;
  otherwise `docker_runsc` when Docker exposes `runsc`; otherwise `docker`;
- macOS (older or Intel): `docker_runsc` when Docker exposes `runsc`; otherwise
  `docker`;
- Linux: `podman_rootless` → `docker_runsc` when configured → `docker`.

Operators may pin an explicit backend instead of `"auto"`. Doctor reports the
resolved backend and why each candidate passed or failed. If no on-platform
backend is available, sandbox actions fail closed. `"auto"` never weakens
default-off: nothing runs until `sandbox.elixir.enabled=true` and doctor is
green.

Apple `container` is included in v0.36 as an optional backend because it gives
excellent macOS isolation when the host qualifies, but v0.36 release readiness
does not depend on a qualifying macOS 26 Apple-silicon host. The implementation
must include doctor detection and mocked/conditional tests; the run path is
accepted only when doctor proves the local `container` CLI can enforce v0.36
policy. A missing or pre-1.0-incompatible Apple backend is an unavailable
candidate, not a failed release.

### 4. Container baseline, gVisor optional

The cross-platform v0.36 baseline is a hardened Docker/Podman container
invocation with:

- read-only project snapshot mount or copy-in root;
- writable temp/work/cache dirs only inside the sandbox;
- no host Docker socket;
- no network by default;
- dropped capabilities, no privileged mode, no host PID/user/network namespace;
- non-root user where backend support allows it;
- CPU, memory, process, disk, output, and wall-clock limits;
- bounded environment allow-list with no secrets.

If `runsc` is installed and configured as a Docker runtime, `docker_runsc` is a
supported hardened backend and is preferred over plain Docker. It is optional
because it adds host setup and platform requirements and does not replace the
cross-platform Docker fallback. Firecracker and remote/microVM isolation remain
future backends.

On supported macOS (Apple silicon, macOS 26+), the optional `apple_container`
backend runs each container in its own lightweight VM, giving stronger host
isolation than a shared-VM Docker Desktop. It is OCI-compatible and
doctor-gated: doctor verifies the host, the `container` CLI, and policy
enforcement (no network, resource limits, non-root copy-in, read-only roots,
and cleanup) before it is selected. Apple `container` is pre-1.0, so its churn
stays contained behind the doctor gate.

### 5. Image and source policy are part of the sandbox

The sandbox never pulls images during a run. `sandbox.elixir.image` must resolve
to an approved local image reference or digest before execution; doctor reports
missing images and fails closed. v0.36 includes an explicit operator image
preparation workflow so the default `allbert-elixir-otp:local` image can be
built, labeled, verified, and diagnosed before sandbox runs. Image preparation
is separate from sandbox execution: the build task may perform the operator's
approved Docker build step, but command execution still uses local-only image
inspection and `--pull=never`.

`AllbertAssist.Sandbox.SourcePolicy` (or an equivalent module) statically scans
draft source and trial files before backend execution. It denies known
dangerous constructs such as `System.cmd`, `Port.open`, `:os.cmd`,
`Code.eval_*`, `Code.compile_*`, `Code.require_*`, `Mix.install`,
`:erlang.load_nif`, broad `File` traversal, and attempts to reach the real
Allbert Home or host paths. Backend isolation remains mandatory; source policy
is defense-in-depth and produces bounded diagnostics.

### 6. Gate runner is deterministic

`AllbertAssist.Sandbox.GateRunner` runs named gate profiles. v0.36 ships:

- `:compile` → `mix compile --warnings-as-errors`
- `:focused_tests` → configured focused `mix test` files
- `:credo` → `mix credo --strict`
- `:dialyzer` → `mix dialyzer`
- `:security_evals` → configured security eval tests
- `:precommit` optional, for final smoke only

Gate profiles are data, not arbitrary command text. Future v0.37/v0.38 work can
add profile entries only through reviewed code and Settings Central policy.

### 7. Settings Central owns all operator policy

New settings are default-off and audited:

| Key | Type | Default | Meaning |
|---|---:|---:|---|
| `sandbox.elixir.enabled` | boolean | `false` | Master switch |
| `sandbox.elixir.backend` | string | `auto` | `auto` (OS-aware resolver), or pin `apple_container`, `docker`, `podman_rootless`, `docker_runsc` |
| `sandbox.elixir.image` | string | project default | Approved local Elixir/OTP image reference or digest; never pulled during run |
| `sandbox.elixir.network` | string | `none` | v0.36 only permits `none` |
| `sandbox.elixir.cpu_limit` | number | bounded default | CPU quota |
| `sandbox.elixir.memory_mb` | integer | bounded default | Memory limit |
| `sandbox.elixir.timeout_ms` | integer | bounded default | Wall-clock limit |
| `sandbox.elixir.output_bytes` | integer | bounded default | stdout/stderr cap |

Settings store selections and bounds, not raw reports, source files, secrets, or
container logs. Reports live under the disposable draft/sandbox report root and
are surfaced read-only.

### 8. Allbert Home roots are explicit

`ensure_home!/0` creates:

- `<ALLBERT_HOME>/sandbox/bundles`
- `<ALLBERT_HOME>/sandbox/reports`
- `<ALLBERT_HOME>/sandbox/cache`

Tests and smokes must use a temporary `ALLBERT_HOME`. The sandbox bundle itself
receives its own temporary home; it never sees the operator's real home.

### 9. Signals, traces, and audit

Sandbox behavior is observable and security-relevant, so it routes through the
signal-driven runtime rather than ad-hoc logging. Backend resolution, bundle
build, command run, gate completion, denial, and cleanup emit signals on the
signal bus and write trace/audit records through the v0.31 audit/redaction
facades. Trace and audit payloads are bounded and redacted (no secrets, no
absolute real-home paths, no oversized output); full reports remain file-backed
under the sandbox report root and are surfaced read-only. Signals carry no
authority — they are evidence and observability only.

### M7: Local sandbox image preparation

## Non-Goals

- No LLM code generation agent committee; that is v0.37.
- No live in-core hot-load or runtime registration; that is v0.37.
- No template gallery, Mix generators, or Canvas Create surface; that is v0.38.
- No broad shell automation. Shell command support is limited to explicit
  allow-listed argv commands required by Elixir/OTP gate profiles.
- No package-manager execution, dependency installation, migrations, NIFs,
  ports, unrestricted network, implicit image pulls, remote builders,
  Firecracker, broader/cross-version Apple Container features beyond the
  optional doctor-gated v0.36 adapter, or multi-language targets.
- No route, action, permission, skill, settings, or active-app authority from a
  sandbox result.

## Definition Of Done

v0.36 is done when Allbert can create a disposable bundle for a generated
Elixir/OTP draft, run explicit allowed `mix` / `elixir` / `erl` gate commands in
an OS-aware sandbox (Docker/Podman baseline, `docker_runsc` preferred over plain
Docker when configured, optional Apple `container` when doctor-green) with no
network/secrets/real-home access, return bounded redacted reports, fail closed
without an on-platform backend, and pass security evals proving denied paths
stay denied. The result is a report only; no generated code is loaded or
registered in the core node in v0.36. Operator and developer docs name the
final setup, doctor, backend, SourcePolicy, report, and report-only authority
contracts.
