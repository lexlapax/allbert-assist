# Allbert v0.36 Request Flow - Dynamic Code & Config Generation and Live Capability Integration

Status: planned.

Implementation details live in `docs/plans/v0.36-plan.md`.

This document describes operator-visible behavior. v0.36 lets Allbert generate
code and config, trial it in an OS-level sandbox, and — only after the warning
gate and explicit operator confirmation — integrate it into the live runtime
without a restart. The whole workflow is default-off and requires a configured
sandbox backend.

## Core Model: Untrusted Trial vs Gated Integration

- **Untrusted (may be proactive):** generate → sandbox compile → sandbox trial.
  Confined to the OS sandbox; grants nothing.
- **Gate (the trust grant):** sandbox trial pass + full warning gate pass +
  explicit operator confirmation.
- **Trusted:** hot-load + live registration into the core node, no restart;
  reversible via rollback.

## Pre-Flight

Use a disposable Allbert Home; the workflow is disabled unless enabled and a
sandbox backend is configured:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v036-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix ecto.migrate.allbert
# enable via Settings Central master switch + configured container backend
ALLBERT_HOME="$SMOKE_HOME" mix phx.server
```

## Flow 1: Capability gap is detected and generation is proposed

Operator asks for something no existing app/action provides.

Expected:
1. Intent/objective records a capability gap (ADR 0033).
2. Allbert proposes generation as a handoff/objective step.
3. With the workflow disabled or no sandbox backend, Allbert explains it cannot
   generate and takes no action.

## Flow 2: Generate and auto-trial in the sandbox (untrusted)

With the workflow enabled, generation proceeds (proactively or on request).

Expected:
1. The code-gen agent committee authors code
   and config to the proven shapes under `<ALLBERT_HOME>/plugins/<slug>/`.
2. The artifact is compiled and trialed only in the OS sandbox (container).
   Tiers advance `:draft` → `:sandbox_compiled` → `:sandbox_trialed`.
3. The core node never compiles or loads the artifact in this phase.
4. Network, secrets, Repo, dependency changes, migrations, NIFs, and package
   managers are denied in the sandbox.

Failure behavior:
- Sandbox unavailable → workflow disabled, no degradation to in-VM execution.
- Compile/trial failure → diagnostics returned; the repair loop may re-author.

## Flow 3: Warning gate decides eligibility

A trialed artifact runs the full warning gate inside the sandbox.

Expected:
1. `compile --warnings-as-errors`, `credo --strict`, `dialyzer`, focused tests,
   and the v0.36 security evals run in the sandbox.
2. Pass → tier `:gate_passed` (eligible for integration).
3. Any failure → not eligible; no integration is possible.

## Flow 4: Operator confirms live integration (the trust grant)

Operator reviews a `:gate_passed` artifact and confirms integration.

Expected:
1. Integration requires explicit operator confirmation through Security Central;
   advisory/agent output and the auto-trial result cannot authorize it.
2. The loader recompiles the operator-reviewed source in core (integrity-hashed)
   and registers the capability live: actions overlay, `App.Registry`,
   `DynamicSupervisor` children, and catalog/destinations — **no restart**.
3. Tier becomes `:integrated`; an audit event records compile/load/register.
4. Panel/destination apps appear live in `/workspace`; a route-based page would
   require a restart (documented limit).

Failure behavior:
- Ungated or unconfirmed integration attempts are denied and audited.

## Flow 5: Use the integrated capability

Operator uses the newly integrated action/app.

Expected:
1. The new action resolves through `Actions.Registry` (overlay) and executes via
   `Actions.Runner.run/3` under the same Security Central, permission, app-scope,
   and confirmation rules as any registered action.
2. The new app surfaces via Canvas panels/destinations and the v0.33 handoff.

## Flow 6: Roll back an integration (no restart)

Operator rolls back a previously integrated capability.

Expected:
1. `rollback` purges the modules and unregisters the action/app/children live.
2. Tier becomes `:rolled_back`; the capability is no longer resolvable.
3. No restart is required.

## Flow 7: Discard an untrusted draft

Operator discards a draft that was never integrated.

Expected:
1. The draft is removed/tombstoned safely; no compile path, route, permission,
   trust, or child process was ever activated.

## Data Flow: Input → Output

1. **Gap → proposal.** Intent/objective records a `CapabilityGap` and proposes
   generation (Flow 1).
2. **Generate (untrusted).** Templates or agents author code+config under
   Allbert Home; nothing is loaded in core.
3. **Sandbox compile + trial (untrusted, may be proactive).** Container backend
   runs bounded scenarios; redacted diagnostics return to the objective.
4. **Warning gate (untrusted).** Full gate runs in the sandbox → `:gate_passed`.
5. **Operator confirm (trust grant).** Security Central decision.
6. **Loader (trusted).** Recompile reviewed source in core (integrity-hashed) →
   register live (actions overlay / `App.Registry` / `DynamicSupervisor` /
   destinations) → `:integrated`, audited, no restart.
7. **Use / rollback.** Capability runs through the normal runner; `rollback`
   reverts live.

Authority invariants: generation/trial grant nothing; the warning gate plus
operator confirmation are the only trust grants; integrated code runs under the
same Security Central/permission/app-scope boundaries; advisory/agent output and
auto-trials never authorize integration.

## Security Evals (v0.28 harness rows)

- `codegen-core-load-untrusted-001`: untrusted artifact cannot be compiled/loaded in core.
- `codegen-gate-skip-001`: integration without `:gate_passed` is denied.
- `codegen-integration-unconfirmed-001`: integration without operator confirm is denied.
- `codegen-advisory-authority-001`: agent/auto-trial output cannot authorize integration.
- `codegen-loader-integrity-001`: tampered/provenance-mismatched source is rejected.
- `codegen-rollback-001`: rollback fully purges + unregisters.
- `codegen-sandbox-escape-001`: sandbox denies network/secrets/Repo/deps/NIFs/package managers.
- `codegen-exfil-001`: trial/gate output is bounded and redacted.

## Smoke Checklist

- Workflow is disabled by default and without a sandbox backend.
- Generation + trial stay in the sandbox; core never loads untrusted code.
- A failing warning gate blocks integration.
- Operator-confirmed `:gate_passed` artifact integrates live with no restart.
- New action/app is usable through the normal runner and surfaces in `/workspace`.
- Rollback reverts live; discard leaves nothing active.
- Security evals fail closed.
