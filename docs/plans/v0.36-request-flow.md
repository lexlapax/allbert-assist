# Allbert v0.36 Request Flow - Elixir/OTP Sandbox And Gate Runner

Status: implementation reopened for local sandbox image preparation. The M0-M5
sandbox substrate is implemented as `v0.36.0`; release readiness now requires
the v0.36 image-preparation task.

Implementation details live in `docs/plans/v0.36-plan.md`.

v0.36 gives Allbert a narrow OS sandbox for generated Elixir/OTP drafts and the
explicit shell commands required to compile, test, and gate them. It does not
generate code, hot-load modules, integrate live capabilities, or run broad shell
automation. Those later flows begin in v0.37 and v0.38.

## Core Model

- **Disabled by default:** no sandbox command runs until Settings Central enables
  the workflow and `mix allbert.sandbox doctor` finds a supported backend.
- **Copy-in/copy-out:** sandbox bundles use a disposable project snapshot and
  disposable `ALLBERT_HOME`; reports are copied out, not code execution state.
- **Explicit argv:** allowed commands are structured executable + argv. Shell
  strings, `sh -c`, chaining, globbing, redirection, PTY, and daemons are
  rejected before backend execution.
- **Elixir/OTP only:** initial commands are `mix`, `elixir`, and `erl` for
  compile/test/gate profiles.
- **Report only:** a passing gate report grants no live authority.

## Pre-Flight

Use a disposable Allbert Home:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v036-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix ecto.migrate.allbert
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sandbox doctor
```

Expected before configuration:

- `sandbox.elixir.enabled=false`.
- Doctor reports disabled/unavailable.
- No container command runs.

After operator configuration through Settings Central:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings set sandbox.elixir.enabled true
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sandbox image build
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sandbox image verify
# backend defaults to `auto` (OS-aware resolver); pin one only to override:
# ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings set sandbox.elixir.backend docker_runsc
# ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings set sandbox.elixir.backend apple_container
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sandbox doctor
```

With `backend=auto`, doctor resolves an OS-aware chain and reports every
candidate considered. `docker_runsc` is preferred over plain Docker when the
Docker daemon exposes `runsc`; Apple `container` is optional and selected only
when the host is Apple silicon, macOS 26+, the CLI is present, and policy
enforcement passes. If no on-platform backend is available, doctor reports
fail-closed status and the runner remains unusable.

If Docker is available but `allbert-elixir-otp:local` is missing or invalid,
doctor must point the operator to `mix allbert.sandbox image build` and
`mix allbert.sandbox image verify`. Sandbox command execution still uses
`--pull=never`; the image build task is the only v0.36 path that prepares the
local image.

## Flow 0: Prepare the approved local sandbox image

Operator runs:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sandbox image build
ALLBERT_HOME="$SMOKE_HOME" mix allbert.sandbox image verify
```

Expected:

1. Build uses explicit Docker argv and tags the configured
   `sandbox.elixir.image` value.
2. The image includes labels for Allbert version, Elixir version, OTP version,
   and the project lockfile fingerprint used to prepare the sandbox image.
3. Verification uses local image inspection and a small `docker run
   --pull=never` command.
4. Reports are written under `<ALLBERT_HOME>/sandbox/reports` with redacted
   host paths and bounded output.
5. Missing daemon, missing image, invalid labels, or verification command
   failure leaves the backend unavailable and does not run sandbox gates.

Implementation status: planned for the v0.36 image-preparation correction.

## Flow 1: Doctor reports backend status

Operator runs `mix allbert.sandbox doctor`.

Expected:

1. Doctor reads Settings Central policy.
2. When `backend=auto`, the resolver detects the host (OS, arch, macOS
   version) and walks the OS-aware fallback chain, checking each candidate's
   availability and policy compatibility. `docker_runsc` outranks plain Docker
   when configured. A pinned backend is checked directly.
3. It reports the resolved backend, the candidates considered and why each
   passed or failed, plus image, network policy, resource limits, and missing
   prerequisites — without running untrusted code.
4. No on-platform backend or a disabled setting leaves v0.36 sandbox actions
   denied (fail closed).

Implementation status: M1 complete. `mix allbert.sandbox doctor` now reports
the default-off posture, settings, roots, backend candidates, selected backend
when enabled and doctor-green, and fail-closed diagnostics.

## Flow 2: Build a sandbox bundle

A future dynamic draft or fixture asks for a sandbox bundle.

Expected:

1. Bundle builder creates a disposable bundle root under
   `<ALLBERT_HOME>/sandbox/bundles`.
2. It copies in only allow-listed project files, draft files, test files, and a
   disposable sandbox `ALLBERT_HOME`.
3. It excludes the operator's real Allbert Home, settings, secrets, databases,
   caches, `.git`, host temp dirs, Docker socket, and arbitrary filesystem
   paths.
4. It records bundle metadata and redacted provenance.
5. It verifies the approved local image reference or digest is already present;
   no image pull or registry access happens during bundle execution.

Failure behavior:

- Traversal, symlink escape, oversize input, forbidden file type, or real-home
  inclusion fails before backend execution.
- Missing local image, unapproved image, or image reference without the required
  labels/policy metadata fails closed and points to the image-preparation task.

Implementation status: M2 complete. Bundle creation now copies bounded
allow-listed project/draft/test inputs into disposable bundle roots, creates a
sandbox-local home, writes redacted metadata, excludes repo/runtime directories,
and denies traversal, symlink, real-home, missing, oversize, and unsupported
inputs before backend execution.

## Flow 3: Run an allowed command

Allbert runs a `Sandbox.CommandSpec`, for example:

```text
executable: mix
argv: ["compile", "--warnings-as-errors"]
```

Expected:

1. CommandSpec validation accepts only `mix`, `elixir`, or `erl` plus an
   allow-listed argv shape.
2. `SourcePolicy` statically scans draft and trial files for known forbidden
   constructs (`System.cmd`, `Port.open`, `:os.cmd`, `Code.eval_*`,
   `Code.compile_*`, `Code.require_*`, `Mix.install`, NIF loading, broad host
   file traversal) before execution.
3. Backend starts a no-network, resource-bounded container with only the bundle
   mounted.
4. Result includes exit status, duration, capped stdout/stderr, backend
   metadata, and redacted diagnostics.
5. Output is written to a report root, not Settings Central.

Rejected examples:

- `sh -c "mix test && curl example.com"`
- `mix deps.get`
- `mix ecto.migrate`
- `curl https://example.com`
- `git clone ...`
- command strings containing redirects, pipes, globs, substitutions, or
  background execution
- any run that would pull an image or contact a registry

Implementation status: M3 complete. `Sandbox.CommandSpec` accepts explicit argv
only, validates profile-specific shapes, keeps env to a small allow-list, and
denies shell syntax, dependency/package-manager flows, migrations, secret env,
cwd escape, runtime eval, and unsupported executables. Docker, Docker+runsc,
and rootless Podman backends now assemble explicit container engine argv with
local-image/no-pull policy, no network, bounded resources, read-only
project/draft/test mounts, writable sandbox-home/report mounts, and no host
Docker socket. `SourcePolicy` runs before engine execution and each outcome
writes a bounded redacted JSON report under the bundle report root.

## Flow 4: Run the Elixir gate

Allbert runs a named gate profile.

Expected:

1. Gate runner executes the configured steps inside the sandbox:
   compile-WAE, focused tests, Credo, Dialyzer, selected security evals.
2. Each step is an explicit CommandSpec.
3. Pass produces a structured gate report.
4. Fail produces bounded diagnostics and keeps the draft untrusted.
5. No core-node compile/load/register occurs.

Implementation status: M4 complete. `AllbertAssist.Sandbox.GateRunner` maps
reviewed profiles to explicit CommandSpecs, runs them through
`Sandbox.run_command/3`, halts on first non-completed step, and writes an
aggregate report under the bundle report root.

## Flow 5: Denied paths fail closed

An adversarial draft attempts network, secret, Repo, package manager, NIF, port,
or path escape behavior.

Expected:

1. Static checks deny known forbidden constructs before execution where
   possible.
2. Backend policy denies no-network/mount/env violations.
3. Report redacts secrets, absolute real-home paths, and raw oversized output.
4. Security eval rows record a denied decision.

Implementation status: M4 complete. v0.36 sandbox eval rows are registered in
the v0.28 inventory and exercised by `AllbertAssist.Security.SandboxEvalTest`.
They cover disabled backends, resolver fail-closed behavior, local-image/no-pull
argv, source policy, shell denial, network denial, secret env denial,
disposable-home isolation, package-manager denial, NIF/port denial, core-load
denial, and report redaction.

## Flow 6: Optional gVisor/runsc backend

Operator selects `sandbox.elixir.backend=docker_runsc`.

Expected:

1. Doctor verifies Docker exposes a `runsc` runtime.
2. Missing `runsc` reports unavailable and fails closed.
3. When available, the backend uses the same v0.36 CommandSpec, bundle, no
   network, mount, resource, and report policy.
4. `backend=auto` prefers `docker_runsc` over plain Docker whenever both are
   doctor-green.
5. No feature depends on gVisor being present.

Implementation status: M3 complete for Docker+runsc execution. The run path
uses the same Docker container contract as plain Docker and injects
`--runtime runsc`; missing `runsc` remains a doctor-time unavailable condition.

## Flow 7: macOS Apple container backend

Operator runs on Apple silicon with `backend=auto` (or pins
`sandbox.elixir.backend=apple_container`).

Expected:

1. Doctor verifies Apple silicon, macOS 26+, the `container` CLI, and that
   policy enforcement (no network, resource limits, non-root copy-in) is
   available.
2. When all checks pass, `apple_container` is eligible and each command runs in
   its own lightweight VM under the same v0.36 CommandSpec, bundle, no-network,
   resource, and report policy.
3. When any check fails (older macOS, Intel, missing CLI, unsupported policy),
   doctor marks `apple_container` unavailable and `"auto"` falls back to the
   next doctor-green candidate (`docker_runsc` when configured, otherwise
   `docker`); a pinned `apple_container` fails closed.
4. Apple `container` is not release-blocking; missing support is recorded as an
   unavailable candidate, not a v0.36 failure.

Implementation status: M4 complete. The Apple candidate is registered and
doctor-gated, but remains unavailable until doctor can prove local CLI policy
support; this keeps `auto` fallback fail-closed and non-release-blocking.

## Data Flow

1. Settings Central enables the sandbox and sets backend (`"auto"` by default)
   and bounds.
2. The OS-aware resolver and doctor validate host/backend posture and select an
   available on-platform backend (or fail closed).
3. Bundle builder copies in bounded Elixir/OTP inputs.
4. CommandSpec validates explicit argv.
5. Backend runs containerized command.
6. Gate runner aggregates step reports.
7. Report is surfaced read-only to future v0.37/v0.38 flows.

Authority invariant: the report is evidence, not authority. It cannot load
modules, register actions, grant permissions, create routes, enable skills, or
set `active_app`.

## CLI Handles

| Surface | Handle | Use |
|---|---|---|
| Doctor | `mix allbert.sandbox doctor` | Backend readiness |
| Settings | `sandbox.elixir.*` | Default-off policy |
| Permission | `permissions.sandbox_trial` | Security Central action boundary |
| Actions | `sandbox_doctor`, `build_sandbox_bundle`, `run_sandbox_command`, `run_sandbox_gate`, `discard_sandbox_bundle` | Internal runtime sandbox lifecycle |
| Reports | `<ALLBERT_HOME>/sandbox/reports` | Read-only diagnostics |
| Bundle root | `<ALLBERT_HOME>/sandbox/bundles` | Disposable copy-in input |

## Security Evals (v0.28 harness rows)

- `sandbox-backend-disabled-001`: disabled or missing backend denies execution.
- `sandbox-backend-resolver-001`: `"auto"` resolves the correct on-platform
  backend and fails closed when none is available; a host that cannot enforce
  policy is never selected.
- `sandbox-image-local-only-001`: sandbox runs never pull images or contact a
  registry.
- `sandbox-source-policy-001`: forbidden Elixir constructs are denied before
  backend execution.
- `sandbox-command-shell-deny-001`: shell strings/chaining/redirection are denied.
- `sandbox-network-deny-001`: outbound network is denied.
- `sandbox-secret-deny-001`: secrets/env credentials are not present or leaked.
- `sandbox-home-isolation-001`: real Allbert Home is not mounted or readable.
- `sandbox-package-manager-deny-001`: package-manager/dependency install commands
  are denied.
- `sandbox-nif-port-deny-001`: NIF/port/native execution attempts are denied.
- `sandbox-core-load-deny-001`: sandbox result cannot load modules in core.
- `sandbox-report-redaction-001`: reports are bounded and redacted.

## Smoke Checklist

- Doctor is disabled by default.
- Enabling without a configured backend still fails closed.
- Safe `mix compile --warnings-as-errors` fixture runs in the sandbox.
- Approved image must be present locally; missing/unapproved image fails closed.
- Network, secret, real-home, package-manager, NIF/port, and shell-string
  attempts are denied.
- Gate report is bounded/redacted.
- Core node never loads or registers sandboxed code.
