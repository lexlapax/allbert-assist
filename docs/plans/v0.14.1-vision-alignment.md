# v0.14.1 — Vision alignment

Status: Shipped

## Summary

v0.14.1 is a patch release that closes the gap between what the docs say has shipped and what an operator actually gets. v0.13 marked adapter training as shipped, but daemon adapter protocol requests still return `adapter_surface_not_implemented` and production training still constructs the fake backend instead of the configured trainer. v0.14 marked self-diagnosis remediation as shipped, but the remediation routes write placeholders that point back at the diagnosis report instead of generating candidate fixes. The fresh-profile local default, Ollama Gemma4, also emits tool-call shapes the strict XML parser rejects.

v0.14.1 has five release-blocking pillars:

1. **Doc reconciliation.** Mark partial surfaces as partial as of v0.14 and tracked by v0.14.1 in plans, README, CHANGELOG, vision, release notes, and operator docs. Add a doc-reality gate so future overclaims fail local validation.
2. **Tool-call parser robustness.** Accept the schema variants local models actually emit, retry once with a corrective system message on parse failure, and add a provider-native `tools` seam to `LlmProvider` for v0.15+.
3. **Daemon adapter protocol wiring.** Replace the v5 adapter stubs with daemon handlers that bridge to the existing local `AdapterStore`, adapter runtime helpers, and `PersonalityAdapterJob` paths.
4. **Adapter trainer factory.** Read `learning.adapter_training.default_backend` for production training. Real configured backends must run; fake is allowed only for tests, preview, or an explicit fake backend.
5. **Diagnosis remediation upgrade.** Generate concrete candidate fixes through a bounded LLM call, preserve the full self-change safety envelope, and keep the report-only scaffold as fallback.

The release does not add a new feature layer. It makes already-promised surfaces honest and usable. v0.14.2 owns the kernel crate split; v0.15 owns the RAG foundation, with growth-loop ingestion parked for a later design pass.

- Make the v0.13 adapter daemon protocol functional for TUI, Telegram, and any v5+ peer.
- Make production adapter training pick the configured backend instead of silently falling back to fake.
- Make `diagnose run --remediate` produce a concrete candidate artifact a reviewer can read.
- Make tool calls work on the default Ollama Gemma4 profile.

- Keep existing v0.12.x/v0.13/v0.14 invariants intact: no automatic patch install, no binary auto-swap, no Lua auto-enable, single-slot adapter activation, hosted providers ignore adapters, review-first remediation.

## Plan hardening decisions

These are release-blocking. Implementation should not invent alternative reconciliation policy, parser shape, factory wiring, remediation prompt, or safety envelope while landing the patch.

### Doc reconciliation gates

Doc claims are reconciled by adding qualifications, not by deleting shipped sections. During implementation, overclaims used the phrase "partial as of v0.14; tracked by v0.14.1". Release closeout replaces that wording with "reconciled in v0.14.1" only for surfaces actually fixed.

`tools/check_doc_reality.sh` enforces three rules:

- Known overclaim phrases require nearby qualification. Allowed qualifiers are "partial as of v0.x", "planned for v0.x.y", "reconciled in v0.x.y", "scaffolded", or `Status: Stub`.
- Code escape hatches such as `_not_implemented`, `todo!`, `unimplemented!`, and production fake-backend construction require a doc-side acknowledgement in a plan, upgrade note, or operator limitation.
- Roadmap rows must match plan statuses: every `Shipped` roadmap row points to a plan whose `Status:` is `Shipped`.

The script is wired into the standard `cargo test` path through a Rust integration test wrapper that invokes the script. CI uses the same path when present.

ADR 0095 records the policy.

### Tool-call parser robustness

The kernel keeps the XML-tagged tool protocol as universal default but makes the parser tolerant to common variants emitted by local/small models:

- `<tool_call>{"name": ..., "input": {...}}</tool_call>` — canonical.
- `<tool_call>{"name": ..., "arguments": {...}}</tool_call>` — OpenAI-style alias.
- `<tool_call>{"tool": ..., "input"|"args"|"arguments"|"parameters": {...}}</tool_call>` — local-model alias with object input.
- `<tool_call>{"function": {"name": ..., "arguments": {...}}}</tool_call>` — nested OpenAI legacy shape.
- `<tool_call>{"program": <str>, "args": [<str>...]}</tool_call>` — direct-spawn shape, normalized to `process_exec`.

Parsing and authorization are two stages. The XML/JSON parser only extracts a
`ParsedToolCall`; the resolver converts it into a runnable `ToolInvocation`
after it has the turn's active tool catalog, active skill tool policy, and
security config.

The direct-spawn shape is resolved only when all of these are true:

- `process_exec` is present in the active tool catalog for the turn.
- Every active skill that constrains tool use permits `process_exec` through `allowed-tools`.
- The program and arguments pass the existing `security.exec_allow` / `security.exec_deny` policy.
- The generated `process_exec` input is the same structured input the native tool already expects; no shell string is introduced.

For direct-spawn calls, `args` is treated as the argv tail. If the first
argument exactly equals `program` or `basename(program)`, the resolver drops
that duplicate command name before policy checks and records the normalization
reason in trace metadata. This handles local models that emit
`{"program":"date","args":["date"]}` without turning it into `date date`.

Unknown shapes fall through to a single corrective retry. Parse/resolve runs
before assistant text persistence and before `AssistantText` is emitted; a
failed first response is replaced by the corrective retry path rather than being
stored or shown as literal `<tool_call>...` output. The retry adds one provider
call at most, uses the same active tool catalog, and is controlled by:

```toml
[intent]
tool_call_retry_enabled = true
```

Default: `true`. Validation accepts only booleans. If disabled or if the retry also fails, the kernel surfaces an operator-facing remediation instead of passing literal `<tool_call>...` text through to the user.

The `LlmProvider` trait gains an additive seam:

```rust

pub struct CompletionRequest {
```

### Daemon adapter protocol wiring

The handler at `crates/allbert-daemon/src/server.rs:1183-1205` is replaced with explicit per-message branches for every protocol v5 adapter request. The daemon remains a thin RPC bridge over disk-backed adapter state:

- list/show/remove/history/status use `AdapterStore` and the existing compute/history helpers.
- activate/deactivate use the existing `activate_adapter` and `deactivate_adapter` runtime helpers.
- external install writes into the existing adapter incoming/quarantine path and creates an `adapter-approval`.
- training start calls the production `PersonalityAdapterJob` constructor from the v0.14.1 trainer factory.
- training start preallocates the run id and cancellation token in the daemon,
  passes both into `PersonalityAdapterJob`, and immediately emits an
  `AdapterTrainingProgress` message with the run id and `phase = "queued"` so a
  peer can cancel before the trainer backend begins work.
- training cancel uses an in-memory active-training registry keyed by run id; each entry stores the supervised task handle and cancellation token. The registry is not a new durable source of truth.

`AdapterTrainingProgress` and `AdapterTrainingFinal` server messages are emitted from the existing training progress/event path. The daemon writes a terminal run-status manifest for success, failure, and cancellation; `AdapterTrainingFinal.manifest_path` points to that manifest even when no trained adapter artifact exists. No new event kind or protocol version is introduced. Per-peer protocol filtering remains: v2/v3/v4 peers do not receive adapter responses; v5+ peers do.

ADR 0097 records the bridge.

### Adapter trainer factory

A new `crates/allbert-kernel/src/adapters/factory.rs` exposes production trainer selection:

```rust
pub fn build_trainer(
    paths: &AllbertPaths,
    config: &Config,
    requested_backend: Option<&str>,
) -> Result<Arc<dyn AdapterTrainer>, KernelError>;
```

Production behavior:

- `learning.adapter_training.enabled = false` returns a refusal with a remediation hint; it does not return fake.
- The effective backend is `requested_backend` when present, otherwise `learning.adapter_training.default_backend`.
- CLI and scheduled training normally pass `None`. Daemon `AdaptersTrainingStart` passes `req.backend.as_deref()` from protocol v5.
- empty or unknown effective backend returns a refusal naming `learning.adapter_training.default_backend`, `learning.adapter_training.allowed_backends`, the request-level `backend`, and `security.exec_allow`.
- request-level backend overrides are allowed only when the requested backend is in `learning.adapter_training.allowed_backends`, passes the same exec-policy gates, and carries `override_reason` in daemon/protocol paths.
- `effective_backend = "mlx-lm-lora"` constructs `MlxLoraTrainer` after the allowlist and exec-policy gates pass.
- `effective_backend = "llama-cpp-finetune"` constructs `LlamaCppLoraTrainer` after the same gates pass.
- `effective_backend = "fake"` constructs `FakeAdapterTrainer` only because the operator, test profile, or explicit request selected `fake` and `fake` is allowed.

Preview and test behavior:

- `PersonalityAdapterJob::default()` and `PersonalityAdapterJob::fake()` remain test-friendly fake constructors.
- `adapters training preview` never invokes a real trainer. It renders corpus/settings/backend intent and may use fake metadata, but it must label preview output as non-training.
- CLI `adapters training start`, daemon `AdaptersTrainingStart`, and scheduled adapter jobs all use the production factory.

ADR 0098 records the factory.

### Diagnosis remediation upgrade

`route_code_remediation`, `route_skill_remediation`, and `route_memory_remediation` gain a candidate-generation pre-step. The kernel issues one bounded provider call with the diagnosis bundle and report:

- **Code:** output only a unified diff under the configured source checkout.
- **Skill:** output only a complete `SKILL.md` body with frontmatter, `allowed-tools`, and a populated `## Behavior` section.
- **Memory:** output only a memory candidate of at least 64 characters, grounded in the bundle evidence.

Generation is gated by:

- `self_diagnosis.allow_remediation = true`;
- explicit command-originated `--remediate <kind> --reason <text>`;
- the daily monetary cost cap;
- `self_diagnosis.remediation_provider_max_tokens`:

```toml
[self_diagnosis]
remediation_provider_max_tokens = 4096
```

Validation clamps the token cap to `[256, 16384]`.

Provider and cost ownership is explicit:

- Daemon-started remediation uses the attached session's effective provider and
  effective model, matching normal assistant turns for that peer/session.
- CLI/offline remediation paths do not silently construct a provider. If no
  attached/effective provider is available, they keep the diagnosis report and
  emit the report-only scaffold with `candidate_status =
  "fallback:offline_no_provider"`.
- Provider construction or availability failure emits `candidate_status =
  "fallback:provider_unavailable"`.
- The daily monetary cost cap is checked before the candidate-generation call.
  Cost-cap refusal emits `candidate_status = "fallback:cost_cap"`.
- Candidate-generation usage is recorded through the same cost ledger and hook
  path as normal model calls. The approval/quarantine/staging frontmatter
  records provider, model, tokens, estimated cost, diagnosis id, and fallback or
  validation status.

Code-shaped remediation preserves the existing self-change envelope:

- create and use a sibling worktree, never the active source checkout;
- reject diffs with absolute paths, parent traversal, `.git` paths, runtime profile paths, generated adapter artifacts, or files outside the source tree;
- apply the candidate diff only inside the sibling worktree;
- run Tier A validation (`cargo fmt --check`, clippy, tests, and CLI help unless the existing self-improvement policy narrows it);
- record validation status, command output paths, candidate provider/model, token usage, and diagnosis id in the `patch-approval` metadata.

A candidate with failed validation can still be routed for review only when it is clearly marked `candidate_status = "validation_failed:<reason>"`; it is never installed automatically. Provider failure, provider unavailability, offline/no-provider paths, empty output, schema failure, cost-cap refusal, or disabled remediation falls back to the existing report-only scaffold and records `candidate_status = "fallback:<reason>"`.

ADR 0099 records candidate generation and the safety envelope.

## Public interfaces

### Config

Add:

```toml
[intent]
tool_call_retry_enabled = true

[self_diagnosis]
remediation_provider_max_tokens = 4096
```

Validation:

- `tool_call_retry_enabled` is boolean, default `true`.
- `remediation_provider_max_tokens` clamps to `[256, 16384]`, default `4096`.

Implementation placement and migration:

- `[intent]` is a new root table represented by `Config.intent:
  IntentRuntimeConfig`. It is intentionally separate from the existing
  `intent_classifier` table, which continues to own classifier thresholds and
  model-selection settings.
- `self_diagnosis.remediation_provider_max_tokens` is added to the existing
  `SelfDiagnosisConfig` struct.
- Existing profile TOML files load through serde defaults; v0.14.1 does not
  rewrite user config just to add default keys.
- Settings descriptors expose `intent.tool_call_retry_enabled` and
  `self_diagnosis.remediation_provider_max_tokens` for `settings show/get/set`
  style surfaces, including validation errors for non-boolean retry settings
  and out-of-range token caps.

### CLI

No new top-level commands.

Existing surfaces gain:

- `allbert-cli doctor` reports tool-call parser variant coverage, daemon adapter-handler health, and trainer factory backend resolution.
- `allbert-cli adapters training start` refuses disabled/missing/unknown production backends instead of silently using fake.
- `allbert-cli diagnose run --remediate ...` prints both the diagnosis report path and candidate artifact path, plus fallback or validation status.

### Provider trait

`LlmProvider::complete` accepts `CompletionRequest.tools` and returns `CompletionResponse.tool_calls`. v0.14.1 providers ignore `tools` and return an empty `tool_calls` vector.

## Implementation design

### Tool-call parser

`crates/allbert-kernel/src/tool_call_parser.rs` owns XML-tag extraction, JSON shape parsing, direct-spawn resolution, and corrective retry text:

```rust
pub enum ParsedToolCall {
    Named {
        name: String,
        input: serde_json::Value,
    },
    DirectSpawn {
        program: String,
        args: Vec<String>,
    },
}

pub fn parse_tool_call_blocks(text: &str) -> Result<Vec<ParsedToolCall>, ToolParseError>;

pub fn resolve_tool_calls(
    parsed: Vec<ParsedToolCall>,
    catalog: &ActiveToolCatalog,
    active_skill_policy: &ActiveSkillToolPolicy,
    security: &SecurityConfig,
) -> Result<Vec<ToolInvocation>, ToolParseError>;

pub fn corrective_retry_message(tool_catalog: &str) -> String;
```

`parse_tool_call_blocks` is deliberately policy-free. `resolve_tool_calls` is where direct-spawn calls prove `process_exec` is active, skill-allowed, and exec-policy permitted before the normal dispatcher sees a `ToolInvocation`. The dispatcher keeps its existing final policy check as defense in depth.

### Trainer factory

`crates/allbert-kernel/src/adapters/factory.rs` owns production trainer selection. Tests that need fake call the explicit fake constructors. Production callers do not construct `FakeAdapterTrainer` directly.

### Diagnosis candidate generator

Candidate generation remains in `crates/allbert-kernel/src/self_diagnosis.rs` for v0.14.1. The later structural release may move it, but this patch release avoids unrelated module movement.
