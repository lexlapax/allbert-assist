# v0.14.1 â€” Vision alignment

Status: Draft

## Summary

v0.14.1 is a patch release that closes the gap between what the docs say has shipped and what an operator actually gets. v0.13 marked adapter training as shipped, but the daemon adapter protocol surfaces return `adapter_surface_not_implemented` and the trainer factory always wires the fake backend. v0.14 marked self-diagnosis remediation as shipped, but the three remediation routes write empty review artifacts that point back at the diagnosis report. The local-default Gemma4 profile cannot reliably call tools because the kernel's XML-tagged tool protocol is parsed strictly against `name`+`input` while small models emit other shapes.

v0.14.1 has six release-blocking pillars:

1. **Doc reconciliation.** Mark partial surfaces as partial in plans, README, CHANGELOG, vision, and operator docs. Add CI grep gates so the same overclaims cannot land again unqualified.
2. **Tool-call parser robustness.** Accept the schema variants local models actually emit, retry once with a corrective system message on parse failure, and add a `tools` seam on `LlmProvider` so v0.15 can switch to provider-native tool calling.
3. **Daemon adapter protocol wiring.** Replace the `adapter_surface_not_implemented` stubs with handlers that bridge to the existing local `AdapterStore` and `PersonalityAdapterJob` paths the CLI already uses.
4. **Adapter trainer factory.** Read `learning.adapter_training.default_backend` and construct `MlxLoraTrainer`, `LlamaCppLoraTrainer`, or `FakeAdapterTrainer` accordingly. The fake backend stops being the silent default whenever real backends are configured.
5. **Diagnosis remediation upgrade.** Generate concrete candidate fixes (a real diff, a populated skill body, a substantive memory candidate) via a bounded LLM call against the diagnosis bundle. Keep the report-only path as fallback. Telegram remains structural-only.
6. **Kernel split.** Move `adapters/`, `self_diagnosis`, `local_utilities`, `self_improvement`, `scripting`, and `replay` from `allbert-kernel` into a sibling `allbert-kernel-services` crate. `lib.rs` shrinks to the kernel core (agent loop, hooks, policy, memory contracts, provider seam) without changing the public re-export surface for clients of `allbert_kernel::*`.

The release does not add new operator-visible surfaces beyond honest tool calling and honest remediation. It does not introduce new approval kinds, new protocol versions, or new ingestion paths. v0.15 owns the growth loop.

## Goals

- Make the v0.13 adapter daemon protocol functional for TUI, Telegram, and any v5+ peer.
- Make adapter training pick the configured real backend instead of falling back to fake.
- Make `self-diagnose --remediate` produce a candidate fix that a reviewer can read, not an empty form.
- Make tool calls work on the default Ollama Gemma4 profile.
- Add a `tools` field on `LlmProvider` so v0.15 can land provider-native tool calling without a second migration.
- Reconcile every doc claim that says "shipped" against the running code, and add CI gates that fail when an unqualified shipped claim regresses.
- Reduce kernel `lib.rs` below 4,000 LOC by moving owned services into a sibling crate without changing public API surface.
- Keep all existing v0.12.x/v0.13/v0.14 invariants intact: no automatic patch install, no binary auto-swap, no Lua auto-enable, single-slot adapter activation, hosted providers ignore adapters, review-first remediation.

## Non-goals

- No new release-defining capability layer. Growth-loop ingestion is v0.15.
- No new protocol version. Protocol v6 is preserved; the kernel split is internal.
- No new inbox kind. Diagnosis-driven fixes still route through `patch-approval`, skill quarantine, and staged memory.
- No native provider tool-calling implementation in v0.14.1. The `tools` seam is added; provider-native dispatch lands additively in v0.15.
- No multi-adapter stacking or hosted training opt-in.
- No removal of the XML-tagged tool protocol. It remains the universal default until provider-native tool calling lands behind a per-provider feature flag.
- No vision/roadmap restructure. v0.14.1 reconciles claims; the deferred-ambitions list stays as is.
- No README rewrite. Reconciliation is surgical; it edits the specific overclaim lines and adds qualifications, not the whole document.

## Plan hardening decisions

These are release-blocking. Implementation should not invent alternative reconciliation policy, parser shape, factory wiring, remediation prompt, or kernel-split boundary while landing the patch.

### Doc reconciliation gates

Doc claims are reconciled by adding qualifications, not by deleting shipped sections. The reconciliation pass edits only the strings the v0.14.1 review flagged. CI gates are the long-lived guardrail:

- `tools/check_doc_reality.sh` greps the docs tree for known overclaim patterns ("adapter daemon protocol", "real backend training", "candidate remediation", "self-diagnosed code/skill/memory fix") and requires each occurrence to be within 5 lines of either "partial", "scaffolding", or "v0.14.1 reconciled" annotations.
- The same script greps the codebase for `adapter_surface_not_implemented`, `unimplemented!`, `todo!`, and `FakeAdapterTrainer` outside test modules; any new instance must be paired with a doc note.
- The script runs in the standard `cargo test` path (Tier A validation). Failure is release-blocking.

### Tool-call parser robustness

The kernel keeps the XML-tagged tool protocol as universal default but makes the parser tolerant to common variants emitted by local/small models. Specifically:

- `<tool_call>{"name": ..., "input": ...}</tool_call>` â€” the existing canonical shape.
- `<tool_call>{"name": ..., "arguments": ...}</tool_call>` â€” OpenAI-style alias.
- `<tool_call>{"tool": ..., "input"|"args"|"arguments"|"parameters": ...}</tool_call>` â€” Gemma/Mistral-style alias.
- `<tool_call>{"function": {"name": ..., "arguments": ...}}</tool_call>` â€” nested OpenAI legacy shape.
- `<tool_call>{"program": <str>, "args": [<str>...]}</tool_call>` â€” direct-spawn shape, mapped only to `process_exec` and only when the program matches the active exec policy.

The normalization is deterministic. Unknown shapes fall through to a single-shot retry: the kernel re-prompts with a corrective system message naming the canonical shape and the active tool catalog, then parses the retry response. If the retry also fails, the kernel surfaces a clear remediation rather than passing the literal `<tool_call>...` text through to the user.

The `LlmProvider` trait gains an additive `tools` seam:

```rust
pub struct ToolDeclaration {
    pub name: String,
    pub description: String,
    pub schema: serde_json::Value,
}

pub struct CompletionRequest {
    pub system: Option<String>,
    pub messages: Vec<ChatMessage>,
    pub model: String,
    pub max_tokens: u32,
    pub tools: Vec<ToolDeclaration>,         // NEW
}

pub struct ToolCallSpan {
    pub call_id: String,
    pub name: String,
    pub input: serde_json::Value,
}

pub struct CompletionResponse {
    pub text: String,
    pub usage: Usage,
    pub tool_calls: Vec<ToolCallSpan>,        // NEW; empty in v0.14.1
}
```

In v0.14.1 every shipped provider sets `tool_calls = vec![]` and the kernel keeps using XML-tagged text. v0.15 implements provider-native tool calling per provider and the kernel's call site can prefer `tool_calls` when non-empty. The seam is added now so we don't change `LlmProvider` again in v0.15.

ADR 0096 records the parser policy and the deferred provider-native seam.

### Adapter trainer factory

A new `crates/allbert-kernel/src/adapters/factory.rs` exposes:

```rust
pub fn build_trainer(
    paths: &AllbertPaths,
    config: &Config,
) -> Result<Arc<dyn AdapterTrainer>, KernelError>;
```

The factory reads `learning.adapter_training.enabled`, `allowed_backends`, and `default_backend`. It constructs:

- `MlxLoraTrainer` when `default_backend == "mlx-lm-lora"`.
- `LlamaCppLoraTrainer` when `default_backend == "llama-cpp-finetune"`.
- `FakeAdapterTrainer` when `default_backend == "fake"`, when adapter training is disabled, or when called from explicit test/preview paths.

`PersonalityAdapterJob::default()` keeps returning fake (test-friendly), but every production caller â€” daemon, CLI `adapters training start`, scheduled `personality-adapter` job â€” must go through `build_trainer`. The factory rejects unknown backends and surfaces a remediation hint naming `learning.adapter_training.allowed_backends` and `security.exec_allow`.

ADR 0098 records the factory.

### Diagnosis remediation upgrade

`route_code_remediation`, `route_skill_remediation`, and `route_memory_remediation` (`crates/allbert-kernel/src/self_diagnosis.rs:598-751`) gain a "candidate generation" pre-step. The kernel issues a bounded provider call with:

- system: "You are reviewing an Allbert self-diagnosis report. Produce one of: a unified diff under the source tree; a SKILL.md body for a remediation skill; or a memory candidate. Output only the artifact, no commentary."
- user: the bounded `TraceDiagnosticBundle.summary` plus `report.md`.
- max_tokens: 4096; cost-cap-aware.

The provider response replaces the empty placeholder body in each route. If the provider call fails, returns empty, or violates the schema (no diff produced for `code`, no SKILL.md body for `skill`, fewer than 64 chars for `memory`), the kernel falls back to the existing report-only scaffold and records the fallback reason in the approval/skill/staged-memory frontmatter.

The candidate generation respects:

- Daily cost cap (ADR 0051) â€” refuses the LLM call if the cap is exceeded.
- `self_diagnosis.allow_remediation` â€” false short-circuits before the provider call.
- Existing review surfaces â€” `patch-approval`, `skill-incoming`, `staging`. No surface change.
- Telegram structural-only posture â€” Telegram still cannot start remediation.

A new config key `self_diagnosis.remediation_provider_max_tokens` (default 4096, range 256..16384) caps generation cost.

ADR 0099 records the candidate-generation policy.

### Kernel split

`allbert-kernel-services` is a new sibling crate with the same workspace dependencies as `allbert-kernel`. The following modules move:

- `adapters/` (8 files, ~3500 LOC)
- `self_diagnosis.rs` (1532 LOC)
- `self_improvement.rs` (1242 LOC)
- `local_utilities.rs` (1223 LOC)
- `replay.rs` (2090 LOC)
- `scripting/` (~530 LOC)

Each module exposes a narrow trait that `allbert-kernel` depends on (e.g. `ServicesRegistry`). Implementations live in `allbert-kernel-services`. `allbert-kernel/src/lib.rs` re-exports the same public names so `allbert_kernel::SelfDiagnosisConfig`, `allbert_kernel::AdapterStore`, etc., keep working for daemon/CLI/tests. No semver-visible API change.

After the split, `allbert-kernel/src/lib.rs` is the kernel core: agent loop, hooks, policy envelope, memory contracts, intent, identity, provider seam, and module re-exports. Target ceiling: 4000 LOC for `lib.rs`, 20000 LOC for the kernel crate total.

ADR 0100 records the boundary.

## Exit criteria

- The default Ollama Gemma4 profile reliably executes a tool call for "what is today's date" (or any equivalent simple prompt) with no `<tool_call>` text leaking to the user.
- A v5 client (TUI, Telegram, future channel) round-trips every `Adapters*` message against a v0.14.1 daemon without `adapter_surface_not_implemented`.
- A profile with a configured real trainer backend uses that backend, not the fake.
- `diagnose run --remediate <kind>` produces a non-empty candidate artifact in 9 of 10 cost-cap-respecting fixture runs; falls back gracefully in the rest.
- `crates/allbert-kernel/src/lib.rs` is under 4000 LOC; the kernel crate is under 20000 LOC total.
- `tools/check_doc_reality.sh` is enforced in CI.
- README, CHANGELOG, vision, plans, and operator docs are honest about partial-completion items.
- v0.13 and v0.14 invariants remain unchanged.
