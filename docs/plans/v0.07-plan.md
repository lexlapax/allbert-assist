# Allbert v0.07 Confirmation Workflow

## Purpose

v0.07 adds the confirmation layer for capability requests that are known,
inspectable, permission-classed, and security-decisioned, but should not run
without explicit operator approval.

This comes after Security Central and action-backed skills because
confirmations need stable settings-backed policy, known permission classes,
registered actions, risk metadata, redaction obligations, and trace metadata
before they can safely interrupt and resume runtime work.

## Expected Inputs

- v0.05 Security Central exists for decision shape, risk, redaction, audit, and
  trace metadata.
- v0.05 exposes editable Settings Central permission defaults and read-only
  effective Security Central status through CLI and `/settings`.
- v0.06 has been released with action-backed skill contracts that map only to
  known Jido actions and known permission classes.
- v0.06 runner, signal, trace, and Security Central metadata include selected
  skill, contract validation, selected action capability, permission decision,
  risk, and outcome context.
- v0.06 local skill helper actions use the explicit `:skill_write` permission
  and are operator-invoked through `Actions.Runner.run/3`, not the intent-agent
  tool surface.
- CLI and LiveView can inspect the same runtime state.

## Expected Direction

- Add pending capability requests for actions that require confirmation.
- Let CLI and LiveView approve or deny a pending request through the same
  runtime boundary.
- Use Security Central and Settings Central for permission defaults,
  confirmation preferences, risk display, and redaction.
- Persist confirmation decisions in traces and memory-friendly audit records.
- Keep command execution and external network adapters inert until this flow is
  tested end-to-end.
- Preserve v0.06's distinction between confirmation and execution: approving a
  pending request should resume only registered actions that already have an
  implemented adapter, and should not make skill metadata, scripts, package
  manifests, or online imports executable.

## Non-Goals

- No arbitrary command execution.
- No external network execution.
- No autonomous approval.
- No channel-specific confirmation forks.
- No long-term policy learning beyond explicit settings and audit records.

## Definition Of Done

v0.07 is done when:

- Confirmation-required actions create pending requests instead of running.
- CLI and LiveView can approve or deny the same pending request.
- Approval resumes through the runtime/action boundary.
- Denial is traceable and does not run the action.
- Confirmation decisions are redacted, audited, and visible in traces.
