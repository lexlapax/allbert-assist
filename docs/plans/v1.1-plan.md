## Purpose

Ship the 1.1 flagship: a prompt that decomposes into multiple tasks returns a
kickoff receipt, then fans out supervised background agents, streams their
status, joins, and reports back to the originating caller — while chat
channels stay open and mid-flight input is contextually routed as steering vs
a new request.

1. **Non-blocking fan-out turns with honest kickoff ordering.** A decomposable
   prompt returns a kickoff ack (the visible task list) in seconds; only after
   the caller confirms that it was delivered or durably recorded does work
   continue in a supervised OTP tree with durable state in Objectives.
   Runtime callers stop owning execution latency for decomposed work.
2. **Parallel child-objectives with an honest join.** Child objectives under
   `parent_objective_id` execute concurrently under DynamicSupervisor with
   explicit backpressure; the join report always enumerates per-child
   outcomes; crash/restart rehydrates from Objectives (ADR 0083).
3. **In-channel steering.** Mid-flight input on the same thread is classified
   (advisory) as steer / adjust / cancel / status query / new request;
   effectful steering routes through registered actions and never lowers a
   confirmation class. Cancellation is real: cooperative cancel between
   operations, supervised shutdown, and OS child-process kill semantics
   (ADR 0085) — retiring the "operators can only wait out a long action" debt.
4. **Authorized report-back on every channel.** A NEW settings-governed
   authority class (default OFF, ADR 0084) lets Allbert autonomously deliver
   status/completion notifications per channel; without opt-in, reports wait
   for the next turn. All nine chat surfaces (tui, web, telegram, email,
   discord, slack, matrix, whatsapp, signal) get kickoff-ack,
   status/completion delivery per their primitives, and steer-by-reply.
5. **The parity contract tells the truth.** `streaming: "turn_complete"` stops
   being hardcoded; every channel declares its streaming capability and the
   parity matrix derives from declarations.
6. **One Runtime contract.** Broad automatic fan-out applies to every
   `Runtime.submit_user_input/1` caller. Chat, CLI, public-protocol, and Jobs
   callers acknowledge the kickoff through the same idempotent two-phase
   continuation boundary before children can execute.

## Locked Decisions

Decisions 1–4 are operator-locked (2026-07-18). Decisions 5–14 are derived
during planning from the locked four plus code evidence; each records its
rationale so the operator can veto at plan review. Decision 11 was
subsequently superseded by an operator fold-in (2026-07-18 intake walk) and
is now operator-locked. Decisions 15–25 are operator-locked readiness-audit
resolutions.

1. **Concurrency = parallel child-objectives (operator).** A fan-out frames
   CHILD objectives under `parent_objective_id`, each executed by a supervised
   run process (one global DynamicSupervisor, `restart: :temporary`,
   Registry-addressed per-run keys); join = `Process.monitor` + child status
   reduction into the parent; durable state lives in Objectives (no private
   goal loops — the OTP tree is the executor); crash/restart ⇒ rehydrate from
   Objectives. ADR 0083 binds the model.
2. **Decomposition = model proposes, visible list (operator).** A Stage-0
   decomposer MAY use the local model to propose the task list (the
   `propose_steps` precedent); the list is ALWAYS surfaced in-channel before
   fan-out starts (default-on ack); deterministic/explicit decomposition works
   without a model; decomposition output is advisory-only and grants nothing
   (AGENTS.md: `objective_id`/`step_id` are never authority).
3. **Report-back authority = settings-level autonomous notify (operator).**
   Per-channel operator settings (Settings Central, default OFF) authorize
   autonomous status/completion notifications on that channel. Security
   Central holds the boundary. This is a NEW authority class with its own ADR
   (0084). The `turn_complete` parity contract is renegotiated per channel
   with an explicit streaming capability field.
4. **Rollout = ALL channels at once; steering extends `:continue_objective`;
   cancellation ships both tiers (operator).** Every adapter
   (tui/web/telegram/email/discord/slack/matrix/whatsapp/signal) gets
   kickoff-ack, status/completion delivery per its primitives, and
   steer-by-reply in 1.1. The steering classifier is advisory; effectful
   cancel/steer routes through registered actions + the confirmation floor
   (steering may never LOWER any action's confirmation class). Cancellation
   ships the merged enablers: cooperative cancel token between operations +
   supervised shutdown, AND OS child-process kill semantics
   (`future-features.md:164-185`). ADR 0085 binds cancellation.
5. **OTP pattern constraints (operator, baked into seams).** Supervised
   GenServer/Jido.Agent per the pragmatic-substrate rule with the choice
   documented in `@moduledoc`; never bare `Task.async` for steerable agents;
   monitors, not polling; Registry-addressed steering; signal-bus push status
   (no polling loops); DynamicSupervisor `max_children` backpressure with an
   explicit FIFO queue policy (refined by Locked Decision 21: fair
   round-robin capacity grants ACROSS fan-outs, durable FIFO preserved
   WITHIN each fan-out); no GenStage/Flow; Security Central authority
   unchanged — a supervised process has no authority by virtue of being
   supervised.
6. **No workflow-YAML routing (derived, ADR 0041).** Fan-out is an
   objectives-native mechanism; `parallel_steps`/`for_each` stay reserved
   vocabulary. Rationale: concurrency semantics belong in the supervised
   runtime, not operator-authored YAML; the ADR 0041 operability rationale
   stands.
7. **Additive-only contract evolution (derived, freeze).** The runtime
   response gains only additive fields (`fanout` summary, `pending_reports`)
   per ADR 0029 Tier-2 rules; channel descriptors gain an optional
   `streaming:` field with the current behavior as default; `release.v1`
   stays green throughout. No non-additive migration: the objectives schema
   change is additive columns + index (the 1.5-horizon migration-runner
   cluster is not pulled forward).
8. **Ack semantics (derived from "default-on ack").** The kickoff ack — the
   decomposition list with child ids/titles — IS the fan-out turn's response,
   delivered synchronously on every channel; visibility cannot be disabled.
   Framing NEVER starts a run: only the caller's idempotent acknowledgement of
   successful delivery makes the queue runnable (Decision 15). A new setting
   `objectives.fanout.confirm_before_start` (default `false`) adds a durable
   confirmation after acknowledgement and before scheduling for operators who
   want a second hard gate.
9. **Settings-pin policy for background runs (derived).** Each run process
   establishes its OWN resolved-settings pin per step execution
   (`Store.with_resolved_settings/1` around each step), because (a) the M8.4
   pin is process-dictionary-scoped and cannot cross into spawned processes,
   and (b) a fan-out may outlive the originating turn by hours — freezing the
   parent turn's snapshot for the whole run would make operator setting
   changes (e.g. disabling autonomous notify) invisible until restart.
   Semantics: deterministic within a step; operator changes take effect at
   the next step/notify boundary. Documented in the RunServer `@moduledoc`
   and tested red-first.
10. **Cancel needs no confirmation; steering never approves (derived).**
    `cancel` of your own fan-out/child is the safety action and follows the
    existing `Objectives.cancel/3` precedent (no confirmation today —
    `objectives.ex:111`); the OS-kill tier executes only against processes
    the run itself spawned (ADR 0085 scope rule). Steering directives are
    advisory reframes recorded durably; they may cause new effectful steps
    which confirm at their own actions, and free-text steering NEVER maps to
    confirmation approval — approvals stay on the typed-command/button path
    (`ConfirmationCallback`). Gate-bound eval rows enforce both.
11. **Capability-declared in-place status (operator fold-in, 2026-07-18
    intake walk — supersedes the earlier no-edit decision).** Adapters
    DECLARE their edit capability in the channel descriptor
    (`status_update_mode: :append_only | :edit_in_place`, default
    `:append_only` — additive, parallel to the S7 `streaming:` field). The
    four providers with a real edit primitive declare `:edit_in_place`:
    telegram (`editMessageText`), discord (message PATCH), slack
    (`chat.update`), matrix (`m.replace` relation). whatsapp (the Cloud API
    has no business-initiated edit), signal (partial/recent CLI support),
    and email stay `:append_only`; TUI/web are local surfaces. Where
    declared, `Channels.Notify` keeps ONE status message per fan-out per
    channel and EDITS it in place, with automatic append fallback on edit
    failure (audited). Completion reports and confirmation requests are
    ALWAYS new messages — never edits: an approval prompt must not replace
    history. Edits do not evade the min-interval throttle — an edit is a
    delivery for throttling purposes. M7 (S7b) carries the implementation.
12. **Signal inbound daemon is wired in-scope; live gating unchanged
    (derived).** "ALL channels" includes signal steer-by-reply, which is
    impossible while `Daemon.daemon_child_spec/2` has no caller — M6 wires
    the MuonTrap daemon receiver under the adapter (ADR 0058 pattern).
    WhatsApp/signal remain `ReleaseAvailability`-gated exactly as today; v1.1
    changes code paths, not release status — their validation runs simulated
    (webhook/notification fixtures) plus live where the operator has a
    configured live channel.
13. **Notify-OFF fallback = next-turn report (derived).** With autonomous
    notify OFF (the default), completed fan-outs deliver their join report as
    part of the user's NEXT turn on that thread (additive `pending_reports`
    response field rendered by every adapter) and are always visible in the
    web/TUI objectives surfaces. Attached live local surfaces (an open
    workspace LiveView, an attached TUI session) render push status without
    the ADR 0084 authority — the user is present and no unattended transport
    send occurs; ADR 0084 defines this boundary precisely.
14. **Fan-out discriminator (derived).** Additive columns `fanout_role`
    (`"parent"`/`"child"`/nil) and `join_policy` (default `"all_terminal"`)
    plus an index on `parent_objective_id` distinguish fan-out children from
    the existing plan/build `parent_objective_id` consumers
    (`plan_build.ex:42`) — the recon's "unused" claim was wrong; coexistence
    is explicit.
15. **Delivery-confirmed start across every Runtime caller (operator).**
    Framing returns a normal additive response plus an opaque, identity-bound,
    single-use start receipt. Children remain durable `open` and cannot start
    until the caller acknowledges successful kickoff delivery: remote chat
    after transport success; web/TUI after render/print; HTTP public protocols
    after response commit; CLI after output; Jobs after a durable kickoff
    event. Duplicate acknowledgement is idempotent. A failed or uncertain
    delivery leaves the fan-out blocked and NEVER starts it; an operator
    retry/status turn re-renders the same kickoff and reuses the durable
    receipt before acknowledging it.
16. **Broad automatic eligibility (operator).** Stage 0 may fan out any prompt
    judged to contain two or more independent tasks; explicit "parallel" or
    "background" language is not required. Single-task, uncertain, unsupported,
    and nested-fan-out proposals return `:single`. If a valid decomposition
    exceeds `max_children`, Allbert clarifies before framing; it never drops,
    truncates, or silently merges tasks. Existing intent accuracy floors
    remain the regression bar.
17. **All Runtime callers participate (operator; restored third pass
    2026-07-18).** Auto-fan-out is not channel-only. Every direct caller of
    `Runtime.submit_user_input/1`, including `mix allbert.ask`, ACP/
    OpenAI-compatible controllers, and Jobs, implements the same delivery
    acknowledgement contract. A caller without an acknowledgement
    implementation fails closed to `:single`; it never starts unacknowledged
    work. **API presentation: OpenAI-compatible/ACP requests HOLD until
    join** — the internal two-phase contract is unchanged (response-commit
    acknowledges the kickoff; children then run), but the HTTP response
    completes with the join report, bounded by the request timeout
    (timeout ⇒ the ack + receipt returns and the report lands via
    `pending_reports`); SSE-streaming clients receive status chunks then
    the report. Synchronous third-party clients always get a complete
    answer in-band; interactive channels keep the async experience. No wire
    format changes (ADR 0029 respected).
18. **Multiple active fan-outs (operator).** An unqualified status query
    aggregates every active fan-out attributed to the thread/session. A
    mutating steer/cancel resolves an explicit child/fan-out reference; when
    more than one target remains plausible, Allbert clarifies instead of
    choosing the newest run. New requests continue through the normal pipeline.
19. **Crash recovery never guesses about external effects (operator).**
    Rehydration auto-resumes only a step whose retry safety/idempotency is
    proved by its action contract. If an effect may have committed before its
    observation became durable, the child becomes `blocked` with an
    `uncertain_effect` review reason; only an explicit retry or skip decision
    may advance it. Registered actions gain optional
    `retry_safety: :safe | :unsafe | :unknown`, default `:unknown`; only
    `:safe` may replay automatically. M2 sweeps the SHIPPED action catalog:
    read-only/idempotent actions (queries, lists, status reads) declare
    `:safe`; possibly-effectful actions declare `:unsafe` or stay `:unknown`;
    the sweep table is recorded in Build Progress so auto-resume is real,
    not vacuously absent.
20. **Durable queue, attempts, and join outcome (operator).** Queue position,
    per-child run-attempt count, delivery state/receipt identity, and
    `join_outcome` are reconstructible from Objectives; the Coordinator owns
    no irreplaceable loop state. `join_outcome` is additive
    (`success|partial|failed|cancelled`) and does not widen the frozen objective
    status enum: all completed → completed/success; any completed plus failed
    or cancelled → completed/partial; no completed and any failed →
    failed/failed; all cancelled → cancelled/cancelled.
21. **Fair global scheduling (operator).** A permanent supervised
    `Objectives.Runs.Scheduler` grants global capacity round-robin across
    active fan-outs while preserving durable FIFO order within each fan-out.
    It is signal/monitor-driven, reconstructs from Objectives, and restarts
    temporary Coordinators after coordinator-only crashes.
22. **Full child lifecycle (operator).** A framed child begins with its task,
    not executable steps. RunServer therefore drives propose → evaluate →
    authorize → execute → observe → advance using the existing Objectives
    helpers directly, without serializing through `Engine.Agent`; settings are
    pinned independently around each lifecycle operation.
23. **Standalone sessions pause honestly (operator).** Attached
    `allbert serve` sessions continue background work. Quitting a standalone
    TUI/CLI pauses at durable boundaries and says that work resumes on the
    next Allbert boot; it does not silently imply a daemon handoff. Resume is
    honest about interruption: a child whose in-flight operation declared
    `retry_safety: :safe` auto-resumes; otherwise it waits `blocked`/
    `uncertain_effect` with an explicit retry affordance, and the quit notice
    and next-boot status both say so (Locked Decision 19).
24. **Restart-safe notify state and honest uncertainty (operator).** An
    additive delivery ledger persists delivery keys, provider message ids,
    throttle timestamps, attempts/errors, and one-time-offer state. Edit-capable
    channels resume the same status message after restart. If a provider may
    have accepted a send before receipt persistence, delivery becomes
    `uncertain`, autonomous retry is suppressed, and the report remains
    available through status/next-turn fallback with an audit warning.
25. **Parked-confirmation steering invalidates before replanning (operator).**
    Redirect/skip against a confirmation-blocked child durably marks the old
    confirmation stale, records the directive, and replans under unchanged
    confirmation rules; the old approval can no longer execute.

## Operator-Visible Win

Ask for a multi-part job on ANY channel — "research X, summarize Y, and draft
Z" from Telegram, the TUI, or the workspace — and Allbert answers in seconds
with the task list and starts working. The channel stays usable: ask for
status, redirect a task, or cancel one, in plain language, from the same
thread. Opt a channel into autonomous notify and completion reports arrive on
their own; leave it off and the report greets the next message. Long-running
actions finally die when told to — including their OS child processes.

## Code Seams

**S1 — objectives fan-out model (M1, ADR 0083).** Additive migration
(`fanout_role`, `join_policy`, `join_outcome`, delivery state/receipt digest,
durable queue position, per-child run-attempt count, review reason, index on
`parent_objective_id`).
`Objectives.Fanout.frame/2` creates the parent + children in one transaction,
copying `source_channel`/`source_surface`/`source_thread_id`/`session_id` to
each child; `children/1` and `join_status/1` are pure reads;
`report/1` builds the enumerated per-child join report (redaction-safe via
`Security.Redactor`). Join reduction is explicit: all completed →
`completed/success`; any completed plus failed or cancelled →
`completed/partial`; no completed and any failed → `failed/failed`; all
cancelled → `cancelled/cancelled`. The report always enumerates. Objective
events record every transition. The serialized engine
agent is untouched except rehydration (S2); existing single-objective flows
prove identity via the objectives suites.

**S2 — supervised run executor (M2, ADR 0083).**
One global `Objectives.Runs.Supervisor` (DynamicSupervisor, `max_children`
from the settings-clamped value) + `Objectives.Runs.Registry` (Elixir Registry,
unique keys) added to the application tree near the coding precedent
(`application.ex:35-36`). A permanent plain-GenServer `Runs.Scheduler`
(pragmatic-substrate choice documented in `@moduledoc`) grants slots fairly
round-robin across fan-outs, preserves each fan-out's durable FIFO order,
monitors Coordinators, and reconstructs/restarts them from Objectives without
polling. `Coordinator` (GenServer per active fan-out,
`restart: :temporary`): derives its FIFO queue from durable queue positions,
starts ≤ `max_concurrent_runs`, monitors runs, applies the bounded restart
policy from the persisted attempt count (one restart per child, then terminal
`failed`), performs join
reduction, emits `fanout.joined`. `RunServer` (plain GenServer — pragmatic
substrate: thin idle→running→cancelling lifecycle, Registry-addressed
steering mailbox; Jido.Agent's routing buys nothing here; documented in
`@moduledoc`): executes the child's full lifecycle
(propose → evaluate → authorize → `Actions.Runner.run/3` → observe → advance)
directly against
the Objectives API — NOT through the serialized engine agent — pinning
settings around each lifecycle operation (Locked Decision 9), emitting
`run.*` signals on the
SignalBus, checking its `CancelToken` between operations, and parking on
`needs_confirmation` (step `confirmation_id`, status `blocked`) without
blocking siblings. Scheduler boot/live reconciliation uses the existing
rehydrate window to restart coordinators for parents with non-terminal
children. Only actions declaring `retry_safety: :safe` auto-resume;
`:unsafe`/`:unknown` and a possibly committed external
effect with no durable observation parks as `blocked`/`uncertain_effect`
until an explicit retry or skip.
Backpressure proof: queued children start only as slots free; forced
`Process.exit(pid, :kill)` → bounded restart → honest join; BEAM restart
mid-fan-out resumes.

**S3 — Stage-0 decomposer + ack (M3).** `Intent.Decomposer.propose/2` runs
BEFORE the router for eligible new turns (no typed command): deterministic
splitter first
(explicit enumerations, "then/and then" chains, numbered lists), model
proposal second (local model, `propose_steps`-style prompt, JSON list,
bounded by `max_children`). Any prompt judged to contain at least two
independent tasks may fan out; explicit parallel language is not required.
Single-task, uncertain, unsupported, and nested proposals return `:single`.
An otherwise valid proposal above `max_children` returns a clarification turn
showing the count and asking the operator to narrow or explicitly batch it;
no task is silently dropped or merged.
Latency guard: the model proposer is consulted ONLY when cheap deterministic
pre-signals (enumeration/conjunction/list shape) indicate plausible
multi-task; single-task prompts must not pay a model round-trip — M3
acceptance carries a measured decomposer-overhead budget on `:single` turns
(cited from the metrics store; the M8.4 −67% turn win is not given back).
Explicit single-task language ("as one task", "together", "don't split")
deterministically forces `:single` — the per-prompt opt-out mirror of the
broad eligibility rule; corpus rows cover both directions.
Decision path: a `:fanout` proposal frames via S1 and returns the kickoff ack
(list + child ids + steering hint, plus the REPORT DELIVERY MODE for this
channel — "reports will arrive here" when autonomous notify is on, or "I'll
report when you next message; enable auto-notify in settings to get it
pushed" when off — the operator always knows where the results will land)
plus a start receipt. It does NOT start S2.
Every Runtime caller acknowledges successful render, transport, response
commit, CLI output, or durable Job event through one idempotent boundary;
only that acknowledgement makes children runnable. Delivery failure leaves
the parent blocked for retry/cancel; operator retry/status re-renders and
reuses the receipt. OpenAI-compatible/ACP callers acknowledge at response
commit and HOLD until join per Locked Decision 17 (timeout ⇒ ack + receipt,
report via `pending_reports`; SSE streams status then report).
`confirm_before_start` adds a durable
confirmation before an acknowledged queue may run. The decomposer NEVER runs
on steering turns (S5 classifies against active fan-outs first).

**S4 — cancellation both tiers (M4, ADR 0085).**
`Objectives.Runs.CancelToken`: per-run token (Registry-keyed atomics/ETS
flag) checked by the RunServer between operations and offered to actions via
runner context (`context[:cancel_token]`, cooperative — long-running actions
opt in by checkpointing). Each OS execution has a supervised
`Execution.ProcessOwner` that owns the helper port independently of
RunServer: a new `Execution.ProcessOwners` DynamicSupervisor
(`restart: :temporary` children, application-tree sibling of the execution
runners — placed in `application.ex` near the coding supervisor precedent,
NOT under `Objectives.Runs.Supervisor`, because non-fan-out executions —
coding turns, direct commands — use it too) plus a unique
`Execution.ProcessRegistry` keyed `{:execution, execution_id}`. Cancellation
addresses that owner before terminating RunServer; helper control-channel
EOF kills its process group on owner/BEAM failure. Supervised shutdown then drains
RunServer to the last durable step. OS tier: `Execution.LocalRunner` (and
`skill_script_runner`, sandbox `command.ex`) move from bare `System.cmd` to a
port-based spawn capturing `:os_pid`, run through a bundled, narrowly scoped
C launcher that creates a process group and reports its leader pid/handle on
macOS and Linux, with kill semantics
SIGTERM → grace (config, default 5s) → SIGKILL to the group; timeout paths
(`local_runner.ex:53-56`) use the same kill instead of orphaning. TUI escape
extends beyond coding turns: escape during an attached fan-out offers
cancel. `cancel_objective_run` action drives: cooperative flag → supervised
shutdown → OS kill, recording tier reached in the objective event.
**Delegate cancel-token propagation (operator fold-in, 2026-07-18 intake
walk — supersedes the earlier out-of-scope note):** the `:delegate_agent`
contract gains the cooperative token — `AgentRegistry.dispatch/4` passes
`cancel_token` in the delegate signal payload; the delegate developer
contract (docs/developer/delegate-agents.md) documents checkpoint polling;
the two shipped consumers (StockSage specialists, research.specialist)
adopt checkpoints at their natural boundaries. A cancelled child's delegate
call now returns `:cancelled` at the next checkpoint instead of computing
to its timeout with a discarded result; unconverted third-party delegates
degrade to exactly the old tier-2 behavior (token unchecked ⇒ supervised
shutdown after grace) — additive, never breaking. ADR 0085's consequence
note updates in the same commit.

**S5 — steering (M8).** Extend the `:continue_objective` seam: fan-out
parents/children in the candidate pool carry fan-out metadata
(`engine.ex:1223-1252` collection + `:273-291` decision attrs). A new
`Intent.Steering` classifier (advisory: deterministic verb/reference
heuristics + optional local-model assist) refines a `:continue_objective`
decision targeting an active fan-out into
`{:steer | :adjust | :cancel | :status | :new_request}` with a child
reference resolution (ordinal, title fuzzy match, "the research one").
Unqualified `status` aggregates all active fan-outs in the thread/session
from durable state (no engine dispatch); `steer/adjust` →
`steer_objective_run` (directive recorded as an objective event, delivered
Registry-addressed to the RunServer, applied at the next step boundary —
never mid-operation); `cancel` → `cancel_objective_run`; `new_request` falls
through to the normal pipeline. A mutating request with more than one
plausible target clarifies; it never silently chooses the newest run. Free
text never approves confirmations
(Locked Decision 10; eval row). Misclassification cost is bounded: steer
directives are advisory and reversible; cancel asks the classifier for high
confidence or falls back to a clarify turn.
Steering a child parked on confirmation first marks that confirmation stale
through the durable confirmation store, then records the directive and
replans; the old callback can no longer execute.

**S6 — autonomous notify + per-adapter delivery (M6, ADR 0084).**
`Channels.Notify` boundary: resolves the fan-out's origin
(`source_channel` + thread ref via the v1.0.1 attribution and
`channel_thread_refs`), re-derives the transport target from the identity
map (never from stored free text), checks — in order — channel live-use,
`channels.<id>.autonomous_notify.enabled`, level, min-interval throttle
(coalescing statuses per fan-out), Security policy, then dispatches via the
adapter's `deliver_outbound/3` with a threading hint in `opts`. Every
decision (delivered/suppressed/failed/uncertain + reason) first updates the
additive delivery ledger and is then signaled. Redacted operator evidence
appends through `Runtime.Audit` kind `:channel_notify` to
`<ALLBERT_HOME>/channels/notify/audit/YYYY-MM.md`. Unique terminal-event
keys deduplicate completion; stored provider message ids and throttle
timestamps restore edit/coalescing state after restart. A send that may have
succeeded without a persisted receipt becomes `uncertain`, is not
autonomously retried, and falls back to status/next-turn retrieval with an
audit warning.
Background-raised confirmations are a distinct notify kind
(`:confirmation_request`) rendered with the channel's approval primitives
(buttons where available, typed `ALLBERT:APPROVE:<id>` everywhere) — the
existing `ConfirmationCallback` path completes the loop untouched.
`pending_reports` rendering rule (all adapters): compact per-fan-out summary
lines (parent title + join outcome + per-child status glyphs), never the full
multi-child report prepended to an unrelated answer — the full report renders
on a status query or in the web/TUI objectives surfaces.

`channels.<id>.autonomous_notify.enabled=true` through the standard audited
settings path (typed-command family, identity re-proved like
`ConfirmationCallback`); the offer is never repeated after answer or
dismissal (durable per-channel offered marker). Free text still never
changes settings — only the typed command/button does.

## Security Posture

This release ADDS an authority class and is planned like it:

- **Autonomous notify (ADR 0084)** is the only new grant: settings-gated per
  channel, default OFF, enforced at the single `Channels.Notify` boundary
  (settings + Security policy + identity-map target re-derivation +
  origin-thread binding + throttle), fully audited, redacted
  (`Security.Redactor` on all status/report text), and gate-bound by `:v11`
  eval rows. Attached-local-surface push (web LiveView, attached TUI) is
  explicitly OUTSIDE the class — no unattended transport send occurs.
- **Nothing else grants.** Decomposition output, steering directives,
  fan-out/run/child ids, and supervised processes carry zero authority; every
  child step executes registered actions with unchanged confirmation classes;
  free text never approves a confirmation; `send_channel_message` keeps
  `confirmation: :required` (operator-initiated sends are NOT relaxed by
  notify settings — different class, different boundary).
- **Cancellation kills only what the run spawned** (ADR 0085 scope rule:
  os_pid/process-group captured at spawn, kill addressed to that group only);
  OTP supervision remains a non-security boundary (AGENTS.md).
- **Inbound trust unchanged:** steering rides the existing channel identity
  mapping/trust classes (ADR 0016/0056/0059); an unmapped sender cannot
  steer, cancel, or receive notifications. E2EE-origin rules (signal) stand.
- Secrets stay encrypted/redacted end-to-end incl. notify payloads, audit
  rows, and release evidence.

## Non-Goals

- Workflow YAML `parallel_steps`/`for_each` promotion (stays reserved,
  ADR 0041).
- Edit-in-place status on channels without a declared edit capability —
  whatsapp (no business-initiated edit in the Cloud API), signal
  (partial/recent CLI support), email stay `:append_only` (Locked
  Decision 11); no adapter is forced to fake edits.
- Nested fan-out (a child framing its own fan-out) — single level in 1.1;
  the model rejects nested proposals.
- Remote/distributed delegate agents, operator no-code agent authoring, and
  the shared delegate-agent behaviour extraction (parked, future-features).
- Flipping whatsapp/signal release availability (code paths ship; release
  status is a separate operator decision).
- Typing indicators, presence, read receipts.
- Cross-user or multi-tenant fan-out semantics (single local-user model
  stands).
