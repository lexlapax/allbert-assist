## Purpose

Expose Objective Runtime as a user-facing Plan/Build mode and allow
operators to author declarative workflow YAML that expands into objective
steps.

Plan/Build is a **surface**, not a runtime. The v0.24 Objective Runtime
remains the only durable cross-turn substrate; v0.44 only renders it and
accepts declarative input that produces objective steps. The runtime
never executes YAML; expansion produces step attrs that flow through
`Actions.Runner.run/3`, Security Central, confirmations, traces, and
audits.

Two distinct operator workflows ship together:

1. **Plan from natural language.** "plan: create a project folder, fetch
   a template, summarize it." The intent layer routes to a one-off
   plan-preview that operates over a synthesized in-memory workflow
   shape. The operator reviews, edits, approves, and executes.
2. **Run an authored workflow.** "run workflow nightly-briefing." The
   loader reads `<ALLBERT_HOME>/workflows/nightly-briefing.yaml`,
   validates against the v1 schema (unknown keys fail closed), and
   expands into the same preview/approve/execute path.

Both paths share the v0.24 Objective Runtime: every produced step is an
`objective_steps` row; every effectful step still resolves through
`Actions.Runner.run/3`.

## M1 Locked Decisions

The first milestone preserves these six decisions. They are no longer
open defaults; later milestones may revisit them only with an explicit
re-decision recorded in this table and in the request-flow document.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Plan/Build surface shape | **Pinnable panel + ephemeral surface over the v0.23/v0.26 workspace canvas**, NOT a separate `/plan` destination route | A destination would duplicate canvas chrome and force the operator to rebuild ambient context (which connector? which prior step output? which thread?). The Devin/Cursor/Claude Plan-Mode prior art (2025-2026) all converged on plan-adjacent panels because plans need the conversation that produced them. |
| 2 | Workflow YAML expression grammar | **Closed function table** + dotted-path references (`${inputs.x}`, `${steps.<id>.<field>}`, `${user.locale}`), AST-parsed at load, no `eval`, no Jinja-with-callables | Jinja-with-callables is the most-cited footgun in agent-workflow systems (LangChain, etc.). GitHub Actions' fixed function table is the proven counter-pattern: a small grammar the validator parses at load time and rejects on unknown identifiers. Choosing an open grammar would create an indirect code-execution surface inside an artifact that is supposed to be inert data. |
| 3 | Schema validation source-of-truth | **Derive the workflow schema from the current action registry snapshot + step-kind module**, not hand-maintained alongside | v0.43 R3 lesson (action/PDF doc drift) and v0.42 R4 lesson (intent-corpus drift) both teach the same rule: when docs and runtime are hand-synced, they drift. GitHub Actions, Argo, and Temporal-DSL projects all hit schema drift the same way. The schema builder uses `Actions.Registry.modules/0` at validation time so source-tree, plugin, and dynamic action overlays are represented without pretending the registry is compile-time static. Plus per-action `params:` validation at load time, not run time, so authors learn about typos before a workflow ever runs. |
| 4 | Workflow file location and id pattern | `<ALLBERT_HOME>/workflows/<workflow-id>.yaml` (already in ADR 0041 amendment); id pattern `^[a-z0-9][a-z0-9_-]*$`; collisions fail closed; discovery is on-demand (no autoload, no scan) | Already pinned in ADR 0041; the lock here adds the id-pattern regex and the on-demand-only rule so M1 can implement the loader without re-deciding. The alternative (autoload + scan) introduces a startup cost and a "what gets loaded when" mystery; on-demand keeps the YAML inert and lets v0.51 export/import treat the directory as a flat operator-readable artifact. |
| 5 | Execution semantics | **Sequential array order**; the only branching is per-step `if:`; **no loops, no parallel/fan-out, no sub-workflow includes, no `on:` triggers, no `env:` block, no retry policies in v0.44**; subagent delegation uses v0.24's `:delegate_agent` step kind unchanged | Serverless Workflow, LangGraph, and Anthropic "Building Effective Agents" all conclude that workflows should start simple and escalate to agents only when needed. Loops + dynamic action names are the most-cited cycle and safety footguns; sub-workflow composition expands the schema surface by 3-4x. Defer everything that doesn't have a real consumer yet; promote later via reserved vocabulary in ADR 0041 §"Reserved Vocabulary". |
| 6 | Confirmation semantics | **Registered-action permission floor is authoritative**; YAML `confirm: true` may only *upgrade* a step (never downgrade); explicit `kind: ask_user` step is the plan-level checkpoint; the v0.44 plan-approval gate at run start is a `:workflow_run_start` permission with `:needs_confirmation` floor | Anthropic Agent-Skills `allowed-tools` precedent: declarative narrowing only, never widening. Aligns with ADR 0021 §4 (advisory output is not authority) and v0.43 cross-operation-grant-deny. If YAML could *lower* a confirmation floor, operators (or attackers) could ship a workflow that bypasses Security Central. The upgrade-only rule keeps every per-step floor honest. |

Rationales (one paragraph per decision):

- **Surface shape.** Plan/Build is a panel because the plan is meaningless
  outside the conversation that produced it. The 2025-2026 winners
  (Cursor 2.2 Plan Mode panel, Claude Code plan side-panel,
  Devin plan card) are all panels alongside the live thread, not
  destinations. A destination would split the operator's mental model
  (objective lives in workspace; the plan to advance the objective lives
  elsewhere). Plus, ADR 0023's workspace canvas + ADR 0024's panel/zone
  substrate is already in place; v0.44 reuses, adding no new shell layer.
- **Expression grammar.** GitHub Actions' `${{ ... }}` is a closed
  mini-language: it has only `==`, `!=`, `<`, `>`, `&&`, `||`, `!`,
  `()`, `[]`, `.`, and a fixed function table. No assignment, no loops,
  no IO except documented helpers. That's the model. Allbert's
  expression set at v1: `${inputs.<name>}`, `${steps.<id>.<field>}`,
  `${user.locale|timezone}`, plus the function table
  `length`, `contains`, `starts_with`, `ends_with`, `lower`, `upper`,
  `default`, `to_json`, `from_json`. The validator AST-parses these and
  verifies every reference resolves to a declared input, an earlier
  step's `save_as` field, or a runtime-provided `user.*` value.
- **Schema source-of-truth.** The workflow schema is assembled
  deterministically from `AllbertAssist.Actions.Registry.modules/0`
  (each action's `schema/0`) and
  `AllbertAssist.Objectives.Step.kinds/0` at validation time. Static
  core branches are ordinary code; plugin and dynamic action overlays are
  included from the current registry snapshot. The operator-facing
  diagnostic refers back to the action's name. When v0.45 adds an action,
  the workflow schema gains that action automatically; when an action's
  params change, the workflow schema reflects the change without a
  separate documentation edit.
- **File location.** `<ALLBERT_HOME>/workflows/<workflow-id>.yaml` is
  consistent with `<ALLBERT_HOME>/memory/`, `<ALLBERT_HOME>/cache/`,
  and `<ALLBERT_HOME>/plugins/`. The id pattern
  `^[a-z0-9][a-z0-9_-]*$` matches plugin id patterns (ADR 0017 plugin
  ids) and skill ids. On-demand discovery means the loader reads a file
  only when an operator references its id; there is no startup cost, no
  ambient autoload, and no "what is loaded when" mystery.
- **Execution semantics.** Sequential array order plus per-step `if:` is
  the smallest semantics that supports useful multi-step work. Adding
  loops/parallel/sub-workflow now would force v0.44 to ship cycle
  detection, fan-in semantics, composition-version-pinning, and
  scheduling — all problems with active research and no current
  consumer. The reserved vocabulary in ADR 0041 §"Reserved" names
  `for_each`, `parallel_steps`, `sub_workflow_include`, `on_schedule`,
  `on_event`, `retry_policy`, `dynamic_action_name`, `env_block`,
  `secret_substitution`, so future versions can promote without
  renaming.
- **Confirmation semantics.** The action's registered permission floor
  is the only floor that counts at execution time. YAML can mark a step
  `confirm: true` to *require* an extra confirmation even when the
  action's floor is `:allowed`, but cannot mark a step `confirm: false`
  to bypass a `:needs_confirmation` floor. The plan-start gate is a
  separate confirmation at the `:workflow_run_start` permission — it
  authorizes the *run*, not any specific step's execution. Each
  effectful step still confirms per its own floor unless an explicit
  remembered grant covers it.

## Research Grounding

The v0.44 design is grounded in:

- **GitHub Actions workflow syntax and expression grammar** ([docs](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax),
  [expressions](https://docs.github.com/en/actions/reference/workflows-and-actions/expressions)):
  the closed function-table model for `${{ ... }}`, the
  `steps[].id/uses/with/if` shape, the `additionalProperties: false`
  precedent in [SchemaStore github-workflow.json](https://github.com/SchemaStore/schemastore/blob/master/src/schemas/json/github-workflow.json),
  the `inputs:` typed-parameter pattern. v0.44 copies the closed grammar
  and the strict-schema pattern but does NOT copy `on:` triggers (those
  belong to a scheduler, not an inert workflow document).
- **Argo Workflows parameter and output references** ([docs](https://argo-workflows.readthedocs.io/en/latest/walk-through/output-parameters/)):
  `{{steps.x.outputs.parameters.y}}` is the closest dotted-path
  precedent for variable references. v0.44 uses
  `${steps.<id>.<field>}` (dollar-brace) to keep the grammar
  visually distinct from YAML's own `{{...}}` patterns and the v0.24
  trace-pointer conventions.
- **Serverless Workflow DSL** ([spec](https://github.com/serverlessworkflow/specification/blob/main/dsl-reference.md)):
  task kinds `Call`, `Do`, `Switch`, `Wait`, `Listen`, `Set`, `Callback`
  are direct analogs of v0.24's six step kinds. The lesson Serverless
  Workflow records: keep retries optional, version every document, never
  promote `For`-loops without a clear consumer.
- **OpenClaw Lobster** ([repo](https://github.com/openclaw/lobster),
  [docs](https://docs.openclaw.ai/tools/lobster)): typed-step + `approval:`
  step type + resume-token pattern is the closest operator-authored
  workflow precedent. The v0.44 plan-start gate fills the same role as
  Lobster's resume-token approval, mapped onto v0.07 confirmations.
- **Anthropic Claude Plan Mode and `ExitPlanMode`** ([Anthropic docs](https://docs.anthropic.com/en/docs/claude-code/plan-mode),
  [Ronacher analysis](https://lucumr.pocoo.org/2025/12/17/what-is-plan-mode/)):
  the "plan is a markdown file the operator can edit in $EDITOR"
  pattern is the precedent for v0.44's expand-to-fullscreen plan editor.
  The plan-mode-as-confirmation-screen pattern (yes-auto / yes-manual /
  keep-planning) maps onto v0.44's plan-start gate plus per-step
  confirmations.
- **Devin and Cursor multi-step UX (2025-2026)** ([Devin release notes](https://docs.devin.ai/release-notes/2025),
  [Cursor long-running agents](https://cursor.com/blog/long-running-agents),
  [Cursor 2.2 changelog](https://cursor.com/changelog/2-2)):
  traffic-lit confidence tiers (green/yellow/red), cost visibility per
  step, and the "approve once at plan, risk-tier per step" pattern are
  research-grounded anti-rubber-stamping defaults. v0.44 ships
  confidence tiers and cost visibility as Plan Preview Contract fields;
  the auto-proceed behavior is opt-in and default-off.
- **Anthropic "Building Effective Agents"** ([article](https://www.anthropic.com/research/building-effective-agents)):
  workflow-vs-agent boundary; HITL checkpoints. v0.44 falls on the
  "workflow" side: deterministic step composition with operator
  oversight, not autonomous agent loops.
- **Anthropic Agent Skills `allowed-tools` declarative-narrowing**
  ([Anthropic engineering](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills),
  [Skill docs](https://code.claude.com/docs/en/skills)): YAML
  frontmatter can only narrow permissions, never widen. v0.44's
  `confirm: true` upgrade-only rule is the direct analog.
- **JSON Schema Draft 2020-12 with `additionalProperties: false`**: the
  established idiom for YAML-validated-by-JSON-Schema, with the
  Kubernetes structural-schema caveat ([blog](https://kubernetes.io/blog/2019/06/20/crd-structural-schema/))
  that nested `additionalProperties: false` interacts badly with
  `oneOf`/`if-then-else`; the v0.44 schema tests assert behavior with
  multiple step kinds in the same workflow.
- **Plan-then-execute separation** ([Architecting Resilient LLM Agents](https://arxiv.org/pdf/2509.08646)):
  the separation between declarative plan (what) and tactical execution
  (how) is the binding architectural principle. v0.44's plan is the
  declarative artifact; v0.24's engine is the execution layer.

Implementation consequence: v0.44 adds one workflow loader, one AST
validator, one schema-from-registry derivation path, one bounded
expander, seven operator-facing registered actions plus one internal
step-confirmation target, one core settings namespace pair
(`workflows.*` and `plan.*`), three permission classes, four
operation classes, one origin kind, two URI schemes (`workflow://` and
`plan://`), one workspace surface provider with two panels, an intent
phrase corpus, and a routing predicate — but no new step kinds (the
six v0.24 kinds are exhaustive), no new objective store, no new
scheduler, no new authority layer.

## v0.24 Objective Runtime Substrate (Consumed)

v0.44 is a surface; v0.24 is the runtime. The substrate already provides:

```text
AllbertAssist.Objectives  (facade — lifecycle public API)
└── Objectives.Engine.Agent  (JidoBacked seven-stage state machine)
    ├── commands: FrameObjective, ProposeSteps, EvaluateSteps,
    │             AuthorizeStep, ExecuteStep, ObserveStep,
    │             AdvanceObjective, CancelObjective, ContinueObjective
    ├── store: SQLite tables `objectives`, `objective_steps`,
    │           `objective_events`
    └── registered actions: list_objectives, show_objective,
                             cancel_objective, continue_objective,
                             delegate_agent
```

The six step kinds (`action`, `ask_user`, `wait`, `observe`, `reflect`,
`delegate_agent`) are frozen at v0.24. v0.44 adds NO new step kinds;
every YAML step lowers to one of the six.

The `:delegate_agent` minimal contract (ADR 0021 §A3) is in place:
`objective_steps.kind = :delegate_agent` with
`objective_steps.delegate_agent_id` populated, dispatched through
`AllbertAssist.Objectives.AgentRegistry` and the registered
`delegate_agent` action. v0.44 surfaces this in plan previews and run
progress; it does not change the contract.

Workflow expansion reuses the v0.24 Objective Runtime storage model: a
workflow expansion produces a step list that is persisted as proposed
`objective_steps` with proposal events after the plan-start confirmation.
M2 writes those rows through the public Objective facade. If a later
milestone uses `Proposer.register_app_proposer/2`, it registers one core
Plan/Build proposer adapter and stores per-run expanded step state in
the objective/proposer hint. It does not register or unregister a new
proposer module per workflow run.

## Workflow YAML Schema (v1)

A workflow YAML file declares a deterministic, inert step list that the
loader expands into `objective_steps.attrs` maps. The schema is JSON
Schema Draft 2020-12, assembled from the current
`AllbertAssist.Actions.Registry.modules/0` snapshot and
`AllbertAssist.Objectives.Step`, and validated at load time. **Unknown
keys fail closed at every level.**

M2 adds a direct JSON Schema validator dependency (planned: `:jsv`,
pinned in `apps/allbert_assist/mix.exs`, not consumed transitively
through another package). If the implementation chooses another
validator, this plan and ADR 0041 must be updated before code lands.

YAML parser policy is binding: the loader bounds file bytes before
parsing; calls `YamlElixir.read_from_string/2` with string-key output
and no atom creation; rejects custom YAML node behavior; and either
rejects anchors/merge keys or normalizes them before validation with a
JSON-Pointer diagnostic. `:ymlr` is available only for deterministic
test/fixture serialization, not for parsing.

### Top-level shape

```yaml
id: nightly-briefing
version: 1
description: |
  Pull GitHub issues assigned to me, summarize, and email me the digest.
owner: local
inputs:
  - name: since
    type: string
    required: false
    default: "1 day ago"
    description: window for issue collection
  - name: recipient
    type: string
    required: true
    description: email address to send to
steps:
  - id: list_issues
    kind: action
    action: github_list_issues
    params:
      assignee: me
      since: ${inputs.since}
    save_as: issues
  - id: summarize
    kind: action
    action: summarize_text
    params:
      input: ${steps.list_issues.summary}
    save_as: summary
  - id: send_email
    kind: action
    action: send_email
    params:
      to: ${inputs.recipient}
      subject: "Nightly briefing"
      body: ${steps.summarize.text}
    if: ${length(steps.list_issues.issues) > 0}
```

Top-level fields:

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | string | yes | matches `workflows.id_pattern`; matches the filename basename (`nightly-briefing.yaml`) |
| `version` | integer | yes | bump on breaking schema change to the workflow document; advisory only — does not gate execution |
| `description` | string | no | operator-visible documentation; bounded to 2000 chars |
| `owner` | string | no | operator-supplied label; advisory |
| `inputs` | array of input descriptors | no | typed and named; defaults supported |
| `steps` | array of step descriptors | yes | one or more; capped at `workflows.max_steps_per_workflow` (default 3, cap 10 in v0.44 to match the existing v0.24 proposer cap) |
| `if` | expression | no | optional workflow-level guard; if false at expansion, workflow yields zero steps with a redacted advisory diagnostic |

Unknown top-level keys reject with a JSON-Pointer diagnostic
(`/version` etc.).

### Input descriptor

```yaml
- name: since               # required; matches `^[a-z][a-z0-9_]*$`
  type: string              # one of: string, integer, number, boolean
  required: false           # default true
  default: "1 day ago"      # type-checked against `type`
  description: "..."        # bounded to 500 chars
```

Inputs are validated when the workflow is invoked, before any step runs.
Missing required inputs reject the run.

### Step descriptor common fields

```yaml
- id: <step_id>             # required; unique within the workflow; ^[a-z][a-z0-9_]*$
  kind: <kind>              # required; enum: action|ask_user|wait|observe|reflect|delegate_agent
  if: <expression>          # optional; evaluated at execution time; false -> step status :skipped with reason :if_false
  save_as: <name>           # optional; ^[a-z][a-z0-9_]*$; aliases the primary result field
  on_error: <abort|continue> # default abort; ask_user-on-error is parked
```

Per-kind discriminator payloads via JSON Schema `oneOf` keyed on `kind`:

#### kind: action

```yaml
- id: list_issues
  kind: action
  action: github_list_issues   # must be a registered action name
  params:                      # validated against the action's registered schema at load time
    assignee: me
    since: ${inputs.since}
```

`params:` is validated at LOAD time against the action's registered
`schema/0`. Type mismatches, missing required params, and unknown params
reject the workflow before any step runs. The v0.42 R4 lesson (intent
corpus drift) applies here: load-time validation prevents drift between
documented action params and shipped behavior.

#### kind: ask_user

```yaml
- id: confirm_recipient
  kind: ask_user
  prompt: "Send digest to ${inputs.recipient}?"
  options:                     # optional; if absent, ask_user is a free-text response
    - {value: yes, label: "Send it"}
    - {value: no, label: "Cancel the workflow"}
```

`kind: ask_user` is the plan-level checkpoint; it does not require
`:workflow_run_start`-level confirmation because the run was already
approved. The operator response is bound into `${steps.<id>.value}`;
when `save_as:` is set, the same value is also available at
`${steps.<id>.<save_as>}`.

#### kind: wait

```yaml
- id: wait_for_callback
  kind: wait
  until: stocksage.analysis.completed   # signal name; matches `^[a-z][a-z0-9_]+(\.[a-z][a-z0-9_]+)+$`
  match:                                # optional subset filter; signal payload must include these keys/values
    objective_id: ${steps.start.objective_id}
```

OR:

```yaml
- id: wait_briefly
  kind: wait
  for_ms: 5000                          # alternative to `until:`; integer, capped at 600_000 (10 min)
```

Exactly one of `until:` or `for_ms:` must be present.

#### kind: observe

```yaml
- id: observe_run
  kind: observe
  signal: allbert.objective.observed
  match:
    objective_id: ${steps.start.objective_id}
  save_as: observation
```

The observed signal payload is bound into declared output fields under
`${steps.<id>.*}`. When `save_as:` is set, the full observed payload is
also available at `${steps.<id>.<save_as>}`.

#### kind: reflect

```yaml
- id: reflect_on_run
  kind: reflect
  prompt: |
    Did the digest reach the recipient? Note any retry-worthy failures.
  inputs:                              # optional; named inputs to the reflection model
    - issues: ${steps.list_issues.issues}
    - send_status: ${steps.send_email.status}
```

`reflect` proposes memory or workflow candidates but never writes; the
v0.21 review surface and v0.46 self-improvement draft path are the
writers.

#### kind: delegate_agent

```yaml
- id: analyze_aapl
  kind: delegate_agent
  delegate_agent_id: stocksage.specialist
  command: run_analysis              # forwarded to the registered specialist
  params:
    symbol: AAPL
  save_as: aapl_analysis
```

Forwards to v0.24's `AllbertAssist.Objectives.AgentRegistry` through the
registered `delegate_agent` action. The agent must be registered at app
boot; unregistered ids reject at load time.

### Reference grammar (closed)

Substitutions use `${...}` and are AST-parsed at load time. The grammar:

```
substitution  := "${" expr "}"
expr          := primary ( binop primary )*
primary       := literal | reference | function_call | "(" expr ")" | unary primary
reference     := path
path          := ident ( "." ident | "[" literal "]" )*
ident         := [a-z][a-z0-9_]*
function_call := func_name "(" arg_list ")"
func_name     := one of: length, contains, starts_with, ends_with,
                         lower, upper, default, to_json, from_json
binop         := "==" | "!=" | "<" | "<=" | ">" | ">=" | "&&" | "||"
unary         := "!"
literal       := string | integer | number | boolean | null
```

When a YAML scalar is exactly one substitution, the resolved value keeps
its native type. When a substitution appears inside a larger string, the
resolved value is stringified with the deterministic redactor applied.
Substitutions are not allowed in `id`, `kind`, `action`,
`delegate_agent_id`, `save_as`, or top-level schema keys.

References must resolve to one of:

- `inputs.<name>` — a declared input
- `steps.<earlier_step_id>.<field>` — a declared output field or
  `save_as` alias from an earlier step (forward refs reject at load)
- `user.locale` or `user.timezone` — runtime-provided
- `workflow.id` or `workflow.version` — the current document

Anything else rejects with a JSON-Pointer diagnostic. **No
`${secrets.x}`. No `${env.x}`.** Secrets resolve through `secret://`
settings refs at the action layer, not in YAML.

### Schema-from-registry derivation

The v1 schema is assembled from the current action registry snapshot:

```elixir
defmodule AllbertAssist.Workflows.Schema do
  @schema_version 1
  def json_schema(action_modules \\ AllbertAssist.Actions.Registry.modules()), do: # ... assembled from:
  #   - top-level fields (this module)
  #   - per-step common fields (this module)
  #   - per-kind oneOf branches (one branch per Step.kinds())
  #   - per-action params (each registered action module's schema/0)
end
```

When an action's `schema/0` changes, the workflow schema changes; when
a new action is registered, the workflow schema gains it
automatically. Hand-maintained schema drift is impossible by construction.
Because plugin and dynamic actions can enter the registry after compile
time, the implementation must not embed a compile-time-only registry
snapshot.

### Validation error shape

```elixir
{:error, %AllbertAssist.Workflows.SchemaError{
  pointer: "/steps/2/params/since",
  reason: :type_mismatch,
  expected: "string",
  got: "integer",
  workflow_id: "nightly-briefing",
  redacted_value: "<int>"   # never echoes raw operator input
}}
```

The diagnostic carries the JSON-Pointer path of the offending key. The
v0.43 R6 lesson (structured doctor diagnostics) applies: error_category
is structured so the operator can fix the issue without parsing free
text. Categories: `:unknown_key`, `:type_mismatch`,
`:missing_required`, `:invalid_reference`, `:unknown_action`,
`:unknown_step_kind`, `:cycle`, `:forward_ref`, `:unknown_delegate_agent`,
`:invalid_id_pattern`, `:cap_exceeded`, `:invalid_expression`,
`:dynamic_action_name`, `:secret_substitution_attempt`,
`:env_substitution_attempt`, `:workflow_not_found`, `:no_workflows_dir`.

### Schema invariants (binding)

1. `additionalProperties: false` at every object level.
2. Top-level `version` is required; missing rejects.
3. `id` matches `^[a-z0-9][a-z0-9_-]*$`; matches filename basename.
4. Step ids match `^[a-z][a-z0-9_]*$`; unique within workflow.
5. Step `kind` is one of the six v0.24 kinds; unknown rejects.
6. References to `inputs.<name>` require a declared input.
7. References to `steps.<id>.<field>` require an earlier step with a
   declared output field or matching `save_as:` alias (forward refs
   reject).
8. Cycles in step dependencies (derived from references) reject at load.
9. Action names must resolve to a registered action; unknown rejects.
10. `params:` for `kind: action` validates against the action's
    registered `schema/0`.
11. `delegate_agent_id` must resolve to a registered agent in
    `AllbertAssist.Objectives.AgentRegistry`.
12. Expressions must parse against the closed grammar; unknown
    functions, dynamic action-name lookups (`action: ${...}`),
    `${secrets.x}`, and `${env.x}` reject.
13. `workflows.max_steps_per_workflow` cap (default 3, cap 10 in v0.44)
    enforced; total
    serialized YAML bytes capped by
    `workflows.max_param_bytes_per_step * step_count` per request.
14. Workflow files larger than 256 KiB reject before parsing.

## Plan Preview Contract

The Plan Preview Contract is the typed packet `preview_plan` emits. It
is descriptive metadata; **never authority** (ADR 0021 §4 applied here).

```elixir
%AllbertAssist.PlanBuild.Preview{
  workflow_id: "nightly-briefing",
  workflow_version: 1,
  resolved_inputs: %{since: "1 day ago", recipient: "me@example.com"},
  objective_title: "Nightly briefing",
  step_count: 3,
  steps: [
    %AllbertAssist.PlanBuild.PreviewStep{
      ordinal: 1,
      id: "list_issues",
      kind: :action,
      action_name: "github_list_issues",
      params_summary: %{assignee: "me", since: "1 day ago"},
      permission: :integration_read,
      safety_floor: :allowed,
      resources_needed: ["mcp://github.com/list_issues"],
      estimated_cost: %{tokens: 0, dollars: 0.0, seconds: 2},
      confidence_tier: :green,
      confirmations_required: false,
      subagent_target: nil,
      failure_blast_radius: %{halts_at: 1, unreachable: ["summarize", "send_email"]}
    },
    # ... summarize step ...
    %AllbertAssist.PlanBuild.PreviewStep{
      ordinal: 3,
      id: "send_email",
      kind: :action,
      action_name: "send_email",
      params_summary: %{to: "me@example.com", subject: "Nightly briefing"},
      permission: :channel_send,
      safety_floor: :needs_confirmation,
      resources_needed: ["channel://email"],
      estimated_cost: %{tokens: 0, dollars: 0.0, seconds: 3},
      confidence_tier: :yellow,
      confirmations_required: true,
      subagent_target: nil,
      failure_blast_radius: %{halts_at: 3, unreachable: []}
    }
  ],
  authority_gates: [
    %{ordinal: 0, gate: :workflow_run_start, scope: "workflow://nightly-briefing"},
    %{ordinal: 3, gate: :channel_send, scope: "channel://email"}
  ],
  warnings: []   # bounded list of operator-facing advisory warnings
}
```

Fields and their roles:

- `workflow_id`, `workflow_version`, `resolved_inputs` — identify the
  source.
- `objective_title` — what the Objective Runtime will use when framing.
- `step_count`, `steps` — full per-step preview.
- For each step: `ordinal` (1-based; matches the per-step confirmation
  chip "step 2 of 5"); `id`, `kind`, `action_name` or kind-specific
  detail; `params_summary` (redacted of secret-shaped values);
  `permission` and `safety_floor` from Security Central; `resources_
  needed` (resolved through `Resources.Grants`); `estimated_cost`
  (advisory; v0.44 ships deterministic estimates; LLM-cost estimators
  are reserved); `confidence_tier` (`:green`/`:yellow`/`:red` per the
  research-grounded auto-proceed gating); `confirmations_required`
  (bool); `subagent_target` (when `delegate_agent`, the registered
  agent id); `failure_blast_radius` (graph-derived; "what becomes
  unreachable if this step fails").
- `authority_gates` — explicit list of where confirmations land in the
  run sequence; lets the operator see "this plan has two confirmations
  total" before approving.
- `warnings` — operator-facing advisory text (e.g., "this workflow
  references a domain you have not granted browser access to").

The packet is rendered identically in the workspace Plan Preview panel,
CLI (`mix allbert.plan show <objective_id>`), email, and Telegram. Each
surface uses its existing renderer affordance; the v0.49 channel
primitive names are documentation vocabulary only until that milestone
lands.

## Workflow Discovery, Persistence, And Cache

v0.44 stores very little durably beyond what v0.24 already does:

- **No new SQLite tables.** Workflow definitions are read-only files;
  workflow runs are existing `objectives` rows with the new
  `source_intent` value `workflow:<workflow_id>:<version>` so they are
  filterable.
- **Workflows directory.**
  `<ALLBERT_HOME>/workflows/<workflow-id>.yaml` per the ADR 0041
  amendment; one file per workflow; collisions on `id` fail closed.
- **Missing directory behavior.** When
  `<ALLBERT_HOME>/workflows/` does not exist, `list_workflows`
  returns an empty list with a `:no_workflows_dir` advisory diagnostic
  (no error, no auto-create). `inspect_workflow`, `expand_workflow`,
  `preview_plan`, and `start_plan_run` reject with
  `error_category: :workflow_not_found` and a JSON-Pointer-style
  message pointing at the requested id. The directory is created
  lazily on the first successful operator write to a workflow file
  (operator action, never plugin or LLM); v0.44 does not ship a
  registered action that writes workflow files.
- **Discovery is on-demand.** The loader reads a file only when an
  operator (or a registered action acting under operator authority)
  references the workflow id. There is no autoload, no startup scan,
  no background watch.
- **No durable schema cache.** The workflow schema is rebuilt
  deterministically from the current action registry snapshot for each
  validation request or held only in process-local ephemeral state; there
  is no persisted cache to invalidate.
- **No expansion cache.** Each expansion re-validates and re-resolves
  references; the cost is bounded by the cap.
- **Audit events** flow through `AllbertAssist.Security.Audit` with
  redaction of secret-shaped operator-supplied input values.
- **Plan preview packets** are ephemeral; they live on the workspace
  surface (per ADR 0023 ephemeral substrate) and in trace metadata, not
  on disk.
- **Run identity.** `plan://run/<objective_id>` is the URI for a
  plan-run; the underlying objective row carries the
  `source_intent: "workflow:<id>:<version>"` field for filtering and
  the new `parent_objective_id` for nested runs (already present in
  v0.24 schema).

The directory is operator-readable and inspectable; v0.51 export/import
preserves it.

## Permission, Operation, And Identity Additions

### Permission classes (`AllbertAssist.Security.Policy`)

| Permission | Default | Floor | Notes |
|---|---|---|---|

- Add `workflow` and `plan` to the supported scheme set.

### Plan-Build action output shape (ADR 0029 normalized responses)

All Plan-Build actions return the map-shaped contract implemented by
`AllbertAssist.Runtime.Response`: every successful runner result is
`{:ok, response_map}` with `:status`, `:message`, `:actions`,
`:diagnostics`, and any Plan/Build payload under `:output_data`.

**v0.44 `:output_data` convention (binding).** v0.43 actions (browser
extract, navigate, screenshot, etc.) place domain-specific result data
at top-level keys (`:extraction`, `:session_id`, …) because
`Runtime.Response`'s `optional(atom()) => term()` catch-all permits any
extra key. v0.44 standardizes on a single nested `:output_data` map for
Plan/Build payloads so panel renderers, CLI formatters, trace surfaces,
and the deterministic `release.v044` gate can pattern-match one path
(`get_in(response, [:output_data, :step_count])`) instead of branching
on N top-level keys. The Response shape is unchanged; v0.44 only
introduces the convention key `:output_data`. v0.43 actions keep their
flat-key shape; v0.45+ may adopt the `:output_data` convention as
release scope permits.

- `expand_workflow` -> `{:ok, %{status: :completed, output_data:
  %{steps: [...], step_count: integer(), preview: %PlanBuild.Preview{}}}}`.
- `preview_plan` -> `{:ok, %{status: :advisory, output_data:
  %{preview: %PlanBuild.Preview{}}}}` (advisory-only, ADR 0021 §4).
- `start_plan_run` first call -> `{:ok, %{status: :needs_confirmation,
  confirmation_id: id, output_data: %{preview: %PlanBuild.Preview{}}}}`;
  approval-resume through `approve_confirmation` must expose the resumed
  target status and the created `objective_id` in normalized output data.
- `cancel_plan_run` -> `{:ok, %{status: :completed, output_data:
  %{objective_id: id, cancelled?: true}}}`; or `{:ok, %{status:
  :denied, ...}}` when the objective is already terminal.
- Validation failures across all actions → `{:error, %SchemaError{}}`
  or `{:ok, %{status: :error, ...}}` per the v0.31 M6 normalizer.

## Operator-Visible Win

An operator can preview multi-step work, inspect required resources and
confirmation points, edit or cancel the plan, then run it step by step.

### First-time walkthrough — natural-language plan

1. Operator: "plan: pull GitHub issues assigned to me, summarize, and
   email me the digest at me@example.com".
2. Intent layer routes to `preview_plan` over a synthesized one-off
   workflow (the natural-language plan path; no file on disk).
3. Allbert pins a **Plan Preview panel** into the workspace canvas,
   adjacent to the chat thread that produced it. The panel shows:
   - workflow title and step count;
   - per-step ordinal, kind, action name, params summary,
     permission and safety floor, resources needed, estimated cost,
     confidence tier (color-coded), confirmations required,
     subagent delegation indicator, failure blast radius;
   - the explicit authority-gates list ("two confirmations total in
     this plan").
4. Operator inspects the third step (`send_email`) and sees its yellow
   tier + `:needs_confirmation` floor.
5. Operator expands the panel to fullscreen (Claude Plan-Mode Ctrl+G
   analog), edits the recipient address, saves, and the preview
   recomputes deterministically.
6. Operator approves the plan-start gate — a single
   `:workflow_run_start` `:needs_confirmation` action; one click, not
   per-step rubber-stamping.
7. The Objective Runtime frames the objective, INSERTs the three steps,
   and the engine begins. Each step's events post into the workspace
   chat timeline and the Plan Run Progress panel.
8. The `send_email` step pauses for its own per-action
   `:needs_confirmation` confirmation; operator approves and the email
   sends. Final objective status: `:completed`.

If the operator denies the plan-start gate, nothing runs. If the
operator approves the gate but denies the `send_email` step, the step
is `:cancelled` and the objective transitions to `:completed` with a
redacted last-observation summary noting the deny.

### Second walkthrough — authored workflow

1. Operator authors `<ALLBERT_HOME>/workflows/nightly-briefing.yaml`
   matching the v1 schema.
2. Operator: "run workflow nightly-briefing recipient=me@example.com".
3. The loader reads the file, validates (unknown keys reject), resolves
   `${inputs.recipient}`, expands into objective-step attrs.
4. `preview_plan` emits the Plan Preview Contract packet; the
   workspace panel renders.
5. Operator approves the plan-start gate; the run proceeds exactly as
   the natural-language path.

### Cancellation walkthrough

1. Mid-run, the operator clicks "cancel plan" on the Run Progress
   panel.
2. `cancel_plan_run` calls `Objectives.cancel/3` with the operator's
   reason. The in-flight step completes normally (v0.24 cooperative
   cancellation rule); no new step starts.
3. The objective transitions to `:cancelled`; the panel shows the
   cancellation reason and which steps did not run.

### Subagent delegation walkthrough

A workflow with a `kind: delegate_agent` step shows the child agent's
target id, command, and params summary inline in the parent step's
expanded view. When the child runs, its lifecycle events render
adjacent to the parent step — the operator can see at a glance which
specialist did what, with what permission scope. (Resolves the
"invisible subagents" UX lesson from Anthropic's plan-mode feedback.)

## Why Plan/Build Is A Panel

Decision 1 in the M1 lock table is the binding rationale; this section
expands it.

The 2025-2026 prior art converged on plan-adjacent panels:

- **Cursor 2.2 Plan Mode panel** — a side panel next to the chat
  thread; not a separate route.
- **Claude Code plan mode** — the plan is a markdown file edited in
  place; the UI is a confirmation screen at `ExitPlanMode` time, not
  a destination.
- **Devin plan card** — pinned card next to the live agent thread;
  cancellable and editable inline.

All three are panels because plans need ambient context — which
connector? which prior step output? which thread? — and a destination
strips that context. Allbert's v0.23/v0.26 canvas + v0.24 panels/zones
substrate is exactly the right place: a destination would duplicate
canvas chrome and split the operator's mental model.

The subagent delegation visibility UX (Anthropic issue feedback) lands
naturally on a panel: child agent calls render inline alongside parent
step events, in the same timeline the operator already watches.

What we give up by choosing panel-over-destination:

- A dedicated `/plan` route with stable bookmarkable URLs. (We
  compensate: `plan://run/<objective_id>` is a stable identity, and
  `/objectives/<id>` from v0.24 is already a stable destination for
  the resulting run.)
- A larger default rendering area. (We compensate: panel expand-to-
  fullscreen, mirroring Claude Plan-Mode's $EDITOR move.)
- Cross-thread plan composition. (Parked under "Multi-User
  Collaborative Plan Editing" in future-features; not a v0.44 user
  story.)

The destination option remains promotable: if a future release ships
team-wide plan boards, the destination can be added without removing
the panel.

## Security Posture

- **YAML is inert data. The runtime never executes it.** The loader is
  read-only. Expansion produces step attrs only; the v0.24 engine
  takes over from there.
- **Closed expression grammar.** `${...}` is AST-parsed at load time
  against a fixed grammar. Unknown identifiers, dynamic action-name
  lookups, `${secrets.x}`, and `${env.x}` reject with structured
  diagnostics. No `eval`, no Jinja-with-callables, no string
  concatenation that produces action names.
- **`additionalProperties: false`** at every schema layer; unknown
  keys reject with JSON-Pointer diagnostics.
- **Confirmation upgrade-only rule.** YAML `confirm: true` may raise a
  step's confirmation requirement above its action's registered floor,
  but never below. The plan-start gate is a separate
  `:workflow_run_start` confirmation; per-step floors still apply at
  execution time.
- **Plan-start gate is the authority transition.** No effectful step
  runs before the plan-start gate is approved. The gate is one
  confirmation per run, not per step.
- **Plan preview is descriptive.** Preview packets are advisory
  metadata per ADR 0021 §4; no permission grant flows from preview.
- **Per-step permissions enforced at execution.** Security Central
  re-authorizes each step at execution time regardless of plan-time
  preview — preview is a forecast, not a contract.
- **No `${secrets.x}`.** Secrets live in Settings Central secret store
  and resolve via `secret://` settings refs at the action layer, not
  in YAML.
- **No dynamic action names.** `action: ${inputs.x}` rejects at load;
  the validator requires `action:` to be a literal string matching a
  registered action.
- **No `env:` block.** Environment comes from the runtime, not the
  document.
- **Anti-rubber-stamping mitigations (research-grounded).** Plan
  Preview Contract carries:
  - **diff against last approved run** — when a workflow has been run
    before, the preview highlights changes in inputs or steps;
  - **risk-tier color coding** — green/yellow/red per the Devin
    precedent;
  - **cost visibility per step** — estimated tokens/dollars/seconds;
    research shows operators rubber-stamp less when cost is shown;
  - **failure blast radius** — graph-derived "what becomes
    unreachable if this step fails";
  - **`auto_proceed_green_tier`** is opt-in (default off) and applies
    only to `:allowed`-floor steps; any `:needs_confirmation` step
    always confirms.
- **Subagent delegation visibility.** The child agent's id, command,
  params summary, and permission scope render inline in the parent
  step's expanded view. Resolves the "invisible subagents" UX lesson.
- **Trace redaction.** Plan preview packets, workflow YAML
  operator-supplied values, and run timeline events are scrubbed of
  secret-shaped strings before any record is written under
  `memory/traces/`. The `Security.Redactor` extension covers
  operator-supplied `inputs:` values, `params:` values, and
  reference-resolved strings.
- **Cooperative cancellation.** `cancel_plan_run` calls
  `Objectives.cancel/3` per v0.24 rule; the in-flight step completes,
  no new step starts, the operator's reason is durable.
- **No new memory promotion.** Workflow expansion writes nothing to
  memory. `kind: reflect` proposes candidates only; the v0.21 review
  surface remains the only writer.
- **No new objective store.** All plan runs are existing `objectives`
  rows; no parallel substrate.

## Non-Goals

- No autonomous plan approval.
- No workflow YAML script execution, shell, or package install.
- No private objective store outside `AllbertAssist.Objectives`.
- No lowering of action-level confirmations.
- No `for_each`, `for`, `while`, or other loop kinds in YAML.
- No `parallel:`, `Fork`, or fan-out in YAML.
- No sub-workflow includes (`include:`, `import:`).
- No `on:` triggers (no schedule, no event); workflows are
  operator-referenced.
- No `env:` block at workflow scope.
- No retry/backoff policies in YAML.
- No dynamic action-name resolution (`action: ${...}`).
- No `${secrets.x}` substitution.
- No LLM-authored workflow YAML in v0.44 (that is v0.46
  self-improvement drafts).
- No plan-time spend enforcement (cost is descriptive only; real
  spend enforcement is a future-feature).
- No multi-user collaborative plan editing.
- No remote workflow distribution.
- No public workflow API outside `mix allbert.workflows`.
- No `/plan` destination route at v0.44; Plan/Build is a panel.
- No new step kinds; the six v0.24 kinds are exhaustive.
- No new permission vocabulary outside the three `:workflow_*`/
  `:plan_*` classes.

## Definition Of Done

v0.44 is done when:

- a disposable-home operator can author or request a plan, preview
  the Plan Preview Contract packet (with all fields rendered), edit
  inputs and reorder/remove steps, approve the plan-start gate, see
  the run proceed through the Objective Runtime with per-step
  events rendering in workspace and existing supported surfaces,
  inspect subagent delegation inline, and cancel cooperatively with
  a durable reason;
- the three `:workflow_*` / `:plan_*` permission classes and floors
  are enforced; the four operation classes and the `:plan_run`
  origin kind are part of `Resources.OperationClass`;
  `workflow://<id>` and `plan://run/<objective_id>` are normalized and
  rendered by `Resources.ResourceURI`; the `workflows.*` + `plan.*`
  settings namespace exists with documented invariants and
  `schema_version: 1`;
- the workflow YAML v1 schema is assembled from
  `Actions.Registry.modules/0` + `Step.kinds()`, validates unknown keys
  closed at every level, AST-parses the closed expression grammar,
  rejects dynamic action names, `${secrets.x}`, `${env.x}`, cycles,
  forward refs, missing inputs, and unknown actions/agents with
  structured `error_category` diagnostics;
- the plan-start gate (`:workflow_run_start`) is the only authority
  transition from preview to run; per-step confirmation floors are
  enforced at execution time and YAML cannot lower them;
- the Plan/Build workspace surface renders Preview + RunProgress
  panels and never calls the loader, validator, expander, or driver
  directly;
- subagent delegation visibility (`expanded_inline`) is the
  shipped default; the issue-#12142 "invisible subagents" lesson is
  resolved;
- the v0.44 security eval set passes on the deterministic gate;
