# v0.5 — Curated memory

Status: Proposed

## Context

v0.1 established markdown memory files as durable ground truth (ADR 0003). v0.2 and v0.3 kept memory simple: read, write, append, grep. That works while sessions are short and context needs are modest. It starts to hurt once:

- sessions span many turns or days (the REPL attach-from-multiple-clients surface),
- scheduled jobs want tailored context without sending an entire memory tree through every prompt,
- new channels (v0.6) need sparse, asynchronous context assembly,
- agents and sub-agents need different slices of memory for different sub-tasks.

v0.5 introduces curated memory: ranked retrieval over structured memory tiers, with a write-back path that stages new learnings before promotion.

## Goal

Give the kernel a memory retrieval surface that is:

- **tiered**: identity/profile files, durable learned memory, and ephemeral session context live in distinct stores with different retention and access semantics;
- **ranked**: BM25-style ranked retrieval over durable memory, replacing naive grep for "what do I know about X";
- **staged**: new learnings land in a staging area that requires explicit promotion before they join durable memory;
- **skill-consumable**: memory retrieval is a small kernel-owned runtime seam, while indexing, summarisation, promotion assistance, and memory-specific workflows remain skill-extensible through the v0.4 folder format.

## Sketch

Three memory tiers, distinguished by purpose and retention:

1. **Identity tier** — bootstrap markdown files (persona, user profile, working conventions). Loaded always. Edited rarely. ADR 0010 already covers the shape.
2. **Durable tier** — long-term learned memory. Written slowly, promoted explicitly, indexed for retrieval.
3. **Ephemeral tier** — per-session context. Discarded at session end. Useful for short-term tracking the user doesn't want persisted.

Retrieval:

- BM25 ranked retrieval is the v0.5 default over the durable tier. Index rebuilds on durable-memory writes and on explicit index-maintenance operations; memory-aware skills may help drive those maintenance flows, but they do not redefine the retrieval contract.
- Retrieval is a kernel-owned tool call (`search_memory(query, tier, limit)`) available to agents and skills.
- Retrieval respects the session policy envelope — skills cannot retrieve from tiers they lack access to.

Write-back (staging → promotion):

- Agent-authored "learnings" write to `~/.allbert/memory/staging/` with timestamps and source attribution (which agent, which session, which turn).
- A promotion step — prompt-native or CLI (`allbert-cli memory promote`) — moves staged entries into the durable tier after user review.
- The staging area is the only place the kernel ever writes durable memory from agent action. Direct writes to the durable tier still require user or skill action through existing tools.

Progressive disclosure (ADR 0036) applies to memory retrieval: tier 1 returns document titles + short snippets, tier 2 returns full bodies only when explicitly requested.

## Runtime defaults

- **Ephemeral tier lifetime**: the ephemeral tier follows the daemon session lifetime. It survives REPL detach/reattach while the session exists, but it is not restart-durable across daemon restarts.
- **Index portability**: the memory index is a derived local artifact and is rebuilt locally. It is never treated as the portable source of truth across machines.
- **Staging retention**: staged entries linger until explicit promotion, rejection, or operator cleanup. v0.5 does not auto-expire them silently.
- **Identity separation**: promotion never writes into bootstrap identity files from ADR 0010. Bootstrap edits remain a separate, explicit flow.

## Expected ADRs

- Memory tiers and retention semantics.
- BM25 retrieval as the v0.5 default (with a pluggable retriever seam for future reranking / embeddings).
- Staging → promotion write-back rule (no agent writes directly to durable memory).
- Memory index is a derived artifact — always rebuildable from the markdown ground truth.

## Dependencies

- v0.4 AgentSkills format — memory-aware skills ship as folder skills that complement, rather than replace, the kernel retrieval surface.
- v0.3 intent routing — `memory_query` intent already in the taxonomy.
- ADR 0003 durable memory rule — v0.5 extends, does not replace.

## Out of scope for v0.5

- Embedding-based retrieval (pluggable seam, default implementation deferred).
- Cross-session memory diffing / merging beyond the staging model.
- Memory compression / summarisation as a kernel feature — skills can own this.
- Per-agent private memory stores.
- Network-replicated memory.

## Remaining planning questions

- What scoring fields beyond BM25 land in v0.5 proper versus a later reranker seam?
- Does staged-entry review happen only through existing CLI/prompt flows, or does v0.5 need a dedicated review surface?
