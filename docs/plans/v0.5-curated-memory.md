# v0.5 — Curated memory

Status: Proposed

## Context

v0.1 established markdown memory files as durable ground truth (ADR 0003). v0.2 and v0.3 added a daemon, scheduled jobs, and first-class agents. v0.4 made the skill system resource-aware and portable. What the runtime still lacks is a complete memory contract for those richer sessions.

The current memory surface is intentionally small:

- `MEMORY.md` head and recent daily context can be injected into prompts.
- `read_memory` and `write_memory` operate on the markdown ground truth directly.
- jobs, sub-agents, and channels can all exist, but they still rely on a thin memory story.

That is no longer enough once:

- sessions span many turns or long-lived daemon attachments,
- sub-agents need narrower memory slices than the root agent,
- scheduled jobs need curated context rather than the whole memory tree,
- sparse asynchronous channels need bounded context assembly,
- the assistant needs to learn durable facts without silently mutating the user's memory corpus.

v0.5 introduces curated memory as a kernel capability, not a prompt trick: tiered memory stores, bounded retrieval, explicit staging and promotion, and a stable turn-flow contract that says when memory is assembled, retrieved, refreshed, and written back.

## Goal

Give the kernel a memory system that is:

- **tiered**: identity, durable memory, staging, and ephemeral working memory are separate stores with different retention and visibility rules;
- **ranked**: durable recall uses a bounded ranked retriever instead of naive grep for "what do I know about X";
- **staged**: autonomous learnings go to a reviewable staging area before they become durable memory;
- **agent-aware**: root agents and sub-agents receive different memory slices on purpose rather than inheriting everything by accident;
- **tool-native**: curated memory extends the existing tool loop and daemon session model rather than introducing a separate memory runtime;
- **skill-extensible**: the kernel owns retrieval and staging rules, while skills package higher-level memory workflows such as review, promotion assistance, compaction, and maintenance.

## Non-goals

v0.5 does not include:

- embeddings or vector retrieval as the default runtime path;
- autonomous direct writes from agent reasoning into approved durable memory;
- replaying full chat transcripts as a second hidden memory system;
- per-agent private durable memory stores;
- network-replicated or multi-user memory;
- nightly retraining or model distillation for memory/personality;
- memory compression or summarisation as a required kernel feature.

## Memory model

v0.5 uses four memory tiers.

| Tier | Purpose | Retention | Default visibility |
| --- | --- | --- | --- |
| `identity` | bootstrap identity and operating context | durable | always loaded |
| `durable` | approved long-term facts, conventions, decisions | durable | bounded prefetch + explicit search/read |
| `staging` | candidate learnings awaiting review | durable until reviewed | never auto-injected; explicit search/review only |
| `ephemeral` | session working memory and scratch coordination state | session-scoped only | bounded session synopsis |

### Identity tier

The identity tier is the existing bootstrap bundle plus `AGENTS.md`:

- `SOUL.md`
- `USER.md`
- `IDENTITY.md`
- `TOOLS.md`
- optional `BOOTSTRAP.md`
- `AGENTS.md`

This tier is already governed by ADR 0010 and ADR 0039. v0.5 does not merge it into memory proper. Identity stays distinct from learned durable memory.

### Durable tier

The durable tier is the approved long-term memory corpus. It remains markdown-grounded and operator-visible:

- `~/.allbert/memory/MEMORY.md`
- `~/.allbert/memory/daily/`
- `~/.allbert/memory/notes/`

`MEMORY.md` remains the human-readable catalog and always-nearby synopsis, not the entire durable corpus. `daily/` remains the recency-oriented note stream. `notes/` holds approved durable memory documents that are indexed for retrieval.

### Staging tier

The staging tier is the only place autonomous learnings land from agent action:

- `~/.allbert/memory/staging/`

Each staged entry carries source attribution:

- agent name
- session ID
- turn ID
- timestamp
- source kind (`explicit_request`, `learned_fact`, `job_summary`, `subagent_result`, etc.)

Staged entries are reviewable, promotable, rejectable, and searchable, but they are not treated as approved durable memory until promoted.

### Ephemeral tier

The ephemeral tier is daemon-session working memory:

- survives REPL detach/reattach while the daemon session exists
- is lost on daemon restart
- may update multiple times in a turn
- is not written to disk in v0.5

It is for:

- temporary task state
- pending approvals
- recent tool outputs
- sub-agent return summaries
- transient facts the operator did not ask to persist

It is not a shadow durable memory store.

## Files and derived artifacts

### Operator-edited ground truth

- `~/.allbert/memory/MEMORY.md`
- `~/.allbert/memory/daily/*.md`
- `~/.allbert/memory/notes/**/*.md`
- `~/.allbert/memory/staging/*.md`

`MEMORY.md` is the human-readable catalog; it summarizes and cross-links the durable corpus but does not replace the files it references.

### Kernel-maintained metadata

- `~/.allbert/memory/manifest.json`

`manifest.json` is a kernel-maintained inventory of approved durable documents with their paths, titles, tags, and content hashes. It is not the user-facing source of truth; it is rebuildable from `notes/` and `daily/` whenever missing or stale. Operators edit markdown; they do not edit `manifest.json` by hand except in recovery scenarios.

### Derived artifacts

- `~/.allbert/memory/index/meta.json` (retriever schema version, last rebuild time, doc counts)
- `~/.allbert/memory/index/tantivy/` (on-disk index directory owned by the retriever)

The index is never the source of truth. It can always be rebuilt from the markdown corpus and `manifest.json`. See [ADR 0046](../adr/0046-v0-5-memory-retrieval-uses-tantivy.md) for the retriever commitment.

### Staged entries

Staged entries live as individual markdown files under `~/.allbert/memory/staging/`, named `YYYYMMDDTHHMMSSZ-<short-hash>.md`. Each file has YAML frontmatter defined in [ADR 0047](../adr/0047-staged-memory-entries-have-a-fixed-schema-and-limits.md). The staging directory is watched for size/age limits; see the same ADR for caps and TTL.

## Budgets and synopsis composition

Bounded means something specific in v0.5. The always-on synopsis assembled at turn start has a hard byte budget, not a soft preference.

| Budget | Default | Purpose |
| --- | --- | --- |
| `max_synopsis_bytes` | 8 192 | total bytes of the always-on memory synopsis after rendering |
| `max_memory_md_head_bytes` | 2 048 | slice of `MEMORY.md` head included in the synopsis |
| `max_daily_head_bytes` | 2 048 | today's daily note head included in the synopsis |
| `max_daily_tail_bytes` | 1 024 | optional yesterday-note tail included only if total fits |
| `max_ephemeral_summary_bytes` | 2 048 | bounded session-summary slice of ephemeral memory |
| `max_prefetch_snippets` | 5 | number of retrieval tier-1 results carried into the root prompt |
| `max_prefetch_snippet_bytes` | 512 | per-snippet ceiling after rendering |
| `max_ephemeral_bytes` | 32 768 | rolling cap on the per-session ephemeral memory buffer; oldest-first eviction |
| `max_staged_entries_per_turn` | 5 | hard cap on how many staged entries one turn may create |

Truncation is deterministic. If the rendered synopsis exceeds `max_synopsis_bytes`, the kernel drops the optional daily tail first, then trims prefetch snippets by score ascending, then trims the daily-note head, then the `MEMORY.md` head, never the ephemeral session summary (which already respects its own cap). A `SynopsisTrimmed` trace event records what was dropped and why.

## Concurrency and atomicity

The memory subsystem has multiple writers: the kernel (staging, ephemeral, promotion, index rebuild), the operator (direct markdown edits, CLI commands), and potentially background jobs. v0.5 commits to the following rules:

- **Staging writes are atomic.** New staged entries are written via `tempfile::NamedTempFile::persist` to produce a single `rename(2)` that is visible to readers only once fully written.
- **Promotion is a two-step atomic move.** The kernel writes the promoted content to a temp file under `notes/`, `fsync`s, renames into place, updates `manifest.json` via the same temp-then-rename pattern, and only then deletes the source staged entry.
- **Index rebuild holds an advisory lock** (`memory/index/.rebuild.lock` via `fs2::FileExt`). Concurrent rebuild attempts wait rather than racing. Incremental reads tolerate a rebuild in progress by using the previous index directory until the new one atomically replaces it.
- **Operator markdown edits are not locked.** The kernel treats `notes/` as operator-owned. A file-watch pass in M3+ reconciles `manifest.json` and the index after operator creates, edits, renames, or deletes files under `notes/`, rather than assuming the manifest changed only through kernel writes.
- **Session ephemeral memory is per-session** and never shared across sessions. No cross-session locking required.

## Bootstrap and initialization flow

v0.5 must define exactly what happens before the first curated-memory turn runs.

### Daemon or direct-kernel boot

1. Ensure the memory roots exist, creating them with restrictive permissions if absent:
   - `memory/`
   - `memory/daily/`
   - `memory/notes/`
   - `memory/staging/`
   - `memory/index/`
2. Ensure `MEMORY.md` exists with a seeded header (a minimal template is shipped with the binary; see [Seed templates](#seed-templates)).
3. Ensure `memory/manifest.json` exists. If it is missing on a profile that already has `notes/` content (e.g. upgrading from v0.4), rebuild it by scanning `notes/` and `daily/` and emit a `MemoryManifestRebuilt` trace event.
4. Load the bootstrap identity bundle and `AGENTS.md`.
5. Load `memory/manifest.json`.
6. Check `memory/index/meta.json`. Rebuild the durable index if meta is missing, its schema version is older than the current retriever, doc counts drift from the manifest, or the last-rebuild content hash does not match. Rebuild emits `BeforeIndexRebuild` and `AfterIndexRebuild` hooks with reason, docs indexed, elapsed, and bytes.
7. Do not load the full durable corpus into the prompt; only prepare the bounded synopsis (see [Budgets and synopsis composition](#budgets-and-synopsis-composition)) used during turn assembly.
8. Age out staged entries past the configured TTL (see ADR 0047). Expired entries are moved to `memory/staging/.expired/` rather than deleted, so operators can recover them manually.

Cold-start rebuild is an operational concern. For a corpus under ~1 000 notes, rebuild should stay under one second on a developer laptop. For larger corpora the daemon emits periodic progress events (`IndexRebuildProgress{indexed, total}`) that CLI can render. Rebuild runs under an advisory file lock (`memory/index/.rebuild.lock`) so a concurrent `allbert-cli memory rebuild-index` call waits for the running rebuild rather than racing it.

### Session creation

1. Create an empty ephemeral memory state for the session.
2. Bind that state to the root agent for the session.
3. Initialize session-local memory budgets and retrieval counters.
4. Do not preload staged entries into the session prompt. They remain review surfaces, not always-on context.

## Turn-flow contract

This is the load-bearing v0.5 addition. Curated memory needs one stable turn loop.

### Ordered flow for an interactive turn

1. **Input arrives.**
   The daemon session receives the user turn from the active channel.
2. **Bootstrap snapshot.**
   Snapshot the identity bundle and `AGENTS.md` for the turn.
3. **Always-on memory snapshot.**
   Build the bounded always-on memory synopsis:
   - `MEMORY.md` head or selected catalog slice
   - today's daily note head
   - optional yesterday note tail if within budget
   - bounded ephemeral session summary
4. **Intent classification.**
   Resolve the turn intent before memory prefetch. The intent classifier sees the current user turn and the identity bundle only; it does **not** consume memory prefetch results or the always-on synopsis. This keeps classification cheap and cache-friendly, and prevents a slow classifier from forcing prefetch to run before classification even decides it is needed.
5. **Automatic memory-prefetch decision.**
   The kernel decides whether to run bounded prefetch based on:
   - resolved intent (see [Automatic memory behavior by intent](#automatic-memory-behavior-by-intent))
   - explicit memory cues in the turn
   - whether the turn is a job, chat, task, schedule, or memory query
6. **Bounded durable retrieval.**
   If prefetch fires, the kernel emits `BeforeMemoryPrefetch(session, intent, budget)`, runs one ranked retrieval pass over the approved durable tier only (staging is never auto-injected), and emits `AfterMemoryPrefetch(results, elapsed)`. Tier-1 results carry:
   - titles
   - paths
   - summaries/snippets (truncated to `max_prefetch_snippet_bytes`)
   - scores and provenance for runtime use
7. **Root prompt assembly.**
   Build the root-agent prompt from:
   - bootstrap bundle
   - `AGENTS.md`
   - always-on memory synopsis
   - prefetched retrieval snippets
   - skill discovery hints
   - current user turn
8. **First root-agent model round.**
   The primary agent receives the initial assembled context and plans its work.
9. **Normal tool/skill/sub-agent loop.**
   The existing v0.1-v0.4 loop remains in force:
   - explicit `invoke_skill`
   - explicit tool calls
   - explicit `spawn_subagent`
   - subsequent model rounds
10. **Optional refresh retrieval.**
    After material external evidence arrives from tools or sub-agents, the kernel may allow one bounded refresh retrieval pass for the turn. This is for "now that I saw X, recall what memory says about X," not for unbounded memory thrashing. Governed by `memory.refresh_after_external_evidence` and `memory.max_refreshes_per_turn`; emits `BeforeMemoryPrefetch` / `AfterMemoryPrefetch` with a `refresh: true` flag so hooks can distinguish prefetch from refresh.
11. **Final synthesis.**
    The root agent produces the final user-facing answer.
12. **Post-response memory write-back.**
    After final synthesis:
    - ephemeral memory may update (no hook; high frequency)
    - candidate durable learnings may be staged **only through explicit `stage_memory` calls** the agent made during the turn — the kernel does not run an implicit extraction LLM call. An optional `memory-curator` skill may run its own extraction pass with its own cost accounting (see [Memory-curator skill](#memory-curator-skill))
    - no autonomous durable promotion occurs
    - each stage emits `BeforeMemoryStage(entry)` and `AfterMemoryStage(id)`
13. **Turn-end surfacing.**
    If the turn created any staged entries and `memory.surface_staged_on_turn_end` is true (default), the assistant appends a short hint to its final response: "I'd like to remember N thing(s) — run `allbert-cli memory staged list` or ask me." This is one line, never inline dialog.
14. **Turn end.**
    Emit hooks, traces, cost records, and any staged-entry metadata.

### What changes across repeated model rounds

- The bootstrap snapshot is fixed for the turn.
- The always-on memory synopsis is fixed for the turn unless a deliberate refresh pass is allowed.
- Explicit memory tools can add more memory material to working context.
- Ephemeral memory can update within the turn.
- Root and sub-agents may therefore see different working context over time, but not by silently rewriting durable memory in the middle of the turn.

## Automatic memory behavior by intent

Default prefetch policy in v0.5:

- `memory_query`: prefetch always
- `task`: prefetch normally
- `schedule`: prefetch normally
- `chat`: prefetch only when the turn or session cues make memory relevant
- `meta`: prefetch rarely

This keeps memory useful without forcing every turn through retrieval work.

## Tools, skills, and memory

Allbert already has a tool-use facility. v0.5 must extend that existing loop rather than replacing it.

### Existing tools that remain relevant

- `read_memory`
- `write_memory`
- `invoke_skill`
- `process_exec`
- the rest of the existing tool surface

### New curated-memory tools for v0.5

Minimum kernel-owned memory tool surface:

- `search_memory(query, tier = "durable", limit = memory.default_search_limit)` — ranked retrieval; returns `{id, path, title, snippet, score, tier, provenance}`. `tier` may be `"durable"` or `"staging"`; `"staging"` is only accepted when called from the curator skill or from an operator-invoked CLI path (the raw tool accepts both, but agent turn-time calls default to durable).
- `stage_memory(content, kind, summary, tags?)` — writes a staged entry with the schema from ADR 0047; returns the generated id.
- `list_staged_memory(limit?, since?, kind?)` — browses the staging queue with filters.
- `promote_staged_memory(id, path?, summary?)` — moves a staged entry into `notes/`, updates the manifest, and schedules re-index. Requires `confirm-trust` unless invoked from an operator CLI command.
- `reject_staged_memory(id, reason?)` — moves the entry into `staging/.rejected/`.

The relationship between the old and new surfaces:

- `read_memory` remains the canonical full-body file read for explicit documents.
- `write_memory` remains the raw file-edit seam for explicit operator- or skill-driven edits to the markdown ground truth.
- autonomous learned-memory capture from agent reasoning must go through `stage_memory`, not direct durable writes (ADR 0042).

### Hook and trace surface

v0.5 extends the existing hook surface with memory-specific events. All hooks fire through the same dispatch as `BeforeTool`/`AfterTool`:

| Hook | When | Payload |
| --- | --- | --- |
| `BeforeMemoryPrefetch` | before a prefetch or refresh pass | `{session, intent, budget, refresh}` |
| `AfterMemoryPrefetch` | after a prefetch or refresh pass | `{results, elapsed, truncated}` |
| `SynopsisTrimmed` | when synopsis composition had to drop a source | `{dropped, bytes_over_budget}` |
| `BeforeMemoryStage` | before `stage_memory` write | `{entry}` |
| `AfterMemoryStage` | after `stage_memory` write | `{id, path}` |
| `BeforeMemoryPromote` | before `promote_staged_memory` | `{id}` |
| `AfterMemoryPromote` | after `promote_staged_memory` | `{id, dest_path}` |
| `BeforeMemoryForget` | before operator-confirmed durable deletion | `{path}` |
| `AfterMemoryForget` | after deletion | `{path, trashed_to}` |
| `BeforeIndexRebuild` | before rebuild | `{reason}` |
| `IndexRebuildProgress` | periodic during rebuild | `{indexed, total}` |
| `AfterIndexRebuild` | after rebuild | `{docs_indexed, elapsed, bytes}` |
| `MemoryManifestRebuilt` | after an upgrade or recovery manifest rebuild | `{docs, source}` |

Cost-tracking records any LLM calls inside memory skills (e.g. curator extraction) against the invoking session, not the kernel.

### Skill interaction

- Skills are discovered before the first root-agent round as they are today.
- Explicit skill activation still happens inside the turn loop.
- Memory-aware skills may help with:
  - review
  - promotion assistance
  - compaction
  - pruning
  - maintenance jobs
- Skills must not redefine the kernel retrieval contract.

## Multi-agent memory semantics

The root agent and sub-agents must not see the exact same memory payload by default.

### Root agent

The root agent receives the full turn assembly:

- bootstrap bundle
- `AGENTS.md`
- always-on memory synopsis
- prefetch retrieval snippets
- skill discovery hints
- user turn

### Sub-agents

Sub-agents receive:

- bootstrap bundle
- `AGENTS.md`
- a parent-provided task brief
- a filtered memory slice selected for the sub-task (default heuristic below)
- their own ephemeral scratch state

They do not inherit the full parent retrieved-memory set automatically. Whether a sub-agent may call `search_memory` or `read_memory` is governed by the skill's `allowed-tools` frontmatter (ADR 0031); those tools are in the default v0.5 allow-list but a narrowly scoped sub-agent skill may restrict them.

### Default sub-agent filtering heuristic (v0.5)

When the parent spawns a sub-agent, the kernel builds the filtered slice as:

1. No prefetch snippets inherited by default.
2. If the parent agent passed an explicit `memory_hints: [paths-or-keywords]` field on the spawn call, the kernel resolves path hints as explicit reads (tier 2) and keyword hints as a single bounded `search_memory` pass (tier 1), capped at `memory.max_subagent_snippets` (default 3).
3. Otherwise the sub-agent starts with no retrieved memory and must call `search_memory` / `read_memory` itself.

This heuristic is deliberately simple. It can be replaced by a skill-contributed policy in a later release; ADR 0044 already reserves the room.

### Attribution

If a sub-agent generates a candidate learning:

- it is attributed to that sub-agent,
- it enters the same staged-memory path,
- it does not bypass the review and promotion rules.

## Memory write policy

### Hot-path writes

Allowed in the turn loop:

- ephemeral memory updates
- explicit `stage_memory`
- explicit user requests to remember something
- explicit operator- or skill-invoked extraction work that is still part of the active turn and goes through normal tool/cost accounting

### Background and maintenance work

Allowed through jobs or explicit skills:

- review queues
- promotion assistance
- index rebuilds
- compaction
- pruning
- wiki-like compilation if a skill package provides it

v0.5 should keep the split explicit:

- **hot path** for lightweight working-state updates and candidate staging
- **background** for maintenance, review, and curation work that would otherwise add latency or complexity to the main turn

## Jobs and memory

Scheduled jobs (v0.2) run inside a daemon session. v0.5 specifies:

- A job session gets **fresh ephemeral memory** per run unless the job declares `session_name: <name>` in its frontmatter, in which case the job shares the named persistent session's ephemeral buffer (and competes with it for the rolling cap).
- Intent classification still runs on the job's prompt; most jobs classify as `task` or `schedule`. The classification cost is accounted to the job, not to any interactive user.
- Prefetch fires on the same intent defaults as interactive turns. Jobs that do not want prefetch can opt out with `memory.prefetch: false` in the job frontmatter.
- Job output may call `stage_memory` and attribute with `kind: job_summary`. Jobs cannot call `promote_staged_memory` or `write_memory` against approved durable files without `confirm-trust` — and because channelless jobs fail closed on confirmation-required actions (per v0.2 job policy), those calls resolve to a staged entry that waits for operator review.
- A job that runs frequently and stages a near-duplicate every run would blow up the staging queue. ADR 0047's duplicate-detection rule applies to job-authored entries just as it does to interactive entries.

## Deletion and forgetting

v0.5 treats forgetting as a first-class operation with the same explicitness posture as promotion.

- **Staged entry rejection** uses `reject_staged_memory` / `allbert-cli memory reject`, which moves the entry to `memory/staging/.rejected/<timestamp>-<id>.md` with the reason appended as frontmatter. Rejected entries are retained for 30 days for audit, then garbage-collected.
- **Approved durable deletion** is always an operator-initiated action. Natural-language "forget that I use Postgres" does not auto-delete anything; the assistant identifies the candidate file(s) via `search_memory`, shows a plain-English preview, and asks for explicit confirmation. On confirmation the file is moved to `memory/.trash/<timestamp>-<basename>.md`, the manifest is updated, and the index is scheduled for incremental re-index.
- **CLI surface:** `allbert-cli memory forget <path-or-query>` performs the same flow non-interactively with a `--confirm` flag and refuses to proceed without it.
- The `.trash/` directory is retained for 30 days and can be restored by the operator; after that it is garbage-collected by a memory-maintenance job.

## Memory-curator skill

v0.5 ships a thin first-party skill at `skills/memory-curator/` that packages the review and optional extraction workflows.

- **Pure-prompt SKILL.md, no bundled scripts** — all curator actions use the kernel-owned memory tools (`list_staged_memory`, `promote_staged_memory`, `reject_staged_memory`, `search_memory`, `read_memory`, `write_memory`). No exec policy exposure beyond what the default skill surface already has.
- **Capabilities:** guided staged-entry review; duplicate detection assistance; batch promotion with operator confirmation; suggested edits for compaction of long-running daily notes into durable notes.
- **Optional LLM-based post-turn extraction:** the skill exposes a `curator.extract_from_turn` agent that the operator can invoke explicitly ("Claude, look at what we just covered and suggest things to remember"). This is where the optional LLM call lives — not in the kernel's turn loop. The extraction agent's LLM usage goes through the normal cost-tracking surface.
- **Security posture:** no network calls, no arbitrary file reads outside `~/.allbert/memory/`, no process execution. The skill ships as the reference example of a memory-aware skill that other community skills can fork.

We surveyed external memory-curation skills (openclaw `active-maintenance`, `agent-memory-ultimate`, `memory-hygiene`, MCP `memory-health-dashboard`) and declined to vendor any of them: all either assume vector/SQLite stores (incompatible with our markdown tier model), lack confirmable permissive licenses, or live in registries with repo-level malicious-content disclaimers. See [ADR 0048](../adr/0048-v0-5-ships-a-first-party-memory-curator-skill.md).

## Upgrading from v0.4 profiles

Operators running v0.4 profiles on first v0.5 boot should see a clean upgrade, not a rebuild.

- `memory/notes/`, `memory/staging/`, and `memory/index/` are created if absent.
- If legacy v0.4 durable buckets exist (`topics/`, `people/`, `projects/`, `decisions/`), first boot imports them into `notes/<bucket>/...` using copy-verify-rename semantics, writes a migration report to `memory/migrations/v0.5-import.json`, and then moves the original directories under `memory/.legacy-v04/`. v0.5 does not keep a dual-root runtime bridge for those directories after import.
- `memory/manifest.json` is generated from the post-import `notes/` tree and `daily/` content.
- Existing `MEMORY.md` is left untouched. If the file pre-dates v0.5's catalog conventions, the curator skill can offer a migration pass on demand.
- The index rebuilds from the generated manifest on first boot.
- No v0.4 config keys are removed in v0.5. New `[memory]` keys have safe defaults. `prefetch_enabled = true` remains the default even on upgrade; on an empty or newly imported corpus that simply yields zero results and negligible overhead rather than changing user-visible behavior.

`allbert-cli memory status` after upgrade must surface: profile version, indexed doc count, staged count, index age, last rebuild reason. This is the operator's one-call confirmation that upgrade succeeded.

## Seed templates

v0.5 ships the following seed templates under the installed binary's resource dir, used at bootstrap when files are missing:

- `MEMORY.md` — a short catalog skeleton with headings (`## Profile`, `## Projects`, `## Notes`) and a one-line "Allbert's kernel maintains the manifest; add durable notes under notes/ and Allbert will index them." footer.
- `notes/.keep` — empty file to preserve the directory in a fresh profile.
- `staging/.keep` — ditto.

These are the minimum needed for the retriever and manifest to operate on a brand-new profile without special-casing empty corpora.

## What belongs in memory

### Good durable candidates

- user preferences
- stable project facts
- environment facts
- working conventions
- durable decisions
- verified lessons learned
- recurring constraints

### Bad durable candidates

- raw transcripts
- speculative claims
- temporary scratch state
- large logs or code dumps
- one-off file paths
- content already represented in bootstrap identity files

## Operator surfaces

v0.5 should ship both prompt-native and CLI/operator memory surfaces.

### Prompt-native

- ask "what do you remember about X?"
- ask "remember that ..."
- inspect staged learnings
- approve or reject a staged learning

### CLI/operator

Minimum operator surface:

- `allbert-cli memory search <query> [--tier durable|staging] [--limit N] [--format text|json]`
- `allbert-cli memory staged list [--kind K] [--since DURATION] [--format text|json]`
- `allbert-cli memory staged show <id>`
- `allbert-cli memory promote <id> [--path PATH] [--summary TEXT]` — shows plain-English preview (title, provenance, tags, destination path) and requires explicit `y` confirmation; `--confirm` skips the prompt for scripted use.
- `allbert-cli memory reject <id> [--reason TEXT]`
- `allbert-cli memory forget <path-or-query> --confirm` — plain-English preview of the target file(s); refuses without `--confirm`.
- `allbert-cli memory rebuild-index [--force]` — waits on any in-flight rebuild via the advisory lock.
- `allbert-cli memory status` — output: profile version, indexed doc count, staged count (by kind), rejected count, index age, last rebuild reason and elapsed, ephemeral summary byte size for the active session (if any), and pending TTL expirations.

Durable promotion, rejection, and forgetting remain explicit operator actions, even when the assistant is the one proposing them. Promotion and forget previews are plain-English (title, provenance, destination), not raw markdown diffs — consistent with v0.4's skill-install preview posture (ADR 0033).

## Config deltas

v0.5 adds a `[memory]` config section (TOML) that is explicit about retrieval, staging, and budget behavior. All keys have defaults; a fresh profile runs without any `[memory]` block.

| Key | Default | Purpose |
| --- | --- | --- |
| `prefetch_enabled` | `true` | master switch for automatic prefetch |
| `prefetch_default_limit` | `5` | number of tier-1 results per prefetch pass |
| `refresh_after_external_evidence` | `true` | allow one refresh retrieval per turn |
| `max_refreshes_per_turn` | `1` | hard cap |
| `max_synopsis_bytes` | `8192` | total always-on synopsis budget |
| `max_memory_md_head_bytes` | `2048` | `MEMORY.md` head slice |
| `max_daily_head_bytes` | `2048` | today's daily note head slice |
| `max_daily_tail_bytes` | `1024` | optional yesterday tail |
| `max_ephemeral_summary_bytes` | `2048` | ephemeral session summary in synopsis |
| `max_prefetch_snippets` | `5` | prefetch snippets carried into root prompt |
| `max_prefetch_snippet_bytes` | `512` | per-snippet byte cap |
| `max_ephemeral_bytes` | `32768` | rolling cap on per-session ephemeral buffer |
| `max_staged_entries_per_turn` | `5` | per-turn cap on new stagings |
| `max_subagent_snippets` | `3` | sub-agent filtered-slice cap |
| `staged_entry_ttl_days` | `90` | auto-expire unreviewed staged entries |
| `staged_total_cap` | `500` | global staged-queue cap; new stagings above this fail with a hook event |
| `rejected_retention_days` | `30` | how long rejected entries stay in `.rejected/` |
| `trash_retention_days` | `30` | how long forgotten durable entries stay in `.trash/` |
| `index_auto_rebuild` | `true` | rebuild on schema/hash drift at boot |
| `default_search_limit` | `10` | `search_memory` default limit |
| `default_daily_recency_days` | `2` | how many recent days feed the synopsis |
| `surface_staged_on_turn_end` | `true` | append the one-line staged hint to assistant output |
| `prefetch_intent_overrides` | `{}` | per-intent prefetch on/off map |

Exact key names can still be tuned during implementation, but the behaviors and defaults above are committed before M1 starts.

## Milestones

### M0 — Spec freeze

Before any M1 code changes:

- flip ADRs 0040, 0041, 0042, 0043, 0044, 0045 from Proposed to Accepted
- write and accept ADRs 0046 (tantivy), 0047 (staging schema), 0048 (memory-curator skill)
- revise ADR 0003 to reference the curated-memory tier model without breaking the "no hidden transcript store" rule

### M1 — Memory substrate and bootstrap

Scope:

- commit `tantivy` and supporting crates (`pulldown-cmark`, `serde_yaml`, `fs2`, `tempfile`) to `Cargo.toml`
- finalize directory/file layout (`memory/`, `notes/`, `staging/`, `index/`, `manifest.json`, `meta.json`)
- add the `[memory]` config section with the full key set from [Config deltas](#config-deltas) and `allbert-cli config doctor` validation
- ship seed templates (`MEMORY.md`, `.keep` files) bundled with the binary
- implement bootstrap flow including advisory-lock index rebuild, manifest rebuild, and staged-entry TTL expiration
- implement `memory/manifest.json` writer/reader with atomic temp-then-rename semantics
- implement `allbert-cli memory status` and `allbert-cli memory rebuild-index`
- implement v0.4→v0.5 upgrade path (no manual steps required on first v0.5 boot)

Tests:

- unit: manifest round-trip, meta.json schema versioning, atomic write/rename, advisory lock held during rebuild
- integration: fresh-profile bootstrap seeds the expected layout; upgrading a canned v0.4 fixture rebuilds the manifest and index; staged TTL expiration moves entries to `.expired/`
- smoke: `memory status` after fresh boot, after upgrade boot, and after a forced rebuild prints legible output
- perf budget: cold-start rebuild under 1s for a 1 000-note fixture on a developer laptop; CI asserts under 3s on the hosted runner

### M2 — Retrieval engine and memory tools

Scope:

- implement the tantivy-backed retriever with fields `title`, `body`, `tags`, `tier`, `date`
- implement `search_memory`, `stage_memory`, `list_staged_memory`, `promote_staged_memory`, `reject_staged_memory`
- wire the memory tools into the existing tool surface and `allowed-tools` skill frontmatter
- implement staged-entry writer with the ADR 0047 schema, duplicate detection, and per-turn/total caps
- implement operator CLI: `memory search`, `memory staged list|show`, `memory promote`, `memory reject`, `memory forget`
- implement plain-English previews for promote and forget

Tests:

- unit: retriever ranking invariants (identity on repeat, monotonic exact-match score, stable tier filter); staged-entry dedup fingerprint
- property: randomized markdown corpus → index → query returns stable ordering; randomized stage/reject/promote sequences preserve manifest integrity
- integration: `stage → promote → search(durable) finds it`; `stage → reject → search(staging) excludes it, finds it in .rejected/` only via operator CLI; duplicate staging within the same turn is rejected with an explanatory hook event
- smoke: `allbert-cli memory search "foo"` against a populated fixture returns deterministic results
- perf budget: single-query retrieval p95 under 10ms for 10 000-doc corpus

### M3 — Turn-flow integration

Scope:

- implement the turn-flow contract including budget-aware synopsis composition and deterministic truncation
- implement bounded prefetch (intent-gated) and the one-pass refresh retrieval after external evidence
- lock the root-agent prompt assembly order with a golden-file test harness
- expose and wire all memory hooks (`BeforeMemoryPrefetch` through `AfterIndexRebuild`)
- implement file-watch-triggered incremental re-index for operator edits to `notes/`

Tests:

- unit: synopsis truncation order (yesterday tail → snippets → daily head → MEMORY head, never ephemeral summary); intent-to-prefetch decision table; refresh-gate logic
- integration: a turn with a memory-cueing prompt triggers prefetch once and a refresh once; a `chat`-intent turn without cues does not prefetch; the intent classifier runs without memory input
- golden-file: canned-LLM turns produce byte-stable root prompts for (a) empty memory, (b) populated memory with prefetch, (c) prefetch + refresh
- integration: operator edit of `notes/*.md` triggers an incremental re-index within 2s
- perf budget: prefetch p95 under 15ms on a 10 000-doc corpus; turn-setup overhead under 30ms total

### M4 — Ephemeral and multi-agent memory

Scope:

- implement session-scoped ephemeral memory with rolling byte cap and oldest-first eviction
- implement the default sub-agent filtering heuristic (`memory_hints` resolution)
- implement staged-entry agent attribution (root vs sub-agent vs job)
- wire job-session memory flow (fresh ephemeral per run, `session_name` opt-in sharing, `memory.prefetch: false` opt-out)

Tests:

- unit: ephemeral eviction preserves recency invariants; attribution frontmatter is populated correctly for each source kind
- integration: multi-session isolation (two concurrent sessions do not see each other's ephemeral); detach/reattach preserves ephemeral across channel reconnects; daemon restart clears ephemeral; sub-agent without `memory_hints` starts with zero retrieved memory; sub-agent with `memory_hints` gets the bounded slice; a job session accumulates staged entries with `kind: job_summary`
- smoke: scheduled job that stages an entry is visible in `memory staged list` with the correct attribution

### M5 — Staging, promotion, and curation skill

Scope:

- ship the first-party `memory-curator` skill (pure-prompt SKILL.md, no scripts)
- implement the `curator.extract_from_turn` agent entrypoint on the skill
- implement the turn-end staged-surfacing hint (`surface_staged_on_turn_end`)
- implement prompt-native review/promote/reject flows through the curator skill
- verify that promotion and forget previews match the plain-English posture of ADR 0033

Tests:

- integration: invoking `memory-curator` with staged entries produces a reviewable summary; batch promotion through the skill routes through `confirm-trust` for non-CLI callers; extract-from-turn agent's LLM usage appears in the session cost record
- smoke: a full conversational loop — user says "remember that we use Postgres" → agent calls `stage_memory` → turn-end hint surfaces → user says "review what's staged" → `memory-curator` lists and offers promote/reject

### M6 — Closeout and end-user usability

Scope:

- update `allbert-cli` help text for all new commands
- update REPL `/help` output and any short aliases so memory flows are discoverable from the terminal without reading the plan
- update onboarding docs (README, getting-started) to cover memory
- write a v0.4→v0.5 upgrade note for operators
- add end-to-end smokes for search, stage, promote, reject, forget, and multi-turn recall
- add end-to-end smokes for the memory-curator discovery path (`skills list`, prompt-native review, CLI review)
- confirm v0.5 remains end-user runnable for the technical source-based target user (still cargo-built, still single binary, still local-first)

Tests:

- end-to-end: scripted conversational memory flow matches documented behavior
- end-to-end: CLI-driven memory review flow matches documented behavior
- docs match shipped behavior (CI grep-gate on command names and config keys)
