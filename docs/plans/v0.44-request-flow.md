## M1 Locked Decisions

Each decision below is recorded in `v0.44-plan.md` §"M1 Locked Decisions"
with a one-paragraph rationale. These are chosen values, not open
defaults.

1. Plan/Build surface shape — pinnable panel + ephemeral surface over
   the v0.23/v0.26 workspace canvas; NOT a separate `/plan` route.
2. Workflow YAML expression grammar — closed function table + dotted-
   path references (`${inputs.x}`, `${steps.<id>.<field>}`,
   `${user.locale|timezone}`, `${workflow.id|version}`); AST-parsed at
   load.
3. Schema validation source-of-truth — derived from the current
   `Actions.Registry.modules/0` snapshot + `Step.kinds()` at validation
   time; not hand-maintained.
4. Workflow file location and id pattern —
   `<ALLBERT_HOME>/workflows/<workflow-id>.yaml`; id pattern
   `^[a-z0-9][a-z0-9_-]*$`; on-demand discovery; collisions fail
   closed.
5. Execution semantics — sequential array order plus per-step `if:`;
   no loops, no parallel/fan-out, no sub-workflow includes, no `on:`
   triggers, no `env:` block, no retry policies in v0.44; subagent
   delegation uses v0.24's `:delegate_agent` step kind unchanged.
6. Confirmation semantics — registered-action permission floor is
   authoritative; YAML `confirm: true` may only upgrade a step (never
   downgrade); explicit `kind: ask_user` is the plan-level checkpoint;
   plan-start gate is `:workflow_run_start` with
   `:needs_confirmation` floor.

## Main User/Operator Flows

- Operator asks Allbert to plan a multi-step task ("plan: ..." prefix);
  the intent layer routes to a one-off synthesized plan-preview path.
- Operator submits a workflow YAML file authored under
  `<ALLBERT_HOME>/workflows/<id>.yaml` ("run workflow <id>"); the
  loader validates, the expander resolves inputs and references, and
  the same preview/approve/execute path runs.
- Operator reviews the Plan Preview Contract packet in the workspace
  Plan/Build Preview panel: per-step ordinal, kind, action name,
  params summary, permission, safety floor, resources needed, estimated
  cost, confidence tier, confirmations required, subagent target,
  failure blast radius.
- Operator expands the panel to fullscreen, edits inputs and per-step
  `confirm:` upgrades, reorders or removes existing steps, saves; the
  preview recomputes deterministically through the registered
  `preview_plan` action. Editor overrides are bounded to existing step
  ids and unknown step ids fail closed.
- Operator approves the plan-start gate (`:workflow_run_start`
  `:needs_confirmation`); the Objective Runtime frames the objective
  and the engine begins.
- Per-step events render in the workspace RunProgress panel and the
  objective timeline; subagent delegation events render inline under
  the parent step (resolves the "invisible subagents" UX lesson).
- Per-step confirmations apply at execution time per each action's
  registered floor; YAML cannot bypass them.
- Operator may cancel mid-run; the in-flight step completes
  cooperatively, no new step starts, the cancellation reason is
  durable.

## Intent Phrase Corpus

Source of truth in `v0.44-plan.md` §"Planned Interfaces" Intent corpus.
M4 implements one routing test per pattern; M5's `release.v044` gate
covers each pattern as a step.

| Pattern | Routes to |
|---|---|
| `plan: <natural-language>` | `preview_plan` over a synthesized one-off workflow |
| `plan <natural-language>` (no colon) | same |
| `run workflow <id>` | `expand_workflow` + `preview_plan` + `start_plan_run` |
| `run <id>` where `<id>` matches `workflows.id_pattern` and the workflow file exists | same |
| `cancel plan <objective_id>` | `cancel_plan_run` |
| `show plan <objective_id>` | `show_objective` (v0.24 read-only) |
| `list workflows` | `list_workflows` |
| `list plans` | `list_plan_runs` |

Patterns match case-insensitively; `<id>` is the rightmost bare token
matching `workflows.id_pattern`. The short `run <id>` form is
Plan/Build-only when `<ALLBERT_HOME>/workflows/<id>.yaml` exists;
otherwise it falls through to the normal intent ranker. The explicit
`run workflow <id>` form always stays in Plan/Build and returns a
refusal advisory when the workflow file is missing. `<objective_id>` is
the v0.24 objective id shape (`obj_` + UUID). The corpus is M4-locked —
additions or edits require an explicit re-decision in this flow and in
the plan.

## Workflow-Vs-Single-Action Routing Predicate

Source of truth in `v0.44-plan.md` §"Planned Interfaces". M4 implements
one test per row.

| Condition | Route |
|---|---|
| Operator explicitly uses `plan:` / `plan ` / `run workflow `, or `run <id>` with an existing workflow file | Plan/Build path |
| Intent decomposes into ≥ 2 dependent steps and ≥ 1 step has `:needs_confirmation` | Plan/Build path |
| Intent matches a single registered action with no dependencies | Single-action path |
| Operator names a missing workflow through explicit `run workflow <id>` | Refusal advisory with `list_workflows` suggestion |

The predicate is M1-locked. No edits without an explicit re-decision
recorded in this flow and in the plan.

## Data Flow

Plan-Build requests flow through `Actions.Runner.run/3`:

1. Intent layer (v0.11/v0.33 descriptors) proposes one of the
   Plan-Build actions; the M1-locked predicate in `v0.44-plan.md`
   §"Workflow-Vs-Single-Action Routing Predicate" picks Plan-Build
   vs the single-action path. M4 implements one routing test per
   predicate row; drift between table and implementation fails the
   test.
2. **Synthesized plan path (natural language).** `preview_plan` builds a
   transient in-memory workflow shape from deterministic intent/objective
   decomposition. It validates that shape against the same v1 schema,
   expands into step attrs, and returns a normalized advisory response
   with the Plan Preview Contract packet under `output_data.preview`.
   v0.44 writes no LLM-authored YAML file.
3. **Authored workflow path (file).** `list_workflows` /
   `inspect_workflow` / `expand_workflow` operate on
   `<ALLBERT_HOME>/workflows/<id>.yaml`. The loader bounds the read
   to `workflows.max_yaml_bytes_per_file`, parses through
   `:yaml_elixir` with string-key output and no atom creation, and hands
   the raw map to the validator.
4. The validator first runs Allbert-specific fail-closed checks for
   caps, declared inputs, step kinds, registered actions, and the
   `${...}` closed grammar so structured diagnostics remain stable.
   It then runs JSON Schema (Draft 2020-12, assembled from the current
   `Actions.Registry.modules/0` snapshot + `Step.kinds()`) with
   `additionalProperties: false` at every level. Cycles, forward
   refs, unknown identifiers, dynamic action names,
   `${secrets.x}`, and `${env.x}` reject with structured
   `error_category` diagnostics and JSON-Pointer paths.
5. The expander resolves inputs (type-checked against declared
   `inputs:`), substitutes references in `params:` values, and
   produces an ordered list of `objective_steps.attrs` maps.
6. `preview_plan` emits the Plan Preview Contract packet
   (advisory-only; ADR 0021 §4 applied). Each per-step entry includes
   permission, safety floor, resources needed, estimated cost,
   confidence tier, confirmations required, subagent target, and
   failure blast radius.
7. `start_plan_run` first call emits a normalized
   `%{status: :needs_confirmation, permission: :workflow_run_start}`
   response with the Plan Preview Contract packet attached.
8. On approval-resume, `start_plan_run` calls `Objectives.frame/2`
   to create the objective row (with `source_intent:
   "workflow:<id>:<version>"`), then persists the already-expanded
   proposed step rows and proposal events through the public Objective
   facade. M2 does not register a proposer module. If a later milestone
   introduces a proposer adapter, v0.44 registers one core Plan/Build
   proposer and stores per-run expanded step state in the
   objective/proposer hint; it does not dynamically register one
   proposer module per workflow. The v0.24 engine takes over after the
   approved plan has durable objective and step rows by selecting the
   next eligible `provider: "plan_build"` proposed step in persisted
   workflow order and executing it through the existing step lifecycle.
9. Each step's lifecycle events (`allbert.objective.step.proposed`,
   `.selected`, `.blocked` if confirmation needed, `.completed`,
   `.failed`) post into the SignalBus. The workspace RunProgress
   panel subscribes via SignalBridge and renders the timeline.
10. For `kind: :delegate_agent` steps, the registered
    `delegate_agent` action dispatches through
    `AllbertAssist.Objectives.AgentRegistry`; child agent events
    render inline under the parent step (the
    `plan.subagent.delegation_visibility: :expanded_inline`
    invariant).
11. For `kind: :ask_user` steps, the runtime emits a per-step
    confirmation via Approval Handoff; the operator's response is
    bound into `${steps.<id>.value}` and, if declared, the
    `${steps.<id>.<save_as>}` alias.
12. `cancel_plan_run` calls `Objectives.cancel/3` with a required
    `reason`. The in-flight step completes (v0.24 cooperative
    cancellation), no new step starts, the cancellation reason is
    durable in `objective_events`.
13. Trace/audit records carry the plan_id, workflow_id, step_id,
    redacted params summary, success/failure, confirmation id;
    `AllbertAssist.Security.Redactor` scrubs secret-shaped
    operator-supplied `inputs:` and `params:` values, raw
    references that resolve to secret-shaped strings, and full
    operator-supplied free text. Raw YAML files live under
    `<ALLBERT_HOME>/workflows/`; trace files under
    `<ALLBERT_HOME>/memory/traces/` never carry raw secret values.

### Runtime Semantics Remediation

The implementation audit after the first v0.44 pass found that the
initial gate proved preview + proposed-step persistence, but not every
runtime semantic described above. M6-M10 close that gap:

- **M6 runtime handoff:** approval starts the existing Objective Runtime
  path for the first eligible workflow step; no step runs before approval.
- **M7 per-step semantics:** runtime enforces `if:`, `on_error:`, and
  YAML `confirm: true` upgrade-only behavior from
  `objective_steps.resource_access`.
- **M8 step-output references:** `${steps.<id>.<field>}` resolves at
  runtime from earlier completed workflow steps and declared `save_as:`
  aliases.
- **M9 evidence repair:** `release.v044` proves execution semantics, not
  only preview/persistence.
- **M10 closeout:** release-record docs and version metadata move to
  `0.44.0` only after M6-M9 evidence is green.

## Authority Invariants

- **YAML is not authority.** Workflow files are inert data. The
  runtime never executes them; expansion produces step attrs only.
- **`workflow_id`, `plan_id`, `step_id` are not authority.** They
  identify; they do not authorize.
- **Plan preview is advisory.** Preview packets are descriptive
  metadata per ADR 0021 §4; no permission grant flows from preview.
- **Plan-start gate is the only authority transition from preview to
  run.** `:workflow_run_start` `:needs_confirmation` floor;
  approval-resume causes the objective to be framed and steps to be
  proposed/inserted; no effectful step runs before this gate.
- **Per-step confirmations are enforced at execution time per the
  action's registered floor.** YAML `confirm: true` may only upgrade
  a step's confirmation requirement; never downgrade. A
  `:needs_confirmation` action stays `:needs_confirmation` regardless
  of YAML.
- **Closed expression grammar.** `${...}` is AST-parsed against a
  fixed grammar. No `eval`. No `${secrets.x}`. No `${env.x}`. No
  dynamic action-name resolution.
- **Schema validation is fail-closed.** Unknown keys, type
  mismatches, missing required inputs, unknown actions, unknown
  delegate-agent ids, cycles, forward refs, and oversized payloads
  reject with JSON-Pointer-path-bearing diagnostics. The structured
  `error_category` enables operator self-service fix.
- **Subagent delegation reuses v0.24's `:delegate_agent` step kind +
  Security Central + Approval Handoff.** YAML never opens a parallel
  delegation path; the registered `delegate_agent` action is the
  only crossing.
- **Cooperative cancellation only.** `cancel_plan_run` per v0.24
  rule; in-flight step completes, no new step starts.
- **Operator confirmation is required for the plan-start gate**
  regardless of any remembered grant on a child action; a remembered
  navigation grant from v0.43 (for example) does not bypass the
  plan-start gate.
- **No new memory promotion.** `kind: reflect` proposes candidates
  only; v0.21 review surface is the only writer.
- **No new objective store.** All plan runs are existing
  `objectives` rows with `source_intent: "workflow:<id>:<version>"`.

## Channel Rendering Forward Pin

v0.44 renders through existing CLI, LiveView, Telegram, and email
affordances. The v0.49 ADR 0016 amendment defines
`{:list, :button, :typed_command, :link}`; v0.44 confirmations are
expressible in that vocabulary but do not require the v0.49 plugin
primitive declaration field.

| Surface | Plan-start gate | `kind: ask_user` | Cancel control | Plan preview review |
|---|---|---|---|---|
| CLI / email | `:typed_command` | `:typed_command` or `:list` | `:typed_command` | `:link` to `/objectives/<id>` |
| LiveView / Telegram | `:button` | `:button` or `:list` | `:button` | `:link` and inline panel |
| Discord / Slack (when v0.49 ships) | `:button` | `:button` or `:list` | `:button` | `:link` |

v0.44 does not depend on v0.49 to ship; the table is a forward-pin so
the v0.49 amendment knows the v0.44 affordance set without
re-litigation.
