## Main User/Operator Flows

- Operator asks Allbert to plan a multi-step task.
- Operator reviews objective steps, resources, confirmations, and failure
  preview.
- Operator edits, approves, or cancels the plan.
- Operator submits workflow YAML that expands to objective steps.
- Background objective progress renders in workspace and origin channel.

## Data Flow

Plan input becomes an objective draft, candidate steps are validated, workflow
YAML is parsed as declarative data, approved steps execute through
`Actions.Runner.run/3`, and objective events drive status rendering.

## Authority Invariants

Plans and YAML do not execute. Objective ids are not authority. Every step keeps
normal action permission and confirmation behavior. Unknown YAML keys fail
closed.

## Operator Validation

Status: planned. Commands are illustrative until the milestone lands; verify and
replace with observed output at closeout.

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v044.XXXXXX)"
export ALLBERT_TRACE_ENABLED=true
mix allbert.ask --trace "plan: create a project folder, fetch a template, and summarize it"
PORT=4044 mix phx.server   # open the /workspace Plan/Build destination to preview
```

Expected behavior:

- the plan preview lists steps, required resources, confirmation points, and a
  failure preview before anything runs.
- editing or cancelling the plan is honored; no step runs without its own
  confirmation; one approved step runs and one denied step does not.
- workflow YAML submitted by the operator expands into the same previewable
  objective steps; unknown YAML keys fail closed.
- objective progress renders in the workspace and the origin channel.
