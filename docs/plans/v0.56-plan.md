# Allbert v0.56 Channel Parity + TUI/Terminal Channel

Request flow: `docs/plans/v0.56-request-flow.md`

Status: planned — **STUB** created in the 2026-06-09 roadmap restructure (Phase A).
Full plan authored in Phase B (research R4). Tracking:
`docs/plans/_wip-restructure-v0.51-v0.57.md`.

ADRs (to consume/amend in Phase B): `docs/adr/0016-*` (channel adapter boundary
+ identity mapping + approval primitives).

## Purpose

Two related goals:
1. **Channel parity** — make the cross-channel capability matrix explicit (which
   approval primitives `{list, button, typed_command, link}`, attachments,
   streaming, identity mapping, and approvals each channel supports), and close
   gaps so behavior is predictable across web, Telegram, email, Discord, Slack,
   and the mobile channels.
2. **A proper TUI/terminal channel** — a real persistent terminal channel under
   the ADR 0016 channel contract (identity, event dedupe, runtime submission,
   Approval Handoff, redaction), rather than relying on the one-shot
   `mix allbert.ask` task.

## Scope (to be detailed in Phase B / R4)
- Build/record the channel capability/parity matrix; decide whether CLI is
  promoted to a first-class channel.
- Research the Elixir TUI substrate (e.g. `ratatouille` vs `owl` vs raw) and how
  a TUI plugs into the channel contract; pick and justify.
- ADR 0016 amendment for the parity matrix + the TUI channel.
- Acceptance + eval rows + `release.v056`.

## Non-Goals (preliminary)
- The TUI channel acquires no authority beyond any other channel (channels are
  delivery adapters; they do not own intent, actions, or Security Central).
