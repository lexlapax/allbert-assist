# v0.6 — Foundation hardening

Status: Shipped

## Context

v0.5 closed on 2026-04-20 with curated memory, a tantivy retrieval path, and the shipped `memory-curator` skill. A retrospective across v0.1–v0.5 the same day surfaced five gaps that do not change what Allbert *can* do, but change how reliably the v0.5 experience lands:

- **The staged-memory notice is too generic.** v0.5 already surfaces a one-line turn-end hint, but it does not always tell an operator enough to review efficiently.
- **Conversation dies on daemon restart.** Sessions survive client reattach, but a daemon restart wipes everything. For a personal assistant whose selling point is "remembers you," this is too fragile.
- **There is no hard daily cost cap yet.** A looping job or a runaway sub-agent has no kernel-level daily spend gate.
- **The "gets smarter over time" loop is not safely defaultable.** `memory-compile` still assumes direct durable writes in its current template, which conflicts with v0.5's staging-first policy, and recurring jobs remain opt-in per profile.
- **Markdown reconciliation lacks operator-visible verification.** The daemon already reconciles note edits while it is running, but operators cannot easily verify daemon-off edits or script against reconciliation state.

v0.6 is a **protect-what-we-shipped** release. No new product capability, no new channel, no new memory tier. Just hardening the v0.5 foundation before v0.7 expands the channel surface onto it.

## Goal

Close the five gaps that prevent v0.5 from delivering its stated experience reliably, cover the daemon + kernel + memory path with a first end-to-end regression test, and finish a doc-alignment pass so `docs/vision.md` and `README.md` match the shipped surface.

## Milestones

### M1 — Turn-end staged-memory notice polish

v0.5 already renders a short turn-end hint when a turn stages memory. v0.6 keeps that shipped behaviour and tightens it into a more review-friendly operator surface rather than inventing a second feature.

- Config key remains `memory.surface_staged_on_turn_end = true|false` (default true on fresh profiles, unchanged on upgrade).
- Rendering may become richer for up to a small number of staged entries, but stays a one-line kernel-rendered pointer rather than an inline review dialog.
- Budget: combined id + summary stays tightly bounded so the notice remains cheap across channels.
- Channel-adaptive rendering remains a v0.7 concern; v0.6 focuses on the shipped REPL and CLI operator path.
- Covered by ADR 0050.

### M2 — Session durability across daemon restart

Each active session persists an append-only markdown journal at `~/.allbert/sessions/<session-id>/turns.md` plus `meta.json`. On daemon restart, `allbert-cli daemon resume [--session <id>]` rehydrates the session.

- Tool outputs in the journal are truncated to `memory.max_journal_tool_output_bytes` (default 4 KiB); full outputs remain in the trace log.
- Sessions older than `daemon.session_max_age_days` (default 30) move to `sessions/.archive/`.
- Sessions interrupted mid-tool resume to the last completed turn boundary; incomplete invocations are discarded with an audit entry.
- Journals are **working state, not durable memory**. Staging/promotion remains the only path into `notes/`.
- Covered by ADR 0049.

### M3 — Daily cost cap enforcement

`limits.daily_usd_cap` is introduced in v0.6 as a hard gate at turn start. Over-cap turns refuse with an operator-facing message and a `/cost --override <reason>` escape hatch for a single turn.

- CostHook reads today's aggregate from `costs.jsonl` keyed by UTC day (cached 60s).
- Enforcement is kernel-level; applies uniformly to REPL, channel-originated, and job-originated turns.
- Jobs refuse silently, log `cap-reached` as the job's last outcome, and surface it in `jobs status`.
- Cap of `null` (default until the operator or setup flow sets it) disables enforcement. The wizard offers a sensible starting cap.
- Covered by ADR 0051.

### M4 — Memory verification and operator visibility

The daemon already reconciles markdown edits while it is running. v0.6 adds an explicit verification surface so operators can see and script against that state, including daemon-off cases.

- `memory status` surfaces manifest/index health, last reconcile time, and whether daemon-driven reconciliation is currently active.
- Add an explicit verify path (`memory verify` or equivalent) that performs the same checksum comparison when the daemon is offline or when an operator wants a scriptable assertion.
- Exit code on the verification path is non-zero when unresolved mismatch remains.
- If a bundled verification job remains at all, it exists only as a narrow daemon-off or audit aid and is not the primary reconciliation mechanism.

### M5 — Policy-correct maintenance loops and defaults

Normalize `memory-compile` so it stages candidates and emits a report, rather than writing durable notes directly. Only after that policy correction does v0.6 revisit fresh-profile defaults for maintenance jobs.

- `memory-compile` uses `stage_memory` plus a durable report artifact, not direct durable writes.
- Fresh-profile defaults apply only when the operator opts into recurring jobs during setup; upgrade profiles preserve their existing job state.
- `trace-triage` and `system-health-check` remain disabled unless later evidence shows they are safe defaults.
- `daily-brief` and `weekly-review` stay disabled: they require operator-confirmed scheduling preferences.

### M6 — End-to-end regression test

Add an integration test under `crates/allbert-daemon/tests/` that covers the daemon + kernel + memory path together. Steps:

1. Boot a fresh profile in a test home.
2. Start a real session and create both ephemeral working state and a staged memory entry.
3. Promote the staged entry.
4. Restart the daemon.
5. Resume the same session.
6. Assert the resumed session still has the expected journal-backed working state from M2 — specifically that the resumed system context still contains the previously established ephemeral working buffer from step 2 — not just durable memory.
7. Run a retrieval turn and assert the promoted durable memory is still found.

This is the first test exercising daemon restart + session journal + durable memory together. It locks the v0.6 behaviour in before v0.7 reshapes the channel surface.

### M7 — Doc alignment pass

- **`docs/vision.md`**: keep the post-v0.5 vision aligned with the v0.6 hardening scope and the new sequencing into v0.7 and v0.8.
- **`README.md`**: update the release-facing sections to describe what v0.6 adds, not to restate v0.5. Document `/cost --override`, `daemon resume`, and the memory verification surface.
- **`docs/onboarding-and-operations.md`**: operator walkthrough for daemon resume, turn-end notice toggle, cost-cap override, and memory verification.

## ADRs

- **ADR 0049** — Session durability is a markdown journal.
- **ADR 0050** — Turn-end staged-memory notice is the shipped kernel-rendered suffix, kept under the existing config key and optionally refined.
- **ADR 0051** — Daily cost cap is a hard gate at turn boundary.
- Memory verification strategy may remain an in-plan decision rather than a standalone ADR; escalate only if the checksum-versus-status split is contested during implementation.

## Dependencies

- v0.5 curated memory — M1 and M4 extend its surface; M3 wraps its staging pipeline in cost accountability.
- v0.2 daemon lifecycle — M2 extends the restart story; M6 tests it end-to-end.
- v0.2 job manager — M5 revisits maintenance-template defaults after policy correction.

## Out of scope for v0.6

- New channels (v0.7).
- New memory tiers or retrieval strategies (v0.5 shape stays).
- Multi-device memory sync and cross-channel continuity (scheduled for v0.8 rather than v0.6).
- Sub-agent depth changes (v0.7).
- Tool-registry refactor to absorb memory/skill kernel-native ops (v0.7).
- A filesystem-event watcher for memory drift (M4 is checksum-based on purpose).

## Remaining planning questions

- Should session journals redact tool outputs by default or preserve them for replay fidelity? Default 4 KiB truncation is a compromise; operators who want full fidelity can raise the limit.
- When a session exceeds retention, should promotion-relevant context be auto-staged before archive, or silently dropped? Current default: silently dropped; staging would require a "salience scorer" that v0.6 explicitly does not introduce.
- Is `/cost --override` REPL-only, or should it extend to channel-originated refusals? v0.6 ships REPL-only; channels negotiate their own override UX in v0.7.
