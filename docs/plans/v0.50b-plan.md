# Allbert v0.50b MCP Server Mode

Request flow: `docs/plans/v0.50b-request-flow.md`
ADR: `docs/adr/0044-public-protocol-exposure.md` (rewritten to MCP-server-only
scope)

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`, post-v0.37 planning pass.

## Purpose

Expose Allbert to external MCP-client agents (Claude Desktop, Cursor, ChatGPT
MCP, JetBrains MCP, other agent runtimes) as an MCP server. Symmetric to the
v0.40 MCP client work. Allbert exposes **registered actions as MCP tools** and
**app memory namespaces as MCP resources**.

This is the **only** public protocol surface in the 1.0 arc. OpenAI-compatible
HTTP API, ACP server mode, and public AG-UI/A2UI bridge are parked post-1.0
(see `future-features.md` "Public Protocol Interop"). The decision rationale
lives in ADR 0044.

## Operator-Visible Win

An operator can configure an external MCP client (e.g., Claude Desktop) to
call Allbert as an MCP server. The operator explicitly enables the tools and
memory resources the external client may see. Effectful calls trigger normal
Approval Handoff through the operator's workspace or origin channel; the
external client never approves its own confirmations.

## Planned Interfaces

- Settings namespace `mcp_server.*`: enabled flag (default off), transport
  (`stdio` default; optional `http_streamable`), per-tool include/exclude
  filters (default empty), per-namespace memory-resource include filters
  (default empty), HTTP transport token refs into Settings Secrets.
- Registered MCP server transport adapters under
  `./plugins/allbert.mcp_server/` (or under the `allbert_assist_web` app,
  depending on supervision placement decided in M1).
- New permission class `:mcp_server_call_inbound` covering inbound MCP
  tool/resource requests; safety floor `:needs_confirmation` for any tool
  whose registered action has a `:needs_confirmation` floor.
- Workspace status panel surfaced through the existing channel/protocol list
  pattern so operators can see which external clients are connected.

## Tool And Resource Exposure Rules

- A registered Allbert action becomes an MCP tool only when explicitly listed
  in `mcp_server.tools_enabled`. The default list is empty.
- An app memory namespace becomes an MCP resource only when explicitly listed
  in `mcp_server.memory_namespaces_enabled`. The default list is empty.
- Tool descriptions, parameter schemas, and resource summaries are bounded
  and redacted before exposure.
- Settings, secrets, raw runtime signals, raw trace bodies, confirmation
  storage internals, plugin internals, and registry internals are **not**
  exposable. The MCP server transport rejects any attempt to surface them as
  a configuration error.

## Authority Rules

- Inbound MCP requests **never** receive more authority than a local
  workspace user.
- External MCP clients **cannot approve their own confirmations**. A
  confirmation-needing tool returns a `:confirmation_pending` response with
  the confirmation id; the operator approves through their usual surface
  (workspace, Telegram, Discord, etc.).
- All effectful calls route through `Actions.Runner.run/3`, Security Central,
  Resource Access, confirmations, traces, and audits — same code path as
  workspace user actions.
- HTTP transport tokens are encrypted at rest in Settings Secrets and never
  logged or returned in responses.
- Inbound prompt payloads are redacted before trace recording per existing
  redaction policy.

## Security Posture

MCP server mode opens a new external-call surface. Security evals must cover:

- Cross-client identity confusion (client A sees client B's session state).
- Prompt injection through MCP request payloads attempting to escalate to
  unconfigured tools.
- External-client self-approval attempts (must be denied).
- Resource-scope leakage across enabled and disabled memory namespaces.
- HTTP transport token replay, redirect handling, and rate-limit bypass.
- Settings/secrets/audit-store exposure attempts.

## Non-Goals

- No OpenAI-compatible local HTTP API.
- No ACP server mode.
- No public AG-UI/A2UI bridge.
- No hosted multi-user authorization.
- No external-client self-approval of confirmations.
- No exposure of settings, secrets, raw signals, raw trace bodies, or
  confirmation storage.
- No auto-enablement of tools or resources from MCP-client request payloads.

## Definition Of Done

v0.50b is done when:

- at least one external MCP client (Claude Desktop or Cursor) can configure
  Allbert as an MCP server in a disposable Allbert Home;
- the operator can explicitly enable specific tools and memory namespaces;
- a confirmation-needing tool call returns `:confirmation_pending` and the
  operator can approve through their workspace;
- security evals for cross-client confusion, self-approval denial, and
  internal-state exposure pass;
- the v0.51 export/import sweep round-trips MCP server settings without
  exposing secrets.
