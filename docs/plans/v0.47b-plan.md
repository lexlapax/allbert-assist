## Purpose

v0.47b is the **handoff** half of operator-supervised self-improvement. On
top of the v0.47 discovery substrate (trace index, generalized suggestion
surface, unified reviewed-draft store), it adds the draft kinds that hand off
to the already-shipped creation, sandbox, gate, and integration paths:

1. **Template-backed drafts** — suggestions that route into the v0.38
   templated-creation path (`Templates.Registry` / `create_from_template`)
   for plugin/app/tool/flow/objective scaffolds.
2. **Marketplace-backed suggestion drafts** — suggestions informed by v0.45
   `Marketplace.list_entries/1` metadata (descriptive only).
3. **Inert delegate-plugin draft requests** — a draft request to scaffold a
   v0.46-style delegate-agent plugin; inert, never an operator no-code agent.
4. **Dynamic capability-gap drafts** — suggestions that record a capability
   gap and route into the v0.37 `DynamicPlugins.request_draft/2` +
   sandbox/gate/loader path.
5. **Objective drafts** — suggestions that turn repeated multi-step shapes
   into inert objective drafts in the unified store.

v0.47b adds **no new authority** and **no new trust tier**: it stays inside
the v0.47 self-improvement tier (ADR 0045). Code-bearing drafts must pass the
existing v0.36 sandbox + v0.37 trusted-validation/gate before any live
integration confirmation; marketplace metadata remains descriptive; delegate-
plugin drafts are inert; operator no-code delegate-agent authoring stays
parked.

## M1 Locked Decisions

The first milestone preserves these four decisions.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Draft home | **The v0.47 unified draft store.** Every v0.47b draft kind is a record in the same store, with `kind` ∈ {`template_backed`, `marketplace_backed`, `delegate_plugin_request`, `capability_gap`, `objective`}. Code-bearing kinds (`capability_gap`, template-backed code) use the v0.37 `sandbox_compiled`/`gate_passed`/`integrated` tiers natively; non-code kinds (`marketplace_backed`, `delegate_plugin_request`, `objective`) use the v0.47 `draft → {discarded | promoted}` path. | A separate store would fragment review/audit. The v0.47 store already models the sandbox/gate tiers code-bearing kinds need. |
| 2 | Code-bearing gate | **Existing v0.36/v0.37 path, unchanged.** Capability-gap and template-backed code drafts route through `DynamicPlugins.request_draft/2` → `Sandbox.run_gate/2` → `Loader.integrate/2`; no live authority before gate + confirmation. | Inventing a parallel gate would bypass `TrustedValidator` and the audited loader. Reuse keeps one trusted path. |
| 3 | Marketplace + delegate inertness | **Descriptive metadata and inert requests only.** Marketplace metadata informs suggestions but grants nothing; a delegate-plugin draft scaffolds an inert plugin request, never a live agent or operator no-code authoring. | Letting marketplace metadata or a delegate draft confer authority would reopen the v0.45/v0.46 trust posture the releases deliberately bounded. |
| 4 | Objective drafts | **Declarative inputs only.** An objective draft is an inert objective definition in the store; promotion frames a real objective through the v0.24 runtime under confirmation. | Auto-framing objectives from patterns would let repetition execute work. Drafts keep framing operator-gated. |

Rationales:

- **Draft home.** Reusing the v0.47 unified store means v0.47b adds *kinds*
  and *promotion targets*, not a new mechanism; the store's tier lifecycle
  already covers the sandbox/gate states code-bearing kinds need.
- **Code-bearing gate.** The v0.37 loader, trusted validator, and v0.36 gate
  runner are the audited path to live authority. v0.47b suggestions feed
  their existing entry points; nothing integrates without gate evidence plus
  operator confirmation.
- **Marketplace + delegate inertness.** v0.45 made marketplace metadata
  descriptive; v0.46 made the delegate boundary metadata-checked. v0.47b
  preserves both: a marketplace-backed suggestion is a hint, and a
  delegate-plugin draft is an inert scaffold request routed through v0.38
  templates, not a no-code agent author.
- **Objective drafts.** A drafted objective is data; framing and running it
  remains a confirmed v0.24 action.

## Operator-Visible Win

On top of v0.47's local drafts, reviewed self-improvement suggestions can hand
off to the heavier creation paths — "draft a plugin from this template," "this
capability gap could be filled, run it through the sandbox," "this marketplace
template matches your pattern" — each inert until reviewed, gated where
code-bearing, and confirmed. In `0.47.1`, these handoff kinds are supported
suggestion/draft rows; the v0.47 `discover_patterns` action does not
automatically infer them from memory-review or objective-event diagnostics.

First-time walkthrough:

1. Operator (self-improvement enabled) reviews suggestions and sees a
   capability-gap suggestion: "you repeatedly needed an X integration."
2. Accepting it creates an inert capability-gap draft in the unified store.
3. Promoting the reviewed draft with `promote_capability_gap_draft` requests a
   v0.37 dynamic draft. Integration confirmation is not created until that
   dynamic draft has passed the sandbox/gate path.
4. The operator confirms live integration; the v0.37 loader trusted-validates,
   compiles, and registers the new action. Rollback remains available.
5. A marketplace-backed or template-backed suggestion instead routes into the
   v0.38 templated-creation path, producing an inert scaffold draft.
6. A delegate-plugin draft request scaffolds an inert v0.46-style plugin; it
   is never enabled by the suggestion.

## Security Posture

- Marketplace metadata is descriptive only; it grants nothing.
- Delegate-plugin drafts are inert scaffold requests; no live agent, no
  operator no-code authoring.
- Code-bearing drafts (capability-gap, template-backed code) must pass v0.36
  sandbox + v0.37 trusted validation/gate before any live-integration
  confirmation; the v0.37 loader, not v0.47b, performs integration.
- Objective drafts are declarative; framing/running stays a confirmed v0.24
  action.
- Inert handoffs require normal registered actions, Security Central, trace, and
  audit. Live promotion/integration, publish, rollback, marketplace submission,
  and objective framing each require durable confirmation in the existing target
  action path.
- No new permission class, operation class, URI scheme, or trust tier; v0.47b
  stays inside the v0.47 self-improvement tier.

## Non-Goals

- No autonomous skill/plugin creation; no auto-enable, auto-publish, package
  install, or remote install from trace patterns.
- No operator no-code delegate-agent authoring (stays parked).
- No new sandbox/gate/loader; v0.47b reuses the v0.36/v0.37 path unchanged.
- No marketplace submission/publishing automation; submission stays a
  confirmed operator action.
- No model distillation, learned system-memory authority, or unsupervised
  self-recompilation.
- No new trust tier; the self-improvement tier (ADR 0045) is unchanged.
