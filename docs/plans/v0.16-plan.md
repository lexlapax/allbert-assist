# v0.16 Plan: Additional Channels

Status: planned.

Formerly the "Additional Channels" future plan. v0.16 is the first concrete
non-CLI/non-LiveView channel milestone in the unified v0.xx sequence.

## Summary

v0.16 adds the channel adapter substrate and proves it with Telegram as the
first remote channel. A channel adapter translates external input into
`AllbertAssist.Runtime.submit_user_input/1`, renders runtime responses back to
the provider, and records enough durable event metadata to dedupe, trace, and
audit the exchange.

This version must not create a second runtime, a second confirmation system, or
a second security policy. Channels are delivery surfaces. All intent, action,
memory, resource access, confirmation, and job behavior continues to flow
through the existing runtime and registered Jido actions.

## Expected Inputs

- v0.11 has shipped execution-aware intent, Approval Handoff, and Resource
  Access Security Posture.
- v0.12 has shipped local string `user_id`, durable `Thread`/`Message`
  conversation history, and default thread selection.
- v0.14 has shipped session scratchpad and active app context.
- v0.15 has shipped the minimal app registration contract and action
  `app_id` tagging.
- ADR 0004, ADR 0007, ADR 0008, ADR 0012, ADR 0014, and ADR 0015 remain
  binding.
- ADR 0016, Channel Adapter Boundary and Identity Mapping, is proposed before
  implementation begins and accepted before release.

## Core Decisions

- Telegram is the only new provider implemented in v0.16. The reusable channel
  abstractions should be provider-neutral, but the implementation should be
  driven by one real adapter instead of a partial matrix of channel sketches.
- External identities never implicitly become local users. Telegram user IDs
  are mapped to local string `user_id` values through explicit Settings Central
  configuration.
- Channel transports may talk only to their configured provider API for
  receiving updates and delivering responses. A Telegram adapter is not a new
  general-purpose HTTP client, browser, crawler, document extractor, skill
  importer, or external service bridge.
- Channel approval UX reuses v0.07 confirmation actions and v0.11 Approval
  Handoff data. A button or callback from Telegram resolves a durable
  confirmation; it does not grant new powers by itself.
- Channel event records store full execution-routing fields needed for dedupe
  and audit, but redacted/truncated summaries for operator display. Bot tokens,
  secrets, and raw provider payload dumps are never written to traces, logs,
  CLI output, LiveView, or channel event summaries.

## Milestones

### M1: ADR, Settings, And Channel Contract

Add ADR 0016 and the initial channel modules:

- `AllbertAssist.Channels` as the public channel context.
- `AllbertAssist.Channels.Message` as the normalized inbound message struct.
- `AllbertAssist.Channels.Identity` for external-to-local user resolution.
- `AllbertAssist.Channels.Telegram.Adapter` as the supervised adapter.
- `AllbertAssist.Channels.Telegram.Client` as the bounded Bot API client.
- `AllbertAssist.Channels.Telegram.Renderer` for response and approval
  rendering.
- `AllbertAssist.Channels.Supervisor` for channel adapter supervision.

Add Settings Central schema entries with defaults:

- `channels.telegram.enabled`: `false`
- `channels.telegram.response_style`: `"concise"`
- `channels.telegram.bot_token_ref`:
  `"secret://channels/telegram/bot_token"`
- `channels.telegram.identity_map`: `[]`
- `channels.telegram.allowed_chat_ids`: `[]`
- `channels.telegram.allow_group_chats`: `false`
- `channels.telegram.poll_interval_ms`: `2000`
- `channels.telegram.poll_timeout_seconds`: `25`
- `channels.telegram.max_text_bytes`: `4096`
- `channels.telegram.render_approval_buttons`: `true`
- `channels.telegram.allow_confirmation_callbacks`: `true`

`identity_map` entries are maps with:

- `external_user_id`: required string
- `user_id`: required local string user id
- `display_name`: optional string
- `enabled`: boolean, default `true`

Validation rules:

- `external_user_id` and `user_id` must be non-empty strings.
- Identity map entries may not duplicate `external_user_id`.
- `allowed_chat_ids` contains strings only.
- `poll_interval_ms`, `poll_timeout_seconds`, and `max_text_bytes` have
  conservative lower and upper bounds.
- Settings output redacts `bot_token_ref` values that resolve to secrets.

### M2: Durable Channel Events And Identity Resolution

Add a SQLite-backed `channel_events` table for dedupe and traceability.

Fields:

- `id`: required
- `channel`, e.g. `"telegram"`: required
- `provider`, e.g. `"telegram_bot_api"`: required
- `direction`, one of `"inbound"`, `"outbound"`, `"callback"`: required
- `external_event_id`: required for inbound/callback events; outbound events
  may use a generated durable id until the provider returns a message id
- `external_user_id`: required when the provider payload has a sender
- `external_chat_id`: required when the provider payload has a chat/target
- `external_message_id`: optional provider message id
- `user_id`: nullable for unmapped rejected events; required once resolved
- `session_id`: nullable until identity resolution
- `thread_id`: nullable until runtime submission returns a thread
- `input_signal_id`: nullable until runtime submission
- `trace_id`: nullable until runtime/action execution creates a trace
- `status`, one of `"received"`, `"processed"`, `"rejected"`, `"failed"`:
  required
- `reason`: optional bounded status reason
- `payload_summary`: optional redacted bounded provider summary
- `error`: optional redacted bounded error summary
- timestamps

Add a unique index on `{channel, external_event_id}` for inbound and callback
provider events. Duplicate provider updates are skipped without resubmitting to
the runtime.

Raw user text belongs in conversation history once accepted by the runtime.
`channel_events.payload_summary` stores only a redacted, bounded summary of the
provider payload and must not duplicate secrets or full raw provider JSON.

Identity resolution:

- An inbound Telegram update resolves `external_user_id` through
  `channels.telegram.identity_map`.
- If there is no enabled mapping, the event is recorded as rejected and the
  runtime is not called.
- Rejected unmapped messages may receive a short provider response explaining
  that the Telegram account is not connected and must be mapped by the local
  operator.
- Resolved requests set both `user_id` and legacy `operator_id` to the mapped
  local user unless a future flow provides a compatible explicit operator.

Session id derivation:

- Telegram sessions use a deterministic string derived from channel,
  `external_user_id`, and `external_chat_id`.
- The value must be bounded and sanitized. If needed, hash the external pieces
  instead of creating atoms or storing unbounded identifiers in process names.

### M3: Telegram Transport And Supervision

Implement Telegram long polling with `Req` and a small client wrapper.

Transport constraints:

- Only Telegram Bot API host(s) are allowed.
- The token is read from Settings Secrets through `bot_token_ref`.
- Requests and errors redact the token and response body fields that could
  contain secrets or excessive user text.
- The adapter handles `getUpdates`, `sendMessage`, and answer/callback
  acknowledgements needed for v0.16 only.
- No arbitrary Telegram method runner is exposed.
- No arbitrary URL fetch, media download, file download, document extraction,
  or browser capture is added.

Supervision behavior:

- `AllbertAssist.Channels.Supervisor` starts as part of the application
  supervision tree.
- Telegram polling is disabled unless `channels.telegram.enabled` is true and
  a token secret is configured.
- Missing or invalid settings produce diagnostics and an idle adapter instead
  of a crash loop.
- Polling is single-flight per adapter instance, uses bounded timeouts, and
  backs off on provider errors.
- Restarted adapters resume from durable provider offset/event dedupe state
  and do not replay already processed updates.

### M4: Runtime Submission And Response Rendering

Normalize supported text messages into `AllbertAssist.Runtime.Request` data:

- `text`: Telegram message text after size validation
- `channel`: `"telegram"`
- `user_id`: resolved local string user id
- `operator_id`: same value for compatibility
- `session_id`: deterministic Telegram session id
- `thread_id`: omitted unless a channel command explicitly supplies one
- `metadata`: external channel, provider, chat, message, and event IDs

Thread behavior:

- Normal messages rely on v0.12 default thread selection: most recently
  updated general thread for the user, or create one if none exists.
- `/new <message>` creates a new thread for the resolved user.
- Conversation history remains SQLite `Thread`/`Message` data. Telegram
  channel events do not become markdown memory and do not promote turns into
  memory.

Rendering behavior:

- Runtime response text is rendered back to Telegram with conservative length
  limits.
- Approval Handoff responses are rendered using the handoff lines and allowed
  actions from v0.11.
- `resource_uri`, `operation_class`, `access_mode`, `scope`, downstream
  consumer, risk, and confirmation id remain inspectable in the rendered
  message, with sensitive values redacted.
- If a message is too large for Telegram, split it into bounded chunks or
  return a clear truncation note and preserve the full trace locally.

Unsupported inbound states:

- Oversized text is rejected before runtime submission.
- Media, voice, stickers, documents, links requiring fetch, and forwarded
  provider payloads are not processed in v0.16.
- Group chats are rejected unless `allow_group_chats` is explicitly true and
  the chat id is listed in `allowed_chat_ids`.

### M5: Confirmation Callback Flow

Telegram callback buttons may resolve durable confirmations created by existing
actions.

Callback data format:

- Compact, versioned strings such as
  `allbert:v1:approve:<confirmation_id>` and
  `allbert:v1:deny:<confirmation_id>`.
- No resource targets, shell commands, URLs, prompt text, or credentials are
  embedded in callback data.
- If a provider payload length limit would make a safe action ambiguous, the
  renderer falls back to a typed command-style instruction instead of adding a
  button.

Resolution flow:

- Resolve the callback sender through the same Settings Central identity map.
- Record a `callback` channel event.
- Call existing confirmation actions through `AllbertAssist.Actions.Runner`
  with channel metadata, not through private confirmation mutation.
- Preserve resolver channel, user id, session id, app id when present, and
  external callback metadata in confirmation trace/audit context.
- If confirmation state has changed, render the current state rather than
  trying to force a second resolution.

Approval scope rules:

- The callback action may approve, deny, or show a confirmation.
- Remembered resource grants are allowed only through scopes already exposed by
  v0.11 Approval Handoff and Security Central.
- A Telegram approval for one operation class does not authorize another
  operation class, provider, resource URI, downstream consumer, user, or app.

### M6: CLI, Tests, Docs, And Release Readiness

Add operator CLI support:

- `mix allbert.channels list`
- `mix allbert.channels show telegram`
- `mix allbert.channels telegram set-token TOKEN`
- `mix allbert.channels telegram map --external-user EXTERNAL --user USER`
- `mix allbert.channels telegram unmap --external-user EXTERNAL`
- `mix allbert.channels telegram simulate --external-user EXTERNAL --chat CHAT "prompt"`
- `mix allbert.channels telegram poll-once`

CLI behavior:

- Output redacts secrets and raw provider payloads.
- `simulate` goes through the same channel normalization, identity resolution,
  event recording, runtime submission, and rendering path as a real inbound
  text update, but never contacts Telegram and does not require polling to be
  enabled or a bot token to be configured. It still requires an explicit
  identity mapping.
- `poll-once` is intended for manual verification and tests. It must not run a
  background polling loop.

Add read-only registered actions where useful for agent/operator inspection:

- `list_channels`
- `show_channel`

These actions summarize settings and diagnostics only. They do not expose bot
tokens, raw provider payloads, or arbitrary provider method calls.

## User Acceptance

- With Telegram disabled, the application boots without polling and
  `mix allbert.channels show telegram` reports disabled diagnostics.
- A Telegram bot token is stored through Settings Secrets and never appears in
  CLI output, traces, logs, or tests.
- `mix allbert.channels telegram map --external-user 123 --user alice` creates
  an explicit mapping; unmapped external users cannot submit runtime input.
- `mix allbert.channels telegram simulate --external-user 123 --chat 456 "hi"`
  produces a runtime response for `user_id: "alice"`, records a channel event,
  and persists the conversation under Alice's SQLite thread history.
- Simulated or test-backed duplicate Telegram update IDs are not processed
  twice.
- `/new hello` creates a new thread for the mapped user.
- Alice and Bob Telegram identities remain isolated in thread history, channel
  event records, confirmation resolution, and traces.
- A v0.11 risky URL/document/import prompt from Telegram produces an
  inspectable Approval Handoff response instead of hidden execution.
- Telegram approve/deny callbacks resolve existing durable confirmations
  through registered actions and preserve resolver channel metadata.
- Unsupported media and oversized text are rejected clearly without calling the
  runtime.
- Restarting the channel adapter does not replay already processed durable
  events or lose the configured identity map.

## Test Plan

Focused tests:

- Settings schema defaults and validation for all `channels.telegram.*` keys.
- Secret storage/redaction for `bot_token_ref`.
- Identity map resolution, duplicate external ID rejection, disabled mapping
  rejection, and no implicit `"local"` fallback.
- `channel_events` insertion, unique dedupe, redacted payload summaries, and
  status transitions.
- Telegram update parser for text, callback query, unsupported media, group
  rejection, and oversized text.
- Telegram renderer for normal response, Approval Handoff response, truncation,
  and callback button fallback.
- Telegram client tests using a test Req adapter; no real network dependency.
- Adapter `poll_once` success, duplicate update skip, provider error backoff,
  and restart/replay behavior.
- Runtime integration proving mapped Telegram input calls
  `Runtime.submit_user_input/1`, persists `Thread`/`Message`, and returns
  `user_id`, `thread_id`, `session_id`, and channel metadata.
- Confirmation callback integration proving approve, deny, already-resolved,
  and unauthorized/unmapped callback paths.
- CLI tests for `list`, `show`, `set-token`, `map`, `unmap`, `simulate`, and
  `poll-once`.

Milestone warning gate:

- `mix format --check-formatted`
- `mix compile --warnings-as-errors`
- `mix credo --strict`
- `mix dialyzer`
- `mix precommit`

## Manual Verification

Use a temporary `ALLBERT_HOME`:

1. Boot with Telegram disabled and confirm no polling side effects.
2. Configure a token secret and identity map with the CLI.
3. Run `simulate` for a mapped user and confirm response, event, thread, and
   trace metadata.
4. Run `simulate` for an unmapped user and confirm rejection without a thread
   message.
5. Trigger a risky prompt that creates Approval Handoff and confirm the
   rendered response includes safe approve/deny/show instructions.
6. Resolve a confirmation callback through the test path and confirm the
   confirmation record keeps channel resolver metadata.
7. Restart the application and verify duplicate events are not replayed.

Real Telegram API verification is optional and should use an operator-created
test bot token. Tests must not require network access or a live provider.

## Non-Goals

- No email, SMS, Discord, Slack, native app, browser, or MCP channel adapter.
- No Telegram webhooks or public inbound HTTP endpoint; v0.16 uses long
  polling only.
- No arbitrary Telegram method execution.
- No arbitrary remote URL fetch, media/file download, document extraction,
  browser capture, package install, skill import, or crawler behavior from
  channel messages.
- No channel-specific intent engine, memory store, confirmation store,
  scheduler, or security policy.
- No implicit external identity claim on local users.
- No proactive broadcast or scheduled channel messaging beyond responses to
  inbound updates and confirmation callbacks.
- No hosted accounts, auth, roles, OAuth, or multi-tenant administration.
- No automatic markdown memory promotion from channel conversation turns.
- No app registry or surface expansion beyond consuming v0.15 metadata when
  present in runtime context.

## Handoffs

- v0.20 can use channel metadata and active app/session context as additional
  inputs for cross-surface intent enrichment.
- v0.23 security evals should include cross-channel identity spoofing,
  callback replay, provider payload injection, resource approval scope leakage,
  and cross-user/thread leakage from channel events.
- v0.26 workspace surface work should treat Telegram as a channel proof point,
  not as a UI architecture.
