## Locked Decisions
Recorded in `v0.55-plan.md`. TUI is a delivery adapter with no extra authority;
library = `owl`; CLI not promoted (TUI is the first-class terminal channel);
TUI primitives `[:typed_command, :list]`; identity via the existing list-shaped
`channels.tui.identity_map`; parity matrix machine-derived from descriptors. The
mix-free operator console (in-TUI slash-commands + `mix allbert.channels status`)
is deferred to v0.55.1 (`docs/plans/v0.55b-plan.md`, ADR 0070); v0.55 ships the
basic `mix allbert.tui` launcher only.

## TUI Data Flow

The TUI inbound path mirrors the Telegram adapter (`plugins/allbert.telegram`):
a supervised GenServer reads terminal input, normalizes it, resolves identity, and
submits through the same `Runtime.submit_user_input/1` seam every channel uses —
the only TUI-specific pieces are the input loop (`Owl.IO.input`), the descriptor's
`:tui_session` strategy, and the scrollback/live-block render surface.

```text
terminal input (Owl.IO.input loop, supervised GenServer)
  -> normalize (trim, drop empty; in-TUI slash-command routing is v0.55.1)
  -> Channels.Identity.resolve (list-shaped channels.tui.identity_map;
     terminal profile "default" unmapped -> {:error, :not_mapped})
  -> Channels.derive_session_id (descriptor session_strategy {:tui_session, prefix: "ch_tui_"}
     through the existing generic strategy branch unless implementation proves otherwise)
  -> Runtime.submit_user_input(%{channel: "tui", user_id, session_id, ...})
  -> Conversations.ChannelThread lookup/link/message-ref recording
  -> channel_events create/update (dedupe by (channel, external_event_id, direction);
     redacted/truncated summary, never raw payload)
  -> response appended as scrollback lines
     + a single Owl.LiveScreen live block for the active prompt / streaming line
  -> approval -> Approval.Handoff.render/2 -> :list / :typed_command in terminal
  -> typed "ALLBERT:APPROVE:<id>" -> Actions.Runner.run("approve_confirmation", ...)
```

The split-payload seam (ADR 0067 extending ADR 0029/0030) adds named typed-response
fields `model_payload` and `surface_payload`. `surface_payload` defaults to
`model_payload` for existing callers. The runtime stores/threads only
`model_payload`; terminal renderers draw `surface_payload`, which may contain
framing, ANSI, paging, and the `LiveScreen` live block. Terminal chrome never
leaks into conversation history.
This is the foundation the v0.57 Pi-mode coding surface builds on.

## Parity Matrix (dimensions, machine-derived)
Per channel (web, Telegram, email, CLI label, Discord, Slack, Matrix, WhatsApp,
Signal, TUI): approval primitives declared; attachments posture; streaming
(none — turn-complete in v0.55); identity mapping; approval rendering; outbound
delivery; release/live-use status. Canonical prose in the ADR 0016 amendment;
derived via `ChannelParity` / `mix allbert.channels --parity`.

## Authority Invariants
- TUI calls only `Runtime.submit_user_input/1` + `Actions.Runner.run/3`; no direct
  Security Central / confirmation-store access (ADR 0016).
- Unmapped terminal identity is rejected; never implicitly `"local"`.
- Approval primitives never embed targets/commands/URLs; confirmations resolve
  only through the durable store via the Runner.
- `channel_events` summaries redacted/truncated; no raw payloads.
- TUI crash isolated under `:one_for_one`.

## Handles
- `mix allbert.channels --parity` (matrix report).
- `mix allbert.tui` (basic launcher: boots the app + the owl input loop;
  persistent terminal session, no slash-commands).
- `plugins/allbert.tui` adapter; `channels.tui.*` settings.

Deferred to v0.55.1 (`docs/plans/v0.55b-request-flow.md`): the mix-free operator
console — the in-session TUI slash-commands `/status`, `/confirmations`, `/events`,
`/channels`, `/settings get`, `/help`, and `mix allbert.channels status`.

## Security / Eval Rows (`:v055`)

## Operator-Validation Punchlist (v0.55, persistent TUI session)

Run A1–A5 against the **persistent TUI session** to confirm the warm-process
behaviour the deterministic gate (`mix allbert.test release.v055`, simulated input)
cannot exercise: the live `owl` input loop, the scrollback/`LiveScreen` render
surface, crash-and-restart under the real supervisor, and the model-facing/surface
payload split. Unlike the v0.54 router punchlist — where every step was a fresh
cold `mix allbert.ask` against a shared `ALLBERT_HOME` — **this punchlist runs
inside one warm BEAM**: a single supervised `mix allbert.tui` session holds the
node, the `Channels.Supervisor` tree, and the SQLite store open for the whole run.
No step restarts the BEAM or reopens the DB; every check is typed at the live
prompt or read from a second shell with an existing `mix` task against that same
running node. (The in-TUI slash-command inspection surface is v0.55.1 —
`docs/plans/v0.55b-request-flow.md`.)

The TUI adapter plus the basic `mix allbert.tui` launcher is built in **M2**
(`plugins/allbert.tui`) and its approval rendering in **M3**; these validation
**instructions** are authored in **M4** (Evals + Docs). The punchlist below is
**run as the M5 operator-validation gate**, and a failing run blocks the **M6**
release closeout / version tag. The mix-free operator console (in-TUI
slash-commands + `mix allbert.channels status`) is v0.55.1 —
`docs/plans/v0.55b-plan.md`.

### Setup (one-time, pre-launch)

These three commands run as one-shot `mix` tasks before the session launches; every
A-check after that happens **inside** the warm session opened by S3.

```bash
# S1. Enable the TUI channel (off by default).
mix allbert.settings set channels.tui.enabled true

# S2. Map the terminal identity. An empty identity_map rejects every inbound
#     (validated in A2); a mapped local profile is required for A1/A3-A5.
#     Reuse the same list-shaped identity_map contract every channel uses.
mix allbert.settings set channels.tui.identity_map '[{"external_user_id":"default","user_id":"local","enabled":true}]'
mix allbert.settings get channels.tui.identity_map        # echoes the list back

# S3. Launch the persistent supervised terminal session in a running node.
#     This is the warm BEAM for the whole punchlist: DB open, Channels.Supervisor
#     up, the owl input loop live. Do NOT exit it between checks.
mix allbert.tui
```

Expected on launch: a banner identifying the `"tui"` channel + provider
`"terminal"`, the resolved local profile, and a live `LiveScreen` prompt. This is
the basic launcher — it boots the app and the owl input loop only. The A-checks
below read state from a second shell using existing `mix` tasks
(`mix allbert.channels show tui` / `list`, `mix allbert.confirmations list`); the
in-TUI slash-command inspection surface (`/status`, `/events`, `/confirmations`,
`/channels`, `/settings get`, `/help`) lands in v0.55.1.

### A. TUI console checks (all inside the one warm session)

**A1 — Full turn + event recording (the live keystone) plus replay dedupe evidence.**

At the live prompt, type a normal message and watch it complete in-session:

```text
> remind me what timezone meetings default to
```

Expect the response to render as **scrollback lines** above the live prompt (not a
new process, not a cold task). Capture the visible terminal evidence: the prompt
stays live, the response appears in scrollback, and the same `mix allbert.tui`
session remains open. A second shell may run `mix allbert.channels show tui` or
`list` only to confirm the channel summary remains secret-free; those tasks are
not event-row inspectors in v0.55.

Do **not** prove dedupe by retyping the same text. A normal re-submit is a new
terminal event and should receive a fresh `external_event_id`. Dedupe is by
`(channel, external_event_id, direction)`, so the duplicate proof comes from the
deterministic `mix allbert.test release.v055` fixture that submits the same
terminal `external_event_id` twice. If implementation adds a validation-only
manual replay helper later, document that helper before the operator punchlist
uses it.

Pass criteria: the live turn completes and renders as scrollback; the first submit
is backed by release evidence showing exactly one processed inbound
`channel_events` row; the replay evidence shows that the same `external_event_id`
produces **no second runtime submission** and no second processed inbound row.
Evidence: capture the live scrollback/prompt behavior and the
`tui-inbound-turn-dedupe-001` release/eval evidence. (The in-TUI `/events`
inspection command itself lands in v0.55.1.)

**A2 — Identity-map rejection.**

Empty the identity map, then submit at the live prompt. With no mapping the
inbound must be **rejected before any runtime submission** — never implicitly
claimed as `"local"`. If TUI settings are read only at session start, relaunch the
same `ALLBERT_HOME` after this one-shot mutation and record the relaunch; if a
future live settings reload is added, use that documented reload path instead.

```bash
mix allbert.settings set channels.tui.identity_map '[]'
```

```text
> hello with no identity mapping
```

Expected: the TUI reports or visibly declines the unmapped input without an
assistant response. The row-level proof that the redacted event records
`status: rejected` carrying `{:error, :not_mapped}`, with no
`Runtime.submit_user_input/1` call, no thread, and no model turn, comes from
`tui-identity-map-001` in `mix allbert.test release.v055`.
Pass criteria: the warm session produces no assistant response for the unmapped
input, and the deterministic release evidence records the rejection. Restore the map
(`[{"external_user_id":"default","user_id":"local","enabled":true}]`) before
continuing. Evidence: the visible rejection/no-response behavior plus the
deterministic rejected-row evidence.

**A3 — Approval primitive render + resolve.**

Trigger a `needs_confirmation` action (e.g. `write_note`) at the live prompt and
confirm the terminal renders the declared primitives — a numbered `:list` plus the
`:typed_command` `ALLBERT:APPROVE:<id>` — never `:button`/`:link`:

```text
> create a note titled v055 with body hello from the tui
```

Expected render (scrollback):

```text
Status: needs_confirmation
Approval Handoff:
  1) approve   2) deny   3) show           # :list (numbered)
  Or type: ALLBERT:APPROVE:<CONF_ID>       # :typed_command
  Confirmation: <CONF_ID> target=write_note status=pending
```

Resolve it by typing the approval line at the same prompt, then read the result —
either from the in-session approval result line, or from a second shell with the
existing `mix allbert.confirmations list` task against the same running node:

```text
> ALLBERT:APPROVE:<CONF_ID>
```

```bash
mix allbert.confirmations list    # existing read task; shows <CONF_ID> resolved/approved
```

Pass criteria: the typed `ALLBERT:APPROVE:<id>` routes through
`Actions.Runner.run("approve_confirmation", ...)` (the same Runner path Telegram
uses — the TUI holds no direct confirmation-store access), the confirmation
resolves, and the `write_note` side effect runs. The in-session approval result
line and `mix allbert.confirmations list` show `<CONF_ID>` moved from pending to
resolved/approved. Evidence: the rendered primitives, the resolved confirmation,
and the written note. (The in-TUI `/confirmations` inspection command itself lands
in v0.55.1.)

**A4 — Crash isolation (`:one_for_one`).**

Kill the live TUI GenServer and confirm the rest of the tree is unaffected and the
TUI restarts. First capture sibling-channel status from a second shell with an
existing read task:

```bash
mix allbert.channels list    # existing read task; shows tui + sibling channels (e.g. telegram)
```

Then from a second shell (or a documented test affordance where the harness
provides one), terminate the TUI adapter process:

```bash
# identify and kill the tui process (e.g. via :observer / a documented test affordance)
```

Pass criteria: killing the TUI GenServer does **not** take down other channels
under `Channels.Supervisor` (Telegram and any other configured channel keep
running), and the supervisor restarts the TUI adapter under `:one_for_one` — a
fresh prompt returns. Evidence: the warm session recovers (a fresh prompt
returns), and `mix allbert.channels list` shows the TUI and sibling channels back
up without secret output.

**A5 — Split-payload seam (the v0.57 dependency).**

Confirm the surface-render framing — `LiveScreen` live block, ANSI styling, paging
— does **not** leak into `model_payload`. The typed-response split (ADR 0067
extending ADR 0029/0030) must keep `model_payload` free of terminal chrome while
allowing `surface_payload` to carry TUI framing so the v0.57 Pi-mode coding
surface can build on it.

```text
> show me a long response that pages and renders styled output
```

Pass criteria: `model_payload` recorded for the turn (the thread message and any
model-facing event summary) contains only the conversational content — **no** ANSI
escape sequences, paging markers, prompt framing, or `LiveScreen` live-block
artifacts. `surface_payload` may be richly framed; the stored/threaded
`model_payload` is clean. Evidence: the rendered terminal output alongside
`split-payload-contract-001` / `split-payload-defaulting-001` release evidence
showing the two payloads diverge exactly at the chrome. Do not use
`mix allbert.channels show tui` as the payload inspector; it is a channel summary
task in v0.55. The v0.55.1 console provides the warm `/events` inspection path.

A second-shell `mix` read against the running node (`mix allbert.channels show
tui` / `list`, `mix allbert.confirmations list`) is read-only channel or
confirmation summary evidence only. If the SQLite store is busy, retry the read
after the live turn finishes; do not make second-shell reads the authority for
row-count or payload checks in v0.55.

### Sign-off

A1–A5 pass with one persistent `mix allbert.tui` validation session except where
A2 explicitly records a settings reload/relaunch requirement; the deterministic
`mix allbert.test release.v055` gate is green; and `mix allbert.channels --parity`
matches the registry. With those, the **M5** operator-validation gate is cleared
and the **M6** release closeout / version tag may proceed.

## Matrix Outbound Real-Provider Smoke

The Matrix generic-outbound check (v0.54-deferred gap, closed in M1) is **not**
part of the warm-TUI punchlist: it requires a configured Matrix homeserver plus
credentials and is run as a separate real-provider smoke. Confirm that a generic
outbound to a **mapped Matrix target** now delivers rather than degrading with
`:outbound_not_implemented`. This validates the v0.54-deferred Matrix
`deliver_outbound/3` that M1 closes through `Channels.Outbound`.

M1 must either update the existing Matrix delivery selector or add a dedicated
v0.55 selector so the release-authoritative smoke exercises
`send_channel_message` → `Channels.Outbound` → Matrix `deliver_outbound/3`, not a
direct Matrix client send. The command documented for closeout must be exact; the
preferred shape is:

```bash
mix allbert.test external-smoke -- matrix
export V055_MATRIX_SMOKE_EVIDENCE="<printed external-smoke-matrix evidence path>"
python3 -m json.tool "$V055_MATRIX_SMOKE_EVIDENCE" | rg '"status": "passed"|"matrix"|"m.room.message"|send_channel_message|Channels.Outbound'
```

Manual fallback, if the smoke selector is blocked by provider state, is the same
action boundary and not an ad-hoc adapter call:

```text
> send a matrix message to <MAPPED_MATRIX_TARGET> saying v055 outbound check
```

The router should choose `send_channel_message` with slots
`channel="matrix"`, `target="<MAPPED_MATRIX_TARGET>"`, and
`body="v055 outbound check"`. Approve the durable confirmation. If a future CLI
helper is added, it must still call the same `send_channel_message` action and
`Channels.Outbound` boundary.

Pass criteria: the message is delivered to the Matrix target; the call **never**
returns `{:error, :outbound_not_implemented}` and is **not** the v0.54 graceful-
degrade for mapped Matrix targets. Evidence: delivery confirmation (or an outbound
`channel_events` row for the Matrix channel) with no `:outbound_not_implemented`.
