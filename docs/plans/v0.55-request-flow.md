## TUI Data Flow

The TUI inbound path mirrors the Telegram adapter (`plugins/allbert.telegram`):
a supervised GenServer reads terminal input, normalizes it, resolves identity, and
submits through the same `Runtime.submit_user_input/1` seam every channel uses —
the only TUI-specific pieces are the input loop (`owl` `IO.input`), the new
`:tui_session` derivation strategy, and the scrollback/live-block render surface.

```text
terminal input (owl IO.input loop, supervised GenServer)
  -> normalize (trim, drop empty, route in-TUI slash-commands /events /confirmations)
  -> Channels.Identity.resolve (channels.tui.identity_map; unmapped -> {:error, :not_mapped})
  -> Channels.derive_session_id ({:tui_session, prefix: "ch_tui_"})
  -> Runtime.submit_user_input(%{channel: "tui", user_id, session_id, ...})
  -> Conversations.ChannelThread lookup/link/message-ref recording
  -> channel_events create/update (dedupe by (channel, external_event_id, direction);
     redacted/truncated summary, never raw payload)
  -> response appended as scrollback lines
     + a single Owl.LiveScreen live block for the active prompt / streaming line
  -> approval -> Approval.Handoff.render/2 -> :list / :typed_command in terminal
  -> typed "ALLBERT:APPROVE:<id>" -> Actions.Runner.run("approve_confirmation", ...)
```

The split-payload seam (ADR 0067 extending ADR 0029/0030) keeps the surface-render
payload (framing, ANSI, paging, the `LiveScreen` live block) separate from the
model-facing payload, so terminal chrome never leaks into conversation history.
This is the foundation the v0.57 Pi-mode coding surface builds on.

## Parity Matrix (dimensions, machine-derived)
Per channel (web, Telegram, email, CLI label, Discord, Slack, mobile, TUI):
approval primitives declared; attachments posture; streaming (none — turn-complete);
identity mapping; approval rendering. Canonical prose in the ADR 0016 amendment;
derived via `ChannelParity` / `mix allbert.channels --parity`.

## Authority Invariants
- TUI calls only `Runtime.submit_user_input/1` + `Actions.Runner.run/3`; no direct
  Security Central / confirmation-store access (ADR 0016).
- Unmapped terminal identity is rejected; never implicitly `"local"`.
- Approval primitives never embed targets/commands/URLs; confirmations resolve
  only through the durable store via the Runner.
- `channel_events` summaries redacted/truncated; no raw payloads.
- TUI crash isolated under `:one_for_one`.

## Handles
- `mix allbert.channels --parity` (matrix report); `plugins/allbert.tui` adapter;
  `channels.tui.*` settings.

## Security / Eval Rows (`:v055`)

crash isolation; approval-primitive-honor (`:list`/`:typed_command` only);

## Operator-Validation Punchlist (v0.55, persistent TUI console)

Run A1–A6 against the **persistent TUI console** to confirm the warm-process
behaviour the deterministic gate (`mix allbert.test release.v055`, simulated input)
cannot exercise: the live `owl` input loop, the scrollback/`LiveScreen` render
surface, crash-and-restart under the real supervisor, and the model-facing/surface
payload split. Unlike the v0.54 router punchlist — where every step was a fresh
cold `mix allbert.ask` against a shared `ALLBERT_HOME` — **this punchlist runs
inside one warm BEAM**: a single supervised `mix allbert.tui` session holds the
node, the `Channels.Supervisor` tree, and the SQLite store open for the whole run.
No step restarts the BEAM or reopens the DB; every check is typed at the live
prompt or issued through an in-TUI slash-command against that same running node.

The TUI adapter is built in **M2** (`plugins/allbert.tui`), its approval rendering
in **M3**, and the persistent operator/validation console in **M4**; these
validation **instructions** are authored in **M5** (Evals + Docs). The punchlist
below is **run as the M6 operator-validation gate**, and a failing run blocks the
**M7** release closeout / version tag.

### Setup (one-time, pre-launch)

These three commands run as one-shot `mix` tasks before the session launches; every
A-check after that happens **inside** the warm session opened by S3.

```bash
# S1. Enable the TUI channel (off by default).
mix allbert.settings set channels.tui.enabled true

# S2. Map the terminal identity. An empty identity_map rejects every inbound
#     (validated in A2); a mapped local profile is required for A1/A3-A6.
#     The map keys the terminal profile to a local user_id (default "local").
mix allbert.settings set channels.tui.identity_map '{"default": "local"}'
mix allbert.settings get channels.tui.identity_map        # echoes the map back

# S3. Launch the persistent supervised terminal session in a running node.
#     This is the warm BEAM for the whole punchlist: DB open, Channels.Supervisor
#     up, the owl input loop live. Do NOT exit it between checks.
mix allbert.tui
```

Expected on launch: a banner identifying the `"tui"` channel + provider
`"terminal"`, the resolved local profile, and a live `LiveScreen` prompt. The
session also exposes in-TUI slash-commands used below — `/events` (recent
`channel_events` rows), `/confirmations` (pending/resolved confirmations),
`/help` — which inspect the same running node, not a separate cold task.

### A. TUI console checks (all inside the one warm session)

**A1 — Full turn + `channel_events` dedupe (the keystone).**

At the live prompt, type a normal message and watch it complete in-session:

```text
> remind me what timezone meetings default to
```

Expect the response to render as **scrollback lines** above the live prompt (not a
new process, not a cold task), and exactly **one** inbound `channel_events` row for
that turn. Inspect it without leaving the session:

```text
> /events
```

Expected:

```text
Status: ok
channel=tui direction=inbound external_event_id=<EVT_ID> status=processed
  summary=<redacted/truncated; no raw payload>
(1 inbound row for this turn)
```

Then retype the **identical** input and re-run `/events`:

```text
> remind me what timezone meetings default to
> /events
```

Pass criteria: the turn completes and renders as scrollback; the first submit
creates exactly one inbound `channel_events` row; the identical re-submit produces
**no new inbound row** (dedupe by `(channel, external_event_id, direction)` — one
runtime submission, not two). `Status: ok` plus the unchanged row count is the
dedupe proof. Evidence: capture the `<EVT_ID>` and the before/after `/events`
counts.

**A2 — Identity-map rejection.**

Empty the identity map (one-shot task), then submit at the live prompt. With no
mapping the inbound must be **rejected before any runtime submission** — never
implicitly claimed as `"local"`.

```bash
mix allbert.settings set channels.tui.identity_map '{}'    # in a second shell, or re-set then relaunch
```

```text
> hello with no identity mapping
> /events
```

Expected: the redacted event records `status: rejected` carrying `{:error,
:not_mapped}`; no `Runtime.submit_user_input/1` call, no thread, no model turn.
Pass criteria: rejection is visible in `/events` and the turn produced no assistant
response. Restore the map (`{"default": "local"}`) before continuing. Evidence:
the rejected row + absence of an outbound/response row.

**A3 — Approval primitive render + resolve.**

Trigger a `needs_confirmation` action (e.g. `write_note`) at the live prompt and
confirm the terminal renders the declared primitives — a numbered `:list` plus the
`:typed_command` `ALLBERT:APPROVE:<id>` — never `:button`/`:link`:

```text
> create a note titled v055 with body hello from the tui
```

Expected render (scrollback):

```text
Status: needs_confirmation
Approval Handoff:
  1) approve   2) deny   3) show           # :list (numbered)
  Or type: ALLBERT:APPROVE:<CONF_ID>       # :typed_command
  Confirmation: <CONF_ID> target=write_note status=pending
```

Resolve it by typing the approval line at the same prompt:

```text
> ALLBERT:APPROVE:<CONF_ID>
> /confirmations
```

Pass criteria: the typed `ALLBERT:APPROVE:<id>` routes through
`Actions.Runner.run("approve_confirmation", ...)` (the same Runner path Telegram
uses — the TUI holds no direct confirmation-store access), the confirmation
resolves, and the `write_note` side effect runs. `/confirmations` shows
`<CONF_ID>` moved from pending to resolved/approved. Evidence: the rendered
primitives, the resolved confirmation, and the written note.

**A4 — Crash isolation (`:one_for_one`).**

Kill the live TUI GenServer and confirm the rest of the tree is unaffected and the
TUI restarts. From a second shell (or an in-TUI `/crash` test affordance where the
harness provides one), terminate the TUI adapter process:

```bash
# identify and kill the TUI channel GenServer in the running node
mix allbert.channels status                # shows tui + other channels (e.g. telegram) running
# kill the tui process (e.g. via :observer / a documented test affordance)
```

Pass criteria: killing the TUI GenServer does **not** take down other channels
under `Channels.Supervisor` (Telegram and any other configured channel keep
running), and the supervisor restarts the TUI adapter under `:one_for_one` — a
fresh prompt returns. Evidence: `mix allbert.channels status` shows the sibling
channels never dropped and the TUI back up after restart.

**A5 — Matrix generic outbound (v0.54-deferred gap, closed in M1).**

Confirm that a generic outbound to a **mapped Matrix target** now delivers rather
than degrading with `:outbound_not_implemented`. This validates the v0.54-deferred
Matrix `deliver_outbound/3` that M1 closes through `Channels.Outbound`:

```text
> send_channel_message to a mapped matrix target
```

(or invoke `send_channel_message` against the mapped Matrix target via the live
session / `Channels.Outbound`).

Pass criteria: the message is delivered to the Matrix target; the call **never**
returns `{:error, :outbound_not_implemented}` and is **not** the v0.54 graceful-
degrade for mapped Matrix targets. Evidence: delivery confirmation (or a
`/events` outbound row for the Matrix channel) with no `:outbound_not_implemented`.

**A6 — Split-payload seam (the v0.57 dependency).**

Confirm the surface-render framing — `LiveScreen` live block, ANSI styling, paging
— does **not** leak into the model-facing payload. The typed-response split (ADR
0067 extending ADR 0029/0030) must keep the model-facing payload free of terminal
chrome so the v0.57 Pi-mode coding surface can build on it.

```text
> show me a long response that pages and renders styled output
> /events
```

Pass criteria: the model-facing payload recorded for the turn (the `channel_events`
summary / thread message) contains only the conversational content — **no** ANSI
escape sequences, paging markers, prompt framing, or `LiveScreen` live-block
artifacts. The surface render in the terminal may be richly framed; the stored
model-facing payload is clean. Evidence: the rendered terminal output alongside the
`/events` summary, showing the two payloads diverge exactly at the chrome.

### Sign-off

A1–A6 pass inside a single uninterrupted `mix allbert.tui` session (no BEAM
restart, no DB reopen between checks); the deterministic `mix allbert.test
release.v055` gate is green; and `mix allbert.channels --parity` matches the
registry. With those, the **M6** operator-validation gate is cleared and the
**M7** release closeout / version tag may proceed.
