# Allbert v0.27 App Surface Contract - StockSage LiveViews

Request flow: `docs/plans/v0.27-request-flow.md`

Formerly tracked as M-D3a, previously planned as v0.23, then v0.24, then
v0.25 before the v0.23 Jido State-Machine Convergence and v0.24 Objective
Runtime Foundation milestones were inserted by the project-direction rethink.
Renamed after the post-v0.26 roadmap reconciliation to make the contract
visible: StockSage is the reference implementation for the app surface
contract, not a special-case app.

## Purpose

v0.27 proves that a plugin-contributed app can expose real Phoenix LiveView
surfaces through `AllbertAssist.App.SurfaceProvider` from day one. StockSage
is the first non-core app to carry that proof.

The milestone ships real StockSage renderers in StockSage-owned
`/stocksage/...` LiveViews. These renderers replace the v0.26 reserved-card
stubs for `:analysis_card`, `:agent_report_card`, `:parity_card`, and
`:debate_round_card` when those cards are displayed in the StockSage app
surface. v0.30 later wires the same proven component types into durable
`/agent` workspace canvas tiles through `canvas_ops` and the audited v0.26
canvas mechanism. v0.27 is renderer and app-surface proof; v0.30 is durable
canvas-emission proof.

v0.27 also adds app memory namespace declaration/registration as an ownership
claim, but does not sync lessons, reflections, or analysis results into
Allbert memory. That separation lets v0.28 audit namespace claim abuse and
namespace isolation before v0.29 adds privileged memory writes.

## Implementation Research Notes

Current Phoenix docs consulted during planning:

- Phoenix LiveView streams support incremental collection rendering with
  `stream/3`, `stream_insert/4`, and `phx-update="stream"`. v0.27 should use
  this shape for analysis progress lists rather than replacing the full page.
- Phoenix.PubSub provides `subscribe/3` and `broadcast/4` over named topics.
  v0.27 progress events should subscribe per analysis/objective/user context
  and catch up from persisted StockSage/objective state on reconnect.
- LiveView JS commands and hooks are appropriate only for client-owned behavior
  such as focus/keyboard helpers or chart integration; StockSage domain state
  remains server-owned behind contexts/actions.

## Routes (host router)

Plugin LiveViews are plugin-owned modules but mount in the host router
(`apps/allbert_assist_web/lib/allbert_assist_web/router.ex`) inside the existing
`scope "/", AllbertAssistWeb` block, alongside `live "/agent", AgentLive`:

```elixir
live "/stocksage", StockSageWeb.WorkspaceLive, :index
live "/stocksage/analyses", StockSageWeb.AnalysisLive, :index
live "/stocksage/analyses/:id", StockSageWeb.AnalysisLive, :show
live "/stocksage/queue", StockSageWeb.QueueLive, :index
live "/stocksage/trends", StockSageWeb.TrendsLive, :index
```

Route mounting stays in the host router (the host owns the routing table);
navigation metadata (labels, ordering, icons) comes from
`StockSage.App.surfaces/0` so the host can build nav without hardcoding
StockSage. The router mount and the provider declaration must agree (validated
by `mix allbert.validate_app`).

### `AllbertAssist.App` behaviour + `AllbertAssist.App.Registry` (new namespace surface)

Add an optional callback and a registry accessor. Declaration is metadata; no
write API is added in v0.27.

```elixir
# apps/allbert_assist/lib/allbert_assist/app.ex
@type memory_namespace_declaration :: %{
        app_id: atom(), namespace: atom(), writable: boolean(),
        description: String.t()
      }
@callback memory_namespace() :: memory_namespace_declaration() | nil
@optional_callbacks memory_namespace: 0

# apps/allbert_assist/lib/allbert_assist/app/registry.ex
@spec registered_memory_namespaces(keyword()) :: [memory_namespace_declaration()]
def registered_memory_namespaces(opts \\ []), do: call(opts, :registered_memory_namespaces, [])
```

### StockSage LiveViews (`plugins/stocksage/lib/stocksage_web/`)

Each is `use AllbertAssistWeb, :live_view`, thin, and delegates to StockSage
contexts + registered actions. They never hold domain logic.

```elixir
defmodule StockSageWeb.AnalysisLive do
  use AllbertAssistWeb, :live_view

  @impl true
  def mount(_params, _session, socket), do: {:ok, set_active_app(socket, :stocksage)}

  @impl true
  def handle_params(%{"id" => id}, _uri, socket) do
    # load persisted analysis + objective state from StockSage contexts;
    # subscribe to the progress topic for live updates.
    {:noreply, load_analysis(socket, id)}
  end

  @impl true
  # PubSub progress event (see Signals); update via stream_insert/4.
  def handle_info({:stocksage_progress, payload}, socket),
    do: {:noreply, stream_insert(socket, :progress, payload)}
end
```

`WorkspaceLive` (landing + nav), `QueueLive` (queue entries), and `TrendsLive`
(trend summaries) follow the same shape. Progress lists use
`stream/3` + `stream_insert/4` + `phx-update="stream"` (per Implementation
Research Notes) so updates never replace the page.

### `StockSage.Actions.RunAnalysis` result extension

`run/2` keeps its `%{message, status, ...}` result and ADDS validated Surface
output. Nodes are validated at action completion via
`AllbertAssist.Surface.validate_surface/1`; undeclared component atoms are
rejected before the result is returned.

```elixir
# completed result gains:
%{
  # ...existing message/status/analysis_id/objective_id...
  surface_nodes: [%AllbertAssist.Surface.Node{component: :analysis_card, props: %{...}}, ...]
}
```

## Settings And Signals

### Settings (Settings Central)

| key | type | default | validation | description |
|---|---|---|---|---|
| `stocksage.web.enabled` | boolean | `true` | strict boolean | Master switch for the `/stocksage/*` LiveView surfaces. Routes remain mounted; each StockSage LiveView checks Settings Central on mount/handle_params and renders or redirects to the host not-found behavior when false. |
| `stocksage.web.progress_stream_enabled` | boolean | `true` | strict boolean | Enables live PubSub progress streaming; when false the detail page still catches up from persisted state on load. |

(No other new settings. StockSage analysis/engine settings are unchanged from
v0.22/v0.25.)

### Signals / PubSub

Progress streaming reuses Phoenix.PubSub (not new SignalBus topics). Topic and
payload convention:

- **Topic**: `stocksage_progress:<user_id>:<analysis_id>`
- **Payload**: `%{analysis_id, objective_id, stage, status, summary, at}`
  (bounded + redacted; `stage` ∈ analyst/debate/risk/synthesis/parity).

LiveViews subscribe on `handle_params` and catch up from persisted
analysis/objective state on (re)connect. No new `allbert.*` SignalBus subjects
are introduced; existing objective/confirmation signals continue to flow
through v0.24/v0.26 SignalBridge.

## Non-Goals

- No durable `/agent` canvas tile emission and no `canvas_ops`; that is v0.30.
- No memory sync, reflections, lesson promotion, or markdown-memory writes;
  that is v0.29.
- No AG-UI protocol emission. Surface nodes are Allbert-native validated
  structures.
- No arbitrary model-generated component shapes.
- No PostgreSQL or Oban dependency.

## Definition Of Done

v0.27 is done when StockSage is a usable plugin-contributed web app through
declared `SurfaceProvider` surfaces; real StockSage renderers replace the
reserved v0.26 card stubs inside `/stocksage/...`; `RunAnalysis` emits
