# Allbert v0.45.1 Gate Transparency And Precommit Decomposition

Request flow: `docs/plans/v0.45.1-request-flow.md`

ADR amendment: `docs/adr/0049-development-gates-and-test-parallelization.md`

Consumed: `docs/plans/v0.41-plan.md`,
`docs/plans/v0.41-request-flow.md`, `docs/developer/test-strategy.md`,
`DEVELOPMENT.md`, `docs/plans/v0.45-plan.md`, and
`docs/plans/v0.45-request-flow.md`.

Status: implemented as the 0.45.1 developer-tooling patch release inserted
after v0.45.0 and before v0.46. It adds no operator-facing assistant
capability.

## Purpose

Close the v0.45 remediation follow-up on slow, opaque gate runs.
v0.41 made precommit a gate matrix in the development methodology, but
the implemented root `mix precommit` alias remained the v0.40-style
monolithic fallback. v0.45 closeout exposed the gap again: the final
release gate was correct and green, but long phases produced little
progress output and the wrapper still delegated to the monolithic
precommit alias before Dialyzer.

v0.45.1 makes the implemented commands match ADR 0049 more closely:

- commit-time feedback is fast and intentionally non-release evidence;
- push/local-handoff feedback is high coverage and partition-aware;
- release feedback remains exhaustive and authoritative;
- every long phase is named, timed, and streamed or summarized as it runs;
- `mix allbert.test release` no longer delegates to raw `mix precommit`;
- historical release strictness is preserved.

## Root Cause

The slow path is not primarily a Python-style `pre-commit` hook scanning
the whole repository. The current root alias is a full local release
sequence:

1. `compile --warnings-as-errors`
2. `deps.unlock --unused`
3. `format --check-formatted`
4. `credo --strict`
5. all core app tests
6. all web app tests
7. StockSage plugin tests
8. Telegram, email, and notes-files plugin tests

Then `mix allbert.test release` runs that whole alias and adds Dialyzer.
The v0.45 release evidence showed the real long poles: core tests, web
LiveView tests, StockSage tests, and then Dialyzer. The earlier v0.41
strategy docs already identified `WorkspaceLiveTest` as the web long
pole and implemented partitioned lane helpers, but the release wrapper
still uses the old precommit oracle.

## Research Grounding

Current tool documentation supports the v0.45.1 direction:

- Mix test supports selecting explicit files/directories, umbrella app
  paths, `--partitions` with `MIX_TEST_PARTITION`, `--max-cases`,
  `--stale`, `--slowest`, and `--slowest-modules`.
- Mix aliases can sequence tasks, but invoking the same task repeatedly
  inside aliases has task-reenable semantics that make a dedicated Mix
  task safer for phase orchestration.
- The `pre-commit` framework distinguishes staged-file commit hooks from
  all-file CI-style runs, supports file/ref scoping, hook stages, and
  cache homes.
- Git hooks distinguish commit-time and push-time enforcement; pre-commit
  hooks are bypassable with `--no-verify`, while pre-push hooks can block
  a push.

Allbert should not import the Python `pre-commit` framework just to solve
this. The correct local implementation is a Mix-native gate runner that
uses the existing v0.41 lane taxonomy and ExUnit partitioning.

## Guardrails

- No coverage reduction. Any command that is release evidence must cover
  the same or stronger suite than the v0.45 final release gate.
- No hidden external smokes. Browser, Docker, provider, or real MCP smokes
  remain explicit `external-smoke` gates.
- No git hook installation in this milestone. Commands are documented and
  runnable; optional hook installation can be a later operator choice.
- No semantic reliance on staged-file heuristics for release evidence.
  Changed-file or `--stale` checks are commit-feedback only.
- No new runtime authority, Settings Central keys, Security Central
  permissions, Resource Access classes, workspace panels, or operator data
  stores.
- Gate evidence paths must derive from a temp Allbert Home unless the
  operator explicitly provides `ALLBERT_HOME` for release evidence.

## Command Contract

v0.45.1 implements the command contract below.

| Command | Intent | Release evidence? | Expected shape |
| --- | --- | --- | --- |
| `mix allbert.test docs` | Docs-only hygiene. | No | Existing docs gate. |
| `mix allbert.test focused -- <files...>` | Milestone-focused tests. | No | Existing focused gate, with clearer phase timing. |
| `mix allbert.test commit` | Fast commit-time confidence. | No | Static checks plus changed/stale-aware focused guidance. |
| `mix precommit` | Compatibility shortcut for the commit gate. | No after v0.45.1 | Routes to `mix allbert.test commit`; docs stop treating it as release evidence. |
| `mix allbert.test prepush` | High-coverage local handoff before push. | No | Static checks plus partitioned high-coverage lanes. |
| `mix allbert.test release` | Authoritative release handoff. | Yes | Named full-suite phases plus Dialyzer, with streaming/timing/evidence. |
| `mix allbert.test release.v045` | Deterministic v0.45 feature gate. | Yes for v0.45 surface | Existing deterministic marketplace gate. |
| `mix allbert.test external-smoke -- <name>` | Machine-dependent smokes. | Only when plan names it | Existing explicit opt-in smoke shape. |

The compatibility line is the core semantic change. Before v0.45.1,
`mix precommit` meant "full release-style local gate." After v0.45.1,
`mix precommit` means "commit-time gate." Release evidence is always
`mix allbert.test release` or a version-specific release gate.

## Release Phase Contract

`mix allbert.test release` becomes a direct phase runner. It must not
shell out to `mix precommit`.

Required phases:

1. `static_compile`: `MIX_ENV=test mix compile --warnings-as-errors`
2. `deps_unused`: `mix deps.unlock --unused`
3. `format`: `mix format --check-formatted`
4. `credo`: `mix credo --strict`
5. `core_tests`: full core app test suite from the core app cwd
6. `web_tests`: full web app test suite from the web app cwd
7. `stocksage_tests`: StockSage plugin tests from the core app cwd
8. `channel_plugin_tests`: Telegram, email, and notes-files plugin tests
9. `dialyzer`: `MIX_ENV=test mix dialyzer`

Each phase records:

- phase id
- cwd
- command args with secrets redacted
- start timestamp
- finish timestamp
- duration milliseconds
- exit status
- test summary when present
- redacted output tail
- full redacted output log path
- ExUnit seed values when present
- failed-test manifest snapshots for failed Mix test phases

The release gate prints each phase start immediately and prints either
streamed child output or a bounded progress heartbeat while the child
process is running. It writes a JSON evidence artifact under
`<ALLBERT_HOME>/release_evidence/gates/` when an Allbert Home is
provided or under a temp Allbert Home when none is provided. Full
redacted phase logs are stored beside the JSON evidence in a per-run
artifact directory. Failed Mix test phases also snapshot the relevant
`.mix_test_failures` files so a one-line test failure summary remains
diagnosable after terminal scrollback is gone.

## Milestones

### M1: Baseline, Inventory, And Spec Lock

Status: implemented.

Goal: lock current behavior before changing commands.

Implementation tasks:

- Record the current root `mix precommit` alias shape and current
  `mix allbert.test release` delegation.
- Run or preserve a fresh baseline timing command for:
  - `MIX_ENV=test mix allbert.test release`
  - `MIX_ENV=test mix allbert.test fast-local`
  - `MIX_ENV=test mix allbert.test fast-local --core-lanes --stocksage-lanes --web-lanes --partitions 4`
  - current slowest-module probes for core, web, and StockSage where
    practical.
- Extend `docs/developer/test-strategy.md` with a v0.45.1 benchmark
  section using the same evidence shape as v0.41.
- Add regression tests that assert `release` is still the only broad
  release-evidence gate.

Development-lane annotation:

- Parallel workstreams: doc updates, baseline command capture, task-test
  fixture design.
- Serial barriers: full release baseline, slowest-module probes, and any
  command touching shared Mix build artifacts.
- Focused gate: existing Mix task tests for `allbert.test` plus docs
  hygiene.
- Rejoin: baseline numbers are recorded before M2 changes command
  behavior.

### M2: Timed Phase Runner And Progress Output

Status: implemented.

Goal: make long gates observable without changing coverage.

Implementation tasks:

- Add a small phase runner in `Mix.Tasks.Allbert.Test` or a private
  helper module under the core app.
- Replace opaque `System.cmd` use for long phases with a streaming or
  heartbeat runner that prints phase start, bounded child output, finish
  status, and duration.
- Keep redaction before output tails are persisted.
- Add JSON evidence writing for release and prepush gates.
- Add tests for success, failure, redaction, duration fields, and bounded

Development-lane annotation:

- Parallel workstreams: runner implementation, evidence schema tests,
  docs/request-flow examples.
- Serial barriers: none beyond focused Mix task tests; full release is
  not required until M3/M5 because M2 should preserve commands.
- Focused gate:
  `MIX_ENV=test mix test apps/allbert_assist/test/mix/tasks/allbert_test.exs`
  plus any new phase-runner unit tests.
- Rejoin: release command still delegates to the old coverage path, but
  its observable phases are now timed.

### M3: Direct Release Phase Decomposition

Status: implemented.

Goal: remove `release -> precommit -> dialyzer` delegation while keeping
coverage equivalent or stronger.

Implementation tasks:

- Replace `release()` delegation with the explicit phase list in
  §Release Phase Contract.
- Preserve child cwd behavior: core and StockSage/plugin tests run from
  the core app cwd; web tests run from the web app cwd.
- Preserve owned `ALLBERT_HOME` and `DATABASE_PATH` behavior for release
  phases.
- Ensure the web test phase receives the same migration setup that the
  monolithic child alias previously provided.
- Add tests that fail if `release` shells out to `mix precommit`.
- Add tests that verify phase ordering, cwd, env ownership, cleanup, and
  failure short-circuit behavior.

Development-lane annotation:

- Parallel workstreams: phase contract implementation, cwd/env tests,
  request-flow evidence updates.
- Serial barriers: web migration setup, full release gate.
- Focused gate:
  `MIX_ENV=test mix test apps/allbert_assist/test/mix/tasks/allbert_test.exs`
  and targeted release-runner tests.
- Release gate: `MIX_ENV=test mix allbert.test release` must pass before
  M4 changes `mix precommit` semantics.
- Rejoin: compare M3 release counts/durations against the M1 baseline;
  any coverage delta blocks M4.

### M4: Commit And Prepush Gate Semantics

Status: implemented.

Goal: make everyday commit/push workflow fast without weakening release
handoff.

Implementation tasks:

- Add `mix allbert.test commit`.
  - Always run static checks needed for code changes.
  - Run `mix allbert.test docs` when the diff is docs-only.
  - Print changed-file guidance and focused-test suggestions.
  - Optionally run Mix `--stale` suites for core/web/plugin owners when
    changed files map cleanly to an owner.
  - Never claim release readiness.
- Add `mix allbert.test prepush`.
  - Run static checks.
  - Run `fast-local --core-lanes --stocksage-lanes --web-lanes
    --partitions N` with a documented default partition count.
  - Run Dialyzer only when the plan chooses it; default prepush should be
    high-signal without becoming the full release gate.
- Rewire root `mix precommit` to the commit gate and update docs that
  currently use `mix precommit` as release evidence.
- Keep `mix allbert.test release` as the only broad release wrapper.

Development-lane annotation:

- Parallel workstreams: changed-file resolver, commit/prepush task tests,
  doc updates.
- Serial barriers: root alias change and docs that mention `mix precommit`
  as current release evidence.
- Focused gate:
  `MIX_ENV=test mix test apps/allbert_assist/test/mix/tasks/allbert_test.exs`
  plus root alias smoke through `mix precommit`.
- Rejoin: `mix precommit` must be fast, non-release evidence; `mix
  allbert.test release` remains green.
