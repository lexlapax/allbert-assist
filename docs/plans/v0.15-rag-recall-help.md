# v0.15 - Vector RAG, recall, and help foundation

Status: M0-M6 implemented; M7 collection-aware user RAG milestone added before
release closeout

v0.15 is the retrieval foundation release. It gives Allbert one bounded,
rebuildable way to retrieve operator docs, CLI commands, settings, skill
metadata, durable memory, facts, episode recall, and session-derived working
history before any new ambient ingestion source expands the corpus.

The release now also makes the retrieval substrate collection-aware. Existing
Allbert-owned retrieval becomes the default `system` collection family, while
operator-requested task/corpus retrieval becomes explicit `user` collections
that share the same SQLite, FTS, vector, maintenance, and prompt-rendering
machinery without sharing trust boundaries.

This release is intentionally vector-first for RAG. SQLite FTS remains the
provider-free lexical baseline and fallback, but v0.15 is not fake-only and not
lexical-only: local Ollama embeddings plus `sqlite-vec` are release scope. Fake
embeddings remain only for deterministic tests.

Growth-loop ingestion, browser capture, proxy/PAC mode, staged ingestion review,
and ingestion-driven learning refresh are parked in [future-plans.md](future-plans.md).
They return only after this RAG service can explain, bound, and maintain the
context Allbert already owns.

## Goals

- Add `RagService` in `allbert-kernel-services`; core owns config and DTOs only.
- Store all RAG artifacts in a derived SQLite database under
  `~/.allbert/index/rag/rag.sqlite`.
- Use SQLite FTS for lexical retrieval and `sqlite-vec` for local vector search.
- Use Ollama `/api/embed` as the first real local embedding provider, defaulting
  to `embeddinggemma`.
- Keep markdown files, generated command/settings descriptors, skill metadata,
  durable memory, and session-derived recall artifacts as source of truth.
- Make help/meta turns retrieve current operator docs, command descriptors,
  settings descriptors, and runbook snippets.
- Make memory-query/recall turns retrieve durable memory, facts, episodes, and
  bounded session-derived working history.
- Add daemon-owned RAG maintenance scheduling and channel-visible RAG status.
- Add logical RAG collections so system sources and user-provided task corpora
  can be indexed, searched, maintained, and prompt-gated independently.
- Add a built-in RAG skill as a thin user/operator interface over
  kernel-services collection calls.

## Non-goals

- No growth-loop ingestion endpoint or browser capture in v0.15.
- No automatic promotion of retrieved, staged, or session-derived content.
- No raw transcript injection as durable memory or ordinary prompt context.
- No replacement of the v0.14.3 schema-bound router.
- No provider-native tool calling unless a later focused release needs it.
- No hosted embedding provider in v0.15. OpenAI or other hosted embeddings are
  future work and must be cost/privacy-gated separately.
- No `sqlite-vss`; v0.15 uses `sqlite-vec` and tracks official SQLite `vec1` as
  a future candidate.
- No ambient web crawling or browser capture in v0.15 M7. URL ingestion is
  allowed only for explicit user collection sources, with HTTP(S)-only fetch
  policy, robots.txt checks, SSRF defenses, and tight page/byte/time caps.
- No automatic prompt injection of user collections. User collections are
  available only when explicitly selected for a task, session, search, or RAG
  skill invocation.

## Web Ingestion Research Anchors

M7 URL ingestion follows established HTTP and crawler safety guidance:

- Use SSRF defenses from the
  [OWASP SSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html):
  normalize and validate URLs with a parser, restrict schemes, reject internal
  address ranges, and revalidate DNS/IP posture after redirects.
- Honor the [Robots Exclusion Protocol, RFC 9309](https://www.rfc-editor.org/rfc/rfc9309)
  for automated URL fetches.
- Store HTTP validators and use conditional refresh behavior aligned with
  [HTTP Semantics, RFC 9110](https://www.rfc-editor.org/rfc/rfc9110), especially
  ETag/If-None-Match and Last-Modified/If-Modified-Since.

## Architecture

`allbert-kernel-core` adds only stable contracts:

- `RagConfig`, `RagVectorConfig`, `RagIndexConfig`, and settings descriptors.
- DTOs for collection type/name filters, source kinds, chunk provenance,
  status, search requests/results, and rebuild summaries.
- Protocol DTOs needed by CLI/TUI/Telegram to read status and search results.

`allbert-kernel-services` owns concrete behavior:

- `RagService` orchestration.
- source collectors for docs, command catalog, settings catalog, skill metadata,
  memory, facts, episodes, and session-derived recall;
- collection cataloging for system and user collections;
- local trusted-root and explicit HTTP(S) ingestion for user collections;
- deterministic chunking and content hashing;
- `SqliteRagStore` with FTS and `sqlite-vec` registration;
- `EmbeddingProvider` implementations for Ollama and fake tests;
- hybrid retrieval, prompt-context rendering, status, rebuild, and stale checks.

The daemon owns runtime scheduling and channel fan-out:

- a non-LLM `RagMaintenanceService` runs rebuilds directly through `RagService`;
- RAG maintenance does not run as a prompt-authored scheduled job and does not
  depend on a model producing a tool call;
- channel surfaces read status/search through daemon protocol v7.

Skills remain review and curation surfaces:

- `memory-curator` and future skills may call RAG/search tools or interpret
  staged results, but they do not own indexing, embedding, source policy, or
  prompt injection limits.
- The built-in RAG skill exposes create/list/show/delete, ingest, rebuild,
  search, and attach-to-task operations by calling kernel-services. It does not
  bypass filesystem trust, collection prompt eligibility, review-only gates, or
  daemon-owned maintenance.

## Turn Control Flow

v0.15 inserts RAG into the existing kernel turn pipeline. It does not create a
second agent and it does not let retrieval draft or execute actions.

Root interactive turns run in this order:

1. The channel attaches to a daemon session and calls `Kernel::run_turn`.
2. The kernel starts the turn, enforces daily/turn budgets, resets per-turn
   state, emits `ClassifyingIntent`, and opens trace spans.
3. If `rag.enabled = true`, the kernel may request a cheap pre-router hint from
   `RagService`. This hint is lexical-only in v0.15 and can query only
   `operator_docs`, `command_catalog`, `settings_catalog`, and bounded
   `skills_metadata`. It returns source ids, titles, and short snippets for
   router context; it never returns full prompt chunks and never runs query
   embeddings.
4. The existing schema-bound intent router runs. It remains the only authority
   for `task`, `chat`, `schedule`, `memory_query`, and `meta` classification
   plus guarded action drafts.
5. If the router returns a high-confidence terminal action such as schedule
   mutation or explicit memory capture, the existing guarded tool path runs
   before root-agent prompt assembly. RAG is not consulted to authorize that
   action.
6. If no terminal router action completes the turn, the kernel appends the user
   message to session history and enters prompt preparation.
7. Existing bootstrap prompt sections are loaded through `BeforePrompt`.
8. The kernel builds the memory synopsis and ephemeral/session summary without
   Tantivy snippet prefetch for RAG-owned source kinds.
9. Post-router RAG retrieval runs only when the resolved intent or explicit
   operator request allows it:
   - `meta` and help-like turns retrieve docs, commands, settings, runbooks, and
     skill metadata.
   - `memory_query` and explicit recall turns retrieve durable memory, approved
     facts, labelled episodes, and session summaries.
   - `task` turns retrieve RAG only when router posture or direct wording asks
     for local/project/operator context.
   - `chat` turns normally skip RAG unless they contain explicit recall/help
     wording.
10. RAG snippets are rendered into the system prompt as labelled evidence with
    source ids, source kind, title/path, score posture, and freshness. They are
    bounded by `rag.max_chunks_per_turn`, `rag.max_chunk_bytes`, and
    `rag.max_prompt_bytes`.
11. The root LLM agent is called with the normal tool catalog plus any active
    skill bodies and RAG evidence.
12. If the model calls tools, the existing tool loop runs. After external
    evidence tools, the runtime may schedule one RAG refresh for a later round
    in the same turn when the output contains memory/help/local-context cues and
    `rag.refresh_after_external_evidence = true`.
13. The turn finalizes through existing hooks, telemetry, traces, activity, and
    session persistence.

Scheduled jobs follow the same prompt-time RAG rules when their resolved intent
or job prompt needs help/local context. RAG index maintenance itself does not
run as a prompt-authored job.

## Memory and RAG Boundary

v0.15 keeps the existing memory synopsis but moves retrieved memory snippets to
RAG.

- `MEMORY.md` head, today's daily note, and ephemeral session summary remain
  lightweight synopsis inputs.
- Tantivy-backed `memory search` remains the user-visible memory CLI/tool
  contract and explicit search implementation.
- Prompt-time retrieval of durable memory, facts, episodes, and session-derived
  summaries is owned by RAG for eligible turns.
- The turn assembler must not inject both Tantivy prefetch snippets and RAG
  snippets for the same durable/fact/episode/session source kind.
- Staged memory stays out of ordinary RAG prompt context. It is visible only to
  explicit review commands/tools and must be labelled `review_only`.
- Session-derived recall is always labelled `working_history`; it never renders
  as approved durable memory.
- The v0.11 `memory.semantic.*` JSON seam is deprecated once RAG vectors land.
  Existing settings should continue to parse, but setup and settings output
  should point operators to `[rag.vector]`.

## Model-Callable RAG Tool

v0.15 exposes one read-only model tool, `search_rag`, in addition to internal
prompt-time retrieval and CLI/channel commands.

Tool schema:

```json
{
  "type": "object",
  "required": ["query"],
  "properties": {
    "query": { "type": "string", "maxLength": 4096 },
    "sources": {
      "type": "array",
      "items": { "type": "string" }
    },
    "collection_type": { "enum": ["system", "user"] },
    "collections": {
      "type": "array",
      "items": { "type": "string" }
    },
    "mode": { "enum": ["hybrid", "vector", "lexical"] },
    "limit": { "type": "integer", "minimum": 1, "maximum": 10 },
    "include_review_only": { "type": "boolean" }
  },
  "additionalProperties": false
}
```

Rules:

- The tool is read-only and cannot mutate memory, jobs, settings, schedules, or
  index state.
- Results are capped by `limit`, `rag.max_chunk_bytes`, and the normal tool
  output byte caps.
- `include_review_only = true` is ignored unless the resolved intent is an
  explicit review/curation surface.
- The tool may search only source kinds enabled in `[rag].sources`.
- `collection_type` and `collections` filter retrieval. When omitted, the tool
  searches the same prompt-eligible `system` collections as current v0.15
  behavior.
- `user` collections are available only when explicitly attached to the current
  task/session or directly named in an operator-approved RAG skill/search
  request.
- Query embeddings obey the same eligibility, timeout, and fallback rules as
  internal post-router RAG.
- Tool results must include collection type/name, source kind, source id, chunk
  id, title/path, retrieval mode, vector posture, and score explanation.

## Collection Model

v0.15 M7 makes RAG collection-aware. A collection is a logical set of sources
and chunks that share prompt eligibility, review posture, privacy tier, source
origin, lifecycle timestamps, and optional vector model state. Collections are
logical rows inside the one derived SQLite RAG database, not separate database
files.

Collection types:

- `system`: Allbert-owned collections built from operator docs, generated
  command/settings descriptors, skill metadata, memory, facts, episode recall,
  session summaries, and staged-review sources. These preserve the current
  v0.15 turn-flow rules and are prompt-eligible only where those rules already
  allow them.
- `user`: operator-created task or corpus collections. They are never
  prompt-eligible by default and enter prompts only when explicitly selected by
  the operator, attached to the current task/session, or named in a scoped RAG
  skill/search call.

System collection mapping:

| Collection | Source kinds | Source URI |
| --- | --- | --- |
| `operator_docs` | `operator_docs` | `allbert://docs/operator` |
| `commands` | `command_catalog` | `allbert://generated/commands` |
| `settings` | `settings_catalog` | `allbert://generated/settings` |
| `skills` | `skills_metadata` | `allbert://skills/metadata` |
| `memory` | `durable_memory` | `allbert://memory/durable` |
| `facts` | `fact_memory` | `allbert://memory/facts` |
| `episodes` | `episode_recall` | `allbert://sessions/episodes` |
| `sessions` | `session_summary` | `allbert://sessions/summaries` |
| `staged_review` | `staged_memory_review` | `allbert://memory/staged-review` |

User collection rules:

- Creation requires an explicit collection name and at least one explicit
  source URI.
- User collection names are operator-facing and must be stable, normalized, and
  unique within `collection_type = 'user'`.
- Source URIs record origin; v0.15 M7 accepts `file://`, `dir://`, `https://`,
  and explicitly allowed `http://`.
- Local file/directory sources must remain under trusted filesystem roots.
- Web URL sources use exact URL ingestion by default. Optional same-origin
  expansion is capped and disabled unless the operator passes an explicit crawl
  depth/page limit.
- Web fetches use HTTP `GET`/`HEAD` only, a dedicated Allbert user agent,
  content-type allowlists, byte/page/time caps, redirect validation, and
  conditional requests using `ETag`/`Last-Modified` when available.
- Web fetches must reject unsupported schemes, embedded credentials, localhost,
  loopback, link-local, private, multicast, broadcast, and cloud-metadata
  addresses. Every redirect target is revalidated before fetch.
- Web ingestion honors the Robots Exclusion Protocol for automated fetches. A
  robots disallow records the source as skipped/degraded rather than fetching.
- `last_ingested_at` records when source files were scanned and source rows were
  refreshed.
- `last_indexed_at` records when chunk/FTS/vector rows were successfully
  rebuilt for the collection.
- `last_accessed_at` updates on explicit collection search, prompt retrieval,
  or RAG skill attachment.
- Deleting a user collection deletes its derived SQLite rows only. It does not
  delete source files.
- Staged/review-only collections remain isolated even when named explicitly
  unless the current intent/surface is a review surface.

## Source Model

Every indexed item has a source kind, stable source id, content hash,
provenance, and source-of-truth path or descriptor id.

Stable source inventory:

| Source kind | Default collection | Collector | Source id formula | Prompt eligibility | Privacy tier | Stale trigger |
| --- | --- | --- | --- | --- | --- | --- |
| `operator_docs` | `system/operator_docs` | `DocsCollector` | `docs:<repo-relative-path>#<heading-slug-or-root>` | help/meta by default; task only on explicit request | local_docs | file mtime/hash change |
| `command_catalog` | `system/commands` | `CommandCatalogCollector` | `command:<descriptor-id>` | help/meta and pre-router hints | generated_public | descriptor hash change |
| `settings_catalog` | `system/settings` | `SettingsCatalogCollector` | `setting:<setting-key>` | help/meta and pre-router hints | generated_local | descriptor hash change |
| `skills_metadata` | `system/skills` | `SkillMetadataCollector` | `skill:<install-scope>:<skill-name>` | help/meta; task when skill routing needs it | local_metadata | skill manifest/frontmatter hash change |
| `durable_memory` | `system/memory` | `MemoryCollector` | `memory:<memory-relative-path>` | memory_query and explicit recall | user_memory | manifest entry/content hash change |
| `fact_memory` | `system/facts` | `FactCollector` | `fact:<memory-relative-path>#<fact-id>` | memory_query and explicit recall | user_memory | source memory/fact metadata hash change |
| `episode_recall` | `system/episodes` | `EpisodeCollector` | `episode:<session-id>#<turn-timestamp-or-ordinal>` | memory_query and explicit recall | working_history | `sessions/<id>/turns.md` hash change |
| `session_summary` | `system/sessions` | `SessionSummaryCollector` | `session:<session-id>#summary` | memory_query and explicit recall | working_history | `sessions/<id>/meta.json` or bounded-turn hash change |
| `staged_memory_review` | `system/staged_review` | `StagedReviewCollector` | `staged:<staged-id>` | explicit review search only | review_only | staged/rejected state change |
| `user_document` | `user/<collection-name>` | `UserCollectionCollector` | `user:<collection-name>:<normalized-source-uri>#<relative-path-or-root>` | explicit task/session attachment only | user_supplied | file mtime/hash change |
| `web_url` | `user/<collection-name>` | `WebUrlCollectionCollector` | `web:<collection-name>:<canonical-url>` | explicit task/session attachment only | user_supplied_web | ETag/Last-Modified/content hash change |

Included in v0.15:

- `operator_docs`: `docs/operator/`, `docs/onboarding-and-operations.md`, and
  release-relevant runbook sections. Historical plans/ADRs are not prompt-time
  help sources unless explicitly queried through CLI search.
- `command_catalog`: entries from `command_catalog`.
- `settings_catalog`: entries from `settings_catalog`, including safety notes.
- `skills_metadata`: bundled and installed skill name, description, intents,
  allowed tools, provenance, and enablement state. Skill bodies are excluded
  unless a skill is active in the current turn.
- `durable_memory`: approved markdown memory under `memory/notes/` and related
  durable memory paths.
- `fact_memory`: approved temporal facts extracted from promoted memory.
- `episode_recall`: bounded role-labelled working-history chunks derived from
  `sessions/<id>/turns.md`.
- `session_summary`: derived per-session summaries using metadata from
  `sessions/<id>/meta.json` and bounded turn counts; these are working history,
  not approved durable memory.
- `user_document`: explicit user collection content ingested from local
  trusted files/directories through the built-in RAG skill or local operator
  RAG commands.
- `web_url`: explicit HTTP(S) user collection content ingested through the
  built-in RAG skill or local operator RAG commands with robots, redirect,
  SSRF, byte, page, and content-type controls.

Excluded in v0.15:

- raw trace spans and provider payloads;
- staged memory in ordinary prompt context;
- rejected/trash memory;
- skill reference files and skill bodies for inactive skills;
- future staged ingestion records.
- ambient link traversal, broad crawling, login-required pages, JavaScript
  execution, non-HTTP schemes, and binary/document parsing that lacks a pinned
  extractor.

Staged memory can be searched only from explicit review surfaces. Session
recall is always labelled as working history and must never be rendered as
approved memory.

## SQLite Store

The M7 schema is schema version 2. Implementation may adjust minor SQLite
syntax for crate APIs, but not table ownership or column meaning. Because
`rag.sqlite` is a derived artifact, schema v1 databases may be deleted and
rebuilt from source truth instead of migrated row-by-row when that is simpler
and safer.


Rebuild behavior:

- Rebuilds run in a temp SQLite file for full rebuilds, then atomically replace
  `rag.sqlite` when complete.
- Schema v1 databases may be treated as stale derived artifacts and rebuilt
  into schema v2 with default `system` collections.
- System startup/scheduled rebuilds default to system collections only. User
  collections rebuild only on explicit CLI/REPL/TUI/RAG-skill request or when
  explicitly included in a future maintenance setting.
- Search/rebuild/status requests accept optional collection filters:
  `collection_type` and `collections`. Omitted filters preserve current system
  prompt-eligible behavior.
- Web-source stale checks use conditional requests with stored `ETag` and
  `Last-Modified` when available, content hash fallback otherwise, and must not
  fetch content when robots or URL safety validation denies access.
- Stale-only rebuilds update sources/chunks in transactions of at most
  `rag.index.max_chunks_per_run` chunks.
- FTS rows are maintained transactionally with chunk writes.
- Vector rows are written only after chunk rows commit. If vector indexing fails,
  lexical rows stay usable and affected chunks remain `embedding_state =
  'missing'` or `stale`.
- Schema version, active provider/model/dimension, sqlite-vec version, and last
  successful source hash live in `rag_meta`.
- GC removes orphan chunks/sources after source deletion and may run `VACUUM`
  only from explicit `rag gc`, not during ordinary turns.
- `last_accessed_at` updates after explicit collection search, prompt
  retrieval, and RAG skill task attachment.

Model changes invalidate all embeddings. The lexical tables remain usable while
vectors are stale; status must show degraded vector posture clearly.

## Chunking

- Chunk markdown by heading and paragraph boundaries first; split oversized
  blocks by sentence-ish boundaries when possible.
- Default target: 800-1200 bytes per chunk, hard capped by
  `rag.max_chunk_bytes`.
- Chunk ids are stable hashes of source kind, source id, content hash, ordinal,
  collection type/name, and active embedding model key.
- Chunks carry titles, source path/descriptor id, section heading, tags, and
  prompt eligibility.
- Duplicate chunks with identical normalized text under the same source kind
  are deduplicated at retrieval time, not removed from source accounting.

## Vector Stack

Local vector RAG uses:

- embedding provider: Ollama `/api/embed`;
- default embedding model: `embeddinggemma`;
- vector database backend: `sqlite-vec` statically linked through its Rust
  crate and registered with `rusqlite`;
- distance metric: cosine;
- batch embedding for indexing; single query embedding for vector searches;
- lexical fallback if Ollama is unavailable or a query embedding fails.

Fake embeddings are allowed only in unit/integration tests and explicit dev
fixtures. A v0.15 release candidate is not considered complete until a local
Ollama vector smoke passes on a maintainer machine.

## Embedding Runtime Contract

- Dimension is discovered by embedding a fixed probe string with the configured
  provider/model before vector schema creation.
- The discovered dimension is stored in `rag_meta`. If a later query/index run
  returns a different dimension, vectors are marked stale and lexical fallback
  is used.
- Indexing batches use `rag.vector.batch_size`; query embeddings are single
  input calls.
- Query embedding input is capped at 4096 bytes after whitespace normalization.
- Query embeddings have timeout `rag.vector.query_timeout_s`; indexing batches
  share `rag.vector.index_timeout_s` across the rebuild phase.
- Ollama calls retry at most two times for transient transport failures, with
  250ms then 1s backoff. Model-missing and dimension-mismatch failures are not
  retried.
- At most two query embeddings run concurrently per daemon process in v0.15.
  Additional query requests fall back to lexical when the semaphore cannot be
  acquired before the query timeout.
- Vector indexing never runs inside ordinary turn preparation. It runs only from
  manual rebuild, startup missing-DB rebuild, or scheduled maintenance.
- `rag doctor` reports base URL reachability, model availability, discovered
  dimension, stored dimension, stale vector count, degraded reason, and the
  remediation command `ollama pull embeddinggemma` when the model is missing.

Official SQLite `vec1` stays tracked because it is promising and maintained by
SQLite, but v0.15 does not select it because it is new and its current
documentation still calls out testing and optimization gaps.

## Retrieval Behavior

Default retrieval mode is `hybrid` when vectors are healthy:

- run vector KNN and lexical FTS in parallel;
- default candidate counts: 30 vector, 30 lexical;
- merge by reciprocal-rank fusion with vector weight `0.70` and lexical weight
  `0.30`;
- deduplicate by chunk id and normalized source/text key;
- return at most `rag.max_chunks_per_turn` prompt chunks;
- enforce `rag.max_chunk_bytes` and `rag.max_prompt_bytes` after merge;
- include path, title, source kind, score, and reason in operator-visible
  results.

If vectors are disabled, missing, stale, or temporarily failing, retrieval falls
back to lexical FTS and reports vector posture as degraded. Query embedding is
allowed only for explicit RAG search, help/meta turns, memory-query turns, and
router-approved retrieval. It must not run for every ordinary chat turn.

RAG provides evidence, not authority:

- the v0.14.3 router remains the authority for intent and guarded action drafts;
- RAG snippets cannot create mutations, approvals, schedule changes, or memory
  promotions;
- RAG hints before the router are limited to command/help/settings sources,
  source ids, titles, and short snippets;
- full prompt RAG context is added only after routing or explicit user request.

Fresh profiles should show a setup question for local vector RAG. If Ollama is

default, but vectors require an explicit settings change or setup refresh.

## Operator and Channel Surface

CLI is the authoritative operator escape hatch:

```bash
allbert-cli rag status [--json]
allbert-cli rag doctor [--json]
allbert-cli rag rebuild [--stale-only] [--source <kind>] [--vectors|--no-vectors]
allbert-cli rag search <query> [--source <kind>] [--mode <hybrid|vector|lexical>] [--json]
allbert-cli rag gc [--dry-run]
```

Interactive surfaces:

- REPL/TUI add `/rag status`, `/rag search <query>`, and `/rag rebuild
  --stale-only`.
- Telegram adds `/rag status` and `/rag search <query>` only. Rebuild remains
  CLI/TUI/REPL in v0.15 because it can be long-running and host-local.
- Prompt-time help/meta answers use RAG automatically when enabled and cite
  compact source labels.
- Status-line support adds optional `rag` item showing `rag ok`, `rag stale`,
  `rag lexical`, or `rag degraded`.

Protocol v7 adds exact RAG messages. v2-v6 clients attach normally and do not
receive v7-only RAG messages.

Client messages:

- `ClientMessage::RagStatus { json: bool }`
- `ClientMessage::RagSearch { query, sources, mode, limit, include_review_only }`
- `ClientMessage::RagRebuildStart { stale_only, sources, include_vectors }`
- `ClientMessage::RagRebuildCancel { run_id }`
- `ClientMessage::RagGc { dry_run }`

Server messages:

- `ServerMessage::RagStatus(RagStatusPayload)`
- `ServerMessage::RagSearchResults(RagSearchResultsPayload)`
- `ServerMessage::RagRebuildStarted(RagRebuildRunPayload)`
- `ServerMessage::RagRebuildProgress(RagRebuildProgressPayload)`
- `ServerMessage::RagRebuildFinished(RagRebuildRunPayload)`
- `ServerMessage::RagRebuildCancelled(RagRebuildRunPayload)`
- `ServerMessage::RagRebuildError(RagRebuildErrorPayload)`
- `ServerMessage::RagGcResult(RagGcResultPayload)`

Payloads must carry `run_id` where relevant, source counts, chunk counts, vector
counts, skipped counts, elapsed time, vector posture, degraded reason, and the
active provider/model/dimension. Search result payloads include bounded snippets
and never include review-only chunks unless explicitly requested and authorized.

## Scheduled Maintenance

RAG indexing is deterministic service work, not a model job.

- The daemon owns a `RagMaintenanceService`.
- It reuses the bounded schedule DSL semantics where practical, but it does not
  create prompt-authored job definitions under `jobs/definitions`.
- It records run history in `rag_index_runs` and surfaces current work through
  daemon activity.
- Only one rebuild runs at a time; manual rebuilds and scheduled maintenance
  share the same lock.
- Manual rebuild requests win over scheduled maintenance. A scheduled run that
  finds the lock held records `status = skipped` with reason `lock_held`.
- Missed scheduled runs coalesce into one stale-only run. The daemon must not
  replay a storm of missed RAG maintenance windows after sleep or shutdown.
- Startup missing-DB rebuild runs lexical first. If vector indexing is enabled
  and Ollama is healthy, vectors run as a second phase under the same `run_id`.
- Shutdown requests cancel in-flight scheduled rebuilds after
  `rag.index.shutdown_grace_s`. Manual rebuild cancellation is exposed through
  `RagRebuildCancel`.
- Scheduled vector phase skips, rather than fails the whole run, when Ollama is
  unavailable or the model is missing. The run records degraded vector posture.
- Transient embedding failures use the embedding retry policy; after retries are
  exhausted, remaining vector work is skipped and lexical index state remains
  successful.
- Stale detection must run before a scheduled rebuild acquires the heavy rebuild
  path. If no stale sources exist, record `status = skipped` with reason
  `nothing_stale`.
- Scheduled runs default to stale-only and may skip vector work if Ollama is
  unavailable.
- Scheduled runs default to system collections only. User collection
  maintenance is explicit in v0.15 M7 so an operator-created corpus cannot
  quietly consume indexing work after the task is finished.
- A missing SQLite file can trigger startup rebuild when
  `rag.index.run_on_startup_if_missing = true`; this rebuild should be lexical
  first, then vector if enabled and available.

The existing jobs system remains for prompt-authored recurring work. v0.15 does
not add a bundled `rag-index-refresh` prompt job because indexing should not
depend on an LLM deciding to call the right tool.

## Exit Criteria

- `allbert-cli rag rebuild --no-vectors` restores lexical RAG after deleting
  `~/.allbert/index/rag/rag.sqlite`.
- `allbert-cli rag rebuild --vectors` builds a real Ollama-backed vector index
  when `embeddinggemma` is available.
- `allbert-cli rag search "configure Telegram" --mode hybrid` returns bounded
  operator-doc snippets with source labels and vector/lexical scoring metadata.
- `help me configure Telegram` retrieves current operator docs before the main
  answer and cites compact source labels.
- Durable memory and approved facts appear in RAG; staged memory does not enter
  ordinary prompt context.
- Session-derived recall is searchable and labelled as working history, not
  durable memory.
- A missing or stale vector index degrades to lexical retrieval with an
  operator-visible warning.
- Scheduled RAG maintenance can run stale-only without invoking an LLM turn.
- Prompt-time memory context contains memory synopsis plus RAG-scored
  memory/fact/episode snippets, without duplicate Tantivy prefetch snippets.
- `search_rag` is available to the root model as a read-only capped tool and
  cannot return review-only content outside explicit review intent.
- Protocol v7 status/search/rebuild/GC messages round-trip, and v2-v6 clients
  never receive v7-only messages.
- SQLite full rebuild can delete and recreate the DB from source truth, while a
  failed vector phase leaves lexical search usable.
- `/rag status` works in REPL/TUI, and Telegram exposes read-only status/search.
- `rag_collections` schema-v2 status shows default system collections and their
  source/chunk counts.
- A user collection can be created from a trusted local file/directory source,
  ingested, rebuilt lexically, rebuilt with real Ollama vectors, searched by
  collection name, and deleted without deleting source files.
- A user collection can be created from an explicit HTTPS URL source, fetched
  with robots/SSRF/redirect/content limits, indexed, searched by collection
  name, refreshed with `ETag`/`Last-Modified` or content-hash stale detection,
  and deleted without deleting any local source files.
- Default help/meta/memory-query RAG behavior remains unchanged when no user
  collection is attached.
- User collections never appear in prompt context unless selected explicitly by
  task/session attachment or a scoped RAG skill/search request.
- `search_rag` supports `collection_type` and `collections` filters and cannot
  return review-only content outside explicit review intent.
