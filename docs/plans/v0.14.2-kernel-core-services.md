# v0.14.2 — Kernel core/services split

Status: Draft

## Summary

v0.14.2 is a structural release after v0.14.1 and before v0.15. It makes the runtime compact again without changing operator-visible behavior. It also closes the v0.14.1 release-readiness oversight where the daemon integration suite could pass serially but intermittently fail under default parallel Cargo execution with local-socket `Operation not permitted` during daemon boot.

The final crate graph has two kernel crates and no compatibility facade:

```text
allbert-kernel-core      # runtime shell, contracts, policy, provider/tool/memory DTOs
allbert-kernel-services  # concrete services plus DefaultKernelServices wiring
```

The current `allbert-kernel` crate is retired by the end of the release. v0.14.2 accepts developer-visible import churn inside the workspace: CLI, daemon, jobs, and tests must import directly from `allbert_kernel_core` or `allbert_kernel_services`. There is no promise to preserve the old `allbert_kernel` Rust API.

## Goals

- Shrink the runtime core so it is auditable independently of owned services.
- Retire the monolithic `allbert-kernel` crate instead of keeping a facade.
- Make crate ownership decision-complete: core owns the runtime shell and contracts; services owns concrete behavior.
- Add size, crate-graph, import-migration, and dependency compactness gates.
- Close the v0.14.1 daemon integration socket flake so default parallel workspace validation is release-blocking again.
- Prepare a service home for v0.15 ingestion without adding ingestion in this release.

## Non-goals

- No growth-loop ingestion.
- No provider-native tool calling.
- No operator-visible behavior change.
- No protocol version bump.
- No storage format change.
- No compatibility facade and no preservation of old `allbert_kernel` imports.
- No per-service crate explosion; one services crate is enough for this release.

## Architecture

### Crate graph

```text
allbert-kernel-core
  - config, paths, settings contracts
  - KernelError and shared result types
  - provider seam and completion DTOs
  - tool contracts, policy, registry traits
  - memory DTOs, frontmatter schemas, and shared contracts
  - hooks, cost, identity, session ids, activity/event contracts
  - KernelRuntime turn shell after concrete services are inverted behind traits

allbert-kernel-services
  - DefaultKernelServices
  - concrete memory/indexing
  - skills store/rendering
  - concrete LLM providers
  - adapters and learning/personality digest
  - self-diagnosis and remediation
  - self-improvement
  - local utilities
  - trace replay and concrete trace storage/export
  - scripting and Lua runtime
  - heartbeat implementation
  - future ingestion service
```

Dependency direction is release-blocking:

- core depends on neither services nor the retired `allbert-kernel` crate;
- services depends on core;
- CLI, daemon, jobs, and tests may depend on core and services directly;
- no workspace crate may depend on `allbert-kernel` at release exit.

During migration only, the existing `allbert-kernel` crate may remain as a temporary holding crate for code that has not moved yet. It must not become a facade, must not re-export moved APIs, and must be removed from the workspace before v0.14.2 closes.

### Runtime ownership

The current `allbert-kernel` crate owns both the large turn runtime and the concrete services it calls. v0.14.2 splits those responsibilities:

- `allbert-kernel-core` owns the turn runtime shell as `KernelRuntime` plus session/turn state, activity state, prompt assembly contracts, tool-dispatch contracts, provider contracts, and service traits.
- `KernelRuntime` moves into core only after direct calls to concrete services are inverted behind core traits. Core must not call concrete memory indexing, concrete providers, Lua, adapter training, self-diagnosis, self-improvement, replay storage, local utilities, or heartbeat file logic directly.
- `allbert-kernel-services` owns `DefaultKernelServices`, the concrete service bundle that wires memory, skills, providers, trace/replay, adapters, learning, local utilities, scripting, self-diagnosis, self-improvement, heartbeat, and future ingestion.
- `allbert-kernel-services` may expose a concrete `Kernel` type alias or constructor for `KernelRuntime<DefaultKernelServices>`. This is the new standard runtime entrypoint for daemon and CLI code.

### Service trait boundaries

Core defines a `KernelServices` trait family. The runtime shell may call concrete behavior only through these traits:

- Provider service: builds effective `LlmProvider` instances from model config and exposes pricing/cost metadata through core provider DTOs.
- Memory service: bootstraps/reconciles memory, builds turn memory snapshots, reports memory status, and owns read/write/search/stage/list/preview/promote/reject/forget operations.
- Skill service: discovers skills, renders `AGENTS.md`, produces routing prompts, validates skill paths, and resolves active skill metadata.
- Service-backed tool registry: registers native tools whose implementations require services while leaving tool invocation/output contracts in core.
- Scripting service: runs Lua skill scripts under existing config, budget, and policy caps.
- Local utility service: owns utility discovery, enablement, doctor output, and `unix_pipe`.
- Trace/replay service: owns concrete trace readers/writers, GC, export, and recovery.
- Adapter/learning service: owns adapter corpus, trainers, activation, evaluation, compute cap accounting, and personality digest jobs.
- Diagnosis/remediation service: owns bounded trace diagnosis, candidate remediation, report storage, and remediation provenance.
- Self-improvement service: owns source-checkout resolution, sibling worktree management, Tier A validation, patch artifacts, and write guards.
- Heartbeat service: owns heartbeat file parsing, quiet-hours checks, and proactive-delivery decisions.

If a runtime call does not fit one of these traits, M0 must classify it before code moves. Core must not add direct dependencies on service implementation modules to avoid introducing the same monolith under a new name.

### Module classification

The implementation must add [v0.14.2 migration inventory](../notes/v0.14.2-migration-inventory-2026-04-26.md) before moving code. The inventory records every current `pub mod` and `pub use`, its destination crate, and its new import path.

| Current surface | Destination | New import rule |
| --- | --- | --- |
| `adapter`, `agent`, `config`, `settings` contracts, `paths`, `error`, `events`, core hook/event/cost/identity/session DTOs | core | Import from `allbert_kernel_core`. |
| `Kernel`, `SessionSnapshot`, `TurnSummary`, turn state, prompt/tool dispatch shell | core runtime shell | Use `allbert_kernel_core::KernelRuntime` for generic runtime and `allbert_kernel_services::Kernel` for default runtime wiring. |
| provider traits and DTOs from `llm::provider` | core | Import contracts from `allbert_kernel_core::llm`. |
| concrete provider clients under `llm::{anthropic, gemini, ollama, openai, openrouter}` | services | Import concrete providers from `allbert_kernel_services::llm`. |
| tool contracts, parser, exec policy DTOs, `ToolRegistry` interfaces | core | Concrete tool implementations that need services call through core service traits. |
| concrete memory storage, staging, Tantivy indexing, memory-curator helpers | services | Import concrete memory operations from `allbert_kernel_services::memory`; shared DTOs remain in core. |
| skills store, skill validation, AGENTS.md rendering | services | Import from `allbert_kernel_services::skills`; `refresh_agents_markdown` lives in services. |
| adapters, learning/personality digest, local utilities, replay/trace storage, scripting/Lua, self-diagnosis, self-improvement, heartbeat implementation | services | Import from `allbert_kernel_services` modules. |
| command catalog descriptors and settings catalog descriptors | core unless they need concrete services | Import from core; move service-specific descriptors to services only if needed. |
| job-manager trait and job DTOs | core | Daemon job implementation remains in `allbert-daemon`. |
| concrete tracing subscriber/appender setup | services | Core may own trace handles/contracts only. |

### Import migration contract

The release intentionally breaks the old Rust crate surface. The implementation must update every workspace import that currently uses `allbert_kernel`:

```rust
use allbert_kernel_core::{AllbertPaths, Config, KernelError};
use allbert_kernel_core::llm::{CompletionRequest, CompletionResponse, LlmProvider};
use allbert_kernel_core::tools::{ToolInvocation, ToolOutput};

use allbert_kernel_services::Kernel;
use allbert_kernel_services::adapters::{AdapterStore, PersonalityAdapterJob};
use allbert_kernel_services::memory::list_staged_memory;
use allbert_kernel_services::skills::{validate_skill_path, SkillProvenance};
use allbert_kernel_services::self_diagnosis::{
    DiagnosisReportArtifact, DiagnosisRemediationRequest,
};
```

Import-migration gates are release-blocking:

- Add [v0.14.2 migration inventory](../notes/v0.14.2-migration-inventory-2026-04-26.md), generated from the current `pub mod` and `pub use` surface, before code movement begins.
- Add compile-only migration fixtures under `crates/allbert-kernel-core/tests/imports.rs` and `crates/allbert-kernel-services/tests/imports.rs` that import representative core and service paths used by CLI, daemon, jobs, and daemon tests.
- Add `tools/check_kernel_import_migration.sh`; it fails if any workspace Rust source, Cargo manifest, or `Cargo.lock` still references `allbert_kernel`, `allbert-kernel`, or `crates/allbert-kernel` outside historical docs and explicit migration notes.
- Move the `integration-trainer` feature to `allbert-kernel-services`; no facade feature forwarding is allowed.

### Size and dependency gates

Add `tools/check_kernel_size.sh` and wire it into local validation:

- `crates/allbert-kernel-core/src/lib.rs` < 4,000 LOC.
- `crates/allbert-kernel-core/src/` total < 20,000 LOC.
- `crates/allbert-kernel-services/src/` total < 30,000 LOC.
- `crates/allbert-kernel/` must not exist as a workspace member at release exit.
- Counts include checked-in `.rs` files under each crate `src/`, including inline module tests, and exclude `target/`, generated artifacts, fixtures outside `src/`, and vendored dependencies.
- M0 records the current baseline before enforcement so reviewers can see how much implementation moved.

Add `tools/check_kernel_crate_graph.sh`:

- fail if core depends on services or any retired `allbert-kernel` package;
- fail if services depends on any retired `allbert-kernel` package;
- fail if any workspace package depends on `allbert-kernel`;
- allow CLI, daemon, jobs, and tests to depend on core and services directly.

Add dependency compactness checks:

- Core normal dependencies may include only contract-level crates needed by config, DTOs, policy, and runtime shell code, such as `allbert-proto`, `serde`, `serde_json`, `toml`, `toml_edit`, `thiserror`, `async-trait`, `time`, `chrono`, `chrono-tz`, `uuid`, `dirs`, `fs2`, `sha2`, and `regex`.
- Core must not depend on concrete service/runtime-heavy crates including `reqwest`, `tantivy`, `mlua`, `tracing-subscriber`, `tracing-appender`, `pulldown-cmark`, `serde_yaml`, `flate2`, `base64`, `tempfile`, or trainer subprocess implementation dependencies as normal dependencies.
- Services owns concrete provider HTTP clients, memory indexing, Lua, trace subscriber/appender setup, markdown/YAML parsing, compression/export helpers, local utility runtime, adapter trainer implementations, and self-improvement process helpers.
- Add `tools/check_kernel_dependency_compactness.sh`; it checks the core/services manifests against the allowed/forbidden dependency lists above.

### Daemon reliability contract

The daemon integration socket failure is a first-class v0.14.2 milestone, not incidental test cleanup. v0.14.1 closed with a known caveat: `env -u RUSTC_WRAPPER cargo test -q -- --test-threads=1` passed, while default parallel `env -u RUSTC_WRAPPER cargo test -q` could intermittently fail during daemon boot with local socket `Operation not permitted`. v0.14.2 must remove that caveat before structural crate movement begins.

Implementation must inspect and harden the daemon boot path around `spawn_with_factory`, `prepare_socket_dir`, `bind_listener`, `try_connect_existing`, socket permission handling, lock acquisition/release, and the daemon integration test `TempHome` helper. The expected fix should make socket and lock state unique, deterministic, and fully cleaned up across parallel tests; it must not hide the failure by forcing serial tests, adding broad ignores, or skipping daemon integration coverage.

No new ADR is required for this milestone if the fix is limited to test/runtime reliability and keeps the operator-visible daemon socket contract unchanged. If the implementation changes the daemon socket location, lock semantics, trust model, or protocol behavior, ADR 0100 must be amended or a new daemon IPC ADR must be added before release.

### M1 — Daemon integration socket reliability

Scope:

- Treat this as the v0.14.1 readiness oversight carry-forward and complete it before M2 starts.
- Reproduce and isolate the intermittent daemon integration failure where local socket boot returns `Operation not permitted` under default parallel Cargo test execution.
- Audit daemon integration test temp-home, socket-path, pid-file, lock-file, permission, stale-socket probe, and cleanup behavior for cross-test collisions and stale state.
- Make daemon socket and lock paths unique per test process/run while preserving operator-visible runtime defaults for real profiles.
- Ensure failed daemon startup and cancelled/crashed tests clean up child processes, socket files, pid files, lock files, and temp directories deterministically.
- Add regression coverage that exercises representative daemon boot, attach, status, second-spawn rejection, shutdown, and restart flows concurrently.
- Record the root cause and the chosen fix in the v0.14.2 closeout notes so the serial-test workaround is not rediscovered later.
- Keep this as test/runtime reliability work only; do not change daemon protocol, command names, or user-facing behavior.

Tests and validation:

- `env -u RUSTC_WRAPPER cargo test -q -p allbert-daemon` passes under default parallel test execution.
- `env -u RUSTC_WRAPPER cargo test -q -p allbert-daemon --test m1_daemon` passes repeatedly under default parallel test execution.
- `env -u RUSTC_WRAPPER cargo test -q` passes without requiring `--test-threads=1`.
- Repeated daemon integration runs leave no long-lived daemon processes or stale local socket artifacts in temp homes.

Exit:

- The daemon integration suite no longer intermittently fails with local socket `Operation not permitted` during daemon boot.
- Serial test execution may remain a diagnostic option but is not required for release validation and must not be documented as the v0.14.2 release workaround.

### M2 — Introduce `allbert-kernel-core`

Scope:

- Create `crates/allbert-kernel-core`.
- Move bottom-layer contracts: config, paths, kernel error, provider trait/DTOs, tool contracts, memory DTOs/frontmatter, hooks/events, cost/identity/session contracts, job-manager trait, and shared utility helpers.
- Add the `KernelServices` trait family used by the runtime shell to call concrete services without depending on services.
- Update CLI, daemon, jobs, tests, and the temporary legacy crate to import moved contracts from `allbert_kernel_core`.
- Keep concrete behavior implementations in the temporary legacy crate until they are represented in `KernelServices` and ready to move.

Tests and validation:

- Workspace builds.
- Core has no dependency on services or the temporary legacy crate.
- Unit tests for moved config/provider/tool/memory contracts remain attached to the moved modules.
- Core has no forbidden concrete-service dependencies as normal dependencies.

### M3 — Introduce `allbert-kernel-services`

Scope:

- Create `crates/allbert-kernel-services`.
- Move the `integration-trainer` feature to services.
- Define `DefaultKernelServices` and the standard `Kernel` constructor/type alias for `KernelRuntime<DefaultKernelServices>`.
- Move concrete service modules in this order to reduce coupling risk:
  1. concrete LLM providers while leaving provider traits in core;
  2. concrete memory storage, staging, and Tantivy indexing while leaving memory DTOs/frontmatter in core;
  3. skills store/rendering and `refresh_agents_markdown`;
  4. local utilities;
  5. replay and concrete trace storage/export;
  6. scripting and Lua runtime;
  7. self-improvement;
  8. adapters and learning/personality digest;
  9. heartbeat implementation;

- Update imports to point directly at `allbert_kernel_services` as each module moves.
- Replace temporary legacy-crate coupling with core contracts or services-local helpers.
- Update self-improvement source-checkout validation, path fixtures, generated remediation paths, and Tier A assumptions so the new core/services crate layout is recognized.

Tests and validation:

- Workspace tests pass after each moved module batch.
- No service imports `allbert_kernel`.
- Existing operator-facing behavior is unchanged.
- Self-improvement tests no longer assume `crates/allbert-kernel` is the only source crate that can be patched or validated.

### M4 — Move runtime shell and retire `allbert-kernel`

## Exit criteria

- Workspace validation passes.
- `crates/allbert-kernel` is removed from the workspace.
- No workspace code depends on or imports `allbert-kernel` / `allbert_kernel`.
- `allbert-kernel-core` owns contracts and the runtime shell without concrete-service dependencies.
- `allbert-kernel-services` owns concrete services, default runtime wiring, and the v0.15 ingestion home.
- Default parallel daemon integration tests pass without local socket `Operation not permitted` boot failures.
- Size, crate-graph, import-migration, and dependency compactness gates pass.
- Self-improvement source-checkout validation, patch fixtures, and Tier A path assumptions recognize core and services crates.
- v0.15 can add ingestion to services without adding new behavior to core.

## Deferred work

- Growth-loop ingestion. v0.15.
- Provider-native tool calling. v0.15+.
- Per-service crates. v0.16+ if the services crate grows too large.
- External crate-level semver compatibility. Deferred until Allbert exposes a supported Rust library API.
