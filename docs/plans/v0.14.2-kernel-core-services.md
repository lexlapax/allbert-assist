# v0.14.2 — Kernel core/services split

Status: Draft

## Summary

v0.14.2 is a structural release after v0.14.1 and before v0.15. It makes the kernel compact again without changing operator-visible behavior.

The current `allbert-kernel` crate contains both runtime core contracts and service implementations. v0.14.2 splits that shape into an acyclic three-crate graph:

```text
allbert-kernel-core      # runtime contracts, core types, policy, provider/tool/memory contracts
allbert-kernel-services  # adapters, diagnosis, remediation, self-improvement, utilities, replay, scripting
allbert-kernel           # compatibility facade preserving allbert_kernel::* imports
```

The compatibility facade depends on both lower crates and re-exports the public surface. Existing frontends and tests should keep compiling against `allbert_kernel::*`.

## Goals

- Shrink the runtime core so it is auditable independently of owned services.
- Preserve existing `allbert_kernel::*` imports through facade re-exports.
- Avoid crate cycles by making core the bottom layer, services depend on core, and facade depend on both.
- Add size and dependency-graph gates so the boundary does not drift again.
- Fix the intermittent daemon integration local-socket startup failure so default parallel workspace validation is release-blocking again.
- Prepare a service home for v0.15 ingestion without adding ingestion in this release.

## Non-goals

- No growth-loop ingestion.
- No provider-native tool calling.
- No operator-visible behavior change.
- No protocol version bump.
- No storage format change.
- No broad frontend import rewrite except narrow cases required to break cycles during migration.
- No per-service crate explosion; one services crate is enough for this release.

## Architecture

### Crate graph

```text
allbert-kernel-core
  - config, paths, settings
  - KernelError and shared result types
  - provider seam and completion DTOs
  - tool contracts, policy, registry traits
  - memory contracts and shared schemas
  - hooks, cost, identity, session ids, activity/event contracts

allbert-kernel-services
  - adapters
  - self-diagnosis and remediation
  - self-improvement
  - local utilities
  - trace replay
  - scripting
  - future ingestion service

allbert-kernel
  - facade re-exports
  - compatibility wrappers only
```

Dependency direction is release-blocking:

- core depends on neither services nor facade;
- services depends on core only;
- facade depends on core and services;
- frontends continue to depend on facade by default.

### Public API compatibility

The implementation must inventory the public API currently exported by `allbert-kernel/src/lib.rs` and preserve it through facade re-exports. Public names may move internally, but existing imports such as these must keep compiling:

```rust
use allbert_kernel::{AllbertPaths, Config, KernelError};
use allbert_kernel::{LlmProvider, ToolInvocation, ToolOutput};
use allbert_kernel::{AdapterStore, PersonalityAdapterJob};
use allbert_kernel::{DiagnosisRunRequest, DiagnosisReportArtifact};
```

If a symbol cannot be preserved without exposing a bad dependency, the implementation must add a facade wrapper with the old name and document the wrapper in the v0.14.2 closeout notes.

### Size gates

Add `tools/check_kernel_size.sh` and wire it into local validation:

- `crates/allbert-kernel/src/lib.rs` < 1,500 LOC.
- `crates/allbert-kernel-core/src/lib.rs` < 4,000 LOC.
- `crates/allbert-kernel-core/src/` total < 20,000 LOC.
- `crates/allbert-kernel-services/src/` total < 25,000 LOC.

Add `tools/check_kernel_crate_graph.sh`:

- fail if core depends on services or facade;
- fail if services depends on facade;
- allow facade to depend on both.

## Milestones

### M0 — ADR, size baseline, and public API inventory

Scope:

- Finalize [ADR 0100](../adr/0100-kernel-splits-into-core-and-services.md).
- Record current module LOC and public exports from `allbert-kernel/src/lib.rs`.
- Add a migration inventory listing each public symbol, target crate, and facade re-export path.
- Add initial size/dependency scripts in report-only mode.
- Update `docs/plans/roadmap.md`, `docs/vision.md`, and v0.15 dependency text to point at v0.14.2.

Tests and validation:

- `git diff --check`.
- `cargo test -q` may run with report-only scripts if they are wired in M0.
- Public API inventory checked into docs or a test fixture.

### M1 — Daemon integration socket reliability

Scope:

- Reproduce and isolate the intermittent daemon integration failure where local socket boot returns `Operation not permitted` under default parallel Cargo test execution.
- Audit daemon integration test temp-home, socket-path, pid-file, lock-file, and cleanup behavior for cross-test collisions and stale state.
- Make daemon socket and lock paths unique per test process/run while preserving operator-visible runtime defaults.
- Ensure failed daemon startup and cancelled/crashed tests clean up child processes, socket files, pid files, and temp directories deterministically.
- Add regression coverage that exercises representative daemon boot, attach, status, and stop flows concurrently.
- Keep this as test/runtime reliability work only; do not change daemon protocol, command names, or user-facing behavior.

Tests and validation:

- `env -u RUSTC_WRAPPER cargo test -q -p allbert-daemon` passes under default parallel test execution.
- `env -u RUSTC_WRAPPER cargo test -q` passes without requiring `--test-threads=1`.
- Repeated daemon integration runs leave no long-lived daemon processes or stale local socket artifacts in temp homes.

Exit:

- The daemon integration suite no longer intermittently fails with local socket `Operation not permitted` during daemon boot.
- Serial test execution may remain a diagnostic option but is not required for release validation.

### M2 — Introduce `allbert-kernel-core`

Scope:

- Create `crates/allbert-kernel-core`.
- Move only bottom-layer contracts: config, paths, kernel error, provider trait/DTOs, tool contracts, memory contracts used across services, hooks/events, cost/identity/session contracts, and shared utility helpers.
- Keep behavior implementations in the existing crate until their dependencies point cleanly at core.
- Re-export moved core symbols from `allbert-kernel` immediately.

Tests and validation:

- Workspace builds.
- Existing imports through `allbert_kernel::*` compile.
- Core has no dependency on facade or services.
- Unit tests for moved config/provider/tool/memory contracts remain attached to the moved modules.

### M3 — Introduce `allbert-kernel-services`

Scope:

- Create `crates/allbert-kernel-services`.
- Define service module boundaries and a services prelude that depends only on core.
- Move no large behavior module until compile boundaries are proven.
- Add facade re-export scaffolding in `allbert-kernel`.
- Update Cargo workspace membership.

Tests and validation:

- Workspace builds with all three crates.
- Dependency graph script passes in enforce mode for the empty services crate.
- No frontend import churn beyond Cargo dependency additions if required.

### M4 — Move service modules one by one

Scope:

- Move service modules in this order to reduce coupling risk:
  1. `local_utilities`;
  2. `replay`;
  3. `scripting`;
  4. `self_improvement`;
  5. `adapters`;
  6. `self_diagnosis` and remediation.
- After each move, restore facade re-exports before moving the next module.
- Keep tests with their modules unless a test is explicitly a facade-compatibility test.
- Replace any hidden facade dependency with a core contract or services-local helper.

Tests and validation:

- Workspace tests pass after each moved module batch.
- Facade import compatibility tests compile.
- No service imports `allbert_kernel`.
- Existing operator-facing behavior is unchanged.

### M5 — Make `allbert-kernel` the compatibility facade

Scope:

- Remove substantial implementations from the facade crate.
- Keep facade `lib.rs` focused on public re-exports, compatibility wrappers, and documentation.
- Update frontends only where necessary; default public imports remain through `allbert_kernel::*`.
- Move integration tests that are really service tests into services; keep public API compatibility tests at the facade.

Tests and validation:

- `cargo build --workspace`.
- `cargo test -q`.
- `cargo doc --workspace` succeeds without broken intra-doc links.
- A compile-only test covers the public import examples in this plan.

### M6 — Size gate, docs, and release closeout

Scope:

- Turn size/dependency scripts from report-only to enforce mode.
- Update `CHANGELOG.md`, release notes, roadmap status, and operator docs if any command help paths changed.
- Confirm v0.15 ingestion plan references the services crate from v0.14.2 rather than v0.14.1.
- Bump crates and lockfile to `0.14.2`.

Tests and validation:

- Default validation from `AGENTS.md`.
- `tools/check_kernel_size.sh`.
- `tools/check_kernel_crate_graph.sh`.
- Public facade imports compile.
- No crate-cycle or duplicate-symbol warnings.

## ADRs

- [ADR 0100](../adr/0100-kernel-splits-into-core-and-services.md): Kernel facade splits into core and services without crate cycles.

## Exit criteria

- Workspace validation passes.
- Existing `allbert_kernel::*` imports still compile.
- No crate cycle exists: core below services below facade.
- The facade remains stable and mostly re-export-only.
- Default parallel daemon integration tests pass without local socket `Operation not permitted` boot failures.
- Size gates pass.
- v0.15 can add ingestion to the services crate without adding new behavior to the core or facade.

## Deferred work

- Growth-loop ingestion. v0.15.
- Provider-native tool calling. v0.15+.
- Per-service crates. v0.16+ if the services crate grows too large.
- Removing compatibility re-exports. Deferred until there is a semver-major policy.
