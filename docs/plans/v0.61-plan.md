## Purpose

Turn first-run from a prototype-grade scatter of steps into a genuine guided
onboarding capability. The v0.58 maturity review found `mix allbert.onboard` is
not interactive (it prints copy-paste shell commands), the web onboarding panel is
not auto-launched on first run, credential entry is punted to a separate form, and
there are no user-category presets. v0.61 closes that experience gap with a
two-track wizard surfaced on both the web and an interactive CLI/TUI over one
shared flow, a first-class provider/model step with masked-key entry and inline
`doctor` verification, and repo-maintained user-category profiles that seed
Settings Central defaults with explicit operator review — adding **no new
authority**.

## Operator-Visible Win

A brand-new operator launches Allbert, the web wizard auto-opens on first run,
they pick QuickStart, enter a masked provider key that is verified inline by
`doctor`, accept a persona preset after reviewing exactly what it seeds, and reach
a working first chat without reading docs or hand-editing config. The same flow is
available as a genuinely interactive terminal wizard for headless installs. The
operator sees the trust spine (confirmation / permission / trace) presented as a
feature during onboarding.

## Planned Interfaces

- Provider/model setup step: masked-key entry; inline `doctor` verification; local

- User-category profile selection (researcher / developer / writer / ops /

- A health-check confirmation the operator can see succeed.

## Guided Wizard Flow (ADR 0069)

The wizard is a single shared flow model with two render surfaces:

- **One flow, two surfaces.** A surface-agnostic wizard state machine owns the
  step sequence, validation, and transitions; the web (ADR 0074) and the
  interactive CLI/TUI (ADR 0067) render it. Neither surface forks the logic.
- **Two tracks.** QuickStart minimizes decisions and skips channel/integration
  setup to reach a working conversation immediately; Advanced exposes the full
  step set.
- **Steps:** provider/model setup → user-category profile selection/review →
  optional channel pairing and integration setup (skipped on the fast path) →
  visible health-check confirmation.
- **Trust spine as a feature.** The wizard presents the confirmation / permission
  / trace model as the safety property it is, not as friction.
- **Vault-ready credentials.** Masked-key entry writes through the existing
  encrypted secrets store now, structured so the v0.62 OS secret-vault swaps in
  underneath without reworking the wizard.

## User-Category Profile System (ADR 0075)

- **Repo-maintained declarative presets**, version-controlled and reviewed in the
  source tree — the same trust posture as reviewed plugins, not a community preset
  marketplace and not learned from traces.
- **Shipped categories** (extensible): `researcher`, `developer`, `writer`, `ops`,
  `general`.
- **Each profile specifies:** Settings Central default values; a suggested set of
  apps to enable; suggested channels; suggested intents; and a recommended
  model-purpose mapping referencing ADR 0072 (purposes, not pinned endpoints).
- **Seed-only semantics.** Applying a profile only pre-populates operator-tunable
  Settings Central values and suggestions the operator can change. It lowers no
  confirmation floor, grants no egress, enables nothing effectful without the
  normal gates, and Security Central (ADR 0006) is unchanged.
- **Explicit application.** A profile is applied during guided onboarding or via a
  registered settings action, always with a review/confirm step that shows exactly
  what will be seeded.

## Security Posture

No new authority. Onboarding is an experience over existing flows: every effectful
step routes through the existing gates and confirmations, and Security Central
(ADR 0006) is unchanged. Profiles seed defaults only — they are not a permission
set, a trust tier, or a per-user authority model, and cannot grant capabilities.

Settings still flow through Settings Central. Onboarding writes settings and
applies profile presets only through Settings Central; it is not a configuration
side channel. Secrets, provider/model, channel pairing, and `doctor` remain the
authoritative flows the wizard drives.

Masked-key entry never echoes or logs raw secret values; credentials are written
through the existing encrypted secrets store and the wizard's diagnostics are
redacted.

## Non-Goals

- No new capability surface or new authority of any kind.
- No runtime authorization, trust tier, or per-user authority model from profiles.
- No learned, distilled, or auto-generated personas (durable non-goal); unrelated
  to the existing free-text `persona.md` identity memory.
- No hosted/multi-user onboarding or multi-tenant role system.
- No configuration side channel — all writes go through Settings Central.
- No OS secret-vault implementation (vault-ready only; the vault lands in v0.62).
- No community preset marketplace.

## Definition Of Done

v0.61 is done when:

- a new operator can complete first-run onboarding through the web wizard
  (auto-launched on first run) and through the interactive CLI/TUI wizard, over
  one shared flow, without reading docs or hand-editing config;
- the QuickStart track reaches a working first chat, and Advanced exposes full
  provider / model / channel / profile control;
- the provider/model step accepts a masked key, verifies it inline with `doctor`,
  supports local and hosted providers, and switches providers without config
  edits, writing credentials vault-ready through the encrypted secrets store;
- repo-maintained user-category profiles seed Settings Central defaults plus
  suggested apps, channels, and intents, applied only with an explicit
  review/confirm step;
- the trust spine (confirmation / permission / trace) is surfaced as a feature;
- ADR 0069 and ADR 0075 are accepted;
- the `:v061` security-eval rows pass and `mix allbert.test release.v061` is
  deterministic — proving no new authority, no secret leak in onboarding,
  settings-central-only writes, and profile-seeds-defaults-only semantics;
- no known warning/security/doc drift remains.
