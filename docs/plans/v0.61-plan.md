## Purpose

Implement the v0.60 information-architecture, navigation, and screen-composition
redesign — filling in the v0.60 walking skeleton with real, redesigned screens —
AND apply the **v0.60b-chosen visual language (Direction C — Soft Modern Depth,
ADR 0079)** — its elevation/depth tokens, rounded geometry, motion roles, brand,
and dark-mode polish — on that redesigned surface, turning the substrate into the
polished primary 1.0 product surface. v0.61 **implements** the chosen language; it
does not re-design it.

**Layout is chosen before it is built.** The one thing neither v0.60 (structure)
nor v0.60b (visual language) fixed is the *per-screen composition/layout* across
the whole IA surface set: v0.60b wireframed only the four hero screens (workspace,
onboarding, trust, launch), and the chat-primary re-layout ADR 0074 deferred is
pinned only in a disposable v0.60b hero wireframe — five of the nine IA surfaces
(jobs, objectives, models, channels, settings) have no chosen layout at all.
Rather than let v0.61 *assume* a layout and build it blind, v0.61 **front-loads a
layout-exploration + operator-choice step** — exactly the v0.60b pattern applied
to layout: it renders **≥3 divergent layout systems** across **all nine IA
surfaces** in the chosen Direction C visual language as disposable previews, the
operator previews them side-by-side and **chooses one** (`CHOSEN_LAYOUT`), and the
build milestones then implement the chosen layout. This closes the "current live
layouts are wrong" gap the operator raised, on the design-first side of the v1.0
freeze.

This closes the gaps the design work surfaced: the v0.60 Product Experience
Design found the app's information architecture and navigation model were
operator-utility-flat and needed a deliberate redesign (delivered as a model and
a walking skeleton, not as shipped screens); the v0.60b Visual Design Language
found the visual language itself was unowned and chose Direction C (delivered as a
spec + token/component delta, not as a built surface); and the v0.58 review found
motion tokens and scattered transitions but no coherent product motion layer, an
absent brand identity, a thin `/` landing with no marketing surface, a
utility-flat hierarchy, and `system` theme behavior that needs explicit
OS-preference validation. v0.61 explores and chooses the layout, then ships the
redesigned IA and screens, the Direction C design-system tokens, brand, motion,
hierarchy, a real landing surface, suggested-action affordances, and validated
OS-driven dark mode — all on the existing token scales, all through the catalog.

## Operator-Visible Win

An operator moves through an app whose information architecture, navigation, AND
per-screen layout were deliberately redesigned and **chosen by the operator**: a
redesigned shell and navigation place workspace, jobs, objectives, settings, and
the operator panels where the v0.60 IA puts them, in the layout the operator
previewed and selected, dressed in the Direction C visual language (tonal violet
depth, rounded geometry, soft elevation). They land on a real `/` marketing
surface with the Allbert brand, navigate into a shell carrying the wordmark
instead of the stock framework logo, see entrance/drawer/skeleton motion that
respects their reduced-motion preference, read redesigned screens with real
visual hierarchy and populated first-run/empty states that suggest what to do
next, and get a dark theme that honors their OS preference when `system` is
selected — with no change to what the app can do on their behalf.

## Cross-Cutting Invariants (apply to every milestone)

- **No new authority, capability, egress, or authority change.** The catalog stays
  the rendering boundary (ADR 0030/0074); every deliverable is presentation-only.
  Suggested actions and landing CTAs render registered-action DTOs (ADR 0073) and
  grant no authority or egress; launcher selection stays view-only. Security
  Central is unchanged.
- **No new rendering path or data-layer path.** The structural redesign recomposes
  screens through the existing catalog/shell/token substrate; the same DTOs render
  through redesigned screens. No model- or data-generated markup.
- **Layout is chosen, not assumed.** The layout is explored as ≥3 divergent systems
  and the operator chooses one (M2) before any build milestone implements it. v0.61
  does not re-open the v0.60 IA or the v0.60b visual-language choice; it fixes the
  per-screen composition the upstream releases left thin, then builds it.
- **Direction C is implemented, not re-decided.** v0.61 realizes the
  operator-chosen visual language and its token/component delta; it does not choose
  a new aesthetic. The delta is built as first-class tokens/variants, not per-page
  styling and not the disposable `[data-visual-direction="c"]` preview CSS.
- **No new route beyond the landing surface.** The nine IA surfaces map onto the
  existing route set plus the rebuilt `/` landing; models/channels/trust/onboarding
  are recomposed as panels/sections/affordances of existing routes, not new routes
  (see **Route & Skeleton Promotion (M3)**).
- **No substrate rebuild.** The v0.58 design-system substrate (tokens, variant
  registry, pattern library, catalog, shell) and the data substrate (canvas,
  tiles, ephemerals, Fragment/HMAC, persistence) are reused as-is.

### M1: Layout-System Exploration

Produce **at least three** coherent, divergent **layout systems** — composition
paradigms, not restyles — each rendered across **all nine IA surfaces** in the
chosen Direction C visual language, so the operator previews real layouts before
one is built. A layout system is a distinct answer to *where things go*: the
placement of the product shell, primary work canvas, context rail, utility
surfaces, and the chat-primary hero; the navigation posture; and the responsive
spine. Illustrative (not prescriptive) axes of divergence:

- **L-A "Focused canvas"** — a single primary canvas with a collapsible context
  rail; secondary surfaces behind a command surface; maximal focus on the chat
  hero.
- **L-B "Workbench"** — a persistent multi-pane layout (nav + work + context)
  always visible; denser; operations-forward.
- **L-C "Progressive shell"** — minimal chrome that expands zones on demand;
  mobile-first spine scaled up to desktop.

Each system is rendered for every one of the nine surfaces (launch, onboarding,
workspace, objectives, jobs, models, channels, settings, trust) — not only the
four v0.60b hero screens — so the five previously-unwireframed surfaces get a real
chosen composition. The renderings are **disposable exploration**: styled variants
of the v0.60 walking-skeleton screens behind the existing `:preview_routes` flag,
rendering placeholder-only compositions through the catalog, reading no live data
and granting no authority. They reuse the v0.60b `data-visual-direction="c"`
token/theme delta so the operator previews layouts **in the chosen visual
language**, before M4 promotes that delta to first-class tokens.

- **Implementation:**
  - **Artifacts:** `docs/design/layout-systems-explored.md` — the ≥3 layout
    systems, each with a per-surface composition spec (zone placement, nav
    posture, chat-primary hero treatment, responsive spine) for all nine IA

  - **Modules / Files (disposable exploration mechanism):** a flag-gated
