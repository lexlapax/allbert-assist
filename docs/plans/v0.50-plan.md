Request flow: `docs/plans/v0.50-request-flow.md`
ADR: `docs/adr/0046-settings-schema-migration-policy.md` (drafted in v0.45,
accepted here, implemented as the migration tool)

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`, post-v0.37 planning pass.

This milestone gained the settings schema migration substrate in the post-v0.37
planning pass: marketplace (v0.45) introduces new settings fragments, and
v0.46-v0.49 add more, so v0.50 implements the migration tool and accepts
ADR 0046 before the 1.0 freeze.

## Purpose

Prepare 1.0 after the final feature milestone. The milestone makes the
v0.39-v0.49b surface portable, auditable, documented, and boring to operate,
implements the settings schema migration substrate per ADR 0046, then gathers
the final release-candidate evidence before the v1.0 contract freeze.

## Expected Inputs

- v0.39 through v0.49b implemented and accepted.
- ADR 0046 draft from v0.45 ready for acceptance.
- Current Allbert Home layout.
- Existing settings, memory, conversation, trace, audit, objective, plugin,
  marketplace, self-improvement, media, channel, and MCP-server-mode storage.

## Operator-Visible Win

An operator can export an Allbert Home, dry-run import on a second machine,
run `mix allbert.settings.migrate` for any pending schema migrations, confirm
behavior, inspect the final cross-surface security posture, and enter the 1.0
contract freeze with evidence.

## Planned Interfaces

- Profile export/import dry run.
- Schema/version metadata summary per Settings Central fragment.
- `mix allbert.settings.migrate` migration runner per ADR 0046.
- Rollback documentation.
- Final security review/status surfaces.
- Final RC acceptance checklist.

## Settings Schema Migration Substrate (ADR 0046)

v0.50 implements the migration policy drafted in v0.45:

- `mix allbert.settings.migrate` inspects each fragment's current
  `schema_version` against the operator's stored settings; enumerates pending
  migrations; shows redacted previews; requires explicit confirmation per
  migration (or `--all` for batch).
- Migrations are explicit Elixir modules under
  `lib/allbert_assist/settings/migrations/`. No LLM-authored migration steps.
- Pending migrations surface in `mix allbert.security status` and the
  workspace Settings panel on boot.
- Each migration writes a record into
  `<ALLBERT_HOME>/audit/settings_migrations/`.
- v0.50 export/import preserves each fragment's `schema_version` so a v0.42
  export imported on v0.50 is correctly identified as needing migrations.

## Security Posture

No new authority. The milestone audits all newly added capability classes:
MCP client, browser, channels (Discord/Slack and WhatsApp/Signal/Matrix),
Plan/Build, marketplace, self-improvement, voice, vision, and MCP server
mode.

OpenAI-compatible API, ACP server, and public AG-UI/A2UI bridge security
evals are NOT part of v0.50; those surfaces are parked post-1.0.

Secret migration is explicit and redacted. The migration tool never logs raw
secret values.

## Non-Goals

- No broad remote sync service.
- No hosted multi-user auth.
- No new capability surface.
- No expansion of v0.46 self-improvement behavior.
- No contract freeze until v1.0.
- No public protocol surface evals beyond MCP server mode (v0.49b).
- No automatic settings migration on boot. Migrations require operator
  confirmation per ADR 0046.

## Definition Of Done

v0.50 is done when:

- a disposable profile can be exported and re-imported with documented
  behavior;
- pending schema migrations surface on boot and `mix allbert.settings.migrate`
  applies them with operator confirmation;
- all v1.0 acceptance-matrix gates (per `docs/plans/roadmap.md`) have
  evidence;
- ADR 0046 is accepted;
- no known warning/security/doc drift remains.
