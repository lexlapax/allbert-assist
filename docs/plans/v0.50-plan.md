## Purpose

v0.50 adds **Artifacts Central**: a uniform, type-agnostic, content-addressable
store (CAS) for durable artifacts that are **uploaded by the operator**,
**created by Allbert**, or **found by Allbert** through approved tools such as
browser research. Artifacts can be any type — audio, video, images, PDFs, text,
office documents, spreadsheets — because the store addresses content by hash and
never interprets it.

The store is built **in-tree on BEAM primitives**, not on a third-party
package. The research-backed reason (recorded in ADR 0053):

- The CAS-specific Hex packages are dead ends: `hashfs` (last release 2017, repo
  gone, SHA-1/git-blob hashing, no metadata) and `scarab` (2016, pre-1.0, no
  metadata) are abandoned; `dasl` is an IPLD/CID *encoding* layer, not a local
  file store.
- The upload utilities are not content-addressed stores: `waffle` keys storage
  by filename/scope (not content hash) and drags in `hackney` + an Ecto/
  ImageMagick-shaped workflow; `Capsule` (Apache-2.0, zero-dep) is a clean
  `Storage`/`Locator` abstraction but explicitly leaves hashing/addressing to
  the caller.
- The "hard part" is stdlib: `:crypto.hash(:sha256, …)` (streaming-capable),
  `Base.encode16/2`, sharded `Path.join`, and atomic `File.rename` cover hashing
  + layout completely in ~100–150 reviewable lines, with zero supply-chain,
  licensing, or maintenance risk and no heavy deps. `Capsule` remains a parked
  option only if a later version needs a pluggable S3 backend.

v0.50 is deliberately **substrate-heavy and primitive-light**: nearly everything
it needs already exists (Allbert Home roots, the resource-URI scheme system,
Security Central operation/permission classes, the Settings Central fragment
pattern, the allow-list redactor, the default-off retention mechanism, the Jido
action Registry/Runner). The genuinely new primitives are the content-addressed
object store itself, the trace-safe artifact metadata index, the
`artifact://sha256/<hex>` identity model, and the codebase's **first Jido
sensor**.

## Readiness Sweep

Status: complete; implementation-ready for M1.

- **Substrate fit.** Artifacts Central is a storage substrate under the runtime,
  not a new runtime. It composes with CLI/workspace/channels through Resource
  Access, Security Central, confirmations, traces, and audits. A stored artifact
  is descriptive evidence, never authority; nothing auto-promotes to memory and
  content-addressed identity never grants permission.
- **Roadmap fit.** v0.50 follows v0.49 vision and precedes v0.51 Channel Pack 1.
  It completes the v0.42 media-resource vision (audio in v0.48, image/screenshot
  in v0.49, durable cross-surface artifacts here) and gives channels a canonical
  artifact home before they forward attachments.
- **Past-plan fit.** Reuses, in place: the `AllbertAssist.Paths` /
  `Runtime.Paths` Home-root facade; the `ResourceURI` scheme-plugin pattern
  (mint/normalize/derive/scope); the `OperationClass` + `Security.Policy`
  closed-vocabulary permission/floor machinery (the v0.48 `voice_floor` and
  `notes_file_write` floors are templates); the `Settings.Schema` voice/vision
  fragment shape (`schema_version` + retention + redact-flag); the allow-list
  `Runtime.Redactor` (which already treats `content_sha256` as trace-safe); the
  default-off `retention_enabled` mechanism + `cleanup_transient_capture`
  lifecycle; the `Actions.Registry` + `Actions.Runner.run/3` action path; and
  the `Marketplace.Bundle` SHA-256 helpers + `sha256:` convention,

- **Future-plan fit.** v0.51+ channel plugins may forward artifact references
  but do not own store authority. v0.52b MCP server mode may expose artifacts as

  `artifacts/` root as another Home subtree to inventory (complementary, not
  overlapping — v0.53 defines no CAS of its own). Nothing here pre-commits the
  v1.0 freeze.
- **Library fit.** No third-party CAS/upload dependency is added (ADR 0053). The
  store is owned in-tree; `Capsule` is parked for a future pluggable-backend
  need only.
- **ADR fit.** ADR 0053 is new and binding. ADR 0042 gains the artifact resource
  class amendment. ADR 0031 (fragment authority) and ADR 0046 (`schema_version`)
  are consumed unchanged.

## Locked Decisions

These choices are binding for v0.50. Changing any requires an explicit
re-decision here and in the request-flow document before code lands.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Store implementation | Thin in-tree CAS over BEAM primitives (`:crypto` SHA-256 + `Base.encode16` + sharded objects + atomic temp-write/`rename`); no third-party CAS/upload dependency. | hashfs/scarab are abandoned and metadata-free; waffle/Capsule are not content-addressed stores; adopting one adds supply-chain/maintenance risk and an abstraction that fights the Allbert-Home layout. |
| 2 | Artifact identity | `artifact://sha256/<hex>` — an immutable, dedup-keyed content address minted by hashing bytes, distinct from the mutable capture-id schemes (`mic://`, `image://`, `screen://`). | A mutable or path-based id loses dedup and breaks cross-surface lookup. |
| 3 | Storage layout | `<ALLBERT_HOME>/artifacts/objects/<ab>/<cd>/<sha256>` (two-level shard from the hash prefix); a real Home root registered through `Paths`/`Runtime.Paths`. | A flat dir bloats; an unregistered ad-hoc path repeats the v0.48/v0.49 mistake of media roots living only as settings strings. |
| 4 | Metadata index | Trace-safe, markdown-first per-artifact record (sidecar under `artifacts/index/`) plus an in-memory lookup index; fields are a strict allow-list. The object layer holds bytes; metadata is decoupled (the good idea from scarab's namespace→hash indirection, without scarab). | Embedding metadata in object paths couples identity to mutable attributes; a binary-only DB loses inspectability/portability. |
| 5 | Metadata allow-list | `{sha256, mime, byte_size, origin, source_resource_uri, created_at, retention, redaction_status, lifecycle}`; raw bytes/filenames-as-content never enter traces/audits/assigns/CLI. | A free-form metadata map risks leaking sensitive content into traces. |
| 6 | Permission floors | `artifact_read` floor `:allowed`; `artifact_write` floor `:allowed` (local durable storage, gated by `artifacts.enabled` + retention policy, no external/billable call); `artifact_delete` floor `:needs_confirmation` (destructive, mirrors `notes_file_write`). | Treating delete as `:allowed` lets durable artifacts vanish without confirmation; treating write as `:needs_confirmation` would spam confirmations on every retained capture. |
| 7 | Retention | Default-off (`artifacts.retention_enabled` false), reusing the v0.48 `retention_enabled`-keyed mechanism; durable artifacts are operator-removable; a mark-and-sweep GC reconciles the metadata index against on-disk objects. | Default-on durable retention stores sensitive media without consent. |
| 8 | Ingestion | `put`/`get`/`list`/`delete` registered actions plus the codebase's first Jido **sensor** for auto-ingesting the durable-retention branch of existing capture flows. The sensor is advisory ingestion only; it never grants authority. | A custom GenServer watcher would re-invent what `Jido.Sensor` provides; making the sensor authoritative would violate the descriptive-evidence rule. |
| 9 | Backfill scope | Migrate the **retained** branch of v0.48 audio (`<ALLBERT_HOME>/audio`) and v0.49 image media (`<ALLBERT_HOME>/images`) into the CAS and formalize those as real Home roots. Ephemeral scratch (`tmp/voice-synthesis`, transient `tmp/voice-captures`, `tmp/telegram-voice`) stays ephemeral. | Migrating ephemeral scratch wastes dedup and complicates lifecycle; leaving retained media outside the CAS keeps two durable-media schemes. |
| 10 | Authority | Capability/identity are routing predicates only. A content address, a doctor success, or a stored metadata record never grants read/write/send permission. | Metadata-as-authority breaks the Security Central boundary. |

## Content-Addressable Store Contract

The object store is a pure blob layer with a decoupled metadata index.

- **Hashing.** `:crypto.hash(:sha256, bytes)` (or the streaming
  `hash_init/:sha256` → `hash_update` → `hash_final` form for large files),
  lowercased Base16. The hash is the artifact's whole identity.
- **Layout.** `<ALLBERT_HOME>/artifacts/objects/<first2>/<next2>/<full_sha256>`.
- **Dedup.** Identical content hashes to the identical path, so a re-`put` is
  idempotent: write to a temp file in the same filesystem, then `File.rename`
  into place (atomic on same-FS), discarding the temp if the object already
  exists. Concurrent writers of identical content converge without locks.
- **Metadata.** Each object has a markdown sidecar
  `<ALLBERT_HOME>/artifacts/index/<first2>/<next2>/<full_sha256>.md` carrying the
  Locked-Decision-5 allow-list, kept inspectable/portable per Allbert's
  markdown-first value. An in-memory index caches `sha256 → metadata` for
  `list`/lookup.
- **Origin** is one of `:operator_upload`, `:allbert_created`, `:allbert_found`
  — provenance, not authority.
- **Retention/GC.** Honors `artifacts.retention_enabled`; a supervised
  mark-and-sweep task reconciles index vs. objects and performs
  operator-policy-driven removal. `delete` removes both object and sidecar and
  is confirmation-gated.

## Artifact Resource And Authority Model

v0.50 adds the artifact slice of ADR 0042:

- `artifact://sha256/<hex>` identifies a durable, content-addressed artifact.
  The id is the lowercase hex SHA-256 (`^[a-f0-9]{64}$`); ids are opaque,
  Allbert-Home-local, and never provider-selected.
- Transport/capture URIs (`image://`, `screen://`, `mic://`, `browser://`,
  generated-media handles) may **point at or derive** an `artifact://` address,
  but they do not become the store's authority model and the store does not
  inherit their mutability.
- Stored content is default-off for retention and redacted from traces by
  default. Traces record bounded metadata only: artifact URI, byte size, MIME,
  origin, content hash, redaction status. Raw bytes never appear in traces,
  audits, diagnostics, LiveView assigns, or CLI output.

New permission classes:

- `:artifact_read` — floor `:allowed` (read durable metadata/bytes through
  Resource Access).
- `:artifact_write` — floor `:allowed` (local durable write, gated by
  `artifacts.enabled` + retention policy; no external/billable call).
- `:artifact_delete` — floor `:needs_confirmation` (destructive; reuses the
  `notes_file_write`-style confirmation floor).

New operation classes / origin:

- origin kind `:artifact_store`
- operation `:artifact_read`
- operation `:artifact_write`
- operation `:artifact_delete`

## Settings

`artifacts.*` fragment (`schema_version: 1`, ADR 0046; fragment authority per
ADR 0031), cloning the voice/vision fragment shape:

- `artifacts.schema_version` (`writable?: false`, 1..1).
- `artifacts.enabled` (bool, default **false**).
- `artifacts.root` (string, default `<ALLBERT_HOME>/artifacts`).
- `artifacts.retention_enabled` (bool, default **false**).
- `artifacts.max_bytes` (bounded_integer) and `artifacts.allowed_mime` /
  `artifacts.allowed_types` (lists; empty = accept any) — type-agnostic by
  default, constrainable by policy.
- `artifacts.dedup` (enum, default `content_sha256`).
- `artifacts.gc` policy keys (retention window / on-demand only).
- `artifacts.trace.redact_bytes` (bool, default **true**).

## Provider Doctor

`mix allbert.artifacts doctor` (or the existing doctor surface) reports, with the
ADR 0047 redacted envelope, additive read-only fields only: `:store_writable`,
`:object_count`, `:index_consistent`, `:retention_enabled`,
`:orphan_object_count`, `:orphan_index_count`. Never emits raw bytes, absolute
paths, or content.

## Security Posture

- A content address, metadata record, or doctor success never grants permission;
  Security Central + Resource Access remain the authority boundary.
- Operator-supplied, Allbert-created, and Allbert-found artifacts only; no
  arbitrary remote fetch beyond already-approved tool paths.
- `artifact_delete` is destructive and confirmation-gated.
- Binary content is bounded, default-off for retention, and absent from
  traces/audits/assigns/CLI; only allow-listed metadata persists.
- The Jido ingestion sensor is advisory: it ingests durable-retention writes but
  never elevates authority.

## Non-Goals

- No third-party CAS/upload dependency (build-own per ADR 0053).
- No arbitrary remote media fetch or sync service.
- No permanent retention by default.
- No content interpretation/transformation (no resize/transcode/OCR) in v0.50.
- No metadata-as-authority; no auto-promotion of artifacts to memory.
- No IPFS/IPLD/CID interchange (dasl-style) in v0.50.

## Milestones

### M1: Content-Addressable Object Store + Metadata Index

Status: planned.

Implementation tasks:

- Add `artifacts_root/0` to `AllbertAssist.Paths` (escape-hatch precedence:
  app-root config → `configured(:artifacts_root)` → `ALLBERT_ARTIFACTS_ROOT` →
  `<ALLBERT_HOME>/artifacts`), wire it through `ensure_home!/0`, and expose it
  via `Runtime.Paths` (`root/1`, `@type root_name`, `roots/0`).
- Implement the object store: SHA-256 hashing (whole-binary and streaming),
  lowercase Base16, two-level shard layout, atomic temp-write + `File.rename`,
  idempotent dedup, and read-back-by-hash.
- Implement the markdown-first metadata sidecar index (Locked Decision 5
  allow-list) plus an in-memory `sha256 → metadata` lookup index, reusing the
  `Marketplace.Bundle` `sha256:` helpers where useful.
- Write ADR 0053 (store identity/layout/dedup/retention/security + the
  build-vs-library decision) and the ADR 0042 artifact resource class amendment.

Acceptance criteria: identical content dedups to one object; atomic writes leave
no partial objects on crash; metadata round-trips through the sidecar; the
`artifacts/` root is a registered Home root; ADRs accepted.

Automated tests: object store (hash/shard/atomic/dedup), metadata index,
`Paths`/`Runtime.Paths` root registration.
