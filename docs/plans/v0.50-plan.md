## Purpose

v0.50 adds **Artifacts Central**: a uniform, type-agnostic, content-addressable
store (CAS) for durable artifacts that are **uploaded by the operator**,
**created by Allbert**, or **found by Allbert** through approved tools such as
browser research. Artifacts can be any type — audio, video, images, PDFs, text,
office documents, spreadsheets — because the store addresses content by hash and
never interprets it.

The store is built **in-tree on BEAM primitives**, not on a third-party
package. The research-backed reason (recorded in ADR 0053):

- The CAS-specific Hex packages are dead ends: `hashfs` (last release 2017, repo
  gone, SHA-1/git-blob hashing, no metadata) and `scarab` (2016, pre-1.0, no
  metadata) are abandoned; `dasl` is an IPLD/CID *encoding* layer, not a local
  file store.
- The upload utilities are not content-addressed stores: `waffle` keys storage
  by filename/scope (not content hash) and drags in `hackney` + an Ecto/
  ImageMagick-shaped workflow; `Capsule` (Apache-2.0, zero-dep) is a clean
  `Storage`/`Locator` abstraction but explicitly leaves hashing/addressing to
  the caller.
- The "hard part" is stdlib: `:crypto.hash(:sha256, …)` (streaming-capable),
  `Base.encode16/2`, sharded `Path.join`, and atomic `File.rename` cover hashing
  + layout completely in ~100–150 reviewable lines, with zero supply-chain,
  licensing, or maintenance risk and no heavy deps. `Capsule` remains a parked
  option only if a later version needs a pluggable S3 backend.

v0.50 is deliberately **substrate-heavy and primitive-light**: nearly everything
it needs already exists (Allbert Home roots, the resource-URI scheme system,
Security Central operation/permission classes, the Settings Central fragment
pattern, the allow-list redactor, the default-off retention mechanism, the Jido
action Registry/Runner). The genuinely new primitives are the content-addressed
object store itself, the trace-safe artifact metadata index, the
`artifact://sha256/<hex>` identity model, the codebase's **first supervised
Jido.Sensor ingestion path**, and an **artifact↔thread/message provenance link**
so an artifact knows which conversation (and turn) created or referenced it.

**Provenance and browsing — the app shape.** Artifacts Central is more than a
blob bucket: it links files to the threads/messages that produced them and is
meant to be *browsed* like a small app (StockSage-style operator surfaces over
a local-first record set). v0.50 ships the **core kernel** — the store, the
`artifact://` scheme, security, settings, the actions, the sensor, and the
artifact↔thread/message links — plus the core read/query actions a browser
consumes. The operator **browsing repository** itself (a workspace LiveView
panel, an `/apps/artifacts/<sha>` detail page, and a `mix allbert.artifacts`
CLI) ships in the focused sidecar **v0.50b Artifacts Browser** as a plugin/app,
modeled on StockSage and `allbert.browser`. The split keeps the core release
shippable and dogfoods the App/SurfaceProvider contract; the store stays core
because channels (v0.51), MCP (v0.52b), and export (v0.53) depend on it and
durable Home data cannot live in a plugin. Both decisions are recorded in
ADR 0054.

## Readiness Sweep

Status: complete; implementation-ready for M1-M7.

- **Substrate fit.** Artifacts Central is a storage substrate under the runtime,
  not a new runtime. It composes with CLI/workspace/channels through Resource
  Access, Security Central, confirmations, traces, and audits. A stored artifact
  is descriptive evidence, never authority; nothing auto-promotes to memory and
  content-addressed identity never grants permission.
- **Roadmap fit.** v0.50 follows v0.49 vision and precedes v0.51 Channel Pack 1.
  It completes the v0.42 media-resource vision (audio in v0.48, image/screenshot
  in v0.49, durable cross-surface artifacts here) and gives channels a canonical
  artifact home before they forward attachments.
- **Past-plan fit.** Reuses, in place: the `AllbertAssist.Paths` /
  `Runtime.Paths` Home-root facade; the `ResourceURI` scheme-plugin pattern
  (mint/normalize/derive/scope); the `OperationClass` + `Security.Policy`
  closed-vocabulary permission/floor machinery (the v0.48 `voice_floor` and
  `notes_file_write` floors are templates); the `Settings.Schema` voice/vision
  fragment shape (`schema_version` + retention + redact-flag); the allow-list
  `Runtime.Redactor` (which already treats `content_sha256` as trace-safe); the
  default-off `retention_enabled` mechanism + `cleanup_transient_capture`
  lifecycle; the `Actions.Registry` + `Actions.Runner.run/3` action path; and
  the `Marketplace.Bundle` SHA-256 helpers + `sha256:` convention,
  `ImageMetadata.content_sha256`, and the `Ref.digest` slot.
- **Future-plan fit.** **v0.50b Artifacts Browser** ships the operator browsing
  surfaces (panel + page route + CLI) as a plugin/app over the core read
  actions. v0.51+ channel plugins may forward artifact references but do not own
  store authority. v0.52b MCP server mode may expose artifacts as read resources
  under the same redaction policy. v0.53 export/import treats the `artifacts/`
  root as another Home subtree to inventory (complementary, not overlapping —
  v0.53 defines no CAS of its own). Nothing here pre-commits the v1.0 freeze.
- **Provenance fit.** The artifact↔thread/message link reuses the established
  durable-record→thread pattern (objectives, canvas tiles, jobs all carry an
  indexed `thread_id` string column populated from `context.request.thread_id`).
  The link is a queryable SQLite join table plus a denormalized originating
  thread/message id in the markdown sidecar; `thread_id`/`message_id` are
  provenance strings, never authority.
- **Surface fit.** The browsing repository reuses the StockSage / `allbert.browser`
  plugin pattern: an `App` + `SurfaceProvider` contributing a `:canvas_panels`
  panel hydrated by `workspace_panel_surfaces/1`, an `/apps/artifacts/<sha>`
  page LiveView (route in the core router, module plugin-owned), and a
  `mix allbert.artifacts` CLI that is a thin shell over the core `:artifact_read`
  actions. A plugin can render the store but cannot own the scheme, permissions,
  Home root, or sensor.
- **Library fit.** No third-party CAS/upload dependency is added (ADR 0053). The
  store is owned in-tree; `Capsule` is parked for a future pluggable-backend
  need only.
- **ADR fit.** ADR 0053 is new and binding. ADR 0054 is new and binding
  (provenance linking + core/plugin surface split); ADR 0053 is amended to point
  at it. ADR 0042 gains the artifact resource class amendment. ADR 0031
  (fragment authority), ADR 0046 (`schema_version`), and the ADR 0015/0017
  app/plugin contract are consumed unchanged.

## Implementation Seams

The names below are the expected implementation seams; implementations may make
small naming adjustments, but must preserve the boundaries and tests.

- **Core context.** Add `AllbertAssist.Artifacts` as the public facade over
  `AllbertAssist.Artifacts.Store`, `MetadataIndex`, `ThreadLinks`, `GC`, and
  `Doctor`. Store code owns object bytes; metadata/index code owns sidecars and
  list/query summaries; thread-link code owns the Repo-backed provenance edge.
- **Ecto layer.** Add `AllbertAssist.Artifacts.ThreadLink` and a
  `CreateArtifactThreadLinks` migration. The link id is a deterministic string
  derived from `{sha256, user_id, thread_id, message_id || input_signal_id ||
  "thread", role}` so repeated puts upsert the same edge even when
  `message_id` is nil. Add indexes on `[:artifact_sha256]`,
  `[:thread_id, :user_id]`, and `[:user_id, :artifact_sha256]`.
- **Action layer.** Add internal registered actions under
  `AllbertAssist.Actions.Artifacts.{PutArtifact,GetArtifact,ListArtifacts,
  DeleteArtifact,Threads,Doctor}` with names `put_artifact`, `get_artifact`,
  `list_artifacts`, `delete_artifact`, `artifact_threads`, and
  `artifact_doctor`. Add `Actions.Capability.execution_mode` values
  `:artifact_read`, `:artifact_write`, `:artifact_delete`, and
  `:artifact_doctor`; `delete_artifact` remains `confirmation: :required`.
- **Settings and policy.** Register `artifacts.*` in `Settings.Schema` defaults,
  schema, and safe-write keys; register `permissions.artifact_read`,
  `permissions.artifact_write`, and `permissions.artifact_delete` in Settings
  Central and `Security.Policy`.
- **Resource Access.** Add `ResourceURI.artifact/1` and `artifact!/1`,
  `artifact://sha256/<hex>` normalization, `derived_fields/1` origin kind
  `:artifact_store`, and a `scope_uri/4` clause for artifact grants if a later
  caller needs scoped Resource Access.
- **Sensor shape.** The ingestion sensor uses `use Jido.Sensor` and runs under
  `Jido.Sensor.Runtime` in the application supervision tree. Per current Jido
  docs, it implements `init/2` and `handle_event/2`, emits ingestion-request
  signals such as `allbert.artifact.ingest_requested` via `{:emit, signal}`, and
  schedules/polls only when needed. It must not write around the core `put`
  path; ingestion either emits a signal consumed by core code or invokes
  `Actions.Runner.run("put_artifact", ...)` with a sensor context that still
  honors `artifacts.enabled`, retention, bounds, redaction, and permission
  floors.
- **Message precision.** `input_signal_id` resolves to a `conversation_messages`
  row by `{user_id, thread_id, input_signal_id}`. If no row exists, the link is
  still recorded at thread level with `message_id: nil` and
  `metadata.input_signal_id`; message precision is best-effort provenance, not
  a release blocker.
- **Backfill scope.** M5 backfills retained v0.48 voice captures, v0.49 vision
  input media, and v0.49 generated-image output roots:
  `<ALLBERT_HOME>/audio`, `<ALLBERT_HOME>/images`, and
  `<ALLBERT_HOME>/generated_images`. Ephemeral scratch and historical Browser
  cache backfill are excluded. Browser/download/cache artifacts may call
  `put_artifact` in future flows, but v0.50 release evidence covers fixtures and
  retained voice/vision/image roots only.

## Locked Decisions

These choices are binding for v0.50. Changing any requires an explicit
re-decision here and in the request-flow document before code lands.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Store implementation | Thin in-tree CAS over BEAM primitives (`:crypto` SHA-256 + `Base.encode16` + sharded objects + atomic temp-write/`rename`); no third-party CAS/upload dependency. | hashfs/scarab are abandoned and metadata-free; waffle/Capsule are not content-addressed stores; adopting one adds supply-chain/maintenance risk and an abstraction that fights the Allbert-Home layout. |
| 2 | Artifact identity | `artifact://sha256/<hex>` — an immutable, dedup-keyed content address minted by hashing bytes, distinct from the mutable capture-id schemes (`mic://`, `image://`, `screen://`). | A mutable or path-based id loses dedup and breaks cross-surface lookup. |
| 3 | Storage layout | `<ALLBERT_HOME>/artifacts/objects/<ab>/<cd>/<sha256>` (two-level shard from the hash prefix); a real Home root registered through `Paths`/`Runtime.Paths`. | A flat dir bloats; an unregistered ad-hoc path repeats the v0.48/v0.49 mistake of media roots living only as settings strings. |
| 4 | Metadata index | Trace-safe, markdown-first per-artifact record (sidecar under `artifacts/index/`) plus an in-memory lookup index; fields are a strict allow-list. The object layer holds bytes; metadata is decoupled (the good idea from scarab's namespace→hash indirection, without scarab). | Embedding metadata in object paths couples identity to mutable attributes; a binary-only DB loses inspectability/portability. |
| 5 | Metadata allow-list | `{sha256, mime, byte_size, origin, source_resource_uri, created_at, retention, redaction_status, lifecycle, provenance}` where `provenance` is a bounded thread/message/id summary only; raw bytes/filenames-as-content never enter traces/audits/assigns/CLI. | A free-form metadata map risks leaking sensitive content into traces. |
| 6 | Permission floors | `artifact_read` floor `:allowed`; `artifact_write` floor `:allowed` (local durable storage, gated by `artifacts.enabled` + retention policy, no external/billable call); `artifact_delete` floor `:needs_confirmation` (destructive, mirrors `notes_file_write`). | Treating delete as `:allowed` lets durable artifacts vanish without confirmation; treating write as `:needs_confirmation` would spam confirmations on every retained capture. |
| 7 | Retention | Default-off (`artifacts.retention_enabled` false), reusing the v0.48 `retention_enabled`-keyed mechanism; durable artifacts are operator-removable; a mark-and-sweep GC reconciles the metadata index against on-disk objects. | Default-on durable retention stores sensitive media without consent. |
| 8 | Ingestion | `put`/`get`/`list`/`delete` registered actions plus the codebase's first supervised `Jido.Sensor` path for auto-ingesting the durable-retention branch of existing capture flows. The sensor is advisory ingestion only; it never grants authority. | A custom GenServer watcher would re-invent what `Jido.Sensor` provides; making the sensor authoritative would violate the descriptive-evidence rule. |
| 9 | Backfill scope | Migrate the **retained** branch of v0.48 audio (`<ALLBERT_HOME>/audio`), v0.49 vision media (`<ALLBERT_HOME>/images`), and v0.49 generated-image media (`<ALLBERT_HOME>/generated_images`) into the CAS and formalize those as real Home roots. Ephemeral scratch (`tmp/voice-synthesis`, transient `tmp/voice-captures`, `tmp/telegram-voice`) and historical Browser cache files stay out of M5. | Migrating ephemeral/cache scratch wastes dedup and complicates lifecycle; leaving retained media outside the CAS keeps multiple durable-media schemes. |
| 10 | Authority | Capability/identity are routing predicates only. A content address, a doctor success, or a stored metadata record never grants read/write/send permission. | Metadata-as-authority breaks the Security Central boundary. |
| 11 | Provenance linking | Artifact↔thread/message links live in a SQLite join table `artifact_thread_links(id, artifact_sha256, thread_id, message_id, role, user_id, metadata)` (`role ∈ created_by \| referenced_by`), indexed on `sha256`, `[thread_id, user_id]`, and `[user_id, artifact_sha256]`, plus a denormalized originating thread/message id in the markdown sidecar. The link id is deterministic from the normalized edge tuple so repeated puts are idempotent. `thread_id`/`message_id` come from `context.request` at put-time (message via `request.input_signal_id`); they are provenance strings, never enforced FKs or authority. | A single column on the artifact under-models a content-addressed object shared across threads with different roles; markdown-only loses the query index; non-idempotent links create duplicate browser rows. |
| 12 | Browsing surfaces | The operator browsing repository (workspace panel + `/apps/artifacts/<sha>` page + CLI) ships in **v0.50b Artifacts Browser** as a plugin/app reading the store only through core `:artifact_read` actions. v0.50 ships the core read/query actions it consumes (including `list_artifacts` by thread). | Building the browser into core bloats it and skips the App/SurfaceProvider contract; a plugin owning the store would invert the channel/MCP/export dependency. |

## Content-Addressable Store Contract

The object store is a pure blob layer with a decoupled metadata index.

- **Hashing.** `:crypto.hash(:sha256, bytes)` (or the streaming
  `hash_init/:sha256` → `hash_update` → `hash_final` form for large files),
  lowercased Base16. The hash is the artifact's whole identity.
- **Layout.** `<ALLBERT_HOME>/artifacts/objects/<first2>/<next2>/<full_sha256>`.
- **Dedup.** Identical content hashes to the identical path, so a re-`put` is
  idempotent: write to a temp file in the same filesystem, then `File.rename`
  into place (atomic on same-FS), discarding the temp if the object already
  exists. Concurrent writers of identical content converge without locks.
- **Metadata.** Each object has a markdown sidecar
  `<ALLBERT_HOME>/artifacts/index/<first2>/<next2>/<full_sha256>.md` carrying the
  Locked-Decision-5 allow-list, kept inspectable/portable per Allbert's
  markdown-first value. An in-memory index caches `sha256 → metadata` for
  `list`/lookup.
- **Origin** is one of `:operator_upload`, `:allbert_created`, `:allbert_found`
  — provenance, not authority.
- **Retention/GC.** Honors `artifacts.retention_enabled`; a supervised
  mark-and-sweep task reconciles index vs. objects and performs
  operator-policy-driven removal. `delete` removes both object and sidecar and
  is confirmation-gated.

## Artifact Resource And Authority Model

v0.50 adds the artifact slice of ADR 0042:

- `artifact://sha256/<hex>` identifies a durable, content-addressed artifact.
  The id is the lowercase hex SHA-256 (`^[a-f0-9]{64}$`); ids are opaque,
  Allbert-Home-local, and never provider-selected.
- Transport/capture URIs (`image://`, `screen://`, `mic://`, `browser://`,
  generated-media handles) may **point at or derive** an `artifact://` address,
  but they do not become the store's authority model and the store does not
  inherit their mutability.
- Stored content is default-off for retention and redacted from traces by
  default. Traces record bounded metadata only: artifact URI, byte size, MIME,
  origin, content hash, redaction status. Raw bytes never appear in traces,
  audits, diagnostics, LiveView assigns, or CLI output.

New permission classes:

- `:artifact_read` — floor `:allowed` (read durable metadata/bytes through
  Resource Access).
- `:artifact_write` — floor `:allowed` (local durable write, gated by
  `artifacts.enabled` + retention policy; no external/billable call).
- `:artifact_delete` — floor `:needs_confirmation` (destructive; reuses the
  `notes_file_write`-style confirmation floor).

New operation classes / origin:

- origin kind `:artifact_store`
- operation `:artifact_read`
- operation `:artifact_write`
- operation `:artifact_delete`

## Provenance And Thread Linking

Artifacts Central links every artifact to the conversation that produced or
referenced it, so an operator (and the v0.50b browser) can ask "what files came
from this thread / this turn?" and, from an artifact, "which thread created
this?".

- **Durable representation.** A SQLite join table on `AllbertAssist.Repo`,
  `artifact_thread_links(id, artifact_sha256, thread_id, message_id, role,
  user_id, metadata, timestamps)`, with `role ∈ {created_by, referenced_by}`,
  indexed on `[:artifact_sha256]` (reverse lookup) and `[:thread_id, :user_id]`
  (browse-by-thread), plus `[:user_id, :artifact_sha256]` for scoped reverse
  lookup. This mirrors the existing durable-record→thread pattern
  (`objectives.source_thread_id`, `workspace_canvas_tiles.thread_id`,
  scheduled-job `thread_id`) — a plain indexed string column, **not** an
  enforced FK, consistent with "thread id is never authority". The deterministic
  link id makes duplicate puts idempotent. The originating
  `thread_id`/`message_id`/`input_signal_id` is additionally denormalized into
  the artifact's markdown sidecar as bounded `provenance` metadata for
  human-readable provenance (mirrors the trace `Thread:` line and the canvas
  tile sidecar).
- **Where the link comes from.** At put-time the artifact actions read
  `request = Map.get(context, :request, %{})` then `request.thread_id`,
  `request.user_id`, `request.session_id` — the canonical read used by
  `append_memory`, `objectives`, and `confirmations/origin`. Message-level
  attribution uses `request.input_signal_id` (persisted on the
  `conversation_messages` row); the runtime resolves it by
  `{user_id, thread_id, input_signal_id}` to a `msg_…` id when possible.
  Otherwise the link is thread-level with the signal id retained in metadata.
- **Query path.** "List artifacts created in thread X" is an Ecto query over the
  join table (user-scoped, role-filtered), the same idiom as
  `Conversations.list_messages/2` and `Objectives.list_objectives/2` — not a
  markdown scan. The reverse ("which thread created this sha256") filters the
  same table by `artifact_sha256`.
- **Authority.** A link is provenance, never permission. Listing artifacts by
  thread still resolves through `:artifact_read`; a thread id grants nothing.

## Settings

`artifacts.*` fragment (`schema_version: 1`, ADR 0046; fragment authority per
ADR 0031), cloning the voice/vision fragment shape:

- `artifacts.schema_version` (`writable?: false`, 1..1).
- `artifacts.enabled` (bool, default **false**).
- `artifacts.root` (string, default `<ALLBERT_HOME>/artifacts`).
- `artifacts.retention_enabled` (bool, default **false**).
- `artifacts.max_bytes` (bounded_integer) and `artifacts.allowed_mime` /
  `artifacts.allowed_types` (lists; empty = accept any) — type-agnostic by
  default, constrainable by policy.
- `artifacts.dedup` (enum, default `content_sha256`).
- `artifacts.gc` policy keys (retention window / on-demand only).
- `artifacts.trace.redact_bytes` (bool, default **true**).

## Provider Doctor

A core `artifact_doctor` read action reports, with the ADR 0047 redacted
envelope, additive read-only fields only: `:store_writable`, `:object_count`,
`:index_consistent`, `:retention_enabled`, `:orphan_object_count`,
`:orphan_index_count`, `:link_count`. Never emits raw bytes, absolute paths, or
content. v0.50 surfaces it through the existing doctor surface; the
`mix allbert.artifacts doctor` CLI wrapper ships with v0.50b.

## Security Posture

- A content address, metadata record, or doctor success never grants permission;
  Security Central + Resource Access remain the authority boundary.
- Operator-supplied, Allbert-created, and Allbert-found artifacts only; no
  arbitrary remote fetch beyond already-approved tool paths.
- `artifact_delete` is destructive and confirmation-gated.
- Binary content is bounded, default-off for retention, and absent from
  traces/audits/assigns/CLI; only allow-listed metadata persists.
- The Jido ingestion sensor is advisory: it ingests durable-retention writes but
  never elevates authority.

## Non-Goals

- No third-party CAS/upload dependency (build-own per ADR 0053).
- No arbitrary remote media fetch or sync service.
- No permanent retention by default.
- No content interpretation/transformation (no resize/transcode/OCR) in v0.50.
- No metadata-as-authority; no auto-promotion of artifacts to memory.
- No IPFS/IPLD/CID interchange (dasl-style) in v0.50.

### M4: Provenance And Thread Linking

Status: planned.

Implementation tasks:

- Add the `artifact_thread_links` join table (Repo migration under
  `apps/allbert_assist/priv/repo/migrations/`): string PK, `artifact_sha256`,
  `thread_id`, `message_id` (nullable), `role` (`created_by`/`referenced_by`),
  `user_id`, `metadata`, timestamps; indexes on `[:artifact_sha256]` and
  `[:thread_id, :user_id]`, and `[:user_id, :artifact_sha256]`. The primary key
  is deterministic from `{sha256, user_id, thread_id, message_id ||
  input_signal_id || "thread", role}` so repeated puts are idempotent.
  `thread_id`/`message_id` are plain strings, not FKs.
- Record links at put-time from `context.request` (`thread_id`, `user_id`,
  `session_id`; message via `request.input_signal_id`, resolved by
  `{user_id, thread_id, input_signal_id}` when the conversation message row
  exists). If the message lookup misses, record the thread-level link with
  `message_id: nil` and `metadata.input_signal_id`, and denormalize only the
  bounded originating thread/message/input-signal summary into the markdown
  sidecar.
- Add `list_artifacts` by-thread filtering and a reverse `artifact_threads`
  lookup, as Ecto queries over the join table (user-scoped, role-filtered),
  mirroring `Conversations.list_messages/2` / `Objectives.list_objectives/2`.

Acceptance criteria: a `put` inside a thread records a `created_by` link;
listing by thread returns it; the reverse lookup returns the thread; repeating
the same put does not duplicate link rows; a thread id grants no permission

## Definition Of Done

v0.50 is done when:

- a type-agnostic content-addressable store exists under
  `<ALLBERT_HOME>/artifacts`, addressing artifacts by `artifact://sha256/<hex>`
  with deduplicated atomic object storage and a trace-safe markdown-first
  metadata index;
- `put`/`get`/`list`/`delete` actions resolve through the Registry/Runner, with
  `delete` confirmation-gated;
- `:artifact_read`/`:artifact_write`/`:artifact_delete` permissions, operation
  classes, floors, and the `artifacts` redaction surface are in place, and
  content-addressed identity never grants permission;
- the `artifacts.*` settings fragment governs enable/retention/bounds/dedup with
  default-off retention;
- artifacts link to the threads/messages that created or referenced them via the
  `artifact_thread_links` join table, populated from `context.request`, with a
  by-thread `list_artifacts` query and reverse lookup, and the link grants no
  authority;
- the retained branch of v0.48 audio, v0.49 vision input media, and v0.49
  generated-image media is migrated into the CAS and those roots are
  formalized, with ephemeral scratch and historical Browser cache untouched;
- the first supervised Jido ingestion sensor auto-ingests durable-retention writes
  advisory-only;
- the v0.50 security eval set + `release.v050` pass on fixtures;
- roadmap/vision/agent-context-map/ADR 0053/0054/0042/CHANGELOG reflect the
  shipped state.

The operator browsing repository (workspace panel, `/apps/artifacts/<sha>` page,
and `mix allbert.artifacts` CLI) is **not** part of v0.50 — it ships in the
**v0.50b Artifacts Browser** plugin/app over the core read actions delivered
here.

Release-state notes: the store is owned in-tree per ADR 0053; no third-party
CAS/upload dependency is added in v0.50. `Capsule` remains a parked option for a
future pluggable-backend need only.
