## Purpose

v0.55 shipped the TUI as a real channel and a basic launcher. v0.55.1 turns that
launcher into the persistent, **mix-free operator/validation console**: interactive
operator work and future-version operator validation run in **one warm BEAM** (the
SQLite store stays open, `Channels.Supervisor` stays up) instead of cold per-turn
`mix allbert.ask` invocations against a shared `ALLBERT_HOME`.

Three concrete deliverables:

1. **In-TUI slash-commands** — a fixed, canonical operator inspection set,
   **defined once here**: `/status`, `/confirmations`, `/events`, `/channels`,
   `/settings get`, `/help`. Each is backed by a **registered READ-ONLY operator
   inspection action resolved through `AllbertAssist.Actions.Runner.run/3`**,
   exposed **operator-only (NOT an intent candidate — the model must not route to
   it)**, performing no mutation, emitting redacted output, granting no new
   authority. Each is a thin view over the **same context read facade the existing
   `mix allbert.*` inspection tasks call** (single source of truth), so the console
   and the cold tasks never drift. Governed by **ADR 0070**.
2. **`mix allbert.channels status`** — an operator-readable channel status report
   (registered channel release/enabled/supervisor status) over the same read
   facade, with no secret output and no new authority. v0.55 listed this command
   in scope as supporting crash/isolation evidence; v0.55.1 lands it as a thin view
   sharing the `/channels`/`/status` facade.
3. **Migrate operator-validation onto the warm console** — move the
   operator-validation and interactive-testing instructions off cold `mix
   allbert.ask`/`mix allbert.*` per-turn invocations and onto the warm in-session
   console, and set the warm console as the **go-forward standard** for
   interactive operator validation. `mix allbert.test` (deterministic CI gates)
   stays unchanged — this targets interactive operator/validation flows only.

This also **strengthens the v0.57 Pi-mode foundation**: Pi-mode runs in this same
persistent console (one warm node, DB open, supervisor up), so the mix-free
operator surface is the surface Pi-mode interactive sessions later attach to.

## Readiness Sweep

- **Contract fit.** The slash-commands add **no authority** and **no domain
  semantics** to the channel. Each resolves a registered action through
  `Actions.Registry` + `Actions.Runner.run/3` (the same path every channel uses;
  DEVELOPMENT.md "Architecture Contract"), reads through a context facade, and
  renders a redacted view. They never touch the Repo, Security Central, the
  confirmation store, or Settings authority directly. The TUI stays a delivery
  adapter (ADR 0016); the console is a thin operator read surface on top of it.
- **Reuse fit.** The existing `mix allbert.channels`, `mix allbert.confirmations`,
  and `mix allbert.settings` tasks already resolve through
  `Runner.run(action_name, params, context())` over context read facades. v0.55.1
  introduces no new read paths: the slash-commands and `mix allbert.channels
  status` are **alternative thin views over those same facades**, so a single
  source of truth backs both the cold task and the warm console.
- **Upstream check.** Confirm v0.55 (`0.55.0`) is tagged and that
  `plugins/allbert.tui` (descriptor, `Channels.Supervisor` child, `:tui_session`
  strategy, scrollback + `Owl.LiveScreen` render, split-payload seam, approval
  rendering) and the **basic** `mix allbert.tui` launcher exist. If the launcher
  is still one-shot or the channel is absent, stop and reconcile v0.55 closeout
  before M0.
- **Operator-only fit.** The inspection actions must be registered such that the
  intent router **cannot** select them — they are not intent candidates. This is
  the same "exposed but not model-routable" posture the eval row
  `tui-slash-readonly-001` locks down. An operator-only registration that the
  model could still route to would be an authority/parity regression, not a detail
  for the coding agent to guess later.

## Locked Decisions
| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Console authority | The console adds **no authority** beyond the v0.55 TUI channel (ADR 0016/0070). Slash-commands only **read**. | A console that could mutate or set active app/grant resource access breaks the channel boundary. |
| 2 | Slash-command backing | Each slash-command is a **registered read-only operator inspection action** resolved through `Actions.Runner.run/3`, **not** ad-hoc TUI code calling the Repo. | Inline TUI reads fork the read path from the `mix allbert.*` tasks and bypass the action boundary. |
| 3 | Routability | Inspection actions are **operator-only**, **NOT intent candidates** — the model must not route to them. | A model-routable inspection action lets a conversation turn invoke operator reads. |
| 4 | Canonical set | The slash set is fixed and defined **once** here: `/status`, `/confirmations`, `/events`, `/channels`, `/settings get`, `/help`. | An open-ended slash set drifts per channel and per doc. |
| 5 | Single source of truth | Slash-commands + `mix allbert.channels status` are **thin views over the same context read facades** the `mix allbert.*` tasks use. | Duplicate read logic drifts the warm console from the cold tasks. |
| 6 | Mix-free standard | The warm `mix allbert.tui` console is the **go-forward standard** for interactive operator validation; `mix allbert.test` (CI gates) is unchanged. | Keeping cold per-turn `mix allbert.ask` validation reopens the DB per turn and cannot prove warm-process behaviour. |
| 7 | Output redaction | Slash-command and `mix allbert.channels status` output is **redacted/truncated**; no secrets, credentials, or raw payloads. | Leaking secrets through an operator read surface is an ADR 0016 redaction regression. |

## Foundation For v0.57 Pi-mode

v0.55.1 hardens the seam v0.57 Pi-mode (ADR 0068) builds on:

- **One warm persistent session.** Pi-mode runs *in* this same persistent console
  — node up, DB open, `Channels.Supervisor` up — not a fresh per-turn process. The
  mix-free console is the lifetime Pi-mode interactive sessions attach to.
- **Operator read surface in-session.** The read-only slash-command pattern
  (registered, operator-only, Runner-resolved, redacted) is the template a Pi-mode
  inspection affordance reuses without adding authority.
- **No boundary assumption blocked.** Every console read goes through
  `Actions.Runner.run/3` with no per-action confirmation or identity assumption
  that blocks v0.57 adding the named "local coding / sandbox level 0" trust tier on
  the same adapter and session.

## Milestones

Lane summary: M0 serial barrier (ADR 0070 + slash-command router) → M1 read-only
inspection actions → M2 `mix allbert.channels status` → M3 migrate
operator-validation onto the warm console → M4 evals + docs → **M5 operator

### M0: ADR 0070 + Slash-Command Router
Draft `docs/adr/0070-tui-operator-console-and-read-only-operator-actions.md`
(Proposed): registered read-only operator inspection actions resolved through the
Runner, operator-only / not intent candidates, redacted, no new authority; the
canonical slash set; the single-source-of-truth read-facade rule. Add the in-TUI
**slash-command router** in `plugins/allbert.tui`: the input-normalize step
recognizes a leading `/` token and dispatches to the inspection-action dispatch
path instead of `Runtime.submit_user_input/1`, so a slash line is never a model
turn. `/help` lists the canonical set; unknown slash → a redacted "unknown
command" line, never a model turn.
- **Lane:** serial barrier — the slash-router changes the TUI normalize/dispatch
  seam (shared adapter path) and ADR 0070 is the shared decision other milestones
  cite. Parallel: ADR 0070 prose (pure doc) ∥ router skeleton (no actions yet).
- **Focused:** a leading `/` dispatches to the router and **never** reaches
  `Runtime.submit_user_input/1`; `/help` lists the six commands; unknown slash is
  inert.
- **Rejoin:** ADR 0070 Proposed; slash-router merged; `/help` green.

### M1: Read-Only Operator Inspection Actions
Register the inspection actions behind `/status`, `/confirmations`, `/events`,
`/channels`, and `/settings get`, each resolved through `Actions.Runner.run/3` over
the **same context read facade** the existing `mix allbert.channels` /
`mix allbert.confirmations` / `mix allbert.settings` tasks call. Register them
**operator-only** so the intent router cannot select them (not intent candidates).
`/status` reports warm-session process/channel status; `/events` reads recent
`channel_events` rows (redacted/truncated); `/confirmations` reads pending/resolved
confirmations; `/channels` reports registered-channel status; `/settings get`
reads a settings key with secret fields redacted.
- **Lane:** serial — registration touches `Actions.Registry` (shared) and the read
  paths hit the store (db_serial for `/events`/`/confirmations`). Parallel: the
  five action modules are distinct (no shared barrier between them once registered)
  ∥ render formatting (pure).
- **Focused:** each action resolves against the warm node and returns redacted
  data; **none** is selectable by the intent router; no action performs a write,
  sets the active app, or grants resource access.
- **Rejoin:** the five inspection actions are registered, operator-only, redacted,
  and wired to their slash-commands.

### M2: `mix allbert.channels status`
Add the `status` subcommand to `mix allbert.channels` — an operator-readable
report of registered channel release/enabled/supervisor status over the **same
read facade** `/channels`/`/status` use, with no secret output and no new
authority. This is the cold-task twin of `/channels` and supporting evidence for
the M5 crash/isolation check; `/status` remains the warm-session proof.
- **Lane:** parallel — a CLI subcommand over an existing facade; no shared barrier
  with M1 beyond the shared facade.
- **Focused:** `mix allbert.channels status` prints truthful TUI + sibling-channel
  status with no secrets and matches the `/channels` view from the same facade.
- **Rejoin:** `mix allbert.channels status` ships and agrees with `/channels`.

### M3: Migrate Operator-Validation Onto The Warm Console
Migrate the operator-validation and interactive-testing **instructions** off cold
`mix allbert.ask`/`mix allbert.*` per-turn invocations and onto the warm
in-session console: a single supervised `mix allbert.tui` session holds the node,
the `Channels.Supervisor` tree, and the store open for a whole validation run, with
every inspection issued through a slash-command against that same running node. Set
the warm console as the **go-forward standard** for interactive operator
validation in the operator guide (`docs/operator/tui-channel.md`) and the
request-flow punchlist. `mix allbert.test` (deterministic CI gates) is unchanged.
- **Lane:** parallel doc-migration lanes (request-flow punchlist ∥ operator guide)
  with no code barrier; depends on M1/M2 for the commands they reference.
- **Focused:** the migrated punchlist runs entirely in-session (one warm
  `mix allbert.tui`, no cold mix per turn); every referenced slash-command and
  `mix allbert.channels status` exists from M1/M2.
- **Rejoin:** operator-validation instructions are warm-console-first and
  mix-free; the go-forward standard is recorded.

### M4: Evals + Docs
Add the `:v0551` eval rows (below) and the deterministic `release.v0551` gate.
**Write** the M5 operator-validation punchlist into
`docs/plans/v0.55b-request-flow.md`, document the canonical slash set + `mix

operator guide `docs/operator/tui-channel.md`, and reconcile the ADR 0067 console

- **Lane:** parallel doc-writing lanes ∥ eval-row authoring.
- **Focused:** `:v0551` evals green on stubs; `release.v0551` deterministic +
  reproducible; the operator-only / not-intent-candidate posture is asserted by

- **Rejoin:** punchlist + slash-set docs written; deterministic gate green.

### M5: Operator Validation (manual gate)
A real operator runs the request-flow punchlist **entirely IN-SESSION** — one warm
`mix allbert.tui`, no cold mix — and records evidence: each slash-command returns
correct redacted data, the inspection actions are operator-only (not
intent-routable), and a full validation turn (send → approve → inspect via
`/confirmations`/`/events`) completes **with no BEAM restart**. This is the manual
real-operator gate; **it blocks release (M6)**.
- **Lane:** serial, manual, single-operator, external-terminal — not
  parallelizable.
- **Focused / gate:** every slash-command returns truthful redacted data; a typed
  conversation turn never routes to an inspection action; the send → approve →
  inspect turn completes in one persistent session with no node restart.
- **Rejoin:** in-session operator-validation evidence recorded.

### M6: Release Closeout

- **Rejoin:** `v0.55.1` tagged with operator-validation + eval evidence.

## Non-Goals
- No new authority for any channel or slash-command; the console only **reads**.
- No mutating slash-commands; `/settings` is `get`-only here.
- No intent-routable inspection actions (operator-only; not intent candidates).
- No change to `mix allbert.test` (CI gates) or to the v0.55 channel/parity matrix,
  Matrix outbound, split-payload seam, or rendering model — those are v0.55 scope.
- No v0.57 Pi-mode coding actions/trust tier; v0.55.1 only hardens the session
  foundation.
