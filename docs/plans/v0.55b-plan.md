## Purpose

v0.55 shipped the TUI as a real channel and a basic launcher. v0.55.1 turns that
launcher into the persistent, **mix-free operator/validation console**: interactive
operator work and future-version operator validation run in **one warm BEAM** (the
SQLite store stays open, `Channels.Supervisor` stays up) instead of cold per-turn
`mix allbert.ask` invocations against a shared `ALLBERT_HOME`.

Three concrete deliverables:

1. **In-TUI slash-commands** — a fixed, canonical operator inspection set,
   **defined once here**: `/status`, `/confirmations`, `/events`, `/channels`,
   `/settings get`, `/help`. Each is backed by a **registered READ-ONLY operator
   inspection action resolved through `AllbertAssist.Actions.Runner.run/3`**,
   registered with `exposure: :internal` and reachable only through the TUI
   slash-command allowlist (NOT an intent candidate — the model must not route to
   it), performing no mutation, emitting redacted output, granting no new
   authority. Each is a thin view over a named read-report interface
   (`AllbertAssist.Operator.Inspection`, or narrower report modules documented in
   M1) shared by the slash actions and their `mix allbert.*` inspection twins
   (single source of truth), so the console and the cold tasks never drift.
   Governed by **ADR 0070**.
2. **`mix allbert.channels status`** — an operator-readable channel status report
   (registered channel release/enabled/supervisor status) over the same read
   report DTO source, with no secret output and no new authority. Earlier v0.55
   planning drafts listed this command for crash/isolation evidence; after the
   split, v0.55.1 lands it as a thin view sharing the `/channels`/`/status`
   report source.
3. **Migrate operator-validation onto the warm console** — move the
   operator-validation and interactive-testing instructions off cold `mix
   allbert.ask`/`mix allbert.*` per-turn invocations and onto the warm in-session
   console, and set the warm console as the **go-forward standard** for
   interactive operator validation. `mix allbert.test` (deterministic CI gates)
   stays unchanged — this targets interactive operator/validation flows only.

This also **strengthens the v0.57 Pi-mode foundation**: Pi-mode runs in this same
persistent console (one warm node, DB open, supervisor up), so the mix-free
operator surface is the surface Pi-mode interactive sessions later attach to.

## Readiness Sweep

- **Contract fit.** The slash-commands add **no authority** and **no domain
  semantics** to the channel. Each resolves a registered action through
  `Actions.Registry` + `Actions.Runner.run/3` (the same path every channel uses;
  DEVELOPMENT.md "Architecture Contract"), reads through a context facade, and
  renders a redacted view. They never touch the Repo, Security Central, the
  confirmation store, or Settings authority directly. The TUI stays a delivery
  adapter (ADR 0016); the console is a thin operator read surface on top of it.
- **Reuse fit.** The existing `mix allbert.channels`, `mix allbert.confirmations`,
  and `mix allbert.settings` tasks already resolve their action-backed reads
  through `Runner.run(action_name, params, context())`, but v0.55.1 must not assume
  every needed operator report already has a named facade. M1 defines the shared
  read-report interface (`AllbertAssist.Operator.Inspection`, or narrower
  documented report modules) for channel status and `channel_events`, and reuses
  existing confirmation/settings read actions where they already fit. The
  slash-commands and `mix allbert.channels status` are **alternative thin views
  over those same report DTOs**, so a single source of truth backs both the cold
  task and the warm console. Caveat: "reuse existing action reads where they fit"
  means sharing the underlying context facade / report DTO, **not** mapping a slash
  to an existing `exposure: :agent` read (e.g. `read_setting`). The slash backings
  are `:internal` reads; an `:agent` read kept as the slash backing would stay
  model-routable and break Locked Decision #3.
- **Upstream check.** Confirm v0.55 (`0.55.0`) is tagged and that
  `plugins/allbert.tui` (descriptor, `Channels.Supervisor` child, `:tui_session`
  strategy, scrollback + `Owl.LiveScreen` render, split-payload seam, approval
  rendering) and the **basic** `mix allbert.tui` launcher exist. If the launcher
  is still one-shot or the channel is absent, stop and reconcile v0.55 closeout
  before M0. As of the second-pass readiness sweep, `v0.55.0` is tagged and this
  upstream prerequisite is no longer a planning blocker; M0 should still verify the
  tag and channel files before editing.
- **Operator-only fit.** The inspection actions use the existing action DSL shape:
  `exposure: :internal`, plus an explicit TUI slash-command allowlist. They must
  be registered such that `Actions.Registry.agent_modules/0`, intent descriptors,
  Stage-1 prefiltering, and Stage-2 disambiguation **cannot** select them. This is
  the same "registered but not model-routable" posture the eval rows lock down. A
  new `:operator` exposure is **not** v0.55.1 scope unless a later ADR deliberately
  widens the action DSL.

## Locked Decisions
| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Console authority | The console adds **no authority** beyond the v0.55 TUI channel (ADR 0016/0070). Slash-commands only **read**. | A console that could mutate or set active app/grant resource access breaks the channel boundary. |
| 2 | Slash-command backing | Each slash-command is a **registered read-only internal inspection action** resolved through `Actions.Runner.run/3`, **not** ad-hoc TUI code calling the Repo. The operator-only surface is the TUI slash allowlist. | Inline TUI reads fork the read path from the `mix allbert.*` tasks and bypass the action boundary. |
| 3 | Routability | Inspection actions are `exposure: :internal`, slash-allowlisted, and **NOT intent candidates** — the model must not route to them. | A model-routable inspection action lets a conversation turn invoke operator reads. |
| 4 | Canonical set | The slash set is fixed and defined **once** here: `/status`, `/confirmations`, `/events`, `/channels`, `/settings get`, `/help`. | An open-ended slash set drifts per channel and per doc. |
| 5 | Single source of truth | Slash-commands + `mix allbert.channels status` are **thin views over the same named read-report interface** (`AllbertAssist.Operator.Inspection`, or narrower documented report modules) and existing action reads where they already fit. | Duplicate read logic drifts the warm console from the cold tasks. |
| 6 | Mix-free standard | The warm `mix allbert.tui` console is the **go-forward standard** for interactive operator validation; `mix allbert.test` (CI gates) is unchanged. | Keeping cold per-turn `mix allbert.ask` validation reopens the DB per turn and cannot prove warm-process behaviour. |
| 7 | Output redaction | Slash-command and `mix allbert.channels status` output is **redacted/truncated**; no secrets, credentials, or raw payloads. | Leaking secrets through an operator read surface is an ADR 0016 redaction regression. |

## Foundation For v0.57 Pi-mode

v0.55.1 hardens the seam v0.57 Pi-mode (ADR 0068) builds on:

- **One warm persistent session.** Pi-mode runs *in* this same persistent console
  — node up, DB open, `Channels.Supervisor` up — not a fresh per-turn process. The
  mix-free console is the lifetime Pi-mode interactive sessions attach to.
- **Operator read surface in-session.** The read-only slash-command pattern
  (registered internal actions, slash-allowlisted, Runner-resolved, redacted) is
  the template a Pi-mode inspection affordance reuses without adding authority.
- **No boundary assumption blocked.** Every console read goes through
  `Actions.Runner.run/3` with no per-action confirmation or identity assumption
  that blocks v0.57 adding the named "local coding / sandbox level 0" trust tier on
  the same adapter and session.

### M1: Read-Only Operator Inspection Actions
Register the inspection actions behind `/status`, `/confirmations`, `/events`,
`/channels`, and `/settings get`, each resolved through `Actions.Runner.run/3`.
Use `exposure: :internal` and wire them only through the TUI slash-command
allowlist, so the intent router cannot select them (not intent candidates).
Define the shared read-report interface in this milestone:
`AllbertAssist.Operator.Inspection` (or narrower report modules if implementation
chooses a smaller naming scheme) returns redacted DTOs for warm session status,
channel status, recent `channel_events`, confirmation summaries, and setting
reads. Slash actions and their `mix allbert.*` twins render those DTOs rather than
duplicating read logic.
`/status` reports warm-session process/channel status; `/events` reads recent
`channel_events` rows (redacted/truncated); `/confirmations` reads pending/resolved
confirmations; `/channels` reports registered-channel status; `/settings get`
reads a settings key with secret fields redacted. The `/settings get` backing is a
**new `:internal` redacting read over `Operator.Inspection`**, **not** the existing
`read_setting` action — `read_setting` is `exposure: :agent` (model-routable, which
would violate Locked Decision #3) and has an agent-facing response shape rather
than an operator DTO with explicit sensitive-value/layer redaction. The redaction
guarantee depends on `Settings.resolve` returning redacted values for
secret-bearing keys plus the operator read redacting sensitive layers; the slash
read must not echo plaintext secrets.
`/status` must include enough
stable, redacted process identity for manual validation to prove a single warm
session served the run: BEAM OS PID (or equivalent VM identity), node name, uptime,
and `Channels.Supervisor` child health. `/status`'s "warm-session process/channel
status" has **no existing read action**, so it is served by a registered internal
read over the shared report interface (`AllbertAssist.Operator.Inspection`): the
action introspects the running node and supervisor and returns a redacted DTO. The
TUI must not call `Supervisor.which_children/1` or the Repo directly from
adapter/console code — that would fork a parallel read path the ADR 0016/0070
boundary forbids.
Pin the inspection-action DSL attributes: `permission: :read_only`,
`exposure: :internal`, `confirmation: :not_required` (a read-only execution mode).
They resolve through `Actions.Runner.run/3` with an `operator_context` carrying the
**resolved channel `user_id`** (from `channels.tui.identity_map`) — not a synthetic
privileged operator identity. Because every Runner action emits
`allbert.action.requested`/`allbert.action.completed` signals, each slash
invocation **is** auditable through those action signals, while creating **no**
`channel_events` row and **no** model turn (so `/events`, which reads
`channel_events`, never shows a slash invocation).
- **Lane:** serial — registration touches `Actions.Registry` (shared) and the read
  paths hit the store (db_serial for `/events`/`/confirmations`). Parallel: the
  five action modules are distinct (no shared barrier between them once registered)
  ∥ render formatting (pure).
- **Focused:** each action resolves against the warm node and returns redacted
  data from the shared report DTOs; **none** is selectable by the intent router or
  generated into intent descriptors; no action performs a write, sets the active
  app, or grants resource access.
- **Rejoin:** the five inspection actions are registered as internal, slash-
  allowlisted, redacted, and wired to their slash-commands.
- **M1 implementation note:** complete. `AllbertAssist.Operator.Inspection` is the
  shared read facade for status, channel inventory, recent `channel_events`,
  confirmation summaries, and redacted setting reads. The slash dispatcher maps
  `/status`, `/confirmations`, `/events`, `/channels`, and `/settings get <key>`
  to `operator_status`, `operator_confirmations`, `operator_events`,
  `operator_channels`, and `operator_setting_get` through
  `Actions.Runner.run/3`; malformed `/settings get` and unknown slash commands
  stay local-only. The TUI adapter resolves `channels.tui.identity_map` before
  Runner dispatch and carries the resolved `user_id`/`operator_id`; unmapped
  profiles get a local unavailable message with no runtime turn and no
  `channel_events` row. Registry and TUI focused tests assert the five actions are
  `permission: :read_only`, `exposure: :internal`,
  `confirmation: :not_required`, absent from `agent_modules/0`, and slash-only.

### M2: `mix allbert.channels status`

report of registered channel release/enabled/supervisor status over the **same

(`list`) subcommand — `status` is the shared channel data plus live
`Channels.Supervisor` child health — and honors Locked Decision #5 (single source

- **M2 implementation note:** complete. `mix allbert.channels status` runs the

- No change to `mix allbert.test` (CI gates) or to the v0.55 channel/parity matrix,
  Matrix outbound, split-payload seam, or rendering model — those are v0.55 scope.
- No v0.57 Pi-mode coding actions/trust tier; v0.55.1 only hardens the session
  foundation.
