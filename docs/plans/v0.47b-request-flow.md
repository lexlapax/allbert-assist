# Allbert v0.47b Request Flow - Operator-Supervised Self-Improvement (Handoff Drafts)

Status: implemented as `0.47.1`.

Implementation details live in `docs/plans/v0.47b-plan.md`. The discovery
substrate (trace index, generalized suggestion surface, unified draft store)
ships in v0.47 (`docs/plans/v0.47-plan.md`).

Shared substrate contract: v0.47b is a consumer of the v0.47 substrate. It may
add handoff suggestion/draft kinds and promotion targets, but it must reuse the
v0.47 suggestion lifecycle, unified reviewed-draft facade, settings namespace,
and authority invariants.

## M1 Locked Decisions

Each is recorded in `v0.47b-plan.md` §"M1 Locked Decisions".

1. Draft home — the v0.47 unified draft store; v0.47b adds kinds and
   promotion targets only.
2. Code-bearing gate — the existing v0.36/v0.37 path, unchanged.
3. Marketplace + delegate inertness — descriptive metadata and inert requests
   only.
4. Objective drafts — declarative inputs; framing/running stays a confirmed
   v0.24 action.

## Main User/Operator Flows

- Operator reviews a self-improvement suggestion with a handoff kind:
  template-backed, marketplace-backed, delegate-plugin request, capability-gap,
  or objective draft. These kinds reuse the v0.47 suggestion lifecycle; in
  `0.47.1` they are supported reviewed rows, not automatically inferred by
  `discover_patterns`.
- Promotion creates an inert draft in the unified store.
- Code-bearing drafts route through the v0.36 sandbox + v0.37 gate/loader.
  Template-backed drafts may route through v0.38 templated creation to create an
  inert dynamic draft; marketplace-backed drafts remain descriptive metadata;
  objective drafts frame a v0.24 objective only under operator confirmation.

## Data Flow

1. A v0.47b handoff suggestion using the v0.47 suggestion lifecycle
   (`template_backed`, `marketplace_backed`, `delegate_plugin_request`,
   `capability_gap`, `objective`) is promoted to an inert draft in the unified
   `Drafts.Store`. `discover_patterns` does not synthesize these handoff kinds
   from memory-review or objective-event diagnostics in `0.47.1`.
2. Suggestion lifecycle stays the v0.47 lifecycle: pending suggestions may
   expire, discard marks them dismissed, promotion accepts them and records
   `draft_id`, and re-promotion returns the same draft idempotently.
3. Handoff routes by kind, each through a normal registered action, Security
   Central, trace, and audit. Inert draft/scaffold handoffs do not themselves
   grant live authority; durable confirmation remains on the existing live
   transition:
   - `template_backed` → M2 records a `Templates.Registry.resolve/2` /
     `Templates.preview/3` packet in the unified draft store; M2
     `promote_template_draft` calls `create_from_template`
     (`permission: :dynamic_codegen_request`, `execution_mode:
     :template_dynamic_draft`, `confirmation: :not_required`) and creates an
     inert dynamic draft with `gate_status: "not_run"`;
   - `marketplace_backed` → M2 reads `Marketplace.list_entries/1` metadata and
     records it as descriptive draft data only (`install_requested: false`,
     `authority: "metadata_only"`); later handoff/publish decisions remain
     separate;
   - `delegate_plugin_request` → M3 stores a v0.38 plugin-template preview plus
     delegate metadata as an inert draft (`scaffold_requested: false`,
     `agent_registered: false`); no plugin directory is written and no objective
     agent is registered;
   - `capability_gap` → M1 records an inert `Codegen.CapabilityGap` summary
     with `dynamic_draft_requested: false`; M4
     `promote_capability_gap_draft` routes the reviewed draft through
     `DynamicPlugins.request_draft/2` and records the resulting v0.37 dynamic
     draft slug. `integrate_dynamic_draft` does not create a confirmation until
     that dynamic draft is `gate_passed`; confirmed integration still runs
     through `Loader.integrate/2`;
   - `objective` → M1 stores declarative objective input with
     `objective_framed: false`; M3 frames a v0.24 objective on confirmation
     (`permission: :objective_write`, `execution_mode:
     :objective_draft_promotion`, `confirmation: :required`).
4. Code-bearing integration occurs only after gate evidence plus operator
   confirmation; `Loader.rollback/3` removes live authority.
5. Trace/audit carry the actor, the draft id, the gate report reference, and
   the integration/rollback decisions.

## Authority Invariants

- Marketplace metadata is descriptive only; it grants nothing.
- Delegate-plugin drafts are inert; no agent is registered by a suggestion or
  a draft.
- Code-bearing drafts reach live authority only through the v0.36 sandbox +
  v0.37 trusted-validation/gate/loader path plus confirmation.
- Objective drafts are declarative; framing/running stays a confirmed v0.24
  action.
- No new permission class, operation class, URI scheme, or trust tier; v0.47b
  stays inside the v0.47 self-improvement tier (ADR 0045).
- Repeated use never grants permission or auto-approval.

## Handles / CLI Surfaces

- New `suggestion_type`/draft kinds: `template_backed`, `marketplace_backed`,
  `delegate_plugin_request`, `capability_gap`, `objective`.
- Registered actions:
  - `promote_template_draft` — `permission: :dynamic_codegen_request`,
    `execution_mode: :template_dynamic_draft`, `confirmation: :not_required`
    for inert dynamic-draft creation.
  - `promote_capability_gap_draft` — `permission: :dynamic_codegen_request`,
    `execution_mode: :dynamic_codegen`, `confirmation: :not_required`.
  - `promote_objective_draft` — `permission: :objective_write`,
    `execution_mode: :objective_draft_promotion`, `confirmation: :required`.
  Marketplace-backed and delegate-plugin request drafts ship without separate
  v0.47b promotion actions in `0.47.1`; marketplace install/publish and plugin
  scaffolding remain separate existing or future reviewed action paths.
  Delegate-plugin drafts reuse the v0.38 plugin template + v0.46 registration
  contract.
- Reused entry points: `Templates.render/3`, `create_from_template`,
  `Marketplace.list_entries/1`, `DynamicPlugins.request_draft/2`,
  `Sandbox.run_gate/2`, `Loader.integrate/2`/`rollback/3`.
- CLI: `mix allbert.self_improvement` `--kind` filtering for the new kinds.

## Smoke Checklist

- Promoting a template-backed draft produces an inert scaffold; nothing is
  enabled.
- Marketplace metadata appears in a suggestion without granting authority.
- A delegate-plugin draft registers no agent.
- A code-bearing draft requires the sandbox/gate path before any live-
  integration confirmation appears; integration occurs only on confirmation;
  rollback removes live authority.
- An objective draft frames an objective only on confirmation.
- `mix allbert.test release.v047b` runs green on a fresh checkout and writes
  reproducible evidence.
