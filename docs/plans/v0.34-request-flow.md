# Allbert v0.34 Request Flow - Workspace UX Refresh

Status: planned.

Implementation details live in `docs/plans/v0.34-plan.md`.

This document describes operator-visible behavior. v0.34 is a layout and
navigation refresh of the v0.32 `/workspace` shell; it changes no domain,
security, or routing authority.

## Two Axes (the core mental model)

v0.34 separates two things the v0.32 shell had fused:

- **What you VIEW** — chosen on the **left launcher**, rendered in **Canvas**.
  Pure navigation; no routing effect.
- **What routing USES** (`active_app`) — set **only** by accepting a v0.33
  conversational handoff, shown by the passive **top-bar context indicator**.

You can view StockSage's dashboard while routing context is still Neutral.
`Neutral` is the UI label for the Allbert/default routing context: the session
may store no app after `Session.clear_active_app`, while runtime turns may
normalize that to `:allbert`.

## Layout States

| State | Wide screen | Narrow screen |
|---|---|---|
| Default | rail · chat (wide) · Canvas = Output | chat tab; ☰ launcher sheet |
| Destination open | rail · chat (narrowed, push) · Canvas = destination | Canvas tab full-screen |
| Focus | rail · chat collapsed to strip · Canvas full | Canvas tab full-screen |

## Flow 1: Open the workspace

Operator opens `/workspace`.

Expected:
1. Chat is the primary, wide column; the left launcher is on the left.
2. Canvas shows the **Output** destination (the thread's durable v0.30 tiles),
   or is collapsed if empty, with chat full width.
3. No permanent Tools column and no floating app/objectives band appear.
4. The top-bar context indicator reads **Neutral**.
5. The URL does not need `app_id` to represent routing context.

## Flow 2: View an app or tool (launcher = pure navigation)

Operator clicks **StockSage** (or **Settings**, **Jobs**, etc.) in the launcher.

Expected:
1. Canvas **replaces** its content with that single destination (StockSage
   dashboard / Settings panel / Jobs panel …). Chat narrows (push); both visible.
2. `active_app` does **not** change; the context indicator still reads
   **Neutral**.
3. No action executes and no confirmation is created by viewing.
4. Selecting **Output** returns Canvas to the durable tiles.

Failure behavior:
- A launcher selection can never set routing context or run an action.
- Visiting `/workspace?app_id=stocksage` or `/workspace?active_app=stocksage`
  does not persist StockSage context. The implementation may ignore or remove
  the stale query params, but the indicator remains **Neutral** until handoff
  accept.

## Flow 3: Reload or deep-link a destination

Operator reloads `/workspace` or opens a view-only destination link.

Expected:
1. `/workspace` opens with Output selected by default.
2. If v0.34 adds a `destination` query param, it is view-only: valid
   destinations select Canvas content, invalid destinations fall back to
   Output with bounded diagnostics.
3. `destination=app:stocksage` can show StockSage in Canvas while the context
   indicator remains **Neutral**.
4. No URL param is allowed to set `active_app`; only accepting a handoff can do
   that.

## Flow 4: Switch threads

Operator selects a different thread from the launcher or thread switcher.

Expected:
1. Chat changes to the selected thread.
2. Canvas destination view state is preserved when valid.
3. If Output is selected, Output shows that thread's durable tiles.
4. Thread switching never changes `active_app`, runs an action, or creates a
   confirmation.

## Flow 5: Enter app context conversationally (handoff sets context)

In Neutral context the operator types `analyze CIEN`.

Expected:
1. The v0.33 engine produces an `:app_handoff` proposal as an ephemeral surface
   (see ADR 0034). If Canvas already shows StockSage, the prompt is the lighter
   inline form ("Run StockSage analysis for CIEN?"); otherwise the cross-app
   form ("StockSage can run a financial analysis for CIEN. Run it?").
2. **Accept** → `set_active_app(:stocksage)`, the prompt re-submits with the
   extracted `ticker` slot, `Actions.Runner.run/3` enforces app scope, Security
   Central applies `:stocksage_analyze`, and the normal `confirmation: :required`
   flow appears. The context indicator updates to **StockSage**.
3. Only after confirmation approval does analysis run; results land as Output
   tiles in Canvas.
4. **Decline** → context unchanged, no confirmation, no action.

## Flow 6: Leave app context

Operator clicks the exit affordance (✕) on the context indicator.

Expected:
1. `Session.clear_active_app` runs; the indicator returns to **Neutral**.
2. Canvas and chat are untouched (you can still be viewing StockSage's board).
3. Subsequent app intents go back through the handoff path.
4. A refresh remains Neutral and does not rehydrate app context from stale
   `app_id` params.

## Flow 7: Settings and workspace tools inside Canvas

Operator opens **Settings** (or Confirmations, Security, Jobs) from the launcher.

Expected:
1. The panel renders in Canvas (no separate column, no separate route).
2. Settings/grants/confirmation writes still flow through the existing
   registered actions and Security Central; provider keys stay redacted.
3. A UI event that tries to mutate settings outside the registered action path
   has no effect.

## Flow 8: Focus mode

Operator toggles canvas-focus while a destination is open.

Expected:
1. Chat collapses to a thin strip; Canvas takes the full width for the
   destination (e.g., a full StockSage dashboard view).
2. Toggling back restores the split.

## Flow 9: Narrow viewport

Operator uses a narrow window / mobile.

Expected:
1. The launcher is a ☰ sheet; the body shows `[Chat | Canvas]` bottom tabs.
2. Canvas shows one destination full-screen; selecting a launcher entry jumps
   to the Canvas tab.
3. No horizontal overflow; tile/menu controls remain reachable.

## Data Flow: Input → Output

Two input surfaces, two axes:

1. **Launcher click (view axis).** `WorkspaceLive` `select_destination` sets the
   `canvas_destination` assign → Canvas renders that destination's existing
   panel(s). No runtime turn, no `set_active_app`, no action. Output is the
   default destination and renders v0.30 durable tiles. Legacy
   `select_workspace_app` / `?app_id=` paths are retired or rewritten so they
   cannot set routing context.
2. **Chat composer (routing axis).** `handle_event("ask", ...)` →
   `Runtime.submit_user_input/1` with the current `active_app`. In Neutral, an
   app intent yields a v0.33 `:app_handoff`/`:clarify_intent` ephemeral; the
   operator accepts → `set_active_app` → re-submit with slots →
   `Actions.Runner.run/3` → Security Central → confirmation → execution →
   durable tiles appear in the Output destination.
3. **Context indicator (read + exit).** Reflects session `active_app`; the ✕
   calls `Session.clear_active_app`. It never sets context.
4. **Optional destination URL (view axis only).** If implemented, a
   `destination` query param may restore Canvas content; it never sets
   `active_app`.

Authority invariants: viewing ≠ acting; the launcher never sets context; routing
context is handoff-only; app-scoped execution still requires matching
`active_app` at the runner; confirmation floors are never lowered (ADR 0019,
ADR 0021, ADR 0024 revision, ADR 0034, v0.28).

## Smoke Checklist

- `/workspace` opens chat-primary; no Tools column; no floating band.
- Launcher selection changes only Canvas; context indicator stays Neutral.
- Stale `app_id` / `active_app` query params do not set routing context.
- Canvas shows exactly one destination; Output is the default.
- Thread switch changes chat and Output tiles but not routing context.
- `analyze CIEN` in Neutral produces a handoff; accept flips context to
  StockSage and reaches the normal confirmation; decline changes nothing.
- Exit affordance returns context to Neutral.
- Settings/tools render in Canvas and still write through registered actions.
- Canvas-focus collapses chat to a strip and back.
- Narrow viewport: sheet + `[Chat | Canvas]` tabs; one destination full-screen;
  no overflow.
- No duplicate launcher entries (one home per concept).
- Security evals: launcher cannot set context/execute; Canvas cannot bypass
  app-scope; settings writes stay action-gated.

## Milestone Update Log

- M0 implementation preflight (2026-05-23):
  - Use `canvas_destination` as view-only LiveView state with optional
    `destination` URL persistence; do not carry `app_id` in workspace paths.
  - Ignore stale `app_id` / `active_app` query params for routing context.
    Session context is read from `Session.get/3`; handoff accept remains the
    only setter.
  - Treat `select_workspace_app` as retired compatibility surface. The launcher
    sends `select_destination` with `output`, `app:<app_id>`, or
    `workspace:<tool>`.
  - Keep existing structural/catalog atoms for compatibility, but remove
    rendered `:utility_drawer` and `:context_rail` regions from the v0.34 shell.
  - Map workspace tools explicitly: Jobs, Objectives, Confirmations, Security,
    and Settings each render their existing CoreApp panel in Canvas.
  - Verify UI changes with Chrome extension control on desktop and narrow
    viewport before release handoff.
- M1 shell cleanup:
  - CoreApp no longer renders the retired `:utility_drawer` column or
    `:context_rail`/badge-band region in the v0.34 workspace shell.
  - The default narrow body tab is Chat, with Canvas as the only peer body tab;
    Nav/Tools/Ephemeral are no longer permanent mobile body tabs.
  - Canvas focus uses the stable `#workspace-canvas-focus` control and a
    shell-level `data-canvas-focus` state.
- M2 view-only launcher:
  - `WorkspaceLive` owns `canvas_destination` and persists it through the
    optional `destination` query param; workspace paths no longer carry
    `app_id`.
  - Launcher clicks send `select_destination` (`output`, `app:<app_id>`, or
    `workspace:<tool>`) and switch the mobile body to Canvas without calling
    `set_active_app`.
  - Stale `app_id` / `active_app` params are ignored for routing; runtime
    prompts remain Neutral/Allbert until handoff accept sets context.
- M3 single-destination Canvas:
  - `Workspace.Catalog` now uses `canvas_destination` for panel visibility;
    `active_app` no longer decides what Canvas renders.
  - `output` renders durable tiles only. `app:<app_id>` renders that app's
    validated panel group. `workspace:<tool>` renders exactly the mapped
    CoreApp tool panel.
  - Settings Central opens fully inside the `workspace:settings` destination
    and keeps writes behind registered actions.
