# Allbert v0.32 Request Flow - Workspace-Only App UI And Settings Central

Status: implementation in progress.

This document describes operator-visible behavior for v0.32. Implementation
details live in `docs/plans/v0.32-plan.md`.

v0.32 fixes the URL-hack part of app context by making app selection explicit
inside `/workspace`. It does not infer app intent from neutral natural-language
prompts; conversational handoff, ambiguity clarification, and the direct-answer
rewrite belong to v0.33.

## Pre-Flight

Use a disposable Allbert Home and open `/workspace`:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v032-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix ecto.migrate
ALLBERT_HOME="$SMOKE_HOME" mix phx.server
```

v0.32 assumes the v0.31 runtime/UI-substrate consolidation, v0.26 workspace
shell, v0.27 StockSage renderers, v0.28 security evals, v0.29 memory/outcomes
contract, and v0.30 canvas emission exist.

## Flow 1: Workspace Is The Only Operator Home

Operator opens `/workspace`.

Expected behavior:

1. The workspace shell renders a left rail, center thread, right canvas, and
   utility drawer affordance.
2. The left rail includes new-thread, recent threads, and app launcher entries.
3. The right canvas preserves durable tiles and app panels.
4. The mobile layout provides reachable nav/chat/canvas/utility states with no
   horizontal overflow.

Failure behavior:

- `/agent`, `/settings`, and `/stocksage/*` are not valid operator routes.
  They do not redirect and do not render compatibility pages.

## Flow 2: Settings Central Opens Inside Workspace

Operator opens the utility drawer in `/workspace`.

Expected behavior:

1. Settings Central renders in the drawer/panel.
2. Settings list/search/update uses the same Settings Central actions as
   before.
3. Provider keys remain redacted and encrypted at rest.
4. Permission settings, confirmations, and remembered grants remain behind
   existing action/security boundaries.

Failure behavior:

- A UI event that tries to mutate settings directly, without the existing
  registered action/context path, is rejected or has no effect.

## Flow 3: Apps Contribute Panels

Operator selects an app in the launcher.

Expected behavior:

1. App selection sets active app context through the existing
   registered/session boundary.
2. App panels appear in host-owned zones.
3. Panels are catalog-validated Surface nodes.
4. Unknown zones, unknown components, raw HTML, and unauthorized action
   bindings are rejected with bounded diagnostics.
5. App metadata influences visibility/placement only; it never grants
   authority.

Failure behavior:

- A rejected panel does not crash the workspace. Other panels and the chat
  thread continue rendering.
- Selecting an app does not execute an action and does not bypass confirmation
  or Security Central.

## Flow 4: CoreApp Uses The Same Panel Path

Operator opens built-in work surfaces such as jobs, objectives, confirmations,
or security/status cards from the workspace.

Expected behavior:

1. Built-in domain cards render through panel composition.
2. There is no private CoreApp-only placement path for those cards.
3. Actions, confirmations, security decisions, and traces remain unchanged.

## Flow 5: StockSage Uses Workspace Panels

Operator selects StockSage in the app launcher.

Expected behavior:

1. Active app context becomes StockSage without manually editing
   `?app_id=stocksage`.
2. StockSage dashboard, recent analyses, queue, and trends render as workspace
   panels.
3. StockSage private nav chrome is gone.
4. Durable StockSage canvas tiles still render with v0.27 cards through the
   v0.30 Fragment/canvas path.
5. Typing `analyze CIEN` while StockSage is selected routes to the existing
   StockSage `run_analysis` confirmation path.
6. Analysis detail either opens as a workspace panel/modal or, only if the
   implementation documents the need, at `/apps/stocksage/analyses/:id`.

Failure behavior:

- `/stocksage`, `/stocksage/analyses`, `/stocksage/queue`, and
  `/stocksage/trends` are absent.
- Typing `analyze CIEN` before selecting StockSage does not silently execute or
  infer StockSage in v0.32; v0.33 owns neutral app-intent handoff.

Implementation decision:

- Analysis detail remains a rare page-shaped route at
  `/apps/stocksage/analyses/:id` because it carries long-form surface-node
  review, objective/progress streaming, rerun controls, reflection generation,
  and explicit lesson-sync confirmation. StockSage dashboard, recent analyses,
  queue, and trends move into `/workspace` panels.

## Flow 6: Route Removal Is Intentional

Operator or test client requests old operator routes.

Expected behavior:

1. `/agent` is absent.
2. `/settings` is absent.
3. `/stocksage/*` is absent.
4. No compatibility redirect is returned.

## Flow 7: Injection Is Rejected

A fixture app attempts to contribute invalid UI.

Expected behavior:

1. Unknown component ids are rejected.
2. Unknown zones are rejected.
3. Raw HTML/JS/CSS payloads are rejected.
4. Foreign-app action bindings are rejected.
5. Secret-like props are redacted or rejected per existing security posture.
6. Rejections are visible in trace/workspace diagnostics.

## DOM Ids And Events

Stable handles for tests and Chrome manual verification:

| Surface | Handle | Expected use |
|---|---|---|
| Workspace shell | `#workspace-shell` | Canonical `/workspace` root |
| Nav rail | `#workspace-nav-rail` | New-thread, recent threads, app launcher |
| Thread list | `#workspace-thread-list` | Recent conversation switching |
| App launcher | `#workspace-app-launcher` | Registered app entries |
| App entry | `[data-app-id="<app-id>"]` | Explicit app selection |
| Zone | `#workspace-zone-<zone>` | Host-owned zone wrapper |
| Panel | `#workspace-panel-<surface-id>` | Catalog-validated panel surface |
| Utility drawer | `#workspace-utility-drawer` | Settings Central and tools |
| Settings panel | `#workspace-settings-panel` | Settings Central inside workspace |
| Mobile nav toggle | `[data-mobile-tab="nav"]` | Narrow viewport nav state |
| Mobile utility toggle | `[data-mobile-tab="utility"]` | Narrow viewport settings/tools state |

## Smoke Checklist

- `/workspace` renders the shell and is usable at desktop and narrow widths.
- `/agent`, `/settings`, and `/stocksage/*` are absent with no redirects.
- Settings Central opens and performs existing safe settings/provider/grant
  workflows inside the utility drawer.
- App launcher switches between Allbert/CoreApp and StockSage.
- Selecting StockSage replaces manual `?app_id=stocksage` URL editing for
  in-context analysis requests.
- Neutral `analyze CIEN` before app selection remains conservative; v0.33 owns
  app-intent handoff.
- CoreApp cards render through panels.
- StockSage dashboard/recent/queue/trends render as panels.
- Durable StockSage tiles still render in the canvas.
- Invalid panels/zones/components are rejected with diagnostics.
- Chrome verification records desktop and narrow viewport screenshots/notes.
