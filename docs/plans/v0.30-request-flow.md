## Pre-Flight

Use a disposable Allbert Home and open `/agent`:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v030-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix ecto.migrate
ALLBERT_HOME="$SMOKE_HOME" mix phx.server
```

v0.30 assumes v0.27 real StockSage renderers exist and v0.28 has audited the
workspace Fragment/canvas path.

Component note: v0.30 does not add `:stock_chart`. Durable canvas tiles use the
four v0.26-reserved and v0.27-proven StockSage atoms:

- `:analysis_card`
- `:agent_report_card`
- `:parity_card`
- `:debate_round_card`

## Flow 1: Emit Durable Canvas Tiles From Analysis

Operator opens `/agent?app_id=stocksage` and asks to analyze a ticker.

Expected behavior:

1. StockSage runs through the existing `RunAnalysis` and confirmation flow.
2. `RunAnalysis` emits the existing `allbert.stocksage.analysis_*` lifecycle
   signals with `user_id`, `thread_id`, analysis id, engine, and bounded
   summary/error metadata.
3. `AllbertAssist.Workspace.Emitters.stocksage_signal/2` turns those lifecycle
   signals into signed `%AllbertAssist.Workspace.Fragment.Envelope{}` structs
   with `scope: :canvas` and StockSage card kinds.
4. `AllbertAssist.Workspace.Fragment.emit/1` validates signature, emitter
   allow-list, catalog, app catalog, rate limit, and payload size, then
   persists the tile through `workspace_canvas_tiles` plus YAML body storage.
5. Tiles render in `/agent` through the v0.27 StockSage card components, not
   the v0.26 stub renderers.
6. The chat/analysis response remains the authoritative analysis result; tile
   emission is a durable workspace side effect.

Failure behavior:

- If tile creation fails validation, the analysis result remains persisted and
  the operator sees a bounded canvas-emission failure.
- No analysis is rerun merely because tile emission failed.
- Missing `thread_id`/`user_id` means no durable canvas tile is emitted; the
  analysis result still returns normally.

## Flow 2: Reload And Resume

Operator refreshes `/agent` or restarts the app.

Expected behavior:

1. Persistent StockSage tiles reappear for the same user/thread.
2. Tile provenance points back to analysis id, objective id, step id, engine,
   emitter id, and emitted timestamp where present.
3. Deleted or evicted tiles follow existing soft-delete/restore behavior.
4. No tiles leak across users or threads.

## Flow 3: Dashboard Uses Persistent Tiles

Operator visits StockSage workspace/dashboard after recent analyses exist, then
opens `/agent` for the same thread.

Expected behavior:

1. Recent analysis summaries are rendered in the StockSage app surface through
   `/stocksage/*` regardless of `/agent` canvas state.
2. Dashboard links and `/agent` tiles point to the same underlying analysis
   records.
3. The StockSage app surface remains usable without opening `/agent`.
4. Pin/remove/inspect affordances on `/agent` tiles use the existing workspace
   tile controls; StockSage does not add private canvas-management controls.

## Flow 4: Invalid Canvas Output Is Rejected

Fixture attempts to emit unknown component ids, cross-app component ids, raw
HTML, secret-like props, or unauthorized action bindings.

Expected behavior:

1. Catalog validation rejects the tile.
2. The rejection is visible in trace/workspace diagnostics.
3. No arbitrary model-generated UI renders.

## DOM Ids And Events

Stable UI handles used by automated tests and Chrome manual verification:

| Surface | Handle | Expected use |
|---|---|---|
| `/agent` canvas tile | `#workspace-node-<tile-id>` | Tile wrapper rendered by workspace renderer |
| StockSage analysis card | `[data-stocksage-component="analysis_card"]` | Real v0.27 card renderer in a durable tile |
| StockSage agent report card | `[data-stocksage-component="agent_report_card"]` | Native specialist summary tile |
| Tile inspector | `#workspace-tile-inspect-<tile-id>` | Existing v0.26c tile inspector action |
| Tile actions menu | `#workspace-tile-menu-button-<tile-id>` | Existing workspace tile menu |

No test should assert a `:stock_chart` component in v0.30.

## Milestone User Testing

### M1 Renderer Registration

1. Seed or create an `analysis_card` canvas tile.
2. Open `/agent?thread_id=<thread-id>`.
3. Confirm the card uses `data-stocksage-component="analysis_card"`.
4. Confirm no visible `v0.26 stub` marker appears.

### M2-M3 Emission And Reload

1. Run an approved StockSage native analysis with `thread_id`.
2. Confirm `workspace_canvas_tiles` contains StockSage tile ids under the
   `stocksage_analysis_*` prefix.
3. Refresh `/agent?thread_id=<thread-id>`.
4. Confirm the StockSage tile still renders and points to the same analysis id.

### M4 Browser Verification

Use Chrome extension browser control:

1. Start Phoenix with a disposable `ALLBERT_HOME`.
2. Open `/agent?thread_id=<thread-id>` where a StockSage analysis tile exists.
3. Capture desktop and narrow viewport screenshots.
4. Confirm text does not overflow controls and tile actions remain reachable.
