## Purpose

Turn first-run from a prototype-grade scatter of steps into a genuine guided
onboarding capability. The current onboarding objective and web panel mostly
sequence existing Mix commands; a non-developer user still needs docs and
checkout-oriented commands to reach first value. v0.62 closes that gap with a
shared two-track wizard in web and terminal, provider/model setup with masked
credential entry and inline doctor verification, and repo-maintained
user-category profiles that seed Settings Central defaults with explicit
operator review.

## Operator-Visible Win

A brand-new operator launches Allbert, the web wizard auto-opens on first run
after `allbert serve`, they pick QuickStart, enter a masked provider key that is
stored through the OS vault and verified inline by doctor, review and apply a
user-category profile, and reach a working first chat without hand-editing config
or reading docs. The same flow is available through an interactive terminal
wizard for headless installs.

## Guided Wizard Flow (ADR 0069)

The wizard is a single shared flow model with two render surfaces:

- **One flow, two surfaces.** Web and terminal render one state machine; neither
  forks onboarding logic.
- **Two tracks.** QuickStart minimizes decisions and skips channel/integration
  setup to reach a working conversation quickly; Advanced exposes provider,
  model, channel, app/profile, and diagnostics choices.
- **Steps:** launch/source check -> provider/model setup -> user-category profile
  selection/review -> optional channel/integration setup -> first useful chat ->
  health/trust-spine confirmation.
- **Trust spine as a feature.** The wizard explains confirmation, permission,
  traces, and local inspectability as safety properties, not setup friction.

## User-Category Profile System (ADR 0075)

- **Repo-maintained declarative presets** live in the source tree and are
  reviewed like other trusted project metadata.
- **Shipped categories:** `researcher`, `developer`, `writer`, `ops`,
  `general`.
- **Each profile specifies:** Settings Central defaults, suggested apps,
  suggested channels, suggested intents, and recommended model-purpose mappings.
- **Seed-only semantics.** Profiles grant no authority, lower no confirmation
  floor, and enable nothing effectful without the normal gates.
- **Explicit application.** Applying a profile always shows exactly what will be
  seeded before confirmation.

## Security Posture

No new authority. Onboarding is an experience over existing flows: every
effectful step routes through existing actions, settings, doctors, gates, and
confirmations. Profiles seed defaults only. Masked credentials write through the
v0.61 OS-vault reference model and never echo/log raw values.

## Non-Goals

- No new capability surface or permission class.
- No runtime authorization, trust tier, or per-user authority model from
  profiles.
- No learned, distilled, or auto-generated personas.
- No native desktop GUI onboarding.
- No packaging or CLI redesign; v0.61 owns entry points.
- No final product RC; v0.63 owns integrated validation.

## Definition Of Done

v0.62 is done when:

- a new operator can complete first-run onboarding through the web wizard and the
  terminal wizard over one shared flow;
- QuickStart reaches a working first chat without docs or config edits;
- provider/model setup stores masked credentials through the OS vault, verifies
  inline with doctor, supports local and hosted providers, and switches providers
  without config edits;
- repo-maintained profiles seed Settings Central defaults and suggestions only
  after explicit review/confirm;
- the trust spine is surfaced as a feature;
- ADR 0069 and ADR 0075 are accepted;
- `mix allbert.test release.v062`, docs, and warning gates pass with no
  onboarding secret leak or profile-authority regression;
- no known warning/security/doc drift remains.
