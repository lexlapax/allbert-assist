## Purpose

Make Allbert installable and invokable as a product, not a checkout. Ship a
release-built `allbert` binary with no Elixir/OTP toolchain required on the user's
machine, a Homebrew + curl install path, a unified grouped CLI dispatcher that
subsumes the flat mix-task sprawl and separates operator from dev/CI commands,
background-daemon management with a visible health check, completion of the
ADR 0070 mix-free TUI console convergence, and OS secret-vault credential handling —
all with no authority change, then settle the entry-point and Home-layout surface so
the v1.0 freeze can lock it.

## Operator-Visible Win

A technical prosumer installs Allbert from a packaged binary — `brew install allbert`
or a one-line curl — with no Elixir/OTP on their machine, runs `allbert serve` to
start the runtime + web workspace + channels as a managed background daemon, watches
a health check go green, drives day-to-day operation through `allbert <group>
<command>` and the mix-free TUI console without ever touching `mix`, and stores
provider credentials in the OS keychain instead of a plaintext settings file.

## Packaged Binary & ERTS Bundling (M1)

Build an OTP release and wrap it as a self-contained `allbert` binary via
Burrito/Bakeware-style ERTS bundling, so the artifact carries its own runtime and
no Elixir/OTP is required on the user's machine. The release config (`rel/`,
`mix.exs` `releases:`) is added here for the first time. The binary boots the same
supervision tree the checkout boots — packaging is a delivery change, not a runtime
change; every entry point still resolves to the ADR 0073 spine.

## Unified CLI Dispatcher & Operator/Dev Split (M3)

A single `allbert <group> <command>` surface subsumes the flat `mix allbert.*`
sprawl behind coherent grouped help. Groups: `ask` and `chat` (conversation),
`tui` (operator console), `serve` (daemon/runtime), `admin <area>` (operator
inspection/management), and `gen` (scaffolding). Operator commands move onto the
binary; developer and CI commands (build, eval gates, scaffolds used only in a
checkout) stay `mix`-only and are not exposed on the binary. The dispatcher
**reorganizes entry points; it adds no capability and no authority** — each command
routes through the same runtime/action/settings spine (ADR 0073) and is subject to
the same Security Central posture and confirmations.

## Daemon Management & `allbert serve` (M4)

`allbert serve` starts the runtime, web workspace, and channels as a managed process
and installs/uninstalls an OS service — launchd on macOS, systemd on Linux, a
Scheduled Task on Windows/WSL2. A health check the operator can watch succeed covers
the three legs (runtime up, web reachable, channels supervised). Service install is
an explicit operator action; the daemon grants no authority beyond what the same
surfaces already hold when launched from a checkout.

## ADR 0070 TUI Console Convergence Completion (M5)

v0.55.1 and v0.58 brought the operator console most of the way: status,
confirmations, events, channels, settings, intents, and model reads are registered
read-only internal actions reachable through the slash-command allowlist. v0.62
absorbs the **remaining admin-inspection reads** so an operator never needs raw `mix`
for day-to-day operation. The remaining reads follow the established ADR 0070
pattern exactly: `exposure: :internal`, routed through `Actions.Runner.run/3`,
slash-allowlisted, excluded from the intent-router candidate set (ADR 0060), backed
by the shared read-report interface — no mutation, no authority, redacted/truncated
output, no new `:operator` exposure. The mix-free console becomes the standing
operator surface the packaged binary ships.

## OS Secret-Vault Credential Handling (M6)

The v0.61 onboarding work (ADR 0069) made the credential UX **vault-ready**:
credentials are referenced through a swap-ready indirection rather than read inline.
v0.62 swaps an OS-vault backend in under that UX — macOS Keychain, the Linux Secret
Service, Windows Credential Manager — so secrets move out of plaintext settings into
the OS keychain, injected at launch and never logged. Settings Central remains the
config authority (ADR 0004 / ADR 0031); the vault holds secret *values*, settings
holds the *references*. Migration of existing plaintext credentials into the vault is
explicit and redacted; no raw secret is ever logged or surfaced in inspection output.

## Allbert Home Layout

Allbert Home layout may shift here to accommodate packaged-install paths (binary,
service unit/plist, install-managed directories) and the OS-vault reference model.
Because this is the **last entry-point/Home-layout churn before the freeze**, the
layout is settled now so the v1.0 freeze can lock it (ADR 0006 / Home-layout
decisions). Any change keeps a documented migration from a v0.61 checkout-style Home.

## Security Posture

**No new authority.** Packaging changes how Allbert is installed and invoked, not
what any surface may do. The binary, the CLI dispatcher, the daemon, the TUI console,
and the vault all route through the existing runtime/action/settings spine (ADR 0073)
and remain subject to Security Central, durable confirmations, the action boundary,
and surface policy (ADR 0006). The CLI dispatcher exposes no command the equivalent
`mix` task did not already expose; it only regroups and renames. The TUI convergence
adds read-only internal actions only — no mutation, no `:operator` exposure, no
intent-router reachability (ADR 0060 / ADR 0070). The OS vault never logs or renders
raw secret values; vault swap and credential migration are explicit and redacted.

## Non-Goals

- Not a native desktop GUI client; the web workspace remains the operator UI through
  v1.0 (the binary serves it). A native client stays post-1.0.
- No hosted/remote distribution and no plugin auto-update.
- No new capability, action, or surface authority.
- No change to confirmation semantics, the action boundary, or Settings Central
  ownership.
- No removal of `mix` tasks for development — dev/CI commands remain `mix`-only.
- No public-protocol surface changes (MCP server, OpenAI-compatible API, ACP).
- No contract freeze in v0.62; the freeze is v1.0.

## Definition Of Done

v0.62 is done when:

- a packaged `allbert` binary installs via Homebrew and curl and runs on a machine
  with no Elixir/OTP toolchain;
- `allbert <group> <command>` exposes the grouped operator surface with the
  operator/dev split, and `mix` tasks remain available for development;
- `allbert serve` runs the runtime + web + channels as a managed launchd/systemd/
  Scheduled-Task daemon with a health check the operator can watch succeed;
- the ADR 0070 convergence is complete — the mix-free TUI console covers day-to-day
  operation with the remaining admin reads as registered read-only internal actions,
  no mutation and no new `:operator` exposure;
- provider credentials are stored in the OS vault under the v0.61 credential UX, with
  explicit redacted migration and no raw secret logged or rendered;
- ADR 0076 is accepted and ADR 0070 is marked converged;
- `:v062` eval rows pass and `mix allbert.test release.v062` is green with evidence
  showing no authority change, spine-only invocation through the binary, secret-vault
  no-leak, and install/daemon health;
- the binary and CLI dispatcher are documented as Tier-1 freeze candidates and the
  Allbert Home layout is settled for the v1.0 freeze;
- no known warning/security/doc drift remains.
