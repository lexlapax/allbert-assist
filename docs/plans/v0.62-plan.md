## Purpose

Make Allbert installable and invokable as a product, not a checkout. Ship a
release-built `allbert` binary with no Elixir/OTP toolchain required on the
user's machine, a Homebrew + curl install path, a unified grouped CLI dispatcher
that separates operator commands from developer/CI commands, background-daemon
management with a visible health check, completion of the ADR 0070 mix-free TUI
console convergence, and OS secret-vault credential handling. This release
**implements** the entry-point / grouped-CLI UX **designed in v0.60 (ADR 0077,
v0.60 M5)** — the command taxonomy is a v0.60 design input, not invented or
"settled" blind here. v0.63 onboarding then teaches the implemented entry points.

## Operator-Visible Win

A technical prosumer installs Allbert from a packaged binary with no Elixir/OTP
on their machine, runs `allbert serve` to start the runtime + web workspace +
channels as a managed background process, watches a health check go green, drives
day-to-day operation through `allbert <group> <command>` and the mix-free TUI
console without touching `mix`, and stores provider credentials in the OS
keychain/secret service instead of a settings file.

## Platform Support Tiers

Cross-platform packaging is a deliberate cost multiplier — a binary, a background
service, and a different OS secret vault per platform — so v0.62 sets **explicit
support tiers** rather than implying uniform support:

- **Tier 1 — macOS and Linux.** Fully supported and **freeze-blocking** for v1.0.
  The `allbert` binary, the Homebrew + curl install, the `launchd` / `systemd`
  daemon, and the macOS Keychain / Linux Secret Service vaults must all work and
  are required by the v1.0 acceptance matrix.
- **Tier 2 — Windows.** Supported via **WSL2** (a Linux target). Native Windows
  packaging, a Scheduled-Task daemon, and Windows Credential Manager are
  **best-effort / beta and explicitly NOT v1.0 freeze-blocking** unless a later
  decision promotes them. The v1.0 acceptance-matrix Windows criterion is
  satisfied by WSL2, not native Windows.

This tiering is an explicit scope decision, not an assumption: Tier 1 is committed
v0.62 work; Tier 2 native-Windows support is out of the v1.0 freeze bar.

## Packaging Feasibility Spike (M0)

Run a time-boxed feasibility spike **before committing M1-M7**, because this
codebase has **no packaging today** and the riskiest unknowns are front-loaded.
The spike must prove, on a single Tier-1 OS, that a release-built, ERTS-bundled
`allbert` binary:

- boots the supervision tree with **no Elixir/OTP toolchain present**;
- opens the **SQLite database** through the `exqlite` NIF from the packaged
  artifact (the highest-risk native-dependency unknown);
- serves the **compiled web assets** (esbuild/tailwind output) over `allbert
  serve`; and
- loads at least **one source-tree plugin** from the packaged layout.

**Explicit gate:** if the spike cannot prove all four on a Tier-1 OS with the
chosen tool (Burrito, Bakeware, or a hand-wrapped OTP release), the M1 packaging
mechanism is reconsidered before any further M1-M7 work begins. The spike result
**chooses** the packaging mechanism; M1 does not assume one in advance.

## Packaged Binary & ERTS Bundling (M1)

Build an OTP release and wrap it as a self-contained `allbert` binary, so the
artifact carries its own runtime and no Elixir/OTP toolchain is required on the
user's machine. The binary boots the same supervision tree the checkout boots.
Packaging is a delivery change, not a runtime authority change.

**First-Model-Path consideration (ADR 0078):** the v0.60 First-Model-Path
decision determines whether the binary/installer also bundles and manages a local
model runtime. If ADR 0078 chose the **assisted local model** path, v0.62
packaging bundles/manages the local model runtime (Ollama) in the binary/installer
so a freshly installed Allbert reaches a first usable model without an external
provider key; if ADR 0078 did **not** choose that path, v0.62 packaging does not
bundle a local model runtime and the binary stays provider-key-only. This is a
design-driven branch, not an assumption made here.

## Install Path (M2)

Ship a Homebrew formula/tap and a curl install script with an uninstall path.
Both paths must verify the release artifact, install only documented files, and
leave Allbert Home/user data alone on uninstall unless the operator explicitly
asks for removal.

## Unified CLI Dispatcher (M3)

A single `allbert <group> <command>` surface subsumes the flat Mix-task sprawl
behind grouped help. Operator commands move onto the binary; developer and CI
commands stay `mix`-only. Each command routes through the same
runtime/action/settings spine and remains subject to Security Central,
confirmations, and surface policy.

## Daemon Management & `allbert serve` (M4)

`allbert serve` starts the runtime, web workspace, and channels as a managed
process and installs/uninstalls an OS service when explicitly requested. A
health check covers runtime up, web reachable, and channels supervised. Service
install grants no authority beyond the surfaces already available when Allbert is
launched from a checkout.

## ADR 0070 TUI Console Completion (M5)

Absorb the remaining day-to-day admin-inspection reads into registered read-only
internal actions so an operator does not need raw `mix` for normal operation.
The reads follow the ADR 0070 pattern: `exposure: :internal`, routed through
`Actions.Runner.run/3`, slash-allowlisted, excluded from intent-router
candidates, redacted/truncated, and mutation-free.

## OS Secret-Vault Credential Handling (M6)

Settings Central holds secret references; the OS vault holds secret values.
Existing plaintext/encrypted settings credentials migrate only through an
explicit, redacted operator step. No raw secret is logged, rendered, included in
release evidence, or written into shell history by the recommended flow.

## Security Posture

No new authority. Packaging changes how Allbert is installed and invoked, not
what any surface may do. The binary, dispatcher, daemon, TUI console, and vault
all route through the existing spine and remain subject to Security Central,
durable confirmations, action boundaries, and redaction policy.

## Non-Goals

- Not a native desktop GUI client. The web workspace remains the operator UI
  through v1.0.
- No hosted/remote distribution service or plugin auto-update.
- No new capability or permission class.
- No guided onboarding wizard. v0.63 builds that on top of these entry points.
- No final product RC. v0.64 owns integrated install/onboard/work validation.

## Definition Of Done

v0.62 is done when:

- the M0 feasibility spike passed (or explicitly re-chose the packaging mechanism)
  before M1-M7 work began;
- a packaged `allbert` binary installs via Homebrew and curl and runs on a Tier-1
  OS (macOS, Linux) with no Elixir/OTP toolchain; Windows is satisfied via WSL2,
  with native-Windows support explicitly Tier 2 / non-freeze-blocking;
- `allbert <group> <command>` exposes the grouped operator surface with a clear
  operator/dev split, while development gates remain `mix`-only;
- `allbert serve` runs runtime + web + channels as a managed daemon with a
  visible health check;
- the mix-free TUI console covers day-to-day inspection through read-only
  internal actions;
- provider credentials are stored in the OS vault through Settings Central
  references, with redacted migration and no raw secret leak;
- ADR 0076 is accepted and ADR 0070 is marked converged;
- `mix allbert.test release.v062`, docs, and warning gates pass;
- no known warning/security/doc drift remains.
