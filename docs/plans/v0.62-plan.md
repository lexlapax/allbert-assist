## Purpose

Make Allbert installable and invokable as a product, not a checkout. Ship a
release-built `allbert` binary with no Elixir/OTP toolchain required on the
user's machine, a Homebrew + curl install path, a unified grouped CLI dispatcher
that separates operator commands from developer/CI commands, background-daemon
management with a visible health check, completion of the ADR 0070 mix-free TUI
console convergence, and OS secret-vault credential handling. This release
**implements** the entry-point / grouped-CLI UX **designed in v0.60 (ADR 0077,
v0.60 M5; concrete artifact: `docs/design/entry-point-cli-ux.md`)** — the
command taxonomy is a v0.60 design input, not invented or "settled" blind here.
The First-Model-Path integration is its own milestone (M4), not a packaging
footnote. v0.63 onboarding then teaches the implemented entry points.

Scope note: the v0.62 **packaging lane** is packaging/entry-point work. The
only non-packaging item intentionally carried here is **M0.1**, a small v0.61b
post-audit web-preflight exception that had no owning milestone. It is
independent of M0-M8 packaging work, may land before or during the release, and
must not be used to reopen presentation scope.

## Operator-Visible Win

A technical prosumer installs Allbert from a packaged binary with no Elixir/OTP
on their machine, runs `allbert serve` to start the runtime + web workspace +
channels as a managed background process, watches a health check go green, drives
day-to-day operation through `allbert <group> <command>` and the mix-free TUI
console without touching `mix`, stores provider credentials in the OS
keychain/secret service instead of a settings file, and on first run reaches a
usable local model through the confirmed guided Ollama path — or lands honestly
on BYOK.

## Platform Support Tiers

Cross-platform packaging is a deliberate cost multiplier — a binary, a background
service, and a different OS secret vault per platform — so v0.62 sets **explicit
support tiers** rather than implying uniform support:

- **Tier 1 — macOS and Linux.** Fully supported and **freeze-blocking** for v1.0.
  The `allbert` binary, the Homebrew + curl install, the `launchd` / `systemd`
  daemon, and the macOS Keychain / Linux Secret Service vaults (on the
  three-tier fallback model) must all work and are required by the v1.0
  acceptance matrix.
- **Tier 2 — Windows.** Supported via **WSL2** (a Linux target). Native Windows
  packaging, a Scheduled-Task daemon, and Windows Credential Manager are
  **best-effort / beta and explicitly NOT v1.0 freeze-blocking** unless a later
  decision promotes them. The v1.0 acceptance-matrix Windows criterion is
  satisfied by WSL2, not native Windows.

This tiering is an explicit scope decision, not an assumption: Tier 1 is committed
v0.62 work; Tier 2 native-Windows support is out of the v1.0 freeze bar.

Architecture matrix note: "macOS Tier 1" means the macOS install/serve/vault
product path is freeze-blocking, but the exact macOS CPU matrix is ratified at
S2 after the M0 spike. The provisional freeze-blocking matrix is
`macos-arm64`, `linux-x64`, and `linux-arm64`; S2 must either promote
`macos-x64` to freeze-blocking or explicitly record it as a non-blocking
compatibility target before implementation continues. Native Windows remains
Tier 2 and is not part of this artifact matrix.

## Research Grounding (surveyed 2026-07-05; sources in the readiness-pass record)

- **Bakeware is dead** (repo archived 2024-09-18, README points at Burrito).
  The spike is a **two-way** choice: Burrito 1.5.x vs a hand-wrapped
  `mix release` tarball + thin launcher.
- **Burrito 1.5.x realities:** build host needs **Zig 0.15.2 pinned** (+xz/7z
  for Windows targets); it fetches "universal machine" OTP/ERTS tarballs from a
  **project-run third-party CDN** at build time (OTP 29.0.1 confirmed served —
  a supply-chain decision to accept or replace with own-CI ERTS); the binary
  **self-extracts** the release to a per-OS cache dir on first run (upgrade
  residue + uninstall must call `maintenance uninstall`); and its cross-build
  NIF recompilation only detects `elixir_make` deps and forces
  `zig cc -shared` — **rebar3-built erlexec is invisible to it and
  muontrap/erlexec are port executables, not shared objects**, so
  cross-compiling this dep tree with Burrito is not viable. **CLI dispatch
  under Burrito:** there is no multi-command plug system — the entry is always
  the release `:mod` Application start; argv arrives via
  `Burrito.Util.Args.argv()` and commands dispatch in-app then
  `System.halt/1` (the Next LS pattern; `serve` holds the tree). The Zig
  wrapper **reserves the `maintenance` subcommand namespace**
  (`maintenance uninstall|directory|meta` are intercepted before the app sees
  argv) — the M3 taxonomy must not collide with it.
- **Therefore: native per-target CI builds** on matching runners regardless of
  the wrapper choice (`mix release` cannot cross-compile either). GitHub
  Actions runners exist for macos-arm64, macos-x64, linux-x64 (ubuntu-22.04
  for the old-glibc floor), linux-arm64.
- **exqlite ships precompiled NIFs** (elixir_make precompiler + cc_precompiler)
  — helps native builds; does not make cross-builds safe.
- **macOS distribution nuance:** curl/scp do **not** set the quarantine
  attribute and Homebrew manages it — so the two v0.62 install paths run
  unsigned binaries without Gatekeeper prompts; browser-downloaded artifacts DO
  get quarantined and post-Sequoia there is no ctrl-click override. Bare
  Mach-O binaries can be signed + notarized (hardened runtime, `notarytool`)
  but **cannot be stapled**; self-extraction vs hardened-runtime library
  validation is a known pain class to test whenever signing lands.
- **Homebrew mechanics:** third-party tap (`homebrew-tap` repo) with
  per-platform `url`+`sha256` pointing at GitHub release artifacts. Casks
  handle prebuilt artifacts and gained Linux support (Homebrew 4.5/5.0), **but
  `brew services` service blocks are formula-only** (no cask equivalent), and
  third-party CLI taps in practice ship formulas with prebuilt binaries (the
  HashiCorp-tap pattern; "no source builds" is a homebrew-core rule, not a tap
  rule) — choosing cask at S3 forecloses `brew services`. Building Elixir from
  source in a formula is a non-starter.
  Precedent: Next LS (Burrito binaries + self-hosted tap); Livebook Desktop
  (hand-rolled per-OS bundles).
- **install.sh conventions:** versioned immutable artifact URLs; SHA256SUMS
  published with the release and verified inside the script; script wrapped in
  `main()` and download-then-inspect friendly; **uninstall manifest** of
  installed paths. Cosign/sigstore-signed checksums are the emerging standard
  and are cheap in GitHub Actions (recorded as a candidate, not required).
- **No maintained Elixir/Erlang OS-keychain library exists (2026).** Realistic
  mechanisms: shell out to `security` (macOS) / `secret-tool` (Linux
  libsecret), or reuse the docker-credential-helper protocol/binaries. Linux
  Secret Service needs a D-Bus session + keyring daemon — **typically absent on
  headless servers**, which is a core daemon deployment. Precedent for the
  fallback tier: gh CLI (keyring → documented file fallback → env), docker.
- **Service management:** `launchctl bootstrap gui/$UID` (legacy
  `launchctl load` is deprecated); `systemd --user` units +
  `loginctl enable-linger` (user units keep the daemon in the user session
  where Secret Service lives — this materially eases the vault story); logs to
  `~/Library/Logs/<name>/` / `journalctl --user -u allbert`.
- **Ollama:** the official download page documents an install script for
  macOS/Linux, and Homebrew publishes a maintained `ollama` formula with
  macOS/Linux bottles. v0.62 treats both as supported upstream paths, but S4
  must choose the exact command Allbert may execute on each Tier-1 OS; Allbert
  never re-hosts Ollama artifacts. **Programmatic pull:** the sanctioned path
  is the localhost REST API (`POST /api/pull`, `stream: true` — structured
  progress events, cancel by connection close with documented resume; strictly
  better trace semantics than scraping the `ollama pull` CLI). **Service
  reality:** `brew install ollama` does NOT auto-start the server (the caveat
  says `brew services start ollama`); the official Linux script installs an
  enabled system systemd unit — so detection is a three-way probe (binary on
  PATH / server answering `127.0.0.1:11434` / service state), and on macOS
  Allbert must also start the server. Hardware floor guidance: ~8 GB RAM floor
  for a 3–4B curated default; 7–8B models want 16 GB.
- **ERTS bundling gotchas:** build Linux artifacts against an old-glibc floor
  (ubuntu-22.04) or ship musl-dynamic; **fully static linking is off the table**
  for a NIF-loading app; OTP's crypto NIF links libcrypto dynamically by
  default — a Homebrew-built dev ERTS breaks on machines without Homebrew
  OpenSSL, so the shipped ERTS must have portable crypto linkage (verify with
  `otool -L`/`ldd` in the gate); pin the ERTS/OTP version as a CI input.

Research evidence ledger for implementers:

- Burrito primary docs: `https://github.com/burrito-elixir/burrito` and
  `https://hexdocs.pm/burrito` — verify Zig/tool requirements, self-extraction,
  maintenance commands, custom ERTS, Phoenix `server: true`, and assets before
  choosing the wrapper.
- Homebrew primary docs: `https://docs.brew.sh/Formula-Cookbook` and
  `https://docs.brew.sh/Cask-Cookbook` — verify formula-vs-cask mechanics,
  service block support, per-platform URLs, checksums, and Linux cask support
  before S3.
- Ollama primary docs: `https://ollama.com/download`,
  `https://formulae.brew.sh/formula/ollama`, and `https://ollama.com/library`
  — verify install commands, package availability, model tags, sizes, and
  current license before S4 selects the curated model or execution path.

### Attach Transport Contract

M3/S2 selected and M8.1 completed the Unix-domain socket transport. The
implemented mechanism is `AllbertAssist.Runtime.Attach` plus
`AllbertAssist.Runtime.Attach.Server`: `:gen_tcp` over `{:local, path}` with
`packet: 4` framing, a per-Allbert-Home token, and an identity handshake that
covers protocol version, expanded Allbert Home, user id, and app version. The
daemon-side listener starts only in `allbert serve`/daemon mode with
`ALLBERT_HOLD_WRITER_LOCK=1`; normal embedded commands do not expose an attach
listener. Erlang distribution was not used.

The **behaviors below are binding contract**:

- **Local-only:** the primary and shipped transport is a Unix-domain socket
  under Allbert Home runtime state. No routable TCP listener is exposed.
- **Path and ownership:** `runtime/attach.sock` and `runtime/attach.token` live
  under Allbert Home. The runtime directory and token are owner-only where the
  platform supports it. The OS socket-path length cap (~104 bytes macOS /
  108 Linux) remains a known operator-support constraint for unusually deep
  Home paths.
- **Mismatch behavior:** a command refuses to attach when daemon version,
  Allbert Home, user id, or protocol version mismatches; it prints repair
  guidance instead of booting a second writer against the same database.
- **Single writer:** the boot/migration lock is **new work (M3/M5)** — a held
  `BEGIN EXCLUSIVE` on a **sidecar lock database** beside the main DB
  (fcntl-backed through the already-shipped exqlite NIF: cross-process,
  auto-released on process death; a second BEAM gets `SQLITE_BUSY`
  immediately → refuse boot with guidance). The attach socket is the
  liveness/discovery probe (connect refused → stale socket → unlink and take
  the lock). SQLite's own `busy_timeout` on the main DB is explicitly **not**
  the guard — WAL serializes transactions, it does not prevent a second
  writer process. The lock+probe composition is proven in M0 proof 7.

### M4 Authority Contract

M4 is allowed to execute only through already-reviewed authority classes. If an
effect cannot be represented by the existing classes, M4 stops and records a
scope blocker instead of adding a new permission atom during implementation.

override (`ALLBERT_VAULT_BACKEND`, ratified at S5); onboarding-state storage
shape is decided in M3 (settings key only if additive under ADR 0046, else a
Home-dir state file); layout preferences remain client-side. Secret
*references* use the existing schema types (Current Code State 10).

## Upgrade, Rollback & Coexistence (scope per Locked Decision 15)

In scope for v0.62: backup-before-migrate on version-changed boot; migration
logging (drop `log: false`); the packaged-vs-checkout coexistence policy
(documented separation via `ALLBERT_HOME`/port + the single-writer guard);
service-worker/asset cache correctness across binary upgrades (the versioned
`?vsn` + SW cache-name mechanism from v0.61b, re-verified in the smoke
harness). **Deferred to v0.64 (recorded, not silent):** automated rollback /
down-migrations; delta upgrades; signed binaries (Locked Decision 14).

Backup-before-migrate means a local copy of the SQLite database plus settings
metadata sufficient for operator-assisted recovery, written outside the migration
transaction before version-changing migrations run. It is not automated rollback:
the packaged boot may refuse to continue if backup creation fails, and v0.64
owns restore/down-migration automation.

## Security Posture

No new authority. Packaging changes how Allbert is installed and invoked, not
what any surface may do. The binary, dispatcher, daemon, TUI console, and vault
all route through the existing spine and remain subject to Security Central,
durable confirmations, action boundaries, and redaction policy.

Recorded with teeth rather than as posture prose:

- The release's **four new network touches** (install-script artifact fetch,
  brew tap fetch, Ollama installer fetch, curated model pull) are enumerated
  in ADR 0076's Distribution Trust section; the latter two execute only behind
  explicit operator confirmations with trace records (Locked Decision 8,
  `ollama-install-and-pull-confirmed-001`).
- The binary carries **no telemetry, no phone-home, no auto-update check**.
- Daemonization must not widen defaults: `WorkingDirectory` pinned; the
  cwd-derived `Scope.directory_subtree` fallbacks audited in M5.
- Release mode must not weaken enforcement: the Mix-probe `production?()`
  class is fixed in M1 (Current Code State 6) and gate-asserted.
- The env-key bypass (`:req_llm`) is documented as vault tier 3 and surfaced
  in inspection — never an invisible side channel.

## Non-Goals

- Not a native desktop GUI client. The web workspace remains the operator UI
  through v1.0.
- No hosted/remote distribution service or plugin auto-update.
- No new capability or permission class.
- No guided onboarding wizard. v0.63 builds that on top of these entry points
  (v0.62 M3 ships the first-run detection + wizard routing hook only).
- No final product RC. v0.64 owns integrated install/onboard/work validation,
  **signing/notarization (Locked Decision 14), and automated rollback (Locked
  Decision 15)**.
- No native-Windows packaging (Tier 2, WSL2 only).
- No new plugin-code loading mechanism for packaged installs (the compile-time
  constraint is documented, not solved, in v0.62).

## Definition Of Done

v0.62 is done when:

- the M0 feasibility spike passed all eight proofs (or explicitly re-chose the
  packaging mechanism) and its report was ratified at S2 before M1-M8 work
  began;
- the M0.1 carried-in web tickets landed with tests;
- a packaged `allbert` binary installs via Homebrew and curl and runs on a
  Tier-1 OS (macOS, Linux) with no Elixir/OTP toolchain — plugins registered,
  stocksage migrations applied, web served; Windows is satisfied via WSL2,
  with native-Windows support explicitly Tier 2 / non-freeze-blocking;
- `allbert <group> <command>` exposes the grouped operator surface per the
  ratified disposition table, attach-first with the single-writer guard, with
  bare-`allbert` first-run detection implementing the v0.60 design;
- first-run reaches a usable model via the confirmed guided Ollama path or
  degrades honestly to BYOK (all 7 states);
- `allbert serve` runs runtime + web + channels as a managed per-user daemon
  with a visible health check and a pinned working directory;
- the mix-free TUI console covers the enumerated day-to-day read set through
  read-only internal actions, from the packaged binary;
- provider credentials resolve through the three-tier vault model with
  redacted migration and no raw secret leak;
- ADR 0076 is accepted (with Distribution Trust) and ADR 0070 is marked
  converged;
- `mix allbert.test release.v062`, docs, and warning gates pass locally; the
  artifact smoke harness passes in the remote matrix before manual
  operator-validation closeout;
- version reads 0.62.0 with the tag held for the operator; no known
  warning/security/doc drift remains.
