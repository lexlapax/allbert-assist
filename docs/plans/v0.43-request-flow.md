## Main User/Operator Flows

- Operator configures Discord or Slack credentials and identity maps.
- External user mentions Allbert in an authorized channel.
- Allbert replies in thread or channel.
- External user approves or denies a confirmation callback.

## Data Flow

Provider events become durable channel events, resolve external identity to
local `user_id`, submit input through `Runtime.submit_user_input/1`, render
responses through provider-specific renderers, and update channel event status.

## Authority Invariants

Channels are delivery adapters only. They do not own runtime logic,
confirmation authority, resource grants, app context, or durable objectives.
Approval Handoff remains action-owned.

Status: planned. Commands are illustrative until the milestone lands; verify and
replace with observed output at closeout.

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v043.XXXXXX)"
export ALLBERT_TRACE_ENABLED=true
# Register Discord and Slack with test tokens (entered as secret refs, never as
# plain arguments), then inspect them.
mix allbert.channels show discord
mix allbert.channels show slack
# In an authorized channel: mention Allbert, get a threaded reply, and approve a
# confirmation callback for one effectful action.
rg -i 'token|secret' "$ALLBERT_HOME" || true
```

Expected behavior:

- only authorized channels/mentions are handled; replies thread correctly and
  identity maps to a stable local user.
- an effectful action renders a confirmation the external user approves or denies
  via callback; denial runs nothing.
- bot tokens are stored as secret refs and never appear in traces/audits.
