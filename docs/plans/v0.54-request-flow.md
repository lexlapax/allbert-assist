## Locked Decisions
Recorded in `v0.54-plan.md`. Authority invariants reaffirmed: model output
advisory re: authority; intent never grants permission; `active_app` ranking-only;
bounded/redacted multi-turn context (no raw history into the prompt);
pending-clarification replies still registry-validated; no code execution during
collection; no new Surface atom. New: the two-stage local router is the **default
selector** (deterministic fast-path + offline fallback retained); approval gating
is a **separate layer** keyed on the resolved action+args; the app-handoff channel
dead-end is removed in favour of a channel-answerable targeted clarify.

## Data Flow
```text
turn input + bounded thread_context summary
  -> [fast-path] deterministic ladder for explicit/unambiguous routes (kept)
  -> Engine.decide_without_explicit_route: collect candidates -> Ranker.rank (multi-turn aware)
  -> Intent.Router (default :two_stage_local):
        Stage 1  embedding prefilter over the in-memory utterance index -> top-K + margin
        Stage 2  ReqLLM.generate_object over the shortlist (enum-constrained):
                 {:execute, action, slots, confidence} | {:clarify, shortlist, question}
                 | {:answer} | {:none}
        confidence gate (logprob + margin) -> act | targeted clarify | abstain
        (model down/timeout -> deterministic fallback)
  -> :execute  -> run_validated_route -> Actions.Runner.run/3 -> Security Central (still requires active_app)
                  -> confirmation:required => Confirmations.create -> ApprovalHandoff.pending
                     -> approval_handoff -> channel ConfirmationCallback (approve/deny buttons | ALLBERT:APPROVE:<id>)
  -> :clarify  -> emit channel-answerable targeted question (real competing candidates)
                  + persist pending-clarification (TTL, thread/user-keyed)
                  -> next turn: resolve reply against offered set -> re-validate via Registry -> Runner.run
  -> :answer/:none -> direct answer | graceful "can't do that" (NO app-handoff dead-end)
```

## Authority Invariants
- Engine/Router/Classifier are proposal infrastructure; the runner + Security
  Central + the confirmation gate are the only authority boundary (ADR 0019 §1,
  ADR 0060).
- The Router selects **which** action within the already-collected,
  registry-validated candidate set; it never grants authority, never sets a
  `confirmation_id`, never lowers a safety floor (ADR 0060).
- A `confirmation: :required` action — however routed (fast-path, router execute,
  or clarify-resolved) — reaches the existing approval primitive; routing and
  approval are independent layers.
- `active_app` carried across turns is ranking signal only; never authorizes.
- Pending-clarification resolution re-validates against the Registry; never
  auto-executes; TTL-bounded; cross-user/thread isolated.
- Router/classifier prompt receives only a bounded/redacted summary + the
  shortlisted action names; raw prompts/completions and raw history are never
  traced (ADR 0019 §2). Embeddings are local-only; hosted escalation (if enabled)
  is audited.

## Handles
- No new operator command for routing; behavior surfaces through chat + ephemeral
  clarification + the existing channel approve/deny primitive. New `intent.router_*`
  + deepening `intent.*` settings. `mix allbert.intent doctor` reports the router
  (embedding availability, index, model tiers), redacted.
- **M9 / ADR 0062 (planned):** descriptor-lifecycle handles —
  `mix allbert.intent list|show|optimize|reindex|edit|disable|review|promote`,
  `doctor` extended with a **coverage report**; an operator **Intents** web panel;
  the `optimize_intent_descriptors` action; `intent.descriptor_autoaccept` setting.

## Descriptor Lifecycle (M9 / ADR 0062, planned)
```text
action set changes at runtime
  (plugin install | marketplace bundle | dynamic-codegen integrate/rollback | write-code)
   -> SignalBus event
      (allbert.dynamic_codegen.{registered,rolled_back,reconcile_completed}
       + NEW allbert.{app,plugin}.registered / allbert.action.registry_changed)
   -> Index subscriber (debounced/coalesced):
        mark index :not_built  +  enqueue descriptor-generation pass for new actions
   -> DescriptorResolver merges (deduped by {app_id, action_name}, later wins):
        code-declared (intent_descriptors/0)
          < generated  (local model: label/examples/synonyms from name+description+capability;
                         no egress; redacted; heuristic fallback)
          < operator overrides  (<ALLBERT_HOME>/intents/overrides/*.md|yaml, trusted-local, audited)
   -> generated descriptors for dynamic/write-code actions land in a REVIEW tier (inert)
        unless intent.descriptor_autoaccept=true; operator review/promote (Drafts + confirmation)
   -> Index.rebuild over the resolved set -> Stage 1 shortlist sees the new actions
   -> routing/authority unchanged: a descriptor is a HINT; the action's own
      permission/confirmation/exposure still govern (ADR 0060). Routable ≠ executable.
operator curation surfaces: CLI (mix allbert.intent ...), Intents web panel,
  md/yaml under <ALLBERT_HOME>/intents/{generated,overrides,drafts,audit}/.
```

## Smoke Checklist
- **Channel approval end-to-end**: a channel message → a `confirmation: :required`
  action → approve/deny buttons render → tap approve resolves; deny cancels.
- Ambiguous "note" request → targeted clarify (create vs search) → reply executes.
- "create a note" routes to `write_note`, not the search descriptor.
- Two-turn follow-up resolves; pronoun/ellipsis carries antecedent.
- Ambiguous prompt lists real competing candidates; no silent selection.
- Clarify reply naming an unregistered action falls back (no execution).
- Stale clarification reply re-classified fresh; cross-user state isolated.
- Embedding model absent → router falls back to deterministic; no hard failure.

## Manual-Validation Punchlist (v0.54, live local model)

Run these against a real local router to confirm the end-to-end workflow that the
deterministic gate (`mix allbert.test release.v054`) cannot exercise. The punchlist
is **CLI-driven and self-contained** — every step is a copy-pasteable shell command
with its expected output, run from the repo root in a single shell that holds a
disposable `ALLBERT_HOME`. No live phx.server, browser, or external provider is
required (an optional channel-button cross-check is noted at the end).

`mix allbert.ask` runs in the **dev** environment, where the test-only
`:intent_router_strategy_override` does **not** apply — so the router is live
(`strategy=two_stage_local` from settings) and genuinely calls Ollama. Each `mix`
invocation is a cold BEAM, but every step persists to the same `ALLBERT_HOME`
SQLite store, so confirmations created by `mix allbert.ask` are approvable by a
later `mix allbert.confirmations` call.

### Setup

```bash
# S1. Pull the local models the router uses by default (all US-origin, A/B-validated
#     — see v0.54-plan.md Appendix A):
#       embedding_local         = nomic-embed-text (Stage-1 prefilter)
#       router_local            = llama3.1:8b      (Stage-2 disambiguator)
#       router_escalation_local = gemma4:26b       (local escalation, on by default)
ollama pull nomic-embed-text
ollama pull llama3.1:8b
ollama pull gemma4:26b
ollama list        # expect nomic-embed-text, llama3.1:8b, gemma4:26b listed

# S2. Disposable home + migrated schema (note the task name: allbert.ecto.migrate).
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v054.XXXXXX)"
echo "$ALLBERT_HOME"
mix allbert.ecto.migrate --quiet      # exits 0, no output with --quiet

# S3. Enable the assist runtime model (so the request reaches IntentAgent at all).
#     The router's own model/embedding profiles are separate (router_local /
#     embedding_local) and need no enabling beyond Ollama serving them.
mix allbert.model use slow --enable-assist     # or: mix allbert.model use local --enable-assist

# S4. Router health.
mix allbert.intent doctor
```

Expected `mix allbert.intent doctor` output (a cold one-shot invocation):

```text
intent router doctor status=ok
strategy=two_stage_local
embedding_profile=embedding_local endpoint=available dim=768
model_profile=router_local escalation=router_escalation_local
index status=not_built size=0
```

- `status=ok` + `endpoint=available` + `dim=768` is the keystone health signal
  (the local embedder responded). `dim` reflects your embedding model
  (`nomic-embed-text` → 768).
- `index status=not_built size=0` is **expected and correct** for a one-shot CLI
  doctor: the utterance index builds lazily on the first route inside a
  long-running process, and a one-shot `mix` task exits before any route runs. The
  index is exercised live in check A1 below.
- If Ollama is down, `status=unavailable endpoint=unavailable dim=` — the router
  then falls back to the deterministic ladder (verified in check D).

### A. Routing via `mix allbert.ask`

**A1 — Execute → approval (the keystone; formerly the channel dead-end).**

```bash
mix allbert.ask --new-thread "create a note titled v054 with body hello world"
```

Expect the router to select `write_note` (a `confirmation: :required` action) and
the runtime to stop at the approval gate — **not** an inert "hand this to the notes
app" dead-end:

```text
Status: needs_confirmation

<a sentence describing the pending write_note and how to approve it>

Signal: <signal id>
Trace: none
User: local
Thread: <THREAD_ID>

Approval Handoff:
  <one or more human-readable handoff lines>
  Details: mix allbert.confirmations show <CONF_ID>
  Approve: mix allbert.confirmations approve <CONF_ID>
  Deny: mix allbert.confirmations deny <CONF_ID>
Actions:
- write_note (pending)
  Confirmation: <CONF_ID>
```

Then approve and confirm the side effect actually runs:

```bash
mix allbert.confirmations list                       # shows <CONF_ID> as pending
mix allbert.confirmations approve <CONF_ID>          # expect Status: completed; note written
mix allbert.confirmations list --resolved            # <CONF_ID> now resolved/approved
```

Pass criteria: status `needs_confirmation` (never a dead-end), a real `<CONF_ID>`
attached to `write_note (pending)`, and `approve` reports the note was written.

**A2 — Create-vs-search routing (the original mis-route).**

```bash
mix allbert.ask --new-thread "create a note about quarterly goals"   # -> write_note (needs_confirmation)
mix allbert.ask --new-thread "find my notes about quarterly goals"   # -> search_notes (completed, read-only)
```

Pass criteria: the first selects `write_note` and stops at approval; the second
selects `search_notes` and completes without a confirmation (read-only). The old
behavior where "create a note" mis-routed to the search descriptor is gone.

**A3 — Clarify loop (multi-turn, thread-keyed).**

```bash
mix allbert.ask --new-thread "note"
```

Expect a targeted clarify question naming the real competing candidates
(create vs. search) — capture the printed `Thread:` value as `<THREAD_ID>`:

```text
Status: needs_clarification

<targeted question, e.g. "Do you want to create a note or search your notes?">

Signal: <id>
Trace: none
User: local
Thread: <THREAD_ID>
```

Resolve it on the **same thread**, against the offered set:

```bash
mix allbert.ask --thread <THREAD_ID> "search"        # binds to the offered search option and runs
```

Negative/stale cases:
- Reply something unrelated on a fresh thread instead → it re-classifies fresh, no
  stale binding.
- Wait past `intent.pending_clarification_ttl_ms` (default 120000 ms) before
  replying → the stale pending clarification is dropped and the reply is
  re-classified fresh.

**A4 — Answer / graceful none (no dead-end on the abstain path).**

```bash
mix allbert.ask --new-thread "what can you help me with?"            # -> Status: completed, a direct answer
mix allbert.ask --new-thread "asdkfj qwioutr nonsense zzz"          # -> graceful "can't do that", NOT an app-handoff dead-end
```

### B. Channel approve/deny cross-check (optional)

The CLI `confirmations approve` in A1 already exercises the same approval gate a
channel button hits. To additionally confirm the channel **rendering** path, use a
provider that supports callback simulation (discord/slack; telegram has only
`simulate`):

```bash
mix allbert.channels discord simulate "create a note titled chan with body hi"
# capture the rendered confirmation id, then:
mix allbert.channels discord simulate-callback --custom-id allbert:v1:approve:<CONF_ID>
```

Pass criteria: the simulated inbound renders an approve/deny primitive and the
approve callback resolves the confirmation (deny cancels it).

### C. Graceful deterministic fallback

```bash
mix allbert.settings set intent.router_strategy deterministic
mix allbert.ask --new-thread "create a note titled fallback with body hi"   # still routes to write_note via the ladder
mix allbert.settings set intent.router_strategy two_stage_local             # restore
```

Equivalently, stop Ollama and repeat A1 with `two_stage_local` — the router
detects the embedder is unavailable and falls back; nothing hard-fails.

### D. Local escalation (on by default → `router_escalation_local`/gemma4:26b)

Escalation is enabled by default to a **local** higher-tier model (no network
egress). When the primary router model returns a low-confidence / ambiguous /
out-of-shortlist selection, the router escalates **once** to gemma4:26b and emits a
redacted `[intent_router_escalation]` Logger.warning; the routed action set is
unchanged (escalation grants no authority). To observe it, watch the server/CLI log
on a borderline input; to A/B the effect, disable it and compare:

```bash
mix allbert.settings get intent.router_escalation_profile   # -> router_escalation_local (default)
mix allbert.settings set intent.router_escalation_profile "" # disable: low-confidence now stays a clarify
# ...repeat a borderline route, observe no [intent_router_escalation] warning...
mix allbert.settings set intent.router_escalation_profile router_escalation_local  # restore default
```

To escalate to a **hosted** model instead, point the profile at an enabled hosted
profile (audited; this one does egress).

### E. Authority + leak gate

```bash
# E1. Authority unchanged: an effectful action still requires its normal
#     confirmation (the router never auto-approves). Re-confirm via A1 that
#     write_note reached needs_confirmation and required an explicit approve.

# E2. No raw secrets in the home tree (router/escalation logs are redacted).
rg -i 'token|password|secret|\+[0-9]{6,}' "$ALLBERT_HOME" || true
#     Expect no raw bot tokens / passwords / phone numbers; resolve any hit.
```
