## Locked Decisions
Recorded in `v0.54-plan.md`. Authority invariants reaffirmed: model output
advisory; intent never grants permission; `active_app` ranking-only; bounded/
redacted multi-turn context (no raw history into the prompt); pending-clarification
replies still registry-validated; no code execution during collection; no new
Surface atom.

## Data Flow
```text
turn input + bounded thread_context summary
  -> Engine.decide: collect candidates -> Ranker.rank (now multi-turn aware)
     -> Classifier (advisory re-rank within collected set, bounded prompt)
  -> decision: direct_answer | registry_action | app_handoff
     | clarify_intent (generalized: top-N competing candidates as options)
  -> if clarify: emit ephemeral options + persist pending-clarification (TTL)
  -> next turn: resolve reply against offered set -> re-validate via Registry
     -> Actions.Runner.run/3 -> Security Central (still requires active_app)
```

## Authority Invariants
- Engine/Classifier are proposal infrastructure; the runner + Security Central
  are the only authority boundary (ADR 0019 §1).
- Classifier may only re-rank/select within already-collected, registry-validated
  candidates (ADR 0019 §2; ADR 0034 §6/§10).
- `active_app` carried across turns is ranking signal only; never authorizes.
- Pending-clarification resolution re-validates against the Registry; never
  auto-executes; TTL-bounded; cross-user/thread isolated.
- Classifier prompt receives only a bounded/redacted summary; raw prompts/
  completions never traced (ADR 0019 §2).

## Handles

## Smoke Checklist
- Two-turn follow-up resolves; pronoun/ellipsis carries antecedent.
- Ambiguous prompt lists real competing candidates; no silent selection.
- Clarify reply naming an unregistered action falls back (no execution).
- Stale clarification reply re-classified fresh; cross-user state isolated.
