# Allbert v0.11 Request Flow: Intent Decisions And Approval Handoff

## Runtime Turn

1. A CLI, LiveView, test, job, or future channel submits input through
   `AllbertAssist.Runtime.submit_user_input/1`.
2. The runtime constructs an inert `AllbertAssist.Intent.Decision` with:
   selected intent, confidence, selected and candidate skills/actions,
   permission, risk summary, confirmation need, execution mode,
   `resource_access`, `approval_handoff`, alternatives, diagnostics, trace
   metadata, and reserved optional `user_id`, `thread_id`, `session_id`, and
   `active_app` context.
3. The decision is validated before any action runs. Validation checks known
   skills, registered actions, known permissions, known execution modes,
   resource operation classes, Security Central policy, Settings Central
   policy, and confirmation state.
4. If the decision is invalid or unsupported, the runtime returns a refusal or
   unavailable result with diagnostics and trace metadata. It does not attempt
   a fallback read, fetch, import, crawl, package install, script run, or MCP
   call.
5. If the decision is safe and does not require confirmation, execution
   proceeds only through registered actions and
   `AllbertAssist.Actions.Runner.run/3`.
6. If the decision requires confirmation, the target action creates or returns
   a durable pending request. The runtime attaches Approval Handoff data for
   the current surface and stops before the risky effect.
7. Resolution happens through `approve_confirmation` or `deny_confirmation`.
   Approval resumes the original registered target action through the runner;
   denial performs no local read/import, fetch, summarization, extraction,
   install, script execution, grant write, or downstream consumer call.
8. Runtime traces record the selected decision, rejected candidates, resource
   posture, confirmation id, handoff state, and final target status when the
   action later resumes.

## Decision Validation

Validation is deterministic and happens before any side effect:

- `selected_action` must resolve through `AllbertAssist.Actions.Registry`.
- `permission`, `execution_mode`, confirmation requirement, and `resumable?`
  come from registry capability metadata, not model output.
- `selected_skill` must resolve through the skill registry when present.
- `resource_access` entries validate through `AllbertAssist.Resources.Ref`,
  `ResourceURI`, and `OperationClass`.
- Security Central must allow or require confirmation for the selected action
  permission before the target action can run.
- Unsupported schemes such as `mcp://`, `agent://`, and `agent+https://`
  route only to `unsupported_resource_workflow` or refusal.

## Resource Access Flow

`resource_access` is descriptive posture data, not an executor. For operations
that read, write, run, fetch, inspect, summarize, import, install, or call a
resource, the decision carries:

- canonical `resource_uri` and redacted/display URI
- origin kind and canonical path or URL when applicable
- operation class, access mode, scope, and downstream consumer
- expected content kind, accepted content types, byte/output caps, redirect or
  retry posture, parser/summarizer/importer requirement, and cache/digest
  expectation where relevant
- origin channel/surface and response target
- allowed approval scopes and remembered approval choices

Operation class is a security boundary. A remembered `summarize_url` grant
does not authorize `import_skill`; an `inspect_document` grant does not
authorize package installation; an `import_local_skill` grant does not
authorize `run_skill_script`.

URL and remote document fetches must use the existing confirmed v0.10
`external_network_request` substrate. Generic local file inspection is
represented as `file://` posture and unavailable unless a registered bounded
read action with confirmation, caps, redaction, and tests is explicitly added;
there is no shell-command fallback for local reads.

## Approval Handoff

Approval Handoff is plain data attached to a decision or action result when
the result is `:needs_confirmation`.

CLI and web surfaces render the same handoff natively:

- redacted action summary
- canonical or redacted resource summary
- operation class, access mode, scope, limits, and downstream consumer
- confirmation id and current status
- approve, deny, details, and allowed remember-scope choices
- same-channel and cross-channel result-return expectations

Surfaces never mutate confirmation records directly. They call registered
confirmation actions through the action runner.

## Unsupported States

v0.11 may represent these requests, but unsupported operations remain inert
until a later plan adds a concrete action, security posture, approval story,
and adapter:

- `mcp://` resources, MCP tools, and future MCP app calls
- `agent://` and `agent+https://` delegation
- broad website crawling, recursive browsing, browser automation, or arbitrary
  multi-page collection
- document formats without an approved extractor/parser
- URL summaries when no summarizer action is registered
- generic local file inspection when no registered bounded read action exists

Unsupported results should name the missing adapter or capability and preserve
the decision/trace metadata. They must not silently downgrade into raw file
reads, `Req` calls, browser automation, shell commands, local crawlers, or
model-generated scripts.

## Guardrails

- No new file, browser, crawler, MCP, agent, package, shell, script, or network
  primitive beyond the v0.08, v0.09, and v0.10 substrates.
- No model-selected side effects without validated decisions.
- No channel-owned approval storage, grant matching, security policy, or
  execution.
- No conversation-history storage; v0.12 plugs thread context into the
  reserved decision fields.
- No app-scoped routing; v0.15 and v0.20 consume `active_app` after the app
  contract and scratchpad exist.

## Acceptance Smoke

- A risky URL summary prompt produces an inspectable decision and pending
  Approval Handoff before any fetch.
- A document inspection prompt produces `inspect_document` posture and either
  pending approval or an explicit unsupported extractor state.
- A direct skill URL import prompt produces `import_skill` posture and does
  not trust, enable, activate, install dependencies, or execute scripts.
- A local skill directory import prompt produces `import_local_skill` posture
  and does not authorize `run_skill_script`.
- A generic local file inspection prompt produces `file://` posture and either
  a pending bounded-read handoff, if such an action exists, or an explicit
  unavailable state without shell fallback.
- `mcp://` and `agent://` prompts return unsupported workflow results without
  execution.
- CLI and web render approval details through the same handoff data and resume
  only through registered confirmation actions.
