# v0.8 M10 regression checklist (preparation)

Status: Prepared (2026-04-21)

This file captures the concrete test targets and fixture expectations for v0.8 M10 so implementation can proceed as a focused test pass instead of another design pass.

## Target scenarios

1. **Cross-channel continuity**
   - REPL turn then Telegram turn under the same identity.
   - Assert one session journal and `channel:` headers per turn.
2. **Concurrent inbound arbitration**
   - Near-simultaneous REPL and Telegram inbound messages.
   - Assert last-speaker-wins at boundary, deterministic queued ordering.
3. **Cross-surface approval resolution**
   - Approval created from Telegram-facing turn.
   - Resolved from REPL/CLI identity surface with correct audit attribution.
4. **Profile import/export roundtrip (overlay)**
   - Export profile tarball.
   - Wipe profile.
   - Import with overlay and assert memory + approvals rehydrate.
5. **Profile import/export roundtrip (replace)**
   - Export profile tarball.
   - Wipe profile.
   - Import with `--replace --yes` while daemon is not running.
   - Assert clean lockfile state and expected session metadata.

## Fixture posture

- Run against the real daemon binary over IPC and filesystem paths.
- Use provider-free scripted LLM responses (same pattern as existing daemon integration tests).
- Use a workspace-local temporary `ALLBERT_HOME` per test process.
- Keep Telegram-path coverage fixture-based and deterministic (no network bot dependency).

## Exit criteria for M10

- New integration tests run under default `cargo test` for `allbert-daemon`.
- Assertions cover both happy-path behavior and at least one failure-mode per scenario.
- No M10 test depends on live providers, secret material, or external network.
