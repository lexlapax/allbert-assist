# Allbert v0.26b Backend Runtime Bugfix Plan

Status: implemented. Date: 2026-05-21.

This document is the planning artifact for **v0.26b - Backend Runtime
Bugfix Pass**, a fast-follow backend/runtime sibling to
`docs/plans/v0.26a-ui-plan.md`.

v0.26a owns the operator workspace UX/UI pass and another agent is already
working on M28-M34. During that audit, three non-UI bugs were flagged:

- H1: setting-shaped prompts fall through to `direct_answer` when no LLM key
  is configured.
- H2: native StockSage failures caused by missing LLM credentials are too easy
  to observe only as `:failed`, without a useful reason in chat or ephemeral
  workspace state.
- H3: the default LiveView prompt value is content copy, not an input
  placeholder.

v0.26b fixes those without expanding the v0.26a UI substrate scope.

## Parallel Scope

v0.26b is intentionally small and backend-biased so it can land while v0.26a
M28-M34 are in flight.

Owned by v0.26b:

- Deterministic intent routing for generic safe Settings Central writes.
- StockSage native failure reason propagation through action response,
  confirmation metadata, objective progress, conversation history, and
  workspace fragment emission.
- Initial prompt state and neutral placeholder copy, with the smallest
  possible LiveView touch.

Not owned by v0.26b:

- Chat timeline rendering, composer keyboard behavior, sticky panes, mobile
  layout, tile inspector, AppBar menus, and other v0.26a M28-M34 work.
- New StockSage card bodies or rich analysis visualization.
- A new authorization model. All mutating behavior remains behind registered
  actions, `Actions.Runner.run/3`, Security Central, and existing settings
  policy.
- Automatic Python fallback from native StockSage failure. ADR 0022 still
  forbids Python as recovery unless the operator explicitly requests Python or
  parity.

Conflict note: H3 touches `AgentLive` and the chat component surface that M28
and M29 may also edit. If that work has already moved the prompt state, apply
H3 after M29 or as a tiny follow-up patch rather than reopening the composer
implementation.

## Target Behavior

### H1 - Setting-Shaped Prompt Routing

Today, the deterministic intent agent only recognizes a narrow setting write:

```text
set my communication style to ...
```

Prompts such as:

```text
Set workspace.theme to dark
Set operator.timezone to America/Los_Angeles
Set settings workspace.theme to dark
```

can fall through to `direct_answer` when model-assisted classification is
disabled or unavailable. The fallback should instead recognize safe dotted
Settings Central keys and route them to the existing registered
`update_setting` action.

Acceptance behavior:

- `Set workspace.theme to dark` selects `update_setting`, not
  `direct_answer`.
- The write still runs through `AllbertAssist.Actions.Runner.run/3`,
  `AllbertAssist.Actions.Settings.UpdateSetting`, `Settings.put/3`, and
  `PermissionGate.authorize(:settings_write, ...)`.
- Unknown, read-only, invalid, or unsafe keys return the existing structured
  settings error from `UpdateSetting`; they do not become natural-language
  direct answers.
- Secret-shaped prompts keep routing to `set_provider_credential` or an
  explicit refusal path. No raw secret should be written, traced, logged, or
  echoed by the generic parser.
- If an operator configures `permissions.settings_write` to require
  confirmation later, the route must preserve that action boundary. Do not
  special-case around Security Central inside the parser.

Implementation outline:

- Extend `AllbertAssist.Agents.IntentAgent.setting_write_route/1` to match
  generic dotted-key forms:
  - `set <key> to <value>`
  - `set setting <key> to <value>`
  - `change <key> to <value>`
- Recognize only keys already known to Settings Central and safe for writes.
  Prefer existing schema helpers such as `Settings.safe_write_keys/0` or
  `Settings.validate/2`; do not maintain a private allow-list in the agent.
- Trim wrapping quotes around values but otherwise let `Settings.put/3` and
  schema validation own type coercion and invalid-value errors.
- Keep provider credential routes before the generic setting parser so
  `openai api key` phrases never become generic setting writes.
- Add bounded trace metadata for the selected setting key, not the raw value
  when the key is secret-shaped or rejected.

Tests:

- `apps/allbert_assist/test/allbert_assist/runtime_intent_agent_test.exs`
  - Prompt `Set workspace.theme to dark` updates the setting and selects
    `update_setting`.
  - Prompt `Set workspace.mobile.breakpoint_px to 900` selects
    `update_setting` and returns the existing read-only setting denial.
  - Prompt `Set providers.openai.api_key_ref to sk-test` does not echo the
    value and does not persist a raw secret.
- `apps/allbert_assist/test/allbert_assist/agents/intent_agent_test.exs`
  - The deterministic agent route returns `update_setting` for safe dotted
    keys without model assistance.
- Optional CLI smoke:
  - `mix allbert.ask "Set workspace.theme to dark"`

### H2 - Visible Native StockSage Failure Reasons

When `stocksage.native_llm_enabled=true` and no usable LLM credential/provider
is configured, native StockSage can end up as a failed objective with too
little operator-visible explanation. `RunAnalysis.persist_failure/5` already
emits `allbert.stocksage.analysis_failed`, and
`Workspace.Emitters.stocksage_signal/2` already has an analysis-failed tile
path. The bug is that the most useful reason is not reliably preserved through
the native path and, on LiveView approval resume, no chat-visible assistant
message is appended when the async target later fails.

Acceptance behavior:

- Missing LLM credentials produce a bounded, specific reason such as
  `native_llm_unavailable: provider credential missing for openai`, not a
  generic `:failed` or quality-gate-only failure.
- The same bounded reason appears in:
  - the `RunAnalysis` response message and action metadata,
  - the failed `stocksage_analyses.summary` and metadata error,
  - `allbert.stocksage.analysis_failed` signal payload,
  - confirmation resolution `target_result`,
  - objective step `result_summary` and objective progress summary,
  - an assistant conversation message when a workspace-threaded async LiveView
    approval resume completes later,
  - a workspace failure fragment or ephemeral body when `thread_id` is present.
- When `stocksage.native_llm_enabled=false`, deterministic advisory packets
  still work for fixture/test/operator-disabled flows.
- Python is not invoked as fallback. Operators must explicitly request
  `engine: "python"` or `engine: "both"` for comparison/reference runs.

Implementation outline:

- Add a native LLM preflight in the StockSage native path before dispatching
  the multi-agent graph. Good candidate owners:
  - `StockSage.Agents.LLM.preflight/0`, or
  - a small helper called by `StockSage.Actions.RunAnalysis.run_native_after_approval/4`.
- The preflight should run only when `stocksage.native_llm_enabled=true`.
  It should report redacted credential/provider status using Settings Central,
  not by probing or printing raw secrets.
- Normalize provider/credential errors from `StockSage.Agents.LLM.JidoAI` into
  a stable failure reason. Preserve the raw provider error only in a bounded,
  redacted detail field.
- In `StockSage.Agents.Commands.Execute`, treat LLM generation errors as
  structured failed advisory packets with `generation_mode: "jido_ai_error"`
  and an explicit `failure_reason`.
- In `StockSage.Agents.NativeCoordinator.Commands.Analyze`, ensure failed
  LLM-backed specialists contribute their reason to warnings and top-level
  failure summaries. A quality-gate rejection may still happen, but it should
  not hide the root provider/credential reason.
- In `StockSage.Actions.RunAnalysis.persist_failure/5`, keep the bounded
  reason as the single canonical summary and pass it to
  `WorkspaceEmitters.stocksage_signal/2`.
- In `AllbertAssist.Actions.Confirmations.ApproveConfirmation`, after an async
  LiveView `run_analysis` resume task finishes, append a bounded assistant
  message to the originating conversation thread when `user_id` and
  `thread_id` are present. The message should mirror the final target
  response and carry an action log with `target_status`, `target_result`, and
  `confirmation_id`.
- Add or reuse a workspace emitter path for failed native analysis that creates
  an operator-visible failure fragment. Prefer existing catalog atoms
  (`:analysis_card`, `:ephemeral_surface`, `:status_badge`) over adding a new
  catalog component.

Tests:

- `plugins/stocksage/test/stocksage/actions/run_analysis_test.exs`
  - With LLM enabled and a fake provider/credential-missing response,
    approved native analysis returns `status: :failed` and a message
    containing the redacted reason.
  - Failed analysis row has `status: "failed"` and the same bounded summary.
  - `allbert.stocksage.analysis_failed` payload includes `thread_id`,
    `objective_id`, and the bounded reason.
- `plugins/stocksage/test/stocksage/agents/native_coordinator/*`
  - LLM failure packets preserve `failure_reason` and do not become silent
    success reports.
  - `stocksage.native_llm_enabled=false` still completes deterministically.
- `apps/allbert_assist/test/allbert_assist/actions/confirmations/*`
  - Async LiveView approval resume appends one assistant message to the
    originating conversation when the target later fails.
  - Confirmation resolution metadata includes `target_status: :failed` and a
    bounded `target_result.error`.
- `apps/allbert_assist/test/allbert_assist/workspace/emitters_test.exs`
  - Analysis-failed payload emits a signed fragment whose visible body includes
    the bounded failure reason.

### H3 - Prompt Placeholder Is Not Prompt Content

The `/agent` LiveView currently seeds the composer with:

```text
Hello Allbert. What can you do right now?
```

That is useful example copy, but it should not be actual textarea content.
The composer should start empty. Any example or hint belongs in an empty-state
card or neutral placeholder.

Acceptance behavior:

- Initial `/agent` render has an empty `prompt` assign.
- The textarea value is empty on first mount.
- The textarea placeholder is neutral, for example:

```text
Ask Allbert anything...
```

- The example prompt may appear in an empty-state hint, but it must not be
  submitted accidentally as the first turn.
- This change should not implement M29 composer reset or Enter-to-submit.
  Those remain v0.26a work.

Implementation outline:

- Change the initial `prompt:` assign in
  `apps/allbert_assist_web/lib/allbert_assist_web/live/agent_live.ex` from the
  example sentence to `""`.
- Keep or update the textarea `placeholder` in
  `apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/chat.ex`
  to neutral copy.
- If the example sentence is retained, move it into the existing empty-state
  copy, not the form value.
- Coordinate with the v0.26a M28/M29 agent before editing the chat component;
  if their composer work already changed this area, assert the acceptance
  behavior and avoid a conflicting rewrite.

Tests:

- `apps/allbert_assist_web/test/allbert_assist_web/live/agent_live_test.exs`
  - Initial render has an empty `#agent-prompt` value.
  - Placeholder text is neutral.
  - The example sentence is absent from the textarea content.
- Existing accessibility tests should continue to pass.

## Milestone Structure

v0.26b uses backend-specific milestone labels to avoid conflicting with
v0.26a M28-M35.

### B1 - Deterministic Settings Write Fallback

Goal: setting-shaped prompts route to `update_setting` without requiring LLM
classification.

Depends on: nothing.

Parallel-safe with: v0.26a M28, M30, M31, M33, M34. Low conflict with M29 if
tests use `/agent`, but code changes are in the core intent agent.

Deliverables:

- Generic dotted-key settings parser.
- Safe-write validation using Settings Central.
- Secret-shaped prompt guard.
- Runtime and intent-agent tests.

Exit signal:

- `Set workspace.theme to dark` changes the setting through the registered
  action path and no longer returns a direct answer.

### B2 - Native StockSage Failure Reason Propagation

Goal: no native StockSage LLM-provider failure is visible only as a bare
`:failed` state.

Depends on: nothing.

Parallel-safe with: v0.26a M28-M34. It touches StockSage native actions,
confirmation resume metadata, conversation persistence, and workspace emitters,
not renderer layout.

Deliverables:

- Native LLM preflight or normalized LLM failure reason.
- Failed advisory packet metadata.
- Failed analysis signal/fragment with bounded reason.
- Async LiveView approval resume conversation message.
- StockSage, confirmation, conversation, and emitter tests.

Exit signal:

- An approved native analysis with missing LLM credentials leaves the operator
  with a chat-visible and workspace-visible reason, plus matching analysis,
  confirmation, objective, and signal metadata.

### B3 - Empty Initial Composer Value

Goal: example prompt copy is not pre-filled into the composer.

Depends on: coordination with v0.26a M29 if it is already editing the composer.

Parallel-safe with: B1 and B2. Potential file conflict with v0.26a M28/M29.

Deliverables:

- Empty initial prompt assign.
- Neutral placeholder copy.
- LiveView test for initial textarea value.

Exit signal:

- Fresh `/agent` opens with an empty composer and no accidental default prompt.

### B4 - Closeout

Goal: prove the three backend fixes together and hand off a warning-free tree.

Depends on: B1, B2, B3.

Deliverables:

- Focused tests for changed modules.
- Update this plan's status notes if implementation reveals a different
  module owner.
- No v0.26a UI milestone acceptance criteria are marked complete here.
