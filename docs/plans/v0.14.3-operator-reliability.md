# v0.14.3 - Operator reliability patch

Status: Draft

## Summary

v0.14.3 is a patch release after v0.14.2 and before v0.15, driven by live operator testing. Its first release-blocking milestone is deterministic conversational scheduling for local models, especially Gemma4/Gemma4:26B.

The motivating transcript is:

```text
allbert> schedule a daily review at 07:00
...
I can set that up for you. I'll create a recurring job named `daily_review` that runs `@daily at 07:00`.

Shall I proceed with scheduling this?
allbert> yes
...
I could not parse the model's tool call safely: unsupported tool call shape: name input/arguments is missing.
```

That flow is wrong for Allbert. Mutating schedule requests should route through the daemon-backed job tool first, and the existing structured durable-change confirmation prompt should be the approval surface.

Additional milestones may be appended as more operator-test findings arrive.

## Goals

- Make mutating conversational scheduling use `upsert_job`, `pause_job`, `resume_job`, or `remove_job` before asking for approval.
- Preserve the structured durable-change confirmation prompt as the only approval surface for job mutations.
- Make local-model tool-call recovery deterministic enough that a plain prose "Shall I proceed?" response does not strand the operator.
- Safely accept the observed flat named local-model tool-call shape without weakening authorization.
- Record enough trace/debug provenance to diagnose malformed tool calls without leaking raw malformed payloads to normal output.

## Non-goals

- No protocol bump.
- No storage migration.
- No new operator command.
- No provider-native tool-calling migration.
- No growth-loop ingestion work.
- No bypass of job confirmation, active tool catalog, skill `allowed-tools`, or exec policy.

## M1 - Deterministic Conversational Scheduling

**Problem.** A user request such as `schedule a daily review at 07:00` is a durable, mutating scheduler request. The model should not ask for plain-text approval and then attempt an ad hoc tool call on the next turn. It should emit the daemon-backed job mutation tool call first so Allbert can render the structured approval preview and persist the job only after operator confirmation.

**Schedule-intent contract.**

- If the resolved intent is `schedule` and the user asks to create, update, pause, resume, or remove a recurring job, the model must emit one of `upsert_job`, `pause_job`, `resume_job`, or `remove_job` first.
- The model must not ask "Shall I proceed?" or similar prose confirmation before calling the job tool.
- The existing durable-change confirmation prompt remains authoritative. For `upsert_job`, approval must show the structured job preview before persistence.
- Read-only scheduling questions such as `what jobs do I have?` must continue to use read-only job tools or ordinary status responses.

**Prompt and intent changes.**

- Tighten the schedule intent guidance so recurring-job mutations say "call the job mutation tool first; Allbert handles approval" rather than encouraging generic confirmation language.
- The job-tool schema in the prompt must continue to include the required `upsert_job` fields: `name`, `description`, `schedule`, and `prompt`.
- The implementation should normalize the natural request `schedule a daily review at 07:00` into a stable job name such as `daily-review`, schedule `@daily at 07:00`, and a bounded prompt describing the daily review.

**Schedule-specific retry guard.**

- When the resolved intent is `schedule`, the original user request is clearly mutating, and the model responds with prose confirmation but no tool call, Allbert retries once.
- The retry message must explicitly require the correct job mutation tool call and state that plain prose confirmation is not accepted for durable schedule changes.
- If the retry still fails or emits malformed tool JSON, Allbert returns a safe operator message naming the CLI fallback:

```bash
allbert-cli jobs upsert <job-definition.md>
```

- The safe failure message should also point the operator to trace inspection for the malformed provider response.
- The retry remains bounded by the normal daily cost cap and provider cost logging.

**Flat named-call normalization.**

The parser should safely accept flat named tool-call objects emitted by local models:

```json
{"name":"upsert_job","description":"Daily review","schedule":"@daily at 07:00","prompt":"Run a concise daily review."}
```

This normalizes to:

```json
{"name":"upsert_job","input":{"description":"Daily review","schedule":"@daily at 07:00","prompt":"Run a concise daily review."}}
```

Rules:

- Apply this only to objects with a string `name`, no `input` or `arguments`, and at least one non-`name` field.
- Preserve every non-`name` field verbatim inside `input`.
- Reject empty flat calls such as `{"name":"upsert_job"}`.
- Reject unknown tools during resolve.
- Keep existing authorization unchanged: active tool catalog, active skill `allowed-tools`, confirmation hooks, and exec policy still apply after normalization.

**Trace/debug provenance.**

- Record bounded, redacted tool-parse failure context in trace when a tool call is malformed.
- Include the parse error, whether flat normalization was attempted, whether schedule retry was attempted, and which retry path was taken.
- Do not print raw malformed provider payloads to ordinary user output unless already permitted by trace capture and redaction settings.

**Acceptance.**

- In `repl --classic`, prompting `schedule a daily review at 07:00` shows the structured durable scheduling confirmation in the same conversational flow.
- After approving with `y`, `jobs status daily-review` shows the job.
- The failure mode "plain prose Shall I proceed? -> yes -> unsupported tool call shape" is not accepted as v0.14.3-ready behavior.

## Future Milestones

M2 and later are intentionally reserved for additional operator-test findings. Every new milestone added to this patch plan should include:

- the observed operator transcript or command output;
- the current safe workaround;
- the planned product fix;
- tests that prove the regression cannot return.

## Public Interfaces

- No new command is added.
- `allbert-cli jobs upsert <job-definition.md>` remains the CLI fallback for schedule creation.
- No protocol version changes.
- No storage migration.
- ADR 0096 is amended to document flat named-call normalization and schedule-specific no-tool retry behavior.
- The operator feature-test runbook gains a v0.14.3 conversational scheduling smoke.

## Test Plan

### Unit Tests

- Parser accepts canonical, alias, nested function, direct-spawn, and flat named `upsert_job` shapes.
- Parser rejects flat named calls with no input fields.
- Parser preserves policy checks after normalization.

### Integration Tests With Scripted Provider

- First response is prose `Shall I proceed?`; retry emits `upsert_job`; REPL receives structured confirmation.
- First response is flat named `upsert_job`; parser normalizes it and confirmation appears.
- Retry remains malformed; user sees safe remediation and trace records parse provenance.
- Non-mutating schedule questions like `what jobs do I have?` still use read-only job tools.
- Non-scheduling chat that mentions `daily` does not trigger schedule mutation retry.

### Live Smoke

With local Gemma4/Gemma4:26B:

```text
schedule a daily review at 07:00
```

Expected: Allbert should not leave the operator at a plain `yes` prompt followed by parser failure. It should show the structured durable scheduling confirmation. After approval:

```bash
allbert-cli jobs status daily-review
```

Expected: the job exists and shows the intended daily schedule.
