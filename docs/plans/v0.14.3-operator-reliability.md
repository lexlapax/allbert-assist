# v0.14.3 - Operator reliability patch

Status: Shipped

## Summary

v0.14.3 is a patch release after v0.14.2 and before v0.15, driven by live operator testing. Its foundation milestone is a schema-bound LLM intent router that runs before full prompt assembly. The router replaces default semantic keyword catching and gives Allbert a safe place to draft schedule and explicit-memory actions before the main assistant turn. The release-blocking operator repairs built on that foundation are conversational scheduling for local models, explicit memory capture for `remember that ...`-style requests, OpenAI Responses assistant-history serialization, and Gemini live-provider response compatibility.

The motivating transcript is:

```text
allbert> schedule a daily review at 07:00
...
I can set that up for you. I'll create a recurring job named `daily_review` that runs `@daily at 07:00`.

Shall I proceed with scheduling this?
allbert> yes
...
I could not parse the model's tool call safely: unsupported tool call shape: name input/arguments is missing.
```

That flow is wrong for Allbert. Mutating schedule requests should route through the daemon-backed job tool first, and the existing structured durable-change confirmation prompt should be the approval surface.

The second motivating transcript is:

```text
allbert> remember that Allbert operator tests use temporary ALLBERT_HOME profiles
...
I could not parse the model's tool call safely: unsupported tool call shape: expected name, tool, function, or program.
allbert> review what's staged
...
There are currently no staged memory entries to review.
```

That flow is also wrong. An explicit operator request to remember a fact should create a staged-memory candidate through the review-first memory pipeline. It should not depend on a local model inventing a valid tool-call shape before the operator can test staged-memory review.

The third motivating transcript is:

```text
allbert> /model openai gpt-5.5 OPENAI_API_KEY
active model: openai / gpt-5.5
allbert> hello
...
llm error: unexpected provider response: openai status 400 Bad Request:
Invalid value: 'input_text'. Supported values are: 'output_text' and 'refusal'.
param: input[1].content[0]
```

That flow exposes a provider-maintenance bug. OpenAI Responses accepts user text as `input_text`, but assistant-history text must be serialized as `output_text`. A multi-turn OpenAI session must not fail as soon as prior assistant history exists.

The release closed with M0 through M4. Additional operator-test findings now belong in a follow-up patch plan unless they are regressions in the shipped v0.14.3 surface.

## Research Direction

- Use a router, not a visible sub-agent. LangChain's router pattern treats routing as lightweight preprocessing, while sub-agents are broader orchestration participants with conversation context: [LangChain router docs](https://docs.langchain.com/oss/javascript/langchain/multi-agent/router).
- Use structured outputs or tool schemas where providers support them. OpenAI recommends Structured Outputs for schema adherence: [OpenAI Structured Outputs](https://platform.openai.com/docs/guides/structured-outputs). Ollama supports JSON-schema `format`, which is useful for local Gemma-style routing: [Ollama Structured Outputs](https://docs.ollama.com/capabilities/structured-outputs).
- Keep execution in Allbert. Anthropic's tool-use guidance still leaves tool execution and policy with client code: [Anthropic tool use](https://docs.anthropic.com/en/docs/agents-and-tools/tool-use/implement-tool-use).

## Goals

- Replace default semantic keyword routing with a schema-bound router call before full prompt assembly.
- Let the router emit guarded schedule and explicit-memory action drafts while preserving Allbert-owned execution, confirmation, and staging.
- Make mutating conversational scheduling use `upsert_job`, `pause_job`, `resume_job`, or `remove_job` before asking for approval.
- Preserve the structured durable-change confirmation prompt as the only approval surface for job mutations.
- Make local-model tool-call recovery deterministic enough that a plain prose "Shall I proceed?" response does not strand the operator.
- Safely accept the observed flat named local-model tool-call shape without weakening authorization.
- Make explicit `remember that ...` memory capture deterministic and review-first.
- Keep hosted-provider OpenAI turns reliable by serializing Responses API conversation history with role-correct content types.
- Record enough trace/debug provenance to diagnose malformed tool calls without leaking raw malformed payloads to normal output.

## Non-goals

- No protocol bump.
- No storage migration.
- No new operator command.
- No provider-native tool-calling migration.
- No growth-loop ingestion work.
- No user-visible router sub-agent.
- No bypass of job confirmation, active tool catalog, skill `allowed-tools`, or exec policy.

## M0 - Schema-Bound Intent Router

**Problem.** The current classifier has semantic keyword fast paths. That makes cases such as scheduling and memory capture feel deterministic, but the determinism is brittle: keywords such as `daily`, `remember`, or `pause` are not intent. v0.14.3 should not add more hardcoded phrase grammar to fix local-model reliability.

**Decision.** Default v0.14.3 routing uses a bounded, schema-validated LLM router before full prompt assembly. The router is an internal kernel step, not a user-visible sub-agent.

**Router position.**

- Runs after turn cost-cap checks and `BeforeIntent`.
- Runs before memory retrieval, skill-body loading, and full system prompt assembly.
- Is attributed as `intent-router` in trace and cost logs.
- Receives only bounded routing context: raw user message, source/channel, current time/timezone, last resolved intent, pending confirmation state, and a bounded job-name index.
- Does not receive bootstrap files, full memory, skill bodies, or prior tool results.

**Default routing policy.**

- Default path is router-first when `intent_classifier.enabled = true`.
- Exact mechanical handling may remain for empty input, REPL slash commands intercepted before the kernel turn, and `intent_classifier.enabled = false`.
- Existing `intent_classifier.rule_only = true` keeps the legacy rule classifier for operators who explicitly choose it.
- Default v0.14.3 behavior must not use semantic keyword lists as routing authority for schedule or memory mutations.

**Config and migration.**

- v0.14.3 reuses the existing `[intent_classifier]` table. It does not add a
  new public config table for the router.
- `intent_classifier.enabled = false` keeps the existing disabled-classifier
  behavior and disables router-generated action drafts.
- `intent_classifier.rule_only = true` uses the legacy rule classifier and does
  not call the LLM router. Semantic keyword rules are allowed only in this
  explicit compatibility mode.
- `intent_classifier.model` selects the router model. An empty value uses the
  turn's effective model.
- `intent_classifier.per_turn_token_budget` bounds router prompt/context
  construction. Router completion output must remain small and schema-shaped;
  implementation should cap router response tokens independently of the main
  turn's `model.max_tokens`.
- Existing profiles need no config rewrite. The shipped defaults
  `enabled = true` and `rule_only = false` become router-first after v0.14.3.

**RouteDecision contract.**

The internal router returns one JSON object with `additionalProperties: false`. Required fields use `null` for absent details so strict-schema providers can validate the same shape:

- `intent`: `task | chat | schedule | memory_query | meta`
- `action`: `none | schedule_upsert | schedule_pause | schedule_resume | schedule_remove | memory_stage_explicit`
- `confidence`: `low | medium | high`
- `needs_clarification`: boolean
- `clarifying_question`: string or null
- `job_name`, `job_description`, `job_schedule`, `job_prompt`: strings or null
- `memory_summary`, `memory_content`: strings or null
- `reason`: bounded string for trace/debug provenance

All fields are required at the JSON-schema level. Absent details are represented
as `null`, not omitted. String fields are trimmed before validation; empty
strings count as absent. `reason` is for trace/debug provenance only and should
be bounded to 512 characters.

Action field requirements:

- `none`: no action fields required. If `needs_clarification = true`,
  `clarifying_question` is required.
- `schedule_upsert`: requires `job_name`, `job_description`, `job_schedule`, and
  `job_prompt`.
- `schedule_pause`, `schedule_resume`, `schedule_remove`: require `job_name`.
- `memory_stage_explicit`: requires `memory_summary` and `memory_content`.

Allbert may turn a router action into a synthetic runtime action only when
schema validation passes, `confidence = high`, `needs_clarification = false`,
and all action-required fields are present.

**Guarded action authority.**

- Router drafts are internal runtime actions, not provider-authored arbitrary
  tool calls. They are limited to the fixed action allowlist above.
- `schedule_upsert` converts to the existing `upsert_job` path with input
  `{name, description, schedule, prompt}` and optional fields omitted.
- `schedule_pause`, `schedule_resume`, and `schedule_remove` convert to the
  existing named job mutation paths with input `{name}`.
- `memory_stage_explicit` converts into the existing `stage_memory` path with
  `kind = explicit_request`; it never promotes durable memory.
- Router drafts cannot invoke `process_exec`, adapters, self-improvement, remediation, local utilities, or arbitrary tools in v0.14.3.
- Low/medium confidence or missing fields produces either a concise clarification question or normal full-turn routing with no mutation.
- Because router drafts run before skill activation, they are not governed by
  active skill `allowed-tools`. Their safety envelope is the hardcoded action
  allowlist plus the existing job preview/confirmation and memory staging
  review surfaces. Provider-emitted tool calls later in the turn remain subject
  to active catalog, active skill `allowed-tools`, confirmation hooks, and exec
  policy.

**Synthetic action lifecycle.**

- Router action conversion runs after `BeforeIntent` and before memory retrieval,
  skill-body loading, and full system prompt assembly.
- The router decision is recorded in the intent trace span with
  `agent = "intent-router"` and bounded decision metadata.
- A converted router action emits the same operator-facing tool/confirmation
  events as the corresponding job or memory tool path where those events already
  exist.
- Successful router-handled `schedule_*` and `memory_stage_explicit` actions are
  terminal for that user turn: Allbert returns a deterministic status/notice and
  does not call the full assistant model afterward.
- Failed router action conversion fails closed. It may ask the router's
  clarifying question when supplied; otherwise the turn falls through to the
  normal assistant path with no synthetic mutation.
- Assistant history must not persist a fake model-authored message for the
  router decision. Persist only the final deterministic operator-facing notice
  or the normal assistant output if the turn falls through.

**Provider seam.**

- Add an internal structured JSON-schema response option to `CompletionRequest`:

```rust
pub enum CompletionResponseFormat {
    Text,
    JsonSchema {
        name: String,
        schema: serde_json::Value,
        strict: bool,
    },
}

pub struct CompletionRequest {
    // existing fields...
    pub response_format: CompletionResponseFormat,
    pub temperature: Option<f32>,
}
```

- Existing normal chat/model turns use `CompletionResponseFormat::Text` and
  preserve current behavior.
- Router calls use `JsonSchema { name: "route_decision", strict: true, ... }`
  and `temperature = Some(0.0)` where the provider supports it.
- OpenAI uses Responses `text.format` JSON schema.
- Ollama uses chat `format` with the router schema and local validation of returned JSON.
- Anthropic, OpenRouter, Gemini, and any provider without native structured
  response support use strict prompt JSON plus local validation and one
  corrective retry until native structured support is added.
- Router failure or invalid JSON fails closed: no synthetic action draft and no durable mutation.

**Acceptance.**

- `schedule a daily review at 07:00` can reach the job preview through a router `schedule_upsert` draft before the full assistant prompt is assembled.
- `remember that Allbert operator tests use temporary ALLBERT_HOME profiles` can stage one `explicit_request` through a router `memory_stage_explicit` draft.
- `I remember when daily notes were chaotic` does not schedule or stage memory.
- `what jobs do I have?` routes read-only and never opens a mutation confirmation.
- Malformed router JSON retries once, records bounded trace provenance, and performs no mutation.
- With default config, semantic keyword rules are not consulted for routing.
  With `intent_classifier.rule_only = true`, the legacy rule classifier still
  behaves as before and no router action drafts are emitted.

## M1 - Deterministic Conversational Scheduling

**Problem.** A user request such as `schedule a daily review at 07:00` is a durable, mutating scheduler request. The main assistant should not ask for plain-text approval and then attempt an ad hoc tool call on the next turn. The router should draft the schedule action first when confident; otherwise the main assistant must emit the daemon-backed job mutation tool call first so Allbert can render the structured approval preview and persist the job only after operator confirmation.

**Schedule-intent contract.**

- If the router resolves a high-confidence schedule mutation, it should emit a `schedule_*` action draft that Allbert converts into one of `upsert_job`, `pause_job`, `resume_job`, or `remove_job`.
- If the router falls through to the full assistant turn for a schedule mutation, the model must emit the job mutation tool before prose confirmation.
- The model must not ask "Shall I proceed?" or similar prose confirmation before calling the job tool.
- The existing durable-change confirmation prompt remains authoritative. For `upsert_job`, approval must show the structured job preview before persistence.
- Read-only scheduling questions such as `what jobs do I have?` must continue to use read-only job tools or ordinary status responses.

**Prompt and router changes.**

- Add route-schema examples that distinguish schedule mutation, read-only job inspection, and ordinary conversation mentioning time or daily routines.
- Tighten fallback schedule guidance so recurring-job mutations say "call the job mutation tool first; Allbert handles approval" rather than encouraging generic confirmation language.
- The job-tool schema in the prompt must continue to include the required `upsert_job` fields: `name`, `description`, `schedule`, and `prompt`.
- The router should normalize the natural request `schedule a daily review at 07:00` into a stable job name such as `daily-review`, schedule `@daily at 07:00`, and a bounded prompt describing the daily review.

**Schedule-specific retry guard.**

- Schedule-specific retry eligibility is based on the validated router decision,
  not a lexical keyword scan. It applies only when the current turn has a valid
  router decision with `intent = schedule` and an action in
  `schedule_upsert | schedule_pause | schedule_resume | schedule_remove`, but
  the turn fell through to the full assistant path instead of executing a
  high-confidence router draft.
- In explicit legacy mode (`intent_classifier.rule_only = true`), the existing
  rule classifier may be used as the retry eligibility source for compatibility.
- When retry is eligible and the model responds with prose confirmation but no
  tool call, Allbert retries once.
- The retry message must explicitly require the correct job mutation tool call and state that plain prose confirmation is not accepted for durable schedule changes.
- The generic malformed-tool retry and the schedule-specific prose-confirmation retry share one retry budget per turn. A turn must not perform one generic retry and then another schedule retry for the same provider-response failure.
- If the retry still fails or emits malformed tool JSON, Allbert returns a safe operator message naming the CLI fallback:

```bash
allbert-cli jobs upsert <job-definition.md>
```

- The safe failure message should also point the operator to trace inspection for the malformed provider response.
- The retry remains bounded by the normal daily cost cap and provider cost logging.

**Flat named-call normalization.**

The parser should safely accept flat named tool-call objects emitted by local models:

```json
{"name":"upsert_job","description":"Daily review","schedule":"@daily at 07:00","prompt":"Run a concise daily review."}
```

This normalizes to:

```json
{"name":"upsert_job","input":{"description":"Daily review","schedule":"@daily at 07:00","prompt":"Run a concise daily review."}}
```

Rules:

- Apply this only to objects with a string `name`, no `input` or `arguments`, and at least one non-`name` field.
- Preserve every non-`name` field verbatim inside `input`.
- Reject empty flat calls such as `{"name":"upsert_job"}`.
- Reject unknown tools during resolve.
- Keep existing authorization unchanged: active tool catalog, active skill `allowed-tools`, confirmation hooks, and exec policy still apply after normalization.

**Trace/debug provenance.**

- Record bounded, redacted tool-parse failure context in trace when a tool call is malformed.
- Include the parse error, whether flat normalization was attempted, whether schedule retry was attempted, and which retry path was taken.
- Do not print raw malformed provider payloads to ordinary user output unless already permitted by trace capture and redaction settings.

**Acceptance.**

- In `repl --classic`, prompting `schedule a daily review at 07:00` shows the structured durable scheduling confirmation in the same conversational flow.
- After approving with `y`, `jobs status daily-review` shows the job.
- The failure mode "plain prose Shall I proceed? -> yes -> unsupported tool call shape" is not accepted as v0.14.3-ready behavior.

## M2 - Deterministic Explicit Memory Capture

**Problem.** The v0.5 curated-memory operator smoke asks:

```text
remember that Allbert operator tests use temporary ALLBERT_HOME profiles
review what's staged
```

With local Gemma4/Gemma4:26B, the first turn can fail with `unsupported tool call shape: expected name, tool, function, or program`. The second turn correctly calls `list_staged_memory`, but there is nothing to review because the explicit memory request never staged a candidate.

**Pre-v0.14.3 workaround.** There was no provider-free public CLI command that created a staged-memory candidate. Operators could inspect, show, promote, reject, verify, and search staged memory through the CLI, but initial staging depended on a live conversational model or internal tests.

**Product fix.**

- Add a router `memory_stage_explicit` action for high-confidence user requests to remember a specific fact.
- `memory_content` must be non-empty and is staged as the candidate body. `memory_summary` is required, deterministic from the router decision, and bounded to 240 characters.
- The router action stages a memory candidate through the existing `stage_memory` pipeline with kind `explicit_request`.
- It must not promote directly to durable memory. Review-first behavior stays intact.
- It must not require the full assistant prompt or a model-generated `stage_memory` tool call after the router decision.
- It should render the same staged-memory notice already used after successful `stage_memory` tool calls.
- It should leave ambiguous memory extraction, summarization, and research-style memory proposals on the model/tool path.
- It must not trigger for stories, quoted examples, or questions such as `do you remember...`; ambiguous memory extraction stays model/tool-driven.
- If a model still emits malformed memory tool JSON for non-deterministic cases, trace provenance should record the bounded, redacted parse failure using the same v0.14.3 tool-parse provenance rules from M1.

**Runbook impact.**

- The v0.5 runbook now treats `remember that Allbert operator tests use temporary ALLBERT_HOME profiles` as a deterministic v0.14.3 smoke that creates one staged entry before `review what's staged`.

**Tests.**

- Scripted-router integration test: `remember that Allbert operator tests use temporary ALLBERT_HOME profiles` creates exactly one staged-memory entry.
- The staged entry is kind `explicit_request`, has a non-empty summary, and does not appear in durable/fact search until promotion.
- `review what's staged` lists the new candidate.
- `memory promote <id> --confirm` promotes it and `memory search "temporary ALLBERT_HOME"` finds it.
- `memory reject <id> --reason ...` removes a separate staged candidate from active staging.
- Non-memory chat that merely uses the word `remember` in a story does not trigger deterministic staging.
- Ambiguous requests still route through the normal model/tool path.

## M3 - OpenAI Responses Assistant-History Serialization

**Problem.** OpenAI Responses rejects assistant-history entries encoded as `input_text`. User text remains `input_text`, but assistant message text must be sent as `output_text`. The current operator failure appears after selecting an OpenAI model and sending a turn once prior assistant history exists.

**Pre-v0.14.3 workaround.** For v0.14.2 testing, use a fresh session with no assistant history, a non-OpenAI provider, or restart after applying the provider fix. That was not release-ready behavior for v0.14.3 because ordinary multi-turn OpenAI REPL use must work.

**Product fix.**

- In the OpenAI Responses request mapper, user text content serializes as `input_text`.
- Assistant-history text serializes as `output_text`.
- User image attachments remain `input_image`.
- Assistant image attachments are rejected locally with a clear provider error because Responses input history does not accept assistant-side image input content.
- Preserve the existing OpenAI request posture: `store: false`, current instructions handling, existing model/usage parsing, and existing cost-ledger path.
- Add provider-free mock HTTP tests that capture the serialized request body for a user/assistant/user history sequence and prove the assistant entry uses `output_text`.

**Acceptance.**

- In `repl --classic`, `/model openai <model> OPENAI_API_KEY` followed by a two-turn conversation succeeds after assistant history exists.
- No OpenAI 400 appears with `Invalid value: 'input_text'` at `input[1].content[0]`.
- Usage/cost parsing still records the OpenAI response through the same provider ledger path as before.

## M4 - Gemini Live-Provider Response Compatibility

**Problem.** v0.14.3 credentialed release checks found that the live Gemini
provider path could fail before the first operator-visible answer with:

```text
profile prompt should succeed: Llm(Response("error decoding response body"))
```

Direct Gemini API probes with the same credential succeeded, so this was not an
API-key or network failure. The issue was Allbert's response decoder: it used
the same strict `GeminiPart` enum for request construction and response parsing.
Live Gemini responses may include non-text response parts such as function-call,
thought, or signature-only metadata alongside normal text. Those parts should
not make the whole response undecodable.

The same live check also showed that the existing ignored Gemini smoke used a
64-token output cap. Gemini 2.5 Flash may spend output budget on hidden thinking,
so the exact-response smoke could truncate `PROFILE_OK` to `PROFILE` even after
the provider decoded correctly. That is a smoke harness issue; Allbert's shipped
default `model.max_tokens` remains 4096.

**Product fix.**

- Split Gemini request parts from response parts.
- Parse Gemini response parts flexibly, extracting text from text-bearing parts
  and ignoring unknown non-text response parts.
- Keep request serialization unchanged for user text, assistant history, and
  image inline data.
- Make Gemini response decode errors provider-specific so future failures say
  `gemini response decode error: ...` instead of a generic body decode error.
- Raise the ignored Gemini release smoke output cap to 512 tokens while keeping
  other hosted-provider smoke caps unchanged.
- Use `gemini-2.5-flash-lite` for the credentialed Gemini release smoke so the
  live check verifies Allbert's Gemini provider path without depending on the
  more frequently saturated Flash endpoint.

**Acceptance.**

- Gemini provider mock tests accept a response with unknown non-text parts before
  a text part and still return the text.
- The credentialed Gemini live smoke no longer fails with generic response-body
  decoding.
- If Gemini returns quota or availability errors, Allbert surfaces the provider
  status and body as an external live-provider failure rather than a local
  decoder failure.

## Test Plan

### Unit Tests

- Route schema validates good decisions and rejects unknown actions, extra fields, missing required fields, malformed JSON, and action/details mismatches.
- Route schema validation enforces the action field matrix for `schedule_*` and
  `memory_stage_explicit`.
- Router config tests cover `enabled = false`, `rule_only = true`, default
  router-first behavior, model selection, and token-budget bounding.
- Legacy `rule_only = true` keeps old behavior; default config does not use semantic keyword rules.
- Router action conversion builds only allowed synthetic runtime actions.
- Router-handled schedule and explicit-memory actions are terminal for the turn
  and do not call the full assistant prompt after successful conversion.
- OpenAI and Ollama provider tests assert the structured-router request shape.
- Providers without native structured output support receive strict prompt JSON
  and perform one validation retry before failing closed.
- Parser accepts canonical, alias, nested function, direct-spawn, and flat named `upsert_job` shapes.
- Parser rejects flat named calls with no input fields.
- Parser preserves policy checks after normalization.
- OpenAI provider serialization tests prove role-specific Responses content mapping without a live provider.
- Gemini provider mock tests prove live response parts with unknown non-text
  shapes do not break text extraction.

### Integration Tests With Scripted Provider

- `schedule a daily review at 07:00` returns `schedule_upsert`, shows job preview, and persists only after approval.
- The schedule path does not require the full assistant prompt before the approval surface.
- First response is prose `Shall I proceed?`; retry emits `upsert_job`; REPL receives structured confirmation.
- Schedule-specific prose retry is triggered from router decision metadata, not
  from matching words such as `daily`, `remind`, or `pause`.
