## Main Operator Flows

- **Stale-CSS class eliminated (R15).** Operator installs/upgrades to a packaged
  1.0.1, opens the web workspace: the stylesheet link is `/assets/css/app-<hash>.css`.
  A later same-version rebuild with changed CSS produces a new hash → the browser
  fetches fresh CSS with no hard reload. The validation-runbook "hard reload after
  same-version rebuilds" caveat from v1.0 becomes obsolete for CSS.
- **Onboarding buttons on design-system tokens.** Operator walks the onboarding
  surface (fresh disposable Home): every button renders through
  `workspace-button[-compact][-variant]` classes; the persona-select button still
  shows selected/unselected emphasis. No visual regression is the contract; the
  styling authority is `Patterns`, not raw daisy classes.
- **Service-worker cache lockstep.** On first load of 1.0.1 the SW's activate-time
  purge fires (cache name changed from the stranded `v0.62.1`), dropping superseded
  caches. Future releases fail the release gate if the SW name is not bumped with the
  version.
- **Evidence matrix complete.** `docs/validation/v1.0/README.md` DIT-5 row resolves to
  a real transcript; the `release.v101` gate mechanically refuses to pass without it.
- **Release consumption.** Operator upgrades via curl installer (cosign-verified) or
  `brew upgrade allbert`; both resolve to 1.0.1; `allbert --version` and the web
  footer/serverInfo agree.

## Data Flow

- Browser → `Plug.Static` (`endpoint.ex:23-28`): root layout renders
  `~p"/assets/css/app.css"` → prod `static_lookup` via
  `priv/static/cache_manifest.json` → `/assets/css/app-<hash>.css` served with
  far-future caching semantics. Dev/test (no manifest): bare path, unchanged behavior.
- Page JS (`app.js:983-1042`) reads the rendered (digested) asset URLs from the DOM and
  posts them to `workspace-sw.js` (`ALLBERT_WORKSPACE_CACHE_ASSETS`); the SW caches
  cache-first keyed by URL — content-hashed URLs make cache-first correct by
  construction.
- Version lockstep: three `mix.exs` files + `workspace-sw.js` `CACHE_NAME` →
  asserted by `version_consistency_test.exs` (compile-time version) → enforced by the
  `v1_version_consistency` step inside both `release.v1` and `release.v101`.

## Operator Validation

Run from the checkout root unless stated. No `mix ... | tail` pipes — they mask exit
codes.

### §A — Focused gates (checkout)

```sh
mix test \
  apps/allbert_assist_web/test/allbert_assist_web/controllers/theme_controller_test.exs \
  apps/allbert_assist_web/test/allbert_assist_web/workspace/design_system_tokens_test.exs \
  apps/allbert_assist_web/test/allbert_assist_web/workspace/offline_test.exs \
  apps/allbert_assist_web/test/allbert_assist_web/version_consistency_test.exs
```

Expected behavior: all four files run, **0 failures**. PASS: exit code 0 and the
failure count line reads `0 failures`.

```sh
grep -rnE 'class=\{?\s*\[?\s*"[^"]*\bbtn\b' apps/allbert_assist_web/lib && echo "FAIL: raw btn found" || echo "PASS: no raw btn"
```

Expected behavior: prints `PASS: no raw btn`.

### §B — R15 packaged proof (disposable Home, packaged binary)

Use a built 1.0.1 artifact (or `mix release allbert` output) — never a real
`~/.allbert`:

```sh
WORK="$(mktemp -d)"
ALLBERT_HOME="$WORK/home" PHX_SERVER=1 PORT=4137 ./allbert serve &
sleep 5
curl -fsS http://127.0.0.1:4137/health
curl -fsS http://127.0.0.1:4137/ > "$WORK/root.html"
grep -Eo 'assets/css/app-[0-9a-f]+\.css' "$WORK/root.html" | head -1
grep -c 'app\.css?v=' "$WORK/root.html" || true
```

Expected behavior: health returns OK; the first grep prints a hashed filename like
`assets/css/app-a925cd4d….css`; the second grep prints `0`. PASS: hashed name present
AND count is 0. (In a dev `mix phx.server` there is no manifest — the bare
`/assets/css/app.css` with no query is the PASS shape there.)

### §C — Service-worker cache name (same session as §B)

```sh
curl -fsS http://127.0.0.1:4137/workspace-sw.js | head -1
```

Expected behavior: prints
`const CACHE_NAME = "allbert-workspace-shell-v1.0.1";` (matching
`allbert --version`). PASS: version in the cache name equals the released version.

### §D — Onboarding buttons `[browser]`

Open `http://127.0.0.1:4137/` in a real browser against a **fresh** disposable Home
(onboarding not yet complete). Walk the onboarding steps.

Expected behavior: every button (continue/save/skip/compact actions, persona
selection) renders styled — no unstyled/native buttons; the selected persona button

PASS: visual check attested, recorded as evidence id `v101-onboarding-buttons`.
