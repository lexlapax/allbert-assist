# Allbert v0.38 Request Flow - Templated Creation

Status: implemented through M6 closeout on 2026-05-27 and ready for operator
manual verification.

Implementation details live in `docs/plans/v0.38-plan.md`.

v0.38 makes common creation patterns first-class for developers and operators:
plugins, apps, LLM tools, scheduled/chron flows, objective workflows, and future
templated code patterns. It reuses v0.36 sandboxing and v0.37 live integration
and adds no new authority.

## Two Output Modes

- **Developer scaffold:** writes inert source under `./plugins/<name>/` for
  human review, compile, and test. No live integration. Available for every
  shipped pattern.
- **Operator live integration:** renders a vetted template, runs the v0.36
  sandbox gate, and uses the v0.37 operator-confirmed live loader. Available
  only for patterns whose artifact shape is loader-eligible in v0.37.5. In
  v0.38 that is the **LLM-tool (action) pattern only**; plugin, app,
  scheduled/chron flow, and objective workflow patterns are
  developer-scaffold-only because the v0.37 loader rejects generated apps,
  panels, settings fragments, memory namespaces, objective wiring, jobs, and
  route pages before in-core compile.

## Flow 1: Developer scaffolds a plugin

Developer runs `mix allbert.gen.plugin my_plugin`.

Expected:

1. Template renders to `./plugins/my_plugin/` because `--target` defaults to
   `./plugins/<name>/`.
2. Output includes manifest, plugin module, skill root, diagnostics, and docs.
3. Output is inert: no compile path, trust, permission, route, or execution
   authority.
4. If `./plugins/my_plugin/` already exists, generation stops and shows a
   preview/diff; overwrite requires explicit `--force`.

## Flow 2: Developer scaffolds an app

Developer runs `mix allbert.gen.app my_app --target ./plugins/my_app`.

Expected:

1. Template renders app module, supervision wiring, sample action, panel
   surface, settings fragment, destination metadata, intent descriptor, memory
   namespace, optional objective/canvas stubs, and theming/layout docs.
2. `mix allbert.validate_app my_app` passes on first run.
3. Output is inert and review-ready.
4. Generated theme/snippet/layout stubs are disabled by default and only
   document the v0.35 contracts.

## Flow 3: Developer scaffolds tool or flow patterns

Developer runs `mix allbert.gen.tool my_tool` or
`mix allbert.gen.flow morning_brief`.

Expected:

1. LLM-tool templates use the v0.25 Jido.AI advisory pattern.
2. Scheduled/chron flow templates use reviewed jobs/objectives contracts.
3. No template auto-enables a schedule, provider, skill, action, or live
   integration.
4. Schedule ids, module namespaces, app ids, destination ids, and output paths
   are normalized from validated params; raw params never create atoms directly.

## Flow 4: Operator creates from a template

Operator opens `workspace:create` in `/workspace`.

Expected:

1. Canvas shows template gallery.
2. Template selection shows validated parameter form.
3. Preview shows generated tree and validation status.
4. Operator chooses developer-scaffold output or live integration. The
   live-integration toggle is hidden or disabled for patterns whose
   `TemplatePattern.live_integration?` is `false` (in v0.38: plugin, app,
   scheduled/chron flow, objective workflow). LLM-tool/action templates expose
   both choices.
5. Surface owns only view/compose state; effectful work uses registered actions.
6. Stable handles are present: `#workspace-create-panel`,
   `#workspace-create-gallery`, `#workspace-create-params`,
   `#workspace-create-preview`, and `#workspace-create-validate`.

## Flow 5: Operator live integration

Operator chooses live integration for a templated artifact whose pattern
declares `live_integration?: true` (LLM-tool/action only in v0.38).

Expected:

1. Draft is written under `<ALLBERT_HOME>/dynamic_plugins/drafts/<slug>/` with
   `producer: "template_pattern"` and the pattern id in `metadata.yaml`.
2. Artifact runs v0.36 sandbox trial/gate through the existing v0.37
   `RunDraftTrial` / `RunDraftGate` actions.
3. v0.37 loader requires explicit operator confirmation. Confirmation is
   accepted only from surfaces allowed by
   `dynamic_codegen.integration_approval_surfaces` (default `["cli",
   "liveview"]`); Telegram, email, and cross-channel approval are denied.
4. On confirm, capability integrates live and is reversible via rollback.
5. Templated creation cannot bypass v0.36 or v0.37 gates.
6. If the operator somehow requests live integration for an unsupported
   pattern (plugin, app, scheduled/chron flow, objective workflow), the
   `CreateFromTemplate` action fails closed with a bounded diagnostic that
   points to developer-scaffold output; no draft is written.

## Flow 6: Invalid template input is rejected

Operator/developer supplies malicious params or traversal-laden names.

Expected:

1. Parameter validation rejects bad input with bounded diagnostics.
2. Path traversal, non-allow-listed files, and oversize output are denied.
3. Existing output roots are not overwritten unless `--force` is explicit.
4. No partial dangerous artifact is written.

## Data Flow

1. Select pattern + params.
2. `RenderTemplate` resolves vetted skeleton.
3. Developer scaffold writes inert source to `--target` (default
   `./plugins/<name>/`) and stops.
4. Operator live integration (LLM-tool only in v0.38) writes the draft under
   `<ALLBERT_HOME>/dynamic_plugins/drafts/<slug>/` with `producer:
   "template_pattern"` and the pattern id in `metadata.yaml`, then hands it to
   the v0.37 `RunDraftTrial` / `RunDraftGate` actions on top of the v0.36
   sandbox gate.
5. v0.37 loader integrates only after operator confirmation from an allowed
   surface (`cli` / `liveview`); other surfaces are denied.
6. Rollback remains available through v0.37 `RollbackIntegration`.
   `mix allbert.dynamic drafts list/show/discard` surfaces templated drafts.

Authority invariants: templates/params grant nothing; developer scaffold never
integrates; operator live integration is exactly the v0.36 + v0.37 gated path
and only for patterns whose `live_integration?` is `true`; `workspace:create`
does not own file, sandbox, or loader authority.

## DOM Ids / CLI Handles

| Surface | Handle | Use |
|---|---|---|
| Mix tasks | `mix allbert.gen.{plugin,app,tool,flow}` | Developer scaffolding (`gen.flow --pattern objective` for objective workflow) |
| Validate | `mix allbert.validate_app` | First-run app validation (existing task, reused) |
| Templated drafts | `mix allbert.dynamic drafts list/show/discard` | Operator inspection of templated drafts beside codegen drafts |
| Create destination | `workspace:create` / `#workspace-dest-create` | Operator gallery |
| Create panel | `#workspace-create-panel` | Create destination root |
| Template gallery | `#workspace-create-gallery` | Pick pattern |
| Parameter form | `#workspace-create-params` | Validate params |
| Preview | `#workspace-create-preview` | Review generated tree/diff |
| Validation | `#workspace-create-validate` | Validation/gate status |
| Live confirm | confirmation record | v0.37 trust grant; approval-surface restricted to `cli`/`liveview` |
| Settings master switch | `templates.create.enabled` (default false) | Enable operator Create surface |
| Live integration switches | `dynamic_codegen.enabled` + `dynamic_codegen.live_loader_enabled` + `sandbox.elixir.enabled` | Required for operator live integration |
| Approval surface allowlist | `dynamic_codegen.integration_approval_surfaces` (default `["cli","liveview"]`) | Telegram/email/cross-channel denied for templated integration |

## Security Evals (v0.28 harness rows)

- `template-create-disabled-001`: the operator `workspace:create` surface is
  denied when `templates.create.enabled=false`, and live integration is denied
  unless the v0.37/v0.36 switches are also enabled.
- `template-param-injection-001`: malicious params cannot inject code/paths.
- `template-path-traversal-001`: output stays within target root.
- `template-overwrite-deny-001`: existing roots are not overwritten without
  explicit force and preview/diff.
- `template-authority-bypass-001`: templates/params grant no authority.
- `template-integration-gate-001`: live integration cannot bypass v0.36/v0.37.
- `template-canvas-authority-001`: Create surface cannot execute/integrate
  without registered action + confirmation.
- `template-scheduled-flow-escalation-001`: chron/scheduled templates cannot
  auto-enable jobs or bypass objective/action confirmation.
- `template-unsupported-live-target-001`: operator live integration is denied
  for plugin, app, scheduled/chron flow, and objective workflow templates
  because v0.37.5 loader rejects their artifact shapes; the request fails
  closed before any draft is written.

## Smoke Checklist

- `mix allbert.gen.plugin/app/tool/flow` produce inert, review-ready trees.
- Default target is `./plugins/<name>`; explicit `--target` works; existing
  directories require `--force`.
- `validate_app` passes on app scaffold.
- `workspace:create` previews and validates templates.
- Developer scaffold never integrates.
- Operator live integration is gated, confirmed, and reversible.
- Invalid params/traversal are denied.
- Security evals fail closed.
