# Allbert v0.35 Dynamic Plugin/App Generation And Sandboxed Module Loading

Request flow: `docs/plans/v0.35-request-flow.md`

ADRs:

- `docs/adr/0032-dynamic-plugin-generation-and-sandboxed-loading.md`
- `docs/adr/0033-capability-gap-acquisition-and-trust-tiers.md`

v0.35 graduates the parked "Dynamic Elixir Code Generation Or Module Loading"
future feature into an implementation-ready plan. It does so with a narrow
safety architecture: generated plugin/app code may be written under Allbert
Home and compiled/tried in a disposable sandbox node or process, but it must
not be compiled, evaluated, or loaded into the core Allbert BEAM node.

## Purpose

When Allbert receives a request that needs a missing capability, it should be
able to propose a generated local plugin or app draft, build it in a sandbox,
try it, and report the result without turning generated code into trusted
runtime authority. v0.35 implements that capability-gap loop for local,
operator-confirmed, sandboxed trials.

This milestone is intentionally not the general developer-facing generator.
It contains the minimum internal scaffolder needed for dynamic draft creation.
v0.36 turns the proven shapes into stable Mix tasks and reusable scaffolds.

## Safety Decision

Elixir `Code.compile_string/2`, `Code.eval_string/3`, `Code.compile_file/2`,
and related APIs execute with the same privileges as the Erlang VM. v0.35
therefore forbids core-node dynamic compilation/evaluation/loading of generated
code. Generated drafts may be compiled and loaded only in an out-of-node
sandbox/trial runner with restricted inputs, no secrets, and a narrow gateway
back to Allbert.

## Expected Inputs

- v0.24 objective runtime exists and owns multi-step/cross-turn work.
- v0.31 consolidates actions, paths, audit, redaction, settings fragments,
  response shapes, and extension registry facades.
- v0.32 through v0.34 precede v0.35 in roadmap order but are sequencing
  context, not hard technical prerequisites. v0.35's real runtime dependencies
  are v0.24 (objective runtime) and v0.31 (consolidated substrates). v0.35
  reuses v0.32 workspace surfaces only to surface draft status, may use v0.33
  intent descriptors to explain missing-capability handoff, and treats the
  v0.34 `<ALLBERT_HOME>` override precedent as draft-root policy guidance.
- A concrete out-of-node, OS-level sandbox backend (a separate OS process with
  dropped privileges, or a container/VM — per ADR 0032 and the parked
  Level-2/Level-3 execution-sandbox work) must exist before M2; absent one, the
  workflow stays disabled.
- ADR 0009 local execution sandbox levels remain true: the core BEAM process
  and child processes are not an OS security boundary.

## Design Decisions

### 1. Generate does not enable

Generated code is inert draft source under:

`<ALLBERT_HOME>/plugins/<slug>/`

Draft creation does not add compile paths, register modules, trust skills,
grant permissions, mount routes, start children, run migrations, or enable
actions.

### 2. Out-of-node sandbox loading only

The sandbox runner may compile and load draft modules only inside a real
OS-level isolation boundary — a separate OS process with dropped privileges, or
a container/VM — never a separate or hidden BEAM/distributed-Erlang node and
never an in-VM "disposable process" (BEAM processes share host privileges; see
ADR 0032 "Sandbox Isolation Requirement" and ADR 0009). The core Allbert node
communicates only through a narrow gateway with bounded request/response data
and redaction.

### 3. Objective-runtime capability gap flow

Dynamic generation starts as an objective step:

1. intent detects missing capability;
2. objective records a capability gap;
3. Allbert proposes "generate local plugin/app draft";
4. operator confirms generation;
5. draft source is generated inertly;
6. sandbox compile validates it;
7. sandbox trial runs bounded scenarios;
8. Allbert reports results;
9. operator discards or marks as promotion candidate.

Advisory/model output never authorizes generation, sandbox execution, or
promotion.

### 4. No new supply-chain or schema authority

v0.35 generated drafts may use only existing project dependencies and approved
post-v0.31 public facades. They may not add Hex dependencies, package-manager
steps, NIFs, ports, migrations, Repo credentials, unrestricted network
adapters, secrets, or external installers.

### 5. Promotion is separate from trial

v0.35 may record a draft as "promotion candidate" after operator review. It
does not automatically promote generated code into reviewed source, compile it
into the project, or enable it in the core node. v0.36 may provide a reviewed
source integration path.

## Planned Interfaces

- `AllbertAssist.DynamicPlugins`: context for draft lifecycle, sandbox compile,
  sandbox trial, discard, and promotion-candidate state.
- `AllbertAssist.DynamicPlugins.Draft`: normalized draft metadata, slug,
  trust tier, path, objective id, capability gap, status, diagnostics, and
  redacted trial summary.
- `AllbertAssist.DynamicPlugins.Sandbox`: compile/trial adapter contract with
  no direct access to secrets, Repo credentials, or core process state.
- `AllbertAssist.Actions.DynamicPlugins.CreateDraft`: confirmed action that
  writes inert draft source.
- `CompileDraft`, `RunDraftTrial`, `DiscardDraft`, and
  `MarkPromotionCandidate`: registered actions with Security Central decisions
  and audit events.
- `AllbertAssist.Objectives.CapabilityGap`: objective helper for representing
  missing capability and acquisition options.

Exact module names may change during implementation, but the public context,
registered actions, sandbox adapter, and objective-gap vocabulary must remain
explicit in docs and request-flow updates.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless the implementation owner records a narrower gate and
why.

### M1: ADRs, paths, trust tiers, and capability-gap vocabulary

Goal: define the safety boundary before generation exists.

Tasks:

- Accept ADR 0032 and ADR 0033.
- Add dynamic plugin draft root under Allbert Home.
- Define trust tiers: `:draft`, `:sandbox_compiled`, `:sandbox_trialed`,
  `:promotion_candidate`, and `:discarded`.
- Add objective capability-gap and acquisition-option vocabulary.
- Add Settings Central policy keys only for enabling the v0.35 workflow; keep
  default disabled if implementation risk warrants it.

Tests:

- Temporary home path tests.
- Objective gap serialization tests.
- Settings/security tests for disabled workflow and confirmation requirement.

Acceptance:

- No draft can be generated or tried without explicit operator action.

### M2: Sandbox compile/load harness

Goal: compile draft code outside the core node.

Tasks:

- Implement sandbox adapter contract and first local sandbox backend.
- Provide copy-in/copy-out or isolated draft path handling.
- Deny secrets, Repo credentials, unrestricted network, package managers,
  dependency changes, migrations, NIFs, and core-node code loading.
- Audit compile start, success, failure, timeout, and policy denial.

Tests:

- Compile success/failure tests with generated fixture drafts.
- Negative tests for forbidden dependency, migration, NIF, secret, network,
  package-manager, and core-load attempts.
- Timeout and cleanup tests.

Acceptance:

- A draft can compile in sandbox; the core BEAM node never loads its modules.

### M3: Minimal dynamic scaffolder

Goal: generate inert plugin/app draft source for a bounded capability gap.

Tasks:

- Generate manifest, README, plugin/app module skeletons, sample action, panel
  surface stub, settings fragment stub, skill stub, validation notes, and trial
  scenarios using post-v0.31 shapes.
- Validate slug, path, file allow-list, generated file size, and no path
  traversal.
- Store generation prompt/provenance in redacted audit/trace records.

Tests:

- Draft tree tests.
- Path traversal and file-size denial tests.
- Generated source static checks for forbidden constructs.

Acceptance:

- Generated source is inspectable and inert before compile/trial.

### M4: Sandboxed trial run and structured report

Goal: try a compiled draft through bounded scenarios and report safely.

Tasks:

- Add trial scenario manifest shape.
- Run scenarios in sandbox with bounded inputs, outputs, time, memory, process,
  and filesystem access.
- Redact results before storing or rendering.
- Return structured success/failure diagnostics to the objective.

Tests:

- Passing/failing trial tests.
- Output limit and redaction tests.
- Audit/trace tests for trial lifecycle.

Acceptance:

- Operator can inspect whether the draft appears useful without enabling it.

### M5: Operator review, discard, and promotion-candidate state

Goal: make trial results reversible and non-authoritative.

Tasks:

- Add discard action that removes or tombstones the draft safely.
- Add mark-promotion-candidate action that records review intent without
  compiling into core.
- Render status in workspace Settings/Security surfaces once v0.32 exists.
- Document manual review expectations before any future promotion path.

Tests:

- Discard/tombstone tests.
- Promotion-candidate tests prove no compile path, route, action permission,
  trust, skill enablement, or child process is activated.

Acceptance:

- Trialed code remains sandbox-only until a later reviewed integration step.

### M6: Security evals and closeout

Goal: prove v0.35 does not weaken the existing non-negotiable.

Tasks:

- Add security evals for core-node compile attempts, promotion bypass,
  dependency injection, migration injection, secret access, network bypass,
  path traversal, sandbox escape signals, output exfiltration, and advisory
  approval bypass.
- Update README, roadmap, future-features, agent map, AGENTS, DEVELOPMENT, and
  ADR forward pointers.

Acceptance:

- Security evals fail closed.
- Docs state the old arbitrary-loading rule remains true except for the
  ADR-defined sandbox trial path.

## Non-Goals

- No core-node dynamic `Code.compile_*`, `Code.eval_*`, `Code.require_*`, or
  BEAM loading of generated code.
- No remote marketplace, remote plugin installation, remote binary loading, or
  code-bearing distribution.
- No package-manager execution, dependency additions, migrations, NIFs, ports,
  or unrestricted network adapters.
- No automatic trust, skill enablement, route authority, permission grants,
  Settings Central authority, child supervision, or app activation.
- No general developer-facing generator; that is v0.36.
- No autonomous trace-to-skill creation; that remains parked future work.

## Definition Of Done

v0.35 is done when Allbert can propose, generate, compile, and try a local
plugin/app draft in a sandboxed out-of-node runtime, report redacted results,
and discard or mark the draft as a promotion candidate without ever loading
generated modules into the core node or granting generated code authority.
