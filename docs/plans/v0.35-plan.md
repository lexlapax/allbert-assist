# Allbert v0.35 User Theming And Layout Overrides

Status: implemented and release-ready as `v0.35.0`; ready for operator manual
verification.

Request flow: `docs/plans/v0.35-request-flow.md`

ADR: `docs/adr/0025-user-theming-and-override-security.md`

v0.35 follows the v0.32 workspace-only UI reset, the v0.33 app-intent handoff
foundation, and the v0.34 workspace UX refresh. It lets operators retheme and
re-lay-out the `/workspace` UI from Allbert Home without editing core code or
rebuilding assets. The Elixir/OTP sandbox substrate moves to v0.36, dynamic
code/config generation and gated live integration moves to v0.37, and the
Plugin And App Generator / templated creation path moves to v0.38 so it can
scaffold the workspace-panel, intent-descriptor, destination, and theming hooks
after each is proven.

## Purpose

Allbert styling is currently host-side: Tailwind/daisyUI compiled into the web
asset bundle, plus workspace variables scoped to the shell. Operators cannot
change colors, fonts, density, or panel layout without editing source.

v0.35 adds a safe, local override system:

- Design-token YAML for common retinting and spacing changes.
- Opt-in sanitized CSS snippets for advanced local customization.
- Validated layout data for enabling, disabling, reordering, and pinning
  v0.34 launcher destinations and Canvas panel groups.

The system reads only from Allbert Home, serves only through host-owned routes,
and is gated by Settings Central keys available from the v0.34 Settings Canvas
destination.

v0.35 is mostly additive, with one deliberate non-additive change: the shipped
scalar `workspace.theme` mode key is renamed to `workspace.theme.mode` so the
new token/snippet keys can live under the `workspace.theme.*` namespace without
a scalar-vs-prefix collision (see Design Decision 5). This is a settings
migration with a compatibility read, not a behavior change to the mode toggle.

## Expected Inputs

- v0.32 ships `/workspace` as the only operator home and the validated
  host-owned panel-surface substrate.
- v0.33 ships app-contributed intent descriptors and explicit handoff
  responses, which theming must not hide or make ambiguous.
- v0.34 ships the chat-primary shell, view-only launcher, single-destination
  Canvas, Settings as a Canvas destination, and handoff-only routing context.
- v0.31 ships consolidated runtime paths and settings fragments.
- `AllbertAssist.Runtime.Paths` resolves Allbert Home.
- Settings Central supports typed keys, auditing, and action-backed writes.
- The web app has the current Tailwind/CSS asset pipeline and workspace shell
  CSS variables.

## v0.34 Carry-Forward Constraints

v0.35 customizes the shipped v0.34 workspace shell; it does not reopen the
workspace IA or routing model:

- Output remains its own launcher section and the neutral Allbert destination.
- Allbert is not exposed as an `app:allbert` launcher/layout destination.
- Launcher selection, layout ordering, and destination URL state are view-only.
  They never set `active_app`, execute actions, or grant permission.
- `active_app` remains handoff-only through the v0.33 explicit accept flow.
- The AppBar is fixed chrome and out of scope for layout override.
- Settings and Output are non-hideable escape hatches.
- Retired `:utility_drawer` / `:context_rail` labels may remain recognized for
  historical panel-zone validation, but v0.35 cannot remount those old regions
  or resurrect the old Tools/context rail IA.

## Design Decisions

### 1. All operator overrides live under Allbert Home

New path helpers:

- `AllbertAssist.Runtime.Paths.themes_root/0` → `<ALLBERT_HOME>/themes`
- `AllbertAssist.Runtime.Paths.theme_snippets_root/0` →
  `<ALLBERT_HOME>/themes/snippets`
- Workspace layout file → `<ALLBERT_HOME>/workspace/layout.yaml`

`ensure_home!/0` creates the roots. Until the v0.31 runtime path facade fully
owns home creation, implementation updates both `AllbertAssist.Runtime.Paths`
and the current legacy `AllbertAssist.Paths`/home-creation path as needed so a
single disposable home exposes `themes`, `themes/snippets`, and `workspace`.
Tests and smoke checks must use a temporary `ALLBERT_HOME`.

### 2. Design tokens are the default path

`<ALLBERT_HOME>/themes/<name>.yaml` contains a constrained token map for
colors, font family, font scale, spacing density, radius, border contrast, and
motion preference. The host renders those tokens as CSS custom properties at
`/theme/user.css`, linked after `app.css`.

**Token allow-list (presentational only).** Only the existing presentational
`--allbert-*` workspace variables are themeable — colors, font family, font
scale, spacing density, radius, border contrast, and motion. The allow-list is
derived from the `#workspace-shell` token set and pinned in
`AllbertAssist.Theme.Tokens` during M2. Layout-structural variables (root-grid
track sizes, rail/canvas widths, AppBar geometry), route selectors, and
Tailwind/daisyUI `--color-*` variables are **excluded** so a token theme can
never break the v0.34 root-grid CSS, globally retheme unrelated UI, or hide
chrome.

**Token theme vs mode (single set over current mode).** The dark/light/system
mode (now `workspace.theme.mode`) still sets the base palette; the selected
token theme is a single set that retints on top of whichever mode is active.
`/theme/user.css` loads after `app.css` and applies in both modes; snippets
(Design Decision 3) load last. v0.35 does not add per-mode token variants;
operators are responsible for choosing tokens that read in their mode, and
defaults stay safe.

Missing files, invalid YAML, unknown tokens, or invalid values fall back to
defaults per-key with bounded operator-visible warnings. No asset rebuild is
required.

Example token file shape:

```yaml
tokens:
  allbert-surface-0: "#101820"
  allbert-surface-1: "#17212b"
  allbert-surface-2: "#1f2c38"
  allbert-text-strong: "#f8fafc"
  allbert-text-soft: "#b6c2cf"
  allbert-line: "#314457"
  allbert-accent: "#7cc7d8"
  allbert-accent-contrast: "#081117"
  allbert-accent-soft: "#17313a"
  allbert-warn: "#fbbf24"
  allbert-warn-soft: "#422006"
  allbert-danger: "#f87171"
  allbert-success: "#4ade80"
```

Invalid example: `color-base-100`, `workspace-root-grid`, or
`workspace-canvas-width` keys are ignored with diagnostics because they are
outside the presentational `--allbert-*` token allow-list.

### 3. CSS snippets are opt-in and sanitized

`<ALLBERT_HOME>/themes/snippets/*.css` files are disabled by default. When
enabled, snippets are sanitized before serving:

- Reject `@import`.
- Strip `url()` and `image-set()`.
- Strip remote font/source constructs.
- Serve empty CSS if a snippet is entirely unsafe.
- Emit bounded warnings naming removed constructs.

Snippets are loaded after token CSS and outside Tailwind layers so they can
override local presentation. They never grant action authority and never load
remote resources.

Example safe snippet:

```css
#workspace-shell .workspace-chat-pane {
  font-size: 0.95rem;
}
```

Invalid example: a snippet containing `@import`, `url(...)`, `image-set(...)`,
or remote font/source constructs is rejected or stripped before serving, with a
bounded warning.

### 4. Layout overrides are data, never code

`<ALLBERT_HOME>/workspace/layout.yaml` can enable/disable/reorder v0.34
launcher destinations, set the default Canvas destination, and pin registered
panels into allowed destination groups. It validates against the destination
registry, retained zone labels used for panel validation, registered panel ids,
and catalog atoms.

Unknown destinations, unknown historical zone labels, unknown panel ids, or
invalid placements are ignored per-key with warnings. Layout override cannot
create components, routes, actions, routing context, or permissions.

Example layout file shape:

```yaml
default_destination: output
launcher_order:
  - output
  - app:stocksage
  - workspace:settings
  - workspace:jobs
  - workspace:objectives
hidden_destinations:
  - workspace:jobs
panel_pins:
  app:stocksage:
    - stocksage.dashboard
```

Invalid example: `app:allbert`, `workspace:unknown`, `component:evil`,
`route:/admin`, `active_app: stocksage`, or attempts to hide `output` /
`workspace:settings` are dropped with diagnostics. Neutral Allbert output is
represented by `output`, not by an app destination.

### 5. Settings Central owns enablement and accountability

**Migration first.** v0.34 shipped a scalar `workspace.theme` key (the
light/dark/system mode, reached from the AppBar `#workspace-theme-toggle`).
v0.35 renames it to `workspace.theme.mode` so the new token/snippet keys can
share the `workspace.theme.*` namespace without a scalar-vs-prefix collision.
The migration:

- adds `workspace.theme.mode` to the schema, defaults, and safe-write list;
- keeps a compatibility read that accepts a legacy stored `workspace.theme`
  value and normalizes it to `workspace.theme.mode` (migrate-on-read/write),
  audited like any settings change;
- updates every mode call site to the new key, including `SetTheme`,
  `WorkspaceLive` assigns/helpers, `theme_attribute/1`, the AppBar
  read/toggle path, `toggle_workspace_theme`, the settings CLI, and
  `mix allbert.workspace`.

New and migrated Settings Central keys:

| Key | Type | Default | Purpose |
|---|---|---|---|
| `workspace.theme.mode` | string | `system` | Light/dark/system mode (migrated from `workspace.theme`) |
| `workspace.theme.active` | string/null | null | Selected token theme name |
| `workspace.theme.snippets_enabled` | boolean | false | Master switch for CSS snippets |
| `workspace.theme.enabled_snippets` | string list | [] | Enabled snippet file basenames |
| `workspace.layout.override_enabled` | boolean | false | Master switch for layout YAML |

These keys are owned by the shipped core `workspace` Settings Central schema
fragment (`core:workspace`), assembled from
`AllbertAssist.Settings.Schema` through the v0.31
`AllbertAssist.Settings.Fragments` facade. They are visible and writable
through the v0.34 Settings Canvas destination and existing settings CLI actions;
v0.35 adds no separate `/settings` route. Every write to these keys uses the
existing Settings Central action/audit path. The
`workspace.layout.override_enabled` master switch is the supported escape hatch
(UI or CLI) if a layout override misbehaves (see Design Decision 8).

Settings Central **does not** store raw token YAML, snippet CSS, or layout YAML
blobs. Those files remain local operator data under Allbert Home. Settings
Central stores and audits the gates/selections that make those files active,
and the Settings Canvas surfaces read-only accountability status:

| Status field | Meaning |
|---|---|
| active token theme basename | Selected theme file, without unsafe path exposure |
| active token fingerprint/mtime | File state used for cache version and operator inspection |
| token parse status | ok, missing, invalid, or partially applied |
| snippet basenames and fingerprints | Enabled snippets after basename validation |
| snippet sanitizer status | ok, stripped, rejected, or empty |
| layout file fingerprint/mtime | Layout file state used for cache/version diagnostics |
| layout validation status | ok, disabled, invalid, or partially applied |
| bounded diagnostics | Capped warnings for ignored keys/values/constructs |

Diagnostics are capped by count and message length, never include raw file
contents, never include secrets, and avoid absolute paths in UI unless a path
is already an intentionally safe project-local display value.

### 6. CSP backs the sanitizer

The `:browser` pipeline adds a Content-Security-Policy baseline (there is no CSP
today beyond `put_secure_browser_headers`). Concrete starting policy, to be
pinned against the running LiveView during M3:

```
default-src 'self';
style-src 'self';
img-src 'self' data:;
font-src 'self';
connect-src 'self' ws: wss:;     # LiveView socket
script-src 'self';
base-uri 'self';
frame-ancestors 'none';
object-src 'none'
```

- If LiveView's runtime requires inline styles/scripts that cannot be served
  from `'self'`, prefer a per-response nonce over `'unsafe-inline'`; document
  any unavoidable allowance in the request-flow.
- `style-src 'self'` (no remote imports) is what backs the sanitizer:
  even a missed dangerous construct cannot fetch remote styles/images.

**`/theme/*` route specifics** (because `Plug.Static` cannot reach Allbert
Home): a dedicated `/theme` scope/pipeline (no session required) serving
`text/css` only; reads are scoped to `themes_root/0` / `theme_snippets_root/0`
with a path-traversal guard (reject `..`, non-`.css`, symlinks out of root);
responses carry `ETag`, and workspace stylesheet links include `?v=` derived
from the active Settings Central selection plus relevant file mtimes or
fingerprints so edits show on reload without a stale cache.

### 7. Layout validates against a pinned destination/panel source

Layout override validation needs an authoritative, enumerable source — v0.34
does not expose one today, so M5 must add it. Pin:

- **Destination grammar:** `output` | `app:<registered_app_id>` |
  `workspace:<core_tool>` (the same ids the shipped launcher and AppBar use via
  `workspace_destination_path/2`). `app:allbert` is not valid; neutral Allbert
  output is `output`.
- **Enumeration API:** add `AllbertAssist.Workspace.Catalog.known_destinations/1`
  (or equivalent), derived from `output`, the registered non-Allbert apps
  (`Extensions.Registry`), and the CoreApp workspace tools. This is the single
  source shared by the shipped launcher/AppBar quick-links and the layout
  validator for destination ids and the default-destination check.
- **Registered panels:** validated against CoreApp/app panel surface ids; pins
  target allowed Canvas destination groups only.
- **Retained zone labels:** the catalog's recognized labels (including the
  v0.34-demounted `:utility_drawer` / `:context_rail` labels that remain
  recognized per ADR 0024) are accepted for panel-zone validation but cannot
  re-mount a region.

Unknown destinations, unknown zone labels, unknown panel ids, or invalid
placements are dropped per-key with bounded warnings. Layout data never creates
components, routes, action bindings, routing context, or permissions.

### 8. No lockout: fixed chrome and non-hideable destinations

Per the v0.34 model, the **AppBar is fixed chrome and out of scope for layout
override**: its brand, context indicator (`#workspace-context-indicator`), theme
toggle (`#workspace-theme-toggle`), and destination quick-links are always
present and not reorderable/hideable by `layout.yaml`. Layout override governs
only the **left launcher and the Canvas**.

To prevent self-lockout:

- The **Settings** and **Output** launcher destinations are non-hideable; a
  layout file that tries to hide them is honored for everything else and warns
  on those entries.
- The `workspace.layout.override_enabled` master switch (UI Settings or the
  settings CLI action) always disables overrides regardless of layout content.
- Because the AppBar is fixed, the context indicator and mode toggle remain
  reachable even with an aggressive layout file.

## Module Shapes

- `AllbertAssist.Theme.Tokens`: parse, validate, normalize, and render token
  YAML into CSS variables.
- `AllbertAssist.Theme.Snippets`: sanitize enabled CSS snippets, reject unsafe
  imports/traversal, strip unsafe fetch constructs, and return warnings.
- `AllbertAssist.Theme.Status`: produce read-only Settings Canvas accountability
  status for selected theme/snippet/layout files, fingerprints/mtimes, and
  bounded diagnostics.
- `AllbertAssist.Theme.Version`: derive deterministic cache stamps from selected
  settings and relevant local file fingerprints/mtimes.
- `AllbertAssist.Theme.Layout`: parse and validate workspace layout YAML
  against `Workspace.Catalog.known_destinations/1`, retained panel-zone labels,
  and registered panel surface ids; enforce non-hideable Settings/Output.
- `AllbertAssist.Workspace.Catalog.known_destinations/1`: new enumerable source
  of valid destinations (`output` | `app:<id>` | `workspace:<tool>`) for layout
  validation (Design Decision 7).
- `AllbertAssistWeb.ThemeController`: serve `/theme/user.css` and sanitized
  `/theme/snippets/<name>.css` from Allbert Home, `text/css` only, traversal-
  guarded, with `ETag` and `?v=` cache/version behavior; mounted on a dedicated
  `/theme` scope.
- `AllbertAssist.Runtime.Paths` and current `AllbertAssist.Paths` home-creation
  path: add theme roots until the runtime facade fully owns creation.
- `AllbertAssist.Settings.Schema` / `AllbertAssist.Settings.Fragments`: the
  shipped core `workspace` fragment owns `workspace.theme.*` (incl. migrated
  `workspace.theme.mode`) and `workspace.layout.override_enabled` through the
  v0.31 fragment contract.
- Settings Canvas diagnostics: render read-only accountability status for
  active files, fingerprints/mtimes, parser/sanitizer/layout status, and
  bounded warnings.
- Workspace shell/catalog: consume validated layout data when the setting is
  enabled; AppBar stays fixed chrome.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless the implementation owner records a narrower gate.

### M1: Theme roots, settings accountability, and migration

Goal: Allbert Home has theme roots and Settings Central has typed enablement
keys, audit coverage, and read-only status for file-backed overrides.

Tasks:

- Add theme path helpers and create roots from `ensure_home!/0` through both
  the runtime facade and current legacy path/home-creation path as needed.
- Migrate `workspace.theme` → `workspace.theme.mode` with a compatibility read,
  and update all mode call sites (`SetTheme`, `WorkspaceLive`,
  `theme_attribute/1`, AppBar read/toggle path, `toggle_workspace_theme`,
  settings CLI, and `mix allbert.workspace`).
- Add the new `workspace.theme.*` / `workspace.layout.override_enabled` keys to
  the shipped core `workspace` Settings Central schema fragment, with audit
  coverage.
- Render the settings controls in the v0.34 Settings Canvas destination.
- Render read-only accountability status for active override basenames,
  fingerprints/mtimes, parser/sanitizer/layout status, and bounded diagnostics.

Tests:

- Path tests with temporary Allbert Home.
- Settings schema/read/write/audit tests, including legacy `workspace.theme`
  migration-on-read.
- Workspace settings-panel tests for the new keys and read-only diagnostics.

Acceptance:

- A disposable home exposes the new roots and settings without touching a real
  user home.
- Settings writes are audited, while raw CSS/YAML file contents remain outside
  Settings Central.

### M2: Token themes

Goal: operator-selected token YAML retints `/workspace`.

Tasks:

- Serve `/theme/user.css`.
- Link token CSS after `app.css`.

Tests:

- Token parsing/rendering tests.
- Controller tests for selected, missing, invalid, and partially invalid
  themes.

### M3: CSP and cache headers

Goal: theme routes and workspace pages have a CSP compatible with LiveView and

- Add `ETag` responses and `?v=` stylesheet links derived from Settings

Tests:

- Cache/version tests for selected, edited, missing, and disabled theme files.
- Regression test for blocked remote style/image fetch attempts where
  feasible.

Acceptance:

  without stale cache behavior.

### M4: CSS snippets

Goal: advanced CSS snippets work only when explicitly enabled and sanitized.

Tasks:

- Add snippet root helper and controller route.
- Implement sanitizer with reject/strip/warn behavior.
- Gate snippets behind Settings Central master switch and enabled basename
  list.
- Prevent path traversal and non-`.css` serving.

Tests:

- Sanitizer tests for `@import`, `url()`, `image-set()`, remote font/source

- Controller tests for disabled, enabled, missing, traversal, and sanitized

  `theme-snippet-url-strip-001`, `theme-css-exfil-001`,
  `theme-path-traversal-001`.
- Chrome verification that safe snippet changes apply and unsafe constructs do
  not fetch.

Acceptance:

- Safe local snippets can restyle the workspace.
- Dangerous constructs are removed with warnings.

### M5: Layout overrides

Goal: layout YAML can reorder/hide zones and pin panels without adding

Tasks:

- Make the destination registry the shared source for the launcher, AppBar
  quick-links, and layout validator; exclude `app:allbert`.
- Enforce non-hideable Settings/Output and AppBar-out-of-scope (Design

- Apply valid layout data in workspace composition when enabled.
- Warn and drop unknown/invalid keys per-key.
- Ensure layout override cannot create routes, components, action bindings,

Tests:

- Layout parser/validator tests.
- Workspace catalog tests for hide/reorder/pin behavior across launcher

- Negative tests for unknown destinations, unknown retained zone labels,
  unknown panels, `app:allbert`, invalid placements, hidden Settings/Output,

- v0.28 eval rows: `layout-override-authority-001`,
  `layout-hide-settings-lockout-001`.

Acceptance:

- Invalid layout data is safely ignored with diagnostics.

## Non-Goals

- No `/settings` route.
- No remote theme marketplace or remote theme fetching.
- No arbitrary JavaScript, plugin code loading, or dynamic Elixir compilation.
- No app generator; that is v0.38.
- No new routing context selector or `active_app` layout control. The
  `workspace.theme` → `workspace.theme.mode` rename is presentational only and
  changes no routing or domain behavior.
- No AppBar reordering: the AppBar (brand, context indicator, theme toggle,
  destination quick-links) is fixed chrome, out of scope for layout override.
- No new workspace destinations beyond the v0.34 launcher/destination registry
  unless required by a documented v0.35 implementation blocker.
- No new Surface catalog atoms solely for theming/layout overrides.
- No AG-UI/A2UI/MCP Apps UI protocol bridge.

### Security evals (v0.28 harness rows)

- `theme-snippet-import-reject-001`: `@import` in a snippet is rejected.
- `theme-snippet-url-strip-001`: `url()` / `image-set()` are stripped.
- `theme-css-exfil-001`: attribute-selector + remote-fetch exfiltration attempt
  is neutralized (sanitizer + CSP).
- `theme-path-traversal-001`: `/theme/*` cannot serve files outside the theme
  roots or non-`.css` files.
- `theme-csp-regression-001`: the workspace/theme CSP blocks remote style/image
  fetches and does not regress.
- `layout-override-authority-001`: layout YAML cannot create routes, components,
  action bindings, routing context, or permissions.
- `layout-hide-settings-lockout-001`: a layout file that hides Settings/Output
  is overridden so they stay reachable.

## Definition Of Done

v0.35 is done when `/workspace` can be rethemed and re-laid out from
Allbert Home through Settings Central-gated token, snippet, and layout files;
unsafe CSS and invalid layout data are bounded with warnings; CSP is in place;
and full gate plus Chrome verification pass.
