# Allbert v0.29 App Memory + Outcomes Contract - StockSage Polish

Request flow: `docs/plans/v0.29-request-flow.md`

Formerly tracked as M-D3b, previously planned as v0.27 before the
project-direction rethink inserted v0.23 Jido State-Machine Convergence and
v0.24 Objective Runtime Foundation. Renamed after the post-v0.26 roadmap
reconciliation to make the contract visible: v0.29 proves memory-consuming app
behavior through StockSage outcomes and lessons.

## Purpose

v0.29 brings StockSage's Elixir implementation to parity with Python
StockSage 0.0.2 user-facing behavior: outcomes, reflections, explicit memory
sync, trends, reruns, and polished responsive UI.

v0.27 declares StockSage's memory namespace as an ownership claim. v0.28 audits
that claim before memory writes exist. v0.29 is the first milestone that
consumes the namespace by syncing resolved StockSage lessons into StockSage
memory records and Allbert memory namespaces through explicit, traceable
actions.

## Memory Sync Write Path (M4 Implemented)

This is the cross-version seam v0.27/v0.28 hand off to. The declaration
(ownership) shipped in v0.27; v0.29 adds the *write*. M4 implements the
following concrete mechanism:

- **Write API**: lesson sync uses the existing markdown memory store, but M4
  first made namespaced writes explicit. `AllbertAssist.Memory.append/1`
  remains the raw append path; `Memory.upsert_app_entry/1` is the namespaced,
  idempotent app-memory write path. v0.29 extends the existing markdown memory
  contract and does not add a second memory store.
- **Memory extension**: `AllbertAssist.Memory.Entry` and markdown
  render/parse/list behavior now preserve optional app-memory fields:
  `app_id`, `namespace`, `kind`, `idempotency_key`, and `source_ref`.
  `Memory.upsert_app_entry/1` validates the v0.27 namespace declaration,
  persists those fields, filters by namespace/kind/idempotency key, and updates
  an existing matching entry instead of appending a duplicate.
- **Namespace posture**: writes carry `%{app_id: :stocksage, namespace:
  :stocksage}` matching the v0.27 declaration.
  `StockSage.App.memory_namespace/0` now declares `writable: true`, so
  `App.Registry.registered_memory_namespaces/0` exposes StockSage as a writable
  namespace only after the v0.28 namespace evals existed.
- **Registered action (the only write entry point)**:
  `AllbertAssist.Actions.Memory.SyncAppLesson` (registered atom
  `:sync_app_lesson`), permission class `:memory_write`, routed through
  `Actions.Runner.run/3` + Security Central. StockSage LiveViews/contexts call
  the registered action; they do not write Allbert markdown memory directly.
- **Advisory context stamp**: `SyncAppLesson` stamps advisory context before
  Security Central authorization on non-approved calls. That central stamp makes
  the first operator-facing sync return `:needs_confirmation`; approved
  confirmation resume writes memory without re-stamping advisory context.
- **Lesson record shape written**:
  ```elixir
  %{
    namespace: :stocksage,
    kind: :stocksage_lesson,
    analysis_id: String.t(),
    objective_id: String.t() | nil,
    ticker: String.t(),
    rating: String.t(),
    realized_return: String.t(),
    holding_period_days: pos_integer(),
    lesson_text: String.t(),     # bounded + redacted
    source: String.t(),
    resolved_at: String.t()
  }
  ```
- **Idempotency**: keyed by `{app_id, analysis_id, holding_period_days}` and
  stored as a durable `idempotency_key`. Re-syncing the same analysis/horizon
  updates the existing lesson entry; it never appends a duplicate.
- **The no-auto-promotion invariant (tested, not prose)**: completing or
  resolving an analysis MUST NOT call `Memory.append/1`, `Memory.upsert_app_entry/1`,
  or any equivalent memory write helper. Only an explicit `:sync_app_lesson`
  invocation writes. M4 tests assert that generating a reflection leaves zero
  Allbert memory entries for the namespace until an operator-facing
  `sync_app_lesson` path queues confirmation.

## Pinned Implementation Shapes

### M1 Outcome Resolver

- Resolver context: `StockSage.Outcomes.resolve_due/2`.
- Registered action: `StockSage.Actions.ResolveOutcomes`, action name
  `resolve_outcomes`, permission `:stocksage_write`, exposure `:internal`,
  app-scoped to `:stocksage`.
- Resolver inputs: `user_id`, `as_of`, optional explicit `prices`, optional
  `limit`, optional `force`, and optional
  `neutral_return_threshold_pct`. v0.29 M1 does not fetch market data; callers
  provide observed prices or the resolver uses already-stored
  `stocksage_outcomes.end_price`.
- Updated fields: `stocksage_outcomes.observed_on`, `end_price`,
  `return_pct`, `label`, and `metadata["resolution"]`.
- Provenance metadata: resolution status, source
  `stocksage_outcome_resolver`, due/as-of dates, recommendation, objective id,
  step id, and threshold.
- Idempotency: normal resolver runs only target pending outcomes. A second run
  after resolution returns no candidates; `force: true` is reserved for
  operator repair flows.
- Settings keys introduced:
  `stocksage.outcomes.default_holding_period_days`,
  `stocksage.outcomes.resolver_cadence`, and
  `stocksage.outcomes.neutral_return_threshold_pct`.
- Non-auto-promotion invariant: M1 writes only StockSage outcome rows. It does
  not write StockSage memory records or Allbert markdown memory.

### M2 Trends And Calibration

- Trend contract remains `StockSage.Analyses.summarize_trends/2` so existing
  `get_trends` and LiveView call sites do not fork.
- New computed shape:
  `accuracy`, `rating_calibration`, and `leaderboard` alongside existing
  `counts`, `returned`, and `outcomes`.
- Accuracy uses resolved labels only (`win`, `loss`, `neutral`, `unknown`);
  pending rows remain visible in counts and the observed-outcomes table but do
  not inflate calibration.
- Return basis is explicitly `:realized_return_no_benchmark` because v0.29 has
  recorded outcome returns but no benchmark/index-adjusted alpha series.
- Rating calibration groups by the persisted analysis recommendation, falling
  back to `"unrated"`.
- The StockSage trends LiveView renders resolved count, win rate, average
  realized return, rating calibration, symbol leaderboard, and raw outcomes.

### M3 Reflection Action

- Reflection context: `StockSage.Reflections.generate/3`.
- Registered action: `StockSage.Actions.GenerateReflection`, action name
  `generate_reflection`, permission `:stocksage_write`, exposure `:internal`,
  app-scoped to `:stocksage`.
- Reflection storage: idempotent `StockSage.Memory.upsert_entry/1` rows with
  `kind: "reflection"`, `source: "analysis"`, `legacy_source:
  "stocksage_reflection"`, and `legacy_id` equal to the resolved outcome id.
- Reflection content is deterministic in v0.29 M3, bounded by
  `stocksage.reflections.max_chars`, and labels itself as StockSage-local
  advisory context.
- Settings keys introduced: `stocksage.reflections.enabled` and
  `stocksage.reflections.max_chars`.
- Operator surface: the StockSage analysis detail LiveView shows resolved
  outcomes, exposes a Generate button for non-pending outcomes, and renders
  local reflection records for the analysis.
- Non-auto-promotion invariant: generating a reflection writes StockSage-local
  memory only. It does not write Allbert markdown memory or mark the local
  record as promoted.

### M4 Explicit Memory Sync

- Core memory extension:
  `AllbertAssist.Memory.Entry` now carries `app_id`, `namespace`, `kind`,
  `idempotency_key`, and `source_ref`; markdown render/parse/list preserves and
  filters those fields.
- Write helper: `AllbertAssist.Memory.upsert_app_entry/1` validates writable app
  namespace declarations through `App.Registry.registered_memory_namespaces/0`
  and idempotently updates matching `{actor, app_id, namespace,
  idempotency_key}` entries.
- Registered action: `AllbertAssist.Actions.Memory.SyncAppLesson`, action name
  `sync_app_lesson`, permission `:memory_write`, confirmation-required and
  resumable through `ApproveConfirmation`.
- Advisory safety: the action stamps advisory context centrally before the
  initial Security Central check, so operator-facing sync requests queue
  confirmation and write no markdown memory until approved.
- StockSage surface: the analysis detail LiveView renders a `Sync lesson`
  control for generated reflections. The click creates a confirmation request,
  stores the pending confirmation/idempotency key on the local reflection, and
  shows the confirmation id. After approval writes the namespaced markdown
  lesson, the next analysis-detail refresh reconciles the local reflection to
  `promoted_to_allbert_memory: true`.
- StockSage namespace: `StockSage.App.memory_namespace/0` is now
  `writable: true` with explicit lesson-sync confirmation as the write posture.

### M5 Rerun Flow

- Rerun uses the existing registered `run_analysis` action instead of a new
  execution path, preserving the v0.22/v0.28 confirmation posture.
- `RunAnalysis` now accepts optional `source_analysis_id`, persists it in
  confirmation params/resume params, action metadata, signals, and new analysis
  metadata, so old and new runs remain distinguishable.
- The StockSage analysis detail LiveView renders `#stocksage-analysis-rerun`
  with explicit `Native`, `Python`, and `Parity` buttons:
  `#stocksage-rerun-native`, `#stocksage-rerun-python`, and
  `#stocksage-rerun-both`.
- Clicking a rerun button queues a normal `run_analysis` confirmation; it does
  not create a new `stocksage_analyses` row until approval resumes the target
  action.
- Pending rerun confirmations are shown on the originating analysis page by
  matching `params_summary.source_analysis_id`.
- Python remains explicit comparison/reference work. The `Python` and `Parity`
  buttons pass explicit engines through `run_analysis`; bridge and Python
  comparison settings still decide whether the request can proceed.

### M6 Product Polish

- StockSage-owned LiveView shells use consistent mobile-safe roots:
  `overflow-x-hidden`, `px-4 sm:px-6`, breakable headings, wrapped navigation,
  and table overflow guards.
- `AppShell.state_panel/1` now has a success tone so confirmation/reflect/sync
  success states do not render like neutral empty states.
- The analysis detail surface renders `#stocksage-analysis-run-context` with
  stable native/Python/parity affordances:
  `#stocksage-engine-native`, `#stocksage-engine-python`, and
  `#stocksage-engine-parity`. Each exposes `data-run-state` so tests and
  manual verification can distinguish current, explicit, included, available,
  and captured states.
- Rerun controls remain backed by the M5 `run_analysis` confirmation path, but
  their buttons now include engine-specific icons and stable minimum hit
  targets.
- Analysis detail empty states are bounded and stable:
  `#stocksage-outcomes-empty`, `#stocksage-reflections-empty`, and
  `#stocksage-progress-empty`.
- M6 adds no new StockSage domain behavior, schema, actions, security policy,
  or memory writes.
