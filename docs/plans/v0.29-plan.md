# Allbert v0.29 App Memory + Outcomes Contract - StockSage Polish

Request flow: `docs/plans/v0.29-request-flow.md`

Formerly tracked as M-D3b, previously planned as v0.27 before the
project-direction rethink inserted v0.23 Jido State-Machine Convergence and
v0.24 Objective Runtime Foundation. Renamed after the post-v0.26 roadmap
reconciliation to make the contract visible: v0.29 proves memory-consuming app
behavior through StockSage outcomes and lessons.

## Purpose

v0.29 brings StockSage's Elixir implementation to parity with Python
StockSage 0.0.2 user-facing behavior: outcomes, reflections, explicit memory
sync, trends, reruns, and polished responsive UI.

v0.27 declares StockSage's memory namespace as an ownership claim. v0.28 audits
that claim before memory writes exist. v0.29 is the first milestone that
consumes the namespace by syncing resolved StockSage lessons into StockSage
memory records and Allbert memory namespaces through explicit, traceable
actions.

## Memory Sync Write Path (seam spec — pin before implementing M4)

This is the cross-version seam v0.27/v0.28 hand off to. The declaration
(ownership) shipped in v0.27; v0.29 adds the *write*. The write path is the one
mechanism that must be concrete before M4 starts:

- **Write API**: lesson sync calls the existing
  `AllbertAssist.Memory.append/1` (`{:ok, entry} | {:error, term}`) — v0.29 adds
  NO new memory store. Writes carry namespace metadata
  `%{app_id: :stocksage, namespace: :stocksage}` matching the v0.27 declaration;
  `App.Registry.registered_memory_namespaces/0` must list `:stocksage` and the
  namespace must be `writable: true` by the time M4 runs (v0.27 declared it
  `false`; flipping to writable is part of M4 and must be a security-reviewed
  change, not silent).
- **Registered action (the only write entry point)**:
  `AllbertAssist.Actions.Memory.SyncAppLesson` (registered atom
  `:sync_app_lesson`), permission class `:memory_write`, routed through
  `Actions.Runner.run/3` + Security Central. StockSage never calls
  `Memory.append/1` directly; LiveViews/contexts call the registered action.
- **Lesson record shape written**:
  ```elixir
  %{
    namespace: :stocksage,
    kind: :stocksage_lesson,
    analysis_id: String.t(),
    objective_id: String.t() | nil,
    ticker: String.t(),
    rating: String.t(),
    realized_return: float() | nil,
    holding_period_days: pos_integer(),
    lesson_text: String.t(),     # bounded + redacted
    source: :outcome_resolution,
    resolved_at: DateTime.t()
  }
  ```
- **Idempotency**: keyed by `{analysis_id, holding_period_days}`. Re-resolving an
  outcome updates the existing lesson entry; it never appends a duplicate.
- **The no-auto-promotion invariant (tested, not prose)**: completing or
  resolving an analysis MUST NOT call `Memory.append/1`. Only an explicit
  `:sync_app_lesson` invocation writes. M4's test asserts: run analysis → resolve
  outcome → assert zero memory entries exist for the namespace until
  `:sync_app_lesson` is invoked by an operator-facing path.

## Blocker Shapes To Pin Before Implementing

This plan is intentionally just-in-time. Before M-level work starts, pin: the
Memory Sync Write Path above (M4), the outcome-resolver module name + the
`stocksage_outcomes` fields it updates (M1), the trend-metrics computed shape
(M2), the reflection action atom + permission (M3), and the `stocksage.*`
settings keys for holding period / resolver cadence / reflection provider. The
remaining milestone bodies should be expanded to the v0.26 milestone shape
(Goal/Depends/Parallel-safe/tasks/Tests/smoke/Acceptance) at that time.

## Milestones

- M1: Outcome resolver - implement idempotent post-holding-period return
  resolution and objective provenance updates.
- M2: Trends and calibration - add alpha-aware accuracy, rating calibration,
  leaderboard/trend calculations, and fixture-backed expectations.
- M3: Reflection action - generate bounded reflections for resolved outcomes
  through registered actions with redaction and objective context.
- M4: Explicit memory sync - write resolved lessons to StockSage memory records
  and Allbert memory namespaces only through traceable operator-facing actions.
- M5: Rerun flow - add LiveView rerun from an existing analysis while
  preserving engine labels, objective context, and confirmation posture.
- M6: Product polish - complete responsive layouts, empty/error states, and
  bridge/native/parity affordances.
- M7: Closeout - focused tests and full warning gate pass.

## Non-Goals

- No new memory namespace declaration; v0.27 owns declaration.
- No canvas tile work; v0.30 owns durable `/agent` canvas integration.
- No hosted deployment requirement.
- No automatic promotion of every analysis into markdown memory.

## Test Plan

- Outcome resolver updates records idempotently.
- Trends calculations match fixture expectations.
- Reflection generation goes through registered actions and preserves objective
  context.
- Memory sync is explicit, traceable, namespace-scoped, and redacted.
- A tested invariant proves analyses are not automatically promoted into
  markdown memory merely because they completed.
- LiveView rerun, empty, error, and mobile layouts pass focused tests/manual
  review.
- Each milestone has focused tests before closeout.
- Full gate at milestone closeout: compile warnings-as-errors, format check,
  Credo strict, Dialyzer, and `mix precommit`.

## Definition Of Done

v0.29 is done when Python StockSage 0.0.2 user-facing outcomes, trends,
reruns, reflections, and explicit lesson sync are replicated in Elixir without
requiring a Python runtime when native parity has passed, and app memory writes
consume the v0.27 namespace contract under the v0.28 security posture.
