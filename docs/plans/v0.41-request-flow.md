## Main User/Operator Flows

- Operator configures an MCP server for calendar/mail/GitHub via Settings
  Central per v0.40 paths.
- Operator opens the calendar/mail/GitHub workspace panel and sees
  MCP-exposed tools/resources.
- Operator asks the workspace for agenda, inbox summary, GitHub overview, or
  notes search; v0.33 intent descriptors route to the right integration.
- Operator approves any effectful integration action through normal Approval
  Handoff.
- Operator (plugin author) reads `./plugins/allbert.notes_files/` as a
  worked-example starter scaffold.

## Data Flow

Integration requests route through intent descriptors, MCP-configured
workspace panels (driven by the v0.40 MCP client) for calendar/mail/GitHub,
or the native `notes_files` plugin for notes operations. All effectful work
flows through `Actions.Runner.run/3`, Security Central, Resource Access,
confirmations, traces, and audits.

v0.41 consumes the v0.40 first-server validation matrix. Calendar/mail/GitHub
panel refreshes may run without repeated prompts only when the needed data is
available through `mcp_read_resource` and an operator-approved `mcp://` Resource
Access grant. If a server exposes required summary data only through
`mcp_call_tool`, that interaction remains operator-triggered and confirmed per
call, is parked, or requires an ADR 0038 amendment before implementation.

## Authority Invariants

- Integrations do not live in Allbert core.
- MCP-configured panels inherit the v0.40 MCP client trust tier (ADR 0038).
- `:mcp_resource_read` grants never authorize `mcp_call_tool`, even for
  read-like tools.
- Native `notes_files` plugin owns only its declared surfaces/actions/settings
  fragment/memory namespace and never grants permission by registration.
- The `notes_files` memory namespace cannot auto-promote; promotion is a
  separate confirmed action.
- Intent descriptors propose handoff; they do not authorize execution.

## Handles / CLI Surfaces

## Smoke Checklist

- Configure at least one MCP server per integration with fake/test
  credentials.
- Run one read-only flow per integration (agenda, inbox, PR list, notes
  search), and record whether it used `mcp_read_resource` or a per-call-
  confirmed MCP tool.
- Run one confirmed effectful flow per integration (event, reply, comment,
  note write).
- Confirm workspace panels render without leaking secrets.
- Confirm disabled integration actions fail closed.
- Open `./plugins/allbert.notes_files/` and verify it is documented as the
  reference scaffold (`docs/developer/how-to-create-an-allbert-app.md` cross-
  links to it).

## Operator Validation

Status: planned. Commands are illustrative until the milestone lands; verify and
replace with observed output at closeout.

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v041.XXXXXX)"
export ALLBERT_TRACE_ENABLED=true
# Configure one test MCP server per integration (calendar/mail/github) with
# fake credentials through the v0.40 MCP client, then exercise each panel.
mix allbert.ask --trace "show me today's agenda"
mix allbert.ask --trace "summarize my inbox"
mix allbert.ask --trace "list my open PRs"
mix allbert.ask --trace "find notes about onboarding"
PORT=4041 mix phx.server   # then open /workspace and view each summary panel
rg -i 'token|secret' "$ALLBERT_HOME/memory/traces" || true
```

Expected behavior:

- each integration renders a workspace summary panel from configured MCP
  tools/resources; resource reads use a remembered grant, not a per-read prompt.
- one effectful action per integration (create event, reply, comment, write
  note) routes through Approval Handoff before running.
- `./plugins/allbert.notes_files/` is present as the reference scaffold.
- disabled integrations show an empty state; traces leak no secrets.
