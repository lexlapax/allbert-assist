# Allbert v0.41 Developer Flow - Developer Velocity And Parallel Test Methodology

Implementation details live in `docs/plans/v0.41-plan.md`.

This is a developer-facing flow document. v0.41 has no operator-facing assistant
request flow; it defines how humans and agents plan, measure, partition, run,
benchmark, and reorder Allbert development work. The filename keeps the normal
`*-request-flow.md` convention so the roadmap and agent-routing docs can still
find the companion flow document.

Primary design references:

- `docs/adr/0049-development-gates-and-test-parallelization.md`

## Developer Planning Flow

1. A milestone owner starts with the active plan and identifies the touched
   subsystem.
2. The owner checks that the milestone has development-lane annotations:
   parallel workstreams, serial barriers, gate evidence, and rejoin points.
3. The owner chooses a focused gate from the plan rather than defaulting to full
   precommit for every small slice.
4. The owner records whether the slice touches any serial resource:
   SQLite/Repo, global app env, Allbert Home/filesystem, named processes,
   external runtimes, LiveView, or security eval harness.
5. Independent lanes may proceed in parallel only when their owned resources and
   merge/rejoin point are documented.
6. The owner runs the focused gate first.
7. The owner runs the warning/static gate required by the change type.
8. Before release closeout, the owner runs the full release gate.

The v0.41 implementation does not lower any gate. It makes the gate choice
explicit so development is fast when it can be fast and strict when it must be
strict.

## Plan Readiness Flow

When asked to create, audit, or make a milestone plan implementation-ready:

1. Add a development-lane annotation to every milestone.
2. Name safe parallel workstreams, for example docs/request-flow edits, pure
   modules, UI shell work, focused tests, browser/manual validation, and external
   smoke setup.
3. Name serial barriers, for example migrations, SQLite/Repo tests, global app
   env changes, shared Allbert Home roots, named processes, LiveView/Repo
   ownership, security evals, and release gate evidence.
4. Mark which tests/gates can run in parallel and which must run with
   `--max-cases 1`, a serial lane, or a single operator-controlled environment.
5. Define the rejoin point: update docs, check drift against the plan, run the
   focused/serial gate evidence, and decide whether the full release gate is
   needed before commit or only at release closeout.
6. If any milestone lacks this annotation, the plan is not implementation-ready.

For v0.41 itself, the readiness gate is stricter: M1 may start, but M6-M9 are
not implementation-ready until M1 records the authoritative v0.40 oracle and
BEFORE benchmark, and M2-M5 freeze taxonomy, ownership, gate, and migration
sequencing decisions.

## Dependency Compatibility Unblock Flow

If the current developer toolchain cannot run the v0.40 oracle before any
v0.41 test-lane migration begins:

1. Treat the failure as correctness-blocked M1 evidence, not as an efficiency
   result.
2. Record the exact failing command, runtime, dependency, and compiler error in
   `docs/developer/test-strategy.md`.
3. Prefer current upstream dependency releases before local overrides.
4. If an override is required, add or update an ADR because dependency
   compatibility constrains future development work.
5. Validate the unblock with focused compatibility tests and the full release
   gate before continuing M1 baseline work.

The 2026-05-29 unblock follows ADR 0050: Jido is updated to the current stack
and `:memento` 0.5.0 is temporarily served from `vendor/memento` with only the
Elixir 1.19 `record/0` typespec collision renamed.

## Test-Lane Classification Flow

Classify at the case-template boundary first (the template's default lane), then
per module only for overrides and the plain-`ExUnit.Case` tail. For each test
module in the inventory:

1. Read the case template and tags.
2. Detect resource use:
   - database: `DataCase`, `ConnCase`, `Repo`, sandbox ownership
   - app env: `Application.put_env/delete_env`
   - home/filesystem: `ALLBERT_HOME`, `DATABASE_PATH`, `System.tmp_dir!`,
     `File.rm_rf!`, generated roots
   - process: named `Agent`, `GenServer`, `Supervisor`, `Registry`, PubSub
   - external runtime: Docker, browser, stdio ports, provider endpoints
   - security eval: `SecurityEvalCase` or eval inventory rows
3. Assign exactly one primary lane and any secondary blockers.
4. Mark migration action:
   - `already_async`
   - `pure_async_candidate`
   - `needs_unique_home`
   - `needs_partition_database` (DB lane: separate file per partition)
   - `needs_partition_isolation` (other VM-global lane — app env / home / process
     / LiveView — owned roots per partition, serial within)
   - `must_remain_serial`
   - `external_smoke_only`
5. Reconcile: lane counts sum to the full suite total — zero unclassified, no
   double-count. Plugin and channel suites are included; StockSage Python-bridge
   tests are `external_runtime_serial`.

## Planned Gate Shapes

M4 freezes these command names for M6 implementation:

- `mix allbert.test docs`
- `mix allbert.test focused -- <files...>`
- `mix allbert.test fast-local`
- `mix allbert.test serial-core --lane <lane> [--partitions N]`
- `mix allbert.test release`
- `mix allbert.test external-smoke -- <smoke-name>`

The intended shapes are:

```sh
# Docs-only changes
mix allbert.test docs

# Focused milestone gate: explicit files from the plan/request-flow doc
mix allbert.test focused -- <focused files>

# Fast local gate: static checks + async-safe lanes
mix allbert.test fast-local

# Partition-parallel VM-global lanes, after the per-partition roots harness exists.
# Each is a separate OS process / BEAM VM with its own ALLBERT_HOME + DATABASE_PATH,
# so app env, named processes, home roots, the SQLite file, and (server: false)
# LiveView are all isolated. Serial within a partition; parallelism = partition
# count (N from the measured core count). One invocation per partition 1..N.
mix allbert.test serial-core --lane db_partition_safe --partitions N
mix allbert.test serial-core --lane app_env_serial --partitions N
mix allbert.test serial-core --lane home_fs_serial --partitions N
mix allbert.test serial-core --lane global_process_serial --partitions N
mix allbert.test serial-core --lane liveview_serial --partitions N

# Single-VM serial lanes (kept conservative): not-yet-partition-safe DB tests and
# security evals.
mix test --only db_serial --max-cases 1
mix test --only security_eval_serial --max-cases 1

# Authoritative release gate
mix allbert.test release
```

The implementation must preserve the lane semantics and the full release gate.

## Isolation Flow

For partition-safe tests:

1. Derive a partition id from `MIX_TEST_PARTITION`; use `0` when unset.
2. Derive `ALLBERT_HOME` from a temp root plus the partition id.
3. Derive `DATABASE_PATH` inside that home or a sibling partition-specific db
   path.
4. Derive settings, secrets, memory, sandbox, plugin, and tmp roots from that
   home.
5. Run migrations for the partition database before tests.
6. Stop supervised test processes and delete only the owned temp roots.

No test may use a real operator `~/.allbert`, a shared default database, or a
shared root while claiming to be async or partition-safe.

This harness is lane-agnostic: the same per-partition roots make any VM-global
lane (DB, app env, home, named processes, LiveView) partition-parallel, not only
DB. SQLite stays single-writer, so DB tests remain serial within a partition.

## Benchmark Flow

1. Record the BEFORE benchmark in M1 before any test migration implementation:
   release + fast-local-equivalent wall-clock, per-lane breakdown, and
   `--slowest-modules`/`--slowest` reports, on a recorded machine context.
2. Store the benchmark record in `docs/developer/test-strategy.md` with commit,
   machine context, exact commands, counts, wall-clock, lane breakdown, slowest
   modules/tests, planned share, actual delta, decision, and follow-up/reorder.
3. After each implementation milestone (M6-M9), re-run the benchmark and record
   the delta versus the previous milestone and the M1 baseline.
4. If `fast-local` did not improve by the milestone's planned share, or the
   dominant slowest-module is a lane not scheduled next, re-sequence the
   remaining batches toward that hotspot and record the reorder in
   `docs/developer/test-strategy.md`.
5. At closeout, run the after-done benchmark versus the M1 baseline and confirm
   the target is met or document the gap.

Reordering stays within the correctness constraint: every batch still reproduces
the v0.40 oracle green set.

## Failure Flow

If an async/partitioned lane flakes:

1. Re-run the suspect module under the v0.40 serial fallback gate. If it passes
   serially, the defect is parallelism/ownership; if it fails serially, it is a
   real regression and blocks.
2. Move the failing module back to its serial lane.
3. Record the resource conflict in `docs/developer/test-strategy.md`.
4. Add or update a helper contract before trying to parallelize it again.
5. Keep the full release gate authoritative (reproducing the v0.40 oracle) while
   the fast gate is repaired.

If full release precommit fails:

1. Do not hand off as complete.
2. Record exact command, failure, and suspected lane.
3. Fix or ask before deferring, following `DEVELOPMENT.md`.

## Developer-Facing Behavior

v0.41 changes developer methodology, test execution, benchmark evidence, and
project documentation only. The visible behavior is for contributors: they choose
focused gates by lane, run partition-aware test commands, record benchmark
artifacts, and reorder remaining batches when efficiency does not improve.

Operator-facing assistant behavior is unchanged. Operator-facing features resume
in v0.42.
