# v0.3 — Agent harness and intent routing

Status: Shipped

## Context

v0.2 delivered a daemon that multiplexes sessions (ADR 0021), a job manager that can run non-interactive sessions on a schedule, and a unified operator surface under `allbert-cli`. Each session still runs a single monolithic agent loop.

v0.3 is about two things:

1. **Agents as a first-class runtime abstraction.** A root agent per session, with the ability to spawn bounded sub-agents for delegated work. Every turn — root or sub-agent — runs through the same kernel hooks, cost tracking, and policy envelope.
2. **Intent routing as a kernel step.** Before an agent turn is constructed, the kernel classifies the user turn against a bounded taxonomy so the right skills, sub-agents, or services can be engaged without relying on fragile prompt-template routing.

Both changes are runtime-core concerns, so both land in the kernel. This preserves the "Kernel is the runtime core" principle (ADR 0001) while giving skills a future home for richer behaviour (ADR 0031) once the AgentSkills folder format lands in v0.4.

## Goal

Ship an agent harness and intent-routing substrate that:

- lets a root agent spawn one level of sub-agents via a built-in `spawn_subagent` tool;
- attributes every sub-agent turn to its agent via `HookCtx.agent_name` and `HookCtx.parent_agent_name`;
- classifies user input against a fixed taxonomy `{task, chat, schedule, memory_query, meta}` with a rule fast-path and an LLM sub-call fallback;
- exposes new hook points `BeforeAgentSpawn`, `AfterAgentSpawn`, `BeforeIntent`, `AfterIntent`, `BeforeJobRun`, `AfterJobRun` on the existing hook API (ADR 0006);
- leaves the accepted v0.1 hook names intact and extends them additively rather than renaming them;
- remains interchangeable with v0.2 runtime behaviour for sessions that do not use sub-agents.

## Current usability assessment

v0.3 is shipped for the same technical source-based user targeted by v0.2.

- Sessions now expose first-class agent and intent state without requiring the user to understand kernel internals.
- `/status` surfaces the root agent, the last active agent stack, and the last resolved intent for the attached session.
- `allbert-cli agents list` and `~/.allbert/AGENTS.md` provide a readable roster of contributed agents.
- The shipped example skill includes explicit `intents:` metadata, validating the preview path before v0.4 standardizes the full AgentSkills format.

## Agent model

- `Agent` is a kernel trait. Each session has exactly one root agent that the kernel constructs on `create_session` (ADR 0021).
- A new built-in tool `spawn_subagent(name, prompt, context)` lets an orchestrator agent spawn a sub-agent inline. The sub-agent's result flows back as tool output.
- v0.3 forbids recursive spawns; sub-agents cannot spawn their own sub-agents. This keeps v0.3 bounded and makes the recursion guard a kernel-level check rather than a skill concern.
- Sub-agents inherit the session's singletons (hook registry, tool registry, provider pool, confirm-trust state) but get their own message history, turn counter, and cost accumulator.
- Cost logs and trace spans attribute every turn to `agent_name`.

See [ADR 0029](../adr/0029-agents-are-first-class-runtime-participants.md).

## Intent routing

- Kernel step that runs before agent turn construction.
- Bounded taxonomy in v0.3: `{task, chat, schedule, memory_query, meta}`.
- Two-stage classifier: rule fast-path (keyword/shape heuristics) first; LLM sub-call fallback when rules are ambiguous.
- `BeforeIntent` and `AfterIntent` hook points.
- Resolved `Intent` lands on `HookCtx` and is surfaced as declarative context to agents and skills. It is a hint, not a hard gate.
- Classifier output is cacheable per turn; the same input is not reclassified within a session.

See [ADR 0030](../adr/0030-intent-routing-is-a-kernel-step-not-a-skill-concern.md).

## Hook system additions

`HookCtx` gains:

- `agent_name: String` — the agent handling the current turn.
- `parent_agent_name: Option<String>` — populated for sub-agent turns.
- `intent: Option<Intent>` — resolved intent, `None` during `BeforeIntent`.

New hook points:

- `BeforeIntent(ctx, input)` → may override the classifier output.
- `AfterIntent(ctx, intent)` → observational.
- `BeforeAgentSpawn(ctx, spawn_request)` → may deny or rewrite a spawn.
- `AfterAgentSpawn(ctx, spawn_result)` → observational.
- `BeforeJobRun(ctx, job_id)` → fires when the job manager launches a scheduled session.
- `AfterJobRun(ctx, job_result)` → observational.

Existing hook points (`BeforePrompt`, `BeforeModel`, `BeforeTool`, `AfterTool`, `OnModelResponse`, `OnTurnEnd`) all fire for sub-agent turns unchanged.

v0.3 does not rename the accepted hook surface from ADR 0006. Exec observability still flows through `BeforeTool` / `AfterTool` on `process_exec`; any future dedicated `BeforeExec` / `AfterExec` points would be additive and require their own ADR.

## Skill frontmatter additions (preview)

v0.3 previews two new optional frontmatter fields on the current minimal `SKILL.md`-centric skill shape, implemented as additive keys so v0.2 skills keep loading:

- `intents: [task, schedule]` — hints to the intent router that the skill is relevant when one of the listed intents is resolved. Tier 1 discovery (ADR 0036) will formalise this in v0.4.
- `agents:` — per-skill agent contributions. v0.3 accepts the field and wires contributions into the kernel agent registry, but the full AgentSkills validation, install, and folder-home story for those same keys lands in v0.4 (ADR 0031).

## Authoring experience

Consistent with the natural-interface principle (ADR 0038), v0.3 introduces no code-authoring requirement for users:

- **The `Agent` trait is kernel-internal Rust.** It is runtime scaffolding, not a user surface.
- **User-authored agents are markdown prompt files**, contributed through skill frontmatter per ADR 0031. In v0.3 they live next to the current `SKILL.md`-centric skill shape; in v0.4 the same contribution moves into canonical AgentSkills folders. No Rust involved.
- **Orchestrator behaviour is expressible in the skill's `SKILL.md` body**, not in code. An orchestrator agent's instructions on when to call `spawn_subagent` live in its markdown prompt, which the kernel runs through the same prompt pipeline as any other agent.
- **Intent routing is kernel-internal.** Users do not configure the classifier beyond the small config keys in `[intent_classifier]`.

## AGENTS.md in the bootstrap bundle

Per ADR 0039, v0.3 adds `AGENTS.md` to the kernel-owned bootstrap bundle established by ADR 0010.

- The kernel regenerates `~/.allbert/AGENTS.md` on daemon startup and whenever the skill catalog is refreshed.
- In v0.3 that means startup plus any explicit skill-file reload or catalog rebuild. Once v0.4 lands install/remove/update flows, those lifecycle events trigger the same regeneration path.
- Content: one section for the root agent (`allbert/root`) plus one section per contributed agent (`<skill>/<agent>`), listing namespaced name, one-line description, `allowed-tools` summary, contributing skill, and optional model override.
- The file is read-only from the user's perspective; a leading comment in the file says so and points at the skill authoring path instead of direct edits: local skill files in v0.3, `allbert-cli skills install/remove` once v0.4 ships.
- `AGENTS.md` is injected into every turn's prompt alongside `SOUL.md` / `USER.md` / `IDENTITY.md` / `TOOLS.md`, bounded by the bootstrap prompt budget.
- `HEARTBEAT.md` remains deferred — revisit in v0.6 channel expansion.

## Runtime defaults

- Root agent in a fresh session: `allbert/root` — a small, stable default agent with the project's canonical system prompt.
- Classifier model: uses the session's effective model by default; overridable via `config.intent_classifier.model`.
- Classifier budget: per-turn token budget capped in config; exceeding the budget short-circuits to the rule classifier only.

## Config schema deltas

```toml
[agents]
max_subagent_depth = 1          # v0.3 enforces 1, exposed for future releases
subagent_turn_limit = 12        # per sub-agent turn cap

[intent_classifier]
enabled = true
model = ""                      # empty = use session effective model
rule_only = false               # true to skip the LLM fallback
per_turn_token_budget = 2000
```

## In scope

- Agent trait + root agent construction + `spawn_subagent` built-in tool.
- Intent classifier with rule fast-path and LLM fallback.
- Hook surface additions (context fields + new points).
- Skill frontmatter preview for `intents` and `agents` (additive, backward compatible).
- Cost/trace attribution to agent names.
- Tests: sub-agent spawn correctness, recursion refusal, classifier taxonomy coverage, hook event ordering.
- Docs: operator docs for agent visibility and intent inspection.

## Out of scope

- AgentSkills folder format adoption (v0.4, ADR 0032).
- Skill install preview + confirm flow (v0.4, ADR 0033).
- Multi-level sub-agent recursion.
- Curated memory retrieval (v0.5).
- New channels (v0.6).

## Milestones

### M1 — Agent trait and root agent

- Define `Agent` trait in `allbert-kernel`.
- Construct a root agent on `create_session`.
- Thread `agent_name` through turn execution, cost logs, and trace spans.
- Update all existing hook points to populate `agent_name` and `parent_agent_name`.
- Tests: root agent is always named, metadata flows to hooks and logs, existing sessions behave identically.

### M2 — Sub-agent spawn

- Implement `spawn_subagent(name, prompt, context)` as a built-in tool.
- Enforce recursion bound = 1 at the kernel level.
- Sub-agent turns run with a fresh message history, own turn counter, own cost accumulator.
- Add `BeforeAgentSpawn` / `AfterAgentSpawn` hook points.
- Tests: spawn/cleanup correctness, recursion refusal, cost attribution per agent, policy envelope respected.

### M3 — Intent classification

- Build rule-based fast-path classifier over the taxonomy.
- Add LLM sub-call fallback using the session's effective model by default.
- Add `BeforeIntent` / `AfterIntent` hook points.
- Plumb resolved `Intent` through `HookCtx` and into skill selection heuristics.
- Config-driven enable/disable and budget controls.
- Tests: taxonomy coverage, rule fast-path correctness, fallback path, budget enforcement, observability via hooks.

### M4 — Skill frontmatter preview

- Extend skill frontmatter parser to accept `intents` and `agents` fields.
- Wire `intents` into intent-aware skill selection hints.
- Wire `agents` into the kernel agent registry (adjacent markdown file within the current minimal skill directory shape; v0.4 later normalises the same key into AgentSkills folders).
- Add job-run hooks (`BeforeJobRun`, `AfterJobRun`) since the job manager now coordinates with the agent surface.
- Tests: frontmatter backward compatibility, agent contribution lifecycle, intent-to-skill match.

### M5 — AGENTS.md bootstrap integration

- Extend the bootstrap prompt pipeline (ADR 0010) to include `AGENTS.md`.
- Implement the kernel-side generator that rebuilds `AGENTS.md` on startup and on skill-catalog refresh events; v0.4 install/remove/update flows must call the same regeneration path.
- Add CLI surface `allbert-cli agents list` that reflects the same data the file surfaces.
- Tests: regeneration on startup and skill-catalog refresh, namespacing correctness, prompt-budget accounting, leading read-only comment present.

### M6 — Closeout

- Update operator docs (CLI commands for inspecting the last active agent stack, last-resolved intent, and `AGENTS.md` contents).
- Migrate bundled skills to add `intents:` where relevant.
- Update `docs/vision.md` references to agent/intent directions.
- Mark ADRs 0029, 0030, 0031, 0038, 0039 Accepted.

## Testing and verification

- Unit tests for classifier rule paths and taxonomy exhaustiveness.
- Integration tests exercising orchestrator-style prompts that spawn sub-agents.
- Recursion-bound test: sub-agent attempting a nested spawn fails cleanly at the kernel boundary.
- Hook ordering tests: `BeforeIntent` → `AfterIntent` → `BeforePrompt` → ... → `OnTurnEnd` for root; sub-agent turns fire with `parent_agent_name` set.
- Cost attribution test: session cost = sum of per-agent costs.
- Scheduled-job smoke test: jobs launched by the daemon fire `BeforeJobRun`/`AfterJobRun` and attribute correctly to their agent.
