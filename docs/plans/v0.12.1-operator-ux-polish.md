# v0.12.1 - Operator UX polish

Status: Draft, implementation-ready

## Summary

v0.12 shipped powerful self-improvement, skill-authoring, scripting, inbox, and review surfaces. The first operator test showed that the runtime can do the work, but the interactive experience still makes users guess what is happening, leave the TUI too often, or know CLI commands from memory.

v0.12.1 is a point release focused on operator confidence. It has three equal release-blocking pillars: shared activity and stuck-state awareness, responsive in-flight TUI behavior, and a first-class settings/command hub. The release also folds in the earlier usability-audit work: legible cost/context/memory/skill state, richer approvals, channel-native Telegram, recovery affordances, discovery, setup resume, error hints, and release-ready docs.

The release includes one narrow protocol bump to v3 so TUI, classic REPL, CLI, Telegram, jobs, and future channels can consume the same daemon-owned `ActivitySnapshot` instead of guessing from frontend timers. v3 is the live activity surface, not a persisted trace or replay protocol. It deliberately does not change self-improvement safety invariants, introduce cancellation, or change classic REPL semantics beyond shared activity/help output.

## Goals

- Keep the TUI responsive during in-flight turns and make activity visible with a spinner, elapsed timer, activity label, current tool summary, result breadcrumb, Tier-2 skill activation badge, and visible input caret.
- Make runtime activity and stuck-state awareness daemon-owned and available across channels through `ActivitySnapshot`.
- Let operators type the next-turn draft while a turn is running without accidentally sending a concurrent turn or leaking draft text into approval/input modals.
- Add a scalable command hierarchy and settings hub as a release-blocking operator surface, so setup choices, daily operations, and later customization share one mental model.
- Make v0.12 workflows discoverable inside the TUI: inbox approvals, staged memory, skills, and self-improvement review/install/garbage-collection paths.
- Make cost, context pressure, memory composition, `AGENTS.md`, and skills legible inline without requiring file spelunking.
- Make approvals explain what, why, and what changes across TUI, classic REPL, CLI, and Telegram.
- Add recovery affordances for memory deletion, staged-memory rejection, skill disablement, and bad config edits.
- Improve discovery and error quality with command grouping, suggestions, examples, setup resume, and remediation hints.
- Make Telegram feel channel-native with typing indication, markdown-aware output, capability-honest fallbacks, and approval feedback.
- Reduce first-run confusion with clearer setup headings, defaults, Ollama readiness guidance, trusted-root wording, a final setup summary, and "what next" commands.
- Make high-value CLI help output self-explanatory for `self-improvement`, `inbox`, `daemon channels`, and `repl`.
- Preserve v0.12 invariants: no automatic patch install, no binary auto-swap, no Lua auto-enable, no hidden config mutation, and no concurrent turn queue.

## Non-goals

- No broad daemon protocol redesign. The v3 bump is limited to additive activity snapshot/update messages and optional inclusion in operator snapshots.
- No turn cancellation. `Ctrl-C` remains non-cancel in v0.12.1 and must say so clearly.
- No concurrent-turn queue. Text typed during an in-flight turn is only a local next-turn draft.
- No transcript scroll, search, copy, or command palette.
- No theming, remappable keys, or full accessibility mode beyond `spinner_style = "off"` as a reduced-motion/static indicator option.
- No arbitrary free-form config editing through the TUI. Settings changes route through a typed allowlist and shared validation.
- No setup `--from-file`, hosted web UI, voice, multimodal status surface, or foundation-model work.
- No memory or session migration. New recovery directories and setup state are created lazily.
- No durable trace persistence, replay, or trace export. v0.12.1 emits bounded live activity transitions; v0.12.2 owns persisted spans and replay surfaces.
- No new ADR. Existing ADRs for TUI boundaries, self-improvement approval, scripting policy, telemetry ownership, and the self-change envelope remain authoritative.
- No classic REPL spinner/caret. Classic may gain shared slash commands, approval context, activity transition text, and help improvements.

## Plan hardening decisions

These details are release-blocking so implementation does not have to invent policy while landing the UX work.

### Protocol compatibility

Protocol v3 is a narrow additive bump, but v0.12.1 must keep the currently shipped v2 clients usable. A v3 daemon accepts `ClientHello.protocol_version` values `2` and `3`, stores the negotiated client protocol version on the connection, and filters outbound serialization through that value. v2 peers receive only v2-safe messages and payload shapes: no `ActivityUpdate`, no direct `ActivitySnapshot` response, and no v3-only fields such as `TelemetrySnapshot.current_activity` or approval `context`. v3 peers receive the full v0.12.1 surface.

The v3 daemon rejects protocol versions below `2` or above `3` with `ProtocolError { code: "version_mismatch", ... }` and an operator-facing remediation that names whether to upgrade the client or daemon. A v3 client connected to a v2 daemon surfaces the existing mismatch as `upgrade the daemon to v0.12.1 to use activity surfaces` and does not fabricate local activity state.

M1 must include handshake tests for v2 and v3 clients against a v3 daemon, a future-version rejection test, and a broadcast-filter test proving a v2 peer never receives v3-only activity messages.

### Activity event ownership

Activity truth flows through one path. The kernel emits bounded internal `KernelEvent::Activity(ActivityTransition)` events around context assembly, intent classification, provider calls, streaming, tool calls, approvals, input waits, validation, scripting, finalization, terminal errors, and turn completion. These internal activity events are consumed by the daemon; they are not forwarded as ordinary transcript events and do not add a new `KernelEventPayload` client event.

The daemon converts activity transitions into per-session `ActivitySnapshot` state, computes elapsed time and stuck hints, broadcasts `ServerMessage::ActivityUpdate` only to v3 peers, and includes `current_activity` only in v3-safe telemetry/operator snapshots. Frontends render the daemon state and may animate local spinners/carets, but they must not infer runtime phase, stuck hints, or next actions from local timers.

### Settings persistence

Settings writes use a typed registry plus path-preserving TOML edits. The implementation should add an explicit `toml_edit` dependency in the crate that owns settings persistence, edit only allowlisted config paths, and persist with the existing atomic-write posture. It must not reserialize the whole `Config` as a fallback for settings changes, because that would erase comments and unrelated formatting.

If a supported setting cannot be edited while preserving unrelated TOML structure, the command fails with a remediation hint that points to manual `config.toml` editing for that key. Unsupported keys, secrets, path escapes, and reserved runtime directories are rejected before any write.

### Recovery schemas

Recovery features write reviewable markdown/TOML metadata rather than opaque tombstones. Durable-memory trash entries live at `memory/trash/<timestamp>-<id>.md` with original id, original path, deleted_at, and delete_reason when available. Restoring refuses to overwrite an existing live memory id; v0.12.1 should prefer a clear collision error over silent rename.

Soft-rejected staged entries live at `memory/reject/<session>/<timestamp>-<id>.md` with original staged id, source session, rejected_at, and reason. `memory reconsider <id>` moves the entry back to the staged pool only when there is no active staged entry with the same id.

Installed skill disablement is represented in skill install metadata, not by deleting or editing the skill body. The loader must treat missing `enabled` as `true` for existing skills. `config.toml.last-good` is written only after successful daemon config load/start, restored atomically, and never overwrites the broken config without first keeping a timestamped copy for operator inspection.

### Milestone shape

The original M0 is split into M0a and M0b. M0a lands command/settings descriptors and validation without file mutation. M0b lands path-preserving settings persistence. M1 and later keep their existing order.

### Future extension seams

v0.12.1 must leave the activity, settings, and setup surfaces open for additive point releases without requiring a redesign. The settings registry and setup state file must accept future groups and steps, including a later `trace` group and trace setup step, without hard-coding a closed enum or fixed terminal step order. Unknown future groups are ignored by v0.12.1 readers unless they are explicitly read through a typed descriptor.

Activity transitions remain live-state inputs in v0.12.1. They may later feed durable span emission, but v0.12.1 does not choose trace storage, trace retention, replay schema, OTLP export, or trace privacy defaults. Protocol v3 remains the live activity protocol; future trace protocol messages must be additive and separately negotiated.

## Public interfaces

### Config

Add optional TUI config under `[repl.tui]`:

```toml
[repl.tui]
spinner_style = "braille" # "braille" | "dots" | "bar" | "off"
tick_ms = 80              # clamped to [40, 250]

[operator_ux.activity]
stuck_notice_after_s = 30
long_tool_notice_after_s = 20
show_activity_breadcrumbs = true

[memory]
trash_retention_days = 30
reject_retention_days = 30

[setup]
state_path = "setup-state.toml"
```

Existing profiles load unchanged with defaults. Invalid `spinner_style` values fall back to `braille` with an operator-visible warning. `tick_ms` values outside `[40, 250]` warn and clamp at config load.

`operator_ux.activity.stuck_notice_after_s` applies to quiet model calls, approval waits, input waits, validation runs, scripts, and daemon-event silence after turn start. `long_tool_notice_after_s` applies to `calling_tool` and tool-like script/validation phases. Values below 5 seconds warn and clamp to 5; values above 600 seconds warn and clamp to 600. `show_activity_breadcrumbs = false` hides transcript breadcrumbs in channels that support them, but does not disable activity tracking or `allbert-cli activity`.

`memory.trash_retention_days` controls soft-deleted durable memory retention. `memory.reject_retention_days` controls soft-rejected staged memory retention. `setup.state_path` is relative to `ALLBERT_HOME`, must stay inside the profile root, and defaults to `setup-state.toml`.

### Settings hub

Add a typed settings registry in the kernel/config layer and expose it through TUI and CLI adapters. This is a v0.12.1 release blocker, not a follow-up nicety. The registry owns labels, descriptions, value types, defaults, validation, privacy/safety notes, restart requirements, and config paths.

Initial settings categories:

- `ui`: `repl.ui`, `repl.tui.spinner_style`, `repl.tui.tick_ms`, status-line item toggles, and activity breadcrumb display.
- `activity`: `operator_ux.activity.stuck_notice_after_s`, `operator_ux.activity.long_tool_notice_after_s`, and `operator_ux.activity.show_activity_breadcrumbs`.
- `memory`: memory routing mode, semantic retrieval enablement, staged/episode auto-prefetch policy where already configurable, trash retention, and reject retention.
- `learning`: personality digest enablement and bounded episode-summary limits.
- `self_improvement`: source checkout, sibling worktree path, install policy, scripting posture, and cleanup policy.
- `setup`: setup resume state path and setup-summary guidance where safe to expose.
- `providers`: non-secret provider/model defaults that can be safely displayed. Secrets remain outside the settings hub and keep their existing secret-management path.

Public commands:

| Command | Behavior |
| --- | --- |
| `/settings list [group]` / `allbert-cli settings list [group]` | List supported settings with current value, default, and restart requirement |
| `/settings show <key>` / `allbert-cli settings show <key>` | Explain one setting, including config path, accepted values, default, safety note, and whether it can be changed live |
| `/settings set <key> <value>` / `allbert-cli settings set <key> <value>` | Validate and persist one supported setting through the registry |
| `/settings reset <key>` / `allbert-cli settings reset <key>` | Remove the explicit override and fall back to default |
| `/settings explain <group>` / `allbert-cli settings explain <group>` | Show why the group exists and common examples |

Settings writes must be atomic, preserve unrelated TOML comments and formatting for successful writes, and reject unsupported keys, secrets, path escapes, and reserved runtime directories. If preservation cannot be guaranteed for a supported key, the command fails with a remediation hint instead of reserializing the whole config. The settings hub may point advanced operators to manual TOML editing for unsupported values, but it must not provide an unsafe generic editor in v0.12.1.

### Daemon protocol

Bump `PROTOCOL_VERSION` to `3` with additive activity messages and fields.

Add:

- `ActivityPhase`: `idle`, `queued`, `preparing_context`, `classifying_intent`, `calling_model`, `streaming_response`, `calling_tool`, `waiting_for_approval`, `waiting_for_input`, `running_validation`, `running_script`, `finalizing`, and `error`.
- `ActivitySnapshot`: `phase`, `label`, `started_at`, `elapsed_ms`, `session_id`, `channel`, optional `tool_name`, optional `tool_summary`, optional `skill_name`, optional `approval_id`, optional `last_progress_at`, optional `stuck_hint`, and optional `next_actions`.
- `TelemetrySnapshot.current_activity: Option<ActivitySnapshot>`.
- `ServerMessage::ActivityUpdate(ActivitySnapshot)` for broadcast updates.
- `ClientMessage::ActivitySnapshot` and `ServerMessage::ActivitySnapshot(ActivitySnapshot)` for direct CLI/channel reads.
- `ApprovalContext` enum plus optional `context: Option<ApprovalContext>` fields on `ConfirmRequestPayload` and `InboxApprovalPayload`.

If `OperatorSnapshotPayload` lands in the same v0.12.1 operator-surface work, include `current_activity: Option<ActivitySnapshot>` there too. If it does not, `TelemetrySnapshot.current_activity` plus the direct activity request are the canonical v0.12.1 surfaces.

Backward compatibility:

- A v2 client connecting to a v3 daemon must continue working. The daemon accepts protocol versions `2` and `3`, tracks the negotiated protocol version per connection, and filters outbound messages and fields for that peer.
- A v2 peer must never receive `ActivityUpdate`, `ActivitySnapshot`, `TelemetrySnapshot.current_activity`, approval `context`, or any other v3-only field/message. Existing v2 requests and responses keep their v2 shape.
- A v3 daemon rejects protocol versions below `2` or above `3` with an actionable `version_mismatch` error naming whether the operator should upgrade the client or daemon.
- A v3 client connecting to a v2 daemon must surface a clear version mismatch with operator-facing remediation (`upgrade the daemon to v0.12.1 to use activity surfaces`) and must not invent activity snapshots locally to fill the gap.
- `ActivityPhase` and `ActivitySnapshot` are additive over time; new phases or fields must be added at the end of their respective enums/structs and clients must tolerate unknown variants without panicking.
- M1 must include roundtrip and handshake tests using v2 envelopes against the v3 daemon to prove old payloads still parse and v3-only broadcasts are suppressed for v2 peers.

Activity snapshots are safe operational summaries. They must not include raw prompts, secret values, full tool inputs, raw tool output, transcript excerpts, or provider payloads.

Approval contexts are also safe summaries. They may include bounded, kind-specific fields such as approval kind, tool name, cwd, validation status, file stats, cost delta, job schedule, staged-memory preview, and the first 10 diff lines for patch approvals. They must not include secrets, full command environments, raw transcript excerpts, or unbounded tool output.

### Runtime behavior

- While a turn is in flight, key input edits `composer_input` as a next-turn draft.
- `Enter` while a turn is in flight must not send a second `RunTurn`; it appends a local notice and preserves the draft.
- `Ctrl-C` while a turn is in flight must not imply cancellation; it appends a local notice that cancel is unavailable.
- Confirmation modals must not use `composer_input`. They use key choices only.
- Input modals must read and write `modal_input`; normal draft text survives modal open, submit, cancel, and close.

## Implementation design

### TUI event loop

Replace the blocking `event::poll` plus turn-specific receive loop in [tui.rs](../../crates/allbert-cli/src/tui.rs) with one async event loop that selects over:

- daemon messages from `client.recv()`;
- key events from `crossterm`'s async `event-stream` feature;
- a `tokio::time::interval` tick using clamped `tick_ms`.

The loop should draw when state is dirty, a turn is in flight, or a transient badge is active. Idle screens should not redraw continuously.

If enabling `crossterm` event-stream introduces a dependency or platform problem during implementation, stop and update this plan instead of silently switching architecture.

The TUI owns animation, caret, draft buffering, modal state, and local transcript rendering. It may show a disconnected notice if activity cannot be refreshed, but it must not invent runtime phases or stuck hints from local timers. The current runtime phase, elapsed time, stuck hint, and next actions come from daemon `ActivityUpdate` messages or a direct `ActivitySnapshot` refresh.

### Shared activity state

Add the shared activity model in `allbert-proto`, with daemon-owned lifecycle management:

```rust
enum ActivityPhase {
    Idle,
    Queued,
    PreparingContext,
    ClassifyingIntent,
    CallingModel,
    StreamingResponse,
    CallingTool,
    WaitingForApproval,
    WaitingForInput,
    RunningValidation,
    RunningScript,
    Finalizing,
    Error,
}

struct ActivitySnapshot {
    phase: ActivityPhase,
    label: String,
    started_at: DateTime<Utc>,
    elapsed_ms: u64,
    session_id: String,
    channel: String,
    tool_name: Option<String>,
    tool_summary: Option<String>,
    skill_name: Option<String>,
    approval_id: Option<String>,
    last_progress_at: Option<DateTime<Utc>>,
    stuck_hint: Option<String>,
    next_actions: Vec<String>,
}
```

Ownership rules:

- Kernel emits precise internal activity transitions around context assembly, intent classification, model calls, response streaming, tool calls, script runs, validation runs, approval waits, input waits, finalization, terminal errors, and turn completion. The implementation adds an internal `KernelEvent::Activity(ActivityTransition)` event and keeps protocol-visible `KernelEventPayload` unchanged.
- Daemon stores the latest `ActivitySnapshot` per session, computes `elapsed_ms`, attaches advisory `stuck_hint` values, broadcasts `ActivityUpdate`, and includes the current snapshot in telemetry/operator snapshots.
- Frontends render activity only. TUI, classic REPL, Telegram, and future channels must not scrape logs or infer stuck state from local frontend timers.
- Activity snapshots must return to `idle` after `TurnResult`, after a terminal turn error has been surfaced, or after a job finishes.
- Activity labels and summaries are operator-facing but conservative. They summarize operational state and never include raw prompt text, secrets, full command lines containing arguments likely to contain secrets, raw tool output, or provider payloads.

Expected turn transitions:

| Runtime point | Activity phase |
| --- | --- |
| No active turn (waiting on operator to type, no scheduled job) | `idle` |
| Turn accepted but not yet running | `queued` |
| Bootstrap, memory, telemetry, skill context assembly | `preparing_context` |
| Intent/router pass | `classifying_intent` |
| Provider request sent and no tokens received yet | `calling_model` |
| Provider tokens or chunks are arriving | `streaming_response` |
| Kernel invokes a tool | `calling_tool` |
| Inbox/approval gate is open | `waiting_for_approval` |
| Kernel asks operator/channel for input | `waiting_for_input` |
| Tier A or other validation command runs | `running_validation` |
| Embedded or policy-governed script runs | `running_script` |
| Turn result, memory staging, cost/session persistence, or cleanup is completing | `finalizing` |
| Terminal error before normal completion | `error`, then `idle` after the error is delivered |

Tool summaries are derived from a conservative extractor table: file paths for read/write tools, first command token plus redacted arguments for process tools, query labels for memory/search tools, validation profile name for test/clippy/build steps, and `...` for unknown tools. Summaries are observability only, not validation.

### Stuck awareness

Stuck state is advisory, daemon-side, and truthful about what is known. It must not imply cancellation exists.

Threshold behavior:

- `calling_model` older than `stuck_notice_after_s` with no provider progress sets a hint such as `calling model for 42s; no provider tokens yet`.
- `calling_tool`, `running_script`, or `running_validation` older than `long_tool_notice_after_s` sets a hint such as `running cargo clippy for 95s`.
- `waiting_for_approval` sets a hint once older than `stuck_notice_after_s`, including `approval_id` when available.
- `waiting_for_input` sets a hint once older than `stuck_notice_after_s`, naming the channel that must answer.
- No daemon event after turn start for more than `stuck_notice_after_s` sets a hint such as `turn started 31s ago; no daemon progress event received`.
- `next_actions` must be accurate for the current phase: wait, inspect logs, resolve approval, provide input, exit/detach, or restart after confirming the daemon is unhealthy. Do not list cancel as an action in v0.12.1.

### Rendering

When in flight, prepend a transient status segment before configured status-line items:

```text
<spinner> calling model 4.2s | model gemma4 | ctx 1240/12% | cost $0.0042 | inbox 1
```

Compact terminals should preserve signal over decoration. Below 60 columns, render spinner plus activity label and omit elapsed time if needed.

Render a visible caret with `frame.set_cursor_position(...)` in the input pane when the composer is active and in modal input when an input modal is active. Cursor blink is delegated to the host terminal; Allbert only sets the cursor position. Do not animate or simulate a blinking cursor inside the renderer, since terminals that disable blink for accessibility or recording reasons must keep that choice.

Render dropped events as compact transcript or status artifacts:

- `ToolCall` renders `> read_file(foo.rs)`.
- `ToolResult` renders `< read_file: 42 lines, 2.1KB` or a truncated first content line.
- `SkillTier2Activated` renders a transient `activated: memory-curator` badge for about 1.5 seconds.
- `Cost` and `TurnDone` continue to update telemetry without adding transcript noise.

### Operator legibility

Unify compact status and inline commands across TUI, classic REPL, and CLI:

- Costs render to six decimals on user-visible status surfaces.
- Context renders as `light`, `moderate`, `heavy`, or `full` when the context window is known, and as `cap unknown` when it is not.
- Memory renders with labels such as `durable`, `staged`, `episode`, and `fact`; avoid opaque abbreviations as the only representation.
- `/context` shows context-window details and a plain-language recommendation.
- `/agents` shows the generated `AGENTS.md` routing summary inline.
- `/skills` and `/skills search` show names, descriptions, source/provenance, enabled state, and activation/routing hints.
- Skill install output includes a one-line capability summary and routing impact hint drawn from skill metadata.

### Approval context

`ApprovalContext` powers TUI modals, classic REPL prompts, CLI `inbox show`, and Telegram approval messages.

Context blocks are kind-specific:

- `tool_confirm`: tool name, cwd, redacted argument summary, and why-this-matters hint when available.
- `cost_cap_override`: requested increase, current daily total, configured cap, and reason.
- `job_approval`: job kind, schedule, next fire time, and recurrence summary.
- `patch_approval`: branch, validation status, file stats, artifact path, and first 10 diff lines.
- `memory_promotion`: staged entry preview, source, and supersession hint.

Renderers use the same three-section shape: summary, context, and next steps. Telegram must fail closed for inline approval if it cannot present enough context for an informed decision.

### Channel-native Telegram behavior

Telegram uses the same daemon activity state as other channels and adds channel-native rendering:

- Send typing indication roughly every four seconds while a session is in flight.
- Use markdown formatting where supported and split long messages at safe markdown boundaries.
- Send explicit approval-resolution feedback back to the chat.
- Gate richer behavior on declared channel capabilities; channels without a capability render a clear fallback or refusal instead of silently dropping context.

### Recovery and reversibility

Recovery features preserve operator trust without weakening review gates:

- `memory forget --confirm` moves durable memory to `memory/trash/<timestamp>-<id>.md`; `memory restore <id>` restores it.
- `memory reject <id>` moves staged entries to `memory/reject/<session>/<timestamp>-<id>.md`; `memory reconsider <id>` returns them to staging.
- `skills disable <name>` sets installed skill metadata to disabled without deleting the directory; `skills enable <name>` reverses it, and loaders honor the flag.
- Daemon writes `config.toml.last-good` after successful config load/start; `config restore-last-good` restores it after bad edits.
- Trash/reject cleanup follows configured retention and is safe to run from maintenance jobs.

File/schema details:

- Durable-memory trash entries include original id, original path, deleted_at, and delete_reason when present. Restore refuses live-id collisions with a clear error.
- Staged-memory reject entries include original staged id, source session, rejected_at, and reason when present. Reconsider refuses active staged-id collisions.
- Skill disablement lives in installed-skill metadata. Missing `enabled` means enabled for backward compatibility.
- `config.toml.last-good` is refreshed only after a successful daemon config load/start. Restore keeps the broken config as a timestamped copy before atomically replacing `config.toml`.

### Discovery, setup, and error hints

Discovery improvements are deliberately small and non-modal:

- Unknown slash commands suggest the closest registered command only when edit distance is at most 2.
- Partial flag entry such as `<command> --` may render a one-line argument hint where the command catalog knows valid flags.
- High-value CLI help includes examples.
- Setup validates each prompt before moving on, shows recommended defaults, writes resumable setup state after each accepted answer, and supports `allbert-cli setup --resume`.
- The top 20 user-reachable errors get remediation hints. Hints may be string-appended in v0.12.1, with structured error fields deferred.

### Shared rendering helpers

Extract CLI rendering helpers for inbox, memory staged entries, skills, and self-improvement outputs where that keeps CLI and TUI consistent. The helpers should return strings or lightweight view models, not write directly to stdout, so tests can assert output without terminal coupling.

### Command and settings ownership

The TUI is only one channel. Command catalog, settings metadata, and activity state must be reusable:

- Kernel/config owns the settings registry, validation, typed parsing, redaction, defaults, and persistence.
- CLI and TUI own command parsing for their channel, but both render from shared command descriptors and lightweight view models where practical.
- Daemon owns runtime activity truth and exposes it over protocol v3; channels consume that state instead of deriving it from logs, transcripts, or frontend timers.
- Telegram can read status/activity from daemon but must not gain broad settings mutation in v0.12.1. Settings changes remain local CLI/TUI operations with the same profile filesystem trust assumptions as existing setup/config commands.
- Setup should finish by showing which settings group changes each answer later, for example `Change UI mode later with /settings set repl.ui classic`.

- Approval context appears for tool, cost, job, patch, and memory-promotion approvals without exposing secrets.
- `/settings` and `allbert-cli settings` can list, explain, set, and reset supported settings with validation.
- `/help` uses the documented command hierarchy rather than a flat command dump.
- Unknown slash commands suggest close matches; CLI help includes examples.
- `memory restore`, `memory reconsider`, `skills disable|enable`, and `config restore-last-good` work as documented.
- Setup validates prompts inline and resumes from setup state.
- The selected top 20 user-reachable errors include remediation hints.
- Typing during an in-flight turn preserves the next-turn draft.
- `Enter` during in-flight does not send a second turn.
- `Ctrl-C` during in-flight gives a truthful non-cancel message.
- Approval/input modals cannot leak normal draft text.
- `/inbox`, `/memory staged`, `/skills`, and `/self-improvement` commands work without leaving the TUI.
- CLI help for `activity`, `settings`, `self-improvement`, `inbox`, `daemon channels`, and `repl` contains useful descriptions.
- Tier A validation passes with zero clippy warnings.

## Deferred work

- Cancel-turn protocol and server-side cancellation boundaries.
- Richer stuck remediation such as provider retry, tool interruption, or validation cancellation.
- Transcript scroll/search/copy and persistent scroll position.
- Command palette and richer command discoverability affordances.
- Theming, color-blind-safe palettes, high-contrast mode, and remappable keys.
- Multi-line composer behavior.
- Full accessibility pass for screen readers and reduced-motion mode beyond `spinner_style = "off"`.
- Voice and multimodal status/activity surfaces.
- Structured `user_message` / `hint` / `doc_link` error trait replacing v0.12.1 string-appended hints.
- Live `AGENTS.md` side panel inside the TUI, beyond slash-command rendering.
- Cross-channel hand-off for starting a turn in one channel and resolving/continuing it in another.
- Non-interactive `allbert-cli setup --from-file` and packaged installer flows.
- Native first-class `allbert-cli scripting status`, if future scripting UX work justifies another public command.

## ADRs

None. This point release stays inside existing ADR boundaries: activity extends the v0.11 daemon-owned telemetry posture rather than introducing a new authority model. If cancel-turn, transcript navigation, broad settings policy, or accessibility work is pulled into a later release, that work may need its own ADR depending on protocol and operator-safety consequences.
