# v0.12.1 - Operator UX polish

Status: Draft, implementation-ready

## Summary

v0.12 shipped powerful self-improvement, skill-authoring, scripting, inbox, and review surfaces. The first operator test showed that the runtime can do the work, but the interactive experience still makes users guess what is happening, leave the TUI too often, or know CLI commands from memory.

v0.12.1 is a point release focused on operator confidence. It broadens the earlier TUI-responsiveness draft into a release-sized UX polish pass: shared activity and stuck-state awareness, responsive in-flight TUI turns, safer input/modal behavior, a consistent command hierarchy, a first-class settings hub, TUI slash-command parity for the main v0.12 workflows, clearer setup, better CLI help, and release-ready docs.

The release includes one narrow protocol bump to v3 so TUI, classic REPL, CLI, Telegram, jobs, and future channels can consume the same daemon-owned `ActivitySnapshot` instead of guessing from frontend timers. It deliberately does not change self-improvement safety invariants, introduce cancellation, or change classic REPL semantics beyond shared activity/help output.

## Goals

- Keep the TUI responsive during in-flight turns and make activity visible with a spinner, elapsed timer, activity label, current tool summary, result breadcrumb, Tier-2 skill activation badge, and visible input caret.
- Make runtime activity and stuck-state awareness daemon-owned and available across channels through `ActivitySnapshot`.
- Let operators type the next-turn draft while a turn is running without accidentally sending a concurrent turn or leaking draft text into approval/input modals.
- Add a scalable command hierarchy and settings hub so setup choices, daily operations, and later customization share one mental model.
- Make v0.12 workflows discoverable inside the TUI: inbox approvals, staged memory, skills, and self-improvement review/install/garbage-collection paths.
- Reduce first-run confusion with clearer setup headings, defaults, Ollama readiness guidance, trusted-root wording, a final setup summary, and "what next" commands.
- Make high-value CLI help output self-explanatory for `self-improvement`, `inbox`, `daemon channels`, and `repl`.
- Preserve v0.12 invariants: no automatic patch install, no binary auto-swap, no Lua auto-enable, no hidden config mutation, and no concurrent turn queue.

## Non-goals

- No broad daemon protocol redesign. The v3 bump is limited to additive activity snapshot/update messages and optional inclusion in operator snapshots.
- No turn cancellation. `Ctrl-C` remains non-cancel in v0.12.1 and must say so clearly.
- No concurrent-turn queue. Text typed during an in-flight turn is only a local next-turn draft.
- No transcript scroll, search, copy, or command palette.
- No theming, remappable keys, or full accessibility mode beyond `spinner_style = "off"` as a reduced-motion/static indicator option.
- No arbitrary free-form config editing through the TUI. Settings changes route through a typed allowlist and shared validation.
- No new ADR. Existing ADRs for TUI boundaries, self-improvement approval, scripting policy, telemetry ownership, and the self-change envelope remain authoritative.
- No change to classic REPL behavior except shared command descriptions/help text where appropriate.

## What-if audit findings

| Scenario | User frustration | v0.12.1 response |
| --- | --- | --- |
| Fresh setup prompt is answered incorrectly | Line-oriented prompts can consume the wrong answer and leave users unsure what was saved | Add numbered step headings, stronger default guidance, final setup summary, and explicit next commands after save |
| Fresh setup chooses Ollama | User can finish setup without realizing Ollama must already be installed and running | Add Ollama readiness hint, model pull/run reminder, and troubleshooting pointer before save |
| Trusted-root prompt appears during setup | User may not understand whether this is project trust, shell trust, or profile trust | Rewrite wording around source checkout trust and what it permits |
| Idle TUI shows `inbox 1` | Status line advertises pending work but `/inbox` is not available locally | Add TUI `/inbox list|show|accept|reject` |
| Idle TUI has no obvious help affordance | Users must already know `/help`; there is no visible command affordance | Add visible input caret and improve `/help` grouping with v0.12 workflow commands |
| Turn is in flight | Screen appears frozen; user cannot tell whether Allbert is thinking, calling tools, or stuck | Add async redraw, spinner, elapsed timer, activity label, current tool summary, and tool result breadcrumb |
| Activity exists only in the TUI | Classic, CLI, Telegram, jobs, and future channels cannot answer "what is Allbert doing?" consistently | Add daemon-owned `ActivitySnapshot` and broadcast `ActivityUpdate` for every channel |
| Provider call is quiet for a long time | User cannot distinguish slow model latency from a broken daemon or lost stream | Daemon computes `stuck_hint`, for example `calling model for 42s; no provider tokens yet` |
| Tool, validation, or script runs long | User sees no current command/tool context and may assume the interface hung | Activity snapshot includes `phase`, `tool_name`, `tool_summary`, elapsed time, and truthful next actions |
| Approval or input is pending | Work appears stalled even though Allbert is waiting on the operator | Activity snapshot uses `waiting_for_approval` or `waiting_for_input` with `approval_id` and next-action guidance |
| User types during in-flight turn | Existing blocking loop prevents typing; future async loop could accidentally send a second turn | Buffer input as a next-turn draft; `Enter` while in-flight must keep the draft and show `turn in progress; draft kept` |
| User presses `Ctrl-C` during in-flight turn | It looks like cancel should happen, but there is no cancellation protocol | Keep turn running and show `cancel is not available yet; turn continues` |
| Modal opens while draft text exists | Current modal shares `app.input`; modal confirmation/input can leak or erase the normal composer draft | Split `composer_input` from `modal_input`; confirmation modals use key choices; input modals use only `modal_input` |
| Patch approval is accepted | User may assume accept installed the patch or swapped the binary | After accept, show next step text and expose `/self-improvement install <approval-id>` |
| Skill-author produces draft skills | User must leave the TUI or know CLI commands to inspect generated artifacts | Add `/skills list|show` and help text for incoming/self-authored skill review |
| Memory staging exists | User sees memory counts but has no local staged-memory inspection path | Add `/memory staged list|show` |
| Lua scripting is opt-in | Operator docs mention scripting, but TUI users may never discover whether Lua is enabled | Surface scripting opt-in guidance through `/help` and relevant self-improvement error/help text; do not add a new `allbert-cli scripting status` command in v0.12.1 |
| Telegram user asks "what is happening?" | Async channel users cannot see TUI status and may duplicate work or restart the daemon | Add `/activity` and include compact activity in `/status` |
| User wants to change an onboarding choice later | Provider, TUI/classic mode, status-line items, activity hints, memory routing, or scripting posture feel buried in TOML | Add `/settings` and `allbert-cli settings` over a typed, validated settings registry |
| Command list grows every release | Flat `/help` output becomes a junk drawer and users cannot predict where new features live | Add a command hierarchy with stable groups: conversation, status, review, memory, skills, self-improvement, settings, and system |
| CLI help is sparse | Top-level and nested help output does not teach the shipped surface | Add non-empty `about`/help text for high-value subcommands |

## Public interfaces

### Config

Add optional TUI config under `[repl.tui]`:

```toml
[repl.tui]
spinner_style = "braille" # "braille" | "dots" | "bar" | "off"
tick_ms = 80              # clamped to [40, 250]

[operator_ux.activity]
stuck_notice_after_s = 30
long_tool_notice_after_s = 20
show_activity_breadcrumbs = true
```

Existing profiles load unchanged with defaults. Invalid `spinner_style` values fall back to `braille` with an operator-visible warning. `tick_ms` values outside `[40, 250]` warn and clamp at config load.

`operator_ux.activity.stuck_notice_after_s` applies to quiet model calls, approval waits, input waits, validation runs, scripts, and daemon-event silence after turn start. `long_tool_notice_after_s` applies to `calling_tool` and tool-like script/validation phases. Values below 5 seconds warn and clamp to 5; values above 600 seconds warn and clamp to 600. `show_activity_breadcrumbs = false` hides transcript breadcrumbs in channels that support them, but does not disable activity tracking or `allbert-cli activity`.

### Command hierarchy

Group TUI slash commands, CLI help, and operator docs around the same hierarchy:

| Group | TUI commands | CLI commands | Purpose |
| --- | --- | --- | --- |
| Conversation | `/help`, `/quit`, `/clear` where supported | `repl` | Start, continue, and understand interactive use |
| Status | `/status`, `/activity`, `/telemetry`, `/statusline` | `status`, `activity`, `telemetry`, `daemon status` | See what Allbert is doing and what resources are in use |
| Review | `/inbox ...` | `inbox ...` | Resolve approvals and review artifacts |
| Memory | `/memory ...` | `memory ...` | Inspect routing, staged memory, facts, episodes, and stats |
| Skills | `/skills ...` | `skills ...` | Inspect installed/incoming skills and self-authored drafts |
| Self-improvement | `/self-improvement ...` | `self-improvement ...` | Review patch approvals, config, install, and cleanup |
| Settings | `/settings ...` | `settings ...` | Change supported profile settings without hand-editing TOML |
| System | `/doctor`, `/logs` if implemented | `doctor`, `home`, `daemon ...` | Diagnose setup, daemon state, and local profile paths |

New command families must choose one of these groups before adding another top-level surface. If a future feature does not fit, update this hierarchy first rather than appending an ungrouped command.

### Settings hub

Add a typed settings registry in the kernel/config layer and expose it through TUI and CLI adapters. The registry owns labels, descriptions, value types, defaults, validation, privacy/safety notes, restart requirements, and config paths.

Initial settings categories:

- `ui`: `repl.ui`, `repl.tui.spinner_style`, `repl.tui.tick_ms`, status-line item toggles, and activity breadcrumb display.
- `activity`: `operator_ux.activity.stuck_notice_after_s`, `operator_ux.activity.long_tool_notice_after_s`, and `operator_ux.activity.show_activity_breadcrumbs`.
- `memory`: memory routing mode, semantic retrieval enablement, and staged/episode auto-prefetch policy where already configurable.
- `learning`: personality digest enablement and bounded episode-summary limits.
- `self_improvement`: source checkout, sibling worktree path, install policy, scripting posture, and cleanup policy.
- `providers`: non-secret provider/model defaults that can be safely displayed. Secrets remain outside the settings hub and keep their existing secret-management path.

Public commands:

| Command | Behavior |
| --- | --- |
| `/settings list [group]` / `allbert-cli settings list [group]` | List supported settings with current value, default, and restart requirement |
| `/settings show <key>` / `allbert-cli settings show <key>` | Explain one setting, including config path, accepted values, default, safety note, and whether it can be changed live |
| `/settings set <key> <value>` / `allbert-cli settings set <key> <value>` | Validate and persist one supported setting through the registry |
| `/settings reset <key>` / `allbert-cli settings reset <key>` | Remove the explicit override and fall back to default |
| `/settings explain <group>` / `allbert-cli settings explain <group>` | Show why the group exists and common examples |

Settings writes must be atomic, preserve unrelated TOML comments where practical, and reject unsupported keys, secrets, path escapes, and reserved runtime directories. The settings hub may point advanced operators to manual TOML editing for unsupported values, but it must not provide an unsafe generic editor in v0.12.1.

### Daemon protocol

Bump `PROTOCOL_VERSION` to `3` with additive activity messages and fields.

Add:

- `ActivityPhase`: `idle`, `queued`, `preparing_context`, `classifying_intent`, `calling_model`, `streaming_response`, `calling_tool`, `waiting_for_approval`, `waiting_for_input`, `running_validation`, `running_script`, `finalizing`, and `error`.
- `ActivitySnapshot`: `phase`, `label`, `started_at`, `elapsed_ms`, `session_id`, `channel`, optional `tool_name`, optional `tool_summary`, optional `skill_name`, optional `approval_id`, optional `last_progress_at`, optional `stuck_hint`, and optional `next_actions`.
- `TelemetrySnapshot.current_activity: Option<ActivitySnapshot>`.
- `ServerMessage::ActivityUpdate(ActivitySnapshot)` for broadcast updates.
- `ClientMessage::ActivitySnapshot` and `ServerMessage::ActivitySnapshot(ActivitySnapshot)` for direct CLI/channel reads.

If `OperatorSnapshotPayload` lands in the same v0.12.1 operator-surface work, include `current_activity: Option<ActivitySnapshot>` there too. If it does not, `TelemetrySnapshot.current_activity` plus the direct activity request are the canonical v0.12.1 surfaces.

Activity snapshots are safe operational summaries. They must not include raw prompts, secret values, full tool inputs, raw tool output, transcript excerpts, or provider payloads.

### Runtime behavior

- While a turn is in flight, key input edits `composer_input` as a next-turn draft.
- `Enter` while a turn is in flight must not send a second `RunTurn`; it appends a local notice and preserves the draft.
- `Ctrl-C` while a turn is in flight must not imply cancellation; it appends a local notice that cancel is unavailable.
- Confirmation modals must not use `composer_input`. They use key choices only.
- Input modals must read and write `modal_input`; normal draft text survives modal open, submit, cancel, and close.

## Implementation design

### TUI event loop

Replace the blocking `event::poll` plus turn-specific receive loop in [tui.rs](../../crates/allbert-cli/src/tui.rs) with one async event loop that selects over:

- daemon messages from `client.recv()`;
- key events from `crossterm`'s async `event-stream` feature;
- a `tokio::time::interval` tick using clamped `tick_ms`.

The loop should draw when state is dirty, a turn is in flight, or a transient badge is active. Idle screens should not redraw continuously.

If enabling `crossterm` event-stream introduces a dependency or platform problem during implementation, stop and update this plan instead of silently switching architecture.

The TUI owns animation, caret, draft buffering, modal state, and local transcript rendering. It may show a disconnected notice if activity cannot be refreshed, but it must not invent runtime phases or stuck hints from local timers. The current runtime phase, elapsed time, stuck hint, and next actions come from daemon `ActivityUpdate` messages or a direct `ActivitySnapshot` refresh.

### Shared activity state

Add the shared activity model in `allbert-proto`, with daemon-owned lifecycle management:

```rust
enum ActivityPhase {
    Idle,
    Queued,
    PreparingContext,
    ClassifyingIntent,
    CallingModel,
    StreamingResponse,
    CallingTool,
    WaitingForApproval,
    WaitingForInput,
    RunningValidation,
    RunningScript,
    Finalizing,
    Error,
}

struct ActivitySnapshot {
    phase: ActivityPhase,
    label: String,
    started_at: DateTime<Utc>,
    elapsed_ms: u64,
    session_id: String,
    channel: String,
    tool_name: Option<String>,
    tool_summary: Option<String>,
    skill_name: Option<String>,
    approval_id: Option<String>,
    last_progress_at: Option<DateTime<Utc>>,
    stuck_hint: Option<String>,
    next_actions: Vec<String>,
}
```

Ownership rules:

- Kernel emits precise activity transitions around context assembly, intent classification, model calls, response streaming, tool calls, script runs, validation runs, approval waits, input waits, finalization, terminal errors, and turn completion.
- Daemon stores the latest `ActivitySnapshot` per session, computes `elapsed_ms`, attaches advisory `stuck_hint` values, broadcasts `ActivityUpdate`, and includes the current snapshot in telemetry/operator snapshots.
- Frontends render activity only. TUI, classic REPL, Telegram, and future channels must not scrape logs or infer stuck state from local frontend timers.
- Activity snapshots must return to `idle` after `TurnResult`, after a terminal turn error has been surfaced, or after a job finishes.
- Activity labels and summaries are operator-facing but conservative. They summarize operational state and never include raw prompt text, secrets, full command lines containing arguments likely to contain secrets, raw tool output, or provider payloads.

Expected turn transitions:

| Runtime point | Activity phase |
| --- | --- |
| Turn accepted but not yet running | `queued` |
| Bootstrap, memory, telemetry, skill context assembly | `preparing_context` |
| Intent/router pass | `classifying_intent` |
| Provider request sent and no tokens received yet | `calling_model` |
| Provider tokens or chunks are arriving | `streaming_response` |
| Kernel invokes a tool | `calling_tool` |
| Inbox/approval gate is open | `waiting_for_approval` |
| Kernel asks operator/channel for input | `waiting_for_input` |
| Tier A or other validation command runs | `running_validation` |
| Embedded or policy-governed script runs | `running_script` |
| Turn result, memory staging, cost/session persistence, or cleanup is completing | `finalizing` |
| Terminal error before normal completion | `error`, then `idle` after the error is delivered |

Tool summaries are derived from a conservative extractor table: file paths for read/write tools, first command token plus redacted arguments for process tools, query labels for memory/search tools, validation profile name for test/clippy/build steps, and `...` for unknown tools. Summaries are observability only, not validation.

### Stuck awareness

Stuck state is advisory, daemon-side, and truthful about what is known. It must not imply cancellation exists.

Threshold behavior:

- `calling_model` older than `stuck_notice_after_s` with no provider progress sets a hint such as `calling model for 42s; no provider tokens yet`.
- `calling_tool`, `running_script`, or `running_validation` older than `long_tool_notice_after_s` sets a hint such as `running cargo clippy for 95s`.
- `waiting_for_approval` sets a hint once older than `stuck_notice_after_s`, including `approval_id` when available.
- `waiting_for_input` sets a hint once older than `stuck_notice_after_s`, naming the channel that must answer.
- No daemon event after turn start for more than `stuck_notice_after_s` sets a hint such as `turn started 31s ago; no daemon progress event received`.
- `next_actions` must be accurate for the current phase: wait, inspect logs, resolve approval, provide input, exit/detach, or restart after confirming the daemon is unhealthy. Do not list cancel as an action in v0.12.1.

### Rendering

When in flight, prepend a transient status segment before configured status-line items:

```text
<spinner> calling model 4.2s | model gemma4 | ctx 1240/12% | cost $0.0042 | inbox 1
```

Compact terminals should preserve signal over decoration. Below 60 columns, render spinner plus activity label and omit elapsed time if needed.

Render a visible caret with `frame.set_cursor_position(...)` in the input pane when the composer is active and in modal input when an input modal is active.

Render dropped events as compact transcript or status artifacts:

- `ToolCall` renders `> read_file(foo.rs)`.
- `ToolResult` renders `< read_file: 42 lines, 2.1KB` or a truncated first content line.
- `SkillTier2Activated` renders a transient `activated: memory-curator` badge for about 1.5 seconds.
- `Cost` and `TurnDone` continue to update telemetry without adding transcript noise.

### Shared rendering helpers

Extract CLI rendering helpers for inbox, memory staged entries, skills, and self-improvement outputs where that keeps CLI and TUI consistent. The helpers should return strings or lightweight view models, not write directly to stdout, so tests can assert output without terminal coupling.

### Command and settings ownership

The TUI is only one channel. Command catalog, settings metadata, and activity state must be reusable:

- Kernel/config owns the settings registry, validation, typed parsing, redaction, defaults, and persistence.
- CLI and TUI own command parsing for their channel, but both render from shared command descriptors and lightweight view models where practical.
- Daemon owns runtime activity truth and exposes it over protocol v3; channels consume that state instead of deriving it from logs, transcripts, or frontend timers.
- Telegram can read status/activity from daemon but must not gain broad settings mutation in v0.12.1. Settings changes remain local CLI/TUI operations with the same profile filesystem trust assumptions as existing setup/config commands.
- Setup should finish by showing which settings group changes each answer later, for example `Change UI mode later with /settings set repl.ui classic`.

## End-user release plan

Release notes must explain:

- The TUI now redraws during turns and shows what Allbert is doing.
- Activity is now daemon-owned and visible across TUI, classic REPL, CLI, Telegram, and job reports.
- `allbert-cli activity [--json]` shows the current phase, elapsed time, stuck hint, and next actions.
- Users can type the next draft while a turn runs, but `Enter` will not send it until the current turn is complete.
- `Ctrl-C` still does not cancel a running turn in v0.12.1.
- Stuck hints are advisory and truthful; they explain what is known and never imply cancellation exists.
- Setup choices can be changed later through `/settings` or `allbert-cli settings` for supported values.
- Commands are grouped consistently across `/help`, CLI help, and operator docs.
- Inbox, staged memory, skill, and self-improvement review workflows are now available from TUI slash commands.
- Setup now ends with a summary and next commands.
- Ollama users still need a running local Ollama service and pulled model.
- `spinner_style = "off"` is available for terminals or recordings that should avoid animation.

Operator acceptance checklist:

- Fresh temp profile setup is understandable without external docs.
- Fresh temp profile can start TUI or classic fallback cleanly.
- A slow fake-provider or real-provider turn shows spinner/activity updates continuously.
- The same slow turn is visible through `allbert-cli activity --json`.
- Telegram `/activity` and `/status` show compact activity without raw prompt or secret data.
- Long model/tool/approval/input waits produce advisory stuck hints with truthful next actions.
- `/settings` and `allbert-cli settings` can list, explain, set, and reset supported settings with validation.
- `/help` uses the documented command hierarchy rather than a flat command dump.
- Typing during an in-flight turn preserves the next-turn draft.
- `Enter` during in-flight does not send a second turn.
- `Ctrl-C` during in-flight gives a truthful non-cancel message.
- Approval/input modals cannot leak normal draft text.
- `/inbox`, `/memory staged`, `/skills`, and `/self-improvement` commands work without leaving the TUI.
- CLI help for `activity`, `settings`, `self-improvement`, `inbox`, `daemon channels`, and `repl` contains useful descriptions.
- Tier A validation passes with zero clippy warnings.

## Deferred work

- Cancel-turn protocol and server-side cancellation boundaries.
- Richer stuck remediation such as provider retry, tool interruption, or validation cancellation.
- Transcript scroll/search/copy and persistent scroll position.
- Command palette and richer command discoverability affordances.
- Theming, color-blind-safe palettes, high-contrast mode, and remappable keys.
- Multi-line composer behavior.
- Full accessibility pass for screen readers and reduced-motion mode beyond `spinner_style = "off"`.
- Native first-class `allbert-cli scripting status`, if future scripting UX work justifies another public command.

## ADRs

None. This point release stays inside existing ADR boundaries: activity extends the v0.11 daemon-owned telemetry posture rather than introducing a new authority model. If cancel-turn, transcript navigation, broad settings policy, or accessibility work is pulled into a later release, that work may need its own ADR depending on protocol and operator-safety consequences.
