## Main User/Operator Flows

1. A newly installed app/plugin/action registers with Allbert.
2. Registration emits a lifecycle signal; the intent index marks itself stale and
   debounces a rebuild + a generation pass for newly-seen actions.
3. Operator or maintenance action runs descriptor optimization.
4. Actions with no active descriptor receive local-model-generated candidates, with
   heuristic fallback.
5. Memory/clarification/confirmation/trace analysis writes learned-review YAML
   proposals only (inert).
6. Operator reviews, promotes, edits, or disables descriptors — **promotion runs the
   ADR 0071 routing-accuracy gate and is rejected on regression or a negative-route
   break.**
7. The resolver rebuilds the active descriptor set; promoted descriptors affect
   routing, but execution still goes through registered actions, Security Central,
   and confirmations.
8. An operator checks **which model to use for what** via the doctor and the model
   recommendation guide, and grows the eval corpus by capturing redacted real
   mis-routes.

## Data Flow

```text
registration event
  -> allbert.{app,plugin}.registered / allbert.action.registry_changed
  -> Intent.Router.Index marks :not_built + debounces rebuild + generation pass
  -> optimize_intent_descriptors action  |  mix allbert.intent optimize
  -> Optimizer.generate:
       action metadata + schema + capability
       -> router_local model (ReqLLM json_schema, bounded, redacted; ADR 0061)
       -> heuristic fallback on unavailable/disabled/timeout/invalid
  -> DescriptorStore:
       generated/        accepted trusted candidates
       learned/review/   memory/trace/confirmation/correction proposals (INERT)
       overrides/        operator edits/disables
  -> promotion (generated/learned -> active)  ─┐
                                                ├─► Intent.Eval.Gate.check(candidate, baseline)
  release.v056 deterministic lane              │      Runner(FakeEmbedder seam + fake disambiguator)
                                                │      Scorer: top-1 / per-domain / per-surface /
                                                │              confusion / slots / clarify-vs-execute /
                                                │              negative-route violations
                                                └────  pass(>=baseline & >=floor & 0 negatives) -> write
                                                       fail -> REJECT promotion / FAIL release  (BLOCKING)
  -> DescriptorResolver active set:
       app/plugin < action < generated < overrides   (learned/review stays inert)
  -> Index rebuild
  -> Router shortlist (Stage 1 embed) -> Stage-2 disambiguation -> Outcome
  -> Runner + Security Central + confirmation gate

model recommendations:
  mix allbert.settings model-doctor | mix allbert.intent doctor
    -> per purpose: recommended (ADR 0072) vs configured (Settings Central) vs status
       (ok | missing | under-capable | not-pulled | remote-egress-warning)   [redacted]
```

## Operator Action Layer (single source of truth)

Every operator operation is a registered Jido action; surfaces are thin views over
the **same** action (extends ADR 0070):

```text
CLI (mix allbert.intent / settings)  ─┐
TUI slash (/intents, /models, ...)    ├─► Actions.Runner.run(<action>, params, operator_context)
v0.58 web panels                      ┘      reads:  exposure: :internal, permission: :read_only
                                              mutations: operator-surface only, gated, audited
                                              -> read-model DTO (redacted)  |  gated mutation
```

Natural-language channel routing sees only `Actions.Registry.agent_modules/0` and
never invokes these internal operator operations.

- Reads: `intent_doctor`, `intent_list_descriptors`, `intent_show_descriptor`,
  `intent_coverage`, `intent_eval_run`, `intent_list_review`, `model_doctor` —
  `:internal`, never in `agent_modules/0`, never intent candidates.
- Mutations: `optimize_intent_descriptors`, `promote_intent_descriptor`,
  `reindex_intent_descriptors`, `enable/disable/edit_intent_descriptor`,
  `intent_eval_baseline`, `intent_eval_capture`, `intent_eval_add` — gate-checked where
  they touch routing.

## Eval Corpus Flow (deterministic gate + operator growth)

```text
committed seed: test/fixtures/intent/eval/<domain>/*.yaml   <-- the ONLY CI gate source
operator runtime:
  intent_eval_capture <trace/clarification ref>
    -> redacted candidate -> <ALLBERT_HOME>/intents/eval/captured/*.yaml
  operator reviews the candidate
  intent_eval_add <candidate>   (dev/repo action)
    -> promotes the reviewed candidate INTO the committed fixture
gate / CI reads only the committed fixture (same input -> identical run)
```

M1 migrated the existing golden anchors into one YAML file per case under the
domain directories above. M2 expanded the corpus across the current routable
inventory, current internal-action negative routes, planned v0.56 operator-action
negative routes, TUI slash negatives, and ambiguous sibling-domain pairs, then
recorded the BEFORE baseline. The committed M2 baseline is intentionally not
release-passing yet (`overall_accuracy: 0.827`, `gate.status: fail`); later
descriptor coverage and promotion-gate milestones must improve from that floor.

## Authority Invariants

- Descriptor generation and learning are advisory only.
- Learned-review proposals never enter active routing until operator promotion.
- A generated/promoted descriptor never grants permission, changes safety floors,
  creates confirmations, or bypasses action schemas.
- The routing-accuracy gate is fail-closed: it only ever **blocks** an unsafe routing
  change; it never enables one and grants no authority.
- Hosted model generation/escalation is not default; local generation uses the v0.54
  model profile stack and fails closed to heuristic generation. A model recommendation
  never enables egress or capability; hosted profiles are explicit, audited opt-ins.
- Registration signals update routing coverage only; they never register actions or
  grant capability authority by themselves.
- All new settings resolve through Settings Central; all security/permission decisions
  resolve through Security Central at the action boundary.

## Per-Surface Routing Invariants

- The same utterance routes to the same action across web, TUI, and every messaging
  channel (Telegram/Discord/Slack/Matrix/WhatsApp/Signal/email) — surface affects
  delivery, not the selected action.
- v0.55.1 operator-inspection actions (`exposure: :internal`), doctor actions, and TUI
  slash lines are **never routable** on any surface (negative-route guarantee; ADR
  0070). A slash line is never a model turn.
- Internal-capability actions (`signal_doctor`, `whatsapp_doctor`) are excluded from
  the routable set.

## UI/UX Surfaces (read-only DTOs)

- **CLI (v0.56):** coverage report, descriptor list with source-layer badges, eval
  results + gate status, and the model-recommendation matrix render through
  `mix allbert.intent doctor|list|eval` and `mix allbert.settings model-doctor`.
- **Web (flagged to v0.58):** an Intents panel (descriptors by domain, source badges,
  coverage, edit/disable, review proposals, trigger optimize) and a Settings/Models
  panel rendering the recommendation DTO. v0.56 ships the DTO contracts; v0.58 renders.
- **TUI (v0.56):** read-only `/intents` and `/models` slash commands reuse the
  v0.55.1 console pattern. They are backed by registered internal read actions
  through `Actions.Runner.run/3`, render the same redacted DTOs as the CLI doctors,
  are reachable only through the TUI slash allowlist, and are never intent candidates.

## Handles / Operator Surfaces

The final v0.56 target is that all of these are **thin views over the Operator Action
Layer** — the Mix task, TUI slash, and the v0.58 web panels invoke the same registered
actions through `Actions.Runner.run/3`; none reaches the Optimizer/Store/Eval modules
directly. M3 ships this guarantee for read operations (`doctor`, `list`, `show`,
`coverage`, `review`, `eval run`). M10 completes the mutation/eval capture handles.

- Registered actions: reads (`intent_doctor`, `intent_list_descriptors`,
  `intent_show_descriptor`, `intent_coverage`, `intent_eval_run`, `intent_list_review`,
  `model_doctor`) + mutations (`optimize_intent_descriptors`, `promote_intent_descriptor`,
  `reindex_intent_descriptors`, `enable/disable/edit_intent_descriptor`,

- `mix allbert.intent optimize | reindex | doctor | review | promote | edit | disable | list | show | bench | coverage`

- TUI slash reads: `/intents` (→ `intent_coverage`), `/models` (→ `model_doctor`) plus

## Smoke Checklist

- Start from a disposable `ALLBERT_HOME`.
- `mix allbert.settings model-doctor` reports per-purpose recommended-vs-configured
  status with no secrets.
- `mix allbert.intent doctor` reports current coverage + the eval baseline + gate
  status.
- `mix allbert.intent optimize` with the local model available → generated YAML is
  data-only and redacted; stop/disable the local model → heuristic fallback.
- Seed a reviewed-memory or clarification-derived proposal → lands in
  `intents/learned/review/` and is not active.
- `mix allbert.intent eval run` is deterministic; a seeded regression makes the gate
  fail; `mix allbert.intent eval capture` adds a redacted real case.
- Promote a proposal → the gate accepts (no regression) and the resolver/index sees it
  after reindex; seed a regressing promotion → the gate rejects it.
- Register a test plugin/action → the registration signal marks the index stale and
  rebuilds after debounce; disable `intent.reindex_on_registration_signal` → the escape
  hatch prevents signal-driven rebuild.
- In one warm `mix allbert.tui` session, `/intents` and `/models` render the same
  redacted coverage/gate/model DTOs as the CLI; slash lines create no model turns and
  no routable intent candidates.
- Run `release.v056` and the docs gate.

### A. Setup And Deterministic Gates (prelaunch)

Run these commands from the umbrella root before launching the TUI. These are
intentionally cold setup/gate commands; the no-cold-Mix rule starts once S6 launches
`mix allbert.tui`. By the time M14 runs, every command below must be real; M10/M13
implementation must update exact subcommand names, flags, `mix help` text, and expected
output before this checklist is handed to an operator.

**S0 — Confirm repository state and create durable evidence paths.**

```sh
git status -sb

export V056_MANUAL_HOME="$(mktemp -d /tmp/allbert-v056-manual.XXXXXX)"
export ALLBERT_HOME="$V056_MANUAL_HOME"
export V056_EVIDENCE_DIR="$HOME/.allbert-release-evidence/v056"
export V056_TRANSCRIPT="$V056_EVIDENCE_DIR/manual-v056-$(date -u +%Y%m%dT%H%M%SZ).log"

mkdir -p "$V056_EVIDENCE_DIR"
printf 'ALLBERT_HOME=%s\n' "$ALLBERT_HOME"
printf 'V056_TRANSCRIPT=%s\n' "$V056_TRANSCRIPT"
test -d "$ALLBERT_HOME" && echo "manual home ok"
test -d "$V056_EVIDENCE_DIR" && echo "evidence dir ok"
```

Expected: clean intended branch, both paths print, and both `test` commands pass.

**S1 — Pull or confirm local models.**


Expected: Notes/files is enabled/trusted, `write_note` is available, notes root is

**S5 — Run deterministic release evidence after corpus/baseline setup.**

### D. Pass Criteria

- model-doctor/doctor report per-purpose recommended-vs-configured status, redacted.
- generated descriptors are local-only, redacted, and data-only YAML.
- learned-review proposals remain inert until promoted.
- the eval gate blocks a regressing or negative-route-breaking promotion/release.
- registration signals rebuild the index when enabled and do nothing when the operator
  disables reindex-on-registration.
- routing changes do not grant authority; effectful actions still require their normal
  permission and confirmation path.
- `/intents` and `/models` are read-only internal TUI affordances, slash-allowlisted
  only, absent from intent candidates, and rendered in the same warm session.
- B1-B5 pass in one persistent `mix allbert.tui` session with no BEAM restart.
- Durable local evidence is retained; raw transcripts/release JSON are not committed.
