## Main User/Operator Flows

1. A newly installed app/plugin/action registers with Allbert.
2. Registration emits a lifecycle signal; the intent index marks itself stale and
   debounces rebuild.
3. Operator or maintenance action runs descriptor optimization.
4. Actions with no active descriptor receive local-model-generated candidates,
   with heuristic fallback.
5. Memory/clarification/confirmation/trace analysis writes learned-review YAML
   proposals only.
6. Operator reviews, promotes, edits, or disables descriptors.
7. The resolver rebuilds the active descriptor set; promoted descriptors affect
   routing, but execution still goes through registered actions, Security Central,
   and confirmations.

## Data Flow

```text
registration event
  -> allbert.{app,plugin}.registered / allbert.action.registry_changed
  -> Intent.Router.Index marks :not_built + debounces rebuild
  -> optimize_intent_descriptors or mix allbert.intent optimize
  -> DescriptorGenerator:
       action metadata + schema + capability
       -> local router model (router_local)
       -> heuristic fallback on unavailable/invalid model
  -> DescriptorStore:
       generated/ for accepted trusted candidates
       learned/review/ for memory/trace/confirmation/correction proposals
       overrides/ for operator edits/disables
  -> DescriptorResolver active set:
       app/plugin < action < generated < overrides
       learned/review remains inert
  -> Index rebuild
  -> Router shortlist / Stage-2 decision
  -> Runner + Security Central + confirmation gate
```

## Authority Invariants

- Descriptor generation and learning are advisory only.
- Learned-review proposals never enter active routing until operator promotion.
- A generated/promoted descriptor never grants permission, changes safety floors,
  creates confirmations, or bypasses action schemas.
- Hosted model generation is not default; local generation uses the v0.54 model
  profile stack and fails closed to heuristic generation.
- Registration signals update routing coverage only; they never register actions
  or grant capability authority by themselves.

## Smoke Checklist

- Start from a disposable `ALLBERT_HOME`.
- Confirm `mix allbert.intent doctor` reports current coverage.
- Run `mix allbert.intent optimize` with the local model available and verify
  generated YAML is data-only and redacted.
- Stop or disable the local model and verify heuristic fallback.
- Seed a reviewed-memory or clarification-derived proposal and confirm it lands
  in `intents/learned/review/` and is not active.
- Promote a proposal and confirm the resolver/index sees it after reindex.
- Register a test plugin/action and confirm the registration signal marks the
  index stale and rebuilds after debounce.
- Disable `intent.reindex_on_registration_signal` and confirm the escape hatch
  prevents signal-driven rebuild.
- Run `release.v056` and the docs gate.

```sh
mix allbert.intent promote <proposal-id>
mix allbert.intent reindex
mix allbert.intent doctor
mix allbert.test release.v056
```

Expected behavior:

- generated descriptors are local-only, redacted, and data-only YAML.
- learned-review proposals remain inert until promoted.
- registration signals rebuild the index when enabled and do nothing when the
  operator disables reindex-on-registration.
- routing changes do not grant authority; effectful actions still require their
  normal permission and confirmation path.
