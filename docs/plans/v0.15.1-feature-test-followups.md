# v0.15.1 - Feature-test follow-ups

Status: Draft

v0.15.1 is a patch-planning bucket for issues found while a human operator runs
[feature-test-runbook.md](../operator/feature-test-runbook.md) against the
v0.15.0 release. It starts as an intake document: each reported user-test
problem should be reproduced or bounded, formulated as a concrete product
failure, and then expanded into a milestone with repair scope and validation.

This release should stay patch-shaped. It fixes shipped behavior, confusing
operator paths, runbook/doc drift, and validation gaps discovered by the
runbook. New feature work belongs in a later focused plan unless it is the
smallest safe way to repair an already shipped surface.

## Goals

- Track user-test findings from the feature test runbook in one place.
- Convert each confirmed finding into a milestone with problem, decision,
  repair scope, and validation.
- Keep fixes provider-free where possible and use live-provider checks only
  when the failing surface requires them.
- Preserve v0.15 trust boundaries around RAG, memory, tools, self-change,
  Telegram, URL ingestion, and local filesystem access.
- Keep runbook, roadmap, operator docs, and release validation aligned with
  actual behavior as fixes land.

## Non-goals

- No new ambient ingestion or browser-capture release scope.
- No broad redesign of the v0.15 RAG architecture.
- No provider-native tool-calling migration unless a reported regression cannot
  be fixed safely inside the current provider seam.
- No weakening of staged-memory review, job confirmation, skill install
  quarantine, trusted-root enforcement, URL fetch policy, or exec policy.
- No live-provider or Telegram requirement for the default contributor gate.

## Intake Workflow

When a user-test problem is reported:

1. Capture the runbook section, exact command or interaction, expected behavior,
   actual behavior, profile/setup context, and any relevant trace/log/session id.
2. Classify the finding as one of: product bug, operator UX bug, doc/runbook
   drift, validation gap, expected live-provider limitation, or deferred feature.
3. Reproduce with a temporary `ALLBERT_HOME` when practical. If reproduction
   needs credentials, local models, Telegram, or network access, record that
   dependency explicitly.
4. Convert confirmed product, UX, doc, or validation failures into milestones
   below. Each milestone should name the owner surface, the minimal repair, and
   the validation command or manual smoke that proves it.
5. Move deferred features to [future-plans.md](future-plans.md) instead of
   expanding v0.15.1.

## Finding Template

Use this shape when filling a reserved milestone:

- **Reported from:** runbook section and command/interaction.
- **Observed:** exact failure or confusing behavior.
- **Expected:** behavior promised by shipped docs or safe operator intuition.
- **Likely area:** crate/module/docs/runbook surface to inspect.
- **Decision:** selected repair direction after code/doc review.
- **Repair scope:** concrete edits needed.
- **Validation:** provider-free command, fake-provider test, or live smoke.
- **Status:** open, investigating, implemented, validated, deferred, or dropped.

## M0 - Intake Ledger and Guardrails

**Problem.** Feature-test findings can otherwise scatter across chat history,
temporary notes, and ad hoc fixes. v0.15.1 needs one ledger that keeps every
reported issue tied back to the runbook and to the validation surface that will
close it.

**Decision.** This document is the ledger. New findings become concrete
milestones below, and the release remains patch-scoped unless a finding proves
that a shipped promise needs a slightly broader repair.

**Done when.**

- Every reported runbook problem has either a milestone, an explicit deferred
  note, or a dropped/expected-limitation note.
- Each implemented milestone names its validation.
- Runbook and roadmap docs are updated when behavior or release posture changes.

## M1 - Manual Job Run Closes the Protocol Connection

- **Reported from:** `feature-test-runbook.md` v0.2 "Daemon, Jobs,
  Sessions", live `memory-compile` check:
  `run daemon start` followed by `run jobs run memory-compile`.
- **Observed:** `daemon status` prints `jobs enabled: no`, then
  `run jobs run memory-compile` returns `protocol error: connection closed`
  while Ollama `gemma:26b` is running separately.
- **Expected:** Manual job runs should never surface a raw protocol disconnect.
  If `memory-compile` is missing, disabled, malformed, blocked by config, or
  unable to boot a provider, the CLI should print a structured, actionable
  daemon error. A valid manual run should produce a `JobRunRecordPayload` and
  `jobs status memory-compile` should show the run outcome. The status label
  `jobs enabled: no` should be clear that scheduled/background ticks are
  disabled, not that manual job commands are unsafe to invoke.
- **Likely area:** `crates/allbert-daemon/src/server.rs` job RPC handlers,
  especially `ClientMessage::RunJob`; `crates/allbert-daemon/src/client.rs`
  already understands `ServerMessage::Error` for job calls. Also inspect
  `crates/allbert-daemon/src/jobs.rs` for manual-run preflight and cleanup, and
  `docs/operator/feature-test-runbook.md` for the v0.2 live job instructions.
- **Decision:** Treat this as a product bug plus runbook clarity bug. Job
  management RPCs should convert user-recoverable `DaemonError`s into
  `ServerMessage::Error` instead of letting `?` terminate the connection. The
  runbook live block should be self-contained enough that an operator who did
  not just enable the template in the same temp profile knows to run
  `jobs template enable memory-compile` first. Do not require
  `[jobs].enabled = true` for explicit manual `jobs run`; that setting should
  continue to control scheduled ticks.
- **Repair scope:** Add structured error handling for `GetJob`, `PauseJob`,
  `ResumeJob`, `RunJob`, `RemoveJob`, and `SweepJobs` where invalid input or
  missing definitions can currently close the connection. Add or update daemon
  integration tests proving `RunJob("memory-compile")` without an enabled
  definition returns `job not found: memory-compile` through the normal error
  frame and leaves the daemon alive. Add a manual-run preflight or CLI hint if
  `memory-compile` is not defined and a bundled template exists. Update daemon
  status or docs so `jobs enabled: no` reads as scheduled-job disablement, not a
  manual job-run failure reason.
- **Validation:** Provider-free regression test for missing-job manual run;
  temp-profile smoke with `jobs.enabled = false`, `jobs template enable
  memory-compile`, `jobs run memory-compile` against a fake or intentionally
  unavailable provider proving the command returns a job record or structured
  provider error rather than `connection closed`; rerun the v0.2 runbook section
  with live Ollama for final release validation.
- **Status:** validated on 2026-04-28 by operator manual retest with live
  Ollama.

## M2 - Curated-Memory Runbook Assumes a Staged Entry Exists

- **Reported from:** `feature-test-runbook.md` v0.5 "Curated Memory", CLI
  review block:
  `memory staged list`, `memory staged show <staged-id>`,
  `memory promote <staged-id> --confirm`, and reject-path commands.
- **Observed:** On a fresh temp profile there may be no staged entries, so the
  CLI review block has no realistic `<staged-id>` to use.
- **Expected:** The runbook should distinguish provider-free status/index
  checks from staged-memory review checks. When a staged id is needed, the
  runbook should tell the operator how to create or obtain one through the
  realistic explicit-memory user path, and should use separate staged ids for
  promotion and rejection.
- **Likely area:** `docs/operator/feature-test-runbook.md`, v0.5 curated-memory
  section. No code change is required because staging is intentionally produced
  by runtime/review-first paths rather than an arbitrary provider-free
  `memory stage` CLI command.
- **Decision:** Treat this as runbook drift. The v0.5 section now says a fresh
  profile may have no staged entries, asks the operator to run the live
  explicit-memory path first when the CLI review flow needs ids, and tells them
  to stage two candidates so promotion and rejection do not race over the same
  id.
- **Repair scope:** Docs only.
- **Validation:** `tools/check_doc_reality.sh`; operator retest of the v0.5
  curated-memory section. Operator manual retest passed on 2026-04-28 after
  the M3 and M4 fixes.
- **Status:** closed on 2026-04-28; validated by operator manual v0.5
  curated-memory retest.

## M3 - Explicit Remember Turns Do Not Stage Memory with Local Gemma

- **Reported from:** `feature-test-runbook.md` v0.5 "Curated Memory", live
  explicit-memory path used to create staged ids for CLI review.
- **Observed:** In a classic REPL session using local `gemma4:26b`, both
  `remember that Allbert operator tests use temporary ALLBERT_HOME profiles`
  and `remember that the v0.5 reject-path smoke uses a disposable staged
  candidate` completed with prose such as "I've noted that." A subsequent
  `review what's staged` turn called `list_staged_memory` successfully, but the
  result was empty: "There is currently nothing in the staging queue awaiting
  review."
- **Expected:** Explicit `remember that ...` requests should create staged
  `explicit_request` memory entries through the v0.14.3 schema-bound router and
  review-first staging pipeline. The turn should not merely acknowledge the fact
  in prose, and `review what's staged` should list the staged candidates.
- **Likely area:** `crates/allbert-kernel-services/src/lib.rs` router
  resolution and terminal router action handling, especially
  `route_intent_with_llm`, `RouteDecision::executable_action`,
  `handle_terminal_router_action`, and `router_decision_invocation`; local
  Ollama/Gemma structured-output behavior; trace provenance for
  `intent-router` decisions. Existing tests cover the scripted happy path where
  the router returns `memory_stage_explicit`, but not the local-model failure
  mode where the router returns `action = none`, low confidence, malformed JSON,
  or missing memory fields for an exact explicit-memory request.
- **Decision:** Treat this as a product bug, not another runbook issue. v0.15.1
  needs a deterministic repair for exact explicit-memory requests that preserves
  the review-first envelope. The preferred fix is to keep the schema-bound
  router as the first step, but add a bounded fail-closed recovery path for
  obvious explicit-memory capture phrases when the local router does not produce
  an executable `memory_stage_explicit` draft. Any fallback must stage only
  `explicit_request` candidates, must never promote durable memory, must record
  provenance that the fallback handled the turn, and must remain narrower than a
  broad semantic keyword classifier.
- **Repair scope:** Add regression coverage for the observed local-model shape:
  a router response with `action = none` or a non-executable memory decision for
  `remember that ...` must still create exactly one staged `explicit_request`
  entry and finish the turn without a full assistant prose-only fallback. Add
  trace/debug provenance that distinguishes router-authored staging from
  fallback-authored staging. Update runbook wording only after the product fix
  lands so the v0.5 CLI review flow can reliably obtain staged ids.
- **Validation:** Provider-free scripted-router tests for non-executable local
  router outputs; temp-profile live Ollama/Gemma smoke with two `remember that
  ...` turns followed by `memory staged list` showing two ids; then rerun the
  v0.5 promote and reject CLI review commands. Operator manual retest passed on
  2026-04-28.
- **Status:** closed on 2026-04-28; validated by operator manual v0.5
  curated-memory retest with live local model staging.

## M4 - Curated-Memory Runbook Promotes the Wrong Staged Candidate by Order

- **Reported from:** `feature-test-runbook.md` v0.5 "Curated Memory", CLI
  promote/search block after the M3 explicit-memory staging fix.
- **Observed:** The operator staged two realistic candidates and
  `memory staged list` showed the disposable reject-path candidate first:
  `the v0.5 reject-path smoke uses a disposable staged candidate`, followed by
  `Allbert operator tests use temporary ALLBERT_HOME profiles`. The runbook said
  to use the first id for promotion, so the disposable candidate was promoted.
  `memory stats` then showed one durable note and one staged note, but
  `memory search "temporary ALLBERT_HOME"` returned no results because the
  temp-profile candidate was still staged rather than durable.
- **Expected:** The runbook should select staged ids by candidate summary or
  excerpt, not by display order. The candidate containing `temporary
  ALLBERT_HOME` must be promoted before the durable search assertion, and the
  disposable candidate should be reserved for the reject path.
- **Likely area:** `docs/operator/feature-test-runbook.md`, v0.5 CLI review
  instructions. Code research confirms `list_staged_memory` sorts staged
  records newest-first by `created_at`, which is why the second `remember that
  ...` turn appears first in the list.
- **Decision:** Treat this as a runbook bug, not a memory-search product bug.
  The transcript is consistent with promoting the wrong staged entry; durable
  search cannot find `temporary ALLBERT_HOME` until that specific note is
  promoted.
- **Repair scope:** Update the v0.5 runbook to say `memory staged list` is
  newest-first, instruct operators to choose ids by summary/excerpt, and name
  the temp-profile candidate as the promote id and the disposable candidate as
  the reject id.
- **Validation:** `tools/check_doc_reality.sh`; operator retest of the v0.5
  CLI review flow by promoting the temp-profile candidate and confirming
  `memory search "temporary ALLBERT_HOME"` returns it, then rejecting the
  disposable candidate. Operator manual retest passed on 2026-04-28.
- **Status:** closed on 2026-04-28; validated by operator manual v0.5
  curated-memory retest.

## M5 - Daily Cost Cap Is Enforced but Missing from Settings Show

- **Reported from:** `feature-test-runbook.md` v0.6 "Hardening, Session
  Durability, Cost Cap, Recovery", settings smoke:
  `memory verify`, `memory recovery-gc`, `settings show limits.daily_usd_cap`,

  `config restore-last-good`.
- **Observed:** `memory verify` and `memory recovery-gc` passed, but
  `settings show limits.daily_usd_cap` failed with:
  `unsupported setting key `limits.daily_usd_cap``.
- **Expected:** `limits.daily_usd_cap` is a shipped v0.6 cost-cap setting and
  is still enforced by the kernel at turn boundaries. The operator settings
  surface should be able to show it, explain its default/current value, and keep
  the v0.6 runbook command valid.
- **Likely area:** `crates/allbert-kernel-core/src/settings.rs` central
  settings catalog and current-value resolver; `crates/allbert-cli/src/settings_cli.rs`
  group examples/descriptions if a dedicated `limits` group is added; settings
  catalog/persistence tests. Research confirms `Config::limits.daily_usd_cap`
  exists, setup writes it, `/cost` reads it, and kernel turn admission enforces
  it, but the typed settings catalog omits `limits.daily_usd_cap`.
- **Decision:** Treat this as a product/settings-catalog bug, not a runbook
  typo. The runbook is asking for an operator-visible key that the product
  already documents and enforces.
- **Repair scope:** Register `limits.daily_usd_cap` in the settings catalog as a
  live, optional daily spend cap with float validation (`null`/absence remains
  the reset/default disabled state; persisted values must be finite and
  non-negative). Add current/default value resolution for
  `config.limits.daily_usd_cap`. Add a `limits` settings group with
  `limits.daily_usd_cap` as its first visible key. Add tests for
  show/validation/persistence/reset so the settings allowlist cannot drift from
  enforced config again.
- **Validation:** Provider-free settings tests covering
  `settings show limits.daily_usd_cap` on a default profile, setting it to a

  disabled/default, and confirming `settings show learning.compute_cap_wall_seconds`

  `cargo fmt --check`; `cargo test -q -p allbert-kernel-core settings_`;
  `cargo test -q -p allbert-cli settings_`; temp-profile CLI smokes for
  `settings show limits.daily_usd_cap`,
  `settings show learning.compute_cap_wall_seconds`,
  `settings set limits.daily_usd_cap 1.25`, and
  `settings reset limits.daily_usd_cap`; targeted clippy for
  `allbert-kernel-core` and `allbert-cli` with `-D warnings`;

- **Status:** closed on 2026-04-28; validated by operator manual v0.6

## M6 - Telegram Setup Does Not Distinguish Message Id from Chat Id

- **Reported from:** `feature-test-runbook.md` v0.7 "Channels, Telegram,
  Normalized Tools, Web-Learning Posture", live Telegram credential setup.
- **Observed:** A `getUpdates` response contains several numeric ids at once:
  `update_id`, `message.message_id`, `message.from.id`, and
  `message.chat.id`. The runbook says to use `message.chat.id` for
  `TELEGRAM_CHAT_ID`, but it does not explicitly say that
  `message.message_id` is only the per-chat Telegram message id and must not be
  used as the allowlisted chat id.
- **Expected:** Operators should be able to identify the correct value without
  knowing Telegram's JSON model. The docs should name the message id separately
  and clearly state that only `message.chat.id` belongs in
  `config/channels.telegram.allowed_chats` and `identity add-channel telegram`.
- **Likely area:** `docs/operator/feature-test-runbook.md` v0.7 live Telegram
  setup and `docs/operator/telegram.md` credential setup.
- **Decision:** Treat this as runbook/operator-guide clarification rather than
  a product bug.
- **Repair scope:** Add exact verbiage distinguishing bot id, update id,
  message id, user id, and chat id, with a small generic JSON example and an
  explicit `TELEGRAM_CHAT_ID=<message.chat.id>` assignment.
- **Validation:** `tools/check_doc_reality.sh`; `git diff --check`; operator
  retest of v0.7 Telegram setup by using `message.chat.id` as
  `TELEGRAM_CHAT_ID`.
- **Status:** closed on 2026-04-28; validated by operator manual v0.7
  Telegram setup retest and reaffirmed by the later M7 setup-helper retest.

## M7 - Telegram Setup Should Not Require Manual Chat/Message ID Discovery

- **Reported from:** v0.7 Telegram setup follow-up after M6. The docs could
  say which id is correct, but the happy path still asked operators to inspect
  raw Telegram Bot API JSON and hand-copy `message.chat.id`.
- **Observed:** Asking for `TELEGRAM_BOT_TOKEN` is reasonable because BotFather
  owns token creation. Asking for a message/update/chat id is avoidable and
  easy to get wrong; Telegram responses contain `update_id`,
  `message.message_id`, `message.from.id`, and `message.chat.id`.
- **Expected:** The primary setup path should discover the chat id from a fresh
  DM and write the existing explicit allowlist and identity binding only after
  local operator confirmation. Manual `curl getUpdates` parsing should remain
  troubleshooting material, not onboarding.
- **Likely area:** `crates/allbert-cli` channel command wiring and a focused
  Telegram setup helper; `docs/operator/feature-test-runbook.md` v0.7;
  `docs/operator/telegram.md`; `docs/onboarding-and-operations.md`; ADR 0057
  and ADR 0058 notes.
- **Decision:** Add `allbert-cli daemon channels setup telegram` as a local
  operator setup helper. Keep runtime trust semantics unchanged: unknown
  Telegram senders still do not create channel state. The helper may use Bot
  API `getMe` and `getUpdates` during explicit CLI setup, extract only
  `message.chat.id`, and mutate profile files only with `--yes`.
- **Repair scope:** Implement `daemon channels setup telegram` with
  `--token-env`, `--chat-id`, `--latest`, `--yes`, `--no-identity`, and
  `--json`. Resolve token from env first, then the existing token file. Validate
  token with `getMe`; discover candidates with `getUpdates`; require
  `--latest` or `--chat-id` when multiple chats are present. With `--yes`,
  write the token file, append the allowlisted chat id idempotently, enable the
  Telegram channel, and add the identity binding unless `--no-identity` is set.
  Update docs and ADR notes so the helper is the happy path and raw JSON parsing
  is the fallback.
- **Validation:** Focused CLI tests cover extracting `message.chat.id` while
  ignoring `message.message_id`, `update_id`, and `message.from.id`;
  deduplicating candidates by chat id with the latest update retained; no-token
  remediation; no-update remediation; dry-run no mutation; `--yes` writes token,
  allowlist, channel enablement, and identity idempotently; and multiple-update
  ambiguity. Implemented validation included `cargo fmt --check`;
  `env -u RUSTC_WRAPPER cargo test -q -p allbert-cli telegram_setup`;
  `env -u RUSTC_WRAPPER cargo test -q telegram`;
  targeted clippy for `allbert-cli` and `allbert-daemon`; `tools/check_doc_reality.sh`;
  `git diff --check`; and an operator live v0.7 Telegram runbook retest with
  real credentials.
- **Status:** closed on 2026-04-28; validated by operator manual v0.7
  Telegram setup-helper retest.

## M9 - Auto-Activated Memory Skill Blocks Ordinary Web Search

- **Reported from:** `feature-test-runbook.md` v0.10 "Provider Expansion",
  live local-model REPL smoke after earlier memory testing:
  `hello what's today's top news?`.
- **Observed:** The turn attempted `web_search`, but the tool result was an
  error and the assistant replied that it did not have permission to use
  `web_search` in the current session. The profile had previously exercised
  memory-router and curated-memory paths.
- **Expected:** Ordinary root turns should be able to use `web_search` when
  web policy and network access allow it. If web access is unavailable, the
  operator should see a clear web/network/policy error, not a stale active-skill
  permission denial from an unrelated earlier memory turn.
- **Likely area:** `crates/allbert-kernel-services/src/lib.rs` memory-routing
  activation and active-skill state management; active-skill tool-policy denial
  wording; prompt/tool catalog filtering while a skill fence is active;
  `docs/operator/feature-test-runbook.md` v0.10 and
  `docs/operator/adaptive-memory.md`.
- **Decision:** Treat this as a product bug plus validation gap. Keep
  active-skill tool fences for explicitly active skills, but make automatic
  memory-router activation current-turn scoped. Do not let `web_search` bypass
  the active-skill policy globally, and do not add a new reset or
  skill-management command in the v0.15.1 patch unless the scoped cleanup fix
  proves insufficient.
- **Repair scope:** Snapshot active skills before memory routing, track only
  skills newly activated by automatic memory routing, and remove those
  auto-activated skills on every root-turn exit path while preserving any skill
  that was already manually active. Improve active-skill tool-denial text so it
  names the blocked tool and active skill. Prefer filtering advertised tool
  declarations and prompt catalog entries while an active-skill fence is in
  force so the model is not invited to call tools that will be denied.
- **Validation:** Provider-free tests prove auto-routed `memory-curator`
  activation is cleared after normal and erroring root turns; manually
  pre-active skills are preserved; a memory-triggering turn followed by a
  general current-info turn is not fenced to memory tools; explicit
  active-skill denial names the blocked tool and active skill; and filtered tool
  catalogs hide disallowed tools while a skill fence is active. Implemented
  validation passed on 2026-04-28: `cargo fmt --check`; focused
  `allbert-kernel-services` memory-routing and active-skill tests; full
  `env -u RUSTC_WRAPPER cargo test -q`; workspace clippy with `-D warnings`;
  `tools/check_doc_reality.sh`; and `git diff --check`. Manual retest should
  run a memory-router turn, then ask `hello what's today's top news?`; the
  result should be search-backed current information or a clear web/network
  error, never an active-skill permission denial. `/telemetry` between ordinary
  turns should not show a stale auto-activated memory skill.
- **Status:** closed on 2026-04-28 for the stale active-skill cleanup. The
  operator retest no longer showed `web_search` blocked by the auto-activated
  memory skill. The remaining v0.10 current-info failure is tracked separately
  as M10 because the model answered in prose without calling the available
  search tool.

## M10 - Router Should Produce a Structured Turn Plan

- **Reported from:** `feature-test-runbook.md` v0.10 "Provider Expansion",
  live local Gemma4 REPL smoke:
  `say hello` followed by `what's today's top news?`.
- **Observed:** The basic chat turn worked. The current-info turn did not emit
  `[activity] calling tool web_search`; instead, Gemma4 answered in prose that
  it did not have direct access to web search in the current session. This is a
  different failure from M9: no active-skill denial or tool result appeared.
- **Expected:** For turns that plainly need a capability before a truthful
  answer, Allbert's router should emit an internal structured turn plan rather
  than only a broad intent. Fresh public-information prompts are the first live
  failure, but explicit operator commands such as `web search for ...` are the
  cleaner acceptance case: when `web_search` is available under policy, the
  runtime should guide or require that tool before answering. If search is
  unavailable because of policy, network, or provider/runtime failure, the
  operator should see that real failure. A no-tool prose apology is not a
  successful v0.10 tool-routing smoke.
- **Likely area:** Router schema and prompt in
  `crates/allbert-kernel-core/src/intent.rs` and
  `crates/allbert-kernel-services/src/lib.rs`; prompt assembly and tool-call
  retry in `run_turn_for_agent_inner`; tool catalog filtering from M9; v0.10
  runbook expectations; ADR 0030 router contract; ADR 0096 local-model
  tool-call reliability.
- **Research note:** Contemporary router patterns separate "classify the
  request" from "select the next capability." LlamaIndex routers select among
  engines, OpenAI Agents handoffs route to a specialized agent, AutoGen's
  selector group chat chooses the next speaker, and model tool APIs expose
  `auto`, required, forced, or subset tool-choice modes. Allbert should take the
  same lesson without copying those systems wholesale: its router should produce
  a bounded, traceable plan that the kernel validates against local policy.
- **Decision:** Treat this as a router/runtime design gap, not a one-off prompt
  typo. The current router can classify broad intent and draft terminal
  schedule or explicit-memory actions, but it cannot express a non-mutating
  execution plan such as "answer directly," "clarify first," "retrieve local
  memory first," "use fresh external evidence," or "call this available tool
  before answering." v0.15.1 should add richer router intelligence without
  turning the router into an autonomous executor, without expanding terminal
  router mutations, without bypassing active-skill policy, and without turning
  web search into automatic memory capture.
- **Repair scope:** Extend the internal router output with a bounded
  non-mutating turn plan. The schema should stay enum- and allowlist-based, for
  example:
  - `execution_path`: `answer_direct`, `clarify`, `tool_first`,
    `terminal_action`.
  - `required_capabilities`: bounded values such as `clear_web`,
    `memory_search`, `rag_search`, `local_files`, `jobs`, `identity`, `inbox`,
    `profile`, `user_input`, or `subagent`.
  - `tool_strategy`: `none`, `prefer`, `require_one`, or `require_any`.
  - `preferred_tools` / `required_tools`: validated against the kernel tool
    registry and then intersected with active-skill policy.
  - `evidence_policy`: `none`, `prefer_local`, `require_local`,
    `require_fresh_external`.
  - `mutation_risk`: `read_only`, `profile_write`, `external_effect`.
  Existing schedule and explicit-memory drafts remain the only terminal router
  actions in this patch.
- **Web-search behavior:** `web_search` is already a built-in kernel tool. It
  accepts `{ "query": "..." }` plus an optional `record_as` for explicit
  research-memory staging, calls DuckDuckGo HTML through the central web policy,
  and returns rendered search results to the model. M10 should make the router
  plan point at that existing path:
  - For `web search for rust shell library`, route as a read-only `tool_first`
    turn with `required_capabilities = ["clear_web"]`,
    `required_tools = ["web_search"]`, and `evidence_policy =
    require_fresh_external`.
  - For `what's today's top news?`, route the same way based on freshness and
    public-world cues, while avoiding local/meta "today" prompts such as cost,
    session history, or profile state.
  - Prompt assembly should tell the root model that the turn plan requires
    `web_search` before answering and should include a concise query hint when
    one is available.
  - If `web_search` is hidden by active-skill policy or denied by web policy,
    the runtime should surface that policy reality instead of encouraging a
    bypass.
  - If the model responds with a no-web/no-browse prose apology and no tool call
    while the validated plan requires `web_search`, retry once with direct
    tool-call guidance. If the retry still does not call the tool, treat that as
    a failed tool-routing turn rather than accepting the apology as success.
  - Do not auto-stage search results into memory. `record_as` remains explicit
    and belongs to ADR 0053.
- **Validation:** Provider-free router/schema tests should cover explicit web
  search commands, current-info prompts, negative local/meta "today" cases,
  unchanged schedule/explicit-memory action validation, and rejection of unknown
  tools/capabilities. Kernel tests with scripted model responses should prove a
  `tool_first` web plan adds web-search guidance, a no-tool no-web apology
  triggers exactly one retry, retry can produce a `web_search` call, active
  skill fences still hide or block `web_search`, and web-search results are not
  staged unless `record_as` is explicit. Manual retest should show `[activity]
  calling tool web_search` for both `web search for today's top news` and
  `what's today's top news?`, followed by search-backed output or a real
  web/network/policy error.
- **Implemented:** Added structured turn-plan fields to `RouteDecision`,
  additive defaults for legacy/no-plan router JSON, lexical enrichment for
  explicit web-search and fresh public-info prompts, turn-plan prompt guidance,
  and a bounded missing-required-tool retry that shares the existing retry
  budget. The runtime validates planned tools against the kernel registry and
  active-skill policy before asking the model to call them. Search results still
  do not stage memory unless `record_as` is explicit.
- **Follow-up:** A live retest with `what's today's top news? use web search`
  still produced a no-tool policy apology. M10 now also synthesizes the
  clear-web `tool_first` plan when no LLM router decision exists, including
  rule-only or router-fallback paths, so explicit/current web-search prompts do
  not depend on a successful router sub-call. If a live transcript still says
  `web_search` is not permitted, run `/telemetry`; an active skill or session
  tool allowlist may be intentionally hiding the tool.
