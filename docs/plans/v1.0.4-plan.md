# Allbert v1.0.4 - Packaged Browser Recovery

Status: **approved; M1-M4 candidate assembled; clean-SHA gate evidence controls tag readiness.**
The operator authorized this immutable hotfix on 2026-07-20 after the published
v1.0.3 macOS rehearsal proved that browser runtime ownership was neither shipped
nor declared: the artifact carried the Allbert bridge but no resolvable host
Playwright/Chromium contract. The `v1.0.3` tag and GitHub Release remain
immutable. v1.0.4 exists to satisfy the binary acceptance that v1.0.3 published
but did not complete.

## Purpose

v1.0.4 is a recovery release, not a new feature release. It must:

1. Keep Node, Playwright, Chromium, and their caches out of every Allbert
   artifact. The binary contains the reviewed Allbert bridge/manifests only;
   external runtimes come from operator-approved host package paths.
2. Make the packaged driver consume explicit host Node, Playwright module, and
   browser executable paths; retain `node_unavailable` and add an actionable
   `playwright_unavailable` diagnostic. Discovery, doctor, and session execution
   must never run npm, npx, an OS package manager, or a downloader.
3. Turn the v1.0.3 escape into a permanent release gate: each matrix artifact
   must prove the forbidden runtime trees are absent and then pass a live
   `browser_doctor` launch against separately provisioned host dependencies and
   a disposable Allbert Home.
4. Repair the operator-flow drift found during the real v1.0.3 rehearsal:
   existing-tap refresh, daemon/TUI single-writer sequencing, `localhost`
   browser origin, explicit browser/research enablement and doctor, and Linux
   container verification against the published cosign bundle.
5. Publish, cosign, tap-fill, and validate v1.0.4 across macOS and both Linux
   artifacts. Stop after those agent-runnable rows and ask the operator to run
   real-host Linux and WSL2. Close only after those results and every remaining
   product row have an operator disposition.

The hard acceptance bar is this Purpose section. Nothing here may be deferred
without an explicit operator decision recorded in Build Progress.

## Why v1.0.4 Exists

The source release was healthy and publication plumbing worked, but product
acceptance did not:

| Evidence | Recorded result |
| --- | --- |
| v1.0.3 tag commit | `329b9d28551315e0a82962ad306927ccd86ee62a` |
| v1.0.3 CI | run `29797899746`, gate/build/rehearsal/publish green |
| Published release | `https://github.com/lexlapax/allbert-assist/releases/tag/v1.0.3` |
| Tap fill | `lexlapax/homebrew-allbert` commit `28ef6c22251519578feaaa978dea1d38126820d8` |
| macOS packaged doctor | FAIL: `Cannot find module 'playwright/package.json'`; persisted category `bridge_exited` |
| Artifact inspection | `bridge.js`, `package.json`, and `package-lock.json` present; `node_modules` and a Chromium executable absent |
| Local evidence root | `/tmp/allbert-v103-evidence.81IzF5/`, including `macos-browser-doctor.json` |

The failure is release-specific: checkout external smoke could use an already
installed local package tree, while the packaged contract neither supplied nor
declared a resolvable external runtime. `scripts/smoke/artifact_smoke.sh`
checked plugin registration but never launched the packaged browser. The green
v1.0.3 matrix therefore could not detect the missing ownership contract.

Moving or replacing `v1.0.3` would make published checksums, signatures,
release assets, and the tap lie about immutability. The only valid repair is a
new version with a new tag and artifacts. This plan, the v1.0.3 transfer entry,
the roadmap, CHANGELOG, and ADR 0040 amendment are the durable decision trail.

## Scope And Locked Decisions

- Product scope is only the unmet v1.0.3 binary acceptance and directly exposed
  release-runbook defects. No v1.1 feature or parked v1.0.3 optimization enters.
- `bridge.js`, `package.json`, and `package-lock.json` remain reviewed Allbert
  source. `node_modules`, `.local-browsers`, browser executables, Node, and
  package-manager caches are forbidden artifact content even when present in a
  developer checkout. Release staging must exclude them rather than relying on
  `.gitignore` or a clean builder.
- Node, the Playwright module, and Chromium/Chrome are explicit host
  prerequisites. The driver resolves Node from `browser.driver.node_path` or
  PATH, accepts a host module root through `browser.driver.node_module_path` /
  `NODE_PATH`, and uses `browser.driver.binary_path` for the OS browser.
- **Operator decision (2026-07-21):** Playwright is installed outside the
  artifact as a pinned host-managed npm package:
  `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1 npm install --prefix <host-root>
  --ignore-scripts --no-audit --no-fund --no-save playwright@1.58.2`.
  Node (18+) and Chromium/Chrome come from OS packages. Each validation row
  records the exact Node, Playwright, and browser versions/paths. Playwright's
  upstream caveat that custom executables are not its bundled-browser support
  path is accepted only when the live doctor/product smoke passes on that host.
- Artifact assembly does not access npm or a browser CDN. Host provisioning is
  an operator/package-manager step before Allbert starts; installed discovery,
  doctor, research, and action execution remain unable to install or download
  dependencies. An empty package-manager/download audit is a gate.
- The 317,912,214-byte bundled macOS prototype is rejected implementation
  evidence, not a size exception. Final artifacts should remain near the direct
  Allbert dependency baseline; every target size is still measured and a
  target more than 5% larger than its v1.0.3 counterpart blocks release absent
  an explicit operator exception.
- v1.0.3 remains published but **not accepted/closed**. v1.0.4 inherits its
  incomplete macOS browser, real-host Linux, WSL2, and any product-smoke rows.
  Previously green source gates may be cited as history but are rerun at the
  v1.0.4 candidate SHA.
- `future-features.md` is unchanged: this is correction of an active release
  obligation, not new operator demand.

## Workstreams And Serial Barriers

- **W1 — packaging boundary:** M1 exclusion and M2 host-runtime resolution.
- **W2 — proof:** M0 red fixture, M3 artifact structure/live-doctor smoke, and
  `release.v104` wiring.
- **W3 — operations/docs:** M4 runbook, install prerequisite, version,
  dependency review, and evidence reconciliation.
- W1 and W3 may proceed in parallel after M0. M3 waits for M1/M2. M5 waits for
  W1-W3 to rejoin and for the complete gate cascade. M6 starts only after the
  immutable tag publishes all three artifacts. M7 is an operator-held barrier.
- Gates run serially; release/artifact builds never share mutable npm/browser
  caches across targets.
