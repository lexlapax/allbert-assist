# Allbert v0.23 Request Flow — Jido State-Machine Convergence

## Purpose

This document describes the operator-visible request flow during and
after v0.23 implementation. **There is no new request flow in v0.23.**
The milestone is a pure internal refactor: `Confirmations.Store` and
`Jobs.Scheduler` become `Jido.Agent` modules instead of plain
`GenServer` modules. Operator-visible output is unchanged.

The "request flow" content of this document is therefore:

- Documentation of which flows are exercised by the conversion (the
  acceptance surface) and must continue to behave identically.
- The Milestone Update Log where each completed milestone records
  actual conversion notes, lifecycle-hook decisions, and any behavior
  that was deferred.

## Acceptance Surface (Flows That Must Continue Unchanged)

### Confirmation Lifecycle

1. Operator runs an action that triggers a confirmation (e.g.,
   `summarize https://example.com` via `mix allbert.ask`).
2. `Actions.Runner.run/3` invokes the action; Security Central returns
   `:needs_confirmation`.
3. `Confirmations.Store` (now a Jido.Agent) creates a pending record
   via the `:create_pending` command. SQLite is authoritative; agent
   state is a cache.
4. Operator runs `mix allbert.confirmations list` and sees the pending
   record.
5. Operator runs `mix allbert.confirmations approve <id>`.
6. The action approval re-checks Security Central, resumes the
   original target action, records the outcome, and updates the
   confirmation record to `:approved`.
7. Agent emits `allbert.confirmation.approved` (same as v0.22 main).
8. Trace section `## Confirmation` appears in the turn's trace
   markdown.

### Job Lifecycle

1. Operator runs `mix allbert.jobs schedule --action <name> --cron
   "..." --name "..."`.
2. `Jobs.Scheduler` (now a Jido.Agent) accepts the `:schedule`
   command, persists to SQLite via `Jobs.Repo`, computes next run,
   and updates its in-memory `next_run_index`.
3. On the next cron tick, the agent's `:tick` command (driven by
   `Jido.Action` `{:emit_after, ...}` directive) evaluates due jobs.
4. Each due job's target action runs through `Actions.Runner.run/3`.
   Risky targets create confirmations via the same flow above.
5. Job run records persist to SQLite `job_runs` table.
6. Operator runs `mix allbert.jobs list` / `show` / `pause` /
   `resume` / `cancel`. All produce identical output to v0.22 main.

### Telegram Confirmation (v0.16 Surface)

1. Operator sends a Telegram message that triggers a confirmation.
2. The Telegram channel adapter renders the Approval Handoff with
   inline buttons.
3. Operator clicks Approve / Deny; the callback resolves via the
   v0.07 confirmation actions.
4. The agent updates state; SQLite persists; the Telegram adapter
   delivers the result message.

### Memory Delete Confirmation (v0.21 Surface)

1. Operator runs `mix allbert.memory delete <path>`.
2. `delete_memory_entry` requests `:needs_confirmation` per the
   `memory.delete_requires_confirmation` setting.
3. Confirmation creates and resolves through the same flow above.
4. On approval, the markdown entry is archived to
   `<ALLBERT_HOME>/memory/deleted/<YYYY-MM>/`.

### Settings Changes Affecting Agent State

1. Operator runs `mix allbert.settings set confirmations.ttl_seconds
   3600`.
2. Settings Central persists the change.
3. New confirmations created after the change use the new TTL.
4. Existing pending confirmations retain their original TTL (no
   retroactive adjustment).

## Lifecycle Hook Wiring

For reference, the Jido.Agent lifecycle hooks used in v0.23:

### `Confirmations.Store`

| Hook | Use |
|------|-----|
| `on_before_validate_state` | Prune expired entries from in-memory caches. |
| `on_after_cmd :create_pending` | Emit `allbert.confirmation.created` signal; update `pending_by_action` index. |
| `on_after_cmd :approve` | Emit `allbert.confirmation.approved` signal; record audit; remove from in-memory caches. |
| `on_after_cmd :deny` | Emit `allbert.confirmation.denied` signal; record audit; remove from caches. |
| `on_after_cmd :expire` | Emit `allbert.confirmation.expired` signal; record audit; remove from caches. |
| `on_error` | Log error; do not crash the agent (preserves v0.07's robustness against single-record failures). |

### `Jobs.Scheduler`

| Hook | Use |
|------|-----|
| `on_before_validate_state` | Validate schedule integrity (no jobs with both `:paused` and `due_at` set). |
| `on_after_cmd :schedule` | Compute next run time; update `next_run_index`. |
| `on_after_cmd :cancel` | Remove from `due_queue` and `next_run_index`. |
| `on_after_cmd :pause` | Move job to `paused` map; remove from `due_queue`. |
| `on_after_cmd :resume` | Compute next run; add back to `due_queue`. |
| `on_after_cmd :tick` | Recompute `due_queue`; schedule next tick via `:emit_after`. |
| `on_error` | Log error; do not crash on a single tick failure (next tick retries). |
