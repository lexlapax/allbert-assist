# Allbert v0.23 Request Flow — Jido State-Machine Convergence

## Purpose

v0.23 does not add a new operator feature. It changes the internal substrate
for confirmations and scheduled jobs while preserving the request flows that
already exist.

This document describes the runtime/user flows that must continue to work
after implementation. Implementation details belong in
`docs/plans/v0.23-plan.md`; this file stays focused on what an operator or
runtime caller observes.

## Pre-Flight For User Testing

Use a disposable Allbert Home and initialize the local SQLite database before
running smoke flows:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v023-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix cmd --app allbert_assist mix ecto.migrate.allbert
```

The confirmation records in these flows live under
`$SMOKE_HOME/confirmations`. Scheduled jobs and runs live in the local SQLite
database for that Allbert Home.

## Flow 1: Confirmation Lifecycle

1. The operator asks for work that requires confirmation, such as a URL
   summary or another risky registered action.
2. The runtime selects a registered action and calls
   `AllbertAssist.Actions.Runner.run/3`.
3. Security Central returns `:needs_confirmation`.
4. The target action creates a durable confirmation record and does not perform
   the target side effect.
5. The operator runs `mix allbert.confirmations list` and sees the pending
   request.
6. The operator runs `mix allbert.confirmations show <id>` and sees the same
   redacted target summary, origin metadata, permission, risk, and approval
   instructions as before v0.23.
7. The operator approves or denies through
   `mix allbert.confirmations approve|deny`.
8. Approval re-checks Security Central and resumes only the original target
   action. Denial records the denial and performs no target work.
9. The resolved record moves to the resolved confirmation area and audit output
   is appended.

Expected operator result: identical confirmation status, display fields, audit
shape, and trace behavior to v0.22.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask "summarize https://example.com"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations show <id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations approve <id> --reason "v0.23 smoke"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --resolved
```

## Flow 2: Confirmation Expiry

1. The operator or scheduler creates a confirmation-required request.
2. The confirmation reaches its configured TTL.
3. The operator runs `mix allbert.confirmations expire`.
4. Expired pending records resolve as expired; target work is not performed.
5. `mix allbert.confirmations list --resolved` shows the expired status.

Expected operator result: same expiry semantics and audit shape as v0.22.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.settings set confirmations.default_ttl_minutes 1
ALLBERT_HOME="$SMOKE_HOME" mix allbert.ask "summarize https://example.com"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations expire
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --resolved
```

## Flow 3: Scheduled Job Lifecycle

1. The operator creates a job through `mix allbert.jobs create ...`.
2. The job is persisted in SQLite.
3. The operator lists/shows the job through `mix allbert.jobs`.
4. A manual or scheduled run claims a durable `scheduled_job_runs` row.
5. The job target enters the same runtime/action path as an interactive
   request.
6. The run finishes as `completed`, `failed`, `skipped`, or
   `needs_confirmation`.
7. The operator inspects run history with `mix allbert.jobs runs <job-id>`.
8. Pause/resume/cancel behavior remains unchanged.

Expected operator result: same job statuses, run records, CLI output, and trace
metadata as v0.22/v0.13.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs create runtime-prompt memory-list \
  --prompt "list my memory entries" --manual --user local
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs list --user local
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs run <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs runs <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs pause <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs resume <job-id>
```

## Flow 4: Risky Scheduled Job

1. The operator creates a scheduled or manual job whose target requires
   confirmation.
2. The scheduler/manual run claims the run row.
3. The target action returns `:needs_confirmation`.
4. The job is marked blocked or needs-confirmation according to existing v0.13
   behavior.
5. The operator approves or denies through the confirmation CLI.
6. Approval resumes through registered confirmation actions; the job does not
   self-approve.

Expected operator result: same blocked-job and confirmation handoff behavior
as v0.13/v0.22.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs create runtime-prompt risky-url \
  --prompt "summarize https://example.com" --manual --user local
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs run <job-id>
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations approve <id> --reason "v0.23 job smoke"
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs show <job-id>
```

## Flow 5: Restart And Rehydration

1. The operator creates a pending confirmation and a job.
2. The supervised application is restarted.
3. The operator lists confirmations and jobs again.
4. Pending confirmations reappear from Allbert Home files.
5. Jobs and run history reappear from SQLite.

Expected operator result: no pending confirmation or scheduled job disappears
because process memory was lost.

Smoke:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --all
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs list --user local
# Restart the Mix/iex process or supervised app.
ALLBERT_HOME="$SMOKE_HOME" mix allbert.confirmations list --all
ALLBERT_HOME="$SMOKE_HOME" mix allbert.jobs list --user local
```

## Flow 6: Existing Consumers

v0.23 must preserve confirmation/job behavior for existing consumers:

- CLI confirmation management.
- Jobs LiveView and confirmation LiveView surfaces.
- Telegram/email confirmation callback simulation where available.
- Memory delete/prune/promotion confirmations.
- StockSage `RunAnalysis` confirmation flow after v0.22 acceptance.

Expected operator result: these consumers continue to use registered actions
and do not learn about internal Jido.Agent mechanics.

## Failure And Safety Cases

| Case | Expected behavior |
| --- | --- |
| Confirmation agent restarts with pending YAML files | Pending records remain listable and approvable. |
| Confirmation pending file is malformed | List/expire skips or reports bounded error without crashing the app. |
| Confirmation was already resolved | Approve/deny returns existing v0.07 resolved/not-pending behavior. |
| Scheduler agent restarts | Due jobs are read from SQLite on the next tick/run-once. |
| Scheduler disabled by config | `run_once` returns the same disabled summary as before. |
| Job target needs confirmation | Job blocks/records needs-confirmation; no self-approval. |
| Blocked job is manually run | Existing blocked-job rejection behavior is preserved. |
| Debug Jido metadata exists internally | Default CLI/trace output does not expose it. |
