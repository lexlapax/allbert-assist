## Readiness Sweep

Status: implementation-ready after this readiness patch. M1 owns the shared
substrate and the locked decisions in §"M1 Locked Decisions"; M2-M5 acceptance
and lane annotations consume those decisions directly.

This version is the first to introduce a long-lived plugin-owned OTP process
(the browser session) instead of stateless per-action work, and the first to
mix native runtime concerns (the driver binary) with operator-visible
capability. The readiness bar is therefore higher than "the milestones are
plausible": the plan must pin the URI/grant model, permission/operation-class
additions, settings namespace, driver/OS posture, network-policy seam, and
trace shape before code starts.

Result:

- **Vision fit.** v0.43 keeps the Allbert kernel small: a browser plugin owns
  the driver subprocess; core gains no browser dependency. Page content is
  descriptive, not authority. Browser results are evidence, never memory
  truth, until manually promoted through existing memory actions. The
  workspace renders results through the v0.27 SurfaceProvider contract.
- **Roadmap fit.** v0.43 correctly follows v0.42 MCP-first integration pack
  and precedes v0.44 channel pack 1 (Discord/Slack). It supplies the
  "everyday research" capability the vision line at
  `docs/plans/allbert-jido-vision.md` §"v0.43" promises.
- **Past-plan fit.** v0.10 `external_network_request` (ADR 0011), v0.11
  inert `summarize_url`/`inspect_document`, v0.17 plugin contract, v0.27
  action DSL, v0.40 MCP doctor pattern, v0.41 lane annotations, and v0.42
  pre-flight pattern (HttpPolicy on every fetch) are all consumed without
  re-litigation.
- **Future-plan fit.** v0.44 channel approval-primitive amendment must cover
  browser navigation confirmations (`:typed_command` and `:button` are
  enough); v0.45 Plan/Build can compose browser actions as workflow steps

  may mine v0.43 redacted trace envelopes as one pattern source (raw content
  remains out of bounds); v0.49 vision/image-generation will reuse the
  bitmap+metadata screenshot shape introduced here.
- **ADR fit.** ADR 0040 deepening, ADR 0013 amendment for `browser://`, and
  no new ADRs beyond those. v0.36 OTP sandbox is for *generated* Elixir/OTP
  drafts; the browser driver is a reviewed runtime dependency and does not
  share the sandbox ADR.

Readiness guardrails:

## Purpose

Let Allbert research web targets through a policy-bounded browser path. The
first release focuses on navigation, extraction, screenshots, and bounded
document extraction rather than arbitrary authenticated browser operation.

The plugin owns the driver subprocess; core never spawns an unbounded
browser. Each operation is a separate Resource Access operation class with
its own safety floor. Remembered grants are per-domain and per-operation.
Browser content is descriptive evidence, never authority — neither for the
agent nor for Allbert memory.

## M1 Locked Decisions

The first milestone preserves these six decisions. They are no longer open
defaults; later milestones may revisit them only with an explicit re-decision
recorded in this table and in the request-flow document.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Browser driver | Playwright Chromium through a reviewed local bridge managed by `AllbertBrowser.Driver` | Wallaby/ChromeDriver is BEAM-native and familiar in Phoenix stacks but ties Allbert to one protocol generation; raw CDP minimizes dependencies but maximizes Allbert-owned driver code. |
| 2 | OS support at v0.43 | macOS + Linux only; Windows/WSL2 parked to v0.43.x | Cross-OS coverage in v0.43 doubles the driver and smoke matrix; v0.39 already established that platform coverage can land per release. |
| 3 | Headless vs headed | Headless only | Headed mode helps debugging but adds a desktop dependency and a separate security/UI posture; park it until an operator-visible browser-window plan exists. |
| 4 | Profile persistence | Ephemeral browser contexts only; cookies/localStorage/IndexedDB are discarded on session close | Persistent profiles require credential storage, profile encryption, revocation UX, and authenticated-operation policy that v0.43 deliberately does not ship. |
| 5 | Routing browser vs HTTP | Keep `external_network_request` and inert `summarize_url`/`inspect_document`; browser is used when extraction needs DOM/JS, screenshots/rendering, PDF extraction, an allowlisted browser domain, or explicit operator request | Retiring HTTP fetch would break v0.10 audit history and force simple JSON/API reads through the heavier browser plugin. |
| 6 | JavaScript default | JavaScript enabled for normal page rendering | JS-disabled is safer but loses the operator-visible value on modern sites; v0.43 instead bounds navigation, subresources, extraction, screenshots, and confirmations. |

Rationales:

- **Driver.** Context7-backed Playwright docs confirm the current primitives
  v0.43 needs: Chromium launch, ephemeral browser contexts/pages,
  `page.route`/context routing for request interception, `page.goto` with
  timeout/wait options, `page.screenshot` options such as `clip`, `fullPage`,
  `quality`, and masking, and explicit context/browser close. The bridge is a
  reviewed local runtime dependency; no package manager runs during plugin
  discovery or action execution.
- **OS support.** macOS and Linux cover the project development environment
  and common CI targets. Windows/WSL2 needs its own driver-install, path, and
  process-lifecycle evidence, so it remains a v0.43.x follow-on.
- **Headless mode.** Headless keeps v0.43 in a non-desktop runtime posture and
  avoids designing operator-visible browser-window controls before the policy
  and trace surfaces exist.
- **Ephemeral profile.** A fresh context per session keeps cookies,
  localStorage, IndexedDB, and HTTP cache out of durable Allbert state. Durable
  credentials and persistent login state remain parked.
- **Routing.** HTTP fetch remains the light path for simple API/HEAD/GET
  workflows, while browser is the explicit graduated path for rendered DOM,
  screenshots, dynamic pages, and bounded PDF extraction.
- **JavaScript.** JS-enabled rendering is required for the milestone's
  research value. Authority still comes only from registered actions,
  Security Central, confirmations, Resource Access grants, and redacted
  traces; page content remains evidence only.

## Research Grounding

The v0.43 design is grounded in:

- **Playwright and Chromium DevTools Protocol** (driver interfaces): request
  interception (`Fetch.enable`, Playwright `page.route` or context routing) is
  the mechanism by which subresource policy is enforced; closing the
  browser/context/page is the mechanism by which session state is bounded;
  `page.screenshot` accepts bounded options such as `clip`, `fullPage`,
  `quality`, and masks that let Allbert bound and redact output. None of these
  grant authority by themselves; the policy that decides whether to call them
  is Allbert's.
- **OWASP SSRF / AI Agent / Prompt Injection guidance**: treat fetched
  HTML/PDF text as untrusted data, deny private-network/loopback/metadata
  hosts at every fetch surface, deny silent redirects across domains, gate
  effectful operations on human approval. Browser-class fetches need a
  separate policy from server-side HTTP because the browser executes
  attacker-controlled JS and follows attacker-controlled redirects.
- **MCP security best practice (ADR 0038 §2)**: server-supplied text is
  descriptive metadata, never authority. The same rule applies to page
  content here.
- **Anthropic and OpenAI computer-use / browser-use docs**: explicit human
  approval for navigation scope, downloads, form submission, account
  operation. Default-deny for credential surfaces.
- **The Allbert Resource Access posture** (ADR 0011/0012/0013): one URI
  substrate, one grant store (`resource_grants.remembered`), one
  operation-class vocabulary, one consumer pattern per workflow. Browser is
  the next consumer; it does not invent a parallel security model.

Implementation consequence: browser adds a new source-tree plugin, six
operation classes, seven permission classes, one URI scheme (`browser://`),
one plugin-contributed settings namespace (`browser.*`), one navigation-policy module
(`AllbertBrowser.NetworkPolicy` parallel to `External.HttpPolicy`), one
bounded extractor family (HTML/markdown/text/PDF), one workspace panel, and
a doctor — but no new permission *vocabulary outside the seven `:browser_*`
classes*, no new grant store, no new memory namespace, no new runtime
substrate beyond the plugin's child supervisor.

## Browser Session Port And `BrowserSession` Contract

`AllbertBrowser.Session` is a supervised GenServer per active session. The
shape:

```text
AllbertBrowser.Supervisor
└── DynamicSupervisor (sessions)
    └── AllbertBrowser.Session  (one per active session_id)
        ├── owns: driver port/channel
        ├── owns: NetworkPolicy state (per-session allowed domains)
        └── exposes: navigate/extract/screenshot/click/fill/download
```

Session-port contract (`AllbertBrowser.Session.Port` behaviour):

- `start_session(opts) :: {:ok, session_id} | {:error, reason}`
- `navigate(session_id, url, opts) :: {:ok, page_meta} | {:error, reason}`
- `extract(session_id, format, opts) :: {:ok, extraction} | {:error, reason}`
- `screenshot(session_id, opts) :: {:ok, screenshot_ref} | {:error, reason}`
- `click(session_id, selector, opts) :: {:ok, page_meta} | {:error, reason}`
- `fill(session_id, selector, value, opts) :: {:ok, page_meta} | {:error, reason}`
- `download(session_id, opts) :: {:ok, download_ref} | {:error, reason}`
- `close_session(session_id) :: :ok`

The default adapter is `AllbertBrowser.Driver.Playwright` (per M1 decision 1);
an alternate adapter exists in tests as `AllbertBrowser.Driver.Stub` that
records every call and returns deterministic fixtures without a real browser.
The stub is the basis of the `release` lane; the real driver is exercised
only in `external-smoke`.

## Persistence And Cache Shape

v0.43 persists very little durably:

- **No new SQLite tables.** Browser session state is in-memory in the session
  GenServer.
- **Audit events** flow through `AllbertAssist.Security.Audit` with the
  redaction rules in ADR 0040 §"Trace and audit".
- **Cache directory.** Screenshots and extracted text payloads live under
  `<ALLBERT_HOME>/cache/browser/<session_id>/` keyed by a content-addressable
  filename. Downloads are denied at v0.43 and store nothing today; the
  cache layout reserves the namespace for a future release without
  pre-creating files. Retention is bounded by `browser.cache.max_bytes`
  and `browser.cache.max_age_ms` settings; expired cache entries are swept
  by a paused-by-default `AllbertAssist.Jobs` job (analog of v0.42's
  discovery scan; gated behind `browser.cache.sweep.schedule`).
- **Doctor state.** `<ALLBERT_HOME>/cache/browser/doctor/state.json`
  records the doctor's last result envelope (`last_verified_at`,
  `live_check_status`, redacted diagnostics). `browser_start_session`
  reads this file at session-start time per §M2.
- **Remembered grants** live in the existing
  `resource_grants.remembered` Settings Central store; no new grant table.

Add the `browser.*` namespace through `AllbertBrowser.Plugin.settings_schema/0`
and the existing `AllbertAssist.Settings.Fragments` composition path. Browser
is off by default. Security Central permission settings remain core-owned
`permissions.browser_*` keys, not plugin-owned browser settings.

    max_pages: 20

    denied_domains: []

    max_redirects: 0                   # follow only with operator re-confirm
    subresource_cdn_allowlist: []      # operator-curated extra origins
  extraction:
    max_bytes: 1048576                 # 1 MiB
    pdf_max_pages: 50
    pdf_parse_timeout_ms: 20000
  screenshot:
    max_bytes: 524288                  # 512 KiB
    full_page: false                   # invariant: read-only false at v0.43

    enabled: false                     # default denied; opt-in still confirms

## Permission, Operation, And Identity Additions

### Permission classes (`AllbertAssist.Security.Policy`)

| Permission | Default | Floor | Notes |
|---|---|---|---|
| `:browser_session_start` | `:needs_confirmation` | `:needs_confirmation` | Allocates browser process; one-time per session. |
| `:browser_navigate` | `:needs_confirmation` | `:needs_confirmation` | Per-domain grant lookup runs before confirmation. |
| `:browser_extract` | `:allowed` | `:allowed` | Pure read of already-loaded page; bounded. |
| `:browser_screenshot` | `:allowed` | `:allowed` | Bounded; credential redaction enforced. |
| `:browser_interact` | `:needs_confirmation` | `:needs_confirmation` | Click; may trigger navigation. |
| `:browser_form_fill` | `:denied` | `:needs_confirmation` | Default-deny; explicit opt-in may only raise to confirmation, never `:allowed`. |
| `:browser_download` | `:denied` | `:needs_confirmation` | Default-deny; explicit opt-in may only raise to confirmation, never `:allowed`. |

`@known_permissions` grows accordingly. `permission_classes/0` returns the
new entries in a stable order. `safety_floor/1` enforces the floor column,
and the settings schema for form fill/download allows only
`needs_confirmation` or `denied`.

### Operation classes (`AllbertAssist.Resources.OperationClass`)

- `@origin_kinds`: add `:browser_session`.
- `@operation_classes`: add `:browser_navigate`, `:browser_extract`,
  `:browser_screenshot`, `:browser_interact`, `:browser_form_fill`,
  `:browser_download`.
- `@scope_kinds`: add `:browser_session` only. Per-domain browser navigation
  grants use the existing `:url_prefix` scope on the navigated target URL.
- `@default_access_modes`: `browser_navigate => :fetch`,
  `browser_extract => :read`, `browser_screenshot => :read`,
  `browser_interact => :execute`, `browser_form_fill => :write`,
  `browser_download => :write`.

### Navigation URL credential rejection (`AllbertAssist.External.HttpPolicy` extension)

v0.42 R9 added URL credential rejection to `AllbertAssist.Mcp.ConnectSpec`
for discovered MCP server URLs. v0.43 lifts that check into the shared
`External.HttpPolicy` so every consumer inherits it — browser navigation
preflight, v0.10 `external_network_request`, v0.42 MCP server connect, and
any future fetch path.

The extension rejects, before any HTTP egress or browser navigation:

- URL userinfo (`https://user:pass@host/...`) — already rejected by
  `HttpPolicy.credentials_absent/1`; preserved unchanged.
- Query parameter names matching the credential-name set:
  `token`, `api_key`, `key`, `secret`, `password`, `bearer`,
  `access_token`, `auth`. Case-insensitive match on the parameter name.
- The opaque-blob heuristic v0.42 R9 records — long, high-entropy values
  with a credential-shaped param name produce the explicit
  `{:error, {:credentialed_remote_url, {:opaque_query_param, name}}}`
  diagnostic, matching the v0.42 `ConnectSpec` test surface
  (`mcp_connect_actions_test.exs:236-244`).

Where the check fires:

- M1 ships the `HttpPolicy` extension and the supporting tests.
- `browser_navigate` preflights through a browser navigation helper that
  builds a constrained `External.RequestSpec` and calls
  `External.HttpPolicy.validate/1`, inheriting the new rejection without
  adding a second credential-URL implementation.
- Existing `external_network_request` and MCP `ConnectSpec` callers
  inherit the same rejection; `ConnectSpec.build/2` can delegate the
  url-credentialed check to the shared module in M1 or in a follow-up
  cleanup commit (not v0.43-blocking).

Operators who need a credential in a URL must move it to `auth_ref` /
header substitution or use the `secret://` settings path (v0.42 R9 lesson).
There is no opt-in toggle to disable the rejection.

### URI scheme (`AllbertAssist.Resources.ResourceURI`)

- Add `browser` to the supported scheme set.
- Add scheme-specific normalization for `browser://session/<id>` (validate
  shape; reject any path beyond `session/<id>`; reject query strings and
  fragments).
- Add scope-URI derivation for in-session references:
  `:browser_session` normalizes to `browser://session/<id>` for lifecycle,
  trace, and audit identity. Remembered navigation grants are not stored
  against the session URI; per-domain grants use the existing `:url_prefix`
  scope on the target URL (`https://<host>/` or explicitly allowed
  `http://<host>/`).
- Add redaction/display rules: the session id is rendered untruncated in
  confirmations and `/settings`, but full target URLs use the existing
  `display_url` redaction for query parameter values.

## Operator-Visible Win

An operator can ask Allbert to research a web target, approve the navigation
scope, see extracted evidence, and inspect trace/audit records.

A first-time research request walks through:

1. Operator: "research https://example.com and summarize it".
2. Allbert: confirms a browser session start (per-domain rationale shown).
3. Operator approves; the session opens.
4. Allbert: confirms navigation to `https://example.com/` (per-domain grant
   offered with a "remember for example.com" checkbox).
5. Operator approves; the page loads.
6. Allbert extracts bounded text, captures a screenshot with credential
   inputs redacted, and renders both in the workspace browser panel.
7. Operator can review extracted text + screenshot, then ask the assistant
   to summarize or to navigate further within the same domain (no new
   confirmation; the remembered grant applies).
8. A new domain triggers a fresh confirmation.

A denied navigation fetches nothing and writes no cache. A redirect chain
that leaves the granted domain pauses for re-confirmation rather than
silently following.

## Browser-Fetch Vs HTTP-Fetch Routing

The intent agent picks between `external_network_request` (v0.10) and the
browser path using this predicate:

| Condition | Route |
|---|---|
| Operator explicitly asks for "browser", "screenshot", "render", or "what does the page look like" | Browser. |
| Target URL is in the operator's `browser.navigation.allowed_domains` allowlist | Browser preferred. |
| Extraction format is `:html` and the page is known dynamic (heuristic: any `.js`-heavy domain, single-page apps, sites in the `browser.routing.dynamic_hosts` setting) | Browser. |
| Target URL is a `.pdf` and the operator asked for a summary | Browser (uses bounded PDF extractor); falls back to a `:denied` advisory if browser disabled. |
| Target URL responds to a HEAD with `text/html` and no operator preference | HTTP path advisory (v0.11 inert) — browser is opt-in, not default. |
| Target URL is a JSON/API endpoint | HTTP path (`external_network_request`). |

The predicate is deterministic and exposed through `IntentAgent` /
`Intent.Engine`. Ownership split:

- **M1 locks** the predicate table above as the authoritative routing
  decision. No edits without an explicit re-decision recorded in this
  plan and the request-flow doc; the M1 acceptance criteria assert the
  lock.
- **M4 implements and tests** the predicate. Each row above becomes one
  intent-routing test asserting the documented input pattern maps to the
  documented route (browser action or HTTP/inert advisory). v0.42 R4 is
  the lesson — plan/CLI/operator-script drift breaks operator validation
  when the implementation diverges from the documented table.

The `browser.routing.dynamic_hosts` setting referenced in row 3 is part
of the `browser.*` settings namespace (see §"Settings").

## Why Plugin-Shaped

ADR 0017 made the plugin contract Allbert's contribution boundary. v0.43 is
the first plugin whose contributions include a long-lived OTP process per
operation (the browser session GenServer), exercising:

- the plugin child supervisor (`AllbertBrowser.Supervisor`) carrying a
  `DynamicSupervisor` of session processes;
- the plugin contributing both actions and a SurfaceProvider;
- the plugin contributing a Settings.Fragment;
- the plugin contributing a doctor;
- the plugin contributing a paused-by-default cache sweep job through
  `AllbertAssist.Jobs`.

Keeping browser in a plugin means core gains no browser dependency and the
contract used by third-party plugin authors (notes_files in v0.42 is the
shallow reference; StockSage is the depth reference) is exercised against a
heavier use case.

**No skill pack at v0.43.** The notes/files plugin (v0.42 M9) is the
canonical plugin-reference scaffold and ships two `SKILL.md` files with
canonical `metadata.allbert.*` frontmatter (v0.42 R8). The browser plugin
does not need to double up: browser is operator-driven via intent phrases
(see §"Planned Interfaces" Intent phrase corpus) and the workspace results
panel, not via skill activation. If a later release identifies a workflow
where skill-mediated browser activation pays back, it ships as a v0.43.x
add-on using the canonical frontmatter shape.

## v0.45 Workflow-Step Compatibility (Forward Pin)

v0.45 Plan/Build composes objective/workflow steps from the registered
action surface. The ten v0.43 browser actions are workflow-step-compatible
by construction:

- Every action runs through `Actions.Runner.run/3` with JSON-serializable
  params and JSON-serializable results.
- Side effects (browser process lifecycle, cache writes, confirmations)
  belong to existing v0.07 / v0.27 / `home_fs_serial` substrates that
  v0.45 already composes.
- `session_id` is opaque and server-issued (M2); a workflow step that
  needs sequential browser ops captures the session id from the
  `browser_start_session` step's result and threads it through subsequent
  steps. Workflows that omit `browser_close_session` are subject to the
  idle/lifetime timeouts from `browser.session.*` settings.
- Confirmation-gated steps (`browser_start_session`, `browser_navigate`,
  `browser_interact`, opt-in `browser_form_fill` / `browser_download`)
  pause the workflow at the standard Approval Handoff seam.

ADR 0041 (workflow YAML) needs no amendment for v0.43; the v0.43.x or
v0.45 implementation can publish v0.43 browser actions in the workflow
catalog without additional plan-side changes.

## Security Posture

- **Page content is not authority.** Extracted HTML/text/PDF/comments never
  alter agent behavior; they render as evidence (ADR 0040, ADR 0038 §2
  precedent).
- **Network policy applies at two layers.** Top-level navigation URLs pass
  through the v0.43 browser navigation preflight helper that reuses
  `External.HttpPolicy` checks; subresources and redirects pass through
  `AllbertBrowser.NetworkPolicy` enforced via the driver's request-interception
  API. Both deny private networks, loopback, link-local, cloud metadata
  endpoints, and embedded URL credentials (userinfo plus credential-shaped
  query parameter names per §"Navigation URL credential rejection").
- **Redirects fail closed.** `browser.navigation.max_redirects` defaults to
  `0`; any cross-domain redirect requires re-confirmation against the new
  host.
- **Per-domain, per-operation grants.** Stored in the existing
  `resource_grants.remembered`; matched on `https://<host>/` `:url_prefix`
  scope plus operation class. Cross-operation reuse is forbidden (ADR 0040
  §"Operation classes").
- **Default-deny on credential-bearing operations.** Form fill, download,
  and any future authenticated operation are denied unless explicitly
  tightened in settings — and even then, gated by confirmation.
- **Screenshot redaction.** Known credential-bearing input types
  (`type=password`, `autocomplete=otp`, `autocomplete=cc-number`) are
  redacted before the bitmap is encoded; the redaction is performed in the
  driver layer (DOM modification or clip masking) so the bitmap never
  contains the secret.
- **Bounded extraction.** Max bytes per page, max pages per PDF, parse
  timeout, no JS execution in PDF parser, no follow-on fetch from PDF
  references, malformed input fails closed.
- **Trace redaction.** Cookies, Authorization, full URLs with credentials,
  and sensitive query parameter values are scrubbed before any record is
  written under `memory/traces/`. Raw HTML/text/PDF/bitmap bodies live
  under `<ALLBERT_HOME>/cache/browser/<session_id>/` with retention bounded
  by settings.
- **Ephemeral profile.** Cookies, local storage, IndexedDB, and cache live
  only inside the session process; `browser_close_session` discards
  everything.
- **Plugin-owned process.** Core spawns no browser; supervision lives under
  `AllbertBrowser.Supervisor`. A driver crash takes the session, not core.
- **Doctor-gated start.** `browser_session_start` refuses to allocate when
  `browser_doctor` reports an unverified driver or missing dependency.

## Non-Goals

- No arbitrary crawling, recursive link following, or sitemap traversal.
- No automatic memory promotion from browser content.
- No broad Office/archive extraction.
- No account-changing browser actions by default.
- No headed mode in v0.43.
- No JS evaluation actions in v0.43 (`evaluate_js`, `add_init_script`,
  `expose_function`).
- No WebSocket, service worker registration, push notifications,
  background sync, geolocation, microphone, camera, or clipboard access.
- No multi-tab, multi-window, or popup orchestration; one active page per
  session.
- No persistent browser profile, no credential storage, no
  captive-portal/SSO flows.
- No retiring of `external_network_request` or v0.11 inert
  `summarize_url`/`inspect_document` in v0.43.
- No new memory namespace; manual promotion uses existing memory actions.
- No new permission vocabulary outside the seven `:browser_*` classes.

## Definition Of Done

v0.43 is done when:

- a disposable-home operator can enable browser, pass `browser_doctor`,
  approve a session start and a navigation against a chosen domain,
  receive bounded extracted evidence (HTML/markdown/text/PDF), see a
  screenshot with credential inputs redacted, and inspect redacted
  trace/audit records;
- the seven `:browser_*` permission classes and floors are enforced; the
  six operation classes are part of `Resources.OperationClass`;
  `browser://session/<id>` is parsed and rendered by `Resources.ResourceURI`;
  the `browser.*` settings namespace exists with documented invariants;
- per-domain remembered grants work end-to-end, cross-operation grant
  reuse is denied, redirects across domains fail closed, and subresource
  policy blocks denied hosts;
- `BrowserFill` and `BrowserDownload` deny by default and require explicit
  opt-in plus confirmation when tightened;
- the workspace browser results panel renders text + screenshot evidence
  and never calls the driver directly;
- the cache layout under `<ALLBERT_HOME>/cache/browser/<session_id>/` is
  swept by a paused-by-default job; raw page content never lands in
  `memory/traces/`;
- the v0.43 security eval set (stub-driver) passes; opt-in `external-smoke`
  passes against a real Playwright when the harness is available;
- `docs/plans/v0.43-request-flow.md`, `docs/plans/roadmap.md` §v0.43,
  `docs/plans/allbert-jido-vision.md` §v0.43, ADR 0040, ADR 0013,
  `docs/developer/agent-context-map.md`, `docs/operator/security-hardening.md`,
  and CHANGELOG reflect the shipped state;
- v0.44 channel approval-primitive amendment is unblocked: v0.43
  confirmations are documented as expressible in `:typed_command`,
  `:button`, and `:link`.
