# Allbert v0.43 Channel Pack 1: Discord And Slack

Request flow: `docs/plans/v0.43-request-flow.md`

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`.

## Purpose

Expand Allbert reach into team and community chat by adding Discord and Slack
channel plugins that reuse the existing channel substrate.

## Expected Inputs

- v0.16 channel adapter boundary.
- v0.17 plugin contract and shipped channel plugins.
- Existing Approval Handoff, confirmations, identity mapping, dedupe, and
  redaction posture.

## Operator-Visible Win

Allbert can be reached from Discord and Slack with safe mention handling,
threaded replies, and confirmation callbacks.

## Planned Interfaces

- `./plugins/allbert.discord/`
- `./plugins/allbert.slack/`
- Discord bot settings and identity map.
- Slack app settings and identity map.
- Channel status surfaced through existing channel list/show paths.

## Security Posture

Channels do not invent permission classes or confirmation shapes. Group/server
authorization, workspace mapping, and callback ownership must be explicit.
Approval Handoff remains provider-neutral and action-owned.

## Non-Goals

- No SMS.
- No WhatsApp, Signal, iMessage, or Matrix in this milestone.
- No channel-specific action authority.
- No private durable goal loop inside a channel adapter.

## Milestones

- M1: Discord plugin skeleton and simulated provider tests.
- M2: Slack plugin skeleton and simulated provider tests.
- M3: Identity mapping, group authorization, and threaded replies.
- M4: Confirmation callbacks and channel rendering.
- M5: Security evals and closeout docs.

## Documentation Deliverables

- Operator channel setup docs for Discord and Slack.
- Security-hardening notes for group/team channels.
- Agent context map update.

## Definition Of Done

v0.43 is done when Discord and Slack adapters can simulate inbound messages,
render replies and approvals, preserve identity/thread mapping, and pass
cross-channel spoofing, replay, group leakage, and callback-scope evals.
