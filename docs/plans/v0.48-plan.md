## Purpose

v0.48 ships the first non-text modality without creating a second provider
system. Voice is the forcing function for a general provider capability and
operator preference layer:

1. providers and model profiles become capability-aware;
2. operators can set a global primary provider/profile and ranked
   per-capability preferences;
3. STT/TTS use the same Settings Central, doctor, secrets, and provider
   routing substrate as text generation;
4. audio resources and voice actions remain policy-bounded through Resource
   Access, Security Central, confirmations, traces, and audits.

v0.49 Vision And Image Generation consumes the provider capability substrate
landed here; it does not get a separate image-provider framework.

## Locked Decisions

These choices are binding for v0.48. Changing any value requires an explicit
re-decision in this plan and in the request-flow document before code lands.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Provider substrate | Reuse the existing provider/model profile framework. Voice providers are still providers. | A parallel voice-provider stack would duplicate secrets, doctor, onboarding, settings, traces, and future vision work. |
| 2 | Generalization | Make provider profiles capability-aware rather than adding voice-only STT/TTS keys. | Voice-only keys are faster, but v0.49 would repeat the same migration for images. |
| 3 | Preference shape | Use a global primary plus ranked preferences per task/capability. | A single profile per capability cannot degrade gracefully when a provider is disabled or not doctored. |
| 4 | Capability authority | Curated catalog defaults are seed data; doctor output may report availability but never grants permission. | Pure auto-detection is provider-fragile; pure catalog metadata cannot reflect live availability. |
| 5 | Compatibility | Keep `intent.model_profile` and `intent.direct_answer_model_profile` as compatibility aliases into the new preference layer. | Immediate migration would churn existing operator settings and focused tests. |
| 6 | Voice default posture | Voice providers support four operator-selectable deployment modes: **fake** (deterministic fixture provider for tests only), **local-endpoint** (the Allbert-owned loopback local voice runtime by default, with advanced override to another OpenAI-compatible loopback server), **bundled-local** (an offline engine invoked by a bounded helper when explicitly configured), and **remote-credentialed** (cloud/provider API through Settings Central secrets). Local/offline is preferred by default. v0.48 release authority requires executable Allbert local-runtime STT/TTS, OpenAI remote STT/TTS, Gemini remote STT/TTS, and a local Ollama text model in the listen-think-speak loop. Remote STT/TTS is used only after the operator opts in. | A fake-only voice release is not operationally usable. A local endpoint that Allbert does not provide is also not a product capability. | 8 | Workspace mic | Live microphone capture is workspace-only and requires explicit per-session confirmation. | Sharing the CLI file path with live capture would blur authority and retention policy. |
| 9 | Cost visibility | v0.48 shows display-only provider/cost/usage metadata in action output and traces. Unified spend dashboards and enforcement remain parked. | A cost subsystem would expand v0.48 beyond the modality substrate. |

Deterministic provider-shape fixtures keep the gate stable while manual smokes prove operator reality. |

| 12 | Capability semantics | Capability names are routing predicates only. ## Capability And Preference Contract

v0.48 adds the following capability vocabulary:

```text
text_generation
speech_to_text
text_to_speech
vision_input
image_generation
video_input
token_streaming
embeddings
tool_use
```

Capabilities are routing predicates, not media schemas. The executable v0.48
voice capabilities are `speech_to_text` and `text_to_speech`. `vision_input`,
`image_generation`, and `video_input` are part of the shared vocabulary so
v0.49+ can reuse the substrate, but v0.48 implements only audio STT/TTS.
`token_streaming` means streaming text tokens or partial text deltas; realtime
audio sessions are represented as transport metadata and are not release
authority for v0.48.

Model profiles may carry optional media metadata. This metadata helps adapters,
doctors, and operator UI explain why a profile was selected, but it never
grants permission:

```yaml
media:
  input_modalities: [audio]
  output_modalities: [text]
  transport_modes: [request_file, local_endpoint]
  deployment_mode: local_endpoint
  audio_formats_supported: [wav, flac]
  audio_sample_rates_supported: [16000, 24000]
  max_audio_bytes: 10485760
  max_audio_duration_ms: 120000
```

Known `deployment_mode` values are `fake`, `local_endpoint`, `bundled_local`,
and `remote_credentialed`. Known `transport_modes` are `request_file`,
`live_upload`, `realtime_session`, `local_endpoint`, and `bundled_local`.
`realtime_session` is metadata only in v0.48; release validation uses bounded
request flows against real providers. `fake` exists only for explicit
automated fixtures.

Fixture-only fake profiles may exist in test settings, but they must not appear
in operator default preferences or release-validation instructions.

Selection is via a shared resolver, e.g. `Models.for/2`:

1. resolve task preference first when the caller asks for a task such as
   `:coding` or `:direct_answer`;
2. otherwise resolve a capability preference such as `:speech_to_text`;
3. skip disabled, missing, or incapable profiles;
4. fall back through the ranked list, then to `model_preferences.primary`
   only when the primary advertises the requested capability;
5. return diagnostics when no capable profile is available.

**Call-failure fallback (Locked Decision 3 made concrete).** Selection above is
availability-aware only through on-demand doctor data, so a selected real
provider can still fail at call time (unreachable local endpoint, remote
timeout/5xx). `transcribe_voice`/`synthesize_voice` therefore make **one bounded
pass** down the resolved ranked list: on a **retryable** adapter error
(connection refused, timeout, 5xx) they advance to the next capable profile; on
a **non-retryable** error (auth/4xx, permission denial, bounds rejection) they
stop and surface it. Each attempt re-applies that profile's deployment-mode
floor and Resource Access checks. The pass never re-orders, never loops, and
records which profiles were skipped and why. This is what makes the
`[local, openai, gemini]` ranking degrade gracefully when a provider is down.

The resolver grants no authority. Action permissions, Resource Access, and
provider secrets remain separate boundaries.

### Provider Shape Matrix

The implementation should avoid hard-coding one vendor's media model as the
Allbert contract:

| Provider shape | Profile signal | v0.48 handling |
|---|---|---|
| Request STT/TTS API | `speech_to_text` or `text_to_speech`, `transport_modes: [request_file]` | Supported through bounded file/capture input, transcode, and adapter call. |
| Realtime audio session | `speech_to_text`/`text_to_speech`, `transport_modes: [realtime_session]` | Metadata only unless wrapped by the bounded request flow; not release authority. |
| Multimodal audio/video input model | `input_modalities: [audio]` or `[video]`, usually no STT/TTS guarantee | Not a v0.48 voice adapter by itself. It may inform future `audio_understanding`/`video_input` work. |
| Local text/vision endpoint | `text_generation`/`vision_input`, `deployment_mode: local_endpoint` | Reuses provider substrate but does not imply local audio support. |
| Dedicated voice provider | `speech_to_text` and/or `text_to_speech` plus format/limit metadata | Supported when configured as a voice-capable profile. OpenAI and Gemini are required remote implementations for v0.48 release authority. |
| Allbert local voice runtime endpoint | `speech_to_text` and/or `text_to_speech`, `deployment_mode: local_endpoint`, `base_url` loopback only | Required real local v0.48 path. The reference contract is the Allbert-owned OpenAI-compatible loopback endpoint for `/v1/audio/transcriptions` and `/v1/audio/speech`. External compatible loopback servers are advanced override targets, not release authority. |
| Local Ollama text endpoint | `text_generation`, `deployment_mode: local_endpoint` | Required local middle-of-chain validation for listen -> think -> speak. Ollama is also a possible local STT backend for the Allbert local voice runtime when an audio-capable Ollama model is configured; it is not the complete voice endpoint because current Ollama docs/source do not provide `/v1/audio/speech`. |
| Bundled offline engine | `deployment_mode: bundled_local` plus `local_runtime_present` doctor field | Deferred executable path unless packaging/model distribution is explicitly added. |

## Voice Provider Adapter Contract

The resolver selects a *profile*; an **adapter** executes the STT/TTS call.
v0.48 defines `AllbertAssist.Voice.ProviderAdapter`, mirroring the v0.43
browser `AllbertBrowser.Driver` behaviour + `Driver.Stub`/`Driver.Playwright`
config-selected seam:

```elixir
@callback transcribe(profile :: map(), request :: map(), opts :: keyword()) ::
            {:ok, transcript_packet} | {:error, term()}
@callback synthesize(profile :: map(), request :: map(), opts :: keyword()) ::
            {:ok, audio_packet} | {:error, term()}
@callback doctor(profile :: map(), opts :: keyword()) :: {:ok, map()} | {:error, term()}
```

Implementations are selected from the resolved profile's
`media.deployment_mode`:

- **Fake** — fixture transcripts/audio for deterministic tests only (the
  `Driver.Stub` analog). No network, no real audio. It is not release
  authority for the voice product.
- **Local-endpoint** — calls the Allbert-owned loopback local voice runtime by
  default; no credential. Operators may override the base URL to another
  OpenAI-compatible loopback server, but v0.48 release authority is the
  Allbert runtime.
- **Bundled-local** — an offline engine invoked through a bounded helper when
  explicitly configured. v0.48 does not require shipping a concrete bundled
  engine in the release gate.
- **Remote** — a credentialed provider via Settings Central secrets. v0.48
  release authority requires concrete OpenAI and Gemini adapters. Anthropic is
  not a voice adapter in v0.48 because the current Claude API does not provide
  native STT/TTS endpoints.

The v0.48 local voice endpoint contract is intentionally minimal and
Allbert-owned:

- `GET /v1/models` reports only the local STT/TTS aliases whose real backends
  are currently available.
- `GET /v1/doctor` reports redacted local-runtime and backend diagnostics.
- `POST /v1/audio/transcriptions` accepts bounded audio and returns
  `{transcript, language?, duration_ms?, usage?}`.
- `POST /v1/audio/speech` accepts bounded text and returns audio bytes plus
  `{mime_type, duration_ms?, usage?}` metadata.

The localhost contract is a configuration target, not a permission grant. The
doctor may report it reachable; `transcribe_voice` and `synthesize_voice` still
run through Resource Access and action permission floors.

Remote voice adapters use provider-native shapes behind the same action
contract:

- **OpenAI** STT uses multipart audio upload to the transcription endpoint and
  TTS uses a JSON speech request that returns bounded audio bytes.
- **Gemini** STT uses the stable provider-native `generateContent` endpoint
  with inline audio parts; TTS uses `generateContent` with audio response
  modality and decodes bounded returned audio bytes.
- **Anthropic/Claude** is text-only for v0.48 voice routing. A Claude model may
  answer the transcribed text as `text_generation`, but it must not be marked
  `speech_to_text` or `text_to_speech` until Anthropic has native audio APIs and
  Allbert implements them.

Provider HTTP policy is voice-specific but inherits ADR 0011's posture:

- local endpoint calls are loopback-only (`localhost`, `127.0.0.1`, `::1`) and
  never accept credentials, redirects, arbitrary paths, or private LAN hosts;
- remote calls are HTTPS-only, credentialed only through Settings Central
  secret refs, no query-string credentials, no redirects, no unbounded retries,
  and bounded request/response bodies;
- tests use deterministic `Req` plugs/stubs for provider request shapes;
- traces/audits record redacted host, provider profile, model, byte counts,
  duration when known, transcript hashes, and bounded usage/cost packets only.

Flow: `transcribe_voice`/`synthesize_voice` action → resolver picks a capable
profile → the **bounded ffmpeg-class transcode helper** (Locked Decision 11)
builds the provider-input spec from the profile's `audio_formats_supported` →
the adapter for that profile's mode executes → returns a packet. `0.48.0`
keeps the helper as a bounded spec boundary; concrete non-fake adapters that
need conversion must execute through that bounded shape, not through arbitrary
provider/model arguments. The adapter performs no permission decisions; the
action's permission/floor and Resource Access bounds are evaluated before the
adapter is called.

**Usage/cost packet (display-only, Locked Decision 9).** `synthesize`/
`transcribe` return a pass-through `usage` packet — never computed or
aggregated by Allbert:

```elixir
%{
  provider: "openai_tts",
  units: 1280,                 # provider-reported (e.g. characters or seconds)
  unit_kind: "characters",
  estimated_cost: nil,         # only when the provider reports it
  source: :provider_reported   # | :unavailable
}
```

`%{source: :unavailable}` is the correct packet for fake/local providers that
report no usage or cost. Allbert must not synthesize byte counts, durations, or
zero-dollar costs into this packet. The action surfaces the packet in its result
and redacted trace metadata; v0.48 adds no spend store, no aggregation, and no
budget enforcement. Eval `voice-tts-cost-metadata-display-only-001` asserts the
field is present, pass-through, and never gates execution.

| Legacy key | New key | Read | Write |
|---|---|---|---|

| `intent.direct_answer_model_profile` (single) | `model_preferences.tasks.direct_answer` (list) | returns the **head** of the list (or `primary` when empty) | legacy write sets the list to `[value]`; editing the list updates the legacy read to its head |

keys. A compatibility alias must never resolve to a profile that fails
capability validation.

## Provider Doctor

`doctor_voice_provider` reuses ADR 0047's canonical redacted envelope. It adds
optional capability fields only:

- `provider_capabilities`
- `provider_deployment_mode`
- `speech_to_text_supported`
- `text_to_speech_supported`
- `audio_sample_rates_supported`
- `audio_formats_supported`
- `provider_usage_metadata_available`
- `local_runtime_present`
- `fixture_probe_ok`

Field names match the profile `media.*` metadata (e.g.
`audio_sample_rates_supported`, `audio_formats_supported`) so curated and
probed values are directly comparable. The curated profile
`media.audio_formats_supported` is the transcode-target default; live doctor
output is advisory and may **narrow** the usable set but never widens it.

The doctor is read-only and may probe only the configured provider/profile
host. It never accepts arbitrary URLs, raw audio paths, or model-output-chosen
probe targets.

## Audio Resource And Authority Model

v0.48 accepts the audio slice of ADR 0042:

- `mic://capture/<id>` identifies workspace microphone captures.
- CLI voice input uses `file://` Resource Access over a bounded local audio
  fixture/file, not live microphone capture.
- Audio content is default-off for retention and redacted from traces by
  default.
- Audio-bounds precedence: the effective cap is `min(global, profile)` — the
  global `voice.audio.max_bytes`/`voice.audio.max_duration_ms` are a hard
  ceiling that a profile's `media.max_audio_bytes`/`media.max_audio_duration_ms`
  can only narrow, never raise.
- Traces may record bounded metadata: resource URI, duration, byte size,
  MIME type, provider profile, transcript hash, and redaction status.
- Raw audio never appears in traces, audits, diagnostics, LiveView assigns, or
  CLI output.

New permission classes:

- `:microphone_capture` - floor `:needs_confirmation` for every deployment mode
  (live capture always confirms).
- `:voice_transcribe` and `:voice_synthesize` - floor is derived from the
  resolved profile's `media.deployment_mode`:

| `deployment_mode` | Audio leaves the BEAM? | `:voice_transcribe` / `:voice_synthesize` floor |
|---|---|---|
| `fake` | no (fixtures) | `:allowed` |
| `bundled_local` | no (in-process offline engine) | `:allowed` |
| `local_endpoint` | yes (localhost socket) | `:needs_confirmation` |
| `remote_credentialed` | yes (network + billable) | `:needs_confirmation` |

Anything that puts audio on a socket confirms; a profile with no resolvable
`deployment_mode` defaults to `:needs_confirmation` (fail-closed). Floors may be
raised by operator policy but never lowered below this table.

New operation classes and origins:

- origin kind `:audio_capture`
- origin kind `:audio_file`
- operation `:microphone_capture`
- operation `:voice_transcribe`
- operation `:voice_synthesize`

## Operator-Visible Win

After onboarding, an operator can say:

- "use local for most things";
- "use qwen coder for coding";
- "use this STT provider for voice transcription";
- "use this TTS provider for spoken output".

Then:

- `mix allbert.ask --voice ./fixtures/hello.wav` transcribes a file and sends
  the transcript through the normal runtime input path;
- workspace microphone capture creates a confirmation, records redacted audio
  metadata, then transcribes approved audio;
- TTS output is a registered action with visible provider/cost metadata;
- Telegram voice notes become transcribed channel input without giving the
  Telegram adapter provider authority.

## Security Posture

- Voice input is input modality only; it does not change action authority.
- Provider/model metadata never grants permission or stores secrets.
- Microphone capture is confirmed per session.
- CLI voice accepts only explicit bounded audio files.
- Remote STT/TTS calls use provider settings and secrets through Settings
  Central.
- Audio retention is default-off; redacted metadata is the trace artifact.
- TTS is effectful and provider-backed; it reports display-only usage/cost
  metadata but does not enforce budgets in v0.48.
- Channel adapters never own STT/TTS authority.

## Non-Goals

- No always-on wake word in 1.0.
- No hidden cloud upload of audio.
- No durable audio retention by default.
- No channel permission bypass through voice.
- No Discord voice in v0.48.
- No unified cost dashboard or spend enforcement.
- No image, screenshot, or image-generation implementation; those are v0.49.
- No removal of legacy `intent.*model_profile` settings in v0.48.

## Definition Of Done

v0.48 is done when:

- provider/model profiles carry capability metadata;
- operator preferences resolve through ranked per-task/per-capability lists;
- STT/TTS uses the shared provider resolver, doctor substrate, and executable
  real local/OpenAI/Gemini provider adapters;
- CLI voice transcribes an explicit audio file through a real selected STT
  provider;
- workspace microphone capture requires confirmation;
- TTS is a registered action with display-only usage/cost metadata;
- Telegram voice notes become text input through the shared STT action;
- raw audio is bounded, default-off for retention, and absent from traces;
- the Allbert-owned local voice runtime exposes OpenAI-compatible loopback
  STT/TTS with real local backends and no credentials;
- Allbert local-runtime STT/TTS, OpenAI remote STT/TTS, Gemini remote
  STT/TTS, and local Ollama text-loop validation pass through
  `release.v048`;
- authoritative post-M8R7 `mix allbert.test release` passes before tag handoff;
- opt-in `.env` live smokes for OpenAI, Gemini, Allbert local voice runtime, and
  Ollama are documented for manual validation before tag.

Release-state notes: the `0.48.0` version stamp and the `## v0.48.0` CHANGELOG
entry stay in place but are **not a tagged release** until manual validation;
the CHANGELOG entry is finalized with the Allbert local-runtime remediation,
not bumped to a new version. `ffmpeg` (or an equivalent transcoder) is a real M8R
runtime dependency for non-native source formats (e.g. Telegram OGG/Opus); its
absence is a doctor/action diagnostic, and the deterministic `release.v048`
lane uses already-supported fixture formats so the gate never hard-requires it.
First-run onboarding must guide the operator to at least one working voice
provider: the Allbert local voice runtime with a configured local STT/TTS
backend and Ollama text model, or OpenAI/Gemini keys. Fake providers are never a
working provider.
Provider model IDs and request shapes (`gpt-4o-mini-transcribe`,
`gpt-4o-mini-tts`, Gemini STT via `generateContent` inline audio, and Gemini
TTS via `generateContent` AUDIO) are verified against live provider docs at
M8R3/M8R4 and rechecked during M8R7 manual validation before coding.
