## Purpose

Turn first-run from a prototype-grade scatter of steps into a genuine guided
onboarding capability. The current onboarding objective and web panel mostly
sequence existing Mix commands; a non-developer user still needs docs and
checkout-oriented commands to reach first value. v0.63 closes that gap by
**implementing the v0.60-designed onboarding flow** (ADR 0077/0069) as a shared
two-track wizard in web and terminal, provider/model setup with masked
credential entry and inline doctor verification, and **applying the v0.60
persona model** (ADR 0077/0075) as repo-maintained user-category profiles that
seed Settings Central defaults with explicit operator review. The wizard UX and
the persona model are inputs from v0.60; v0.63 builds and ships them on the
v0.61 overhauled surface and the v0.62 packaging.

## Operator-Visible Win

A brand-new operator launches Allbert, the web wizard auto-opens on first run
after `allbert serve`, they pick QuickStart, reach a working model through the
**First-Model Path resolved in ADR 0078** (assisted-local QuickStart by guided
Ollama install/pull, with BYOK as Advanced/fallback — never an assumed
pre-existing key or managed-hosted default), store/verify any BYOK credential
through the OS vault and inline doctor, review and apply a user-category profile,
and reach a working first chat without
hand-editing config or reading docs. The same flow is available through an
interactive terminal wizard for headless installs.

## First-Model Path (consumed from ADR 0078 — decided in v0.60 M3)

QuickStart's "fastest first chat" requires a **reachable model**, and the flow
must define — explicitly, not by assumption — what happens for a user who arrives
with **neither a hosted API key nor a local model installed**. This is the single
biggest "does it feel 1.0" risk in onboarding (the competitive zero-config bar set
by LM Studio and Jan is met by managing the first model). v0.63 **does not decide
this**: the decision was made upstream in v0.60 M3 and recorded in ADR 0078, which
selected among:

- **(a) Assisted local model.** Detect or instruct an Ollama install and one-click
  pull a small default model, so the fast path needs no key. Cost: a managed local
  dependency and model download weight.
- **(b) Managed hosted default.** Ship a default hosted free/low-tier provider so
  the user can chat before configuring their own key. Cost: an Allbert-operated
  credential/relay, its abuse/cost policy, and an ongoing-cost owner.
- **(c) Bring-your-own-key, honestly framed.** QuickStart explicitly requires a
  key or a running local model and says so up front, with the most assisted
  possible key / Ollama setup. Cost: not truly zero-config; a weaker first-value
  story versus the competitive bar.

**Explicit gate:** v0.63 does not assume a key or model already exists, and it
does not re-litigate the option. It **builds QuickStart against the path ADR 0078
resolved**: the QuickStart fast path, the Operator-Visible Win, and the
acceptance-matrix "first useful chat" criterion are all defined against the
ADR 0078 decision — never against an assumed pre-existing model.

## Guided Wizard Flow (implements the v0.60 design — ADR 0077/0069)

v0.63 implements the onboarding flow designed in v0.60 M4 (ADR 0077); ADR 0069
is the implementation ADR accepted here. The wizard is a single shared flow model
with two render surfaces:

- **One flow, two surfaces.** Web and terminal render one state machine; neither
  forks onboarding logic.
- **Two tracks.** QuickStart minimizes decisions and skips channel/integration
  setup to reach a working conversation quickly; Advanced exposes provider,
  model, channel, app/profile, and diagnostics choices.
- **Steps:** launch/source check -> provider/model setup -> user-category profile
  selection/review -> optional channel/integration setup -> first useful chat ->
  health/trust-spine confirmation.
- **Trust spine as a feature.** The wizard explains confirmation, permission,
  traces, and local inspectability as safety properties, not setup friction.

## User-Category Profile System (applies the v0.60 persona model — ADR 0077/0075)

v0.63 applies the persona model designed in v0.60 M4 (ADR 0077); ADR 0075 is the
application ADR accepted here. The shipped categories and what each persona means
come from v0.60; v0.63 realizes them as repo-maintained settings profiles.

- **Repo-maintained declarative presets** live in the source tree and are
  reviewed like other trusted project metadata.
- **Shipped categories:** `researcher`, `developer`, `writer`, `ops`,
  `general`.
- **Each profile specifies:** Settings Central defaults, suggested apps,
  suggested channels, suggested intents, and recommended model-purpose mappings.
- **Seed-only semantics.** Profiles grant no authority, lower no confirmation
  floor, and enable nothing effectful without the normal gates.
- **Explicit application.** Applying a profile always shows exactly what will be
  seeded before confirmation.

## Security Posture

No new authority. Onboarding is an experience over existing flows: every
effectful step routes through existing actions, settings, doctors, gates, and
confirmations. Profiles seed defaults only. Masked credentials write through the
v0.62 OS-vault reference model and never echo/log raw values.

## Non-Goals

- No new capability surface or permission class.
- No runtime authorization, trust tier, or per-user authority model from
  profiles.
- No learned, distilled, or auto-generated personas.
- No native desktop GUI onboarding.
- No onboarding-flow or persona-model design, and no First-Model-Path decision;
  v0.60 owns experience design (ADR 0077) and the First-Model Path (ADR 0078).
- No web-surface or affordance overhaul; v0.61 owns the surface this seats in.
- No packaging or CLI redesign; v0.62 owns entry points.
- No final product RC; v0.64 owns integrated validation.

## Definition Of Done

v0.63 is done when:

- a new operator can complete first-run onboarding through the web wizard and the
  terminal wizard over one shared flow that implements the v0.60 design;
- QuickStart is built against the ADR 0078 First-Model Path (decided in v0.60 M3)
  and reaches a working first chat through it without docs or config edits — for a
  user starting from the state that path explicitly assumes, not from an assumed
  pre-existing key;
- provider/model setup stores masked credentials through the OS vault, verifies
  inline with doctor, supports local and hosted providers, and switches providers
  without config edits;
- repo-maintained profiles seed Settings Central defaults and suggestions only
  after explicit review/confirm;
- the trust spine is surfaced as a feature;
- ADR 0069 and ADR 0075 are accepted;
- `mix allbert.test release.v063`, docs, and warning gates pass with no
  onboarding secret leak or profile-authority regression;
- no known warning/security/doc drift remains.
