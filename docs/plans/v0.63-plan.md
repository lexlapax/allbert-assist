## Purpose

Turn first-run from a prototype-grade scatter of steps into a genuine guided
onboarding capability. The current onboarding objective and web panel mostly
sequence existing Mix commands; a non-developer user still needs docs and
checkout-oriented commands to reach first value. v0.63 closes that gap by
**implementing the v0.60-designed onboarding flow** (ADR 0077/0069) as a shared
two-track wizard in web and terminal, provider/model setup with masked
credential entry and inline doctor verification, and **applying the v0.60
persona model** (ADR 0077/0075) as repo-maintained user-category profiles that
seed Settings Central defaults with explicit operator review. The wizard UX and
the persona model are inputs from v0.60; v0.63 builds and ships them on the
v0.61 overhauled surface and the v0.62 packaging.

## Operator-Visible Win

A brand-new operator launches Allbert, the web wizard auto-opens on first run
after `allbert serve`, they pick QuickStart, reach a working model through the
**First-Model Path resolved in ADR 0078** (assisted-local QuickStart by guided
Ollama install/pull, with BYOK as Advanced/fallback — never an assumed
pre-existing key or managed-hosted default), store/verify any BYOK credential
through the three-tier vault and inline doctor, review and apply a user-category
profile, and reach a working first chat without hand-editing config or reading
docs. The operator sees one next action at every point: Ready, Needs model,
Needs credentials, Needs runtime, or Needs review. The same flow is available
through an interactive terminal wizard for headless installs.

## Current Code State (the ground truth v0.63 changes)

v0.63 is **not greenfield onboarding** — it reconciles and upgrades existing
scaffolding and adds the greenfield persona layer on a mature Settings Central.
The as-built substrate this release touches (verified 2026-07-07, post-v0.62.0):

1. **Objective-backed onboarding wizard (v0.39-era, command-sequencing grade).**
   `apps/allbert_assist/lib/allbert_assist/onboarding.ex` — a plain context module
   (no process state) that frames/resumes one first-run objective on the Objectives
   runtime tables, with a hardcoded 9-step flow (`welcome_scope`,
   `pick_provider_profile`, `resolve_credential_or_endpoint`, `run_doctor`,
   `pick_model_profile`, `toggle_model_assisted_intent`,
   `optional_channel_registration`, `identity_slot_preview`, `done`). Each step
   carries an evidence snapshot and a suggested **Mix command** to run — i.e. it
   sequences existing commands rather than performing guided setup. Action
   `.../actions/onboarding/step_complete.ex` (`onboarding_step_complete`,
   `:objective_write`, internal, no confirmation) records progress. Web
   LiveComponent `apps/allbert_assist_web/.../workspace/components/onboarding.ex`
   (`#workspace-onboarding-panel`). Catalog wiring: `surface/catalog.ex`
   (`:onboarding_panel`) and `workspace/catalog.ex` (`:core_onboarding_panel`) —
   already cross-surface aware. Mix task `lib/mix/tasks/allbert.onboard.ex`.
2. **First-run detection state machine (v0.62 M3), built as a v0.63 seam.**
   `apps/allbert_assist/lib/allbert_assist/cli/first_run.ex` — `detect/0` returns
   one of six product states (`:home_missing`, `:schema_incompatible`,
   `:onboarding_incomplete`, `:first_model_not_ready`, `:profile_unreviewed`,
   `:product_ready`); "onboarding complete" is a **Home marker file
   `<Home>/onboarding.json`** (deliberately outside Settings Central). Its own
   moduledoc says "v0.63 owns the wizard semantics this hooks into," and both
   `profile_reviewed` (marker key) and `:profile_unreviewed` (state) are
   **placeholders** — `mark_onboarding_complete/0` just sets them true. `cli.ex`'s
   bare-`allbert` `first_run/0` prints guidance strings, several literal "(v0.63)"
   stubs. The v0.62 CLI taxonomy maps `onboard → admin onboarding`
   (`cli/commands.ex:113`), which today backs a **summary read**
   (`first_run.ex` `onboarding_summary/0`), not an interactive wizard.
   v0.63 owns the operator-facing wizard semantics; it must not expose this raw
   six-state product taxonomy as the onboarding UX.
3. **First-Model Path (v0.62 M4) — shipped and mature; consumed as-is.**
   `cli/first_run.ex` `first_model_state/1` probes the model path and returns the
   technical readiness reasons the current code knows (`local_ready`,
   `runtime_missing`, `runtime_unhealthy`, `model_missing`,
   `below_hardware_floor`, `byok_ready`). Probe code lives in
   `first_model/ollama.ex` (binary / loopback server / curated model) and
   `first_model/hardware.ex`; confirmation-gated actions
   `first_model/detect.ex` (read-only), `install_ollama.ex` (`:command_execute`,
   resumable), and `pull_model.ex` (`:external_network`, loopback-only,
   resumable) are the implementation seam. ADR 0078's assisted-local path **is
   real code**, so the ADR 0078 BYOK-primary degrade branch does not apply to
   v0.63. v0.63 adds an operator-facing readiness mapper over these **six** probe
   states per the **Readiness Label Mapping Contract** (below). Note there is **no
   `:blocked` state** in the shipped code — earlier drafts speculated one; the plan
   does not surface it. `Needs credentials` and `Needs review` are produced by the
   provider step and the `detect/0` profile layer, not by `first_model_state/1`.
4. **Settings Central — mature; the only seam profiles use.** `settings.ex`
   (`get/put/resolve/list/validate`), `settings/schema.ex` (`@safe_write_keys`
   allowlist, dotted-key namespacing, `safe_write_key?/known_key?` gates),
   `settings/fragments.ex` + `fragment.ex` (core/app/plugin fragment owners),
   `settings/vault.ex` + `secrets.ex` + `key_custody.ex` — secrets are `*_ref`
   pointers into the **three-tier** backend (v0.62 Locked Decision 12: OS vault →
   encrypted-store fallback → env), never inline. Model/provider profiles already
   exist (`list_model_profiles`, `set_active_model_profile`, ADR 0072 recs) and are
   distinct from personas.
5. **Persona / user-category layer — fully greenfield.** No persona data model,
   registry, seed file, or Settings key exists; only ADR 0075 and
   `docs/design/persona-model.md`. The existing free-text `persona.md`/identity
   memory is explicitly *unrelated* (ADR 0075).
6. **Web workspace shell — ADR 0080 consolidated sidebar (v0.61b as-built).** The
   `onboarding_panel` already renders inside the shell on web/CLI/TUI; v0.63
   extends this surface, it does not add new surface plumbing.

**Reconciliation this creates (Locked Decision 1):** the objective-status notion
of "onboarding complete" and the `<Home>/onboarding.json` marker are today two
disjoint notions. v0.63 unifies them: `onboarding.ex` becomes the single shared
state machine, and the marker + first-model states + profile-review become its
inputs — one source of truth.

## Research Grounding

Surveyed 2026-07-07 (post-v0.62.0). Verifiable local-first / BYOK reference tools
whose patterns map to Allbert's constraints:

- **Goose (`goose configure`)** — interactive provider+model wizard, auto-detects
  provider from a pasted key, stores the key in the **OS keychain with a
  `secrets.yaml` file fallback** when the keyring is unavailable, and warns "do not
  store API keys in project files." This is precisely Allbert's three-tier vault;
  v0.63 improves on Goose by **surfacing which tier the key landed in** (Keychain /
  encrypted-store / env) as a trust feature rather than a silent fallback.
- **Ollama** — first run = install → pull a small default model → chat, with a
  three-way **local / hosted / both** mode prompt that discovers models and
  auto-pulls when missing. Validates ADR 0078's assisted-local QuickStart; v0.63
  keeps the pull confirmation-gated (no silent install).
- **LM Studio / Jan** — GUI model-browser set the "zero-config first model" bar
  the plan already cites; met by *managing* the first model, not assuming a key.
- **Cline / Aider / OpenCode** — BYOK-first with a "manually enter API key" path
  and an **OpenAI-compatible custom endpoint** (base URL + key), confirming
  "provider switch without config edit" and a custom-endpoint option are
  table-stakes.
- **OpenClaw** — official public docs verify a terminal `openclaw onboard` path,
  app onboarding, QuickStart vs Advanced flow selection, re-run/reset semantics,
  custom providers (including OpenAI-compatible endpoints), and non-interactive
  secret-reference automation. Treat it as a competitive operator-UX reference,
  not a protocol/API precedent: the earlier pass-2 notes that named a specific
  risk flag or protocol shape are not relied on as verified requirements.
  Allbert should copy the operator contract it needs: official
  `allbert onboard --quickstart`, `--advanced`, default resume, confirmed reset,
  automation with explicit input refs, and one shared state machine.
- **Hermes Agent** — official docs keep the easy path small: `hermes setup`,
  choose or connect a provider, run the first chat, then add gateway/tools/skills.
  They also separate non-secret config from secrets and use `SOUL.md` as a
  behaviour persona. Allbert should preserve the "first chat before optional
  integrations" posture while keeping ADR 0075 profiles seed-only, not a system
  prompt or behaviour override.

**Persona vs system-prompt distinction (name it explicitly):** competitor
"persona" features are often a `SOUL.md` system-prompt that *changes behaviour*.
Allbert's ADR 0075 persona is the opposite — a **seed-only settings profile** that
pre-populates operator-tunable defaults + suggestions after review and grants
nothing. ADR 0075's own context already cites OpenClaw QuickStart defaults and Msty
personas as the competitive bar; the seed-only choice is the deliberate
trust-preserving contrast.

Official references used for the implementation-readiness contract: OpenClaw's onboarding
wizard (`https://docs.openclaw.ai/start/wizard`), CLI setup reference
(`https://docs.openclaw.ai/start/wizard-cli-reference`), and automation docs
(`https://docs.openclaw.ai/start/wizard-cli-automation`); Hermes Agent's
quickstart
(`https://hermes-agent.nousresearch.com/docs/getting-started/quickstart`),
configuration (`https://hermes-agent.nousresearch.com/docs/user-guide/configuration`),
personality (`https://hermes-agent.nousresearch.com/docs/user-guide/features/personality`),
and security (`https://hermes-agent.nousresearch.com/docs/user-guide/security`)
docs. "Hermes Agent" replaces the earlier ambiguous Hermes reference.

## Cross-Cutting Invariants (apply to every milestone)

- **No new authority or permission class.** Onboarding is an experience over
  existing flows; every effectful step routes through existing registered actions,
  doctors, gates, and confirmations. Security Central (ADR 0006) is unchanged.
- **Settings-Central-only writes.** All settings and profile seeds write through
  Settings Central over `@safe_write_keys`; onboarding is not a config side channel.
- **Seed-only profiles.** Personas seed defaults + suggestions after review/confirm;
  they lower no confirmation floor, grant no egress, connect no channel, store no
  secret.
- **No secret leak.** Masked-key entry writes only `*_ref` pointers through the
  three-tier vault; raw values never echo, log, or appear in traces/diagnostics.
  The env tier is read-only: if a secret resolves from env, the wizard reports it
  as operator-provided and verifies it, but new masked entry stores to OS vault or
  encrypted-store fallback only.
- **No silent egress or silent install.** Wizard convenience never bypasses the
  v0.62 M4 confirmation floor on Ollama install / model pull / hosted egress.
- **Operator UX over internal atoms.** Wizard surfaces show one next action in
  operator language; raw product/model probe atoms are for tests, traces, and
  diagnostics.
- **One shared flow, no surface fork.** Web and terminal render the same state
  machine and step IDs; neither forks onboarding logic.
- **Docs-in-triad.** Implementation detail lives in this plan / the request flow /
  ADR 0069 + 0075 — no sidecar planning docs.

### Readiness Label Mapping Contract

The shipped `first_run.ex` `first_model_state/1` emits exactly **six** states (there
is **no `:blocked`** — do not surface it). The operator sees one of five readiness
labels; the mapping is authoritative and every label has a real source:

| Source | Origin | Operator label |
|---|---|---|
| `local_ready` | model probe | **Ready** |
| `byok_ready` | model probe | **Ready** (hosted egress posture visible) |
| `runtime_missing` | model probe | **Needs runtime** |
| `runtime_unhealthy` | model probe | **Needs runtime** |
| `model_missing` | model probe | **Needs model** |
| `below_hardware_floor` | model probe | **Needs review** (local box below the RAM floor → recommend BYOK/existing endpoint; *not* "Needs credentials") |
| hosted/BYOK provider chosen, no key present | **provider step** (not a model-probe state) | **Needs credentials** |
| `:profile_unreviewed` | **`detect/0` product state** (profile layer, not the model probe) | **Needs review** |

`Needs credentials` and `Needs review` are produced by the provider step and the
product/profile layer respectively — not by `first_model_state/1`. Raw internal
atoms are for traces/tests only (invariant "Operator UX over internal atoms").

### Non-Interactive Authorization & Input Contract

`allbert onboard --non-interactive` suppresses prompts for automation. It does
**not** bypass the confirmation floor. The confirmation-gated onboarding steps
(`install_ollama` `:command_execute`, `pull_model` `:external_network`,
`apply_persona_profile` `:settings_write`) proceed only when the operator supplies
the canonical `--authorize` flag. `--accept-risk` remains a deprecated
compatibility alias for older automation; it must warn, stay out of primary help
copy, and route to the exact same durable approval path as `--authorize`.
`--authorize` is an **explicit, durable, logged authorization request** for gated
automation. Install/pull gates may create + approve a durable `Confirmations`
record with a trace, routed through the existing confirmation approve path
(reusing the v0.62 M8.14 confirmation-completion + `admin confirmations approve`
mechanism) — they never set `confirmation.approved?` ad hoc and never skip the
floor. Persona apply is stricter: `apply-persona <id> --authorize` renders the
review diff and writes nothing; only `apply-persona <id> --authorize --yes` may
run the durable create+approve path. This grants **no new authority**: it is a
pre-recorded operator approval of *existing* gated actions, fully auditable.

**Required inputs (refuse on any missing required input; secrets only as `*_ref`
or env, never literals):**

| Step | Required in `--non-interactive` | Supplying flags |
|---|---|---|
| Track | yes | `--quickstart` \| `--advanced` |
| Provider/model | if not `local_ready` | `--provider`, `--model`, `--endpoint` (custom) |
| BYOK credential | if hosted/BYOK chosen | `--key-ref` (a vault/env ref, never a literal) |
| Ollama install / model pull | authorization | `--authorize` (records + approves the durable confirmation) |
| Persona | optional (skip allowed) | `--profile`; review with `--authorize`; apply with `--authorize --yes` |

Missing a required input → the run **refuses** with the specific missing input;
it never prompts, never silently defaults a credential/model/profile, and never
stores an implicit secret.

### Provider / credential keys (M3)

Beyond persona seeds, the M3 provider/model step reuses the **existing** provider
and secret keys — it introduces no new persona-style schema:

- Provider/model selection: `providers.*` (provider config, incl. a custom
  OpenAI-compatible **base-URL** key under the chosen provider) and
  `model_preferences.primary` — written through Settings Central after review
  (`set_provider_credential` and the provider settings path; `set_provider_credential`
  is `:settings_secret_write`, `confirmation: :not_required`, so masked-key *writes*
  are operator-initiated and not a floor concern — distinct from the confirmation-
  gated install/pull steps).
- Secrets: `secret://providers/<name>/api_key`-style `*_ref` pointers only; the
  literal value lands in the resolved vault tier (OS vault → encrypted-store → env)
  and the env tier is read-only (Locked Decision 4). No raw key is written to
  settings.

If M3 needs a genuinely new provider/endpoint key not already in the schema, it is
an explicit schema decision called out in M3, never a silent write.

## First-Model Path (consumed from ADR 0078 — decided in v0.60 M3)

QuickStart's "fastest first chat" requires a **reachable model**, and the flow
must define — explicitly, not by assumption — what happens for a user who arrives
with **neither a hosted API key nor a local model installed**. This is the single
biggest "does it feel 1.0" risk in onboarding (the competitive zero-config bar set
by LM Studio and Jan is met by managing the first model). v0.63 **does not decide
this**: the decision was made upstream in v0.60 M3 and recorded in ADR 0078, which
selected among:

- **(a) Assisted local model.** Detect or instruct an Ollama install and one-click
  pull a small default model, so the fast path needs no key. Cost: a managed local
  dependency and model download weight.
- **(b) Managed hosted default.** Ship a default hosted free/low-tier provider so
  the user can chat before configuring their own key. Cost: an Allbert-operated
  credential/relay, its abuse/cost policy, and an ongoing-cost owner.
- **(c) Bring-your-own-key, honestly framed.** QuickStart explicitly requires a
  key or a running local model and says so up front, with the most assisted
  possible key / Ollama setup. Cost: not truly zero-config; a weaker first-value
  story versus the competitive bar.

**Explicit gate:** v0.63 does not assume a key or model already exists, and it
does not re-litigate the option. It **builds QuickStart against the path ADR 0078
resolved**: the QuickStart fast path, the Operator-Visible Win, and the
acceptance-matrix "first useful chat" criterion are all defined against the
ADR 0078 decision — never against an assumed pre-existing model.

## Guided Wizard Flow (implements the v0.60 design — ADR 0077/0069)

v0.63 implements the onboarding flow designed in v0.60 M4 (ADR 0077); ADR 0069
is the implementation ADR accepted here. The wizard is a single shared flow model
with two render surfaces:

- **One flow, two surfaces.** Web and terminal render one state machine; neither
  forks onboarding logic.
- **Two tracks.** QuickStart minimizes decisions and skips channel/integration
  setup to reach a working conversation quickly; Advanced exposes provider,
  model, channel, app/profile, and diagnostics choices.
- **Steps (canonical 8 step IDs — one order across the triad):** `welcome` ->
  `track_select` -> `model_path` -> `profile_select` -> `profile_review` ->
  `health_check` -> `first_chat` -> `optional_connect` (health verification
  precedes first useful chat; optional channel/integration setup is deferred to
  last). This is the single authoritative order — ADR 0069 and the request-flow
  match it.
- **Operator readiness language:** the model/provider step maps internal probe
  reasons to operator states per the **Readiness Label Mapping Contract**
  (`Ready` / `Needs model` / `Needs credentials` / `Needs runtime` / `Needs
  review`) and always shows the next action, not the raw probe atom.
- **Trust spine as a feature.** The wizard explains confirmation, permission,
  traces, and local inspectability as safety properties, not setup friction.

## User-Category Profile System (applies the v0.60 persona model — ADR 0077/0075)

v0.63 applies the persona model designed in v0.60 M4 (ADR 0077); ADR 0075 is the
application ADR accepted here. The shipped categories and what each persona means
come from v0.60; v0.63 realizes them as repo-maintained settings profiles.

- **Repo-maintained declarative presets** live in the source tree and are
  reviewed like other trusted project metadata.
- **Shipped categories:** `researcher`, `developer`, `writer`, `ops`,
  `general`.
- **Each profile specifies:** Settings Central defaults, suggested apps,
  suggested channels, suggested intents, and recommended model-purpose mappings.
- **Seed-only semantics.** Profiles grant no authority, lower no confirmation
  floor, and enable nothing effectful without the normal gates.
- **Explicit application.** Applying a profile always shows exactly what will be
  seeded before confirmation.

## Security Posture

No new authority. Onboarding is an experience over existing flows: every
effectful step routes through existing actions, settings, doctors, gates, and
confirmations. Profiles seed defaults only. Masked credentials write through the
v0.62 **three-tier** secret backend (OS vault → encrypted-store fallback → env) as
Settings `*_ref` pointers and never echo/log raw values; new entries land in OS
vault or encrypted-store fallback, env-provided secrets are read-only, and the
wizard surfaces the tier (Locked Decision 4). See the Cross-Cutting Invariants
and Locked Decisions sections above for the full set.

- No runtime authorization, trust tier, or per-user authority model from
  profiles.
- No learned, distilled, or auto-generated personas.
- No native desktop GUI onboarding.
- No onboarding-flow or persona-model design, and no First-Model-Path decision;
  v0.60 owns experience design (ADR 0077) and the First-Model Path (ADR 0078).
- No web-surface or affordance overhaul; v0.61 owns the surface this seats in.
- No packaging or CLI redesign; v0.62 owns entry points.
- No final product RC; v0.64 owns integrated validation.

## Vision Check

v0.63 preserves the v0.62/1.0 vision: onboarding is an **experience over existing
flows**, not a new product surface. It grants no new authority, adds no permission
class or Settings key beyond seeding existing safe-write keys, and keeps the
runtime/action/settings spine (ADR 0073) and Security Central (ADR 0006) unchanged.
Personas are configuration seeds, not behaviour — distribution of a better first
run, not new power.

## Upgrade, Rollback & Coexistence

The persistence unification (Locked Decision 1) changes how "onboarding complete"
is stored, so existing Homes need a defined migration:

- **Already-fully-onboarded Homes (v0.62).** `<Home>/onboarding.json` with
  `onboarding_complete=true` (and the placeholder `profile_reviewed=true`) is read
  as complete by the unified state machine — no re-onboarding is forced. Because
  `profile_reviewed` was a placeholder (always set by `mark_onboarding_complete/0`),
  such Homes are treated as **profile-review-skipped**, not profile-applied; the
  operator can apply a persona later via `apply_persona_profile` without re-running
  first-run.
- **Partially-onboarded Homes (objective mid-flight).** The old objective-status
  flow and the marker were disjoint; on first v0.63 launch the state machine
  **reconciles to the marker** as the authoritative source and reframes/cancels any
  stale in-flight onboarding objective (the reconcile is one-way; the Objectives
  runtime tables are no longer a second source of "complete").
- **Retirement.** `:profile_unreviewed`/`profile_reviewed` become real states
  written by `profile_review`; the objective-status notion of onboarding completion
  is retired as a completion source (the objective may remain as a trace artifact,
  but `detect/0` reads only the marker).
- **Rollback.** Reverting to a v0.62 binary still reads `onboarding.json`; no schema
  migration is destructive. `--reset` (confirmation-gated) is the supported way to
  re-run onboarding, and it preserves all other Home data.
