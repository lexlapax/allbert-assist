# v0.12.2 - Tracing and replay

Status: Shipped

## Summary

v0.12.1 made Allbert legible in the moment: operators can see what the runtime is doing right now through `ActivitySnapshot`, spinners, stuck hints, and channel-native activity surfaces. v0.12.2 makes Allbert legible after the fact. When an operator asks "what happened?", "why did skill X activate?", "did the model retry?", or "which tool ran for 12 seconds?", v0.12.2 produces a structured, persisted, replayable answer.

The release introduces a span-based session trace aligned with the current OpenTelemetry GenAI semantic conventions as an export target. Allbert's internal trace schema remains versioned and stable for replay even when external semantic conventions evolve. The same kernel transitions that emit `ActivityUpdate` in v0.12.1 also emit `Span` begin/end events, captured to `~/.allbert/sessions/<id>/trace.jsonl` and surfaced through `/trace`, `allbert-cli trace`, and a Telegram trace summary. An optional file-based OTLP exporter lets advanced operators ship traces into their own observability stack (Datadog, Langfuse, Jaeger, Grafana) without coupling the daemon to a remote endpoint.

The default capture posture is `capture_messages = true`. Allbert is a personal assistant; the operator owns their own data and gets the most value when traces include actual prompts, responses, tool args, and tool results so replay is meaningful. Privacy guards remain unconditional: secrets are always redacted, the disk cap is bounded, retention is finite, and the operator can lower capture fidelity per field or disable tracing entirely through `/settings`.

The release includes an additive protocol bump to `4` so daemon and channels can carry trace span messages alongside existing v3 activity messages, plus a setup wizard step and a path-preserving default-write upgrade path so both fresh and existing profiles get tracing without hidden whole-file TOML rewrites.

## Goals

- Capture every kernel decision and runtime call as a structured span with parent-child relationships: turn, prepare-context, classify-intent, route-skill, invoke-agent, execute-tool, chat, run-validation, run-script, wait-approval, wait-input, finalize.
- Persist spans per session at `~/.allbert/sessions/<id>/trace.jsonl`, append-only, line-oriented, with disk-cap-aware rotation and crash recovery for in-flight spans.
- Align span names and attributes with the OpenTelemetry GenAI semantic conventions so traces are portable to external observability tools.
- Default `capture_messages = true` so operators get full prompt/response replay; allow per-field policy (`capture` / `summary` / `drop`) for finer privacy control.
- Always redact secrets at write time, regardless of `capture_messages`, using a hardened pattern allowlist plus pluggable redactor seam.
- Make traces useful in the same channels operators already live in: `allbert-cli trace`, TUI `/trace`, Telegram `/trace last`.
- Provide an opt-in file-based OTLP-JSON exporter so operators with their own observability stacks can ship traces without coupling Allbert to remote endpoints.
- Fold tracing posture into the v0.12.1 settings hub so capture fidelity, retention, and disk cap are inspectable and changeable from `/settings`.
- Fold tracing into onboarding/bootstrap so fresh profiles answer the question explicitly during setup and existing profiles get safe defaults through the v0.12.1 path-preserving settings machinery.
- Reuse v0.12.1 boundaries: same kernel transitions, same redaction posture, same channel capability gating; do not invent a parallel observability stack.

## Non-goals

- No network OTLP exporter. v0.12.2 ships only file-based OTLP-JSON export. Network exporters can land in a later release once policy on outbound traffic is decided.
- No automatic anomaly detection, eval scheduling, golden-set replay, or annotation queue. These are observability-platform features, not in scope here.
- No retroactive trace synthesis. Sessions that ran before v0.12.2 will not gain spans; tracing starts forward.
- No cross-session aggregation analytics in v0.12.2. Each session's trace stands alone.
- No tracing of pre-daemon process startup (the setup wizard before the kernel starts has its own logging path).
- No new approval kinds. Tracing inherits existing approval/inbox behavior.
- No structured error trait migration; v0.12.1's deferral remains in force.
- No span sampling beyond the `capture_messages` flag and per-field redaction policy in v0.12.2.
- No automatic redaction of model output text. Operators who want output redaction must lower `provider_payloads` to `summary` or `drop`.
- No removal of existing telemetry, cost log, daemon log, or session-scoped trace text files. They remain alongside the new span trace.
- No change to top-level `~/.allbert/traces/` treatment. That legacy/debug trace directory remains derived and excluded from profile export/sync; v0.12.2's new session trace artifacts live under `sessions/`.

## Plan hardening decisions

These decisions are release-blocking. Implementation should not invent alternative protocol, storage, privacy, or migration policy while landing trace/replay.

### Protocol compatibility

Protocol v4 is additive. A v4 daemon accepts `ClientHello.protocol_version` values `2`, `3`, and `4`, stores the negotiated client protocol version on each connection, and filters outbound serialization through that value. v2 peers receive only v2-safe messages and fields. v3 peers receive the v0.12.1 activity surface but no v4 trace messages or fields. v4 peers receive trace read responses, trace session summaries, and completed-span broadcasts only after an explicit trace subscription.

The v4 daemon rejects protocol versions below `2` or above `4` with an actionable `ProtocolError { code: "version_mismatch", ... }` that names whether to upgrade the client or daemon. A v4 client connected to a v3 daemon surfaces `upgrade the daemon to v0.12.2 to use trace surfaces` and must not synthesize spans locally.

M0 must include proto-level handshake and envelope-decode tests for v2, v3, and v4 peers plus future-version rejection coverage. Daemon-side broadcast-filter tests land in M1 alongside the daemon wiring that owns the per-peer filter.

### Trace storage and continuity

Durable span traces are session artifacts. The current writer appends to `~/.allbert/sessions/<session_id>/trace.jsonl`, rotates to `trace.<n>.jsonl.gz`, and uses `current_spans/<span_id>.json` only for crash recovery. These files are continuity-bearing because they are needed for cross-device replay, v0.14 self-diagnosis, and profile export/import fidelity.

Profile export/sync includes `sessions/*/trace.jsonl`, `sessions/*/trace.<n>.jsonl.gz`, and `sessions/*/current_spans/` recovery state with the rest of `sessions/` by default. Export manifests should report trace counts and byte totals so operators see the size/sensitivity footprint. Top-level `~/.allbert/traces/` remains a derived legacy/debug output directory and stays excluded by ADR 0061.

Trace GC owns only trace artifacts. Retention and disk-cap enforcement may remove `trace.jsonl`, rotated trace archives, and stale `current_spans/` files after recovery, but must not remove `turns.md`, `meta.json`, approvals, patch artifacts, attachments, or other non-trace session files.

Trace readers for `trace show`, `trace list`, `trace export`, and later v0.14 bounded read APIs must read both the active `trace.jsonl` and rotated `trace.<n>.jsonl.gz` archives. Archives are read in numeric order before the active file, spans are ordered by `started_at` with span id as a stable tie-breaker, malformed trailing JSONL records are skipped with an operator-visible warning, and unsupported future schema versions fail with remediation instead of being partially interpreted.

### Settings and default-write behavior

All trace settings use the v0.12.1 typed settings registry and path-preserving TOML edit discipline. The implementation must not reserialize the whole config as a fallback for `[trace]` default-write or `/settings set trace.*`.

For an existing profile with no `[trace]` table, daemon start attempts to insert the documented default block atomically while preserving unrelated TOML comments and table order. If preservation cannot be guaranteed, the daemon skips the write, loads in-memory trace defaults for that boot, and emits a remediation hint naming the default block and `/settings show trace`. This is a documented configuration-scaffolding exception, not a hidden behavior change.

For an existing profile with a partial `[trace]` table, the daemon never auto-rewrites the table. Missing keys load defaults, validation warnings name the effective values, and `/settings show trace` displays both file values and defaulted values where the settings registry supports that distinction.

### OpenTelemetry alignment

OpenTelemetry GenAI semantic conventions are the export alignment target, not Allbert's durable replay contract. The persisted JSONL contract is an Allbert-owned `TraceRecord` envelope with `schema_version = 1`, `record_type = "span"`, and a `span` payload. The protocol `Span` type remains the live wire payload and does not carry file-format migration state. The OTLP-JSON exporter maps internal spans to the current OTLP trace payload shape and current GenAI attribute names at export time.

The ADR and implementation notes must cite the official OpenTelemetry GenAI semantic-conventions page as Development and the official OTLP specification as Stable for trace signals. If OTel GenAI names change, Allbert updates the exporter mapping without migrating historical session trace files unless the internal schema itself changes.

### ADR readiness

The v0.12.2 ADRs are accepted. M0 must re-read them before code implementation begins and patch the ADRs first if implementation needs to diverge:

- ADR 0081: durable session trace artifacts and replay envelope.
- ADR 0082: trace capture, privacy, and redaction posture.
- ADR 0083: protocol v4 trace messages and OTLP-JSON export.

## Public interfaces

### Config

```toml
[trace]
enabled = true                  # set to false to disable persistence entirely
capture_messages = true         # include prompt/response text and tool args/results
session_disk_cap_mb = 50        # per-session cap; oldest spans archived/evicted past cap
total_disk_cap_mb = 2048        # total cap across all sessions; oldest trace artifacts GCed past cap
retention_days = 30             # sessions older than this are GCed by `trace gc`
otel_export_dir = ""            # empty uses ALLBERT_HOME/exports/traces for explicit exports
otel_service_name = "allbert"

[trace.redaction]
secrets = "always"              # "always" only; setting "never" is rejected with operator-visible error
tool_args = "capture"           # "capture" | "summary" | "drop"
tool_results = "capture"        # "capture" | "summary" | "drop"
provider_payloads = "capture"   # "capture" | "summary" | "drop"
```

When `capture_messages = false`, the loader coerces `tool_args`, `tool_results`, and `provider_payloads` from `capture` to `summary` for that load. `provider_payloads` covers raw prompt text, response text, request/response bodies, and streaming chunks when adapters expose them. The override does not rewrite the file; the operator still sees their original choice in `/settings`. `capture_messages = true` means the per-field policy applies as written.

`session_disk_cap_mb` clamps to `[5, 500]`; `total_disk_cap_mb` clamps to `[100, 51200]`; `retention_days` clamps to `[1, 365]`. Out-of-range values warn and clamp at config load. `otel_export_dir` and `trace export --out` must stay inside `ALLBERT_HOME`; absolute paths or path escapes are rejected with an operator-visible error.

`secrets = "never"` is intentionally rejected. Secret redaction is unconditional, and removing it would break the privacy contract documented in operator docs and the upgrade note.

### Daemon protocol

Bump `PROTOCOL_VERSION` from `3` to `4` with additive span messages and fields.

Add:

- `SpanKind`: `Internal`, `Client`, `Server`, `Producer`, `Consumer` (mirrors OTel).
- `SpanStatus`: `Ok`, `Error { message: String }`.
- `SpanEvent`: `{ timestamp, name, attributes }` for events nested inside a span.
- `Span`: `{ id, parent_id, session_id, trace_id, name, kind, started_at, ended_at, duration_ms, status, attributes, events }`.
- `TraceSessionSummary`: `{ session_id, span_count, root_span_count, started_at, last_touched_at, total_duration_ms, bytes, has_rotated_archives, truncated_count }`.
- `ClientMessage::TraceSubscribe { session_id: Option<String> }` and `ClientMessage::TraceUnsubscribe { session_id: Option<String> }` for completed-span broadcast opt-in.
- `ServerMessage::TraceSubscribed { session_id }` for the resolved subscription.
- `ServerMessage::TraceSpan(Span)` for completed-span broadcast to subscribed v4 peers only.
- `ClientMessage::TraceShow { session_id: Option<String> }` and `ServerMessage::TraceSpans(Vec<Span>)` for direct CLI/channel reads.
- `ClientMessage::TraceShowSpan { session_id: Option<String>, span_id }` and `ServerMessage::TraceSpanDetail(Span)` for one-span reads.
- `ClientMessage::TraceList` and `ServerMessage::TraceSessions(Vec<TraceSessionSummary>)`.

When `session_id` is omitted in trace read/tail requests, the daemon resolves it to the connection's active session if one exists, otherwise to the latest session with trace data. If a span id is not unique in the resolved scope, the daemon returns a remediation hint asking for `--session`.

Backward compatibility:

- v2/v3 clients attached to a v4 daemon must continue working. The daemon must omit unknown-to-peer messages and fields when serializing for older peers, preserve v0.12.1 activity behavior for v3 peers, and never send v4 trace messages to v2/v3 peers.
- A v4 client connecting to a v3 daemon must surface a clear version mismatch with operator-facing remediation (`upgrade the daemon to v0.12.2 to use trace surfaces`) and must not synthesize spans locally.
- M0 must include roundtrip and version-mismatch tests for v2, v3, and v4 envelopes at the proto deserializer; daemon-side broadcast-filter coverage moves to M1 with the daemon wiring.

### Bootstrap and onboarding

Setup wizard adds a "Trace and replay" step after the memory step and before the self-improvement step. The step must:

1. Briefly explain what tracing captures (decisions, routing, tool calls, model calls, durations) and how it differs from live activity.
2. Ask whether to enable tracing (default `yes`).
3. Ask whether to capture full prompt and response text (`capture_messages` default `yes`), with one-paragraph explanation that secrets are always redacted, retention is bounded, and the operator can change this later via `/settings`.
4. Show the storage path, disk caps, and retention defaults; allow override.
5. Print a summary line at the end naming `/settings show trace` and `allbert-cli trace --help` for later inspection.

The setup state file (v0.12.1's `setup.state_path`) gains a `trace` block so `allbert-cli setup --resume` resumes the trace step correctly.

Existing-profile upgrade behavior:

- On daemon start, if the loaded `config.toml` has no `[trace]` section, the daemon attempts to insert the default block atomically with the v0.12.1 path-preserving TOML editor and emits an informational message to `daemon.log`: `Trace defaults installed (capture_messages=true, retention_days=30). Use \`/settings show trace\` or \`allbert-cli trace --help\` to inspect.`
- If the default block cannot be inserted without preserving unrelated comments and paths, the daemon skips the write, loads the same defaults in memory for that boot, and emits a remediation hint naming manual `config.toml` editing and `/settings show trace`.
- Existing profiles do not get a setup-wizard re-run.
- Default-write must be idempotent and must skip if any partial `[trace]` block exists; the operator's partial choice wins and missing keys load defaults.
- The default-write is logged but not surfaced as an inbox approval; this is configuration scaffolding, not a behavior change requiring operator review.

## Implementation design

### Rust dependencies and module layout

v0.12.2 adds no new top-level crate dependencies. The implementation reuses crates already in the workspace:

- `flate2` for `trace.<n>.jsonl.gz` rotation. Currently declared only in `allbert-cli`. Add it to `allbert-kernel` since the trace writer/reader live there.
- `serde_json` for JSONL records and OTLP-JSON export shape.
- `uuid` for `id`, `parent_id`, and `trace_id` values.
- `chrono` for `DateTime<Utc>` span timestamps. Add `chrono` to `allbert-proto` with `default-features = false` and `features = ["serde"]` because shared `Span` payloads derive serde; keep `allbert-kernel`'s existing `clock` feature for timestamp creation.
- `toml_edit` for the path-preserving `[trace]` default-write, already in `allbert-kernel` from v0.12.1.

The shared span types live in `allbert-proto`. The persisted-record envelope, writer, reader, rotation, GC, and exporter live in `allbert-kernel`. The new module must not be named `trace.rs` because [`crates/allbert-kernel/src/trace.rs`](../../crates/allbert-kernel/src/trace.rs) already owns the `tracing-subscriber` daemon-logging bootstrap. Use `replay.rs` (preferred) or `trace_store.rs` for the new module to avoid the collision; the existing logging module stays where it is. M1 must not overwrite the existing `trace.rs`.

v0.12.2 deliberately does **not** adopt the OpenTelemetry SDK crates (`opentelemetry`, `opentelemetry-sdk`, `opentelemetry-otlp`, `tracing-opentelemetry`). The rationale, captured fully in [CARGO_DEPS.md](../../CARGO_DEPS.md):

- Export is file-only in v0.12.2; network OTLP is deferred. `opentelemetry-otlp` would be unused weight today.
- Redaction must run before persistence, display, and export. Owning the span emission keeps that discipline simple; layering on top of `tracing-opentelemetry` would add a second span source to reconcile with the kernel-owned activity boundary established in v0.12.1.
- The existing `tracing` crate in `allbert-kernel` drives diagnostic daemon logging only; v0.12.2 does not change that subsystem and does not graft span-replay onto it.

OTel alignment is at the **export shape** (OTLP/JSON) and **attribute names** (GenAI semconv) only; the durable replay contract is Allbert-owned per ADR 0083.

### Span model

Add the shared span types in `allbert-proto` and the persisted record envelope in `allbert-kernel`'s new replay module:

```rust
struct TraceRecord {
    schema_version: u16, // starts at 1
    record_type: TraceRecordType,
    span: Span,
}

#[serde(rename_all = "snake_case")]
enum TraceRecordType { Span }

struct Span {
    id: String,
    parent_id: Option<String>,
    session_id: String,
    trace_id: String,
    name: String,
    kind: SpanKind,
    started_at: DateTime<Utc>,
    ended_at: Option<DateTime<Utc>>,
    duration_ms: Option<u64>,
    status: SpanStatus,
    attributes: BTreeMap<String, AttributeValue>,
    events: Vec<SpanEvent>,
}

enum SpanKind { Internal, Client, Server, Producer, Consumer }
enum SpanStatus { Ok, Error { message: String } }

struct SpanEvent {
    timestamp: DateTime<Utc>,
    name: String,
    attributes: BTreeMap<String, AttributeValue>,
}

enum AttributeValue {
    String(String),
    Int(i64),
    Float(f64),
    Bool(bool),
    StringArray(Vec<String>),
    IntArray(Vec<i64>),
}
```

`trace_id` is per top-level turn; `session_id` is the cross-turn session identifier. A turn span is the root of one trace; sub-agent spans inherit the same `trace_id` with a chain of `parent_id` references. Protocol messages exchange `Span`; persisted JSONL writes `TraceRecord` so file-format migrations are explicit and separate from live protocol evolution.

Span identifiers are OTel-compatible lowercase hex strings, not hyphenated UUID display strings. `trace_id` is 32 hex characters (16 bytes). `id` and `parent_id` are 16 hex characters (8 bytes). The implementation may generate UUID v4 bytes and encode the full 16 bytes for `trace_id` plus a collision-resistant 8-byte slice or random value for span ids, but the stored/protocol strings must already be valid OTLP `traceId`/`spanId` values.

### OTel attribute alignment

Attribute keys follow the current OTel GenAI conventions where they exist and use the `allbert.*` prefix for assistant-specific attributes. OTel GenAI alignment is an export target; the persisted JSONL schema remains Allbert-owned and versioned for replay:

- `gen_ai.operation.name`: e.g., `chat`, `invoke_agent`, `execute_tool`, `route_skill`.
- `gen_ai.provider.name`: provider name (`anthropic`, `openai`, `gemini`, `ollama`, `openrouter`).
- `gen_ai.request.model`: model id.
- `gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`, `gen_ai.usage.cache_read_tokens`, `gen_ai.usage.cache_create_tokens`.
- `gen_ai.agent.name`: for `invoke_agent` spans, names the agent invoked.
- `allbert.session.id`, `allbert.channel`, `allbert.skill.name`, `allbert.skill.tier`, `allbert.tool.name`, `allbert.approval.id`.

Unknown attributes are tolerated by readers; new attributes can land additively.

### Capture points

| Span name | When emitted | Typical children |
| --- | --- | --- |
| `turn` | Top-level user input → final response | All below |
| `prepare_context` | Bootstrap, memory load, telemetry assembly | none |
| `classify_intent` | Intent/router pass | none |
| `route_skill` | Per skill considered/eligible (one span per candidate, with eligibility outcome attribute) | none |
| `invoke_agent` | Agent or sub-agent invocation | nested spans |
| `chat` (`gen_ai.operation.name=chat`) | Logical provider call | none (retry attempts and streaming chunks are span events when available; retries are not sibling spans in v0.12.2) |
| `execute_tool` | Tool call (kernel-driven) | none |
| `run_validation` | Tier A or other validation command | none |
| `run_script` | Script (Lua) execution | none |
| `wait_approval` | Approval gate open → resolved | none |
| `wait_input` | Input request open → resolved | none |
| `finalize` | Memory staging, cost persist, cleanup | none |

The same kernel transitions that emit `ActivityUpdate` in v0.12.1 emit `SpanBegin`/`SpanEnd`. The wiring is shared; only the destinations differ. v0.12.1 sends activity to channels live; v0.12.2 also writes spans to the trace sink.

### Persistence

- Each session has a directory `~/.allbert/sessions/<session_id>/`.
- Spans land in `trace.jsonl` (current file) as one `TraceRecord` JSON object per line, written when the span ends.
- In-flight span snapshots live in `current_spans/<span_id>.json`. On daemon restart, any leftover snapshot is finalized with `status = Error { message: "truncated_at_restart" }` and `ended_at = now()` plus a span event marking the restart, then appended to `trace.jsonl` and removed from `current_spans/`. Snapshot-on-begin plus remove-on-end is two filesystem ops per span; this is an explicit trade for crash recoverability against per-span fs cost. M1 may coalesce the per-span snapshots into a single `current_spans/in_flight.json` rewritten on transitions if measured cost warrants, without changing the recovery semantics.
- When `trace.jsonl` exceeds `session_disk_cap_mb`, the file is rotated to `trace.<n>.jsonl.gz` (gzip) and a fresh `trace.jsonl` started. Rotation runs synchronously inside the writer to avoid races.
- When a session's archived files exceed the per-session cap, the oldest archive is evicted.
- Trace readers load `trace.<n>.jsonl.gz` archives plus the active `trace.jsonl`, normalize ordering by span timestamps, skip malformed trailing JSONL records with a warning, and reject unsupported `schema_version` values with remediation.
- When the total trace footprint exceeds `total_disk_cap_mb`, trace artifacts from the oldest sessions are removed as a group; non-trace artifacts in those session directories are not touched.
- GC runs on demand via `allbert-cli trace gc` and as a maintenance job per existing job scheduling. GC must be safe to interrupt and idempotent.
- Session trace artifacts under `sessions/*/trace*` are continuity-bearing and included with `sessions/` in profile export/sync by default. Profile export manifests include trace artifact counts and byte totals. Top-level `~/.allbert/traces/` remains derived legacy/debug output and stays excluded from export/sync.

### Redaction

Secret detection runs on every attribute value at write time:

- Built-in patterns: AWS keys (`AKIA[0-9A-Z]{16}`), generic API keys (`(api[_-]?key|token|secret)\s*[:=]\s*['"]?[A-Za-z0-9_\-]{20,}['"]?`), JWTs, env-style assignments matching common credential names, OpenAI/Anthropic key prefixes (`sk-`, `sk-ant-`).
- Detected secrets are replaced with `<redacted:secret>` in span attributes before persistence.
- The redactor is wrapped behind a `SecretRedactor` trait so the pattern set can extend without re-architecting.

`tool_args`, `tool_results`, and `provider_payloads` policies apply after secret redaction:

- `capture`: full content with secret redaction applied.
- `summary`: first 200 chars of the stringified value plus `total_len_chars` and `total_size_bytes`.
- `drop`: field omitted entirely; span retains a `<field>_dropped: true` marker so readers know data was suppressed by policy.

`capture_messages = false` overrides the loaded policy to `summary` for `tool_args`, `tool_results`, and `provider_payloads` for that boot. The override is logged once at startup so operators understand why their TOML choice was downgraded.

### Cross-cutting with v0.12.1

- Same emission boundaries: every kernel transition that produces an `ActivityUpdate` also produces a span begin/end. The kernel exposes both through one combined hook so adding new boundaries is one edit, not two.
- Same redaction posture: v0.12.1 already declares activity must not include raw prompts, secrets, full tool inputs, or provider payloads. v0.12.2's redactor is the implementation; v0.12.1's declarations stay the contract.
- Same settings hub: v0.12.1 ships the registry; v0.12.2 adds the `trace` group keys.
- Same async loop and protocol seam: v0.12.1 bumps protocol to `3`; v0.12.2 bumps to `4` additively.

### OTLP-JSON export

Add an `exports_traces` field to [`AllbertPaths`](../../crates/allbert-kernel/src/paths.rs) computed as `root.join("exports").join("traces")`. The directory is created lazily on first export, not in `ensure()`, because exports are operator-driven and not always present.

`OtlpJsonExporter` writes OTel-aligned JSON to `paths.exports_traces.join("<session_id>.json")` by default, or to `<ALLBERT_HOME>/<trace.otel_export_dir>/<session_id>.json` when the configured relative directory stays inside `ALLBERT_HOME`. `trace export --out` may override the file path, but the resolved path must still stay under `ALLBERT_HOME`. The format follows the OTLP/JSON specification for `ResourceSpans`:

- `resource.attributes`: includes `service.name = otel_service_name`, `service.version = "0.12.2"`, and `allbert.host` when known.
- `scope_spans[0].scope.name`: `allbert-kernel`.
- `scope_spans[0].spans`: array of OTel-shaped spans with `traceId`, `spanId`, `parentSpanId`, `name`, `kind`, `startTimeUnixNano`, `endTimeUnixNano`, `attributes`, `status`, `events`.

Operators wire their own pipeline: file watcher, periodic upload, or batched ingestion into Datadog, Langfuse, Jaeger, or any OTLP-compatible backend. Allbert does not push to a network endpoint in v0.12.2.

## End-user release plan

Release notes must explain:

- Allbert now persists a structured trace of every session at `~/.allbert/sessions/<id>/trace.jsonl`.
- The default capture posture is `capture_messages = true`: full prompt and response text, tool args, and tool results are captured for replay value.
- Secrets are always redacted at write time. The `trace.redaction.secrets = "never"` value is intentionally rejected.
- Operators can lower fidelity per field (`tool_args`, `tool_results`, `provider_payloads`) to `summary` or `drop`, or set `trace.capture_messages = false` to coerce the per-field policy in one switch.
- `allbert-cli trace show|tail|list|show-span|export|gc` covers inspection, tail, single-span detail, OTLP-JSON export, and disk-cap GC.
- TUI `/trace` renders the current session's span tree inline; `/trace show-span` shows one span in detail.
- Telegram `/trace last` returns a compact, redacted summary; full content stays on local disk.
- An optional file-based OTLP-JSON exporter writes per-session `ResourceSpans` for ingestion into Datadog, Langfuse, Jaeger, Grafana, or any OTLP-compatible backend.
- Setup now includes a "Trace and replay" step. Existing profiles get `[trace]` defaults installed through the path-preserving settings writer when safe, or boot with in-memory defaults plus a remediation hint when insertion would risk rewriting operator comments or unrelated paths.
- Disk caps and retention are bounded; `allbert-cli trace gc` enforces them.
- The protocol bumps from `3` to `4`; v2/v3 clients attached to v4 daemons keep working, and v4 clients connecting to older daemons surface a clear version-mismatch remediation.

Operator acceptance checklist:

- Fresh setup ends with the trace step accepted, defaults written, and `/settings show trace` reflecting the choices.
- Existing profile starts the daemon and either sees `[trace]` defaults installed in `config.toml` with the informational log line or, when safe insertion is not possible, sees in-memory defaults plus a remediation hint.
- A turn produces a span tree visible in `/trace show` and `allbert-cli trace show`.
- Sub-agent invocations show as nested `invoke_agent` spans.
- Tool calls show with `duration_ms` populated.
- Provider retry shows as retry events on the parent `chat` span.
- Setting `capture_messages = false` coerces tool-arg/result/provider-payload fields to `summary` and logs the coercion.
- `trace.redaction.secrets = "never"` is rejected with operator-visible remediation.
- A session trace exceeding the per-session cap rotates and archives without losing trailing spans.
- `allbert-cli trace export <session> --format otlp-json` writes a valid OTLP-JSON document.
- `allbert-cli trace show` and `trace export` include spans from both active and rotated session trace files.
- `allbert-cli trace gc` removes sessions past `retention_days` and trims to `total_disk_cap_mb` without touching non-trace artifacts.
- A simulated daemon crash mid-turn finalizes the in-flight span on next boot rather than losing it.
- Telegram `/trace last` replies with structural data only and does not inline prompt/response text.
- Tier A validation passes with zero clippy warnings.

## ADRs

- ADR 0081: Durable session trace artifacts and replay envelope. Captures span/event schema versioning, `sessions/<id>/trace.jsonl`, rotation, crash recovery, retention/GC ownership, profile export/sync inclusion, and the rule that GC removes trace artifacts only.
- ADR 0082: Trace capture, privacy, and redaction posture. Captures `capture_messages = true`, unconditional secret redaction, per-field capture/summary/drop, `capture_messages=false` coercion, Telegram structural-only replies, read-only `trace.redaction.secrets`, and the documented `[trace]` default-write exception.
- ADR 0083: Protocol v4 trace messages and OTLP-JSON export. Captures v2/v3/v4 daemon compatibility, per-peer filtering, v4-to-old-daemon remediation, file-only OTLP-JSON export under `ALLBERT_HOME`, no network exporter, and OTel alignment without making in-development GenAI conventions the durable internal contract.
