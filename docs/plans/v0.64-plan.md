# Allbert v0.64 Trusted Install And Non-Developer First Run

Request flow: `docs/plans/v0.64-request-flow.md`
ADR: ADR 0076 (Packaging, Distribution, And Unified CLI), ADR 0077 (Product
Experience Design And Information Architecture), and ADR 0078 (First-Model
Path) constrain this release. Update those ADRs only if implementation discovers
a durable decision change.

Status: planned (inserted after v0.63 to replace the old validation-only v0.64
scope).

Source: post-v0.63 product-readiness review. v0.63 proved the guided onboarding
shape but still left the pre-1.0 target framed as a technical prosumer. The new
1.0 target is a non-developer local-first operator who can install Allbert,
start it, choose either a local or BYOK model path, repair common first-run
blocks, and reach a first useful chat without reading developer docs.

## Purpose

Make the first install and first run trustworthy and repairable for a
non-developer. v0.64 moves the v0.62 packaging-trust deferrals into the first-run
release because a normal operator cannot be expected to distinguish "installer
trust" from "onboarding trust." The release does not add new authority; it makes
existing distribution, first-run, model, vault, and onboarding behavior coherent
as one user-facing path.

The 1.0 target is **two-tier**: a **consumer default** path that reaches a first useful
chat with no terminal beyond starting the service, no `ollama` CLI, and no API key; and a
**prosumer advanced** path that keeps BYOK, custom endpoints, and CLI-driven control. The
consumer default is what a fresh non-developer sees first; the advanced path is opt-in.

## Expected Inputs

- v0.62 packaged binary, unified CLI, daemon, Home layout, backup-before-migrate
  behavior, and three-tier vault accepted.
- v0.62b reusable release/install evidence accepted where applicable.
- v0.63 guided onboarding, QuickStart/Advanced split, provider/model setup, and
  persona profile review accepted.
- ADR 0078 accepted: assisted-local default, BYOK hosted fallback, no
  managed-hosted default.
- v0.63 packaged operator-validation findings reviewed so first-run repair work
  targets real operator blockers, not hypothetical polish.

## Operator-Visible Win

A non-developer installs Allbert from a trusted package path, starts it as a persistent
background service (`brew services` / `systemctl --user`, so they never re-run `serve`),
opens the web workspace, and on the **consumer default** path downloads a curated local
model with one click (in-app progress, no `ollama` CLI, no API key) to reach a first
useful chat — understanding any blocked first-run state and taking one clear repair
action, with no source checkout, raw `mix`, hand-edited config, or unexplained
security/trust state. The **advanced** path offers BYOK hosted setup and custom
endpoints for prosumers who prefer them.

## Planned Interfaces

- Package-first install docs and onboarding entry path.
- Installer-side signature/trust verification and clear failure messages.
- Upgrade rollback/restore automation from the backup-before-migrate artifact.
- First-run state copy and repair-action contract for web, CLI, and TUI.
- Web-first QuickStart with terminal onboarding as secondary/resume path.
- Model setup repair contract for local Ollama and BYOK hosted providers.
- Trust-spine summary that explains local data, hosted-provider egress, secrets,
  confirmations, traces, and memory review in plain language.
- `mix allbert.test release.v064` deterministic gate.

## Security Posture

No new permission class, authority source, model privilege, or bypass. Installer
trust verifies release artifacts before install. Onboarding and repair screens
may explain or route to existing actions, but they do not grant authority by
themselves. Secrets continue to resolve through the v0.62 vault tiers, and all
effectful repair work stays behind registered actions, Security Central,
confirmations, traces, and audits.

## Non-Goals

- No native desktop GUI client.
- No new integration surface beyond existing model/provider and onboarding
  paths.
- No local files/notes/memory launch-path work; v0.65 owns that.
- No final product RC; v0.66 owns full no-docs validation.
- No contract freeze; v1.0 owns the freeze after v0.66 is green.

## Milestones

- M0: Packaging-trust intake from v0.62. Implement Developer-ID
  signing/notarization decision where applicable; test the self-extraction x
  hardened-runtime interaction; require installer-side signature verification
  for curl/Homebrew; and implement or explicitly decide the rollback/restore
  automation path from the backup-before-migrate artifact. If anything is
  consciously re-parked post-1.0, record the exception in ADR 0076 and the v1.0
  plan, not only in release notes.
- M0.a: One-click model-download mechanism spike (ADR-0078-constrained). Choose and
  record how the consumer-default in-app model download is delivered — drive Ollama's
  pull API behind a friendly progress surface (auto-detect/guide Ollama presence), embed
  a downloader, or bundle a small model in the artifact — with the size/licensing/offline
  trade-offs weighed. This is the one genuinely new engineering commitment in v0.64; the
  chosen mechanism is recorded as-built and amends ADR 0078.
- M1: Package-first documentation contract. Operator docs start with packaged
  install and `allbert serve`; source checkout and `mix setup` move to
  contributor/developer orientation.
- M2: First-run state taxonomy and repair copy. Every blocked first-run state
  maps to user-facing language and exactly one primary repair action.
- M3: Web-first QuickStart on a persistent service. The browser path is the primary 1.0
  product surface, reached by starting Allbert as a background service (`brew services
  start allbert` / `systemctl --user`) once — the non-developer never re-runs `serve`.
  Terminal onboarding can resume or drive headless use but is not the default first-run
  story.
- M4: Two-tier model path. **Consumer default** — one-click in-app download of a curated
  local model with an in-web progress surface (no `ollama` CLI, no API key), carrying a
  fresh operator to a first useful chat. **Advanced** — BYOK hosted setup and custom
  endpoints for prosumers. Both tiers surface readiness, progress, credential storage
  tier, network egress, and failure recovery without raw internal probe terms; the
  consumer default is the primary first-run path and the advanced path is opt-in.
- M5: Trust-spine presentation. Onboarding explains local data, hosted-provider
  egress, secrets, confirmations, traces, and memory review as product controls.
- M6: Fresh-home operator script. Define the no-docs first-run script that v0.66
  later executes with a fresh non-developer operator.
- M7: Docs drift, `release.v064`, docs gate, warning gate, and v0.65 handoff.

## Definition Of Done

v0.64 is done when:

- packaged install verification and rollback/restore posture are implemented or
  explicitly dispositioned with accepted ADR/v1.0-plan language;
- operator-facing docs start with packaged install and browser-first onboarding,
  not source checkout or `mix`;
- every first-run blocked state has plain-language copy and one primary repair
  action across web/CLI/TUI;
- the consumer-default path carries a fresh operator to a first useful chat with **no
  external tool install and no API key** (one-click curated local-model download), and
  the advanced BYOK/custom path is available and clearly labeled as opt-in;
- Allbert runs as a persistent background service so the operator does not re-run
  `serve`;
- secrets, logs, traces, evidence, and screenshots stay redacted;
- `mix allbert.test release.v064`, docs, warning, and drift checks pass;
- v0.65 has current inputs for local files, notes, and reviewed memory.
