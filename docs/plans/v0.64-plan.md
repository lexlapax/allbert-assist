# Allbert v0.64 Trusted Install And Non-Developer First Run

Request flow: `docs/plans/v0.64-request-flow.md`
ADR: ADR 0076 (Packaging, Distribution, And Unified CLI), ADR 0077 (Product
Experience Design And Information Architecture), and ADR 0078 (First-Model
Path) constrain this release. Update those ADRs only if implementation discovers
a durable decision change.

Status: planned (inserted after v0.63 to replace the old validation-only v0.64
scope).

Source: post-v0.63 product-readiness review. v0.63 proved the guided onboarding
shape but still left the pre-1.0 target framed as a technical prosumer. The new
1.0 target is a non-developer local-first operator who can install Allbert,
start it, use the consumer-default local model path before opting into advanced
BYOK/custom setup, repair common first-run blocks, and reach a first useful
chat without reading developer docs.

## Purpose

Make the first install and first run trustworthy and repairable for a
non-developer. v0.64 moves the v0.62 packaging-trust deferrals into the first-run
release because a normal operator cannot be expected to distinguish "installer
trust" from "onboarding trust." The release does not add new authority; it makes
existing distribution, first-run, model, vault, and onboarding behavior coherent
as one user-facing path.

The 1.0 target is **two-tier**: a **consumer default** path that reaches a first useful
chat with no terminal beyond starting the service, no manual model CLI, and no API key;
and a **prosumer advanced** path that keeps BYOK, custom endpoints, and CLI-driven
control. The consumer default may guide installation of the supported local model
runtime through an explicit in-app confirmation, but it never asks the operator to run
`ollama` commands or paste a hosted key. The consumer default is what a fresh
non-developer sees first; the advanced path is opt-in.

## Operator-Visible Win

A non-developer installs Allbert from a trusted package path, starts it as a persistent
background service (`brew services` / `systemctl --user`, so they never re-run `serve`),
opens the web workspace, and on the **consumer default** path uses guided local runtime
setup if needed, then starts a curated local-model pull with one click (in-app progress,
no manual `ollama` CLI, no API key) to reach a first useful chat. They understand any
blocked first-run state and take one clear repair action, with no source checkout, raw
`mix`, hand-edited config, or unexplained security/trust state. The **advanced** path
offers BYOK hosted setup and custom endpoints for prosumers who prefer them.

## Planned Interfaces

- Package-first install docs and onboarding entry path.
- Installer-side signature/trust verification, guided verification-tool bootstrap, and
  clear fail-closed messages.
- Upgrade rollback/restore automation or an explicit restore contract from the
  backup-before-migrate artifact.
- First-run state copy and repair-action contract for web, CLI, and TUI.
- Web-first QuickStart with terminal onboarding as secondary/resume path.
- Model setup repair contract for guided local-runtime install, Ollama pull progress,
  and BYOK/custom advanced providers.
- Trust-spine summary that explains local data, hosted-provider egress, secrets,
  confirmations, traces, and memory review in plain language.
- `mix allbert.test release.v064` deterministic gate.

## Security Posture

No new permission class, authority source, model privilege, or bypass. Installer
trust verifies release artifacts before install and fails closed when verification
cannot be completed; any verification-tool bootstrap is itself guided and visible.
Onboarding and repair screens may explain or route to existing actions, but they do not
grant authority by themselves. Secrets continue to resolve through the v0.62 vault
tiers, and all effectful repair work stays behind registered actions, Security Central,
confirmations, traces, and audits.

## Non-Goals

- No native desktop GUI client.
- No new integration surface beyond existing model/provider and onboarding
  paths.
- No bundled model or bundled local model runtime in v0.64; Ollama remains the
  managed external runtime selected by ADR 0078.
- No local files/notes/memory launch-path work; v0.65 owns that.
- No final product RC; v0.66 owns full no-docs validation.
- No contract freeze; v1.0 owns the freeze after v0.66 is green.

## Current Implementation Gaps

The release is **not implementation-ready** until these gaps are closed or explicitly
re-scoped:

- `mix allbert.test release.v064` is documented but not implemented in the Mix test
  dispatcher.
- Current first-model actions can detect Ollama, run the confirmed Ollama install path,
  and call `/api/pull`, but the pull path returns only a bounded summary; it does not yet
  expose web-visible progress, cancellation, retry, or resume.
- Web onboarding is still provider-key/doctor centric. It does not yet present the
  consumer-default guided runtime install and one-click model pull as the first path.
- CLI onboarding still asks for a provider key as the visible model step and does not
  offer the same guided consumer-default local setup.
- The web first-run auto-open path does not yet route `first_model_not_ready` and
  related repair states back into a model repair panel after onboarding.
- Bare CLI first-run copy can expose raw internal model state instead of one primary
  repair action (`cli.ex:405-407` interpolates the raw probe atom and re-probes Ollama).
- The web has **no standalone (non-wizard) model-repair destination**: model repair copy
  lives only inside the wizard's `model_path` step, unreachable once onboarding completes.
- The **TUI has no first-run/blocked-state gating** (`cli/tui.ex` launches the console
  regardless of readiness); the trust spine is only 4 static lines; and the built service
  action has no read-only `status` operation.
- The curl installer verifies SHA256 integrity from the release origin, but mandatory
  installer-side cosign verification, guided verification-tool bootstrap, and automated
  rollback/restore are still unimplemented.
- v0.66/v1.0 validation depends on v0.64 evidence, so downstream docs must not treat the
  consumer-default path as proven until these gaps close.

## Locked Decisions

- **Consumer model mechanism:** v0.64 uses the ADR 0078 assisted-local path: a managed
  Ollama runtime plus curated model pull. A literal no-external-runtime artifact is not
  the contract; the contract is no manual model CLI and no API key. If Ollama is missing,
  Allbert may guide its install behind explicit confirmation.
- **Trust verification:** installer-side verification is fail-closed. If cosign or an
  equivalent verifier is missing, the installer guides the operator through a supported
  verifier install path, then verifies before installing; warning-and-continue is not a
  v0.64 pass.
- **Rollback minimum:** every upgrade must have a named restore path from the
  backup-before-migrate artifact. If full automatic rollback is not shipped, the docs and
  release gate must explicitly prove the manual restore command and failure behavior.
- **Service posture:** Tier-1 validation starts Allbert through the persistent service
  path. Foreground `allbert serve` remains a diagnostic and repair fallback, not the
  happy path for non-developer validation. The service surface itself is **already built**
  (v0.62 M5: `AllbertAssist.Service`, `ServiceControl` = `admin service`, `ServeHealth` =
  `admin health`, the Homebrew `service do` block); v0.64's only service work is a
  read-only `status` view and the brew-services-vs-`systemctl --user` reconciliation.
- **Single curated model, not a catalog (2026-07-09).** The consumer-default one-click
  download pulls the **one** curated model (`Ollama.curated_model/0` = `llama3.2:3b`, 8 GB
  floor), not a browsable model catalog. A model chooser/catalog is explicitly post-v0.64.
- **TUI first-run scope (2026-07-09).** M2's blocked-state repair is delivered on **web
  and CLI** in v0.64. The TUI (`cli/tui.ex`) has no first-run gating today and gets only a
  **minimal guard** — if the home/onboarding/model is not ready, it prints a one-line
  "run onboarding first (`allbert onboard`)" pointer instead of launching the console.
  Full TUI repair panels are post-v0.64. (Rationale: web is the primary 1.0 surface; a
  greenfield TUI repair surface is disproportionate for the non-developer path.)
- **Cosign sign↔verify coupling (2026-07-09).** Adding fail-closed installer-side cosign
  verification requires **re-hardening the release workflow's cosign *sign* step in the
  same release** — it is currently `continue-on-error: true` (a v0.63 Fulcio-outage
  mitigation). A fail-closed verifier against an optional signer would ship unverifiable
  releases. Both changes land together and are recorded in ADR 0076.

## Code Seams (v0.64 recon, 2026-07-09)

Grounded from a read-only code recon so an engineer builds against real seams, not
reconstructs the plan. What exists is named; what is net-new is called out.

- **M0 installer verify (`scripts/install/install.sh:66-85`).** SHA256 fail-closed exists;
  add a `cosign verify-blob --bundle SHA256SUMS.cosign.bundle SHA256SUMS` (pinned to the
  repo OIDC identity + issuer) between checksum download and compare, plus a
  `command -v cosign` guided-bootstrap guard mirroring the existing `sha256sum` fallback.
  Re-harden `.github/workflows/release-artifacts.yml` cosign sign step (drop
  `continue-on-error`).
- **M0 restore (`apps/allbert_assist/lib/allbert_assist/database.ex:169-213`).**
  Backup-only today; add a `restore_from_backup/1` alongside `backup_database!/1` + a list
  helper over `db/backups/`, exposed as an `admin db restore` action
  (`actions/serve/`-style, `:command_execute` + `confirmation: :required`, mirroring
  `service_control.ex`).
- **M0.a/M4 pull + progress (`actions/first_model/pull_model.ex`).** The default puller
  (`~:141`) hardcodes `json: %{name: model, stream: false}` → single summary; flip to
  `stream: true` + `into:` a Req streaming callback consuming Ollama's
  `status/completed/total/digest` frames. This widens the injectable-puller contract
  (the `~:129` test seam), the run/output path, and the action's `output_schema` to carry
  progress. Broadcast frames onto the existing per-thread
  PubSub topic (`AllbertAssistWeb.SignalBridge.workspace_topic_for/2`) that
  `WorkspaceLive` already subscribes to (`workspace_live.ex:82-88`) — nothing in
  first_model broadcasts today.
- **M4 onboarding wiring (web `components/onboarding.ex:296-354`, CLI
  `cli/areas/onboarding.ex:217-226`).** Both model steps are BYOK/provider-key-first and
  only *name* `install_ollama`/`pull_model` as guidance strings. Add install + one-click
  pull affordances that dispatch the existing actions through the working create+approve
  paths — web `apply_persona/2` pattern (`components/onboarding.ex:455-478`), CLI
  `run_authorized/4`+`approve_gated/4` (`cli/areas/onboarding.ex:379-421`). The readiness
  atoms `:install_runtime`/`:pull_model` are already declared in `build_guidance/2`
  (`onboarding.ex:408-485`) but **no surface dispatches on them** — that is the hook.
- **M2 web dead-end (`workspace/first_run.ex:18`, `live/workspace_live.ex:1336`).**
  `auto_open?` covers only `:onboarding_incomplete`/`:profile_unreviewed`; a
  completed-onboarding-but-model-not-ready operator is routed to the default canvas, and
  **no standalone (non-wizard) model-repair destination exists** — build one.
- **M2 bare-CLI atom leak (`cli.ex:405-407`).** `first_run/0` interpolates the raw probe
  atom into copy and re-probes Ollama a second time; route it through
  `Onboarding.model_guidance_for/2` (already exists) like the `onboard` area does.
- **M5 trust spine (`onboarding.ex:95-104`).** 4 static lines cover
  confirmations/traces/local-data/permission; add **hosted-provider egress, secrets/vault,
  and memory review** and present as product controls, at that attribute + both render
  sites (`cli/areas/onboarding.ex:153-160`, `components/onboarding.ex:281-288`).
- **M7 gate (`lib/mix/tasks/allbert.test.ex`).** Copy the v063 quintet (usage `:43`,
  dispatch `:155`, `@release_v063_steps` `:4461`, `release_v063` `:4570`,
  `run_release_v063_step` `:4607`) to v064. Each proof row shells to `mix test <files>`, so
  the **cosign-verify, restore-path, and pull-progress test files must be written with the
  gate** — the rows have nothing to run otherwise.

## Milestones

- M0: Packaging-trust intake from v0.62. Implement guided, fail-closed
  installer-side signature verification for curl/Homebrew; implement the
  Developer-ID signing/notarization decision where applicable; test the
  self-extraction x hardened-runtime interaction; and implement or document a
  proven restore path from the backup-before-migrate artifact. If anything is
  consciously re-parked post-1.0, record the exception in ADR 0076 and the v1.0
  plan, not only in release notes.
- M0.a: Consumer-default model setup implementation. Deliver the locked ADR
  0078 mechanism: detect the managed Ollama runtime, guide confirmed runtime
  install if absent, then drive Ollama's pull API behind a friendly web
  progress surface for the curated model. Do not bundle a model or runtime in
  v0.64. Do not ask the operator to run the `ollama` CLI.
- M1: Package-first documentation contract. Operator docs start with packaged
  install, persistent service start, browser workspace, and `/health`/admin
  inspection. Source checkout, `mix setup`, and foreground `allbert serve`
  move to contributor or diagnostic orientation.
- M2: First-run state taxonomy and repair copy. Every blocked first-run state
  maps to user-facing language and exactly one primary repair action.
- M3: Web-first QuickStart on a persistent service. The browser path is the primary 1.0
  product surface, reached by starting Allbert as a background service (`brew services
  start allbert` / `systemctl --user`) once. The non-developer does not re-run `serve`.
  Terminal onboarding can resume or drive headless use but is not the default first-run
  story. If the platform service manager is absent or degraded, the product surfaces that
  as a repair state and may offer foreground `serve` as a fallback.
- M4: Two-tier model path. **Consumer default** — guided local runtime setup if needed,
  then one-click in-app download of a curated local model with an in-web progress surface
  (no manual `ollama` CLI, no API key), carrying a fresh operator to a first useful chat.
  **Advanced** — BYOK hosted setup and custom endpoints for prosumers. Both tiers
  surface readiness, progress, credential storage tier, network egress, and failure
  recovery without raw internal probe terms; the consumer default is the primary first-run
  path and the advanced path is opt-in.
- M5: Trust-spine presentation. Onboarding explains local data, hosted-provider
  egress, secrets, confirmations, traces, and memory review as product controls.
- M6: Fresh-home operator script. Define the no-docs first-run script that v0.66
  later executes with a fresh non-developer operator.
- M7: Implement and document `release.v064` by copying the v063 gate quintet (see Code
  Seams). Its proof rows shell to `mix test <files>`, so the **backing test files must be
  written with the gate** — focused tests for installer cosign verify, DB restore path,
  guided local-runtime setup, model-pull progress, first-run repair states (incl. the
  bare-CLI atom-leak fix and the web standalone repair destination), and natural-prompt
  routing — plus docs drift, warning gate, and the v0.65 handoff.

## Definition Of Done

v0.64 is done when:

- packaged install verification is fail-closed, verifier bootstrap is guided, and
  rollback/restore posture is implemented or explicitly dispositioned with accepted
  ADR/v1.0-plan language;
- operator-facing docs start with packaged install and browser-first onboarding,
  not source checkout or `mix`;
- every first-run blocked state has plain-language copy and one primary repair
  action across web/CLI/TUI;
- the consumer-default path carries a fresh operator to a first useful chat through
  guided local runtime setup if needed and one-click curated local-model download, with
  **no manual model CLI and no API key**, and the advanced BYOK/custom path is available
  and clearly labeled as opt-in;
- Allbert runs as a persistent background service on Tier-1 validation hosts so the
  operator does not re-run `serve`; service-manager absence is a documented repair state,
  not the happy path;
- secrets, logs, traces, evidence, and screenshots stay redacted;
- `mix allbert.test release.v064`, docs, warning, and drift checks pass;
- v0.65 has current inputs for local files, notes, and reviewed memory.
