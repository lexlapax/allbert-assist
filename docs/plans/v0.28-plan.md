# Allbert v0.28 Security Hardening And Evals

Request flow: `docs/plans/v0.28-request-flow.md`

## Purpose

v0.28 returns to Security Central after real execution, imports, jobs, channels,
memory review, richer intent, a full app contract, an objective runtime, and
LiveView surfaces exist. The goal is to harden the system with evidence from
actual runtime surfaces rather than guessing too early.

This milestone was formerly tracked as v0.16 before the post-v0.10 roadmap
was unified into one v0.xx sequence, then tracked as v0.24, then v0.25, then
v0.26, and finally v0.28 after the project-direction rethink inserted
v0.23 Jido State-Machine Convergence and v0.24 Objective Runtime Foundation.

## Expected Direction

- Add security eval fixtures for prompt injection, tool argument injection,
  SSRF attempts, unsafe redirects/retries, untrusted skill activation,
  malicious imports, package-manager lifecycle script abuse, unpinned package
  specs, URL/git/tarball/path/global package specs, command approval bypass,
  resource access operation-scope bypass, local path traversal, symlink escape,
  unsafe summarizer handoff, unsupported document extraction, credential
  leakage, cross-session data access, channel spoofing, and unsafe background
  execution.
- Add cross-user and thread leakage evals for attempts to read another user's
  thread history, inject a different `user_id` into a request, or
  cross-contaminate ETS scratchpad entries between sessions.
- Add plugin-boundary evals for malicious or malformed manifests, duplicate
  plugin ids, action-name collisions, disabled-plugin bypass attempts,
  path traversal in plugin skill roots, attempts to claim code-bearing
  contributions from `<ALLBERT_HOME>/plugins`, attempts to smuggle arbitrary
  `./plugins/*/lib` directories into compilation, and attempts to gain trust,
  permissions, resource grants, or confirmation bypass from plugin metadata.
- Add app-scoped action routing evals. StockSage actions registered through
  `AllbertAssist.App` must not be reachable when `active_app` is not
  `:stocksage` — including the default `:allbert` context established by v0.18;
  `StockSage.Actions.RunAnalysis` requires explicit StockSage session context.
- Add app-registration security evals. v0.18 app/surface contract is complete;
  these evals are required: attempts to register a malicious app that bypasses
  `validate/1`, claims another app's `app_id`, injects unauthorized actions
  into the registry, or registers skill paths outside the app's declared scope.
- Add memory namespace ownership evals before v0.29 adds memory writes:
  attempts to claim another app's namespace, register overlapping namespace
  prefixes, write through an undeclared namespace, or use namespace metadata to
  bypass Memory review, Security Central, redaction, or action-boundary
  requirements.
- Add surface and SurfaceProvider security evals: attempts to register
  component ids not declared in the app's catalog, Surface.Node structures
  carrying component types from a different app's catalog, action result surface
  nodes claiming elevated permissions through component metadata, SurfaceProvider
  returning route paths outside the app's declared scope, and
  `AllbertAssist.Surface.Encoder.to_a2ui/1` emitting fields that would bypass
  Allbert redaction or security policy if consumed by a downstream client.
- Add workspace surface security evals for v0.26: surface events that attempt
  to bypass the registered-action boundary, canvas tiles that claim persistence
  or provenance they were not granted, ephemeral surfaces that survive discard,
  and workspace shell events that mutate confirmations, memory, or resource
  grants directly.
- Add objective runtime security evals for v0.24: `objective_id` used as
  authority, cross-user objective lookup, cross-thread objective resume,
  unbounded loop_count, advisory provider output treated as observed fact,
  predictions about user behavior short-circuiting confirmation,
  `cancel_objective` race conditions with in-flight action, simulated state
  written as memory truth without operator confirmation.
- Treat the Python bridge as a security surface. Cover arbitrary command
  injection through the bridge JSON protocol, path traversal in bridge
  arguments, and bridge crash handling that does not leak partial analysis data.
- Add financial workflow authorization evals. StockSage analysis actions that
  trigger external market-data API calls must flow through Resource Access
  Security Posture and confirmation workflow; a remembered
  `external_service_request` grant for a general API must not automatically
  authorize financial analysis API calls.
- Add operator-visible security review workflows for recent denials,
  confirmations, imports, external calls, and redaction incidents.
- Reassess sandbox, allowlist, safe-bin, Resource Access Security Posture,
  document extraction, summarizer handoff, and supply-chain policies against
  real traces.
- Add hardening docs for deployment posture, channel pairing, exposed services,
  local file permissions, and emergency disable switches.

## Eval Harness Shape

Security evals are ExUnit tests, not a new runtime subsystem. They drive real
boundaries (`Actions.Runner.run/3`, `Resource Access Security Posture`,
`Workspace.Fragment.validate_received/1`, `App.Registry`, the objective runtime)
with adversarial fixtures and assert the boundary held.

### Locations (net-new; no eval dir exists today)

- Shared helper: `apps/allbert_assist/test/support/security_eval_case.ex`
  (`AllbertAssist.SecurityEvalCase`).
- Core eval modules: `apps/allbert_assist/test/security/*_eval_test.exs`.
- StockSage eval modules:
  `plugins/stocksage/test/security/*_eval_test.exs` (run via the precommit
  plugin-test path).
- Adversarial fixtures: `apps/allbert_assist/test/support/security_fixtures/`
  (and the StockSage equivalent), reusing the v0.10 M9 preserved cases.

### Harness contract

```elixir
defmodule AllbertAssist.SecurityEvalCase do
  use ExUnit.CaseTemplate
  # run_eval(fixture) -> %{decision: :allowed | :needs_confirmation | :denied
  #                                   | :dropped | :error,
  #                        result: term(), trace: map()}
  #   Drives the named boundary for the fixture and captures the decision + trace.
  # assert_denied(eval, opts)            — safety floor held; no side effect ran
  # assert_needs_confirmation(eval)      — gated by confirmation, not auto-run
  # assert_dropped(eval)                 — surface/fragment rejected pre-render/persist
  # assert_no_cross_user_leak(eval, other_user_id)
  # assert_no_secret_in(eval)            — output/trace carry no secret/credential
  # assert_trace_records(eval, keys)     — decision is auditable in the trace
end
```

### Eval fixture shape

```elixir
%{
  id: "ssrf-redirect-001",
  surface: :resource_access,
  scenario: "summarize_url follows a 302 to a 169.254.x.x internal address",
  boundary: {AllbertAssist.Actions.Runner, "external_network_request"},
  input: %{params: %{url: "http://attacker.example/redirect-to-metadata"}},
  expected: :denied,
  assert: [:denied, {:trace_records, [:resource_decision]}, {:no_call, Req}],
  test_module: "AllbertAssist.Security.ResourceExecutionEvalTest"
}
```

## Milestones

Dependency note: M1 (inventory + harness) gates M2–M6; M2–M6 are parallel-safe
with one another; M7 (operator surfaces) is parallel-safe with M2–M6; M8 is
terminal.

### M1 — Eval harness + inventory

Goal: ship `AllbertAssist.SecurityEvalCase`, the fixture dir, and an inventory
mapping every surface (v0.08→v0.27) to a catalog row. Depends on: nothing.

Tasks: implement the harness contract (`run_eval` + assert helpers); create the
fixture dirs; turn the Eval Catalog above into the full row inventory.

Tests: harness self-test (a known-denied fixture asserts denied; a known-allowed
fixture asserts allowed); inventory covers every mandated surface.

Acceptance: harness + fixture layout exist; every surface group has ≥1 row.

### M2 — Resource + execution evals

Goal: implement `resource_execution_eval_test.exs` covering injection, SSRF,
redirects, package abuse, command bypass, path traversal, summarizer/document
handoff, external-service grants. Depends on: M1. Parallel-safe with: M3–M7.

Tests: every catalog M2 row passes; the v0.10 M9 preserved grant cases still
hold. Acceptance: all M2 rows green; safety floors held.

### M3 — Identity + context evals

Goal: `identity_context_eval_test.exs` — cross-user/thread/session leakage,
spoofed `user_id`, scratchpad contamination, channel spoofing, job context.
Depends on: M1. Parallel-safe with: M2, M4–M7.

Tests: every M3 row passes; `assert_no_cross_user_leak` holds. Acceptance: no
cross-identity leak path remains.

### M4 — Plugin + app registry evals

Goal: `plugin_app_registry_eval_test.exs` — malformed/duplicate manifests,
disabled-plugin bypass, route/surface scope, app-scoped action routing, foreign
`app_id` claim. Depends on: M1. Parallel-safe with: M2–M3, M5–M7.

Tests: every M4 row passes; StockSage actions unreachable outside `:stocksage`.
Acceptance: registry + app-scope boundaries hold.

### M5 — Surface + workspace + namespace evals

Goal: `surface_workspace_eval_test.exs` — catalog bypass, component injection,
cross-app component theft, fragment/canvas provenance, ephemeral lifecycle,
`to_a2ui/1` redaction, namespace claim abuse (pre-write). Depends on: M1.
Parallel-safe with: M2–M4, M6–M7.

Tests: every M5 row passes; forged fragments/foreign components dropped;
namespace claims isolated. Acceptance: surface + namespace boundaries hold
before v0.29 enables writes.

### M6 — Objective + financial + bridge evals

Goal: `objective_financial_eval_test.exs` — objective_id authority misuse,
cross-objective resume, loop bound, advisory-as-fact, cancel race, bridge
protocol/path/crash, market-data authorization. Depends on: M1. Parallel-safe
with: M2–M5, M7.

Tests: every M6 row passes; generic grant ≠ market-data; bridge crash leaks no
partial data. Acceptance: objective + financial + bridge boundaries hold.

### M7 — Operator review surfaces

Goal: operator-visible review of recent denials, confirmations, imports,
external calls, redaction incidents — plus the emergency disable switches.
Depends on: M1. Parallel-safe with: M2–M6.

Tasks: define the review surface (CLI subcommands and/or a `/security` review
LiveView) + the disable-switch settings keys; wire to existing audit/trace.

Tests: review surface lists recent decisions; disable switch flips the gated
behavior. Acceptance: operators can inspect recent security decisions + flip
emergency switches.

### M8 — Policy reassessment + closeout

Goal: hardening docs, roadmap risk reassessment for v0.29+/v0.30/post-v0.31,
full gate. Depends on: M1–M7.

Tests: full gate (compile warnings-as-errors, format, Credo strict, Dialyzer,
`mix precommit`). Acceptance: all evals green; docs + roadmap risk section
current.

## Non-Goals

- No broad hosted multi-user authorization model.
- No replacement for Settings Central or Security Central.
- No new execution capabilities.

## Definition Of Done

v0.28 is done when:

- Security evals cover the high-risk agent failure modes that now exist.
- Cross-user and thread leakage fixtures are present and passing.
- App-scoped action routing fixtures are present and passing. v0.15 and v0.18
  are complete before this milestone.
- Plugin-boundary fixtures cover v0.17 plugin discovery, disabled-plugin
  behavior, skill-root path safety, duplicate contributions, and no runtime
  code loading from home plugins or arbitrary source-tree plugin folders.
- App-registration security fixtures are present and passing. v0.18 app/surface
  contract is complete before this milestone; no pending condition applies.
- Memory namespace ownership/isolation fixtures are present and passing before
  v0.29 consumes StockSage's declared namespace for memory sync.
- Surface catalog bypass, component injection, cross-app component type theft,
  SurfaceProvider scope violation, `to_a2ui/1` encoder redaction-bypass, and
  workspace shell boundary-bypass fixtures are present and passing.
- Operators can inspect recent security decisions and risky capability history.
- Security Central policy has been reviewed against real execution, channel,
  job, import, resource access, summarizer handoff, memory, surface, and
  financial-analysis behavior.
- The roadmap has a fresh risk reassessment for v0.29+, v0.30, and
  post-v0.31 work.
