# Allbert v0.14 Session Scratchpad And Active App Context

Formerly tracked as M-D1b.

## Purpose

v0.14 adds volatile local session state keyed by `{user_id, session_id}`. The
scratchpad lets Allbert carry short-lived workspace context, especially
`active_app`, across turns in the same local session without treating that
state as durable memory, authorization, or app registration.

This milestone is deliberately small. It makes `session_id` useful, gives the
runtime a safe place to read `active_app`, and prepares v0.15 app registration
and v0.20 cross-surface intent enrichment. It does not add the app registry,
app action routing, canvas state, hosted sessions, or a new UI surface.

The acceptance surface is runtime, CLI, signals, traces, and tests. Phoenix
LiveView may pass a `session_id` later, but v0.14 should not add an AgentLive
session sidebar, workspace shell, or canvas controls.

## Expected Inputs

- v0.11 is released and tagged. `AllbertAssist.Intent.Decision` already
  reserves `user_id`, `thread_id`, `session_id`, and `active_app`.
- v0.12 local workspace identity and conversation history are implemented.
  Runtime requests and responses carry canonical string `user_id`,
  compatibility `operator_id`, `thread_id`, and optional `session_id`.
- v0.13 scheduled jobs are implemented. Jobs may carry `session_id`; when they
  enter the runtime after v0.14, the runtime may read scratchpad context for
  that session but jobs must not mutate scratchpad implicitly.
- ADR 0014 local workspace identity is accepted.
- `AllbertAssist.Runtime.submit_user_input/1` remains the runtime entrypoint,
  and channels/jobs continue to call it instead of calling agents directly.

## Roadmap And Vision Fit

The Allbert/Jido vision treats session scratchpad state as local, volatile
context around the signal-driven runtime. It is useful because the workspace is
becoming multi-app, but it is intentionally weaker than durable storage:

- SQLite `Thread` and `Message` rows remain durable conversation history.
- Markdown memory remains the durable, operator-edited source of truth.
- ETS scratchpad entries are fast, local, TTL-expiring session context.

This milestone prepares later work without implementing it:

- v0.15 app registration can use `active_app` as context while registering app
  actions and skill paths through explicit contracts.
- v0.16 channels can map external session/thread identities into local
  `session_id` values without inventing per-channel scratchpads.
- v0.20 intent enrichment can use `active_app` to prioritize app-registered
  actions only after the app registry exists.
- v0.22 StockSage LiveViews can set `active_app: :stocksage` when the user is
  inside StockSage context.
- v0.23 security evals can test cross-session leakage and app-scoped routing.
- v0.26/v0.27 canvas work can add durable or task-scoped surface state later;
  v0.14 reserves no canvas storage shape.

v0.14 must keep Allbert local-first and inspectable. The scratchpad should be a
supervised process with clear APIs, predictable expiry, redacted observability,
and boring failure behavior.

## Handoffs

### v0.12 Handoff

v0.14 keys scratchpad entries from the canonical v0.12 `user_id` and optional
`session_id`. It must not move SQLite conversation history into ETS, change
thread selection semantics, or use scratchpad presence as authorization.

When no `session_id` is present, the runtime behaves as v0.12/v0.13 do today:
no scratchpad lookup, `active_app: nil`, and no scratchpad entry creation.

### v0.13 Handoff

Scheduled jobs may pass a stored `session_id` to the runtime. If a matching
scratchpad entry exists and has not expired, the job's runtime prompt receives
that session context. Jobs must not create, update, or clear scratchpad entries
as a side effect of running. Registered action jobs keep using action context
and do not read scratchpad unless a later plan explicitly adds that behavior.

### v0.15 And v0.20 Handoff

v0.14 carries `active_app`; it does not route on it. The runtime and intent
agent receive `active_app` so later app-aware ranking can consume the field.
Until v0.15 registers apps and v0.20 enriches intent, `active_app` is context
and trace metadata only.

## Implementation Boundaries

Authoritative existing boundaries:

- Runtime entrypoint: `AllbertAssist.Runtime.submit_user_input/1`.
- Runtime agent path: `AllbertAssist.Agents.IntentAgent.respond/1`.
- Conversation history: `AllbertAssist.Conversations`.
- Intent decision contract: `AllbertAssist.Intent.Decision`.
- Tracing: `AllbertAssist.Trace` and the `record_trace` action.
- Settings: `AllbertAssist.Settings`.
- Supervision: `AllbertAssist.Application`.

Preferred new implementation shape:

- `AllbertAssist.Session.Scratchpad`: supervised GenServer that owns an ETS
  table and exposes the public scratchpad API.
- `AllbertAssist.Session.Entry`: optional struct or documented map shape for
  normalized entry data.
- `AllbertAssist.Session.AppId`: small normalization helper for `active_app`
  values, if keeping that logic separate makes the boundary clearer.
- `Mix.Tasks.Allbert.Sessions`: CLI inspection and explicit active-app control
  for manual verification.

The ETS table should be owned by the supervised scratchpad process. Runtime,
CLI, tests, jobs, and future channel adapters should use the scratchpad API
rather than reading or writing ETS directly.

The scratchpad process belongs in the application supervision tree before
anything that can submit runtime work on startup. It may be disabled or named
through application config for tests, but operator-tunable values belong in
Settings Central.

## Settings

Add one writable Settings Central key:

- `sessions.scratchpad_ttl_minutes`: integer, default `30`, minimum `1`,
  maximum `1440`.

Runtime/process config may also expose test-only or deployment-only options:

- `enabled?`: default `true`.
- `table_name`: default `:allbert_session_scratchpad`.
- `sweep_interval_ms`: default `60_000`, test-configurable.

The TTL setting controls new writes and runtime touches. Existing entries keep
their stored `expires_at_ms` until touched, updated, or swept.

## Scratchpad State Contract

Entries are keyed by `{user_id, session_id}` after normalization.

Stored entry shape:

```elixir
%{
  user_id: "alice",
  session_id: "sess-1",
  active_app: :stocksage,
  working_memory: %{},
  inserted_at_ms: 12_345,
  updated_at_ms: 12_345,
  expires_at_ms: 1_812_345,
  metadata: %{}
}
```

Rules:

- `user_id` is a non-empty string and defaults upstream to `"local"`.
- `session_id` is required for scratchpad operations, non-empty after trim,
  and should be capped at 128 characters.
- `active_app` is `atom | nil`.
- `working_memory` is a bounded map for transient, non-secret context.
- `metadata` is a bounded map for non-secret diagnostics such as channel or
  surface.
- Expiry uses monotonic milliseconds, not wall-clock datetimes.
- CLI/log/trace output may show `active_app`, key names, counts, and expiry
  metadata, but must not dump arbitrary `working_memory` values by default.

`working_memory` is intentionally small:

- Accept maps only.
- Reject obvious secrets and rely on existing redaction for rendering.
- Reject oversize payloads before storing. A reasonable initial cap is 64 KiB
  measured with `:erlang.external_size/1`.
- Merge semantics are shallow by default: new keys replace existing keys; a
  later milestone may add nested patch semantics if needed.

## Active App Semantics

`active_app` means "the session currently has evidence that the operator is in
an app context." It does not mean "the app exists," "the user is authorized,"
or "app actions should run."

Normalization rules:

- nil, `""`, `"none"`, and `"general"` normalize to nil.
- Atoms are accepted by internal APIs.
- Binary CLI input must not create arbitrary atoms. Before v0.15, CLI input
  may map only documented reserved values such as `"stocksage"` to a
  compile-time atom (`:stocksage`) plus nil/general values.
- v0.15 may replace or extend this with app-registry validation.

v0.14 does not prioritize StockSage actions, discover app skill paths, or
enforce app-scoped action routing. Those are v0.15/v0.20/v0.23 responsibilities.

## Runtime Request Contract

Extend runtime request normalization with scratchpad lookup:

1. Normalize `user_id`, `operator_id`, `thread_id`, and `session_id` exactly as
   v0.12 defines.
2. If `session_id` is nil, set `active_app: nil` and skip scratchpad lookup.
3. If `session_id` is present, fetch `{user_id, session_id}` from
   `AllbertAssist.Session.Scratchpad`.
4. If no entry exists or the entry expired, use `active_app: nil`.
5. If an entry exists, read `active_app` and pass it into runtime request,
   input signal, intent-agent request, response signal, response map, trace
   context, and assistant message metadata.
6. A runtime read may touch the entry and extend TTL; it must not create a new
   entry when none exists.

The runtime agent request should include:

```elixir
%{
  text: request.text,
  channel: request.channel,
  user_id: request.user_id,
  operator_id: request.operator_id,
  thread_id: request.thread_id,
  session_id: request.session_id,
  active_app: request.active_app,
  thread_context: request.thread_context,
  metadata: request.metadata,
  timeout_ms: request.timeout_ms,
  input_signal_id: signal.id
}
```

Runtime responses should include `active_app` even when nil so channel adapters
have one stable field to render or ignore.

## CLI Contract

Update `mix allbert.ask`:

- Add `--session SESSION_ID`.
- Preserve existing `--user`, `--operator`, `--thread`, and `--new-thread`
  behavior.
- Render `Session: SESSION_ID` when present.
- Render `Active app: stocksage` or `Active app: none`.
- Do not add `--active-app` to `ask` in v0.14; setting session state should be
  explicit through `mix allbert.sessions`.

Add `mix allbert.sessions`:

- `mix allbert.sessions list [--user USER]`
- `mix allbert.sessions show --user USER --session SESSION_ID`
- `mix allbert.sessions set-active-app --user USER --session SESSION_ID APP`
- `mix allbert.sessions clear-active-app --user USER --session SESSION_ID`
- `mix allbert.sessions clear --user USER --session SESSION_ID`
- `mix allbert.sessions sweep`

CLI rules:

- Omitted `--user` defaults to `"local"`.
- `--operator` is accepted as a compatibility alias and must match `--user`
  when both are present.
- `set-active-app` creates or updates the scratchpad entry.
- `clear-active-app` keeps the entry but sets `active_app: nil`.
- `clear` deletes the entry.
- `show` prints `user_id`, `session_id`, `active_app`, remaining TTL, metadata
  keys, and working-memory key count. It does not print raw working-memory
  values by default.

## Signals, Traces, And Message Metadata

Add `active_app` to:

- `allbert.input.received` signal data.
- `allbert.agent.responded` signal data.
- runtime response maps.
- intent-agent request maps.
- `AllbertAssist.Intent.Decision` context passed through the runtime.
- trace request/context summaries.
- assistant message metadata.

Do not include raw scratchpad `working_memory` in signals, traces, logs, or
conversation message metadata. If a future action needs to expose part of
working memory, it should do so through an explicit redacted summary.

Scratchpad operations may emit log lines or diagnostics, but v0.14 does not
need a durable scratchpad audit table. Security-sensitive or app-routing evals
arrive in v0.23.

## Expiry, Sweep, And Restart Semantics

- `get` returns nil/not found for expired entries and may delete them.
- `touch` extends `expires_at_ms` from the current monotonic time using the
  current TTL setting.
- The supervised scratchpad process periodically sweeps expired entries.
- `sweep` is exposed for tests and CLI manual verification.
- If the scratchpad process crashes, OTP restarts it with an empty ETS table.
- Application restart starts with an empty scratchpad.
- Expiry or restart must not delete SQLite threads, messages, jobs, memory, or
  settings.

## Milestones

### Milestone 1: Scratchpad Process And API

Status: planned.

Goal: add the supervised volatile session store without touching runtime
behavior.

Implementation expectations:

- Add `AllbertAssist.Session.Scratchpad` under supervision.
- Own a protected ETS table from the scratchpad process.
- Implement public APIs for `get`, `put`, `set_active_app`,
  `clear_active_app`, `merge_working_memory`, `clear`, `list`, `touch`, and
  `sweep_expired`.
- Normalize `{user_id, session_id}` keys and reject blank session ids.
- Enforce TTL using `sessions.scratchpad_ttl_minutes`.
- Keep tests isolated and able to start named scratchpad processes/tables.

### Milestone 2: Active App Normalization And CLI Sessions Surface

Status: planned.

Goal: make session state inspectable and manually controllable without adding
workspace UI.

Implementation expectations:

- Add safe `active_app` normalization that never creates atoms from arbitrary
  user input.
- Add `mix allbert.sessions` list/show/set-active-app/clear-active-app/clear/
  sweep commands.
- Preserve `--user`/`--operator` compatibility and conflict behavior.
- Redact or omit raw working-memory values from CLI output.
- Add CLI tests for alice/bob isolation, pending TTL display, active-app
  setting/clearing, invalid app ids, and missing sessions.

### Milestone 3: Runtime Scratchpad Lookup And Active App Propagation

Status: planned.

Goal: make `active_app` visible to the runtime and intent agent when a session
has scratchpad state.

Implementation expectations:

- Add `--session` to `mix allbert.ask`.
- Add `active_app` to runtime request/response structs/maps and types.
- Read the scratchpad after identity/thread normalization and before input
  signal emission.
- Pass `active_app` to input signals, intent-agent request maps, response
  signals, response maps, trace context, and assistant message metadata.
- Touch existing scratchpad entries on runtime reads, but do not create empty
  entries implicitly.
- Preserve v0.11-v0.13 risky execution and confirmation behavior unchanged.

### Milestone 4: Expiry, Restart, And Observability Hardening

Status: planned.

Goal: prove scratchpad volatility and make failure behavior boring.

Implementation expectations:

- Add deterministic tests for expired entries, sweep, touch/TTL extension, and
  process restart emptying the table.
- Ensure expired entries produce `active_app: nil` in runtime context.
- Ensure runtime continues gracefully if scratchpad lookup fails, with a
  diagnostic rather than a crashed turn when possible.
- Ensure traces/logs do not contain raw working-memory values.
- Confirm no conversation history, markdown memory, jobs, settings, or
  confirmations are deleted by scratchpad expiry or restart.

### Milestone 5: Release Readiness And Manual Verification Docs

Status: planned.

Goal: close v0.14 with operator-readable verification and future handoffs.

Implementation expectations:

- Update README, CHANGELOG, roadmap, v0.14 plan/request-flow status, and
  version metadata.
- Include manual verification for setting active app, asking through a session,
  clearing active app, alice/bob isolation, omitted-session behavior, and
  restart volatility.
- State release/tag readiness and the operator manual verification path.

## Acceptance

- `mix allbert.sessions set-active-app --user alice --session sess-1 stocksage`
  creates a volatile entry with `active_app: :stocksage`.
- `mix allbert.ask --user alice --session sess-1 "hello"` passes
  `active_app: :stocksage` to the intent agent and returns/renders that
  context.
- `mix allbert.ask --user alice "hello"` keeps `active_app: nil`.
- Alice and Bob using the same `session_id` do not see each other's
  scratchpad entries.
- `clear-active-app` sets active app back to nil without deleting the session
  entry.
- Expired entries are ignored and swept.
- Restarting the scratchpad process or application starts with no session
  entries.
- Existing v0.11-v0.13 URL/document/import/package/shell/script/job
  confirmation behavior is unchanged.
- No markdown memory, SQLite conversation rows, job rows, or settings rows are
  created from ordinary scratchpad reads.

## Non-Goals

- No durable session persistence.
- No browser cookie/session-token/auth model.
- No hosted accounts, roles, or permission grants.
- No app registry, app action routing, app skill ranking, or StockSage domain
  behavior.
- No AgentLive session sidebar, workspace shell, canvas state, or ephemeral UI.
- No automatic session creation from arbitrary `session_id` reads.
- No automatic markdown memory or conversation-history promotion from
  scratchpad contents.
- No generic dynamic atom creation from user input.
- No durable scratchpad audit table.

## Test Plan

- Scratchpad API tests cover create/get/touch/list/clear, active-app
  set/clear, working-memory merge, TTL expiry, sweep, oversize rejection, and
  alice/bob isolation.
- Supervision tests prove process restart creates an empty table and does not
  affect durable data.
- Runtime tests with a stub agent prove `active_app` reaches input signals,
  agent request maps, response signals, response maps, traces, and assistant
  metadata when a matching entry exists.
- Runtime tests prove no-session, missing-session, expired-session, and
  scratchpad-error paths return `active_app: nil` or diagnostics without
  changing v0.12 thread behavior.
- CLI tests cover `mix allbert.ask --session`, `mix allbert.sessions`
  list/show/set-active-app/clear-active-app/clear/sweep, identity aliasing,
  invalid app ids, and redacted output.
- Integration tests prove jobs with a stored `session_id` can carry active app
  context through runtime prompt jobs without jobs mutating scratchpad.
- Regression tests prove risky v0.11-v0.13 prompts still stop at existing
  Resource Access and confirmation boundaries.
- Final code gates follow `DEVELOPMENT.md`: `mix compile --warnings-as-errors`,
  `mix format --check-formatted`, `mix credo --strict`, `mix dialyzer`, and
  `mix precommit`.

## Definition Of Done

v0.14 is done when session scratchpad state is supervised, TTL-expiring,
runtime-visible, CLI-inspectable, trace-safe, and covered by focused tests
without changing durable memory, conversation history, app routing, or
authorization behavior.
