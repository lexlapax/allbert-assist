# Allbert v0.30 App Canvas Contract - StockSage Canvas Integration

Request flow: `docs/plans/v0.30-request-flow.md`

Formerly tracked as M-Canvas, previously planned as v0.28 before the
project-direction rethink inserted v0.23 Jido State-Machine Convergence and
v0.24 Objective Runtime Foundation. Renamed after the post-v0.26 roadmap
reconciliation to make the contract visible: v0.30 proves app-owned canvas
emission through StockSage.

## Purpose

v0.30 wires the StockSage components proven in v0.27 into the v0.26 workspace
canvas substrate. It is not new StockSage domain work and it is not a new
renderer milestone.

The renderer-vs-emission boundary is deliberate:

- v0.27 renders real StockSage components in StockSage-owned `/stocksage/...`
  LiveViews and validates `RunAnalysis` Surface-node output.
- v0.30 emits those proven components as durable `/agent` workspace canvas
  tiles through `canvas_ops` and the v0.26/v0.28-audited canvas mechanism.

## Expected Inputs

- v0.26 workspace surface and canvas substrate exist.
- v0.27 App Surface Contract is complete: real StockSage renderers,
  component/catalog declarations, and validated analysis Surface nodes exist.
- v0.28 Security Hardening And Evals has audited workspace fragments, canvas
  provenance, catalog validation, and app-scoped Surface behavior.
- v0.29 App Memory + Outcomes Contract has completed StockSage product polish.
- v0.18 app/surface contract includes canvas/catalog extension points.

## Implementation Decisions

### Component Catalog

v0.30 does **not** add a `:stock_chart` catalog atom. The v0.26 catalog already
reserved the four StockSage atoms that v0.27 proved in StockSage-owned
LiveViews:

- `:analysis_card`
- `:agent_report_card`
- `:parity_card`
- `:debate_round_card`

Durable `/agent` canvas tiles must render those same four atoms. Chart-like
data, trend snippets, and analysis metrics remain properties or children of
the existing StockSage cards until a later catalog-expansion ADR explicitly
adds a chart atom. This keeps ADR 0015's 42-component catalog stable and avoids
a second StockSage card shape.

### Renderer Ownership

v0.27 owns the StockSage card UI in `StockSageWeb.Components.Cards` and
dispatches it through `StockSageWeb.Components.SurfaceRenderer`. v0.30 wires
the workspace renderer to those existing renderers for durable canvas tiles:

- `AllbertAssistWeb.Workspace.Components.AnalysisCard`
- `AllbertAssistWeb.Workspace.Components.AgentReportCard`
- `AllbertAssistWeb.Workspace.Components.ParityCard`
- `AllbertAssistWeb.Workspace.Components.DebateRoundCard`

Those workspace LiveComponents are thin adapters. They assign the existing
workspace renderer context and call the v0.27 StockSage function components.
They must not reimplement StockSage card markup or bring back v0.26 stub text.

### Emission Path

The existing `AllbertAssist.Workspace.Emitters.stocksage_signal/2` function is
the v0.30 canvas operation adapter. `StockSage.Actions.RunAnalysis` already
emits the three canonical lifecycle signals:

- `allbert.stocksage.analysis_requested`
- `allbert.stocksage.analysis_completed`
- `allbert.stocksage.analysis_failed`

v0.30 treats the corresponding emitted Fragment envelopes as the concrete
`canvas_ops` output. The action response does not need a new top-level
`canvas_ops` field unless a later operator API needs to display them directly;
the authoritative side effect is the signed, catalog-validated Fragment path.

## `canvas_op` Payload

v0.30 introduces NO new emission mechanism. A `canvas_op` is a thin wrapper that
StockSage uses to emit a durable tile through the **v0.26-audited
`AllbertAssist.Workspace.Fragment` path** — the same signed `FragmentEnvelope`
already used by `AllbertAssist.Workspace.Emitters` for confirmation/objective
fragments. v0.30 is StockSage *calling* that audited path, not a parallel one.

- **Envelope reuse**: a `canvas_op` produces a
  `AllbertAssist.Workspace.Fragment.Envelope` with
  `scope: :canvas`, `kind: <component atom>`, `emitter_id: "StockSage.Actions.RunAnalysis"`,
  the analysis `user_id`/`thread_id`, a stable `id`
  (`"stocksage_analysis_<analysis_id>"` so re-emit updates, not duplicates), and
  the validated Surface nodes from v0.27 as the body. It is signed and validated
  by `Fragment.emit/1` (signature, catalog, emitter allow-list, rate limit,
  size) exactly like every other fragment. No new validation, no new signing.
- **Emitter module**: `AllbertAssist.Workspace.Emitters.stocksage_signal/2`
  builds StockSage Fragment envelopes and calls `Workspace.Fragment.emit/1`.
  StockSage does not write `workspace_canvas_tiles` directly — persistence is
  the v0.26 receiver's job after validation.
- **Persistence**: reuses the v0.26 `workspace_canvas_tiles` table — **no
  migration**. The tile `kind` is the component atom; `body_yaml_path` carries
  the rendered node body; provenance/`emitter_id` come from the envelope.
- **Tile ids**: requested, completed, and failed fragments use stable ids under
  the `stocksage_analysis_*` prefix. Re-emitting the same body is idempotent;
  emitting the same id with a different body must fail through the existing
  Fragment/Canvas conflict path and must not overwrite the stored YAML body.
- **Settings**: no new `workspace.*` settings. v0.30 reuses
  `workspace.fragment.*`, `workspace.canvas.*`, `stocksage.web.enabled`, and
  the existing StockSage analysis settings.
- **Failure contract**: canvas emission is best effort after analysis
  authority has already completed. A tile-emission error must not rerun or
  roll back analysis persistence. It is visible through workspace dropped
  signals/log diagnostics and focused tests.

## Module Shapes

### Workspace StockSage Card Adapters

`apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/app_cards.ex`
defines one LiveComponent module per app-card atom. In v0.30 the four
StockSage modules become adapters:

```elixir
defmodule AllbertAssistWeb.Workspace.Components.AnalysisCard do
  use AllbertAssistWeb, :live_component

  @impl true
  def update(assigns, socket) do
    {:ok,
     socket
     |> assign(assigns)
     |> assign_new(:renderer_context, fn -> %{} end)
     |> assign_new(:workspace_state, fn -> %{} end)}
  end

  @impl true
  def render(assigns) do
    ~H"""
    <StockSageWeb.Components.Cards.analysis_card node={@node} />
    """
  end
end
```

The same adapter shape applies to `AgentReportCard`, `ParityCard`, and
`DebateRoundCard`. Tests assert the workspace renderer output includes
`data-stocksage-component="<atom>"` and does not include `v0.26 stub`.

### StockSage Canvas Operation Adapter

`AllbertAssist.Workspace.Emitters.stocksage_signal/2` is the only v0.30
canvas-op builder:

```elixir
@spec stocksage_signal(String.t(), map()) :: :ok
def stocksage_signal(type, payload) when is_binary(type) and is_map(payload)
```

It must continue to:

- require `user_id` and `thread_id`;
- redact payloads before building props;
- emit only the four known StockSage card atoms;
- sign envelopes with `Workspace.Fragment.SigningSecret`;
- call `Workspace.Fragment.emit/1`;
- return `:ok` on emission failures so analysis persistence remains durable.

### `RunAnalysis` Integration

`StockSage.Actions.RunAnalysis.emit_analysis_signal/2` remains the integration
point. It logs the StockSage lifecycle signal and then calls
`WorkspaceEmitters.stocksage_signal(type, payload)`. v0.30 tests cover that the
approved run path creates durable canvas tiles through the Fragment path when a
thread id is present.

## Milestones

### M1: Canvas Renderer Registration

Goal: `/agent` durable canvas tiles render StockSage card atoms through the
v0.27 StockSage renderers instead of the v0.26 stubs.

Depends on: v0.27 StockSage card components and v0.26 workspace renderer
dispatch.

Parallel-safe: yes; this is a web-renderer adapter change only.

Tasks:

- Replace the four StockSage app-card stubs in
  `AllbertAssistWeb.Workspace.Components.AppCards` with thin adapters to
  `StockSageWeb.Components.Cards`.
- Keep `AllbertAssistWeb.Workspace.Renderer.renderer_for/1` mapped to the
  same four component modules.
- Do not add `:stock_chart` or any new catalog atom.
- Update renderer tests to assert no StockSage workspace card emits
  `v0.26 stub`.

Tests:

- `mix test apps/allbert_assist_web/test/allbert_assist_web/workspace/renderer_test.exs`
- `mix test apps/allbert_assist_web/test/stocksage_web/components/cards_test.exs`

Acceptance:

- `:analysis_card`, `:agent_report_card`, `:parity_card`, and
  `:debate_round_card` render in the workspace renderer using
  `data-stocksage-component`.
- Unknown catalog components still render through the safe placeholder.

### M2: Canvas Operation Emission Contract

Goal: StockSage analysis lifecycle signals create signed durable Fragment
envelopes through the existing workspace emitter path.

Depends on: M1 for renderer output; existing RunAnalysis lifecycle signals.

Parallel-safe: no; this touches the action-to-workspace seam.

Tasks:

- Keep `RunAnalysis.emit_analysis_signal/2` as the single lifecycle hook.
- Ensure completed, failed, and requested StockSage signals carry enough
  workspace context (`user_id`, `thread_id`, analysis id, engine, objective id,
  step id, summary/error) to build durable tile envelopes.
- Extend focused tests so an approved `run_analysis` call with `thread_id`
  creates persistent `workspace_canvas_tiles` rows for the emitted StockSage
  cards.
- Assert emission does not happen without a workspace context and does not
  change the analysis result.

Tests:

- `mix test plugins/stocksage/test/stocksage/actions/run_analysis_native_test.exs`
- `mix test apps/allbert_assist/test/allbert_assist/workspace/emitters_test.exs`
- `mix test apps/allbert_assist/test/allbert_assist/workspace/fragment_test.exs`

Acceptance:

- An approved native run with `thread_id` persists an analysis row and durable
  StockSage canvas tiles.
- Emission failures remain bounded and do not fail the completed analysis.

### M3: Persistence, Reload, And Provenance

Goal: StockSage canvas tiles reload by user/thread with preserved provenance and
safe idempotency.

Depends on: M2 durable emission.

Parallel-safe: yes after M2 lands.

Tasks:

- Add or extend tests that load `Workspace.canvas_tiles(thread_id, user_id)`
  after emission and assert the tile body contains encoded Surface nodes.
- Assert tile metadata includes `emitter_id`, `scope`, and emitted timestamp.
- Assert tile ids are stable and idempotent for same-body re-emission.
- Assert same id with different body fails through existing conflict behavior.

Tests:

- `mix test apps/allbert_assist/test/allbert_assist/workspace/fragment_test.exs`
- `mix test apps/allbert_assist/test/allbert_assist/workspace/emitters_test.exs`

Acceptance:

- Refresh/reload paths can reconstruct StockSage tile bodies from stored YAML.
- No tile leaks across user/thread boundaries.

### M4: Operator Flow And UI Verification

Goal: `/agent` visibly shows durable StockSage tiles rendered with the same card

Depends on: M1-M3.

## Non-Goals

- No new trading agents, analysis engines, outcome records, or StockSage domain
  behavior.
- No new app renderer contract; v0.27 owns renderers.
- No arbitrary model-generated components.
- No canvas contract changes outside v0.26 extension points.
- No bypass of the v0.28 security posture.

## Test Plan

- An approved StockSage run can push registered `analysis_card`,
  `agent_report_card`, `debate_round_card`, and `parity_card` tiles when an
  analysis completes with workspace context.
- Persistent tiles survive reload when marked persistent.
- Unknown StockSage component ids are rejected by catalog validation.
- Canvas operations use audited workspace fragment/canvas paths and preserve
  user/thread provenance.
- Full gate at milestone closeout: compile warnings-as-errors, format check,
  Credo strict, Dialyzer, and `mix precommit`.

## Definition Of Done

v0.30 is done when StockSage analysis can produce validated durable workspace
canvas tiles using the proven StockSage components and audited Allbert canvas
mechanism, without adding new domain behavior or renderer semantics.
