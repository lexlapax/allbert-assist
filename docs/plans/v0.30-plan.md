# Allbert v0.30 App Canvas Contract - StockSage Canvas Integration

Request flow: `docs/plans/v0.30-request-flow.md`

Formerly tracked as M-Canvas, previously planned as v0.28 before the
project-direction rethink inserted v0.23 Jido State-Machine Convergence and
v0.24 Objective Runtime Foundation. Renamed after the post-v0.26 roadmap
reconciliation to make the contract visible: v0.30 proves app-owned canvas
emission through StockSage.

## Purpose

v0.30 wires the StockSage components proven in v0.27 into the v0.26 workspace
canvas substrate. It is not new StockSage domain work and it is not a new
renderer milestone.

The renderer-vs-emission boundary is deliberate:

- v0.27 renders real StockSage components in StockSage-owned `/stocksage/...`
  LiveViews and validates `RunAnalysis` Surface-node output.
- v0.30 emits those proven components as durable `/agent` workspace canvas
  tiles through `canvas_ops` and the v0.26/v0.28-audited canvas mechanism.

## canvas_op Payload (seam spec — pin before implementing M2)

v0.30 introduces NO new emission mechanism. A `canvas_op` is a thin wrapper that
StockSage uses to emit a durable tile through the **v0.26-audited
`AllbertAssist.Workspace.Fragment` path** — the same signed `FragmentEnvelope`
already used by `AllbertAssist.Workspace.Emitters` for confirmation/objective
fragments. v0.30 is StockSage *calling* that audited path, not a parallel one.

- **Envelope reuse**: a `canvas_op` produces a
  `AllbertAssist.Workspace.Fragment.Envelope` with
  `scope: :canvas`, `kind: <component atom>`, `emitter_id: "StockSage.Actions.RunAnalysis"`,
  the analysis `user_id`/`thread_id`, a stable `id`
  (`"stocksage_analysis_<analysis_id>"` so re-emit updates, not duplicates), and
  the validated Surface nodes from v0.27 as the body. It is signed and validated
  by `Fragment.emit/1` (signature, catalog, emitter allow-list, rate limit,
  size) exactly like every other fragment. No new validation, no new signing.
- **Emitter module**: `StockSage.Canvas.Emitter` (or extend the existing
  `AllbertAssist.Workspace.Emitters.stocksage_signal/2` path already present from
  v0.26) builds the envelope and calls `Workspace.Fragment.emit/1`. StockSage
  does not write `workspace_canvas_tiles` directly — persistence is the v0.26
  receiver's job after validation.
- **Persistence**: reuses the v0.26 `workspace_canvas_tiles` table — **no
  migration**. The tile `kind` is the component atom; `body_yaml_path` carries
  the rendered node body; provenance/`emitter_id` come from the envelope.
- **Component-atom reconciliation (decision to pin)**: `:analysis_card` reuses
  the reserved v0.26 catalog atom (good). `:stock_chart` is **NOT** in the v0.26
  42-component catalog — v0.30 must either (a) add `:stock_chart` as a new
  catalog atom (catalog 42 → 43, an ADR 0015 amendment), or (b) render charts
  inside the existing `:analysis_card` body or existing primitive nodes instead
  of a new atom. Pin this before M1; option (b) avoids a catalog change.

## Blocker Shapes To Pin Before Implementing

Just-in-time plan. Before M-level work: resolve the `:stock_chart` atom decision
above; pin the `canvas_op` → `FragmentEnvelope` mapping; confirm "no migration —
reuses `workspace_canvas_tiles`"; and confirm no new `workspace.*` settings (it
reuses v0.26 canvas caps + fragment rate limits). Then expand the milestone
bodies to the v0.26 milestone shape (Goal/Depends/Parallel-safe/tasks/Tests/
smoke/Acceptance).

## Milestones

- M1: Canvas component registration - register StockSage chart and analysis
  card canvas components through the existing Surface/catalog path.
- M2: Canvas ops contract - add analysis-result `canvas_ops` for durable tile
  creation without changing `RunAnalysis` domain semantics.
- M3: Persistence and reload - ensure emitted tiles persist by user/thread and
  survive browser refresh and app restart.
- M4: Dashboard integration - let StockSage workspace/dashboard flows expose
  recent-analysis persistent tiles.
- M5: Validation and closeout - prove unknown components are rejected and the
  full warning gate passes.

## Non-Goals

- No new trading agents, analysis engines, outcome records, or StockSage domain
  behavior.
- No new app renderer contract; v0.27 owns renderers.
- No arbitrary model-generated components.
- No canvas contract changes outside v0.26 extension points.
- No bypass of the v0.28 security posture.

## Test Plan

- `mix allbert.ask --user alice "analyze AAPL"` can push a registered
  `stock_chart` or `analysis_card` tile when an analysis completes in
  StockSage context.
- Persistent tiles survive reload when marked persistent.
- Unknown StockSage component ids are rejected by catalog validation.
- Canvas operations use audited workspace fragment/canvas paths and preserve
  user/thread provenance.
- Full gate at milestone closeout: compile warnings-as-errors, format check,
  Credo strict, Dialyzer, and `mix precommit`.

## Definition Of Done

v0.30 is done when StockSage analysis can produce validated durable workspace
canvas tiles using the proven StockSage components and audited Allbert canvas
mechanism, without adding new domain behavior or renderer semantics.
