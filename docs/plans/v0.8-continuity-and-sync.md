# v0.8 — Continuity and sync

Status: Proposed

## Context

By the end of v0.7, Allbert has a hardened daemon, curated memory, at least one approval-capable non-REPL channel, and a real `Channel` trait. That is enough to make the assistant available from more than one surface, but not enough to make it feel continuous across them.

The likely user pain after channel expansion is not yet self-improvement. It is continuity:

- one person may reach Allbert from REPL, CLI, Telegram, and future channels;
- approvals and pending actions should survive outside the surface that created them;
- sessions should be resumable across surfaces without losing operator legibility;
- memory and session artifacts need a clear sync posture instead of an accidental one.

v0.8 addresses that gap directly. It is still a local-first release, not a hosted platform or a network-addressable daemon. The focus is continuity, approval UX, and explicit sync semantics for a personal assistant used from more than one place.

## Goal

Deliver a continuity layer that makes Allbert feel like one assistant across channels and devices:

1. cross-channel identity mapping for one user's allowed identities;
2. durable session routing and reattachment across channels;
3. an operator-visible approval inbox / pending-action surface outside the REPL;
4. a clearly documented sync posture for memory, session, and approval artifacts;
5. early operator-surface groundwork for future native/web UI without turning the daemon into a hosted app.

## Sketch

### Identity mapping

- Introduce a stable local user identity record that maps:
  - local REPL / CLI operator identity;
  - allowlisted Telegram sender ids;
  - future channel-local identifiers.
- Channel-specific sender ids remain part of the trust model, but they resolve to one Allbert profile rather than silently creating unrelated personas.
- Identity mapping stays explicit and operator-editable. No fuzzy merge based on display name or phone number.

### Durable session routing

- Sessions gain a continuity layer above raw channel attachments:
  - one session can be resumed from another allowed surface;
  - channel-local threads/messages point to the same logical session when policy allows;
  - resumable session metadata is inspectable from operator surfaces.
- Session continuity remains bounded and local-user scoped:
  - no multi-user shared sessions;
  - no network-exposed session broker;
  - no hidden cross-account merges.

### Approval inbox and pending-action UX

- Pending approvals become durable first-class artifacts rather than transient channel prompts.
- v0.7's boundary stays intact here:
  - v0.7 already lets operators inspect pending approvals from REPL / CLI
  - v0.7 still resolves async approvals only on the originating async channel
  - v0.8 is where cross-surface approval resolution begins
- Operators can inspect and resolve pending approvals from:
  - REPL / CLI
  - the originating async channel
  - future richer operator surfaces
- The inbox is the canonical view of:
  - pending confirmations
  - cost-cap overrides awaiting operator action
  - deferred job or channel approvals
- Approval history remains auditable and local-first.

### Sync posture

v0.8 must make sync explicit instead of accidental.

- Minimum release bar:
  - define which artifacts are continuity-bearing (`memory/`, `sessions/`, approval inbox state, identity map);
  - define which are derived and rebuildable (`index/`, caches, channel-local render state);
  - document the supported local-only posture clearly if automatic multi-device sync is still deferred.
- Preferred v0.8 direction:
  - keep the daemon local-only;
  - make continuity-bearing artifacts sync-friendly and stable enough for explicit export/import or user-managed filesystem sync;
  - do not ship a hosted sync backend in v0.8.

### Operator surfaces

- Add a continuity-oriented operator surface that makes "what is waiting for me?" legible without dropping into raw filesystem inspection.
- This is groundwork for future native/web UI, but v0.8 itself does not require a full GUI.
- The CLI and REPL remain first-class operator paths.

## Expected ADRs

- Stable identity mapping for one local user across multiple channels.
- Pending approvals are durable inbox items, not transient channel-only state.
- Continuity-bearing artifacts vs derived artifacts for sync purposes.
- Local-only continuity and explicit sync posture before any hosted sync service.
- Session-routing rules across channels for one local user.

## Dependencies

- v0.6 foundation hardening — session durability, cost caps, and memory verification are prerequisites.
- v0.7 channel expansion — there must be at least one real non-REPL channel before cross-channel continuity is meaningful, but trust and continuity remain channel-local there. v0.8 is where identity mapping, cross-channel session routing, and cross-surface approval resolution begin.
- v0.5 curated memory — sync posture must respect markdown-ground-truth memory and rebuildable indices.

## Out of scope for v0.8

- Network-addressable daemon.
- Hosted multi-user sync service.
- Collaborative multi-user sessions.
- Full web UI or mobile app.
- Self-improvement and embedded scripting (moved to v0.9).

## Remaining planning questions

- Is the first v0.8 sync posture local-only plus explicit export/import, or does user-managed filesystem sync count as supported from day one?
- How much session history should travel with cross-channel continuity versus staying surface-local?
- Do approvals always follow the session, or can they be resolved globally from the inbox regardless of originating session?
- Which artifacts need stable ids for sync conflict handling?
