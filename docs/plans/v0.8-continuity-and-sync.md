# v0.8 — Continuity and sync

Status: Shipped

## Context

By the end of v0.7, Allbert has a hardened daemon, curated memory, at least one approval-capable non-REPL channel, and a real `Channel` trait. That is enough to make the assistant available from more than one surface, but not enough to make it feel continuous across them.

The likely user pain after channel expansion is not yet self-improvement. It is continuity:

- one person may reach Allbert from REPL, CLI, Telegram, and future channels;
- approvals and pending actions should survive outside the surface that created them;
- sessions should be resumable across surfaces without losing operator legibility;
- memory and session artifacts need a clear sync posture instead of an accidental one.

v0.8 addresses that gap directly. It is still a local-first release, not a hosted platform or a network-addressable daemon. The focus is continuity, approval UX, and explicit sync semantics for a personal assistant used from more than one place.

## Goal

Deliver a continuity layer that makes Allbert feel like one assistant across channels and devices:

1. cross-channel identity mapping for one user's allowed identities;
2. durable session routing and reattachment across channels;
3. an operator-visible approval inbox / pending-action surface outside the REPL;
4. a clearly documented sync posture for memory, session, and approval artifacts;
5. early operator-surface groundwork for future native/web UI without turning the daemon into a hosted app.

Implementation note: before coding starts, flip ADRs 0058 through 0062 from `Proposed` to `Accepted` as the v0.8 spec freeze.

## Release assessment

A closeout review on 2026-04-23 reopened the plan after earlier status claims proved overstated. The closeout pass completed the same day: M3, M4, M6, M10, and M11 are now closed, and this document remains the authoritative record of the shipped v0.8 surface.

- Complete: M1 through M11.
- The former M10/M11 checklist sidecars were absorbed into this document during closeout and then deleted.

## Milestones

### M1 — Local user identity record (ADR 0058)

Status: Complete

- Seed `~/.allbert/identity/user.md` on first daemon start after v0.8 upgrade. Template populates `id: usr_<ulid>`, `name: "primary"`, a single `channels` entry for the local REPL sender, and an empty body.
- Implement the frontmatter schema validator. Errors are operator-readable, not stack traces.
- CLI:
  - `allbert-cli identity show`
  - `allbert-cli identity add-channel <kind> <sender>`
  - `allbert-cli identity remove-channel <kind> <sender>`
  - `allbert-cli identity rename <new-name>`
  - all mutations write atomically (fsync+rename).
- Overlay with v0.7 per-channel allowlists. Document precedence: when a sender is in both, `user.md` wins for routing but the per-channel allowlist still admits the turn. The CLI warns when it detects inconsistency between the two surfaces.
- First-boot migration path: kernel scans any pre-existing channel allowlists (e.g. the v0.7 Telegram allowlist) and prompts the operator (CLI hint on first `identity show`) to promote senders into `user.md` if they want cross-channel continuity.

### M2 — Session routing by identity + cross-channel resume (ADR 0059)

Status: Complete

- `sessions/<sid>/meta.json` gains `identity_id: usr_<ulid>`. Old journals without the field load under a sentinel identity; first subsequent mutation writes the correct value.
- Inbound routing order: sender → identity (via `user.md`) → MRU session for that identity across all channels → reattach if younger than `daemon.session_max_age_days`, else create new session tagged with the identity.
- `turns.md` entries gain a `channel:` field in their per-turn header so one journal can hold a cross-channel conversation legibly.
- Config key `sessions.cross_channel_routing = inherit | scoped` (default `inherit`). `scoped` preserves v0.7 per-channel-sender routing for operators who prefer per-channel silos.
- Concurrent-inbound state machine: last-speaker-wins at turn boundary, in-flight turns complete at their originating channel, queued inbound becomes its own turn. Exclusive-attach guard arbitrates exact-tie races by timestamp then channel priority.
- Fallback: senders not mapped to any identity route per v0.7 per-channel-sender rules (no identity attachment, no cross-channel continuity, but the turn still runs).

### M3 — Approval inbox + ADR 0056 amendment (ADR 0060)

Status: Complete

Shipped outcome:

- `meta.json.pending_approval` (scalar) becomes `pending_approvals` (list). First-load migration upgrades existing journals in place.
- Retire the `prior-approval-pending` refusal in the policy gate path.
- In-memory inbox keyed by identity:
  - enumerate from `sessions/*/approvals/*.md` on daemon start;
  - maintain on approval create/resolve/expire;
  - rebuild is always valid (inbox never owns data).
- CLI:
  - `allbert-cli inbox list [--identity <id>] [--kind <kind>] [--include-resolved] [--json]`
  - `allbert-cli inbox show <aid>`
  - `allbert-cli inbox accept <aid> [--reason <text>]`
  - `allbert-cli inbox reject <aid> [--reason <text>]`
  - v0.7's `allbert-cli approvals list | show` continues to work for per-session inspection.
- Identity-scoped resolution: any surface belonging to the approval's identity can resolve. Channel adapters look up by id; channel of origin no longer constrains resolver. Approver identity (channel + sender) is recorded on the resolution entry for audit.
- New approval kinds added to the shared file layout, all with a `kind:` frontmatter field:
  - `tool-approval` (the existing ADR 0056 kind);
  - `cost-cap-override` — written when an async-channel turn refuses under `limits.daily_usd_cap` instead of silent failure;
  - `job-approval` — written when a scheduled job (ADR 0015) hits an interactive gate, rather than only logging `fail-closed`. Resolving flips the job to a retriable outcome.
- `channels.approval_inbox_retention_days` ships with default `30`.
- ADR 0056 gains the amendment banner noting these lifts.
- Bootstrap prompt guidance (`AGENTS.md`, `TOOLS.md`) gets a line: agents should avoid triggering multiple gates in quick succession. The kernel no longer forbids it, but the UX is still nicer without.

### M4 — Sync artifact catalog + daemon lockfile + atomic-write audit (ADR 0061)

Status: Complete

- Codify the artifact catalog in kernel code and in `docs/operator/continuity.md` — four categories (continuity-bearing, derived, sensitive, device-scoped) with the exact path list.
- `~/.allbert/daemon.lock` written on daemon start with `{pid, host, started_at}`. Lifecycle:
  - no lock → write and proceed;
  - lock exists, same host, pid alive → refuse with a clear error;
  - lock exists, same host, pid dead → stale takeover with audit entry;
  - lock exists, different host → refuse unless `--force`; `--force` logs a loud warning and flags a reconcile on next start.
  - graceful shutdown removes the lock; crashes leave it behind.
- `allbert-cli daemon status` gains continuity diagnostics:
  - lockfile owner (`pid`, `host`, `started_at`) when present;
  - the configured `model.api_key_env` name;
  - whether that env var is visible to the running daemon process.
  This closes the v0.7 operator-debugging gap where the shell could have an API key but a long-lived daemon might still be missing it.
- Atomic-write audit: continuity-bearing writes now route through a shared fsync-and-rename helper, and the daemon integration suite includes a guardrail test that fails on newly reintroduced raw `fs::write` call sites in those modules.

### M5 — Profile export / import (ADR 0061)

Status: Complete

- `allbert-cli profile export <path.tgz> [--include-secrets] [--identity <id>]`:
  - gzipped tarball of all continuity-bearing paths plus `.allbert-manifest.json`;
  - manifest includes `version`, `exported_at`, `exported_from_host`, `identity_id`, counts by artifact type, and an explicit `excluded` list;
  - safe against a live daemon: read-lock on the profile, snapshot-copy strategy;
  - `--include-secrets` is the documented opt-in; default excludes `secrets/`.
- `allbert-cli profile import <path.tgz> [--overlay | --replace] [--yes]`:
  - `--overlay` (default): per-file mtime compare, tarball wins iff newer, conflicts logged and local file preserved;
  - `--replace`: destructive; requires `--yes` and a clean lockfile (no live daemon);
  - post-import auto-run: the equivalent of `memory rebuild-index --force` plus a `memory verify`-style sanity check (counts vs what was imported).
- Explicit test: export → wipe profile → import → open a session → memory and approvals behave as they did.

### M6 — HEARTBEAT.md bootstrap + cadence parsing (ADR 0062)

Status: Complete

- Seed `~/.allbert/HEARTBEAT.md` from a template on first boot after v0.8 upgrade. Template ships with:
  - `version: 1`;
  - `timezone` populated from the system setting, fallback `UTC`;
  - `primary_channel: <first non-REPL channel if any, else repl>`;
  - `quiet_hours: ["22:00-07:00"]` (operator edits at will);
  - all `check_ins.*` disabled by default;
  - `inbox_nag: { enabled: true, cadence: daily, time: "09:00", channel: <primary_channel> }`.
- Frontmatter validator: IANA timezone names, `HH:MM-HH:MM` ranges, known channel kinds. Validation warnings surface through `allbert-cli heartbeat show`, never silent.
- Snapshot `HEARTBEAT.md` into the bootstrap bundle at start of each user turn, under the same dedicated bootstrap budget as `SOUL.md` et al. (ADR 0010).
- Budget semantics are fixed in v0.8: `HEARTBEAT.md` uses the existing bootstrap limits (`limits.max_bootstrap_file_bytes` per file and `limits.max_prompt_bootstrap_bytes` aggregate). v0.8 does not add a separate sub-budget knob for heartbeat context.

- Kernel consultation points:
  - push nudges (unsolicited) respect `quiet_hours`; inside the range, queue until the window ends;
  - scheduled jobs (`daily-brief`, `weekly-review`, and any future proactive job) execute only if `check_ins.<job>.enabled = true`;
  - inbox nag renders a one-line pending-items summary from the ADR 0060 inbox view and sends to the configured `channel` at `time` on `cadence`;
  - fail-silent: missing or malformed `HEARTBEAT.md` → no push, no proactive check-ins, no inbox nag.
- CLI:
  - `allbert-cli heartbeat show` (renders file + validation warnings);
  - `allbert-cli heartbeat edit` (opens in `$EDITOR`);
  - `allbert-cli heartbeat suggest [--channel <kind>]` (wizard proposes defaults based on current identity and registered channels; writes a proposal to a scratch path for operator review).
- v0.6 M5's `daily-brief` and `weekly-review` are unblocked but remain default-off — `HEARTBEAT.md` is the opt-in surface, not the flip.

### M7 — Session lifecycle CLI

Status: Complete

- Uniform vocabulary with the inbox command family (list/show/verb):
  - `allbert-cli sessions list [--identity <id>] [--channel <kind>] [--limit N] [--json]`;
  - `allbert-cli sessions show <sid>` — render journal (markdown, optionally paged);
  - `allbert-cli sessions resume <sid>` — reattach REPL to a specific session (overrides MRU);
  - `allbert-cli sessions forget <sid>` — archive (move to `.archive/`); approvals travel with the session per ADR 0060 retention rules.
- List output surfaces identity, channel(s) contributing turns, last-activity timestamp, pending-approval count.

### M8 — REPL idle-time inbox summary

Status: Complete

- On REPL attach / reattach, if the identity has pending inbox items, render a one-line summary above the prompt:
  - `3 pending approvals, 1 cost-cap override; see allbert-cli inbox list`.
- Toggle: `repl.show_inbox_on_attach = true` (default). Operators who want a silent REPL can disable.
- Does not modify the running session state — pure read-side UX.

### M9 — Per-device cost cap posture documented (doc-only)

Status: Complete

- `docs/operator/cost-caps.md` (new) calls out the ADR 0061 decision: `limits.daily_usd_cap` is per-device. Multi-device operators see an effective aggregate of ≈N × cap on N devices.
- Explicit non-goal statement: cross-device aggregation is out of scope through v0.11. Future redesign would need either a central counter or a file-sync-safe CRDT, both of which depend on a hosted/sync backend not planned before a later release.

### M10 — End-to-end cross-channel regression test

Status: Complete

Shipped outcome:

- The daemon integration suite now covers the shared continuity surfaces at the daemon/process boundary, not only through CLI unit tests.
- Existing profile import/export unit tests remain in place, but they are now complemented by daemon-level coverage for inbox resolution, async cost-cap override flow, job approvals, heartbeat gating, and the continuity-bearing write guardrail.

Target scenarios:

- Integration suite under `crates/allbert-daemon/tests/` (or equivalent):
  1. REPL turn → Telegram turn (same identity) → verify single journal, correct per-turn channel attribution, reattachment logic.
  2. Concurrent REPL + Telegram inbound → verify last-speaker-wins, queued-message ordering, exclusive-attach guard.
  3. Approval from Telegram, resolve from REPL/CLI → verify status update, audit trail, and agent resume.
  4. Profile export → wipe → import (overlay) → memory and approvals behave as before.
  5. Profile export → wipe → import (replace) with `--yes` → clean state, no stale lock.
- Text turns and image attachments are covered in v0.8's shared-journal model as session-scoped artifact references. Voice notes and richer media continuity remain out of scope through v0.8 and are explicitly deferred to v0.11 or later.

Fixture posture:

- Run against the real daemon binary over IPC and filesystem paths.
- Use provider-free scripted LLM responses (same pattern as existing daemon integration tests).
- Use a temporary `ALLBERT_HOME` per test process.
- Keep Telegram-path coverage fixture-based and deterministic (no network bot dependency).

Release evidence for M10:

- New integration tests run under default `cargo test` for `allbert-daemon`.
- Assertions cover both happy-path behavior and at least one failure-mode per scenario.
- No M10 test depends on live providers, secret material, or external network.

## Dependencies

- v0.6 foundation hardening — session durability, cost caps, and memory verification are prerequisites.
- v0.7 channel expansion — there must be at least one real non-REPL channel before cross-channel continuity is meaningful, but trust and continuity remain channel-local there. v0.8 is where identity mapping, cross-channel session routing, and cross-surface approval resolution begin.
- v0.5 curated memory — sync posture must respect markdown-ground-truth memory and rebuildable indices.

## Out of scope for v0.8

- Network-addressable daemon.
- Hosted multi-user sync service.
- Collaborative multi-user sessions.
- Full web UI or mobile app.
- Provider expansion and local-first model defaults (moved to v0.10).
- Self-improvement and embedded scripting (moved to v0.11).
