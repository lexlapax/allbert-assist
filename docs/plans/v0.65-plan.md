# Allbert v0.65 Local Knowledge: Files, Notes, And Agent Memory

Request flow: `docs/plans/v0.65-request-flow.md`
Design: `docs/design/local-knowledge-path.md`
ADR: **no new authority** class, source, or floor change. v0.65 does add
first-class product surfaces, so it records a **product-experience/IA amendment to
ADR 0077** (the two new `workspace:notes` and `workspace:memory` destinations and
the config-free notes-root connect affordance) rather than a new authority ADR. It
consumes the existing native notes/files plugin contract, the memory review +
active-memory substrate, Security Central, Settings Central, and the v0.64
first-run path unchanged.

Status: planned; implementation-readiness deepened 2026-07-10 against the shipped
code (Current Code State + Code Seams below). Inserted before the product RC because
local files, notes, and reviewed agent memory are the 1.0 launch-critical integration
path. **Scope: Full** — v0.65 builds the operator-facing surfaces (notes-root connect
affordance, `workspace:notes`, an interactive `workspace:memory` review panel,
launch-path prompts) on top of the already-built engine, not only documentation.

Source: post-v0.63 product-readiness review and operator direction. For the 1.0
launch, remote channels, MCP, browser research, and public protocol surfaces
must continue to work because they are implemented capabilities, but they are not
the first-run product path. The launch path is local files/notes plus reviewed
agent memory.

## Purpose

Turn Allbert's local knowledge substrate into an obvious product workflow:
connect a local notes/files root, search or read a note, confirm a safe write,
review what Allbert may remember, and recall that reviewed memory in a later
chat. v0.65 does not add a new authority class; it makes the existing
notes/files, Resource Access, and memory-review paths understandable and
testable as the default first useful assistant workflow.

**Already built (v0.65 documents + wires, does not re-implement):**

- **Notes/files native action plugin** (`plugins/allbert.notes_files/`), NOT an MCP
  server. `search_notes` + `read_note` are `permission: :read_only,
  confirmation: :not_required`; `write_note` is `permission: :notes_file_write,
  confirmation: :required, resumable?: true` with a real durable-confirmation resume
  (`.../actions/write_note.ex:4-10,104-148,174-177`). Shipped-module map:
  `plugin/discovery.ex:17`.
- **Notes root setting** `apps.notes_files.notes_root` (default `<ALLBERT_HOME>/notes`),
  writable/safe-write (`plugins/allbert.notes_files/.../settings_fragment.ex:11-15`,
  `safe_write_keys` at `:34-46`), read at `notes.ex:28-37`.
- **Non-writable `:notes_files` memory namespace**, enforced — a write returns
  `{:error, {:memory_namespace_not_writable, …}}` (`app.ex:37-46`, `memory.ex:770-781`).
- **Memory review loop** as registered actions: `ReviewMemoryEntry`
  (`:memory_write`), `DeleteMemoryEntry` (`:memory_write, confirmation: :required`,
  archives), prune/promote (`confirmation: :required`); statuses
  `:unreviewed/:kept/:flagged/:prune_nominated` (`memory/entry.ex:29`).
- **Recall is real end-to-end.** `DirectAnswer` retrieves active memory
  (`direct_answer.ex:105,206`) and injects it into the live LLM prompt
  (`req_llm_answerer.ex:123`); only `:kept` entries are retrieved
  (`active_memory.ex:122`).
- **Auto-promotion guard enforced.** New entries default `:unreviewed`
  (`memory.ex:411`); only `:kept` recalls; the sole transition is a permissioned human
  review action. An agent (`AppendMemory`) can create a candidate but cannot make it
  recallable.
- **CLI memory review** — `allbert admin memory {list,show,review,delete,retrieve,
  prune,search,…}` (`cli/areas/memory.ex`).

2. **Onboarding notes step.** `optional_connect` is a no-op placeholder ("Continue when

3. **First-class `workspace:notes` destination + nav.** Panel *surfaces* exist
   (`app.ex:49-65`) but there is no nav item / `workspace_nav_key` /

4. **Interactive web/TUI memory-review surface.** Only an unwired stub

5. **Local-knowledge starter prompts + empty states.** Persona `first_chat_prompts`
   plumbing exists (`personas.ex:106`) but there is no local-knowledge prompt set and no
   empty-state mechanism.
6. **`allbert admin memory status`** — the command does not exist (closest: `list`,

7. **`release.v065` gate** — entirely absent; needs a v0.64-shaped deterministic

## Locked Decisions

1. **Full scope.** v0.65 builds the operator-facing surfaces in items 1–7 above, not
   only documentation.
2. **Notes-root connect = a dedicated, config-free affordance** (an onboarding step +
   a `set_notes_root` settings action with validation), not the generic Settings Central
   key/value editor. The CLI companion is **`allbert admin notes set-root PATH`**,
   routed through the same action; `admin settings set apps.notes_files.notes_root PATH`
   remains a low-level fallback, not the product path.
3. **Memory review UI = a new `workspace:memory` destination** that wires the existing
   memory actions (`ReviewMemoryEntry`/`DeleteMemoryEntry`) through the Runner. It adds
   **no new authority** — it surfaces an already-permissioned loop. UI **Reject** maps
   to `review_memory_entry status=flagged`; Delete remains the confirmation-gated archive
   path.
4. **Resource Access = provenance, not the enforcement seam.** Notes reads/writes are
   enforced by `PermissionGate` + root-and-extension bounding (`inside_root?`,
   `@max_read_bytes`), with `ResourceURI`/`Scope` refs emitted for **audit/provenance**.
   v0.65 does **not** route file I/O through `Grants`; docs must describe the real seam
   (a grants-enforcement change is a separate future decision).
5. **`admin memory status` is built** (a read-only summary command), not rewritten away.
6. **No authority change.** No new permission class, floor, filesystem grant, or
   auto-promotion; the new surfaces route through the existing action/Security/Settings
   spine.

## Security Posture

No new local file authority, permission class, or confirmation floor. Notes reads and
writes stay scoped to the configured root by `PermissionGate` + **root-and-extension
path bounding** (`inside_root?`, extension allowlist, `@max_read_bytes` in
`notes.ex`); the emitted `ResourceURI`/`Scope` refs are **provenance/audit metadata**,
not the file-access enforcement seam (Locked Decision 4). `write_note` remains
confirmation-gated. The `:notes_files` memory namespace remains non-writable and never
auto-promotes. Agent memory is reviewed before it becomes durable/recallable
(`:unreviewed` default; `:kept`-only recall). The new `workspace:notes`/`workspace:memory`
surfaces and the connect affordance make these controls easier to use, but they do not
lower confirmation floors or infer authority from a plugin/app surface.

## Non-Goals

- No remote sync service.
- No automatic indexing of arbitrary home directories.
- No auto-promotion of notes, files, chat output, or plugin output into memory.
- No new file permission class or broad filesystem grant.
- No remote channel, MCP, browser, or public protocol expansion; v0.66 validates
  those implemented surfaces as regression blockers.
- No final product RC; v0.66 owns no-docs validation.

## Definition Of Done

v0.65 is done when:

- local files/notes/memory is the documented 1.0 primary launch integration, recorded
  in the design doc and the ADR 0077 amendment;
- the operator can configure a notes root **from onboarding, the post-first-chat
  QuickStart affordance, web/settings affordance, or `allbert admin notes set-root PATH`**
  without hand-editing config (the M2 affordance, not the generic key/value editor);
- `workspace:notes` and an interactive `workspace:memory` review panel are first-class
  destinations, and search/read/confirm-write and keep/reject-as-`:flagged`/delete work
  from the UI;
- note reads/writes are documented with their **real** enforcement (`PermissionGate` +
  root/extension bounding) and confirmation boundaries, and Resource Access refs are
  described as provenance, not the access gate;
- memory review is explicit user control (web + CLI), never automatic promotion;
- launch-path starter prompts and empty states guide a first useful local-knowledge chat;
- `admin memory status` reports review-status counts;
- no docs or surfaces imply broad filesystem authority, automatic memory promotion, or
  silent file writes, and no new authority class/floor is introduced;
- `mix allbert.test release.v065` (with real backing tests), docs, warning, and drift
  checks pass;
- v0.66 has current inputs for no-docs product RC validation.
