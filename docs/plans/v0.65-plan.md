# Allbert v0.65 Local Knowledge: Files, Notes, And Agent Memory

Request flow: `docs/plans/v0.65-request-flow.md`
ADR: no new authority ADR expected. This release consumes the existing
notes/files plugin contract, Resource Access, memory review substrate, Security
Central, Settings Central, and the v0.64 first-run path.

Status: planned (inserted before the product RC because local files, notes, and
agent memory are the 1.0 launch-critical integration path).

Source: post-v0.63 product-readiness review and operator direction. For the 1.0
launch, remote channels, MCP, browser research, and public protocol surfaces
must continue to work because they are implemented capabilities, but they are not
the first-run product path. The launch path is local files/notes plus reviewed
agent memory.

## Purpose

Turn Allbert's local knowledge substrate into an obvious product workflow:
connect a local notes/files root, search or read a note, confirm a safe write,
review what Allbert may remember, and recall that reviewed memory in a later
chat. v0.65 does not add a new authority class; it makes the existing
notes/files, Resource Access, and memory-review paths understandable and
testable as the default first useful assistant workflow.

## Expected Inputs

- v0.42 notes/files native reference plugin accepted: `search_notes`,
  `read_note`, `write_note`, Resource Access file refs, non-writable
  `:notes_files` memory namespace, and confirmation-gated writes.
- v0.21/v0.39b memory review and active-memory substrate accepted.
- v0.64 trusted install and non-developer first-run path accepted.
- Settings Central and Security Central remain the authority boundaries for app
  settings, local file roots, memory writes, and confirmations.

## Operator-Visible Win

A non-developer can point Allbert at a local notes folder, ask about those notes,
approve a note write when needed, decide what Allbert is allowed to remember,
and later see the reviewed memory improve a chat response. The product explains
where the files live, what Allbert can read or write, what becomes memory, and
how to inspect or undo it.

## Planned Interfaces

- Local knowledge setup path in onboarding or immediately after onboarding.
- Notes/files workspace panel and settings copy framed as the default local
  knowledge path, not only a reference plugin.
- Starter prompts for "ask about my notes", "summarize this note", "remember
  this after review", and "show what you remember."
- Confirmation copy for `write_note` and memory write/review.
- Memory review docs and UI copy centered on user intent, retention, and
  removal.
- `mix allbert.test release.v065` deterministic gate.

## Security Posture

No new local file authority. Reads and writes stay scoped to configured roots
through Resource Access. `write_note` remains confirmation-gated. The
`:notes_files` memory namespace remains non-writable and never auto-promotes.
Agent memory is reviewed before it becomes durable user memory. The product may
make these controls easier to understand, but it must not lower confirmation
floors or infer authority from a plugin/app surface.

## Non-Goals

- No remote sync service.
- No automatic indexing of arbitrary home directories.
- No auto-promotion of notes, files, chat output, or plugin output into memory.
- No new file permission class or broad filesystem grant.
- No remote channel, MCP, browser, or public protocol expansion; v0.66 validates
  those implemented surfaces as regression blockers.
- No final product RC; v0.66 owns no-docs validation.

## Milestones

- M1: Local knowledge product contract. Define the exact first-use loop:
  configure notes root -> search/read -> confirm write -> review memory ->
  recall later.
- M2: Notes/files operator docs. Reframe notes/files as the launch integration
  path while preserving its native plugin and Resource Access boundaries.
- M3: Memory review operator docs. Explain what can be remembered, what requires
  review, where memory lives, how to inspect it, and how to delete or reject it.
- M4: Starter prompts and empty states. Provide launch-path prompts without
  implying hidden authority or automatic memory promotion.
- M5: Confirmation and trust copy. Align note-write and memory-write wording
  across web, CLI/TUI, traces, and evidence.
- M6: Product acceptance script. Define the fresh-home path v0.66 will validate:
  connect notes root, ask a grounded question, confirm a write, review memory,
  and recall it.
- M7: Docs drift, `release.v065`, docs gate, warning gate, and v0.66 handoff.

## Documentation Deliverables

- Local files/notes/memory launch-path guide.
- Notes/files settings and Resource Access explanation for non-developers.
- Memory review guide focused on inspect, keep, reject, delete, and recall.
- Confirmation/trust copy contract for note writes and memory writes.
- v0.66 no-docs validation handoff for local knowledge.

## Definition Of Done

v0.65 is done when:

- local files/notes/memory is documented as the 1.0 primary launch integration;
- the operator can configure a notes root without hand-editing config;
- search/read/write note flows are documented with their Resource Access and
  confirmation boundaries;
- memory review is documented as explicit user control, not automatic assistant
  behavior;
- starter prompts and empty states guide a first useful local-knowledge chat;
- no docs imply broad filesystem authority, automatic memory promotion, or
  silent file writes;
- `mix allbert.test release.v065`, docs, warning, and drift checks pass;
- v0.66 has current inputs for no-docs product RC validation.
