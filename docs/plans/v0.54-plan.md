# Allbert v0.54 Intent Deepening

Request flow: `docs/plans/v0.54-request-flow.md`

ADRs:
- `docs/adr/0019-cross-surface-intent-enrichment.md` (consumed; amendment note
  for bounded multi-turn context).
- `docs/adr/0034-conversational-app-intent-handoff-and-clarification.md`
  (consumed; amendment note for generalized disambiguation + pending-clarification
  turn state).

Status: planned. NEW in the 2026-06-09 restructure; authored Phase B (research
R2). Sequenced **before** v0.55 Web UX because chat quality depends on intent.

## Purpose

Deepen the intent subsystem so a chat-primary conversation surface (v0.55) routes
reliably: carry bounded multi-turn context, strengthen classification, generalize
disambiguation beyond app descriptors, and make clarification a smooth multi-turn
loop — **without changing the authority model**. Intent stays proposal
infrastructure: model output is advisory, intent never grants permission, and
`active_app` remains ranking context only.

Today (per the survey): `Intent.Engine.decide/1` collects ranked candidates and
chooses `:direct_answer` / `:registry_action` / `:app_handoff` / `:clarify_intent`
/ `:continue_objective`; the bounded `Intent.Classifier` is advisory. The weak
spots for natural chat: **conversation context (`thread_context`, 12 recent
messages) is built by the runtime but never read by intent**; classification is
single-turn keyword/regex; clarification fires only on missing descriptor slots
or close descriptor margins (no general cross-kind disambiguation); and there is
**no persisted "awaiting clarification" turn state**, so a natural reply ("the
second one", "what about NVDA") cannot bind back to the offered candidates.

## Readiness Sweep
- **Authority fit.** Stays inside ADR 0019/0034: advisory-only model output, no
  permission grants, `active_app` ranking-only, runner still enforces matching
  `active_app`, no code execution during candidate collection.
- **Reuse fit.** `Conversations.recent_context/2` + the runtime-supplied
  `thread_context` already exist (just unused by intent); `Intent.Engine`/
  `Ranker`/`Classifier`/`Handoff` are the seams; ephemeral primitives render
  clarification (no new Surface catalog atom, per ADR 0034 §11).
- **Sequencing fit.** Lands before v0.55 so the chat-primary UI builds on it.

## Locked Decisions
| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Model output | Advisory only; the classifier re-ranks/selects within the already-collected, registry-validated candidate set (ADR 0019 §2, ADR 0034 §6/§10). | Authoritative model output is an ADR 0019 §violation requiring a new ADR. |
| 2 | Authority | Intent never grants permission; safety filter + permission floors unchanged; runner still requires matching `active_app`. | Breaks the Security Central boundary. |
| 3 | active_app | Remains ranking context only (ADR 0019 §3). | Using it to grant/bypass authority is an ADR 0019 §violation. |
| 4 | Multi-turn context | A **bounded, redacted** recent-turn summary feeds ranking + the classifier prompt — never raw thread history into the prompt (ADR 0019 §2). | Raw history into the model is an ADR 0019 §violation; unbounded context risks leakage. |
| 5 | Pending clarification | A short-lived (TTL) "awaiting clarification" turn state keyed to thread/user; the next turn resolves against the offered candidate set, **still registry-validated and proposal-only**. | Auto-executing a clarification reply without re-validation is an ADR 0019 §violation. |
| 6 | Settings | New `intent.*` keys (context window, disambiguation margin, pending-clarification TTL, multi-turn enable) via Settings Central / ADR 0031; behavior-changing flags default conservative/off (classifier-disabled-by-default precedent). | Default-on risks regressing existing routing. |
| 7 | Surfaces | No new Surface catalog atom; clarification/disambiguation reuse existing ephemeral primitives (ADR 0034 §11). | A new atom would need an ADR 0030 amendment. |

## Milestones

### M0: Preflight & Contracts
Confirm `thread_context` shape; define the bounded/redacted "conversation context
summary" contract and the "pending clarification" state record; draft `intent.*`
settings keys. No behavior change.

### M1: Multi-Turn Intent Context
Thread a bounded/redacted recent-turn summary from `thread_context` into
`Ranker.ranking_context/1` and the `Classifier` prompt (bounded per ADR 0019);
add follow-up/antecedent signals (prior selected action/app, prior clarification).
Settings-gated. Seams: `intent/ranker.ex`, `intent/classifier.ex`,
`intent/engine.ex`, `agents/intent_agent.ex`.
Acceptance: classifier prompt carries only a bounded/redacted summary (no raw
history; no raw prompt in traces); ranking uses prior-turn signal.

### M2: Classification Improvements
Stronger deterministic normalization/synonyms/follow-up handling; tighten
descriptor + keyword matching; classifier stays advisory re-ranker.
Seams: `intent/ranker.ex`, `intent/descriptor.ex`, `intent/classifier.ex`.

### M3: Generalized Disambiguation
When the top-N collected candidates (any kind) fall within
`intent.disambiguation_margin`, emit `:clarify_intent` listing the **actual
competing candidates** as options (each re-validated). Generalize
`descriptor_margin`/`descriptor_options` beyond descriptors.
Seams: `intent/engine.ex`, `intent/handoff.ex`, `workspace/emitters.ex`.

### M4: Clarification Loop / Multi-Turn UX
Persist short-lived pending-clarification state (TTL) keyed to thread/user; the
next turn resolves the reply against the offered candidate set (registry-validated,
proposal-only); progressive multi-slot fill.
Seams: pending-clarification store (plain GenServer or session scratchpad),
`agents/intent_agent.ex`, `conversations.ex`/`session.ex`.

### M5: Intent Traces
Extend the `intent_candidates` trace block with context-carry contribution,
disambiguation margins, and pending-clarification resolution — bounded/redacted,
no raw prompts.

### M6: Evals, Docs, Closeout
`:v054` eval rows + `release.v054` lane; ADR 0019/0034 amendment notes;
agent-context-map/security-hardening docs; version/CHANGELOG/roadmap closeout +
full release gate.

## Security Eval Rows (`:v054`)
- `intent-multiturn-followup-resolves-001` (turn-2 "yes" binds to offered set;
  still requires `active_app` at runner)
- `intent-pronoun-antecedent-001` ("what about NVDA" carries prior app, no new
  authority)
- `intent-generalized-disambiguation-001` (two close candidates → `:clarify_intent`
  listing both; no silent selection)
- `intent-clarify-reply-revalidated-001` (reply naming an unregistered action →
  fallback, no execution; ADR 0019 §2/§4)
- `intent-active-app-no-escalation-multiturn-001` (prior-turn app context does not
  auto-execute an app action in neutral context)
- `intent-context-bound-no-raw-history-001` (classifier prompt bounded/redacted;
  no raw prompt traced)
- `intent-slot-progressive-fill-001` (missing slots clarified, never guessed)
- `intent-pending-clarification-ttl-001` (stale reply re-classified fresh)
- `intent-pending-clarification-cross-user-isolation-001`

## Non-Goals
- No change to the Security Central / action-boundary authority model.
- No model-output-as-authority; no intent-driven permission grants.
- No `active_app`-as-authorization; no raw thread history into the model prompt.
- No code execution during candidate collection; no new Surface catalog atom.

## Definition Of Done
v0.54 is done when:
- bounded multi-turn context feeds ranking + the (advisory) classifier without
  raw history into the prompt or traces;
- general cross-kind disambiguation lists real competing candidates as
  clarification options, never silently selecting;
- a pending-clarification turn state lets the next reply resolve against the
  offered set, still registry-validated and proposal-only, with TTL + cross-user
  isolation;
- the authority model is provably unchanged (eval rows above pass);
- `:v054` evals + `release.v054` + the full release gate pass;
- ADR 0019/0034 amendment notes, roadmap, vision, and agent-context-map agree.
