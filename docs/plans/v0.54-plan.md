v0.54 is now sequenced **ahead of completing v0.53** as well as before v0.55:

## Purpose

Deepen the intent subsystem so every surface routes reliably — **without changing
the authority model**. Intent stays proposal infrastructure: model output is
advisory, intent never grants permission, the runner/permission/confirmation gates
are unchanged, and `active_app` remains ranking context only. v0.54 does this by:

- replacing the single-stage deterministic keyword/descriptor matcher with a
  **local-first two-stage intent router** (embedding prefilter → constrained LLM
  disambiguation → confidence gate) as the **default selector**, with the
  deterministic ladder kept as fast-path + offline fallback (ADR 0060);
- adding a **local text-embedding capability** and **router model tiers**
  (embedding + 7–8B local default + optional audited hosted escalation, ADR 0061);
- carrying **bounded, redacted multi-turn context** into ranking and the router
  prompt (never raw history; ADR 0019);
- generalizing **disambiguation** beyond app descriptors and making
  **clarification** a smooth, channel-answerable multi-turn loop with a TTL'd
  pending-clarification turn state (ADR 0034);
- **removing the app-handoff channel dead-end**, replacing it with a targeted
  clarify that re-enters and executes, so a `confirmation: :required` action —
  however routed — reaches the existing `ConfirmationCallback` approve/deny
  primitive;
- closing the **action/descriptor coverage** gap (write/create descriptors,
  pretraining-aligned names) that made the right capability invisible;
- adding a **golden-set intent eval** + the channel approval end-to-end regression
  to the release gate.

The original draft kept model output advisory **as a re-ranker only**. The
deepened plan promotes the LLM router to the **default selector within the same
registry-validated candidate set** — still advisory re: *authority* (the chosen
action runs only through the runner + permission + confirmation gates), still
proposal-only, still registry-validated. The shift is in *selection quality*, not
in the authority model.

### Today's weak spots (the survey, retained and extended)

`Intent.Engine.decide/1` collects ranked candidates and chooses `:direct_answer` /
`:registry_action` / `:app_handoff` / `:clarify_intent` / `:continue_objective`;
the bounded `Intent.Classifier` is advisory and **off by default**. The failures:

- **No LLM in the loop by default** — routing is deterministic keyword/regex; the
  classifier only re-ranks pre-collected candidates and is gated off.
- **Mis-route from descriptor-coverage gaps** — "create a note" matches the
  notes_files **`search_notes`** descriptor because there is **no `write_note`
  descriptor** and the bare synonym `"notes"` wins the single-token match.
- **App-handoff dead-end on channels** — on a neutral surface (`active_app` ∈
  `[nil, :allbert]`, i.e. every channel message), `descriptor_decision_attrs/5`
  emits `:app_handoff`; `run_validated_route/4` returns inert text with
  `approval_handoff: nil`; no confirmation is created; the only "accept"
  affordance is a workspace-canvas fragment never wired to channels. The action
  never runs and approve/deny buttons never render.
- **Conversation context (`thread_context`, 12 recent messages) is built by the
  runtime but never read by intent**; classification is single-turn; clarification
  fires only on missing descriptor slots / close descriptor margins; and there is
  **no persisted "awaiting clarification" turn state**, so a natural reply ("the
  second one", "what about NVDA") cannot bind back to the offered candidates.

## Readiness Sweep

- **Authority fit.** Stays inside ADR 0019/0034 and ADR 0060's approval-gate
  separation: advisory-only model output, no permission grants, `active_app`
  ranking-only, the runner still enforces matching `active_app`, no code execution
  during candidate/ shortlist collection. The router selects *which* action; the
  `confirmation`/`ConfirmationCallback` gate decides *whether* an effectful action
  runs; `:channel_message_inbound` (ADR 0056) and Security Central are unchanged.
- **Reuse fit.** `Intent.Classifier.Behaviour` (the extension pattern the
  `Intent.Router` mirrors); `ReqLLM.generate_object` constrained output;
  `Models.for/2` + `model_profiles` + `model_routing`; `Conversations.recent_context/2`
  + the runtime `thread_context` (already built, unused by intent);
  `Candidate`/`Ranker`/`Handoff`/`ApprovalHandoff`; the `ConfirmationCallback`
  channel approval primitive; ephemeral primitives render clarification (no new
  Surface catalog atom, ADR 0034 §11); the ADR 0047 doctor envelope; the `:v05x`
  eval harness.
- **Net-new.** A local **embedding** path (none exists today), the
  `Intent.Router` behaviour + two-stage default impl, an in-memory utterance
  index, the confidence gate, the targeted-clarify primitive replacing the
  app-handoff dead-end, and the golden-set intent eval.
- **Sequencing fit.** Resequenced ahead of completing v0.53 (channel approval
  depends on it) and before v0.55 (chat quality depends on intent).

## Locked Decisions

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Model output | Advisory re: authority; the router **selects within the already-collected, registry-validated candidate set** and the selection runs only through the runner + permission + confirmation gates (ADR 0019 §2, ADR 0034 §6/§10, ADR 0060). | Authoritative model output is an ADR 0019 violation needing a new ADR. |
| 2 | Authority | Intent never grants permission; safety filter + permission floors unchanged; runner still requires matching `active_app`; approval gate independent of routing (ADR 0060). | Breaks the Security Central boundary. |
| 3 | active_app | Remains ranking context only (ADR 0019 §3). | Using it to grant/bypass authority is an ADR 0019 violation. |
| 4 | Multi-turn context | A **bounded, redacted** recent-turn summary feeds ranking + the router/classifier prompt — never raw thread history (ADR 0019 §2). | Raw history into the model is an ADR 0019 violation; unbounded context risks leakage. |
| 5 | Pending clarification | A short-lived (TTL) "awaiting clarification" turn state keyed to thread/user; the next turn resolves against the offered candidate set, **still registry-validated and proposal-only**. | Auto-executing a clarification reply without re-validation is an ADR 0019 violation. |
| 6 | Router strategy + default | Two-stage local router is the **default** (`intent.router_strategy = :two_stage_local`); deterministic ladder is the fast-path + offline fallback (ADR 0060). **This supersedes the original "behavior-changing flags default off" stance** — the channel workflow requires it and the `:v054` eval + deterministic fallback are the safety net. Operators may pin `:deterministic`. | Default-off would leave the channel approval workflow broken; an ungated default-on would risk routing regressions (mitigated by the eval gate). |
| 7 | Model tiers | Local embedding (Stage 1) + **7–8B local** Stage-2 default + **optional, audited, off-by-default** hosted escalation for the low-confidence tail (ADR 0061). | A 3B Stage-2 default is unreliable at multi-slot; a default-on hosted tier breaks local-first/offline. |
| 8 | Embeddings | **Local-only** Ollama embeddings, in-memory cosine index from the registry, no external vector DB, no egress (ADR 0061). | An external vector DB / hosted embeddings breaks local-first and adds a dependency. |
| 9 | No dead-ends | The app-handoff `:approval_card`-only path is removed for channel surfaces and replaced everywhere by a targeted, channel-answerable clarify (ADR 0016 `:button`/`:typed_command`/`:list`, ADR 0034 §11). | A channel dead-end leaves the workflow broken. |
| 10 | Action-surface hygiene | Pretraining-aligned action names + descriptor coverage for write/create actions + tighter matching (no single-token wins). | The mis-route persists even with a perfect router. |
| 11 | Surfaces | No new Surface catalog atom; clarification/disambiguation reuse existing ephemeral primitives (ADR 0034 §11). | A new atom would need an ADR 0030 amendment. |
| 12 | Settings | New `intent.router_*` + the original `intent.*` (context window, disambiguation margin, pending-clarification TTL, multi-turn enable) via Settings Central / ADR 0031; `@safe_write_keys` updated. | Off-schema keys are rejected by `validate_key_value`. |

## Router Architecture (ADR 0060)

```
free text (any surface; channel requests have no active_app)
 → [fast-path] deterministic ladder for explicit/unambiguous routes  (kept)
 → Stage 1  EMBEDDING PREFILTER (local Ollama embeddings, in-memory cosine index)
        rank actions by cos(utterance + bounded multi-turn summary, per-action utterances) → top-K + margin
 → Stage 2  LLM DISAMBIGUATOR over the shortlist (ReqLLM.generate_object, enum-constrained)
        → {:execute, action, slots, confidence} | {:clarify, shortlist, question}
          | {:answer} | {:none}      (reason free-form, answer label constrained)
 → CONFIDENCE GATE (avg-token logprob where available + Stage-1 margin; intent.router_min_confidence)
        act · ask ONE targeted clarifying question (TTL turn-state) · abstain
 → [existing] Runner.run → confirmation:required → approval_handoff → ConfirmationCallback (approve/deny)
```

Inserted at `Engine.decide_without_explicit_route/1` (the single fork every
surface passes through after candidates are collected and ranked); its outcome is
consumed by `IntentAgent.run_validated_route/4`, which today unconditionally
dead-ends `:app_handoff`.

| Key | Default | Purpose |
|---|---|---|

| `intent.router_top_k` | 5 | Stage 1 shortlist size |
| `intent.router_min_confidence` | per-model (≈0.6) | act vs clarify vs abstain |

`intent.context_window`, `intent.disambiguation_margin`,
`intent.pending_clarification_ttl_ms`. All follow the existing `intent.*` schema

`intent.model_assist_enabled`/`descriptors_enabled`/`handoff_*` keys remain for
the deterministic fast-path + fallback and are reconciled in M5.

### M5: Engine Wiring + Dead-End Removal + Generalized Disambiguation + Clarification Loop
Status: **implemented (2026-06-16)**; this milestone makes the end-to-end workflow work.

Implementation tasks:
- Consume `Router.Outcome` at `Engine.decide_without_explicit_route/1`: `:execute`
  → direct execution decision (carrying `slots`); `:clarify` → targeted-clarify
  decision listing the **real competing candidates** (generalizing

- In `IntentAgent.run_validated_route/4`, route `:execute` through
  `run_route`/`Runner.run` so a `confirmation: :required` action creates a

  dead-end.**
- Render `:clarify` as a channel-answerable targeted question (ADR 0016
  primitives); persist short-lived (TTL) pending-clarification state keyed to
  thread/user; the next turn resolves the reply against the offered candidate set
  (registry-validated, proposal-only); progressive multi-slot fill. Preserve the
  workspace `:approval_card` only where it already worked (web canvas), now fed by
  clarify.
- Flip the default: `intent.router_strategy = :two_stage_local`; keep the
  deterministic ladder as fast-path + offline fallback.

Acceptance: a channel message mapping to a `confirmation: :required` action →
executes → **approve/deny buttons render** → tap resolves (end-to-end, stubs);
ambiguous request → targeted clarify listing real candidates → reply binds and
executes (re-validated); explicit commands take the deterministic fast-path;
model-down → deterministic fallback; no app-handoff dead-end on channels;
authority unchanged.

Automated tests: end-to-end intent→execute→approval_handoff (fake model + fake
provider); clarify→reply→execute (TTL, cross-user isolation); generalized
disambiguation; fast-path + fallback; lanes `db_serial` + `global_process_serial`.

Implementation result (M5, 2026-06-16): wired the router into
`IntentAgent.run_validated_route/4`. A turn first resolves any pending
clarification (`ClarifyResolver` over the offered, registry-validated options),
then routes via `Router.route` with a built `ConversationContext`:
- **`:execute`** → `run_action`/`Runner.run` (slots merged as existing-atom
  params only), reaching the `confirmation: :required` → `approval_handoff` →
  `ConfirmationCallback` channel approve/deny primitive — the keystone.

### M9: Comprehensive Intent Golden-Set, Descriptor Coverage & Router Optimization
Status: **planned** — queued after the v0.54 manual-validation punchlist. **Do not
execute until approved.** Authored 2026-06-16 from the manual-validation findings
+ a full zoom-out of the action surface.

**Why this milestone exists (findings).** Live local-model validation
(2026-06-16) did two things: it surfaced real ReqLLM↔Ollama integration defects
and an over-conservative gate (all fixed — Appendix B), and — more importantly — a
zoom-out of the action/intent surface showed the router universe is far too narrow
to call "intent routing" validated:

- **154 registered actions**, **42 agent-exposed**, but only **18 intent
  descriptors** feed the router's Stage-1 index (via
  `Extensions.Registry.registered_intent_descriptors/0`, gathered from each
  app/plugin's `intent_descriptors/0`).
- Those 18 are **notes / stocks / marketplace / workspace-panel-centric**. Whole
  domains a user expresses in natural language have **no routable descriptor** and
  therefore cannot route regardless of model quality: **send/draft an email**
  (only `open_mail_panel` exists), **create a calendar event** (only
  `open_calendar_panel`), **"remember that…" / recall memory**, **research a
  topic** (inert handoff only), **generate an image**, **write a plugin / author a
  skill**, **send to a channel / resume a thread**, **change a setting / switch
  model**, **MCP tool calls**, **objectives / workflows / plans**.
- Therefore "create a note" — the case we optimized during validation — is **1 of
  18 narrow cases**. Routing quality is gated **first by descriptor coverage**,
  then by model/threshold tuning. M9 fixes coverage, builds a comprehensive
  golden set across the real action taxonomy, and optimizes the router against
  *that*.

**Zoom-out / architecture constraint (respect registration).** Coverage is added
the canonical way — declare `intent_descriptors/0` in each app/plugin module
(gathered by `ExtensionsRegistry`, deduped by `{app_id, action_name}`); selection
stays in Stage 2; authority stays in Security Central + the confirmation gate. No
router-side special-casing, no bypass of the Actions Registry or Settings Central.

**Scope.**
1. **Descriptor coverage expansion.** Build a taxonomy of routable user verbs from
   the 42 agent-exposed actions; for each that a user would naturally speak,
   declare a descriptor (label, 3–6 example utterances, synonyms, required/optional
   slots + extractors). Explicitly mark infra/orchestration/doctor/registry-health
   actions as **intentionally non-routable** (internal exposure). Candidate
   domains: notes, email (send/draft/summarize), calendar (create/show), memory
   (remember/recall/search), research, browser, stocks, image-gen, channels
   (send/resume), settings/model, skills+plugin authoring (gen/dynamic), marketplace
   /packages, objectives/workflows/plans, voice, MCP tools, confirmations.
2. **Comprehensive intent golden-set** (the durable eval artifact, stored as
   fixtures consumed by both an offline-fakes harness and a live-local harness).
   Per 2026 eval guidance: **50–100 hand-crafted anchor cases** first, growing
   toward **≥500** via *validated* synthetic variation + later production-trace
   mining. Each case = `{utterance, context(active_app, thread), expected_outcome
   (execute<action,slots> | clarify<shortlist> | answer | none | defer),
   rationale}`. Categories: per-domain execute happy-paths; **cross-domain
   disambiguation** (e.g. "summarize my inbox" → `open_mail_panel` vs research;
   "analyze AAPL" → `run_analysis` vs answer); **slot extraction** (ticker, email
   recipient/subject, note title/body, calendar time); **negatives / out-of-scope**
   (→ `none`/`answer`); **ambiguous** (→ `clarify`); **multi-turn** follow-ups;
   **adversarial / authority-probing** (must never grant authority or bypass
   confirmation).
3. **Optimization harness + metric.** A replay bench (`mix allbert.intent bench` or
   a `:v055` lane) that runs the golden set against the live local router and
   reports **per-category accuracy, calibration (confidence vs correctness),
   clarify/none rates, latency, escalation rate**. Tunable knobs: descriptor text,
   Stage-1 `router_top_k`, `intent.disambiguation_margin`,
   `intent.router_min_confidence`, `@decisive_confidence`, the Stage-2 prompt,
   `router_local` model, escalation profile + triggers. Method: anchors first;
   **fixed train/holdout split** so thresholds are not overfit; LLM-as-judge only
   where exact-match is too brittle; a **CI replay subset (~30 cases, <5 min)** as
   a merge gate; the full set as a pre-tag gate.
4. **Acceptance.** Per-category execute accuracy ≥ target on holdout; calibration
   good (wrong picks carry low confidence so the gate/escalation catch them);
   warm-latency budget; **zero authority/security regressions**; doctor reports
   descriptor count/coverage; new eval rows added.

**Version-scoping (decision needed at kickoff).** This is a substantial, separable
effort. v0.54's keystone (no dead-end + working live local routing + US model +
escalation) is already met. **Recommendation: land M9 as v0.55**, keeping v0.54
scoped to the router mechanism + the validation fixes already shipped. Flagged for
operator decision before execution.

## Appendix A: Router model A/B benchmark (2026-06-16, for posterity)

Measured on the **live** router pipeline against local Ollama: real Stage-1
prefilter shortlist → Stage-2 `generate_object` with `openai_structured_output_mode:
:json_schema`, temperature 0.0, on a 6-case notes set (3 create → `write_note`,
2 search → `search_notes`, 1 ambiguous `note` → `__clarify__`). Accuracy is
exact-match on the expected outcome; latency is avg wall-clock per Stage-2 call
(first call includes cold model load).

| Model | Origin | Accuracy | Avg latency | Calibration / notes |
|---|---|---|---|---|
| qwen2.5:7b | Alibaba (CN) | 6/6 | 1.6s | Excellent (0.5 on ambiguous). Incumbent; not US-origin. |
| **gemma4:26b** | Google (US) | **6/6** | 6.4s | Excellent (0.8 ambiguous). Only US model at 6/6; 26B, ~3× slower. → **escalation tier**. |
| **llama3.1:8b** | Meta (US) | 5/6 | **2.1s** | Good; its 1 miss was a safe low-conf clarify. → **default `router_local`**. |
| gemma3n:e2b | Google (US) | 5/6 | 2.2s | OK; miss = `__none__`@0.3. |
| gemma4:e4b | Google (US) | 5/6 | 5.1s | Poor — confidently wrong (`__none__`@1.0); slow. |
| nemotron-3-nano:4b | NVIDIA (US) | 5/6 | 4.3s | **Hallucinated `write_quote`** (out-of-shortlist)@0.9 — disqualifying. |
| phi4-mini-reasoning | Microsoft (US) | 3/6 | 4.3s | Over-clarifies; reasoning interferes. |
| phi4-mini | Microsoft (US) | 1/6 | 2.0s | Clarifies everything — unusable as a router. |

**Decision:** default `router_local` = **llama3.1:8b** (US, fast, well-calibrated);
escalation `router_escalation_local` = **gemma4:26b** (US, 6/6, local → no egress),
enabled by default. Calibration matters more than raw accuracy for a
confidence-gated router: well-calibrated wrong answers carry low confidence so the
gate/escalation catch them; confidently-wrong models (gemma4:e4b) and
out-of-shortlist hallucinators (nemotron) are unsafe regardless of headline score.
This benchmark is the seed harness M9 generalizes across all domains.

## Appendix B: Manual-validation fixes (2026-06-16)

Found running the live punchlist (`docs/plans/v0.54-request-flow.md`) against a
real local model; all shipped on `main` before M9:

1. **Embedding probe failed (`endpoint=unavailable`).** ReqLLM 1.13 gates `embed/3`
   on registry capability metadata that local Ollama embedding models lack. Fix:
   the default embedder asserts the `embeddings` capability the profile already
   declares (ADR 0061) on the inline spec. → `nomic-embed-text` accepted, 768-dim.
2. **Stage-2 returned an empty object (router silently deferred → `direct_answer`).**
   ReqLLM's `:auto` structured-output mode chose OpenAI strict tool-calling for
   unregistered local models, which Ollama's `/v1` ignores. Fix: the disambiguator
   forces `openai_structured_output_mode: :json_schema` (Ollama-native; hosted
   OpenAI supports it too).
3. **Over-conservative gate (confident, correct picks clarified).** The Stage-1
   embedding margin is noisy and length-sensitive (e.g. "create a note titled v054
   with body hello world" → margin 0.007 even though write_note is top-1 and
   Stage 2 picks it @1.0). Fix: a tight margin no longer vetoes a **decisive**
   (≥ `@decisive_confidence` = 0.8) Stage-2 selection — Stage 2 is the selection
   authority (ADR 0060).
4. **Default router model → US-origin + local escalation.** `router_local`
   qwen2.5:7b → **llama3.1:8b**; added **`router_escalation_local`** (gemma4:26b),
   set as the default `intent.router_escalation_profile` (local, audited);
   `intent.router_model_timeout_ms` 4000 → **20000** to cover local cold-start.
   The escalation eval row was amended from "off by default" to "local by default
   (no egress)".
