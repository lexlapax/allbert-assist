## Purpose

v0.23 converges the two clearest existing state-machine components onto the
Jido substrate before v0.24 ships the objective runtime:

- `AllbertAssist.Confirmations.Store`
- `AllbertAssist.Jobs.Scheduler`

This is an architectural refactor, not a feature milestone. It should make the
runtime more internally consistent with the Allbert/Jido vision while keeping
the v0.07 confirmation workflow and v0.13 scheduled-job workflow unchanged to
operators.

The current code matters:

- `AllbertAssist.Confirmations.Store` is a plain file-backed module, not a
  `GenServer` and not SQLite-backed. ADR 0008 makes confirmation records
  durable action state under Allbert Home. v0.23 must preserve that storage
  model. The Jido.Agent layer becomes the state-machine/coordinator around
  those files; it does not migrate confirmations into SQLite.
- `AllbertAssist.Jobs.Scheduler` is a supervised `GenServer` that keeps no
  authoritative job queue in memory. Each tick reads due jobs from SQLite and
  claims runs durably. v0.23 must preserve that contract. The Jido.Agent layer
  owns scheduler lifecycle and tick commands; SQLite remains authoritative for
  jobs and job runs.

## Readiness Decision

v0.23 is ready for implementation when:

- v0.22 has passed operator acceptance and any documented v0.22 bridge/routing
  fixes are closed or explicitly deferred in docs.
- The project-direction decision is locked: v0.23 converts only
  Confirmations.Store and Jobs.Scheduler. It does not implement objectives.
- The pragmatic "Jido.Agent where it earns its keep" rule is present in
  `docs/plans/allbert-jido-vision.md`, `AGENTS.md`, `DEVELOPMENT.md`, and
  `docs/developer/agent-context-map.md`.
- Implementers have checked current Jido docs before coding. As of this pass,
  Context7 shows `use Jido.Agent`, `signal_routes`, `Jido.AgentServer`,
  action modules, module-level `cmd` helpers on agent structs, documented
  `on_before_cmd/2` and `on_after_cmd/3` hooks, and
  `Jido.Agent.Directive.schedule/2` / `Directive.cron/2` scheduling helpers.
  The plan should not use non-existent `defcommand` macros or assume
  undocumented lifecycle hooks.

## Architecture Readiness Audit

v0.23 is consistent with the current architecture if it preserves these
boundaries:

- ADR 0006: Security Central remains the permission, risk, redaction, and
  audit boundary. No permission classes, safety floors, risk tiers, or
  redaction obligations change.
- ADR 0007: runtime-facing, effectful, security-relevant, or observable work
  still enters through registered Jido actions and `Actions.Runner.run/3`.
  Converting an internal state holder to `Jido.Agent` does not create a second
  action boundary.
- ADR 0008: confirmation records stay durable under Allbert Home. The agent's
  in-memory state is a rebuildable projection and transition coordinator. It
  is not authoritative state.
- ADR 0014: local identity semantics are unchanged. `user_id`, `operator_id`,
  `thread_id`, and `session_id` fields in confirmation/job records preserve
  their existing meanings.
- ADR 0021: v0.24 objectives will consume confirmations/jobs after this
  convergence, but v0.23 does not add `objective_id`, `step_id`, objective
  signals, objective settings, or objective CLI commands.

No new ADR is required for v0.23. If implementation discovers that Jido's
current API forces a behavior change visible outside the internals, stop and
write or update an ADR before proceeding.

## Vision And Roadmap Alignment

The Allbert/Jido vision says:

- Use `Jido.Signal` for events.
- Use `Jido.Action` for capability boundaries.
- Use `Jido.Agent` where state machines, lifecycle hooks, or successor-agent
  stories earn their keep.
- Use plain `GenServer` or plain modules for storage wrappers where an agent
  abstraction adds ceremony without value.

v0.23 implements the clearest convergence case without overcorrecting. It does
not turn Settings, Trace, Memory storage IO, Session.Scratchpad,
Memory.Compiler, or Memory.Promotion into Jido.Agents.

Roadmap dependency order:

- v0.22 supplied the StockSage Python bridge.
- v0.23 converges Confirmations.Store and Jobs.Scheduler onto Jido.Agent.
- v0.24 adds Objective Runtime Foundation on top of this substrate.
- v0.25 adds native Jido trading agents consuming objective state from day
  one.

## Expected Inputs

Confirmations:

- `AllbertAssist.Confirmations` public facade delegates to
  `AllbertAssist.Confirmations.Store`.
- `AllbertAssist.Confirmations.Store` owns Allbert Home paths:
  `confirmations/pending`, `confirmations/resolved`, and
  `confirmations/audit`.
- `AllbertAssist.Confirmations.Record` validates durable record shape.
- `AllbertAssist.Actions.Confirmations.*` registered actions call the
  facade/store; CLI and LiveView use those actions, not private file writes.
- Settings keys under `confirmations.*` control TTL, display, and approval
  surfaces.

Jobs:

- `AllbertAssist.Jobs` context owns job creation, listing, pausing, resuming,
  blocking, run records, and due-job claims.
- `AllbertAssist.Jobs.Scheduler` is the supervised polling process.
- `AllbertAssist.Jobs.Runner` executes claimed/manual runs through
  `Runtime.submit_user_input/1` or `Actions.Runner.run/3`.
- `scheduled_jobs` and `scheduled_job_runs` SQLite tables remain unchanged.
- Settings keys under `jobs.*` control timezone, default state, schedule
  policy, and scheduler posture.

Jido:

- `Jido.Agent`, `Jido.Action`, `Jido.AgentServer`, `Jido.Signal`, and
  `Jido.Agent.Directive` are available.
- Implementation must verify exact function names/options against current
  docs/source before coding; do not rely on stale examples.

## Guiding Rules

1. **No operator-visible behavior change.** CLI output, LiveView behavior,
   trace sections, audit file content, signal payloads, settings, and error
   messages must remain byte-identical or trivially equivalent. Trivially
   equivalent means timestamps, process identifiers, or internal debug
   metadata may differ; operator-meaningful fields may not.

2. **No durable schema changes.** No Ecto migration. Confirmation files stay
   under Allbert Home. Job tables stay unchanged.

3. **No new permissions, actions, settings, or signals.** Existing registered
   actions remain the public surface. v0.23 may introduce private internal
   Jido.Action modules for agent commands, but those are not registered
   operator capabilities.

4. **State is rebuildable.** On start/restart, agent state is reconstructed
   from Allbert Home confirmation files or SQLite job rows. In-memory agent
   state is never the source of truth.

5. **Actions remain the authority boundary.** A Jido.Agent command may
   coordinate internal state, but all externally meaningful capability work
   still flows through registered actions, Security Central, confirmations,
   and the action runner.

6. **Use documented Jido APIs.** Prefer Jido.Action command modules,
   `signal_routes`, `Jido.AgentServer` calls, `on_before_cmd/2`,
   `on_after_cmd/3`, and documented directives such as
   `Jido.Agent.Directive.schedule/2`. Do not invent macros or lifecycle hooks.

7. **Keep storage helpers pure.** Path helpers, YAML codec helpers, record
   rendering, schedule parsing, and Ecto changesets remain plain helper
   modules/functions. Do not wrap everything in an agent just because this
   milestone exists.

## Implementation Shape

### Confirmations.Store

Goal: make `AllbertAssist.Confirmations.Store` a supervised Jido.Agent-backed
state machine while preserving its public API and Allbert Home file layout.

Public API must remain:

- `root/0`
- `pending_root/0`
- `resolved_root/0`
- `audit_root/0`
- `ensure_root!/0`
- `create/2`
- `read/1`
- `list/1`
- `resolve/4`
- `expire/1`
- `pending_path/1`
- `resolved_path/2`
- `audit_path/1`

Implementation direction:

- Keep path helpers and YAML/record helper logic as pure functions.
- Add Jido.Agent state with bounded, rebuildable projection fields such as:
  - `pending_ids`
  - `pending_by_target`
  - `last_rebuilt_at`
  - `last_sweep_at`
  - `last_command`
- Add private internal Jido.Action command modules, for example:
  - `AllbertAssist.Confirmations.Store.Actions.Create`
  - `AllbertAssist.Confirmations.Store.Actions.Read`
  - `AllbertAssist.Confirmations.Store.Actions.List`
  - `AllbertAssist.Confirmations.Store.Actions.Resolve`
  - `AllbertAssist.Confirmations.Store.Actions.Expire`
  - `AllbertAssist.Confirmations.Store.Actions.Rebuild`
- Route those internal actions through the Store agent using documented
  Jido.Agent/Jido.AgentServer APIs. Exact wrapper mechanics are an
  implementation detail, but public callers still use the existing store
  functions or `AllbertAssist.Confirmations`.
- On `create`, write the pending YAML record and audit entry before updating
  the projection.
- On `resolve`, re-read the pending YAML record from disk, write the resolved
  record, remove the pending file, append audit, then update the projection.
- On `expire`, enumerate pending files and resolve expired records.
- On restart, rebuild the projection from `pending/*.yml`; do not trust stale
  process state.
- Use `on_before_cmd/2` and/or `on_after_cmd/3` only where they simplify
  projection upkeep, validation, or audit metadata. Error handling belongs in
  action return values and existing caller paths unless current Jido docs prove
  a better supported callback.

Do not:

- Migrate confirmations to SQLite.
- Add a second confirmation queue.
- Change confirmation record YAML shape.
- Add new confirmation signals unless the existing v0.07 signal path already
  emits them and tests assert no payload drift.

### Jobs.Scheduler

Goal: make `AllbertAssist.Jobs.Scheduler` a supervised Jido.Agent-backed
scheduler while preserving the v0.13 durable job/run contract.

Public API must remain:

- `start_link/1`
- `run_once/2`
- `cleanup_stale_runs/2`

Implementation direction:

- Preserve the current principle: the scheduler keeps no authoritative queue
  in memory. Each tick calls `Jobs.due_jobs/2`, claims due jobs durably, and
  executes claimed runs through `Jobs.Runner`.
- Jido.Agent state should mirror scheduler runtime configuration and recent
  diagnostic state, not copy every durable job row:
  - `interval_ms`
  - `initial_delay_ms`
  - `batch_size`
  - `stale_run_ms`
  - `enabled?`
  - `poll_on_start?`
  - `cleanup_on_start?`
  - `last_tick_at`
  - `last_summary`
  - `last_error`
- Add private internal Jido.Action command modules, for example:
  - `AllbertAssist.Jobs.Scheduler.Actions.RunOnce`
  - `AllbertAssist.Jobs.Scheduler.Actions.CleanupStaleRuns`
  - `AllbertAssist.Jobs.Scheduler.Actions.Tick`
  - `AllbertAssist.Jobs.Scheduler.Actions.ScheduleNextTick`
- Replace direct `Process.send_after/3` use in scheduler code with the
  current documented Jido scheduling directive where practical, currently
  `Jido.Agent.Directive.schedule/2`. If implementation proves the directive
  cannot target the supervised scheduler process without behavior drift, stop
  and document the constraint before falling back to a tiny timer wrapper.
- Keep existing job lifecycle signals (`allbert.job.*`) and payload shapes.
- Keep blocked-job behavior from v0.13: blocked jobs are not manually or
  automatically run while their referenced confirmation remains pending.
- Keep stale-run cleanup behavior on startup.

Do not:

- Move job creation/pause/resume/cancel semantics into Scheduler. Those stay
  in `AllbertAssist.Jobs` and registered actions.
- Add a new in-memory durable queue.
- Change `scheduled_jobs` or `scheduled_job_runs` schemas.
- Change job CLI output.
