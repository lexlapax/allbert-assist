## Purpose

v0.23 converts `AllbertAssist.Confirmations.Store` and
`AllbertAssist.Jobs.Scheduler` from plain `GenServer` modules to
`Jido.Agent` modules. Both are state-machine components with plausible
successor-agent stories (smarter confirmation policies, smarter
schedulers). Converting them aligns the existing codebase with the
project-direction rethink before v0.24 ships `Objectives.Engine` as the
next new Jido.Agent.

The milestone is a pure architectural refactor: no new user-visible
features, no schema changes, no semantic changes to the v0.07
confirmation workflow or v0.13 scheduler workflow. The full v0.07 and
v0.13 acceptance criteria continue to hold after v0.23.

`AllbertAssist.Settings`, `AllbertAssist.Trace`, `AllbertAssist.Memory`
storage IO, `AllbertAssist.Session.Scratchpad`, `Memory.Compiler`, and
`Memory.Promotion` stay as plain `GenServer` modules. They are stateful
storage with fixed algorithms; converting them would add ceremony
without unlocking value.

## Architecture Readiness Audit

The v0.23 plan is consistent with current architecture:

- ADR 0006 keeps Security Central as the permission, risk, redaction,
  and audit boundary. The conversion does not change permission classes,
  safety floors, redaction posture, or audit fields.
- ADR 0007 requires runtime-facing, effectful, observable operations to
  enter through registered actions. v0.07 confirmation actions
  (`create_pending`, `approve_confirmation`, `deny_confirmation`,
  `expire_confirmation`, `list_confirmations`, `show_confirmation`) and
  v0.13 job actions (`schedule_job`, `cancel_job`, `list_jobs`,
  `pause_job`, `resume_job`) remain registered Jido actions. The agents
  are state holders that those actions invoke; they do not replace the
  action boundary.
- ADR 0008 defines durable confirmation requests in SQLite. SQLite
  remains authoritative for durable state. The agent's in-memory state
  is a bounded projection used for fast lookups during a turn; on
  restart it is rebuilt from SQLite.
- ADR 0014 (local workspace identity) is unchanged.

No new ADR is required for v0.23. The pragmatic rule for new components
is documented in the vision and `AGENTS.md`. If future drift requires
tightening to a hard decision-making vs storage rule, that revision
would touch the vision and `AGENTS.md`, not v0.23 internals.

## Vision And Roadmap Alignment

The vision says Allbert should use `Jido.Signal` for events,
`Jido.Action` for capabilities, and `Jido.Agent` where state machines
or lifecycle hooks earn their keep. Until v0.23, the only
`Jido.AI.Agent` in the codebase is `IntentAgent`. v0.23 closes the
clearest part of that gap: two existing state-machine components become
Jido.Agents, and v0.24 (Objective Runtime Foundation) adds
`Objectives.Engine` as the third.

Roadmap dependency order under the project-direction rethink:

- v0.22 supplied StockSage Python bridge.
- v0.23 (this milestone) converts Confirmations.Store + Jobs.Scheduler.
- v0.24 ships Objective Runtime Foundation, building on Confirmations
  and Jobs as Jido.Agents.
- v0.25 (Native Jido Trading Agents) consumes objective state.
- v0.26+ continues per the renumbered sequence.

v0.23 must not pull later work forward. It does not add objective
state, new step kinds, new advisory provider behaviours, new world-model
abstractions, new permissions, new confirmations, or new jobs. It does
not change the v0.07 confirmation surface or the v0.13 scheduler
surface.

## Guiding Rules

1. **No behavior change visible to operators.** All CLI commands,
   LiveView surfaces, signals, settings, audit records, and trace
   sections must produce byte-identical or trivially-equivalent output
   before and after the conversion. "Trivially equivalent" means
   numeric timestamps may differ by milliseconds and signal source
   metadata may carry the Jido.Agent name where the GenServer name
   appeared, but no operator-meaningful field changes.

2. **No durable schema changes.** `confirmations`, `jobs`, and
   `job_runs` SQLite tables, columns, and indexes are unchanged.
   Migrations are zero. If the agent's in-memory state shape changes,
   it is rebuilt from SQLite on every start.

3. **No new permissions, safety floors, or risk tiers.** All
   confirmation permission classes (`:confirmation_decide`,
   `:settings_write`, etc.) and job permission classes
   (`:job_schedule`, `:job_cancel`) remain identical.

4. **No new registered actions.** The existing v0.07 and v0.13 action
   surfaces continue. Internal helpers may be added or refactored, but
   no action enters or leaves the registry as part of v0.23.

5. **State is rebuildable.** On agent start, in-memory state is
   reconstructed by reading SQLite. Crash recovery uses the same path.
   Agent state is a cache, not authoritative.

6. **Lifecycle hooks are the win.** The Jido.Agent value is in
   `on_before_validate_state`, `on_after_validate_state`,
   `on_before_run`, `on_after_run`, `on_error`, `on_before_cmd`,
   `on_after_cmd`. Use them where they make existing logic cleaner:
   status-transition audit emission, pending-cache eviction on resolve,
   next-run computation, schedule replay on settings change. Do not
   force hooks into places where existing GenServer callbacks were
   adequate.

7. **Pragmatic rule for the rest.** Settings, Trace, Memory storage,
   Scratchpad, Memory.Compiler, and Memory.Promotion stay as plain
   GenServers. The vision update in M3 codifies this without forcing
   converts.

## Module Shapes

### `AllbertAssist.Confirmations.Store` (after)

```elixir
defmodule AllbertAssist.Confirmations.Store do
  use Jido.Agent,
    name: "confirmations_store",
    description: "Durable pending-confirmation state machine",
    schema: [
      pending_by_id: [type: :map, default: %{}],
      pending_by_action: [type: :map, default: %{}],
      ttl_index: [type: :map, default: %{}]
    ]

  alias AllbertAssist.Confirmations.Record
  alias AllbertAssist.Repo

  # state-machine transitions become declared commands
  defcommand :create_pending, schema: [...] do ... end
  defcommand :approve, schema: [...] do ... end
  defcommand :deny, schema: [...] do ... end
  defcommand :expire, schema: [...] do ... end
  defcommand :rebuild_from_repo, schema: [] do ... end

  # lifecycle hooks
  def on_before_validate_state(state), do: {:ok, prune_expired(state)}
  def on_after_cmd(state, :approve, %{record: record}) do
    AllbertAssist.Trace.record_confirmation_approved(record)
    {:ok, state}
  end
  def on_after_cmd(state, :deny, %{record: record}) do
    AllbertAssist.Trace.record_confirmation_denied(record)
    {:ok, state}
  end
  def on_error(state, command, reason, _ctx) do
    Logger.error("Confirmations.Store #{command} failed: #{inspect(reason)}")
    {:ok, state}
  end
end
```

In-memory state:

- `pending_by_id` — `%{confirmation_id => Record.t()}` for fast lookup
  during action resume.
- `pending_by_action` — `%{action_key => [confirmation_id]}` for
  duplicate detection.
- `ttl_index` — `%{epoch_ms => [confirmation_id]}` for expiry sweep.

Durable state: unchanged SQLite `confirmations` table.

### `AllbertAssist.Confirmations.Store` (before — for reference)

The current implementation is a `GenServer` with `handle_call` for
`create_pending`, `approve`, `deny`, `expire`, and `list`. State holds
the same three caches. The conversion preserves state shape and behavior;
only the dispatch mechanism changes from `handle_call/3` clauses to
`defcommand` blocks.

### `AllbertAssist.Jobs.Scheduler` (after)

```elixir
defmodule AllbertAssist.Jobs.Scheduler do
  use Jido.Agent,
    name: "jobs_scheduler",
    description: "Cron-based scheduled job state machine",
    schema: [
      jobs_by_id: [type: :map, default: %{}],
      due_queue: [type: :list, default: []],
      paused: [type: :map, default: %{}],
      next_run_index: [type: :map, default: %{}]
    ]

  defcommand :schedule, schema: [...] do ... end
  defcommand :cancel, schema: [...] do ... end
  defcommand :pause, schema: [...] do ... end
  defcommand :resume, schema: [...] do ... end
  defcommand :tick, schema: [] do ... end
  defcommand :rebuild_from_repo, schema: [] do ... end

  def on_after_cmd(state, :schedule, %{job: job}) do
    compute_and_emit_next_run(job)
    {:ok, recompute_due(state)}
  end
  def on_after_cmd(state, :tick, _result) do
    {:ok, recompute_due(state)}
  end
  def on_error(state, command, reason, _ctx) do
    Logger.error("Jobs.Scheduler #{command} failed: #{inspect(reason)}")
    {:ok, state}
  end
end
```

In-memory state:

- `jobs_by_id` — `%{job_id => Job.t()}` for fast lookup.
- `due_queue` — sorted list of `{next_run_at, job_id}` for cron tick.
- `paused` — `%{job_id => paused_at}` for paused jobs.
- `next_run_index` — `%{cron_expr => [job_id]}` for evaluating one cron
  schedule's targets in O(1).

Durable state: unchanged SQLite `jobs` and `job_runs` tables.

### `AllbertAssist.Jobs.Scheduler` (before — for reference)

The current implementation is a `GenServer` using `Process.send_after`
for the tick cycle and `handle_call`/`handle_info` for schedule, cancel,
pause, resume, and tick. Conversion replaces `handle_*` clauses with
`defcommand` blocks; the tick uses `Jido.Action` directive
`{:emit_after, delay, signal}` instead of `Process.send_after`.

## Actions (unchanged surface)

All registered actions under `AllbertAssist.Actions.Confirmations.*`
and `AllbertAssist.Actions.Jobs.*` keep their:

- module names
- permission classes
- argument schemas
- result shapes
- side-effect semantics
- trace/audit metadata

The action implementations now dispatch to the agent via
`Jido.Agent.cmd/3` instead of `GenServer.call/3`. Operator-visible
behavior is unchanged.

## Settings (unchanged)

`confirmations.*` and `jobs.*` settings schemas, defaults, writability,
and Settings Central validation are unchanged.

## CLI (unchanged)

`mix allbert.confirmations` and `mix allbert.jobs` commands, output
formats, error messages, and confirmation prompts are unchanged.

## Signals

Existing signals continue:

```text
allbert.confirmation.created
allbert.confirmation.approved
allbert.confirmation.denied
allbert.confirmation.expired
allbert.job.scheduled
allbert.job.cancelled
allbert.job.due
allbert.job.started
allbert.job.completed
allbert.job.failed
allbert.job.skipped
allbert.job.needs_confirmation
```

Jido.Agent adds standard lifecycle signals automatically. v0.23 does
**not** suppress them; they appear in trace at `:debug` level and are
filtered out of operator-facing trace sections so operator output is
unchanged:

```text
allbert.confirmations_store.cmd.started        # Jido lifecycle
allbert.confirmations_store.cmd.completed      # Jido lifecycle
allbert.confirmations_store.state.validated    # Jido lifecycle
allbert.jobs_scheduler.cmd.started             # Jido lifecycle
allbert.jobs_scheduler.cmd.completed           # Jido lifecycle
allbert.jobs_scheduler.state.validated         # Jido lifecycle
```

Trace level mapping rule: Jido-lifecycle signals render only when
`traces.detail` is `:debug` or higher. The default
`traces.detail = :operator` does not surface them.

Release documentation must preserve the safety story:

- v0.23 is a refactor. No new user-visible features, no permission
  changes, no schema changes, no semantic changes.
- Both converted modules continue to delegate authority to Security
  Central via the action runner.
- SQLite remains authoritative for durable state; agents are caches.
- The pragmatic rule means new components author chooses
  Jido.Agent or plain GenServer based on plausible value; no automatic
  conversion of remaining GenServers.

## Non-Goals

- No conversion of `Settings`, `Trace`, `Memory` storage IO,
  `Session.Scratchpad`, `Memory.Compiler`, or `Memory.Promotion` to
  Jido.Agents.
- No new permissions, safety floors, or risk tiers.
- No new registered actions; no removal of existing actions.
- No durable schema changes for `confirmations`, `jobs`, or `job_runs`.
- No new operator-facing CLI commands or LiveView surfaces.
- No new Settings Central keys.
- No new Resource Access Security Posture consumers.
- No new channel-adapter behavior.
- No StockSage native trading agents (v0.25 and later).
- No objective runtime modules, tables, or signals (v0.24).
