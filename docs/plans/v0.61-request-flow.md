## Main User/Operator Flows

- **Before the build:** the operator previews **≥3 divergent layout systems**
  rendered across all nine IA surfaces in Direction C, side-by-side in Chrome, and
  **chooses one** (`CHOSEN_LAYOUT`) — the layout the rest of the release implements.
- An operator navigates an app whose information architecture and navigation were
  redesigned in v0.60 and implemented here in the chosen layout: the redesigned
  shell and navigation place workspace, jobs, objectives, settings, and the operator
  panels per the v0.60 IA model, dressed in Direction C.
- A visitor lands on a real `/` marketing surface with the Allbert brand, SEO/OG
  metadata, and variant-registry CTAs, composed per the v0.60 IA in the chosen
  layout, then enters the app shell.
- An operator works in a redesigned shell carrying the Allbert logo/wordmark instead
  of the stock Phoenix asset.
- An operator sees entrance/drawer/skeleton motion that collapses to instant when
  their reduced-motion preference is set.
- An operator reads the redesigned `/workspace`, `/jobs`, `/objectives`, and
  operator panels with real visual hierarchy and populated empty/first-run states.
- A first-run operator on an empty workspace sees "what can I do" suggested-action
  affordances shaped by the v0.60 onboarding-flow design and First-Model Path
  (view-only; seats the v0.63 guided-onboarding wizard).
- An operator with `system` theme selected gets a dark theme resolved from the OS
  `prefers-color-scheme` preference.

## Data Flow

The layout exploration, IA/navigation redesign, redesigned screens, Direction C
tokens, brand assets, motion, hierarchy, dark-mode resolution, and SEO/OG metadata
are presentation concerns rendered through the unified catalog inside one
redesigned app shell. The structural redesign recomposes screens per the v0.60
information architecture in the M2-chosen layout through the existing
catalog/shell/token substrate — it adds no new rendering path and no new data-layer
path; the same DTOs render through redesigned screens. The walking skeleton from
v0.60 becomes the real navigation frame the redesigned screens fill, built by
extending the variant registry and pattern library per the v0.60 design-system gap
analysis and the Direction C delta. The layout-system previews (M1) and the
visual-direction previews (v0.60b) are disposable exploration behind the
`:preview_routes` flag, retired at closeout. The landing surface emits the shell
data contract, consumes the Direction C tokens and the variant registry. Motion
durations/easings come from the Direction C motion roles over the v0.58 motion-scale
tokens; the reduced-motion axis gates them. Dark mode resolves app tokens from the
operator's theme setting plus OS `prefers-color-scheme`. Suggested-action
affordances and landing CTAs compose registered-action DTOs (ADR 0073) for display
only, along the v0.60 onboarding-flow path; no backend, model, or data-layer path is
added.

## Authority Invariants

No new capability and no authority change. The catalog stays the rendering boundary
(ADR 0030/0074). The layout exploration, IA/navigation redesign, redesigned screens,
Direction C tokens, brand, motion, hierarchy, landing, dark mode, and
suggested-action affordances are presentation-only — the structural redesign
recomposes screens through the existing substrate and grants no authority. The
layout-system previews read no live data and grant no authority. Suggested actions
and landing CTAs render registered-action DTOs and grant no authority or egress;
launcher selection stays view-only. No model- or data-generated markup. No new route
beyond the landing surface (the nine IA surfaces map onto existing routes plus the
rebuilt `/` per the plan's M3 table). No rebuild of the rendering or data substrate
(tokens/registry/pattern-library/catalog/shell and canvas/tiles/ephemerals/
Fragment-HMAC/persistence are reused as-is). No internal rename. SEO/OG metadata is
static and exposes no operator data or secrets. v0.61 implements Direction C; it
does not re-decide the visual language.

## Operator Validation

Run in order; stop at the first non-zero status, missing artifact, unresolved route,
a11y failure, regression, or an absent recorded choice. This is a **presentation +
design-review** validation on the LIVE product surface (not the `/preview/*`
scaffold, except S3–S4 which validate the disposable layout previews the operator
chooses from). There is no new authority to exercise; the professional-UX bar is
judged against the v0.60b `visual-language-brief.md` rubric, not taste alone.

**Two-phase timing of the layout choice.** The layout choice is MADE and formalized
during **M2** — an initial design review scores the rendered layout systems, picks
one, and writes it into `layout-systems-selected.md` so the gate can go green. The
**S5** design-review pass RE-CONFIRMS that already-formalized choice against the same
rendered systems; it does not invent a new one. **S6** binds the S5-recorded
`CHOSEN_LAYOUT` to the direction named in `layout-systems-selected.md`, so the two
phases cannot drift. The heart of the design validation is **S5**, where the operator
views the ≥3 layout systems side-by-side across all nine IA surfaces and re-confirms
the chosen layout.
