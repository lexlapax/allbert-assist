## Main User/Operator Flows

- New operator launches Allbert; the web wizard auto-opens on first run.
- Operator picks a track: QuickStart (fastest first chat) or Advanced (full
  control).
- Operator sets up a provider/model: enters a masked key, sees inline `doctor`
  verification, chooses local (Ollama) or hosted (OpenAI / Anthropic / OpenRouter),
  and can switch providers without editing config.
- Operator selects a user-category profile, reviews exactly what it seeds, and
  confirms; Settings Central defaults plus suggested apps, channels, and intents
  are applied.
- QuickStart skips channel/integration setup and reaches a working first chat;
  Advanced continues into optional channel pairing and integration setup.
- Operator sees a health-check confirmation succeed and the trust spine surfaced
  as a feature.
- Headless operator runs the same flow through the interactive CLI/TUI wizard.

## Data Flow

The shared wizard state machine drives the step sequence and validation; the web
(ADR 0074) and CLI/TUI (ADR 0067) surfaces render it without forking logic.
Provider/model setup writes credentials through the existing encrypted secrets
store (vault-ready for v0.62) and verifies them through the existing `doctor`
flow. Profile selection reads repo-maintained declarative presets and, on explicit
confirm, seeds Settings Central default values plus suggested apps, channels, and
intents through Settings Central. Channel pairing and integration setup reuse the
existing pairing flows. No step introduces a new write path: settings and profile
seeds flow only through Settings Central, and every effectful step routes through
the existing gates and confirmations.

## Authority Invariants

No new capability is introduced. Onboarding is an experience over existing flows;
Security Central (ADR 0006) is unchanged. Profiles seed defaults only — they lower
no confirmation floor, grant no egress, and enable nothing effectful without the
normal gates. Settings and profile seeds flow only through Settings Central.
Masked-key entry never echoes or logs raw secret values; wizard diagnostics are
redacted.

## Handles / CLI Surfaces

- Web auto-first-run wizard (ADR 0074 design system).
- Interactive CLI/TUI wizard over the ADR 0067 terminal channel (replaces the
  copy-paste `mix allbert.onboard` output).
- Provider/model setup step: masked-key entry, inline `doctor`, provider switch.
- Registered settings action to apply a user-category profile preset with
  review/confirm.
- Health-check confirmation surface.

## Smoke Checklist

- Launch a disposable home; confirm the web wizard auto-opens on first run.
- Run QuickStart to a working first chat; confirm channel/integration setup is
  skipped on the fast path.
- Enter a masked provider key; confirm inline `doctor` verification and that the
  raw key is never echoed or logged.
- Switch provider (local Ollama ↔ hosted OpenAI / Anthropic / OpenRouter) without
  editing config.
- Select each shipped profile (researcher / developer / writer / ops / general);
  confirm the review step shows exactly what is seeded and nothing is written
  before confirm.
- Confirm a profile apply seeds Settings Central defaults plus suggested apps,
  channels, and intents only — no authority change, no confirmation-floor change.
- Run the same flow through the interactive CLI/TUI wizard; confirm it prompts and
  verifies in place (no copy-paste shell commands).
- Confirm the trust spine (confirmation / permission / trace) is surfaced.
- Confirm docs and gates are clean.
