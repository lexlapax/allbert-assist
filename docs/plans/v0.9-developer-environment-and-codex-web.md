# v0.9 — Developer environment and Codex Web readiness

Status: Shipped

## Context

Through v0.8, Allbert is documented and validated primarily as a source-based end-user product for a local technical operator. That is enough for product use, but it is not yet a complete development contract for contributors or coding agents working in ephemeral environments.

The repo currently assumes a lot of contributor knowledge implicitly:

- Rust is installed, but the toolchain is not pinned in-repo.
- `rustfmt` and `clippy` are expected, but not declared in a toolchain file.
- `RUSTC_WRAPPER` may be set locally, but should not be assumed.
- normal contributor validation is expected to be provider-free, but that distinction is not documented cleanly.
- safe temp-profile usage through `ALLBERT_HOME` is common practice, but it is not yet the documented default contributor path.
- there is no repo-level contributor guide and no repo-level `AGENTS.md` for coding agents.

That gap matters before self-improvement. A Rust rebuild skill or coding agent cannot be treated as first-class until the repo itself declares what a valid development environment looks like on macOS, Linux, and Codex Web.

v0.9 fills that gap. It is not about hosted deployment and it is not about making the daemon a web service. It is about making the source tree reproducible and legible for local contributors and remote ephemeral coding workspaces.

## Goal

Define a reproducible contributor contract for the Allbert source tree:

1. pin the Rust toolchain and required components in-repo;
2. document a canonical macOS/Linux development setup;
3. document a Codex Web / ephemeral-workspace development workflow;
4. separate required provider-free validation from optional live-provider verification;
5. add repo-level contributor instructions that are explicitly distinct from runtime bootstrap artifacts;
6. keep the product runtime local-first while making the source tree easy to work on from more than one development environment.

## Milestones

### M1 — Toolchain and platform contract (ADR 0063)

- Add `rust-toolchain.toml` at repo root.
- Pin the toolchain to the current known-good version: `1.94.0`.
- Include required components:
  - `rustfmt`
  - `clippy`
- Freeze the supported contributor platforms:
  - macOS
  - Linux
- Freeze unsupported-through-v0.9:
  - Windows native
- Document that `rustup` is the required toolchain manager for contributor setup.
- Document that no additional Node/Python/database service is required for the default workspace build/test path.
- Document that OpenSSL is not a required local prerequisite because the workspace uses `reqwest` with `rustls-tls`.

### M2 — Contributor docs and repo agent instructions (ADR 0065)

- Add `DEVELOPMENT.md` at repo root.
- Add repo root `AGENTS.md`.
- Add `.env.example`.
- `DEVELOPMENT.md` must cover:
  - local macOS/Linux setup via `rustup`
  - Codex Web setup expectations
  - required vs optional environment variables
  - canonical validation commands
  - temp-profile guidance using `ALLBERT_HOME`
- Root `AGENTS.md` must:
  - state that it is for repository contributors and coding agents
  - explicitly distinguish itself from runtime-generated `~/.allbert/AGENTS.md`
  - document expected workflow discipline: temp profiles, fake-provider-first validation, and doc/code consistency checks
- `.env.example` must enumerate only optional live-check variables and never imply that secrets are required for routine development.

### M3 — Validation matrix (ADR 0064)

- Document one canonical green path that works on macOS, Linux, and Codex Web:
  1. `cargo fmt --check`
  2. `env -u RUSTC_WRAPPER cargo clippy --workspace --all-targets -- -D warnings`
  3. `env -u RUSTC_WRAPPER cargo test -q`
  4. `env -u RUSTC_WRAPPER cargo run -q -p allbert-cli -- --help`
- Document that `RUSTC_WRAPPER` is optional and must not be assumed.
- Document that normal contributor validation is:
  - provider-free
  - network-optional
  - safe to run without touching the developer's real `~/.allbert`
- Document that full green status does not require live Anthropic/OpenRouter turns.

### M4 — Temp-profile smoke workflow

- Standardize the safe contributor smoke posture around temporary `ALLBERT_HOME`.
- Provide one macOS/Linux shell recipe that:
  - uses a temp profile directory
  - runs a safe CLI/daemon smoke
  - does not require provider secrets
- Provide a Codex Web equivalent that uses a workspace-local temp path when `mktemp` is inconvenient.
- Document the rule that contributor smokes should never rely on a pre-existing real profile.

### M5 — Secrets and live-provider posture

- Split contributor checks into two explicit tiers.

**Tier A — required**
- `cargo fmt --check`
- clippy zero warnings
- full test suite
- CLI help smoke
- temp-home smoke without live providers

**Tier B — optional**
- real Anthropic/OpenRouter verification
- Telegram bot/channel verification
- any checks requiring secrets or network

- Document that Codex Web is expected to pass Tier A without secrets.
- Document live env vars in `.env.example`, including at least:
  - `ANTHROPIC_API_KEY`
  - `OPENROUTER_API_KEY`
  - `OPENROUTER_API_KEY_BOOTSTRAP`
- Document Telegram testing prerequisites as optional local-only setup, not default contributor requirements.

### M6 — Codex Web workflow guide

- Document Codex Web as an ephemeral developer environment, not a hosted Allbert runtime.
- Freeze the expected workflow:
  - clone/open the repo
  - ensure the pinned Rust toolchain is available
  - use Tier A validation
  - use temp `ALLBERT_HOME`
  - do not assume a persistent daemon across workspace lifetimes
  - treat live-provider checks as optional
- Document that the primary Codex Web expectations are:
  - reproducible local compilation
  - provider-free tests
  - no reliance on the contributor's personal shell state
- Document the known limitation that local machine integrations and long-lived secret-backed channels are better exercised from a real macOS/Linux workstation.

### M7 — Closeout

- Update `README.md` with a short contributor pointer to `DEVELOPMENT.md`.
- Update `docs/vision.md` and `docs/plans/roadmap.md` so the release ordering is consistent.
- Update any v0.8 docs that currently point non-dev follow-on work at `v0.9` when that work now really belongs to `v0.10` or later.
- Add the final consistency pass:
  - no roadmap entry still calls self-improvement `v0.9`
  - no contributor doc implies live-provider checks are required for normal development
  - no repo doc conflates root `AGENTS.md` with runtime `~/.allbert/AGENTS.md`

## Defaults

| Setting | Default |
| --- | --- |
| contributor platforms | macOS and Linux |
| unsupported platform | Windows native |
| toolchain manager | `rustup` |
| Rust version | `1.94.0` |
| required rustup components | `rustfmt`, `clippy` |
| default contributor validation | provider-free, network-optional |
| default profile posture for smokes | temp `ALLBERT_HOME` |
| live-provider verification | optional |

## Dependencies

- v0.1 through v0.8 establish the product/runtime surface this repo builds and tests.
- v0.10 self-improvement depends on this release so rebuild skills and coding agents operate against an explicit, reproducible development contract instead of unstated workstation assumptions.

## Out of scope for v0.9

- Hosted daemon deployment.
- Webhooks or public network listeners.
- CI/CD rollout design.
- Windows-native contributor support.
- Self-improvement and embedded scripting (moved to v0.10).
