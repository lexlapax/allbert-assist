## §A — The Fan-Out Lifecycle (frame → deliver → acknowledge → run → join)

```
user prompt (any Runtime caller)
  │
  ▼
Runtime.submit_user_input  ──────────────  synchronous, unchanged entry
  │
  ▼
IntentAgent turn (settings-pinned)
  │
  ├─ Stage-0 Decomposer (S3)         :single ──▶ today's pipeline, unchanged
  │      broad automatic eligibility: ≥2 independent tasks; explicit
  │      parallel language not required; uncertain/nested ⇒ :single
  │      │ :fanout (task list, advisory)
  │      ▼
  ├─ Objectives.Fanout.frame/2       parent + N children (durable, one txn,
  │                                  origin attribution copied to children)
  │      │
  │      ▼
  └─ response = KICKOFF ACK + opaque start receipt
         │   children remain durable open; NOTHING executes yet
         ▼
caller delivers/records kickoff
  ├─ remote chat: transport success      ├─ HTTP/ACP/OpenAI: response committed
  ├─ web/TUI: rendered/printed           ├─ CLI: output completed
  └─ Jobs: durable kickoff event
         │
         ├─ failure ──▶ persist delivery_blocked; retry idempotently or cancel
         ▼
Runtime acknowledges receipt (single-use, identity-bound, idempotent,
non-authoritative) ──▶ queue runnable ──▶ Coordinator starts
  (confirm_before_start=true adds its durable confirmation before start)
         │
         ▼
caller is free; duplicate acknowledgement cannot start work twice

background (per child, in parallel, supervised):
  RunServer: pin settings ──▶ authorize step ──▶ Actions.Runner.run/3
     ▲                │            (cancel-token checkpoints between operations)
     │                ▼
     │        observe/advance ──▶ next step … until terminal
     │                │
     │                ├─ needs_confirmation ──▶ park (step blocked +
     │                │                          confirmation_id); siblings
     │                │                          unaffected; notify kind
     │                │                          :confirmation_request (§E)
     │                └─ run.* signals on the SignalBus (status stream)
     ▼
  Coordinator (monitors): child terminal ──▶ durable status reduction
     │  … all children terminal
     ▼
  JOIN: parent terminal + enumerated per-child report (fanout.joined)
     │
     ├─ autonomous notify ON  ──▶ Channels.Notify ──▶ adapter deliver_outbound
     └─ autonomous notify OFF ──▶ pending_reports on the NEXT turn +
                                  web/TUI objectives surfaces (always)
```

Status streaming while running: `run.progress` signals fan to (a) the web
workspace via the existing SignalBridge objective-event path, (b) an attached
TUI session's live region, (c) remote channels ONLY through `Channels.Notify`
when `level = "status_and_completion"` (coalesced, throttled; on channels
declaring `status_update_mode: :edit_in_place` the one status message is
edited in place, with append fallback — S7b/M7).

```
│   ├── Runs.RunServer (child A1, restart: :temporary)           [NEW]
│   │     plain GenServer (@moduledoc-documented substrate choice)
│   │     owns: step pipeline, per-step settings pin, CancelToken
│   │     checkpoints, run.* signal emission
│   ├── Runs.RunServer (child A2) …
│   └── Runs.Coordinator (parent B) …
├── Channels.Supervisor (one_for_one)                            [exists]
│   ├── Telegram.Adapter … (eight adapters, one child each)
│   └── Signal.Adapter
│         └── Signal daemon receiver (MuonTrap.Daemon,           [NEW wiring]
│             ADR 0058 — child spec existed, never mounted)
└── (web) AllbertAssistWeb.Endpoint ── WorkspaceLive             [exists]
          subscribes SignalBridge topics (objective events, fragments)
```

Rules (ADR 0083 / plan Locked Decision 5): `restart: :temporary` everywhere in
the run scope (restart is a COORDINATOR decision from durable state, not a
supervisor reflex); monitors not polling; Registry-addressed steering; no bare
`Task.async` for steerable work; OS processes spawned by steps are captured as
os_pid/process groups (ADR 0085) — supervision is never a security boundary.
Delivery state/receipt digest, queue position, run-attempt count, review
reason, and join outcome are durable. Coordinator memory is a rebuildable
projection and owns no private goal-loop truth.

## §D — Per-Channel Delivery Matrix

Streaming declarations land in the channel descriptor (S7) and the parity
matrix derives from them (no more `channel_parity.ex:98` hardcode). "Steer"
means: a reply on the origin thread is classified against the active fan-out
(§E); available on every surface because inbound is unchanged.

| Surface | streaming decl | Kickoff ack | Status updates | Completion report | Confirmation notify | Steer-by-reply | v1.1 adapter work |
|---|---|---|---|---|---|---|---|
| web | `:live_region` | chat thread + objectives tree | SignalBridge push (no Notify authority needed — attached local surface) | in-thread + tree | existing approval surface | chat input | fan-out tree, steering affordances, notify settings panel (M9) |
| tui | `:live_region` | rendered list | live region while attached (stream_event_sink reuse); nothing when detached | attached: printed; else next-turn | typed `ALLBERT:APPROVE:<id>` | reply / escape-offer | ack + live region + escape cancel + settings CLI (M6/M9) |
| telegram | `:progress_messages`, `status_update_mode: :edit_in_place` | reply message | single status message edited in place (opt-in; append fallback) | reply message (opt-in) or next-turn | inline buttons + typed command | reply in chat | threading hint passthrough (has outbound `adapter.ex:915`); NEW `editMessageText` client (M7) |
| email | `:turn_complete` | reply email | none (digest-class; level forced `"completion"`) | threaded reply email (opt-in) or next-turn | typed command in reply | reply email | **implement `deliver_outbound/3`** via SmtpClient (M6) |
| discord | `:progress_messages`, `status_update_mode: :edit_in_place` | reply message | single status message edited in place (opt-in; append fallback) | reply/thread message (opt-in) or next-turn | buttons + typed command | reply/thread | threading hint → message_reference (has outbound `:608`); NEW message-PATCH client (M7) |
| slack | `:progress_messages`, `status_update_mode: :edit_in_place` | thread reply | single status message edited in place (opt-in; append fallback) | thread message (opt-in) or next-turn | buttons + typed command | thread reply | threading hint → thread_ts (has outbound `:650`); NEW `chat.update` client (M7) |
| matrix | `:progress_messages`, `status_update_mode: :edit_in_place` | reply event | single status message edited in place (opt-in; append fallback) | thread event (opt-in) or next-turn | typed command | room reply | threading hint → m.thread relation; allowed_room_ids stands (has outbound `:601`); NEW `m.replace` edit via whitelisted PUT (M7) |
| whatsapp | `:progress_messages` | reply message | new messages (opt-in) | reply (opt-in) or next-turn | buttons + typed command | reply | threading hint → context_message_id; ReleaseAvailability-gated (has outbound `:793`) |
| signal | `:progress_messages` | reply message | new messages (opt-in) | reply (opt-in) or next-turn | typed command | reply | **wire MuonTrap daemon receiver** (inbound gap); ReleaseAvailability-gated (has outbound `:526`) |

Notes: "opt-in" = `channels.<id>.autonomous_notify.enabled = true` (default
false) with `level` selecting status vs completion-only; the kickoff ack is
NEVER opt-in (it is the turn response, Locked Decision 8); the ack always
states the report delivery mode for this channel (push here vs next-turn,
with the enable hint when off) so the operator knows where results will
land; `pending_reports` renders as compact per-fan-out summaries (full
report on a status query). `status_update_mode` is a second additive
descriptor declaration (S7b/M7, Locked Decision 11): telegram/discord/
slack/matrix declare `:edit_in_place` — Notify keeps ONE status message per
fan-out per channel and edits it, falling back automatically to an audited
append on edit failure; whatsapp (no business-initiated Cloud API edit),
signal (partial/recent CLI support), and email stay `:append_only`.
Completion reports and confirmation requests are ALWAYS new messages —
never edits — and an edit counts as a delivery for the min-interval
throttle.

## §E — Steering Flows

**Classification (advisory, S5).** Mid-flight input on a thread with an
active fan-out extends the existing `:continue_objective` seam:

```
inbound text ──▶ engine candidates (fan-out parent/children present)
  ├─ no fan-out candidate wins ──▶ normal pipeline (new request)
  └─ fan-out candidate wins ──▶ Intent.Steering.classify
        ├─ :status      ──▶ aggregate every active fan-out in thread/session
        ├─ :steer/:adjust ─▶ steer_objective_run action ──▶ directive event
        │                     (durable) ──▶ Registry send to RunServer ──▶
        │                     applied at next STEP boundary, never mid-op
        ├─ :cancel      ──▶ cancel_objective_run (child ref or whole fan-out)
        │                     ──▶ tiers: cooperative → supervised → OS kill
        ├─ :new_request ──▶ normal pipeline (decomposer may run)
        └─ low confidence ─▶ clarify turn (list children, ask which/what)
```

Invariants (gate-bound): the classifier is advisory; free text NEVER approves
a pending confirmation (approval stays on `ALLBERT:APPROVE`/buttons through
`ConfirmationCallback`); steering can raise but never lower a confirmation
class; an identity-unmapped sender cannot steer at all (inbound trust
unchanged); steering another user's objective is denied at the action.
Mutating steering resolves an explicit child/fan-out reference. If multiple
targets remain plausible, Allbert clarifies; it never silently chooses the
newest run.

**Background-raised confirmations, out-of-band.** A run hitting
`needs_confirmation` parks its step (`confirmation_id`) and emits
`run.blocked`:

```
run.blocked ──▶ Channels.Notify kind=:confirmation_request
  ├─ notify ON  ──▶ origin channel gets the request rendered with its
  │                 approval primitives (buttons / ALLBERT:APPROVE:<id>)
  └─ notify OFF ──▶ visible in web/TUI approval surfaces (as today) and
                    surfaced with pending_reports on the next turn
user approves (any surface, existing ConfirmationCallback path, identity
re-proved) ──▶ confirmation resolves ──▶ RunServer resumes the parked step;
siblings were never blocked
```

## §F — Failure, Crash, and Rehydration Flows

- **Run crash:** RunServer dies → Coordinator `:DOWN` → crash event and
  attempt count recorded. ONE restart is allowed only for a step whose action
  contract proves retry safety/idempotency. A possibly committed external
  effect without a durable observation becomes
  `blocked`/`uncertain_effect`; explicit retry or skip is required. A second
  crash → child `failed`.
- **Coordinator crash:** children keep running (they are supervisor
  siblings, not linked); boot-style rehydration logic restarts the
  coordinator from Objectives + Registry lookups; monitors re-established;
  join still honest (reduction is durable-state-driven).
- **BEAM restart mid-fan-out:** boot rehydration (engine `rebuild_state/1`
  extension) finds parents with non-terminal children inside the rehydrate
  window → coordinators restart → runs resume from last durable step;
  outside the window the existing stale-abandonment path applies and the
  report says so.
- **Backpressure:** children beyond `max_concurrent_runs` stay durable
  `open`; FIFO positions are persisted, so restart reconstructs the same
  order. Status shows queued positions; a cancelled child never starts.
- **Notify failure:** adapter delivery error → `notify.failed` + audit; one
  bounded retry after the min-interval; then suppressed with reason (report
  still reaches the user via next-turn fallback — delivery failure never
  loses the report).
- **Turn-timeout interplay:** the fan-out turn itself only frames + acks
  (well under the 120s channel default); child runs are not bound by the
  turn deadline (they are bounded per-step by action/delegate timeouts and
  by cancellation).
- **Join outcome:** all completed → `completed/success`; any completed plus
  failed or cancelled → `completed/partial`; no completed and any failed →
  `failed/failed`; all cancelled → `cancelled/cancelled`. Reports enumerate
  every child.

## §G — Data Flow And Authority Invariants

- Durable truth: Objectives rows/steps/events (SQLite). Process state is a
  rebuildable projection (ADR 0083); signals are notifications, not storage.
- Kickoff receipts are opaque, identity-bound, single-use continuation
  handles, not authority. Acknowledgement only makes framed children runnable;
  it never grants an action, approves confirmation, or weakens policy.
- New authority surface: EXACTLY ONE — autonomous notify (ADR 0084),
  settings-gated per channel, default OFF, enforced at `Channels.Notify`,
  audited, redacted. Kickoff acks and next-turn reports are turn responses,
  not autonomous sends. Attached web/TUI push rendering is outside the class.
- Not authority: decomposition output, fan-out/run/objective/step ids,
  steering directives, supervised processes, signals. Every effectful step
  goes through registered actions with unchanged confirmation classes
  (`Actions.Registry` + `Runner.run/3`).
- `send_channel_message` (ADR 0063) keeps `confirmation: :required` —
  operator-initiated compose and autonomous notify are different classes on
  the same `deliver_outbound/3` transport.
- Notify targets are re-derived from the identity map + origin thread refs at
  send time; stored text never becomes an address. Identity-unmapped targets
  are unreachable.
- Cancellation kills only processes the run spawned (os_pid/process-group
  captured at spawn, ADR 0085); OTP supervision is not a security boundary.

- Runtime callers share the delivery acknowledgement contract: the nine chat
  surfaces, `mix allbert.ask`, ACP/OpenAI-compatible controllers, and Jobs.
  Callers without it fail closed to `:single`. API presentation:
  OpenAI-compatible/ACP requests hold until join (bounded by the request
  timeout; SSE clients stream status then the report; timeout falls back to
  ack + `pending_reports`) — Locked Decision 17.
- `mix allbert.test release.v11` — release gate (v1 prefix ++ v11 focused);

  objectives actions and `send_channel_message`, unchanged).
