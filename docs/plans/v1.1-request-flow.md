## §A — The Fan-Out Lifecycle (frame → deliver → acknowledge → run → join)

```
user prompt (Runtime caller declaring exact `fanout_delivery_ack_v1` capability)
  │
  ▼
Runtime.submit_user_input  ──────────────  synchronous, unchanged entry
  │
  ├─ capability absent/malformed/unsupported ──▶ single-turn pipeline
  │      (no decomposition and no durable fan-out framing)
  │
  ▼
IntentAgent turn (settings-pinned)
  │
  ├─ Stage-0 Decomposer (S3)         :single ──▶ today's pipeline, unchanged
  │      broad automatic eligibility: ≥2 independent tasks; explicit
  │      parallel language not required; uncertain/nested ⇒ :single
  │      count > max_children_per_fanout ⇒ clarify before framing; no task loss
  │      │ :fanout (task list, advisory)
  │      ▼
  ├─ Objectives.Fanout.frame/2       parent + N children (durable, one txn,
  │                                  origin attribution copied to children)
  │      │
  │      ▼
  └─ response = KICKOFF ACK + opaque start receipt
         │   children remain durable open; NOTHING executes yet
         ▼
caller delivers/records kickoff
  ├─ remote chat: transport success      ├─ HTTP non-streaming: durable record
  ├─ SSE: kickoff event flushed
  ├─ web/TUI: rendered/printed           ├─ CLI: output completed
  └─ Jobs: durable kickoff event
         │
         ├─ failure ──▶ persist delivery_blocked; retry idempotently or cancel
         ▼
Runtime acknowledges receipt (single-use, identity-bound, idempotent,
non-authoritative) ──▶ queue runnable ──▶ Coordinator starts
  (confirm_before_start=true adds its durable confirmation before start)
         │
         ▼
caller is free; duplicate acknowledgement cannot start work twice

background (per child, in parallel, supervised):
  RunServer: propose ──▶ evaluate ──▶ authorize ──▶ Actions.Runner.run/3
     ▲                │            (cancel-token checkpoints between operations)
     │                ▼
     │        observe/advance ──▶ next step … until terminal
     │                │
     │                ├─ needs_confirmation ──▶ park (step blocked +
     │                │                          confirmation_id); siblings
     │                │                          unaffected; notify kind
     │                │                          :confirmation_request (§E)
     │                └─ run.* signals on the SignalBus (status stream)
     ▼
  Coordinator (monitors): child terminal ──▶ durable status reduction
     │  … all children terminal
     ▼
  JOIN: parent terminal + enumerated per-child report (fanout.joined)
     │
     ├─ autonomous notify ON  ──▶ Channels.Notify ──▶ adapter deliver_outbound
     └─ autonomous notify OFF ──▶ pending_reports + delivery receipt on the NEXT turn +
                                  web/TUI objectives surfaces (always)
```

OpenAI-compatible/ACP presentation (Locked Decisions 17/26/29; clarified
2026-07-19 and 2026-07-22): the request HOLDS until join. For non-streaming
requests, durable server-side kickoff recording satisfies the start
precondition; for SSE, successful kickoff-event flush does. Children then
run, and the HTTP response completes with the join report, bounded
by the request timeout (timeout ⇒ kickoff returns; the report remains behind
its delivery receipt in `pending_reports`); SSE clients stream status chunks then the report.
Disconnect-before-record/flush never starts work; disconnect after durable
recording retains the report for restart-safe `pending_reports` retrieval.
The controller/server awaits through the additive fan-out continuation API,
outside `Runtime.submit_user_input/1`. OpenAI SSE is genuinely chunked:
kickoff flush → start acknowledgement → status chunks → join → `[DONE]`.
ACP prompt work runs concurrently with the stdio reader, so `session/cancel`
remains serviceable and routes to the active run. Timeout returns the kickoff;
the report remains pending until its report-delivery receipt is acknowledged.
No existing request field or terminal response shape is removed.

Status streaming while running: `run.progress` signals fan to (a) the web
workspace via the existing SignalBridge objective-event path, (b) an attached
TUI session registered ephemerally in `Channels.TUI.Subscriptions` and rendered
through its live region (no PID/function is durable), (c) remote channels ONLY through `Channels.Notify`
when `level = "status_and_completion"` (coalesced, throttled; on channels
declaring `status_update_mode: :edit_in_place` the one status message is
edited in place, with append fallback — S7b/M7).

Rules (ADR 0083 / plan Locked Decision 5): `restart: :temporary` everywhere in
the run scope (restart is a COORDINATOR decision from durable state, not a
supervisor reflex); monitors not polling; Registry-addressed steering; no bare
`Task.async` for steerable work; OS processes spawned by steps are captured as
os_pid/process groups (ADR 0085) — supervision is never a security boundary.
Delivery state/receipt digest, queue position, run-attempt count, review
reason, and join outcome are durable. Coordinator memory is a rebuildable
projection and owns no private goal-loop truth.
RunServer executes the full child lifecycle; a task title is never treated as
an already-authorized executable step. It calls `Objectives.Lifecycle`, whose
transactional/CAS operations own transitions, events, attempt policy,
Registry/Runner invocation, confirmation parking, observation, and advance;
private Engine Jido command modules are not its execution API.

Notes: "opt-in" = `channels.<id>.autonomous_notify.enabled = true` (default

with the enable hint when off) so the operator knows where results will
land; `pending_reports` renders as compact per-fan-out summaries (full
report on a status query). `status_update_mode` is a second additive
descriptor declaration (S7b/M7, Locked Decision 11): telegram/discord/

fan-out per channel and edits it, falling back automatically to an audited
append on edit failure; whatsapp (no business-initiated Cloud API edit),
signal (partial/recent CLI support), and email stay `:append_only`.
Completion reports and confirmation requests are ALWAYS new messages —
never edits — and an edit counts as a delivery for the min-interval
throttle.

## §E — Steering Flows

**Classification (advisory, S5).** Mid-flight input on a thread with an
active fan-out extends the existing `:continue_objective` seam:

```
inbound text ──▶ engine candidates (fan-out parent/children present)
  ├─ no fan-out candidate wins ──▶ normal pipeline (new request)
  └─ fan-out candidate wins ──▶ Intent.Steering.classify
        ├─ :status      ──▶ aggregate every active fan-out in thread/session
        ├─ :steer/:adjust ─▶ steer_objective_run action ──▶ directive event
        │                     (durable) ──▶ Registry send to RunServer ──▶
        │                     applied at next lifecycle-operation boundary,
        │                     never during an effect
        ├─ :cancel      ──▶ cancel_objective_run (child ref or whole fan-out)
        │                     ──▶ tiers: cooperative → supervised → OS kill
        ├─ :new_request ──▶ normal pipeline (decomposer may run)
        └─ low confidence ─▶ clarify turn (list children, ask which/what)
```

Invariants (gate-bound): the classifier is advisory; free text NEVER approves
a pending confirmation (approval stays on `ALLBERT:APPROVE`/buttons through
`ConfirmationCallback`); steering can raise but never lower a confirmation
class; an identity-unmapped sender cannot steer at all (inbound trust
unchanged); steering another user's objective is denied at the action.
Mutating steering resolves an explicit child/fan-out reference. If multiple
targets remain plausible, Allbert clarifies; it never silently chooses the
newest run.

`adjust` refines how an existing child pursues its goal (for example, shorten
the answer or add citations); `steer` redirects the goal. They remain distinct
for acknowledgement, audit wording, clarification, and accuracy metrics, but
both use `steer_objective_run` with identical ownership, authority, and
confirmation posture. Mentioning two ordinal targets is ambiguous and produces
a clarification rather than selecting the first.
Redirect/skip against a confirmation-blocked child resolves the old
confirmation as `cancelled` with reason `superseded_by_steer` and the steering
event id before replanning; the prior button/typed callback can no longer run it.

**Background-raised confirmations, out-of-band.** A run hitting
`needs_confirmation` parks its step (`confirmation_id`) and emits
`run.blocked`:

```
run.blocked ──▶ Channels.Notify kind=:confirmation_request
  ├─ notify ON  ──▶ origin channel gets the request rendered with its
  │                 approval primitives (buttons / ALLBERT:APPROVE:<id>)
  └─ notify OFF ──▶ visible in web/TUI approval surfaces (as today) and
                    surfaced with pending_reports on the next turn
user approves (any surface, existing ConfirmationCallback path, identity
re-proved) ──▶ confirmation resolves ──▶ RunServer resumes the parked step;
siblings were never blocked
```

## §F — Failure, Crash, and Rehydration Flows

- **Run crash:** RunServer dies → Coordinator `:DOWN` → crash event and
  attempt count recorded. ONE restart is allowed only for a step whose action
  contract proves retry safety/idempotency. A possibly committed external
  effect without a durable observation becomes
  `blocked`/`uncertain_effect`; explicit retry or skip is required. A second
  crash → child `failed`.
- **Coordinator crash:** children keep running (they are supervisor
  siblings, not linked); Scheduler observes `:DOWN`, restarts the temporary
  Coordinator from Objectives + Registry lookups, and re-establishes monitors;
  join still honest (reduction is durable-state-driven).
- **Scheduler crash:** its supervisor restarts it permanently; it rebuilds
  active fan-outs and fair-queue membership from Objectives and Registry
  without changing durable FIFO order.
- **BEAM restart mid-fan-out:** boot rehydration (engine `rebuild_state/1`
  extension) finds parents with non-terminal children inside the rehydrate
  window → coordinators restart → runs resume from last durable step;
  outside the window the existing stale-abandonment path applies and the
  report says so.
- **Backpressure:** Scheduler grants at most
  `max_concurrent_runs_global`, round-robin across fan-outs; each fan-out uses
  at most `max_concurrent_runs_per_fanout`. Extra children stay durable `open`;
  `max_children_per_fanout` only limits framing. FIFO positions are persisted,
  so restart reconstructs the same order. Status shows queued positions; a
  cancelled child never starts.
- **Notify failure:** adapter delivery error → `notify.failed` + audit; one
  bounded retry after the min-interval; then suppressed with reason (report
  still reaches the user via next-turn fallback — delivery failure never
  loses the report).
- **Uncertain notify:** provider acceptance with no persisted receipt →
  ledger state `uncertain`; no autonomous retry; status/next-turn retrieval
  retains the report and the markdown audit records the warning.
- **Delivery-ack uncertainty:** kickoff may have rendered but acknowledgement
  did not persist → remain blocked; operator status/retry re-renders the same
  kickoff, reuses the receipt, then acknowledges it.
- **Pending-report delivery uncertainty:** reading does not consume a report.
  Failed/uncertain transport retains the same report receipt; only successful
  transport/render/print or durable public-protocol recording acknowledges it.
- **Turn-timeout interplay:** interactive-channel Runtime work only frames +
  returns the kickoff (well under the 120s default); child runs are not bound
  by that turn deadline. OpenAI/ACP await separately through the continuation
  API and apply their public-request timeout without cancelling the fan-out.
- **Join outcome:** all completed → `completed/success`; any completed plus
  failed or cancelled → `completed/partial`; no completed and any failed →
  `failed/failed`; all cancelled → `cancelled/cancelled`. Reports enumerate
  every child.

## §G — Data Flow And Authority Invariants

- Durable truth: Objectives rows/steps/events (SQLite). Process state is a
  rebuildable projection (ADR 0083); signals are notifications, not storage.
- Kickoff receipts are opaque, identity-bound, single-use continuation
  handles, not authority. Acknowledgement only makes framed children runnable;
  it never grants an action, approves confirmation, or weakens policy.
- Report receipts are likewise identity-bound delivery handles. Reading a
  report is non-destructive; acknowledgement after successful delivery is the
  only consumption transition.
- Notify operational state lives in the additive delivery ledger; redacted
  append-only evidence lives under Allbert Home through
  `Runtime.Audit :channel_notify`. Neither grants authority.
- New authority surface: EXACTLY ONE — autonomous notify (ADR 0084),
  settings-gated per channel, default OFF, enforced at `Channels.Notify`,
  audited, redacted. Kickoff acks and next-turn reports are turn responses,
  not autonomous sends. Attached web/TUI push rendering is outside the class.
- Not authority: decomposition output, fan-out/run/objective/step ids,
  steering directives, supervised processes, signals. Every effectful step
  goes through registered actions with unchanged confirmation classes
  (`Actions.Registry` + `Runner.run/3`).
- `send_channel_message` (ADR 0063) keeps `confirmation: :required` —
  operator-initiated compose and autonomous notify are different classes on
  the same `deliver_outbound/3` transport.
- Fan-out framing persists the exact origin thread-ref id/digest,
  receiver-account ref, channel, and local user. Notify loads that exact ref
  and re-proves its current identity mapping at send time; missing,
  conflicting, cross-account, or cross-user refs are suppressed. Stored text
  never becomes an address.
- Cancellation kills only processes the run spawned (os_pid/process-group
  captured at spawn, ADR 0085); OTP supervision is not a security boundary.
  Each ProcessOwner pins `execution.cancel.grace_ms` from Settings Central at
  spawn; an explicit bounded internal proof/test override wins for that spawn.

## §H — Handles / CLI Surfaces

- Named web chat objective chips patch within the current shell to
  `destination=workspace:objectives&objective_id=<owned-id>`. The LiveView
  bounds the id and re-proves ownership before the catalog receives objective
  data. Missing, malformed/oversized, and foreign ids fail safely to the generic
  objectives panel. The count chip and `+N more` remain generic index links.

- All Runtime callers share the delivery acknowledgement contract: the nine
  chat surfaces, `mix allbert.ask` and the `allbert` CLI command groups
  (ask/chat/tui — `cli/commands.ex:186`), ACP/OpenAI-compatible controllers,
  and Jobs. Callers without it fail closed to `:single`. API presentation:
  The declaration is the closed internal request value
  `delivery_ack_capability: "fanout_delivery_ack_v1"`; it is caller capability,
  not user input, provider metadata, a public protocol field, or authority.
  OpenAI-compatible/ACP requests hold until join through the continuation API
  (bounded by the request timeout; real SSE streams kickoff/status/report;
  ACP continues reading cancellation; timeout returns kickoff while the
  report receipt remains pending) — Locked Decisions 17/26/29. Standalone
  CLI/TUI exit waits a bounded grace for a durable boundary, then invokes
  tiered cancellation and parks non-safe interruption as `uncertain_effect`;
  attached-service (`allbert serve`) work continues.
- `mix allbert.test release.v11` — release gate (v1 prefix ++ v11 focused);
  evidence under `<ALLBERT_HOME>/release_evidence/v11/`.
- Settings (Settings Central + CLI): `objectives.fanout.*`,
  `channels.<id>.autonomous_notify.*` (defaults per plan Planned Interfaces).
- Actions: `steer_objective_run`, `cancel_objective_run` (plus the existing
  objectives actions and `send_channel_message`, unchanged).
- Existing gates consumed: `fast-local`, prepush, `inventory --check-tags`,
  `release.v1`, `mix allbert.test release`, `mix allbert.test metrics`.

asking the operator to copy it. The first invocation in each mode must exit 1:

```sh
set -eu
export ALLBERT_BIN="${ALLBERT_BIN:-allbert}"
export ALLBERT_HOME="$(mktemp -d "${TMPDIR:-/tmp}/allbert-ov12.XXXXXX")"
"$ALLBERT_BIN" admin settings set permissions.command_execute needs_confirmation

ov12_run() {
  mode="$1"
  evidence="$V11_EVIDENCE_ROOT/OV-12-${OV12_HOST}-${mode}.txt"
  set +e
  request_output="$("$ALLBERT_BIN" admin cancellation-proof "$mode" 2>&1)"
  request_rc=$?
  set -e
  printf '%s\n' "$request_output" | tee "$evidence"
  test "$request_rc" -eq 1
  confirmation_id="$(printf '%s\n' "$request_output" |
    sed -n 's/.*Confirmation: \(conf_[^.[:space:]]*\).*/\1/p' | head -1)"
  test -n "$confirmation_id"
  set +e
  approval_output="$("$ALLBERT_BIN" admin confirmations approve "$confirmation_id" 2>&1)"
  approval_rc=$?
  set -e
  printf '%s\n' "$approval_output" | tee -a "$evidence"
  test "$approval_rc" -eq 0
  if [ "$mode" = session-escape ] && printf '%s\n' "$approval_output" |
    grep -q 'status=UNSUPPORTED'; then
    : # approval succeeded; the proof records an unavailable platform primitive
  else
    printf '%s\n' "$approval_output" | grep -q 'OV12 status=PASS'
  fi
}

: "${OV12_HOST:?set OV12_HOST to macos, linux-x64, linux-arm64, or wsl2}"
ov12_run cancel
ov12_run timeout
ov12_run session-escape
"$ALLBERT_BIN" admin settings set permissions.command_execute denied
```
