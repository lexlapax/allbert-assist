## Pre-Flight

Use a disposable Allbert Home and migrate the local database:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v027-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix ecto.migrate
ALLBERT_HOME="$SMOKE_HOME" mix phx.server
```

For deterministic smoke, StockSage native LLM generation may be disabled using
the existing StockSage settings path. Provider-backed smoke remains explicit
operator choice.

## Flow 1: Enter The StockSage App Surface

Operator navigates to `/stocksage` or a concrete `/stocksage/...` route.

Expected behavior:

1. The route renders a StockSage-owned LiveView, not the core `/agent` shell.
2. `StockSage.App` is discoverable through `AllbertAssist.App.Registry`.
3. The surface is declared by `AllbertAssist.App.SurfaceProvider`; navigation
   metadata and route shape agree.
4. The session scratchpad records `active_app: :stocksage`.
5. Empty states are specific and useful: no imported analyses, no queued runs,
   or no trends yet.

Failure behavior:

- Unknown route paths render the host's normal not-found behavior.
- If StockSage app registration fails validation, the host keeps running and
  reports app diagnostics rather than mounting a partial app.

## Flow 2: Run A Native Analysis From StockSage

Operator submits "analyze AAPL" from a StockSage surface.

Expected behavior:

1. Intent routing treats StockSage as the active app.
2. `StockSage.Actions.RunAnalysis` creates or joins an objective.
3. Security Central requests confirmation when required.
4. After approval, progress updates stream into the StockSage LiveView without
   a page reload.
5. The final analysis renders with real `:analysis_card` content, not the v0.26
   placeholder.
6. The page shows objective id, current status, engine label, trace/provenance
   links, warnings, and bounded failure reason if the run fails.
7. `RunAnalysis` returns validated Surface-node output for the result.

Failure behavior:

- Missing provider credentials render the bounded native failure reason from
  v0.26b.
- Python is not invoked as fallback. The operator must explicitly request
  Python or parity comparison.
- If progress events arrive after the user leaves the page, the analysis detail
  page catches up from persisted analysis/objective state.

## Flow 3: Inspect Specialist Output

Operator opens an analysis detail page.

Expected behavior:

1. Native specialist reports render through real `:agent_report_card` bodies.
2. Debate rounds render through real `:debate_round_card` bodies.
3. Parity comparison runs render through real `:parity_card` bodies.
4. Bridge/native/parity labels are visually clear and present in accessible
   text.
5. Objective step links navigate to existing objective inspection routes where
   available.

Failure behavior:

- Missing detail JSON renders a bounded "detail unavailable" state and keeps
  the analysis summary visible.
- Unknown component types are rejected by catalog validation and never render
  arbitrary HTML or model-generated UI.

## Flow 4: Namespace Declaration Is Visible But Inert

Operator or developer validates StockSage app registration.

Expected behavior:

1. `StockSage.App` declares its memory namespace.
2. App validation reports the namespace declaration as part of the app contract.
3. No action syncs lessons, reflections, or analyses into Allbert memory in
   v0.27.
4. The namespace declaration gives v0.28 something concrete to attack and audit
   before v0.29 writes through it.

## Flow 5: App-Flow UX

Operator uses StockSage on desktop and mobile viewport widths.

Expected behavior:

1. Loading states are visible during analysis start and progress updates.
2. Empty states distinguish queue, analyses, and trends.
3. Error states include bounded reasons and recovery actions.
4. Keyboard focus remains inside dialogs/confirmations and returns to the
   initiating control.
5. Mobile layout avoids hidden controls and horizontal overflow.

## Boundary With v0.30

v0.27 renders StockSage cards in StockSage-owned `/stocksage/...` surfaces and
validates Surface-node output. v0.30 later emits those proven components into
durable `/agent` canvas tiles through `canvas_ops`. v0.27 must not implement
durable canvas emission.

## Smoke Checklist

- `mix allbert.validate_app StockSage.App` passes.
- `/stocksage`, analysis, queue, and trends routes load.
- `active_app: :stocksage` is set under StockSage routes.
- Native and bridge fixture analyses render through real card bodies.
- Progress streams without page reload.
- Namespace declaration is visible; no memory sync action exists.
- Focused tests pass for LiveView routes, renderer fixtures, Surface
  validation, active app context, and app-flow UX.
- Milestone closeout runs `mix precommit`.
