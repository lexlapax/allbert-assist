# Allbert v0.41 Developer Velocity And Parallel Test Methodology

Request flow: `docs/plans/v0.41-request-flow.md`

ADR: `docs/adr/0049-development-gates-and-test-parallelization.md`

Developer guide: `docs/developer/test-strategy.md`

Status: planned.

Source: post-v0.40 closeout. The v0.40 full `mix precommit` gate took roughly
22 minutes in local development, with most time in synchronous ExUnit lanes. The
next feature milestone is deferred until Allbert has an explicit development
gate and test-parallelization methodology.

## Purpose

Make Allbert's development loop match its Elixir/OTP ambition: parallel when
resources are isolated, explicit and serial when they are not, and still strict
enough that release gates remain authoritative.

This release is about methodology and developer substrate. It does not add
operator-facing assistant capability. It prepares the implementation work that
will split the current monolithic precommit mindset into a gate matrix, classify
tests by resource ownership, create a safe path to parallel precommit lanes, and
make future implementation plans describe which work can proceed in parallel and
which work must rejoin through explicit serial gates.

## Current Baseline

Observed during v0.40 M6 closeout:

- `mix precommit` passed but took roughly 22 minutes wall-clock.
- The core app suite reported 1146 tests, 0 failures, 2 skipped.
- The web suite reported 107 tests, 0 failures.
- Plugin/channel suites reported 197 tests and 2 tests, 0 failures.
- A repository sweep found hundreds of `*_test.exs` files, many more
  `async: false` modules than `async: true` modules, and heavy use of
  `Application.put_env`, temporary Allbert Home roots, named processes, SQLite
  sandbox ownership, and filesystem cleanup.

These numbers are starting evidence, not final truth. M1 produces the
authoritative census and records exact commands, machine context, and timing.

## Design Principles

- Async eligibility is proven by owned resources, not guessed from test speed.
- SQLite-backed tests remain synchronous inside a single VM unless they run in
  isolated OS-process partitions with per-partition `DATABASE_PATH` and
  `ALLBERT_HOME`.
- `Application.put_env`, global process names, shared registry names, Req/Mox
  style global stubs, and shared filesystem roots are serial resources unless a
  test helper provides unique ownership and cleanup.
- Security evals, external runtime smokes, Docker/gate runs, and browser/LiveView
  flows are explicit lanes. They are not hidden inside a vague "precommit."
- Full release precommit remains authoritative. Faster local gates add feedback,
  they do not weaken the release bar.
- Implementation-ready plans must annotate concurrency. Milestones name
  parallel workstreams, serial barriers, focused gates, external smokes, and the
  rejoin point where evidence is checked before commit or release closeout.

## Planned Gate Matrix

Future implementation defines these gates as named commands or Mix aliases:

| Gate | Purpose | Expected shape |
| --- | --- | --- |
| `docs` | Docs-only verification | `git diff --check` plus link/reference checks. |
| `focused` | Milestone-local feedback | Explicit test file list named in the milestone plan. |
| `fast-local` | Daily local confidence | Format, Credo, compile warning gate, pure async tests, partition-safe fast lanes. |
| `serial-core` | Known global-resource tests | SQLite/shared-home/app-env/global-process lanes run intentionally serial. |
| `release` | Authoritative handoff gate | All lanes plus Dialyzer and security evals. Equivalent or stronger than current `mix precommit`. |
| `external-smoke` | Opt-in local integrations | Docker, real MCP servers, provider endpoints, browser drivers, and other machine-dependent checks. |

## Implementation-Plan Parallelization Contract

After v0.41, a plan is not implementation-ready merely because each milestone
has scope and acceptance criteria. Each milestone must also include a development
lane annotation:

- **Parallel workstreams**: docs, schemas/migrations, pure modules, UI surface
  work, focused tests, browser/manual validation, or external smoke preparation
  that can proceed independently.
- **Serial barriers**: SQLite/Repo ownership, app env mutation, shared Allbert
  Home or filesystem roots, named processes/supervisors, LiveView/Repo process
  trees, security eval harnesses, external runtimes, and release evidence.
- **Gate evidence**: focused test files, static checks, serial lane command,
  external smoke command, and whether full release precommit is required before
  the milestone commit or only at release closeout.
- **Rejoin point**: where parallel branches must be integrated, request-flow
  docs updated, drift checked against the plan, and gate evidence reviewed.

Plan audits must call out missing annotations as readiness gaps. This is the
planning-side analogue of test-lane classification: parallelism is allowed when
resource ownership and ordering are explicit.

## Milestones

### M1: Baseline Measurement And Test Inventory

Status: planned.

Work:

- Measure current gates with `--slowest-modules`, `--slowest`, and normal
  precommit timing.
- Produce a test inventory with file path, app/plugin owner, case template,
  `async` value, tags, approximate timing, and detected resource classes.
- Record current contention classes: SQLite sandbox, `Application` env, Allbert
  Home/filesystem roots, named processes/supervisors, global HTTP stubs, LiveView
  ownership, external runtime smokes.

Acceptance:

- `docs/developer/test-strategy.md` has a current baseline table.
- The inventory identifies which files are candidates for `pure_async`,
  `db_serial`, `app_env_serial`, `home_fs_serial`,
  `global_process_serial`, `external_runtime_serial`, `liveview_serial`, and
  `security_eval_serial`.

### M2: Test Taxonomy And Ownership Rules

Status: planned.

Work:

- Finalize the lane taxonomy in `docs/developer/test-strategy.md`.
- Define async-safe requirements: no shared database writes, no global app env
  mutation, unique tmp roots, unique process names, local-only stubs, and
  deterministic cleanup.
- Define serial-only requirements and how to document exceptions.
- Define the milestone planning annotation format for parallel workstreams,
  serial barriers, gate evidence, and rejoin points.

Acceptance:

- Every future test can be classified without an implementer inventing new
  labels.
- Case templates (`DataCase`, `ConnCase`, security eval case, plain
  `ExUnit.Case`) have documented default lane expectations.
- Future milestone plans can be audited for implementation readiness using the
  same annotation format.

### M3: Isolation Contract

Status: planned.

Work:

- Specify per-test and per-partition ownership for `ALLBERT_HOME`,
  `ALLBERT_HOME_DIR`, `DATABASE_PATH`, Settings Central roots, secrets roots,
  memory roots, sandbox roots, and tmp roots.
- Specify process naming rules: no global test name unless the test is serial;
  async tests use unique names or supervised anonymous processes.
- Specify cleanup rules for filesystem roots, ports, supervised processes,
  registry entries, and global env.

Acceptance:

- The contract is strong enough to support OS-process partitioning with
  `MIX_TEST_PARTITION`.
- The contract forbids tests from writing to a real operator home.

### M4: Gate Matrix Design

Status: planned.

Work:

- Design future command names and sequencing for docs, focused, fast-local,
  serial-core, release, and external-smoke gates.
- Decide which gates may run in parallel OS processes and which must be serial.
- Document how agents choose a focused gate per milestone and when full release
  precommit is mandatory.
- Update agent/developer guidance so planning passes require milestone-level
  parallelization annotations before implementation starts.

Acceptance:

- `DEVELOPMENT.md` and `AGENTS.md` describe the matrix and keep the
  no-known-failing-gate handoff rule.
- `DEVELOPMENT.md`, `AGENTS.md`, and
  `docs/developer/agent-context-map.md` route plan-readiness audits to the
  development-lane annotation contract.
- `docs/plans/v0.41-request-flow.md` shows exact planned command shapes.

### M5: Migration Plan For Tests

Status: planned.

Work:

- Convert only proven pure tests first.
- Move DB tests into partition-safe lanes only after per-partition database and
  home roots are implemented and measured.
- Keep security evals and external smokes explicit until proven isolated.
- Add a regression checklist for flaky tests introduced by async conversion.

Acceptance:

- The migration order reduces risk: pure helpers first, then filesystem-isolated
  tests, then DB partitions, then LiveView/process-heavy tests.
- Rollback is documented: any flaky lane can be moved back to serial without
  weakening release coverage.

### M6: Acceptance, Metrics, And Rollback

Status: planned.

Work:

- Define target metrics for wall-clock improvement and flake tolerance.
- Define release evidence: normal release gate, slowest-module report, and lane
  summary.
- Document rollback path for newly flaky lanes.

Acceptance:

- The later implementation can be judged with objective evidence: faster wall
  time, no weakened coverage, no hidden external dependencies, and no increase in
  flaky failures.

## Definition Of Done

v0.41 is done when Allbert has an accepted development-gate ADR, an explicit test
strategy, a measured baseline, a decision-complete isolation contract, and a
clear migration path for parallel precommit lanes plus implementation-plan
parallelization annotations without weakening the release bar.
