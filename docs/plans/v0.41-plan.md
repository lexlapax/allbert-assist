# Allbert v0.41 MCP-First Integration Pack 1

Request flow: `docs/plans/v0.41-request-flow.md`

ADR: `docs/adr/0039-mcp-first-native-plugin-second-integrations.md`

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`, post-v0.37 planning pass.

This milestone was tightened in the post-v0.37 planning pass to **MCP-server
configured panels only**, plus a single `notes/files` **native reference
plugin** that serves as the developer-onboarding scaffold. Native plugin
variants for calendar/mail/GitHub are post-1.0 follow-on (v0.41.x) and ship
only when MCP coverage proves insufficient for a specific workspace surface
or memory-namespace need.

## Purpose

Make Allbert useful for ordinary personal work without growing
provider-specific dependencies in core. Calendar, mail, GitHub, and
notes/files arrive as MCP-configured workspace panels driven by the v0.40 MCP
client. The `notes/files` integration additionally ships as a small native
reference plugin to give plugin authors a concrete starter scaffold.

## Expected Inputs

- v0.40 MCP client integration accepted through operator use: the
  `mcp_list_tools`, `mcp_list_resources`, `mcp_read_resource`, and
  `mcp_call_tool` actions, the `mcp.servers.*` settings namespace, and the
  grant-gated resource-read model (reads via a remembered Resource Access grant
  per `mcp://` scope; tool calls confirmation-gated). v0.40 validates calendar
  and mail server shapes specifically so v0.41 panels do not discover gaps.
  The v0.40 validation matrix must identify which first-wave panel data is
  exposed as MCP resources and which data is exposed only as MCP tools. v0.41
  does not assume read-like tool calls can be silently or remembered-approved.
- v0.27 app/surface contract and memory namespace declaration.
- v0.33 app intent descriptor and handoff substrate.
- v0.32/v0.34/v0.35 workspace shell with panel-zone composition.
- Existing Settings Central secrets and Settings.Fragment substrate.
- Proposed ADR 0039 accepted before implementation closeout.

## Operator-Visible Win

An operator can ask for an agenda, mail summary, GitHub overview, or notes
search from `/workspace` or a supported channel. Each integration renders a
workspace summary panel driven by configured MCP tools and resources. The
operator approves any effectful action through the normal Approval Handoff
path.

A plugin author opening `./plugins/allbert.notes_files/` sees a complete,
minimal worked example of the plugin contract: app behaviour, surface
provider, panel declarations, memory namespace, intent descriptor, registered
actions, and Settings.Fragment — small enough to read in one sitting.

## Planned Interfaces

- Workspace summary panels (one per integration) under existing
  `:canvas_panels` zone.
- Intent descriptors per integration so v0.33 handoff works (e.g., "show me
  today's agenda" → calendar panel).
- `./plugins/allbert.notes_files/`: native reference plugin.
  - `AllbertNotesFiles.Plugin` (plugin contract).
  - `AllbertNotesFiles.App` (app + SurfaceProvider contract).
  - Memory namespace declaration: `notes_files`.
  - Intent descriptor: `search_notes`, `read_note`.
  - Registered actions: `SearchNotes`, `ReadNote`, `WriteNote` (writes
    confirmation-gated).
  - Workspace panel: notes list + note detail.
  - Settings.Fragment for notes-files config.
  - Skill pack with two example skills.
- MCP-configured panels for calendar/mail/GitHub (no native plugin in v0.41):
  - MCP server `mcp.servers.calendar` (operator points at e.g. Google
    Calendar MCP server).
  - MCP server `mcp.servers.mail` (operator-chosen mail MCP server).
  - MCP server `mcp.servers.github` (official GitHub MCP server).
  - Each panel renders through the v0.40 actions: `mcp_list_tools` /
    `mcp_list_resources` for discovery, `mcp_read_resource` for summary content
    (gated by a remembered Resource Access grant the operator approves once per
    `mcp://` server scope), and `mcp_call_tool` for effectful actions
    (confirmation-gated). Panels never call MCP servers directly; every effect
    goes through `Actions.Runner.run/3`, Security Central, and confirmations.

## v0.40 Capability Matrix Dependency

Before v0.41 implementation starts, the v0.40 closeout notes must include this
matrix for each first-wave integration:

| Integration | Required panel data | Expected MCP exposure | v0.41 implication |
|---|---|---|---|
| Calendar | upcoming events, event detail, create/update event | resources for read-heavy agenda data; tools for writes | Resource-backed reads may refresh after a remembered grant. Tool-only agenda reads stay behind per-call confirmation or are parked. |
| Mail | inbox/message summaries, message detail, reply/send | resources for read-heavy message data; tools for sends/replies | Resource-backed summaries may refresh after a remembered grant. Tool-only summaries stay behind per-call confirmation or are parked. |
| GitHub | PR/issue/review summaries, repo file/readme detail, comments | resources for read-heavy repo/issue/PR data; tools for comments/mutations | Resource-backed summaries may refresh after a remembered grant. Tool-only summaries stay behind per-call confirmation or are parked. |

If a real or mock MCP server exposes essential summary-panel data only through
tools, v0.41 has three allowed choices: keep that interaction operator-triggered
with per-call confirmation, park that panel detail, or amend ADR 0038 before
implementation to introduce a narrower trust tier. It must not treat
`:mcp_resource_read` grants as authority for `mcp_call_tool`.

## Why notes/files Is The Native Reference

`notes/files` is the cheapest integration to implement natively (just file IO
plus a tiny workspace panel) and the most relatable for plugin authors
(everyone has notes). StockSage remains the depth reference for financial
workflows, native specialist agents, Python bridge integration, and parity
runs; `notes/files` is the shallow reference that proves the contract is
small enough to use as a starter scaffold.

The `notes/files` plugin is also the only native plugin in v0.41 because the
notes/files use case benefits most from a native memory namespace (operators
want notes promoted into Allbert memory) and a custom workspace panel
(MCP-only would lose the markdown-render affordance).

## Why Other Integrations Stay MCP-Only

- **Calendar**: MCP servers for Google Calendar, CalDAV, and Microsoft Graph
  exist or are emerging. Native plugin would lock Allbert into one provider.
- **Mail**: v0.16 email channel already covers the mail-as-channel surface.
  Mail-as-app via MCP avoids duplicating IMAP/SMTP code.
- **GitHub**: official GitHub MCP server is comprehensive. Native plugin
  would replicate GitHub API client surface in core.

Each can graduate to a native plugin in **v0.41.x** follow-on releases if a
workspace surface, memory namespace, or intent descriptor need proves the MCP
coverage insufficient. v0.41.x releases are post-1.0 and not freeze-blocking.

## Security Posture

Integrations are plugin-shaped (not core dependencies). MCP-configured panels
inherit the v0.40 MCP client trust tier (per ADR 0038). Native `notes/files`
plugin uses Resource Access for file IO and Security Central for
permission/confirmation. The notes namespace uses the v0.27 namespace
declaration contract; v0.46 may suggest memory promotion candidates but never
auto-promotes.

## Non-Goals

- No native plugin for calendar, mail, or GitHub in v0.41 (post-1.0
  follow-on if needed).
- No core dependency on Google APIs, GitHub APIs, or mail provider SDKs.
- No silent credential reuse across unrelated integrations.
- No automatic memory promotion from notes or mail.
- No hosted multi-user authorization model.
- No replacement of StockSage as the depth reference app.

## Definition Of Done

v0.41 is done when:

- each of calendar/mail/GitHub has a usable MCP-configured workspace panel
  driven by an operator-configured MCP server (or a clear "no MCP server
  configured" empty state);
- `./plugins/allbert.notes_files/` is a working native plugin a developer can
  copy as a starting point for their own plugin;
- at least one effectful flow per integration runs end-to-end through
  registered actions + confirmation (e.g., create calendar event, reply to
  mail, comment on PR, write note);
- focused tests/evals prove no integration bypasses Allbert's action boundary
  and no integration auto-promotes content into markdown memory.
