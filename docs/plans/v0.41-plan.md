# Allbert v0.41 Developer Velocity And Parallel Test Methodology

Request flow: `docs/plans/v0.41-request-flow.md`

ADR: `docs/adr/0049-development-gates-and-test-parallelization.md`

Developer guide: `docs/developer/test-strategy.md`

Status: planned.

Source: post-v0.40 closeout. The v0.40 full `mix precommit` gate took roughly
22 minutes in local development, with most time in synchronous ExUnit lanes. The
next feature milestone (v0.42 Tool Discovery + MCP-First Integration Pack 1) is
deferred until Allbert has an explicit development gate and test-parallelization
methodology — and until the existing suite is migrated onto it.

## Purpose

Make Allbert's development loop match its Elixir/OTP ambition: parallel when
resources are isolated, explicit and serial when they are not, and still strict
enough that release gates remain authoritative.

This release does two things. First, it defines the methodology and developer
substrate: a gate matrix, a test-lane taxonomy by resource ownership, an
isolation contract, and a milestone parallelization-annotation contract that all
future plans must follow. Second — because the methodology only pays off once the
suite uses it — v0.41 **executes the full migration of the existing test suite
onto the new lanes and gates**, leaving a clean slate for development going
forward. It adds no operator-facing assistant capability; operator features
resume in v0.42.

The migration is the dominant work of this release: it touches essentially every
test in the repository (see "Scope Of The Test Migration"). It is staged,
reversible, and validated at every step against the v0.40 regression oracle
below, so the suite never loses coverage and the release bar never drops.

## v0.40 Regression Oracle And Fallback Gate

The v0.40 closeout commit (`ed84a73`, "v0.40 M6 MCP security closeout") is the
regression oracle for this entire release. Its full `mix precommit` runs green:
1146 core + 107 web + 197 + 2 plugin/channel tests, 0 failures, 2 skipped.

This is baked into the migration:

- **Oracle.** Every implementation milestone and every migration batch must
  reproduce that exact green result through the full release gate before the
  batch is accepted. "Faster" is never accepted in exchange for fewer passing
  tests.
- **Fallback gate.** The monolithic v0.40-style serial `mix precommit` stays
  runnable throughout the migration. Development never loses a known-good gate:
  if the new lanes are mid-flight or suspect, the v0.40 serial precommit is the
  authoritative fallback.
- **Flake triage.** When a newly parallel/partitioned lane flakes, re-run the
  suspect module under the serial fallback. If it passes serially, the failure is
  a parallelism/ownership defect (fix the ownership contract or move the module
  back to serial); if it fails serially too, it is a real regression and blocks.

## Current Baseline

Observed during v0.40 M6 closeout (preliminary; M1 replaces these with an
authoritative census):

- `mix precommit` passed but took roughly 22 minutes wall-clock, and is a
  monolithic serial sequence (compile, format, Credo, then four separate `mix
  test` invocations for core, web, StockSage, and telegram/email). It does not
  run Dialyzer.
- The core app suite reported 1146 tests, 0 failures, 2 skipped; web 107; plugin/
  channel 197 and 2 — across ≈233 `*_test.exs` files (≈180 core, 16 web, 37
  plugin).
- The suite is conservative, not untagged: async is almost always explicit
  (≈195 of 196 app test files declare it) but skewed serial — about 160
  `async: false` modules versus 35 `async: true`.
- Heavy serial-resource use: ≈623 `Application.put_env` sites in tests, temporary
  Allbert Home roots, ≈21 named-process spawns, `Ecto.Adapters.SQL.Sandbox` in
  `:manual` mode (`shared: not tags[:async]`), and filesystem cleanup.
- Partition-readiness is partial, not greenfield: `config/test.exs` already
  derives `DATABASE_PATH` from env vars. `MIX_TEST_PARTITION` is referenced only
  in a comment and is not yet wired.

These numbers are starting evidence, not final truth. M1 produces the
authoritative census and records exact commands, machine context, and timing.

## Scope Of The Test Migration

The implementation reaches essentially every test (≈233 files / ≈1,450 tests;
≈160 `async: false` promotion candidates). To keep that tractable and reviewable,
the migration is **classified at the case-template boundary, not per file**:

- **Template defaults carry the lane.** Each shared case template
  (`AllbertAssist.DataCase`, web `ConnCase`, `SecurityEvalCase`, channel/plugin
  cases) sets its default lane tag and `async` default in one place. A handful of
  template edits classify the majority of files. (`SecurityEvalCase` already
  takes `async:` as an option, so this extends an existing pattern.)
- **Per-file edits only for exceptions.** Promotions (e.g. a `DataCase` test that
  is actually pure → `pure_async`) and the plain-`ExUnit.Case` tail (files using
  no shared template) get an explicit `@moduletag` override. The async flip is
  then a small, reviewable, ownership-proof diff per promoted module — not a
  blanket rewrite of 233 files.
- **Reconciliation is the exit criterion.** Lane test-counts must sum to the full
  suite total: zero unclassified files, no file double-counted. A file touching
  multiple serial resources records a primary lane plus secondary blockers and is
  excluded from async if any blocker is present.

## Design Principles

- Async eligibility is proven by owned resources, not guessed from test speed.
- SQLite-backed tests remain synchronous inside a single VM unless they run in
  isolated OS-process partitions with per-partition `DATABASE_PATH` and
  `ALLBERT_HOME`.
- `Application.put_env`, global process names, shared registry names, Req/Mox
  style global stubs, and shared filesystem roots are serial resources unless a
  test helper provides unique ownership and cleanup.
- Security evals, external runtime smokes, Docker/gate runs, and browser/LiveView
  flows are explicit lanes. They are not hidden inside a vague "precommit."
- Full release precommit remains authoritative, and the v0.40 green set is the
  regression oracle. Faster local gates add feedback; they do not weaken the
  release bar.
- Implementation-ready plans must annotate concurrency. Milestones name
  parallel workstreams, serial barriers, focused gates, external smokes, and the
  rejoin point where evidence is checked before commit or release closeout.

## Planned Gate Matrix

Future implementation defines these gates as named commands or Mix aliases:

| Gate | Purpose | Expected shape |
| --- | --- | --- |
| `docs` | Docs-only verification | `git diff --check` plus link/reference checks. |
| `focused` | Milestone-local feedback | Explicit test file list named in the milestone plan. |
| `fast-local` | Daily local confidence | Format, Credo, compile warning gate, pure async tests, partition-safe fast lanes. |
| `serial-core` | Known global-resource tests | SQLite/shared-home/app-env/global-process lanes run intentionally serial. |
| `release` | Authoritative handoff gate | All lanes plus Dialyzer and security evals. Equivalent or stronger than current `mix precommit`. |
| `external-smoke` | Opt-in local integrations | Docker, real MCP servers, provider endpoints, browser drivers, and other machine-dependent checks. |

The `release` gate adds Dialyzer, which today's `mix precommit` does not run; it
is a superset of the v0.40 precommit, never a subset.

## Implementation-Plan Parallelization Contract

After v0.41, a plan is not implementation-ready merely because each milestone
has scope and acceptance criteria. Each milestone must also include a development
lane annotation:

- **Parallel workstreams**: docs, schemas/migrations, pure modules, UI surface
  work, focused tests, browser/manual validation, or external smoke preparation
  that can proceed independently.
- **Serial barriers**: SQLite/Repo ownership, app env mutation, shared Allbert
  Home or filesystem roots, named processes/supervisors, LiveView/Repo process
  trees, security eval harnesses, external runtimes, and release evidence.
- **Gate evidence**: focused test files, static checks, serial lane command,
  external smoke command, and whether full release precommit is required before
  the milestone commit or only at release closeout.
- **Rejoin point**: where parallel branches must be integrated, request-flow
  docs updated, drift checked against the plan, and gate evidence reviewed.

Plan audits must call out missing annotations as readiness gaps. This is the
planning-side analogue of test-lane classification: parallelism is allowed when
resource ownership and ordering are explicit. The contract applies to plans
authored or audited after v0.41; existing downstream plans (v0.42 onward) get
their annotations back-filled at the start of their own implementation, and v0.42
is back-filled now as the worked example.

This plan dogfoods the contract: every milestone below carries a Lane annotation.

## Milestones

M1-M5 are the methodology/design pass (docs + measurement, no test edits).
M6-M9 are the implementation pass that migrates the suite. The split keeps the
"design the contract" work reviewable before any test is touched.

### M8: Async Promotion And Ownership Retrofits

Status: planned.

Acceptance:

a known failing/flaky lane.

## Definition Of Done

v0.41 is done when Allbert has an accepted development-gate ADR, an explicit test
strategy, a measured baseline with the recorded v0.40 oracle, a decision-complete
isolation contract, the gate matrix and isolation helpers implemented, and **the
existing test suite migrated onto the lanes** — every test classified, proven-safe
modules promoted to async/partition-safe, security evals and external smokes kept
serial — with the full release gate reproducing the v0.40 green set (plus
Dialyzer), measurable fast-local speedup, zero net-new flakes, and v0.42 carrying
development-lane annotations. The clean slate: development going forward chooses a
gate by lane, and every new plan is auditable against the parallelization
contract, without weakening the release bar.
