# v0.13 — Local personalization (LoRA / adapter training)

Status: Future

## Context

The origin note asked for a nightly pipeline that distills or retrains a small local model so Allbert's memory and personality tune to the user over time. v0.11 takes the first reviewable step: a deterministic personality digest that compiles a markdown learned overlay (`PERSONALITY.md`) from approved durable memory, approved fact memory, and bounded recent episode summaries, plus a `LearningJob` trait seam. v0.11 does not train any model and does not replace `SOUL.md`.

v0.13 is the release that actually trains. It plugs a local adapter-training job into the same `LearningJob` seam, consuming the same approved durable/fact plus bounded episode-summary corpus contract the digest already honors, plus the two distinct bootstrap persona surfaces: `SOUL.md` as baseline persona/constraints and accepted `PERSONALITY.md` as reviewed learned adaptation input. v0.12 (self-improvement) runs in parallel conceptually but addresses a different axis (Allbert authoring code and skills). v0.13 focuses on personalization: a small local model that adapts to the user's voice, preferences, and durable memory without sending any of that data to a hosted provider.

v0.13 is marked Future because it depends on:

- v0.11's `LearningJob` trait and `PersonalityDigestJob` semantics shipping and stabilising.
- v0.11's approved durable/fact plus bounded episode-summary corpus contract proving reliable in practice.
- v0.12's self-improvement posture (worktree isolation, patch-approval inbox) establishing the pattern for "Allbert changes things, operator reviews before they take effect." Adapter swaps follow the same review-first pattern.
- A decision about the local-model target and trainer toolchain that the roadmap is deliberately not pinning yet.

This file is a stub. It captures the shape of the release and the invariants it must honor; concrete milestones are written when v0.13 is the current release.

## Goal

Ship a review-first local personalization pipeline:

1. A `PersonalityAdapterJob` implementing the v0.11 `LearningJob` trait. It trains or distils a small local adapter (LoRA or equivalent) from approved durable memory, approved facts, bounded recent episode summaries, `SOUL.md` baseline persona/constraints, and accepted `PERSONALITY.md` learned adaptation input. Adapter artifacts carry `provenance: self-trained` metadata (trainer backend, base model, corpus digest hash, training date, operator acceptance status).
2. A `ModelAdapterStore` that keeps adapter artifacts, metadata, and a rollback trail under `~/.allbert/adapters/`. Adapters do not reuse v0.12's sibling-worktree infrastructure; they are distinct artifact kinds with their own isolation directory.
3. An `adapter-approval` inbox flow so the operator reviews every new adapter before it becomes the active local-model enhancement. Mirrors v0.12's `patch-approval` pattern: same review state machine, additive kind-specific fields (base model, adapter rank, corpus digest, eval summary).
4. Local-first execution: no training data leaves the machine unless the operator explicitly opts into a hosted training provider.
5. A trainer-backend allowlist distinct from the LLM provider seam (ADR 0066). The trainer is a separate executable (mlx, llama.cpp, peft sidecar, or equivalent) invoked through the same exec-policy gates as any other external tool; the LLM provider seam is not repurposed for training.

## Non-goals

- No training on staged, unapproved, or ephemeral content.
- No automatic adapter swap of the active model. Adapter activation requires explicit operator acceptance, just like patch approvals in v0.12.
- No hosted fine-tuning by default. Any hosted-training path is opt-in and carries the same corpus-upload preview/confirm flow the v0.11 digest already uses.
- No distillation or retraining of the foundation model. v0.13 trains adapters on top of a local base, not full models.
- No dependence on the Rust rebuild skill from v0.12. Adapter training does not require recompiling the Allbert binary.

## Invariants v0.13 must preserve from v0.11/v0.12

- The `LearningJob` trait shape. `PersonalityAdapterJob` must fit the same `name()` / `describe_corpus()` / `run(ctx) -> LearningJobReport` surface without removing or renaming existing fields. Adding new report fields (trainer-backend identity, compute-wall-seconds detail, rollback handles) is explicitly permitted under v0.11's additive-only clause.
- The approved durable/fact plus bounded episode-summary corpus contract. Training inputs come from the same durable-memory, approved-fact, and working-history-derived episode-summary selection rules the digest uses.
- `SOUL.md` and `PERSONALITY.md` remain distinct. Adapter training uses `SOUL.md` as baseline persona/constraints and accepted `PERSONALITY.md` as reviewed learned adaptation input. `PERSONALITY.md` is not deprecated by v0.13, and adapter artifacts live outside both markdown files and do not share their bootstrap load path.
- Spend cap (ADR 0051) applies to any hosted-training opt-in path. A local training run is near-zero spend but non-trivial wall-clock, so v0.13 introduces a distinct compute cap (`learning.compute_cap_wall_hours`) that gates local training at the turn boundary the same way ADR 0051 gates spend. Fail-closed scheduling (ADR 0015) applies to both caps.
- `adapter-approval` inbox kind mirrors v0.12's `patch-approval` state machine. Accept/reject/install semantics, rendering contract, and artifact linkage all follow the same shape. v0.14 inherits neither directly; its code-shaped remediation reuses `patch-approval`, its adapter-shaped work (if any emerges) would reuse `adapter-approval`.
- Provenance frontmatter pattern. v0.13 adds `self-trained` as an additive enum value alongside v0.12's `self-authored`; the frontmatter slot is not reshaped.
- Review-first. v0.13 never auto-activates a new adapter, and adapter artifacts follow the self-change envelope from [ADR 0080](../adr/0080-self-change-artifacts-share-approval-provenance-and-rollback-envelope.md).

## Open design questions

- Base-model target: sticks with v0.10's default Ollama/Gemma4 or widens to any local-model provider the v0.10 seam supports?
- Trainer toolchain: mlx, llama.cpp LoRA, peft via python sidecar, or a Rust-native path?
- Adapter format: GGUF adapter, safetensors LoRA, or a provider-specific format with a conversion shim?
- Corpus shaping: how much of the durable/fact memory graph and bounded episode-summary set does the adapter need vs. what accepted `PERSONALITY.md` already compresses, while preserving `SOUL.md` as non-generated persona constraints?
- Evaluation harness: what does "this adapter is a regression" look like to a reviewer who is not an ML engineer?
- Compute-cap unit: wall-clock hours (simple, coarse) vs resident GPU-memory × time (accurate, complex)? v0.13 plan will pick the simpler default and allow the richer metric as a future option.
- Trainer-backend allowlist shape: per-backend executable allowlist under `security.exec_allow`, or a dedicated `learning.training.allowed_backends` list? The former reuses existing policy; the latter is clearer but adds surface.

These questions are intentionally not answered here. They will be resolved when v0.13 becomes the current release and the state of local-model tooling is re-examined.

## ADRs

To be written when v0.13 is the current release. Expected shapes:

- Adapter training is a `LearningJob` implementation, not a new kernel runtime.
- Adapter activation is review-first, mirroring `patch-approval`.
- Local-first training is the default; hosted training follows the digest-style corpus-upload preview/confirm flow.
- Adapter artifacts and metadata live under `~/.allbert/adapters/` with a rollback trail.
- Approved durable/fact plus bounded episode-summary corpus contract is reused verbatim from v0.11, with `SOUL.md` and accepted `PERSONALITY.md` consumed as distinct persona/adaptation inputs.
- Compute cap is a first-class policy distinct from the ADR 0051 spend cap.
- Trainer-backend identity is distinct from the LLM provider seam.
- Adapter artifacts participate in the cross-release self-change envelope from ADR 0080.

## Exit criteria (placeholder)

- `PersonalityAdapterJob` trains a fixture adapter from the approved durable/fact plus bounded episode-summary corpus, `SOUL.md`, and accepted `PERSONALITY.md`, then emits a `LearningJobReport` the operator can review.
- New adapters route through `adapter-approval` and are never auto-activated.
- The ADR 0051 daily spend cap and v0.13 compute cap gate training at the turn boundary.
- Operator docs cover corpus scope, review posture, rollback, and the hosted-training opt-in.
- Tier A validation is green.
