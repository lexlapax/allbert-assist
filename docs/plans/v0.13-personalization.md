# v0.13 — Local personalization (LoRA / adapter training)

Status: Future

## Context

The origin note asked for a nightly pipeline that distills or retrains a small local model so Allbert's memory and personality tune to the user over time. v0.11 takes the first reviewable step: a personality digest that compiles a markdown profile (`PERSONALITY.md`) from approved durable and fact memory, plus a `LearningJob` trait seam. v0.11 does not train any model.

v0.13 is the release that actually trains. It plugs a local adapter-training job into the same `LearningJob` seam, consuming the same approved-only corpus contract the digest already honors. v0.12 (self-improvement) runs in parallel conceptually but addresses a different axis (Allbert authoring code and skills). v0.13 focuses on personalization: a small local model that adapts to the user's voice, preferences, and durable memory without sending any of that data to a hosted provider.

v0.13 is marked Future because it depends on:

- v0.11's `LearningJob` trait and `PersonalityDigestJob` semantics shipping and stabilising.
- v0.11's approved-only corpus contract (durable + fact tiers, approved episode summaries) proving reliable in practice.
- v0.12's self-improvement posture (worktree isolation, patch-approval inbox) establishing the pattern for "Allbert changes things, operator reviews before they take effect." Adapter swaps follow the same review-first pattern.
- A decision about the local-model target and trainer toolchain that the roadmap is deliberately not pinning yet.

This file is a stub. It captures the shape of the release and the invariants it must honor; concrete milestones are written when v0.13 is the current release.

## Goal

Ship a review-first local personalization pipeline:

1. A `PersonalityAdapterJob` implementing the v0.11 `LearningJob` trait. It trains or distils a small local adapter (LoRA or equivalent) from an approved-only corpus.
2. A `ModelAdapterStore` that keeps adapter artifacts, metadata, and a rollback trail under `~/.allbert/adapters/`.
3. An `adapter-approval` inbox flow so the operator reviews every new adapter before it becomes the active local-model enhancement. Mirrors v0.12's `patch-approval` pattern.
4. Local-first execution: no training data leaves the machine unless the operator explicitly opts into a hosted training provider.

## Non-goals

- No training on staged, unapproved, or ephemeral content.
- No automatic adapter swap of the active model. Adapter activation requires explicit operator acceptance, just like patch approvals in v0.12.
- No hosted fine-tuning by default. Any hosted-training path is opt-in and carries the same corpus-upload preview/confirm flow the v0.11 digest already uses.
- No distillation or retraining of the foundation model. v0.13 trains adapters on top of a local base, not full models.
- No dependence on the Rust rebuild skill from v0.12. Adapter training does not require recompiling the Allbert binary.

## Invariants v0.13 must preserve from v0.11/v0.12

- The `LearningJob` trait shape. `PersonalityAdapterJob` must fit the same `name()` / `describe_corpus()` / `run(ctx) -> LearningJobReport` surface without reshaping it.
- The approved-only corpus contract. Training inputs come from the same durable-memory + approved-fact + approved-episode selection rules the digest uses.
- `PERSONALITY.md` remains the human-readable profile. Adapter training uses it as one input and as an evaluation anchor; it is not deprecated by v0.13.
- Daily cost cap (ADR 0051) and fail-closed scheduling (ADR 0015) apply unchanged. A local job has near-zero hosted cost but still accrues compute budget.
- Review-first. v0.13 never auto-activates a new adapter.

## Open design questions

- Base-model target: sticks with v0.10's default Ollama/Gemma4 or widens to any local-model provider the v0.10 seam supports?
- Trainer toolchain: mlx, llama.cpp LoRA, peft via python sidecar, or a Rust-native path?
- Adapter format: GGUF adapter, safetensors LoRA, or a provider-specific format with a conversion shim?
- Corpus shaping: how much of the durable-memory graph does the adapter need vs. what `PERSONALITY.md` already compresses?
- Evaluation harness: what does "this adapter is a regression" look like to a reviewer who is not an ML engineer?

These questions are intentionally not answered here. They will be resolved when v0.13 becomes the current release and the state of local-model tooling is re-examined.

## ADRs

To be written when v0.13 is the current release. Expected shapes:

- Adapter training is a `LearningJob` implementation, not a new kernel runtime.
- Adapter activation is review-first, mirroring `patch-approval`.
- Local-first training is the default; hosted training follows the digest-style corpus-upload preview/confirm flow.
- Adapter artifacts and metadata live under `~/.allbert/adapters/` with a rollback trail.
- Approved-only corpus contract is reused verbatim from v0.11.

## Exit criteria (placeholder)

- `PersonalityAdapterJob` trains a fixture adapter from an approved-only corpus and emits a `LearningJobReport` the operator can review.
- New adapters route through `adapter-approval` and are never auto-activated.
- Daily cost/compute caps gate training at the turn boundary.
- Operator docs cover corpus scope, review posture, rollback, and the hosted-training opt-in.
- Tier A validation is green.
