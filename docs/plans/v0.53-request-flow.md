## Main User/Operator Flows

- Operator first retro-validates existing Telegram + email channels to
  Discord/Slack parity: Telegram bot token + chat/user ids; email SMTP/IMAP
  mailbox + app password; doctors; per-provider external smokes; manual
  inbound/outbound/approval/rejection/poll-resume checks; cleanup.
- Operator pairs/configures a new channel: Matrix (bot user + access token),
  WhatsApp Cloud API when a working WABA/phone-number-id is available (app
  secret + a public webhook URL / tunnel), and Signal (link the `signal-cli`
  daemon to a number via QR — an audited operator action).
  All secrets are `secret://` refs served through KeyCustody.
- Inbound: Telegram long-poll and email IMAP for M5 validation; Matrix via
  `/sync`; WhatsApp via the signed webhook; Signal via the daemon JSON-RPC
  control channel. Each resolves external identity → local `user_id`,
  records provider thread/message refs with a `trust_class`, and submits through
  `Runtime.submit_user_input/1`.
- Allbert replies through the adapter, placing the reply per the channel's
  `threading:` capability (degradation ladder) and rendering approvals through
  `Approval.Handoff`'s highest-fidelity primitive. Every v0.53 descriptor keeps
  ADR 0016's mandatory `:list` fallback even when the provider also supports
  `:button`, `:typed_command`, or `:link`.
- Unified history view aggregates a canonical thread across channels but
  **excludes `:e2ee_origin` (Signal) content from a different channel by
  default**; resuming E2EE-origin content onto a weaker class requires an audited
  operator confirmation (ADR 0059).

## Data Flow

```text
inbound (Telegram getUpdates | Email IMAP | Matrix /sync |
         WhatsApp signed webhook | Signal daemon JSON-RPC)
  -> WhatsApp only:
     GET verification answers hub.challenge without runtime authority
     POST enters the shared HTTP ingress controls (body cap, secure headers,
       provider/install rate-limit bucket) — REUSED from v0.51
     NET-NEW: body reader caches RAW body in conn.private before Plug.Parsers
       (v0.51's reader streams/discards it)
     NET-NEW: PublicProtocolWebhookAuth (NOT the bearer-token PublicProtocolAuth)
       verifies X-Hub-Signature-256 = crypto.mac(:hmac,:sha256,app_secret,raw)
       with secure_compare, BEFORE parse; app_secret via Settings.KeyCustody
  -> Channels.<Provider>.Parser: normalize, redact (phone numbers, tokens),
     derive receiver_account_ref + provider_thread_ref + provider_thread_key,
     stamp trust_class (Signal :e2ee_origin; WhatsApp/Matrix :server_readable)
  -> ChannelThread.echo?/1 (drop Allbert's own redelivered outbound)
  -> allowlist + identity resolution (Signal keyed on ACI, not phone)
  -> ChannelThread.lookup_thread -> Runtime.submit_user_input/1
       (canonical thread_id; provider ids never authority)
  -> Security Central -> Resource Access -> confirmations
  -> Runtime.Redactor -> Renderer (threading: degradation ladder;
     reply_key_type :timestamp for Signal; quote_ttl_ms degrade for WhatsApp;
     resolver returns declared/effective threading, reply key, quote window,
     and degradation reason)
  -> Client outbound (Matrix PUT send | WhatsApp Cloud API | Signal daemon)
  -> ChannelThread.link_thread + record_message_ref (direction=out)

secrets: every token via Settings.KeyCustody.fetch/1 (decrypt-once, :sensitive)

unified view / resume (ADR 0059 gating):
  show_thread -> exclude cross-channel :e2ee_origin by default (audited opt-in)
  resume_thread_on_channel -> downgrade to weaker trust_class requires audited
    operator confirmation; never silent live mirroring
```

## Authority Invariants

- Phone number, ACI/PNI, device session, MXID, WABA id, and provider thread/
  message ids are **never** permission authority. Channel metadata never grants
  action permissions; effectful work routes through registered actions, Security
  Central, confirmations, traces, audits.
- Adapters invent no new permission classes or confirmation shapes
  (`:channel_message_inbound` from v0.52 applies).
- **KeyCustody (ADR 0058):** secret material decrypted once in one supervised
  process with `:sensitive` enabled, closure-backed state, masked status, custom
  `Inspect`, and audited fetches. M1 verified local `:sys.get_state` can still
  return GenServer state on this OTP, so the enforced guarantee is no raw secret
  material in returned/inspectable state, not denial of same-VM introspection.
  No claim of locked/zeroed/protected memory or in-VM module isolation.
- **`signal-cli` daemon:** control endpoint is local-only (preferred `0600` UNIX
  socket; loopback TCP/HTTP only with auth/ACL controls); key files `0600` under
  `<ALLBERT_HOME>/signal/`; pairing audited; daemon is transport, not authority.
- **Webhook:** raw-body HMAC verified before parse; absent/invalid rejected
  before parser/runtime; challenge handshake grants no authority.
- **Trust-class (ADR 0059):** `:e2ee_origin` content excluded from the default
  cross-channel unified view (audited per-request or Settings Central opt-in);
  resume-downgrade confirmed + audited. Trust metadata never grants authority.
- Phone numbers redacted everywhere, including durable `channel_events`,
  traces/audits, and unified-history content/refs; Signal identity links key on
  ACI UUID (`aci:<uuid>` normalized), not phone/PNI.
