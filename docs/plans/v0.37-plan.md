# Allbert v0.37 Templated Creation: Plugins, Apps, Tools, and Code Patterns

Request flow: `docs/plans/v0.37-request-flow.md`

ADRs:

- `docs/adr/0017-allbert-plugin-contract.md` (plugin contract)
- `docs/adr/0015-allbert-app-contract-and-surface-dsl.md` (app/surface contract)
- `docs/adr/0032-dynamic-plugin-generation-and-sandboxed-loading.md` (sandbox +
  gated integration this milestone reuses)
- `docs/adr/0035-codegen-agents-and-live-integration-loader.md` (the generation
  engine this milestone builds on)
- `docs/adr/0036-templated-creation-and-pattern-registry.md` (this milestone's
  template/pattern registry and creation surfaces)

v0.37 is the **templated creation** milestone. Where v0.36 is the dynamic,
LLM-driven generation/integration *engine*, v0.37 is the curated,
deterministic, operator- and developer-facing *creation experience*: a registry
of vetted templates (plugin, app, LLM tool, and an extensible "templated code
pattern" system) exposed through Mix tasks, operator-facing workspace flows, and
a Canvas creation surface. Templates produce artifacts to the proven v0.27–v0.35
contract shapes and reuse the v0.36 sandbox/gate/loader to validate and,
optionally, integrate them live.

## Purpose

A developer should be able to scaffold a correct plugin/app/tool from a vetted
template in one command, and an operator should be able to create one from a
guided workspace flow — without hand-writing boilerplate or risking the
open-ended LLM path. v0.36 proved that Allbert can generate, trial, gate, and
integrate code; v0.37 makes the *common, well-known* creation patterns
first-class, deterministic, and low-risk, with two output modes:

- **Developer scaffold** (default, safe): write source under `./plugins/<name>/`
  for human review, compile, and test — the classic generator path; no live
  integration.
- **Operator templated creation**: a parameterized template runs through the
  v0.36 sandbox trial + warning gate + operator-confirmed live integration
  (ADR 0032/0035), so an operator can stand up a templated capability live.

## Expected Inputs

- v0.36 generation engine: the OS sandbox backend, warning gate, and the gated
  in-core integration loader (ADR 0032/0033/0035) — v0.37 reuses these; it adds
  no new isolation or integration authority.
- v0.27–v0.35 contract shapes: plugin manifest, `AllbertAssist.Action` actions,
  `SurfaceProvider` panels + `intent_descriptors` (v0.33), settings fragments,
  memory namespace, objectives, v0.34 destinations, and v0.35 theming/layout.
- v0.25 native-agent + Jido.AI pattern (the basis for the LLM-tool template).
- v0.34 Canvas/destination model and v0.31 extension registry (for the Canvas
  creation surface and template discovery).

## Design Decisions

### 1. A template/pattern registry, not hard-coded generators
Templates are vetted, parameterized skeletons resolved deterministically
(parameter substitution into reviewed files), discovered through a
`TemplatePattern` behaviour and registry. Shipped patterns:

- **plugin** — minimal source-tree plugin (manifest, plugin module, skill root,
  diagnostics, validation docs, compile-path/integration notes);
- **app** — app plugin with the full app/surface contract layers (app module,
  supervision wiring, sample `AllbertAssist.Action`, `SKILL.md`, panel surface,
  optional page-surface notes, settings fragment stub, v0.34 destination
  metadata, intent descriptor stub, memory namespace stub, optional Ecto domain
  stub, optional objective scaffold, canvas wiring stub, theming/layout docs);
- **LLM tool** — a Jido.AI-backed tool/agent (model profile, prompt files,
  `AllbertAssist.Action` wrapper, advisory boundary) reusing the v0.25 pattern;
- extensible **templated code patterns** — future patterns register through the
  same behaviour without new milestones.

Templates and parameters are metadata; they grant no authority.

### 2. Mix tasks for developers
- `mix allbert.gen.plugin NAME [opts]` → `./plugins/<name>/`.
- `mix allbert.gen.app NAME [opts]` → app plugin with the contract layers.
- `mix allbert.gen.tool NAME [opts]` → an LLM-tool scaffold.
- `mix allbert.gen.<pattern> NAME [opts]` → any registered pattern.
- `mix allbert.validate_app NAME` passes on first run.
Scaffolded source is inert: no automatic compile-path changes, trust, skill
enablement, route authority, permission grants, or execution authority.

### 3. Operator-facing creation flows
In `/workspace`, a guided "create from template" flow: pick a template → fill
parameters (validated) → preview the generated tree → validate (reuse v0.36
sandbox/gate) → choose developer-scaffold output or operator live integration
(reuse the v0.36 loader, operator-confirmed). All effectful steps stay behind
registered actions and Security Central.

### 4. Canvas creation surface
A dedicated Canvas destination — a **Create** launcher entry (`workspace:create`)
— renders the template gallery, the parameter form, the preview, and
validation/integration status, composed from existing v0.34 panels/destinations
and v0.35 layout. It is view/compose only; creation runs through the registered
actions above.

### 5. Reuse v0.36 safety; add none of its risk by default
v0.37 introduces no new isolation, hot-load, or integration authority. The
developer-scaffold mode never integrates. The operator live-integration mode is
exactly the v0.36 gated path (sandbox trial + warning gate + operator confirm +
audited, reversible loader). Templates being deterministic and vetted makes this
the lower-risk creation path versus v0.36's open-ended LLM generation.

### 6. Language-pluggable, Elixir first
Templates target the v0.36 `CodeGenTarget` abstraction; v0.37 ships Elixir/OTP
templates. Other-language templates remain parked (Scripting Engine / Execution
Sandboxes future work).

## Planned Interfaces

- `AllbertAssist.Templates`: pattern registry + lifecycle (list, render,
  validate, scaffold, create).
- `AllbertAssist.Templates.Pattern`: `TemplatePattern` behaviour (params schema,
  file set, target shapes).
- Built-in patterns: `Templates.Plugin`, `Templates.App`, `Templates.Tool`.
- Mix tasks: `Mix.Tasks.Allbert.Gen.{Plugin,App,Tool}`,
  `Mix.Tasks.Allbert.ValidateApp`.
- Registered actions: `RenderTemplate`, `ScaffoldTemplate` (developer output),
  `CreateFromTemplate` (operator live integration via v0.36 loader; confirmed).
- Canvas: a `workspace:create` destination + template-gallery/parameter panels.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless a narrower gate is recorded with why.

### M1: ADR + template/pattern registry
Accept ADR 0036; add the `TemplatePattern` behaviour and `AllbertAssist.Templates`
registry; pin the params schema and target shapes. Tests: registry/list/validate.

### M2: Plugin + app templates and Mix tasks
Ship the plugin and app templates and `mix allbert.gen.plugin`/`gen.app` +
`validate_app`. Tests: generated tree, `validate_app` passes first run, inert
output, traversal/size denial.

### M3: LLM-tool template
Ship the LLM-tool template (Jido.AI model profile + prompts +
`AllbertAssist.Action` wrapper, advisory boundary) and `mix allbert.gen.tool`.
Tests: generated tool scaffold, advisory boundary, no auto-enable.

### M4: Operator creation flow + Canvas surface
Add the `workspace:create` destination, template gallery, parameter form,
preview, and the developer-scaffold vs operator-live-integration choice; wire
`RenderTemplate`/`ScaffoldTemplate`/`CreateFromTemplate` actions. Tests:
workspace flow, validation, view-only surface, no integration without confirm.

### M5: Live templated integration via v0.36
Operator live integration of a templated artifact reuses the v0.36 sandbox/gate/
loader and is operator-confirmed and reversible. Tests: gated integration,
rollback, and that templated creation cannot bypass the v0.36 gate.

### M6: Security evals and closeout
Add evals for template parameter injection, traversal, authority bypass,
ungated/unconfirmed templated integration, and Canvas-surface authority. Update
README, roadmap, agent map, future-features, ADR pointers. Bump version metadata
to `0.37.0`.

## Non-Goals

- No new isolation, hot-load, or integration authority beyond v0.36; templated
  live integration is exactly the v0.36 gated path.
- No remote template marketplace or remote code-bearing templates.
- No autonomous creation; operator/developer initiates every template run, and
  live integration is operator-confirmed.
- No dependency/migration/NIF additions in generated templates.
- No multi-language templates beyond Elixir/OTP (parked).

## Definition Of Done

v0.37 is done when a developer can scaffold a plugin, app, or LLM tool from a
vetted template via Mix tasks (with `validate_app` passing first run), an
operator can create one through a guided `/workspace` Canvas flow, registered
templated patterns are extensible through the `TemplatePattern` behaviour, and
operator live integration reuses the v0.36 gated sandbox/warning-gate/loader path
with security evals proving the bypass paths fail closed.
