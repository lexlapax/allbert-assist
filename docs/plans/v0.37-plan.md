# Allbert v0.37 Dynamic Code & Config Generation and Live Capability Integration

Request flow: `docs/plans/v0.37-request-flow.md`

ADRs:

- `docs/adr/0032-dynamic-plugin-generation-and-sandboxed-loading.md` (untrusted
  trial + gated in-core integration)
- `docs/adr/0033-capability-gap-acquisition-and-trust-tiers.md` (capability gap,
  trust tiers incl. `:integrated`)
- `docs/adr/0035-codegen-agents-and-live-integration-loader.md` (code-gen agents
  and the live-integration loader)
- `docs/adr/0037-elixir-otp-sandbox-backend-and-gate-runner.md` (v0.36 sandbox
  prerequisite)

v0.37 is the self-extending-runtime milestone. It consumes the v0.36
Elixir/OTP sandbox and gate runner, detects a capability gap, generates
**Elixir/OTP code and config** to the proven v0.27-v0.35 contract shapes,
stores the draft as file-backed local metadata under Allbert Home, trials it in
the v0.36 sandbox, and — only after the sandbox gate passes and the operator
confirms — hot-loads and registers it into the live core node without a restart,
with audit and rollback.

The deterministic, template-driven developer/operator creation experience (Mix
tasks, operator flows, Canvas Create surface, scheduled/chron pattern templates)
is v0.38 and reuses the v0.37 engine.

## Purpose

When a request needs a capability no existing plugin/app/action provides,
Allbert should be able to build that capability and make it live — but only
through a gate that keeps generation advisory and authority explicit. v0.37
delivers the full loop:

1. capability gap;
2. advisory generation;
3. v0.36 sandbox trial and warning gate;
4. operator-confirmed trust grant;
5. live loader registration;
6. use through normal action/app boundaries;
7. rollback.

## Safety / Trust Model

### 1. v0.36 remains the untrusted boundary

Generated Elixir/OTP code is compiled, tested, and trialed only through the
v0.36 sandbox/gate runner until integration is explicitly confirmed. v0.37 does
not add a second sandbox path, hidden BEAM node boundary, or host-process
fallback.

### 2. Code-gen agents are advisory

The Planner, Author, TrialAuthor, Critic, and Repair agents may propose source,
config, tests, and repair patches. Their output cannot enable the workflow,
advance trust tiers, bypass the gate, authorize integration, grant
permissions, create routes, or set `active_app`.

### 3. The trust grant is narrow

An artifact is eligible for integration only when:

- v0.36 sandbox trial passed;
- v0.36 in-sandbox warning gate passed; before v0.37 implementation starts,
  v0.36 M10's `ALLBERT_DOCKER_FULL_GATE_TEST=1` smoke should be green on a
  local Docker host to prove the real umbrella full-gate path, not only a
  trivial compile fixture;
- v0.37 static/integrity checks passed;
- the operator explicitly confirms integration through Security Central.

Only that combination permits transition to `:integrated`.

### 4. Trusted loader recompiles reviewed source

The live loader recompiles the operator-reviewed source in core with an
integrity hash. It does not load opaque sandbox-built binaries. It registers
only manifest-declared modules/actions/apps/children/destinations and records
audit events for compile, load, register, and rollback.

### 5. Existing authority boundaries still apply

Dynamically integrated actions resolve through the actions registry overlay and
execute through `AllbertAssist.Actions.Runner.run/3`. App surfaces register
through `App.Registry` and v0.34 Canvas/destination boundaries. Settings writes,
permissions, confirmations, Resource Access, and app-scope rules remain
unchanged.

## Design Decisions

### 1. Generation targets Elixir/OTP code and config

v0.37 generates only Elixir/OTP artifacts to the proven project shapes. Other
language targets, package-manager installs, migrations, NIFs, ports, and broad
shell automation stay parked.

Generated artifacts may include plugin/app manifest and modules,
`AllbertAssist.Action` actions, panel surfaces, intent descriptors, settings
fragments, memory namespace stubs, objective wiring, and theming/layout docs.

### 2. Code-gen agent committee owns authoring, not authority

Add advisory `Jido.Agent` agents:

- `Codegen.Planner`
- `Codegen.Author`
- `Codegen.TrialAuthor`
- `Codegen.Critic`
- `Codegen.Repair`

Raw prompts/completions are redacted or omitted from traces; structured
provenance and bounded diagnostics remain.

### 3. Draft lifecycle consumes v0.36 reports

Draft state is file-backed local data, not a new database table. v0.37 stores
drafts under `<ALLBERT_HOME>/dynamic_plugins/drafts/<slug>/` with:

- `metadata.yaml` (tier, provenance, source hash, selected target shapes,
  static-validation status, operator decisions, timestamps);
- generated source/config/test files under reviewed subdirectories;
- sandbox reports copied from v0.36;
- bounded diagnostics and repair history.

The untrusted draft root is deliberately separate from
`<ALLBERT_HOME>/plugins/<slug>` so ordinary plugin discovery cannot mistake a
draft for a trusted plugin. Integration may copy or materialize a reviewed
artifact into the dynamic integration root only after the trust gate.

Draft tiers:

- `:draft`
- `:sandbox_compiled`
- `:sandbox_trialed`
- `:gate_passed`
- `:integrated`
- `:rolled_back`
- `:discarded`

The first four tiers are evidence states. They grant no live authority. Only the
operator-confirmed loader transition reaches `:integrated`.

### 4. Loader is explicit and reversible

`AllbertAssist.DynamicPlugins.Loader` integrates an approved artifact:

- verifies source hash and manifest-declared module set;
- enforces generated namespace rules (`AllbertAssist.DynamicPlugins.Generated`
  or another reviewed generated namespace), and rejects replacement of core,
  static plugin, or shipped app modules;
- rejects undeclared modules, route-page integration without restart, or
  forbidden constructs, including protocols, router edits, application env
  changes, migrations, dependency additions, NIFs, ports, and package-manager
  hooks;
- recompiles reviewed source in core;
- registers actions through a runtime overlay;
- registers apps/panels/destinations through runtime registries;
- starts declared children through a `DynamicSupervisor`;
- writes audit events;
- rolls back by unregistering actions/apps/children and purging loaded modules.

Rollback's hard guarantee is authority removal: actions no longer resolve,
apps/destinations disappear from registries, and children stop. BEAM module
purge/delete is attempted and audited, but is best effort because loaded code
may be retained by the VM while old references drain.

Rollback, like integration, requires an explicit operator confirmation through
Security Central.

### 5. Route pages remain a documented limit

Panel/destination apps integrate live. Compile-time Phoenix router page routes
require restart or a later route-bridge plan. The loader must reject or mark
route-page artifacts as restart-required rather than silently half-integrating.

### 6. Actions overlay has collision and precedence rules

`AllbertAssist.DynamicPlugins.ActionsOverlay` sits beside the existing static
and plugin action registry paths. It must:

- deny action-name collisions with static, plugin, app, or other dynamic
  actions;
- deny app/plugin id collisions;
- expose diagnostics for rejected dynamic contributions;
- preserve `Actions.Runner.run/3`, app-scope, confirmation, Resource Access,
  and Security Central behavior;
- unregister entries atomically during rollback.

Dynamic actions never outrank static actions. Collision is denial, not
shadowing.

### 7. Settings Central owns workflow policy

New settings are default-off and audited:

| Key | Type | Default | Meaning |
|---|---:|---:|---|
| `dynamic_codegen.enabled` | boolean | `false` | Master switch for draft generation |
| `dynamic_codegen.provider_profile` | string/null | null | Optional advisory model/provider profile |
| `dynamic_codegen.max_repair_iterations` | integer | bounded default | Repair loop cap |
| `dynamic_codegen.max_files` | integer | bounded default | Draft file-count cap |
| `dynamic_codegen.max_bytes` | integer | bounded default | Draft total-size cap |
| `dynamic_codegen.allowed_targets` | string list | reviewed defaults | Generated target shapes permitted |
| `dynamic_codegen.live_loader_enabled` | boolean | `false` | Separate switch for operator-confirmed live integration |
| `dynamic_codegen.retention_days` | integer | bounded default | Draft/report cleanup horizon |

Settings store enablement, bounds, and selections. Draft source, metadata,
reports, and diagnostics remain file-backed Allbert Home data surfaced
read-only.

### 8. Signals, traces, and audit

Draft and integration behavior is security-relevant and observable, so it routes
through the signal-driven runtime. Capability-gap proposal, generation, draft
tier transitions, sandbox trial/gate consumption, integration, rollback,
discard, and every denial emit signals on the signal bus and write trace/audit
records through the v0.31 audit/redaction facades. The loader's compile, load,
register, and rollback audit events (ADR 0035) are the same signal/trace path,
not a side channel. Raw prompts/completions are never traced; structured
provenance, tier, and bounded diagnostics are. Signals and audit records carry
no authority: only the operator confirmation plus the v0.36 gate grant trust.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless a narrower gate is recorded with why.

### M1: ADRs, tiers, capability-gap vocabulary, settings

Accept ADR 0032/0033 amendments and ADR 0035. Add trust tiers, capability-gap
vocabulary, default-off Settings Central switches, file-backed draft metadata
store, and draft roots under Allbert Home.

Tests: gap serialization, disabled-workflow behavior, file-backed metadata
round-trip/atomic-write behavior, trust-tier transitions, confirmation-required
security tests.

### M2: Draft output contract and static validation

Pin the generated Elixir/OTP code/config contract and static validator:
slug/path/allow-list/size/no-traversal, manifest-declared modules, forbidden
constructs, no dependency/migration/NIF/port/package-manager authority.

Tests: valid draft tree, traversal/size denial, undeclared module denial,
forbidden construct denial, generated-namespace enforcement, core-module
replacement denial.

### M3: Code-gen agent committee and repair loop

Add advisory agents and sandbox-grounded repair loop.

Tests: agents propose only; agent output cannot enable/trust/integrate; repair
uses v0.36 diagnostics; raw prompts/completions are not traced.

### M4: Sandbox trial and warning gate integration

Wire dynamic draft compile/trial/gate calls to the v0.36 sandbox facade.

Tests: sandbox unavailable blocks trial; gate pass/fail changes only evidence
tiers; failed artifacts cannot integrate.

### M5: Live integration loader and actions overlay

Add runtime-mutable actions overlay, dynamic app/panel registration, supervisor
integration, integrity checks, audit, and rollback.

Tests: operator-confirmed `:gate_passed` artifact becomes usable without
restart; rollback requires confirmation and unregisters authority surfaces;
module purge is best-effort/audited; route-page restart limit is enforced;
unconfirmed/ungated/tampered integration is denied; action collisions deny
rather than shadowing.

### M6: Security evals and closeout

Add evals for untrusted core-load attempt, gate skip, unapproved/auto
integration, loader integrity tamper, rollback correctness, exfiltration,
supply-chain denial, sandbox bypass, and advisory-approval bypass. Update
README, roadmap, agent map, future-features, AGENTS, DEVELOPMENT, and ADR
pointers. Create/update the operator and developer docs listed in
Documentation Deliverables, especially
`docs/operator/dynamic-capability-integration.md` and
`docs/developer/dynamic-plugin-drafts.md`. Bump version metadata to `0.37.0`.

## Non-Goals

- No sandbox backend implementation; v0.37 consumes v0.36.
- No live integration without v0.36 gate pass and operator confirmation.
- No integration authorized by advisory/agent output or auto-trial result.
- No package-manager execution, dependency additions, migrations, NIFs, ports,
  unrestricted network, remote marketplace, remote binary loading, dynamic
  protocol/router generation, or application env mutation.
- No multi-language target beyond Elixir/OTP.
- No templated Create destination or Mix generator UX; that is v0.38.

## Definition Of Done

v0.37 is done when Allbert can detect an objective-owned capability gap,
generate Elixir/OTP code and config through advisory agents, trial and gate it
through the v0.36 sandbox, and — only after explicit operator confirmation —
hot-load and register it live without a restart, with audit, normal action/app
security semantics, file-backed draft provenance, collision-safe actions
overlay, and operator-confirmed rollback that removes live authority. Security
evals prove untrusted, ungated, unconfirmed, tampered, shadowing, and
advisory-authorized paths fail closed. Operator docs explain enablement,
evidence review, integration, rollback, and emergency disablement; developer
docs explain the draft metadata/root, validator, loader, overlay, and plugin
discovery separation.
