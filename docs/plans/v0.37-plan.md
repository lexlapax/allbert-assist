# Allbert v0.37 Dynamic Code & Config Generation and Live Capability Integration

Request flow: `docs/plans/v0.37-request-flow.md`

ADRs:

- `docs/adr/0032-dynamic-plugin-generation-and-sandboxed-loading.md` (untrusted
  trial + gated in-core integration)
- `docs/adr/0033-capability-gap-acquisition-and-trust-tiers.md` (capability gap,
  trust tiers incl. `:integrated`)
- `docs/adr/0035-codegen-agents-and-live-integration-loader.md` (code-gen agents
  and the live-integration loader)
- `docs/adr/0037-elixir-otp-sandbox-backend-and-gate-runner.md` (v0.36 sandbox
  prerequisite)

v0.37 is the self-extending-runtime milestone. It consumes the v0.36
Elixir/OTP sandbox and gate runner, detects a capability gap, generates
**Elixir/OTP code and config** to the proven v0.27-v0.35 contract shapes,
stores the draft as file-backed local metadata under Allbert Home, trials it in
the v0.36 sandbox, and — only after the sandbox gate passes and the operator
confirms — hot-loads and registers it into the live core node without a restart,
with audit and rollback.

The deterministic, template-driven developer/operator creation experience (Mix
tasks, operator flows, Canvas Create surface, scheduled/chron pattern templates)
is v0.38 and reuses the v0.37 engine.

## Purpose

When a request needs a capability no existing plugin/app/action provides,
Allbert should be able to build that capability and make it live — but only
through a gate that keeps generation advisory and authority explicit. v0.37
delivers the full loop:

1. capability gap;
2. advisory generation;
3. v0.36 sandbox trial and warning gate;
4. operator-confirmed trust grant;
5. live loader registration;
6. use through normal action/app boundaries;
7. rollback.

v0.37 prefers existing capability over generated code, in line with the vision's
preference for declarative, validated surfaces over opaque generated code
(`docs/plans/allbert-jido-vision.md`). For a capability gap the resolution order
is: reuse or compose existing registered actions/skills/apps first; then a
declarative `AllbertAssist.Surface` over existing capabilities; and only when
neither can satisfy the gap, generate new Elixir/OTP code. Code generation is the
last resort, not the default response to a gap.

v0.37 is the bounded, non-autonomous realization of the vision's "Deferred Until
The Foundation Settles" self-recompilation item. Generation stays advisory,
integration stays operator-confirmed, and generated authority stays narrow (the
Design Decision 6 permission ceiling). "Fully autonomous skill creation with
broad execution permissions" remains deferred.

## Safety / Trust Model

### 1. v0.36 remains the untrusted boundary

Generated Elixir/OTP code is compiled, tested, and trialed only through the
v0.36 sandbox/gate runner until integration is explicitly confirmed. v0.37 does
not add a second sandbox path, hidden BEAM node boundary, or host-process
fallback.

v0.37 also owns the handoff detail between durable draft files and the v0.36
bundle: generated modules that must compile are materialized into a disposable
project-shaped staging tree under the reserved generated namespace, and those
same generated bytes are scanned before any sandbox command runs. The separate
v0.36 `drafts` and `tests` mounts are evidence inputs, not a compile path by
themselves.

### 2. Code-gen agents are advisory

The Planner, Author, TrialAuthor, Critic, and Repair agents may propose source,
config, tests, and repair patches. Their output cannot enable the workflow,
advance trust tiers, bypass the gate, authorize integration, grant
permissions, create routes, or set `active_app`.

### 3. The trust grant is narrow

An artifact is eligible for integration only when:

- v0.36 sandbox trial passed;
- v0.36 in-sandbox warning gate passed; before v0.37 implementation starts,
  v0.36 M10's `ALLBERT_DOCKER_FULL_GATE_TEST=1` smoke should be green on a
  local Docker host to prove the real umbrella full-gate path, not only a
  trivial compile fixture;
- v0.37 static/integrity checks passed;
- the operator explicitly confirms integration through Security Central.

Only that combination permits transition to `:integrated`.

### 4. Trusted loader recompiles reviewed source

The live loader recompiles the operator-reviewed source in core with an
integrity hash. It does not load opaque sandbox-built binaries. It registers
only manifest-declared read-only action modules and records audit events for
compile, load, register, and rollback. Generated app, panel, child,
destination, settings-fragment, memory-namespace, and objective-wiring targets
remain rejected live targets in v0.37.

### 5. Existing authority boundaries still apply

Dynamically integrated actions resolve through the actions registry overlay and
execute through `AllbertAssist.Actions.Runner.run/3`. App surfaces register
through `App.Registry` and v0.34 Canvas/destination boundaries. Settings writes,
permissions, confirmations, Resource Access, and app-scope rules remain
unchanged.

Generated action metadata is not trusted by declaration alone. Static reviewed
actions self-authorize inside their `run/2` bodies; v0.37 must therefore validate
generated action permissions, confirmation metadata, body call targets, and
response action metadata before integration. A generated action cannot declare a
lower permission while performing higher-authority work in its body.

## Design Decisions

### 1. Generation targets Elixir/OTP code and config

v0.37 generates only Elixir/OTP artifacts to the proven project shapes. Other
language targets, package-manager installs, migrations, NIFs, ports, and broad
shell automation stay parked.

Generated artifacts may include plugin/app manifest and modules,
`AllbertAssist.Action` actions, panel surfaces, intent descriptors, settings
fragments, memory namespace stubs, objective wiring, and theming/layout docs.
Generated objective wiring may use only the shared `AllbertAssist.Objectives`
facade or registered objective actions. Generated apps/plugins must not create
private durable goal loops, private objective tables, or private objective
engines; `objective_id` and `step_id` remain correlation data, never authority.

### 2. Code-gen agent committee owns authoring, not authority

Add advisory `Jido.Agent` agents:

- `Codegen.Planner`
- `Codegen.Author`
- `Codegen.TrialAuthor`
- `Codegen.Critic`
- `Codegen.Repair`

Raw prompts/completions are redacted or omitted from traces; structured
provenance and bounded diagnostics remain.

The committee runs on the shared `AllbertAssist.JidoBacked` substrate used by the
v0.25 StockSage native coordinator (standard AgentServer child specs, private
command routing, restart rehydration, and the `allbert.jido.debug_trace`
diagnostic gate), rather than a private agent runtime. v0.37 uses the single
Settings Central selection `dynamic_codegen.provider_profile` for the committee
instead of hard-coding provider/model names in code. Per-role model profiles such
as separate fast/capable/thinking aliases are deferred until there is a real
operator need and a settings/audit story for them.

### 3. Draft lifecycle consumes v0.36 reports

Draft state is file-backed local data, not a new database table. v0.37 stores
drafts under `<ALLBERT_HOME>/dynamic_plugins/drafts/<slug>/` with:

- `metadata.yaml` (tier, provenance, source hash, selected target shapes,
  static-validation status, operator decisions, timestamps);
- generated source/config/test files under reviewed subdirectories;
- sandbox reports copied from v0.36;
- bounded diagnostics and repair history.

Minimal metadata shape:

```yaml
schema_version: 1
slug: "weather_summary"
revision: "rev_2026_05_25_001"
tier: "draft"
producer: "codegen_committee"
provider_profile: "local"
target_shapes: ["action", "panel"]
source_hashes:
  lib/allbert_assist/dynamic_plugins/generated/weather_summary/action.ex: "sha256:..."
compiled_paths:
  - lib/allbert_assist/dynamic_plugins/generated/weather_summary/action.ex
scan_paths:
  - lib/allbert_assist/dynamic_plugins/generated/weather_summary/action.ex
budget:
  provider_calls_used: 2
  provider_usage_units_used: 1800
gate:
  status: "not_run"
  sandbox_report_id: null
static_validation:
  status: "not_run"
confirmations:
  integration_id: null
  rollback_id: null
diagnostics:
  - code: "draft_created"
    message: "Draft written; no sandbox evidence yet."
repair_history: []
timestamps:
  created_at: "2026-05-25T00:00:00Z"
  updated_at: "2026-05-25T00:00:00Z"
```

Metadata is operator-inspectable and useful for reconciliation, but it is not a
trust anchor by itself. Security Central confirmation records, current file
hashes, and the v0.36 gate report remain the authority checks for integration.

The untrusted draft root is deliberately separate from
`<ALLBERT_HOME>/plugins/<slug>` so ordinary plugin discovery cannot mistake a
draft for a trusted plugin. Integration may copy or materialize a reviewed
artifact into the dynamic integration root only after the trust gate.

Draft tiers:

- `:draft`
- `:sandbox_compiled`
- `:sandbox_trialed`
- `:gate_passed`
- `:integrated`
- `:rolled_back`
- `:discarded`

The first four tiers are evidence states. They grant no live authority. Only the
operator-confirmed loader transition reaches `:integrated`.

Draft creation is a pluggable producer. The v0.37 code-gen committee is one
producer; v0.38 templated creation is another. Everything from `:draft` onward —
the metadata store, staging, `RunDraftTrial`, `RunDraftGate`, the loader, the
actions overlay, and rollback — is producer-agnostic and records the producer in
provenance. v0.37 builds the lifecycle so a later deterministic template producer
can enter at `:draft` without the advisory committee.

### 4. Capability gaps are explicit in v0.37

v0.37 records a capability gap only from an explicit operator request to
create/generate a missing capability or from an explicit objective-owned step
proposal that no existing registered action/app can satisfy. Intent/action
planning may report that no eligible capability exists, and low-confidence
ranking may suggest generation as an option, but classifier output, rejected
candidates, or weak confidence cannot start generation by themselves.
Proactive auto-acquisition from intent confidence is deferred.

The draft lifecycle is owned by the v0.24 objective runtime, not a private loop.
Gap, generation, trial, gate, integration, and rollback are recorded as objective
steps and `objective_events` on `Objectives.Engine`'s state machine: the gap and
acquisition options surface in the propose/evaluate stages;
`GenerateDraft`/`RunDraftTrial`/`RunDraftGate` execute as advisory/evidence steps
that grant no authority; and `IntegrateDraft`/`RollbackIntegration` are the
authorize-then-execute steps gated by Security Central confirmation. The
objective's acceptance criteria reference the v0.36 gate plus operator
confirmation. `objective_id`/`step_id` are correlation only, never authority.

### 5. Draft staging has one compile and scan contract

Durable draft source remains under
`<ALLBERT_HOME>/dynamic_plugins/drafts/<slug>/`, outside ordinary plugin
discovery. For sandbox trial/gate, v0.37 builds a disposable staging project
root and materializes generated files into compile-visible, reviewed paths
inside that staged project:

- generated core modules live under the reserved namespace path for
  `AllbertAssist.DynamicPlugins.Generated.<Slug>...`;
- generated action, app, panel, and support modules must be declared in the
  draft manifest and live under that generated namespace;
- generated tests live under staged umbrella test paths and are invoked by the
  v0.36 gate through `focused_test_paths` / `security_eval_paths`, with
  `focused_test_cwd` / `security_eval_cwd` only when the reviewed shape needs a
  bundle-local subdirectory;
- settings fragments, manifests, and docs are copied as inert data and cannot
  set operator values, secrets, permissions, provider credentials, or resource
  grants.

The staged project includes the same umbrella gate configuration used by the
v0.36 default project bundle: root Mix files, formatter config, Credo config,
Dialyzer ignore config, root config, app source, and reviewed source-tree plugin
inputs needed by the current warning gate. Generated drafts must meet the real
project bar, not a reduced gate fixture.

The staging builder records a source hash for every generated file and passes
the same generated source/test files through `Sandbox.SourcePolicy.scan/2`'s
path-list form before sandbox backend resolution. The gate may compile only the
staged project tree whose generated-file hashes match the scanned draft
manifest. v0.37 can express this with the v0.36 bundle builder's
`project_paths`/`test_paths` inputs plus the path-list scanner; a v0.36 change
is required only if hash enforcement must move into the sandbox facade or
container itself. A draft that compiles from unscanned bytes, or scans bytes
that are not the ones compiled, is invalid.

Generated focused tests and security evals are functional evidence, not a
security boundary. They are authored by advisory code-gen agents and can only
show that selected behavior passed in the sandbox. Security rests on static
validation, body call-target validation, the generated-permission ceiling,
operator review, Security Central confirmation, and normal runtime boundaries.

### 6. Loader is explicit and reversible

`AllbertAssist.DynamicPlugins.Loader` integrates an approved artifact:

- verifies source hash and manifest-declared module set;
- enforces generated namespace rules (`AllbertAssist.DynamicPlugins.Generated`
  or another reviewed generated namespace), and rejects replacement of core,
  static plugin, or shipped app modules;
- rejects undeclared modules, route-page integration without restart, or
  forbidden constructs, including protocols, router edits, application env
  changes, migrations, dependency additions, NIFs, ports, package-manager
  hooks, top-level side effects, `@on_load`, dynamic compile/eval/require, and
  generated macro code that would run during trusted core compilation;
- reconciles the manifest and AST bidirectionally: every discovered `defmodule`
  must be under the generated namespace and listed in the manifest, and every
  manifest-declared module must exist in reviewed source;
- validates generated action metadata against the v0.37 permission ceiling and
  against the action body's call targets;
- reruns v0.37 static/integrity validation over the reviewed source immediately
  before trusted compile;
- recompiles reviewed source in core;
- registers actions through a runtime overlay;
- rejects generated apps, panels, children, settings fragments, memory
  namespaces, objective wiring, and route-page artifacts before core
  compilation;
- writes audit events;
- rolls back by unregistering actions and purging loaded modules.

Trusted-phase validation is default-deny and is not the v0.36 regex scanner.
Before the loader calls any in-core compile API, it parses reviewed generated
source with `Code.string_to_quoted!/2` (or an equivalent non-executing parser)
and walks the AST against an allowlist. Unknown AST forms are denial.

Allowed top-level forms are limited to reviewed `defmodule` declarations under
the generated namespace, normal `def`/`defp` function bodies, a fixed small set
of inert module attributes, and `alias`/`use`/`import`/`require` only for
reviewed allowlisted modules. v0.37's generated macro/use allowlist is
enumerated: `use AllbertAssist.Action` for generated action modules and
`use AllbertAssist.App` for generated app modules. Generated UI is declarative
`AllbertAssist.Surface` data using existing catalog components; custom Phoenix
components, LiveViews, HEEx sigils, and route modules are denied in v0.37 unless
a later plan adds an explicit reviewed allowlist. Macro options, action DSL
options, schema entries, tags, module attributes, and manifest fields must be
inert literals; compile-time expressions such as `System.get_env/1`,
`Application.get_env/2`, dynamic atom/module construction, or function calls in
macro arguments are denial.

The validator also scans all call sites inside generated function bodies,
callbacks, helper modules, action `run/2` implementations, app callbacks,
surface data builders, and child process callbacks. It allows only local calls
within the generated namespace and an explicit module/function allowlist of
side-effect-free Elixir/Allbert helpers or approved runtime facades. It denies
direct calls to protected authority surfaces including `System`, `Port`, `Code`,
`Mix`, `Application` mutation, `File`/`Path` write-delete traversal operations,
`Node`, `:rpc`, `:os`, dangerous `:erlang` functions, NIF loading, ETS or
`:persistent_term` access to core tables, direct Repo writes, Settings writes,
secret reads/writes, confirmations, Resource grants, local command runners,
package/skill execution, sandbox actions, integration/rollback/disable actions,
and any `apply/3` or dynamic dispatch escape route to those targets. Declared
permission, confirmation metadata, response action metadata, and body call
targets must agree; a mismatch is denial.

Generated action permissions have a ceiling. The hard-coded v0.37 ceiling
allows `:read_only` by default and may allow only operator-enabled,
validator-proven delegations for `:external_network`, `:memory_write`,
`:objective_write`, or `:workspace_canvas_write` through existing registered
actions/facades whose normal Security Central floors still apply. Settings may
select a subset of this ceiling but cannot expand the hard-coded ceiling.
Generated actions are hard-denied from
declaring or implementing host execution, package install, skill import/script
execution, sandbox trial, secret read/write, confirmation decision, Security
Central trust-control, dynamic integration/rollback/disablement, direct Settings
mutation, or Resource-grant mutation. Generated actions are `resumable?: false`
in v0.37; dynamic confirmation resume adapters are deferred.

Generated child processes are not live-loaded in the shipped v0.37 loader.
Future child support must keep the documented state-only rule: callbacks pass
the same call-target validator and do not create autonomous timers, network
calls, shell/package/script execution, durable goal loops, or direct
protected-subsystem writes. Effectful generated workers and private scheduled
loops are deferred.

v0.37's trusted-compile threat model is explicit: the sandbox gate plus
operator review plus AST allowlist plus runtime call-target validation plus
generated-permission ceiling defend the core node against advisory-agent
mistakes, malformed drafts, permission/body mismatch, and obvious injection
before a generated artifact is granted live authority. They are not a
tamper-proof boundary against an operator or local attacker who already has
write access to Allbert Home or the source tree; that remains the local trust
boundary.

Integration is all-or-nothing. The loader records each step it completes during
an integration attempt. If source copy, trusted validation, compile, actions
overlay registration, or audit append fails
before the artifact reaches stable `:integrated`, the loader automatically
unwinds registrations/modules from that attempt and leaves the draft at
`:gate_passed` with bounded diagnostics. Operator-confirmed rollback is needed
only after a stable `:integrated` state has been reached.

Integration copies the approved, reviewed source and metadata into
`<ALLBERT_HOME>/dynamic_plugins/integrated/<slug>/<revision>/`. Ordinary plugin
discovery still ignores this root; the live loader is the only authority that
may register from it. Each source change creates a new revision that returns to
`:draft`, reruns the sandbox gate, and requires a new operator confirmation.
v0.37 does not support in-place mutation of an integrated artifact. A same-name
upgrade must first roll back the currently integrated revision, then integrate
the new gate-passing revision with its own confirmation. Atomic supersede
without an intermediate rollback is deferred.

On boot, the loader performs reconciliation: it may re-register only artifacts
whose metadata says `:integrated`, whose source hash still matches, whose gate
record is present, whose integration confirmation is valid in Security Central's
durable confirmation/audit store, and whose current loader policy and settings
still allow live dynamic code. `metadata.yaml` may cache confirmation ids and
hashes, but it is not a trust anchor by itself. If
`dynamic_codegen.live_loader_enabled=false`, reconciliation must not register
dynamic authority surfaces. The emergency disable action turns the switch off
and clears the action overlay without deleting source. This anchors
reconciliation inside Allbert's local trust boundary; it does not defend against
an operator or local attacker who can rewrite Allbert Home and the Security
Central store.

Rollback's hard guarantee is authority removal: actions no longer resolve,
apps/destinations disappear from registries, and children stop. BEAM module
purge/delete is attempted and audited, but is best effort because loaded code
may be retained by the VM while old references drain.

Rollback, like integration, requires an explicit operator confirmation through
Security Central.

Integration and rollback confirmations are restricted to high-trust operator
surfaces. The confirmation subsystem (ADR 0008/0016) is channel-aware, so the
ordinary `approve_confirmation` path can otherwise be resolved from CLI,
LiveView, Telegram callbacks, email typed commands, or any cross-channel
approver. Because integration hot-loads code into the core node, v0.37 confines
`IntegrateDraft`/`RollbackIntegration` approval to the surfaces named by
`dynamic_codegen.integration_approval_surfaces`, excludes low-trust channel
approval (Telegram, email) and cross-channel approval regardless of
`confirmations.allow_cross_channel_approval`, and denies and audits an approval
that arrives from a disallowed surface.

### 7. Route pages remain a documented limit

Panel/destination apps integrate live. Compile-time Phoenix router page routes
require restart or a later route-bridge plan. The loader must reject or mark
route-page artifacts as restart-required rather than silently half-integrating.

### 8. Actions overlay has collision and precedence rules

`AllbertAssist.DynamicPlugins.ActionsOverlay` sits beside the existing static
and plugin/app action registry paths. It mirrors the existing
`plugin_actions/0` collision-denial precedent and is merged by
`AllbertAssist.Actions.Registry` through the public seams that callers already
use: `modules/0`, `agent_modules/0`, `capabilities/0`,
`agent_capabilities/0`, `internal_capabilities/0`, `resolve/1`,
`capability/1`, `registered_module?/1`, `capabilities_for_app/1`, and
`diagnostics/0`. App attribution for dynamic action capabilities is provided by
the app registry seam (`App.Registry.app_id_for_action/2`) after dynamic app
registration.

It must:

- deny action-name collisions with static, plugin, app, or other dynamic
  actions;
- deny module, app id, plugin id, memory namespace, surface id, and destination
  id collisions before registration;
- expose diagnostics for rejected dynamic contributions;
- preserve `Actions.Runner.run/3`, app-scope, confirmation, Resource Access,
  and Security Central behavior;
- unregister entries atomically during rollback.

Resolution/display precedence is static reviewed actions, then reviewed
source-tree plugin/app actions, then integrated dynamic actions. Collision is
denial, not shadowing, so dynamic actions never outrank static or reviewed
source-tree actions. Integrated dynamic actions enter `capabilities/0`; only
those whose reviewed capability declares `exposure: :agent` enter
`agent_capabilities/0` and intent ranking. Internal-only dynamic actions are
callable only through explicit runtime/operator paths.

### 9. Settings Central owns workflow policy

New settings are default-off and audited:

| Key | Type | Default | Meaning |
|---|---:|---:|---|
| `dynamic_codegen.enabled` | boolean | `false` | Master switch for draft generation |
| `dynamic_codegen.provider_profile` | string/null | null | Required advisory model profile when generation is enabled; `null` fails closed |
| `dynamic_codegen.max_repair_iterations` | integer | bounded default | Repair loop cap |
| `dynamic_codegen.max_provider_calls_per_gap` | integer | bounded default | Planner/Author/TrialAuthor/Critic/Repair provider-call cap for one capability gap |
| `dynamic_codegen.max_provider_usage_units_per_gap` | integer/null | bounded default | Optional usage/token budget when provider usage data is available |
| `dynamic_codegen.max_files` | integer | bounded default | Draft file-count cap |
| `dynamic_codegen.max_bytes` | integer | bounded default | Draft total-size cap |
| `dynamic_codegen.allowed_targets` | string list | reviewed defaults | Generated target shapes permitted |
| `dynamic_codegen.allowed_action_permissions` | string list | `["read_only"]` | Operator-configured subset of the v0.37 generated-permission ceiling |
| `dynamic_codegen.live_loader_enabled` | boolean | `false` | Separate switch for operator-confirmed live integration |
| `dynamic_codegen.integration_approval_surfaces` | string list | `["cli", "liveview"]` | Surfaces allowed to approve integration/rollback; excludes Telegram/email/cross-channel |
| `dynamic_codegen.retention_days` | integer | bounded default | Draft/report cleanup horizon |

Settings store enablement, bounds, and selections. Draft source, metadata,
reports, and diagnostics remain file-backed Allbert Home data surfaced
read-only.

`dynamic_codegen.provider_profile` resolves through the existing
`Settings.resolve_model_profile/1` model-profile path used by the ReqLLM and
Jido.AI consumers. Generation fails closed when the setting is unset, the named
profile is unknown, the provider is disabled, credentials are unavailable,
ReqLLM/Jido.AI is unavailable, or the provider type is unsupported. These
provider calls use the settings-gated provider/model-profile path (the same one
`direct_answer` and the classifier use), not the Resource Access external-service
consumer path; they are bounded by settings rather than per-call resource grants.
Advisory providers never receive secrets, operator memory, or raw conversation in
prompts, traces, draft metadata, or sandbox bundles.

After an explicit generation step is accepted, Planner/Author/TrialAuthor/Critic
and Repair provider calls do not prompt for per-call Security Central
confirmation. Their cost and network use are bounded by
`dynamic_codegen.provider_profile`, `dynamic_codegen.max_repair_iterations`,
provider-call and usage budgets, and audit/trace records. Budget exhaustion
fails closed and leaves the draft at its current evidence tier.

`dynamic_codegen.allowed_action_permissions` can only select permission classes
inside the hard-coded generated-permission ceiling described in Design Decision
6. Unknown permission names, hard-denied permission classes, or attempts to
enable host execution, secret access, sandbox execution, confirmation decisions,
integration control, or direct trust-control permissions are rejected at settings
validation and again by the loader.

### 10. Signals, traces, and audit

Draft and integration behavior is security-relevant and observable, so it routes
through the signal-driven runtime. Capability-gap proposal, generation, draft
tier transitions, sandbox trial/gate consumption, integration, rollback,
discard, and every denial emit signals on the signal bus and write trace/audit
records through the v0.31 audit/redaction facades. The loader's compile, load,
register, and rollback audit events (ADR 0035) are the same signal/trace path,
not a side channel. Raw prompts/completions are never traced; structured
provenance, tier, and bounded diagnostics are. Signals and audit records carry
no authority: only the operator confirmation plus the v0.36 gate grant trust.

## Non-Goals

- No sandbox backend implementation; v0.37 consumes v0.36.
- No live integration without v0.36 gate pass and operator confirmation.
- No integration authorized by advisory/agent output or auto-trial result.
- No package-manager execution, dependency additions, migrations, NIFs, ports,
  unrestricted network, remote marketplace, remote binary loading, dynamic
  protocol/router generation, or application env mutation.
- No generated action body may hide host execution, package install, skill/script
  execution, sandbox execution, secret access, confirmation decisions,
  integration control, or direct protected-subsystem writes behind a lower
  declared permission.
- No multi-language target beyond Elixir/OTP.
- No templated Create destination or Mix generator UX; that is v0.38.

generate Elixir/OTP code and config through advisory agents, trial and gate it
through the v0.36 sandbox from a scanned, hash-matched project-shaped staging

live without a restart, with audit, validated action/app security semantics,
file-backed draft provenance, trusted-compile AST validation, integration-root
