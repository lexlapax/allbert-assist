## Locked Decisions

1. Voice reuses the existing provider/model profile framework.
2. Provider profiles become capability-aware; no voice-only provider stack.
3. Operator preferences are a global primary plus ranked task/capability
   lists.
4. Catalog capabilities are curated seed metadata; doctor output reports live
   availability but never grants authority.
5. `intent.model_profile` and `intent.direct_answer_model_profile` remain
   compatibility aliases.
6. CLI voice transcribes an explicit audio file/fixture.
7. Live microphone capture is workspace-only and confirmation-gated.
8. v0.48 cost visibility is display-only metadata, not spend enforcement.
9. Fake STT/TTS providers are fixture-only. The release lane uses fixture
   audio plus deterministic provider-shape fixtures for real adapter code.
10. Captured audio is normalized to a provider-accepted format via a bounded
    ffmpeg-class transcode step.
11. Voice providers support fake, local-endpoint, bundled-local, and
    remote-credentialed deployment modes, all operator-selectable; the resolver
    picks a profile and a voice adapter executes it.
12. Capability names are routing predicates. Media input/output, accepted
    formats, size/duration limits, and realtime/session support live in profile
    metadata and never grant permission.
13. v0.48 release authority requires executable Allbert local-runtime STT/TTS,
    OpenAI remote STT/TTS, Gemini remote STT/TTS, and a local Ollama
    text-generation turn in the listen -> think -> speak loop.
14. Anthropic/Claude is a text-generation middle-of-chain provider in v0.48,
    not a native STT/TTS provider.

## Main User/Operator Flows

- Operator configures a primary provider/profile and per-capability voice
  preferences during onboarding or settings review.
- Operator doctors an STT/TTS profile and sees the ADR 0047 redacted
  envelope plus voice capability fields.
- Operator runs `mix allbert.ask --voice AUDIO_FILE`; Allbert validates the
  file, transcribes it through the selected STT provider, and submits the
  transcript as ordinary runtime text.
- Operator uses the workspace microphone affordance; Allbert requests
  microphone confirmation, captures bounded audio after approval, transcribes
  it, and submits text.
- Allbert emits spoken output through a registered TTS action when requested.
- Telegram voice notes are parsed as voice inputs, downloaded only through the
  channel's `External.HttpPolicy`-bounded fetch path, capped by
  `min(20 MB, voice.audio.max_bytes)`, and transcribed through the shared STT
  action.
- Operator can validate a fully local voice loop: Allbert local voice runtime
  STT -> Ollama text model -> Allbert local voice runtime TTS.

## Profile Resolution Matrix

The resolver and adapter contract must handle provider shapes without making
one vendor's API the Allbert media model:

| Requested use | Required capability | Required media/profile metadata | v0.48 result |
|---|---|---|---|
| CLI audio file transcription | `speech_to_text` | `input_modalities: [audio]`, `transport_modes: [request_file]`, audio bounds/formats | `transcribe_voice` returns text input metadata. |
| Workspace microphone transcription | `speech_to_text` | `input_modalities: [audio]`, `transport_modes: [live_upload]` or `[request_file]`, audio bounds/formats | confirmed capture becomes `mic://capture/<id>`, then text input. |
| TTS action | `text_to_speech` | `output_modalities: [audio]`, audio output format metadata | `synthesize_voice` returns audio plus display-only usage metadata. |
| Realtime speech session | `speech_to_text` and/or `text_to_speech` | `transport_modes: [realtime_session]` | metadata only in v0.48 unless wrapped by the bounded request flow. |
| Generic audio/video-understanding model | no v0.48 STT/TTS guarantee | `input_modalities: [audio]` or `[video]` | not a v0.48 voice adapter by itself; reserved for future planning. |
| Local text/vision endpoint | `text_generation` or `vision_input` | `deployment_mode: local_endpoint` | can be a provider profile but does not imply audio support. |
| Allbert local voice runtime endpoint | `speech_to_text` and/or `text_to_speech` | `deployment_mode: local_endpoint`, loopback `base_url`, bounded request-file transport | required real local STT/TTS path. |
| OpenAI remote voice | `speech_to_text` and/or `text_to_speech` | `deployment_mode: remote_credentialed`, Settings Secret API key | required real remote path. |
| Gemini remote voice | `speech_to_text` and/or `text_to_speech` | `deployment_mode: remote_credentialed`, Settings Secret API key, provider-native audio request shape | required real remote path. |
| Ollama local text turn | `text_generation` | `deployment_mode: local_endpoint` | required local middle-of-chain validation; not an STT/TTS adapter. |

## Preference Resolution Flow

1. Caller asks for a task or capability, for example `:direct_answer`,
   `:speech_to_text`, or `:text_to_speech`.
2. The shared resolver reads Settings Central:
   - `model_preferences.tasks.<task>` first for task requests;
   - `model_preferences.capabilities.<capability>` for capability requests;
   - `model_preferences.primary` as fallback only when capable.
3. The resolver validates each candidate profile:
   - profile exists;
   - provider exists and is enabled by Settings Central;
   - profile advertises the requested capability;
   - doctor data, when present, does not mark the capability unavailable.
4. The first valid profile is returned with diagnostics about skipped
   candidates.
5. If none qualify, the action returns a denied/unavailable response before
   provider calls.
6. On a real-provider **call** failure, the action makes one bounded pass to the
   next capable profile in the ranked list — retryable errors only (timeout,
   5xx, connection refused); non-retryable errors (auth/4xx, permission, bounds)
   stop and surface. No re-ordering, no loop.

Compatibility:

- `intent.model_profile` maps to `model_preferences.primary`.
- `intent.direct_answer_model_profile` maps to
  `model_preferences.tasks.direct_answer`.
- Existing consumers migrate to the resolver but keep their operator-visible
  behavior.

## CLI Voice File Flow

1. Operator runs:

   ```sh
   mix allbert.settings set voice.enabled true
   mix allbert.ask --voice test/fixtures/audio/hello.wav --trace
   ```

   Release live validation uses `scripts/v048_voice_live_smoke.exs` because
   real local-endpoint and remote-credentialed provider calls require durable
   confirmation/resume.

2. The Mix task validates that a file path was provided and that no live
   microphone mode was requested.
3. Resource Access validates the local file path and audio bounds.
4. `transcribe_voice` resolves `:speech_to_text`.
5. The bounded transcode helper materializes provider input for a format the
   resolved provider accepts (`audio_formats_supported`) when the source format
   is not directly accepted. The helper uses only the fixed ADR 0042 command
   shape and writes to Allbert temp paths.
6. The selected voice adapter returns a transcript packet. For release
   authority, this must include the executable OpenAI-compatible local,
   OpenAI remote, and Gemini remote adapter paths; fake may be used only as a
   fixture fallback in tests. If the resolved provider deployment mode requires
   confirmation, `transcribe_voice` creates a durable confirmation and
   `approve_confirmation` resumes the same target action after Security
   Central re-checks policy.
7. The runtime receives normal text input with voice metadata:
   - original resource URI;
   - duration when known, plus byte size;
   - provider profile;
   - transcript hash;
   - redaction status.
8. Trace output stores only redacted voice metadata and the normal text turn.

Authority invariants:

- CLI voice never opens the microphone.
- File denial happens before provider calls.
- Provider output is text input, not action authority.
- Fake-only transcription is not sufficient release coverage. Local-endpoint,
  OpenAI remote, and Gemini remote adapter execution are release-blocking in
  M8R.

## Workspace Microphone Flow

1. Operator clicks the workspace microphone affordance.
2. Allbert creates a `:microphone_capture` confirmation for the session.
3. Denial ends the flow and writes no audio resource.
4. Approval creates a bounded `mic://capture/<id>` resource and arms the
   workspace for one capture.
5. The browser MediaRecorder hook writes the final audio blob into a Phoenix
   LiveView binary upload and submits the voice form.
6. The server consumes the upload only after the approved capture id exists,
   enforces `voice.audio.max_bytes`, and enforces a server-side capture time
   window from the approval time rather than trusting client-reported duration.
7. Transient audio is copied under the Allbert temp root only long enough for
   transcription unless `voice.audio.retention_enabled` explicitly writes it
   under `voice.audio.retention_root`.
8. Audio metadata is redacted and passed to `transcribe_voice` with the
   `mic://capture/<id>` resource identity.
9. Transcript enters the normal runtime input path with `metadata.voice`.

Authority invariants:

- Microphone capture is confirmation-gated per session.
- `mic://capture/<id>` is identity, not permission.
- Denial writes no retained or transient audio resource.
- Raw audio is not stored in traces or audits.

## TTS Flow

1. A registered action requests `:text_to_speech`.
2. The resolver selects a capable profile.
3. `synthesize_voice` runs under `:voice_synthesize`.
4. The response includes redacted display-only `usage` and `cost` metadata.
   Providers pass through metadata when available. Allbert does not compute
   byte-count costs, does not synthesize zero-dollar packets, and does not
   aggregate usage into a spend dashboard in v0.48.
5. Generated audio is returned as a bounded local output file with redacted
   output-resource metadata.

Authority invariants:

- TTS is effectful and provider-backed.
- Usage/cost metadata is informational in v0.48.
- No budget enforcement or spend dashboard ships in v0.48.

## Telegram Voice-Note Flow

1. Telegram parser recognizes a voice-note update.
2. The channel adapter records bounded message metadata.
3. The adapter fetches the voice payload with Bot API `getFile` plus the
   `https://api.telegram.org/file/bot<token>/<file_path>` download path. Both
   calls preflight through `External.HttpPolicy`; the effective download cap is
   `min(20 MB, voice.audio.max_bytes)`, and Allbert keeps the downloaded file in
   a bounded temp path for the transcription turn only.
4. `transcribe_voice` resolves `:speech_to_text` and transcribes.
5. The transcript is submitted as channel text input with redacted STT metadata
   plus bounded Telegram voice-note identifiers, size, MIME type, and duration.

Authority invariants:

- The Telegram adapter does not own provider choice or STT authority.
- Raw audio bytes, local temp paths, bot tokens, and download URLs are not
  stored in runtime metadata.
- Unsupported media remains unsupported unless v0.48 explicitly adds a path.
- Channel identity and provider identity stay separate.

## Data Flow

```text
provider catalog + settings
  -> capability-aware model profiles
  -> ranked model_preferences
  -> Models.for(task_or_capability)
  -> doctor/action validates provider availability
  -> STT/TTS registered action
  -> runtime text input or audio output metadata
  -> redacted trace/audit evidence
```

Audio resource flow:

```text
file://... or mic://capture/<id>
  -> Resource Access validation
  -> permission gate
  -> local/OpenAI/Gemini provider call, or fake fixture in tests
  -> transcript or generated audio packet
  -> redacted metadata in trace
```

End-to-end local voice loop:

```text
file://... or mic://capture/<id>
  -> Allbert local voice runtime STT profile
  -> transcript submitted as normal runtime text
  -> direct_answer/text_generation resolves voice_text_local on local_ollama
  -> Allbert local voice runtime TTS profile
  -> bounded audio output resource
```

The loop is opt-in via an explicit surface: CLI `mix allbert.ask --voice FILE
--speak` (writes the reply audio to a printed `file://` path) or the workspace
voice panel (in-panel playback). Speak is never automatic; the call-failure
fallback applies to each STT/TTS hop.

## Authority Invariants

- Provider capabilities and preferences are routing metadata, not authority.
- The voice adapter executes provider calls but makes no permission decisions;
  action permission/floor and Resource Access are evaluated before it runs.
- `0.48.0` release authority requires executable Allbert local-runtime,
  OpenAI remote, and Gemini remote adapter calls. Fake is concrete only as a
  deterministic fixture adapter.
- The transcode helper is bounded (no arbitrary args, size/duration-capped) and
  never accepts operator- or model-chosen codec flags.
- Voice input is input modality only; it does not lower action permissions.
- `:microphone_capture` is confirmation-gated for every deployment mode.
- `:voice_transcribe`/`:voice_synthesize` run as registered actions whose floor
  derives from the resolved profile's `media.deployment_mode`: `fake`/
  `bundled_local` may be `:allowed`; `local_endpoint`/`remote_credentialed`
  confirm; an unresolvable mode fails closed to `:needs_confirmation`.
- Voice provider confirmations persist only redacted summaries for operator
  inspection; internal resume params are used to restart the approved action,
  and the resolution stores redacted target metadata.
- Effective audio bounds are `min(global voice.audio.*, profile media.*)`: a
  profile can narrow the global ceiling but never raise it.
- Raw audio is redacted from traces by default and retained only by explicit
  operator policy.
- Channel adapters do not own STT/TTS provider authority.
- Model/provider doctor output is redacted and additive-only per ADR 0047.
- Anthropic/Claude text models may be selected for the text turn after STT, but
  cannot satisfy `speech_to_text` or `text_to_speech` in v0.48.

## Voice Doctor Dispatch

`mix allbert.model doctor <profile>` remains the operator-facing command. The
task resolves the configured profile first:

- profiles advertising only text-generation capabilities dispatch to
  `doctor_model_profile`;
- profiles advertising `speech_to_text` or `text_to_speech` dispatch to
  `doctor_voice_provider`;
- future vision profiles dispatch to the v0.49 vision doctor.

Every branch returns the ADR 0047 canonical envelope. Voice branches add
`provider_capabilities`, `provider_deployment_mode`,
`speech_to_text_supported`, `text_to_speech_supported`,
`audio_formats_supported`, `audio_sample_rates_supported`,
`provider_usage_metadata_available`, `local_runtime_present`, and
`fixture_probe_ok` only when known.

## M8R Real-Provider Remediation Flow

M8R turned the pre-remediation fixture-only stack into an operational voice
release:

1. Added provider HTTP execution with two explicit policy branches:
   loopback-only local endpoint and HTTPS-only credentialed remote.
2. Added bounded transcode materialization so Telegram OGG/Opus and workspace
   captures can become provider-accepted audio without arbitrary ffmpeg args.
3. Implemented the client-side OpenAI-compatible local adapter against the
   `/v1/audio/transcriptions` and `/v1/audio/speech` contract.
4. Implemented OpenAI remote STT/TTS with multipart transcription requests and
   bounded raw speech responses.
5. Implemented Gemini remote STT/TTS through provider-native `v1beta/interactions`
   audio input for STT and `generateContent` AUDIO responses for TTS.
6. Added `voice_text_local` and proved the
   listen -> think -> speak route through the resolver.
7. Updated `release.v048` so deterministic fixtures exercise the real adapter
   code paths without live network.
8. Added durable confirmation/resume for real provider STT/TTS actions so
   approval re-checks Security Central and then resumes the target action.
9. Added opt-in `.env` manual smokes for OpenAI, Gemini, local voice endpoint,
   and Ollama before release tag.

M8R7 adds the missing product side of item 3:

1. Allbert provides and starts the loopback local voice runtime at the default
   product base URL `http://127.0.0.1:5050/v1`.
2. The runtime implements `GET /v1/models`, `GET /v1/doctor`,
   `POST /v1/audio/transcriptions`, and `POST /v1/audio/speech`.
3. The runtime uses a configured local Ollama transcription model for STT and
   a real local/offline TTS backend for speech output.
   - `gemma4:e2b` is the v0.48 validated default for Mac local STT.
   - `gemma4:e4b` is an accepted higher-quality optional STT model when the
     machine has sufficient memory.
   - `gemma4:e2b-mlx` and `gemma3n:e2b` are not release-validation defaults:
     manual probes returned an empty transcript for the MLX tag and no
     multimodal support in local Ollama metadata for Gemma 3n.
4. Doctor output distinguishes endpoint readiness from backend readiness:
   missing Ollama, missing an audio-capable STT model, missing `say`, and
   missing `ffmpeg` are actionable diagnostics.
5. Settings Central owns `voice.local_runtime.*` configuration, Security
   Central owns `permissions.voice_local_runtime_manage`, and STT/TTS HTTP
   requests require the per-Allbert-Home local runtime token that Allbert's
   local adapter attaches automatically.
6. The local runtime starts Bandit with supported options only and decodes
   Ollama `application/x-ndjson` transcription bodies before extracting text.
