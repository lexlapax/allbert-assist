# Allbert v0.34 Workspace UX Refresh

Request flow: `docs/plans/v0.34-request-flow.md`

Visual mockup (for validation): [`v0.34-mockup.png`](v0.34-mockup.png) — desktop
shell, mobile layout, width states, and numbered design callouts.

ADR:

- `docs/adr/0024-app-ui-contribution-and-workspace-zones.md` (v0.34 revision of
  the workspace zone model)

v0.34 is a composition/navigation refresh of the v0.32 `/workspace` shell. The
shipped shell renders too many simultaneous regions — left rail, a floating
app/objectives band, center chat, a permanent Canvas column, and a permanent
Tools column — with no clear primary, and it puts an empty-state app dashboard
in the visual hero. v0.34 restructures the shell around a chat spine, a
view-only left launcher, and a single Canvas, and uses the v0.33 conversational
handoff as the only way to set routing context.

v0.34 changes layout and navigation only. It adds no domain behavior, no
analysis change, no new execution authority, no new routing authority, no
theming, no dynamic code, and no new Surface catalog atom.

## Purpose

The v0.32 workspace proved panels, zones, and Settings-in-workspace, but the
operator UI drifted from the intended zone semantics:

- `:utility_drawer` (Settings/tools) was rendered as an always-on fourth
  column instead of an on-demand surface.
- App dashboards and CoreApp objective cards landed in a floating top band
  instead of the canvas.
- Canvas and Tools are both permanent, so chat — the actual workspace spine —
  is cramped and never primary.
- The largest, darkest element on screen is frequently an empty-state app
  dashboard, so the eye is pulled to the least useful content.

v0.33 then made conversational handoff the way to enter an app context. v0.34
finishes that arc: it separates **what you view** (left launcher → Canvas) from
**what routing uses** (`active_app`, set only by handoff), and collapses the
right side to one Canvas.

## Expected Inputs

- v0.32 workspace route, host-owned zones, `:panel` surfaces, structural
  catalog atoms, Settings Central inside `/workspace`, and the registered
  `set_active_app` / `dismiss_workspace_ephemeral` actions.
- v0.33 conversational app-intent handoff: `Intent.Handoff`,
  `:app_handoff` / `:clarify_intent` decisions, ephemeral-surface surfacing,
  and handoff-accept as the path that calls `set_active_app`.
- v0.31 unified `Surface.Catalog`, `Workspace.Catalog` panel composition,
  `Extensions.Registry`, and settings fragments.
- v0.30 durable canvas tiles (the "Output" view) through the signed Fragment
  path.

## Design Decisions

### 1. Chat is the primary spine

`/workspace` keeps the left rail and the center chat. Chat becomes the visual
primary and the default-wide column. The shell has three width states:

- **chat-only** — Canvas closed; chat full width.
- **split (push)** — Canvas open; chat narrows so both stay visible.
- **canvas-focus** — a focus toggle collapses chat to a thin strip for a full
  app/destination view.

Opening Canvas **pushes** (resizes) rather than overlays, so output stays
visible while chatting.

### 2. The left rail is a view-only launcher

The rail has sections, and selecting an entry only changes **what Canvas
shows** — it never changes routing context, permissions, or executes anything:

- **THREADS** — switch the active conversation (changes chat, not Canvas).
- **APPS** — Allbert (home) and registered apps (e.g., StockSage); selecting
  one renders that app's dashboard in Canvas.
- **WORKSPACE** — Output, Jobs, Objectives, Confirmations, Security, Settings;
  each renders its existing panel in Canvas.

Launcher selection is a pure view switch. It does not call `set_active_app`.

### 3. Canvas shows exactly one destination (replace model)

Canvas renders the single selected destination at a time — no tabs, no
stacking, no floating band. **Output** (the thread's durable v0.30 tiles) is its
own launcher entry and the default Canvas view when a thread opens. Selecting an
App/Workspace destination replaces the Canvas content; returning to Output is
selecting the Output entry.

### 4. Routing context is conversational-only

`active_app` is set **only** by accepting a v0.33 handoff. There is no manual
context selector. A passive top-bar **context indicator** shows the current
routing context — `Neutral` (Allbert) or the active app — with an
**exit-to-neutral** affordance (clears `active_app` via
`Session.clear_active_app`). Viewing an app's dashboard in Canvas does **not**
put you in that app's context.

Consequence to keep in mind: you can be **viewing** StockSage's dashboard while
routing context is still Neutral. Asking `analyze CIEN` then still produces the
v0.33 handoff. When Canvas is already showing app X, that handoff should be
phrased as a lighter inline confirm ("Run StockSage analysis for CIEN?") rather
than a full cross-app handoff, but it travels the same accept → `set_active_app`
→ runner → confirmation path. This phrasing nuance is an M5 refinement, not a
new authority path.

### 5. Apps live in Canvas and take input through chat or scoped pop-ups

Apps render their surfaces inside Canvas. App input comes from the **chat
composer** (the normal runtime turn) or **scoped pop-ups** (ephemeral surfaces)
for app-specific quick inputs. No app-private shell chrome, side nav, or extra
operator routes return; the v0.32 page exception (`/apps/stocksage/analyses/:id`)
stands.

### 6. Remove the permanent Tools column and floating band

The always-on Tools column and the floating top band are removed. Settings and
tools become WORKSPACE launcher destinations rendered in Canvas. The
`:utility_drawer` zone is retired as a region (see ADR 0024 revision).

### 7. Desktop and mobile are both first-class

- **Wide:** rail | chat | Canvas, with push + canvas-focus.
- **Narrow:** rail becomes a launcher **sheet** (☰); the body is a single
  surface with `[Chat | Canvas]` bottom tabs; Canvas shows one destination
  full-screen. The replace model + view-only rail make this collapse work with
  no special cases.

### 8. Security, routing, and domain are unchanged

- Panels/destinations stay catalog-validated; metadata grants no authority.
- Settings writes still flow through registered actions and Security Central.
- App-scoped actions still require explicit `active_app` at
  `Actions.Runner.run/3`.
- `active_app` is still set only by handoff accept.
- No new Surface catalog atom; the context indicator reuses existing top-bar
  chrome.

## Architecture And Contracts

### Zone model revision (ADR 0024)

| v0.32 zone | v0.34 |
|---|---|
| `:nav_apps` | broadened to one **view-only launcher** (Threads/Apps/Workspace/Output); no `active_app` side-effect |
| `:canvas_panels` | the **single Canvas destination host** (replace model) |
| `:utility_drawer` | **retired as a region**; Settings/tools become launcher destinations rendered in Canvas |
| `:context_rail` | **retired as a region**; routing context moves to the top-bar indicator; compact app status folds into the Canvas header |
| `:ephemeral` | unchanged; handoff/confirmation/scoped app pop-ups |

### Module shapes

- `AllbertAssistWeb.WorkspaceLive` — tracks a `canvas_destination` assign
  (default `:output`); adds a `select_destination` event (view switch only, no
  `set_active_app`); renders rail | chat (push, with a `canvas_focus` toggle) |
  Canvas (one destination). Adds the top-bar context indicator and an
  `exit_app_context` event (`Session.clear_active_app`). Removes the permanent
  Tools column and the floating band.
- `AllbertAssist.App.CoreApp.surfaces/0` — re-declares the shell: rail lists
  destinations; Canvas hosts the selected destination; no utility-drawer
  column. CoreApp Output/Jobs/Objectives/Confirmations/Security/Settings remain
  `:panel` surfaces, now reached as launcher destinations.
- `StockSage.App` — dashboard/recent/queue/trends remain `:panel` surfaces
  rendered as the StockSage destination in Canvas; app input via chat composer
  / scoped ephemeral pop-ups.
- `AllbertAssist.Surface.Catalog` / `Workspace.Catalog` — zone registry updated
  per the table above; structural atoms reused (no new atom). The Canvas
  composition renders one destination instead of a stacked panel set.
- Context indicator — bound to session `active_app`; passive (display + exit),
  never a setter.

### What does not change

`active_app` setting (handoff-only), Security Central, confirmations, durable
Fragment/canvas emission, app-scope enforcement, the v0.32 page route exception,
and the v0.33 intent/descriptor/handoff contract.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, Chrome
desktop + narrow verification where UI changes, and `mix precommit`.

### M0: Preflight
Verify the shell/zone seams against the shipped v0.32/v0.33 tree (WorkspaceLive
render path, `Surface.Catalog` zones, `Workspace.Catalog` composition,
`set_active_app` / `dismiss_workspace_ephemeral`, the v0.33 handoff ephemeral
path). Confirm no new catalog atom is required. Confirm the ADR 0024 revision
captures the zone changes. `git diff --check`.

### M1: Chat-primary shell + remove permanent Tools column and floating band
Restructure WorkspaceLive to rail | chat | Canvas with push and a canvas-focus
toggle. Remove the always-on Tools column and the floating app/objectives band.
Tests: shell renders chat-primary; no Tools column; no floating band; width
states behave. Chrome desktop verification.

### M2: View-only left launcher
Make the rail a launcher with Threads/Apps/Workspace/Output sections; add
`select_destination` that sets `canvas_destination` only. Prove selection never
calls `set_active_app` and never executes an action. Tests: selecting StockSage
shows its dashboard with `active_app` still Neutral; selecting Settings shows
settings; routing is unaffected by selection.

### M3: Single-destination Canvas
Implement the replace model; Output is the default destination and renders v0.30
durable tiles; App/Workspace destinations render their existing panels. Tests:
exactly one destination renders; Output is default; switching replaces content;
no floating band.

### M4: Context indicator + exit, handoff-only confirmed
Add the passive top-bar context indicator and `exit_app_context`
(`Session.clear_active_app`). Re-assert that the only setter of `active_app` is
the v0.33 handoff accept. Tests: indicator reflects Neutral vs active app;
exit returns to Neutral; rail/launcher cannot set context; handoff still sets it.

### M5: App-in-canvas input + dedupe
StockSage operates inside Canvas using the chat composer and scoped ephemeral
pop-ups; no app-private shell returns. Soften the same-app handoff phrasing when
Canvas already shows that app. Remove duplicate launcher entries (the shipped UI
shows Objectives ×3 and Security ×2 — one home per concept). Tests: StockSage
flows reachable in Canvas; no duplicate destinations.

### M6: Mobile and accessibility
Narrow-width launcher sheet + `[Chat | Canvas]` tabs; one destination
full-screen; keyboard/focus order; no horizontal overflow. Chrome narrow
verification.

### M7: Security, docs, and closeout
Extend the v0.28/v0.32 eval harness: launcher selection cannot set `active_app`
or execute; Canvas destinations cannot bypass app-scope; Settings still flow
through registered actions. Accept the ADR 0024 revision; bump version metadata;
update README, roadmap, agent-context map, CHANGELOG, and this plan/request-flow.
Full gate.

## Non-Goals

- No user theming, layout YAML, or CSS overrides; that is v0.35.
- No dynamic plugin/app generation or sandboxed loading; that is v0.36.
- No generator; that is v0.37.
- No new domain behavior, analysis engine, outcome model, memory write path, or
  canvas emission mechanism.
- No new routing authority; `active_app` stays handoff-only (no manual selector,
  no rail-driven context).
- No new Surface catalog atom, no route compatibility shims, and no return of
  app-private shells beyond the v0.32 `/apps/<app_id>/...` page exception.

## Definition Of Done

v0.34 is done when `/workspace` is a chat-primary shell with a view-only left
launcher, a single-destination Canvas (Output by default), a passive top-bar
context indicator with exit-to-neutral, app surfaces operating inside Canvas via
chat/pop-up input, the permanent Tools column and floating band gone, desktop
and mobile both verified, the ADR 0024 zone revision accepted, and the full gate
plus security evals passing — with routing, security, and domain behavior
unchanged.
