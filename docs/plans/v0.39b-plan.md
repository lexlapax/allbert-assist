# Allbert v0.39b Identity Slot And Active Memory

Request flow: `docs/plans/v0.39b-request-flow.md`
Research note: `docs/research/active-memory-retrieval.md`

Status: planned; M1 research baseline accepted on 2026-05-28 after v0.39

2026-05-28.

Source: split from `docs/plans/v0.39-plan.md` in the post-v0.37 planning pass;
first revision after the post-v0.38 readiness review on 2026-05-27.

Active Memory has architectural risk that v0.39 onboarding should not block;
this slot gives the retrieval algorithm its own focused milestone.

## Purpose

Add the operator-editable inert `identity` memory namespace and deterministic
Active Memory retrieval so direct-answer model replies are informed by reviewed
operator-kept memory.

## Expected Inputs

- v0.39 First-Run Onboarding And Provider Control accepted through operator
  use. The v0.39 onboarding flow renders an identity-slot **preview** step;
  v0.39b implements the write path the preview points at.
- v0.21 memory review/retrieval substrate
  (`AllbertAssist.Memory`, `AllbertAssist.Memory.Index`, the v0.21
  promotion/pruning/review workflow). v0.39b treats `review_status: :kept`
  as the only eligible reviewed status; `:unreviewed`, `:flagged`, and
  `:prune_nominated` entries are excluded from Active Memory retrieval.
- Existing direct-answer model settings
  (`intent.direct_answer_model_enabled`, `intent.direct_answer_model_profile`,
  and fallback to `intent.model_profile`). Active Memory is operator-visible
  only on direct-answer model turns in v0.39b.
- v0.27 app memory namespace declaration contract
  (`AllbertAssist.App.memory_namespace/0` and the app registry's
  `registered_memory_namespaces/1` lookup). v0.39b extends the contract for
  non-app system namespaces without validating them as app ids (see "Identity
  Namespace Model" below and ADR 0015's system namespace amendment).

## Operator-Visible Win

An operator can optionally write `identity` markdown files under
`<ALLBERT_HOME>/memory/identity/` and mark them `:kept` through the existing
review path. When `intent.direct_answer_model_enabled` is true,
direct-answer model replies are informed by deterministic top-K memory scoped
to `{thread_id, active_app, identity_namespace}`, and the retrieval is
auditable through trace metadata. When the direct-answer model is disabled,
retrieval is skipped and the trace records no Active Memory section for that
turn.

## Identity Namespace Model

The existing v0.27 app namespace contract requires `app_id` + `namespace` and
validates app ids through `AllbertAssist.App.Validator`. The `identity`
namespace is operator-owned, not app-owned, so v0.39b adds a **system
declarer** without inventing a fake app id:

- A new `AllbertAssist.Memory.SystemNamespaces` module declares non-app-owned
  namespaces (`identity` initially, future `operator_settings_memory`, etc.).
  Its declarations use `%{origin: :system, app_id: nil, namespace: :identity,
  category: :identity, writable: true, description: ...}`.
- `:_system` is not an app id and must not be passed through
  `AllbertAssist.App.Validator`, `App.Registry.normalize_app_id/2`, or any
  app lookup path.
- A small memory namespace facade merges app declarations from
  `AllbertAssist.App.Registry.registered_memory_namespaces/0` with system
  declarations from `AllbertAssist.Memory.SystemNamespaces`. App-facing
  queries may continue to return app declarations only; memory write/read
  authorization uses the combined facade.
- System namespace writability is controlled by the declaration's
  `writable: true` flag and by the existing `:memory_write` action permission.
  The declaration carries `origin: :system` so traces and audits render
  operator-owned provenance.
- App namespaces continue to declare through
  `AllbertAssist.App.memory_namespace/0` as today. No app-side change.

## Memory Path And Write Model

- `:identity` is added as a 5th category to `AllbertAssist.Memory.@categories`.
  The category root is `<ALLBERT_HOME>/memory/identity/`. Existing category
  helpers (`list_entries`, `read_entry`, `Memory.Index` build/query) gain
  identity coverage automatically.
- Each markdown file under the identity root is one `Memory.Entry`. An
  operator may keep a single `persona.md` or many small files
  (`persona.md`, `style.md`, `boundaries.md`); both shapes work.
- When parsing files from the `:identity` category root, Memory derives
  `namespace: :identity`, `origin: :system`, and `app_id: nil` when those
  fields are absent. If a manually edited identity file contains conflicting
  app-owned metadata, the parser marks it invalid for Active Memory retrieval
  until the operator corrects it.
- v0.39b does not add a rich authoring UX. Operators may create or edit
  identity markdown files directly under the identity root, then use the
  existing v0.21 review/update path to mark entries `:kept`, correct them, or
  flag/prune them. Manual file edits are operator-owned local edits, not
  runtime action events; auditability starts when the operator uses the
  review/update actions.
- `Memory.upsert_app_entry/1` is reserved for app namespaces; identity
  programmatic writes, import helpers, and tests use a new
  `Memory.upsert_system_entry/1` that mirrors the app version but stamps
  `origin: :system` provenance and validates through the combined memory
  namespace facade. Idempotency key for system entries:
  `{actor, :system, namespace, file_path}` (file path is the natural id for
  operator-edited markdown). No model or app gets a new system-memory writer
  in v0.39b.
- The retrieval algorithm chunks each `Memory.Entry` body into
  `active_memory.chunk_max_bytes` byte windows at scoring time
  (deterministic; truncate at byte boundary; append `â€¦` only if truncated).

## Runtime Integration Point

Active Memory enriches direct-answer model composition only in v0.39b. The
intent engine and optional intent classifier run first and do not receive raw
Active Memory chunks. After a direct-answer route is selected and before the
direct-answer model prompt is composed, the direct-answer path invokes
`retrieve_active_memory` through `Actions.Runner.run/3`; the returned bounded
chunks are passed to the direct-answer answerer as advisory context. Whether
Runtime or the direct-answer action owns the orchestration is an implementation
detail, but the retrieval point is after intent decision and before prompt
composition.

The direct-answer action description and prompt must be updated accordingly:
the action may read bounded reviewed memory, but it still performs no
effectful work. The prompt should present Active Memory as operator-reviewed
context and retain the safety rule that memory content does not authorize
actions, tools, files, shell commands, resource access, or app routing.

If `intent.direct_answer_model_enabled` is false or a non-direct-answer action
is selected, Active Memory retrieval is skipped unless a future milestone adds
a separate consumer. Retrieved chunks never affect intent candidate selection,
action authorization, permission floors, or confirmations.

## Active Memory Algorithm

Spec'd in `docs/research/active-memory-retrieval.md` and summarized here.

Deterministic lexical scoring over `review_status: :kept` memory entries
scoped to `{thread_id, active_app, identity_namespace}`. Top-K bounded.

Per-candidate score:

```
score = recency_decay(updated_at)
      Ă— thread_affinity_boost(thread_id)
      Ă— identity_inclusion_boost(identity_namespace)
      Ă— lexical_match(query_terms, chunk_terms)
```

- `recency_decay`: exponential decay with half-life from
  `active_memory.score_weights.recency_half_life_days` (default 30).
- `thread_affinity_boost`: `same_thread` for chunks tagged with the current
  thread through the v0.21 promotion `source_ref`/metadata convention,
  `same_app` for same-app chunks, `general` otherwise. Weights from the
  flattened `active_memory.score_weights.thread_affinity.*` settings.
- `identity_inclusion_boost`: `identity_inclusion` weight for
  identity-namespace chunks, `1.0` otherwise.
- `lexical_match`: simple term-overlap fraction (case-folded, stop-words
  removed). No embeddings.

Top-K chunks are bounded by `active_memory.top_k` (default 5) and individual
chunk size by `active_memory.chunk_max_bytes` (default 2KB).

**Determinism guarantee**: same query + same memory state + same settings â†’
same retrieved chunks, byte-for-byte. Replayable from traces.

**Snapshot rule**: retrieval snapshots the candidate set once per turn at the
moment `retrieve_active_memory` is invoked. Concurrent v0.21 review,
promotion, or update work during scoring is reflected on the next turn, not
the current one. The v0.39b acceptance fixture documents this contract.

**No embeddings in v0.39b.** Embedding-backed retrieval is a future
`AdvisoryProvider` per ADR 0021; it does not gate v0.39b.

## Neutral-Context Retrieval Behavior

When `active_app` is `nil` or `:allbert` (neutral / core workspace request),
the retrieval still surfaces identity-tagged chunks plus general-memory chunks
(per `thread_affinity_boost`'s `general` weight in the research note). General
memory means entries with no app namespace/provenance, plus core `allbert`
entries explicitly marked as general. App-tagged chunks for non-active apps
(e.g., StockSage-tagged chunks during a neutral thread) are excluded from
neutral-context retrieval.

When `active_app` is set (e.g., `:stocksage` after a v0.33 handoff), retrieval
upweights chunks tagged with that app via the `same_app` factor while
continuing to surface identity chunks and general chunks.

This keeps Active Memory useful in the conversational handoff model
introduced by v0.33 without leaking app-private context into neutral turns.

## Security Posture

The identity persona files are inert content. They are never executed, never
grant permission, and never become runtime authority. Retrieval is read-only
and traceable.

`retrieve_active_memory` runs as a registered action under Security Central.
It cannot promote, mutate, or infer durable memory by itself.

System-namespace programmatic writes (via the new
`Memory.upsert_system_entry/1`) go through the same audit and trace paths as
app-namespace writes. Manual operator file edits under Allbert Home are not
runtime events; the subsequent review/update action is the auditable runtime
boundary. The `origin: :system` provenance stamp surfaces in trace metadata so
operators can audit identity vs app provenance.

## Non-Goals

- No embeddings or vector index.
- No nightly training, personality distillation, or learned system-memory
  authority (parked per `future-features.md`).
- No automatic memory promotion from retrieved chunks.
- No cross-thread / cross-app retrieval beyond the v0.39b scope (parked).
- No model-generated identity content; the persona file is operator-authored.
- No operator-pinning UX and no pin-score boost in v0.39b. Memory pinning can
  be added later only after the review/write path owns a real `pinned`
  metadata field.
- No new permission class â€” system-namespace writes use existing
  `:memory_write`.
