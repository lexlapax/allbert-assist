# Allbert v0.39b Identity Slot And Active Memory

Request flow: `docs/plans/v0.39b-request-flow.md`
Research note: `docs/research/active-memory-retrieval.md`

Status: planned; first revision after post-v0.38 readiness review.

Source: split from `docs/plans/v0.39-plan.md` in the post-v0.37 planning pass;
first revision after the post-v0.38 readiness review on 2026-05-27.

Active Memory has architectural risk that v0.39 onboarding should not block;
this slot gives the retrieval algorithm its own focused milestone.

## Purpose

Add the operator-editable inert `identity` memory namespace and deterministic
pre-reply Active Memory retrieval so Allbert replies are informed by reviewed
operator-promoted memory.

## Expected Inputs

- v0.39 First-Run Onboarding And Provider Control accepted through operator
  use. The v0.39 onboarding flow renders an identity-slot **preview** step;
  v0.39b implements the write path the preview points at.
- v0.21 memory review/retrieval substrate
  (`AllbertAssist.Memory`, `AllbertAssist.Memory.Index`, the v0.21
  promotion/pruning/review workflow).
- v0.27 app memory namespace declaration contract
  (`AllbertAssist.App.memory_namespace/0` and the app registry's
  `registered_memory_namespaces/1` lookup). v0.39b extends the contract for
  non-app system namespaces (see "Identity Namespace Model" below).

## Operator-Visible Win

An operator can optionally write `identity` markdown files under
`<ALLBERT_HOME>/memory/identity/`. Each reply is informed by reviewed-promoted
memory scoped to `{thread_id, active_app, identity_namespace}`, and the
retrieval is auditable through trace metadata.

## Planned Interfaces

- `identity` declared as a **system memory namespace** (see "Identity
  Namespace Model" below).
- `:identity` added as a 5th value to `AllbertAssist.Memory.@categories`
  (alongside `:notes`, `:preferences`, `:traces`, `:skills`).
  `<ALLBERT_HOME>/memory/identity/` becomes the category root.
- Registered `retrieve_active_memory` action (`:read_only`,
  `:not_required`) that wraps the v0.21 retrieval substrate with the

- Trace metadata: `### Active Memory` subsection per turn rendering retrieved

  - `mix allbert.memory list --namespace identity` lists identity-namespace

  - `mix allbert.memory list --category identity` lists by the new
    category root (deterministic alternative to `--namespace`).
  - `mix allbert.memory retrieve --query "..."` developer/operator helper
    that runs `retrieve_active_memory` and prints the deterministic top-K

- Settings keys:
  - `active_memory.enabled` — boolean, default `true`.
  - `active_memory.top_k` — integer, default `5`.
  - `active_memory.chunk_max_bytes` — integer, default `2048`.
  - `active_memory.score_weights.recency_half_life_days` — integer, default
    `30`.
  - `active_memory.score_weights.thread_affinity` — map with `same_thread`,
    `same_app`, `general` numeric weights (defaults `1.0`, `0.6`, `0.3`).
  - `active_memory.score_weights.identity_inclusion` — float, default `1.5`.
  - `active_memory.score_weights.explicit_pin` — float, default `2.0`.

The settings shape matches `docs/research/active-memory-retrieval.md` exactly.

## Identity Namespace Model

The existing v0.27 namespace contract requires `app_id` + `namespace`. The
`identity` namespace is operator-owned, not app-owned, so v0.39b extends the
registry with a **system declarer** alongside app declarers:

- A new `AllbertAssist.Memory.SystemNamespaces` module declares
  non-app-owned namespaces (`identity` initially, future
  `operator_settings_memory`, etc.). It registers each system namespace
  through the same `AllbertAssist.App.Registry` namespace surface using a
  reserved `:_system` app id (chosen to never collide with a real app id;
  the app registry already normalizes app ids and rejects invalid
  candidates).
- `Memory.validate_writable_namespace/2` accepts the system origin without
  requiring a matching app to be enabled. Writability remains controlled
  by the namespace declaration's `writable: true` flag.
- The namespace declaration carries `system?: true` so writes and reads
  can be audited as operator-owned rather than app-owned.
- App namespaces continue to declare through `AllbertAssist.App
  .memory_namespace/0` as today. No app-side change.

## Memory Path And Write Model

- `:identity` is added as a 5th category to `AllbertAssist.Memory.@categories`.
  The category root is `<ALLBERT_HOME>/memory/identity/`. Existing category
  helpers (`list_entries`, `read_entry`, `Memory.Index` build/query) gain
  identity coverage automatically.
- Each markdown file under the identity root is one `Memory.Entry`. An
  operator may keep a single `persona.md` or many small files
  (`persona.md`, `style.md`, `boundaries.md`); both shapes work.
- Writes go through the **existing v0.21 review surface**, not through a
  new authoring path. Operators edit identity entries the same way they
  edit any other reviewed-promoted entry. The system namespace declarer
  enforces the `:identity` category and the system-origin metadata.
- `Memory.upsert_app_entry/1` is reserved for app namespaces; identity
  writes go through the v0.21 review flow plus a new
  `Memory.upsert_system_entry/1` that mirrors the app version but stamps
  `system?: true` provenance. Idempotency key for system entries:
  `{actor, :_system, namespace, file_path}` (file path is the natural id
  for operator-edited markdown).
- The retrieval algorithm chunks each `Memory.Entry` body into
  `active_memory.chunk_max_bytes` byte windows at scoring time
  (deterministic; truncate at byte boundary; append `…` only if truncated).

## Active Memory Algorithm

Spec'd in `docs/research/active-memory-retrieval.md` and summarized here.

Deterministic lexical scoring over reviewed-promoted memory entries scoped to
`{thread_id, active_app, identity_namespace}`. Top-K bounded.

Per-candidate score:

```
score = recency_decay(updated_at)
      × thread_affinity_boost(thread_id)
      × identity_inclusion_boost(identity_namespace)
      × explicit_pin_boost(metadata.pin?)
      × lexical_match(query_terms, chunk_terms)
```

- `recency_decay`: exponential decay with half-life from
  `active_memory.score_weights.recency_half_life_days` (default 30).
- `thread_affinity_boost`: `same_thread` for chunks tagged with the current
  thread, `same_app` for same-app chunks, `general` otherwise. Weights from
  `active_memory.score_weights.thread_affinity` map.
- `identity_inclusion_boost`: `identity_inclusion` weight for
  identity-namespace chunks, `1.0` otherwise.
- `explicit_pin_boost`: `explicit_pin` weight for chunks the operator pinned
  during memory review (v0.21).
- `lexical_match`: simple term-overlap fraction (case-folded, stop-words
  removed). No embeddings.

Top-K chunks are bounded by `active_memory.top_k` (default 5) and individual
chunk size by `active_memory.chunk_max_bytes` (default 2KB).

**Determinism guarantee**: same query + same memory state + same settings →
same retrieved chunks, byte-for-byte. Replayable from traces.

**Snapshot rule**: retrieval snapshots the candidate set once per turn at the
moment `retrieve_active_memory` is invoked. Concurrent v0.21 review/promotion
during scoring is reflected on the next turn, not the current one. The
v0.39b acceptance fixture documents this contract.

**No embeddings in v0.39b.** Embedding-backed retrieval is a future
`AdvisoryProvider` per ADR 0021; it does not gate v0.39b.

## Neutral-Context Retrieval Behavior

When `active_app` is `nil` (neutral / pre-handoff workspace request), the
retrieval still surfaces identity-tagged chunks plus general-memory chunks
(per `thread_affinity_boost`'s `general` weight in the research note).
App-tagged chunks for non-active apps (e.g., StockSage-tagged chunks during
a neutral thread) are excluded from neutral-context retrieval.

When `active_app` is set (e.g., `:stocksage` after a v0.33 handoff), retrieval
upweights chunks tagged with that app via the `same_app` factor while
continuing to surface identity chunks and general chunks.

This keeps Active Memory useful in the conversational handoff model
introduced by v0.33 without leaking app-private context into neutral turns.

## Security Posture

The identity persona files are inert content. They are never executed, never
grant permission, and never become runtime authority. Retrieval is read-only
and traceable.

`retrieve_active_memory` runs as a registered action under Security Central.
It cannot promote, mutate, or infer durable memory by itself. Operator-pin
metadata is honored from the v0.21 review surface; v0.39b does not add a
pin-write path.

System-namespace writes (via the new `Memory.upsert_system_entry/1`) go
through the same audit and trace paths as app-namespace writes. The
`system?: true` provenance stamp surfaces in trace metadata so operators can
audit identity vs app provenance.

## Non-Goals

- No embeddings or vector index.
- No nightly training, personality distillation, or learned system-memory
  authority (parked per `future-features.md`).
- No automatic memory promotion from retrieved chunks.
- No cross-thread / cross-app retrieval beyond the v0.39b scope (parked).
- No model-generated identity content; the persona file is operator-authored.
- No operator-pinning UX in v0.39b — pin metadata is already honored by the
  scoring function, but the v0.21 review surface owns the pin write path.
- No new permission class — system-namespace writes use existing
  `:memory_write`.

## Definition Of Done

v0.39b is done when:

- the `identity` system namespace exists, is registered through the app
  registry's namespace surface, and is operator-editable through the v0.21
  review surface;
- `:identity` is the 5th category in `Memory.@categories` and
  `<ALLBERT_HOME>/memory/identity/` is the category root;
- `retrieve_active_memory` returns deterministic top-K chunks for a given
  query and memory state, matching the algorithm in
  `docs/research/active-memory-retrieval.md`;
- traces render `### Active Memory` metadata at the pinned placement that
  operators can inspect;
- a replayability test fixture proves same-state determinism byte-for-byte;
- neutral-context retrieval surfaces identity + general chunks without
  leaking app-private context, verified by a security eval row;
- security evals pass (identity inert, retrieval read-only, no memory
  mutation, no cross-namespace leak, deterministic replay);
- version metadata bumped to `0.39b` (target `0.39.1` or per project
  versioning convention chosen at M5);
- `docs/operator/active-memory.md` is filled in (no longer a stub);
- README, CHANGELOG, roadmap, and agent-context-map reflect the shipped
  state.
