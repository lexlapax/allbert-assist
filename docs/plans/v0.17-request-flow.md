# Allbert v0.17 Plugin Request Flow

Status: implemented through M6 closeout on 2026-05-14.

## Purpose

This document turns the v0.17 plugin plan into implementation flows and user
acceptance checks. It should be read with:

- `docs/plans/v0.17-plan.md`
- `docs/plans/v0.16-plan.md`
- `docs/plans/v0.15-plan.md`
- `docs/adr/0017-allbert-plugin-contract.md`
- `docs/adr/0016-channel-adapter-boundary-and-identity-mapping.md`
- `docs/adr/0015-allbert-app-contract-and-surface-dsl.md`

## Core Flow: Application Startup

1. `AllbertAssist.Application` starts Repo, migrations, PubSub, SignalBus, and
   Jido as before.
2. `AllbertAssist.Plugin.Supervisor` starts:
   - `AllbertAssist.Plugin.Registry`
   - `AllbertAssist.Plugin.ChildSupervisor`
   - `AllbertAssist.Plugin.Bootstrap`
3. Plugin bootstrap reads core plugin settings:
   - `plugins.enabled`
   - `plugins.disabled`
   - `plugins.scan_paths`
   - `plugins.trusted_project_roots`
   - `plugins.load_policy`
4. Plugin bootstrap registers shipped plugin modules.
5. Plugin bootstrap scans direct child folders of each configured plugin path,
   including the shipped `./plugins/allbert.telegram` and
   `./plugins/allbert.email` folders.
6. Valid folder manifests are normalized into plugin entries.
7. Invalid folders produce diagnostics and do not crash boot.
8. Bootstrap starts compiled plugin-owned child specs through
   `Plugin.ChildSupervisor`; `:ignore` child specs are accepted.
9. `AllbertAssist.App.Supervisor` starts after plugin bootstrap and registers
   configured apps plus plugin-contributed apps.
10. `AllbertAssist.Channels.Supervisor` starts after plugin bootstrap and starts
   registered channel adapter children.

Boot invariant: a broken optional plugin produces diagnostics. It does not
stop the runtime, scheduler, app registry, or other channel plugins.

Shipped plugins are enabled by default unless listed in `plugins.disabled`.
`plugins.enabled` is for optional folder plugins and does not need to list
`allbert.telegram` or `allbert.email` for v0.16 channel behavior to continue.

## Plugin Path Resolution

Default scan paths:

```text
./plugins
<ALLBERT_HOME>/plugins
```

Resolution rules:

1. Relative paths resolve from the project root.
2. `<ALLBERT_HOME>` is expanded from `AllbertAssist.Paths.home()`.
3. Duplicate expanded paths are scanned once.
4. Missing paths are ignored with an informational diagnostic.
5. Hidden child directories are skipped.
6. Discovery scans direct children only.
7. The shipped channel plugins live at:
   - `./plugins/allbert.telegram`
   - `./plugins/allbert.email`
8. Skill paths listed in a manifest resolve relative to the plugin directory
   and must stay inside that directory.
9. `plugins.scan_paths` controls discovery only. It does not make runtime code
   compilation dynamic.

## Manifest Flow

Given:

```text
<ALLBERT_HOME>/plugins/example-skills/allbert_plugin.json
```

with:

```json
{
  "schema_version": 1,
  "plugin_id": "example.skills",
  "name": "Example Skills",
  "version": "0.1.0",
  "kind": "skills",
  "skill_paths": ["skills"]
}
```

Flow:

1. Discovery reads `allbert_plugin.json`.
2. Manifest parser decodes JSON with Jason.
3. Validator checks:
   - `schema_version == 1`
   - `plugin_id` is a bounded lowercase dotted string
   - `name`, `version`, and `kind` are bounded strings
   - `skill_paths` are relative paths that stay under the plugin root
   - no unsupported code-bearing contribution is accepted from home plugins
4. Registry stores an entry with source `:home`, trust status from policy, and
   diagnostics.
5. `Skills.Registry` sees the plugin skill root when loading skills.

No `.ex`, `.beam`, package manager, shell command, script, or dependency
installer is run during this flow.

Normalized entries contain only bounded, validated fields: plugin id, display
name, version, kind, source, status, trust status, root path, manifest path,
contribution summaries, and diagnostics. CLI/actions render these normalized
entries, not raw manifests.

## Shipped Plugin Compile Flow

The v0.16 channel-specific files move out of the core app:

```text
apps/allbert_assist/lib/allbert_assist/channels/telegram/*
apps/allbert_assist/lib/allbert_assist/channels/email/*
```

and into:

```text
plugins/allbert.telegram/lib/allbert_assist/channels/telegram/*
plugins/allbert.email/lib/allbert_assist/channels/email/*
```

The plugin wrapper modules live beside them:

```text
plugins/allbert.telegram/lib/allbert_assist/plugins/telegram.ex
plugins/allbert.email/lib/allbert_assist/plugins/email.ex
```

Implementation flow:

1. `apps/allbert_assist/mix.exs` explicitly adds the two shipped plugin `lib/`
   roots to `elixirc_paths/1`.
2. The modules compile with `allbert_assist`.
3. Runtime discovery reads the manifests from `./plugins`.
4. Bootstrap registers only already-compiled modules that match Allbert's
   shipped plugin id/module allowlist.
5. A manifest in `<ALLBERT_HOME>/plugins` that names a module does not trigger
   compilation, code loading, or atom creation.

## Shipped Channel Plugin Flow

Shipped Telegram:

1. Discovery reads `./plugins/allbert.telegram/allbert_plugin.json`.
2. Bootstrap verifies `plugin_id: "allbert.telegram"` and module
   `AllbertAssist.Plugins.Telegram` against the shipped allowlist.
3. Plugin bootstrap calls `AllbertAssist.Plugin.Registry.register_module/2`
   for the already-compiled module.
4. The module validates its plugin metadata.
5. Registry stores:
   - `plugin_id: "allbert.telegram"`
   - `source: :shipped`
   - channel descriptor for `"telegram"`
   - settings metadata matching `channels.telegram.*`
   - secret refs for Telegram token status
   - adapter child spec for `AllbertAssist.Channels.Telegram.Adapter`
6. `Channels.Supervisor` starts the adapter from the registered descriptor.
7. `Channels.list_channels/0` renders the Telegram summary from the descriptor
   and existing settings/event substrate.

Shipped email follows the same flow with `allbert.email`.

The stable v0.16 settings schema entries for `channels.telegram.*` and
`channels.email.*` remain statically declared in core for this milestone.
Plugin descriptors expose matching metadata for inspection and channel
summaries, but existing user settings must validate before and after the file
move.

## Channel Descriptor Lookup Flow

v0.17 uses `AllbertAssist.Plugin.Registry` as the single source for registered
channel descriptors. There is no separate `AllbertAssist.Channels.Registry` in
this milestone.

Descriptor lookup:

1. `AllbertAssist.Channels.list_channels/0` asks
   `Plugin.Registry.registered_channels/0`.
2. Channel summaries render provider, source plugin, status, redacted settings
   fields, credential status, and diagnostics from descriptors plus Settings
   Central.
3. `AllbertAssist.Channels.channel_settings/1` resolves the descriptor by
   string channel id and then reads the descriptor's stable settings prefix.
4. `AllbertAssist.Channels.credential_status/1` uses descriptor secret refs and
   must return a safe status even if settings are temporarily unavailable.
5. `AllbertAssist.Channels.Supervisor` starts adapter child specs from
   descriptors with `status: :enabled`.
6. Unknown channel ids return `{:error, :unknown_channel}` or equivalent clear
   diagnostics; they do not create atoms.

Session derivation strategies:

- Telegram descriptor: `{:telegram_chat, prefix: "ch_tg_"}` derives from
  channel id, external Telegram user id, and chat id.
- Email descriptor: `{:email_sender, prefix: "ch_em_"}` derives from channel
  id and normalized sender address.

`AllbertAssist.Channels.derive_session_id/3` keeps the v0.16 public behavior
but routes strategy selection through the descriptor.

## Channel Dispatch Flow

Inbound Telegram or email after v0.17:

1. Provider adapter receives inbound/callback event.
2. Adapter inserts or dedupes `channel_events` exactly as v0.16.
3. Adapter resolves external identity through Settings Central.
4. Adapter derives session id through channel descriptor/session strategy.
5. Adapter calls `AllbertAssist.Runtime.submit_user_input/1` or registered
   confirmation actions exactly as v0.16.
6. Adapter renders response through channel renderer.
7. Adapter updates `channel_events` with status, thread id, trace id, and error
   metadata.

Plugin registration does not alter runtime authority. It only changes how the
adapter is discovered and supervised.

The shared channel substrate stays in `allbert_assist`. Provider-specific
behavior lives in the plugin folders. `AllbertAssist.Channels` no longer owns
a hardcoded provider map for Telegram/email; it asks the registered channel
descriptors for provider, secret refs, settings keys, and session strategy.

Channel-specific CLI subcommands keep their v0.16 names
(`telegram set-token`, `telegram map`, `email set-password`, and so on), but
they resolve channel metadata through descriptors rather than hardcoded
provider maps.

## Skill-Only Plugin Flow

Operator creates:

```text
<ALLBERT_HOME>/plugins/research-pack/
  allbert_plugin.json
  skills/
    summarize-paper/
      SKILL.md
```

Then:

```sh
mix allbert.plugins list
mix allbert.plugins show example.research_pack
mix allbert.skills list
```

Expected behavior:

- Plugin appears in plugin list.
- Skill root appears in skill diagnostics/sources.
- Skill follows the existing skill trust and enablement policy.
- No code is compiled or loaded.
- A duplicate skill name follows existing skill duplicate diagnostics.

## App Contribution Flow

For v0.17 fixture tests and v0.20 StockSage:

1. A compiled plugin module returns an app module from `apps/0`.
2. Plugin registry records the app contribution.
3. App bootstrap passes the app module to `AllbertAssist.App.Registry`.
4. App registry validates and registers the app exactly as v0.15.
5. App actions and skill paths remain app-owned and permission-neutral.

Important: a folder manifest in `<ALLBERT_HOME>/plugins` that names an app
module must not cause that module to load or register in v0.17.

## Action Contribution Flow

For v0.17 fixture tests:

1. A compiled plugin module returns action modules from `actions/0`.
2. Action registry merges plugin action modules after static core actions.
3. Duplicate action names produce diagnostics and are not silently replaced.
4. `Actions.Registry.capability_for_module!/1` includes plugin provenance.
5. `Actions.Runner.run/3` remains the only runtime execution path.

Expected denial path:

- If a plugin action has unsafe permission metadata or missing capability data,
  the registry records diagnostics and the action is not agent-facing.

## Plugin Child Supervision Flow

For compiled plugin modules:

1. Registry stores a normalized child spec returned by `child_spec/1`, or
   records `:ignore`.
2. Bootstrap starts non-ignored child specs under
   `AllbertAssist.Plugin.ChildSupervisor`.
3. Duplicate child ids are diagnostics and do not replace an existing child.
4. Optional child start failures are diagnostics; required shipped child
   failures may fail boot only when they would make a previously released core
   feature silently unavailable.
5. Channel adapters are not started by the plugin child supervisor. They are
   started by `AllbertAssist.Channels.Supervisor` from channel descriptors.

This is the path future app plugins such as StockSage use for
`StockSage.Supervisor`.

## Settings Schema Flow

1. Static settings schema includes `plugins.*`.
2. Static schema still includes the stable shipped channel settings keys from
   v0.16: `channels.telegram.*` and `channels.email.*`.
3. Plugin bootstrap reads only static plugin settings.
4. Plugin registry records plugin-contributed settings schema metadata.
5. Settings schema reads merge static entries with available plugin entries.
6. Writes to plugin settings are validated once the plugin registry is
   available.

If plugin registry is unavailable, plugin settings should fail closed or report
clear diagnostics rather than crash Settings Central.

## CLI Flow

### `mix allbert.plugins list`

Expected columns:

```text
plugin_id          source   kind      status    contributions
allbert.telegram   shipped  channel   enabled   channels=1 skills=0 apps=0 actions=0
allbert.email      shipped  channel   enabled   channels=1 skills=0 apps=0 actions=0
```

### `mix allbert.plugins show allbert.telegram`

Expected safe output:

```text
Plugin: allbert.telegram
Name: Telegram
Version: 0.17.0
Source: shipped
Status: enabled
Channels: telegram
Actions: 0
Apps: 0
Skill paths: 0
Diagnostics: none
```

Do not print raw secrets, token refs if they are considered sensitive, raw
provider payloads, or unbounded manifest content.

### `mix allbert.plugins diagnostics`

Expected behavior:

- Prints invalid manifest errors.
- Prints disabled plugin notes.
- Prints duplicate id/action/channel diagnostics.
- Exits successfully if diagnostics exist; diagnostics are inspectable state,
  not command failure.

## Manual Verification Matrix

1. Boot Allbert with the shipped `./plugins/allbert.telegram` and
   `./plugins/allbert.email` folders present and no extra
   `<ALLBERT_HOME>/plugins` folder. Boot succeeds.
2. Run `mix allbert.plugins list`. Shipped Telegram and email are listed with
   source `shipped`.
3. Run `mix allbert.channels list`. Telegram and email are listed and match
   v0.16 disabled-by-default posture.
4. Verify `apps/allbert_assist/lib/allbert_assist/channels/telegram/` and
   `apps/allbert_assist/lib/allbert_assist/channels/email/` no longer contain
   provider-specific implementation files.
5. Verify the provider-specific files compile from
   `./plugins/allbert.telegram/lib/` and `./plugins/allbert.email/lib/`.
6. Create a temp home plugin with a valid skill-only manifest and `SKILL.md`.
7. Run `mix allbert.plugins show example.skills`. It shows plugin metadata and
   no code contributions.
8. Run `mix allbert.skills list`. The plugin skill is discoverable under the
   expected source/trust policy.
9. Create an invalid manifest with a duplicate `plugin_id`. Diagnostics show
   the duplicate without crashing boot.
10. Create a home plugin manifest that claims an action/app/channel module.
   Diagnostics reject the code-bearing contribution in v0.17.
11. Run Telegram simulate after mapping an identity. Runtime response,
   `channel_events`, thread id, and trace behavior match v0.16.
12. Run email simulate after mapping an identity. Runtime response,
    `channel_events`, thread id, and trace behavior match v0.16.
13. Restart the application. Plugin registry, app registry, and channel
    adapters rebuild from configured sources.
14. Add a fixture compiled plugin whose `child_spec/1` returns `:ignore`; it
    registers cleanly.
15. Add a fixture compiled plugin with a duplicate child id; diagnostics show
    the conflict without replacing an existing child.

## Explicitly Unsupported In v0.17

- Loading arbitrary code from `<ALLBERT_HOME>/plugins`.
- Running package managers during plugin discovery.
- Installing remote plugins.
- Auto-trusting plugin skills.
- Granting permissions from plugin metadata.
- Mounting plugin LiveView routes.
- Compiling plugin migrations from home folders.
- Plugin hot reload.
- Remote plugin marketplace publishing.
- StockSage implementation.
