## Main User/Operator Flows

- Operator connects or confirms a local notes/files root **from onboarding or a
  settings action** (config-free; the new `set_notes_root` affordance, not the generic
  key/value editor).
- Operator asks Allbert to find or summarize a local note.
- Operator reads a note through the `workspace:notes` destination or `search_notes`/
  `read_note`.
- Operator confirms a proposed note write (`write_note`, confirmation-gated).
- Operator reviews a memory candidate in the `workspace:memory` panel (or `admin
  memory`) before it becomes durable/recallable.
- Operator asks a later question that uses reviewed (`:kept`) memory.
- Maintainer runs docs, warning, drift, leak-scan, and `release.v065` gates.

## Data Flow

No new authority data flow is introduced. Notes/files reads and writes are enforced by
`PermissionGate` + **root-and-extension path bounding** under the configured
`apps.notes_files.notes_root`; the `ResourceURI`/`Scope` refs the plugin emits are
**provenance/audit metadata**, not the file-access enforcement seam. Note writes remain
confirmation-gated. Memory candidates are written `:unreviewed` and only become
recallable when a human review action marks them `:kept`; nothing derives authority from
notes/files output, plugin metadata, or model output.

## Authority Invariants

Local file roots are explicit and set through the settings action. Read/write operations
stay bounded to the configured root (path + extension bounding), not an arbitrary
filesystem grant. `write_note` requires confirmation. The `:notes_files` memory namespace
stays non-writable and does not auto-promote. Reviewed agent memory remains
user-controlled, inspectable, and removable; the new `workspace:memory` panel surfaces the
existing permissioned review actions and adds no authority.

## Handles / CLI Surfaces

- `workspace:notes` destination + nav and the `set_notes_root` connect affordance.
- `workspace:memory` interactive review panel (keep/reject/delete).
- `search_notes`, `read_note`, and `write_note`.
- `allbert admin memory {list,show,review,delete,retrieve,status}` (with `status`
  introduced by v0.65).
- Confirmation, trace, and audit views for note writes and memory decisions.
- `mix allbert.test release.v065` deterministic gate.

## Smoke Checklist

- Configure a notes root from the connect affordance (onboarding/settings action) in a
  disposable Home, with no hand-edited config.
- Search/read a sample note and verify access is bounded to the configured root
  (path/extension bounding) and the emitted resource refs are correct provenance.
- Confirm a note write and verify denial/approval behavior.
- Open the `workspace:memory` panel, surface a memory candidate, and keep/reject/delete it
  (delete confirmation-gated).
- Ask a later question that uses the kept memory; run `admin memory status`.
- Verify no notes content or secrets leak into evidence beyond explicit test
  fixtures.
- Run docs gate, warning gate, drift checks, and `release.v065`.

Expected behavior:

- local files/notes setup is understandable without developer docs;
- note reads are scoped and note writes require confirmation;
- memory candidates require review before durable recall;
- reviewed memory can influence a later response;
- no secret or unintended local content appears in logs, evidence, output, docs,
  or screenshots;
- `release.v065`, docs, warning, and drift checks pass.
