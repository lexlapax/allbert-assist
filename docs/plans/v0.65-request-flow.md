## Main User/Operator Flows

- Operator connects or confirms a local notes/files root **from onboarding, the
  post-first-chat QuickStart affordance, web/settings affordance, or
  `allbert admin notes set-root PATH`** (config-free; the new `set_notes_root`
  affordance, not the generic key/value editor).
- Operator asks Allbert to find or summarize a local note.
- Operator searches/reads a note through the action-backed `workspace:notes`
  destination or `search_notes`/`read_note`.
- Operator confirms a proposed note write (`write_note`, confirmation-gated).
- Operator reviews a memory candidate in the `workspace:memory` panel (or `admin
  memory`) before it becomes recallable; Keep maps to `:kept`, Reject maps to
  `:flagged`, and Delete archives through confirmation.
- Operator asks a later question that uses reviewed (`:kept`) memory.
- Maintainer runs docs, warning, drift, leak-scan, and `release.v065` gates.

## Data Flow

No new authority data flow is introduced. Notes/files reads and writes are enforced by
`PermissionGate` + **root-and-extension path bounding** under the configured
`apps.notes_files.notes_root`; the `ResourceURI`/`Scope` refs the plugin emits are
**provenance/audit metadata**, not the file-access enforcement seam. Note writes remain
confirmation-gated. Memory candidates are written `:unreviewed` and only become
recallable when a human review action marks them `:kept`; nothing derives authority from
notes/files output, plugin metadata, or model output.

## Authority Invariants

Local file roots are explicit and set through the `set_notes_root` action, whether
entered from onboarding, web, or `allbert admin notes set-root PATH`. Read/write
operations stay bounded to the configured root (path + extension bounding), not an
arbitrary filesystem grant. `write_note` requires confirmation. The `:notes_files`
memory namespace stays non-writable and does not auto-promote. Reviewed agent memory
remains user-controlled, inspectable, and removable; the new `workspace:memory` panel
surfaces the existing permissioned review actions and adds no authority.

## Handles / CLI Surfaces

- `workspace:notes` destination + nav, action-backed search/read panel, and the
  `set_notes_root` connect affordance.
- `allbert admin notes set-root PATH`, routed through `set_notes_root`.
- `workspace:memory` interactive review panel (keep/reject-as-`:flagged`/delete).
- `search_notes`, `read_note`, and `write_note`.
- `allbert admin memory {list,show,review,delete,retrieve,status}` (with `status`
  introduced by v0.65, scoped to the current user by default, `--all-users` for the
  explicit aggregate, and backed by an exact aggregate helper rather than a bounded list
  sample).
- Confirmation, trace, and audit views for note writes and memory decisions.
- `mix allbert.test release.v065` deterministic gate.

## Smoke Checklist

- Configure a notes root from the connect affordance (onboarding/web) and from
  `allbert admin notes set-root PATH` in a disposable Home, with no hand-edited config.
- Confirm the connect affordance remains reachable for QuickStart after first chat, since
  QuickStart currently defers `optional_connect`.
- Search/read a sample note through Runner-backed `search_notes`/`read_note` with
  `active_app: :notes_files`; verify access is bounded to the configured root
  (path/extension bounding) and the emitted resource refs are correct provenance.
- Confirm a note write and verify denial/approval behavior.
- Open the `workspace:memory` panel, surface a memory candidate, keep one candidate,
  reject one as `:flagged`, and delete/archive one (delete confirmation-gated).
- Ask a later question that uses the kept memory; run `allbert admin memory status`.
- Verify no notes content or secrets leak into evidence beyond explicit test
  fixtures.
- Run docs gate, warning gate, drift checks, and `release.v065`.

## Operator Validation

The deterministic, model-free core (notes-root connect + fail-closed + Runner-backed
notes search/read + memory review round-trip) is a runnable acceptance script —
`scripts/smoke/v065_local_knowledge.sh` — that exits non-zero on any failed check. The
model-dependent chat/recall steps below are the operator checklist v0.66 runs against the
packaged binary with a model configured.

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v065.XXXXXX)"
notes_root="$ALLBERT_HOME/launch-notes"
mkdir -p "$notes_root"
printf '# Onboarding\n\nBring the local knowledge checklist.\n' > "$notes_root/onboarding.md"

# Start the packaged persistent service through the platform path, then verify health.
curl -fsS http://localhost:4000/health
allbert admin health

allbert admin notes set-root "$notes_root"
allbert ask "find notes about onboarding"
allbert ask "write a note titled follow up with body ask about memory review"
# Approve the resulting note-write confirmation through the packaged confirmation path.
allbert admin confirmations list

allbert ask "remember this preference after review: prefer concise answers"
# Open /workspace?destination=workspace:memory or use admin memory to keep/reject/delete.
allbert admin memory review PATH --status kept --note "operator accepted"
allbert admin memory status
allbert admin memory status --all-users
mix allbert.test release.v065
```

Expected behavior:

- local files/notes setup is understandable without developer docs;
- note reads are scoped and note writes require confirmation;
- memory candidates require review before durable recall;
- reject maps to `:flagged`, delete archives only after confirmation, and `memory status`
  reports review-status counts;
- reviewed memory can influence a later response;
- no secret or unintended local content appears in logs, evidence, output, docs,
  or screenshots;
- `release.v065`, docs, warning, and drift checks pass.
