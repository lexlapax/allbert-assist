# Allbert v0.11 Execution-Aware Intent, Approval Handoff, And Remote Network Content Security Posture

## Purpose

v0.11 introduces the first structured intent decision contract over real risky
capabilities, the shared Approval Handoff contract for channel-native
approvals, and the remote network content security posture that lets higher
level consumers use v0.10 network access safely.

Allbert needs this before scheduled jobs or new channels consume shell, script,
package, network, and online import capabilities. The contract makes selection,
confirmation, denial, and refusal explainable without adding new execution
powers.

Remote network content security posture is the key v0.11 generalization. A
user-facing request such as "check this URL and summarize it" is not a special
skill flow and not a generic browser. It is an execution-aware decision that
would fetch network content, ask the user for approval with the URL, limits,
operation class, and destination clearly shown, then hand approved content to a
summarizer agent/action only if that summarizer is available. Direct skill URL
import, online skill search, package metadata inspection, and document
inspection use the same posture with different operation classes.

Approval Handoff is the bridge between an action that needs confirmation and
the user's current channel. CLI/REPL, web chat, scheduled jobs, and future
Telegram-style adapters should all translate a risky action result into native
approve, deny, and details affordances without owning confirmation storage,
security policy, or execution.

## Expected Inputs

- v0.05 Security Central decision shape exists.
- v0.06 action-backed skills and action runner exist.
- v0.07 confirmation workflow exists.
- v0.08 shell execution exists behind Level 1 local sandbox policy.
- v0.09 skill script execution exists behind trusted resource policy.
- v0.10 external service, package-install, and online import adapters exist:
  `external_network_request`, `plan_package_install`, `run_package_install`,
  `search_online_skills`, `show_online_skill`, `audit_online_skill`, and
  `import_online_skill`.
- v0.10 request/audit/trace metadata can be treated as the substrate for
  remote network content references: canonical URL, method, source/profile,
  operation class, size limits, redacted summaries, confirmation id, and
  target execution result.
- v0.10 adds distinct `:package_install` and `:online_skill_import`
  permissions, so the intent decision contract must not collapse those into
  generic shell or network intent.

## Intent Decision Contract

Introduce a structured intent decision, tentatively
`AllbertAssist.Intent.Decision`, with fields equivalent to:

- `:intent`
- `:confidence`
- `:reason`
- `:risk_summary`
- `:selected_skill`
- `:candidate_skills`
- `:selected_action`
- `:candidate_actions`
- `:permission`
- `:confirmation`
- `:execution_mode`
- `:remote_network_content`
- `:foreground_or_background`
- `:approval_handoff`
- `:alternatives`
- `:diagnostics`
- `:trace_metadata`

The decision object is plain data. It does not execute anything. Runtime still
executes through registered actions, the action lifecycle runner, Security
Central, the permission gate compatibility layer, and confirmation state.

When present, `:remote_network_content` is posture metadata, not an executor.
It should carry canonical URL, method, source/profile label, operation class,
expected content kind, accepted content types, byte cap, redirect/retry
posture, digest/cache expectation, downstream consumer, summarizer
requirement, origin channel/surface, response target when available, and
allowed approval scopes. Initial operation classes should include:

- `external_service_request`
- `summarize_url`
- `inspect_document`
- `import_skill`
- `online_skill_search`
- `online_skill_detail`
- `online_skill_audit`
- `online_skill_import`

Operation class is a security boundary. A remembered approval for
`summarize_url` must not authorize `import_skill`; an `inspect_document`
approval must not authorize package installation; an import approval must not
authorize skill activation or script execution.

## Approval Handoff Contract

Introduce Approval Handoff as plain data attached to the intent decision or
confirmation-needed action result. It should carry:

- confirmation id
- redacted action summary
- remote network content summary when the risky action fetches URL content:
  canonical URL, method, operation class, source/profile, expected content
  kind, byte cap, destination consumer, and whether summarization/import is
  expected after the fetch
- origin channel and surface
- origin response target when the channel supports durable follow-up
- allowed approval surfaces
- remembered approval choices, scoped to exact URL or URL hierarchy/prefix and
  operation class
- same-channel and cross-channel resolution expectations
- channel render hints for approve, deny, details, pending, resolved, and
  result-return states

Channels may render the handoff in their native UX, but approval and denial
must still call `approve_confirmation` and `deny_confirmation`. Channels must
not mutate confirmation records directly, bypass Security Central, or invoke
execution adapters.

## Milestones

### Milestone 1: Execution-Aware Intent Decision Contract

Goal: make risky capability selection inspectable before execution or
confirmation.

Implementation expectations:

- Define the intent decision data shape and validation rules.
- Include selected and candidate skills/actions, permission, risk,
  confirmation, execution mode, alternatives, diagnostics, and trace metadata.
- Validate decisions against known skills, registered actions, known
  permissions, Security Central, Settings Central, and confirmation state.
- Cover shell, skill script, package-manager, external service, online skill,
  and remote network content decisions so Approval Handoff can show the actual
  risky target, not just "network" or "install".
- For package installs, include exact package specs, target root, dry-run argv,
  npm execution argv preview, timeout/output caps, and pip preview-only
  warnings.
- Keep the decision object inert; it never executes work.

### Milestone 2: Remote Network Content Security Posture Contract

Goal: define the shared posture object used by URL summarization, document
inspection, direct skill URL import, and online source consumers.

Implementation expectations:

- Add `remote_network_content` to intent decisions and confirmation-needed
  action results when work would fetch network content.
- Include canonical URL, method, source/profile label, operation class,
  expected content kind, accepted content types, byte cap, redirect/retry
  posture, digest/cache expectation, downstream consumer, summarizer
  requirement, origin channel/surface, response target, and allowed approval
  scopes.
- Treat `skills.sh` as one source profile. Direct `SKILL.md` or skill archive
  URLs should use operation class `import_skill` and the same posture object.
- Treat "check this URL and summarize it" as operation class `summarize_url`:
  fetch approval happens first; summarization receives only approved, bounded,
  extracted content.
- If no summarizer agent/action exists, the approved fetch result must report
  that summarization is unavailable instead of hiding the missing capability.
- Keep HTTPS GET as the initial posture default for URL/document consumers.
  Block local/private-network targets, credentials in URLs, hidden redirects,
  unbounded downloads, and raw secret-bearing output.

### Milestone 3: Security-Aware Approval Handoff Contract

Goal: define the shared data contract that lets every channel render pending
approval UX consistently.

Implementation expectations:

- Add `approval_handoff` to the decision/action result shape for work that
  resolves to `:needs_confirmation`.
- Include confirmation id, redacted summary, origin channel/surface, response
  target, allowed approval surfaces, and render hints.
- Include remote network content details for URL/document/skill-source
  consumers: URL, operation class, source/profile, content limits, expected
  consumer, summarizer availability, import destination when relevant, and
  redaction/cache posture.
- Include remembered approval options as data, not policy: approve once,
  remember exact URL for this operation class, or remember URL hierarchy/prefix
  for this operation class when policy allows it.
- Define result-return expectations for same-channel and cross-channel
  approval so a resolver sees the immediate outcome and the origin channel can
  show durable follow-up when it supports callbacks. Channels must preserve the
  v0.10 distinction between an approved confirmation and a failed resumed
  target, such as an online source HTTP failure with `target_status=failed`.
- Keep approval handoff data descriptive only; it must not become authority to
  execute, approve, or deny.

### Milestone 4: Existing Channel Approval UX Baseline

Goal: make the first CLI/REPL-style and web surfaces render channel-native
approvals over the shared handoff.

Implementation expectations:

- CLI/REPL-style surfaces should show pending approval details and expose
  approve, deny, and details commands over the confirmation actions.
- For remote network content, CLI/REPL output should show the URL, operation
  class, content limit, destination consumer, and remember-scope choices before
  approval.
- Web LiveView/chat surfaces should render the same handoff as native approve,
  deny, and details controls.
- For remote network content, web surfaces should show the same URL, operation
  class, content limit, destination consumer, same-channel/cross-channel
  resolution expectation, and remembered-scope options.
- Both surfaces must call `approve_confirmation` and `deny_confirmation`
  through the action runner.
- Neither surface may write confirmation files directly, make policy
  decisions, or invoke shell, script, package, network, import, or other
  execution adapters directly.

### Milestone 5: Network Content Consumer UX Baselines

Goal: prove the posture supports multiple consumers without making any one
consumer the architecture center.

Implementation expectations:

- Document and implement the baseline UX for "check this URL and summarize
  it" as operation class `summarize_url`.
- Document and implement the baseline UX for "inspect/read this document URL"
  as operation class `inspect_document`, with bounded fetch/extraction and a
  clear unsupported-content message for formats not yet extractable.
- Document direct skill URL import as operation class `import_skill`; it should
  fetch through the same posture, validate with the existing Agent Skills
  parser/registry, audit, and import disabled/untrusted.
- Preserve v0.10 online skill search/detail/audit/import as source-profile
  consumers of the same posture, not a separate marketplace-specific path.
- Ensure remembered `summarize_url` or `inspect_document` approvals cannot be
  reused for `import_skill`, package install, skill activation, or script
  execution.

### Milestone 6: Release Readiness And User Testing Docs

Goal: close v0.11 with implementation-ready user tests and future handoffs.

Implementation expectations:

- Update README, CHANGELOG, roadmap, v0.11 plan, v0.12/v0.13/v0.16 handoffs,
  and ADR links with final names and implemented behavior.
- Include manual tests for CLI/REPL URL summarization approval, web approval,
  denial-before-fetch, approved fetch plus summary, summarizer unavailable,
  unsupported document extraction, remembered exact URL, remembered hierarchy,
  and direct skill URL import.
- Include negative tests showing remembered `summarize_url` does not authorize
  `import_skill`, and direct skill import does not authorize activation or
  script execution.
- State release/tag readiness and expected tag after operator acceptance.

## Test Plan

Focused implementation tests should cover:

- intent decision validation for known actions, permissions, execution modes,
  remote network content operation classes, and invalid model output
- Approval Handoff rendering data for CLI/REPL and web surfaces
- pending URL summary request creates confirmation before fetch
- denial performs no fetch, extraction, summarization, import, or grant write
- approved URL summary fetches bounded content through the v0.10 adapter and
  returns a summary when a summarizer agent/action is available
- approved URL summary reports summarizer unavailable when no summarizer exists
- document inspection reports unsupported content types or extractor failures
  without hiding the approved fetch outcome
- remembered exact-URL and URL-prefix grants are scoped by operation class
- `summarize_url` and `inspect_document` grants do not authorize
  `import_skill`
- direct skill URL import uses the remote network content posture, parser and
  registry validation, import audit, disabled/untrusted result flags, and no
  activation or script execution
- jobs and future channel adapters can consume the same handoff metadata
  without owning confirmation storage, policy, fetch, summarization, or import

Final gates should match the code-change gate in `DEVELOPMENT.md`:

```sh
mix compile --warnings-as-errors
mix format --check-formatted
mix credo --strict
mix dialyzer
mix precommit
git diff --check
```

## Non-Goals

- No new network primitive beyond the v0.10 external adapter.
- No model-selected side effects without validation.
- No new external channels; v0.13 consumes the handoff contract for
  Telegram/email/SMS-style adapters.
- No broad cross-channel routing; later channel-routing work owns that.
- No memory retrieval or embeddings work.
- No channel-specific intent fork.
- No browser, crawler, arbitrary recursive fetcher, or unbounded document
  ingestion.
- No automatic trust, enablement, activation, package install, or script
  execution from fetched network content.

## Definition Of Done

v0.11 is done when:

- Runtime turns can include an intent decision object.
- Risky capability prompts produce inspectable selected/candidate decisions
  before execution or confirmation.
- Remote network content prompts produce operation-scoped posture data before
  any fetch, including URL, method, content limits, consumer, and allowed
  approval scopes.
- Confirmation-needed prompts produce an Approval Handoff that CLI/REPL-style
  and web surfaces can render as channel-native approvals.
- URL summarization, document inspection, and direct skill URL import are
  represented as consumers of the same posture, with clear unavailable or
  unsupported states when summarization/extraction/import cannot proceed.
- Remembered approvals are operation-scoped and cannot cross from summaries or
  document inspection into skill import, package install, activation, or script
  execution.
- Invalid model or route output cannot invent skill names, action names,
  permissions, or execution modes.
- Intent traces explain selected and rejected candidates plus approval handoff
  state when a confirmation is pending or resolved.
- Approval still resumes work only through `approve_confirmation`; no channel
  gets direct execution authority.
