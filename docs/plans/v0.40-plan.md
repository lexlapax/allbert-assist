## Purpose

Add MCP client support as the first ecosystem-scale capability expansion after
onboarding. MCP servers become configured resources and tools that Allbert can
inspect and call only through registered actions, Resource Access, Security
Central, confirmations, traces, and audits.

v0.40 turns the parked `mcp://` identity (ADR 0013, reserved as inert in
`AllbertAssist.Resources.ResourceURI`) into a real, bounded adapter. MCP tool
schemas, resource lists, and result bodies remain descriptive metadata; they
never become authority (ADR 0038, ADR 0021).

v0.40 is also the substrate v0.42 builds on. v0.42's calendar, mail, and GitHub
workspace summary panels are driven by the MCP client actions and the
grant-gated resource-read model that v0.40 ships. The v0.40 validation matrix
must prove which first-wave integration data can be fetched as MCP resources and
which data is exposed only through MCP tools. If a v0.42 summary panel would
need repeated read-like tool calls, v0.42 must either keep that panel behind
per-call confirmation or revise the plan/ADR before implementation; v0.40 does
not add remembered approval for any tool call. The v0.40 request flow must be
kept current in `docs/plans/v0.40-request-flow.md`; every milestone below has a
request-flow handoff requirement.

## Research Grounding

The v0.40 design is grounded in:

- **MCP protocol and transports**: MCP defines tools, resources, and prompts
  over JSON-RPC, with stdio (local subprocess), streamable HTTP, and SSE
  transports. v0.40 consumes tools and resources only; prompts and MCP Apps UI
  are out of scope.
- **Hermes MCP (`hermes_mcp`)**: a mature, OTP-native Elixir MCP
  client/server library with `initialize`, `list_tools`, `list_resources`,
  `call_tool`, and `read_resource` client operations plus a validated MCP
  message codec. It was evaluated during M1 and revisited after M2. v0.40 uses
  `Hermes.MCP.Message` for MCP JSON-RPC message encoding and validation, but it
  does not use `Hermes.Client` or Hermes transport modules in the runtime call
  path. Its v0.14.1 production client transport schema accepts only built-in
  transport modules, and those built-ins own HTTP/Port egress.
- **Allbert external posture**: `AllbertAssist.External.HttpClient` /
  `HttpPolicy` already implement SSRF blocking, host allowlist, bounded
  timeouts, bounded response bodies, redirect denial, and redaction (ADR 0011,
  ADR 0047). MCP HTTP/SSE transports reuse this rather than opening a second
  network path.
- **Local execution sandbox levels (ADR 0009)**: stdio MCP servers are local
  OS processes; their startup is bounded under existing local-execution policy
  (explicit argv, no shell string, bounded lifecycle, audited startup).
- **Resource Access Security Posture (ADR 0011/0012/0013)**: MCP is one more
  resource consumer. ADR 0011 already names "future MCP resources"; `:mcp_resource`
  already exists as an origin kind in `AllbertAssist.Resources.OperationClass`.
  Operation class is part of the security boundary: a read grant must not
  authorize a tool call.
- **Provider doctor contract (ADR 0047)**: `mcp_doctor_server` reuses the
  canonical doctor envelope and redaction policy proven by v0.39's
  `doctor_model_profile` (`AllbertAssist.Settings.ModelDoctor`).

Implementation consequence: MCP adds new actions, a new Settings namespace, two
permission classes, and MCP operation classes — but no new network primitive, no
new secret store, and no new runtime.

## Operator-Visible Win

An operator can configure an MCP server (HTTP/SSE or stdio) with secret refs,
run `mcp_doctor_server` and see a redacted reachability/discovery envelope, list
its tools and resources, approve a Resource Access grant to read resources,
approve a tool call, and inspect trace/audit evidence — without MCP metadata ever
acting as authority.

## Dependencies

M1/M2 spike result: v0.40 uses `:hermes_mcp` for protocol message validation
and wire encoding, with a minimal Allbert MCP client over Allbert-owned
transports for runtime calls.

`hermes_mcp` v0.14.1 documentation and source confirm the needed client
operations and built-in stdio, streamable HTTP, WebSocket, and SSE transports.
They do not provide a production custom client transport hook suitable for
forcing HTTP/SSE through `AllbertAssist.External.HttpClient` or stdio through an
ADR 0009 launcher. The library's production client transport schema enumerates
only its built-in transport modules, and those modules own egress.

Therefore M2 implements `AllbertAssist.Mcp.Codec` as the narrow Hermes protocol
wrapper and `AllbertAssist.Mcp.Client` / `AllbertAssist.Mcp.Transport` as the
Allbert-owned sequencing and egress boundary. SSE remains supported for
compatibility, but it is treated as legacy; streamable HTTP is the preferred
remote transport.

Module naming follows the project's acronym convention (`Http`, not `HTTP`):
new modules live under `AllbertAssist.Mcp.*` and `AllbertAssist.Actions.Mcp.*`.

## Settings Central Schema

Add the `mcp.servers.*` namespace to `AllbertAssist.Settings.Schema`.

```yaml
mcp:
  servers:
    github:
      enabled: false
      transport: "streamable_http"   # streamable_http | sse | stdio
      base_url: "https://example-github-mcp.invalid/mcp/"  # http/sse only
      command: null                  # stdio only, e.g. "npx"
      args: []                       # stdio only
      env: {}                        # stdio only; secret-bearing values are refs
      headers: {}                    # http/sse only; secret-bearing values are refs
      auth_ref: "secret://mcp/github/token"
      tool_allowlist: []             # empty = all advertised tools (still gated)
      tool_denylist: []
      confirmation: "required"       # tighten-only: required | denied
  stdio:
    allowed_launchers: []            # explicit operator allowlist; empty = deny
```

`mcp.servers` defaults to an empty map. No server is shipped enabled.
`mcp.stdio.allowed_launchers` defaults to an empty list; a stdio server cannot
start until the operator explicitly allows the launcher command or absolute
path.

Safe-write keys (writable through `update_setting`):

- `mcp.stdio.allowed_launchers`
- `mcp.servers.*.enabled`
- `mcp.servers.*.transport`
- `mcp.servers.*.base_url`
- `mcp.servers.*.command`
- `mcp.servers.*.args`
- `mcp.servers.*.env`
- `mcp.servers.*.headers`
- `mcp.servers.*.auth_ref`, only when the referenced secret exists or is written
  in the same operation
- `mcp.servers.*.tool_allowlist`
- `mcp.servers.*.tool_denylist`
- `mcp.servers.*.confirmation`

Validation rules:

- `transport`: one of `streamable_http`, `sse`, `stdio`.
- `base_url`: required for `streamable_http`/`sse`; must parse as `http`/`https`
  with host and must pass `HttpPolicy` host rules at call time. Forbidden for
  `stdio`.
- `command`: required for `stdio`; non-empty string; resolved against an
  operator-controlled `mcp.stdio.allowed_launchers` list (e.g. `npx`, `uvx`, or
  an absolute path). Forbidden for `streamable_http`/`sse`.
- `args`: list of strings.
- `env`: `stdio` only; map of string to string. `headers`:
  `streamable_http`/`sse` only; map of string to string. For both, any value
  that looks secret-bearing must be a `secret://mcp/<server-id>/<name>` ref, not
  a literal.
- `auth_ref`: `nil` or a valid `secret://mcp/<server-id>/<name>` ref.
- `confirmation`: `required` or `denied`. `allowed` is rejected for tool calls;
  the value can only tighten, never loosen, the `:mcp_tool_call` floor.
- `tool_allowlist` / `tool_denylist`: lists of strings.

Secret refs:

- v0.40 adds `secret://mcp/<server-id>/<name>` to
  `AllbertAssist.Settings.Secrets.validate_secret_ref/1`.
- `<server-id>` and `<name>` match `[a-zA-Z0-9_-]+`.
- Stored encrypted in `secrets.yml.enc`; only redacted status is ever exposed.

CLI/config input requirement:

- `mcp.servers.*.args`, `tool_allowlist`, and `tool_denylist` are string lists
  and can reuse the existing Settings `:string_list` type, which
  `mix allbert.settings set` already parses (comma-separated). They are the easy
  case.
- `mcp.servers.*.env` and `headers` are string-to-string maps, for which the
  settings CLI has no input path today. v0.40 must add one: either JSON-aware
  `mix allbert.settings set` parsing for these known map keys or dedicated
  `mix allbert.mcp config ...` helpers.
- The operator guide examples must match the implemented input path before M2
  starts.

## Permission Classes

Add two permission classes to `AllbertAssist.Security.Policy`, following the
`:package_install` / `:online_skill_import` precedent (ADR 0011):

- `:mcp_tool_call` — safety floor `:needs_confirmation`, default decision
  `:needs_confirmation`. Setting key `permissions.mcp_tool_call`. Per-server
  `confirmation` policy may tighten to `:denied` but never below confirmation.
- `:mcp_resource_read` — safety floor `:allowed`, default decision `:allowed`.
  Setting key `permissions.mcp_resource_read`. Authority comes from a remembered
  Resource Access grant per `mcp://` server/URI (see Resource Identity); the
  permission class does not itself force confirmation.

Each class is wired into `@permission_settings`, `@default_decisions`,
`safety_floor/1`, `@known_permissions`, and `reason/5`. Settings can tighten
either class; settings cannot loosen `:mcp_tool_call` below `:needs_confirmation`
and cannot configure `:mcp_resource_read` to bypass the grant requirement.

## Resource Identity And Operation Classes

`AllbertAssist.Resources.ResourceURI`:

- Remove `mcp` from `@unsupported_schemes` (leaving `agent agent+https`).
- Add `mcp://<server-id>/<encoded-uri>` support: an encode helper
  (`ResourceURI.mcp(server_id, server_resource_uri)`), `normalize/1` handling,
  and a `derive/2` clause returning
  `%{origin_kind: :mcp_resource, canonical_id: ..., unsupported?: false,
  server_id: ..., server_resource_uri: ...}`.
- `<server-id>` matches `[a-zA-Z0-9_-]+`; `<encoded-uri>` is the percent-encoded
  server resource URI. Round-trip encode/decode is lossless.

`AllbertAssist.Resources.OperationClass`:

- Add operation classes `:mcp_tool_call` and `:mcp_resource_read` to
  `@operation_classes` and the `operation_class` type.
- Add access mode `:call` to `@access_modes`.
- Add scope kinds `mcp_server` and `mcp_tool` to `@scope_kinds`.
- Add `@default_access_modes`: `mcp_tool_call: :call`, `mcp_resource_read: :read`.
- `:mcp_resource` already exists as an origin kind; no change there.

`AllbertAssist.Resources.Grants`:

- Match grants on the canonical `mcp://` resource URI, operation class, scope
  kind (`mcp_server` for whole-server reads, `mcp_tool` for tool scope), and
  downstream consumer.
- An `mcp_resource_read` grant for server A does not authorize server B, and
  never authorizes `mcp_tool_call`.
- `mcp_read_resource` uses the existing
  `AllbertAssist.Resources.GrantHandoff` / `remember_resource_grant` path for
  first-access approvals and remembered grants. If the existing handoff helpers
  need small generic extensions for `mcp_server`/`mcp_tool` scope kinds, those
  extensions happen in Resource Access, not in a private MCP grant store.

## Action Schemas And Capability Metadata

All MCP actions are registered through `AllbertAssist.Actions.Registry` and run
through `AllbertAssist.Actions.Runner.run/3`. v0.40 adds these
`AllbertAssist.Actions.Capability.execution_mode` atoms:

- `:mcp_doctor`
- `:mcp_discovery`
- `:mcp_resource_read`
- `:mcp_tool_call`

Action parameter schemas:

| Action | Required params | Optional params | Notes |
|---|---|---|---|
| `mcp_doctor_server` | `server_id` | `include_discovery` boolean, default `true` | Reads only the configured server entry and returns the ADR 0047 envelope plus MCP fields. |
| `mcp_list_tools` | `server_id` | `cursor`, `limit` | Returns descriptive metadata only; creates no grants. |
| `mcp_list_resources` | `server_id` | `cursor`, `limit` | Returns descriptive metadata only; creates no grants. |
| `mcp_read_resource` | `server_id`, `uri` | `resource_uri`, `scope_kind`, `downstream_consumer`, `remember_scope` | `uri` is the server-native resource URI; `resource_uri` may be the canonical `mcp://...` form. The action normalizes to canonical `mcp://<server-id>/<encoded-uri>`. |
| `mcp_call_tool` | `server_id`, `tool_name`, `arguments` | `downstream_consumer`, `idempotency_key` | `arguments` is a decoded JSON object/map. Every call confirms before transport execution. |

`server_id` and `tool_name` use `[A-Za-z0-9_-]+`. `arguments`, `env`, and
`headers` are redacted before traces/audits; full resource or tool bodies are
never embedded in confirmations.

## Confirmation Policy

- `mcp_doctor_server`, `mcp_list_tools`, `mcp_list_resources`: read-only,
  confirmation `:not_required`. They return descriptive metadata only.
- `mcp_read_resource`: permission `:mcp_resource_read` (floor `:allowed`); first
  access to a new `mcp://` server/scope requires operator approval that creates
  a remembered Resource Access grant; in-scope reads thereafter need no new
  confirmation.
- `mcp_call_tool`: permission `:mcp_tool_call` (floor `:needs_confirmation`);
  every call is confirmed. Per-server/per-tool policy may tighten to `:denied`.
  Disabled, denied, or out-of-allowlist tools cannot run.

Confirmation requests carry `Confirmations.ResourceMetadata` with the MCP server
id, tool/resource name, `mcp://` resource URI, operation class, and redacted
argument summary. Confirmations never embed raw secrets or raw tool/resource
bodies.

## Doctor Contract

`mcp_doctor_server` reuses the ADR 0047 envelope and redaction policy. Base
fields (`endpoint_kind`, `endpoint_ok`, `credential_ok`, `redacted_host`,
`diagnostics`, ...) apply; `endpoint_kind` is `:local_endpoint` for stdio and
`:credentialed_remote` for authenticated HTTP/SSE. v0.40 adds these additive
fields (ADR 0047 §2 additive-only; mirrored into ADR 0047):

```elixir
%{
  transport_kind: :streamable_http | :sse | :stdio,
  tools_listable: boolean(),
  resources_listable: boolean(),
  tool_count: non_neg_integer() | :unknown,
  resource_count: non_neg_integer() | :unknown,
  protocol_version: String.t() | nil
}
```

The doctor probes only the configured server entry, grants no tool/resource
authority, and creates no remembered grants. HTTP/SSE probes use the ADR 0047
SSRF/timeout/redirect/redaction subset; stdio probes start the configured
process under ADR 0009 bounds and read only the initialize/list handshake.
Because stdio maps to `:local_endpoint`, `credential_ok` is `nil` for stdio even
when the server uses a secret env; stdio secret-env presence is reported through
a `diagnostics` entry, not `credential_ok`.

## Security Posture

MCP schemas are descriptive only. MCP calls get the new `:mcp_tool_call` and
`:mcp_resource_read` permission classes, with the floors above that settings
cannot loosen. Server env and headers use secret refs. HTTP/SSE transports route
through `External.HttpPolicy`; stdio startup is bounded under ADR 0009.
`mcp_doctor_server` reuses ADR 0047's redacted envelope for reachability and
discovery and does not grant tool/resource authority. Tool/resource confusion,
prompt injection, server impersonation, secret-env redaction, and stdio process
startup all require security eval coverage (see M6).

## Non-Goals

- No MCP Apps iframe or third-party UI model (parked; MCP server mode is
  v0.52b and other public protocol surfaces are post-1.0).
- No `agent://` or `agent+https://` endpoint execution (parked).
- No arbitrary auto-enablement of MCP tools.
- No MCP server authority over Allbert permissions, grants, or confirmations.
- No silent or remembered MCP tool-call approval in v0.40.
- No WebSocket transport in v0.40 (HTTP/SSE and stdio only).
- No code-bearing plugin install through MCP.
- No MCP prompts consumption, no MCP server mode (v0.52b owns server mode).
- No memory writes from MCP output (no auto-promotion).

## Definition Of Done

v0.40 is done when:

- At least one MCP server can be configured in a disposable Allbert Home over
  both an HTTP/SSE and a stdio transport, with secrets stored as
  `secret://mcp/...` refs.
- `mcp_doctor_server` returns the ADR 0047 envelope plus MCP additive fields for
  a reachable and an unreachable server, redacted.
- `mcp_list_tools` and `mcp_list_resources` return descriptive metadata that
  authorizes nothing.
- `mcp_read_resource` is gated by a remembered Resource Access grant per `mcp://`
  scope, and `mcp_call_tool` is confirmed per call and cannot be loosened below
  the `:mcp_tool_call` floor.
- `mcp://` is supported in `ResourceURI`; intent routes `mcp://` to MCP actions
  while `agent://` remains unsupported, and the prior "MCP unsupported" tests are
  rewritten.
- MCP security evals (schema-not-authority, tool/resource confusion, prompt
  injection, server impersonation, secret-env redaction, stdio startup policy,
  doctor redacted envelope) pass.
- v0.42 has a stable substrate: the `mcp_list_tools`, `mcp_list_resources`,
  `mcp_read_resource`, and `mcp_call_tool` actions plus the grant-gated read
  model are documented as the interfaces v0.42 panels consume.
- `docs/plans/v0.40-request-flow.md` matches the implementation; roadmap,
  future-features, ADRs, operator/developer guides, agent-context-map, and
  security-hardening reflect the shipped state.
- Release gate (`mix compile --warnings-as-errors`, `mix credo --strict`,
  `mix dialyzer`, `mix precommit`, `git diff --check`) passes.
