# v0.15 — Growth loop (stub)

Status: Stub

This is a placeholder plan. v0.15 is the release that finally addresses the origin note's central ask: "when I search things in Google… it records things that I search and extracts and remembers it and stores it for training the engines." Allbert was supposed to *grow with the user* by ingesting daily activity. Through v0.14, the only ingestion path is the agent's own in-turn `web_search` with an opt-in `record_as` parameter. v0.15 fills that gap without adding a hosted platform.

The stub records the must-dos. The fully detailed milestone-by-milestone plan is drafted only after v0.14.1 and v0.14.2 land.

## Must-dos

### 1. One ingestion endpoint

Add a single daemon-owned ingestion entry point. Protocol bumps to v7.

- New client message `IngestRecord(IngestRecordRequest)`.
- Payload: `{ source: String, kind: IngestKind, captured_at: ISO8601, text: Option<String>, url: Option<String>, attribution: Option<String>, redaction_hint: Option<RedactionHint> }`.
- `IngestKind` enum: `search_query`, `page_visit`, `clipboard`, `chat_message`, `note`, `other`.
- Filesystem-trust scoped, identical to existing daemon IPC. No network listener beyond localhost.
- Per-record byte cap (default 16 KiB) and per-day record cap (default 5000) enforced at the daemon.

### 2. Two ingest adapters as proof-of-concept set

The endpoint is generic; v0.15 ships two adapters as proof:

1. **CLI piped feed.** `allbert-cli ingest stdin --source <name> [--kind <kind>]` — operators can pipe anything (tail of a chat log, output of a script, clipboard manager) into Allbert. This is the lowest-risk adapter and validates the endpoint shape.
2. **Browser extension.** A minimal Chrome/Firefox/Safari extension repo (`allbert-browser-ingest/`, separate repo) that posts page visits and search queries to `http://127.0.0.1:<daemon_port>/ingest`. Operator-installed only; the extension repo is not bundled with the kernel build.

Skip Slack/Gmail/iMessage/SMS/screenshots in v0.15. v0.16+ picks the next one based on operator demand.

### 3. Staging-only by default

Every ingested record lands in `~/.allbert/memory/staging/<source>/<id>.md` with frontmatter:

```yaml
kind: ingestion
source: <source>
ingest_kind: <search_query|page_visit|...>
captured_at: <iso>
attribution: <url-or-other>
provenance:
  source: ingest
  redacted: true
```

Promotion to durable memory remains the existing memory-curator review path. No record auto-promotes. No record affects the running turn unless explicitly retrieved.

A new staged-memory `kind: ingestion` is added to the existing schema (additive per ADR 0047). Memory-curator review workflow gains an "ingestion" filter so an operator can review the day's ingest separately from agent-staged candidates.

### 4. Adapter corpus opt-in

Once an ingested record is promoted to durable memory, it becomes eligible corpus material for the v0.13 adapter training pipeline under the existing `learning.adapter_training.include_tiers` rules. No new corpus tier; promotion is the gate.

The v0.12.2 secret redactor runs at ingest time (defensive first pass) and again at corpus-build time (existing second pass), preserving the current redaction posture.

### 5. Scheduled review job

A new bundled job template `ingestion-review` (disabled by default) compiles a daily markdown digest of the day's staged ingest records, grouped by source and kind, with one-line summaries. Operator runs through the existing memory-curator review surface. The job follows the v0.2 bounded schedule DSL and routes through `JobManager`.

### 6. Privacy posture documented

Add `docs/operator/ingestion.md` describing:

- What each ingest source captures and what it does not.
- How redaction applies (defensive double-pass at ingest and at corpus build).
- How to disable an ingest source (`/settings set ingestion.<source>.enabled false`).
- How to purge ingested data (`allbert-cli memory staged forget --kind ingestion --since <date>`).
- The default-off posture: every ingest source is opt-in per setup wizard or settings.

Add an ADR (0101) recording the staging-only, daemon-owned, single-endpoint posture.

## Non-goals (v0.15 only)

- No additional ingest sources beyond CLI stdin and the browser extension.
- No automatic promotion of ingested records.
- No silent ingestion. Every source is opt-in and visible in `/settings show ingestion`.
- No ingestion of full email or chat-message bodies — v0.15 captures metadata and explicitly-shared snippets.
- No remote daemon or hosted ingestion API. Local socket only.
- No ingestion → live turn context shortcut. Ingested records flow through staging → durable → retrieval, like every other memory artifact.

## Open questions for the full plan

- Should the browser extension live in this repo or a sibling repo? (Lean: sibling, so kernel build doesn't depend on browser-extension toolchain.)
- Do we want a `kind: ingestion` staged-memory tier, or reuse `kind: research` from the `record_as` flow? (Lean: distinct kind so reviewers can filter.)
- Does the daemon expose a localhost HTTP listener for the browser extension, or does the extension write to a watched file? (Lean: localhost HTTP behind the existing daemon socket; trust model unchanged from ADR 0023.)
- Do we add a per-source rate limit, or rely on global daemon caps? (Lean: per-source cap, since browser visit rates dwarf clipboard rates.)
- Does the personality digest (v0.11) consume ingested-then-promoted memory the same way it consumes other durable memory? (Lean: yes, no special-casing.)

## Dependencies

- v0.14.1 must land first. The doc-reality gate, daemon adapter wiring, and tool-call parser fix are prerequisites for adding new daemon protocol surfaces and new memory tiers without compounding the existing partial-completion problem.
- The v0.14.2 kernel split must land before implementation starts. New ingestion code lands in the services crate created by v0.14.2, not in the runtime core or compatibility facade.

## ADRs (planned)

- ADR 0101: Growth-loop ingestion is daemon-owned, single-endpoint, staging-only.
- ADR 0102: Browser extension lives in a sibling repo and posts to localhost.
- ADR 0103: `kind: ingestion` is a new staged-memory tier with promotion-gated corpus eligibility.
- ADR 0104: Protocol v7 ingestion message and per-peer filtering for v2-v6 clients.

## Exit criteria (full plan)

- A user piping `tail -f ~/.zsh_history | allbert-cli ingest stdin --source shell --kind note` produces staged-memory entries the operator can review.
- A browser extension installed locally posts each Google/DuckDuckGo search and each page visit to the daemon; entries land in staging.
- The daily `ingestion-review` job produces a markdown digest the operator reviews through memory-curator.
- Promoted ingested records flow into the v0.13 adapter corpus with full redaction; the resulting adapter training run shows the operator's voice/preferences are reflected.
- No ingest source is active by default on a fresh profile.
- `docs/vision.md` and `docs/notes/origin-2026-04-17.md` can both be checked off for the "grows with you" promise.

## Deferred to v0.16+

- Slack/Discord/iMessage/email/SMS ingestion.
- Screenshot/clipboard image OCR ingestion.
- Active-window/keystroke ingestion (privacy-heavy; needs separate ADR).
- Ingestion-driven adapter scheduling ("retrain when N new records").
- Ingestion-driven personality-digest refresh.
- Cross-device ingestion sync.
