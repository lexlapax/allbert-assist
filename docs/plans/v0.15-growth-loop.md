# v0.15 - RAG foundation and growth loop (stub)

Status: Stub

v0.15 is the release that finally addresses the origin note's central ask:
"when I search things in Google... it records things that I search and extracts
and remembers it and stores it for training the engines." Allbert was supposed
to *grow with the user* by ingesting daily activity.

The first implementation step is not ingestion itself. v0.15 starts with a
RAG foundation so Allbert can retrieve and explain its own operator docs, CLI
commands, settings, memory, and later ingested records through one bounded,
derived retrieval service. Without that substrate, ingestion would create more
candidate context than the runtime can use responsibly.

Through v0.14.3, the only semantic retrieval surface is memory-local,
JSON-backed, disabled by default, and fake-provider-only. Through v0.14.3, the
only ingestion path is the agent's own in-turn `web_search` with an opt-in
`record_as` parameter. v0.14.3 repairs router reliability first. v0.15 then
builds on that router by adding a SQLite-backed RAG substrate before adding the
growth-loop ingestion endpoint and browser capture proofs.

The original v0.15 growth-loop scope is still part of this plan. It moves later
in the release after the RAG foundation: the daemon ingestion endpoint,
CLI-piped feed, browser extension, browser proxy/PAC mode, staging-only ingest
records, daily ingestion review job, operator ingestion docs, and promoted
records joining adapter corpus eligibility are preserved as M5-M8. Sources that
were already out of scope remain deferred to v0.16+.

## Goals

- Add a rebuildable `RagService` in `allbert-kernel-services`.
- Keep markdown memory, docs, skills, and generated descriptors as ground
  truth; SQLite is a derived local index.
- Use SQLite metadata tables plus FTS for provider-free lexical retrieval.
- Add optional vector retrieval behind an owned embedding-provider seam.
- Make help-like questions retrieve operator docs, command descriptors,
  settings descriptors, and runbook snippets before the full prompt grows.
- Let memory queries retrieve durable memory, facts, and labelled episode
  history through the same bounded retrieval shape.
- Add growth-loop ingestion only after the retrieval substrate exists.

## Non-goals

- No RAG work landed in v0.14.3.
- No automatic promotion of ingested records.
- No ingestion source is active by default.
- No silent ingestion into live prompt context.
- No additional ingest sources beyond CLI stdin, browser extension, and browser
  proxy mode in v0.15.
- No full email or chat-message body ingestion in v0.15; only metadata and
  explicitly shared snippets are in scope.
- No remote daemon or hosted ingestion API. Browser-facing listeners bind to
  loopback only.
- No replacement of v0.14.3's schema-bound LLM router.
- No vector embedding call on every turn by default.
- No hosted embedding provider is required for provider-free validation.
- No `sqlite-vss`; v0.15 uses `sqlite-vec` for the first vector backend and
  tracks official SQLite `vec1` as a future candidate.

## Milestones

### M0 - RAG ADR, dependency proof, schema, config, and source inventory

Record the RAG decision in ADR 0106 and freeze the first implementation shape
before code starts.

- `allbert-kernel-services` owns concrete `RagService` behavior.
- `allbert-kernel-core` owns only config and DTO/contract types that frontends
  need.
- Derived artifacts live under `~/.allbert/index/rag/rag.sqlite`.
- SQLite tables hold source metadata, chunks, chunk-source provenance,
  lifecycle timestamps, content hashes, and retrieval scores.
- SQLite FTS is the provider-free lexical baseline.
- `sqlite-vec` is the first vector backend behind an internal trait.
- Official SQLite `vec1` is tracked but not used first because it is new and
  still documents testing limitations.
- `sqlite-vss` is excluded because its upstream project moved active work to
  `sqlite-vec`.

Config planned for v0.15:

```toml
[rag]
enabled = true
vector_enabled = false
embedding_provider = "none"
embedding_model = ""
max_chunks_per_turn = 6
max_chunk_bytes = 1200
sources = ["operator_docs", "commands", "settings", "memory"]
```

Allowed embedding providers are `none`, `ollama`, `openai`, and `fake`.
`fake` is for deterministic tests. Ollama is the preferred real local path.
OpenAI is optional, cost-logged, and daily-cost-cap gated.

### M1 - SQLite lexical RAG over operator docs, help, settings, and commands

Build provider-free RAG first.

- Index current operator docs, focused guides, feature-test runbook sections,
  CLI command descriptors, and settings descriptors.
- Chunk text deterministically with stable source IDs and content hashes.
- Provide lexical search over SQLite FTS with bounded snippets and provenance.
- Add operator diagnostics:

```bash
allbert-cli rag status
allbert-cli rag rebuild
allbert-cli rag search <query> [--source <source>]
```

- `rag rebuild` must be idempotent and recover from a deleted SQLite file.
- `rag status` reports schema version, source counts, chunk counts, vector
  posture, last rebuild time, and stale-source count.

### M2 - Embedding provider seam and SQLite vector backend

Layer semantic retrieval without making it mandatory.

- Introduce an owned `EmbeddingProvider` seam separate from chat completion.
- Implement fake embeddings for deterministic tests.
- Implement local Ollama embeddings as the preferred real provider.
- Implement optional OpenAI embeddings with existing provider cost logging and
  daily-cost-cap checks.
- Store vectors in SQLite through the internal vector backend trait.
- Use `sqlite-vec` as the first backend and keep it isolated behind one module.
- Do not run query embeddings for every turn; vector retrieval is used only
  when the selected source, router posture, or explicit operator query needs it.

### M3 - Memory, fact, and episode RAG integration

Bridge existing memory surfaces into the broader RAG substrate without weakening
review gates.

- Index approved durable memory, approved temporal facts, and labelled episode
  working history.
- Staged memory is searchable only for explicit review surfaces unless the
  operator requests staged review.
- Staged ingestion records are never trusted prompt context.
- Promoted ingested records enter RAG through the same durable-memory path as
  other promoted memory.
- Tantivy/BM25 remains the v0.5 memory search baseline while RAG becomes the
  cross-source retrieval substrate.

### M4 - Router and full-turn RAG context integration

Use RAG as evidence, not authority.

- Keep v0.14.3's schema-bound LLM router as the authority for intent and action
  drafts.
- Add a cheap pre-router RAG hint over command/help/settings sources only.
- Feed the router bounded retrieval hints, never full docs.
- For `meta` and help-like questions, retrieve operator docs and command
  snippets before the main assistant prompt.
- For `memory_query`, retrieve durable memory/facts/episode snippets through
  the RAG service.
- Normal task turns receive bounded RAG snippets only when router confidence or
  explicit user request justifies it.
- RAG hints must never create synthetic mutations, bypass tool policy, or
  replace the router's structured decision.

### M5 - Growth-loop ingestion endpoint

Add the single daemon-owned ingestion entry point after RAG exists.

- Protocol bumps to v7.
- New client message `IngestRecord(IngestRecordRequest)`.
- Payload:
  `{ source: String, kind: IngestKind, captured_at: ISO8601, text: Option<String>, url: Option<String>, attribution: Option<String>, redaction_hint: Option<RedactionHint> }`.
- `IngestKind`: `search_query`, `page_visit`, `host_visit`, `clipboard`,
  `chat_message`, `note`, `other`.
- Filesystem-trust scoped, identical to existing daemon IPC.
- Per-record byte cap defaults to 16 KiB.
- Per-source and per-day record caps are enforced by the daemon.

### M6 - Initial ingest adapters and browser capture paths

The endpoint is generic; v0.15 ships one low-risk CLI adapter plus two browser
capture paths:

1. **CLI piped feed.** `allbert-cli ingest stdin --source <name> [--kind <kind>]`
   lets operators pipe shell history, logs, notes, or clipboard-manager output
   into Allbert.
2. **Browser extension.** A minimal Chrome/Firefox/Safari extension repo
   (`allbert-browser-ingest/`, separate repo) posts exact page visits and
   search queries to `http://127.0.0.1:<daemon_port>/ingest`.
3. **Browser proxy mode.** An optional daemon-owned local HTTP proxy plus PAC
   file lets an operator point a browser profile at Allbert when an extension
   is unavailable or undesirable.

Browser proxy mode is intentionally narrower than the extension:

- It binds to loopback only, defaults off, and is visible under
  `/settings show ingestion`.
- It forwards traffic and emits normalized `IngestRecord` entries.
- For plaintext HTTP, it may capture the full URL after redaction.
- For HTTPS, it records only host, port, timestamp, and source browser profile.
- It does not install a root certificate, intercept TLS, inspect bodies, cache
  responses, or claim exact search-query capture through proxy traffic.
- `CONNECT` is restricted to web-safe ports by default.

Skip Slack, Gmail, iMessage, SMS, and screenshots in v0.15. v0.16+ picks the
next source based on operator demand.

### M7 - Staged ingestion review and promoted-record indexing

Every ingested record lands in staging first:

```yaml
kind: ingestion
source: <source>
ingest_kind: <search_query|page_visit|host_visit|...>
captured_at: <iso>
attribution: <url-or-other>
provenance:
  source: ingest
  redacted: true
```

- Promotion to durable memory remains the existing memory-curator review path.
- No record auto-promotes.
- No staged record affects the running turn unless explicitly retrieved through
  a review surface.
- Memory-curator gains an `ingestion` filter.
- Promoted ingested records become eligible for RAG indexing and for the v0.13
  adapter corpus under existing `learning.adapter_training.include_tiers`
  rules.
- The v0.12.2 secret redactor runs at ingest time and again at corpus-build
  time.

### M8 - Scheduled review job and operator docs

- Add a bundled `ingestion-review` job template, disabled by default.
- The job compiles a daily markdown digest of staged ingest records grouped by
  source and kind.
- Operators review the digest through the existing memory-curator workflow.
- Add `docs/operator/ingestion.md` documenting each source, what it captures,
  what it does not capture, redaction, disabling sources, purging staged ingest,
  and the default-off posture.

## Open questions for the full plan

- Should the browser extension live in this repo or a sibling repo? Lean:
  sibling, so the kernel build does not depend on browser-extension tooling.
- Is browser proxy mode release-blocking or experimental? Lean: release-blocking
  because it directly addresses "point the browser at Allbert," but keep it
  host/URL metadata only.
- Does the daemon expose one localhost HTTP listener for extension and proxy
  ingestion, or separate listeners? Lean: separate loopback listeners with
  shared ingestion normalization.
- How much RAG hint context is safe before the router? Lean: source/title/snippet
  only, command/help/settings sources only, and no vector query by default.
- Does personality digest consume ingested-then-promoted memory the same way it
  consumes other durable memory? Lean: yes, no special-casing.

## Dependencies

- v0.14.1 reconciliation is shipped. The doc-reality gate, daemon adapter
  wiring, trainer factory, remediation provenance, and tool-call parser fixes
  keep new daemon protocol surfaces and memory tiers from compounding older
  partial-completion drift.
- v0.14.2 is the shipped architectural baseline. New RAG and ingestion code
  lands in `allbert-kernel-services`, not in `allbert-kernel-core`.
- v0.14.3 lands first. Its schema-bound router gives RAG hints a bounded
  consumer and keeps schedule/memory mutations away from keyword routing.

## ADRs (planned)

- ADR 0101: Growth-loop ingestion is daemon-owned, single-endpoint,
  staging-only.
- ADR 0102: Browser extension lives in a sibling repo and posts to localhost.
- ADR 0105: Browser proxy mode is loopback-only, host/URL metadata only, and no
  TLS interception in v0.15.
- ADR 0103: `kind: ingestion` is a new staged-memory tier with
  promotion-gated corpus eligibility.
- ADR 0104: Protocol v7 ingestion message and per-peer filtering for v2-v6
  clients.
- ADR 0106: RAG index is a derived SQLite lexical/vector store.

## Exit criteria (full plan)

- Deleting `~/.allbert/index/rag/rag.sqlite` and running `allbert-cli rag rebuild`
  restores provider-free lexical retrieval from docs/help/settings/commands.
- `allbert-cli rag search "configure Telegram"` returns operator-doc snippets
  with source paths and bounded excerpts.
- A help-like prompt such as `help me configure Telegram` retrieves current
  operator docs/runbook snippets before the full assistant answer.
- A promoted durable memory entry appears in RAG; an unpromoted staged memory
  entry does not auto-inject into ordinary prompt context.
- Fake-vector tests pass deterministically; optional Ollama/OpenAI vector smokes
  are documented but not required for contributor validation.
- A user piping `tail -f ~/.zsh_history | allbert-cli ingest stdin --source shell --kind note`
  produces staged-memory entries the operator can review.
- A browser extension installed locally posts each Google/DuckDuckGo search and
  each page visit to the daemon; entries land in staging.
- A browser profile pointed at the local proxy produces staged host-visit
  records for HTTPS `CONNECT` traffic and full-URL visit records only for
  plaintext HTTP.
- The daily `ingestion-review` job produces a markdown digest the operator
  reviews through memory-curator.
- Promoted ingested records flow into RAG and the v0.13 adapter corpus with full
  redaction.
- No ingest source is active by default on a fresh profile.
- `docs/vision.md` and `docs/notes/origin-2026-04-17.md` can both be checked
  off for the "grows with you" promise.

## Test plan

- Provider-free tests build and rebuild the SQLite RAG index from docs, command
  descriptors, and settings descriptors.
- Lexical search tests assert source filters, stable snippets, deleted-DB
  recovery, and stale-source detection.
- Vector tests use fake embeddings for deterministic CI.
- Optional live smokes cover Ollama embeddings and OpenAI embeddings outside the
  default contributor gate.
- Router tests assert RAG hints improve help/schedule/memory routing without
  becoming keyword authority or mutation authority.
- Help smoke: `help me configure Telegram` retrieves operator-doc snippets.
- Memory smoke: promoted durable memory appears in RAG; staged memory does not
  auto-inject.
- Ingestion smoke: staged browser/search records are searchable for review, and
  only promoted records join normal RAG context.
- Validation: `cargo fmt --check`, clippy with `-D warnings`, full tests, CLI
  help, and `tools/check_doc_reality.sh`.

## Deferred to v0.16+

- Slack/Discord/iMessage/email/SMS ingestion.
- Screenshot/clipboard image OCR ingestion.
- Active-window/keystroke ingestion.
- TLS-intercepting browser proxy mode with local CA installation, if ever
  accepted; separate security ADR required.
- Ingestion-driven adapter scheduling.
- Ingestion-driven personality-digest refresh.
- Cross-device ingestion sync.
