# Allbert v0.31 Plugin And App Generator

Request flow: `docs/plans/v0.31-request-flow.md`

Previously planned as v0.29 before the project-direction rethink inserted
v0.23 Jido State-Machine Convergence and v0.24 Objective Runtime Foundation.
The post-v0.26 roadmap reconciliation keeps the generator at v0.31.

## Purpose

v0.31 adds generators only after StockSage proves the plugin, app surface,
app memory, and app canvas contracts manually. The generator should encode the
boring, proven shape of an Allbert plugin and app rather than inventing
abstractions early.

## Expected Inputs

- v0.17 plugin contract has been exercised by shipped channel plugins and
  StockSage's plugin entrypoint.
- v0.18 app/surface contract exists.
- v0.20 proves StockSage as a plugin-contributed app.
- v0.24 Objective Runtime Foundation exists; generated apps may opt into
  objective scaffolding when their actions are multi-step.
- v0.26 workspace and surface substrate exists.
- v0.27 App Surface Contract proves StockSage `SurfaceProvider` LiveViews,
  real app-owned renderers, validated Surface-node output, and memory
  namespace declaration.
- v0.29 App Memory + Outcomes Contract proves explicit namespace-scoped memory
  consumption and app outcome workflows.
- v0.30 App Canvas Contract proves durable canvas emission through audited
  workspace APIs.

## Expected Direction

- Add `mix allbert.gen.plugin MyPlugin` to scaffold
  `./plugins/my_plugin/` with `allbert_plugin.json`, README, `lib/`,
  optional `skills/`, and validation docs.
- Add `mix allbert.gen.app MyApp` as an app-oriented plugin scaffold.
- Generate the boring local plugin/app shape that v0.15, v0.17, v0.18, v0.20,
  v0.26, v0.27, v0.29, and v0.30 proved: plugin manifest, plugin module, app
  module, app supervision wiring, sample Jido action, sample `SKILL.md`, sample
  SurfaceProvider or Surface node, sample Ecto domain stub, memory namespace
  stub, optional objective scaffolding, canvas wiring stub, and validation
  documentation.
- Ensure `mix allbert.validate_plugin MyPlugin` passes immediately.
- Ensure `mix allbert.validate_app MyApp` passes immediately.
- Generated code is inert by default. It does not automatically add arbitrary
  plugin `lib/` paths to compilation, trust or enable generated skills, grant
  action permissions, register external publishing, or perform side effects
  without explicit operator action through existing Allbert boundaries.
- Consider optional `mix allbert.publish_skills` only after local validation,
  trust, and explicit operator review are boring and safe.

## Blocker Shapes To Pin Before Implementing

This plan is intentionally just-in-time (v0.31 is five versions out and would
drift if fully specified now). Before M-level work starts, pin:

- **Mix task arg signatures**: `mix allbert.gen.plugin NAME [--into PATH]
  [--force]`, `mix allbert.gen.app NAME [--namespace ATOM] [--into PATH]
  [--force]`, `mix allbert.validate_plugin NAME`, `mix allbert.validate_app
  NAME`, and (optional) `mix allbert.publish_skills`.
- **Scaffold output manifest**: the exact file tree each task writes and a
  template skeleton per file — `allbert_plugin.json` keys (incl.
  `trusted: false`, `enabled: false`), plugin module, app module + the
  `AllbertAssist.App` callbacks it stubs (incl. `memory_namespace/0` from v0.27),
  supervision wiring, sample action, `SKILL.md`, SurfaceProvider/Surface node,
  Ecto stub, objective scaffold, canvas wiring stub, validation docs.
- **"Inert by default" as testable assertions**: generated manifest has
  `trusted: false`; generated skill `enabled: false`; no `config/` compile-path
  entry added; `Actions.Registry` gains zero entries until the operator wires it;
  the generator itself runs no registered action and makes no Security Central
  call.

Then expand the milestone bodies to the v0.26 milestone shape
(Goal/Depends/Parallel-safe/tasks/Tests/smoke/Acceptance).

## Milestones

- M1: Generator contract inventory - freeze the scaffold fields from the
  proven plugin, app surface, memory, objective, action, and canvas contracts.
- M2: Plugin generator - add `mix allbert.gen.plugin` with inert manifest,
  README, optional skill root, and validation docs.
- M3: App generator - add `mix allbert.gen.app` with app module,
  SurfaceProvider stub, action stub, memory namespace stub, objective stub, and
  canvas wiring stub.
- M4: Validation tasks and docs - ensure generated outputs pass
  `mix allbert.validate_plugin` and `mix allbert.validate_app` immediately and
  explain explicit compile-path/project integration.
- M5: Safety closeout - prove generated code grants no trust, permissions,
  skill enablement, external publishing, dynamic loading, or execution
  authority.

## Non-Goals

- No generator before the contract is proven.
- No autonomous app or skill creation from traces, prompts, or repeated tasks.
- No runtime dynamic module loading from arbitrary plugin, app, or skill
  folders.
- No automatic compile-path changes for generated plugin code without explicit
  operator/developer review.
- No generated execution authority, automatic trust, automatic skill
  enablement, or automatic permission grants.
- No marketplace publish without explicit operator action and review.
- No AG-UI, A2UI, MCP Apps, or external UI protocol bridge.

## Test Plan

- Generated plugin compiles and validates.
- Generated app plugin compiles and validates.
- Generated docs explain the explicit compile-path or project dependency step
  needed for code-bearing source-tree plugins.
- Generated skill is inert until explicitly enabled/trusted.
- Generated action goes through the registry, action runner, and Security
  Central.
- Generated surface data validates through `AllbertAssist.Surface` and does
  not invent component ids or action bindings.
- Generated memory namespace stub is declaration-only until app code adds
  explicit memory actions.
- Generated canvas wiring uses the proven v0.30 path and grants no authority.
- Generator output includes clear validation docs for the created plugin/app.

## Definition Of Done

v0.31 is done when a new local Allbert plugin and app plugin can be
scaffolded, validated, and inspected without granting unsafe capability,
publishing externally, loading untrusted runtime code, or scaffolding
unproven app-contract layers.
