# Allbert v0.31 Workspace-Only App UI And Settings Central

Request flow: `docs/plans/v0.31-request-flow.md`

ADR: `docs/adr/0024-app-ui-contribution-and-workspace-zones.md`

Inserted after v0.30. v0.31 is the workspace product reset: `/workspace`
becomes the only operator home, app UI moves into host-owned workspace panels,
and Settings Central becomes a workspace utility panel. User theming moves to
v0.32. The Plugin And App Generator moves to v0.33 so it can scaffold the
workspace-panel and theming contracts after both are proven manually.

## Purpose

Allbert currently has multiple operator UI worlds:

- `/agent` is the dynamic workspace shell with chat, durable canvas tiles, and
  ephemeral surfaces.
- `/settings` is a separate Settings Central page.
- StockSage has its own `/stocksage/*` pages and app shell.

v0.31 collapses those into one product surface. The default operator
experience is `/workspace`; apps contribute catalog-validated panels into
named workspace zones; Settings Central lives inside the workspace utility
drawer; rare app-specific pages live under `/apps/<app_id>/...`.

There is no compatibility requirement in v0.31. `/agent`, `/settings`, and
`/stocksage/*` are removed rather than redirected. Historical docs may still
describe those routes for v0.30 and earlier, but new implementation work must
target `/workspace`.

v0.31 changes UI composition and navigation. It adds no new domain behavior, no
new analysis engine, no new execution authority, no theming override system,
and no generator.

## Expected Inputs

- v0.18/ADR 0015 app contract, Surface DSL, `SurfaceProvider`, and catalog
  validation exist.
- v0.26/ADR 0023 workspace shell exists as a Surface tree with per-thread
  canvas, ephemeral surfaces, signed Fragment emission, offline editing, and
  the `Workspace.Fragment.Guard` emitter allow-list.
- v0.27 StockSage owns real `:analysis_card`, `:agent_report_card`,
  `:parity_card`, and `:debate_round_card` renderers and page LiveViews.
- v0.28 security harness covers catalog bypass, component injection, fragment
  replay/tampering, emitter allow-list, app-scope routing, and namespace
  isolation.
- v0.30 StockSage emits durable canvas tiles through
  `AllbertAssist.Workspace.Emitters.stocksage_signal/2`.
- Settings Central actions, confirmation decisions, remembered grants, and
  provider-key controls already exist behind action/security boundaries.

## Design Decisions

### 1. `/workspace` is canonical

The host router exposes `/workspace` as the operator product surface. The old
operator routes are intentionally removed:

| Old route | v0.31 behavior |
|---|---|
| `/agent` | no route |
| `/settings` | no route; Settings Central is a workspace utility panel |
| `/stocksage` | no route |
| `/stocksage/analyses` | no route |
| `/stocksage/queue` | no route |
| `/stocksage/trends` | no route |

Tests should assert absence, not redirects. Links, docs, nav entries, service
worker scope, and Chrome manual verification all use `/workspace`.

### 2. App UI has two tiers: panels by default, pages only when necessary

`AllbertAssist.App.SurfaceProvider` gains a workspace panel contribution tier:

- **Panel surface**: a catalog-validated Surface subtree targeting a host-owned
  workspace zone. This is the default for dashboards, lists, status, settings,
  queues, trend summaries, and operational controls.
- **Page surface**: a host-declared route under `/apps/<app_id>/...`, reserved
  for self-contained, deep-linkable, page-shaped flows that cannot reasonably
  fit as panels.

Routes remain compile-time declared in the host router. v0.31 does not add
dynamic route creation. App metadata can request panel placement and
visibility, but the host owns final layout and route authority.

### 3. Host-owned workspace zones

`AllbertAssist.Workspace.Catalog` declares a fixed zone set for v0.31:

| Zone | Location | Purpose |
|---|---|---|
| `:nav_apps` | left rail | app launcher entries |
| `:context_rail` | left rail / contextual stack | active app status, filters, compact panels |
| `:canvas_panels` | right canvas | app dashboards, lists, summaries, durable tile-adjacent panels |
| `:utility_drawer` | drawer/sheet | Settings Central, security review shortcuts, workspace tools |
| `:ephemeral` | overlay | existing v0.26 ephemeral surfaces |

Expanding this set later is a catalog amendment. Unknown zones are rejected.
`zone`, `visible_when`, and ordering metadata are never authority.

### 4. ChatGPT-style workspace shell

`CoreApp.surfaces/0` declares a workspace shell with:

- Collapsible left rail: new-thread action, recent thread list, app launcher,
  and context panels.
- Center thread: existing chat/runtime interaction.
- Right workspace: durable canvas tiles and `:canvas_panels`.
- Utility drawer: Settings Central and workspace tools.
- Mobile navigation: stable drawer/tab states for nav, chat, canvas, and
  utility surfaces.

New structural catalog atoms are added for host chrome:
`:workspace_shell`, `:nav_rail`, `:thread_list`, `:app_launcher`,
`:utility_drawer`, and `:workspace_panel`. These are host structural atoms, not
model-facing arbitrary UI.

### 5. Settings Central moves into the workspace

The existing Settings Central behavior is preserved but rendered inside
`/workspace`:

- Settings list/search/update.
- Provider-key storage and redaction.
- Permission and confirmation policy controls.
- Remembered resource grants review/revoke.
- Security Central status and review shortcuts when already exposed.

The panel calls the same registered actions and contexts as the current
Settings LiveView. LiveView state does not own settings semantics, secret
storage, permission decisions, or confirmation policy.

### 6. StockSage becomes workspace-panel based

StockSage's dashboard, recent analyses, queue, and trends become workspace
panels. StockSage private navigation and app-shell chrome are removed.

`/apps/stocksage/analyses/:id` is allowed only if analysis detail needs
page-level focus for deep-linking, printing, or long-form review. If the
implementation can provide equivalent ergonomics as a workspace panel/modal,
that page route should be omitted and analysis detail should remain inside
`/workspace`.

### 7. One panel mechanism for built-in and plugin apps

CoreApp domain UI, such as objective, job, confirmation, and security/status
cards, uses the same panel-zone mechanism as StockSage. Built-in UI does not
get a private composition path.

### 8. Security boundaries remain unchanged

Panels and zones add no authority:

- Panel nodes validate through the Surface catalog.
- Dynamic durable tile data still flows through the v0.30 signed Fragment path.
- Settings changes still flow through registered actions and Security Central.
- App-scoped actions still require explicit active app scope.
- Plugin/app metadata never grants permissions.

## Module Shapes

- `AllbertAssist.Surface` / validator: add `kind: :panel` with `zone`, `node`,
  `visible_when`, `order`, and stable `id`; reject unknown zones/components and
  unauthorized action bindings.
- `AllbertAssist.Workspace.Catalog`: add zone registry, structural atoms, panel
  composition, and `/workspace` tree builder.
- `AllbertAssist.App.CoreApp`: declare the canonical `/workspace` shell and
  CoreApp panel surfaces.
- `AllbertAssistWeb.WorkspaceLive`: becomes the single workspace LiveView; old
  route names are not retained.
- `AllbertAssistWeb.Workspace.Components`: add nav rail, thread list, app
  launcher, utility drawer, panel wrapper, and Settings Central panel
  components.
- `StockSage.App` / `StockSageWeb`: declare dashboard/recent/queue/trends as
  panel surfaces; remove private app shell and old page routes; keep or omit
  `/apps/stocksage/analyses/:id` per the analysis-detail decision above.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless the implementation owner explicitly records a narrower
gate and why.

### M1: Canonical `/workspace` route and route removal

Goal: `/workspace` is the only operator home; old operator routes are gone.

Tasks:

- Add host route for `/workspace`.
- Remove `/agent`, `/settings`, and `/stocksage/*` routes.
- Update app links, nav targets, service-worker scope, tests, and docs to use
  `/workspace`.
- Leave non-operator/deep-link routes such as `/jobs` and `/objectives/:id`
  unchanged unless a focused implementation blocker requires a separate plan.

Tests:

- Router tests assert `/workspace` renders.
- Router tests assert `/agent`, `/settings`, and `/stocksage/*` are absent.
- Existing runtime/chat tests run through `/workspace`.

Acceptance:

- A fresh browser can start at `/workspace`.
- Old operator route requests do not redirect or render compatibility pages.

### M2: Named zones and panel surface validation

Goal: apps can declare validated panels into host-owned zones.

Tasks:

- Add zone registry and `:panel` surface kind.
- Validate panel node, zone, visibility metadata, action bindings, and catalog
  atoms at registration/render time.
- Add diagnostics for dropped/rejected panels.
- Add a fixture app that contributes one valid and several invalid panels.

Tests:

- Surface validator tests for valid panels, unknown zones, unknown components,
  raw HTML attempts, foreign action bindings, and secret-like props.
- Security eval additions for panel catalog bypass and zone injection.

Acceptance:

- A fixture panel renders in the expected zone.
- Invalid panels are rejected with bounded diagnostics and no workspace crash.

### M3: Workspace shell and navigation

Goal: `/workspace` renders the new shell with rail, thread, canvas, and utility
drawer.

Tasks:

- Add structural catalog atoms and components.
- Render recent threads from existing conversation history.
- Render registered apps in the app launcher.
- Preserve current canvas tile rendering, ephemeral surfaces, offline editing,
  split behavior, and accessibility affordances.
- Add mobile drawer/tab behavior for nav, chat, canvas, and utility.

Tests:

- Workspace LiveView tests for rail, thread switching, app launcher, canvas
  persistence, ephemeral surfaces, keyboard/focus, and mobile layout handles.

Acceptance:

- Chrome verification at desktop and narrow widths shows no overflow, no
  overlapping text, reachable controls, and preserved chat/canvas behavior.

### M4: Workspace Settings Central

Goal: Settings Central is available inside `/workspace`, not `/settings`.

Tasks:

- Move settings list/search/edit, provider keys, permission controls,
  confirmation decisions, and remembered grants into the utility drawer.
- Reuse existing settings actions and redaction helpers.
- Ensure writes still produce audit records and security decisions exactly as
  before.
- Remove standalone Settings LiveView navigation and tests.

Tests:

- Workspace tests for opening the utility drawer, listing settings, updating a
  safe setting, redacted provider-key flow, approving/denying confirmations,
  and revoking grants.
- Negative tests for direct UI bypass of settings/security actions.

Acceptance:

- Operator can complete the current Settings Central smoke entirely from
  `/workspace`.
- `/settings` is absent.

### M5: CoreApp panels

Goal: built-in domain UI proves the same panel path plugins use.

Tasks:

- Re-declare objective, job, confirmation, security/status, and relevant
  workspace tool cards as panel surfaces.
- Remove private placement paths that bypass the zone model.
- Keep domain behavior behind existing contexts/actions.

Tests:

- Panel rendering tests for each built-in card family.
- Regression tests that action execution and confirmations remain unchanged.

Acceptance:

- CoreApp cards appear in zones through panel composition.
- No built-in-only UI composition path remains for these cards.

### M6: StockSage workspace panels

Goal: StockSage dashboard/recent/queue/trends are workspace panels.

Tasks:

- Declare StockSage dashboard, recent analyses, queue, and trends as panel
  surfaces.
- Source panel data through existing read contexts/actions, not page LiveView
  authority.
- Delete private StockSage app shell/nav chrome and removed page LiveViews.
- Decide during implementation whether analysis detail remains
  `/apps/stocksage/analyses/:id` or becomes a workspace panel/modal; document
  the chosen outcome in the request-flow closeout.

Tests:

- StockSage panel tests for empty/loading/error states and real card renderers.
- Workspace tests for app switching to StockSage and rendering all panels.
- Route tests for removed `/stocksage/*` paths.

Acceptance:

- StockSage operator workflows are reachable from `/workspace`.
- No StockSage dashboard/list/queue/trend route or private nav remains.

### M7: Security, route, and documentation closeout

Goal: v0.31 is release-ready and documented.

Tasks:

- Extend v0.28 eval harness for panel catalog bypass, zone injection, settings
  action bypass, app-scope preservation, and removed-route assertions.
- Update README, roadmap, ADR 0024, agent context map, and request-flow logs.
- Bump version metadata to `0.31.0`.
- Run focused suites, Chrome verification, and full gate.

Tests:

- `mix format --check-formatted`
- `mix compile --warnings-as-errors`
- `mix credo --strict`
- `mix dialyzer`
- `mix precommit`
- `git diff --check`

Acceptance:

- Docs, tests, and Chrome manual verification agree that `/workspace` is the
  only operator home and app UI is panel-based inside the workspace.

## Non-Goals

- No `/agent`, `/settings`, or `/stocksage/*` redirects or compatibility pages.
- No user theming, token files, CSS snippets, layout YAML, or theme controller;
  that is v0.32.
- No plugin/app generator; that is v0.33.
- No dynamic/runtime route creation.
- No arbitrary plugin HTML/CSS/JS execution.
- No AG-UI/A2UI/MCP Apps public protocol bridge.
- No new StockSage analysis behavior, outcome model, memory write path, or
  canvas emission mechanism.

## Test Plan

- Router tests prove canonical `/workspace` and removed legacy operator routes.
- Surface tests prove panel validation, named-zone validation, and rejection of
  arbitrary UI.
- Workspace LiveView tests cover shell, thread switching, app launcher,
  utility drawer, settings flows, CoreApp panels, StockSage panels, canvas
  persistence, ephemeral surfaces, focus, keyboard, and mobile behavior.
- Security evals cover panel catalog bypass, zone injection, settings action
  bypass, app-scope preservation, and removed-route access attempts.
- Chrome verification covers desktop and narrow viewport `/workspace`, app
  switching, Settings Central drawer, StockSage panels, and any retained
  `/apps/stocksage/analyses/:id` detail route.

## Definition Of Done

v0.31 is done when `/workspace` is the single operator home, Settings Central
is a workspace utility panel, CoreApp and StockSage UI compose through the same
catalog-validated panel/zone mechanism, old operator routes are removed without
compatibility redirects, and the release gate plus Chrome verification pass.
