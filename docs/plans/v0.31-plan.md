# Allbert v0.31 Workspace-Native Plugin UI And User Theming

Request flow: `docs/plans/v0.31-request-flow.md`

ADR: `docs/adr/0024-app-ui-contribution-and-workspace-zones.md`
ADR: `docs/adr/0025-user-theming-and-override-security.md`

Inserted into the roadmap after v0.30. The Plugin And App Generator that
previously held the v0.31 slot moved to v0.32 (`docs/plans/v0.32-plan.md`) so
the generator can scaffold the panel-surface, named-zone, `/apps/<app_id>`
route, and theming shape that this release proves.

## Purpose

v0.31 closes the gap between Allbert's two parallel UI worlds. Today `/agent`
is a fully dynamic Surface-tree workspace, while apps like StockSage ship their
own hand-written full-page LiveViews under hardcoded `/stocksage/*` routes with
private navigation and ~2,000 lines of embedded Tailwind. v0.31 makes the
workspace the primary surface for app UI, reserves dedicated routes only for
genuinely page-shaped views, and lets operators retheme and re-lay-out the UI
from `<ALLBERT_HOME>` without editing core code.

It graduates the "Workspace Hooks" / plugin-contributed workspace regions
reserved in ADR 0023 §1, and adds the first user-facing theming/override
substrate (net-new; not previously parked in `future-features.md`).

v0.31 changes how UI is *contributed and styled*. It adds no new domain
behavior, no new analysis engine, and no new execution authority.

- `AllbertAssist.Paths` resolves `<ALLBERT_HOME>` subroots with a
  config/env/default precedence the theming root will extend.

## Design Overview

Two contribution tiers, one shell, three theming layers. Prior art (VS Code
`viewsContainers` vs `views`, Grafana app-page vs panel, Backstage
`PageBlueprint` vs `EntityCardBlueprint`, Home Assistant panel vs Lovelace
card, JupyterLab named shell areas, Discourse plugin outlets) converges on the
same rule: the host owns the layout and named slots; an app declares *what* and
*which slot*, never the container. Allbert already has the substrate for this —
a declarative Surface tree dispatched through a curated catalog — so v0.31 adds
a contribution tier and a zone registry rather than a new rendering mechanism.

## Implementation Decisions

### 1. Two-tier app UI contribution (page vs panel)

`AllbertAssist.App.SurfaceProvider` surfaces gain an explicit kind split:

- **Page surface** (existing behavior, now namespaced): owns a route. Used only
  for self-contained, deep-linkable, page-shaped views (e.g. a full analysis
  detail/print view). Rare.
- **Panel surface** (new): a validated Surface subtree that targets a
  host-owned named zone in the workspace. This is the default contribution for
  dashboards, lists, summaries, and status. It is composed into `/agent`, never
  given a route.

Panel surfaces are declarative Surface nodes validated against the catalog at
registration and render time — identical to every other Surface. No app ships
arbitrary HTML/CSS/JS for a panel; it composes catalog atoms (StockSage's four
cards are already catalog atoms).

### 2. Host-owned named zones

`AllbertAssist.Workspace.Catalog` declares a small, fixed set of named zones.
v0.31 ships exactly these; expanding the set later is a catalog amendment, not
an open registration surface:

| Zone | Location | Purpose |
|---|---|---|
| `:nav_apps` | left rail | app launcher entries (one per registered app) |
| `:context_rail` | left rail (lower) | contextual/status panels |
| `:canvas_panels` | right canvas | app dashboards/lists/summaries as panels |
| `:ephemeral` | overlay | existing v0.26 ephemeral surfaces |

`workspace_tree/1` composes the `CoreApp` base shell tree plus every registered
app's panel surfaces, each rendered into its declared zone. A panel's `zone`
and optional `visible_when` (e.g. `%{active_app: :stocksage}`) are
ranking/visibility metadata only — never authority (consistent with ADR
0019/0021).

### 3. `/apps/<app_id>` route namespace

Page surfaces standardize on `/apps/<app_id>/...`. Routes remain
**compile-time declared** in the host router; v0.31 does not add dynamic route
creation (explicitly out of scope per the agent-context-map workspace note and
to avoid Phoenix macro/runtime risk). The generator (v0.32) templates the route
block. StockSage reduces to a single page route:
`/apps/stocksage/analyses/:id` → `StockSageWeb.AnalysisLive` (`:show`). A
courtesy redirect from `/stocksage/*` to the new locations is optional, not a
maintained compatibility layer.

### 4. ChatGPT-style workspace shell

`CoreApp.surfaces/0`'s workspace tree becomes a three-zone shell: a collapsible
left rail (new-chat + thread history from existing `recent_threads/1` + the
`:nav_apps` app launcher + `:context_rail`), a center chat thread, and a right
canvas that holds `:canvas_panels` and durable tiles. New structural catalog
atoms `:nav_rail`, `:thread_list`, and `:app_launcher` are required; this is an
ADR 0015 catalog amendment for host structural chrome (not model-facing
component types). The existing two-pane behaviors (offline tile editing,
WCAG-AA accessibility, mobile responsiveness, split resizer) are preserved; the
mobile `data-mobile-tab` system gains a `nav` drawer state.

### 5. Uniform path for built-in and plugin apps

For consistency, `CoreApp`'s own domain cards (objective, job, confirmation)
migrate to panel surfaces in zones using the same tier-2 mechanism. `CoreApp`
is the reference implementer of the panel contract; StockSage is the second.
Neither gets a private composition path.

### 6. User theming and override layers

A new `Paths.themes_root/0` → `<ALLBERT_HOME>/themes` (added to
`ensure_home!/0`) and a new `AllbertAssistWeb.ThemeController` serve
operator-owned styling, because `Plug.Static` only serves app `priv` and cannot
reach `<ALLBERT_HOME>`. Three layers, each opt-in via Settings Central
`workspace.theme.*`:

1. **Design tokens (default, safe).** `<ALLBERT_HOME>/themes/<name>.yaml` is a
   constrained token map (colors, spacing, fonts, radii). The host emits a
   `:root { --allbert-*: …; --color-*: … }` block served at `/theme/user.css`,
   linked after `app.css` in `root.html.heex`. Tailwind v4 compiles theme
   tokens to runtime `:root` variables, so reassigning them retints the whole
   UI with no asset rebuild.
2. **CSS snippets (opt-in, strip-and-warn).** `<ALLBERT_HOME>/themes/snippets/`
   holds raw `.css`, enabled per-file. Served last (outside Tailwind's `@layer`s
   so it wins). A sanitizer **rejects `@import` and strips `url()`/`image-set()`
   with an operator-visible warning** before serving — raw user CSS is a real
   data-exfiltration surface (see ADR 0025). Default disabled.
3. **Layout override (data, validated).** `<ALLBERT_HOME>/workspace/layout.yaml`
   enables/disables/reorders zones and pins which panels appear where. It is a
   selection of catalog-allowed atoms and registered zones — never code —
   validated against the catalog and the registered surfaces from decisions
   1–2.

A Content-Security-Policy is added to the `:browser` pipeline (`style-src
'self'`, `img-src 'self'`, no remote fetch); there is none today beyond
`put_secure_browser_headers`. ADR 0025 owns the theming/override security
posture.

## Module Shapes

- `AllbertAssist.Surface` / `AllbertAssist.Surface.Validator`: add the `:panel`
  surface kind with `zone`, `node`, and `visible_when`; reject unknown zones and
  non-catalog nodes.
- `AllbertAssist.Workspace.Catalog`: add the named-zone registry, the
  `:nav_rail`/`:thread_list`/`:app_launcher` atoms, and panel composition in
  `workspace_tree/1`.
- `AllbertAssistWeb.Workspace.Components.*`: new `NavRail`, `ThreadList`,
  `AppLauncher` LiveComponents; existing renderer dispatch unchanged.
- `AllbertAssist.App.CoreApp`: three-zone shell tree; objective/job/confirmation
  domain cards re-declared as panel surfaces.
- `StockSage.App` / `StockSageWeb`: dashboard/list/queue/trends become panel
  surfaces; only `AnalysisLive` (`:show`) survives as a page; `AppShell.nav`
  deleted.
- `AllbertAssist.Paths`: `themes_root/0`, `themes_snippets_root/0`; both added
  to `ensure_home!/0`.
- `AllbertAssistWeb.ThemeController` + `AllbertAssist.Theme.{Tokens,Sanitizer,Layout}`:
  read, validate, render, sanitize, and serve token CSS, snippet CSS, and layout
  config.
- Settings Central: `workspace.theme.active`, `workspace.theme.snippets_enabled`,
  `workspace.theme.enabled_snippets`, `workspace.layout.override_enabled`.

## Milestones

Each milestone is independently buildable, testable, and leaves the app in a
working state.

### M1: Named-zone substrate and panel composition

Status: planned.

Goal: `Workspace.Catalog` declares host-owned zones and `workspace_tree/1`
composes registered panel surfaces into them. No app contributes panels yet;
zones render empty.

Depends on: v0.26 catalog/renderer.
Parallel-safe: foundational; nothing else can land first.

Tasks:

- Add the zone registry (`:nav_apps`, `:context_rail`, `:canvas_panels`,
  `:ephemeral`) and zone validation.
- Compose base shell + (empty) per-app panel lists in `workspace_tree/1`.
- Renderer renders an empty zone as a stable, accessible placeholder.

Tests: `workspace/catalog_test`, `workspace/renderer_test`.

Acceptance: zones resolve and render; unknown zones are rejected; existing
`/agent` behavior is unchanged.

### M2: `:panel` surface kind and validation

Status: planned.

Goal: `SurfaceProvider` apps can declare panel surfaces validated against the
catalog and zones.

Depends on: M1.
Parallel-safe: no (Surface contract change).

Tasks:

- Extend `AllbertAssist.Surface` with `kind: :panel`, `zone`, `node`,
  `visible_when`.
- Validate panel nodes against the catalog; reject arbitrary HTML, unknown
  components, unknown zones, and action bindings the app may not bind.
- Keep `zone`/`visible_when` as metadata, never authority.

Tests: `surface_test`, `surface/validator_test`, v0.28 eval addition for panel
catalog bypass.

Acceptance: a fixture app contributes a valid panel into `:canvas_panels`;
invalid panels are rejected with diagnostics.

### M3: ChatGPT-style shell

Status: planned.

## Non-Goals

- No dynamic/runtime route creation; page routes stay compile-time.
- No arbitrary model-generated HTML/JS; panels remain catalog-validated Surface
  nodes.
- No new domain behavior, analysis engine, outcome record, or execution
  authority.
- No new emission mechanism; dynamic tiles still use the v0.30 signed Fragment
  path and emitter allow-list.
- No AG-UI/A2UI/MCP-Apps public protocol bridge (post-v0.31 UI Protocol Interop).
- No remote theme fetching; theming reads only `<ALLBERT_HOME>`.
- No multi-user collaborative cursors or canvas snapshot/time-travel (still
  reserved in ADR 0023).

## Test Plan

- A fixture app and StockSage contribute catalog-valid panels into zones; bad
  panels are rejected.
- `/agent` renders the three-zone ChatGPT-style shell; existing offline/a11y/

- CoreApp domain cards render through the same zone path.
- Token files retint with no rebuild; snippet sanitizer strips/rejects unsafe

- Security evals: zone injection, panel catalog bypass, CSS exfiltration, CSP

- Full gate at closeout.

## Definition Of Done

v0.31 is done when apps contribute UI as catalog-validated panels into
host-owned workspace zones (with dedicated `/apps/<app_id>` routes reserved for
genuinely page-shaped views), StockSage's dashboard/list/queue/trends live in
the `/agent` workspace, the workspace renders the ChatGPT-style three-zone
shell, and operators can retheme and re-lay-out the UI from `<ALLBERT_HOME>`
through tokens, sanitized snippets, and validated layout config — all without
editing core code, granting new authority, or rendering arbitrary
model-generated UI.
