Inserted after v0.30. v0.31 is a behavior-preserving consolidation release
that makes the v0.32 workspace-only UI, v0.33 theming, v0.34 dynamic plugin
trial substrate, and v0.35 generator cheaper and safer to implement.

## Purpose

The v0.26-v0.30 arc proved the app/surface, security, memory, and canvas
contracts through StockSage. It also left several duplicated implementation
seams that should not become the foundation for the next UI and generation
arc:

- action modules repeat raw `use Jido.Action`, permission, and registry
  boilerplate;
- surface catalog, renderer dispatch, app catalog metadata, and StockSage card
  adapters encode component truth in multiple places;
- app and plugin registries both expose extension contribution data;
- Settings Central schema remains monolithic;
- redaction, path, audit, trace, and persistence helpers have overlapping
  local variants;
- response maps from actions, intent, objectives, and UI surfaces do not have a
  single typed contract.

v0.31 consolidates those seams without changing user-visible product routes,
domain behavior, persistence format, or permissions. The point is to make the
following releases consume one substrate instead of adding new work around old
duplication.

## Expected Inputs

- v0.24 objective runtime exists and is the only durable multi-step work
  substrate.
- v0.26 workspace/canvas and Surface substrate exists.
- v0.27-v0.30 StockSage app contracts are implemented and tested.
- v0.28 security evals cover app scope, Resource Access, fragments, objective
  boundaries, redaction tripwires, and namespace ownership.
- Current route shape remains released v0.30 behavior until v0.32 intentionally
  removes `/agent`, `/settings`, and `/stocksage/*`.

## Design Decisions

### 1. Behavior-preserving release

v0.31 must not change operator routes, domain workflow, permission floors,
database schemas, memory format, StockSage analysis behavior, or workspace tile
semantics. Any implementation that needs a user-visible behavior change should
move that change to v0.32+ or record a separate ADR amendment.

### 2. Public facades before deletion

Each consolidation starts by naming the public facade, internal helpers, and
compatibility shims. Deletion is allowed only after tests prove all callers use
the facade and released behavior still passes.

### 3. Actions use one Allbert-facing DSL

Runtime-facing actions should use a thin `AllbertAssist.Action` wrapper over
Jido action definitions. The wrapper owns common metadata, permission
declaration, capability registration, response helpers, and compile-time
validation. Private Jido command modules inside agents remain private and do
not become Allbert capability actions.

### 4. One extension contribution index

Plugins and apps remain distinct concepts, but contribution discovery should
flow through one normalized extension index. App surfaces, settings fragments,
actions, skill roots, child specs, and diagnostics should be discoverable from
one registry facade. Metadata still grants no authority.

### 5. One Surface catalog and renderer dispatch path

Component atoms, primitive status, catalog validation, app-declared catalog
metadata, and web renderer dispatch should share one source of truth. v0.31
should retire the permanent need for separate StockSage-only render dispatch
and v0.30 pass-through card adapters.

### 6. Settings schema fragments

Settings Central remains the authority for operator configuration, but schema
ownership moves from one monolithic module to registered fragments owned by
runtime contexts, apps, and plugins. Writes still go through Settings Central
actions and Security Central.

### 7. Dynamic code is still forbidden

v0.31 adds no dynamic code generation or loading. The current rule remains:
Allbert does not auto-generate, compile, or load Elixir modules from arbitrary
skill, plugin, YAML, or user-created folders. v0.34 will define a narrow
sandboxed exception in ADR 0032.

## Module Shapes To Pin

- `AllbertAssist.Boundary`: documentation and compile-time helpers for public
  facades versus internal modules.
- `AllbertAssist.Action`: wrapper macro and helpers for registered Allbert
  capability actions.
- `AllbertAssist.Actions.Capability`: normalized capability metadata emitted by
  actions and consumed by the registry.
- `AllbertAssist.Runtime.Response`: typed response helpers for completed,
  denied, confirmation-needed, advisory, and error outcomes.
- `AllbertAssist.Runtime.Paths`, `Runtime.Redactor`, `Runtime.Audit`, and
  `Runtime.Persistence`: shared substrates or facades over existing modules.
- `AllbertAssist.Extensions.Registry`: normalized contribution index over
  compiled plugins and apps.
- `AllbertAssist.Surface.Catalog`: single component, primitive, renderer, and
  app-surface validation source.
- `AllbertAssist.Settings.Fragment`: schema-fragment contract registered by
  core contexts, apps, and plugins.

Names may be adjusted during implementation only if the request-flow update
records the final public facade and all references are changed consistently.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless the implementation owner records a narrower gate and
why.

### M1: Boundary inventory and facade map

Goal: freeze the public/internal boundary before code movement.

Tasks:

- Inventory action, app/plugin registry, surface/catalog/renderer, settings,
  audit, trace, path, persistence, redaction, and intent response modules.
- Mark public facades, internal modules, compatibility shims, and deletion
  candidates in docs and module docs.
- Add compile-time or test helpers that make accidental use of private shims
  visible where feasible.

Tests:

- Boundary inventory tests or static assertions cover the named public
  facades.
- Existing focused tests for actions, registry, settings, workspace, and
  StockSage still pass without route changes.

Acceptance:

- Implementers know which module owns each substrate and which shims are
  temporary.

### M2: Dead-code and compatibility cleanup

Goal: remove unused compatibility layers that are provably no longer called.

Tasks:

- Delete modules, functions, fixtures, and docs that M1 marks unused.
- Keep any shim that still has external callers, but add a retirement note.
- Avoid opportunistic refactors outside the inventoried seams.

Tests:

- Focused compile and caller tests prove no deleted module is referenced.
- Route, action, settings, workspace, and StockSage behavior remains v0.30
  compatible.

Acceptance:

- Dead code is gone; surviving compatibility shims have owners and exit
  criteria.

### M3: Shared redaction and path substrate

Goal: one path vocabulary and one redaction surface for runtime-facing output.

Tasks:

- Consolidate Allbert Home path helpers behind one facade.
- Consolidate redaction helpers used by actions, traces, audits, LiveViews,
  StockSage, Resource Access, and future dynamic-trial reports.
- Preserve existing path locations and redaction behavior.

Tests:

- Temporary `ALLBERT_HOME` tests cover path roots.
- Redaction regression tests cover secrets, Resource Access traces, StockSage
  payloads, and audit/LiveView output.

Acceptance:

- New code has one obvious path helper and one obvious redaction helper.

### M4: Shared audit, persistence, and trace substrate

Goal: reduce overlapping IO/audit helpers while keeping durable formats stable.

Tasks:

- Add or clarify facades for audit events, hybrid SQLite/YAML persistence, and
  trace writes.
- Route existing callers through the facade without changing files on disk.
- Keep security-relevant audit vocabulary stable.

Tests:

- Persistence round-trip tests prove existing data can still be read.
- Audit and trace tests prove event shape and redaction are unchanged.

Acceptance:

- v0.34 sandbox-trial work can audit and persist through one documented path.

### M5: `AllbertAssist.Action` and capability registry

Goal: action modules declare Allbert capability metadata once.

Tasks:

- Introduce `use AllbertAssist.Action` as a thin wrapper over Jido actions.
- Move common permission, app-scope, capability, response, and registration
  metadata into the wrapper or companion helpers.
- Migrate representative core and StockSage actions, then the remaining
  runtime-facing actions.
- Keep private Jido command modules out of the Allbert action registry.

Tests:

- Registry tests prove migrated actions retain ids, permissions, app scope,
  safety floors, and confirmation behavior.
- Security evals covering app-scope, Resource Access, and confirmation still
  pass.

Acceptance:

- New generated and handwritten actions have one approved shape.

### M6: Typed runtime response contract

Goal: action/intent/objective responses have one inspectable shape.

Tasks:

- Define completed, denied, confirmation-needed, advisory, and error response
  helpers.
- Normalize action runner, intent, objective, and UI consumption paths where
  they currently inspect ad hoc maps.
- Preserve external CLI/LiveView copy unless a request-flow note records a
  deliberate text cleanup.

Tests:

- Response helpers cover all statuses.
- Existing CLI/LiveView/action tests assert equivalent behavior.

Acceptance:

- Downstream UI panels and generators can rely on one response contract.

### M7: Unified surface catalog, renderer, and extension registry

Goal: one source of truth for component registration and app/plugin
contributions.

Risk note: unlike M1-M6 and M8, M7 is **not invisible** — it retires live
render-dispatch paths (the StockSage-only renderer and the v0.30 pass-through
card adapters). Treat M7 as the one consolidation milestone that carries
rendering-regression risk: gate it on render/DOM equivalence, and if it
destabilizes during implementation, split it into its own release (v0.31b) ahead
of the v0.32 workspace rebuild rather than forcing it under the
behavior-preserving tag.

Tasks:

- Move component atoms, primitive markers, catalog metadata, and renderer
  dispatch into one catalog facade.
- Register StockSage cards through the shared catalog path.
- Retire separate StockSage-only surface renderer dispatch and permanent
  workspace pass-through card adapters when the shared catalog can render the
  same components directly.
- Merge app/plugin contribution discovery behind one extension registry facade
  while preserving app and plugin concepts.

Tests:

- Catalog tests prove known components, primitive components, renderer modules,
  and app-declared catalog entries agree.
- Workspace and StockSage render tests prove real cards still render in
  `/agent` and `/stocksage/*` during v0.31.
- Registry tests prove apps, plugins, settings fragments, skill roots, actions,
  and diagnostics are discoverable through the facade.

Acceptance:

- v0.32 can add panels/zones by extending one catalog and registry path.
- `/agent` and `/stocksage/*` render the same cards before and after M7
  (render/DOM equivalence proven), with no `v0.26 stub` text and no visual
  regression.

### M8: Settings fragments and single authority

Goal: Settings Central becomes fragment-owned without losing policy control.

Tasks:

- Add a settings-fragment contract for core contexts, apps, and plugins.
- Split the monolithic schema into registered fragments without changing key
  names, defaults, secret handling, or audit behavior.
- Ensure Settings Central actions remain the only write path.
- Retire or amend compatibility permission helpers only after Security Central
  parity and eval coverage are explicit.

Tests:

- Settings schema tests prove every existing key remains present with the same
  default, type, validation, and secret behavior.
- Write/action/audit tests prove settings updates still pass through Security
  Central.

Acceptance:

- v0.32 can render workspace Settings Central from fragments instead of a
  monolith.

### M9: Closeout and downstream doc reconciliation

Goal: v0.31 is release-ready and the v0.32-v0.35 plans consume the new
substrate.

Tasks:

- Update README, roadmap, agent context map, ADR pointers, request-flow logs,
  and future-features pointers.
- Reconcile v0.32-v0.35 plan prerequisites with the final facade names.
- Run full warning gate and `mix precommit`.

Acceptance:

- Docs, tests, and code agree that v0.31 changed substrates only, not product
  behavior.

## Non-Goals

- No `/workspace` route reset; that is v0.32.
- No `/agent`, `/settings`, or `/stocksage/*` removal; that is v0.32.
- No user theming or layout overrides; that is v0.33.
- No dynamic plugin/app generation or sandboxed module loading; that is v0.34.
- No developer-facing generator; that is v0.35.
- No migration, data-format rewrite, permission relaxation, or route
  compatibility change.

## Definition Of Done

v0.31 is done when the runtime-facing action, settings, catalog/renderer,
registry, path/redaction/audit/persistence, and response seams have one
documented substrate; released v0.30 behavior still passes; and downstream
plans target the consolidated shapes instead of the pre-v0.31 duplication.
