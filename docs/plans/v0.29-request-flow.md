## Pre-Flight

Use a disposable Allbert Home with StockSage fixture analyses and outcomes:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v029-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix ecto.migrate
```

The StockSage namespace must already be declared from v0.27, and v0.28
namespace/security evals must be passing.

## Flow 1: Resolve Outcomes

Operator or scheduled job runs the outcome resolver after holding periods have
elapsed.

Expected behavior:

1. Pending `stocksage_outcomes` rows resolve idempotently.
2. Resolved outcomes retain analysis id, ticker, objective id, user/thread
   context, price basis, return, and resolution timestamp.
3. Running the resolver again does not duplicate results or overwrite observed
   facts incorrectly.
4. The analysis detail page shows resolved outcome status.

Failure behavior:

- Missing price data leaves the outcome pending or failed with a bounded reason.
- Resolver failure does not promote memory or create reflections implicitly.

Implementation notes:

- `resolve_outcomes` is an internal registered StockSage action. It requires
  explicit `active_app: :stocksage` runner context and `:stocksage_write`
  permission.
- M1 is deterministic and fixture-friendly: no market-data adapter is called by
  the resolver. Operator/scheduled paths supply observed prices or rely on
  already-recorded `end_price` values.
- The stable test handles for this flow are
  `plugins/stocksage/test/stocksage/outcomes_test.exs` and the
  `resolve_outcomes` runner coverage in
  `plugins/stocksage/test/stocksage/actions/actions_test.exs`.

## Flow 2: View Trends And Calibration

Operator opens StockSage trends.

Expected behavior:

1. Trend cards show alpha-aware accuracy, rating calibration, and leaderboard
   summaries from resolved outcomes.
2. Empty states explain that trends need resolved outcomes.
3. Filters or time windows preserve user/thread/app context.
4. Calculations match fixture expectations.

Implementation notes:

- `StockSage.Analyses.summarize_trends/2` now returns `accuracy`,
  `rating_calibration`, and `leaderboard` in addition to existing counts and raw
  outcomes.
- `accuracy.return_basis` is `:realized_return_no_benchmark`; v0.29 does not
  invent benchmark-adjusted alpha data.
- The trends surface exposes stable DOM anchors:
  `#stocksage-trends-summary`, `#stocksage-rating-calibration`,
  `#stocksage-leaderboard`, and `#stocksage-trend-outcomes`.

## Flow 3: Generate Reflection

Operator requests a reflection for a resolved outcome.

Expected behavior:

1. The reflection is generated through a registered action.
2. The action carries objective/analysis context and redacts sensitive data.
3. The reflection distinguishes observed outcome from advisory interpretation.
4. The result is visible in StockSage surfaces and trace metadata.

Implementation notes:

- `generate_reflection` writes one idempotent StockSage-local
  `stocksage_memory_entries` row per outcome using `legacy_source:
  "stocksage_reflection"` and `legacy_id: outcome.id`.
- The analysis detail surface exposes
  `#stocksage-outcome-reflection-actions`,
  `#stocksage-generate-reflection-<outcome_id>`, and
  `#stocksage-reflections`.
- M3 uses deterministic bounded reflections. Provider-backed prose can be
  added later behind the same registered action after a provider/confirmation
  story is explicitly planned.

Failure behavior:

- Provider errors become bounded failure states.
- Reflection text is not written into durable Allbert memory unless the
  operator explicitly invokes memory sync.

## Flow 4: Explicitly Sync A Lesson

Operator reviews a resolved reflection and chooses to sync the lesson.

Expected behavior:

1. The analysis detail page renders `#stocksage-sync-lesson-<entry_id>` for
   each generated StockSage-local reflection.
2. Clicking sync calls the registered `sync_app_lesson` action with
   `active_app: :stocksage`, user/session/objective context, source analysis,
   outcome id, ticker, rating, realized return, holding period, and reflection
   lesson text.
3. The action centrally stamps advisory context and Security Central returns
   `:needs_confirmation`; no Allbert markdown memory is written on the first
   click.
4. The operator sees `#stocksage-sync-notice` with the confirmation id and can
   review the pending confirmation in the existing confirmation surfaces.
5. The local reflection records the pending confirmation id and app-memory
   idempotency key, rendering as `Allbert sync pending` with the sync button
   disabled.
6. Approval resumes `sync_app_lesson` and writes one namespaced markdown memory
   entry through `Memory.upsert_app_entry/1`.
7. The next analysis-detail refresh reconciles the local reflection against the
   idempotency key and marks it as synced if the Allbert memory entry exists.
8. Re-syncing the same `{app_id, analysis_id, holding_period_days}` updates the
   existing namespaced lesson entry instead of appending a duplicate.
9. The trace and markdown metadata record app id, namespace, kind,
   idempotency key, source analysis, objective id, and a redacted lesson body.
10. Memory review semantics still distinguish conversation history from durable
   markdown memory.

Implementation notes:

- The app-memory markdown metadata fields are `app_id`, `namespace`, `kind`,
  `idempotency_key`, and `source_ref`.
- `StockSage.App.memory_namespace/0` is `writable: true` in v0.29, but writes
  remain confirmation-gated by `sync_app_lesson`.
- Stable DOM anchors for this flow are `#stocksage-sync-lesson-<entry_id>`,
  `#stocksage-sync-notice`, and `#stocksage-sync-error`.

Non-automatic invariant:

- Completing an analysis, resolving an outcome, or generating a reflection does
  not automatically promote anything into markdown memory.

## Flow 5: Rerun Analysis

Operator reruns an analysis from the StockSage UI.

Expected behavior:

1. The analysis detail page renders `#stocksage-analysis-rerun` when an
   analysis is loaded.
2. The operator chooses an explicit engine:
   `#stocksage-rerun-native`, `#stocksage-rerun-python`, or
   `#stocksage-rerun-both`.
3. The LiveView invokes the existing `run_analysis` registered action with the
   original ticker, original analysis date when present (otherwise today's
   date), thread/session context, explicit engine, and `source_analysis_id`.
4. The first click queues a normal `:stocksage_analyze` confirmation and does
   not create a new analysis row before approval.
5. The page renders `#stocksage-rerun-notice` with the confirmation id and
   includes the pending confirmation in `#stocksage-confirmation-links`.
6. Approval resumes `run_analysis`, which creates the new analysis/objective as
   the existing action already does.
7. Old and new runs remain distinguishable because confirmation params, action
   metadata, signals, and new analysis metadata carry `source_analysis_id`.
8. Python and parity reruns remain explicit comparison/reference paths; the
   existing bridge and Python-comparison settings still gate them.

Failure behavior:

- Missing ticker, invalid engine, disabled native engine, disabled Python
  comparison, disabled bridge, or Security Central denial render
  `#stocksage-rerun-error`.
