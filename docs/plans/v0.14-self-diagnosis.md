# v0.14 — Self-diagnosis and Unix co-tenant posture

Status: Future

## Context

Two related ambitions from the origin note are not covered by v0.11 (TUI + adaptive memory), v0.12 (self-improvement), or v0.13 (personalization):

1. **Allbert learns from its own traces.** The origin note asked for tracing that Allbert itself could use to diagnose what went wrong, not just a human-facing log. Today Allbert can emit traces (ADR 0025 hooks + telemetry) but it cannot read and reason about those traces to propose fixes.
2. **Composable small utility programs as co-tenants.** The origin note valued Unix-style composition: Allbert should run alongside small, focused utilities on the system, shelling out to them rather than absorbing their functionality into the kernel. The current tool surface supports shell exec under policy, but there is no curated posture for "let Allbert learn about and call the local utility programs that already exist on this machine."

v0.14 ships both as one coherent release because they share infrastructure: trace introspection, exec-policy-aware tool discovery, candidate-fix routing through existing approval surfaces, and a review-first posture on anything that might change durable state. This release is Future, pinned after v0.13, because:

- It needs v0.11's telemetry + memory routing to expose trace metadata cleanly.
- It needs v0.12's patch-approval and skill-author patterns so that candidate self-fixes route through familiar review UX.
- It needs v0.13's personalization to have landed without pulling in training complexity; otherwise the combined scope becomes unreviewable.

This file is a stub. Concrete milestones are written when v0.14 is the current release.

## Goal

Ship three capabilities framed as "Allbert diagnoses itself and plays well with the rest of the system, under explicit review":

1. A `self-diagnose` skill that reads recent traces, correlates failures with agent/turn/tool state, and drafts candidate explanations plus optional remediation patches. Remediation routes through v0.12's `patch-approval` inbox; memory fixes route through v0.11's staging pipeline.
2. A `local-utilities` surface where Allbert can enumerate, describe, and safely invoke small system utilities (jq, ripgrep, fd, bat, pandoc, etc.) through the existing exec policy. Discovery is explicit, not implicit; activation remains allow-listed.
3. A `unix-pipe` tool posture that composes local utilities through bounded shell pipelines while keeping the policy envelope intact. No new scripting runtime.

## Non-goals

- No autonomous application of self-diagnosis fixes. Every candidate fix routes through the same review surfaces as any other patch or memory change.
- No expansion of the exec-policy model. Utility discovery and invocation reuse `security.exec_allow` / `security.exec_deny` unchanged.
- No kernel-level daemon for trace streaming. Self-diagnosis reads persisted trace artifacts; live trace tailing can land later if needed.
- No new scripting language. Unix-pipe composition is shell exec with bounded timeouts and output caps, not a scripting runtime. The v0.12 `ScriptingEngine` seam remains the path for embedded scripting.
- No automatic installation of system utilities. Allbert reports missing tools; the operator installs them.

## Invariants v0.14 must preserve

- **Security envelope.** Trace introspection never escapes sandboxed read access. Utility invocation stays under exec policy. Candidate fixes route through `patch-approval` or staging, never through a direct write.
- **Markdown as ground truth.** Self-diagnosis outputs are markdown artifacts the operator can read and edit before any change lands.
- **Progressive disclosure.** Enumeration of local utilities surfaces names + short descriptions cheaply; full man-page or help-text reading happens on demand.
- **Review-first.** Every state-changing proposal from self-diagnosis — memory staging, patch approval, adapter rebuild trigger — goes through the same review path the originating subsystem already owns.

## Open design questions

- Trace storage scope. What slice of the hook-emitted trace stream is durable vs. rolling? How long does the self-diagnose skill's lookback window reach?
- Correlation model. Does self-diagnosis index traces into the same Tantivy store as memory, or stay a separate derived view?
- Utility discovery surface. Does Allbert maintain a curated `~/.allbert/utilities/` manifest, or detect on demand by probing the operator's `PATH`?
- Pipe bounds. What are the default timeouts, output caps, and stderr/stdout merge policies for `unix-pipe` tool invocations?
- Failure taxonomy. How does self-diagnosis categorise failures it cannot explain from local data alone (e.g., provider-side errors)?

## ADRs

To be written when v0.14 is the current release. Expected shapes:

- Self-diagnosis candidate fixes route through existing inbox kinds (`patch-approval` for code, staging for memory).
- Local utility invocation stays under the existing exec policy with no privileged bypass.
- Unix-pipe composition is a bounded tool shape, not a new runtime.
- Trace introspection uses persisted artifacts from the existing hook surface (ADR 0025) rather than requiring a new live-trace channel.

## Exit criteria (placeholder)

- `self-diagnose` can read a fixture trace, produce a markdown explanation, and optionally route a candidate fix through `patch-approval` or staging without changing state directly.
- Local-utility enumeration lists available tools from an allowlist plus operator-approved discoveries.
- `unix-pipe` tool invocation runs composed pipelines under exec policy with bounded timeouts and output caps.
- Operator docs cover trace inspection, utility allowlisting, and review flow for self-diagnosis candidate fixes.
- Tier A validation is green.
