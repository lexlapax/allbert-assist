# v0.14 — Self-diagnosis and Unix co-tenant posture

Status: Future

## Context

Two related ambitions from the origin note are not covered by v0.11 (TUI + adaptive memory), v0.12 (self-improvement), or v0.13 (personalization):

1. **Allbert learns from its own traces.** The origin note asked for tracing that Allbert itself could use to diagnose what went wrong, not just a human-facing log. Today Allbert can emit traces (ADR 0025 hooks + telemetry) but it cannot read and reason about those traces to propose fixes.
2. **Composable small utility programs as co-tenants.** The origin note valued Unix-style composition: Allbert should run alongside small, focused utilities on the system, shelling out to them rather than absorbing their functionality into the kernel. The current tool surface supports shell exec under policy, but there is no curated posture for "let Allbert learn about and call the local utility programs that already exist on this machine."

v0.14 ships both as one coherent release because they share infrastructure: trace introspection, exec-policy-aware tool discovery, candidate-fix routing through existing approval surfaces, and a review-first posture on anything that might change durable state.

### Hard dependencies

v0.14's technical dependencies are on v0.11 and v0.12 specifically. v0.13 is a sequencing neighbor, not a runtime dependency:

- **v0.11 — staging pipeline, telemetry, memory routing.** Memory-shaped remediation (e.g., "this session reveals a fact worth capturing") flows through the existing staging pipeline. Trace correlation consumes `TelemetrySnapshot` (last resolved intent, active agent stack, cost, context pressure) as ground truth. Always-eligible routing tells the self-diagnose skill when to surface itself.
- **v0.12 — sibling-worktree, Tier A validation, `patch-approval`, `skill-author`.** Code-shaped remediation produces a diff in a sibling worktree, runs Tier A, and routes through `patch-approval`; it does not introduce a second isolation mechanism. Skill-shaped remediation drafts a diagnostic skill through v0.12's `skill-author` invoked as a sub-agent, and the draft carries `provenance: self-diagnosed` through the same install quarantine.
- **v0.13 — sequencing only.** v0.14 does not read adapters, trigger training, or modify `~/.allbert/adapters/`. The v0.13→v0.14 ordering is a scope/complexity choice (avoid overlapping two hard reviews); if personalization were deprioritized, v0.14 could ship directly after v0.12.

This file is a stub. Concrete milestones are written when v0.14 is the current release.

## Goal

Ship three capabilities framed as "Allbert diagnoses itself and plays well with the rest of the system, under explicit review":

1. A `self-diagnose` skill that reads recent traces, correlates failures with agent/turn/tool state, and drafts candidate explanations plus optional remediation. Remediation shape is dispatched by failure kind:
   - **Code-shaped** — produce a diff in a v0.12 sibling worktree, run Tier A, route through `patch-approval`.
   - **Skill-shaped** — invoke v0.12's `skill-author` as a sub-agent; the resulting draft carries `provenance: self-diagnosed` and flows through the standard install quarantine + preview.
   - **Memory-shaped** — route through v0.11's staging pipeline (never direct-write durable memory).
2. A `local-utilities` surface where Allbert can enumerate, describe, and safely invoke small system utilities (jq, ripgrep, fd, bat, pandoc, etc.) through the existing exec policy. Discovery is explicit, not implicit; activation remains allow-listed.
3. A `unix-pipe` tool posture that composes local utilities through bounded shell pipelines while keeping the policy envelope intact. This is a new tool shape, not a new runtime — distinct from v0.12's `ScriptingEngine` (Lua is in-process, sandboxed, JSON-IO; `unix-pipe` is shell exec under policy). Operator docs must explain when to reach for each.

## Non-goals

- No autonomous application of self-diagnosis fixes. Every candidate fix routes through the same review surfaces as any other patch or memory change.
- No expansion of the exec-policy model. Utility discovery and invocation reuse `security.exec_allow` / `security.exec_deny` unchanged.
- No kernel-level daemon for trace streaming. Self-diagnosis reads persisted trace artifacts; live trace tailing can land later if needed.
- No new scripting language. Unix-pipe composition is shell exec with bounded timeouts and output caps, not a scripting runtime. The v0.12 `ScriptingEngine` (Lua) seam remains the path for embedded in-process scripting. Lua and `unix-pipe` are complementary: Lua for safe in-process transforms over JSON inputs; `unix-pipe` for invoking utilities that already exist on the operator's system.
- No second isolation mechanism for code-shaped remediation. v0.12's sibling-worktree is the single worktree path; v0.14 reuses it.
- No new inbox kinds for code or skill remediation. v0.12's `patch-approval` and the existing install-quarantine preview cover both; v0.14 adds only kind-specific fields if needed, never a new kind.
- No automatic installation of system utilities. Allbert reports missing tools; the operator installs them.

## Invariants v0.14 must preserve

- **Security envelope.** Trace introspection never escapes sandboxed read access. Utility invocation stays under exec policy. Candidate fixes route through `patch-approval` or staging, never through a direct write.
- **Markdown as ground truth.** Self-diagnosis outputs are markdown artifacts the operator can read and edit before any change lands.
- **Progressive disclosure.** Enumeration of local utilities surfaces names + short descriptions cheaply; full man-page or help-text reading happens on demand.
- **Review-first.** Every state-changing proposal from self-diagnosis — memory staging, patch approval, diagnostic-skill install — goes through the same review path the originating subsystem already owns, following the cross-release self-change envelope in [ADR 0080](../adr/0080-self-change-artifacts-share-approval-provenance-and-rollback-envelope.md).
- **Provenance pattern.** Diagnostic skills drafted by `self-diagnose` via v0.12's `skill-author` carry `provenance: self-diagnosed`, extending the additive enum v0.12 (`self-authored`) and v0.13 (`self-trained`) already populated.

## Open design questions

- **Trace persistence is a hard prereq, not just a storage-scope question.** ADR 0025 defines the hook surface; what it does not define is durable retention of emitted traces for later consumption by a skill. v0.14 needs: a storage location (`~/.allbert/traces/` is the natural candidate), a retention policy (size cap, age cap, or both), a schema version so traces can be re-read across upgrades, and a read API surfaced through the existing tool surface (ADR 0052). If no prior release lands trace persistence, v0.14's M0-M1 must own it.
- Trace storage scope. What slice of the hook-emitted trace stream is durable vs. rolling? How long does the self-diagnose skill's lookback window reach?
- Correlation model. Does self-diagnosis index traces into the same Tantivy store as memory, or stay a separate derived view? If Tantivy, this is a new `trace` tier — analogous to v0.11's `episode`/`fact` additions and subject to the same schema-version bump discipline.
- Utility discovery surface. Does Allbert maintain a curated `~/.allbert/utilities/` manifest, or detect on demand by probing the operator's `PATH`?
- Pipe bounds. What are the default timeouts, output caps, and stderr/stdout merge policies for `unix-pipe` tool invocations?
- Failure taxonomy. How does self-diagnosis categorise failures it cannot explain from local data alone (e.g., provider-side errors)?

## ADRs

To be written when v0.14 is the current release. Expected shapes:

- Self-diagnosis candidate fixes route through existing inbox kinds (`patch-approval` for code, staging for memory).
- Local utility invocation stays under the existing exec policy with no privileged bypass.
- Unix-pipe composition is a bounded tool shape, not a new runtime.
- Trace introspection uses persisted artifacts from the existing hook surface (ADR 0025) rather than requiring a new live-trace channel.

## Exit criteria (placeholder)

- `self-diagnose` can read a fixture trace, produce a markdown explanation, and optionally route a candidate fix through `patch-approval` or staging without changing state directly.
- Local-utility enumeration lists available tools from an allowlist plus operator-approved discoveries.
- `unix-pipe` tool invocation runs composed pipelines under exec policy with bounded timeouts and output caps.
- Operator docs cover trace inspection, utility allowlisting, and review flow for self-diagnosis candidate fixes.
- Tier A validation is green.
