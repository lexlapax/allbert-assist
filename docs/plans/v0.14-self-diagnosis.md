# v0.14 — Self-diagnosis and Unix co-tenant posture

Status: Shipped

Reality note: remediation routing shipped, but concrete candidate generation is partial as of v0.14; tracked by v0.14.1. This plan remains the v0.14 design record, while v0.14.1 owns replacing report-oriented scaffolds with substantive candidate artifacts.

This document is now the implementation record for the v0.14 release. Milestone language is retained below for traceability.

## Summary

v0.14 is the release where Allbert learns from its own trace history and becomes a better Unix co-tenant without bloating the kernel. It ships three complementary surfaces:

1. `self-diagnose`, a first-party skill and CLI/TUI surface that reads bounded v0.12.2 session traces, correlates failures, and writes operator-readable markdown reports.
2. `local-utilities`, a curated manifest of small system utilities the operator has explicitly enabled for Allbert.
3. `unix_pipe`, a structured direct-spawn pipeline tool for composing those enabled utilities under the existing exec policy.

The release is review-first. Diagnosis reports explain by default. Candidate remediations are opt-in per run and route through existing v0.11/v0.12 review surfaces: staged memory, `patch-approval`, or skill install quarantine. v0.14 does not create a new trace persistence format, live trace channel, shell runtime, inbox kind, or self-change bypass.

## Lineage

The origin note asked for two things that are now ready to converge: tracing Allbert can learn from, and Unix-style composition with small focused utility programs already on the system. The release lineage is:

- **Origin note**: tracing should help Allbert diagnose what went wrong; Rust stays compact; local utilities should be co-tenants rather than absorbed into the kernel.
- **v0.11**: daemon-owned telemetry, memory staging, adaptive memory routing, and the first review-first `LearningJob` seam.
- **v0.12**: sibling-worktree self-modification, `patch-approval`, `skill-author`, and the sandboxed Lua scripting seam.
- **v0.12.1**: daemon-owned live `ActivitySnapshot`, responsive operator surfaces, command/settings descriptors, and path-preserving settings writes.
- **v0.12.2**: durable session trace artifacts, redaction policy, bounded replay/read surfaces, and protocol v4 trace messages.
- **v0.13**: a second self-change artifact kind, `adapter-approval`, which makes review-before-activation familiar before v0.14 proposes diagnostic fixes.

v0.14 consumes those surfaces. It does not revisit their core decisions.

## Goals

- Add bounded trace-diagnosis reads over v0.12.2 session artifacts.
- Ship a first-party `self-diagnose` skill that produces markdown reports under the active session.
- Add an explicit `allbert-cli diagnose` command surface for provider-free fixture validation and operator use.
- Allow optional candidate remediation only when the operator asks for it, routing through existing review surfaces.
- Add a curated local utility catalog and host-specific enablement manifest.
- Add a bounded `unix_pipe` tool that composes enabled utilities without shell parsing.
- Add protocol v6 self-diagnosis and utility status messages with backward-compatible per-peer filtering.

## Non-goals

- No autonomous application of diagnostic fixes.
- No new inbox kind. Code fixes reuse `patch-approval`; memory fixes reuse staging; skill fixes reuse install quarantine.
- No new trace persistence location, trace schema, trace retention policy, protocol v4 change, or trace index.
- No Tantivy trace tier in v0.14. Trace correlation uses bounded scans and summarization over `TraceReader`.
- No automatic installation of system utilities.
- No shell-string execution, `sh -c`, glob expansion, redirection, process substitution, or free-form environment override in `unix_pipe`.
- No new scripting language. Lua remains the embedded JSON-in/JSON-out scripting seam; `unix_pipe` is out-of-process utility composition.
- No default profile export/sync of host-specific utility manifests.
- README, changelog, release notes, and shipped-behavior operator docs are updated only during M7 release closeout.

## Plan hardening decisions

### Bounded trace diagnosis

Self-diagnosis reads `TraceDiagnosticBundle` values built from v0.12.2 `TraceReader`. The bundle is a summarized, redacted, bounded diagnostic input, not a new durable trace format.

Defaults:

```toml
[self_diagnosis]
enabled = true
lookback_days = 7
max_sessions = 5
max_spans = 1000
max_events = 2000
max_text_snippet_bytes = 32768
max_report_bytes = 262144
allow_remediation = false
```

The read algorithm lists trace sessions through `TraceReader::list_sessions()`, filters to the active session plus most recent sessions within `lookback_days`, applies `max_sessions`, reads spans through `TraceReader::read_session()`, sorts by timestamp, then summarizes to the span/event caps. If caps are hit, the bundle records truncation metadata. Unsupported future trace schemas surface an actionable remediation hint and do not silently skip data.

The bundle includes session ids, span ids, parent ids, names, status, duration, bounded attributes/events, trace read warnings, and a failure classification. Text-bearing fields are already redacted by v0.12.2 before persistence; v0.14 still re-runs the `SecretRedactor` before report or prompt inclusion as a defensive pass.

Implementation DTOs are owned by `allbert-kernel::self_diagnosis`; proto owns only the daemon wire equivalents. The internal bundle shape is:

```text
TraceDiagnosticBundle {
  bundle_version: 1,
  active_session_id,
  selected_session_ids,
  generated_at,
  bounds,
  spans,
  events,
  warnings,
  truncation,
  classification,
}
```

`bounds` records the effective settings used for the read. `truncation` records whether session, span, event, text, or report caps were hit. `classification` records one primary failure kind, optional secondary failure kinds, confidence `0.0..=1.0`, and a bounded rationale string.

Failure taxonomy is fixed for v0.14:

- `provider_error`
- `tool_denied`
- `tool_failed`
- `timeout`
- `approval_abandoned`
- `cost_cap`
- `context_pressure`
- `memory_mismatch`
- `adapter_training_failure`
- `unknown_local`

`unknown_local` is a valid result, not a failure of the diagnostic tool.

### Reports and remediation

Diagnosis reports are markdown session artifacts:

```text
~/.allbert/sessions/<session_id>/artifacts/diagnostics/<diagnosis_id>/report.md
~/.allbert/sessions/<session_id>/artifacts/diagnostics/<diagnosis_id>/bundle.summary.json
```

`report.md` is the operator-facing artifact and contains summary, evidence, confidence, skipped/truncated data, and recommended next actions. `bundle.summary.json` is a bounded machine-readable summary for tests and CLI rendering; it is not a replay trace and is not a new source of truth.

Diagnosis ids use `diag_<utc_timestamp>_<shortid>`, where the timestamp is UTC `YYYYMMDDTHHMMSSZ` and the short id is at least 8 lowercase base32 or hex characters generated from local randomness. The id is stable once the artifact directory is created.

`bundle.summary.json` contains:

```text
DiagnosisReportSummary {
  schema_version: 1,
  diagnosis_id,
  session_id,
  created_at,
  selected_session_ids,
  classification,
  confidence,
  report_path,
  bounds,
  truncation,
  warnings,
  remediation,
}
```

`report.md` uses these sections in order: `Summary`, `Classification`, `Evidence`, `Skipped Or Truncated Data`, `Recommended Next Actions`, and `Remediation Status`. Sections with no data remain present with a short explanation so CLI/TUI renderers have stable anchors.

By default, self-diagnosis only explains. Remediation requires `self_diagnosis.allow_remediation = true` and explicit operator intent through `allbert-cli diagnose run --remediate <code|skill|memory> --reason <text>` or an equivalent REPL/TUI slash command. A vague natural-language request inside an ordinary chat turn may suggest the exact command to run, but it must not start remediation. `--reason` is mandatory, must be non-empty after trimming, and is written into the generated approval, staged memory candidate, or skill draft metadata.

Remediation authorization is daemon-owned and command-originated. The daemon builds `DiagnosisRemediationRequest` only from parsed CLI/REPL/TUI command input, never from a model-emitted tool call. The skill-callable `self_diagnose` tool is report-only in v0.14 and rejects any remediation field.

Telegram is structural-only for v0.14 remediation. It may show that a report recommends code, skill, or memory remediation, but it never starts remediation and never accepts `--remediate` equivalents.

Remediation routing:

- **Code-shaped**: create a v0.12 sibling worktree, produce a diff, run Tier A validation, and emit existing `patch-approval`. Accept still does not install.
- **Skill-shaped**: invoke the v0.12 `skill-author` flow into `skills/incoming/<name>/` with `provenance: self-diagnosed`; standard install preview/confirm remains mandatory.
- **Memory-shaped**: write staged memory candidates only. Promotion remains the v0.11 staging review path.

No remediation path writes directly to durable memory, installed skills, active source checkout, adapters, or bootstrap files.

### Local utilities

The built-in utility catalog is compiled into `allbert-kernel` and starts with conservative Unix tools that are useful for inspection/transforms: `jq`, `rg`, `fd`, `bat`, `pandoc`, `sed`, `awk`, `sort`, `uniq`, `wc`, `head`, and `tail`. The catalog stores id, display name, description, executable candidates, safe usage notes, and catalog-defined bounded version/help probes.

The runtime enablement manifest is host-specific:

```text
~/.allbert/utilities/enabled.toml
```

Manifest schema:

```toml
schema_version = 1

[[utility]]
id = "rg"
path = "/opt/homebrew/bin/rg"
path_canonical = "/opt/homebrew/bin/rg"
version = "ripgrep 14.1.1"
help_summary = "rg [OPTIONS] PATTERN [PATH ...]"
enabled_at = "2026-04-26T00:00:00Z"
verified_at = "2026-04-26T00:00:00Z"
status = "ok"
size_bytes = 123456
modified_at = "2026-04-01T00:00:00Z"
pipe_allowed = true
```

`status` is one of `ok`, `missing`, `changed`, `denied`, or `needs-review`. Enabling canonicalizes symlinks, records the canonical path, executable size, and modified timestamp, and stores bounded version/help text if the catalog defines a probe. Verification marks an entry `missing` when the path is gone, `changed` when canonical path, size, or modified timestamp differs, `denied` when central exec policy hard-denies the executable, and `needs-review` when metadata cannot be safely verified. Only `ok` entries with `pipe_allowed = true` may be used by `unix_pipe`.

`utilities enable` never silently edits `security.exec_allow`. If the executable is currently allowed, the command records that fact. If it is not hard-denied but will require confirmation at run time, the command succeeds and prints an actionable note. If it is hard-denied, enablement refuses until the operator changes exec policy through the existing settings surface.

The manifest is excluded from profile export/sync by default because it points at host-local binaries. `profile export --dry-run` must name `utilities/enabled.toml` in the excluded host-specific set once v0.14 ships.

### `unix_pipe`

`unix_pipe` is a structured tool, not a shell runtime. Input shape is a JSON object with ordered stages:

```json
{
  "stages": [
    { "utility_id": "rg", "args": ["TODO"] },
    { "utility_id": "head", "args": ["-n", "20"] }
  ],
  "stdin": null,
  "cwd": "/workspace",
  "timeout_s": 30
}
```

Each stage references an enabled utility id, not a program path. The kernel resolves the id through the utility manifest, constructs each process with direct spawn, wires stdout to the next stage stdin, and captures final stdout plus per-stage stderr summaries.

Defaults and ceilings:

- `local_utilities.unix_pipe_max_stages = 5`
- `local_utilities.unix_pipe_timeout_s = 30`
- `local_utilities.unix_pipe_max_stdin_bytes = 1048576`
- `local_utilities.unix_pipe_max_stdout_bytes = 1048576`
- `local_utilities.unix_pipe_max_stderr_bytes = 262144`
- `local_utilities.unix_pipe_max_args_per_stage = 64`
- `local_utilities.unix_pipe_max_arg_bytes = 4096`
- `local_utilities.unix_pipe_max_argv_bytes = 32768`

There is no shell parsing, redirection, glob expansion, env override, or arbitrary pipeline string. `cwd` must stay inside configured filesystem roots. Every stage runs through the existing `security.exec_allow`/`security.exec_deny` policy. Utility enablement is a second gate; it never bypasses central exec policy.

`cwd` is optional per stage. When omitted, the daemon uses the active session workspace root if one is configured and otherwise `ALLBERT_HOME`; the resolved cwd must stay inside configured filesystem roots. Each stage may pass at most `unix_pipe_max_args_per_stage` args; each arg must be valid UTF-8 and at most `unix_pipe_max_arg_bytes` bytes; the full argv vector for a stage must be at most `unix_pipe_max_argv_bytes` bytes after UTF-8 encoding.

All stages are preflighted before any process starts. Preflight validates stage count, argument count/length, enabled utility status, `cwd`, byte caps, timeout, active-skill `allowed-tools`, and central exec policy. After preflight succeeds, stages launch as one pipeline. If any stage exits non-zero, times out, or hits a cap, the pipeline result is unsuccessful; already-started stages are allowed to drain unless the failure is timeout/cap related, in which case the kernel kills every running child.

v0.14 supports text I/O only. `stdin` is UTF-8 text, final stdout is rendered as UTF-8 text, and invalid UTF-8 from child processes is lossy-rendered with an explicit `lossy_utf8 = true` flag. Binary-safe streaming is deferred.

The tool returns a structured JSON string:

```json
{
  "ok": false,
  "timed_out": false,
  "cap_violated": false,
  "stdout": "bounded final stdout",
  "stdout_bytes": 1024,
  "stdout_truncated": false,
  "lossy_utf8": false,
  "stages": [
    {
      "utility_id": "rg",
      "exit_code": 1,
      "stdout_bytes": 0,
      "stderr_bytes": 0,
      "stderr_summary": "",
      "stderr_truncated": false
    }
  ],
}
```

Hook metadata records utility ids, resolved programs, stage count, caps, exit codes, byte counts, and truncation flags. Trace spans record `tool.name = "unix_pipe"` plus per-stage events.

### Protocol v6

v0.14 bumps the daemon protocol from `5` to `6` additively.

A v6 daemon accepts client protocol versions `2`, `3`, `4`, `5`, and `6`, records the negotiated protocol per connection, and filters outbound messages and fields per peer:

- v2 peers receive only v2-safe status/session messages.
- v3 peers additionally receive live activity messages.
- v4 peers additionally receive trace messages.
- v5 peers additionally receive adapter messages.
- v6 peers additionally receive self-diagnosis and local-utility messages.

Unsupported versions below `2` or above `6` receive `ProtocolError { code: "version_mismatch", ... }` with remediation that names whether to upgrade the client or daemon. A v6 client connected to a v5 daemon must surface `upgrade the daemon to v0.14 to use self-diagnosis and local utility surfaces` and must not synthesize diagnosis or utility state locally.

Protocol v6 adds daemon-owned payloads for diagnosis status/report summaries, utility catalog/enabled status, and `unix_pipe` run summaries. Frontends render these messages; they do not infer diagnosis state from traces or utility state from local PATH.

Client messages:

- `DiagnoseRun(DiagnosisRunRequest)`
- `DiagnoseList { session_id: Option<String> }`
- `DiagnoseShow { diagnosis_id: String }`
- `UtilitiesDiscover`
- `UtilitiesList`
- `UtilitiesShow { utility_id: String }`
- `UtilitiesEnable(UtilityEnableRequest)`
- `UtilitiesDisable { utility_id: String }`
- `UtilitiesDoctor`

Server messages:

- `DiagnosisStarted { diagnosis_id: String }`
- `Diagnosis(DiagnosisSummary)`
- `Diagnoses(Vec<DiagnosisSummary>)`
- `DiagnosisReport { diagnosis_id: String, report_markdown: String, summary: DiagnosisReportSummary }`
- `UtilityCatalog(Vec<UtilityCatalogEntryPayload>)`
- `Utility(UtilityCatalogEntryPayload)`
- `EnabledUtilities(Vec<EnabledUtilityPayload>)`
- `UtilitiesDoctor(Vec<EnabledUtilityPayload>)`
- `UnixPipeRun(UnixPipeRunSummary)`

Payload structs:

```text
DiagnosisRunRequest {
  session_id: Option<String>,
  lookback_days: Option<u16>,
  remediation: Option<DiagnosisRemediationRequest>,
  json: bool,
}

DiagnosisRemediationRequest {
  kind: DiagnosisRemediationKind, // code | skill | memory
  reason: String,
}

DiagnosisSummary {
  diagnosis_id,
  session_id,
  created_at,
  classification,
  confidence,
  report_path,
  truncated,
  remediation,
}

UtilityCatalogEntryPayload {
  id,
  name,
  description,
  executable_candidates,
  pipe_allowed,
  installed,
  enabled,
  status,
}

UtilityEnableRequest {
  utility_id,
  path: Option<String>,
}

EnabledUtilityPayload {
  id,
  path,
  path_canonical,
  version,
  help_summary,
  enabled_at,
  verified_at,
  status,
  pipe_allowed,
}

UnixPipeRunSummary {
  ok,
  timed_out,
  cap_violated,
  stdout,
  stdout_bytes,
  stdout_truncated,
  lossy_utf8,
  stages,
}
```

`ActivityPhase::Diagnosing` is added for diagnosis runs. `unix_pipe` uses the existing tool activity posture with `tool_name = "unix_pipe"`.

```toml
[self_diagnosis]
enabled = true

unix_pipe_max_stdout_bytes = 1048576
unix_pipe_max_stderr_bytes = 262144
unix_pipe_max_args_per_stage = 64
unix_pipe_max_arg_bytes = 4096
unix_pipe_max_argv_bytes = 32768
```

Validation:

- `lookback_days` must be `1..=90`.
- `max_sessions` must be `1..=50`.
- `max_spans` must be `1..=10000`.
- `max_events` must be `0..=50000`.
- `max_text_snippet_bytes` must be `1024..=262144`.
- `max_report_bytes` must be `8192..=1048576`.
- `unix_pipe_max_stages` must be `1..=10`.
- `unix_pipe_timeout_s` must be `1..=300`.
- `unix_pipe_max_args_per_stage` must be `0..=256`.
- `unix_pipe_max_arg_bytes` must be `64..=65536`.
- `unix_pipe_max_argv_bytes` must be `1024..=262144` and at least `unix_pipe_max_arg_bytes`.
- byte caps must be `1024..=16777216`.

When `self_diagnosis.enabled = false`, daemon and CLI `diagnose run` and the `self_diagnose` tool fail closed with a remediation hint naming `/settings show self_diagnosis` and the settings key to enable. `diagnose list/show --offline` may still read existing report artifacts because those are passive session artifacts.

When `local_utilities.enabled = false`, `utilities enable`, `utilities disable`, `utilities doctor`, and `unix_pipe` fail closed with a remediation hint naming `/settings show local_utilities`. `utilities discover/list/show --offline` may still inspect the built-in catalog and existing manifest, but must label results as disabled/offline and must not verify or mutate host state.

The settings catalog gains `SelfDiagnosis` and `LocalUtilities` groups plus descriptors for all keys above. Settings writes use the v0.12.1 typed allowlist and path-preserving TOML edit discipline.

### CLI

Add:

```text
allbert-cli diagnose run [--session <id>] [--lookback-days <n>] [--remediate code|skill|memory --reason <text>] [--json]
allbert-cli diagnose show <diagnosis-id> [--offline] [--json]
allbert-cli diagnose list [--session <id>] [--offline] [--json]

allbert-cli utilities discover [--offline] [--json]
allbert-cli utilities list [--offline] [--json]
allbert-cli utilities show <utility-id> [--offline] [--json]
allbert-cli utilities enable <utility-id> [--path <path>]
allbert-cli utilities disable <utility-id>
allbert-cli utilities doctor [--json]
```

All v0.14 CLI commands attach to the daemon by default. If the daemon is not running, they follow the existing `daemon.auto_spawn` setting: auto-spawn when enabled, otherwise fail with the standard `allbert-cli daemon start` remediation hint. Mutating or live commands must use daemon protocol v6: `diagnose run`, any remediation, `utilities enable`, `utilities disable`, and `utilities doctor`. Read-only inspection commands support `--offline` to bypass the daemon and read local artifacts/catalog data directly; offline mode must label results as offline and omit live daemon-only status.

`diagnose run` writes the markdown report and prints its path plus top findings. `--session <id>` limits the trace read to that session. Without `--session`, the daemon uses the active session plus recent sessions within the configured bounds. `--remediate` refuses unless `self_diagnosis.allow_remediation = true` and `--reason <text>` is present.

`utilities discover` is read-only and probes candidate executables. `utilities enable` sends `UtilityEnableRequest { utility_id, path }` and writes `utilities/enabled.toml` only after the resolved path passes catalog validation and exec policy does not hard-deny the executable. If `path` is omitted, the daemon uses the first catalog candidate in catalog order that exists, canonicalizes, is executable, and is not hard-denied. If `path` is present, it must be absolute, must canonicalize successfully, and must match the requested catalog utility id by catalog-defined validation; otherwise enablement refuses.

### REPL/TUI and Telegram

Add local slash commands:

```text
/diagnose [run|list|show <id>]
/diagnose run --remediate <code|skill|memory> --reason <text>
/utilities [list|show <id>|doctor]
```

TUI renders diagnosis progress through `ActivityPhase::Diagnosing` and uses shared report/list renderers. Telegram is structural-only:

- `/diagnose last` shows latest diagnosis id, confidence, top classification, and report path.
- `/utilities status` shows enabled count and utilities needing review.

Telegram does not inline full trace excerpts, full report bodies, stdout/stderr payloads, or utility help text.

### First-party skill and tool

Add a built-in kernel tool named `self_diagnose`. It is the only trace-read tool exposed to skills in v0.14. Input:

```json
{
  "session_id": null,
  "lookback_days": null
}
```

The schema is closed: any `remediation` field or unknown field is rejected. Output is the same bounded `DiagnosisReportSummary` plus report path used by CLI/protocol rendering. The tool never returns raw trace files or unbounded span/event payloads.

The first-party `self-diagnose` skill includes `self_diagnose` in `allowed-tools`. It explains the report, asks the operator to run explicit commands when remediation is needed, and does not parse trace files or call lower-level trace APIs directly.

### Skill provenance

Skill frontmatter provenance gains:

```yaml
provenance: self-diagnosed
```

This is additive to `external | local-path | git | self-authored`. Existing skills still default to `external`. A self-diagnosed skill follows the same quarantine and install preview path as every other generated skill.

## Implementation design

### Kernel seams

- Add `crates/allbert-kernel/src/self_diagnosis.rs` for `SelfDiagnosisConfig` helpers, `TraceDiagnosticBundle`, failure classification, markdown report rendering, and report artifact writes.
- Add `crates/allbert-kernel/src/local_utilities.rs` for the built-in catalog, enablement manifest, verification, and utility resolution.
- Extend the kernel tool registry with `self_diagnose` so the first-party skill uses the same diagnosis service as CLI/daemon surfaces.
- Extend `crates/allbert-kernel/src/tools/mod.rs` with `UnixPipeTool`; keep direct `process_exec` unchanged.
- Extend `crates/allbert-kernel/src/security/mod.rs` so `unix_pipe` invokes per-stage exec policy checks and active-skill allowed-tools checks can explicitly allow `unix_pipe`.
- Extend `crates/allbert-kernel/src/config.rs` and `settings.rs` with `[self_diagnosis]` and `[local_utilities]`.
- Extend `SkillProvenance` with `SelfDiagnosed`.
- Reuse `TraceReader`, `SecretRedactor`, staging memory APIs, self-improvement worktree APIs, and skill creation/quarantine APIs rather than introducing parallel modules.

### Daemon and protocol seams

- Bump `allbert-proto::PROTOCOL_VERSION` to `6`.
- Add v6 diagnosis/utility request/response payloads.
- Add `ActivityPhase::Diagnosing`.
- Extend daemon per-peer filtering for v6 messages.
- Make daemon protocol v6 the default path for v0.14 CLI/TUI/Telegram surfaces. Only explicit `--offline` read-only CLI commands are allowed to use local filesystem/catalog readers.

### CLI seams

- Add `crates/allbert-cli/src/diagnose_cli.rs`.
- Add `crates/allbert-cli/src/utilities_cli.rs`.
- Add top-level `diagnose` and `utilities` subcommands and REPL/TUI slash command dispatch.
- Reuse existing output conventions: human-readable by default, `--json` stable payloads for automation.

### Filesystem layout

```text
~/.allbert/utilities/enabled.toml
~/.allbert/sessions/<session_id>/artifacts/diagnostics/<diagnosis_id>/report.md
~/.allbert/sessions/<session_id>/artifacts/diagnostics/<diagnosis_id>/bundle.summary.json
```

All writes are atomic. Diagnosis artifacts are session artifacts and travel with sessions. Utility enablement is host-specific and excluded by default.

### Setup and upgrade state

Bump the setup state version to `6` when implementing v0.14. Add a `local_utilities` setup state block that records whether the fresh-profile utility discovery prompt has been shown and which catalog ids the operator enabled during setup.

Fresh profiles are prompted after tracing/personalization setup to discover local utilities. The default answer enables none. Existing profiles are not forced through setup again; they load in-memory defaults for `[self_diagnosis]` and `[local_utilities]` until the operator changes settings or runs `utilities enable`.
