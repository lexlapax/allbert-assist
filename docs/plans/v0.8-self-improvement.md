# v0.8 — Self-improvement

Status: Proposed

## Context

By v0.7, Allbert has a hardened kernel (v0.6), a real agent harness, a rich skill format, curated memory, and at least one approval-capable non-REPL channel with a settled review pattern. v0.8 is about the assistant helping improve itself: two specialised coding skills (Rust for the runtime, a skill-authoring skill for Python/etc.) plus a principled scripting seam so future embedded runtimes (Lua, etc.) enter through one controlled door rather than by the kernel picking up a new heavyweight dependency.

This is both the most interesting and the highest-risk release on the roadmap. Self-authored code can extend Allbert; self-authored code can also break Allbert. The design has to make the "improve Allbert" path first-class while routing every artifact through existing trust gates.

## End-user framing

Consistent with ADR 0038, v0.8 does **not** require end users to write Rust, Python, or Lua.

- The Rust coding skill is Allbert coding on itself, under explicit operator diff review. The user reviews and approves diffs; the user does not author them.
- The skill-authoring skill scaffolds new skills through natural-language conversation. The user describes what they want; Allbert produces a folder-format skill that lands in the same install quarantine as any external install. This is the user-facing front door for skill creation — not a back-office developer tool.
- The embedded Lua scripting engine is an opt-in advanced surface for users who want a small, sandboxed DSL. It is not a requirement for any end-user workflow and stays disabled in `exec_policy` by default.

Self-improvement is a capability Allbert uses on the user's behalf, within the trust gates that govern everything else.

## Goal

Ship three capabilities, all framed as "Allbert does this for the user under review":

1. A **Rust coding skill** that can read, modify, build, and test the Allbert codebase inside a sibling worktree, producing diffs the operator reviews before merge.
2. A **skill-authoring skill** (user-facing) that scaffolds new AgentSkills-format skills through natural-language conversation and submits them through the same install preview + confirm flow every other skill uses (ADR 0033). This replaces the v0.4 `skills init` CLI wizard as the default skill-authoring front door; `skills init` remains as the CLI escape hatch for technical users.
3. A **scripting engine seam** (`ScriptingEngine` trait) behind which embedded runtimes (Lua, later others) can attach, with one concrete default implementation, all of it opt-in per `exec_policy`.

## Sketch

### Rust rebuild skill

- Operates in `~/.allbert/worktrees/<branch>/` created from the active Allbert source checkout.
- Uses the existing exec seam (ADR 0004) for `cargo build`, `cargo test`, and `git` operations.
- Produces patches as diffs reviewed in the REPL or via a channel (v0.7). No automatic push, no automatic merge.
- Distinct from the primary checkout so a broken build cannot take down the running daemon.
- Confirm-trust gates any destructive git action (force push, branch deletion) independently of the build itself.
- Operator review includes the v0.7 cost-cap posture: rebuilds that exceed the daily cap refuse the same way any other turn does.

### Skill-authoring skill (user-facing)

- Scaffolds a folder-format skill (ADR 0032) with frontmatter, `SKILL.md`, optional `scripts/` and `references/`.
- Conversational intake: the skill asks the user what the skill should do, what inputs it needs, what outputs it should produce, and what tools it should use. The user answers in plain English.
- Uses the in-memory tier 1/2 prompt builder (ADR 0036) to preview what the new skill would contribute before materialising it.
- Persists the draft to `~/.allbert/skills/incoming/` — the same quarantine directory as any other install (ADR 0033) — and goes through the standard preview + confirm flow.
- Python is the recommended scripting interpreter for new skills (already default-allowed per ADR 0034); Lua becomes available if the user opts into the scripting engine.
- Provenance: skills authored this way carry a `provenance: self-authored` frontmatter tag so operators can distinguish them from external installs during review.

### Scripting engine seam

- `ScriptingEngine` trait with methods for loading, invoking, and isolating scripts.
- v0.8 ships one default implementation: **Lua** (small, embeddable, well-understood).
- Opt-in via `config.exec_policy.allowed_interpreters` (adding `"lua"`) plus a new `config.scripting.engine = "lua"` setting.
- Lua scripts invoked through the exec seam just like any other script (ADR 0034); the engine seam is specifically for embedded, in-process execution paths, which is a distinct trust story.
- Sandboxed Lua runtime (sandboxed builtins, no raw filesystem access by default, no network access by default). Sandbox policy declared in config and audited by hooks.
- `BeforeTool`/`AfterTool` observation on embedded-script invocations relies on v0.7's normalized tool surface (ADR 0052).

### Self-improvement trust model

- Every self-generated artifact routes through an existing gate:
  - Rust patches → operator diff review.
  - New skills → install preview + confirm (ADR 0033).
  - Script snippets run by agents → exec policy + hook observability (ADR 0034).
  - Embedded Lua snippets → scripting engine sandbox + exec policy.
- No self-improvement path gets a privileged bypass. The kernel remains the policy surface.

## Expected ADRs

- First-class sibling worktree model for self-modification.
- `ScriptingEngine` trait + Lua as the v0.8 default embedded runtime.
- Embedded-script sandboxing policy (builtins allowlist, no filesystem/network by default).
- Self-authored skill persistence follows the same install quarantine path as external installs.
- Skill-authoring as a first-party user-facing flow (natural-language conversational intake rather than CLI wizard as default front door).

## Dependencies

- v0.4 AgentSkills format — the skill-authoring skill produces folder-format skills.
- v0.4 install preview + confirm — the path self-authored skills take.
- v0.4 script policy — interpreter allowlist governs what new skills can execute.
- v0.6 hardening — cost cap enforcement and session durability apply to rebuild turns just like any other work.
- v0.7 channel expansion — operators might approve rebuilds or skill installs from at least one non-REPL approval-capable channel instead of the REPL; tool-surface normalization (ADR 0052) makes embedded-script hook observation uniform.

## Out of scope for v0.8

- Autonomous self-modification (no human in the loop).
- Automatic publishing of self-authored skills outside the local machine.
- Scripting engines beyond the one shipped default (Lua) — other engines slot in behind the same trait without a new ADR pass.
- A kernel-wide plugin system. Skills + scripting engine seam is the stated extension surface; heavier plugin work is explicitly deferred.
- Local model retrain / micro-GPT distillation for memory or personality. Origin-note ambition; still philosophically in scope but requires curated memory to produce clean training data first. Revisit after v0.8 lands.

## Remaining planning questions

- What's the rebuild story on a machine without a working Rust toolchain — refuse, offer to install, or warn?
- How much of the Allbert test suite must pass for a rebuild to be marked "safe to merge" vs. "needs review"?
- Are self-authored skills marked differently in the installed-skills list beyond the `provenance: self-authored` frontmatter tag?
- Do embedded Lua scripts share memory with the host session, or are they fully isolated?
- Does the conversational skill-authoring flow live as a shipped first-party skill (analogous to `memory-curator` in v0.5) or as a kernel-native agent? Preference: shipped first-party skill for consistency with ADR 0048.
