## Purpose

Add reviewed discovery before self-improvement, without adding arbitrary
remote code authority. v0.45 gives operators a curated way to discover
skills, templates, and reviewed-source plugin metadata that later
suggestion flows can reference. The catalog is **single-vendor** at v1.0
(Allbert-author bundles only); operators get a small but fully-reviewed
catalog.

v0.45 is also the forcing function for ADR 0046's `schema_version`
convention: the new `marketplace.*` settings fragment declares
`schema_version: 1`, establishing the field convention that v0.58's
runtime migration tool consumes.

The first milestone preserves these seven decisions. They are no
longer open defaults; later milestones may revisit them only with an
explicit re-decision recorded in this table and in the request-flow
doc.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Hash algorithm | **SHA-256** via `:crypto.hash(:sha256, basis)`. Matches v0.42 R9 `Discovery.stable_hash` and v0.43 browser cache; established in-repo. | SHA-512 is more future-proof but no in-repo precedent; switching later requires a hash-format migration. |
| 2 | Permission strategy | **New `:marketplace_install` permission class** (default `:allowed`, floor `:allowed`). ## Research Grounding

The v0.45 design is grounded in:

- **npm registry** (`registry.npmjs.org`): single-source catalog with
  manifest files (`package.json`) + tarball bundles, SHA hashes (npm
  v6+ uses SHA-512 but earlier was SHA-1; v0.45 picks SHA-256 to
  match the in-repo v0.42 R9 precedent). Key lesson: keep the
  manifest small, point at bundle artifacts by hash.
- **VS Code Marketplace** (`marketplace.visualstudio.com`):
  single-vendor curated index with extension manifests. Lesson:
  publisher identity + version pinning + extension id are the
  durable identity tuple; install is per-version.
- **Anthropic Skills repo** (`github.com/anthropics/skills`):
  reviewed skill marketplace with directory-per-skill layout. Lesson:
  ship skill bundles as directory trees, not single archives;
  filesystem mirrors the install target.
- **PURL spec** (`pkg:` URI form): inspiration for the entry id
  pattern. v0.45 entry ids are scoped `<author>/<name>` (e.g.
  `allbert/research-helpers`); future community submissions can add
  a scope namespace without breaking the pattern.
- **In-repo precedents:**
  - **v0.03 `Skills.Registry`** owns the in-process skill catalog;
    marketplace installs register through this with `source_scope:
    :marketplace_install`, `status: :disabled`, `trust: :untrusted`.
    No parallel registry. The `:marketplace_install` value is a new
    `:source_scope` enum entry added in v0.45 (sibling of v0.10's
    `:imported_cache`); the v0.03 substrate gets a one-value
    additive schema change at M1.
  - **v0.10 `:online_skill_import`** establishes the disabled-on-
    install + cache-write contract for imported skills. Marketplace
    install reuses the disabled/untrusted invariant.
  - **v0.31 `Settings.Fragments`** is the substrate for plugin-owned
    or feature-owned settings namespaces. `marketplace.*` registers
    through this composition path.
  - **v0.42 R9 `Discovery.stable_hash`** SHA-256 pattern is the
    direct precedent for marketplace bundle hashes.
  - **v0.43 doctor live-verify pattern** (ADR 0047 shape) is the
    template for `marketplace_doctor`.
  - **v0.44 `workflows.schema_version: 1`** at `schema.ex:1077,
    :2245` is the precedent for the ADR 0046 draft `schema_version`
    field on every plugin/feature-owned settings fragment.
  - **v0.10 ResourceURI source_profile** at `resource_uri.ex:56-65`
    establishes `allbert://sources/<kind>/<id>` for source profiles;
    marketplace adopts a distinct `marketplace://entry/<author>/<name>` scheme
    because marketplace entries are first-class identity, not source
    profiles.

Implementation consequence: v0.45 adds one settings fragment, one
permission class, three operation classes, one URI scheme, one workspace
panel, one CLI module, seven registered actions (including doctor), one
`priv/marketplace/{index.json,bundles/}` directory tree, and the ADR
0043 + ADR 0046 draft. No new SQLite tables, no new runtime
processes, no external network, no JavaScript / Node dependencies.

## Catalog Schema (v1)

### `priv/marketplace/index.json` shape

```json
{
  "schema_version": 1,
  "catalog_version": "0.45.0",
  "source": "shipped",
  "generated_at": "2026-06-01T00:00:00Z",
  "source_git_commit": "abc123…",
  "entries": [
    {
      "id": "allbert/research-helpers",
      "version": "1.0.0",
      "kind": "skill",
      "name": "Research Helpers",
      "description": "Skill pack for research workflows.",
      "author": "Allbert Maintainers",
      "license": "MIT",
      "bundle_path": "bundles/allbert-research-helpers-1.0.0/",
      "bundle_hash": "sha256:abcdef…",
      "provenance": {
        "scheme": "shipped",
        "source_git_commit": "abc123…",
        "review_date": "2026-05-30"
      },
      "tags": ["research", "skills"]
    }
  ]
}
```

### Bundle manifest shape

Each bundle directory contains a `bundle.json`:

```json
{
  "schema_version": 1,
  "id": "allbert/research-helpers",
  "version": "1.0.0",
  "kind": "skill",
  "files": [
    {
      "path": "SKILL.md",
      "sha256": "abcdef…"
    },
    {
      "path": "scripts/helper.sh",
      "sha256": "012345…"
    }
  ],
  "bundle_hash": "sha256:abcdef…",
  "install_target": "<ALLBERT_HOME>/marketplace/skills/allbert-research-helpers/",
  "install_state": "disabled_untrusted"
}
```

### Identity rules

- **Entry id pattern:** `^[a-z0-9][a-z0-9_-]*/[a-z0-9][a-z0-9_-]*$`
  (`<author>/<name>`). v0.45 ships `allbert/*` only.
- **Version pattern:** semver-shaped (`^\d+\.\d+\.\d+$`). Pre-release
  / build metadata parked.
- **Kind enum:** `skill | template | plugin_index`. Closed set.
  `skill` and `template` are installable; `plugin_index` is
  browse/inspect metadata only.
- **Bundle hash:** `sha256:<64-hex>` format (M1 decision 1).
- **`bundle_path`:** relative to `priv/marketplace/`. No `..`
  traversal; reject paths outside the prefix.
- **`install_target`:** required for `skill` and `template` bundle
  manifests; omitted for `plugin_index`. When present, it is a
  template literal with `<ALLBERT_HOME>` token and resolves to the
  per-kind directory per M1 decision 4.

### Validation invariants

- Unknown top-level keys reject with JSON-Pointer-bearing
  diagnostic.
- Unknown per-entry keys reject.
- Missing required fields reject.
- Bundle hash MUST match the recursive SHA-256 of bundle contents
  (manifest excluded from the hash to allow regeneration).
- `bundle_path` MUST exist under `priv/marketplace/bundles/`.
- `bundle_path` MUST contain a `bundle.json` matching the bundle
  manifest shape.
- `install_target` MUST resolve to a path under `<ALLBERT_HOME>/
  marketplace/` for installable kinds.
- `plugin_index` entries MUST NOT declare `install_target`; install
  attempts reject with a structured diagnostic.
- Duplicate `id` across entries rejects.
- `install_state` MUST be `disabled_untrusted` at v0.45 (invariant).
- `schema_version` MUST be `1` at v0.45 (invariant; ADR 0046 draft).
- `provenance.scheme` MUST be `"shipped"` at v0.45; reserved values
  (`"signed"`, `"community_reviewed"`, `"vendor_published"`) reject
  at catalog parse time with a JSON-Pointer-bearing diagnostic
  (`error_category: :catalog_unknown_provenance_scheme`).
- Bundle manifest `schema_version` MUST be `1` at v0.45. The ADR
  0046 draft `schema_version` convention applies to bundle
  manifests as well as the catalog index; the parser rejects other
  values with `error_category: :bundle_manifest_invalid` and a
  JSON-Pointer-bearing diagnostic.

## Bundle Distribution And Mirror

- **Source of truth:** `priv/marketplace/{index.json, bundles/}` ships
  in the Allbert release. Operators do not edit `priv/`.
- **Mirror trigger:** lazily on first `mix allbert.marketplace *`
  invocation, or eagerly at `mix allbert.marketplace mirror`.
  Application boot does NOT mirror.
- **Mirror target:** `<ALLBERT_HOME>/marketplace/cache/index.json`
  (a read-only copy of the shipped index for operator inspection).
- **Operator-modification handling:** if the mirror is operator-
  modified, marketplace actions log a warning and re-read from
  `priv/marketplace/index.json` as the authority. Operator-mirror
  edits are advisory only.
- **Re-mirror on Allbert upgrade:** when Allbert is upgraded, the
  next marketplace action detects the source git commit drift (via
  `index.json:source_git_commit`) and re-mirrors silently. Installed
  bundles in `<ALLBERT_HOME>/marketplace/<kind>/<entry-id>/` are
  preserved; only the cached index is replaced.
- **No upstream remote fetch:** v0.45 never makes an HTTP request
  for catalog data. Updates to the catalog ship with new Allbert
  releases.
- **Mirror write is idempotent.** Concurrent marketplace actions
  that race the first lazy mirror write resolve to the same final
  state via the write-temp + rename pattern (POSIX-atomic
  same-filesystem rename). A per-Allbert-Home advisory lock
  serializes the write so partial reads cannot observe a half-
  written index; subsequent reads see the rename's atomic finish.

## Provenance / Hash / Version / Rollback Model

### Provenance

At v0.45 single-vendor scope, provenance = "shipped in Allbert
v0.45.x release N." The `index.json:source_git_commit` field carries
the git commit hash of the Allbert source tree the catalog was
built from. Per-entry `provenance.scheme: "shipped"` +
`provenance.source_git_commit` + `provenance.review_date` document
the seed bundle's provenance.

v0.45 is single-vendor; signing is not required. Future community
submissions (parked) will require a signing scheme; the schema field
`provenance.scheme` is forward-compatible.

### Hash

`sha256:<64-hex>` format. `bundle_hash` is the recursive SHA-256 of
bundle contents:

```
sha256(
  sort(files).map(f => f.path + "\0" + sha256(file_content)).join("\n")
)
```

Each file's SHA-256 also appears in `bundle.json:files[].sha256` for
spot-checks without recomputing the whole bundle hash.

Hash verification runs:
- when `marketplace_doctor` runs
- before `install_marketplace_bundle` writes
- in `verify_marketplace_bundle_hash` action (on demand)

Hash mismatch fails closed with a structured diagnostic; the install
does not proceed.

### Version

Semver-shaped `<major>.<minor>.<patch>`. Bundles ship as
`<bundle_path>/` immutable per-version directories. Updating a
bundle is a new version + new bundle directory + new index entry;
the old version remains addressable for rollback.

### Rollback

`rollback_marketplace_install <entry-id>` removes
`<ALLBERT_HOME>/marketplace/<kind>/<entry-id>/` and updates
`<ALLBERT_HOME>/marketplace/installed.json` to mark the entry as
rolled-back. No previous-version restore is needed because the
shipped catalog is always re-installable from `priv/marketplace/`.

If multiple versions of the same entry-id are installed (e.g., an
operator installed v1.0.0 and then upgraded to v1.1.0), rollback
removes only the most recently installed version; the prior version
directory is not auto-restored (operators re-install if needed).

## Settings (`marketplace.*`)

Add the `marketplace.*` namespace through
`AllbertAssist.Marketplace.SettingsFragment` and the existing
`Settings.Fragments` composition path (ADR 0031).

```yaml
marketplace:
  schema_version: 1                       # ADR 0046 draft; invariant at v0.45
  enabled: true                           # opt-out master switch
  catalog:
    source: "shipped"                     # invariant: "shipped" only at v0.45
    cache_path: "<ALLBERT_HOME>/marketplace/cache"
    mirror_on_first_action: true          # invariant: true at v0.45
  install:
    default_state: "disabled_untrusted"   # invariant
    target_dir_skills: "<ALLBERT_HOME>/marketplace/skills"
    target_dir_templates: "<ALLBERT_HOME>/marketplace/templates"
  provenance:
    hash_algorithm: "sha256"              # invariant; matches v0.42 R9
    require_hash_match: true              # invariant
  installed_state_path: "<ALLBERT_HOME>/marketplace/installed.json"
```

Safe-write keys: `marketplace.enabled`,
`marketplace.catalog.cache_path` (operator may relocate cache only;
source remains shipped), `marketplace.install.target_dir_skills`,
`marketplace.install.target_dir_templates`,
`marketplace.installed_state_path`.
Every custom safe-write path must resolve under Allbert Home; paths
outside Allbert Home fail closed before writes.

Core permission safe-write key: `permissions.marketplace_install`.
Accepted values are `allowed`, `needs_confirmation`, and `denied`.
The default is `allowed`; operators may tighten marketplace installs
to confirmation or denial without changing the safety floor.

Invariants enforced at the fragment schema layer:

- `marketplace.schema_version` is read-only at `1` (ADR 0046 draft
  contract; the runtime migration tool at v0.58 manages bumps).
- `marketplace.catalog.source` is read-only at `"shipped"` at v0.45.
- `marketplace.catalog.mirror_on_first_action` is read-only at
  `true` at v0.45.
- `marketplace.install.default_state` is read-only at
  `"disabled_untrusted"`.
- `marketplace.provenance.hash_algorithm` is read-only at `"sha256"`.
- `marketplace.provenance.require_hash_match` is read-only at
  `true`.

## ADR 0046 Draft Scope At v0.45

ADR 0046 (`Settings Central Schema Migration Policy`) is `Status:
Proposed` and explicitly accepted at v0.59 when the runtime migration
tool ships. v0.45 only adopts the **field convention**:

- Every new plugin-owned or feature-owned settings fragment SHOULD
  declare `<namespace>.schema_version: <integer>` at version 1.
- v0.45's `marketplace.schema_version: 1` is the first new fragment
  to follow this convention.
- v0.44 retroactively shipped `workflows.schema_version` (already in
  `schema.ex:1077, :2245`) at `1`; v0.45 marketplace fragment is the
  second.

v0.45 does NOT ship:
- A schema_version migration runner.
- Bump semantics for `schema_version`.
- Fail-closed handling for unknown schema versions at boot.

Those land at v0.58 per the accepted ADR. v0.45 only establishes the
field exists.

## Permission, Operation, And Identity Additions

### Permission classes (`AllbertAssist.Security.Policy`)

| Permission | Default | Floor | Notes |
|---|---|---|---|
| `:marketplace_install` | `:allowed` | `:allowed` | Writes a marketplace bundle in disabled/untrusted state to Allbert Home; grants no execution authority. Reading the disabled bundle requires the existing `:skill_write` / enablement action path. |

`@permission_settings`, `@default_decisions`, and
`@known_permissions` grow by `:marketplace_install`.
`permission_classes/0` returns the new entry in a stable order.
`safety_floor/1` enforces the `:allowed` floor. Because the floor is
`allowed`, settings may tighten `permissions.marketplace_install` to
`needs_confirmation` or `denied`; there is no cap to apply unless a
later plan raises the floor.

### Operation classes (`AllbertAssist.Resources.OperationClass`)

- `@origin_kinds`: add `:marketplace_entry`.
- `@operation_classes`: add `:marketplace_browse`,
  `:marketplace_install_bundle`, `:marketplace_rollback`.
- `@scope_kinds`: add `:marketplace_entry` (per-entry identity for
  trace/audit). No grants stored against marketplace entries — the
  install writes disabled/untrusted state, not a grant.
- `@default_access_modes`: `marketplace_browse => :read`,
  `marketplace_install_bundle => :write`, `marketplace_rollback =>
  :write`.

### URI scheme (`AllbertAssist.Resources.ResourceURI`)

- Add `marketplace` to the supported scheme set.
- `marketplace://entry/<author>/<name>` normalizes to a per-entry
  identity URI. The URI authority must be `entry`; the path must
  contain exactly two entry-id segments.
- Reject any path beyond `entry/<author>/<name>`; reject query
  strings; reject fragments.
- Add scope-URI derivation: `:marketplace_entry` scope normalizes to
  the per-entry URI.
- Display/redaction: entry-id rendered untruncated in confirmations
  when the operator has tightened policy to require one, and in
  `/settings`; provenance fields untruncated; bundle hashes shown as
  `sha256:abcdef…` prefix-only in trace.

### ADR 0013 amendment

Add `marketplace://entry/<author>/<name>` to the initial URI mappings:

> - marketplace entries: `marketplace://entry/<author>/<name>` (added at v0.45
>   for the reviewed marketplace catalog, see ADR 0043). Per-entry
>   identity for trace/audit; install authority comes from
>   `:marketplace_install` permission, not the URI itself.

## Action Schemas And Capability Metadata

All actions register through `AllbertAssist.Actions.Registry` and run

## Operator-Visible Win

Operators can discover Allbert-author reviewed skills and templates
without granting them execution authority. Installed skills and
metadata are visible, local, and disabled/untrusted by default. The
catalog is small but fully reviewed.

A first-time browse walks through:

1. Operator opens `/workspace` and pins the Marketplace Catalog
   panel; sees the shipped catalog entries grouped by kind (skill,
   template, plugin_index) with version, provenance, hash, and
   install state for installable entries.
2. Operator runs `mix allbert.marketplace show
   allbert/research-helpers` (or clicks the entry in the panel) and
   sees the full bundle manifest, file list, install target when
   applicable, and provenance.
3. Operator runs `mix allbert.marketplace install
   allbert/research-helpers`. The action verifies the bundle hash,
   writes the bundle to
   `<ALLBERT_HOME>/marketplace/skills/allbert-research-helpers/`
   in disabled/untrusted state, and updates `installed.json`. No
   skill becomes executable; no permission grant fires.
4. Operator runs `mix allbert.skills list` and sees the new skill
   with `source_scope: :marketplace_install, status: :disabled,
   trust: :untrusted`.
5. Enabling and trusting the skill is a separate operator action
   that goes through the existing v0.03 + v0.06 skill
   enablement/trust path; marketplace ships only the install side.
6. If the operator wants to remove the install, `mix
   allbert.marketplace rollback allbert/research-helpers` deletes
   the install directory and updates `installed.json`. The shipped
   bundle remains in `priv/marketplace/` for future re-install.

## v0.44 Workflow Forward-Pin

Marketplace metadata at v0.45 MAY include workflow-id descriptors
as part of the bundle metadata schema (e.g., a reviewed seed bundle
could advertise that it ships an example workflow), but the v0.45
catalog **never installs workflow YAML files into the live
`<ALLBERT_HOME>/workflows/` directory**. Workflow files distributed
through the marketplace add an additional trust tier (workflows
execute through registered actions, so a marketplace-shipped workflow
can still drive any allowed-floor action without operator
confirmation), and that tier is parked under "Remote Workflow
Distribution / Marketplace Workflows" in
`docs/plans/future-features.md`.

v0.45 marketplace catalog metadata format MAY reference a
`workflow://<id>` URI to advertise that an installed seed bundle has
an example workflow available; resolving that URI loads from
`<ALLBERT_HOME>/workflows/` only if the operator has authored the
workflow themselves. The marketplace never writes to that directory.

## v0.47 Self-Improvement Forward-Pin

v0.47 (Operator-Supervised Self-Improvement) consumes v0.45
marketplace metadata as **one trace pattern source**, not as authority
or execution input. v0.45 commits to:

- A stable catalog snapshot shape per `marketplace.schema_version: 1`
  (the v0.45 wire format is preserved unless ADR 0046 ships a
  migration at v0.58).
- `list_marketplace_entries` and `list_installed_marketplace_bundles`
  return JSON-serializable shapes safe for v0.47 trace mining.
- Provenance / hash / version metadata is operator-visible and stays
  in the redacted envelope shape (no raw bundle content in traces).
- No new authority transitions: v0.47 may suggest "install bundle X"
  as a draft suggestion, but the install itself still goes through
  `install_marketplace_bundle` + the standard policy path. If the
  operator has tightened `permissions.marketplace_install` to
  `needs_confirmation`, the durable confirmation path applies.

v0.47 does NOT inherit:
- The ability to auto-install bundles (no auto-install at v0.45 or
  v0.47).
- Permission grant authority from marketplace metadata.
- Workflow YAML install from marketplace (parked per the v0.44
  forward-pin above).

## Why Core-Owned (and not plugin-shaped)

Unlike v0.43 browser and v0.44 Plan/Build, v0.45 marketplace is
**core-owned, not plugin-owned**. Reasons:

- The marketplace catalog is shipped data, not extension code; the
  installer logic + descriptor validation is core substrate.
- Marketplace installs ARE skills / templates / plugin metadata —
  the consumed contract is already core (v0.03 Skills.Registry,
  v0.38 template registry, v0.17 plugin contract).
- The `marketplace.*` settings fragment composes through the v0.31
  substrate but its owner module lives in core
  (`AllbertAssist.Marketplace.SettingsFragment`).

This matches the v0.42 `find_tools` substrate pattern (core-owned
discovery substrate; per-source providers can be added later) rather
than the v0.42 notes_files reference plugin pattern (plugin-owned).

## Security Posture

- **Marketplace metadata is descriptive only.** Catalog entries grant
  no permission, no skill enablement, no resource grants, no
  confirmation bypass.
- **Install writes disabled/untrusted state only.** Installed bundles
  appear in `Skills.Registry` (for skill kind) or the template
  registry (for template kind) with `status: :disabled, trust:
  :untrusted`. Enablement and trust are separate operator actions
  that go through existing v0.03 + v0.06 paths.
- **Hash mismatch fails closed.** `install_marketplace_bundle`
  rejects with a structured diagnostic if the recursive SHA-256
  doesn't match the manifest's `bundle_hash`. No partial install.
- **`bundle_path` is sandboxed.** `..` traversal rejected; paths
  outside `priv/marketplace/bundles/` rejected.
- **`install_target` is sandboxed.** For installable kinds, resolves
  to a path under `<ALLBERT_HOME>/marketplace/`; any other prefix
  rejected. `plugin_index` entries must not declare an install target.
- **Unknown schema_version rejects.** Catalog index with
  `schema_version != 1` rejects at v0.45 with a JSON-Pointer
  diagnostic (preview of ADR 0046's v0.59 runtime migration
  contract).
- **No remote network.** Catalog source is `"shipped"`; mirror is
  file-IO from `priv/marketplace/` to
  `<ALLBERT_HOME>/marketplace/cache/`. No HTTP, no DNS, no remote
  fetch.
- **Operator-modified mirror is advisory.** Modifications to
  `<ALLBERT_HOME>/marketplace/cache/index.json` are logged as
  warnings; the shipped `priv/marketplace/index.json` remains the
  authority.
- **Workflow files are never written.** Marketplace bundle kinds
  are `skill | template | plugin_index` only; no `workflow` kind at
  v0.45 (parked per v0.44 forward-pin).
- **Code-bearing remote plugin install is denied.** Reviewed-source
  plugin index entries are descriptive metadata only; v0.45 never
  fetches or compiles plugin code from the marketplace.
- **No theme/snippet distribution at v0.45** (parked).

## Non-Goals

- No community-submission process in v1.0 (parked at "Marketplace
  Community Submission / Review Governance" in future-features.md).
- No arbitrary remote code-bearing plugin install.
- No remote dependency resolution.
- No marketplace-provided permission grants.
- No marketplace theme/snippet distribution.
- No public API / ACP / MCP-server / AG-UI / A2UI protocol exposure
  for the marketplace (catalog reads are local actions only).
- No remote upstream fetch for catalog updates (catalog updates
  ship with Allbert releases).
- No marketplace-installed workflow YAML files in
  `<ALLBERT_HOME>/workflows/` (parked as "Remote Workflow
  Distribution / Marketplace Workflows").
- No bundle signing at v0.45 (single-vendor scope; `provenance.scheme`
  field is forward-compatible for future signing).
- No automatic skill/template enablement on install (disabled/
  untrusted invariant).
- No upgrade-in-place semantics — versions ship as immutable
  per-version directories; "upgrade" is install-new-version +
  rollback-old-version.
- No multi-author scope namespace at v0.45 (entries are
  `allbert/*` only; future community submissions will add scope
  per the entry id pattern).
