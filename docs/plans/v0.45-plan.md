# Allbert v0.45 Marketplace Lite (Data Shape + Allbert-Author Seeds)

Request flow: `docs/plans/v0.45-request-flow.md`
ADRs:
- `docs/adr/0043-marketplace-lite-trust-tier.md`
- `docs/adr/0046-settings-schema-migration-policy.md` (drafted here; accepted

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`, post-v0.37 planning pass.

This milestone tightened in the post-v0.37 planning pass: v0.45 ships the
**data shape and Allbert-author seed bundles only**. Community submission /
review governance is parked in `docs/plans/future-features.md` and may be
promoted post-1.0 when the project decides on governance. v0.45 also begins
drafting ADR 0046 (Settings schema migration) because marketplace adds new
settings fragments.

## Purpose

Add reviewed discovery before self-improvement, without adding arbitrary remote
code authority. v0.45 gives operators a curated way to discover skills,
templates, and reviewed-source plugin metadata that later suggestion flows can
reference. The catalog is **single-vendor** at v1.0 (Allbert-author bundles
only); operators get a small but fully-reviewed catalog.

## Expected Inputs

- v0.38 templated creation.
- v0.40 MCP client lessons.
- Existing plugin, skill, Settings Central, Security Central, and Allbert Home
  contracts.
- Proposed ADR 0043 accepted before implementation closeout.

## Operator-Visible Win

Operators can discover Allbert-author reviewed skills and templates without
granting them execution authority. Installed skills and metadata are visible,
local, and disabled/untrusted by default. The catalog is small but fully
reviewed.

## Planned Interfaces

- Marketplace-lite catalog **schema** (data shape) under a versioned index
  format.
- Reviewed skill install into Allbert Home as disabled/untrusted by default.
- Reviewed-source plugin index metadata without automatic code install.
- Template catalog metadata for `workspace:create`.
- Provenance, hash, version, and rollback metadata.
- **Allbert-author seed bundles** loaded into the catalog at v0.45 (the
  initial small reviewed set of skills/templates the Allbert maintainers
  publish). Operators see these immediately; no community submissions in
  v1.0.
- Settings Central fragment `marketplace.*` declaring `schema_version: 1`
  per the ADR 0046 draft.

## Catalog Source Of Truth

v0.45 ships the data shape **and** the v1.0 catalog content via shipped
Allbert-author seed bundles. The catalog is single-vendor by design for v1.0:

- the maintainer-curated seed set lives in the Allbert source tree under
  `priv/marketplace/index.json` and `priv/marketplace/bundles/`;
- the index is mirrored into Allbert Home on first run for operator
  inspection;
- there is no upstream remote fetch in v0.45; updating the catalog is a
  matter of upgrading Allbert.

A separate post-1.0 governance decision is required to open the catalog to
community submissions. See `future-features.md` "Marketplace Community
Submission / Review Governance".

## Security Posture

Marketplace-lite metadata never grants permission. Code-bearing remote plugin
install, remote dependency resolution, remote theme/snippet distribution, and
MCP Apps iframe execution are excluded from v0.45. Any later enablement still
routes through registered actions, Security Central, confirmations, traces, and
audits.

## v0.44 Workflow Forward-Pin

Marketplace metadata at v0.45 MAY include workflow-id descriptors as
part of the bundle metadata schema (e.g., a reviewed seed bundle could
advertise that it ships an example workflow), but the v0.45 catalog
**never installs workflow YAML files into the live
`<ALLBERT_HOME>/workflows/` directory**. Workflow files distributed
through the marketplace add an additional trust tier (workflows
execute through registered actions, so a marketplace-shipped workflow
can still drive any allowed-floor action without operator
confirmation), and that tier is parked under "Remote Workflow
Distribution / Marketplace Workflows" in `docs/plans/future-features.md`.

v0.45 marketplace catalog metadata format MAY reference a
`workflow://<id>` URI to advertise that an installed seed bundle has
an example workflow available; resolving that URI loads from
`<ALLBERT_HOME>/workflows/` only if the operator has authored the
workflow themselves. The marketplace never writes to that directory.

## Non-Goals

- No community-submission process in v1.0 (parked).
- No arbitrary remote code-bearing plugin install.
- No remote dependency resolution.
- No marketplace-provided permission grants.
- No marketplace theme/snippet distribution.
- No public API/ACP/MCP-server/AG-UI/A2UI protocol exposure.
- No remote upstream fetch for catalog updates (catalog updates ship with
  Allbert releases).
- No marketplace-installed workflow YAML files in `<ALLBERT_HOME>/workflows/`
  (per "v0.44 Workflow Forward-Pin"; parked as "Remote Workflow
  Distribution / Marketplace Workflows").

## Definition Of Done

v0.45 is done when:

- Allbert-author reviewed skills/templates can be discovered and installed
  from the catalog without granting authority;
- reviewed-source plugin metadata is inspectable;
- security evals prove marketplace metadata cannot enable execution or trust;
- ADR 0046 draft exists for v0.51 to accept;
- the marketplace settings fragment declares `schema_version: 1`.
