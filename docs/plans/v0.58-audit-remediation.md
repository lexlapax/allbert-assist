# Allbert v0.58 Post-Implementation Audit — Remediation Plan (Pass 1)

## Purpose

This document records the findings of the **first post-implementation audit** of
v0.58 (milestones M0–M13 committed) and specifies the remediation work required
**before M14 manual operator validation**. It is the audit-1 output in the
agreed sequence: *full audit → manual operator validation → final audit*, each
round producing remediation steps.

The audit combined six parallel static auditors (one per pillar/milestone group)
with a live UI pass driven through the Chrome browser-control extension against a
running dev server (`localhost:4000`). It is grounded in the committed code, not
the plan prose: every finding carries `file:line` evidence.

**Headline verdict.** The architecture spine is genuinely implemented and the core
M0–M13 claims hold — single `Surface.Renderer`, shared `ContextBuilder`/
`Actions.Helper`, Settings Central enforcement with a CI guard, behavior-preserving
redundancy consolidation, the `release.v058` gate, clean warning-as-error compile +
docs gate, and real behavioral test coverage for every milestone. **No
security/authority BLOCKER.** What the audit found is **"uniformity" leaking at the
edges** — exactly what a consolidation release's first audit should catch.

**Tally:** 1 BLOCKER, 8 MAJOR, ~10 MINOR/accepted-with-note.

## Scope & status

- **This pass produces the plan only.** No code has been changed. Implementation of
  R1–R3 follows on approval.
- **Settled approach decisions** (operator, 2026-06-25):
  - The eval behavioral gap (R3-A) and surface-policy coverage gap (R3-B) are to be
    **fixed in code**, not resolved by tightening prose.
  - All BLOCKER + MAJOR items are v0.58 remediation (before M14). MINOR/accepted
    items are deferred to the v0.59 cleanup unless promoted here.

## Remediation Rounds

Ordered by risk. R1 (contract/security) lands first; R2 (design-system) and R3
(eval/policy code) can proceed in parallel once R1 is underway. Each round ends with
a re-run of `mix allbert.test release.v058` before re-entering manual validation.

### R1: Contract & Security Hardening

#### R1-A — Redact provider/model profile DTOs at the source (load-bearing)
- **Finding (MAJOR):** `list_provider_profiles`/`list_model_profiles` return raw
  `base_url` + `api_key_ref` (and `provider_base_url` + `provider_api_key_ref`) into
  LiveView socket assigns. No secret is painted in the DOM today (the template
  renders only `endpoint_kind`/`credential_status`, and `ExposureFilter` blocks
  these from public surfaces — both verified), but the ADR-0070/0073 contract is
  that the **DTO itself** is redacted, not that today's template happens to omit the
  fields. The raw secret-ref + endpoint sit in server-side assigns where any future
  field render, `inspect`, or debug dump leaks them.
- **Evidence:** `apps/allbert_assist/lib/allbert_assist/settings.ex:136-137,153-154`;
  returned unmodified by
  `apps/allbert_assist/lib/allbert_assist/actions/settings/list_provider_profiles.ex:40`
  and `.../list_model_profiles.ex:40`. Also consumed by
  `settings_central.ex:892-893`.
- **Fix:** project the returned `providers:`/`models:` lists down to non-secret
  fields (drop `base_url`/`api_key_ref`/`provider_base_url`/`provider_api_key_ref`),
  keeping `endpoint_kind`/`credential_status`. Mirror the already-redacted `message`.
- **Tests/Eval:** assert the action result contains no secret-ref/endpoint keys; add
  this assertion to the `:v058` `models-redaction-001` eval (see R3-A).
- **Doc edits:** none required beyond noting the redaction in ADR 0070's v0.58 note.

#### R1-B — Reconcile `exposure: :agent` on the two profile reads
- **Finding (MAJOR):** the two profile reads are `exposure: :agent` (router-promoted
  per the v0.54 M9.1 note), making them intent-router candidates that carry the R1-A
  DTO — a user turn can reach them. This contradicts the ADR-0070 discipline the
  panels embody ("operator reads are `:internal`, not intent candidates"). The other
  panel reads (`model_doctor`) are correctly `:internal`.
- **Evidence:**
  `.../actions/settings/list_provider_profiles.ex:6`, `.../list_model_profiles.ex:6`;
  registered as agent candidates `registry.ex:224-225,417-423`.
- **Fix (choose one, decide during implementation):** (a) add `:internal`
  operator-read variants for the Models-panel inventory and point the panel at them;
  or (b) keep `:agent` and record an explicit **ADR 0070 amendment** carve-out — but
  only valid once R1-A makes the agent-routable DTO secret-free. R1-A is the
  load-bearing prerequisite either way.
- **Doc edits:** `docs/adr/0070-*` — amendment documenting the chosen disposition.

#### R1-C — Record pre-dispatch protocol rejections as audit events
- **Finding (MAJOR):** ACP `session/prompt` records inbound only after its guards
  pass; failures (uninitialized, surface-disabled, bad session, bad prompt) fall to
  the `else` and return an error with **no** `record_rejection`. MCP stdio
  `read_resource` likewise records no event. OpenAI + MCP-HTTP already close this,
  making ACP/stdio-resource the outliers (ADR 0073 req 4).
- **Evidence:**
  `apps/allbert_assist/lib/allbert_assist/public_protocol/acp/server.ex:136-143`;
  `apps/allbert_assist/lib/allbert_assist/public_protocol/mcp/runtime.ex:124-141`.
- **Fix:** emit `EventRecorder.record_rejection(surface, …)` in the ACP guard `else`
  branch and around the MCP `read_resource`/disabled-surface paths, matching the
  OpenAI controller pattern.
- **Tests/Eval:** assert a rejection event is persisted on each guard failure;
  fold into the `event-audit-parity-001` behavioral eval (R3-A).
- **Doc edits:** `docs/plans/v0.58-request-flow.md` S4/S6 expected output if the
  operator-visible event report changes.

#### R1-D — Remove residual direct Settings reads in web surface code
- **Finding (MAJOR):** web reads operator-tunable config directly via the sanctioned
  `Settings.get/1` accessor (softer than raw `Settings.Store`, hence MAJOR not
  BLOCKER), violating ADR 0073 req 2/5 and the DoD "no operator-tunable config read
  outside Settings Central."
- **Evidence:** `apps/allbert_assist_web/.../components/layouts.ex:225`
  (theme/high-contrast/reduce-motion);
  `apps/allbert_assist_web/.../workspace/components/template_create.ex:368,389`
  (`templates.create.enabled`, `templates.allowed_patterns`).
- **Fix:** route through the `resolved_settings_snapshot` action / a registered
  read-only action, or resolve layout attributes server-side via the same snapshot.
  Confirm the M5 `settings_central_no_bypass` Credo guard covers (or is extended to)
  these web `Settings.get/1` call classes.
- **Tests:** existing web suites; assert no direct `Settings.get/1` in the two
  modules (or guard-allowlisted with rationale).

### R2: Design-System Completion

#### R2-A — Bring the home page `/` onto the design system (BLOCKER)
- **Finding (BLOCKER):** the home page bypasses the entire system — bespoke HEEx, no
  catalog, generic `Layouts.app` (not `operator_shell`), hardcoded hex
  `bg-[#e8f0f7]`/`text-[#3b5b7a]`, `text-slate-*` utility soup, raw `btn btn-primary`/
  `btn btn-outline`, zero `workspace-button`, and no `data-theme`. Live-confirmed via
  DOM inspection (`has_operator_shell=false`, `has_catalog=false`) and screenshot.
  ADR 0074 §4 makes the catalog/shell the boundary for **every** web page; the home
  page was overlooked (the original M6–M9 scope named only `/jobs`/`/objectives`).
- **Evidence:**
  `apps/allbert_assist_web/.../controllers/page_html/home.html.heex:1-39`.
- **Fix:** fold `/` into the catalog + `operator_shell`; OR, if `/` is intentionally
  a thin landing page, at minimum replace the hex/slate literals with token vars and
  the `btn` classes with `workspace-button`, and emit the shell `data-*` contract so
  theming inherits.
- **Tests:** add `/` to the shell+catalog conformance check; assert no arbitrary
  hex/slate/`btn` in the rendered home markup.

#### R2-B — Adopt the variant registry across the ~40 raw `btn` sites
- **Finding (MAJOR):** ~40 button sites use raw DaisyUI `btn btn-*` instead of
  `Patterns.button_class!/1` / the `:button` catalog atom — including the M10
  operator panels v0.58 itself added. `template_create.ex:279-296` does conditional
  class branching, the exact anti-pattern the registry replaced. Live-confirmed:
  `btn btn-secondary btn-sm` present in the workspace shell.
- **Evidence:** `operator_panels.ex` (172,236,266,277,429,487,668,726);
  `settings_central.ex` (343,453,589,604,619,643,746,785,795,830);
  `plan_build.ex` (101,109,127,200,365); `onboarding.ex` (192-309);
  `template_create.ex` (279-296,345). Competing registry:
  `core_components.ex:94-119` (`<.button>`).
- **Fix:** route these through `Patterns.button_class!/1` or the `:button` atom;
  retire or explicitly out-of-system-document `core_components.ex` `<.button>` and
  stop using it on operator pages.
- **Tests:** extend `patterns_test`/`renderer_test` to assert panels emit registry
  classes, not raw `btn`.

#### R2-C — Complete the shared pattern library (or formally defer)
- **Finding (MAJOR):** the pattern library shipped **modal-only**
  (`workspace_modal` + button/badge variants). The error/status/loading/streaming/
  drawer/table patterns ADR 0074 §3 promises remain page-local — each page re-rolls
  `role=status`/`role=alert` blocks. This reads as incomplete against the M7 DoD, not
  a clean deferral.
- **Evidence:** `objective_live.ex:70,264-270` (hand-built `workspace-status-callout`/
  `workspace-error-callout`); ADR 0074 lines 28,118-122.
- **Fix:** add `Patterns.error_callout/1`, `Patterns.status_callout/1`, and the
  drawer/loading patterns; replace the page-local copies. **OR** (operator's call
  during implementation) formally narrow ADR 0074 §3 to the patterns actually shipped
  and move the rest to v0.59 with an explicit note.
- **Doc edits:** `docs/adr/0074-*` — reconcile the pattern-library scope to the
  decision taken; `docs/developer/web-design-system.md` accordingly.

### R3: Eval & Policy — Code Fixes

#### R3-A — Add behavioral assertions to the `:v058` eval module
- **Finding (MAJOR):** the `:v058` eval test file is **inventory-only** — its tests
  assert rows exist/are grouped/carry `:allowed`/`:denied` with non-empty assert
  lists, but run **no** surface-consistency behavior (no `Runner.run`,
  `SurfacePolicy`, `Renderer`, or `EventRecorder` calls). The behavior *is* covered
  by the unit lanes bundled in `release.v058`, but this diverges from the v0.57
  `V057CodingEvalTest` convention (inventory **plus** behavior in one file) and reads
  as a coverage-shape regression.
- **Evidence:**
  `apps/allbert_assist/test/security/v058_surface_consistency_eval_test.exs:48-76`.
- **Fix (code, per operator decision):** add real behavioral assertions to the eval
  module covering the load-bearing invariants — renderer no-`surface_payload`-leak,
  `EventRecorder` records by `surface_id` (incl. the R1-C rejections), profile-DTO
  redaction (R1-A), and panel no-authority — matching the v0.57 convention. Keep the
  inventory/exact-set check.
- **Verification:** the eval module exercises behavior, not just registration;
  `release.v058` stays green.

#### R3-B — Extend surface-policy enforcement to the remaining operator reads
- **Finding (MAJOR):** `SurfacePolicy.report_policy/3` is consulted in only 4 list
  actions; ~83 other `:read_only`/`:internal` operator reads (intents list,
  `model_doctor`, status, jobs) never consult the policy, while the plan/ADR prose
  implies coverage "across the uniform surfaces."
- **Evidence:** `report_policy` consulted only in `list_settings.ex:33`,
  `list_model_profiles.ex:32`, `list_provider_profiles.ex:32`, `list_channels.ex:32`.
- **Fix (code, per operator decision):** extend `report_policy` consultation to the
  remaining inventory/raw-capable operator reads (start with the operator-panel reads
  — intents list, `model_doctor` — then the broader read set), so policy genuinely
  governs report shape/row bounds across the operator surface. Keep the policy
  fail-closed default.
- **Tests/Eval:** extend `surface_policy_actions_test`; add a `raw-report-affordance`
  assertion for a newly-covered read.
- **Doc edits:** update `docs/plans/v0.58-plan.md` + `docs/adr/0073-*` to state the
  (now-widened) actual coverage rather than an aspirational claim.

## Deferred to v0.59 (accepted-with-note)

These are real but were deliberately scoped out (Locked Decision 9 / M-drift notes)
or are cosmetic; they belong to the v0.59 final-cleanup unless the operator promotes
them:

- **`field/3` long tail** — ~128 domain-local copies untouched (the bulk of
  pillar-4's "100+ sites"); the single largest remaining duplication family.
- **`limit/1` clamp folding** — 5 inline clamp-with-bounds sites not folded into
  `Validation.clamp_limit`.
- **Identity literals** — `@default_user_id "local"`/`@default_external_user_id
  "web-local"` (`workspace_live.ex:54-57`) fed through `Identity.resolve` (substance
  met); "User local" subtitle painted in `/jobs`. Source from Settings Central / a
  named constant.
- **Objectives nav wart** — the "Objectives" nav item routes to `/workspace`
  (documented; no list route).
- **Theming inheritance** — `data-theme` is `null` on `<html>` across pages
  (tokens at `:root` still apply; theme-variant inheritance for non-workspace pages
  needs a check).
- **`ErrorExtraction` unit test** — covered only transitively.
- **Bespoke `response_error/1`** — `allbert.voice.local.ex:100-102`,
  `marketplace/panels/catalog.ex:190-191` (sanctioned legacy operator copy).
- **`Serialization.stringify_keys` struct passthrough** / **`Maps.get_any` falsy
  semantics** — latent, no real caller affected; document the intended behavior.
- **Stale `CHANGELOG.md:296`** ("Intents panel deferred to v0.58") — the M15
  changelog sweep will replace it.

## Sequencing

1. Implement **R1** (A→D); re-run `mix allbert.test release.v058`.
2. Implement **R2** and **R3** (parallelisable); re-run the gate.
3. Update the conformance matrix + ADR/plan prose listed per item.
4. Re-run the gate green, then proceed to **M14 manual operator validation** against
   the remediated code.
5. Any substantial R-fix that changes an operator-facing flow is re-validated live
   (not just by paper audit) in the final audit pass.

## Definition of Done (this remediation round)

- BLOCKER R2-A and all eight MAJOR items closed in code, or explicitly converted to a
  documented v0.59 deferral with operator sign-off.
- `mix allbert.test release.v058` green; `:v058` eval module exercises behavior
  (R3-A); surface-policy coverage widened (R3-B).
- Warning-as-error compile + `mix allbert.test docs` clean.
- Conformance matrix and the named ADR/plan prose reflect the remediated reality.
- Pass-1 audit findings either fixed or recorded as accepted v0.59 deferrals above.

## Non-Goals

- No v0.59 scope (settings migration ADR 0046, ADR 0065 central param contracts,
  onboarding simplification) is pulled into this round.
- No new authority, exposure, or surface is introduced; remediation only brings the
  existing surfaces into conformance.
