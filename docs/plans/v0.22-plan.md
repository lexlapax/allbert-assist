# Allbert v0.22 StockSage Python Bridge

Formerly tracked as M-D2b.

## Purpose

v0.22 brings the existing Python StockSage/TradingAgents behavior into Allbert
through a supervised bridge. The bridge is transitional: it gives Allbert a
working baseline before native Jido trading agents replace it after parity.

## Expected Inputs

- v0.20 StockSage plugin, app, domain records, safe local actions, and queue
  records exist.
- v0.15 app registration is available.
- External market-data and service calls remain governed by Resource Access
  Security Posture, Settings Central, confirmations, redaction, traces, and
  audits.

## Expected Direction

- Add `StockSage.TraderBridge` as a supervised bridge. Start with a spike
  between ErlPort and JSON-over-stdio Port; choose the simpler local
  operational path.
- Start `StockSage.TraderBridge` as a child of `StockSage.Supervisor`, the
  supervisor returned by `StockSage.Plugin.child_spec/1`. Do not start the
  bridge from a separate Allbert core supervisor or one-off task; the bridge
  must share the plugin lifecycle.
- Add `priv/python/bridge.py` wrapping the frozen Python TradingAgents
  baseline.
- Add `StockSage.Actions.RunAnalysis` as a registered Jido action calling the
  bridge.
- Update `StockSage.Plugin.actions/0` and the `allbert_plugin.json`
  `contributions.actions` list so `StockSage.Actions.RunAnalysis` is visible
  through `mix allbert.plugins show stocksage` and through plugin-contributed
  action registration.
- Add `mix stocksage.analyze AAPL 2026-05-01`.
- Consume v0.20 queue records and persist bridge results into the existing
  `stocksage_analyses` and `stocksage_analysis_details` tables.
- Add or expand the `run-analysis` StockSage skill now that execution exists.
- Route "analyze AAPL" through StockSage skill/action only when app context
  makes StockSage in scope.

## Non-Goals

- No native Jido trading agents as default.
- No broad shell escape through the Python bridge.
- No PostgreSQL/Oban requirement.
- No LiveView StockSage UI.

## Test Plan

- Bridge accepts only structured JSON/protocol input.
- Invalid ticker/date/engine arguments fail safely.
- Bridge crash returns an error without leaking partial private data.
- `mix stocksage.analyze AAPL 2026-05-01` returns and persists a decision.
- The action is reachable through Allbert routing only with explicit StockSage
  context.
- `mix allbert.plugins show stocksage` lists `StockSage.Actions.RunAnalysis`
  as a StockSage plugin contribution.
- OTP supervision tests show `StockSage.TraderBridge` starts under
  `StockSage.Supervisor` and stops/restarts with the plugin supervisor.

## Definition Of Done

v0.22 is done when Python baseline analysis runs through a supervised,
traceable, plugin-contributed registered Allbert action, the bridge is owned by
`StockSage.Supervisor`, plugin inspection shows the action contribution, and
results persist to StockSage SQLite storage.
