## Locked Decisions

Each is recorded in `v0.50-plan.md` Â§"Locked Decisions".

1. Thin in-tree CAS over BEAM primitives; no third-party CAS/upload dependency.
2. Identity is `artifact://sha256/<hex>` â€” immutable, dedup-keyed content
   address.
3. Layout `<ALLBERT_HOME>/artifacts/objects/<ab>/<cd>/<sha256>`; a registered
   Home root.
4. Trace-safe, markdown-first per-artifact metadata sidecar + in-memory lookup;
   metadata decoupled from the object layer.
5. Metadata allow-list `{sha256, mime, byte_size, origin, source_resource_uri,
   created_at, retention, redaction_status, lifecycle, provenance}`; provenance
   is a bounded thread/message/id summary only; raw bytes never traced.
6. Floors: `artifact_read` `:allowed`; `artifact_write` `:allowed`;
   `artifact_delete` `:needs_confirmation`.
7. Retention default-off, operator-removable; mark-and-sweep GC.
8. `put_artifact`/`get_artifact`/`list_artifacts`/`delete_artifact` actions +
   the first supervised Jido ingestion sensor (`use Jido.Sensor`, supervised by
   `Jido.Sensor.Runtime`, advisory only).
9. Backfill the retained branch of v0.48 audio + v0.49 vision/generated-image
   media; ephemeral scratch and historical Browser cache stay out of M5.
10. Content address / metadata / doctor success never grant permission.
11. Artifactâ†”thread/message links in a SQLite `artifact_thread_links` join table
    (`role` created_by/referenced_by) from `context.request`; provenance, never
    authority.
12. The browsing repository (panel + `/apps/artifacts/<sha>` page + CLI) ships in
    v0.50b over core `:artifact_read` actions; v0.50 ships the read/query actions.

## Main User/Operator Flows

- Operator uploads a file (any type); Allbert stores it as a content-addressed
  artifact and returns its `artifact://sha256/<hex>` handle.
- Allbert creates an artifact (e.g. a generated image) or finds one (e.g. a
  browser-research download); it is stored through the same CAS.
- Operator lists/inspects stored artifacts (metadata only) and removes one
  (confirmation-gated delete).
- Operator (or the v0.50b browser) lists artifacts created/referenced in a
  thread, and from an artifact looks up the thread/turn that produced it.
- A retained voice capture, vision image, or generated image is ingested into
  the CAS instead of a standalone media root.

## Provider/Store Resolution

There is no provider here â€” the store is local and in-tree. A `put` hashes the
bytes (`:crypto.hash(:sha256, â€¦)`), derives `artifact://sha256/<hex>`, and writes
the object atomically (temp + `File.rename`) under the sharded layout, skipping
the write if the object already exists (dedup). A `get` resolves the hash to its
object path; `list` reads the metadata index; `delete` removes object + sidecar
after confirmation.

## Data Flow

```text
operator upload | Allbert-created | Allbert-found (approved tool)
  -> :artifact_write permission (floor :allowed) + ingest bounds (max_bytes, mime)
  -> sha256(bytes) -> artifact://sha256/<hex>
  -> dedup check -> atomic write to objects/<ab>/<cd>/<sha256> (skip if present)
  -> metadata sidecar (allow-list) + in-memory index; trace stores redacted
     metadata only (content_sha256 is trace-safe; raw bytes never traced)
  -> artifact_thread_links row from context.request (thread_id, message via
     input_signal_id, role=created_by, deterministic link id for idempotence);
     originating ids denormalized to sidecar provenance metadata

list by thread -> :artifact_read -> Ecto query over artifact_thread_links
  (user-scoped, role-filtered); reverse lookup filters by artifact_sha256

retained v0.48 audio / v0.49 vision or generated-image write
  -> Jido.Sensor emits allbert.artifact.ingest_requested (advisory)
  -> explicit IngestionConsumer dispatch target publishes the redacted signal
  -> caller runs core put_artifact path (if artifacts.enabled + retention)

get/list -> :artifact_read (floor :allowed) -> Resource Access -> metadata/bytes
delete -> :artifact_delete (floor :needs_confirmation) -> remove object + sidecar
gc -> mark-and-sweep reconcile index vs objects -> operator-policy removal
```

## Authority Invariants

- A content address, metadata record, or doctor success never grants action
  authority.
- Operator-supplied, Allbert-created, and Allbert-found artifacts only; no
  arbitrary remote fetch beyond already-approved tool paths.
- `:artifact_read`/`:artifact_write` are `:allowed`; `:artifact_delete` is
  confirmation-gated.
- `artifact://sha256/<hex>` ids are inert identifiers; malformed ids are
  rejected and valid ids do not grant permission.
- Binary content is bounded, default-off for retention, and redacted from
  traces/audits/assigns/CLI; only allow-listed metadata persists.
- The Jido ingestion sensor is advisory; it never elevates authority or
  auto-promotes to memory, and it has no private writer around `put_artifact`.
- An artifactâ†”thread/message link is provenance only; a thread id grants no
  permission, and by-thread listing still resolves through `:artifact_read`.

## Handles / CLI Surfaces

- Resource `artifact://sha256/<hex>`.
- Registered core actions `put_artifact`, `get_artifact`, `list_artifacts`
  (incl. by-thread filter), `delete_artifact`, `artifact_threads` (reverse
  lookup), `artifact_doctor` (redacted store health).
- Permission classes `:artifact_read`, `:artifact_write`, and
  `:artifact_delete`; a separate action capability `execution_mode` descriptor
  such as `:artifact_store` (the execution descriptor is not the permission
  atom).
- `artifacts.*` settings (enable, root, retention, bounds, dedup, redact).
- The `mix allbert.artifacts` CLI and the browsing panel/page ship in v0.50b as
  thin shells over the core read actions above.

- Store fixture artifacts of several types (audio, image, PDF, text); confirm
  dedup on a repeat `put`.
- Confirm oversized/disallowed artifact denial before write.
- Confirm traces redact the binary payload (metadata only).
- Confirm `delete` requires confirmation and removes object + sidecar.
- Confirm retention default-off and operator removal.
- Confirm the Jido sensor ingests a retained-media fixture only when
  `artifacts.enabled`, through the same `put_artifact` path as actions.
- Confirm a `put` inside a thread records a `created_by` link, by-thread listing
  returns it, duplicate puts do not create duplicate links, the reverse lookup
  returns the thread, and the link grants nothing.
- Confirm a content address grants no permission by itself.
