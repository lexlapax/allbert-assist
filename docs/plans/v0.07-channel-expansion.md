# v0.7 — Channel expansion

Status: Shipped

## Context

v0.2's daemon (ADR 0018) makes attachable channels a first-class idea (ADR 0013), but in practice only the REPL/CLI channel has existed. v0.6 hardens the v0.5 substrate (session durability, cost cap enforcement, staged-memory review polish, and memory verification). v0.7 turns the channel abstraction into reality by adding one new channel — Telegram as the pilot — while formalising the `Channel` trait so every future channel (Discord, WhatsApp, web, email, SMS) fits the same shape.

Channels differ from a REPL in important ways: they are asynchronous, they have their own presentation idioms, they sometimes lack any confirmation path at all, and each one has its own trust story. The Channel trait must encode those differences as capability flags rather than forcing every channel to pretend it is a terminal.

v0.7 also folds in four retrospective findings that naturally belong alongside the channel work:

- Normalising the **two-tier tool world** (kernel-native vs registry) so hooks observe every tool call uniformly. Channels without direct REPL supervision need uniform policy observability.
- Gating **background web-learning** behind explicit user intent. Channels make this urgent: a Telegram user saying "look that up" should stage research with the same privacy posture as a REPL user.
- Replacing the **one-level sub-agent cap** with a budget-governed model. Channel-driven research workflows need depth; nesting count is a crude proxy.
- **Intent-guided routing that does real work** (not just labels): channels with sparse turns need the intent step to shape prompt construction, agent choice, confirmation style, and tool preference, not just annotate the trace.

## Goal

Deliver the `Channel` trait and its capability model, ship the Telegram pilot, normalize the tool surface so hooks are uniform, wire explicit-intent web-learning, lift sub-agent depth to a budget-governed bound, and upgrade intent routing from label to real routing defaults without turning it into a hard gate.

## Milestones

Milestones are ordered by implementation dependency. Each one can land in its own PR; later milestones assume the earlier ones are merged.

### M1 — `Channel` trait and capability flags

Scope: the trait and capability model that every later milestone depends on.

- New `allbert-channels` crate. Rationale: adapter dependency graphs (teloxide, reqwest) should not be mandatory for tests that do not exercise channels. Daemon and proto crates stay unencumbered.
- `Channel` trait with `kind`, `capabilities`, `receive`, `send`, `confirm`, `shutdown`.
- `ChannelCapabilities` struct with the full flag set: `supports_inline_confirm`, `supports_async_confirm`, `supports_rich_output`, `supports_file_attach`, `supports_image_input`, `supports_image_output`, `supports_voice_input`, `supports_voice_output`, `supports_audio_attach`, `max_message_size`, `latency_class` (`Synchronous | Asynchronous | Batch`).
- Per-channel allowlists alongside `~/.allbert/config`. Unknown senders dropped with audit log entry — no implicit session creation.
- Channel secrets under `~/.allbert/secrets/<channel>/` with IPC-socket-mode permissions (ADR 0023).
- Daemon consults capability flags at turn time: confirm-trust path selection, staged-memory notice rendering (ADR 0050), multimodal gating.
- v0.7 uses one shared bootstrap persona/identity bundle across REPL, CLI, and Telegram. Channel-specific identity overlays are deferred to v0.8's continuity work.
- The REPL and CLI adapters are refitted to the trait (declaring `supports_inline_confirm: true`, `latency_class: Synchronous`) with no behavioural change.
- Covered by ADR 0055.

### M2 — Tool-surface normalization (incremental)

Scope: promote kernel-native ops to `Tool` implementations so hooks observe uniformly. Sequenced to reduce blast radius.

- **Phase 2a — memory tools.** Promote `read_memory`, `write_memory`, `search_memory`, `stage_memory`, `list_staged_memory`, `promote_staged_memory`, `reject_staged_memory`, `forget_memory`. Wire `ToolCtx.memory`.
- **Phase 2b — skill tools.** Promote `list_skills`, `invoke_skill`, `create_skill`, `read_reference`, `run_skill_script`. Wire `ToolCtx.skills`.
- **Phase 2c — `spawn_subagent`.** Promote last; overlaps with M4 budget plumbing and is the most complex surface.
- Existing `BeforeTool`/`AfterTool` hooks gain observation of the newly-promoted tools from day one.
- SecurityHook on the newly-observable memory and skill tools is **actionable from v0.7** (not log-only). A denied memory write, skill invocation, or sub-agent spawn returns a structured tool error the calling agent can read and react to.
- Bootstrap prompts (`TOOLS.md`, `AGENTS.md`) are regenerated to describe the unified surface.
- Incremental rollout rationale: a big-bang flip would couple to every other v0.7 milestone; phased promotion lets us ship M3/M4 behind stable tool-call observation.
- Covered by ADR 0052.

### M3 — Async-confirm state machine

Scope: give async channels a durable confirmation path.

- Suspend-for-approval turn state. Pending approval persisted at `~/.allbert/sessions/<sid>/approvals/<approval-id>.md` (journal-compatible markdown per ADR 0049).
- `meta.json` gains `pending_approval: <approval-id>`. **One pending approval per session at a time** in v0.7; a second gated action refuses with `prior-approval-pending` and can be retried after resolution. Multi-approval inbox is v0.8.
- Only the channel-local sender that triggered the turn may resolve the approval. Other allowlisted senders replying are ignored with an audit entry.
- REPL and CLI may inspect pending approvals in v0.7, but they do not resolve channel-originated approvals. Cross-surface approval resolution is deferred to v0.8's approval inbox.
- Timeout: `channels.approval_timeout_s` (default `3600` = 1h). Expiry auto-rejects with a `confirm-timeout` error the agent can surface.
- Daemon-restart recovery: pending approvals survive because the markdown file is durable. On resume the kernel re-enumerates pending approvals; expired or orphaned approval replies are dropped with an audit entry.
- REPL and CLI keep the v0.2 inline blocking prompt — no behavioural change.
- Covered by ADR 0056.

### M4 — Budget-governed sub-agent depth

Scope: lift the one-level cap; express depth control in the cost/time currency operators already reason in.

- Each sub-agent inherits `remaining_budget_usd` and `remaining_budget_s` from its parent at spawn time.
- Further spawns subdivide the same budgets. Spawn refuses with `budget-exhausted` when either budget reaches zero.
- `spawn_subagent` grows one optional input field:
  - `budget.usd?: number`
  - `budget.seconds?: integer`
- If `budget` is omitted, the kernel applies the default even split of the parent's remaining budget across sibling spawns in that model round.
- If `budget` is provided, each provided bound must be less than or equal to the parent's remaining bound. Impossible allocations fail with `budget-invalid`; zero remaining usable budget fails with `budget-exhausted`.
- Unspecified dimensions continue to use kernel defaults derived from the parent's remaining budget.
- Root-turn budgets: `limits.max_turn_usd` (default `$0.50`) and `limits.max_turn_s` (default `120`). REPL overrides: `/cost --turn-budget <usd>`, `/cost --turn-time <s>`.
- **Concurrent sub-agents allowed.** The parent may request explicit allocation through `spawn_subagent.budget`; otherwise the kernel divides evenly.
- Channel-originated turns inherit channel defaults; job-originated turns inherit the job's declared budget (ADR 0022 frontmatter + defaults).
- Daily-cap interaction (ADR 0051): the lower of per-turn and remaining daily budget applies.
- Bootstrap prompts document expected agent behaviour on `budget-exhausted` (return partial, request more budget, narrow scope).
- Covered by ADR 0054.

### M5 — Intent-guided routing defaults (hint-based)

Scope: make intent operational without turning it into a hard gate. ADR 0030's "intent is a hint, not a hard gate" rule remains intact.

- `IntentShape { prompt_preamble: Option<String>, tool_priority_order: Vec<ToolName> }` computed per classified intent.
- The shape is injected into the agent prompt as advisory; the agent may ignore it.
- No intent removes tools from the runtime. Policy envelope, skill fences, and agent judgment still decide final availability.
- Low-confidence classification silently falls back to `task` (unchanged from v0.3). v0.7 does not require a visible "I'm interpreting this as X" preamble.
- Reference shapes documented in prose, not a matrix ADR:
  - `chat` — conversational preamble, lower side-effect bias, prefers staying in natural-language explanation unless the user explicitly asks the assistant to act.
  - `task` — full problem-solving preamble, normal tool ordering, default sub-agent budget.
  - `schedule` — scheduler-oriented preamble, job-management and durable-mutation review flows preferred, confirmation language foregrounds durable effects.
  - `memory_query` — retrieval-first preamble, memory tools preferred, concise evidence-oriented response shaping.
  - `meta` — operator/meta preamble, status/setup/cost surfaces preferred, side-effect-avoiding guidance.
- If the shape grows enough independent surface to warrant an ADR in implementation, promote it then; the v0.7 plan does not commit to one.

### M6 — Telegram pilot

Scope: the reference non-REPL `Channel` implementation.

- `teloxide` crate for the Bot API; **long-polling only** in v0.7. Webhooks are documented as a future option but are not shipped code.
- `TelegramChannel` impl declares capabilities per ADR 0057's contract (`supports_async_confirm: true`, `supports_inline_confirm: false`, `supports_rich_output: true`, `supports_file_attach: true`, `supports_image_input: true`, `latency_class: Asynchronous`, `max_message_size: 4096`).
- Per-chat allowlist at `~/.allbert/config/channels.telegram.allowed_chats`. Bot token at `~/.allbert/secrets/telegram/bot_token`.
- **Session model**: inbound message from an allowlisted sender resumes the sender's most-recent session if younger than `daemon.session_max_age_days` (ADR 0049); else creates a new session. `/start` is treated identically to any other inbound message — no "reset on /start." `/reset` forces a new session.
- Telegram's resume-most-recent model is the v0.7 reference implementation, not a universal mandate for future channels. Other adapters may choose different session strategies as long as the `Channel` contract and adapter docs make that strategy explicit.
- **Rate limits** per ADR 0057:
  - `channels.telegram.min_interval_ms_per_chat` (default `1200` — safe margin over the 1 msg/sec per-chat limit),
  - `channels.telegram.min_interval_ms_global` (default `40` — ~25 msg/sec under the 30 msg/sec global ceiling).
  - Per-chat FIFO queue; global FIFO queue. Queued messages observable in trace.
- **Approval vocabulary** (per ADR 0056): `/approve <approval-id>`, `/reject <approval-id>`. Reaction-emoji approval is a stretch goal, not required.
- **Outbound rendering**: Telegram MarkdownV2 with safe-escape; long responses chunked at `max_message_size` with continuation markers.
- Cost-cap override from Telegram: `/override <reason>` reply to a cap-refusal message applies for one turn (closes a v0.6 remaining planning question).
- Covered by ADR 0057.

### M7 — Background web-learning under explicit intent

Scope: give channel-driven research a privacy-respecting path into staged memory.

- `web_search` and `fetch_url` gain an optional `record_as: "<summary>"` parameter.
- When populated (agent response to explicit phrases like "remember this" or "look that up and save it"), the result stages a memory entry with `kind: research`, `source_url`, `summary`, `fetched_at`, and fingerprint.
- Absent `record_as`, behaviour is unchanged from v0.5.
- `research`-kind entries route through the same 5-per-turn / 90-day / dedup limits as any other staged memory (ADR 0047).
- No auto-stage. Privacy-by-default.
- Covered by ADR 0053.

### M8 — CLI surface additions

Scope: make the new runtime surfaces operable without dropping into raw filesystem inspection.

- `allbert-cli daemon channels list | status [<kind>] | add <kind> | remove <kind>` — enumerate and manage configured channels.
- `allbert-cli approvals list | show <approval-id>` — inspect pending async approvals from the REPL/CLI side without resolving channel-originated approvals.
- `/cost --turn-budget <usd>` and `/cost --turn-time <s>` REPL commands (from M4).
- `/cost --override <reason>` extended to Telegram as `/override <reason>` (from M6).
- Output format follows the v0.5 status-surface idioms: human text by default, `--json` flag for scripting.
- `daemon channels add|remove` is generic at the CLI surface but limited to built-in known kinds in v0.7. `telegram` is the only non-REPL addable kind in this release.
- `daemon channels add <kind>` creates or updates the channel config entry, scaffolds required allowlist/secret file paths if missing, and marks the channel enabled. It does not inline-prompt for secrets; missing token or allowlist leaves the channel in a visible `misconfigured` / `needs_setup` state.
- `daemon channels remove <kind>` disables the channel in config and stops loading it on next daemon start or reload. It does not delete secret or allowlist files.
- `daemon channels status` exposes at least: enabled or disabled, configured or misconfigured, last error, queue depth if running, and effective capability flags.

### M9 — Multimodal: image input

Scope: the minimum multimodal surface that lands cleanly with the v0.7 timeline.

- `supports_image_input: true` on TelegramChannel when the provider declares vision support. Anthropic: always. OpenRouter: model-dependent, queried at provider-init time.
- Inbound Telegram photo → staged as a file under the session working directory → referenced in the provider message payload per the provider's image contract.
- Inbound images are session artifacts, not durable memory artifacts. They archive, forget, and purge with the parent session under the same session-retention rules from v0.6; v0.7 does not add a separate media-retention subsystem.
- Bootstrap prompts (`TOOLS.md`) document that incoming images are referenced by session-scoped path; no raw binary in prompt text.
- **Voice notes and image output are explicitly deferred to v0.8 or later.** The v0.7 `ChannelCapabilities` struct carries flags for them so adapters can declare them later without trait churn, but no v0.7 milestone relies on them.

### M10 — Doc alignment pass

Scope: shipped docs match the shipped surface.

- **`docs/vision.md`** — refresh the Channel Direction section (Telegram pilot, capability flags, async-confirm, image input); confirm the Multimodal sentence aligns with M9 scope.
- **`README.md`** — release-facing sections: `Channel` trait, Telegram pilot (long-polling, approvals, rate-limits), budget-governed sub-agents, CLI additions (`daemon channels`, `approvals list|show`).
- **`docs/onboarding-and-operations.md`** — Telegram setup walkthrough (bot token creation, allowlist, long-polling behaviour), approval inspection from REPL/CLI and resolution from Telegram, intent-shape behaviour, image-input expectations.
- `AGENTS.md` and `TOOLS.md` are regenerated from the M2 unified tool surface and the M4 budget vocabulary.

## Runtime defaults

- **Async confirm timeout**: pending async confirmations expire after a bounded timeout and fail closed by default.
- **Trace attribution**: channel-originated turns are tagged with channel kind and channel-local sender identifier in the trace surface.
- **`HEARTBEAT.md`**: remains deferred. Channel rhythm alone is not enough reason to add another bootstrap artifact — revisit only if async channels produce demonstrable operator pain.
- **Session durability** (v0.6 M2): every channel session is journaled. Telegram threads survive daemon restart.

## ADRs

- **ADR 0052** — Kernel-native operations promote to `Tool` implementations (normalises hook observation). Covers M2.
- **ADR 0053** — Background web-learning stages only under explicit user intent. Covers M7.
- **ADR 0054** — Sub-agent depth is governed by cost/time budget, not nesting count. Covers M4.
- **ADR 0055** — `Channel` trait with capability flags. Covers M1.
- **ADR 0056** — Async-confirm is a suspend/resume turn state. Covers M3.
- **ADR 0057** — Telegram pilot uses teloxide and long-polling. Covers M6.
- Per-channel allowlist trust model is documented inside ADR 0055 rather than its own ADR.
- Intent-shape vocabulary (M5) stays in-plan; promote to an ADR only if implementation surfaces contested trade-offs.
- Secrets layout (`~/.allbert/secrets/<channel>/`) and CommonMark→channel rendering (M6 outbound path) stay in-plan; promote only if policy-loaded in a later release.
- *Deferred, tracked here for v0.11+*: skill-format adapter that imports legacy / MCP / OpenClaw shapes into AgentSkills format.

## Dependencies

- v0.6 hardening — session durability, cost cap enforcement, and staged-memory notice all become cross-channel concerns here. v0.7 depends on v0.6 landing first.
- v0.5 curated memory — channels with sparse turns need cheap context assembly; ADR 0053 builds on staging schema.
- v0.3 intent routing — channel input goes through the same intent step as REPL input, now with prompt-shaping defaults.
- ADR 0013 (channels) and ADR 0018 (daemon host) as runtime substrate.

## Out of scope for v0.7

- Full web UI (deferred).
- Email and SMS channels (follow Telegram pilot into their own tracking once the trait is stable).
- Cross-channel conversation merging (e.g. starting a turn in Telegram and continuing in REPL).
- Network-addressable daemon (out of scope through v0.12; deferred per [roadmap.md](roadmap.md)).
- Cross-channel continuity and sync (scheduled for v0.8 rather than bundled into v0.7).
