This release has two tracks in one version. The **discovery track** (M1-M5)
ships first because it is the foundational primitive: a `find_tools` capability
search that fans out to local tools and to internet MCP registries, plus the
confirmation-gated connect gate by which a discovered server becomes real. The
**integration track** (M6-M11) is the original v0.42 scope — calendar, mail, and
GitHub as **MCP-server-configured workspace panels** plus a single `notes/files`
**native reference plugin** — and it now becomes a clean consumer of `find_tools`
(an unconfigured panel can discover and connect a server through the gate rather
than assuming hand configuration). Native plugin variants for
calendar/mail/GitHub remain post-1.0 follow-on (v0.42.x) and ship only when MCP
coverage proves insufficient for a specific workspace surface or memory-namespace
need.

## Purpose

Make Allbert useful for ordinary personal work without growing
provider-specific dependencies in core, and give Allbert a general,
authority-bounded way to answer "is there a tool for this, and if not, can I get
one?"

The discovery track adds `find_tools`: when the operator needs a capability,
Allbert searches its own registered actions, skills, and already-connected MCP
servers (`find_local_tools`) and, in parallel, searches internet MCP registries
(`find_mcp_tools`), returning one ranked list of candidates. A candidate that is
not already usable is connected only through an explicit, confirmation-gated
connect gate that shows the operator the exact run command or remote URL. This is
the reusable primitive the intent engine, the objective runtime (ADR 0033), and
operator queries can all call.

The integration track makes calendar, mail, GitHub, and notes/files arrive as
MCP-configured workspace panels driven by the v0.40 MCP client. The `notes/files`
integration additionally ships as a small native reference plugin to give plugin
authors a concrete starter scaffold.

## Research Grounding

The v0.42 design is grounded in:

- **The official MCP Registry** (`registry.modelcontextprotocol.io`): read API
  frozen at v0.1, defining `GET /v0.1/servers` (params: `search` = name
  substring only, `updated_since`, cursor pagination, `version`), per-version
  endpoints, and a `server.json` schema (`name` reverse-DNS, `description`,
  `version`, `repository`, `packages[]` with `registryType`/`identifier`/
  `transport`/`runtimeHint`/`runtimeArguments`/`environmentVariables`/
  `fileSha256`, and `remotes[]` with `type`/`url`/`headers`). It is **preview
  with no durability guarantee** and performs **no cryptographic payload
  validation**, so Allbert treats it as a non-durable cache and does its own
  verification.
- **Subregistries / aggregators**: third parties (PulseMCP, Glama, Smithery)
  re-implement the same Generic Registry API and add the search/enrichment the
  official registry lacks (semantic search, popularity, security scans). v0.42
  targets the official registry plus one **no-auth** aggregator (PulseMCP) behind
  a provider port; semantic / API-keyed sources (Smithery) are parked.
- **The cascade / progressive-disclosure pattern** (Anthropic Tool Search,
  Composio tool-calling guide): defer loading tool definitions, search for a
  capability, expand only matching candidates, then act. v0.42 retargets this
  from "pick among tools I already have" to "find a *server* I might add," so the
  terminal step is a human-gated connect, not a tool call.
- **MCP discovery/connection threats** (MCP security best practices, security
  research): SSRF during discovery (the client fetches server-controlled URLs),
  malicious `npx`/`uvx` start commands, tool-poisoning / prompt-injection via
  tool descriptions, registry name-squatting, and rug-pull (post-approval
  definition changes). Mitigations: route every discovery fetch through
  `External.HttpPolicy`; treat all server-supplied text as untrusted metadata;
  mandate pre-configuration consent showing the exact command; hash tool
  definitions at connect and re-verify on reconnect.
- **The Allbert plugin/app contract** for the `notes/files` reference plugin:
  ADR 0017 (plugin contract), ADR 0027 (app/surface contract and memory namespace
  declaration), ADR 0033 (capability vocabulary), the v0.33 intent descriptor and
  handoff substrate, and the v0.32/v0.34/v0.35 workspace shell.

Implementation consequence: discovery adds new read-only actions, one
confirmation-gated connect action, a discovery settings namespace, a tool-source
port, and a passive workspace surface — but no new network primitive (reuses
`External.HttpPolicy`), no new secret store, no new runtime (reuses
`AllbertAssist.Jobs` for scanning), and no new authority root.

## find_tools Port And ToolCandidate Contract

`find_tools` is an orchestrator action that fans out in parallel to adapters
behind a tool-source port and returns one ranked, deduplicated candidate list.

```text
find_tools(need)                       # orchestrator; parallel fan-out + merge + rank
├── find_local_tools(need)             # read-only, no network, no confirmation
│     • Actions.Registry.capabilities/0      (registered actions)
│     • Skills.Registry.list/1               (skills)
│     • configured MCP servers' tools/list   (already connected)
└── find_mcp_tools(need)               # internet cascade, SSRF-guarded, read-only
      • discover_servers → fetch_server_manifest → evaluate_server
      • across official MCP Registry + PulseMCP, behind a registry provider port
```

Source-port contract (`AllbertAssist.Tools.SourcePort` behaviour):

- `search(need :: String.t(), opts :: map()) :: {:ok, [ToolCandidate.t()]} | {:error, term()}`
- adapters: `AllbertAssist.Tools.Source.Local`, `AllbertAssist.Tools.Source.McpRegistry`
  (the registry adapter itself fans out to registry-provider adapters
  `AllbertAssist.Mcp.Registry.Official` and `AllbertAssist.Mcp.Registry.PulseMcp`
  behind `AllbertAssist.Mcp.Registry.Provider`).

Normalized `ToolCandidate`:

```elixir
%{
  name: String.t(),                 # action name, skill name, or server.json name
  description: String.t(),          # untrusted when source is :remote_mcp
  source: :local_action | :local_skill | :configured_mcp | :remote_mcp,
  usable_now?: boolean(),           # local => true; remote_mcp => false
  requires: :none | :connect_confirmation,
  provenance: map(),                # namespace match, verified/official, signing
  signals: map()                    # popularity, recency, security scan, registry id
}
```

The `usable_now?` / `requires` pair is the enforcement seam: a `:remote_mcp`
candidate is inert descriptive metadata until it passes the connect gate. Tool
descriptions from `:remote_mcp` candidates are surfaced verbatim but never alter
agent behavior (ADR 0038 §2; ADR 0048 §2).

## Discovery Settings (`mcp.discovery.*`)

Add the `mcp.discovery.*` namespace to `AllbertAssist.Settings.Schema`.
Discovery is off by default; the background scan is a paused job by default.

```yaml
mcp:
  discovery:
    enabled: false                 # opt-in master switch for find_mcp_tools
    sources:
      official:
        enabled: true              # registry.modelcontextprotocol.io (no auth)
      pulsemcp:
        enabled: true              # no-auth aggregator
    scan:
      schedule: "paused"           # operator_approved cron; job created paused
      max_results: 25
    registry_allowlist: []         # optional reverse-DNS namespace pins
    registry_denylist: []
    auto_connect: false            # invariant: must stay false (connect is confirmed)
```

Safe-write keys: `mcp.discovery.enabled`, `mcp.discovery.sources.*.enabled`,
`mcp.discovery.scan.schedule`, `mcp.discovery.scan.max_results`,
`mcp.discovery.registry_allowlist`, `mcp.discovery.registry_denylist`.
`mcp.discovery.auto_connect` is read-only and validated to remain `false`.

No secret refs are needed for the no-auth sources. A future Smithery adapter
would add `secret://mcp/discovery/smithery_key` and is parked.

The connect gate writes the v0.40 `mcp.servers.*` namespace (ADR 0038 §1); no new
server-config shape is introduced.

## Discovery Permission And Operation Classes

Add to `AllbertAssist.Security.Policy` (following the v0.40 MCP precedent):

- `:tool_discovery` — safety floor `:allowed`, default decision `:allowed`.
  Gates `find_mcp_tools` registry egress. Read-only; creates no grants. (Local
  fan-out needs no new authority.)
- `:mcp_server_connect` — safety floor `:needs_confirmation`, default decision
  `:needs_confirmation`. Gates `mcp_server_connect`. Settings can tighten to
  `:denied` but can never loosen below confirmation.

Add to `AllbertAssist.Actions.Capability` execution modes: `:tool_discovery`
(read-only) and `:mcp_server_connect` (confirmation-gated). No new
`OperationClass` is required: discovery search touches no Resource Access
identity, and the connect gate writes settings + creates a confirmation.

## Action Schemas And Capability Metadata

All actions register through `AllbertAssist.Actions.Registry` and run through
`AllbertAssist.Actions.Runner.run/3`.

| Action | Required params | Optional params | Mode / exposure | Notes |
|---|---|---|---|---|
| `find_tools` | `need` | `sources`, `limit` | `:read_only` / `:internal` | Orchestrates local + mcp adapters in parallel; merges + ranks `ToolCandidate`s. |
| `find_local_tools` | `need` | `limit` | `:read_only` / `:internal` | Inventories actions, skills, configured-MCP tools. No network. |
| `find_mcp_tools` | `need` | `sources`, `limit` | `:read_only` / `:internal` | Registry cascade; permission `:tool_discovery`; egress via `External.HttpPolicy`. |
| `mcp_fetch_server_manifest` | `name` | `version`, `source` | `:read_only` / `:internal` | Normalizes `server.json`; surfaces the exact run command / remote URL. |
| `mcp_evaluate_server` | `name` | `version`, `source` | `:read_only` / `:internal` | Provenance + danger-flag + bounded health probe (extends doctor) + baseline tool-def hash. |
| `mcp_server_connect` | `name` | `version`, `source`, `server_id`, `enable_on_connect` | `:mcp_server_connect` (confirmation-gated) / `:internal` | Pre-config consent → write `mcp.servers.<id>` → persist baseline hash. Never auto-connects. |

`name`/`server_id` use `[A-Za-z0-9_.\-]+`. Resolved run commands/argv, remote
URLs, env, and headers are shown untruncated in the connect consent but redacted
to names/counts in traces and audits; secret values are never rendered.

Integration-track actions (M9 `notes/files` plugin):

| Action | Required params | Mode | Notes |
|---|---|---|---|
| `SearchNotes` (`search_notes`) | `query` | `:read_only` | Reads notes via Resource Access file scope. |
| `ReadNote` (`read_note`) | `path` | `:read_only` | Reads one note via Resource Access. |
| `WriteNote` (`write_note`) | `path`, `content` | confirmation-gated | Writes confirmation-gated via Resource Access. |

## Operator-Visible Win

An operator can ask "find me a tool for X" (or "find an MCP server for
calendar") and get a ranked list mixing tools Allbert already has with internet
MCP servers it could add. Connecting a discovered server always shows the exact
command/URL and requires explicit approval; nothing connects silently. With the
opt-in background scan enabled, suggestions accumulate on a passive Discovery
Suggestions panel the operator reviews on their own time.

An operator can also ask for an agenda, mail summary, GitHub overview, or notes
search from `/workspace` or a supported channel. Each integration renders a
workspace summary panel driven by configured MCP tools and resources; an
unconfigured panel can offer to discover and connect a server through the gate.
The operator approves any effectful action through the normal Approval Handoff
path.

A plugin author opening `./plugins/allbert.notes_files/` sees a complete,
minimal worked example of the plugin contract: app behaviour, surface provider,
panel declarations, memory namespace, intent descriptor, registered actions, and
Settings.Fragment — small enough to read in one sitting.

## v0.40 Capability Matrix Dependency

v0.40 M5 **shipped** the first-server validation matrix (see
`docs/plans/v0.40-plan.md` M5 implementation notes). v0.42 consumes it directly
rather than re-deriving it; the resolved implications are:

| Integration | Required panel data | v0.40 shipped exposure finding | v0.42 resolution |
|---|---|---|---|
| Calendar | upcoming events, event detail, create/update event | **mostly tools** | Agenda reads are **per-call-confirmed tool calls**, or the panel shows an empty/"configure a resource-exposing server" state. v0.42 does **not** promise prompt-free agenda refresh. (Decided in M7.) |
| Mail | inbox/message summaries, message detail, reply/send | mixed resources and tools | Header/detail summaries use `mcp_read_resource` (remembered grant) when the configured server exposes them; otherwise per-call-confirmed. Sends/replies are confirmed tool calls. |
| GitHub | PR/issue/review summaries, repo/readme detail, comments | mixed resources and tools | Prefer resource reads for inspectable artifacts; mutations and searches stay behind confirmed tool calls. |
| In-repo HTTP / stdio mocks | doctor/list/read/call coverage | resources and tools | CI exercises the canonical grant-gated read and confirmed tool-call paths. |

If a real or mock MCP server exposes essential summary-panel data only through
tools, v0.42 keeps that interaction operator-triggered with per-call
confirmation, parks that panel detail, or amends ADR 0038 before implementation.
It must not treat `:mcp_resource_read` grants as authority for `mcp_call_tool`.

## Why notes/files Is The Native Reference

`notes/files` is the cheapest integration to implement natively (just file IO
plus a tiny workspace panel) and the most relatable for plugin authors
(everyone has notes). StockSage remains the depth reference for financial
workflows, native specialist agents, Python bridge integration, and parity
runs; `notes/files` is the shallow reference that proves the contract is
small enough to use as a starter scaffold.

The `notes/files` plugin is also the only native plugin in v0.42 because the
notes/files use case benefits most from a native memory namespace (operators
want notes promoted into Allbert memory) and a custom workspace panel
(MCP-only would lose the markdown-render affordance).

## Why Other Integrations Stay MCP-Only

- **Calendar**: MCP servers for Google Calendar, CalDAV, and Microsoft Graph
  exist or are emerging. Native plugin would lock Allbert into one provider.
- **Mail**: v0.16 email channel already covers the mail-as-channel surface.
  Mail-as-app via MCP avoids duplicating IMAP/SMTP code.
- **GitHub**: official GitHub MCP server is comprehensive. Native plugin
  would replicate GitHub API client surface in core.

Each can graduate to a native plugin in **v0.42.x** follow-on releases if a
workspace surface, memory namespace, or intent descriptor need proves the MCP
coverage insufficient. v0.42.x releases are post-1.0 and not freeze-blocking.

## Security Posture

Discovery:

- `find_mcp_tools` and the manifest/evaluate actions are read-only; every
  registry/server fetch routes through `External.HttpPolicy` (SSRF, host rules,
  bounded timeout/body, redirect denial, redaction). The official registry is a
  non-durable cache: discovery degrades to local-only when it is unreachable.
- Server-supplied text (descriptions, tool schemas) is untrusted metadata and
  never alters agent behavior (ADR 0038 §2).
- `mcp_evaluate_server` flags dangerous run-command patterns (`sudo`, `rm -rf`,
  network egress baked into argv, paths outside expected dirs) and scores
  provenance (namespace match, verified/official, signing); these are advisory
  inputs to the consent, never auto-approve.
- `mcp_server_connect` is the single dangerous transition: confirmation-gated
  (`:mcp_server_connect`, floor `:needs_confirmation`), shows the exact
  untruncated command/URL + required secret refs, writes `mcp.servers.<id>` only
  on approval, and records a tool-definition baseline hash. Reconnect /
  `mcp_doctor_server` re-verifies the hash; a change forces re-consent.
- Background scanning is opt-in (`mcp.discovery.enabled: false` default), a
  paused-by-default `Jobs` job, and writes to a passive surface only — no
  unprompted messaging, no scan-initiated connect.

Integration:

- Integrations are plugin-shaped (not core dependencies). MCP-configured panels
  inherit the v0.40 MCP client trust tier (per ADR 0038). Native `notes/files`
  plugin uses Resource Access for file IO and Security Central for
  permission/confirmation. The notes namespace uses the v0.27 namespace
  declaration contract; v0.47 may suggest memory promotion candidates but never
  auto-promotes.

## Non-Goals

Discovery:

- No auto-connect and no remembered/silent connect approval.
- No proactive push messaging about discovery results (passive surface only).
- No capability-gap-triggered remote acquisition in v0.42 (ADR 0033 objectives
  may consult `find_local_tools`; gap-triggered discovery is parked).
- No semantic / API-keyed registry source in v0.42 (Smithery parked).
- No code-bearing plugin install through discovery — MCP servers only.
- No community trust scoring or registry-moderation authority.

Integration:

- No native plugin for calendar, mail, or GitHub in v0.42 (post-1.0
  follow-on if needed).
- No core dependency on Google APIs, GitHub APIs, or mail provider SDKs.
- No silent credential reuse across unrelated integrations.
- No automatic memory promotion from notes or mail.
- No hosted multi-user authorization model.
- No replacement of StockSage as the depth reference app.

## Definition Of Done

v0.42 is done when:

- `find_tools` fans out to `find_local_tools` and `find_mcp_tools` and returns one
  ranked, normalized candidate list distinguishing usable-now local tools from
  connect-required remote MCP servers;
- a discovered MCP server can be connected only through the confirmation-gated
  `mcp_server_connect` consent that shows the exact command/URL, writes
  `mcp.servers.<id>`, and records a tool-definition baseline hash, with reconnect
  re-verification; nothing auto-connects;
- background discovery is opt-in, paused-by-default, and writes only to a passive
  Discovery Suggestions surface with no unprompted messaging;
- each of calendar/mail/GitHub has a usable MCP-configured workspace panel driven
  by an operator-configured MCP server (or a clear empty state with a
  discover/connect affordance), with the calendar read model decided;
- `./plugins/allbert.notes_files/` is a working native plugin a developer can
  copy as a starting point for their own plugin;
- at least one effectful flow per integration runs end-to-end through registered
  actions + confirmation (create event, reply, comment, write note);
- focused tests/evals prove discovery fetches stay within `External.HttpPolicy`,
  server metadata never becomes authority, connect requires consent, rug-pulls
  are detected, no integration bypasses Allbert's action boundary, and no
  integration auto-promotes content into markdown memory;
- `docs/plans/v0.42-request-flow.md` matches the implementation; roadmap,
  future-features, vision, ADRs, operator/developer guides, agent-context-map,
  and security-hardening reflect the shipped state.
