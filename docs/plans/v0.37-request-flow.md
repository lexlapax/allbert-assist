# Allbert v0.37 Request Flow - Dynamic Code & Config Generation and Live Capability Integration

Status: implemented as `0.37.0` on 2026-05-25; v0.37.1
post-implementation audit hardening and final gates completed on 2026-05-25.
Reopened for v0.37.2 capability-first generator and bounded model-backed
committee implementation before release tagging; v0.37 is not accepted until a
useful generated action can be created, repaired from evidence, sandbox-gated,
approved, run live, and rolled back. Reopened again for v0.37.3 before tagging
so generated actions can express delegated `:memory_write` and
`:external_network` capabilities through reviewed facades. Reopened again for
v0.37.4 before tagging to close audit gaps in discard, security evals, request
permission separation, validator coherence, version metadata, and docs.

Implementation details live in `docs/plans/v0.37-plan.md`.

v0.37 lets Allbert generate Elixir/OTP code and config, trial and gate it using
the v0.36 sandbox runner, and — only after the gate and explicit operator
confirmation — integrate it into the live runtime without a restart. v0.37.2
proved the first source-bearing generator for useful read-only action targets
through an explicit Planner/Author/TrialAuthor/Critic/Repair loop. v0.37.3 adds
delegated write actions whose effects can only reach reviewed facades through
`AllbertAssist.DynamicPlugins.Delegate.run/3`. The whole workflow is
default-off.

## Core Model: Evidence vs Trust

- **Evidence:** generation, sandbox compile/trial, sandbox gate report.
- **Trust grant:** explicit operator confirmation after the evidence is present.
- **Trusted integration:** loader recompiles reviewed source in core and
  registers reviewed actions live through the actions overlay. The shipped
  target remains action-only: pure read-only actions plus delegated
  `:memory_write`/`:external_network` actions whose effects pass through
  allowlisted reviewed facades.
  Generated apps, panels, settings fragments, memory namespaces, objective
  wiring, route pages, and children are rejected in the v0.37 live loader until
  they have separate trusted validators.

Agent output, sandbox success, and gate success are not authority by themselves.

That proves the real umbrella gate path with toolchain-backed deps, writable

## Flow 1: Capability gap is detected

Operator explicitly asks Allbert to create/generate a missing capability, or an
existing objective reaches a step where no registered action/app can satisfy the
operator-approved objective and proposes generation as an explicit next step.

Expected:

1. Intent/action planning may report no eligible existing capability, but that
   report is advisory.
2. The objective runtime records a capability gap only after an explicit
   operator request or an explicit objective step proposal (ADR 0033).
3. Allbert proposes generation as an objective step; generation does not begin
   from mere low intent confidence, rejected candidates, or classifier output.
4. With dynamic generation disabled, `dynamic_codegen.provider_profile` unset
   or unresolved, provider credentials unavailable, or the v0.36 sandbox
   unavailable, Allbert explains it cannot generate and takes no action.
5. Code generation is a last resort. Allbert prefers reusing or composing
   existing registered actions/skills/apps, then a declarative
   `AllbertAssist.Surface` over existing capabilities, and only proposes
   generating new code when neither can satisfy the gap.

## Flow 2: Advisory producer authors a draft

With the workflow enabled, the v0.37 codegen producer writes a source-bearing
action draft for explicit operator/objective capability gaps. The generator is
advisory: it writes draft files and evidence metadata only. It does not sandbox,
approve, integrate, or grant live authority.

Expected:

1. `DynamicPlugins.Codegen.Agent` accepts only explicit operator/objective
   capability-gap requests.
2. `Codegen.Producer` resolves `dynamic_codegen.provider_profile`, checks the
   provider is enabled, and fails closed when a required credential is missing.
3. Provider-call and usage requests are capped by
   `dynamic_codegen.max_provider_calls_per_gap` and
   `dynamic_codegen.max_provider_usage_units_per_gap`.
4. The producer resolves the capability gap through model-backed role packets:
   Planner creates the generation spec and acceptance criteria, Author creates
   action source, TrialAuthor creates focused trial tests, Critic reviews static
   and sandbox evidence, and Repair produces a new revision when validation or
   trial fails. Calls go through `Jido.AI.generate_object/3` behind injectable
   provider boundaries. For effectful drafts, Author may only use
   `AllbertAssist.DynamicPlugins.Delegate.run/3` with a literal configured
   facade name; direct writes, direct network calls, confirmation decisions, and
   protected runtime calls remain forbidden.
5. Deterministic tests use a fake provider with the same structured object
   contract. Opt-in remote smoke tests source `.env` for provider credentials
   without printing secrets.
6. The producer records a draft revision id and source hash for every generated
   file; repair creates a new revision rather than mutating the evidence for a
   prior revision.
7. Draft metadata for source-bearing drafts includes compile-visible generated
   paths, scan paths, budget counters, confirmation ids, and repair history.
   Objective events record the same budget consumption so repair/resume cannot
   silently double-spend provider calls.
8. Draft tier remains `:draft`; no core compile/load/register occurs, and
   ordinary plugin discovery does not scan the draft root.

Provider calls during generation and repair remain bounded by
`dynamic_codegen.max_repair_iterations`, `dynamic_codegen.max_provider_calls_per_gap`,
and any configured provider usage budget. After this explicit generation step is
accepted, those provider calls do not prompt for per-call confirmation; budget
exhaustion fails closed with bounded diagnostics.

The authoring loop is evidence-driven:

1. Planner, Author, and TrialAuthor produce schema-constrained packets.
2. Deterministic prechecks reject missing placeholders, malformed source/test
   files, target-shape mismatches, and obvious authority violations.
3. The v0.36 sandbox runs compile and focused generated tests.
4. Critic receives bounded source/test summaries plus validation and sandbox
   diagnostics. Critic may veto or request repair, but it cannot accept, trust,
   integrate, or advance a tier.
5. Repair is invoked only when repair is requested and produces a new draft
   revision from bounded diagnostics. Prior revision evidence is immutable.
6. The loop stops when deterministic prechecks, focused tests, warning gate,
   selected security evals, and trusted validation pass, or when iteration,
   provider-call, provider-usage, wall-clock, or repeated-identical-failure
   limits are reached.

The durable draft root is not itself a compile root. Before any sandbox run,
v0.37 prepares a disposable project-shaped staging tree whose generated source
files live under the reserved `AllbertAssist.DynamicPlugins.Generated`
namespace paths and whose generated tests live under staged umbrella test
paths. Generated settings fragments remain inert declarations; they cannot set
operator values, secrets, permissions, provider credentials, or resource
grants.

The LLM-backed committee is one draft producer. Everything from `:draft` onward
(staging, trial, gate, loader, overlay, rollback) is producer-agnostic, so v0.38
templated creation can enter the same lifecycle without a codegen committee.

The M16 closeout exposes that lifecycle through
`DynamicPlugins.request_draft_with_gate/3` for generated read-only actions. This
facade requests the draft, runs trial evidence, runs gate evidence, then runs
trusted validation over the gate-passed revision. Failures from sandbox reports,
source-policy denials, timeouts, or trusted validation are converted to bounded
Repair evidence and rerun until the gate/validation pass or a repair,
whole-workflow provider-call/usage, wall-clock, or repeated-identical-failure
limit stops the attempt. The facade returns evidence only; operator-confirmed
integration and rollback remain separate Security Central flows.

The v0.37.3 delegated-write path uses the same lifecycle. A generated
`:memory_write` or `:external_network` action can pass the trusted validator only
when the operator has enabled both its declared permission and the matching
reviewed facade, and when the generated body delegates through
`AllbertAssist.DynamicPlugins.Delegate.run/3` with a binary string literal
facade name. The initial facade ceiling is `append_memory` and
`external_network_request`. The delegated facade keeps its own Security Central
behavior and approval surfaces; `dynamic_codegen.integration_approval_surfaces`
continues to apply only to integration and rollback hot-load confirmations.

## Flow 3: Trial and gate through v0.36 sandbox

Allbert invokes v0.36 sandbox actions for the draft.

Expected:

1. v0.37 builds a v0.36 bundle from the disposable staged project tree, not
   directly from the durable draft root.
2. Generated files that must compile are present inside the bundle's
   `/workspace/project` tree under reviewed generated-namespace paths.
3. The same generated bytes are passed through `Sandbox.SourcePolicy.scan/2`
   using the scanner's path-list form before backend resolution; the compiled
   generated paths and scanned generated paths must match by source hash.
4. The staged project includes root Mix files, `.formatter.exs`, `.credo.exs`,
   `.dialyzer_ignore.exs`, root config, app source, and reviewed source-tree
   plugin inputs needed by the real umbrella warning gate.
5. Gate runs compile-WAE, generated focused tests, Credo, Dialyzer, and
   selected security evals through v0.36 gate profiles. Generated tests are
   invoked with `focused_test_paths` / `security_eval_paths`, and any
   `focused_test_cwd` / `security_eval_cwd` stays inside the bundle project.
6. Passing gate moves the draft to `:gate_passed`.
7. Failure returns bounded diagnostics, can drive the Repair role while budgets
   remain, and cannot integrate.

Failure behavior:

- If generated modules exist only in the v0.36 `drafts` mount and are not on
  the staged project compile path, the gate is invalid.
- If generated source compiles from bytes that were not scanned, or scanned
  bytes differ from compiled bytes, the gate is invalid.
- A gate pass remains evidence only; it grants no action, route, app,
  settings, child-process, or module-loading authority.
- Generated tests are advisory functional evidence only. They do not replace the
  trusted AST/body validator, generated-permission ceiling, operator review, or
  Security Central confirmation.
- Repaired source/test packets create new draft revisions. Prior sandbox reports
  remain immutable evidence and cannot be overwritten by a later repair.

## Flow 4: Operator confirms live integration

Operator reviews a `:gate_passed` artifact and confirms integration.

Expected:

1. Security Central creates a confirmation record for integration. The
   confirmation is resolvable only from the high-trust operator surfaces named by
   `dynamic_codegen.integration_approval_surfaces` (CLI/LiveView by default);
   Telegram, email, and cross-channel approval are excluded regardless of
   `confirmations.allow_cross_channel_approval`.
   `IntegrateDraft`/`RollbackIntegration` must check the resolver/origin surface
   at approval time and deny disallowed surfaces even if the global confirmation
   subsystem would normally accept them.
2. The loader verifies the source hash and reconciles the manifest with the AST:
   every discovered `defmodule` is generated-namespace scoped and declared, and
   every manifest-declared module exists in reviewed source.
3. The loader parses reviewed generated source without executing it and applies
   the default-deny trusted-phase AST allowlist. Allowed top-level forms are
   limited to generated-namespace `defmodule` declarations, inert literal module
   attributes, normal function definitions, and tightly allowlisted `alias`/`use`
   targets.
4. The allowed generated macro/use set is explicit: `use AllbertAssist.Action`.
   Action DSL options, schema entries, tags, attributes, and manifest values
   must be inert literals; custom Phoenix components, LiveViews, HEEx sigils,
   routes, and `use AllbertAssist.App` are denied in the shipped v0.37 loader.
5. The loader scans generated action `run/2` and helper function call sites.
   Calls must be local generated calls or reviewed allowlisted module/function
   calls. For delegated writes, the only approved effect path is
   `AllbertAssist.DynamicPlugins.Delegate.run/3` with a binary string literal
   facade name in `dynamic_codegen.allowed_facades`.
6. The loader validates each generated action's declared permission,
   confirmation metadata, `resumable?` flag, response action metadata, and body
   call targets against the v0.37 generated-permission ceiling. For delegated
   writes, the generated permission must match the reviewed facade permission:
   `append_memory` maps to `:memory_write`; `external_network_request` maps to
   `:external_network`. Permission/body mismatch is denial.
7. The loader denies core/static module replacement, undeclared modules,
   protocols, router edits, application env mutation, migrations, dependencies,
   NIFs, ports, package-manager hooks, top-level side effects, `@on_load`,
   dynamic compile/eval/require, dynamic module construction, custom compile
   hooks, unapproved macros, `apply/3` escape routes, and direct calls to
   protected authority surfaces such as shell/process runners, package/skill
   execution, sandbox actions, Settings writes, secrets, confirmations,
   Resource grants, Repo writes, distributed Erlang, or trust-control actions.
8. Generated apps, panels, settings fragments, memory namespaces, objective
   wiring, route pages, and children are denied before core compilation.
9. The loader copies reviewed source and metadata to
   `<ALLBERT_HOME>/dynamic_plugins/integrated/<slug>/<revision>/`.
10. The loader recompiles reviewed source in core.
11. The loader registers action overlay entries through
   `AllbertAssist.DynamicPlugins.ActionsOverlay`.
12. Tier becomes `:integrated`; audit records compile/load/register.
13. Route-page artifacts are rejected or marked restart-required.

After integration, delegated writes are invoked like any other registered
dynamic action through `AllbertAssist.Actions.Runner.run/3`. The dynamic action
does not resume confirmations itself (`resumable?: false`). If the delegated
reviewed facade creates a confirmation, the existing facade-specific approval
and resume path owns that confirmation. Its approval surfaces are the facade's
normal surfaces, not the dynamic integration/rollback surface list.

Failure behavior:

- Ungated, unconfirmed, tampered, undeclared-module, route-page, or forbidden
  construct attempts are denied and audited.
- Unknown AST forms, macro options that are not literals, manifest/module
  mismatches, protected runtime calls, generated-permission ceiling violations,
  permission/body mismatches, and `resumable?: true` dynamic actions are denied.
- An integration approval that arrives from a disallowed or low-trust surface
  (Telegram, email, or cross-channel) is denied and audited.
- Any failure before stable `:integrated` auto-unwinds registrations and loaded
  modules from that attempt and leaves the artifact at
  `:gate_passed` with diagnostics; no operator rollback is required.
- Dynamic actions cannot shadow static/plugin/app/dynamic action names; a
  collision is denial with diagnostics.
- A new source hash cannot update an integrated artifact in place; it creates a
  new draft revision that must pass the gate and receive confirmation.
- A same-name revision cannot integrate while an older revision is live. v0.37
  requires rolling back the live revision first, then integrating the new
  revision; atomic supersede is deferred.

## Flow 5: Use integrated capability

Operator uses the new action.

Expected:

1. New actions resolve through the registry overlay and execute through
   `Actions.Runner.run/3`.
2. Generated app/panel surfaces are not live-loaded in v0.37.
3. Dynamic action behavior remains bounded by its validated body call targets
   and generated-permission ceiling. Static actions may self-authorize inside
   `run/2`; generated actions cannot rely on declaration alone.
4. Settings writes, Resource Access, confirmations, app-scope, and permissions
   behave like static reviewed code only when reached through the approved
   registered action/facade path for the generated action's validated
   permission.
5. Integrated dynamic actions appear in `Actions.Registry.capabilities/0`;
   only reviewed `exposure: :agent` actions appear in
   `agent_capabilities/0` and intent ranking.

## Flow 6: Restart or emergency-disable dynamic integration

The daemon restarts, or the operator disables the live loader.

Expected:

1. On boot, the loader reconciles only artifacts with `:integrated` metadata,
   matching source hash, present gate evidence, present operator confirmation
   validated against Security Central's durable confirmation/audit store, and
   current settings/policy that still allow live dynamic code.
2. If `dynamic_codegen.live_loader_enabled=false`, the loader does not
   register dynamic authority surfaces.
3. `disable_dynamic_live_loader` turns the setting false and clears dynamic
   action authority. Source files remain inspectable.
4. A failed reconciliation is denied and audited; it does not fall back to
   ordinary plugin discovery or untrusted core compile.
5. `metadata.yaml` may cache confirmation ids and hashes, but it is not a trust
   anchor by itself. Reconciliation remains bounded by local operator trust; it
   is not tamper-proof against someone who can rewrite Allbert Home and the
   Security Central store.

## Flow 7: Roll back integration

Operator rolls back a live integration.

Expected:

1. Security Central creates a confirmation record for rollback, resolvable only
   from the same high-trust operator surfaces as integration
   (`dynamic_codegen.integration_approval_surfaces`); low-trust channel and
   cross-channel approval are excluded.
2. Action overlay registrations are removed.
4. Loaded modules are purged/deleted where safe; this is best effort and
   audited.
5. Tier becomes `:rolled_back`.
6. Capability is no longer resolvable without a new gate pass and operator
   confirmation.

## Flow 8: Discard draft

Operator discards an untrusted or failed draft.

Expected:

1. A non-integrated or rolled-back draft is tombstoned or removed safely.
2. No compile path, route, permission, trust, or child process was ever active.
3. An `:integrated` artifact must be rolled back before discard.
4. `:discarded` is terminal; restoring requires a new draft revision.
5. The operator surface is `discard_dynamic_draft` or
   `mix allbert.dynamic drafts discard <slug>`.

## Data Flow

1. Gap → objective-owned proposal recorded as objective steps/`objective_events`
   on `Objectives.Engine` (propose/evaluate → authorize → execute → observe).
2. Codegen producer, future repair producer, or later template producer →
   inert draft metadata plus source-bearing draft code/config/tests when the
   target shape is implemented.
3. v0.36 sandbox → compile/trial/gate report.
4. Gate pass → `:gate_passed` evidence tier.
5. Operator confirmation → trust grant.
6. Loader → live registration + audit → `:integrated`.
7. Boot reconciliation or emergency disable → audited keep/remove decision.
8. Use through normal runner/app surfaces.
9. Rollback/discard → reversible cleanup.

Authority invariants: generation/trial grant nothing; the warning gate plus
operator confirmation are the only trust grants; advisory/agent output never
authorizes integration; `:integrated` dynamic actions remain bounded by the
generated-permission ceiling and validated runtime call targets.

## Security Evals (v0.28 harness rows)

- `codegen-core-load-untrusted-001`: untrusted artifact cannot be loaded in core.
- `codegen-gate-skip-001`: integration without `:gate_passed` is denied.
- `codegen-integration-unconfirmed-001`: integration without operator confirm is denied.
- `codegen-advisory-authority-001`: agent output cannot authorize integration.
- `codegen-loader-integrity-001`: tampered/provenance-mismatched source is rejected.
- `codegen-unscanned-compile-path-001`: generated source cannot compile unless
  the same bytes were statically scanned.
- `codegen-scanned-but-not-compiled-001`: scanned source outside the staged
  compile path cannot satisfy gate evidence for compiled modules.
- `codegen-low-confidence-autogen-001`: low intent confidence or rejected
  candidates cannot start generation without an explicit operator/objective
  proposal.
- `codegen-trusted-compile-side-effect-001`: `@on_load`, top-level side
  effects, dynamic compile/eval/require, and generated compile-time macro
  execution are rejected before trusted compile.
- `codegen-trusted-ast-allowlist-001`: trusted compile rejects any generated
  source outside the reviewed AST allowlist.
- `codegen-macro-literal-options-001`: generated macro/use options, attributes,
  schemas, tags, and manifest values must be inert literals.
- `codegen-manifest-defmodule-reconcile-001`: every generated `defmodule` is in
  the manifest and every manifest module is present in reviewed source.
- `codegen-generated-runtime-call-deny-001`: generated function bodies cannot
  call protected runtime authorities directly.
- `codegen-permission-ceiling-001`: generated actions cannot declare permission
  classes outside the v0.37 generated-permission ceiling.
- `codegen-permission-body-mismatch-001`: declared permission, response action
  metadata, and body call targets must agree.
- `codegen-delegated-write-001`: generated `:memory_write` and
  `:external_network` actions can effect work only by delegating to a matching
  reviewed facade.
- `codegen-facade-allowlist-001`: generated delegation to any facade outside
  `dynamic_codegen.allowed_facades` is denied.
- `codegen-facade-name-literal-001`: generated delegation requires a binary
  string literal facade name; variables, interpolation, atoms, and computed
  names are denied.
- `codegen-delegated-permission-match-001`: generated permission, response
  metadata, and delegated facade permission must match.
- `codegen-discard-draft-001`: non-integrated draft can be terminally
  discarded; integrated artifact requires rollback first.
- `codegen-request-permission-001`: `request_dynamic_draft` is controlled by
  `:dynamic_codegen_request`, not historical `:skill_write`.
- `codegen-delegated-memory-allow-001`: delegated `:memory_write` validates only
  when both generated permission and `append_memory` facade are enabled.
- `codegen-delegated-network-confirmation-001`: delegated
  `:external_network` creates the normal `external_network_request`
  confirmation through the facade.
- `codegen-delegated-runtime-facade-disabled-001`: disabling a facade after
  integration makes runtime `Delegate.run/3` fail closed.
- `codegen-delegated-rollback-authority-001`: rollback removes delegated action
  registry authority.
- `codegen-delegate-only-effects-001`: direct protected writes, network calls,
  confirmation decisions, Settings mutation, and trust-control calls remain
  denied even when a generated write permission is enabled.
- `codegen-generated-resumable-deny-001`: generated actions cannot register as
  resumable in v0.37.
- `codegen-dynamic-child-effect-deny-001`: generated children are denied in the
  v0.37 loader; future child callbacks with timers, process execution, network
  calls, or private durable loops remain denied.
- `codegen-undeclared-module-001`: undeclared modules are not loaded.
- `codegen-core-module-replace-001`: generated artifacts cannot replace core or
  shipped modules.
- `codegen-action-shadow-deny-001`: dynamic actions cannot shadow static,
  plugin, app, or other dynamic actions.
- `codegen-route-page-live-deny-001`: page routes cannot silently integrate live.
- `codegen-settings-fragment-authority-001`: generated settings fragments cannot
  set values, secrets, permissions, provider credentials, or resource grants.
- `codegen-private-objective-loop-001`: generated objective wiring cannot create
  private durable goal loops or treat `objective_id` as authority.
- `codegen-integration-partial-failure-001`: mid-integration failure
  auto-unwinds partial registrations and leaves the draft non-integrated.
- `codegen-revision-upgrade-live-collision-001`: same-name revisions cannot
  integrate over a live revision without rollback.
- `codegen-integration-approval-surface-001`: integration/rollback approval is
  accepted only from high-trust operator surfaces and denied from low-trust
  channel or cross-channel approval.
- `codegen-rollback-001`: rollback requires confirmation and removes live
  authority; module purge is best-effort/audited.
- `codegen-emergency-disable-001`: disabling the live loader removes or blocks
  dynamic authority surfaces.
- `codegen-restart-reconcile-001`: restart re-registers only valid integrated
  artifacts and fails closed on tamper or disabled settings.
- `codegen-v036-sandbox-bypass-001`: v0.37 cannot bypass the v0.36 sandbox.
- `codegen-exfil-001`: trial/gate/integration output is bounded and redacted.
- `codegen-generation-budget-001`: provider-call or usage-budget exhaustion stops
  generation/repair without integration.
- `codegen-critic-authority-001`: Critic acceptance cannot advance tiers,
  trust a draft, or authorize integration.

## Smoke Checklist

- Workflow is disabled by default.
- v0.37 is unavailable when v0.36 sandbox doctor is not green.
- v0.37 generation is unavailable when `dynamic_codegen.provider_profile` is
  unset, unresolved, disabled, or missing credentials.
- Provider-call and usage budgets cap generation and repair after the explicit
  generation step.
- Planner, Author, TrialAuthor, and Critic produce separate role packets; Repair
  produces packets when repair is requested. Only Author calling the LLM is
  insufficient for v0.37.2 acceptance.
- Trial/gate failures can drive bounded Repair, but repair never mutates prior
  revision evidence.
- Agent output cannot enable, trust, or integrate.
- Low-confidence intent ranking cannot start generation by itself.
- Generated files are compiled only from a staged project tree whose generated
  bytes were scanned and hash-matched, with real umbrella gate config included.
- Generated tests are functional evidence only, not the trusted security
  boundary.
- Trusted compile uses an AST allowlist, not the v0.36 regex scanner.
- Trusted validation is default-deny, reconciles manifest and `defmodule`s,
  requires literal macro/attribute options, scans generated runtime call targets,
  and enforces the generated-permission ceiling.
- Sandbox gate failure blocks integration.
- Operator-confirmed `:gate_passed` action artifact integrates live with no
  restart after generated permission and call-target validation.
- Generated actions cannot be resumable in v0.37, and dynamic child processes
  are not live-loaded.
- Mid-integration failure auto-unwinds and leaves the artifact non-integrated.
- Same-name revision upgrades require rollback before re-integration.
- New dynamic action is usable through the normal runner and registry surfaces.
- Read-only CLI/action status can show draft tier, gate report, integration
  registration state, and rollback state without opening the workspace UI.
- Rollback requires confirmation and removes live authority.
- Integration/rollback approval is accepted only from high-trust operator
  surfaces, not Telegram/email/cross-channel.
- Live-loader disablement removes or blocks dynamic authority surfaces.
- File-backed draft metadata stays under Allbert Home and is not confused with
  ordinary plugin discovery.
- Security evals fail closed.
