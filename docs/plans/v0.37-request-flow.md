# Allbert v0.37 Request Flow - Templated Creation

Status: planned.

Implementation details live in `docs/plans/v0.37-plan.md`.

v0.37 makes common creation patterns (plugin, app, LLM tool, future templated
code) first-class for developers (Mix tasks) and operators (a guided `/workspace`
Canvas flow). It reuses the v0.36 sandbox/warning-gate/loader and adds no new
integration authority.

## Two Output Modes

- **Developer scaffold (default, safe):** writes inert source under
  `./plugins/<name>/` for human review/compile/test; no live integration.
- **Operator templated creation:** runs the v0.36 gated path (sandbox trial +
  warning gate + operator confirmation + reversible loader) to integrate the
  templated capability live.

## Flow 1: Developer scaffolds a plugin

Developer runs `mix allbert.gen.plugin my_plugin`.

Expected:
1. A vetted plugin template renders deterministically to `./plugins/my_plugin/`
   (manifest, plugin module, skill root, diagnostics, validation docs).
2. Output is inert: no compile-path change, trust, permission, or route.
3. `mix allbert.validate_app my_plugin` (for app templates) passes on first run.

## Flow 2: Developer scaffolds an app

Developer runs `mix allbert.gen.app my_app`.

Expected:
1. The app template renders the full contract layers: app module, supervision
   wiring, sample `AllbertAssist.Action`, `SKILL.md`, panel surface, settings
   fragment stub, v0.34 destination metadata, intent descriptor stub, memory
   namespace stub, optional Ecto/objective/canvas stubs, theming/layout docs.
2. Output is inert and review-ready.

## Flow 3: Developer scaffolds an LLM tool

Developer runs `mix allbert.gen.tool my_tool`.

Expected:
1. The LLM-tool template renders a Jido.AI-backed tool/agent (model profile,
   prompt files, `AllbertAssist.Action` wrapper, advisory boundary), reusing the
   v0.25 pattern.
2. The tool is advisory and inert until reviewed and registered; no auto-enable.

## Flow 4: Operator creates from a template in the workspace

Operator opens the **Create** destination (`workspace:create`) in `/workspace`.

Expected:
1. The Canvas shows a template gallery (plugin, app, LLM tool, registered
   patterns).
2. Selecting a template shows a validated parameter form and a preview of the
   generated tree.
3. The operator chooses developer-scaffold output or live integration.
4. The surface is view/compose only; generation/integration runs through
   registered actions and Security Central.

## Flow 5: Operator live integration (reuses v0.36)

Operator chooses live integration for a templated artifact.

Expected:
1. The artifact runs the v0.36 sandbox trial and the full warning gate.
2. Integration requires explicit operator confirmation (the v0.36 trust grant);
   advisory output and validation results cannot authorize it.
3. On confirm, the v0.36 loader integrates it live without a restart; it is
   reversible via the v0.36 rollback.
4. Templated creation cannot bypass the v0.36 gate.

## Flow 6: Invalid template input is rejected

Operator/developer supplies bad parameters or a traversal-laden name.

Expected:
1. Parameter validation rejects bad input with bounded diagnostics.
2. Path traversal, non-allow-listed files, and oversize output are denied.
3. No partial/inert-but-dangerous artifact is written.

## Data Flow: Input → Output

1. **Select pattern + params** (Mix task or Canvas) → `RenderTemplate` resolves
   the vetted skeleton deterministically.
2. **Developer scaffold** → `ScaffoldTemplate` writes inert source to
   `./plugins/<name>/`; stop.
3. **Operator live integration** → `CreateFromTemplate` hands the artifact to
   the v0.36 sandbox trial → warning gate → operator confirm → loader →
   `:integrated`, audited and reversible.

Authority invariants: templates/params grant nothing; the developer-scaffold mode
never integrates; operator live integration is exactly the v0.36 gated path
(sandbox + gate + operator confirm); the Canvas surface is view/compose only.

## DOM Ids / CLI Handles

| Surface | Handle | Use |
|---|---|---|
| Mix tasks | `mix allbert.gen.{plugin,app,tool}`, `mix allbert.validate_app` | Developer scaffolding |
| Create destination | `workspace:create` / `#workspace-dest-create` | Operator template gallery |
| Template gallery / param form | Canvas panels | Pick + parameterize + preview |
| Live integration confirm | confirmation record (Security Central) | v0.36 trust grant |

## Security Evals (v0.28 harness rows)

- `template-param-injection-001`: malicious params cannot inject code/paths.
- `template-path-traversal-001`: output stays within the target root, allow-list only.
- `template-authority-bypass-001`: templates/params grant no trust/permission/route.
- `template-integration-gate-001`: operator live integration cannot bypass the v0.36 gate.
- `template-canvas-authority-001`: the Create surface cannot execute or integrate without a registered action + confirmation.

## Smoke Checklist

- `mix allbert.gen.plugin/app/tool` produce inert, review-ready trees;
  `validate_app` passes first run.
- The `/workspace` Create flow previews and validates templates.
- Developer scaffold never integrates; operator integration is gated + confirmed
  + reversible via v0.36.
- Invalid params/traversal are denied.
- Security evals fail closed.
