## M0 Locked Decisions

Each decision below is recorded in `v0.52-plan.md` §"M0 Locked Decisions"
with a one-paragraph rationale. These are chosen values, not open defaults.

1. `Approval.Handoff` module — new `AllbertAssist.Approval.Handoff`
   module owning `render/2`; separate from existing
   `Intent.ApprovalHandoff` (text formatter) and `Intent.Handoff` (v0.33
   app-intent handoff).
2. Discord transport — Gateway WebSocket through a reviewed BEAM
   bridge; single shard; no public URL. Interactions HTTP +
   sharding parked.
3. Slack transport — Socket Mode through App-Level Token; no public
   URL. Events API HTTP parked.
4. Settings ownership — plugin-owned `Settings.Fragment` for
   `channels.discord.*` and `channels.slack.*`, enabled by an M0 trusted
   channel-settings bridge because the existing settings schema reserves
   `channels.*`. Telegram + email settings-fragment migration parked as
   v0.52.x follow-on.
5. Slack workspace scope — single-workspace bot token at v0.52.
   Multi-workspace OAuth parked.
6. Transport implementation vehicle — locked by the M0 spike as raw `Req`
   REST plus direct `websockex` clients for Discord Gateway and Slack Socket
   Mode. Nostrum and `slack_elixir` are not part of the v0.52 baseline; any
   later change requires an explicit re-decision before M2/M3 consumes it.
7. Provider thread / Allbert conversation mapping — provider thread roots scope
   channel `session_id` continuity and reply routing, but do not become Allbert
   internal `thread_id` authority. Slack keys continuity by `team_id +
   channel_id + (thread_ts || message ts) + external_user_id`; Discord keys it
   by `guild_id_or_dm + channel_id_or_thread_channel_id + external_user_id`,
   with ordinary `message_reference` used only for reply routing.

8. Canonical thread model (ADR 0057) — the existing `conversation_threads.id`
   IS the canonical thread id; two durable join tables map it per channel:
   `thread_channel_refs` and `conversation_message_refs` (with `owner_scope`,
   `receiver_account_ref`, deterministic `provider_thread_key`, `direction`,
   and `part_id`) plus `cross_channel_identity_links`. v0.52 uses
   `owner_scope: "local"`; the field prevents a post-1.0 multi-tenant schema
   redo but grants no authority.
9. Threading capability descriptor (ADR 0057) — each channel declares
   `threading:` (`:native_threads | :reply_chain | :flat | :rich`);
   `Conversations.ChannelThread` drives reply placement + the degradation ladder;
   the registry rejects a missing/invalid declaration.
10. Cross-channel model (ADR 0057) — continuity within each channel + a unified
    read-only history view + an explicit `resume_thread_on_channel` action; no
    silent live mirroring; cross-channel ordering by Allbert ingest sequence.
11. Cross-channel identity links (ADR 0057) — explicit operator-configured links
    group already-mapped per-channel identities; never auto-merged.

This v0.52 cycle also adds the ADR 0056 `:channel_message_inbound` inbound
permission class (floor `:needs_confirmation`) and reads Discord/Slack free-text
only on @mention or DM (Discord uses the privileged `MESSAGE_CONTENT` intent).

## Main User/Operator Flows

- Operator configures Discord or Slack credentials (bot token, app
  token, application id, workspace team id) through `mix
  allbert.channels <provider> *` subcommands; all tokens stored as
  `secret://` references.
- Operator adds external identities to `identity_map` and adds
  guild/workspace/channel ids to allowlists; enables the channel via
  Settings.
- Operator runs `mix allbert.channels <provider> doctor` to verify the
  bot can authenticate against the provider; doctor returns an ADR
  0047-style redacted channel-doctor envelope.
- External user mentions Allbert in an allowlisted channel; the
  adapter resolves the external identity to a local `user_id`,
  submits through `Runtime.submit_user_input/1`, and renders the
  response in-channel or in-thread.
- For effectful actions, Allbert renders a confirmation through
  `Approval.Handoff.render/2` (selecting the highest-fidelity
  primitive the descriptor declares). The external user approves or
  denies; the callback resolves the pending through the standard
  Approval Handoff path.
- Cross-provider confirmation isolation holds: a Discord-origin pending
  cannot be approved via Slack and vice versa.
- An adapter crash takes its own session; the other adapter remains
  available (`:one_for_one` per v0.16).

## Provider Conversation / Thread Mapping

The request flow keeps these identifiers separate:

- `user_id`: local Allbert user resolved from provider identity mapping.
- `session_id`: channel session continuity key derived from provider/user/thread
  root metadata.
- `thread_id`: internal Allbert conversation-history id selected by
  `Conversations.resolve_thread/1` inside `Runtime.submit_user_input/1`.
- `provider_thread_ref`: redacted provider reply context used for dedupe,
  reply placement, and smoke evidence.
- `receiver_account_ref`: redacted Allbert-side configured receiver/install
  reference (Slack team binding, Discord bot/application + guild/DM scope,
  Telegram bot/chat, email mailbox, web workspace, CLI profile).
- `provider_thread_key`: deterministic bounded string/hash derived from the
  canonicalized provider thread ref; used for database uniqueness.

Adapters submit provider-thread metadata as request metadata and event
metadata, never as internal Allbert conversation authority. **ADR 0057 adds the
plan-accepted durable canonical mapping**: `thread_channel_refs`
(`owner_scope`, `canonical_thread_id`, `channel`, `receiver_account_ref`,
`provider_thread_key`, redacted `provider_thread_ref`) and
`conversation_message_refs` (`owner_scope`, `canonical_message_id`,
`canonical_thread_id`, `channel`, `receiver_account_ref`,
`provider_message_id`, `part_id`, `direction`), both built in M1 before any
adapter consumes them. `Conversations.ChannelThread` maps canonical ⇄ provider
refs, suppresses echoes (inbound matching a recorded `:out` ref is dropped), and
picks the reply-placement strategy from the channel's `threading:` capability
via the degradation ladder (`:native_threads` → `:reply_chain` → `:flat` →
unified-view). M6 retrofits Telegram/email/web/CLI onto this substrate; M7 adds
the unified history view + `resume_thread_on_channel`. Provider thread metadata
is never `thread_id` authority (ADR 0056); a channel may pass `thread_id` only
when `ChannelThread.lookup_thread/1` has returned a previously bound canonical
Allbert thread id.

## Data Flow

Channel messages flow through `Runtime.submit_user_input/1`:

1. Provider event arrives over Gateway WS (Discord) or Socket Mode WS
   (Slack); `Channels.<Provider>.Adapter` GenServer receives it.
2. `Channels.<Provider>.Parser` normalizes the payload, derives
   `receiver_account_ref`, redacted `provider_thread_ref`, and deterministic
   `provider_thread_key`, applies
   provider-payload redaction (bot tokens, signing secrets, full
   Authorization headers, full URLs with credentials are scrubbed),
   resolves the inbound channel id / guild id / workspace id.
3. Allowlist check: messages from non-allowlisted guilds/channels/
   workspaces produce a `channel_events` rejected record; no runtime
   submission.
4. Identity resolution: the external user id resolves via
   `channels.<provider>.identity_map` using the existing v0.16
   list-of-maps shape (`external_user_id`, `user_id`, `enabled`) to a local
   `user_id`. Unmapped identities produce a `channel_events`
   rejected-unknown record; no runtime submission.
5. Dedupe and echo suppression: `channel_events` unique-index check on the
   provider's external event id / message id; duplicates are recorded but not
   re-submitted. `ChannelThread.echo?/1` drops inbound provider messages that
   match a recorded outbound `conversation_message_refs` row.
6. Thread lookup: `ChannelThread.lookup_thread/1` checks
   `{owner_scope, channel, receiver_account_ref, provider_thread_key}`. If a row
   exists, it returns the canonical Allbert `thread_id`; if not, no
   `thread_id` is supplied to runtime.
7. Runtime submission: `Runtime.submit_user_input/1` with the resolved
   `user_id`, channel-id metadata, `provider_thread_ref`,
   `provider_thread_key`, and a stable `session_id` derived from the v0.16
   session rules plus the v0.52 provider thread root policy. The adapter does
   not set Allbert's internal `thread_id` from Slack `thread_ts`, Discord thread
   ids, or Discord `message_reference`; it only supplies a `thread_id` returned
   by `ChannelThread.lookup_thread/1`. Otherwise `Conversations.resolve_thread/1`
   remains the internal conversation authority and creates/selects the canonical
   thread.
8. Post-runtime linkage: `ChannelThread.link_thread/1` binds the response's
   canonical thread id to the provider key when no row already existed, and
   `record_message_ref/1` stores the inbound message ref.
9. Response rendering: `Channels.<Provider>.Renderer` produces text +
   chunks per `max_text_bytes`. If the response carries an
   `approval_handoff`, the renderer builds an effective descriptor from
   descriptor + settings (removing `:button` when
   `render_approval_buttons: false`) and calls
   `Approval.Handoff.render(handoff_payload, effective_descriptor)` to pick
   the primitive (`:button > :typed_command > :link > :list`; `:link`
   requires a workspace URL or falls through to `:list`). It then formats
   the provider-specific wire shape (Discord components / Slack Block Kit /
   typed-command text). The same call site is used by Telegram and email
   after the M0 retroactive refactor.
10. Outbound delivery: `Channels.<Provider>.Client` posts the reply
   via provider HTTP (Discord `POST /channels/{channel.id}/messages`;
   Slack `chat.postMessage`); threaded replies preserve provider thread
   identity (Discord thread channel / `message_reference`, Slack `thread_ts`)
   from `provider_thread_ref`. `ChannelThread.record_message_ref/1` records each
   native outbound id with `direction: :out` and `part_id`.
11. `channel_events` records the outbound delivery status (success,
   provider error, retry) and stores canonical `thread_id` only; provider ids
   remain in redacted metadata and ADR 0057 tables.
12. Callback events (Discord INTERACTION_CREATE, Slack `interactive`
    envelope) carry `custom_id` / `action_id` in the
    `allbert:v1:<verb>:<id>` format; the Parser validates the verb is
    in `{approve, deny, show}` and the id matches an open pending
    owned by the resolved local user and originating from this
    channel. Discord component interactions are acknowledged through the
    interaction callback endpoint with the interaction id/token. Slack
    Socket Mode envelopes are acknowledged by exact `envelope_id`.
    Cross-provider callbacks reject with a structured diagnostic.
13. The registered confirmation action resolves the pending through
    `Confirmations.resolve/4`; the outbound result re-renders through
    the same Renderer path.

## Authority Invariants

- **Channels are delivery adapters only.** They do not own runtime
  logic, confirmation authority, resource grants, app context, or
  durable objectives. Approval Handoff remains action-owned.
- **One ADR-introduced inbound permission class; no new operation/URI
  classes.** v0.52 adds `:channel_message_inbound` (ADR 0056, floor
  `:needs_confirmation`, registered at every `Security.Policy` /
  `Security.Risk` / `Settings.Schema` spot, cannot be lowered below the floor),
  the channel counterpart to ADR 0055. It records the trust tier of a resolved
  inbound call; it does not replace identity mapping + allowlist, which remain
  the authentication gate. No new operation classes or URI schemes.
- **Clicker re-authorized per interaction (ADR 0056).** A button/component tap
  re-resolves the clicker's external id through `Channels.Identity` on every
  tap; payload-embedded user fields are never authority. Unmapped/non-allowlisted
  clickers are rejected (ephemeral "not authorized" + rejected `channel_events`),
  and the confirmation action requires the resolved clicker to own the pending.
- **Ack before runtime.** The provider ack (Slack `envelope_id` / Discord
  deferred component response) precedes the runtime submission; `channel_events`
  dedupe absorbs Slack retries and Discord RESUME redelivery so nothing
  double-submits or double-approves.
- **Descriptors must declare primitives.** Channel registry rejects
  descriptors missing the `primitives:` field, declaring an empty
  list, declaring an unknown primitive, or omitting `:list` (universal
  fallback).
- **Descriptors must declare threading.** Channel registry rejects descriptors
  missing `threading:` or declaring a value outside
  `:native_threads | :reply_chain | :flat | :rich`. Renderers consume the
  descriptor through `Conversations.ChannelThread`; they do not hard-code a
  private provider ladder.
- **Trusted channel settings bridge only.** Source-tree channel plugins may
  contribute only their own `channels.<provider>.*` subtree through Settings
  Central. Untrusted/home plugin metadata and cross-provider namespace claims
  remain rejected.
- **Provider thread refs are routing metadata only.** Slack `thread_ts`,
  Discord thread-channel ids, and Discord `message_reference` values may scope
  channel `session_id` continuity and outbound reply placement, but they are
  never Allbert `thread_id` authority and never grant permission.
- **Canonical mapping is owner/account/key scoped.** `thread_channel_refs` and
  `conversation_message_refs` are unique by `owner_scope + channel +
  receiver_account_ref + provider_thread_key/message_id`. v0.52 always writes
  `owner_scope: "local"`; future multi-tenant ownership must use this field
  instead of adding a second canonical thread id or reusing provider ids.
- **`channel_events.thread_id` is canonical-only.** It stores
  `conversation_threads.id` when known. Provider ids stay in redacted
  `provider_thread_ref` metadata and ADR 0057 mapping rows.
- **Primitive selection is module-owned.** `Approval.Handoff.render/2`
  is the single boundary for primitive selection; renderers consume
  the selection, they do not make it.
- **Allowlist before runtime submission.** Guild/workspace/channel
  allowlists gate inbound handling before `submit_user_input/1`. No
  implicit channel admission.
- **Identity must be mapped.** Unmapped external identities produce
  rejected `channel_events`; no implicit `user_id` assignment, no
  fall-through to `"local"`.
- **Bot tokens are secret refs only.** Discord/Slack channel CLIs accept only
  `secret://` references; raw values never appear in settings output,
  `channel_events`, traces, or audits. Key names such as `bot_token_ref` and
  values such as `secret://...` are expected and are not leaks.
- **Provider-payload redaction.** Bot tokens, signing secrets,
  Authorization headers, and credential-bearing URLs scrubbed at the
  Parser stage; v0.16 redaction discipline applies identically to
  Discord and Slack.
- **No callback-data smuggling.** `allbert:v1:<verb>:<id>` is the
  only accepted format; Parser/callback guard rejects malformed callback
  data, unknown verbs, unknown ids, wrong-user ids, wrong-channel ids, and
  cross-channel callbacks before invoking the registered confirmation action.
- **Cross-provider confirmation isolation.** A pending originating in
  Discord cannot be resolved by Slack and vice versa (origin channel
  is recorded on the pending and re-checked at resolve).
- **DM vs workspace surface auth.** DM-routed messages are
  authenticated only when the external user has an `identity_map`
  entry. `response_style: dm_only` restricts the bot to never reply
  in shared channels.

  `AllbertDiscord.Settings.Fragment`)

  `AllbertSlack.Settings.Fragment`)

  detail for Discord/Slack doctor state and settings summaries)

### CLI surface

Shared:

```text
mix allbert.channels list
mix allbert.channels show discord
mix allbert.channels show slack
```

Discord-specific:

```text
mix allbert.channels discord set-token TOKEN_REF
mix allbert.channels discord set-application-id APPLICATION_ID
mix allbert.channels discord add-guild GUILD_ID
mix allbert.channels discord remove-guild GUILD_ID
mix allbert.channels discord add-channel CHANNEL_ID
mix allbert.channels discord remove-channel CHANNEL_ID

```

`set-token` accepts only a `secret://` reference and rejects raw token
strings.

Slack-specific:

```text
mix allbert.channels slack set-token TOKEN_REF

```

`set-token` and `set-app-token` accept only `secret://` references and reject
raw `xoxb-...` / `xapp-...` token strings.

### Cross-channel threading (ADR 0057)

- M1 implementation note: `Conversations.ChannelThread` owns normalized
  `owner_scope + channel + receiver_account_ref + provider_thread_key` refs,
  links canonical threads, records provider message refs, exposes `echo?/1`,
  and returns reply-target strategies from descriptor `threading:`. Runtime
  now accepts optional `channel_thread_ref` + `provider_message_id`, looks up an
  existing canonical thread only through `ChannelThread.lookup_thread/1`, links
  new provider roots after runtime thread resolution, and returns user +
  assistant conversation message ids for post-delivery outbound ref recording.
  M6 still owns full Telegram/email/web/CLI retrofit onto the substrate.
- `provider-thread-not-authority-001` (a provider thread ref never becomes an
  Allbert `thread_id` or grants permission)
- `owner-account-thread-key-isolation-001` (the same provider thread ids under
  different `receiver_account_ref` values do not collide; `owner_scope` is
  `"local"` and not a permission source)
- `echo-loop-suppression-001` (replaying Allbert's own outbound
  `provider_message_id` as inbound is dropped)
- `cross-channel-resume-same-user-001` (`resume_thread_on_channel` rejects a
  different `user_id`)
- `threading-capability-missing-rejected-001` (a descriptor missing/invalid
  `threading:` is rejected by the registry)
- `identity-link-no-auto-merge-001` (cross-channel identity links are never
  auto-derived)
- `unified-view-redaction-001` (the unified history view redacts via
  `Runtime.Redactor`)
