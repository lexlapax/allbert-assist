## M0 Locked Decisions

Each decision below is recorded in `v0.52-plan.md` §"M0 Locked Decisions"
with a one-paragraph rationale. These are chosen values, not open defaults.

1. `Approval.Handoff` module — new `AllbertAssist.Approval.Handoff`
   module owning `render/2`; separate from existing
   `Intent.ApprovalHandoff` (text formatter) and `Intent.Handoff` (v0.33
   app-intent handoff).
2. Discord transport — Gateway WebSocket through a reviewed BEAM
   bridge; single shard; no public URL. Interactions HTTP +
   sharding parked.
3. Slack transport — Socket Mode through App-Level Token; no public
   URL. Events API HTTP parked.
4. Settings ownership — plugin-owned `Settings.Fragment` for
   `channels.discord.*` and `channels.slack.*`, enabled by an M0 trusted
   channel-settings bridge because the existing settings schema reserves
   `channels.*`. Telegram + email migration parked as v0.52.x follow-on.
5. Slack workspace scope — single-workspace bot token at v0.52.
   Multi-workspace OAuth parked.

## Main User/Operator Flows

- Operator configures Discord or Slack credentials (bot token, app
  token, application id, workspace team id) through `mix
  allbert.channels <provider> *` subcommands; all tokens stored as
  `secret://` references.
- Operator adds external identities to `identity_map` and adds
  guild/workspace/channel ids to allowlists; enables the channel via
  Settings.
- Operator runs `mix allbert.channels <provider> doctor` to verify the
  bot can authenticate against the provider; doctor returns an ADR
  0047-style redacted channel-doctor envelope.
- External user mentions Allbert in an allowlisted channel; the
  adapter resolves the external identity to a local `user_id`,
  submits through `Runtime.submit_user_input/1`, and renders the
  response in-channel or in-thread.
- For effectful actions, Allbert renders a confirmation through
  `Approval.Handoff.render/2` (selecting the highest-fidelity
  primitive the descriptor declares). The external user approves or
  denies; the callback resolves the pending through the standard
  Approval Handoff path.
- Cross-provider confirmation isolation holds: a Discord-origin pending
  cannot be approved via Slack and vice versa.
- An adapter crash takes its own session; the other adapter remains
  available (`:one_for_one` per v0.16).

## Data Flow

Channel messages flow through `Runtime.submit_user_input/1`:

1. Provider event arrives over Gateway WS (Discord) or Socket Mode WS
   (Slack); `Channels.<Provider>.Adapter` GenServer receives it.
2. `Channels.<Provider>.Parser` normalizes the payload, applies
   provider-payload redaction (bot tokens, signing secrets, full
   Authorization headers, full URLs with credentials are scrubbed),
   resolves the inbound channel id / guild id / workspace id.
3. Allowlist check: messages from non-allowlisted guilds/channels/
   workspaces produce a `channel_events` rejected record; no runtime
   submission.
4. Identity resolution: the external user id resolves via
   `channels.<provider>.identity_map` using the existing v0.16
   list-of-maps shape (`external_user_id`, `user_id`, `enabled`) to a local
   `user_id`. Unmapped identities produce a `channel_events`
   rejected-unknown record; no runtime submission.
5. Dedupe: `channel_events` unique-index check on the provider's
   external event id / message id; duplicates are recorded but not
   re-submitted.
6. Runtime submission: `Runtime.submit_user_input/1` with the resolved
   `user_id`, channel-id metadata, and a stable `session_id` derived
   per the v0.16 session_id rules.
7. Response rendering: `Channels.<Provider>.Renderer` produces text +
   chunks per `max_text_bytes`. If the response carries an
   `approval_handoff`, the renderer builds an effective descriptor from
   descriptor + settings (removing `:button` when
   `render_approval_buttons: false`) and calls
   `Approval.Handoff.render(handoff_payload, effective_descriptor)` to pick
   the primitive (`:button > :typed_command > :link > :list`; `:link`
   requires a workspace URL or falls through to `:list`). It then formats
   the provider-specific wire shape (Discord components / Slack Block Kit /
   typed-command text). The same call site is used by Telegram and email
   after the M0 retroactive refactor.
8. Outbound delivery: `Channels.<Provider>.Client` posts the reply
   via provider HTTP (Discord `POST /channels/{channel.id}/messages`;
   Slack `chat.postMessage`); threaded replies preserve provider thread
   identity (Discord `message_reference`, Slack `thread_ts`).
9. `channel_events` records the outbound delivery status (success,
   provider error, retry).
10. Callback events (Discord INTERACTION_CREATE, Slack `interactive`
    envelope) carry `custom_id` / `action_id` in the
    `allbert:v1:<verb>:<id>` format; the Parser validates the verb is
    in `{approve, deny, show}` and the id matches an open pending
    owned by the resolved local user and originating from this
    channel. Discord component interactions are acknowledged through the
    interaction callback endpoint with the interaction id/token. Slack
    Socket Mode envelopes are acknowledged by exact `envelope_id`.
    Cross-provider callbacks reject with a structured diagnostic.
11. The registered confirmation action resolves the pending through
    `Confirmations.resolve/4`; the outbound result re-renders through
    the same Renderer path.

## Authority Invariants

- **Channels are delivery adapters only.** They do not own runtime
  logic, confirmation authority, resource grants, app context, or
  durable objectives. Approval Handoff remains action-owned.
- **No new permission, operation, or URI classes.** v0.52 introduces
  zero new authority vocabulary.
- **Descriptors must declare primitives.** Channel registry rejects
  descriptors missing the `primitives:` field, declaring an empty
  list, declaring an unknown primitive, or omitting `:list` (universal
  fallback).
- **Trusted channel settings bridge only.** Source-tree channel plugins may
  contribute only their own `channels.<provider>.*` subtree through Settings
  Central. Untrusted/home plugin metadata and cross-provider namespace claims
  remain rejected.
- **Primitive selection is module-owned.** `Approval.Handoff.render/2`
  is the single boundary for primitive selection; renderers consume
  the selection, they do not make it.
- **Allowlist before runtime submission.** Guild/workspace/channel
  allowlists gate inbound handling before `submit_user_input/1`. No
  implicit channel admission.
- **Identity must be mapped.** Unmapped external identities produce
  rejected `channel_events`; no implicit `user_id` assignment, no
  fall-through to `"local"`.
- **Bot tokens are secret refs only.** Discord/Slack channel CLIs accept only
  `secret://` references; raw values never appear in settings output,
  `channel_events`, traces, or audits. Key names such as `bot_token_ref` and
  values such as `secret://...` are expected and are not leaks.
- **Provider-payload redaction.** Bot tokens, signing secrets,
  Authorization headers, and credential-bearing URLs scrubbed at the
  Parser stage; v0.16 redaction discipline applies identically to
  Discord and Slack.
- **No callback-data smuggling.** `allbert:v1:<verb>:<id>` is the
  only accepted format; Parser/callback guard rejects malformed callback
  data, unknown verbs, unknown ids, wrong-user ids, wrong-channel ids, and
  cross-channel callbacks before invoking the registered confirmation action.
- **Cross-provider confirmation isolation.** A pending originating in
  Discord cannot be resolved by Slack and vice versa (origin channel
  is recorded on the pending and re-checked at resolve).
- **DM vs workspace surface auth.** DM-routed messages are
  authenticated only when the external user has an `identity_map`
  entry. `response_style: dm_only` restricts the bot to never reply
  in shared channels.

## Handles / CLI Surfaces

### Settings

- `channels.discord.*` (plugin-owned fragment via
  `AllbertDiscord.Settings.Fragment`)
- `channels.slack.*` (plugin-owned fragment via
  `AllbertSlack.Settings.Fragment`)

### Actions

- `list_channels` (existing, v0.16)
- `show_channel` (existing, v0.16; v0.52 extends generic plugin-channel
  detail for Discord/Slack doctor state and settings summaries)
- `discord_doctor` (v0.52 M1 — ADR 0047-style redacted channel-doctor envelope)
- `slack_doctor` (v0.52 M2 — ADR 0047-style redacted channel-doctor envelope)

### CLI surface

Shared:

```text
mix allbert.channels list
mix allbert.channels show discord
mix allbert.channels show slack
```

Discord-specific:

```text
mix allbert.channels discord set-token TOKEN_REF
mix allbert.channels discord set-application-id APPLICATION_ID
mix allbert.channels discord add-guild GUILD_ID
mix allbert.channels discord remove-guild GUILD_ID
mix allbert.channels discord add-channel CHANNEL_ID
mix allbert.channels discord remove-channel CHANNEL_ID
mix allbert.channels discord map --external-user EXTERNAL --user USER
mix allbert.channels discord unmap --external-user EXTERNAL
mix allbert.channels discord simulate --guild GUILD --channel CHANNEL --user EXTERNAL "prompt"
mix allbert.channels discord simulate-callback --user EXTERNAL --custom-id allbert:v1:<verb>:<id>
mix allbert.channels discord doctor
```

`set-token` accepts only a `secret://` reference and rejects raw token
strings.

Slack-specific:

```text
mix allbert.channels slack set-token TOKEN_REF
mix allbert.channels slack set-app-token APP_TOKEN_REF
mix allbert.channels slack set-team-id TEAM_ID
mix allbert.channels slack add-channel CHANNEL_ID
mix allbert.channels slack remove-channel CHANNEL_ID
mix allbert.channels slack map --external-user EXTERNAL --user USER
mix allbert.channels slack unmap --external-user EXTERNAL
mix allbert.channels slack simulate --channel CHANNEL --user EXTERNAL "prompt"
mix allbert.channels slack simulate-callback --user EXTERNAL --action-id allbert:v1:<verb>:<id>
mix allbert.channels slack doctor
```

`set-token` and `set-app-token` accept only `secret://` references and reject
raw `xoxb-...` / `xapp-...` token strings.
