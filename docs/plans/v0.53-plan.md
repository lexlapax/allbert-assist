# Allbert v0.53 Channel Pack 2 - WhatsApp, Signal, And Matrix

Request flow: `docs/plans/v0.53-request-flow.md`

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`, post-v0.37 planning pass.

## Purpose

Expand Allbert into mobile/personal messaging channels through the existing
v0.16 channel adapter boundary, v0.17 plugin contract, and the v0.52 ADR 0016
amendment for channel approval primitives plus ADR 0057 cross-channel
threading. Scope is tightened in the post-v0.37 planning pass:

- **iMessage parked** — macOS-only platform constraint warrants a dedicated
  platform-policy decision rather than 1.0-arc bundling. See
  `future-features.md` "iMessage Channel Adapter".
- **Public protocol interop** split into v0.51, including MCP server,
  OpenAI-compatible API, and ACP server.
- **Public AG-UI/A2UI bridge** remains parked post-1.0. See
  `future-features.md` "Public UI Protocol Interop Remainder".

## Expected Inputs

- v0.16 channel adapter boundary.
- v0.17 plugin contract and shipped channel plugins.
- v0.52 ADR 0016 amendment for the four approval primitives (`list`,
  `button`, `typed_command`, `link`).
- v0.52 ADR 0057 conversation-threading substrate:
  `Conversations.ChannelThread`, `thread_channel_refs`,
  `conversation_message_refs`, `cross_channel_identity_links`, per-channel
  `threading:` descriptors, `owner_scope: "local"`, `receiver_account_ref`, and
  deterministic `provider_thread_key`.
- Existing Settings Central secret handling and Approval Handoff.

## Operator-Visible Win

Allbert becomes reachable on common mobile/personal messaging surfaces:
WhatsApp Business or whatsapp-web bridge (operator-chosen), Signal via
signald/libsignal (operator-chosen), and Matrix (Element/Synapse-compatible).

## Planned Interfaces

- `./plugins/allbert.whatsapp/`
- `./plugins/allbert.signal/`
- `./plugins/allbert.matrix/`
- Provider-specific setup and pairing flows under Settings Central.
- Per-provider identity mapping into local `user_id`.
- Channel status surfaced through existing channel list/show paths.

## Channel Approval Primitives

Each adapter declares its supported primitive subset per the v0.52 ADR 0016
amendment:

- **WhatsApp**: `typed_command`, `link`. WhatsApp Business interactive
  buttons require API approval; default ships typed-command and link.
- **Signal**: `typed_command`, `link`. Signal protocol is text-first; no
  interactive button primitive.
- **Matrix**: `button` for bot-room contexts (where the bot has formatting
  permissions), `typed_command` and `link` otherwise.

`Approval.Handoff` picks the highest-fidelity primitive supported by the
adapter at delivery time.

## Channel Threading Capabilities

Each adapter consumes ADR 0057 instead of inventing a mobile-specific
conversation model:

- **WhatsApp**: `threading: :reply_chain` by default. Provider reply/quote ids
  and quote TTLs are stored through `ChannelThread`; expired quotes degrade to
  `:flat` rendering without changing the canonical Allbert thread.
- **Signal**: `threading: :reply_chain`. Quote timestamp/id metadata is routing
  metadata only and is scoped by `owner_scope + receiver_account_ref`.
- **Matrix**: `threading: :native_threads` when the room supports `m.thread`,
  with `:reply_chain` fallback for rooms/bridges that expose only replies.

All three use `conversation_threads.id` as the canonical id and inherit the
v0.52 rule that provider ids never grant authority.

## Security Posture

Each channel provider gets explicit Settings Central schema, identity mapping,
retry/dedupe policy, and pairing recovery docs. Channel metadata never grants
action permissions. All effectful work still routes through registered
actions, Security Central, confirmations, traces, and audits.

## Non-Goals

- No SMS.
- No iMessage (parked).
- No public protocol interop in this slot (see v0.51 for shipped public
  protocol surfaces; public AG-UI/A2UI remains parked in future-features).
- No hidden device/session pairing.
- No channel-specific confirmation model.
- No hosted multi-user authorization model.
- No cloud sync service.

## Definition Of Done

v0.53 is done when each planned channel has simulated inbound/reply coverage,
identity mapping is explicit, the v0.52 approval-primitive and ADR 0057
threading contracts are honored, and evals cover pairing/session replay, phone
identity confusion, mobile link handling, quote/thread degradation, and
provider-thread-not-authority.
