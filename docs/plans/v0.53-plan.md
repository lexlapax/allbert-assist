## Purpose

Expand Allbert into mobile/personal messaging channels through the existing v0.16
channel boundary, v0.17 plugin contract, the v0.52 ADR 0016 approval-primitive
amendment, and the v0.52 ADR 0057 threading substrate — **and finish the
system-wide constructs v0.52 declared but did not build**, because Matrix,
WhatsApp, and Signal force them:

- **Signal** is always end-to-end encrypted and integrates only by supervising an
  external `signal-cli` daemon that holds account keys on disk.
- **WhatsApp Cloud API** is a public signed webhook (server-readable plaintext to
  Allbert), not an outbound WS like Discord/Slack.
- **Matrix** is a pure HTTP `/sync` client; encrypted rooms are out of scope (no
  Elixir E2EE support), so v0.53 targets unencrypted Matrix rooms.

These break v0.52's assumptions (in-process WS clients, no E2EE, no phone PII), so
v0.53 adds: **Key Custody** (ADR 0058), **channel trust-class gating** (ADR 0059,
completing ADR 0057's E2EE-origin promise), a **public signed-webhook ingress**
(ADR 0056 amendment, reusing v0.51), **descriptor reply-key/quote-TTL
consumption**, and **phone-PII redaction**. Scope tightened per prior passes:

- **Viber** is documented on paper as a validated WhatsApp-twin (signed-webhook
  reuse, `:flat`, `:button`+`:link`, non-E2EE, opaque no-PII id) but **not built**
  in v0.53 — its ~€100/month standing bot fee makes it poor value to run. See
  `future-features.md` "Viber Channel Adapter".
- **iMessage parked** (macOS-only platform policy). **SMS parked.**
- **Signal kept in scope** via the `signal-cli` daemon (ADR 0058); `signald` is
  deprecated and "libsignal" is a library, not a runnable bot — neither is used.
- **Channel Pack 1 retro-validation is release-blocking.** v0.53 opens by
  bringing the already-shipped Telegram and email channels to Discord/Slack
  validation parity before new adapters land: live provider smokes, doctors,
  operator guides, manual checks, evidence, and leak scans.

## Key Custody (ADR 0058)

`AllbertAssist.Settings.KeyCustody` is a supervised GenServer that decrypts the
`secrets.yml.enc` envelope **once** (via the existing `Settings.Secrets`
master-key chain + `:crypto` AES-256-GCM) and serves secret material to in-VM
callers through `fetch/1`, replacing today's decrypt-on-every-read. Hardening
(achievable): `process_flag(:sensitive, true)` (crash-dump/observer hardening,
but not treated as a local `:sys.get_state` denial), masked `format_status/1`,
an `Inspect` that prints no secrets, secrets held as zero-arity closures,
`Plug.Crypto.prune_args_from_stacktrace/1`
+ `secure_compare/2`, and audited fetch. **Not claimed:** locked/zeroed/protected
memory or in-VM isolation between Allbert's own modules (the BEAM provides
neither; ADR 0058 honesty clause). `put_secret`/`delete_secret` invalidate the
cache. Zero new dependencies.

## Channel Trust-Class And Relay Gating (ADR 0059)

Every channel declares a `trust_class:` (`:e2ee_origin` | `:server_readable` |
`:local`); `ChannelThread` stamps it onto `thread_channel_refs` /
`conversation_message_refs` (new `trust_class` column, default
`:server_readable`). Signal = `:e2ee_origin`; WhatsApp/Viber/Telegram/Slack/
Discord/email = `:server_readable`; web/CLI = `:local`. The **unified history
view** excludes `:e2ee_origin` content from a *different* channel by default
(audited per-request/setting opt-in); **`resume_thread_on_channel`** onto a
weaker class requires an audited operator confirmation naming the downgrade.
v0.53 retrofits the web + CLI unified-view surfaces to honor `trust_class`. The
construct controls audited *exposure decisions*, not cryptographic guarantees
after E2EE termination (stated plainly in docs).

## Descriptor Flag Consumption + Phone Redaction

- **`reply_key_type` / `quote_ttl_ms` (declared in v0.52, now consumed):**
  `ChannelThread.resolve_reply_target/2` reads them. Signal uses
  `reply_key_type: :timestamp` (quote-by-timestamp+author); WhatsApp uses
  `quote_ttl_ms` (~30-day heuristic upper bound) to degrade `:reply_chain` →
  `:flat` when the quote window closes, without changing the canonical thread.
- **Phone-PII redaction:** `Security.Redactor` gains a phone-number rule
  (E.164) so WhatsApp/Signal `external_user_id` and any phone-bearing
  `receiver_account_ref` are redacted in `channel_events`, traces, audits, and
  the unified view. **Signal identity keys on ACI (UUID)**, not the rotatable
  phone number/PNI.

## Public Signed Webhook Ingress (ADR 0056 amendment)

WhatsApp Cloud API POSTs inbound events to a public HTTPS endpoint. **Reuse is
partial and the net-new plumbing is named honestly** (verified against the
shipped v0.51 ingress):

- **Reused as-is:** the v0.51 `RateLimiter` (its `{surface, client_id}` key is
  generic — buckets scope by provider install, e.g. `whatsapp_webhook` + WABA /
  `phone_number_id`), the `PublicProtocolHeaders` secure-header posture,
  `PublicProtocolBodyCap`, the router `:public_protocol_api` pipeline, and the
  `HttpIngress` error envelope.
- **Net-new (does not exist in v0.51):**
  1. **Raw-body preservation.** v0.51's `PublicProtocolBodyReader` delegates to
     `Plug.Conn.read_body/2`, which streams and discards the body — after
     `Plug.Parsers` the raw bytes are gone. M4 enhances the body reader (or adds a
     pre-parser step) to cache the raw bytes in `conn.private` **before**
     `Plug.Parsers` JSON decoding, so an HMAC can be computed over them.
  2. **A signature-auth plug** (`PublicProtocolWebhookAuth`), **distinct from the
     bearer-token `PublicProtocolAuth`** — the v0.51 token auth
     (`HttpIngress.authenticate` extracts `x-allbert-client-id` +
     `Authorization: Bearer`) is **not reused**. The webhook plug verifies
     `X-Hub-Signature-256` = `:crypto.mac(:hmac, :sha256, app_secret, raw_body)`
     (not the removed `:crypto.hmac/3`) with `Plug.Crypto.secure_compare/2`, the
     `app_secret` fetched via `Settings.KeyCustody`, then rate-limits on the
     install bucket.

The controller answers the `hub.challenge` verification handshake without
granting runtime authority. Absent/invalid signature → rejected before parse and
before runtime work. Self-host without a public URL uses a tunnel (documented).
The same construct is the (deferred) Viber twin's ingress with a Viber-specific
verifier (`X-Viber-Content-Signature`, keyed by the auth token).

## Channel Approval Primitives

Each adapter declares its supported subset per the v0.52 ADR 0016 amendment;
`Approval.Handoff` picks the highest-fidelity primitive at delivery time.
`Plugin.Validator` still rejects any channel descriptor that omits `:list`.

- **Matrix**: `typed_command`, `link`, `list`. No stable portable Matrix
  bot-button primitive is claimed for v0.53; bots render formatted HTML + links,
  with `:list` as the universal fallback.
- **WhatsApp**: `button`, `typed_command`, `link`, `list`. Interactive reply
  buttons need **no template approval** (only *templates* are reviewed); buttons
  work in the 24h customer-service session, with `typed_command`/`link`/`list`
  outside it.
- **Signal**: `typed_command`, `link`, `list`. Text-first; no interactive
  buttons.
- **Viber (deferred)**: `button`, `typed_command`, `link`, `list` (keyboard
  `reply` buttons round-trip the payload as a normal message).

## Security Posture

Each provider gets explicit Settings Central schema, identity mapping,
retry/dedupe, and pairing/recovery docs. Channel metadata never grants action
permissions; all effectful work routes through registered actions, Security
Central, confirmations, traces, audits. v0.53-specific invariants: the
`signal-cli` control endpoint is local-only (preferred `0600` UNIX socket, or
`127.0.0.1`-only TCP/HTTP with auth/ACL controls) and its key files `0600` (ADR 0058);
the WhatsApp webhook verifies the raw-body HMAC before parsing (ADR 0056
amendment); phone numbers are redacted everywhere; KeyCustody never leaks secret
material via inspect/`:sys.get_state`/crash dump; `:e2ee_origin` content is gated
out of the default cross-channel view and downgrade-on-resume is confirmed +
audited (ADR 0059).

## Definition Of Done

v0.53 is done when:
- Telegram and email reach Discord/Slack validation parity: provider doctors,
  provider-targeted outbound + inbound external smokes, operator guides, manual
  inbound/outbound/approval/rejection/poll-resume checks, recorded evidence, and
  clean raw-token/credential leak scans, with the M5R1/M5R2 remediations covered
  by deterministic `:v053` eval rows so they cannot silently regress;

  inbound/reply, map identity explicitly, and honor the v0.52 approval-primitive
  + ADR 0057 threading contracts;
- **KeyCustody (ADR 0058)** serves secrets from one `:sensitive` decrypt-once
  process (no decrypt-on-every-read), excluded from crash dumps/introspection,
  never logged, redacted on inspect — with **no** locked/zeroed-memory claim;
- the **`signal-cli` daemon** is supervised (muontrap/erlexec), its control
  endpoint is local-only (preferred `0600` UNIX socket; loopback TCP/HTTP only
  with auth/ACL controls), key files are `0600`, and pairing is audited;
- **trust-class gating (ADR 0059)** stamps every ref, excludes `:e2ee_origin`
  from the default cross-channel unified view (audited opt-in), and requires an
  audited confirmation to resume E2EE-origin content onto a weaker class; web +
  CLI honor it;
