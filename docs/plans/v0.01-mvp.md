# Allbert v0.1 — MVP Plan

## Context

[`docs/vision.md`](../vision.md) lays out the durable vision for Allbert: a personal AI assistant with a **small Rust kernel**, a markdown-defined bootstrap identity layer, markdown-based skills, markdown memory, built-in tools (process exec, input, skills), security checks at the kernel level, cost tracking, and toggleable tracing. RAG, scripting engines, training, cron jobs, multi-channel gateways, and self-recompile are deferred.

The rough original project note is preserved at [`docs/notes/origin-2026-04-17.md`](../notes/origin-2026-04-17.md). The user's explicit MVP guidance from that origin note:

> "Start off with we should probably have a simple agent [with] a few tools like the ability to call shell commands and gather input and ability to read skills and run them and [a] loop [that's] always listening for input. After that, we can have scheduled and cron jobs."

**Goal of v0.1:** ship a kernel that runs an agentic loop, owns bootstrap context / tool dispatch / skills / memory / security / cost / trace, and exposes a clean API that any frontend can drive. The REPL is the first frontend. Nothing in the kernel should know the REPL exists.

### Planning history
- **rev 1**: REPL-centric single-crate plan. Rejected.
- **rev 2** (user feedback): kernel-as-center + REPL as one frontend; two providers day-one (Anthropic + OpenRouter); streaming OFF.
- **rev 3** (openclaw architecture study): hook points in agent loop; session lanes stubbed; three-tier memory shape; skill capability fence; SKILL.md treated as untrusted prompt input; skill self-creation seam reserved.
- **rev 4** (tool-surface expansion; confirmed in-scope for v0.1): MVP tool set grew beyond the origin-note list. Added `InputPrompter` seam and `request_input` tool (aligned with the vision's "input gathering"), promoted `create_skill` from reserved seam to first-class tool, added clear-web `web_search` and `fetch_url` with a centralized `web_policy`, added `invoke_skill.args` for activation-time guidance, added `write_memory.summary` with `MEMORY.md` index deduplication, and reduced `HookOutcome` to `{Continue, Abort}`. See [ADR 0006](../adr/0006-hook-api-is-public-from-day-one.md) and [ADR 0009](../adr/0009-v0-1-tool-surface-expansion-and-policy-envelope.md).
- **rev 5** (bootstrap personality layer): added a runtime-owned bootstrap context bundle — `SOUL.md`, `USER.md`, `IDENTITY.md`, `TOOLS.md`, and optional `BOOTSTRAP.md` — loaded ahead of memory and skills on each turn. This layer is distinct from both long-term memory and `SKILL.md` activation. OpenClaw-style `AGENTS.md` / `HEARTBEAT.md` are intentionally deferred until Allbert has those product surfaces. See [ADR 0010](../adr/0010-bootstrap-personality-files-are-first-class-runtime-context.md).
- **rev 6** (closeout and source-based usability): M1-M6 completed the kernel feature milestones, but v0.1 is not considered wrapped yet. Closeout now adds guided onboarding, source-based user docs, shipped-state reconciliation, and dual-provider live smoke gates before the plan can move from `in_closeout` to `shipped`. - **rev 7** (ship closeout): guided setup, `/setup`, `/status`, README/operator docs, clean-home setup validation, trace validation, and live Anthropic/OpenRouter smoke all passed. v0.1 is now marked `shipped` for the intended technical source-based CLI audience.

Research caveat on the openclaw study: some citations returned by the research agent looked fabricated; only architectural patterns that are (a) consistent with the project vision, (b) standard in agent-runtime design, or (c) both were retained.

## Current usability assessment

Allbert is now wrapped for the intended v0.1 audience: a technical source-based CLI user. Guided setup, source-based quickstart docs, explicit workspace trust, and live Anthropic/OpenRouter smoke validation are all part of the shipped release.

## Kernel API surface (the seam)

```rust
// crates/allbert-kernel/src/lib.rs
pub struct Kernel { /* config, tool registry, skill store, memory store, llm, cost, security, hooks */ }

impl Kernel {
    pub async fn boot(config: Config, adapter: FrontendAdapter) -> Result<Self, KernelError>;

    // Drive one user turn to completion. Emits events via adapter.on_event.
    pub async fn run_turn(&mut self, user_input: &str) -> Result<TurnSummary, KernelError>;

    // Introspection.
    pub fn list_skills(&self) -> &[Skill];
    pub fn active_skills(&self) -> &[ActiveSkill];
    pub fn model(&self) -> &ModelConfig;
    pub fn provider_name(&self) -> &'static str;
    pub fn reset_session(&mut self);
    pub fn session_cost_usd(&self) -> f64;
    pub fn today_cost_usd(&self) -> Result<f64, KernelError>;
    pub fn session_id(&self) -> &str;
    pub async fn set_model(&mut self, model: ModelConfig) -> Result<(), KernelError>;

    // Hook registration — public from day one.
    pub fn register_hook(&mut self, point: HookPoint, hook: Arc<dyn Hook>);
}

// The frontend's side of the seam: events outbound + confirm/input inbound.
pub struct FrontendAdapter {
    pub on_event: Box<dyn Fn(&KernelEvent) + Send + Sync>,
    pub confirm: Arc<dyn ConfirmPrompter>,
    pub input: Arc<dyn InputPrompter>,
}

#[async_trait]
pub trait ConfirmPrompter: Send + Sync {
    async fn confirm(&self, req: ConfirmRequest) -> ConfirmDecision;
}
pub enum ConfirmDecision { Deny, AllowOnce, AllowSession }

#[async_trait]
pub trait InputPrompter: Send + Sync {
    async fn request_input(&self, req: InputRequest) -> InputResponse;
}
pub struct InputRequest { pub prompt: String, pub allow_empty: bool }
pub enum InputResponse { Submitted(String), Cancelled }

pub enum KernelEvent {
    AssistantText(String),
    ToolCall { name: String, input: serde_json::Value },
    ToolResult { name: String, ok: bool, content: String },
    Cost(CostEntry),
    TurnDone { hit_turn_limit: bool },
}

// Hook points — stable extension surface (openclaw-style).
pub enum HookPoint { BeforePrompt, BeforeModel, BeforeTool, AfterTool, OnModelResponse, OnTurnEnd }

#[async_trait]
pub trait Hook: Send + Sync {
    async fn call(&self, ctx: &mut HookCtx) -> HookOutcome;
}
pub enum HookOutcome { Continue, Abort(String) }
```

**Hook outcome semantics in v0.1:** hooks mutate `HookCtx` in place and then return `Continue` to let the stage proceed or `Abort(message)` to stop the current turn with a structured, model-visible error. v0.1 intentionally omits a generic `Skip`; if future work needs stage-specific short-circuit behavior, it should be added explicitly rather than overloading one catch-all outcome.

**v0.1 built-in hooks** (shipped as `pub` types; registered by default in `Kernel::boot`; external code may compose with or replace them):
- `BeforePrompt` → `BootstrapContextHook` (loads and snapshots `SOUL.md`, `USER.md`, `IDENTITY.md`, `TOOLS.md`, and optional `BOOTSTRAP.md` for the current turn).
- `BeforePrompt` → `MemoryIndexHook` (injects `MEMORY.md` head and today's daily note after bootstrap context).
- `BeforeTool` → `SecurityHook` (runs confirm prompt for `process_exec`, enforces sandbox for fs tools, enforces skill capability fence).
- `OnModelResponse` → `CostHook` (extracts usage, writes JSONL, emits `Cost` event).

**Why this shape:** kernel owns the loop; frontend owns I/O. Any v0.2 frontend (web, cron, message adapter) implements `ConfirmPrompter`, `InputPrompter`, and consumes `KernelEvent`. Hooks are the extension point the project vision wants — for policy, tracing probes, auto-skill-creation, cost caps, etc.

## Provider abstraction (two providers day-one)

```rust
// crates/allbert-kernel/src/llm/provider.rs
#[async_trait]
pub trait LlmProvider: Send + Sync {
    async fn complete(&self, req: CompletionRequest) -> Result<CompletionResponse, LlmError>;
    fn pricing(&self, model: &str) -> Option<Pricing>;   // for cost estimation
    fn provider_name(&self) -> &'static str;
}
```

- `llm/anthropic.rs` — direct Anthropic Messages API client; pricing table for Sonnet/Haiku/Opus.
- `llm/openrouter.rs` — direct OpenRouter Chat Completions API client; pricing table fetched best-effort at boot from OpenRouter's `/api/v1/models` and cached in memory (fallback to a hardcoded table for common models if the fetch fails; the kernel still boots and the provider still works).
- Config picks provider by enum; each provider has its own `api_key_env` and `model_id`.

```toml
# ~/.allbert/config.toml
[model]
provider     = "anthropic"         # or "openrouter"
model_id     = "claude-sonnet-4-5" # provider-specific id
api_key_env  = "ANTHROPIC_API_KEY" # or "OPENROUTER_API_KEY"
max_tokens   = 4096

[security.web]
allow_hosts  = []                  # exact host match; empty = rely on default SSRF guards
deny_hosts   = []                  # exact host match; wins over allow
timeout_s    = 15
```

At boot, kernel instantiates the right `LlmProvider` implementor based on `provider`. `/model` slash command (REPL frontend) lets you swap at runtime.

## Agent loop (inside the kernel)

```
Kernel::run_turn(user_input):
    state.messages.push(User(user_input))
    bootstrap = snapshot_bootstrap_files(~/.allbert/{SOUL,USER,IDENTITY,TOOLS,BOOTSTRAP})
    memory = load_prompt_memory(MEMORY.md head + today's daily)
    for turn in 0..limits.max_turns:
        sys = build_system_prompt(base, bootstrap, memory, skill_manifest, active_skill_bodies)
        resp = provider.complete(sys, messages, tool_schemas).await
        cost.record(resp.usage); hooks.emit(Cost(..))
        state.messages.push(Assistant{text, tool_calls})
        hooks.emit(AssistantText(text))
        if tool_calls.empty(): hooks.emit(TurnDone); return
        for call in tool_calls:
            if tool_call_budget_exhausted(): return tool_budget_error()
            hooks.emit(ToolCall{..})
            result = tools.dispatch(call, &mut ctx).await  // ctx carries ConfirmPrompter handle
            hooks.emit(ToolResult{..})
            state.messages.push(ToolResult(result))
    // limits.max_turns reached → emit TurnDone with hit_limit=true
```

Streaming OFF (confirmed). `dispatch` awaits `ConfirmPrompter::confirm` for `process_exec` when needed — the kernel does not know the prompt happens in a terminal.

## Runtime limits (v0.1)

- `limits.max_turns` bounds model/tool rounds per user turn.
- `limits.max_tool_calls_per_turn` bounds the total number of tool invocations before the turn is aborted with a structured error.
- `limits.max_tool_output_bytes_per_call` truncates each tool result before it is added to context.
- `limits.max_tool_output_bytes_total` caps cumulative tool output added to a single turn's context.
- `limits.max_bootstrap_file_bytes` caps any one bootstrap file before prompt injection.
- `limits.max_prompt_bootstrap_bytes` caps the total bootstrap bundle injected into the prompt for a turn.
- `limits.max_prompt_memory_bytes` caps how much memory content (`MEMORY.md` head + today's daily note) can be injected into the prompt.
- `limits.max_skill_args_bytes` caps the serialized `invoke_skill.args` payload stored per active skill; oversized args are rejected by the `invoke_skill` tool with a model-visible error to bound prompt bloat and injection surface.

The kernel should fail closed when a limit is exceeded: emit a tool/error event, add the error as model-visible context if there is remaining budget, and stop the turn if continuing would exceed another limit.

## Bootstrap context (identity/persona)

- Disk: `~/.allbert/{SOUL.md,USER.md,IDENTITY.md,TOOLS.md,BOOTSTRAP.md}`
- Creation: on first boot, the kernel seeds missing bootstrap files with minimal markdown templates so the runtime surface is immediately inspectable and editable.
- `SOUL.md`: core values, tone, boundaries, and behavioral stance.
- `USER.md`: the human's preferred name, timezone, ongoing projects, and stable working preferences.
- `IDENTITY.md`: presentation metadata such as assistant name and vibe.
- `TOOLS.md`: local environment notes and durable conventions that are not shared skill logic.
- `BOOTSTRAP.md`: optional one-time first-run ritual. It is injected only while present; completing the ritual normally deletes it.
- Prompt strategy: bootstrap files are loaded by the kernel at turn start, snapshotted for the duration of that user turn, and injected ahead of memory and skill context on every model round in that turn.
- Budgeting: bootstrap files are bounded separately from memory by `limits.max_bootstrap_file_bytes` and `limits.max_prompt_bootstrap_bytes`.
- Separation of concerns: bootstrap files are always-on runtime context; they are **not** skills, are not discovered through the skill registry, and are not a substitute for durable memory notes.
- Deliberate non-goal for v0.1: OpenClaw-style `AGENTS.md` and `HEARTBEAT.md` are not adopted yet. Their workspace norms, proactive heartbeat checklists, and group-surface behavior belong to later milestones once Allbert actually ships those surfaces.

Seeded default templates:

- `SOUL.md` starts with `Purpose`, `Values`, `Tone`, and `Boundaries` sections. It defines how Allbert should generally behave, not what to do on a particular task.
- `USER.md` starts with `Preferred name`, `Timezone`, `Working style`, and `Current priorities`. It records stable user preferences and durable context, not transient task notes.
- `IDENTITY.md` starts with `Name`, `Role`, and `Style`. It defines presentation and self-description, not memory or operational policy.
- `TOOLS.md` starts with `Environment notes` and `Editing rules`. It holds durable local conventions and environment facts that are too specific for the product vision but too stable for per-task prompts.
- `BOOTSTRAP.md` starts with a short first-run checklist that tells the agent to fill in the durable bootstrap files, ask only for missing essentials, and remove `BOOTSTRAP.md` after the bootstrap pass is complete.

Ownership and mutation rules:

- `SOUL.md` is user-owned. The agent may suggest edits, but must not rewrite its values, tone, or boundaries without explicit user intent and confirmation.
- `USER.md` is shared but user-led. The agent may only write facts or preferences the user explicitly provided or confirmed during bootstrap, and the write still follows the durable confirm path in v0.1.
- `IDENTITY.md` is user-owned by default. Changes to assistant name, role, or vibe require explicit user intent and confirmation.
- `TOOLS.md` is shared-maintenance. The agent may propose durable environment or workflow notes when they appear stable and useful, but every write remains a confirmed durable prompt-surface mutation in v0.1.
- `BOOTSTRAP.md` is kernel-seeded and ephemeral. The agent may delete it only after the user has finished or confirmed the bootstrap pass; otherwise it should remain present so the next turn keeps nudging setup forward.

## Tools (v0.1)

| Tool | Purpose | Security |
|---|---|---|
| `process_exec` | `{program, args?, cwd?, timeout_s?}` | Direct process spawn only; no shell parsing in v0.1. Confirm-before-run unless matched by `exec_allow`; `exec_deny` hard-blocks. Common shell/interpreter entrypoints (`sh`, `bash`, `zsh`, `fish`, `python`, `node`, etc.) are deny-by-default unless explicitly allowlisted. 30s timeout; output truncated to 8KB |
| `read_file` | `{path, max_bytes?}` | Canonicalize + must be under a `fs_roots` entry; UTF-8 only; 1MB cap |
| `write_file` | `{path, content, create_dirs?}` | Sandboxed; reject symlink escape; confirm overwrite |
| `request_input` | `{prompt, allow_empty?}` | Delegates to the frontend's `InputPrompter`; no extra kernel confirm; non-interactive frontends may return `Cancelled` |
| `list_skills` | `{}` | read-only |
| `invoke_skill` | `{name, args?}` | validates skill's `allowed-tools` ⊆ registry; activates the skill and stores optional invocation args for prompt rendering in the current session |
| `create_skill` | `{name, description, allowed_tools, body}` | Writes under `~/.allbert/skills/<name>/SKILL.md` via `SkillStore::create`; location is fixed to the skills root by kernel path construction and overwrite triggers the same confirm path as `write_file`. Not in the always-permitted fence set — active skills must declare `create_skill` in their `allowed-tools` to invoke it. New skills still pass through the activation gate before their body enters context. Emits log warning if body contains prompt-injection signatures. See [ADR 0009](../adr/0009-v0-1-tool-surface-expansion-and-policy-envelope.md). |
| `web_search` | `{query}` | Routed through `web_policy`; clear-web search (e.g., DuckDuckGo HTML) only. See [ADR 0009](../adr/0009-v0-1-tool-surface-expansion-and-policy-envelope.md). |
| `fetch_url` | `{url}` | Routed through `web_policy`: deny non-`http(s)`, `file://`, `data:`; deny RFC1918 / loopback / link-local / unique-local after DNS resolution; require resolvable host; timeout-bound; output capped by `limits.max_tool_output_bytes_per_call`; every URL logged at INFO. See [ADR 0009](../adr/0009-v0-1-tool-surface-expansion-and-policy-envelope.md). |
| `read_memory` | `{path}` | relative path only; sandboxed to `~/.allbert/memory/` |
| `write_memory` | `{path?, content, mode: write\|append\|daily, summary?}` | relative path only when present; sandboxed; `daily` appends to today's note and ignores `summary`; non-daily writes auto-update `MEMORY.md` index using `summary` when provided or a derived fallback on first insert |

## Skills system

- Disk: `~/.allbert/skills/<name>/SKILL.md`
- Frontmatter: `name`, `description`, `allowed-tools` (space-separated) — Claude-Code compatible subset. Unknown fields ignored fail-soft so future capability fields (`fs_paths_read`, `net_allow`) don't break old parsers.
- Startup: `SkillStore::discover()` walks dir, `gray_matter` → `Skill`, fail-soft on parse errors.
- Bootstrap files (`SOUL.md`, `USER.md`, `IDENTITY.md`, `TOOLS.md`, `BOOTSTRAP.md`) are a separate runtime layer and are governed by [ADR 0010](../adr/0010-bootstrap-personality-files-are-first-class-runtime-context.md), not by skill discovery or activation.
- **Prompt strategy:** manifest (name + description only) always in system prompt; full body injected only after explicit `invoke_skill` tool call. Keeps base context small **and** treats third-party SKILL.md as untrusted input that must be explicitly activated.
- `invoke_skill.args`, when present, is stored with the active skill and rendered immediately before that skill's body as a small `Invocation arguments (JSON)` block on later prompts. Re-invoking the same skill replaces the stored args for that skill in the current session. Args guide the model; they are not passed directly to tools and do not bypass policy.
- **Capability fence:** while a skill is "active," tool calls outside its `allowed-tools` list are rejected by the `SecurityHook` before reaching the tool. Multiple skills active = union of allowed-tools.
- **Important limitation:** `allowed-tools` is a coarse-grained narrowing mechanism, not a sandbox. It never expands privileges beyond global policy, does not bypass confirm prompts, and does not replace filesystem roots or exec deny rules.
- **Auto-creation:** `SkillStore::create(name, fm, body)` is exposed via the `create_skill` tool in v0.1, allowing the agent to persist new routines for future use.
- Log-only warning if SKILL.md body contains obvious prompt-injection patterns (`IGNORE ALL PREVIOUS`, base64 blobs, etc.). No blocking in MVP.

## Memory system (three-tier shape, two tiers implemented)

Directory layout (all tiers' directories created at boot so the shape is locked in):
```
~/.allbert/memory/
├── MEMORY.md              # tier 1: always-in-context index (first ~40 lines → system prompt)
├── daily/
│   └── 2026-04-17.md      # tier 2: dated recent notes (today auto-loaded into context)
├── topics/                # tier 3: searchable — empty in v0.1
├── people/                # tier 3
├── projects/              # tier 3
└── decisions/             # tier 3
```

- Tier 1 (`MEMORY.md`): human-editable index. For `mode: write|append`, `write_memory` ensures one `- [[path]] — summary` entry exists per note path. `summary` comes from `write_memory.summary` when provided; otherwise, on first insert, the kernel derives it from the first markdown heading or first non-empty line (truncated for readability). Later writes without `summary` preserve the existing index text.
- Tier 2 (`daily/`): `write_memory` with `mode: "daily"` appends to today's file.
- Tier 3: directories exist; no RAG/index; `read_memory` can reach them by relative path. Embeddings + index land in v0.2.
- Memory is distinct from the bootstrap persona layer. `SOUL.md` / `USER.md` / `IDENTITY.md` / `TOOLS.md` define always-on runtime stance and profile; `memory/` holds learned facts and durable notes.
- No RAG in v0.1. The model reads `MEMORY.md` head + today's daily from the system prompt and pulls specific files via `read_memory`.

## Security

Centralized in a `SecurityHook` registered on `HookPoint::BeforeTool`. Nothing else enforces policy — one choke point, easy to audit.

- `security::exec_policy(req) → Deny | AutoAllow | NeedsConfirm`, where `req` is a normalized `{program, args, cwd}` request, not a raw shell string.
- Confirm goes through `ConfirmPrompter` (frontend-supplied). REPL's impl renders `Allbert wants to run: <rendered command>\n[y/N/always]`.
- `AllowSession` caches the exact normalized exec request for the lifetime of the `Kernel`; v0.1 does not grant session trust based only on a program name.
- `security::sandbox::check(path, roots)`: canonicalize, `starts_with` one root, reject symlink escapes. Used by all fs/memory tools.
- **Default deny for general interpreters:** shell entrypoints and general-purpose interpreters are blocked by default in v0.1 unless explicitly allowlisted in config. The goal is to avoid sneaking shell semantics back in through `sh -c`, `python -c`, etc.
- **Bootstrap-file mutation is privileged:** edits to `SOUL.md`, `USER.md`, `IDENTITY.md`, `TOOLS.md`, or `BOOTSTRAP.md` are durable prompt-surface changes and must go through the same explicit confirm path as other sensitive writes, even though the files live under the normal Allbert root.
- **Skill capability fence:** when one or more skills are active, `SecurityHook` computes the union of their `allowed-tools`; tool calls outside that set → `HookOutcome::Abort("tool not permitted by active skill(s)")`. Built-in low-risk tools (`request_input`, `read_memory`, `invoke_skill`, `list_skills`) are always permitted. `create_skill`, `web_search`, and `fetch_url` are **not** always-permitted; skills must declare them explicitly.
- **Web policy:** `security::web_policy(url) → Deny | AutoAllow | NeedsConfirm`, enforced in `SecurityHook` for `web_search` and `fetch_url`. Deny non-`http(s)` schemes, `file://`, and `data:`; deny RFC1918, loopback, link-local, and unique-local ranges after DNS resolution; reject unresolvable hosts; apply per-request timeouts; cap output by `limits.max_tool_output_bytes_per_call`. Host allow/deny patterns live under a new `[security.web]` config block.
- **Skill creation sandboxing:** `create_skill` writes are confined to the skills root (`~/.allbert/skills/`) via the same `sandbox::check` used by fs tools; overwrites confirm just like `write_file`; body content is written verbatim but a log warning is emitted on prompt-injection signatures. Creation never bypasses the activation gate — a newly written skill's body enters prompt context only through an explicit `invoke_skill` call.
- `--yes` CLI flag on the REPL frontend flips `auto_confirm=true`.

## Cost tracking

- After every LLM call, append JSONL to `~/.allbert/costs.jsonl`:
  `{ts, session, provider, model, input_tokens, output_tokens, cache_read, cache_create, usd_estimate}`
- Per-provider pricing: Anthropic hardcoded table; OpenRouter fetched best-effort at boot from `/api/v1/models` with hardcoded fallback. Pricing metadata failure must not block boot or provider use.
- `/cost` slash command (REPL) sums today's entries.

## Tracing

- `trace.rs` in kernel exposes `init_tracing(cfg)`; REPL calls it at boot.
- Layered: stderr INFO always; file layer at DEBUG when `--trace` (to `~/.allbert/traces/<session>-<ts>.log`).
- `#[tracing::instrument]` on `Kernel::run_turn`, `Tool::call`, `LlmProvider::complete`.
- Runtime trace toggling is out of scope for v0.1; tracing is enabled at startup with `--trace`.

## Testing and verification

Verification for v0.1 is two-layered:

1. **Automated tests** cover deterministic kernel behavior and ship with each milestone.
2. **Manual smoke tests** cover end-to-end UX and live-provider integration where API keys, network, or terminal UX are involved.

Testing should bias toward the kernel, not the REPL. The REPL gets lightweight tests for adapter behavior and slash-command handling; policy, memory, skills, and the agent loop are primarily exercised against `allbert-kernel`.

**Test harness expectations**
- Use temporary directories for config, skills, memory, traces, and costs so tests never depend on the real `~/.allbert/`.
- Provide a fake `LlmProvider` for scripted model responses and usage metadata.
- Provide test doubles for `ConfirmPrompter` and `InputPrompter` so confirm/input flows are deterministic.
- Prefer black-box integration tests at the kernel seam (`boot()`, `run_turn()`, tool dispatch, hooks) for cross-module behavior; use unit tests for pure helpers like path checks, config parsing, and summary derivation.
- Live Anthropic/OpenRouter checks remain smoke tests, not required in CI, because they depend on credentials and external network behavior.

## Resolved decisions

Each bullet links to the ADR that carries its durable reasoning. The prose here is a short operational summary; the ADR is the source of truth.

1. **Skill body injection = on-demand via `invoke_skill`.** Manifest (name + description) always in system prompt; body only enters context after an explicit `invoke_skill` tool call. Keeps base context small and treats third-party SKILL.md bodies as untrusted input requiring explicit activation. See [ADR 0002](../adr/0002-skill-bodies-require-explicit-activation.md).

2. **Session lifetime = fresh per `cargo run`.** `AgentState.messages` lives in memory only; each invocation starts a new session with a new UUID. Durable state belongs in the memory system, not in hidden chat transcripts. Cost tracking and traces persist; memory persists; conversation does not. See [ADR 0003](../adr/0003-memory-files-are-durable-chat-history-is-not.md).

3. **"always" confirm scope = session-only exact request.** When the user picks `always` at a confirm prompt, the normalized exec request is added to an in-memory allow-list for the current `Kernel` lifetime. Never written back to `config.toml`. Exact-match scoping prevents bare-program-name approvals from covering unintended argument shapes; persistent trust goes through the deliberate path of editing `exec_allow`. See [ADR 0007](../adr/0007-session-scoped-exact-match-confirm-trust.md).

4. **OpenRouter pricing = best-effort live fetch at boot with hardcoded fallback.** On `Kernel::boot` with provider=OpenRouter, fetch `/api/v1/models` once, cache in memory. On fetch failure, fall back to a hardcoded table covering the most common models. Log a warning when the fallback is used, but do not fail boot or provider selection. Anthropic keeps a pure hardcoded table. See [ADR 0005](../adr/0005-support-anthropic-and-openrouter-in-v0-1.md).

5. **Hook API = public from day one.** `Kernel::register_hook(HookPoint, Arc<dyn Hook>)` is a public method. v0.1 ships with `BootstrapContextHook`, `MemoryIndexHook`, `SecurityHook`, and `CostHook` registered by `Kernel::boot`; external callers can append their own. The built-in hooks are also exposed as `pub` types so external code can compose with them. See [ADR 0006](../adr/0006-hook-api-is-public-from-day-one.md).

6. **Process execution = direct spawn, not shell-string evaluation.** v0.1's execution tool takes a normalized `{program, args, cwd}` request and spawns it directly. Shell parsing, pipes, redirects, and glob expansion are out of scope for the MVP. Direct spawn keeps the behavior auditable and the security model legible. See [ADR 0004](../adr/0004-process-exec-uses-direct-spawn-and-central-policy.md).

7. **Skill permissions are a fence, not a sandbox.** A skill's `allowed-tools` can only narrow the tool set available during its activation; it does not grant new privileges or relax global security policy. The fence framing avoids implying isolation the MVP cannot enforce. See [ADR 0008](../adr/0008-skill-allowed-tools-is-a-fence-not-a-sandbox.md).

8. **Kernel is the core; frontends are adapters.** The workspace split between `allbert-kernel` (library) and `allbert-cli` (binary), with an explicit `FrontendAdapter` / `ConfirmPrompter` seam, is a load-bearing architectural choice rather than a packaging detail. See [ADR 0001](../adr/0001-kernel-is-runtime-core-frontends-are-adapters.md).

9. **v0.1 tool surface is broader than the origin-note list, paired with an explicit policy envelope.** MVP ships `request_input`, `create_skill`, `web_search`, `fetch_url`, `invoke_skill.args`, and `write_memory.summary` in addition to the original exec/fs/memory/skills tools. Each addition has documented sandboxing, web policy, or byte limits so the central-policy claim stays intact. See [ADR 0009](../adr/0009-v0-1-tool-surface-expansion-and-policy-envelope.md).

10. **Bootstrap personality files are a first-class runtime context layer.** v0.1 treats `SOUL.md`, `USER.md`, `IDENTITY.md`, `TOOLS.md`, and optional `BOOTSTRAP.md` as kernel-owned prompt inputs loaded ahead of memory and skills. This layer is distinct from both task skills and durable memory notes; edits to it are durable prompt-surface changes. See [ADR 0010](../adr/0010-bootstrap-personality-files-are-first-class-runtime-context.md).

11. **v0.1 closeout targets a source-based CLI with guided onboarding and explicit workspace trust.** The v0.1 usability bar is a technical terminal user who can build from source, complete guided setup, and explicitly choose trusted roots; packaged installers and auto-trust are out of scope. See [ADR 0011](../adr/0011-v0-1-closeout-targets-a-source-based-cli-with-guided-onboarding-and-explicit-workspace-trust.md).
