# v0.11 — TUI and adaptive memory

Status: Proposed

## Context

v0.10 shipped the provider expansion and made fresh profiles local-first with Ollama/Gemma4. The next operator-facing gap is not self-improvement yet; it is visibility and memory confidence during normal use.

The current REPL is reliable, but it is intentionally plain: it streams assistant text and local slash-command output without a persistent view of model posture, context pressure, token usage, spend, memory state, active skills, or pending approvals. At the same time, the curated-memory system from v0.5 has the right safety model, but it is too passive: the `memory-curator` skill is available, but the runtime does not make memory curation feel like an always-nearby part of the assistant's working loop.

v0.11 therefore ships two related improvements before self-improvement:

1. A TUI operator surface that makes Allbert's runtime state legible while preserving the existing classic REPL fallback.
2. Adaptive memory routing and recall enhancements that keep Allbert markdown-first and review-first while making memory more reliably available.

This moves the previously planned self-improvement release to v0.12. That sequencing is deliberate: before Allbert authors code and patches itself, the operator should have a better terminal surface, stronger telemetry, and a clearer memory posture.

## Research checkpoint

| Topic | Decision |
| --- | --- |
| Codex-style TUI | Use a split between daemon/protocol event handling and terminal rendering. The TUI is an adapter over existing daemon sessions, not a second runtime. |
| Claude Code-style status line | Add native session telemetry and configurable status-line items. v0.11 does not run arbitrary status-line shell commands; status data comes from the daemon protocol. |
| MemPalace | Add searchable session-episode recall for verbatim or near-verbatim past working context, but do not promote chat history into durable memory. |
| Zep / Graphiti | Add provenance-aware temporal fact metadata and supersession semantics for staged/promoted facts. Keep review gates before durable promotion. |
| Mem0 graph memory | Treat entity/relation extraction as a structured overlay on top of markdown memory, not as the source of truth. |
| Letta memory model | Preserve a small always-visible memory synopsis and searchable deeper memory; improve routing rather than stuffing every memory surface into every prompt. |

## Goal

Ship an end-user-ready release with:

1. A Ratatui/Crossterm TUI as the default interactive surface for fresh profiles.
2. The existing Reedline REPL preserved as `classic` fallback and upgrade default.
3. Kernel-owned telemetry exposed through the daemon protocol.
4. Configurable status-line items for model, context, tokens, cost, memory, intent, skills, inbox, channel, and trace.
5. Configurable memory routing where `memory-curator` is always eligible but not always active.
6. Searchable session-episode recall that never bypasses the durable-memory staging/promotion contract.
7. Temporal fact metadata for staged/promoted fact memory.
8. Optional semantic/hybrid retrieval behind an off-by-default config flag.
9. Operator docs and upgrade notes that make the new defaults and fallback path clear.

## Non-goals

- No self-modifying Rust rebuild skill in v0.11; that is v0.12.
- No skill-authoring skill in v0.11; that is v0.12.
- No embedded Lua scripting engine in v0.11; that is v0.12.
- No arbitrary shell command status-line renderer.
- No hosted sync, hosted web UI, or cross-network daemon.
- No vendored third-party memory system.
- No hidden durable transcript memory.
- No automatic promotion of staged facts into approved durable memory.
- No semantic retrieval enabled by default.

## Design decisions

### TUI posture

- `repl.ui = "tui"` is the fresh-profile default.
- Existing profiles that lack the key migrate to `repl.ui = "classic"` to avoid surprising operators on upgrade. Migration is gated on the stored `setup.version`: any profile whose setup predates v0.11 writes `classic` explicitly on first load; fresh setups write `tui`.
- `allbert-cli` with no subcommand launches the configured UI.
- `allbert-cli repl --classic` and `repl.ui = "classic"` keep the current Reedline behavior available.
- `allbert-cli repl --tui` is the opt-in switch for existing profiles and does not mutate config unless `--save` is passed.
- The TUI attaches to the daemon via the same local IPC and session model as the current REPL and replays a bounded tail of the session journal on attach so the rendered transcript matches daemon-held state.
- The TUI never boots a private kernel.
- The TUI renderer owns layout, focus, input editing, modals, transcript display, and status-line rendering.
- The daemon/kernel own session state, tools, memory, cost, model usage, approvals, and telemetry.
- Keyboard is primary; `repl.tui.mouse = true` enables selection and scroll-wheel handling only.
- If the terminal refuses alt-screen or raw mode, the CLI falls back to classic REPL with a one-line notice rather than failing the launch.

### Telemetry posture

Telemetry is runtime state, not terminal decoration. v0.11 adds a protocol payload that can be consumed by the TUI, classic REPL slash commands, tests, and future frontends.

The telemetry payload includes:

- session id, channel, turn state, and current agent stack
- provider, model id, API-key state, max output tokens, and optional known context window
- latest model usage and cumulative session usage
- latest context percentage when a context window is known
- session cost, today's cost, daily cap, and active turn budget
- memory synopsis bytes, ephemeral bytes, prefetch hit count, staged count, indexed durable docs, episode count, and fact count
- active skills, always-eligible skills, and last resolved intent
- pending inbox count and trace state

Context percentages use provider-reported usage from the latest response only. v0.11 does not ship tokenizer-estimated preflight context accounting.

`SessionStatus` stays as a concise backward-compatible surface for `/status` and existing clients; `TelemetrySnapshot` is additive and is the canonical source for `/telemetry`, the TUI status line, and any future status-aware frontend. Fields shared with `SessionStatus` are sourced once in the kernel so they cannot drift.

### Status-line posture

The TUI status line is native and configurable:

```toml
[repl.tui.status_line]
enabled = true
items = ["model", "context", "tokens", "cost", "memory", "intent", "skills", "inbox", "channel", "trace"]
```

Supported items:

- `model`: provider/model
- `context`: latest input+output tokens over known context window, or `ctx ?` when unknown
- `tokens`: latest and session cumulative tokens
- `cost`: session and today spend
- `memory`: durable/staged/episode/fact counts and synopsis pressure
- `intent`: last resolved intent
- `skills`: active and always-eligible skill count
- `inbox`: pending approval count
- `channel`: attached channel/session
- `trace`: trace on/off

The `/statusline` slash command changes the active session view immediately and can persist changes to config when called with `--save`. In classic REPL mode the command mutates the same session-scoped view that `/telemetry` renders; the list itself is unchanged, only which items are shown in the rendered snapshot.

The status-line item catalog is closed in v0.11. Unknown names fail fast in two places: at config load (config validation refuses the profile) and at slash-command runtime (the command returns a local error without contacting the daemon). Operator-defined or shell-backed items are a non-goal.

### Memory routing posture

The new default is `always_eligible`, not `always_active`.

```toml
[memory.routing]
mode = "always_eligible"
always_eligible_skills = ["memory-curator"]
auto_activate_intents = ["memory_query"]
auto_activate_cues = ["remember", "what do you remember", "review staged", "promote that", "forget"]
```

Rules:

- Always-eligible skills are surfaced in a dedicated prompt stage inserted between the catalog-of-skill-names stage and the full skill-manifest/body stages. The stage names the skill and a one-line description only; bodies stay unloaded.
- Their full skill bodies are not loaded unless routing policy activates them.
- `memory-curator` auto-activates for the `MemoryQuery` intent (config string `memory_query`, matching the existing intent enum's serde rename) and explicit memory-review cues.
- Cues are a supplemental trigger: if the existing kernel intent classifier already resolved `MemoryQuery`, activation happens through the intent path; cues only activate when the classifier falls back to `Chat` but the utterance contains a configured token. The rule-based classifier's existing memory keywords remain authoritative and are not duplicated into config defaults unless an operator opts in.
- Operators can change routing through TOML, setup, or `allbert-cli memory routing show|set`. A successful `set` invocation issues `ClientMessage::ReloadSessionConfig` so attached sessions observe the new policy at the next root turn without restart.
- Runtime-generated `~/.allbert/AGENTS.md` remains generated product state; it gains a short "Routing" section that summarises active always-eligible skills and auto-activate triggers. The section is regenerated on skill lifecycle events and on `reload-config`. The config remains the source of truth; AGENTS.md is a derived view.

### Episode recall posture

Session journals are indexed as `episode` recall material. This lets Allbert answer questions like "what did we decide in that debugging session?" without pretending transcripts are durable learned memory.

Rules:

- Episode recall is derived from `~/.allbert/sessions/*/turns.md`. Sessions moved to `sessions_trash` or `sessions_archive` (the existing daemon forget/archive targets) are skipped during reconciliation; v0.11 does not ship incremental delete-by-query.
- Episode docs are produced per turn. Granularity is one Tantivy document per user turn and one per assistant turn; tool-output spans are already truncated in the journal to `memory.max_journal_tool_output_bytes` and the indexer reuses that truncation so episode snippets stay bounded.
- Episode docs carry `session_id`, `turn_id`, `role`, `timestamp`, `channel`, and `source_path`. Adding these fields and the new `Episode`/`Fact` tier values bumps `INDEX_SCHEMA_VERSION`. First launch after upgrade triggers the existing version-mismatch rebuild path (ADR 0045); the rebuild is idempotent from markdown.
- Episode search is explicit (`tier = "episode"`) unless `memory.episodes.prefetch_enabled = true`.
- The default auto-prefetch path excludes episodes.
- Episode hits are labelled as working history (`source.kind = "session_working_history"`) in search output, not approved memory.
- Any learning extracted from an episode still goes to staging before durable promotion.

### Temporal fact posture

Facts are a structured overlay over staged/promoted markdown memory:

```yaml
facts:
  - subject: "project storage"
    predicate: "uses"
    object: "Postgres"
    valid_from: "2026-04-24T00:00:00Z"
    valid_until: null
    source:
      kind: "staged_memory"
      id: "stg_..."
    supersedes: []
```

Rules:

- Fact metadata can appear on staged entries and promoted durable notes. `StageFrontmatter` parsing is extended to accept optional `facts`, and any unknown/forward-compatible keys are preserved through promotion (the promotion path must not silently drop unrecognised frontmatter).
- Fact metadata is indexed as one Tantivy document per fact, linked back to its parent note by `source.id`. `tier = "fact"` queries filter by `source.kind == "durable_memory"` so that staged facts never appear as approved facts.
- Facts do not become durable merely by being extracted.
- The `max_facts_per_entry` cap is enforced at staging time: over-cap entries fail staging with a clear error and are not partially accepted.
- Supersession at promotion does not rewrite prior durable notes. A new durable fact with `supersedes: [old_fact_id]` causes the derived fact index to filter the superseded entries from "current facts" views while keeping them queryable via an explicit `include_superseded` flag.
- Superseded facts remain auditable and are filtered from default fact summaries unless explicitly requested.
- The markdown file remains the source of truth.

### Semantic retrieval posture

BM25/Tantivy remains the default retrieval engine. v0.11 may add optional semantic retrieval as a derived index:

```toml
[memory.semantic]
enabled = false
provider = "none"
embedding_model = ""
hybrid_weight = 0.35
```

Rules:

- When disabled, no embedding calls are made and no vector index is required.
- When enabled, embedding calls route through the v0.10 provider seam (ADR 0066). Candidate adapters are Ollama, OpenAI, and Gemini; a fake deterministic provider is used for provider-free validation per ADR 0064.
- When enabled, semantic hits are fused with BM25 using a deterministic weighted-sum over normalised scores: `score = bm25_norm * (1 - hybrid_weight) + semantic_norm * hybrid_weight`. Normalisation and tie-breaking rules are fixed so the ordering is reproducible across runs.
- The semantic index lives under `memory/index/semantic/<provider>/<model>/` so multiple providers or models can coexist without colliding; its schema version is independent of the BM25 index schema version.
- The semantic index is derived and rebuildable from markdown ground truth.
- Enabling a hosted embedding provider sends note bodies to that provider. v0.11 exposes semantic configuration through TOML and setup only (no dedicated CLI sub-verb); setup and the operator docs must flag this privacy implication explicitly before any embedding call is issued.

### Daemon protocol

Add:

- `ClientMessage::SessionTelemetry`
- `ServerMessage::SessionTelemetry(TelemetrySnapshot)`

`TelemetrySnapshot` is additive. Existing `SessionStatus` remains for backward-compatible status rendering.

### Memory search

Extend `search_memory` tiers:

- `durable`
- `staging`
- `episode`
- `fact`
- `all`

Auto-prefetch remains conservative:

- durable tier only by default
- staging never auto-injected
- episode excluded unless explicitly enabled
- fact summaries can join durable prefetch only when `memory.facts.enabled = true` and the fact source is approved durable memory

## Implementation prerequisites

The milestones below depend on a small number of kernel, protocol, and config seams that do not exist in the v0.10 code. Each milestone lists its concrete work, but the seams are grouped here so reviewers can confirm the plan is implementation-ready end to end.

### Kernel seams

- `AgentState::last_response_usage: Option<Usage>` — populated after each provider response, reset at turn start; source for context-percentage rendering. The `Usage` type (input/output/cache tokens) already exists on the provider trait.
- `AgentState` memory counters: accessor helpers for durable doc count, staged count, episode count, fact count, synopsis bytes, ephemeral bytes. Counts are derived on demand from `CuratedMemory` so they cannot drift from markdown ground truth.
- `Kernel::memory_status()` — returns the counters above plus index health.
- `Kernel::session_telemetry()` — composer that assembles `TelemetrySnapshot` from existing accessors (cost, provider, model, intent, agent stack, active skills) plus the new memory counters and last-response usage.
- Always-eligible skill surface: a new `SkillStore::always_eligible_prompt()` or equivalent that emits a short bullet list for the prompt stage between catalog-names and active-skill bodies. It must not load skill bodies.
- `SkillStore::render_agents_markdown()` extended with a "Routing" section driven by `config.memory.routing`.
- Intent → auto-activation wiring: on `BeforeIntent`/`AfterIntent` completion, the kernel consults `memory.routing.auto_activate_intents` and `auto_activate_cues` and activates configured skills for the current turn only.
- Memory search: `MemoryTier` enum gains `Episode` and `Fact` variants; `search_memory` filters accordingly. Tantivy schema adds `session_id`, `turn_id`, `role`, `timestamp`, `channel`, `source_path`, and fact fields (`subject`, `predicate`, `object`, `valid_from`, `valid_until`, `supersedes`). `INDEX_SCHEMA_VERSION` bumps; reconcile rebuilds from markdown on first launch after upgrade.
- Episode indexer: `reconcile_curated_memory` scans `~/.allbert/sessions/*/turns.md` (excluding `sessions_trash` and `sessions_archive`) and produces per-turn episode documents capped by the existing journal truncation rules.
- Staged frontmatter: `StageFrontmatter` extended with optional `facts: Vec<FactMeta>` and a preserve-unknown strategy so promotion does not drop forward-compatible keys.

### Protocol seams

- `ClientMessage::SessionTelemetry` and `ServerMessage::SessionTelemetry(TelemetrySnapshot)` — handler mirrors the existing `SessionStatus` path in the daemon.
- `TelemetrySnapshot` type in `allbert-proto` with the fields listed under "Telemetry posture". Additive; `SessionStatus` is unchanged.

### Config seams

- `ReplConfig` gains `ui: ReplUiMode` (`tui` | `classic`) and nested `tui: TuiConfig` (`enabled`, `mouse`, `max_transcript_events`) and `tui.status_line: StatusLineConfig` (`enabled`, `items: Vec<StatusItem>`).
- `ModelConfig` gains `context_window_tokens: u32` (0 = unknown). Known provider defaults are documented but not auto-populated to keep config authoritative.
- `MemoryConfig` gains nested `routing`, `episodes`, `facts`, and `semantic` sub-configs. Validation rejects unknown status-line items, routing modes, intent names outside the kernel intent enum, `hybrid_weight` outside `[0.0, 1.0]`, and negative caps.
- Migration: profiles whose `setup.version` predates v0.11 serialise `repl.ui = "classic"` on first load; fresh setups write `tui`.

### CLI seams

- `allbert-cli memory` subcommand tree gains `routing show`, `routing set`, and `stats`. These follow the existing precedent of `allbert-cli daemon channels {add|remove}`: mutating commands call `Config::persist()` and issue `ReloadSessionConfig` for attached sessions.
- `allbert-cli repl --tui` / `--classic` switches the current invocation without mutating config unless `--save` is passed.
- `allbert-cli telemetry --json` prints a `TelemetrySnapshot` acquired via the new protocol pair.

Everything above is strictly additive: `SessionStatus`, existing memory tiers, existing `[memory]` keys, the Reedline REPL, and current skill activation semantics all remain working and untouched on the "classic" path.

- Bump `INDEX_SCHEMA_VERSION`. First launch after upgrade triggers the existing version-mismatch rebuild (ADR 0045); the rebuild reads only markdown ground truth, so it is idempotent and safe to run repeatedly.
- Extend the Tantivy schema with `session_id`, `turn_id`, `role`, `timestamp`, `channel`, and `source_path` fields. Add `Episode` (and `Fact`) variants to `MemoryTier`.
- Extend the memory indexer to add `episode` documents derived from session journals under `~/.allbert/sessions/*/turns.md`, skipping `sessions_trash` and `sessions_archive`. One document per user turn and one per assistant turn; tool-output spans inherit the journal's existing truncation to `memory.max_journal_tool_output_bytes`.
- Add `search_memory(tier = "episode")` and `tier = "all"` merging behaviour for episodes.
- Add `allbert-cli memory stats` reporting durable, staged, episode, and fact counts plus index schema version and last rebuild time.
- Keep default prefetch durable-only; episodes are never injected unless `memory.episodes.prefetch_enabled = true`.
- Episode hits surface `source.kind = "session_working_history"` so UI and tools can label them distinctly from approved durable memory.

Tests:

- Session journal fixture indexes into `episode` tier.
- Episode search finds prior session text.
- Durable prefetch excludes episodes by default.
- Deleting/forgetting a session removes derived episode docs on rebuild.
- Episode hits are clearly labeled as session working history.

### M7 — Temporal fact overlay

Scope:

- Extend `StageFrontmatter` parsing to tolerate and validate optional `facts`. Switch the frontmatter model to preserve unknown/forward-compatible keys so promotion does not silently drop them.
- Enforce `memory.facts.max_facts_per_entry` at staging time; reject over-cap entries with a clear error rather than truncating.
- Preserve fact metadata on promotion. The promotion path copies `facts` into the durable note frontmatter untouched.
- Index approved durable facts into `tier = "fact"` as one Tantivy document per fact, linked to its parent note by `source.id`. Fact docs carry `subject`, `predicate`, `object`, `valid_from`, `valid_until`, `supersedes`, and `source.kind`.
- Add supersession metadata and current-fact filtering helpers. A new durable fact with `supersedes: [id]` does not rewrite prior notes; the filter simply excludes superseded entries from default fact summaries. Queries with `include_superseded = true` see the full history.
- Render fact counts in telemetry and memory stats.

Tests:

- Promotion preserves fact metadata.
- Fact search returns approved durable facts.
- Staged facts are not returned by durable/fact auto-prefetch before promotion.
- Superseded facts remain searchable only when explicitly requested.

### M8 — Optional semantic retrieval

Scope:

- Add semantic config (already covered in M1) and a fake deterministic embedding provider for provider-free tests (ADR 0064).
- Extend the v0.10 provider seam (ADR 0066) with an `EmbeddingProvider` surface; ship at minimum the fake provider in-tree. Real adapters (Ollama, OpenAI, Gemini) can be added here or in a follow-up without reshaping the seam.
- Add derived semantic index metadata under `memory/index/semantic/<provider>/<model>/` with its own schema version independent of the BM25 index version.
- Implement deterministic hybrid fusion when semantic retrieval is enabled: weighted sum over normalised scores, `score = bm25_norm * (1 - hybrid_weight) + semantic_norm * hybrid_weight`, with a fixed tie-breaker (doc id) so ordering is reproducible.
- Keep the disabled path dependency-light at runtime. The semantic crate(s) gate behind a cargo feature so operators who never enable semantic do not pull the heavier dependency tree.
- Document provider requirements and privacy posture. Setup and operator docs must flag that enabling a hosted embedding provider sends note bodies to that provider.

- Semantic index can be deleted and rebuilt from markdown.
- BM25-only search output remains unchanged when semantic is disabled.

## End-user release plan

### Upgrade behavior

- Existing profiles keep classic REPL unless the operator opts into TUI. Migration is gated on `setup.version`.
- Fresh profiles default to TUI.
- Operators can switch with setup, config, `allbert-cli repl --tui`, or `allbert-cli repl --classic`.
- Memory routing defaults are safe and non-destructive; `always_eligible` does not load skill bodies.
- Episode/fact indexes are derived and rebuildable. First launch after upgrade rebuilds the curated-memory index once because `INDEX_SCHEMA_VERSION` bumps; the rebuild is idempotent and reads only markdown ground truth.
- Semantic retrieval is disabled unless explicitly configured.

### Operator messaging

Release notes must explain:

- "The terminal UI is new; classic REPL remains available."
- "Status-line data is local daemon telemetry, not a shell hook."
- "`memory-curator` is always eligible by default, not always loaded into every prompt."
- "Session episodes are searchable working history, not durable memory."
- "Temporal facts still go through staged review before durable promotion."
- "Semantic retrieval is opt-in."

### Acceptance checklist

- Fresh setup can complete without hosted provider keys.
- Upgrading a v0.10 profile retains classic REPL and triggers exactly one memory-index rebuild on first launch.
- `allbert-cli` opens the configured UI.
- `allbert-cli repl --classic` and `allbert-cli repl --tui` work as session-scoped overrides without mutating config.
- TUI renders model, cost, token, memory, and inbox status without a live provider.
- `/status` (concise, legacy) and `/telemetry` (verbose, new) work in both TUI and classic REPL with identical data.
- Memory routing can be shown and changed without editing code; attached sessions pick up the new routing on the next root turn via `ReloadSessionConfig`.
- Episode/fact search is explicit and clearly labeled as session working history or approved fact.
- Semantic retrieval stays disabled by default; enabling it with the fake provider produces stable hybrid ordering in tests.
- Docs and upgrade notes are end-user ready and name the first-launch rebuild explicitly.

## Handoff to v0.12

v0.12 depends on the following v0.11 surfaces staying stable across that release cycle. Changes after v0.11 ships must treat them as public API:

- `TelemetrySnapshot` shape in `allbert-proto`. v0.12's patch-review UX and rebuild status surfaces consume it.
- The always-eligible routing stage and the `SkillStore::always_eligible_prompt()` contract. v0.12's skill-author and rust-rebuild skills are expected to participate in the same routing mechanism.
- `MemoryTier::{Episode, Fact}` variants and their `source.kind` labels. v0.12's self-improvement docs refer back to episode recall when narrating session context for patch review.
- `Kernel::memory_status()` counters. v0.12 surfaces them in self-improvement tooling.
- The status-line catalog. v0.12 may add items for patch-approval counts, scripting budgets, and skill-author drafts; those are additive and must not reshape existing items.

v0.12's M0 already promises "recheck this plan against the shipped v0.11 telemetry/TUI surfaces"; this section names what that recheck covers.
