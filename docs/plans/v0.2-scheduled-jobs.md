# Allbert v0.2 â€” Scheduled Jobs and Maintenance

## Context

[`docs/vision.md`](../vision.md) says that after the current foundation is solid, Allbert can grow into "scheduled jobs, broader integrations, richer memory retrieval, self-generated skills, and additional frontends" [docs/vision.md](../vision.md). The original project note is even more explicit:

> "Start off with we should probably have a simple agent [with] a few tools like the ability to call shell commands and gather input and ability to read skills and run them and [a] loop [that's] always listening for input. After that, we can have scheduled and cron jobs."  
> â€” [docs/notes/origin-2026-04-17.md](../notes/origin-2026-04-17.md)

v0.1 shipped the simple assistant foundation:

- kernel-owned agent loop
- bootstrap identity files
- tools
- skills
- durable markdown memory
- security checks
- cost tracking
- tracing
- guided setup and release docs

That means the next target should follow the original sequencing instead of jumping ahead to vector retrieval, scripting engines, retraining, or a new GUI surface.

## Goal of v0.2

Ship a **local scheduled-jobs layer** that reuses the existing kernel and makes Allbert useful when the user is not actively in the REPL.

This target should support:

- recurring maintenance work for memory and traces
- recurring prompt-based jobs for technical users
- explicit, auditable local scheduling
- non-interactive execution with fail-closed safety behavior

The target is still deliberately local-first and source-based. v0.2 is not a hosted scheduler, a daemon-heavy automation platform, or a packaged desktop app.

## Why This Next

This is the cleanest next step for three reasons:

1. It is the next thing the original note asked for after the simple agent loop.
2. It expands capability without bloating the kernel: jobs can be a new frontend over the existing runtime seam.
3. It directly unlocks the memory direction in the vision by giving Allbert a place to do maintenance, pruning, summaries, and compilation later.

## Product Shape

v0.2 adds a new workspace crate:

```text
crates/
  allbert-kernel/
  allbert-cli/
  allbert-jobs/    # new: non-interactive scheduled-jobs frontend
```

`allbert-jobs` should:

- load job definitions from `~/.allbert/jobs/`
- execute one named job or all due jobs
- expose lifecycle commands to create, list, update, pause, resume, run, remove, and inspect job status
- reuse the kernel rather than reimplementing agent behavior
- emit structured job-run history and cost information
- run each scheduled execution in a fresh non-interactive session by default

The scheduler trigger itself should remain external in v0.2:

- macOS `launchd`
- Linux `cron` and `systemd` timers
- Windows Task Scheduler

Allbert should not grow a resident always-running daemon in this version.

The important compatibility promise is at the **command interface** level: `allbert-jobs run <job-id>` and `allbert-jobs run-due` should be stable and documented so they can be scheduled cleanly on macOS, Linux, and Windows. Convenience installer helpers for those schedulers can still come later.

## OS Scheduler Support Matrix

| OS family | Scheduler surface in v0.2 | v0.2 expectation | Explicitly deferred |
| --- | --- | --- | --- |
| macOS | `launchd` | documented invocation path and manual smoke coverage | installer/helper generation |
| Linux | `cron` and `systemd` timers | documented invocation path and manual smoke coverage | installer/helper generation |
| Windows | Task Scheduler | documented invocation path and manual smoke coverage | installer/helper generation |

The compatibility target is therefore **cross-platform command-interface support in v0.2**, not "best effort on Unix first." The portable surface is the `allbert-jobs` CLI contract; the per-OS scheduler wiring remains user-visible and documented in this release.

## Scheduled Jobs vs Future Heartbeat

OpenClaw's docs are useful here because they separate exact scheduled work from ambient periodic awareness. Allbert should do the same.

In v0.2:

- scheduled jobs are for exact, auditable, non-interactive work
- jobs run in fresh sessions by default
- jobs are the right fit for recurring reports, maintenance, and bounded background chores

Deferred beyond v0.2:

- heartbeat-style "ambient check-in" behavior
- approximate periodic awareness loops
- main-session wakeups that inherit open chat context

That boundary keeps v0.2 smaller and more auditable. It also avoids mixing precise automation with a later, looser "check what needs attention" model.

## Job Model (sketch)

Each job definition should be a small TOML file under `~/.allbert/jobs/`.

Likely fields:

- `id`
- `description`
- `enabled`
- `schedule`
- `prompt`
- optional `model` override
- optional `working_dir`
- optional ordered `skills`
- optional `report_policy`
- optional `output_path`
- optional `history_retention`
- policy settings for non-interactive execution

Illustrative shape:

```toml
id = "daily-review"
description = "Summarize recent notes and prepare a daily review."
enabled = false
schedule = "daily@09:00"
prompt = """
Review recent daily notes and produce a concise daily summary.
"""
working_dir = "/absolute/path/to/workspace"
skills = ["daily-review"]
report_policy = "always"
output_path = "/absolute/path/to/output/daily-review.md"

[policy]
allow_exec = false
allow_web = false
```

The exact schedule syntax can stay small in v0.2. This is a technical-user release, so it does not need a universal automation grammar on day one.

Jobs should be self-contained. A scheduled run gets a fresh session and should not depend on ambient chat state. If a job needs reusable procedure, it should attach one or more ordered skills rather than assuming prior conversation context.

## Job Lifecycle

v0.2 should treat jobs as durable objects with an explicit lifecycle, not just one-off config files.

The planned command surface should include:

- `create`
- `list`
- `update`
- `pause`
- `resume`
- `run`
- `remove`
- `status`

The jobs frontend should also expose local run history so users can see what ran, when it ran, what it cost, and why a run failed or stopped.

## Runtime Semantics

Scheduled jobs are **non-interactive**.

That means:

- `ConfirmPrompter` cannot assume a live human is present.
- `InputPrompter` cannot assume a live human is present.
- jobs must fail closed when they need missing human input or risky confirmation that has not been explicitly preconfigured.

The job frontend should therefore supply a non-interactive adapter implementation with explicit behavior:

- confirm requests deny by default unless the job policy already allows the action
- `request_input` returns cancelled / unavailable
- jobs record why they stopped instead of hanging indefinitely
- scheduled runs cannot create, edit, pause, resume, or remove other scheduled jobs while executing
- report policy should be explicit, with support for at least `always`, `on_failure`, and `on_anomaly`

This keeps the kernel boundary intact and keeps automation behavior auditable.

## First Shipped Scope

v0.2 should be **maintenance-first**, not "general automation platform first."

The first bundled jobs should be things the vision and original note already imply:

- memory review / compaction
- daily summary or daily carry-forward
- trace review / failure triage

On top of that, v0.2 should ship one generic scheduled prompt job path so technical users can automate bounded recurring work without waiting for a larger integration layer.

## Bundled Default Job Templates

v0.2 should ship a small set of **bundled job templates** under the Allbert home directory or installable examples. These should be present but **disabled by default** so the user opts into automation explicitly.

The initial bundled set should be:

- `daily-brief`
  - morning summary of recent notes, unfinished items, and suggested next actions
- `weekly-review`
  - weekly review of progress, repeated blockers, unresolved work, and likely follow-ups
- `memory-compile`
  - promote stable facts from recent notes into `MEMORY.md` and deduplicate existing entries conservatively
- `trace-triage`
  - review recent failures or suspicious traces and write a short anomaly report
- `system-health-check`
  - verify provider env vars, bootstrap files, trusted roots, and recurring job health

Default reporting expectations:

- `daily-brief` and `weekly-review` report on success
- `memory-compile` reports when it made changes or hit a problem
- `trace-triage` and `system-health-check` default to anomaly-first reporting

These bundled jobs should prefer first-party skills or well-bounded built-in prompts rather than giant inline prompt blobs duplicated across files.

## In Scope

- `allbert-jobs` frontend crate
- job definitions under `~/.allbert/jobs/`
- durable lifecycle commands for job management
- run-one and run-due execution paths
- non-interactive adapter behavior
- OS-scheduler-driven invocation model
- job-run history and cost visibility
- fresh-session scheduled execution
- skill-backed jobs
- bundled disabled job templates
- docs for macOS `launchd`, Linux `cron` / `systemd`, and Windows Task Scheduler usage at a technical-user level

## Out of Scope

- vector RAG / embeddings
- resident internal scheduler daemon
- heartbeat-style background awareness loop
- embedded scripting engines
- retraining or personality-model pipelines
- packaged installers
- platform-specific scheduler installer helpers
- web frontend
- broad third-party integration marketplace

## Milestones

**M1 â€” Job model + workspace skeleton.** Add `allbert-jobs` as a third crate, define the job-definition shape, lifecycle model, skill attachment fields, job discovery, due-job selection, and a small run-history format. *Verify:* automated tests cover TOML parsing, schedule parsing, lifecycle-state parsing, and due-job selection against frozen timestamps.

**M2 â€” Non-interactive frontend adapter.** Implement a jobs adapter over `allbert-kernel` that reuses the same kernel loop but supplies non-interactive confirm/input semantics. *Verify:* automated tests prove jobs fail closed on confirm/input and emit structured stop reasons rather than hanging.

**M3 â€” OS-scheduler-driven execution path.** Add `allbert-jobs create`, `update`, `run <job-id>`, `run-due`, `list`, `pause`, `resume`, `remove`, and `status`, plus state/history updates and docs for macOS `launchd`, Linux `cron` / `systemd`, and Windows Task Scheduler invocation. *Verify:* automated tests cover state advancement, create/update parsing, pause/resume behavior, and repeated due-run behavior; manual smoke shows an OS-level scheduled invocation can run `run-due`, and docs cover one supported scheduler path on each OS family.

**M4 â€” First-party maintenance jobs.** Ship the bundled disabled templates for `daily-brief`, `weekly-review`, `memory-compile`, `trace-triage`, and `system-health-check`, plus any first-party skills they depend on. *Verify:* fake-provider tests cover the bundled jobs' expected file outputs, report-policy behavior, and fail-closed behavior.

**M5 â€” Generic recurring prompt jobs + release docs.** Add a generic prompt-job path for technical users, document common policy patterns, and run end-to-end local smoke on the supported OS-scheduler interfaces. *Verify:* clean-home job setup works, job history is readable, costs are recorded, recurring jobs respect non-interactive safety policy, and the documented macOS/Linux/Windows scheduler entrypoints match the shipped command interface.
