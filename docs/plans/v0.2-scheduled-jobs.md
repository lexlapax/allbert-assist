# Allbert v0.2 — Daemon Host, Channels, and Internal Job Manager

## Context

[`docs/vision.md`](../vision.md) now points toward a daemon-capable micro-kernel that can host lightweight internal services and accept attachable user-facing channels. The original project note still matters because it identified scheduled jobs as the next meaningful value unlock after the simple assistant loop, but the runtime shape now needs to be spelled out more concretely.

> "Start off with we should probably have a simple agent [with] a few tools like the ability to call shell commands and gather input and ability to read skills and run them and [a] loop [that's] always listening for input. After that, we can have scheduled and cron jobs."  
> — [docs/notes/origin-2026-04-17.md](../notes/origin-2026-04-17.md)

v0.1 shipped the simple assistant foundation:

- kernel-owned agent loop
- bootstrap identity files
- tools
- skills
- durable markdown memory
- security checks
- cost tracking
- tracing
- guided setup and release docs

The remaining gap is that the current runtime is still frontend-owned. The REPL boots the kernel directly, and the old v0.2 plan assumed jobs could arrive as another runtime host plus OS scheduler glue. That is too small for the intended architecture. The next target should build the daemon substrate that future channels and services can share, while still delivering scheduled jobs as the main user-visible value unlock.

## Goal of v0.2

Ship a **local daemon-capable runtime** that hosts the kernel, exposes attachable local channels, and owns an internal job manager.

This target should support:

- a long-lived local daemon host
- attachable channels for REPL, CLI, and jobs
- recurring maintenance work for memory and traces
- recurring prompt-based jobs for technical users
- explicit, auditable local scheduling owned by the daemon
- non-interactive execution with fail-closed safety behavior

The target is still deliberately local-first and source-based. v0.2 is not a hosted control plane, a large gateway platform, or a packaged desktop app.

## Why This Next

This is the cleanest next step for three reasons:

1. It is the next thing the original note asked for after the simple agent loop.
2. It expands capability without bloating the kernel: the daemon can host small services while keeping the runtime compact and local-first.
3. It directly unlocks the memory direction in the vision by giving Allbert a place to do maintenance, pruning, summaries, and compilation later.

## Product Shape

v0.2 should reshape the workspace like this:

```text
crates/
  allbert-kernel/
  allbert-proto/   # shared local protocol types
  allbert-daemon/  # long-lived local daemon host
  allbert-cli/
  allbert-jobs/    # thin jobs client over the daemon
```

`allbert-kernel` remains the runtime core. The daemon owns it and hosts a small service set:

- `ChannelService`
  - accepts local IPC clients
  - performs handshake and channel attach
  - forwards events, confirm requests, and input requests
- `SessionService`
  - owns interactive kernel sessions
  - allows clients such as the REPL to attach and detach while the daemon stays alive
- `JobManagerService`
  - owns job definitions, mutable job state, due-job evaluation, and job execution
  - launches scheduled runs as fresh isolated sessions
- `Supervisor`
  - starts and stops services
  - coordinates shutdown
  - publishes daemon health/status

`allbert-cli` and `allbert-jobs` become thin clients. They should:

- connect to the daemon over local IPC
- auto-spawn the daemon if it is missing
- attach to the appropriate channel kind
- avoid booting their own primary runtime host

## Control Plane

v0.2 should use **local IPC first**.

Transport expectations:

- macOS/Linux: Unix domain sockets
- Windows: named pipes or an equivalent local IPC transport
- protocol framed and versioned from day one

The daemon should remain local-user scoped rather than network-addressable in v0.2.

## Runtime Defaults

The v0.2 defaults should be:

- local IPC as the control-plane default
- auto-spawn clients
- supervised threads/tasks first, subprocess seams later
- one local user daemon in v0.2
- in-memory interactive session continuity across channel attaches
- no restart-durable sessions yet
- no boot-time service install in v0.2

That means the daemon can be long-lived and useful, but it is still intentionally modest in operational scope for this release.

## Job Manager Model

The internal scheduler now belongs to the daemon.

`JobManagerService` should:

- load job definitions from `~/.allbert/jobs/definitions/`
- maintain mutable job state under `~/.allbert/jobs/state/`
- append run history under `~/.allbert/jobs/runs/`
- expose lifecycle commands to create, list, update, pause, resume, run, remove, and inspect job status
- execute due jobs internally rather than deferring to OS cron
- emit structured job-run history and cost information
- run each scheduled execution in a fresh non-interactive session by default

The thin `allbert-jobs` client should only talk to the daemon-owned job manager. It should not be a second runtime host.

## Scheduled Runs

Scheduled runs are **non-interactive** and **fresh-session by default**.

That means:

- `ConfirmPrompter` cannot assume a live human is present
- `InputPrompter` cannot assume a live human is present
- jobs must fail closed when they need missing human input or risky confirmation that has not been explicitly preconfigured
- scheduled runs cannot create, edit, pause, resume, or remove other scheduled jobs while executing
- jobs may attach zero or more ordered skills
- job prompts should be self-contained unless those skills provide reusable procedure

## First Shipped Scope

v0.2 should still be **maintenance-first**, not "general automation platform first."

The first bundled jobs should be things the vision and original note already imply:

- memory review / compaction
- daily summary or daily carry-forward
- trace review / failure triage

On top of that, v0.2 should ship one generic scheduled prompt job path so technical users can automate bounded recurring work without waiting for a larger integration layer.

## Bundled Default Job Templates

v0.2 should ship a small set of **bundled job templates** owned by the daemon-managed jobs system. These should be present but **disabled by default** so the user opts into automation explicitly.

The initial bundled set should be:

- `daily-brief`
  - morning summary of recent notes, unfinished items, and suggested next actions
- `weekly-review`
  - weekly review of progress, repeated blockers, unresolved work, and likely follow-ups
- `memory-compile`
  - promote stable facts from recent notes into `MEMORY.md` and deduplicate existing entries conservatively
- `trace-triage`
  - review recent failures or suspicious traces and write a short anomaly report
- `system-health-check`
  - verify provider env vars, bootstrap files, trusted roots, and recurring job health

Default reporting expectations:

- `daily-brief` and `weekly-review` report on success
- `memory-compile` reports when it made changes or hit a problem
- `trace-triage` and `system-health-check` default to anomaly-first reporting

These bundled jobs should prefer first-party skills or well-bounded built-in prompts rather than giant inline prompt blobs duplicated across files.

## In Scope

- `allbert-proto` shared protocol crate
- `allbert-daemon` host crate
- local IPC control plane
- auto-spawn client behavior
- daemon-owned `ChannelService`, `SessionService`, `JobManagerService`, and `Supervisor`
- `allbert-cli` as a thin REPL and control client
- `allbert-jobs` as a thin jobs client
- daemon-owned lifecycle commands for job management
- job-run history and cost visibility
- fresh-session scheduled execution
- skill-backed jobs
- bundled disabled job templates

## Out of Scope

- vector RAG / embeddings
- network-first control plane
- boot-time OS service install and management
- restart-durable interactive sessions
- heartbeat-style background awareness loop
- embedded scripting engines
- retraining or personality-model pipelines
- web frontend
- broad third-party integration marketplace

## Milestones

**M1 — Daemon substrate and protocol.** Add `allbert-proto` and `allbert-daemon`, define the local IPC protocol, channel attach semantics, daemon boot path, and supervisor scaffolding. *Verify:* automated tests cover daemon boot, single-instance behavior, client attach, handshake/version checks, and client auto-spawn fallback.

**M2 — Channelized CLI and REPL.** Refactor the kernel/frontend seam so REPL and CLI behavior runs through daemon-owned sessions and local channels rather than directly booting the kernel in the client. *Verify:* automated tests cover channel-backed confirm/input round-trips, client disconnect handling, and in-memory session continuity while the daemon is alive.

**M3 — Internal `JobManagerService`.** Add daemon-owned job definitions, mutable job state, due-job evaluation, run history, and the thin `allbert-jobs` client surface. *Verify:* automated tests cover create/update parsing, pause/resume behavior, due-job selection, skip-if-running behavior, and reload of definitions/state after daemon restart.

**M4 — First-party maintenance jobs.** Ship the bundled disabled templates for `daily-brief`, `weekly-review`, `memory-compile`, `trace-triage`, and `system-health-check`, plus any first-party skills they depend on. *Verify:* fake-provider tests cover the bundled jobs' expected file outputs, report-policy behavior, and fail-closed behavior.

**M5 — v0.2 closeout and local daemon smoke.** Add end-user docs for daemon operation, channel attach behavior, and jobs usage, then run end-to-end local smoke for daemon startup, attach/detach, and scheduled execution while the daemon is alive. *Verify:* clean-home daemon setup works, clients auto-spawn and attach, job history is readable, costs are recorded, recurring jobs respect non-interactive safety policy, and known limitations are documented clearly.

## Testing and Verification

Automated tests should cover:

- daemon boot and shutdown
- local IPC handshake and attach behavior
- client auto-spawn and reconnect path
- session continuity across channel attach/detach while the daemon is alive
- job-definition parsing
- due-job selection
- lifecycle command behavior
- non-interactive confirm/input behavior
- job-run history writes
- cost tracking from scheduled runs
- skill-backed fresh-session execution
- bundled maintenance job behavior

Manual smoke should cover:

- daemon startup and status check
- REPL attach, detach, and reattach
- one named job run
- one due-job sweep
- one pause/resume cycle
- one maintenance job producing durable markdown output
