# Allbert v0.2 — Daemon Host, Channels, and Internal Job Manager

## Context

[`docs/vision.md`](../vision.md) now points toward a daemon-capable micro-kernel that can host lightweight internal services and accept attachable user-facing channels. The original project note still matters because it identified scheduled jobs as the next meaningful value unlock after the simple assistant loop, but the runtime shape now needs to be spelled out more concretely.

> "Start off with we should probably have a simple agent [with] a few tools like the ability to call shell commands and gather input and ability to read skills and run them and [a] loop [that's] always listening for input. After that, we can have scheduled and cron jobs."  
> — [docs/notes/origin-2026-04-17.md](../notes/origin-2026-04-17.md)

v0.1 shipped the simple assistant foundation:

- kernel-owned agent loop
- bootstrap identity files
- tools
- skills
- durable markdown memory
- security checks
- cost tracking
- tracing
- guided setup and release docs

The remaining gap is that the current runtime is still frontend-owned. The REPL boots the kernel directly, and the old v0.2 plan assumed jobs could arrive as another runtime host plus OS scheduler glue. That is too small for the intended architecture. The next target should build the daemon substrate that future channels and services can share, while still delivering scheduled jobs as the main user-visible value unlock.

## Goal of v0.2

Ship a **local daemon-capable runtime** that hosts the kernel, exposes attachable local channels, and owns an internal job manager.

This target should support:

- a long-lived local daemon host
- attachable channels for REPL, CLI, and jobs
- recurring maintenance work for memory and traces
- recurring prompt-based jobs for technical users
- explicit, auditable local scheduling owned by the daemon
- non-interactive execution with fail-closed safety behavior

The target is still deliberately local-first and source-based. v0.2 is not a hosted control plane, a large gateway platform, or a packaged desktop app.

## Why This Next

This is the cleanest next step for three reasons:

1. It is the next thing the original note asked for after the simple agent loop.
2. It expands capability without bloating the kernel: the daemon can host small services while keeping the runtime compact and local-first.
3. It directly unlocks the memory direction in the vision by giving Allbert a place to do maintenance, pruning, summaries, and compilation later.

## Product Shape

v0.2 should reshape the workspace like this:

```text
crates/
  allbert-kernel/
  allbert-proto/   # shared local protocol types
  allbert-daemon/  # long-lived local daemon host
  allbert-cli/
  allbert-jobs/    # thin jobs client over the daemon
```

`allbert-kernel` remains the runtime core and stays usable as a standalone library. Integration tests, future embedded uses, and one-shot tools must still be able to construct a `Kernel` directly and drive it without the daemon; the daemon is a host *around* the library, not a replacement for it. The v0.1 single-session shape expands into a multi-session kernel (see [ADR 0021](../adr/0021-kernel-multiplexes-sessions-shared-runtime-per-session-state.md)): one `Kernel` owns many sessions concurrently, with shared singletons (config defaults, hooks, tool registry, provider factory, provider client pool, tracing, cost log) and per-session state (messages, active skills, turn counter, session cost total, confirm-trust approvals, adapter handles, effective model config). The daemon owns that one Kernel and hosts a small service set:

- `ChannelService`
  - accepts local IPC clients
  - performs handshake and channel attach
  - forwards events, confirm requests, and input requests
- `SessionService`
  - owns interactive kernel sessions
  - allows clients such as the REPL to attach and detach while the daemon stays alive
- `JobManagerService`
  - owns job definitions, mutable job state, due-job evaluation, and job execution
  - launches scheduled runs as fresh isolated sessions
- `Supervisor`
  - starts and stops services
  - coordinates shutdown
  - publishes daemon health/status

`allbert-cli` and `allbert-jobs` become thin clients. They should:

- connect to the daemon over local IPC
- auto-spawn the daemon if it is missing
- attach to the appropriate channel kind
- avoid booting their own primary runtime host

`allbert-cli` remains the primary user-facing binary and is the home of the REPL, `/setup`, `/status`, `/cost`, and `/model` surfaces from v0.1. Jobs commands are exposed as `allbert-cli` subcommands (`allbert jobs list`, `allbert jobs run <name>`, `allbert jobs pause <name>`, etc.). `allbert-jobs` ships as the same subcommand group installed under a short alias binary for users who prefer it; both routes call into the same thin client code in `allbert-jobs`.

## Control Plane

v0.2 should use **local IPC first**.

Transport expectations:

- macOS/Linux: Unix domain sockets at `~/.allbert/run/daemon.sock` with mode `0600`, parent directory `0700`
- Windows: named pipes with an ACL restricted to the current user SID
- protocol framed and versioned from day one
- stale-socket handling: daemon attempts a connect before unlinking; refuses to start if socket or parent directory has broader permissions than specified

The daemon should remain local-user scoped rather than network-addressable in v0.2. Trust is filesystem-scoped: any process that can open the endpoint is trusted as the user, with no token handshake in v0.2 (see [ADR 0023](../adr/0023-local-ipc-trust-is-filesystem-scoped-no-token-auth-in-v0-2.md)). Tool-level policy from v0.1 (exec_policy, web_policy, confirm-trust, security hooks) continues to run on every turn regardless of channel.

Protocol compatibility in v0.2 is strict: client and daemon must agree on the exact protocol version during handshake, otherwise the client refuses to proceed and prints a remediation message (typically "upgrade one side"). v0.2 is not obligated to wire-accept older clients; the shared `allbert-proto` crate is versioned together with the daemon. A compatibility window across minor versions is a future concern, explicitly deferred so the protocol stays easy to evolve through v0.2 milestones.

## Runtime Defaults

The v0.2 defaults should be:

- local IPC as the control-plane default
- auto-spawn clients
- supervised threads/tasks first, subprocess seams later
- one local user daemon in v0.2
- in-memory interactive session continuity across channel attaches
- no restart-durable sessions yet
- no boot-time service install in v0.2

That means the daemon can be long-lived and useful, but it is still intentionally modest in operational scope for this release.

## Job Manager Model

The internal scheduler now belongs to the daemon.

`JobManagerService` should:

- load job definitions from `~/.allbert/jobs/definitions/`
- maintain mutable job state under `~/.allbert/jobs/state/`
- append run history under `~/.allbert/jobs/runs/`
- expose lifecycle commands to create, list, update, pause, resume, run, remove, and inspect job status
- execute due jobs internally rather than deferring to OS cron
- emit structured job-run history and cost information
- run each scheduled execution in a fresh non-interactive session by default

Job definitions are markdown files with YAML frontmatter, filename = job name, and a schedule DSL plus cron escape hatch (see [ADR 0022](../adr/0022-job-definitions-are-markdown-with-frontmatter-and-a-bounded-schedule-dsl.md)). Frontmatter may also carry per-job `timezone` and `model` overrides. Mutable job state (`~/.allbert/jobs/state/<name>.json`) holds last-run timestamp, next-due timestamp, failure streak, and pause flag. Run history (`~/.allbert/jobs/runs/<YYYY-MM-DD>.jsonl`) is append-only JSONL with one record per run: `{run_id, job_name, session_id, started_at, ended_at, outcome, cost_usd, skills_attached, stop_reason?}`.

Cost shape does not fork: the existing `CostEntry` at `~/.allbert/costs.jsonl` gains an optional `source: "repl" | "job:<name>"` and `run_id?` field so the single cost log still carries every provider call. Job-run records in `~/.allbert/jobs/runs/` summarize cost per run by joining on `run_id`.

The thin `allbert-jobs` client should only talk to the daemon-owned job manager. It should not be a second runtime host.

Missed-run and interval semantics are explicit in v0.2:

- `JobManagerService` persists `next_due_at` for every enabled job.
- If the daemon was down and `next_due_at <= now` at startup, the job becomes due **once** immediately as a coalesced catch-up run.
- After that catch-up run, `next_due_at` is advanced repeatedly until it lands in the future; missed intervals are not replayed one-by-one.
- `every <duration>` schedules are anchored to persisted schedule state, not to daemon start time. A restart does not reset the cadence.
- Fixed-time schedules (`@daily`, `@weekly`, `cron:`) also coalesce missed slots into at most one immediate catch-up run after downtime.

## Model Resolution

The daemon owns global config defaults, but model selection is resolved per session.

- The daemon-wide `[model]` config remains the default for newly created sessions.
- Each interactive session stores its own effective model config at session creation time.
- `/model` changes the attached interactive session only. It does not silently change every session on the daemon.
- Job definitions may carry an optional model override; if they do not, each job run resolves its model from the daemon's current default at run creation time.
- Reloading config changes the default used for **new** sessions and **future** job runs. It does not preempt in-flight turns or mutate the model of an already-running session.

## Scheduled Runs

Scheduled runs are **non-interactive** and **fresh-session by default**.

That means:

- `ConfirmPrompter` cannot assume a live human is present
- `InputPrompter` cannot assume a live human is present
- jobs must fail closed when they need missing human input or risky confirmation that has not been explicitly preconfigured
- scheduled runs cannot create, edit, pause, resume, or remove other scheduled jobs while executing
- jobs may attach zero or more ordered skills
- job prompts should be self-contained unless those skills provide reusable procedure

## Channel Attach Semantics

The REPL session is a single persistent interactive session owned by the daemon. Any `allbert-cli` client attach connects to that one session: its history, cost total, active skills, and confirm-trust approvals are preserved across disconnect/reattach while the daemon stays alive. Two concurrent `allbert-cli` attaches see the same session (last-write-wins for input, broadcast for events). A client that wants an isolated session uses a one-shot subcommand; those get their own fresh session and are ended when the command returns. Scheduled job sessions always run fresh (ADR 0016) and are never attachable by a client.

## Daemon Logging and Tracing

The daemon runs without an attached terminal, so v0.1's stderr-first tracing shape shifts:

- All tracing is written to files under `~/.allbert/logs/`: `daemon.log` for INFO and above, `daemon.debug.log` for DEBUG and above when `config.trace` is on. Files rotate daily with a small retention cap.
- Attached clients receive live event streams (assistant text, tool calls, tool results, cost) over the channel. Clients render those to their own stderr/stdout as they did in v0.1.
- A client-facing trace toggle (`allbert trace on|off`) flips the DEBUG file sink on the daemon without restarting it.
- Daemon startup writes a one-line boot record (pid, socket path, trace state, provider) so operators can inspect recent daemon behavior without attaching.

## Hook System Reach

The v0.1 hook API stays public and keeps its existing points (`BeforePrompt`, `BeforeModel`, `BeforeTool`, `AfterTool`, `OnModelResponse`, `OnTurnEnd`). v0.2 adds two job-oriented points so `JobManagerService` gets the same auditability:

- `BeforeJobRun` — fires when the job manager is about to create a fresh session for a due job; can `Abort` with a reason (recorded to run history).
- `AfterJobRun` — fires after the fresh session is torn down; receives outcome, duration, cost, and stop reason.

`HookCtx` grows a `session_id` field and a `job_run_id?` field so hooks can distinguish REPL turns from scheduled runs. The hook registry itself is one shared registry on the Kernel (ADR 0021), and hooks must be safe to invoke concurrently across sessions.

## Bundled First-Party Skills

Bundled jobs (M4) that depend on first-party skills require those skills to exist on disk. The daemon seeds bundled skills on first run into `~/.allbert/skills/` using the same template-seeding pattern v0.1 uses for bootstrap files in `paths.rs`. Seeded skills are written with an explicit `bundled: true` frontmatter marker; users can edit or delete them, and the seeder will not overwrite a user-modified copy.

## Config Schema Deltas

v0.2 extends `~/.allbert/config.toml`:

- New `[daemon]` block: `socket_path?` (override default), `log_dir?`, `log_retention_days`, `auto_spawn: bool` (default `true`).
- New `[jobs]` block: `enabled: bool` (master switch, default `false` until user opts in via setup), `max_concurrent_runs` (default `1`), `default_timeout_s` (default `600`), `default_timezone?` (defaults from the user's configured timezone when available, otherwise system local time).
- `SetupConfig.version` bumps from the shipped v0.1 value `1` to `2`; loader migrates older configs by filling daemon and jobs defaults without invalidating existing onboarding state.
- Existing `[model]`, `[security]`, `[limits]`, and `trace` blocks are unchanged.

## Setup Flow

`/setup` remains the guided onboarding surface. In v0.2 it runs client-side under `allbert-cli` (same as v0.1) and writes the updated config file on disk; the daemon watches for config changes and reloads on the next session boundary. The wizard gains two additional steps: (1) confirm daemon autoboot preferences and (2) offer to enable one or more bundled job templates. Both steps are opt-in and default to off. The wizard also seeds `jobs.default_timezone` from the user's existing timezone profile answer unless the user explicitly overrides it. Existing v0.1 users with `setup.version = 1` are treated as already onboarded; loading v0.2 simply migrates their config to `setup.version = 2` by filling daemon/jobs defaults and preserving prior answers.

## First-Run Flow

A brand-new user running `allbert` on a clean home directory should not have to know about the daemon. The flow:

1. The CLI probes the local IPC endpoint. It is absent.
2. The CLI detects no config at `~/.allbert/config.toml` and enters the `/setup` wizard **before** attempting auto-spawn, so the daemon starts against a complete config rather than a template.
3. The wizard writes config, seeds bootstrap files, seeds bundled skills (disabled) and bundled job templates (disabled).
4. The CLI then auto-spawns the daemon, handshakes, and attaches to a fresh REPL session.

If the CLI finds a config but no daemon, it skips the wizard and auto-spawns directly. If both are present, it just attaches. The daemon must be able to start against a freshly written config without requiring a restart of its own process tree.

## Daemon Lifecycle Commands

`allbert-cli` exposes explicit daemon controls so the user is never forced to reach for process tools:

- `allbert daemon status` — reports running state, pid, socket path, uptime, session count, and trace state.
- `allbert daemon start` — explicit start (useful after a manual stop or on boot).
- `allbert daemon stop` — graceful shutdown: stop accepting new clients, drain in-flight turns up to a timeout, cancel pending scheduled runs, flush cost/trace/run-history writes, then exit.
- `allbert daemon restart` — stop + start, preserving config.
- `allbert daemon logs [--debug] [--follow]` — tail `daemon.log` or `daemon.debug.log`.

Ctrl-C in the REPL client detaches the client but does **not** stop the daemon; stopping the daemon is an explicit command. This protects scheduled jobs and other attached clients from incidental shutdown.

## Failure Surfacing for Scheduled Runs

A scheduled run that fails closed or errors out needs to reach the user without requiring JSONL spelunking:

- Every failure appends to `~/.allbert/jobs/failures/<YYYY-MM-DD>.jsonl` with `{run_id, job_name, stop_reason, ended_at}`, in addition to the full record under `~/.allbert/jobs/runs/`.
- When any `allbert-cli` client is attached, the daemon broadcasts a structured `JobFailed` event over the channel; the REPL renders a one-line notice at the next prompt.
- `allbert jobs status` surfaces the most recent failure and failure streak per job.
- A bundled `trace-triage` run (M4) reads the failures log and writes a short anomaly report to memory so the next REPL session can see it surfaced naturally.

## First Shipped Scope

v0.2 should still be **maintenance-first**, not "general automation platform first."

The first bundled jobs should be things the vision and original note already imply:

- memory review / compaction
- daily summary or daily carry-forward
- trace review / failure triage

On top of that, v0.2 should ship one generic scheduled prompt job path so technical users can automate bounded recurring work without waiting for a larger integration layer.

## Bundled Default Job Templates

v0.2 should ship a small set of **bundled job templates** owned by the daemon-managed jobs system. These should be present but **disabled by default** so the user opts into automation explicitly.

The initial bundled set should be:

- `daily-brief`
  - morning summary of recent notes, unfinished items, and suggested next actions
- `weekly-review`
  - weekly review of progress, repeated blockers, unresolved work, and likely follow-ups
- `memory-compile`
  - promote stable facts from recent notes into `MEMORY.md` and deduplicate existing entries conservatively
- `trace-triage`
  - review recent failures or suspicious traces and write a short anomaly report
- `system-health-check`
  - verify provider env vars, bootstrap files, trusted roots, and recurring job health

Default reporting expectations:

- `daily-brief` and `weekly-review` report on success
- `memory-compile` reports when it made changes or hit a problem
- `trace-triage` and `system-health-check` default to anomaly-first reporting

These bundled jobs should prefer first-party skills or well-bounded built-in prompts rather than giant inline prompt blobs duplicated across files.

## In Scope

- `allbert-proto` shared protocol crate
- `allbert-daemon` host crate
- local IPC control plane
- auto-spawn client behavior
- daemon-owned `ChannelService`, `SessionService`, `JobManagerService`, and `Supervisor`
- `allbert-cli` as a thin REPL and control client
- `allbert-jobs` as a thin jobs client
- daemon-owned lifecycle commands for job management
- job-run history and cost visibility
- fresh-session scheduled execution
- skill-backed jobs
- bundled disabled job templates

## Out of Scope

- vector RAG / embeddings
- network-first control plane
- boot-time OS service install and management
- restart-durable interactive sessions
- heartbeat-style background awareness loop
- embedded scripting engines
- retraining or personality-model pipelines
- web frontend
- broad third-party integration marketplace

## Milestones

**M1 — Daemon substrate and protocol.** Add `allbert-proto` and `allbert-daemon`, define the local IPC protocol, channel attach semantics, daemon boot path, and supervisor scaffolding. *Verify:* automated tests cover daemon boot, single-instance behavior, client attach, handshake/version checks, and client auto-spawn fallback.

**M2 — Channelized CLI and REPL.** Refactor the kernel/frontend seam so REPL and CLI behavior runs through daemon-owned sessions and local channels rather than directly booting the kernel in the client. *Verify:* automated tests cover channel-backed confirm/input round-trips, client disconnect handling, in-memory session continuity while the daemon is alive, and session-local `/model` behavior that does not leak across sessions.

**M3 — Internal `JobManagerService`.** Add daemon-owned job definitions, mutable job state, due-job evaluation, run history, and the thin `allbert-jobs` client surface. *Verify:* automated tests cover create/update parsing, pause/resume behavior, due-job selection, skipped-overlap behavior, coalesced catch-up behavior after downtime, timezone resolution, and reload of definitions/state after daemon restart.

**M4 — First-party maintenance jobs.** Ship the bundled disabled templates for `daily-brief`, `weekly-review`, `memory-compile`, `trace-triage`, and `system-health-check`, plus any first-party skills they depend on. *Verify:* fake-provider tests cover the bundled jobs' expected file outputs, report-policy behavior, and fail-closed behavior.

**M5 — v0.2 closeout and local daemon smoke.** Add end-user docs for daemon operation, channel attach behavior, and jobs usage, then run end-to-end local smoke for daemon startup, attach/detach, and scheduled execution while the daemon is alive. *Verify:* clean-home daemon setup works, clients auto-spawn and attach, auto-spawn failure paths surface actionable error messages (missing daemon binary, stale socket, permission drift), job history is readable, costs are recorded, recurring jobs respect non-interactive safety policy, and known limitations are documented clearly.

## Testing and Verification

Automated tests should cover:

- daemon boot and shutdown
- local IPC handshake and attach behavior
- client auto-spawn and reconnect path
- session continuity across channel attach/detach while the daemon is alive
- config migration from shipped `setup.version = 1` to v0.2 `setup.version = 2`
- job-definition parsing
- due-job selection
- lifecycle command behavior
- non-interactive confirm/input behavior
- session-local model override behavior
- coalesced catch-up behavior after daemon downtime
- timezone resolution precedence for scheduled jobs
- job-run history writes
- cost tracking from scheduled runs
- skill-backed fresh-session execution
- bundled maintenance job behavior

Manual smoke should cover:

- daemon startup and status check
- REPL attach, detach, and reattach
- one named job run
- one due-job sweep
- one pause/resume cycle
- one maintenance job producing durable markdown output
