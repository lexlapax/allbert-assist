# Allbert v0.57 Request Flow - Pi-mode Coding Surface

Status: in progress; M0-M9 were implemented 2026-06-23, then the
post-implementation audits and warm-validation findings added M9.1-M9.23
remediation work. M9.1 is the session-scope remediation:
the six coding actions are internal registered capabilities, absent from
`agent_modules/0`, rejected by public protocol exposure, and guarded at runtime by
active Pi-mode session metadata plus the local-coding operator tier. M9.2 wires
the production coding turn to `ReqLLM.stream_text`, incremental stream-event
emission, the TUI live sink, and provider-level `StreamResponse.cancel`
registration. M9.3-M9.8 are implemented agent-loop remediation: bind the six internal
coding actions as session-local `ReqLLM.Tool` definitions, execute model-proposed
tool calls only through `Actions.Runner.run/3`, append bounded tool results back to
`ReqLLM.Context`, persist the updated session context, update eval/operator
validation coverage, and prepare for a third implementation audit. Release closeout
requires deterministic gates plus warm operator validation against a real configured
streaming/tool-capable coding profile.

Implementation details live in `docs/plans/v0.57-plan.md`.

## Locked Decisions
Recorded in `v0.57-plan.md`. Pi-mode is a channel/app on the one spine with no
extra authority; **six default tools** — read-only/sensitive `read`/`grep`/`glob`
and effectful `write`/`edit`/`bash` — through `Actions.Runner.run/3` + Security
Central; read/search runs without a confirmation prompt but still enforces cwd
jail, ignore policy, caps, redaction, trace, and audit; a named
**local-coding operator** trust tier (ADR 0056 lineage, sandbox **Level 1**; single
trusted operator, main session, `tui`) that is never the default and never for
channel-originated or generated-code sessions; **coder-facing approval modes**
(`default`/`accept-edits`/`plan`/`tier`) + `Resources.Grants`-backed per-repo "always
allow" as a **confirmation-cost seam** in `Security.Decision` (drops only the prompt;
preserves decision/trace/audit; no new authority); `bash` is tier-restricted **raw
shell** (argv-only `run_shell_command` for everyone else), cwd-jailed + timed out;
deterministic acceptance gates for effectful/generated-code work (the ordinary turn
ends when the model stops calling tools); sub-1000-token **prompt + tool-defs**
budget + chunked-read context discipline; a coding turn runs under a **supervised
Task** (async boundary) so **net-new v0.57** live rendering — assistant-text token
streaming + progressive tool-argument diff streaming over the v0.55 **static** split,
plus a real **Esc-to-cancel** (registered stream cancel callback + bounded registry
task shutdown) + queued steering — is live after M9.2 and must be validated against
a configured streaming/tool-capable provider profile. The coding **slash set**
(`/help`/`/mode`/`/model`/`/clear`/`/init`/`/diff`/
`/compact`; `/mode`/`/model`/`/clear`/`/compact` ungated session ops, no `:coding_session_write`);
Pi-mode **session entry** (`/pi [path]`) pins the working repo to the cwd jail;
coding tool I/O is session-scoped, not long-term memory; mid-session model switch
over req_llm message-history; delegate agents (no bash-spawned sub-agents).

## Main User / Operator Flows

### Opening a Pi-mode coding session
A single trusted operator, at their own terminal, opens the v0.55 `tui` channel in
the **main session** and **enters Pi-mode** (`/pi [path]`) for the working repo. On
entry the working repo is resolved to an absolute root, symlink-checked, and
**pinned to `coding.workspace.cwd_jail` for the session's life** — every one of the
six tools and raw shell is jailed to it. Security Central resolves the **local-coding
operator** tier (at sandbox **Level 1**) *only* because the caller is the trusted
operator, in the main session, on `tui`. The session loads a coding system prompt
**+ six tool definitions under the combined token budget**, assembled from the
`AGENTS.md` hierarchy + the v0.03 skill loader, exposing the six default tools
(read-only/sensitive `read`/`grep`/`glob` + effectful `write`/`edit`/`bash`) and the
operator's chosen approval mode. The objective is "pair on this repo"
(stage 3 trivial); step proposal (stage 4) is trivial; authorization (stage 5)
runs cheaply at the tier — *present, not absent* — and policy/redaction/trace/audit
still run at Level 1.

### The read / grep / glob / write / edit / bash loop at the local-coding tier
The operator types an instruction. The model proposes a tool call as **advisory
output** — it does not execute anything itself. `read`/`grep`/`glob` are
read-only/sensitive and run without a confirmation prompt, as in every peer CLI,
but still become signals through `Actions.Runner.run/3` and Security Central.
Each effectful proposal (`write`/`edit`/`bash`) follows the same runner/gate path.
The operator's **approval mode** sets the confirmation cost:
`default` prompts effects, `accept-edits` auto-approves `write`/`edit` but prompts
`bash`, `plan` executes only read/search (write/edit/bash proposals never execute),
`tier` is cheapest for the trusted local caller; a persisted "always allow `<cmd>`"
is a **`Resources.Grants` remembered grant** scoped by repo fingerprint + canonical
(normalized/redacted) command + permission + cwd that suppresses a specific prompt.
The cheap path is a **confirmation-cost seam** in `Security.Decision`: it drops only
the interactive prompt while the `:needs_confirmation` decision, policy, redaction,
trace, and audit **still run** at Level 1. `read`/`grep`/`glob`/
`write`/`edit` are confined to `coding.workspace.cwd_jail` (normalized path
and symlink checks); `bash` is host execution behind the registered action: raw
shell **only at this tier** (argv-only otherwise), cwd-jailed and timed out (reuse
`skill_script_runner`); the host/container, not OTP/BEAM, is the security boundary.
Context is gathered through **chunked reads** (offset/limit) and file artifacts.
M9.3-M9.4 implement the production loop by passing only session-local
`ReqLLM.Tool` definitions to `ReqLLM.stream_text`, consuming the stream once,
materializing the assistant tool calls, executing them through
`Actions.Runner.run/3`, appending `ReqLLM.Context.tool_result/3` messages, and
continuing until the model returns final prose with no actionable tool calls.
In `default` mode, a model-proposed effect that returns `needs_confirmation` is a
pending result, not an applied effect: the current model stream may finish after
observing the pending result, and approval resumes the registered action later
through the confirmation path. In `accept-edits`, `tier`, or exact command-grant
cases, same-loop effect completion happens only because Security Central suppresses
the prompt while preserving the `:needs_confirmation` decision, trace, and audit.

### Streamed diffs + token streaming
`write`/`edit` produce the split tool-result payload (`model_payload` vs.
`surface_payload`, ADR 0067 extending ADR 0029): the model receives the clean
`model_payload`; the terminal renders the `surface_payload` diff through the
unified renderer (ADR 0030) / `owl`. The streaming is **net-new v0.57** work — a
coding-diff live-region renderer that renders the diff by progressively parsing the
tool-call **arguments** as they arrive, **and token-by-token streaming of the
assistant's prose** so the turn feels live. Token streaming needs the runtime to
emit model token deltas (v0.55 is turn-complete). The `surface_payload` never leaks
back into model context. Post-audit status: M9.2 connects the production coding-turn
LLM path to `ReqLLM.stream_text`, emits provider chunks incrementally through
`Coding.StreamPipeline`, and reconciles with `turn_complete`. M9.5 extends this
path so model-proposed tool calls emit tool-argument and tool-result events before
final reconciliation.

### Interrupt + steer
A coding turn runs under a **supervised Task** (the M5 async turn-execution boundary
the synchronous v0.55 runtime lacks), so it is addressable and killable. The operator
can press **Esc to cancel** a running turn: the in-flight model stream is aborted
via the running turn's registered
`ReqLLM.StreamResponse.cancel` callback and the supervised turn task is shut down
through the `Coding.TurnSupervisor` registry-level cancel path
(`coding.cancel.grace_ms`, then hard-kill fallback if still alive). Any
running action is bounded by task shutdown plus its own Runner/action timeout and
brutal-kill policy; v0.57 does not promise an additional child-process cancel
callback beyond those action boundaries. Work done so far is kept, a
`turn_cancelled` stream event is emitted, and a **partial-turn trace** is written.
Where a provider stream cannot be aborted or no stream callback was registered, the
registry task shutdown still frees the turn and the late result is discarded. A
**queued correction** lets the operator redirect without killing the turn (applied on
the next turn). No authority path changes.

### Mid-session model switch
The operator invokes the model-switch affordance (a `tui` slash command
`/model <name>`). The session's req_llm message history is serialized and handed to
the new model via `ReqLLM.Context.merge_response/2`; cross-provider history conversion
is **best-effort** (provider reasoning/tool-call shapes differ). M8 verifies the
concrete API against vendored `deps/req_llm` source (Context7 resolves `Req`, not a
first-class `req_llm` package). The
conversation continues with **no change to the authority path** — same runner, same
Security Central, same trace.

### Completion
The ordinary interactive turn ends naturally when the model **stops emitting tool
calls** and control returns to the operator prompt — the same REPL norm as Claude
Code / Codex / Gemini. The separate rule, "the model never decides it's *done*,"
governs **deterministic acceptance gates** for effectful or generated-code work
(evidence + confirmation), the same posture Allbert took against Hermes-style
model-confidence completion — not the end of a conversational turn.

```text
  -> [turn runs under a SUPERVISED Task: addressable + cancellable (M5 boundary)]
  -> Security Central gate  [confirmation-cost seam: approval mode + local-coding tier @ Level 1]
       seam drops only the prompt; :needs_confirmation decision + policy + trace + audit unchanged
       mode: default(prompt) | accept-edits(auto write/edit) | plan(read/search only) | tier(cheapest)
       + Resources.Grants remembered "always allow <cmd>"  (confirmation cost only; no authority)
       trusted operator + main session + tui  -> cheap gate, raw-shell bash, full trace
       channel-originated / scheduled / generated-code -> full gate, argv-only (no tier)
  -> action executes (read[chunked] | grep/glob[read-only/sensitive] | write[cwd-jail] | edit[cwd-jail] | bash[Level 1])
  -> split tool-result (ADR 0067/0029): model_payload + surface_payload diff
  -> terminal render: NET-NEW
       v0.57 live region (ADR 0030 / owl) for progressive tool-argument diff

  -> channel_events (dedupe, redacted summary) + trace [coding tool I/O is session-scoped,

  -> [operator] mid-session model switch -> req_llm Context.merge_response handoff (no authority change)
```

## How the gates stay present-but-cheap at the tier
The local-coding operator tier changes the *cost* of the gate, never its
*existence* or the *sandbox level*. For the single trusted operator in the main
session on `tui`, Security Central skips the per-action confirmation prompt **but
still runs at Level 1**: it evaluates policy, enforces the `bash` cwd-jail +
timeout, redacts/truncates `channel_events` summaries, and writes the full trace.
The tier is a **named, auditable trust tier** (ADR 0056 lineage), never the default
— every other caller (channel-originated inbound, scheduled jobs, generated-code
sessions) gets the full gate and argv-only `bash` with no tier escalation. This is
the rationale's "stages 3–5 intact; gates present but cheap" made concrete, without
relaxing the sandbox level.

## Authority Invariants
- Pi-mode calls only `Runtime.submit_user_input/1` + `Actions.Runner.run/3`; no
  direct Security Central / confirmation-store / Resource Access access (ADR 0016).
- Advisory model output never authorizes execution; every effect routes through the
  runner + Security Central.
- The six coding actions are registered internal capabilities, not general
  intent-agent or public protocol tools. They deny out-of-session invocation before
  path policy, search, file effects, or shell normalization.
- The local-coding operator tier resolves only for trusted operator + main session
  + `tui` (at sandbox Level 1); never the default; unreachable by
  channel-originated, scheduled, or generated-code sessions.
- `bash` is cwd-jailed + timed out; raw shell resolves only at the tier (argv-only
  otherwise); OTP/BEAM supervision is not an OS security boundary — the
  host/container is.
- `read`/`grep`/`glob` are read-only/sensitive and run without a confirmation
  prompt; they still route through Runner/Security Central and enforce cwd jail,
  ignore policy, output caps, redaction, trace, and audit.
- Approval modes and the `Resources.Grants`-backed "always allow" change only
  confirmation cost via the `Security.Decision` seam — the `:needs_confirmation`
  decision, policy, trace, and audit are preserved; only the prompt is dropped.
  `plan` mode executes only read/search; no mode grants authority beyond its
  definition.
- A coding turn runs under a supervised Task. Provider-level
  `StreamResponse.cancel` is registered for live coding provider streams before
  bounded task shutdown preserves the partial trace.
- Coding tool I/O stays in the session trace; it is not auto-promoted to
  markdown-first long-term memory unless the operator explicitly saves it.
- `/mode`/`/model`/`/clear`/`/compact` are ungated, non-routable session ops (no
  `:coding_session_write`); `/init` writes through `:coding_file_write`; `/diff` and
  `@file` read through `:coding_file_read`.
- Acceptance for effectful/generated-code work is deterministic (evidence +
  confirmation); the ordinary turn ends when the model stops calling tools.
- The split-result UI payload never leaks into the model payload (diffs **and**
  streamed assistant tokens render from `surface_payload`/`model_payload` only).
- Mid-session model switch changes the model only; the authority path is unchanged.
- Coding slash commands are non-routable (allowlist only); never intent candidates.
- Delegation uses restricted-registry delegate agents; no bash-spawned sub-agent.

  Level 1, raw-shell `bash`); a simulated channel-originated or generated-code
  caller gets the full gate and argv-only `bash`.
- Approval modes behave: `plan` blocks a `write` (read/search-only); `accept-edits`
  auto-applies an `edit` but still prompts `bash`; the cheap path preserves the
  `:needs_confirmation` decision + trace + audit and only drops the prompt; a
  `Resources.Grants` remembered "always allow `<cmd>`" suppresses only that prompt,
  per repo fingerprint + canonical command, and is listable, revocable, expirable,
  and auditable.
- `write`/`edit` refuse path traversal, symlink escape, and destinations outside
  `coding.workspace.cwd_jail`; the same simulated non-tier callers get the full
  gate for file effects.
- A `write`/`edit` keeps diff previews in `surface_payload` and model receipts in
  `model_payload`; streamed assistant text renders from provider token deltas.
- **Esc** mid-turn cancels cleanly: the registered provider
  `StreamResponse.cancel` callback runs before bounded registry task shutdown and
  partial-turn trace preservation.
- The coding slash set (`/help`/`/mode`/`/model`/`/clear`/`/init`/`/diff`/`/compact`) creates
  no model turn and is non-routable; `/init` is action-backed (`:coding_file_write`),
  `/diff`+`@file` read-backed (`:coding_file_read`), and `/mode`/`/model`/`/clear`/
  `/compact` are ungated session ops that never prompt.
- A mid-session model switch preserves session history and changes no authority.
