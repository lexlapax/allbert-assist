## Locked Decisions
Recorded in `v0.57-plan.md`. Pi-mode is a channel/app on the one spine with no
extra authority; four default boundary actions (`read`/`write`/`edit`/`bash`)
through `Actions.Runner.run/3` + Security Central; a named **local-coding
operator** trust tier (ADR 0056 lineage, sandbox **Level 1**; single trusted
operator, main session, `tui`) that is never the default and never for
channel-originated or generated-code sessions; `bash` is tier-restricted **raw
shell** (argv-only `run_shell_command` for everyone else), cwd-jailed + timed out;
deterministic acceptance (no model-decides-done); sub-1000-token **prompt +
tool-defs** budget + chunked-read context discipline (offset/limit + artifacts,
not whole-file); streamed diffs are **net-new v0.57** (a coding diff live-region
renderer + progressive tool-argument rendering) over the v0.55 **static** split payload;
mid-session model switch over req_llm message-history; delegate agents (no
bash-spawned sub-agents).

## Main User / Operator Flows

### Opening a Pi-mode coding session
A single trusted operator, at their own terminal, opens the v0.55 `tui` channel in
the **main session**. They request Pi-mode for the working repo; Security Central
resolves the **local-coding operator** tier (at sandbox **Level 1**) *only* because
the caller is the trusted operator, in the main session, on `tui`. The session
loads a coding system prompt **+ four tool definitions under the combined token
budget**, assembled from the `AGENTS.md` hierarchy + the v0.03 skill loader,
exposing exactly the four boundary actions. The objective is "pair on this repo"
(stage 3 trivial); step proposal (stage 4) is trivial; authorization (stage 5)
runs cheaply at the tier — *present, not absent* — and policy/redaction/trace/audit
still run at Level 1.

### The read / write / edit / bash loop at the local-coding tier
The operator types an instruction. The model proposes a `read`/`write`/`edit`/`bash`
call as **advisory output** — it does not execute anything itself. Each proposed
call becomes a signal → `Actions.Runner.run/3` → Security Central gate. At the
local-coding tier for the trusted local caller the gate runs *cheap* (no per-action
confirmation prompt) but **runs** — it still evaluates policy, redacts, and writes
the trace at Level 1. `read`/`write`/`edit` are confined to
`coding.workspace.cwd_jail`, including normalized path and symlink checks; `bash`
is host execution behind the registered action: raw shell **only at this tier**
(argv-only otherwise), cwd-jailed and timed out (reuse `skill_script_runner`); the
host/container, not OTP/BEAM, is the security boundary. Context is gathered through
**chunked reads** (offset/limit) and file artifacts, not whole-file ingestion.

### Streamed diffs
`write`/`edit` produce the split tool-result payload (`model_payload` vs.
`surface_payload`, ADR 0067 extending ADR 0029): the model receives the clean
`model_payload`; the terminal renders the `surface_payload` diff through the
unified renderer (ADR 0030) / `owl`. The streaming is **net-new v0.57** work — a
coding diff live-region renderer that renders the diff by progressively parsing the
tool-call **arguments** as they arrive (not by streaming the tool result; v0.55
ships only the static split). The `surface_payload` never leaks back into model
context.

### Mid-session model switch
The operator invokes the model-switch affordance (a `tui` slash command such as
`/model <name>`). The session's req_llm message history is serialized and handed to
the new model; cross-provider history conversion is **best-effort** (provider
reasoning/tool-call shapes differ). M4 verifies the concrete API through Context7
when available and otherwise through vendored `deps/req_llm` docs/source. The
conversation continues with **no change to the authority path** — same runner, same
Security Central, same trace.

### Completion
For effectful or generated-code work the model never "decides it's done."
Acceptance is **deterministic** (evidence + confirmation), the same posture Allbert
took against Hermes-style model-confidence completion.

## Data Flow
```text
operator terminal input (v0.55 tui channel, supervised GenServer)
  -> normalize -> Identity.resolve (list-shaped channels.tui.identity_map)
  -> Channels.derive_session_id (main session) -> Runtime.submit_user_input
  -> coding loop: model emits advisory read/write/edit/bash proposal
       (sub-1000-token prompt+tool-defs, chunked-read context, 4 default tools)
  -> signal -> Actions.Registry -> Actions.Runner.run/3
  -> Security Central gate  [local-coding operator tier @ Level 1: present-but-cheap]
       trusted operator + main session + tui  -> cheap gate, raw-shell bash, full trace
       channel-originated / scheduled / generated-code -> full gate, argv-only (no tier)
  -> boundary action executes (read[chunked] | write[cwd-jail] | edit[cwd-jail] | bash[Level 1])
  -> split tool-result (ADR 0067/0029): model_payload + surface_payload diff
  -> terminal render via NET-NEW v0.57 coding diff live-region renderer (ADR 0030 / owl); progressive
       tool-argument streaming; model sees only model_payload
  -> channel_events (dedupe, redacted summary) + trace + markdown-first memory
  -> [operator] mid-session model switch -> req_llm message-history handoff (no authority change)
```

## How the gates stay present-but-cheap at the tier
The local-coding operator tier changes the *cost* of the gate, never its
*existence* or the *sandbox level*. For the single trusted operator in the main
session on `tui`, Security Central skips the per-action confirmation prompt **but
still runs at Level 1**: it evaluates policy, enforces the `bash` cwd-jail +
timeout, redacts/truncates `channel_events` summaries, and writes the full trace.
The tier is a **named, auditable trust tier** (ADR 0056 lineage), never the default
— every other caller (channel-originated inbound, scheduled jobs, generated-code
sessions) gets the full gate and argv-only `bash` with no tier escalation. This is
the rationale's "stages 3–5 intact; gates present but cheap" made concrete, without
relaxing the sandbox level.

## Authority Invariants
- Pi-mode calls only `Runtime.submit_user_input/1` + `Actions.Runner.run/3`; no
  direct Security Central / confirmation-store / Resource Access access (ADR 0016).
- Advisory model output never authorizes execution; every effect routes through the
  runner + Security Central.
- The local-coding operator tier resolves only for trusted operator + main session
  + `tui` (at sandbox Level 1); never the default; unreachable by
  channel-originated, scheduled, or generated-code sessions.
- `bash` is cwd-jailed + timed out; raw shell resolves only at the tier (argv-only
  otherwise); OTP/BEAM supervision is not an OS security boundary — the
  host/container is.
- Acceptance for effectful/generated-code work is deterministic (evidence +
  confirmation); no model-confidence completion.
- The split-result UI payload never leaks into the model payload.
- Mid-session model switch changes the model only; the authority path is unchanged.
- Delegation uses restricted-registry delegate agents; no bash-spawned sub-agent.

## Smoke Checklist
- Pi-mode opens on `tui` for the trusted operator; a read/write/edit/bash turn
  drives a full loop with streamed diffs and `channel_events` dedupe.

  Level 1, raw-shell `bash`); a simulated channel-originated or generated-code
  caller gets the full gate and argv-only `bash`.
- `write`/`edit` refuse path traversal, symlink escape, and destinations outside
  `coding.workspace.cwd_jail`; the same simulated non-tier callers get the full
  gate for file effects.

  no `surface_payload` in model context.
- A mid-session model switch preserves session history and changes no authority.
