## Locked Decisions
Recorded in `v0.57-plan.md`. Pi-mode is a channel/app on the one spine with no
extra authority; four default boundary actions (`read`/`write`/`edit`/`bash`)
through `Actions.Runner.run/3` + Security Central; a named "local coding / sandbox
level 0" tier (single trusted operator, main session, `tui`) that is never the
default and never for channel-originated or generated-code sessions; deterministic
acceptance (no model-decides-done); sub-1000-token prompt + full-file context;
streamed diffs via the v0.55 split payload; mid-session model switch over req_llm
`Context`; delegate agents (no bash-spawned sub-agents).

## Main User / Operator Flows

### Opening a Pi-mode coding session
A single trusted operator, at their own terminal, opens the v0.55 `tui` channel in
the **main session**. They request Pi-mode for the working repo; Security Central
resolves the **"local coding / sandbox level 0"** tier *only* because the caller is
the trusted operator, in the main session, on `tui`. The session loads a
sub-1000-token coding system prompt assembled from the `AGENTS.md` hierarchy + v0.03
skill loader, exposing exactly the four boundary actions. The objective is "pair on
this repo" (stage 3 trivial); step proposal (stage 4) is trivial; authorization
(stage 5) runs cheaply at level 0 — *present, not absent*.

### The read / write / edit / bash loop at level 0
The operator types an instruction. The model proposes a `read`/`write`/`edit`/`bash`
call as **advisory output** — it does not execute anything itself. Each proposed
call becomes a signal → `Actions.Runner.run/3` → Security Central gate. At level 0
for the trusted local caller the gate runs *cheap* (no per-action confirmation
prompt) but **runs** — it still evaluates policy, redacts, and writes the trace.
`bash` is policy-bounded host execution behind the registered action (OTP/BEAM is
not the security boundary). The full-file context discipline gathers full files
into the spec rather than snippets.

### Streamed diffs
`write`/`edit` produce the **split tool-result payload** (ADR 0029): the model
receives the clean text payload; the terminal renders the UI/diff payload through
the unified renderer (ADR 0030) / `owl`, streamed as chunks arrive. The UI payload
never leaks back into model context.

### Mid-session model switch
The operator invokes the model-switch affordance. The session's req_llm canonical
`Context` (ADR 0047/0051) is handed to the new model; the conversation continues
with no loss of context and **no change to the authority path** — same runner, same
Security Central, same trace.

### Completion
For effectful or generated-code work the model never "decides it's done."
Acceptance is **deterministic** (evidence + confirmation), the same posture Allbert
took against Hermes-style model-confidence completion.

## Data Flow
```text
operator terminal input (v0.55 tui channel, supervised GenServer)
  -> normalize -> Identity.resolve (channels.tui.identity_map)
  -> Channels.derive_session_id (main session) -> Runtime.submit_user_input
  -> coding loop: model emits advisory read/write/edit/bash proposal
       (sub-1000-token prompt, full-file context, 4 default tools)
  -> signal -> Actions.Registry -> Actions.Runner.run/3
  -> Security Central gate  [local coding / sandbox level 0: present-but-cheap]
       trusted operator + main session + tui  -> cheap gate, full trace
       channel-originated / scheduled / generated-code -> full gate (no level 0)
  -> boundary action executes (read | write | edit | bash[policy-bounded])
  -> split tool-result (ADR 0029): model text payload  +  UI/diff render payload
  -> terminal render of streamed diff (ADR 0030 / owl) ; model sees clean payload
  -> channel_events (dedupe, redacted summary) + trace + markdown-first memory
  -> [operator] mid-session model switch -> req_llm Context handoff (no authority change)
```

## How the gates stay present-but-cheap at level 0
Level 0 changes the *cost* of the gate, never its *existence*. For the single
trusted operator in the main session on `tui`, Security Central skips the
per-action confirmation prompt **but still runs**: it evaluates policy, enforces
the `bash` policy bound, redacts/truncates `channel_events` summaries, and writes
the full trace. Level 0 is a **named, auditable tier**, never the default — every
other caller (channel-originated inbound, scheduled jobs, generated-code sessions)
gets the full gate with no level-0 escalation. This is the rationale's "stages 3–5
intact; gates present but cheap" made concrete.

## Authority Invariants
- Pi-mode calls only `Runtime.submit_user_input/1` + `Actions.Runner.run/3`; no
  direct Security Central / confirmation-store / Resource Access access (ADR 0016).
- Advisory model output never authorizes execution; every effect routes through the
  runner + Security Central.
- Level 0 resolves only for trusted operator + main session + `tui`; never the
  default; unreachable by channel-originated, scheduled, or generated-code sessions.
- `bash` is policy-bounded; OTP/BEAM supervision is not an OS security boundary.
- Acceptance for effectful/generated-code work is deterministic (evidence +
  confirmation); no model-confidence completion.
- The split-result UI payload never leaks into the model payload.
- Mid-session model switch changes the model only; the authority path is unchanged.
- Delegation uses restricted-registry delegate agents; no bash-spawned sub-agent.
