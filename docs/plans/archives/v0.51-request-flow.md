## Locked Decisions
Recorded in `v0.51-plan.md`. Summary:
1. Three public surfaces ship: MCP server, OpenAI-compatible HTTP API, ACP
   server. AG-UI/A2UI stays parked post-1.0.
2. All three are adapters over Allbert's runtime/action boundaries:
   conversational prompts enter through `Runtime.submit_user_input/1`; exposed
   tool/effectful calls execute through `Actions.Runner.run/3`; no surface gets
   more authority than a local workspace user; none self-approves.
3. Default-off, default-empty exposure; tools/namespaces opt-in via Settings.
4. Inbound trust tier (ADR 0055): new `:public_surface_call_inbound` permission
   (floor `:needs_confirmation`); HTTP surfaces require per-client Settings-Central
   tokens (encrypted, never logged, `secure_compare`, rotatable/revocable) behind
   an API secure-header posture + net-new inbound rate-limit + redaction; stdio
   surfaces under ADR 0009.
5. Stateless clients read confirmation-gated results via a poll-by-id readback
   action (`:agent`-exposable, client-scoped, never before approval).
6. OpenAI adapter flattens `messages[]` into one bounded conversational turn and
   rejects client-supplied `tools`/`functions` and non-text content in v0.51
   (no client-driven tool or media authority).
7. `hermes_mcp` for MCP protocol framing only; Allbert owns ingress + authority.
   v0.51 targets the pinned Hermes-supported MCP versions (`2025-03-26` /
   `2025-06-18` where available), not unverified `2025-11-25` parity.
8. Public-surface operator configuration comes only from Settings Central. Do
   not add Application env, process-env, CLI-flag, protocol metadata, or Hermes
   transport-option config paths for enablement, clients, tokens, rate limits,
   body caps, tool allowlists, namespace allowlists, models, or sessions.

## Protocol Subset

v0.51 implements a text-first public protocol subset. This is intentional:
provider profiles can describe multiple media modalities, but executable media
flows remain capability-specific (`speech_to_text`, `vision_input`,
`artifact_read`, etc.) and do not become public just because a protocol can
carry image/audio/resource content.

- MCP stdio: stdout is JSON-RPC/MCP messages only; logs go to stderr. Tool and
  resource exposure comes only from the shared exposure filter. Advertise only
  initialize/lifecycle, `tools/list`, `tools/call`, `resources/list`, and
  `resources/read`; no prompts, resource templates, subscriptions, or
  `listChanged` unless M3/M4 implement and test them.
- MCP artifacts: not served as resources in M3/M4. If artifact serving is added,
  the resource adapter must resolve through Artifacts Central /
  `:artifact_read`, with Settings Central allowlists and redaction/bounds tests;
  protocol resource metadata or `artifact://` URIs alone are never authority.
- MCP streamable HTTP: M4 implements JSON-only `POST /mcp` for JSON-RPC
  `initialize`, `tools/list`, `tools/call`, `resources/list`, and
  `resources/read`; SSE/streaming is not claimed. `mcp-protocol-version` is
  optional but, when present, must be `2025-06-18` or `2025-03-26`;
  unsupported newer versions are rejected before runtime work.
  `mcp-session-id` is echoed when provided, but M4 does not create durable HTTP
  MCP sessions. Authenticated `DELETE /mcp` returns explicit `405`. Origin
  headers are allowed only for loopback origins against loopback request hosts;
  absent Origin is accepted. HTTP ingress is Allbert-owned Phoenix/Plug request
  handling first; do not mount `Hermes.Server.Transport.StreamableHTTP.Plug`
  directly or start a Hermes-owned HTTP listener outside Allbert's
  body-cap/auth/rate-limit/header pipeline.
- OpenAI-compatible HTTP: `GET /v1/models` lists only
  `openai_api.models_enabled` aliases from Settings Central. `POST
  /v1/chat/completions` accepts only `model`, `messages`, `stream`, `user`,
  `response_format`, and Allbert extension fields `allbert_user_id`,
  `allbert_thread_id`, and `allbert_session_id`. Accepted `messages` are string
  content or text content parts with roles `system`, `developer`, `user`, or
  `assistant`. Reject `tools`, `functions`, `tool_calls`, `tool_choice`,
  non-text content parts (`image_url`, `input_audio`, file/resource-like parts),
  `modalities`, `audio`, unsupported `response_format`, `stream_options`,
  incoming `tool` role messages, assistant messages containing `tool_calls` or
  `function_call`, and request fields that change retention, routing, authority,
  output semantics, provider behavior, artifact access, or media handling unless
  explicitly implemented.
- ACP stdio: v0.51 implements ACP protocol version `1` over newline-delimited
  stdio JSON-RPC. Accept text content blocks only. `initialize` advertises no
  image/audio/embedded-resource, filesystem, terminal, MCP, load/resume, or
  additional-directory capability. `cwd` is inert metadata; non-empty
  `additionalDirectories`, non-empty `mcpServers`, and `permissionMode` are
  rejected. Unsupported session or content capabilities are not advertised and
  are rejected if sent. Client-supplied `mcpServers` are never imported into
  Allbert MCP client settings or discovery.

## Main Flows
- Operator enables a surface + specific tools/namespaces in Settings Central.
- External client connects (MCP stdio/HTTP, OpenAI `/v1/chat/completions`, or ACP
  stdio) and lists/calls exposed tools / reads exposed resources.
- A confirmation-needing call returns `:confirmation_pending` (+ id); operator
  approves on workspace/origin channel; client cannot self-approve.

## Data Flow
```text
inbound request (MCP tool call | OpenAI chat completion | ACP session/prompt)
  -> surface adapter (Hermes.Server stdio | /mcp controller | /v1 controller | ACP stdio)
  -> body/stdio frame bounds, protocol subset validation
  -> HTTP? token auth + rate-limit + secure-header policy before runtime work
  -> auth/exposure filter (deny-before-allow; Registry exposure==:agent ∩ surface allowlist;
     Memory.Namespaces.app_namespaces/0 ∩ allowlist; system namespaces never)
  -> conversational prompt? -> Runtime.submit_user_input/1
       -> intent/objectives -> Runner only for selected effectful actions
  -> MCP exposed tool call? -> Actions.Runner.run/3
  -> Security Central -> Resource Access -> confirmations
  -> Runtime.Redactor -> surface-shaped response
     (effectful + floor :needs_confirmation -> :confirmation_pending + id)

readback poll (stateless client retrieves a confirmation-gated result):
  inbound get_public_call_result(id)
  -> auth/exposure filter (action exposure==:agent)
  -> client-scope check from caller context, not params
       (caller originated id? else deny — no cross-client leak)
  -> public_protocol_call_results ownership row
  -> operator approved? no -> pending (never a result before approval)
                         yes -> Runtime.Redactor -> approved_with_result | denied
                         expired -> expired (no result bytes)
```

## Authority Invariants
- Metadata/protocol/transport never grant authority (ADR 0038). Runtime turns
  may propose work; the runner + Security Central remain the effectful authority
  boundary.
- No external-client self-approval; confirmation-decision actions never exposed;
  ACP `session/request_permission` requests are advisory UI signals only, and
  ACP permission responses never authorize Allbert execution.
- Readback never returns a result before operator approval, and is client-scoped
  (a client reads only ids it originated).
- HTTP tokens encrypted at rest, never logged, compared with `secure_compare`,
  rotatable/revocable. Absent/invalid/revoked tokens are denied before runtime
  work. Bearer-token replay denial is not claimed unless nonce/signature or
  idempotency semantics are implemented and tested.
- JSON APIs use the ADR 0055 API secure-header posture (`default-src 'none'`/
  `frame-ancestors 'none'`), distinct from the ADR 0025 §3 browser CSP baseline
  (a category mismatch for JSON APIs).
- Non-exposable internals (settings/secrets/signals/traces/confirmation store/
  registry/`:internal` actions/system namespaces) rejected as config errors.
- Artifact bytes/metadata are not exposed as MCP resources by M3/M4 or through
  the M5 OpenAI-compatible surface. Existing artifact actions are internal;
  future MCP artifact serving must go through Artifacts Central and
  `:artifact_read`, not direct store access.
- Settings/profile/credential actions can currently be `:agent`; they are still
  denied before allowlisting by capability id, execution mode, and permission
  class (`:settings_read`, `:settings_write`, `:secret_write`, confirmation/
  trace/registry/local-process modes).
- Dynamic/generated/plugin actions require both existing reviewed/gated
  authority and explicit per-surface allowlisting before public exposure.

  `openai_api.clients.*`.
- `public_protocol.schema_version`, `public_protocol.result_readback_ttl_ms`,
  `public_protocol.max_body_bytes`.
- `acp_server.schema_version`, `acp_server.enabled`,
  `acp_server.stdio.enabled`, `acp_server.tools_enabled`,
  `acp_server.memory_namespaces_enabled`, `acp_server.session.*`.
- `permissions.public_surface_call_inbound`.

HTTP client entries reference Settings Secrets for token material and carry
per-client rate-limit settings. Raw token material is never valid setting output,
trace content, audit content, or test fixture evidence.

MCP HTTP authentication uses `x-allbert-client-id` plus
`authorization: Bearer <token>`. The controller accepts a request only after
Settings Central confirms `mcp_server.enabled`,
`mcp_server.streamable_http.enabled`, the client entry, token ref, and
per-client rate limit. `mcp_server.stdio.enabled` does not authorize HTTP, and
`mcp_server.streamable_http.enabled` does not authorize stdio.

No public protocol enablement or client/auth/rate-limit/body-cap/tool/resource
configuration is read from Application env, process env, ad hoc CLI flags, MCP
client metadata, ACP session metadata, OpenAI request fields, or Hermes
transport options.

Validation rejects unknown keys, invalid client ids, invalid port/rate-limit
ranges, non-local bind hosts unless a later hardening plan enables them,
allowlisted tools that fail the public exposure deny rules, namespaces outside
`Memory.Namespaces.app_namespaces/0`, and model aliases that do not resolve to
configured profiles.

Token operator surface:
`mix allbert.public_protocol token create|rotate|revoke|list --surface <mcp_http|openai_api> --client <id>`.
Only `create` and `rotate` print the new raw token, once.

## Response Shapes

- MCP: list responses expose only bounded enabled tools/resources. Protocol,
  parse, version, auth, and exposure failures are JSON-RPC/MCP errors.
  Action-level denial returns an error tool result. Confirmation-required calls
  return a successful tool result with `status: "confirmation_pending"` and
  `public_call_id`.
- OpenAI-compatible: `/v1/models` returns an OpenAI-shaped list. Chat success
  returns `chat.completion`; streaming returns `chat.completion.chunk` SSE
  deltas and `[DONE]`. M5 emits one bounded completion chunk for the completed
  turn; it does not claim token-by-token provider streaming. Pending
  confirmation uses `allbert_status: "pending"` and
  `allbert_public_call_id` extension fields. Errors use the OpenAI
  `%{"error" => ...}` body shape for validation, auth, authorization, and
  rate-limit failures. This is not full OpenAI API or Responses API parity.
- ACP: initialize/session setup advertises only implemented text-session
  capabilities. Prompts emit `session/update` text chunks and return
  `stopReason: "end_turn"` or pending/readback ids. Confirmation-pending turns
  may emit an advisory `session/request_permission` request for client UI, but
  the result remains operator/readback controlled. Errors are JSON-RPC-shaped
  and redacted.

## Smoke Checklist
- Each surface: enable one tool + one namespace; list/call/read; confirm
  unexposed tools/namespaces are invisible.
- Each surface: an effectful call returns confirmation-pending; operator approves
  on workspace; client cannot self-approve. Client polls `get_public_call_result`:
  `pending` before approval, `approved_with_result` after; a second client's id
  is denied.
- HTTP surfaces: token required; bad/revoked token denied; raw token redacted;
  rate-limit enforced before runtime work; Origin/session/version behavior
  matches the documented transport subset.
- OpenAI-compatible surface: success, pending, streaming, and error responses
  match the documented shapes, including `allbert_` pending fields and
  OpenAI-shaped errors.
- OpenAI/ACP: non-text media/resource content is denied unless a later
  capability-specific plan exposes it.
- ACP: `cwd`, `additionalDirectories`, `mcpServers`, and `permissionMode` do not
  create filesystem roots, start MCP clients, or bypass confirmation.
  Confirmation-pending turns may send ACP `session/request_permission` for
  advisory client display; client responses are ignored as Allbert execution
  authority.
- No public AG-UI/A2UI endpoint responds.
