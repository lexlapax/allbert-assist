## Purpose

v0.50b adds **Artifacts Browser**: the operator-facing browsing repository for
Artifacts Central, shipped as a plugin/app (not core), modeled on StockSage and
the `allbert.browser` plugin. It lets an operator browse, search, filter, and
inspect stored artifacts — by type, origin, thread, and date — from the
workspace LiveView surface, a deep-linkable detail page, and the command line.

It owns **no store authority**. It reads the v0.50 core store exclusively through
registered `:artifact_read` actions (`list_artifacts`, `get_artifact`,
`artifact_threads`, `artifact_doctor`), renders metadata-only (raw bytes stay
redacted per ADR 0053), and routes delete through the core confirmation-gated
`delete_artifact` action. A content address or a rendered row grants nothing.

## Readiness Sweep

Status: complete; v0.50 has shipped the core read actions and v0.50b is
released as `v0.50.1`.

- **Surface fit.** Reuses the established plugin/app + Surface DSL pattern: an
  `AllbertAssist.Plugin` package, an `AllbertAssist.App` + `SurfaceProvider`
  contributing a `:canvas_panels` panel hydrated by `workspace_panel_surfaces/1`
  (the `allbert.browser` Results panel is the direct template), an
  `/apps/artifacts/<sha>` page LiveView (route in the core router, module
  plugin-owned, rendered via the host `Surface.Renderer` like StockSage's
  `AnalysisLive`), and a `mix allbert.artifacts` CLI that calls
  `Actions.Runner.run/3` with `channel: :cli` (the `mix allbert.browser` task is
  the template).
- **Authority fit.** A plugin contributes data only and acquires no authority
  (ADR 0017). The browser cannot define the `artifact://` scheme, register
  permissions, add a Home root, or own the sensor — all of that is v0.50 core.
  It reads through `:artifact_read` and writes nothing except via the core
  confirmation-gated `delete_artifact`.
- **Provenance fit.** Browse-by-thread and the reverse "which thread made this"
  lookup consume the v0.50 `artifact_thread_links` query actions; no new linking
  primitive is added here.
- **Past-plan fit.** No core changes except a single page-route line in the web
  router (the web layer is host-owned). A panel-only fallback exists if a
  zero-core-edit build is preferred.

## Implementation Seams

The browser is intentionally a plugin/app shell over core actions.

- **Plugin id and modules.** Use `plugins/allbert.artifacts/` with
  `allbert_plugin.json` `plugin_id: "allbert.artifacts"`, module
  `AllbertArtifacts.Plugin`, app module `AllbertArtifacts.App`, surface provider
  `AllbertArtifacts.SurfaceProvider`, panel module
  `AllbertArtifacts.Panels.Browser`, and task module
  `Mix.Tasks.Allbert.Artifacts`. App id is `:allbert_artifacts`.
- **Reference templates.** Copy the plugin/action/panel/CLI structure from
  `plugins/allbert.browser/` and the host-owned page route pattern from
  StockSage (`/apps/stocksage/analyses/:id` route in the core router with a
  plugin-owned LiveView module).
- **Core router edit.** Add exactly one route in
  `AllbertAssistWeb.Router`: `live "/apps/artifacts/:sha",
  AllbertArtifactsWeb.ArtifactLive, :show` under the host browser pipeline.
  The route validates lowercase 64-character SHA-256 before calling core
  actions; invalid SHA values render not-found/error state without probing the
  store.
- **Action calls.** Panel/page/CLI call only `Actions.Runner.run/3` for
  `list_artifacts`, `get_artifact`, `artifact_threads`, `artifact_doctor`, and
  `delete_artifact`. They never call `AllbertAssist.Artifacts.Store`,
  `Runtime.Paths.artifacts_root/0`, or raw filesystem APIs.
- **Panel failure posture.** If a core read action fails, the panel renders a
  redacted unavailable/error node through the host catalog and logs no raw
  paths/bytes. A failed browse read grants no fallback store access.
- **Surface catalog.** Declare any plugin-specific components
  (`:artifact_row`, `:artifact_filter_summary`, `:artifact_provenance_list`) in
  `AllbertArtifacts.App.surface_catalog/0`; host validation must pass before
  rendering.
- **Release gate and eval inventory.** Add `:v050b` to
  `AllbertAssist.SecurityFixtures.EvalInventory`'s milestone type, add
  `:artifact_browser` to the eval surface vocabulary, and add
  `release.v050b` to `Mix.Tasks.Allbert.Test` usage, dispatch,
  `@release_v050b_steps`, evidence cleanup, secret scan, and
  `allbert_test_task_test.exs` coverage. Evidence writes under
  `<ALLBERT_HOME>/release_evidence/v050b/`.
- **Browser fixture seeding.** Add a deterministic browser-smoke helper (for
  example `scripts/v050b_artifacts_browser_smoke.exs`) that seeds an artifact
  fixture through core `put_artifact`, prints the fixture SHA/thread id, and
  records that pair in release evidence. Browser validation must use the printed
  SHA; `<fixture-sha>` is not allowed to remain an unresolved placeholder.
- **Host asset scan.** If plugin-owned web modules/components introduce Tailwind
  classes outside host-rendered catalog nodes, add a scoped Tailwind `@source`
  entry in `apps/allbert_assist_web/assets/css/app.css` for the
  `plugins/allbert.artifacts/` web namespace. If the browser renders entirely
  through existing host catalog components, record that no CSS source update was
  needed.

## Locked Decisions

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Packaging | A plugin/app `plugins/allbert.artifacts/` (`kind: app`), implementing `Plugin` + `App` + `SurfaceProvider`; not core. | Building into core bloats it and skips the app contract; a plugin owning the store would invert the channel/MCP/export dependency. |
| 2 | Store access | Read-only through core `:artifact_read` actions (`list_artifacts`, `get_artifact`, `artifact_threads`, `artifact_doctor`); delete via the core confirmation-gated `delete_artifact`. No direct object-store access. | Direct FS access would bypass Security Central + redaction and duplicate store logic. |
| 3 | Workspace surface | A `:canvas_panels` panel declared in `App.surfaces/0` and hydrated by `workspace_panel_surfaces/1`; rendered through the host catalog (no model-generated UI code). | Mounting its own route/socket would break the host-owned web boundary (ADR 0024). |
| 4 | Detail page | An `/apps/artifacts/<sha>` page LiveView; route registered in the core web router, module plugin-owned, rendered via the host `Surface.Renderer`. | Omitting it loses deep-linkable detail; self-mounting violates the host-owned router. |
| 5 | CLI | `mix allbert.artifacts list|show|threads|doctor|rm` as a thin shell over `Actions.Runner.run/3` (`channel: :cli`); the task owns no logic and no authority. | Task-side logic would bypass the action boundary. |
| 6 | Search/filter | In-memory/SQLite filtering over the core list/query actions (by type, origin, thread, date); no new index beyond the v0.50 metadata index + `artifact_thread_links`. | A bespoke search index duplicates the core index and risks drift. |

## Surfaces

- **Workspace panel** (`:browser`-style): a `:canvas_panels` Artifacts panel
  listing recent/filtered artifacts as catalog nodes (type icon, size, origin,
  thread link, created-at), `visible_when: :operator_opened`; row navigation
  can be wired after the route and filter controls exist. Metadata only; no raw
  bytes in assigns.
- **Detail page** `/apps/artifacts/<sha>`: implemented in M2 with full redacted
  metadata, provenance (origin + linked threads/messages with role), retention
  state, and a confirmation-gated remove action.
- **CLI** `mix allbert.artifacts`: M3 implements `list [--limit N]`,
  `show <sha>`, `threads <sha>`, `doctor`, and `rm <sha>` (confirms). M4 adds
  `list --type --origin --thread --since --retention --lifecycle --limit`
  filters over the core `list_artifacts` action.

## Non-Goals

- No store ownership (scheme, permissions, Home root, settings authority, sensor
  all stay v0.50 core).
- No direct object-store/filesystem access; all reads go through actions.
- No content interpretation/preview rendering of raw bytes; metadata + redacted
  references only.
- No new provenance primitive; consumes the v0.50 `artifact_thread_links` query.

## Definition Of Done

v0.50b is done when:

- a `plugins/allbert.artifacts/` app contributes a workspace Artifacts panel, an
  `/apps/artifacts/<sha>` detail page, and a `mix allbert.artifacts` CLI;
- all three read the store only through core `:artifact_read` actions and route
  delete through the core confirmation-gated `delete_artifact`, rendering
  metadata-only with raw bytes redacted;
- browse/search/filter by type/origin/thread/date works over the core list/query
  actions, including by-thread and reverse-thread lookup;
- the plugin grants no authority and owns no store internals;
- the `:v050b` eval set + `release.v050b` pass on fixtures;
- browser validation uses a seeded real SHA from release evidence and records
  screenshots;
- roadmap/vision/agent-context-map/ADR 0054/CHANGELOG reflect the shipped state.
