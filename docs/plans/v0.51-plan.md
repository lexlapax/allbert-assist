Source: `docs/archives/version-1.0-planning-03.md`, post-v0.37 planning pass;

## Purpose

Expose Allbert as a public **agent surface** across three protocols so external
agent runtimes can call Allbert's registered actions as tools and read enabled
memory namespaces as resources:

1. **MCP server** — Claude Desktop, Cursor, ChatGPT MCP, JetBrains MCP, other
   MCP agents. Symmetric to the v0.40 MCP client work. (The original v0.52b
   scope.)
2. **OpenAI-compatible HTTP API** — `POST /v1/chat/completions` (+ optional
   `/v1/models`, SSE streaming) so the large OpenAI-API client ecosystem can
   reach Allbert.
3. **ACP server** — Agent Client Protocol (editor agents such as Zed) over
   stdio JSON-RPC.

All three are **thin adapters over Allbert's existing runtime and action
boundaries**: conversational requests enter through `Runtime.submit_user_input/1`;
exposed tool/effectful calls execute through `Actions.Runner.run/3`; none gains
more authority than a local workspace user; none can self-approve confirmations.
The public **AG-UI/A2UI bridge stays parked** post-1.0 (remains internal/
test-only; ADR 0044 rationale holds).

  `Codec` (Hermes message wrapper), `Transport` (stdio `Port` + HTTP/SSE),
  `Client` (protocol lifecycle, `@protocol_version`), trust/audit substrate —
  reusable for server mode.
- `hermes_mcp` 0.14.1 (already a dep): provides **MCP server** support
  (`use Hermes.Server`, `register_tool/3`, `handle_tool/3`,

- `Actions.Registry` + `Capability` (name/permission/`exposure`/execution_mode/

- `Runtime.submit_user_input/1`, `Actions.Runner.run/3`, Security Central,

- Web `:api` pipeline + `ContentSecurityPolicy` plug (`apps/allbert_assist_web`).
- ADR 0044 amendment accepted before implementation closeout.

## Operator-Visible Win

An operator configures an external client — an MCP client (Claude Desktop), an
OpenAI-API client, or an ACP editor (Zed) — to reach Allbert. The operator
explicitly enables the tools and memory resources the external client may see.
Effectful calls trigger normal Approval Handoff through the operator's workspace
or origin channel; **the external client never approves its own confirmations**.

## Planned Interfaces

- Settings namespaces `mcp_server.*`, `openai_api.*`, `acp_server.*` (each:
  enabled flag default **off**; per-tool include filters default **empty**;
  per-namespace memory-resource include filters default **empty**; HTTP token
  refs into Settings Secrets where applicable). Mirror the v0.40 `mcp.servers.*`
  client shape.
- MCP server via `Hermes.Server` (+ `Hermes.Server.Registry`); stdio default,
  optional streamable-HTTP via `StreamableHTTP.Plug` mounted **inside** Allbert's
  authenticated/CSP/rate-limited ingress pipeline (not a second ingress).
- OpenAI-compatible API on the web `:api` pipeline: `/v1/chat/completions`
  (+ `/v1/models`), each request -> one bounded Allbert turn through
  `Runtime.submit_user_input/1`; any selected effectful action still reaches
  `Actions.Runner.run/3`; SSE streaming reuses the v0.40 `Transport`
  SSE-framing pattern.
- ACP server: stdio JSON-RPC adapter (`initialize`/`session/new`/`session/prompt`/
  `session/request_permission`) reusing the v0.40 stdio `Port` pattern under
  ADR 0009 bounds; `session/prompt` submits through `Runtime.submit_user_input/1`
  and permission requests map to advisory Approval Handoff state.
- Shared **exposure-filter** module backed by `Actions.Registry`
  (`exposure == :agent` ∩ operator allowlist) and `Memory.Namespaces`
  (app namespaces ∩ allowlist; `system_namespaces` never exposed).
- Shared inbound surface policy (e.g. `:public_surface_call_inbound` for
  exposed tool/effectful calls, floor `:needs_confirmation`), applied across
  all three surfaces without treating ordinary conversational prompts as
  authority grants.
- Workspace status panel listing connected external clients per surface.

## Exposure Rules (all three surfaces)

- A registered action becomes an exposable tool only when `exposure == :agent`
  AND explicitly listed in the surface's `tools_enabled` (default empty).
- An app memory namespace becomes a resource only when explicitly listed in
  `memory_namespaces_enabled` (default empty).
- Tool descriptions, parameter schemas, and resource summaries are bounded and
  redacted before exposure.
- **Not exposable** (config error if attempted, across all surfaces): settings,
  secrets, raw runtime signals, raw trace bodies, confirmation storage,
  plugin/registry internals, `:internal`-exposure actions, system memory
  namespaces. **Confirmation-decision actions (`approve_confirmation`/
  `deny_confirmation`) are never exposable** — this is the self-approval-denial
  enforcement point.

## Authority Rules (all three surfaces)

- Inbound requests **never** receive more authority than a local workspace user.
- External clients **cannot approve their own confirmations**. A confirmation-
  needing call returns a `:confirmation_pending` response (MCP/ACP) or an
  OpenAI-shaped equivalent carrying the confirmation id; the operator approves
  through their usual surface. `ApprovalHandoff` stays operator-owned.
- ACP `session/request_permission` maps to a confirmation, but the ACP client's
  permission response is **advisory only and never authorizes execution**
  (ADR 0038 "metadata is never authority" applied to ACP).
- All effectful calls route through `Actions.Runner.run/3`, Security Central,
  Resource Access, confirmations, traces, and audits.
- HTTP-bearing surfaces (MCP streamable-HTTP + OpenAI API) require a per-client
  token issued via Settings Central, encrypted at rest, never logged, behind the
  v0.35 CSP baseline + a rate-limit + redaction pipeline. stdio surfaces (MCP
  stdio + ACP) run under ADR 0009 process bounds.
- `hermes_mcp` provides MCP protocol/lifecycle/framing only; Allbert owns ingress
  auth and authority. Native `Req`/`Plug` fallback decided at M1 (mirrors
  ADR 0038 §6) if the Plug cannot be cleanly constrained.

## Security Posture

Three new external-call surfaces. Security evals must cover, per surface:
cross-client identity confusion; prompt-injection escalation to unconfigured
tools; self-approval-attempt denial; resource-scope leakage; HTTP token replay /
redirect / rate-limit; settings/secrets/audit exposure attempts; the ACP
permission-response-not-authoritative rule; and that no public AG-UI/A2UI
endpoint exists.

## Non-Goals

- No public AG-UI/A2UI bridge (stays parked post-1.0).
- No hosted multi-user authorization.
- No external-client self-approval of confirmations.
- No exposure of settings, secrets, raw signals, raw trace bodies, or
  confirmation storage; no auto-enablement of tools/resources from request
  payloads.

## Definition Of Done

v0.51 is done when:

- an external MCP client, an OpenAI-API client, and an ACP editor can each reach
  Allbert in a disposable Allbert Home, seeing only operator-enabled tools and
  memory namespaces;
- a confirmation-needing call on each surface returns `:confirmation_pending`
  (or its OpenAI-shaped equivalent) and the operator approves through their
  workspace; no surface can self-approve;
- conversational prompts enter through `Runtime.submit_user_input/1`; every
  effectful call routes through `Actions.Runner.run/3` + Security Central +
  confirmations; non-exposable internals are rejected as config errors;
- the security eval set (`:v051`, all three surfaces) + `release.v051` pass; the
  AG-UI/A2UI bridge remains internal-only;
- the v0.57 export/import sweep round-trips the three surfaces' settings without
  exposing secrets.
