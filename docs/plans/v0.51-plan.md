## Purpose

Expose Allbert as a public **agent surface** across three protocols so external
agent runtimes can call Allbert's registered actions as tools and read enabled
memory namespaces as resources:

1. **MCP server** — Claude Desktop, Cursor, ChatGPT MCP, JetBrains MCP, other
   MCP agents. Symmetric to the v0.40 MCP client work. (The original v0.52b scope.)
2. **OpenAI-compatible HTTP API** — `POST /v1/chat/completions` (+ optional
   `/v1/models`, SSE streaming) so the large OpenAI-API client ecosystem can
   reach Allbert.
3. **ACP server** — Agent Client Protocol (editor agents such as Zed) over stdio
   JSON-RPC.

The request path is a thin adapter over Allbert's existing boundaries:
conversational requests enter through `Runtime.submit_user_input/1`; exposed
tool/effectful calls execute through `Actions.Runner.run/3`; none gains more
authority than a local workspace user; none can self-approve confirmations. The
**return path for confirmation-gated calls is genuinely new substrate** — a
poll-by-id result readback (§"Result Readback"), since nothing today returns a
post-approval result to a stateless external client. The public **AG-UI/A2UI
bridge and MCP Apps iframe UI stay parked** post-1.0 (ADR 0044).

## Protocol / Content Subset

v0.51 is a public **text-first protocol subset**, not a generic media ingress
release. OpenAI and ACP both define image/audio/resource content forms, but
Allbert already keeps executable media routing capability-specific (v0.48 voice,
v0.49 vision, v0.50 artifacts) and has no catch-all `multimodal` authority.

- MCP server mode exposes only the operator-enabled tools/resources generated
  from the exposure filter. Tool outputs use MCP content blocks, but v0.51
  adapters must not treat embedded image/audio/resource payloads as implicit
  authority to read artifacts, fetch URLs, transcribe audio, or run vision.
  Artifacts are not MCP resources in M3/M4. If a later v0.51 milestone or
  follow-on release serves artifacts over MCP, it must use Artifacts Central
  (`AllbertAssist.Artifacts` and registered `:artifact_read` actions or a
  public-safe adapter over them), Settings Central allowlists, redaction, and
  explicit tests. It must not expose raw artifact store paths, bypass the
  `:artifact_read` permission, or make existing `exposure: :internal` artifact
  actions public by metadata alone.
- MCP v0.51 subset is intentionally small: initialize/lifecycle, `tools/list`,
  `tools/call`, `resources/list`, and `resources/read`. No prompts, resource
  templates, resource subscriptions, or `listChanged` capability are advertised
  in v0.51 unless implemented with tests in M3/M4. `tools/list` and
  `resources/list` either return the complete bounded allowlisted set with no
  `nextCursor` or implement real cursor pagination; they must not emit a cursor
  that cannot be resumed.
- MCP stdio keeps stdout reserved for JSON-RPC/MCP messages; all logs go to
  stderr. The fixture server must fail if non-protocol text reaches stdout.
- MCP streamable HTTP must document and test the exact transport subset it
  implements: POST JSON and/or SSE responses, session id handling when enabled,
  `MCP-Protocol-Version` validation, concurrent stream behavior, DELETE
  session handling or explicit 405, Origin validation, and localhost binding for
  local/private deployments. If the client sends a protocol version newer than
  the Hermes-supported set, Allbert returns a protocol-shaped unsupported-version
  error before token-authenticated runtime work; it does not claim
  `2025-11-25` compatibility until a later dependency audit.
- OpenAI-compatible `/v1/chat/completions` accepts only `model`, `messages`,
  `stream`, `user`, `response_format`, and the Allbert extension fields
  `allbert_user_id`, `allbert_thread_id`, and `allbert_session_id`. `messages`
  may contain string content or text content parts only. It rejects `tools`,
  `functions`, `tool_calls`, `tool_choice`, non-text content parts
  (`image_url`, `input_audio`, file/resource-like parts), `modalities`, `audio`,
  model-driven tool selection, structured output requests that would imply a
  different authority path, and every unimplemented request field that changes
  authority, retention, routing, output semantics, provider behavior, or media
  handling. `response_format: {"type": "text"}` is a no-op; other
  `response_format` values are rejected. Incoming `messages[]` with role `tool`
  and assistant messages containing `tool_calls` or `function_call` are rejected
  with an OpenAI-shaped error.
- OpenAI streaming in M5 is a bounded text SSE facade over the same completed
  turn: one `chat.completion.chunk` event followed by `data: [DONE]`.
  `stream_options` and unsupported streaming options are rejected rather than
  silently ignored.
- ACP supports a v0.51 text-only ACP v1 stdio subset. `initialize` negotiates
  protocol version `1` and advertises no image/audio/embedded-resource,
  filesystem, terminal, MCP, load/resume, or additional-directory capability.
  `session/new` accepts inert `cwd` metadata but rejects non-empty
  `mcpServers`, non-empty `additionalDirectories`, and `permissionMode`.
  `session/prompt` accepts text `ContentBlock` prompts only. ACP image/audio/
  resource/resource-link blocks, `embeddedContext`, filesystem roots, client-
  supplied `mcpServers`, and `permissionMode` are not authority. Allbert
  advertises only the ACP capabilities it implements.

These constraints are implementation requirements, not optional compatibility
notes. Unsupported protocol features must be rejected with bounded, redacted
errors and covered by the v0.51 eval rows.

### Protocol Response / Error Contract

M3-M6 lock response shapes before implementation:

- MCP `tools/list` returns only enabled public tools with `name`, `title` when
  available, bounded/redacted `description`, and JSON `inputSchema` generated
  from `Capability`; `resources/list` returns enabled app namespaces with URI,
  name, description, and MIME type. Protocol/parse/version/auth/exposure failures
  return JSON-RPC/MCP errors. Action-level denial returns a tool result marked as
  an error; confirmation-required calls return a successful tool result whose
  structured payload carries `status: "confirmation_pending"` and
  `public_call_id`.
- OpenAI-compatible success returns Chat Completions-shaped objects:
  `chat.completion` for non-streaming and `chat.completion.chunk` SSE deltas for
  streaming, with `id`, `created`, `model`, `choices`, assistant text content,
  `finish_reason`, and best-effort or `null` usage. Streaming ends with
  `data: [DONE]`. Confirmation-required turns return the same object family with
  Allbert extension fields prefixed `allbert_` (`allbert_status:
  "pending"`, `allbert_public_call_id`) rather than exposing confirmation-store
  internals. Error responses use an OpenAI-shaped
  `%{"error" => %{"message" => ..., "type" => ..., "param" => ..., "code" => ...}}`
  body for `400`, `401`, `403`, and `429`; no stack traces, raw settings, tokens,
  traces, or provider payloads appear.
- `/v1/models` returns a bounded OpenAI-shaped list object for enabled Allbert
  model/profile aliases only. Model names are routing metadata, not authority;
  unsupported model selection is advisory/denied rather than a way to bypass the
  configured runtime.
- ACP `initialize`/session setup advertises only implemented text session
  capabilities. `session/prompt` emits `session/update` agent text chunks and
  returns `stopReason: "end_turn"` or a pending/readback id in ACP-shaped
  response bodies. Confirmation-pending turns may emit an ACP
  `session/request_permission` request for client UI display, but any ACP
  permission response is advisory only and never authorizes Allbert execution.
  ACP errors stay JSON-RPC-shaped and redacted. Client-supplied `mcpServers` are
  rejected and never imported into Allbert MCP client settings, discovery, or
  background connection flows.

## Exposure Rules (all three surfaces)

- The exposure filter is **deny-before-allow**. A registered action becomes an
  exposable tool only when it first passes the non-exposable deny rules, then has
  `exposure == :agent`, then is explicitly listed in the surface's
  `tools_enabled` (default empty). The source of truth is
  `Actions.Registry.capabilities/0`; public exposure is a narrower projection
  over agent-exposable actions, not the intent-agent candidate list itself.
- Current `:agent` actions include settings/profile/credential operations, so
  `exposure == :agent` alone is insufficient. Public filtering must reject by
  capability id, execution mode, and permission class before consulting the
  operator allowlist. `:settings_read`, `:settings_write`, `:secret_write`,
  confirmation-decision/storage modes, raw trace/signal access, plugin/registry
  internals, and local-process/shell-like modes are non-exposable unless a later
  ADR/plan makes a specific action public-safe.
- Dynamic/generated/plugin actions do not become public because metadata says
  `exposure: :agent`; they also need the existing reviewed/gated authority path
  plus an operator allowlist entry for the surface.
- An app memory namespace becomes a resource only when explicitly listed in
  `memory_namespaces_enabled` (default empty). Implementers use
  `Memory.Namespaces.app_namespaces/0` as the source set, never `all/0`
  filtered late; system namespaces such as identity never reach public
  resource enumeration.
- Artifacts are a distinct Artifacts Central authority, not memory namespaces.
  M3/M4 do not serve artifact resources. Any future artifact MCP resource must
  be backed by Artifacts Central and `:artifact_read` policy, plus a Settings
  Central allowlist and explicit byte/redaction/bounds tests.
- Tool descriptions, parameter schemas, and resource summaries are bounded and
  redacted before exposure.
- **Not exposable** (config error if attempted, across all surfaces): settings,
  secrets, raw runtime signals, raw trace bodies, confirmation storage,
  plugin/registry internals, `:internal`-exposure actions, system memory
  namespaces. **Confirmation-decision actions (`approve_confirmation`/
  `deny_confirmation`) are never exposable** — the self-approval-denial point.
  The read-only result-readback action (Locked Decision 4) is the *only* new
  `:agent`-exposable confirmation-related surface, and it is client-scoped.

## Authority Rules (all three surfaces)

- Inbound requests **never** receive more authority than a local workspace user.
- External clients **cannot approve their own confirmations**. A confirmation-
  needing call returns `:confirmation_pending` (MCP/ACP) or an OpenAI-shaped
  equivalent carrying the confirmation id; the operator approves through their
  usual surface; the client retrieves the result via the poll-by-id readback.
- ACP `session/request_permission` maps to a confirmation but is **advisory only
  and never authorizes execution** (ADR 0038 / ADR 0055).
- All effectful calls route through `Actions.Runner.run/3`, Security Central,
  Resource Access, confirmations, traces, and audits.
- HTTP-bearing surfaces (MCP streamable-HTTP + OpenAI API) require a per-client
  token (ADR 0055), behind an API secure-header policy + inbound rate-limit +
  redaction. stdio surfaces (MCP stdio + ACP) run under ADR 0009 bounds.

## Settings Contract

M1 adds schema-versioned Settings Central fragments per ADR 0046. All fragments
are default-off and default-empty; unknown keys are rejected.

- `mcp_server.schema_version: 1`, `mcp_server.enabled: false`,
  `mcp_server.stdio.enabled: false`, `mcp_server.streamable_http.enabled:
  false`, `mcp_server.streamable_http.bind_host: "127.0.0.1"`,
  `mcp_server.streamable_http.port: null`, `mcp_server.tools_enabled: []`,
  `mcp_server.memory_namespaces_enabled: []`, `mcp_server.clients: %{}`.
- `openai_api.schema_version: 1`, `openai_api.enabled: false`,
  `openai_api.path_prefix: "/v1"`, `openai_api.models_enabled: []`,
  `openai_api.tools_enabled: []`, `openai_api.memory_namespaces_enabled: []`,
  `openai_api.clients: %{}`.
- `public_protocol.schema_version: 1`,
  `public_protocol.result_readback_ttl_ms: 3_600_000`,
  `public_protocol.max_body_bytes: 1_048_576`.
- `acp_server.schema_version: 1`, `acp_server.enabled: false`,
  `acp_server.stdio.enabled: false`, `acp_server.tools_enabled: []`,
  `acp_server.memory_namespaces_enabled: []`,
  `acp_server.session.load_enabled: false`,
  `acp_server.session.resume_enabled: false`,
  `acp_server.session.additional_directories_enabled: false`.
- HTTP client entries are maps such as
  `%{"enabled" => true, "token_ref" => "settings-secret://...",
  "rate_limit" => %{"limit" => 60, "period_ms" => 60_000, "burst" => 10}}`.
  Token material lives only in Settings Secrets. Token refs, client ids, and
  rate-limit decisions may be audited; raw tokens never are.
- Validation invariants are part of M1 acceptance: client ids use a bounded
  slug-safe alphabet; ports are `1..65535`; HTTP bind host defaults to
  `127.0.0.1`; non-local binds are rejected unless a later operator-facing
  hardening plan enables them; rate-limit `limit`, `period_ms`, and `burst` have
  minimum/maximum bounds; `tools_enabled` entries must resolve to public-action
  candidates after the deny rules; `memory_namespaces_enabled` entries must
  resolve through `Memory.Namespaces.app_namespaces/0`; model aliases must resolve
  to configured profiles without changing provider authority.
- M1/M4 add one operator token surface:
  `mix allbert.public_protocol token create|rotate|revoke|list --surface <mcp_http|openai_api> --client <id>`.
  `create` and `rotate` print the new bearer token once, while `list`, traces,
  audits, and test evidence show only redacted/token-ref metadata.
- `permissions.public_surface_call_inbound` is added with configured default
  `needs_confirmation` and safety floor `needs_confirmation`. It also receives
  a `Security.Risk` tier/reason so operator security status can explain it.

M1 tests must prove defaults, safe-write keys, validation errors, secret-ref
shape, token redaction, and cross-fragment consistency.

## Result Readback

Confirmation-gated effectful calls need a return path for stateless clients.
Today: an action returns `:needs_confirmation`, the operator approves out-of-band
(`approve_confirmation` re-runs a resumable action and returns the result to the
*operator*), and the result persists only in the confirmation record — the read-
by-id actions are `:internal`, so an external client can never retrieve it. v0.51
adds:

- An **Ecto-backed public protocol readback table** in the Allbert Assist DB,
  rather than an Allbert Home flat file or extra confirmation-store metadata.
  M2 owns the migration, TTL cleanup/sweeper behavior, and `db_serial` tests.
- A new **read-only, `:agent`-exposable** action (e.g. `get_public_call_result`)
  keyed by a public call id, returning `pending | approved_with_result | denied |
  expired`. Unknown ids and cross-client ids return authorization/error shapes,
  not confirmation data. The result is the redacted action result (via
  `Runtime.Redactor`), not confirmation-store internals.
- **Client-scoped**: a client may read only confirmations/calls it originated; a
  per-client call→confirmation ownership record backs this (no cross-client
  leak). The record stores public call id, surface, client id, action/turn
  label, confirmation id when present, trace id, created/resolved/expires
  timestamps, status, and redacted result/error metadata only.
- **Never before approval**: returns `pending` until the operator resolves the
  confirmation. Expired entries return `expired` with no result bytes.
- **Retention bounded**: the default TTL is documented in Settings and enforced
  in tests; expiry is audited. Readback does not expose `show_confirmation` or
  `list_confirmations`.
- Each surface shapes this idiomatically: MCP as a tool / resource read; OpenAI
  as a follow-up request carrying the id; ACP as a `session/*` read.

ADR 0055 records the trust decision; this section is the design seam.

## Non-Goals
- No public AG-UI/A2UI bridge or MCP Apps iframe UI (parked post-1.0).
- No hosted multi-user authorization.
- No MCP OAuth 2.1 protected-resource / authorization-server implementation.
  v0.51 HTTP auth is an Allbert local/private bearer-token ingress, not a claim
  of public remote MCP OAuth parity.
- No external-client self-approval of confirmations.
- No client-supplied tool selection (OpenAI `tools` rejected).
- No full OpenAI API or Responses API parity. v0.51 is a bounded
  Chat Completions compatibility shim for text turns and fixture clients.
- No generic multimodal/media ingress. OpenAI and ACP image/audio/resource
  protocol payloads are rejected unless a later plan exposes a capability-
  specific route through existing media/artifact authority.
- No exposure of settings, secrets, raw signals, raw trace bodies, or
  confirmation storage; no auto-enablement from request payloads.

## Definition Of Done

v0.51 is done when:
- an MCP client, an OpenAI-API client, and an ACP editor can each reach Allbert
  in a disposable Allbert Home, seeing only operator-enabled tools and memory
  namespaces;
- a confirmation-gated call on each surface returns `:confirmation_pending` (or
  its OpenAI-shaped equivalent), the operator approves through their workspace,
  and the client retrieves the result via the **client-scoped poll-by-id
  readback** (never before approval, never another client's);
- conversational prompts enter through `Runtime.submit_user_input/1`; every
  effectful call routes through `Actions.Runner.run/3` + Security Central +
  confirmations; non-exposable internals are config errors; no surface can
  self-approve;
- HTTP surfaces require a per-client token (rotatable/revocable) behind the API
  secure-header policy + inbound rate-limit; the `:public_surface_call_inbound`
  permission + floor are in `Security.Policy`;
- the `:v051` eval set + `release.v051` + the full release gate pass; AG-UI/A2UI
  + MCP Apps remain internal/parked;
- roadmap/vision/agent-context-map/ADR 0044/0055/CHANGELOG reflect the shipped
  state; the v0.57 export/import sweep round-trips the three surfaces' settings
  without exposing secrets.
