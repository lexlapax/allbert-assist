# Allbert v0.47 Operator-Supervised Self-Improvement

Request flow: `docs/plans/v0.47-request-flow.md`
ADR: `docs/adr/0045-operator-supervised-self-improvement-trust-tier.md`

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`,
`docs/plans/allbert-jido-vision.md`, and `docs/plans/future-features.md`.

## Purpose

Add the safe self-improvement precursor after marketplace-lite exists, without
granting Allbert autonomous authority. v0.47 covers trace-to-skill draft
suggestions, workflow/intention draft suggestions, marketplace-aware template
suggestions, and dynamic capability review loops that reuse the v0.36
sandbox/gate runner, v0.37 dynamic integration gate, and v0.38 templated
creation substrate.

## Operator-Visible Win

Allbert can notice repeated operator patterns and propose useful inert drafts:
new skills, workflow YAML, template inputs, marketplace-backed suggestions, or
capability-gap drafts. The operator remains in charge of review, validation,
sandbox evidence, approval, enablement, and publication.

## Planned Interfaces

- Workspace suggestion queue for trace-to-skill, trace-to-workflow,
  marketplace-backed template, and capability-gap drafts.
- CLI inspection command for suggestions, for example
  `mix allbert.self_improvement list`.
- Registered read-only analysis action for pattern discovery over traces,
  objectives, memory review decisions, marketplace metadata, and
  operator-marked examples.
- Registered draft creation action that writes inert drafts only into reviewed
  draft roots.
- Reviewed memory promotion/update draft facades.
- Reviewed objective/workflow draft-write facades.
- Handoff into v0.38 templates for plugin/app/tool/flow scaffolds.
- Handoff into the v0.36/v0.37 sandbox/gate/integration path for code-bearing
  drafts.

## Security Posture

The milestone is advisory and inert by default.

- Trace analysis does not grant permissions.
- Draft suggestions do not enable skills, actions, plugins, workflows, or
  package installs.
- Code-bearing drafts must pass the existing sandbox/gate and trusted
  validation path before any live authority can be requested.
- Instruction-only skill drafts are created disabled and untrusted.
- Workflow YAML drafts are declarative objective inputs only, not an execution
  engine.
- Memory promotion/update and objective/workflow facades write draft state only.
- Any enablement, integration, rollback, publish, or marketplace submission
  requires normal registered actions, Security Central, durable confirmation,
  traces, and audits.

## Non-Goals

- No autonomous skill creation.
- No auto-enable, auto-publish, or remote install from trace patterns.
- No broad execution permission derived from model output or repeated use.
- No hidden model training, personality distillation, or learned system-memory
  authority.
- No unsupervised self-recompilation, compiler-loop bootstrapping, or runtime
  mutation outside the v0.36/v0.37/v0.38 review path.
- No settings, secrets, shell, package, confirmation-decision, trust-control,
  or live workspace/canvas write facades.
- No hosted multi-user or distributed coordination model.
- No release-candidate freeze; v0.51 owns final hardening and RC evidence.

## Definition Of Done

v0.47 is done when Allbert can propose useful drafts from repeated operator
patterns and marketplace metadata while every effectful transition still
requires explicit operator review, registered actions, Security Central,
confirmation, traces, and audit evidence.
