## Purpose

Add the safe self-improvement precursor after marketplace-lite (v0.45) and
delegation hardening (v0.46), without granting Allbert autonomous authority.

v0.47 is the **discovery + local-drafts** half of operator-supervised
self-improvement. It ships:

1. a **trace index** that makes repeated operator activity queryable without
   widening trace authority (ADR 0045 A1);
2. a **generalized suggestion surface** that reuses the v0.42 discovery
   suggestion schema and passive workspace panel (ADR 0045 A2 + ADR 0048
   v0.47 amendment);
3. a **read-only pattern-discovery action** that proposes inert suggestions
   from trace-index patterns while reporting structured signals (objective
   events and memory review decisions, including entries the operator marked
   `:kept`/`:flagged` via `review_memory_entry`) as diagnostics/source
   context;
4. a **unified reviewed-draft store** generalized from the v0.37
   `DynamicPlugins.Draft` lifecycle, holding every inert draft kind in one
   place (ADR 0045 A3 + ADR 0032 v0.47 amendment);
5. **skill, workflow, and memory draft** creation into that store, where each
   draft is inert until the operator promotes it through the already-shipped
   live write path.

The **handoff half** â€” template-backed and marketplace-backed suggestion
drafts, inert delegate-plugin draft requests, dynamic capability-gap drafts,
and objective drafts that fan into the v0.36/v0.37/v0.38 sandbox / gate /
templated-creation path â€” ships in **v0.47b** (`docs/plans/v0.47b-plan.md`).
The split mirrors v0.39 â†’ v0.39b: v0.47 lands the discovery substrate and the
non-code local drafts carefully, then v0.47b adds the code-bearing and
catalog-backed draft kinds on top of a proven base.

v0.47 adds **no new authority**: pattern discovery is read-only, suggestion
packets are advisory metadata, and every draft is inert until a normal
registered action, Security Central, durable confirmation, trace, and audit
promote it. No trace signal, suggestion score, or repeated operator approval
grants permission by itself (ADR 0045).

v0.47b) may revisit them only with an explicit re-decision recorded here and
in the request-flow document.

Rationales:

- **Draft store.** The v0.37 store already distinguishes inert drafts from
  integrated authority through a tier lifecycle (`draft`, `sandbox_compiled`,
  `sandbox_trialed`, `gate_passed`, `integrated`, `rolled_back`,
  `discarded`). Generalizing it â€” a generic `kind` plus per-kind promotion
  targets â€” gives memory, workflow, and skill drafts the same inert-by-default
  guarantees, listing, and audit without a second review mechanism. The facade
  is the single operator review surface; **promotion** is what calls
  `Memory.append/1`, writes a live `<ALLBERT_HOME>/workflows/<id>.yaml`, or
  enables a skill â€” each through its existing confirmed action.
- **Suggestion surface.** Reusing `Tools.Discovery.Suggestion` means one
  schema (status, `candidate_snapshot`, `evaluation_snapshot`, provenance,
  expiry) and one passive panel. v0.47 adds suggestion types and a
  provenance discriminator; it does not invent a queue. future-features
  Â§"Push/notification" already treats a self-improvement suggestion and a
  discovery suggestion as the same kind of passive, opt-in surface event.
- **Discovery substrate.** Traces are redacted markdown, not a queryable
  event log. The index is a read-only compiled view (the vision's "compiled
  runtime views for speed") that records *that* a pattern repeated, never new
  authority. It inherits the v0.40â€“v0.43 trace redaction; raw content stays in
  its source files. M1 records the indexed signal set and the privacy policy.
- **Release scope.** The split keeps v0.47 honest: the trace index and the
  unified draft store are real substrate work, and the non-code draft kinds
  (skill/workflow/memory) exercise the store and the suggestion surface
  end-to-end without touching the sandbox/gate path. v0.47b then adds the
  code-bearing and catalog-backed kinds.
- **Discovery action shape.** `find_tools` is the shipped template for a
  read-only, internal, diagnostics-returning discovery action. Mirroring it
  keeps the security floor (`:read_only`) and the passive-return contract
  consistent and testable.

## The Self-Improvement Substrate (Built And Generalized)

v0.47 builds one new read-only substrate (the trace index) and generalizes
two shipped ones (the suggestion surface and the draft store):

```text
AllbertAssist.SelfImprovement.TraceIndex            (NEW, read-only)
  index(opts)                                        # compile a redaction-safe
                                                     # view over memory/traces
  query(filter)                                      # repeated prompts, action
                                                     # chains, corrections,
                                                     # failed intents

AllbertAssist.Tools.Discovery.Suggestion            (GENERALIZED, v0.42)
  suggestion_type                                    # + self_improvement kinds
  status / candidate_snapshot / evaluation_snapshot  # reused as-is
AllbertAssist.Workspace.DiscoverySuggestions        (REUSED panel)

AllbertAssist.Drafts.Store                          (GENERALIZED from v0.37
  put(kind, payload, provenance) -> draft            #  DynamicPlugins.Draft)
  list(filter) / show(id) / discard(id)              # one inert holding area
  promote(id, context)                               # -> per-kind live path

AllbertAssist.Actions.SelfImprovement.DiscoverPatterns  (NEW, :read_only)
  the read-only boundary that reads the index + structured signals and
  writes inert suggestions; mirrors Actions.Tools.FindTools + Tools.Finder
```

Shared substrate contract for v0.47/v0.47b:

- v0.47 owns `SelfImprovement.TraceIndex`, the generalized
  `Tools.Discovery.Suggestion` surface, and the logical
  `AllbertAssist.Drafts.Store` facade.
- `AllbertAssist.Drafts.Store` is the single review/list/show/discard/promote
  contract. New v0.47 non-code drafts use `<ALLBERT_HOME>/drafts/` subroots
  such as `<ALLBERT_HOME>/drafts/workflows/<id>.yaml`; existing v0.37
  source-bearing dynamic drafts remain under
  `<ALLBERT_HOME>/dynamic_plugins/drafts/` and are exposed through the same
  facade as a compatibility root.
- v0.47b may add draft `kind`s and promotion targets only; it must reuse this
  suggestion lifecycle, draft facade, settings namespace, and authority
  boundary.

The discovery action never writes live authority. It may write advisory
suggestion rows, but it must not write live settings, skills, workflows,
memory, plugins, code, permissions, or confirmations. Suggestions are
advisory packets. Drafts are inert store records. Only an operator-confirmed
**promotion** action crosses into a live write, and it does so through the
already-shipped path for that kind (skill enablement, workflow file, memory
append).

## Settings (`self_improvement.*`)

Add the `self_improvement.*` namespace through the generated
`core:self_improvement` settings fragment (ADR 0031) with `schema_version: 1`
(ADR 0046). Self-improvement discovery is off by default.

```yaml
self_improvement:
  enabled: false                 # opt-in master switch
  trace_index:
    enabled: false               # opt-in; index only when both flags on
    max_indexed_entries: 5000    # bound the compiled view
    min_repetitions: 3           # a pattern must recur this many times
  suggestions:
    max_open: 25                 # cap pending suggestions in the queue
    ttl_days: 14                 # suggestions expire (reuse v0.42 expiry)
  drafts:
    max_open: 50                 # cap inert drafts in the unified store
  schema_version: 1              # invariant read-only at v0.47
```

Safe-write keys: `self_improvement.enabled`,
`self_improvement.trace_index.enabled`,
`self_improvement.trace_index.max_indexed_entries`,
`self_improvement.trace_index.min_repetitions`,
`self_improvement.suggestions.max_open`,
`self_improvement.suggestions.ttl_days`,
`self_improvement.drafts.max_open`.

Invariants: `self_improvement.schema_version` is read-only at `1`;
`trace_index.max_indexed_entries` cannot exceed 50000;
`trace_index.min_repetitions` cannot be below 2;
`suggestions.max_open` cannot exceed 200; `drafts.max_open` cannot exceed 500.
All caps and read-only invariants are enforced at settings-write time (a
rejected write returns an error), not only clamped at runtime.

No new permission settings: discovery uses `:read_only`; inert draft-store
writes use the existing `:dynamic_codegen_request` permission as the generalized
reviewed-draft write gate, but v0.47 draft creation must not call a provider,
generate code, require `dynamic_codegen.enabled`, or create live authority.
Promotion uses each kind's existing permission (`:skill_write` for skill
enablement, `:objective_write` for workflows, `:memory_write` for memory) and
requires durable confirmation.

## Operator-Visible Win

An operator who has opted into trace indexing can ask Allbert what repeated
work it has noticed and receive **inert suggestions** â€” a skill draft, a
workflow draft, or a memory promotion draft â€” each reviewable, editable, and
discardable, and none enabled until the operator promotes it.

First-time walkthrough:

1. Operator (self-improvement + trace index enabled, after some real use):
   "show me what you could turn into a skill or workflow."
2. Intent routes to the read-only `discover_patterns` action.
3. The action converts trace-index patterns into inert suggestions and reports
   objective events plus memory review decisions (including entries the
   operator marked `:kept`/`:flagged`) as diagnostics/source context. No
   message is pushed; the panel updates.
4. The operator opens the DiscoverySuggestions panel (or
   `mix allbert.self_improvement list`) and sees, e.g., "you ran this
   three-step browser+summarize sequence 4 times â€” draft a workflow?"
5. The operator inspects a suggestion; promoting it creates an **inert
   draft** in the unified store (a disabled/untrusted skill, a
   `<ALLBERT_HOME>/drafts/workflows/<id>.yaml` workflow draft, or a
   memory-promotion draft) â€” still nothing enabled.
6. Promoting the draft to live is a separate confirmed action: enabling the
   skill, writing the live `<ALLBERT_HOME>/workflows/<id>.yaml`, or appending
   the memory entry â€” each through its existing Security Central + confirmation
   path.
7. Discarding a suggestion or a draft leaves no enabled capability behind.

Repeated prior approvals never auto-approve a new suggestion; model
confidence and pattern frequency are advisory only.

## Security Posture

- **Pattern discovery is read-only.** `discover_patterns` is `:read_only`;
  it reads the index and diagnostic structured signals and writes only inert
  advisory suggestions from trace-index patterns. It writes no live settings,
  skills, workflows, memory, plugins, code, permissions, or confirmations.
- **Suggestions are advisory, never authority.** A suggestion packet is
  descriptive metadata (ADR 0021 Â§4 / ADR 0045). It does not alter agent
  behavior and does not auto-promote.
- **Drafts are inert.** Every draft is a store record with no live effect.
  Skill drafts are disabled and untrusted; workflow drafts are declarative
  objective inputs only; memory drafts are draft store state, not memory
  entries, until promotion.
- **Promotion is the only effectful boundary.** Enabling a skill, writing a
  live workflow file, or appending memory each requires a normal registered
  action, Security Central, durable confirmation, trace, and audit.
- **The trace index inherits trace redaction.** It compiles over already-
  secret-redacted trace markdown (the prompt text and action names it reads
  are what traces already store); it adds no new exposure beyond the traces,
  is scoped per-user/per-app, and records no new secrets.
- **No repeated-use grant.** Pattern frequency, suggestion score, and prior
  approvals never become a permission grant or an auto-approval (ADR 0045).
- **Bounded surfaces.** `suggestions.max_open`, `suggestions.ttl_days`, and
  `drafts.max_open` bound the queue and store; `trace_index.max_indexed_entries`
  bounds the compiled view.
- **No new attack surface.** No new permission class, operation class, URI
  scheme, or memory namespace; the only new settings are an enable toggle and
  bounded caps. The unified draft store reuses the v0.37 tier lifecycle and
  audit.

## Non-Goals

- No autonomous skill creation; no auto-enable, auto-publish, or remote
  install from trace patterns.
- No broad execution permission derived from model output or repeated use.
- No hidden model training, personality distillation, or learned
  system-memory authority.
- No unsupervised self-recompilation, compiler-loop bootstrapping, or runtime
  mutation outside the v0.36/v0.37/v0.38 review path.
- No settings, secrets, shell, package-install, confirmation-decision,
  trust-control, or live workspace/canvas write facades.
- No code-bearing, template-backed, marketplace-backed, delegate-plugin,
  capability-gap, or objective drafts in v0.47 â€” those are v0.47b.
- No free-text NLP mining of trace prose beyond the bounded repetition index;
  the index records structural repetition, not semantic summarization.
- No hosted multi-user or distributed coordination model.
- No release-candidate freeze; v0.59 owns final hardening and RC evidence.
