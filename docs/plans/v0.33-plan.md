# Allbert v0.33 User Theming And Layout Overrides

Request flow: `docs/plans/v0.33-request-flow.md`

ADR: `docs/adr/0025-user-theming-and-override-security.md`

v0.33 follows the v0.32 workspace-only UI reset. It lets operators retheme and
re-lay-out the `/workspace` UI from Allbert Home without editing core code or
rebuilding assets. The dynamic plugin trial substrate moves to v0.34 and the
Plugin And App Generator moves to v0.35 so it can scaffold the workspace-panel
and theming hooks after both are proven.

## Purpose

Allbert styling is currently host-side: Tailwind/daisyUI compiled into the web
asset bundle, plus workspace variables scoped to the shell. Operators cannot
change colors, fonts, density, or panel layout without editing source.

v0.33 adds a safe, local override system:

- Design-token YAML for common retinting and spacing changes.
- Opt-in sanitized CSS snippets for advanced local customization.
- Validated layout data for enabling, disabling, reordering, and pinning
  workspace zones/panels.

The system reads only from Allbert Home, serves only through host-owned routes,
and is gated by Settings Central keys available in the v0.32 workspace
settings panel.

## Expected Inputs

- v0.32 ships `/workspace` as the only operator home.
- v0.32 ships host-owned zones, panel surfaces, utility drawer, and Settings
  Central inside the workspace.
- v0.31 ships consolidated runtime paths and settings fragments.
- `AllbertAssist.Runtime.Paths` resolves Allbert Home.
- Settings Central supports typed keys, auditing, and action-backed writes.
- The web app has the current Tailwind/CSS asset pipeline and workspace shell
  CSS variables.

## Design Decisions

### 1. All operator overrides live under Allbert Home

New path helpers:

- `AllbertAssist.Runtime.Paths.themes_root/0` → `<ALLBERT_HOME>/themes`
- `AllbertAssist.Runtime.Paths.theme_snippets_root/0` →
  `<ALLBERT_HOME>/themes/snippets`
- Workspace layout file → `<ALLBERT_HOME>/workspace/layout.yaml`

`ensure_home!/0` creates the roots. Tests and smoke checks must use a
temporary `ALLBERT_HOME`.

### 2. Design tokens are the default path

`<ALLBERT_HOME>/themes/<name>.yaml` contains a constrained token map for
colors, font family, font scale, spacing density, radius, border contrast, and
motion preference. The host renders those tokens as CSS custom properties at
`/theme/user.css`, linked after `app.css`.

Missing files, invalid YAML, unknown tokens, or invalid values fall back to
defaults per-key with bounded operator-visible warnings. No asset rebuild is
required.

### 3. CSS snippets are opt-in and sanitized

`<ALLBERT_HOME>/themes/snippets/*.css` files are disabled by default. When
enabled, snippets are sanitized before serving:

- Reject `@import`.
- Strip `url()` and `image-set()`.
- Strip remote font/source constructs.
- Serve empty CSS if a snippet is entirely unsafe.
- Emit bounded warnings naming removed constructs.

Snippets are loaded after token CSS and outside Tailwind layers so they can
override local presentation. They never grant action authority and never load
remote resources.

### 4. Layout overrides are data, never code

`<ALLBERT_HOME>/workspace/layout.yaml` can enable/disable/reorder v0.32 zones
and pin registered panels into allowed zones. It validates against the zone
registry, registered panel ids, and catalog atoms.

Unknown zones, unknown panel ids, or invalid placements are ignored per-key
with warnings. Layout override cannot create components, routes, actions, or
permissions.

### 5. Settings Central owns enablement

New Settings Central keys:

| Key | Type | Default | Purpose |
|---|---|---|---|
| `workspace.theme.active` | string/null | null | Selected token theme name |
| `workspace.theme.snippets_enabled` | boolean | false | Master switch for CSS snippets |
| `workspace.theme.enabled_snippets` | string list | [] | Enabled snippet file basenames |
| `workspace.layout.override_enabled` | boolean | false | Master switch for layout YAML |

These keys are visible and writable through the v0.32 workspace Settings
Central panel. CLI support may exist through existing settings actions, but
v0.33 does not add a separate `/settings` route.

### 6. CSP backs the sanitizer

The browser pipeline adds a Content-Security-Policy baseline for workspace
pages and theme routes:

- Styles served only from self.
- Images served only from self/data where already required by existing UI.
- No remote CSS imports.
- No inline script loosening beyond the current Phoenix/LiveView requirement.

The exact CSP must be pinned during implementation against existing Phoenix
LiveView requirements, then covered by regression tests.

## Module Shapes

- `AllbertAssist.Theme.Tokens`: parse, validate, normalize, and render token
  YAML into CSS variables.
- `AllbertAssist.Theme.Sanitizer`: sanitize CSS snippets and return warnings.
- `AllbertAssist.Theme.Layout`: parse and validate workspace layout YAML
  against v0.32 zones and registered panels.
- `AllbertAssistWeb.ThemeController`: serve `/theme/user.css` and sanitized
  snippet CSS from Allbert Home.
- `AllbertAssist.Runtime.Paths`: add theme roots.
- Settings fragments: add `workspace.theme.*` and
  `workspace.layout.override_enabled` through the v0.31 fragment contract.
- Workspace shell/catalog: consume validated layout data when the setting is
  enabled.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless the implementation owner records a narrower gate.

### M1: Theme roots and settings

Goal: Allbert Home has theme roots and Settings Central has typed enablement
keys.

Tasks:

- Add theme path helpers and create roots from `ensure_home!/0`.
- Add settings schema keys and audit coverage.
- Render the settings controls in the workspace Settings Central panel.

Tests:

- Path tests with temporary Allbert Home.
- Settings schema/read/write/audit tests.
- Workspace settings-panel tests for the new keys.

Acceptance:

- A disposable home exposes the new roots and settings without touching a real
  user home.

### M2: Token themes

Goal: operator-selected token YAML retints `/workspace`.

Tasks:

- Implement token parser/validator with a constrained token allow-list.
- Serve `/theme/user.css`.
- Link token CSS after `app.css`.
- Add warnings for invalid or missing token files.

Tests:

- Token parsing/rendering tests.
- Controller tests for selected, missing, invalid, and partially invalid
  themes.
- Chrome verification for visible retinting with no rebuild.

Acceptance:

- A token file changes workspace color/density on reload.
- Invalid values fall back safely.

### M3: CSP baseline

Goal: theme routes and workspace pages have a CSP compatible with LiveView and
the override system.

Tasks:

- Pin the CSP needed for Phoenix LiveView and theme CSS.
- Add headers to browser/theme routes.
- Document any unavoidable LiveView allowances.

Tests:

- Header tests for workspace and theme routes.
- Regression test for blocked remote style/image fetch attempts where
  feasible.

Acceptance:

- CSP is present and does not break LiveView.

### M4: CSS snippets

Goal: advanced CSS snippets work only when explicitly enabled and sanitized.

Tasks:

- Add snippet root helper and controller route.
- Implement sanitizer with reject/strip/warn behavior.
- Gate snippets behind Settings Central master switch and enabled basename
  list.
- Prevent path traversal and non-`.css` serving.

Tests:

- Sanitizer tests for `@import`, `url()`, `image-set()`, remote font/source
  constructs, safe CSS, and entirely unsafe snippets.
- Controller tests for disabled, enabled, missing, traversal, and sanitized
  output.
- Chrome verification that safe snippet changes apply and unsafe constructs do
  not fetch.

Acceptance:

- Safe local snippets can restyle the workspace.
- Dangerous constructs are removed with warnings.

### M5: Layout overrides

Goal: layout YAML can reorder/hide zones and pin panels without adding
authority.

Tasks:

- Parse and validate layout YAML against v0.32 zone/panel registries.
- Apply valid layout data in workspace composition when enabled.
- Warn and drop unknown/invalid keys per-key.
- Ensure layout override cannot create routes, components, action bindings, or
  permissions.

Tests:

- Layout parser/validator tests.
- Workspace catalog tests for hide/reorder/pin behavior.
- Negative tests for unknown zones, unknown panels, invalid placements, and
  attempted action/component creation.

Acceptance:

- Valid layout changes are reflected in `/workspace`.
- Invalid layout data is safely ignored with diagnostics.

### M6: Closeout

Goal: v0.33 is release-ready.

Tasks:

- Finalize ADR 0025.
- Update README, roadmap, agent context map, future-features, and release
  notes.
- Bump version metadata to `0.32.0`.
- Run focused tests, Chrome verification, and full gate.

Tests:

- `mix format --check-formatted`
- `mix compile --warnings-as-errors`
- `mix credo --strict`
- `mix dialyzer`
- `mix precommit`
- `git diff --check`

Acceptance:

- Operators can retheme and re-layout `/workspace` from Allbert Home without
  editing core source, loading remote resources, or granting new authority.

## Non-Goals

- No `/settings` route.
- No remote theme marketplace or remote theme fetching.
- No arbitrary JavaScript, plugin code loading, or dynamic Elixir compilation.
- No app generator; that is v0.35.
- No new workspace zones beyond the v0.32 zone registry unless required by a
  documented v0.33 implementation blocker.
- No AG-UI/A2UI/MCP Apps UI protocol bridge.

## Test Plan

- Path/settings tests for Allbert Home roots and Settings Central keys.
- Token parser/controller tests and Chrome retint verification.
- CSP header tests and unsafe remote-fetch regression coverage.
- CSS sanitizer/controller tests for safe, unsafe, disabled, and traversal
  cases.
- Layout parser/workspace composition tests for hide/reorder/pin and invalid
  data.
- Chrome verification at desktop and narrow widths for token theme, snippet
  application, layout override, fallback behavior, and Settings Central
  controls.

## Definition Of Done

v0.33 is done when `/workspace` can be rethemed and re-laid out from
Allbert Home through Settings Central-gated token, snippet, and layout files;
unsafe CSS and invalid layout data are bounded with warnings; CSP is in place;
and full gate plus Chrome verification pass.
