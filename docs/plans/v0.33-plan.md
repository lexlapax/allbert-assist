# Allbert v0.33 Conversational App Intent Handoff And Direct Answer Foundation

Request flow: `docs/plans/v0.33-request-flow.md`

ADR:

- `docs/adr/0034-conversational-app-intent-handoff-and-clarification.md`

v0.33 fixes the gap exposed after v0.31: a neutral workspace prompt such as
`analyze CIEN` currently falls to the static `direct_answer` echo unless the
operator has already selected StockSage. That behavior is safe, but not useful.

v0.33 keeps the safety invariant intact. Neutral context must not silently run
app-owned actions. Instead, Allbert learns to answer ordinary questions,
recognize app-owned capability candidates, ask targeted clarification when
needed, and offer an explicit app handoff before the normal action
confirmation flow begins.

## Purpose

Allbert currently has two separate shortcomings:

- `direct_answer` is a side-effect-free static echo, not a real answer path.
- Plausible app-owned candidates are collected and ranked, but neutral context
  does not surface them as operator choices.

v0.33 introduces a generic conversational bridge between neutral workspace
chat and registered app capabilities. StockSage proves the bridge with
`run_analysis`, but the contract is app-agnostic and descriptor-driven.

## Design Decisions

### 1. Direct answer becomes a real side-effect-free answer path

`direct_answer` must no longer be a static echo. It should answer plain
general questions through the configured model/provider boundary when enabled,
with no tools, no app handoff, no memory write, no resource access, and no
execution. If no model is configured or the provider fails safely, it returns a
bounded fallback that explains no action was taken.

The response text must not include stale version-specific runtime strings.

### 2. Apps contribute intent descriptors

App-owned routing hints come from registered descriptors, not hardcoded
StockSage predicates in core. Descriptor metadata includes:

- `app_id`
- `action_name`
- display label
- examples and synonyms
- required slots
- slot extractors or slot names to fill through clarification
- whether neutral context requires app handoff before execution

Descriptors are proposal metadata only. They never grant permission, trust,
route authority, or settings authority.

### 3. Neutral app intent produces handoff, not execution

When a neutral request strongly matches an app-owned descriptor, Allbert emits
an app handoff decision such as:

`StockSage can run a financial analysis for CIEN. This requires confirmation.
Run it?`

Accepting the handoff sets explicit app context for the turn/session/thread
through the existing session/action boundary and then enters the normal
registered action path. The target action still reaches Security Central and,
for StockSage `run_analysis`, the existing confirmation flow.

Declining the handoff leaves context unchanged and creates no pending action
confirmation.

### 4. Ambiguity produces clarification

When required slots are missing, or when top candidates are close under the
configured threshold and margin, Allbert asks a concise clarification question
instead of echoing.

Examples:

- `analyze` -> ask which ticker.
- Multiple plausible app actions -> present a short list of choices.
- A low-confidence match -> answer generally or ask for rephrase, depending on
  candidate evidence.

### 5. Existing classifier remains advisory

`AllbertAssist.Intent.Classifier` already provides bounded model assistance.
v0.33 reuses it as an advisory selector over already-collected candidates and
descriptor summaries. It may support handoff/clarification decisions, but it
cannot invent actions, alter permissions, lower confirmation floors, set app
context, or bypass runner app-scope checks.

Classifier disabled, low-confidence, invalid, or timed-out paths fall back to
deterministic descriptor routing and direct-answer behavior.

### 6. Retire StockSage-specific core routing

The existing StockSage-specific ranker keyword branch is removed only after the
generic descriptor path covers the same behavior. StockSage remains the
reference app; it is not a special case in the core router.

## Architecture And Contracts

This section pins the concrete seams so v0.33 is implementable without
re-deriving them. Every new path is descriptive/proposal infrastructure; none
grants authority (ADR 0019, ADR 0021, ADR 0034). End-to-end operator and data
flow lives in `docs/plans/v0.33-request-flow.md`.

### Descriptor contribution and collection

- Apps/plugins declare intent descriptors through a new optional
  `AllbertAssist.App.SurfaceProvider` callback `intent_descriptors/0`
  (default `[]` via `__using__`, like the existing `surface_catalog/0`).
  Plugin app modules without a SurfaceProvider may expose the same callback.
- `AllbertAssist.Extensions.Registry.contributions/1` gains an
  `intent_descriptors` key collected from registered apps/plugins, alongside
  the existing surfaces/actions/skill_paths/settings_schema contributions.
- `AllbertAssist.Intent.Descriptor` validates and normalizes each descriptor and
  drops it (with bounded diagnostics) when its `app_id`/`action_name` is not a
  registered, enabled, agent-exposed action. A descriptor never registers an
  action, route, permission, trust tier, or setting.

Descriptor fields (all metadata-only):

| Field | Meaning |
|---|---|
| `app_id` | owning registered app (atom) |
| `action_name` | target registered action (must exist and be agent-exposed) |
| `label` | operator-facing capability label |
| `examples` / `synonyms` | deterministic match phrases (e.g. "analyze AAPL") |
| `required_slots` | named slots required before execution (e.g. `ticker`) |
| `slot_extractors` | descriptor-declared, conservative per-slot extractors |
| `handoff_required?` | whether neutral context must hand off before execution |

### Engine decision rule

`AllbertAssist.Intent.Engine` collects descriptor candidates during
`collect_candidates/2` and scores them deterministically in
`AllbertAssist.Intent.Ranker` from descriptor `examples`/`synonyms`/`label`
(not from hardcoded app keywords). After ranking, the engine chooses among
three outcomes using Settings-Central thresholds:

- `:app_handoff` — top descriptor score >= `intent.handoff_threshold` AND its
  margin over the next distinct candidate >= `intent.handoff_margin` AND all
  `required_slots` are filled. Produces a handoff proposal; executes nothing.
- `:clarify_intent` — a plausible descriptor candidate exists but a
  `required_slot` is missing, or the top-two margin is below
  `intent.handoff_margin`, or the score is in
  `[intent.clarify_floor, intent.handoff_threshold)`. Produces a targeted
  question or a short candidate choice list; executes nothing.
- `:direct_answer` — no descriptor candidate clears `intent.clarify_floor`;
  falls through to the real direct-answer path below.

The deterministic rule runs first. The bounded `Intent.Classifier`
(`intent.model_assist_enabled`) may only re-rank/select among already-collected
candidates or recommend clarification; it never invents candidates, sets app
context, lowers confirmation floors, or changes the rule's authority outcome.

### Decision and handoff payload

- `AllbertAssist.Intent.Decision` adds `:app_handoff` and `:clarify_intent` as
  valid `intent` values (the struct already accepts atom intents).
- `AllbertAssist.Intent.Handoff` is the structured, inert proposal carried in
  the response and decision `trace_metadata`: `kind` (`:app_handoff` |
  `:clarify_intent`), `app_id`, `action_name`, `label`, `extracted_slots`,
  `missing_slots`, `candidate_options` (for ambiguity), `prompt_text`, and
  `accept`/`decline` labels.

### IntentAgent response branches

`AllbertAssist.Agents.IntentAgent.respond/1` gains branches for the new intents.
When `Engine.decide/1` returns `:app_handoff` or `:clarify_intent`, the agent
builds a side-effect-free response (`status: :completed`, a handoff/clarify
payload, no `Actions.Runner.run/3` call, no confirmation record). All existing
intents stay unchanged.

### Operator surfacing (no new catalog atom)

The handoff/clarify proposal is surfaced as a v0.26 **ephemeral surface** in the
v0.32 workspace, reusing the existing `ephemeral_surfaces/2` render path,
`dismiss_workspace_ephemeral` event, and ephemeral tab. It is composed from
existing catalog primitives (`:section`, `:text`, `:action_button`/`:button`);
**v0.33 adds no catalog atom and needs no ADR 0030 amendment**. The proposal
carries stable handles `#intent-handoff`, `#intent-handoff-accept`,
`#intent-handoff-decline`, `#intent-clarification`, and `[data-intent-option]`.

### Accept / decline / clarify data path

- **Accept** (`#intent-handoff-accept`): `AllbertAssistWeb.WorkspaceLive`
  invokes the existing registered `set_active_app` action for the proposed
  `app_id`, then re-submits the original prompt through
  `Runtime.submit_user_input/1` carrying the descriptor-extracted slots. With
  `active_app` set, the normal in-context path reaches `Actions.Runner.run/3`,
  Security Central, and the existing confirmation flow (for StockSage
  `run_analysis`, `confirmation: :required`).
- **Decline** (`#intent-handoff-decline`): dismisses the ephemeral surface via
  the existing `dismiss_workspace_ephemeral` action. No `active_app` change, no
  confirmation, no action.
- **Clarify answer**: the operator reply (a ticker, or a chosen
  `[data-intent-option]`) re-enters `submit_user_input/1`; the engine fills the
  slot / selects the option and proceeds to `:app_handoff`.

### Direct-answer model boundary

`AllbertAssist.Actions.Intent.DirectAnswer` stops returning the static echo. A
bounded model boundary (mirroring `Intent.Classifier.DefaultClassifier`'s
ReqLLM pattern) answers plain prompts when `intent.direct_answer_model_enabled`
is true and a profile resolves; otherwise it returns a deterministic bounded
fallback. The call grants no Resource Access, no tools, and no app actions;
prompts/completions are redacted in traces/audits (ADR 0019). The stale `v0.26`
string is removed.

### Retire scope (StockSage-specific core code)

v0.33 retires the StockSage-specific *routing/slot* code once descriptors cover
it:

- `AllbertAssist.Intent.Ranker.active_app_keyword_match?(:stocksage,
  "run_analysis", _)` → replaced by descriptor scoring.
- `AllbertAssist.Agents.IntentAgent.maybe_put_analysis_params/3` ticker special
  case → replaced by descriptor slot extraction (used by both the
  handoff-accept and in-context paths).

Out of scope for v0.33 (explicitly deferred; still `active_app`-gated and safe):
the StockSage-specific *objective framing* in `IntentAgent`
(`objective_framing_candidate?/1`, `objective_frame_params/2`). Generalizing
objective framing is a later objective-runtime concern, not part of the
conversational-handoff contract. Record this carryover in the closeout.

### Trace and audit shape

Handoff/clarify decisions extend the existing
`decision.trace_metadata.intent_candidates` block with: decision kind, selected
descriptor (`app_id`, `action_name`), `extracted_slots`, `missing_slots`, the
threshold/margin values used, and the classifier diagnostic when model assist
ran. No raw model prompt/completion is traced.

### Module shapes summary

- `AllbertAssist.Intent.Descriptor` — descriptor validation/normalization.
- `AllbertAssist.Intent.Handoff` — inert handoff/clarify proposal struct.
- `AllbertAssist.Intent.Engine` / `Ranker` — descriptor candidates + decision rule.
- `AllbertAssist.Agents.IntentAgent` — `:app_handoff`/`:clarify_intent` branches.
- `AllbertAssist.Actions.Intent.DirectAnswer` — model-backed, bounded-fallback answer.
- `AllbertAssist.Extensions.Registry` / `App.SurfaceProvider` — descriptor discovery.
- `AllbertAssistWeb.WorkspaceLive` — ephemeral handoff/clarify rendering + accept/decline.
- `StockSage.App` — contributes the `run_analysis` descriptor (`analyze AAPL`,
  `run StockSage analysis for TSLA`; required slot `ticker`).

## Settings Keys

New keys live in the v0.31 intent settings fragment (ADR 0031). All are
operator-owned, validated, default conservative, and written only through
Settings Central actions.

| Key | Type | Default | Purpose |
|---|---|---|---|
| `intent.descriptors_enabled` | bool | `true` | master switch for descriptor-driven handoff |
| `intent.handoff_threshold` | float | `0.6` | min top descriptor score to propose handoff |
| `intent.handoff_margin` | float | `0.15` | min top-two score gap to avoid clarification |
| `intent.clarify_floor` | float | `0.3` | min score to clarify instead of direct answer |
| `intent.direct_answer_model_enabled` | bool | `false` | enable model-backed direct answers |
| `intent.direct_answer_model_profile` | string | `"fast"` (same release default as `intent.model_profile`) | model profile for direct answers |

Defaults are starting points to tune during M3/M4 against fixtures; the
implementation owner may adjust them in the settings fragment with a
request-flow note.

M1 implementation note: model-backed direct answers require
`intent.direct_answer_model_enabled=true`, a valid
`intent.direct_answer_model_profile`, and an enabled provider profile. Missing,
disabled, or failing providers return the bounded fallback.

M2 implementation note: app intent descriptors are collected from compiled app
modules through `AllbertAssist.Extensions.Registry.intent_descriptors`, validated
by `AllbertAssist.Intent.Descriptor` against registered agent-exposed actions,
and surfaced as inert `:app_intent` candidates with descriptor slot metadata in
intent traces. They do not execute, set app context, or create confirmations;
M3 owns converting selected descriptor candidates into handoff/clarification.

M3 implementation note: `AllbertAssist.Intent.Engine` now converts neutral
descriptor matches into `:app_handoff` or `:clarify_intent` decisions using the
Settings Central gates `intent.descriptors_enabled`,
`intent.handoff_threshold`, `intent.handoff_margin`, and
`intent.clarify_floor`. `AllbertAssist.Intent.Handoff` is an inert payload in
decision trace metadata; `IntentAgent.respond/1` emits a completed response and
best-effort v0.26 ephemeral workspace fragment without invoking
`Actions.Runner.run/3` for the app action. `WorkspaceLive` accepts handoff only
through `set_active_app`, then re-submits the original prompt with explicit app
context; decline only dismisses the ephemeral surface.

- Enabled classifier can select among registered candidates.
- Classifier cannot invent action/app ids.

  boundary.

M4 status: implemented and covered by focused classifier and engine tests.

## Non-Goals

- No silent app action execution from neutral context.
- No permission, trust, confirmation, or resource-access relaxation.
- No StockSage-specific core predicates.
- No sandboxed Elixir/OTP gate runner; that is v0.36.
- No dynamic plugin/app generation or gated live loading; that is v0.37.
- No generator; that is v0.38.
- No route compatibility changes beyond what v0.32 already owns.
