# Allbert v0.33 Conversational App Intent Handoff And Direct Answer Foundation

Request flow: `docs/plans/v0.33-request-flow.md`

ADR:

- `docs/adr/0034-conversational-app-intent-handoff-and-clarification.md`

v0.33 fixes the gap exposed after v0.31: a neutral workspace prompt such as
`analyze CIEN` currently falls to the static `direct_answer` echo unless the
operator has already selected StockSage. That behavior is safe, but not useful.

v0.33 keeps the safety invariant intact. Neutral context must not silently run
app-owned actions. Instead, Allbert learns to answer ordinary questions,
recognize app-owned capability candidates, ask targeted clarification when
needed, and offer an explicit app handoff before the normal action
confirmation flow begins.

## Purpose

Allbert currently has two separate shortcomings:

- `direct_answer` is a side-effect-free static echo, not a real answer path.
- Plausible app-owned candidates are collected and ranked, but neutral context
  does not surface them as operator choices.

v0.33 introduces a generic conversational bridge between neutral workspace
chat and registered app capabilities. StockSage proves the bridge with
`run_analysis`, but the contract is app-agnostic and descriptor-driven.

## Expected Inputs

- v0.19 registry-aware intent engine and bounded model-assist classifier exist.
- v0.24 objective vocabulary and advisory-provider authority rules exist.
- v0.28 app-scope hardening is enforced: app-owned actions require explicit
  matching `active_app` at `Actions.Runner.run/3`.
- v0.31 action DSL, typed response, extension registry, catalog, audit, trace,
  redaction, and settings-fragment substrates exist.
- v0.32 workspace route and app launcher exist. Explicit app selection in
  `:nav_apps` is the simple, non-inferred way to enter an app context.

## Design Decisions

### 1. Direct answer becomes a real side-effect-free answer path

`direct_answer` must no longer be a static echo. It should answer plain
general questions through the configured model/provider boundary when enabled,
with no tools, no app handoff, no memory write, no resource access, and no
execution. If no model is configured or the provider fails safely, it returns a
bounded fallback that explains no action was taken.

The response text must not include stale version-specific runtime strings.

### 2. Apps contribute intent descriptors

App-owned routing hints come from registered descriptors, not hardcoded
StockSage predicates in core. Descriptor metadata includes:

- `app_id`
- `action_name`
- display label
- examples and synonyms
- required slots
- slot extractors or slot names to fill through clarification
- whether neutral context requires app handoff before execution

Descriptors are proposal metadata only. They never grant permission, trust,
route authority, or settings authority.

### 3. Neutral app intent produces handoff, not execution

When a neutral request strongly matches an app-owned descriptor, Allbert emits
an app handoff decision such as:

`StockSage can run a financial analysis for CIEN. This requires confirmation.
Run it?`

Accepting the handoff sets explicit app context for the turn/session/thread
through the existing session/action boundary and then enters the normal
registered action path. The target action still reaches Security Central and,
for StockSage `run_analysis`, the existing confirmation flow.

Declining the handoff leaves context unchanged and creates no pending action
confirmation.

### 4. Ambiguity produces clarification

When required slots are missing, or when top candidates are close under the
configured threshold and margin, Allbert asks a concise clarification question
instead of echoing.

Examples:

- `analyze` -> ask which ticker.
- Multiple plausible app actions -> present a short list of choices.
- A low-confidence match -> answer generally or ask for rephrase, depending on
  candidate evidence.

### 5. Existing classifier remains advisory

`AllbertAssist.Intent.Classifier` already provides bounded model assistance.
v0.33 reuses it as an advisory selector over already-collected candidates and
descriptor summaries. It may support handoff/clarification decisions, but it
cannot invent actions, alter permissions, lower confirmation floors, set app
context, or bypass runner app-scope checks.

Classifier disabled, low-confidence, invalid, or timed-out paths fall back to
deterministic descriptor routing and direct-answer behavior.

### 6. Retire StockSage-specific core routing

The existing StockSage-specific ranker keyword branch is removed only after the
generic descriptor path covers the same behavior. StockSage remains the
reference app; it is not a special case in the core router.

## Module Shapes

- `AllbertAssist.Intent.Descriptor`: validates app-contributed descriptor
  metadata and exposes a normalized candidate summary.
- `AllbertAssist.Intent.Handoff`: represents explicit app-handoff and
  clarification proposals, including candidate id, app id, action name,
  extracted slots, missing slots, and decline/accept labels.
- `AllbertAssist.Intent.Engine`: collects descriptor candidates, applies
  threshold/margin rules, and returns `:app_handoff` or `:clarify_intent`
  decisions when appropriate.
- `AllbertAssist.Actions.Intent.DirectAnswer`: becomes the model-backed,
  side-effect-free answer action with bounded fallback behavior.
- `StockSage.App` or plugin metadata: contributes the `run_analysis`
  descriptor with examples such as `analyze AAPL`, `run StockSage analysis for
  TSLA`, and required slot `ticker`.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless the implementation owner records a narrower gate and
why.

### M1: Direct-answer foundation

Goal: plain general questions receive a useful side-effect-free answer.

Tasks:

- Remove stale release-specific static copy from `direct_answer`.
- Add the safe model-backed answer path and bounded no-provider fallback.
- Ensure direct answers cannot call tools, resources, app actions, memory
  writes, shell, package, skill, browser, or provider-specific side effects
  beyond the configured answer model call.

Tests:

- Plain question returns an answer, not an echo.
- Missing/disabled provider returns bounded fallback.
- No action except `direct_answer` is executed.
- Trace/audit output redacts provider details and records no app handoff.

Acceptance:

- A neutral workspace prompt such as `what is Allbert?` is answered usefully
  without creating confirmations or app context.

### M2: App intent descriptors

Goal: app-owned capabilities describe how they can be recognized without core
app-specific predicates.

Tasks:

- Add descriptor validation and registry collection.
- Add StockSage `run_analysis` descriptor as the first app contribution.
- Add descriptor candidate trace output.
- Keep descriptors inert if their app/action is not registered or disabled.

Tests:

- Valid descriptor appears in candidate collection.
- Invalid app/action/slot metadata is rejected.
- StockSage descriptor is collected from app/plugin metadata.
- Descriptor metadata alone cannot execute an action.

Acceptance:

- `analyze CIEN` in neutral context produces a plausible StockSage descriptor
  candidate, but no StockSage action execution.

### M3: Handoff and clarification decisions

Goal: neutral app intent becomes an explicit choice, and ambiguity becomes a
targeted question.

Tasks:

- Add `:app_handoff` and `:clarify_intent` decision rendering.
- Apply deterministic threshold and top-two margin rules.
- Extract simple required slots from descriptor metadata.
- Persist no app context until the operator accepts a handoff.

Tests:

- `analyze CIEN` -> app handoff proposal, no pending action confirmation yet.
- Accepting handoff sets `active_app: :stocksage` and enters normal
  `run_analysis` confirmation.
- Declining handoff leaves context unchanged and creates no confirmation.
- `analyze` asks for ticker.
- Multiple plausible candidates render concise options.
- App-owned action without matching active app remains denied by runner.

Acceptance:

- The operator can start at `/workspace`, ask `analyze CIEN`, and proceed
  through explicit handoff plus the normal StockSage confirmation flow.

### M4: Classifier advisory integration

Goal: model assist improves ambiguous cases without becoming authority.

Tasks:

- Feed descriptor/handoff summaries to the existing classifier when enabled.
- Accept classifier suggestions only when they reference collected candidates
  and meet confidence thresholds.
- Fall back deterministically on disabled, invalid, low-confidence, or timeout
  responses.
- Trace classifier use without raw prompts or completions.

Tests:

- Enabled classifier can select among registered candidates.
- Classifier cannot invent action/app ids.
- Low-confidence classifier output produces deterministic fallback or
  clarification.
- Classifier-selected app action still requires handoff and confirmation.

Acceptance:

- Model assist improves ambiguous routing without changing any authority
  boundary.

### M5: Retire app-specific ranker hardcode and closeout

Goal: v0.33 is release-ready and the generic descriptor path replaces the
current StockSage keyword special case.

Tasks:

- Remove the StockSage-specific ranker keyword branch once tests cover the
  descriptor path.
- Update ADR 0019/0021 amendments, roadmap, README, agent-context map,
  future-features, and release docs.
- Add security/eval coverage for handoff bypass attempts.

Tests:

- Full release gate.
- Security evals prove app handoff cannot bypass `active_app` app-scope
  denial.
- Regression: in StockSage-selected context, `analyze CIEN` still routes to
  StockSage confirmation.
- Regression: neutral context never silently executes app actions.

Acceptance:

- v0.33 is done when direct answers are useful, neutral app intents become
  explicit handoff/clarification, and all effectful app actions still require
  explicit app context plus normal confirmation/security checks.

## Non-Goals

- No silent app action execution from neutral context.
- No permission, trust, confirmation, or resource-access relaxation.
- No StockSage-specific core predicates.
- No dynamic plugin/app generation; that is v0.35.
- No generator; that is v0.36.
- No route compatibility changes beyond what v0.32 already owns.
