# Allbert v0.33 Plugin And App Generator

Request flow: `docs/plans/v0.33-request-flow.md`

Previously planned as v0.29 before the project-direction rethink inserted
v0.23 Jido State-Machine Convergence and v0.24 Objective Runtime Foundation.
The post-v0.26 roadmap reconciliation kept the generator at v0.31; it moved to
v0.32 when workspace-only app UI was inserted; it now moves to v0.33 so v0.31
can prove the workspace-only panel contract and v0.32 can prove user theming
and layout overrides before generators scaffold either shape.

## Purpose

v0.33 adds generators only after StockSage and CoreApp prove the plugin, app
surface, app memory, app canvas, workspace-panel, settings-panel, and theming
contracts manually. The generator should encode the boring, proven shape of an
Allbert plugin and app rather than invent abstractions early.

## Expected Inputs

- v0.17 plugin contract has been exercised by shipped channel plugins and
  StockSage's plugin entrypoint.
- v0.18 app/surface contract exists.
- v0.20 proves StockSage as a plugin-contributed app.
- v0.24 Objective Runtime Foundation exists; generated apps may opt into
  objective scaffolding when their actions are multi-step.
- v0.26 workspace and surface substrate exists.
- v0.27 App Surface Contract proves StockSage `SurfaceProvider`, real app-owned
  renderers, validated Surface-node output, and memory namespace declaration.
- v0.29 App Memory + Outcomes Contract proves explicit namespace-scoped memory
  consumption and app outcome workflows.
- v0.30 App Canvas Contract proves durable canvas emission through audited
  workspace APIs.
- v0.31 Workspace-Only App UI proves `:panel` surfaces, host-owned named
  workspace zones, Settings Central inside `/workspace`, and the
  `/apps/<app_id>` convention for rare app pages.
- v0.32 User Theming And Layout Overrides proves token, snippet, and layout
  hooks from Allbert Home.

## Expected Direction

- Add `mix allbert.gen.plugin MyPlugin` to scaffold
  `./plugins/my_plugin/` with `allbert_plugin.json`, README, `lib/`,
  optional `skills/`, and validation docs.
- Add `mix allbert.gen.app MyApp` as an app-oriented plugin scaffold.
- Generate the proven local plugin/app shape: plugin manifest, plugin module,
  app module, app supervision wiring, sample registered Jido action, sample
  `SKILL.md`, sample panel surfaces, optional page surface under
  `/apps/<app_id>`, settings schema stub, memory namespace stub, optional
  objective scaffold, canvas wiring stub, theming/layout documentation, and
  validation documentation.
- Ensure `mix allbert.validate_plugin MyPlugin` passes immediately.
- Ensure `mix allbert.validate_app MyApp` passes immediately.
- Generated code is inert by default. It does not automatically add arbitrary
  plugin `lib/` paths to compilation, trust or enable generated skills, grant
  action permissions, register external publishing, load runtime code, or
  perform side effects without explicit operator action through existing
  Allbert boundaries.
- Consider optional `mix allbert.publish_skills` only after local validation,
  trust, and explicit operator review are boring and safe.

## Blocker Shapes To Pin Before Implementing

This plan is intentionally just-in-time. Before M-level work starts, pin:

- **Mix task arg signatures**: `mix allbert.gen.plugin NAME [--into PATH]
  [--force]`, `mix allbert.gen.app NAME [--namespace ATOM] [--into PATH]
  [--force]`, `mix allbert.validate_plugin NAME`, `mix allbert.validate_app
  NAME`, and optional `mix allbert.publish_skills`.
- **Scaffold output manifest**: exact file tree and template skeletons for
  plugin manifest, plugin module, app module, settings schema, panel surface
  stubs, optional page route notes, action stub, skill stub, memory namespace,
  objective scaffold, canvas emitter stub, theming/layout docs, validation
  docs, and README.
- **Inert-by-default assertions**: generated manifest has `trusted: false`;
  generated skill is disabled; no compile-path entry is added; no action is
  registered until explicitly wired; no Security Central policy is changed; no
  route is added except as developer-reviewed source diff.

Then expand milestone bodies to the v0.26 milestone shape
(Goal/Depends/Parallel-safe/tasks/Tests/smoke/Acceptance).

## Milestones

- M1: Generator contract inventory - freeze scaffold fields from proven plugin,
  app surface, workspace panel, settings, memory, objective, action, canvas,
  and theming contracts.
- M2: Plugin generator - add `mix allbert.gen.plugin` with inert manifest,
  README, optional skill root, and validation docs.
- M3: App generator - add `mix allbert.gen.app` with app module, panel-surface
  stub, optional page-surface notes, action stub, settings schema stub, memory
  namespace stub, objective stub, canvas wiring stub, and theming docs.
- M4: Validation tasks and docs - ensure generated outputs pass
  `mix allbert.validate_plugin` and `mix allbert.validate_app` immediately and
  explain explicit compile-path/project integration.
- M5: Safety closeout - prove generated code grants no trust, permissions,
  skill enablement, external publishing, dynamic loading, route authority, or
  execution authority.

## Non-Goals

- No generator before contracts are proven.
- No autonomous app or skill creation from traces, prompts, or repeated tasks.
- No runtime dynamic module loading from arbitrary plugin, app, or skill
  folders.
- No automatic compile-path changes for generated plugin code without explicit
  operator/developer review.
- No generated execution authority, automatic trust, automatic skill
  enablement, route authority, or permission grants.
- No marketplace publish without explicit operator action and review.
- No AG-UI, A2UI, MCP Apps, or external UI protocol bridge.

## Test Plan

- Generated plugin compiles and validates.
- Generated app plugin compiles and validates.
- Generated docs explain the explicit compile-path or project dependency step
  needed for code-bearing source-tree plugins.
- Generated skill is inert until explicitly enabled/trusted.
- Generated action goes through the registry, action runner, and Security
  Central.
- Generated panel surface data validates through `AllbertAssist.Surface` and
  cannot invent component ids, zones, or action bindings.
- Generated settings schema stub grants no authority by itself.
- Generated memory namespace stub is declaration-only until app code adds
  explicit memory actions.
- Generated canvas wiring uses the proven v0.30 path and grants no authority.
- Generated theming docs point at v0.32 token/snippet/layout hooks without
  creating arbitrary CSS/JS execution.

## Definition Of Done

v0.33 is done when a new local Allbert plugin and app plugin can be scaffolded,
validated, inspected, and reviewed without granting unsafe capability,
publishing externally, loading untrusted runtime code, or scaffolding unproven
app-contract layers.
