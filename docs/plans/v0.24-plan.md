## Purpose

v0.24 introduces an objective runtime layer between intent selection
and action execution. The intent layer continues to interpret the
user's immediate input and rank candidate routes. The action layer
continues to execute one bounded, authorized capability at a time.
Objectives become responsible for durable outcome state across one or
more steps, confirmations, jobs, channels, and turns.

The milestone adds three durable record types (`objectives`,
`objective_steps`, `objective_events`), one supervised Jido.Agent
(`AllbertAssist.Objectives.Engine`), a seven-stage state machine, four
registered CLI/LiveView actions for inspection and lifecycle, eleven
new signals, and four new settings keys. It threads `objective_id` and
`step_id` through Confirmations, Jobs, traces, and StockSage
`RunAnalysis`. It does not ship advisory provider behaviours, world-
model abstractions, capability inventory aggregation, route proposal,
acquisition options, or any planner beyond the deterministic step
proposer needed for the first one-step and two-step objectives.

Scope discipline matters here. The earlier rethink draft proposed a
sixteen-stage pipeline with 60+ named hook points, two provider
umbrellas, nine reserved provider roles, and 15+ settings keys. v0.24
ships the minimum implementation slice the rethink doc now describes:
seven engine stages, five step kinds, five named hook points, four
settings keys, three tables, eleven signals. Reserved vocabulary
(advisory providers, world model, capability/route/acquisition,
parallelism settings) is documented in ADR 0021 and the research note
but not implemented.

## Architecture Readiness Audit

- ADR 0006 keeps Security Central as the permission, risk, redaction,
  and audit boundary. Objective steps that mutate, fetch, send,
  execute, or contact external systems still flow through
  `Actions.Runner.run/3` and Security Central. `objective_id` carries
  no permission.
- ADR 0007 requires runtime-facing effectful behavior to enter through
  registered Jido actions. All objective lifecycle actions
  (`list_objectives`, `show_objective`, `cancel_objective`,
  `continue_objective`) are registered Jido actions. The engine agent
  itself dispatches commands internally; external callers go through
  the action runner.
- ADR 0008 keeps durable confirmation requests in SQLite. Confirmation
  records gain optional `objective_id` and `step_id` columns. Existing
  confirmations from prior versions remain valid (NULL in both
  columns).
- ADR 0014 keeps `user_id` as the identity boundary. Every objective
  has `user_id`. Cross-user objective access returns `:not_found`.
- ADR 0015 keeps `active_app` as session context, not authorization.
  Objectives capture `active_app` at creation; ranking and rendering
  can use it; permission decisions cannot.
- ADR 0019 invariants for intent candidates hold. v0.24 adds
  `:objective` as a candidate kind. Objective candidates are proposal
  data; they do not grant permissions or change execution mode.
- ADR 0021 (new in this milestone) defines the intent / objective /
  step / observation / capability / route / acquisition / advisory
  provider / world model / action authority boundary. This plan
  derives from ADR 0021.

## Vision And Roadmap Alignment

The vision describes Allbert as a signal-driven runtime where intent
is interpreted, capabilities are selected through registered actions,
permissions are checked, results and traces are recorded, and the
operator can inspect and steer work. v0.24 makes the "across multiple
turns" part of that loop a first-class concept.

Roadmap dependency order:

- v0.07 supplies the confirmation workflow that objective steps
  depend on for risky-action gating.
- v0.12 supplies the `Thread` + `Message` records that objectives
  reference via `source_thread_id`.
- v0.13 supplies the scheduled-job runtime that may create or advance
  objectives.
- v0.18 supplies the app/surface contract that objective views are
  rendered through.
- v0.19 supplies the intent engine that v0.24 extends with
  `:objective` candidate kind.
- v0.20 supplies StockSage as the first proving app.
- v0.21 supplies memory promotion that objective reflection steps may
  propose candidates for.
- v0.22 supplies the `RunAnalysis` confirmation flow that v0.24
  rewraps inside an objective.
- v0.23 supplies Confirmations.Store and Jobs.Scheduler as Jido.Agents
  so the engine can compose with them via the same substrate.

v0.24 must not pull later work forward. It does not implement:

- v0.25 native trading agents (Objectives.Engine has only a
  deterministic step proposer; LLM-assisted planners and StockSage
  specialist agents are v0.25+).
- v0.26 workspace shell or canvas (objective views render through
  existing LiveView surfaces; canvas integration is v0.30).
- v0.28 security hardening evals over objective surfaces (the
  engine's security tests cover correctness; adversarial evals are
  v0.28).
- Advisory provider behaviours, world-model abstractions, capability
  inventory aggregation, route proposal, acquisition options,
  diffusion proposers, market allocators, or probabilistic inference
  providers. These are reserved vocabulary in ADR 0021 only.

## Guiding Rules

1. **Objective state is not authority.** `objective_id` never grants
   permission. `active_app` may scope ranking and objective context,
   but not permission. Every step that mutates, fetches, sends,
   executes, or contacts external systems grounds to a registered
   action and Security Central.

2. **SQLite is authoritative.** `objectives`, `objective_steps`, and
   `objective_events` rows are durable truth. The engine agent's
   in-memory state is a bounded projection of the current active
   objective(s) for the current user/session; it rebuilds from SQLite
   on restart or cold start.

3. **Bounded loops.** Every objective has explicit caps:
   `objectives.max_steps_per_turn` (default 3),
   `objectives.max_loop_count` (default 5). Exceeding a cap records
   an `impasse` event, not a silent failure.

4. **Cooperative cancellation only.** `cancel_objective` transitions
   status to `:cancelled`, blocks new step creation, and lets in-
   flight registered actions complete normally. Mid-action
   interruption is deferred to v0.25+.

5. **Coexisting signals.** `allbert.objective.*` signals emit
   alongside existing `allbert.action.*` and `allbert.runtime.turn.*`
   events. No existing signal is removed or renamed. Trace volume is
   bounded by `objectives.trace_detail`.

6. **Cross-user isolation.** All objective operations filter by
   `user_id`. The CLI defaults to `"local"` only at the operator
   boundary. Actions never default `user_id` internally.

7. **No automatic completion.** Acceptance criteria are evaluated
   deterministically against observation summaries and step results.
   An LLM-assisted evaluator is reserved as vocabulary in ADR 0021
   but not implemented; v0.24 completion is a function of structured
   data, not free-form model output.

8. **Reserved vocabulary stays reserved.** Capability inventory,
   capability gap, route, acquisition option, advisory provider,
   world model, diffusion proposer, market allocator, and
   probabilistic inference provider are documented in ADR 0021 and
   the research note. They are not modules, settings, signals, or
   step kinds in v0.24.

9. **Pragmatic substrate rule.** `Objectives.Engine` is a Jido.Agent
   because it is a stage state machine with lifecycle hooks that
   carry their weight. Per v0.23's pragmatic rule, this is a per-
   component decision, not a uniform policy.

## Engine Stage Pipeline

The engine is a Jido.Agent implementing a seven-stage state machine.
Each stage is a declared command on the agent. Stage transitions are
driven by `on_after_cmd` hooks and `Jido.Action` directives.

| # | Stage | Command | Description |
|---|---|---|---|
| 1 | Receive | `:receive_input` | Accept a runtime turn; attach `objective_id` if resuming. |
| 2 | Interpret intent | `:interpret_intent` | Pass through to `Intent.Engine`; receive ranked candidates including `:objective` kind. |
| 3 | Frame/resume objective | `:frame_objective` | Decide: create new, resume existing, or no objective. Deterministic rules in v0.24. |
| 4 | Propose and evaluate steps | `:propose_steps` + `:evaluate_steps` | Deterministic proposer turns intent candidate into one or more step records. Deterministic evaluator scores feasibility. |
| 5 | Authorize selected step | `:authorize_step` | Pass through to `Actions.Runner` + Security Central + Confirmations. |
| 6 | Execute | `:execute_step` | Pass through to `Actions.Runner.run/3`. |
| 7 | Observe and advance | `:observe_step` | Record observation; evaluate against acceptance criteria; emit progress event; decide continue / block / complete. |

The reserved sixteen-stage breakdown from the rethink doc is not
implemented. The seven stages above are enough for the v0.24 acceptance
surface (single-action objectives, two-step "analyze and compare"
objectives, cancellation, resumption from confirmation approval).

### Engine state schema

```elixir
defmodule AllbertAssist.Objectives.Engine do
  use Jido.Agent,
    name: "objectives_engine",
    description: "Stage state machine for durable objective work",
    schema: [
      active_objectives: [type: :map, default: %{}],  # objective_id => projection
      current_stage: [type: :map, default: %{}],      # objective_id => stage atom
      loop_counts: [type: :map, default: %{}]         # objective_id => integer
    ]

  # commands ...
end
```

In-memory state is bounded by `objectives.max_active_per_user`
(reserved; default 16). Engine state is a cache; SQLite is
authoritative.

### Lifecycle hook wiring

| Hook | Use |
|------|-----|
| `on_before_validate_state` | Prune completed/cancelled/failed objectives older than 1 hour from in-memory projection (SQLite retains them). |
| `on_after_cmd :frame_objective` | Persist objective row; emit `allbert.objective.created` or `allbert.objective.updated`; record `objective_events` row. |
| `on_after_cmd :propose_steps` | Persist `objective_steps` row with `status: :proposed`; emit `allbert.objective.step.proposed`. |
| `on_after_cmd :authorize_step` | Update step `status` to `:selected` or `:blocked`; thread `objective_id` + `step_id` to `Confirmations.Store` when needed. |
| `on_after_cmd :execute_step` | Update step `status` to `:running` then `:completed` or `:failed`; emit `allbert.objective.step.completed` or `.step.failed`. |
| `on_after_cmd :observe_step` | Update objective `last_observation_summary` and `loop_count`; emit `allbert.objective.observed`. |
| `on_error` | Log; do not crash the engine on a single objective failure; record an `impasse` event for the affected objective. |

## Actions

All under `AllbertAssist.Actions.Objectives.*`.

### Read actions (`:read_only`)

- `list_objectives` — list objectives for `user_id` with optional
  filters: `status`, `active_app`, `since`, `limit`. Returns bounded
  Objective maps.
- `show_objective` — read one objective by `id`. Returns full
  Objective map plus steps list (capped at 50) plus recent events
  (capped at 20).

### Lifecycle actions (`:objective_write` — new permission class)

- `cancel_objective` — transition `status` to `:cancelled`. Blocks new
  step creation. Lets any in-flight registered action for this
  objective complete normally. Records audit event. Requires
  `:objective_write` permission (default `:allow`; safety floor
  `:allow`; not high-risk).
- `continue_objective` — resume an objective in `:blocked` status by
  re-running the framing stage. Used after a confirmation approval
  or after a paused step's blocking condition resolves. Requires
  `:objective_write` permission.

### `:objective_write` permission

New permission class added to Security Central:

- Default policy: `:allow` for the objective's owning `user_id`.
- Safety floor: `:allow` (this is not a high-risk class; the work
  that an objective performs goes through other action-level
  permission classes).
- Settings Central key: `permissions.objective_write` (writable).

The class exists for symmetry with other write classes
(`:memory_write`, `:stocksage_write`, `:settings_write`) and for
future ability to scope objective creation/cancellation when hosted
multi-user lands (v0.31+).

## Trace Sections

Trace markdown gains:

- `## Objective` — appears for any runtime turn that creates,
  resumes, advances, blocks, completes, or cancels an objective.
  Contents: objective id, title, status, current step, acceptance
  criteria summary, last observation summary, loop count.
- `## Objective Steps` — list of steps for the objective in this

## StockSage Integration

`StockSage.Actions.RunAnalysis` gains optional parameters:

- `objective_id` (string, optional)
- `step_id` (string, optional)

When invoked through the objective engine, both are present and
threaded into:

- The confirmation record (via `Confirmations.Store :create_pending`).
- The audit event (via Security Central).
- The trace markdown.
- The `stocksage_analyses` row (via two new nullable columns).

When invoked directly (e.g., `mix stocksage.analyze TICKER DATE`
without an active objective), both are NULL. Behavior is otherwise
unchanged from v0.22.

### Two acceptance flows for v0.24

**Single-step objective:**

```
User: "analyze AAPL" (with active_app: :stocksage)
  → Intent.Engine ranks RunAnalysis candidate
  → Engine frames objective: title "Analyze AAPL"
  → Engine proposes step: kind=action, candidate=RunAnalysis
  → Engine authorizes; Security Central returns :needs_confirmation
  → Confirmation created with objective_id + step_id
  → Operator approves
  → Step transitions selected → running
  → RunAnalysis executes
  → Step transitions running → completed
  → Engine evaluates acceptance: one completed analysis for AAPL on today's date → met
  → Objective transitions to :completed
```

**Two-step "analyze and compare" objective:**

```
User: "analyze AAPL and compare to MSFT"
  → Engine frames objective: title "Analyze AAPL and compare to MSFT"
  → Engine proposes step 1: action=RunAnalysis(AAPL)
  → Step 1 follows the single-step flow above → completed
  → Engine proposes step 2: action=RunAnalysis(MSFT), references step 1 result
  → Step 2 follows the same flow → completed
  → Engine evaluates acceptance: two completed analyses, comparison summary attached → met
  → Objective transitions to :completed
```

Both flows are exercised in M5 acceptance smokes.

## Intent Engine Integration

`AllbertAssist.Intent.Engine.collect_candidates/1` gains an
`:objective` candidate kind:

- When the operator has open objectives matching the current turn's
  topic, the engine returns an objective-resume candidate.
- The candidate's `source` is `:objective`; `score` is 0.2–0.5 based
  on recency and topic match; `reason` is the objective title.
- ADR 0019 invariants hold: objective candidates do not grant
  permissions; do not authorize actions; do not skip Security
  Central.

`Candidate.bound/2` adds `objective: 5` to `@default_kind_limits`.

Memory candidates (v0.21) and objective candidates can coexist in
the same ranking pass.

## Non-Goals

- No advisory provider behaviour (reserved in ADR 0021).
- No world-model abstraction (reserved).
- No capability inventory aggregation as a module (reserved).
- No route, acquisition option, or planner beyond the deterministic
  proposer.
- No diffusion proposer, market allocator, probabilistic inference
  provider.
- No parallel step execution (`max_parallel_steps` reserved at 1).
- No mid-action interruption (cooperative cancellation only).
- No LLM-assisted acceptance evaluator (deterministic only).
- No automatic memory promotion from objective observations; the
  v0.21 review surface remains the only path.
- No native StockSage trading agents (v0.25).
- No workspace shell upgrade (v0.26).
- No canvas integration (v0.30).
- No public plugin hook contribution API.
- No automatic capability acquisition (install plugin, write code,
  request credential) from an objective step.
