## Main User/Operator Flows

The flow below implements the v0.60-designed onboarding (ADR 0077/0069) and
applies the v0.60 persona model (ADR 0077/0075); QuickStart's first-model step
follows the First-Model Path resolved in ADR 0078 (v0.60 M3), not a path decided
here.

- New operator starts Allbert with the packaged entry points; the web wizard
  auto-opens on first run.
- Operator picks QuickStart or Advanced, or uses the equivalent terminal entry
  points: `allbert onboard --quickstart`, `allbert onboard --advanced`, or bare
  `allbert onboard` to resume.
- Operator reaches a first model through the ADR 0078 First-Model Path
  (assisted-local QuickStart by guided Ollama install/pull, with BYOK as
  Advanced/fallback), enters a masked provider key where required, stores it
  through the writable vault tiers (with the tier surfaced), sees inline doctor
  verification, chooses local or hosted provider/model, and can switch providers
  without editing config. Env-provided secrets are detected and verified as
  read-only.
- Operator selects a user-category profile, reviews exactly what it seeds, and
  confirms.
- QuickStart reaches a working first chat; Advanced may continue into optional
  channel/integration setup.
- Operator sees health/trust-spine confirmation.
- Headless operator runs the same flow through the interactive terminal wizard,
  the line-oriented fallback, or explicit automation with
  `--non-interactive --authorize` (deprecated `--accept-risk` compatibility alias
  warns and follows the same approval path).

## Data Flow

The shared wizard state machine (an upgrade of `onboarding.ex` per plan Locked
Decision 1) implements the v0.60-designed 8-step flow (ADR 0077/0069) and owns step
sequence, validation, transitions, and the unified onboarding-complete persistence
that `first_run.ex` reads. The web and terminal surfaces render it without forking
logic, seated in the **ADR 0080 consolidated sidebar** (v0.61b as-built). The
first-model step realizes the ADR 0078 First-Model Path (assisted-local, shipped in
v0.62 M4); v0.63 builds against that resolved decision and does not branch a new
one. Provider/model setup writes newly entered credential values through writable
secret tiers (OS vault → encrypted-store fallback) as Settings Central `*_ref`
pointers, detects env-provided secrets as read-only inputs, surfaces the tier, and
verifies them through existing doctor flows. Readiness labels follow the plan's
**Readiness Label Mapping Contract** — the six model-probe states map to
`Ready` / `Needs runtime` / `Needs model`, `below_hardware_floor` maps to
`Needs review`, and `Needs credentials` / `Needs review` come from the
provider-step / profile layer (not the model probe) — always shown as one next
action, never a raw atom. Profile selection applies the v0.60 persona
model (ADR 0077/0075): it reads repo-maintained declarative presets and, after
explicit confirmation, seeds Settings Central defaults plus suggested apps,
channels, and intents. Every effectful step remains behind the existing
action/settings/security gates.

## Authority Invariants

No new capability is introduced. Profiles seed defaults only; they lower no
confirmation floor, grant no egress, and enable nothing effectful without normal
gates. Settings/profile writes flow only through Settings Central. Masked-key
entry never echoes or logs raw secrets; wizard diagnostics are redacted.
Non-interactive `--authorize` does **not** bypass the confirmation floor: each
gated step (Ollama install, model pull, persona apply) is created and approved as
a durable, traced `Confirmations` record through the existing approve path — an
explicit, logged operator pre-authorization of existing gated actions, granting no
new authority.
