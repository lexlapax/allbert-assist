## Main User/Operator Flows

The flow below implements the v0.60-designed onboarding (ADR 0077/0069) and
applies the v0.60 persona model (ADR 0077/0075); QuickStart's first-model step
follows the First-Model Path resolved in ADR 0078 (v0.60 M3), not a path decided
here.

- New operator starts Allbert with the packaged entry points; the web wizard
  auto-opens on first run.
- Operator picks QuickStart or Advanced.
- Operator reaches a first model through the ADR 0078 First-Model Path
  (assisted-local QuickStart by guided Ollama install/pull, with BYOK as
  Advanced/fallback), enters a masked provider key where required, stores it
  through the three-tier vault (with the tier surfaced), sees inline doctor verification, chooses local or hosted
  provider/model, and can switch providers without editing config.
- Operator selects a user-category profile, reviews exactly what it seeds, and
  confirms.
- QuickStart reaches a working first chat; Advanced may continue into optional
  channel/integration setup.
- Operator sees health/trust-spine confirmation.
- Headless operator runs the same flow through the interactive terminal wizard.

## Data Flow

The shared wizard state machine (an upgrade of `onboarding.ex` per plan Locked
Decision 1) implements the v0.60-designed 8-step flow (ADR 0077/0069) and owns step
sequence, validation, transitions, and the unified onboarding-complete persistence
that `first_run.ex` reads. The web and terminal surfaces render it without forking
logic, seated in the **ADR 0080 consolidated sidebar** (v0.61b as-built). The
first-model step realizes the ADR 0078 First-Model Path (assisted-local, shipped in
v0.62 M4); v0.63 builds against that resolved decision and does not branch a new
one. Provider/model setup writes credential values through the **three-tier** secret
backend (OS vault → encrypted-store fallback → env) as Settings Central `*_ref`
pointers — surfacing the tier the value landed in — and verifies them through
existing doctor flows. Profile selection applies the v0.60 persona
model (ADR 0077/0075): it reads repo-maintained declarative presets and, after
explicit confirmation, seeds Settings Central defaults plus suggested apps,
channels, and intents. Every effectful step remains behind the existing
action/settings/security gates.

## Authority Invariants

No new capability is introduced. Profiles seed defaults only; they lower no
confirmation floor, grant no egress, and enable nothing effectful without normal
gates. Settings/profile writes flow only through Settings Central. Masked-key
entry never echoes or logs raw secrets; wizard diagnostics are redacted.

## Handles / CLI Surfaces

- Web auto-first-run wizard.
- Interactive terminal wizard through the packaged `allbert` CLI.
- Provider/model setup with masked three-tier-vault entry, inline doctor, and provider
  switch.
- Registered settings action to apply profile presets with review/confirm.
- Health/trust-spine confirmation surface.
- `mix allbert.test release.v063` deterministic gate.

Each S-gate states what it gates, its preconditions, and its pass criteria;
sign-offs are stamped inline as milestones land.

### S1 — Shared state-machine contract (gates M2–M6)

Precondition: M1's step IDs, transitions, track branching, and the **unified
persistence** (one "onboarding complete" source; `first_run.ex` reads it;
placeholders retired) are ratified. Pass: both surfaces provably drive identical
step IDs; `allbert onboard` launches the wizard.

### S2 — Provider/model + First-Model Path (gates the model steps of M5/M6)

Precondition: M2 (7-state track behavior) + M3 (three-tier masked vault write,
inline doctor, provider switch incl. custom endpoint) pass. Pass: QuickStart reaches
first chat from each first-model state; no raw secret leaks; the storage tier is
surfaced.

### S3 — Persona system (gates the persona step of M5/M6)

Precondition: M4's `priv/` catalog + registry + apply action + exact seed values
pass, seeding only `@safe_write_keys`. Pass: review/confirm shows exactly what is
seeded and writes nothing before confirm; grants no authority; usable outside
first-run.

### S4 — Web wizard (gates the web closeout)

Precondition: M5 auto-opens on first run and renders M1–M4 through ADR 0074 on the
ADR 0080 consolidated sidebar. Pass: a new operator reaches first useful chat on web
with no docs/config edits.

### S5 — Terminal wizard (gates the headless closeout)

Precondition: M6 renders the same machine via the TUI console (TTY) and the line-CLI
`allbert onboard` (headless), no surface fork. Pass: a headless operator completes
the same flow.

### S6 — Trust-spine + eval gate + ADR acceptance (release closeout)

Precondition: M7 trust-spine surface + first-chat checkpoint; the 10 `:v063` rows +
ADR-accept rows registered; `mix allbert.test release.v063` deterministic and green;
ADR 0069 + 0075 Accepted. Pass: no warning/security/doc drift.
