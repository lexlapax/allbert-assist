# v0.11 — Self-improvement

Status: Proposed

## Context

By v0.10, Allbert has a hardened kernel (v0.6), a real agent harness, a rich skill format, curated memory, at least one approval-capable non-REPL channel, a continuity model for cross-channel review and resumable work, an explicit contributor/development contract for rebuilding the source tree, and a broader local-first provider layer with Anthropic, OpenRouter, OpenAI, Gemini, and Ollama. v0.11 is about the assistant helping improve itself: two specialised coding skills (Rust for the runtime, a skill-authoring skill for Python/Bash/etc.) plus a principled scripting seam so future embedded runtimes (Lua, etc.) enter through one controlled door rather than by the kernel picking up a new heavyweight dependency.

This is both the most interesting and the highest-risk release on the roadmap. Self-authored code can extend Allbert; self-authored code can also break Allbert. The design has to make the "improve Allbert" path first-class while routing every artifact through existing trust gates.

## End-user framing

Consistent with ADR 0038, v0.11 does **not** require end users to write Rust, Python, or Lua.

- The Rust rebuild skill is Allbert coding on itself, under explicit operator diff review. The user reviews and approves diffs; the user does not author them, and the user never sees an auto-swapped binary. Installing the built binary is always an explicit operator action (ADR 0068).
- The skill-authoring skill scaffolds new skills through natural-language conversation. The user describes what they want; Allbert produces a folder-format skill that lands in the same install quarantine as any external install. This is the user-facing front door for skill creation — not a back-office developer tool.
- The embedded Lua scripting engine is an opt-in advanced surface for users who want a small, sandboxed DSL. It is not a requirement for any end-user workflow and stays disabled in `security.exec_allow` by default.

Self-improvement is a capability Allbert uses on the user's behalf, within the trust gates that govern everything else.

## Goal

Ship three capabilities, all framed as "Allbert does this for the user under review":

1. A **Rust rebuild skill** (`rust-rebuild`) that can read, modify, build, and test the Allbert codebase inside a sibling worktree, producing diffs the operator reviews through a new `patch-approval` inbox kind before any merge.
2. A **skill-authoring skill** (`skill-author`, user-facing) that scaffolds new AgentSkills-format skills through natural-language conversation and submits them through the same install preview + confirm flow every other skill uses (ADR 0033). This replaces the v0.4 `skills init` CLI wizard as the default skill-authoring front door; `skills init` remains as the CLI escape hatch for technical users.
3. A **scripting engine seam** (`ScriptingEngine` trait) behind which embedded runtimes (Lua, later others) can attach, with one concrete default implementation (`mlua` in sandbox mode), all of it opt-in through the existing security allow/deny policy.

## Planning-question answers (decisions baked in)

These resolve the open questions flagged in pre-implementation review. They are design decisions for v0.11 implementation and are reflected in the milestones and ADRs below.

| Question | Decision for v0.11 |
| --- | --- |
| What test-pass threshold makes a rebuild "safe to merge"? | Tier A validation from ADR 0064 (`cargo fmt --check`, `cargo clippy -D warnings`, `cargo test`, `cargo run -- --help`, temp-home `daemon status` smoke) must pass cleanly; optional live-provider smokes (Tier B) are **not** required. Any Tier A failure routes the patch as `needs-review`, not `safe-to-merge`. |
| Are self-authored skills marked differently from external installs? | Yes. The folder-format frontmatter gains a `provenance:` field (`external`, `local-path`, `git`, or `self-authored`). It is surfaced in `allbert-cli skills list` output and in the install preview. Trust posture is unchanged — the same preview + confirm flow runs regardless. |
| Do embedded Lua scripts share memory with the host session? | No. Scripts receive serialized inputs and return serialized outputs. No shared Rust references, no pointers into session state, no direct memory service access. Any data flow goes through the same tool surface everything else uses. |
| Is conversational skill-authoring a first-party skill or a kernel-native agent? | First-party skill, matching ADR 0048's pattern for `memory-curator`. It ships under `skills/skill-author/` in the repo and installs to `~/.allbert/skills/installed/skill-author/` on first run. Intent routing (ADR 0030) dispatches natural-language requests ("make me a skill that...") to it. The CLI wizard (`allbert-cli skills init`) remains available as an escape hatch for users who prefer it. |

## Research checkpoint

| Topic | Decision |
| --- | --- |
| Lua runtime crate | Use `mlua` with `Lua::sandbox(true)` as the embedded runtime for v0.11. Rationale: actively maintained, well-understood sandbox primitive, sync and async APIs, explicit memory-limit and instruction-count hooks. `rlua`, `piccolo`, and `hlua` were considered; `mlua` is the closest match to the "small, auditable seam" goal. |
| Worktree strategy | Isolated `target/` directory per worktree. Each `~/.allbert/worktrees/<branch>/` is a full `cargo` working copy with its own build artifacts. Slower first build, but zero risk of corrupting the main checkout's incremental state. |
| Binary swap policy | No automatic swap. A successful Tier A rebuild produces a patch (`cargo build`-ready) plus a built binary under the worktree's `target/release/`. The operator either accepts the diff and runs their own `cargo install`/`daemon upgrade` sequence, or rejects it. The daemon never replaces its own running binary (ADR 0068). |
| Patch-approval surface | Extend the v0.8 approval inbox (ADR 0060) with a new `kind: patch-approval`. Renders diffs with summary + file-by-file navigation rather than the tool-approval renderer's single-shot confirmation shape. |
| cargo network access | `cargo build` hitting crates.io is a build step, not background web learning. Governed by the exec policy (ADR 0004/0034) only; explicitly **not** subject to ADR 0053's explicit-intent web-learning gate. |

## Milestones

### M0 — ADR freeze

Before any code ships, flip ADRs 0067 through 0073 from `Proposed` to `Accepted` as the v0.11 spec freeze. Update amendment banners on ADRs 0032, 0056, and 0060 to reference the new ADRs (provenance field, inbox amendment chain, patch-approval kind respectively).

### M1 — Worktree model and source-checkout detection

- Add `self_improvement.source_checkout: Option<PathBuf>` to config, default `None`. The kernel resolves the active source checkout in this order:
  1. `self_improvement.source_checkout` if set;
  2. `$ALLBERT_SOURCE_CHECKOUT` env var;
  3. walk up from the current executable's symlink-resolved path looking for a `Cargo.toml` with `[workspace].members` including `allbert-kernel`.
- If none resolves, `rust-rebuild` refuses to activate with an operator-readable error directing the user to `allbert-cli self-improvement config`.
- `self_improvement.worktree_root: PathBuf` defaults to `~/.allbert/worktrees/`.
- `self_improvement.max_worktree_gb: u32` defaults to `10`. The daemon refuses to create a new worktree when the existing tree already exceeds the cap; stale worktree GC is operator-driven (`allbert-cli self-improvement gc`) in v0.11.
- Verify the pinned Rust toolchain (ADR 0063) is present on `rustup show`. If not, the rebuild skill refuses with a clear error rather than silently falling back to host `rustc`.

### M2 — Rust rebuild skill scaffold

- Ship `skills/rust-rebuild/` as a first-party skill.
- Skill workflow:
  1. Create a sibling worktree from the resolved source checkout: `git worktree add ~/.allbert/worktrees/<branch> <branch-name>`.
  2. Apply agent-generated edits inside the worktree only. The skill's exec policy explicitly forbids writing to the source checkout's working tree.
  3. Run the Tier A validation chain through the existing exec seam (`cargo fmt --check`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo test -q`, `cargo run -q -p allbert-cli -- --help`, temp-home `daemon status` smoke).
  4. On success, generate a unified diff (`git -C <worktree> diff <branch-point>..HEAD`) and emit a patch artifact.
  5. On any failure, collect failing test output and emit a `needs-review` artifact with the same diff plus the failure context.
- Cost-cap semantics (resolving v0.11's cost-cap-plus-long-builds ambiguity): Tier A validation counts as a single turn-unit for ADR 0051's daily cap. Wall-clock build time is **not** counted against the cap; only token usage by the model driving the rebuild is metered. A rebuild turn that exceeds the daily cap mid-run refuses at the next turn boundary per ADR 0051.
- The skill MUST NOT modify files within `skills/rust-rebuild/` itself in the active source checkout. That is enforced by a path-allowlist check in the skill's exec policy; violations fail the turn.
- The skill has no read or write access to `~/.allbert/` user data (sessions, memory, identity, secrets). Only `~/.allbert/worktrees/<branch>/` is readable/writable.

### M3 — Patch-approval inbox kind (amends ADR 0060)

- Add `kind: patch-approval` to the inbox file layout. Frontmatter mirrors existing approval kinds with additional fields: `source_checkout`, `branch`, `worktree_path`, `validation: { fmt, clippy, test, cli_help, daemon_smoke }` (each a `passed | failed | skipped`), `artifact_path` (absolute path to the diff file on disk).
- The diff itself is not inlined into the approval markdown (diffs can be thousands of lines). Instead, the approval references `artifact_path` pointing at `sessions/<sid>/artifacts/<aid>/patch.diff`.
- Renderer: `allbert-cli inbox show <aid>` for `patch-approval` surfaces:
  - a one-line summary (`N files changed, +X/-Y lines, Tier A: <status>`);
  - file-by-file summary (file path, additions, deletions);
  - a reminder that accepting the patch only marks it approved — it does not apply the binary.
- Resolution flow:
  - `inbox accept <aid>` marks the patch approved, records approver identity + reason, and does nothing else. Operator is directed to the next step: `allbert-cli self-improvement install <aid>` (see M4).
  - `inbox reject <aid>` marks the patch rejected and deletes the worktree (configurable via `self_improvement.keep_rejected_worktree`, default `false`).
- The v0.8 inbox retention policy (`channels.approval_inbox_retention_days`, default 30) applies to `patch-approval` alongside existing kinds.

### M4 — Rebuild install flow (no auto-swap)

- Add `allbert-cli self-improvement install <aid>` — requires the patch approval to be in `accepted` state, then:
  1. Applies the patch to the source checkout (`git apply` or `git merge` depending on `self_improvement.install_mode`, default `apply-to-current-branch`);
  2. Emits a build-and-install hint rather than running `cargo install` directly: the operator runs `cargo install --path crates/allbert-cli` (or equivalent) themselves. The rationale is recorded in ADR 0068: a long-lived daemon must not swap its own running binary.
  3. Records the install event in `~/.allbert/self-improvement/history.md` (append-only operator log).
- `allbert-cli self-improvement diff <aid>` renders the stored diff without applying it (for review from any surface).
- `allbert-cli self-improvement gc` removes worktrees older than `self_improvement.worktree_max_age_days` (default `14`) whose parent approval is resolved. Never runs automatically.

### M5 — Skill-authoring skill (`skill-author`)

- Ship `skills/skill-author/` as a first-party user-facing skill. Installed under `~/.allbert/skills/installed/skill-author/` on first run.
- Intent routing (ADR 0030) dispatches natural-language requests like "make me a skill that drafts standup updates" or "I want a skill to summarize my calendar" to the skill. Operator-facing phrasing is markdown-first, no code-writing step.
- Conversational intake sequence:
  1. Name (kebab-case, unique, ≤64 chars per ADR 0032);
  2. Description (≤1024 chars);
  3. Capability summary (what it does, what inputs/outputs);
  4. Interpreter choice if scripts are needed: Python (recommended), Bash, or other interpreters already in `security.exec_allow`. Lua appears as an option only if `security.exec_allow` contains `"lua"` and the scripting engine is enabled (M7/M8);
  5. Tool needs (`allowed-tools` field);
  6. Agent contribution (optional, per ADR 0031).
- Draft persistence: iterative drafts live at `~/.allbert/skills/incoming/<draft-name>/` across turns, surviving session exit. The drafts directory IS the standard quarantine path (ADR 0033). No auxiliary scratch directory.
- Each refinement round re-runs the in-memory tier-1/tier-2 prompt preview (ADR 0036) so the operator sees what the skill would contribute before the final preview step.
- Final submission routes through the standard install preview + confirm flow (ADR 0033). The skill-authoring skill does **not** call the `create_skill` kernel tool (which writes directly to `installed/`). That tool is explicitly forbidden to `skill-author` via its `allowed-tools` fence. (See also: `create_skill` hardening in M6.)
- Capability scope for v0.11:
  - ✅ Authors skills with `scripts/` (bash, python, or any interpreter already on the allowlist);
  - ✅ Authors skills that contribute agents via ADR 0031 frontmatter;
  - ✅ Authors skills that reference other installed skills by name in their body;
  - ✅ Authors skills with `references/` and `assets/` subdirectories;
  - ❌ Authors skills declaring Lua scripts unless the scripting engine is enabled and `lua` is in `security.exec_allow`. This is enforced by preview-time validation.
- Provenance: the draft frontmatter carries `provenance: self-authored`. On promotion from `incoming/` to `installed/`, the tag persists. (ADR 0071.)

### M6 — Skill frontmatter `provenance` field + `create_skill` hardening (amends ADR 0032)

- Add `provenance` to ADR 0032's optional frontmatter fields: `external` (default for git/path installs), `local-path`, `git`, or `self-authored`. Validation is additive; existing skills without the field load as `external`.
- Install preview (ADR 0033) surfaces `provenance` alongside `name`, `description`, and `allowed-tools`.
- `allbert-cli skills list` adds a `Source` column showing the provenance tag.
- Harden `create_skill`: it continues to write to `installed/` for existing callers (kernel-native use), but gains a new required argument `skip_quarantine: bool`. When `false` (the new default), it writes to `~/.allbert/skills/incoming/` and emits an install-preview event instead. `skill-author` uses `skip_quarantine: false`; first-party kernel seeding still uses `skip_quarantine: true` and is audited in hook metadata.
- ADR 0032 gains an amendment banner noting the new field (M0).

### M7 — `ScriptingEngine` trait and `mlua` implementation

- New kernel module `crates/allbert-kernel/src/scripting/` with:
  ```rust
  trait ScriptingEngine: Send + Sync {
      fn name(&self) -> &'static str;
      fn capabilities(&self) -> ScriptingCapabilities;
      fn load(&self, source: &str, source_ref: &str) -> Result<LoadedScript>;
      fn invoke(
          &self,
          script: &LoadedScript,
          inputs: serde_json::Value,
          budget: ScriptBudget,
      ) -> Result<ScriptOutcome>;
      fn reset(&self) -> Result<()>;
  }
  ```
- Inputs and outputs are serde-JSON only. No shared references, no pointers, no direct memory service access. Serialization boundary is the trust boundary (answers planning question #3).
- `LuaEngine` implements the trait using `mlua` with `Lua::sandbox(true)` plus an explicit stdlib allowlist (see M8).
- Integration with skills: Lua scripts are declared in `SKILL.md` frontmatter under `scripts:` just like any other interpreter (ADR 0034), with interpreter string `lua`. This keeps the skill-author contract uniform.
- Integration with exec hooks (ADR 0052): embedded Lua invocations surface as a synthetic tool `exec.lua:<skill>/<script-path>` in `BeforeTool` / `AfterTool` events, carrying the same metadata shape as process exec (interpreter, skill name, script path, budget used). This is the naming convention for any future embedded engine: `exec.<name>:<ref>`.
- The scripting engine is a kernel-owned extension surface. Adding a future engine (WASM, QuickJS) requires a new trait implementation but does not require a new top-level ADR — ADR 0069 covers the seam, ADR 0070 covers the sandbox policy contract any implementation must satisfy.

### M8 — Sandbox policy and config schema

- Extend `security` config with a `scripting` subsection:
  ```toml
  [security.scripting.lua]
  sandbox_mode = "strict"          # strict = mlua sandbox + stdlib allowlist below
  max_execution_ms = 1000
  max_memory_kb = 65536
  max_output_bytes = 1048576
  allow_stdlib = ["string", "math", "table"]
  deny_stdlib = ["io", "os", "package", "require", "debug"]
  ```
- `sandbox_mode = "strict"` is the only supported value in v0.11. Future modes are intentionally not introduced.
- Enabling Lua end-to-end requires both:
  - `security.exec_allow` includes `"lua"` (per ADR 0034's interpreter opt-in pattern);
  - `scripting.engine = "lua"` at the top level of config.
- Neither is defaulted on. Fresh profiles ship with Lua disabled. The install preview surfaces required interpreters (M6) so a Lua-using skill tells the operator up front.
- Hooks observability: every embedded-script invocation records `budget_used` (instructions executed, bytes allocated, wall-ms) in the `AfterTool` metadata. Operators can audit Lua usage per session from the existing hook log surface.
- A violation of any cap (time, memory, output size) terminates the script, emits an `AfterTool` event with `outcome: cap-exceeded`, and returns an error result to the calling skill. The host is never panicked by a runaway script.

### M9 — Integration tests

New tests under `crates/allbert-daemon/tests/` and `crates/allbert-kernel/tests/` (or equivalent):

1. **Rebuild round-trip**: scripted LLM response produces a known patch → `rust-rebuild` runs Tier A → inbox receives a `patch-approval` → operator accept → install command applies the patch to a fixture source tree. No live provider.
2. **Rebuild Tier A failure**: patch introduces a clippy warning → `rust-rebuild` emits `needs-review` → inbox renders failure context. Accept path refuses with a clear error.
3. **Rebuild source isolation**: patch attempts to modify `skills/rust-rebuild/` in the source checkout → exec policy denies → turn fails with path-allowlist violation.
4. **Self-authored skill install**: scripted LLM response drives `skill-author` through the intake sequence → draft lands in `incoming/<name>/` → install preview surfaces `provenance: self-authored` → confirm promotes to `installed/<name>/` with provenance intact.
5. **Lua script invocation**: a fixture skill declares a Lua `scripts:` entry → kernel loads the engine → `BeforeTool` / `AfterTool` events fire with the `exec.lua:<ref>` tool name → budget accounting matches declared caps.
6. **Lua sandbox escape attempts**: scripts attempting `io.open`, `os.execute`, `require "socket"` → all denied with clear errors, no host effect.
7. **Lua cap enforcement**: scripts exceeding `max_execution_ms`, `max_memory_kb`, and `max_output_bytes` respectively → each terminates cleanly, emits `outcome: cap-exceeded`, returns an error to the caller.
8. **cargo-network policy**: a rebuild turn making a `cargo build` call that fetches a new dep is allowed (build step, not web-learning); an in-skill attempt to wrap `cargo` around a direct crates.io HTTP fetch is governed by ADR 0053 (explicit-intent web learning).

Required validation remains ADR 0064 Tier A: `cargo fmt --check`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo test -q`, `cargo run -q -p allbert-cli -- --help`, temp-home `daemon status` smoke.

## ADRs

- [ADR 0067](../adr/0067-self-modification-uses-a-sibling-worktree-with-operator-diff-review.md) — self-modification happens in a sibling worktree; operator reviews diffs through the inbox.
- [ADR 0068](../adr/0068-rebuild-binary-swap-requires-explicit-operator-action.md) — rebuild never auto-swaps the running binary; install is an explicit operator step.
- [ADR 0069](../adr/0069-scripting-engine-trait-with-lua-as-the-v0-11-default-embedded-runtime.md) — `ScriptingEngine` trait; `mlua`-backed Lua as the v0.11 default.
- [ADR 0070](../adr/0070-embedded-script-sandbox-policy.md) — embedded-script sandbox policy: builtins allowlist, time/memory/output caps, no fs/net by default.
- [ADR 0071](../adr/0071-self-authored-skills-route-through-the-standard-install-quarantine.md) — self-authored skills use the standard install quarantine; amends ADR 0032 to add the `provenance` frontmatter field.
- [ADR 0072](../adr/0072-skill-authoring-is-a-first-party-natural-language-skill.md) — skill-authoring is a first-party natural-language skill; CLI wizard is the escape hatch; reinforces ADR 0038 and ADR 0048.
- [ADR 0073](../adr/0073-rebuild-patch-approval-is-a-new-inbox-kind.md) — rebuild patch approval is a new `patch-approval` inbox kind; amends ADR 0060.

## Dependencies

- v0.4 AgentSkills format — `skill-author` produces folder-format skills; `rust-rebuild` ships as a folder-format skill.
- v0.4 install preview + confirm — the path self-authored skills take.
- v0.4 script policy — interpreter allowlist governs what new skills can execute; `lua` is a new optional entry.
- v0.6 hardening — cost cap enforcement and session durability apply to rebuild turns just like any other work.
- v0.7 channel expansion — operators may approve rebuilds or skill installs from at least one non-REPL channel; tool-surface normalization (ADR 0052) makes embedded-script hook observation uniform.
- v0.8 continuity and sync — approval inbox (ADR 0060) is extended here with `patch-approval`; cross-channel continuity makes self-improvement review legible outside the REPL.
- v0.9 developer-environment readiness — rebuild and coding flows operate against an explicit contributor/toolchain contract (ADR 0063); Tier A validation (ADR 0064) is the rebuild safe-to-merge bar.
- v0.10 provider expansion — self-improvement can run in more operator environments (local Ollama, hosted providers) without forcing one vendor path.

## Non-goals

- Autonomous self-modification (no human in the loop).
- Automatic binary swap of the running daemon. Install is always operator-driven (ADR 0068).
- Automatic publishing of self-authored skills outside the local machine.
- Scripting engines beyond the one shipped default (Lua/mlua). Other engines slot in behind `ScriptingEngine` without a new top-level ADR pass.
- A kernel-wide plugin system. Skills + scripting engine seam is the stated extension surface; heavier plugin work is explicitly deferred.
- Rebuild skill writing to `~/.allbert/` user data, or patching its own source files in the active checkout.
- In-process Lua sharing memory with the host session.
- Auto-promotion of self-authored skills (they still go through preview + confirm).
- Local model retrain / micro-GPT distillation for memory or personality. Origin-note ambition; still philosophically in scope but requires curated memory, continuity, provider-management seams, and the new contributor contract to be stable first. Revisit after v0.11 lands.

## Exit criteria

- `rust-rebuild` can produce a Tier-A-passing patch against a fixture source checkout, route it through a `patch-approval` inbox entry, and be accepted/installed via the documented operator flow.
- `skill-author` can drive a natural-language intake to a `provenance: self-authored` skill in `incoming/`, survive session exit, and promote through the standard install preview + confirm flow.
- `mlua` engine loads, invokes, and sandboxes Lua scripts with all caps enforced; sandbox escape attempts are denied; hook metadata shows `exec.lua:<ref>` tool names.
- Tier A validation is green; integration tests M9.1–M9.8 pass.
- All new operator docs shipped and referenced from README and onboarding.
- Roadmap and vision updated; crate versions bumped.
