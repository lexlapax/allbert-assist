## Purpose

Give Allbert a **Pi-mode coding surface**: a gated terminal coding loop that
feels as minimal as Pi (four tools, a sub-1000-token prompt, streamed diffs,
full-file context) **while running on the one authority spine** — the same
`AllbertAssist.Actions.Runner.run/3`, Security Central, trace, and markdown-first
memory as every other surface.

Stated structurally (rationale §4): Allbert's cognitive runtime is the
seven-stage machine **Receive → Interpret → Frame/resume objective →
Propose/evaluate steps → Authorize step → Execute (`Actions.Runner.run/3`) →
Observe/advance.** Pi is `{Receive → Interpret(implicit) → Execute(YOLO) →
Observe}` — it deletes stages 3–5. v0.57 makes **stages 6–7 and the tool surface
Pi-minimal** (four boundary actions, streamed split results, observable,
sub-1000-token prompt) **while leaving stages 3–5 fully intact**. A Pi-mode
session is one where the objective is "pair on this repo," step proposal is
trivial, and authorization runs at **sandbox level 0** for a single trusted local
operator. The gates are *present but cheap*, never *absent*.

This is a **channel/app under ADR 0016 on the existing spine** — NOT a sibling
runtime, NOT a new authority. Four things land:

1. Four boundary actions (`read`/`write`/`edit`/`bash`) routed through the action
   runner and Security Central.
2. A sub-1000-token coding system prompt + full-file context-gathering discipline.
3. A named **"local coding / sandbox level 0"** trust tier (extends ADR 0009) —
   single trusted operator, main session, terminal channel; never the default,
   never for channel-originated or generated-code sessions.
4. Streamed diffs via the v0.55 split tool-result payload, plus an operator
   affordance for mid-session model switch (`req_llm` context handoff), and the
   minimalism budget as a standing coding policy.

## Readiness Sweep
- **Contract fit.** The Pi-mode surface is a delivery+driver experience over the
  v0.55 `tui` channel; it owns no intent/Security Central/Resource Access/
  confirmation storage/conversation history. Coding effects reach the host *only*
  through registered actions executed by `Actions.Runner.run/3`. The runner +
  Security Central decide; the surface renders and dispatches.
- **Reuse fit.** Reuse `plugins/allbert.tui` (normalize → `Identity.resolve` →
  `Channels.derive_session_id` → `Runtime.submit_user_input` → `ChannelThread` →
  `channel_events`), the v0.55 split-result contract (ADR 0029) and unified
  renderer (ADR 0030) for streamed diffs, the existing `actions/` boundary as the
  template for the four new coding actions, ADR 0009 sandbox levels for the trust
  tier, the v0.03 skill loader + `AGENTS.md` hierarchy for context, and req_llm's
  canonical `Context` (ADR 0047/0051) for the model switch.
- **Upstream check.** Confirm v0.55 delivered the `tui` channel AND the split
  tool-result payload through the typed response contract + renderer. If either
  is missing, stop and reconcile v0.55/v0.56 before implementing v0.57.
- **Authority check (AGENTS.md Non-Negotiables).** Advisory output is never
  authority: the coding model may propose `read`/`write`/`edit`/`bash` calls but
  may not bypass `Actions.Runner.run/3`, Security Central, or confirmations. The
  bash action is policy-bounded host execution behind a registered action — OTP
  supervision is **not** an OS security boundary. No model-decides-it's-done for
  effectful or generated-code work. No sub-agent-via-bash-spawn.
- **Library fit.** No new library is required; `owl` (v0.55, terminal render) and
  `req_llm` (provider neutrality + `Context` handoff) cover the surface. Context7
  confirms req_llm `Context` shape before the M4 switch affordance.

## Locked Decisions
| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Surface authority | Pi-mode is a **channel/app on the one spine** (ADR 0016): every coding effect routes through `Actions.Runner.run/3` + Security Central. The surface has **no authority beyond any other channel**. | A sibling runtime or a direct-effect coding loop reintroduces Pi's YOLO threat model — the thing Allbert exists to reject. |
| 2 | Tool surface | Exactly **four default boundary actions** for coding — `read`, `write`, `edit`, `bash` — each a registered Jido action via `Actions.Registry`/`Runner`. Everything else is lazy-disclosed (MCP-first, ADR 0038/0039), not pre-loaded. | A wide default tool surface defeats the minimalism budget and the lazy-disclosure discipline Allbert already has from v0.03. |
| 3 | Trust tier | A named **"local coding / sandbox level 0"** tier (extends ADR 0009): single trusted operator, main session, `tui` channel. **Never** the default; **never** for channel-originated inbound or generated-code sessions. Gates stay present, run cheaply. | An implicit or default level-0 tier turns a low-friction convenience into Pi-by-default and breaks the action boundary. |
| 4 | Acceptance | **Deterministic acceptance rules**, not model confidence. For effectful or generated-code work the model never "decides it's done"; evidence + confirmation rule (same posture Allbert took against Hermes). | Model-confidence completion contradicts "advisory output is never authority" and the codegen committee's deterministic gates. |
| 5 | Prompt + context | A **sub-1000-token** coding system prompt; full-file context-gathering discipline (gather context deliberately, feed full files not snippets). Reuse the `AGENTS.md` hierarchy + v0.03 skill loader. | A large default prompt or snippet-context contradicts Pi's strongest empirical findings and the minimalism budget. |
| 6 | Diff rendering | Streamed diffs ride the **v0.55 split tool-result payload** (`model_payload` vs. `surface_payload`) through the unified renderer (ADR 0030); the terminal renders the surface payload while the model sees the clean model payload. | A single conflated payload re-bloats model context and loses the streamed-diff ergonomics v0.57 builds over the v0.55 substrate. |
| 7 | Model switch | Mid-session model switch is an **operator affordance** over req_llm's canonical `Context` handoff (ADR 0047/0051), surfaced in the `tui` session; it changes the model, never the authority path. | Hiding the switch wastes a structurally-cheap req_llm capability Pi proved is worth surfacing. |
| 8 | Sub-agents | Delegation uses **restricted-registry delegate agents**, never an opaque `bash`-spawned sub-agent. | Self-spawn via bash is an unobservable, unrestricted child — the Hermes "narrow the child's tools" lesson, violated. |

## Milestones

### M0: Contracts + ADR + Trust-Tier Definition  *(serial barrier)*
Land ADR 0068 (Pi-mode coding surface + "local coding / sandbox level 0" trust
tier) as the authority contract; confirm the v0.55 upstreams (tui channel + split
tool-result) exist or block. Define the coding-action contract shape (the four
actions' signals, results, and Security Central policy hooks) and the level-0 tier
vocabulary as an ADR 0009 extension. **No effectful code lands here** — this is the
barrier the parallel lanes rejoin against.
- Focused tests: ADR-fixture trust-tier resolution (level 0 resolves only for
  single trusted operator + main session + `tui`; rejects channel-originated /
  generated-code / non-`tui` callers).
- Gate: ADR 0068 accepted; upstream check green; tier contract reviewed.
- Rejoin point for M1 and M2.

### M1: Split-Result Consumption + Streamed-Diff Terminal Render  *(parallel lane A)*
Consume the v0.55 split tool-result payload (ADR 0029) in the `tui` surface:
render the UI/diff payload via the unified renderer (ADR 0030) / `owl`, while the
model receives only the clean text payload. Stream `write`/`edit` diffs to the
terminal as they're produced. Touches rendering only — no authority.
- Focused tests: split payload renders diff to terminal without leaking the UI
  payload into model context; streamed-diff chunks render in order; turn-complete
  fallback when streaming unavailable.
- Lane: parallel with M2 (renderer surface, no runner coupling).

### M2: Four Boundary Actions + Runner Routing  *(parallel lane B)*
Register `read`, `write`, `edit`, `bash` as Jido actions via `Actions.Registry`,
each executed through `Actions.Runner.run/3` and gated by Security Central. `bash`
is policy-bounded host execution behind the registered action (not an OS boundary;
not a sub-agent spawn). Results emit the split payload (model text + render diff)
for M1.
- Focused tests: each action resolves through `Actions.Registry` and executes via
  `Runner.run/3`; `bash` is policy-bounded and refuses outside policy; `write`/
  `edit` produce a split result; no action bypasses Security Central; no
  bash-spawned sub-agent path exists.
- Lane: parallel with M1 (action boundary, no render coupling).

### M3: Local-Coding Trust Tier  *(serial — gates M2's cheapness)*
Implement the "local coding / sandbox level 0" tier from M0: Security Central
resolves level 0 **only** for a single trusted operator in the main session on the
`tui` channel, making the gate *present but cheap* (no per-action confirmation
prompt for the trusted local case) while keeping the full gate for every other
caller. Channel-originated inbound, scheduled jobs, and generated-code sessions
**cannot** reach level 0.
- Focused tests: trusted-operator/main/`tui` → level 0, cheap gate, full trace;
  channel-originated → full gate, no level-0 escalation; generated-code session →
  full gate; level 0 never the default; tier is auditable in the trace.
- Serial after M2 (the tier modulates how M2's gate runs); rejoins before M4.

### M4: Minimal Prompt + Context Discipline + Model-Switch Affordance  *(parallel lanes)*
- Lane A: the **sub-1000-token** coding system prompt + full-file
  context-gathering discipline (reuse `AGENTS.md` hierarchy + v0.03 skill loader);
  encode the **minimalism budget** as a standing coding policy (small prompt, four
  default tools, lazy disclosure of the rest).
- Lane B: the **mid-session model switch** operator affordance over req_llm
  `Context` handoff (ADR 0047/0051), surfaced in the `tui` session.
- Focused tests: assembled coding prompt is under the token budget; context
  gathering feeds full files; default turn exposes only the four actions (rest
  lazy); model switch preserves session `Context` and changes no authority path.
- Lanes A/B parallel; rejoin before M5.

### M5: Evals, Hardening, Closeout  *(serial barrier)*
`:v057` security/coding eval lane + `release.v057` deterministic gate; opt-in
interactive coding-session smoke where the harness allows; ADR 0068 accepted and
cross-linked; docs (request-flow, roadmap §v0.57, vision, agent-context-map,
operator Pi-mode coding guide); the `pi-integration-rethink.md` sequencing items
3–5 reconciled; version/CHANGELOG closeout + full release gate.

## Non-Goals
*(from the Pi analysis §8 "what not to do")*
- **No YOLO-by-default.** Level 0 is a named, gated, non-default tier — never the
  default, never for channel-originated or generated-code sessions.
- **No weakening of the action boundary / Security Central / confirmations.** The
  gates are made *cheap at level 0*, not *absent*; every effect still routes
  through `Actions.Runner.run/3`.
- **No model-decides-it's-done** for effectful or generated-code work;
  deterministic acceptance rules.
- **Keep MCP-first with lazy disclosure** (ADR 0038/0039); do not adopt Pi's
  no-MCP stance. The minimalism is in the *default* surface, not a ban.
- **No sub-agent-via-bash-spawn**; use restricted-registry delegate agents.
- **No sibling runtime**; Pi-mode is a channel/app on the one spine — same
  authority boundary, same trace, same memory.
- No new TUI renderer choice; reuse the v0.55 `owl` terminal surface (Pi's
  scrollback-native model stays reserved until evidence).

## Definition Of Done
v0.57 is done when:
- a Pi-mode coding surface runs as a channel/app on the `tui` channel, exposing
  exactly four boundary actions (`read`/`write`/`edit`/`bash`), each resolving
  through `Actions.Registry` and executing via `Actions.Runner.run/3` under
  Security Central;
- a named "local coding / sandbox level 0" trust tier (ADR 0009 extension, ADR
  0068) makes the gate present-but-cheap **only** for a single trusted operator in
  the main session on `tui`, and provably cannot be reached by channel-originated,
  scheduled, or generated-code sessions, and is never the default;
- streamed diffs render in the terminal via the v0.55 split tool-result payload
  (ADR 0029/0030) with no UI payload leaking into model context;
- the coding system prompt is under the 1000-token budget, context gathering feeds
  full files, the default turn exposes only the four actions, and a mid-session
  model switch preserves the req_llm `Context` without touching the authority path;
- acceptance for effectful/generated-code work is deterministic (evidence +
  confirmation), with no model-confidence completion and no bash-spawned sub-agent;
- `:v057` evals + `release.v057` + the full release gate pass;
- ADR 0068, the minimalism-budget coding policy, roadmap, vision, and
  agent-context-map agree, and `pi-integration-rethink.md` items 3–5 are delivered.
