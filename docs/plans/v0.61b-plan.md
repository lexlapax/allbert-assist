## Purpose

v0.61 implemented the v0.60 IA in the v0.60b-chosen Direction C language and
shipped correct-as-built. The operator's manual validation then did what manual
validation is for: it judged the shipped composition **as a product**, and found
the *structure of the chrome* wanting while confirming the content is right. The
eight feedback items decompose into two natures:

1. **A coherent shell-composition critique (items 1, 3, 6, 7).** Two navigation
   columns where one would do; persistent top bars spending vertical space on
   context that belongs in the navigation panel; tool panels floating over — and
   occluding — the chat they should sit beside; and no way to collapse any of it.
   Read together these are one decision, not four fixes: **navigation and context
   consolidate into a single collapsible sidebar; content (chat, canvas, docked
   tools) gets everything else.** ADR 0080 records this; the Research Grounding
   section below grounds the desktop-navigation and token-role choices while
   treating contemporary AI/workspace products as pattern observations.
2. **Contained refinements (items 2, 4, 5, 8).** A chat type-scale bug (the
   timestamp has *no* CSS rule and inherits 16px, out-weighing the 13px body), a
   mislabeled navigating status chip, a missing thread-rename affordance the
   schema already supports (`title` field exists; `:conversation_write`
   permission exists), and a dark palette one notch too loud. Each is within an
   existing contract (ADR 0074 type scale, NN/g menu-labeling guidance via ADR 0080 §5,
   ADR 0073 registered-action path, ADR 0079 Direction C) — plan-level work.

v0.61b is the point release that lands both natures before v0.62 shifts attention
to packaging — presentation feedback is cheapest to apply while the presentation
release's context is fresh, and the v1.0 presentation-contract freeze makes
pre-freeze composition fixes time-sensitive.

One precedent note, stated rather than implied: v0.61b exceeds the prior point
releases in kind — v0.60b was design-only and v0.55b read-only, while v0.61b
ships production shell code and one operator **write** affordance (thread
rename). It still honors every formal point-release invariant (no renumbering
of v0.62–v0.64; no new authority, permission, capability class, route, or
Settings key); the rename is the single operator write affordance and it rides an
existing permission through the existing action spine.

## Cross-Cutting Invariants (apply to every milestone)
- **No new authority.** No new capability class, permission, confirmation-floor
  change, egress grant, or Settings Central key. Thread rename uses the existing
  `:conversation_write` permission (already `: allowed`-defaulted in policy and
  used by `resume_thread_on_channel`); Security Central is unchanged.
- **The catalog stays the rendering boundary.** All shell recomposition is
  catalog/shell/component-variant work through the established renderer; no
  model- or data-generated markup; no new rendering mechanism; no route added or
  removed (the additive-only carve-out is untouched; `?destination=workspace:*`
  deep links keep working). On the carve-out's **component** axis: the retired
  shell node atoms are single-emitter workspace atoms the v1.0 freeze exempts
  from name-freezing; they stay registered-but-unused in `Surface.Catalog`
  (see M5), so nothing frozen is renamed or removed.
- **All writes go through registered actions** (ADR 0073): server-derived,
  ownership-scoped identity via `ContextBuilder.live_view_context`, params never
  carry identity, PermissionGate enforces the permission. The JobsLive
  registered-action migration is the template.
- **No internal rename.** `Conversations.Thread` modules/schema/atoms/topics/
  settings keys and `Session.Scratchpad` are untouched (v0.58 invariant, ADR 0074
  guardrail). The thread title is a persisted *field value*, surfaced in UI
  strings.
- **Direction C holds; a11y axes hold.** Every theme × high-contrast ×
  reduced-motion × OS-preference cell stays readable (status tokens ≥ AA); focus
  management, `aria-expanded`, focus return, and keyboard reachability are
  gate-checked. The `:v061` proof suite is **reconciled, not held verbatim**:
  where a milestone deliberately changes what a proof asserts literally
  (`direction_c_tokens_test.exs:71-76` and `dark_mode_test.exs:72-73` pin the
  exact dark hex values M3 changes; `operator_shell_nav_test.exs` pins the nav
  groups/pills M5 restructures; `accessibility_conformance_test.exs:106-120`
  hardcodes the `#14121f` contrast denominator), the owning milestone updates
  the proof as a deliberate, reviewed edit — never a weakening — and the suite
  then re-runs green. v0.61b refines, it does not regress.
- **Client-side layout preferences, not settings.** Sidebar collapsed state and
  docked-pane width persist client-side (localStorage via a JS hook, the
  ThemeSync pattern); **no new Settings Central key**.
- **One spine.** Nothing reaches `Settings.Store`, `Confirmations`, or business
  logic directly; reads come through the renderer context / existing actions.

## Research Grounding

This release uses research to lock the direction, not to expand scope. The cited
sources are limited to decisions that need external support; product examples are
pattern observations only.

- **Visible desktop navigation:** NN/g's hidden-navigation study says desktop
  navigation should remain visible or partially visible; the shell therefore
  defaults to an expanded sidebar, collapses first to an icon rail, and reserves
  full hide for opt-in focus mode:
  <https://www.nngroup.com/articles/hamburger-menus/>.
- **Left-side app navigation and disclosure rules:** NN/g's menu checklist places
  application primary navigation on the left side, calls for current-location
  cues, clear labels, caret/arrow submenu signifiers, click-activated submenus,
  and no deep cascading menus. That maps to one left product sidebar, contextual
  Workspace disclosures, clear labels, max two visible levels, and active
  highlighting: <https://www.nngroup.com/articles/menu-design/>.
- **Token-role discipline:** Radix Colors' scale guidance separates backgrounds,
  component backgrounds, borders, solid accents, and text roles. v0.61b uses that
  framing for the dark-mode subtlety pass: keep Direction C, reduce over-loud
  accent/surface choices, and preserve text/contrast roles:
  <https://www.radix-ui.com/colors/docs/palette-composition/understanding-the-scale>.
- **Product-pattern observations:** ChatGPT Canvas, Claude Artifacts, Slack,
  Linear, VS Code, Notion, and GitLab are observed as contemporary workspace
  examples of docked panes, single sidebars, rails, or slim per-view headers.
  They are not treated as normative sources and do not override the Allbert IA or
  Security Central invariants.

## Non-Goals

- **Not a re-choice of the visual language.** Direction C holds; the dark pass
  tunes values inside it (ADR 0079 untouched beyond a pointer if needed).
- **No IA surface-inventory or journey change.** No screen or route is added or
  removed; no new authority or capability class. Navigation presentation changes
  only, and thread rename remains the one contained operator write affordance on
  the existing `:conversation_write` spine.
- **No new authority, permission, Settings key, or confirmation change.**
- **Not the v0.62 packaging release.** v0.62 scope (binary, CLI dispatcher,
  daemon, vault) is untouched; only its UX carryover section moves here.
- **No mobile-first redesign.** Mobile variants degrade per spec (today's
  mobile patterns adapt to the consolidated shell) but mobile is not this
  release's design target.
- **No guided onboarding or persona work** (v0.63) and **no product-RC
  validation scope** (v0.64).
