# v0.4 — AgentSkills adoption

Status: Proposed

## Context

v0.1 and v0.2 shipped skills as single markdown files. That served the early releases well but limits three capabilities we now need:

1. Skills with supporting scripts have no clean home for them.
2. Reference material either inflates every load (inline) or disappears from the skill's portability (out-of-band).
3. Allbert cannot easily consume skills authored for the wider agent-assistant ecosystem.

The AgentSkills open standard at [agentskills.io](https://agentskills.io/home) already defines the richer shape we need: `skill-name/SKILL.md` plus optional `scripts/`, `references/`, `assets/` directories, with a defined frontmatter schema and a `skills-ref` validator. Progressive disclosure across discovery, activation, and resource tiers is a first-class idea in the spec.

v0.3 introduced first-class agents (ADR 0029) and made skills a plausible place to contribute agents (ADR 0031). v0.4 builds on that by making the skill format itself richer and by codifying the trust model around shared skills.

## Goal

Adopt the AgentSkills folder format as Allbert's canonical skill shape, preserve a one-release read path for v0.2-style single-file skills, add an explicit install-with-preview trust model, and route skill scripts through the existing exec policy.

## In scope

- AgentSkills folder format as canonical (ADR 0032).
- Skill frontmatter validation against the AgentSkills schema plus Allbert extensions (`agents`, `bundles`, `intents`).
- Progressive disclosure wired into prompt construction (ADR 0036).
- Install sources: local path and git URL only (ADR 0035).
- Explicit preview + confirm flow for every install (ADR 0033).
- Skill scripts executed through `process_exec` seam under `exec_policy` (ADR 0034); Bash + Python default, Node opt-in.
- Deprecation warnings and migration helper for single-file skills; removal scheduled for v0.5 (ADR 0037).
- Bundled first-party skills re-shipped in folder format.
- Lifecycle commands: `allbert-cli skills list|show|install|update|remove|validate|migrate`.

## Out of scope

- Remote curated skill registry (deferred; ADR 0035).
- Cross-runtime AgentSkills ecosystem tooling beyond validation compatibility.
- Sandboxed script execution beyond the existing exec policy surface.
- Skill signing / provenance attestation (possible v0.5+).

## Skill format

Canonical layout:

```
<skill-name>/
├── SKILL.md            # required: frontmatter + body
├── scripts/            # optional
│   ├── helper.sh
│   └── analyse.py
├── references/         # optional
│   └── notes.md
├── assets/             # optional
│   └── diagram.png
└── agents/             # optional (Allbert extension for ADR 0031)
    └── researcher.md
```

Frontmatter:

```yaml
---
name: research-assistant                       # required, kebab-case, ≤64 chars, matches directory
description: Helps with literature review ...  # required, ≤1024 chars
license: MIT                                   # optional
compatibility:                                 # optional
  agentskills: ">=0.1"
  allbert: ">=0.4"
metadata:                                      # optional, free-form
  tags: [research, reading]
allowed-tools: [read_memory, web_fetch]        # optional, intersects session policy
intents: [task, memory_query]                  # Allbert extension (from v0.3)
agents:                                        # Allbert extension (ADR 0031)
  - name: researcher
    path: agents/researcher.md
scripts:                                       # optional
  - name: summarise
    path: scripts/summarise.py
    interpreter: python
---
```

Validation uses an Allbert-internal validator that is compatible with the upstream `skills-ref` schema plus the Allbert extensions above. Skills that pass upstream validation pass Allbert validation provided extensions are syntactically correct.

## Progressive disclosure

The prompt builder surfaces tiers explicitly (ADR 0036):

- **Tier 1 — discovery**: `name` + `description` indexed into a lightweight catalog at skill load time. Used by the intent router (ADR 0030) and skill-pick heuristics.
- **Tier 2 — activation**: when a skill is activated for a turn, its `SKILL.md` body is loaded into the active-skills prompt stage.
- **Tier 3 — resource**: references and assets resolved on demand via explicit tool calls (`read_reference(skill, path)`); scripts invoked through the exec seam (ADR 0034).

Hooks emit tier-transition events (`SkillTier1Surfaced`, `SkillTier2Activated`, `SkillTier3Referenced`).

## Install + distribution

v0.4 supports two sources (ADR 0035):

1. **Local filesystem path** — existing directory that matches the folder format.
2. **Git URL** — `https://` or `ssh://`, optionally pinned to ref/commit. Fetch runs through `git` via the exec seam.

Install flow:

1. Fetch into `~/.allbert/skills/incoming/<skill-name>/` (quarantine).
2. Validate frontmatter + structure.
3. Preview surfaces: skill name, description, first N lines of `SKILL.md`, list of scripts with SHA-256 hashes, declared `allowed-tools`, declared `agents`, source identity, resolved commit / path SHA.
4. User approves or rejects. Rejection deletes the quarantine copy.
5. On approval, move to `~/.allbert/skills/installed/<skill-name>/` and register in the skill store.

Install metadata records source identity, commit hash (for git), tree SHA-256 (for both), and approval timestamp. Approvals are remembered per `(source identity, tree SHA-256)`; any SHA change forces re-preview.

## Trust model

- Every install goes through preview + confirm (ADR 0033) — no auto-install.
- Scripts execute under `exec_policy` (ADR 0034); Bash and Python interpreters default-allowed, Node requires explicit opt-in.
- Each script invocation fires `BeforeExec`/`AfterExec` hooks tagged with the skill name + script path.
- `allowed-tools` in skill frontmatter intersects the session policy — skills cannot broaden access.
- Install preview surfaces interpreter requirements so users see upfront whether they will need to widen `exec_policy`.
- **End users do not author scripts.** Scripts are authored by skill authors and ship inside the skill folder; end users encounter them at install time through the preview, not through a "write a script" step. This aligns with ADR 0038's natural-interface principle.
- **Preview UX is plain-English rendering**, not a raw code diff. Each script is summarised by name, interpreter, and SHA-256 hash. Users who want to audit the script body can still inspect the quarantine directory directly.

## Script execution policy

- Declaration: `scripts:` array in frontmatter, each entry with `name`, `path`, `interpreter`.
- Invocation: kernel resolves the interpreter against `exec_policy.allowed_interpreters`, rejecting with an actionable message if missing.
- Default allowlist: `["bash", "python"]`.
- Node and other interpreters require explicit opt-in via `config.exec_policy.allowed_interpreters`.
- Script env and cwd are set by the kernel, not by prompt content — no env injection from the model.

## Migration bridge

v0.4 reads both formats:

- Single-file skills load with a deprecation warning in the console, trace log, and skill status output.
- `allbert-cli skills migrate [--apply]` converts single-file skills to folder format. Preview by default; `--apply` writes the conversion.
- Bundled skills ship in folder format from v0.4 onward.
- v0.5 removes the single-file read path entirely (ADR 0037). The migration helper remains available after v0.5 as a one-shot conversion tool.

## Lifecycle commands

`allbert-cli skills` subcommands in v0.4:

- `list` — show installed skills (name, version/ref, enabled state, last-used).
- `show <name>` — print frontmatter, tier 1 description, tier 2 body summary, scripts, references, current install metadata.
- `install <source>` — local path or git URL. Triggers quarantine + preview + confirm.
- `update <name>` — re-fetch the original source, diff, re-preview, confirm.
- `remove <name>` — deregister and delete from installed skills directory.
- `validate <path>` — schema-validate a skill directory without installing.
- `migrate [<path>] [--apply]` — convert single-file skill(s) to folder format.
- `init <name>` — interactive scaffold that creates a valid folder-format skill from prompts. Asks for `name` (validates kebab-case) and `description`, offers to create empty `scripts/` and `references/` subdirectories, and writes a stub `SKILL.md` with passing frontmatter. Complements, rather than replaces, the v0.7 natural-language skill-authoring skill — `init` is for technical users who want to start a skill at the CLI without hand-editing YAML.

## Bundled skills reshipment

All bundled first-party skills (introduced across v0.1 and v0.2) are re-shipped in folder format in v0.4. The migration helper is used internally for conversion; the result is committed as the canonical form. No bundled skill remains in single-file form after v0.4.

## Config schema deltas

```toml
[skills]
installed_dir = "~/.allbert/skills/installed"
incoming_dir = "~/.allbert/skills/incoming"

[exec_policy]
allowed_interpreters = ["bash", "python"]   # add "node" to opt in

[install]
remember_approvals = true
```

## Milestones

### M1 — Folder loader and validator

- Skill loader walks `<root>/*/SKILL.md` layout.
- Frontmatter validator against AgentSkills schema + Allbert extensions.
- `allbert-cli skills validate <path>` command.
- Tests: schema coverage, malformed skill rejection, Allbert extension handling.

### M2 — Progressive disclosure + prompt builder integration

- Tier 1 catalog built at skill load time.
- Tier 2 activation path uses folder-resident `SKILL.md` body.
- Tier 3 resource resolution via `read_reference` tool.
- Tier-transition hook events.
- Tests: tier-aware loading, cache correctness, hook ordering.

### M3 — Install flow (local path)

- Quarantine directory, plain-English preview renderer, confirm prompter.
- Install metadata persisted with SHA-256 tree hash.
- Remembered approvals keyed by `(source, tree_sha)`.
- `allbert-cli skills install <path>` + `remove <name>` + `init <name>`.
- Tests: approve/reject paths, re-preview on SHA change, metadata correctness, `init` produces a skill that validates without modification.

### M4 — Install flow (git URL) + script execution

- Git fetch via exec seam; default branch resolution and ref pinning.
- Script declaration in frontmatter → kernel exec seam integration.
- Interpreter allowlist enforcement + actionable error when interpreter missing.
- `allbert-cli skills update <name>`.
- Tests: git fetch, pinning, update flow, script interpreter allowlist.

### M5 — Migration bridge

- Single-file skill deprecation warning surface.
- `allbert-cli skills migrate` helper (preview + apply).
- Convert all bundled single-file skills to folder format.
- Document v0.5 removal schedule (ADR 0037).
- Tests: migration correctness, deprecation warning fires once per session per skill.

### M6 — Closeout

- Operator docs for `allbert-cli skills` commands and preview UX.
- Re-ship bundled skills in folder format.
- Update `docs/vision.md` skill direction references.
- Mark ADRs 0032, 0033, 0034, 0035, 0036, 0037 Accepted.

## Testing and verification

- Round-trip tests: an AgentSkills-compatible sample skill loads, validates, activates, and runs scripts.
- Trust tests: install preview cannot be bypassed; rejection leaves no trace in `installed/`.
- Script tests: Bash and Python scripts run; Node script is refused until opted in.
- Migration tests: every bundled single-file skill migrates cleanly to folder format.
- Progressive disclosure tests: tier 1 description is available without loading tier 2; tier 3 loads only on explicit reference.
- Compatibility tests: validator passes a representative corpus of public AgentSkills-format skills.
