# v0.6 — Foundation hardening

Status: Proposed

## Context

v0.5 closed on 2026-04-20 with curated memory, a tantivy retrieval path, and the shipped `memory-curator` skill. A retrospective across v0.1–v0.5 the same day surfaced five gaps that do not change what Allbert *can* do, but change how reliably the v0.5 experience lands:

- **Staging is silent.** The 5-per-turn cap and 90-day TTL bound the blast radius, but users do not know what was captured unless they run `memory staged list`. Entries that do not surface at the moment of creation rarely get promoted.
- **Conversation dies on daemon restart.** Sessions survive client reattach, but a daemon restart wipes everything. For a personal assistant whose selling point is "remembers you," this is too fragile.
- **The cost cap is decorative.** `limits.daily_usd_cap` is read and logged, not enforced. A looping job or a runaway sub-agent has no hard gate.
- **The "gets smarter over time" loop ships disabled.** `memory-compile`, `system-health-check`, and `trace-triage` bundle as disabled-by-default templates. Most profiles will never turn them on.
- **Markdown drift is undetected.** Users can edit `~/.allbert/memory/notes/` directly — the whole point of markdown ground truth — but `manifest.json` and the tantivy index drift silently until the next explicit rebuild.

v0.6 is a **protect-what-we-shipped** release. No new product capability, no new channel, no new memory tier. Just hardening the v0.5 foundation before v0.7 expands the channel surface onto it.

## Goal

Close the five gaps that prevent v0.5 from delivering its stated experience reliably, cover the daemon + kernel + memory path with a first end-to-end regression test, and finish a doc-alignment pass so `docs/vision.md` and `README.md` match the shipped surface.

## Milestones

### M1 — Turn-end staged-memory notice

After every turn that stages at least one memory entry, the kernel renders a short togglable notice on the assistant's final message summarising what was staged, with a `memory staged show <id>` pointer. Kernel-rendered because staging is kernel-native.

- Config: `memory.turn_end_notice = true|false` (default true on fresh profiles, unchanged on upgrade).
- Budget: combined id + summary truncated to 120 chars per entry.
- Channel-adaptive: plain-text for channels without rich output; omitted for batch channels (future v0.7 concern).
- Covered by ADR 0050.

### M2 — Session durability across daemon restart

Each active session persists an append-only markdown journal at `~/.allbert/sessions/<session-id>/turns.md` plus `meta.json`. On daemon restart, `allbert-cli daemon resume [--session <id>]` rehydrates the session.

- Tool outputs in the journal are truncated to `memory.max_journal_tool_output_bytes` (default 4 KiB); full outputs remain in the trace log.
- Sessions older than `daemon.session_max_age_days` (default 30) move to `sessions/.archive/`.
- Sessions interrupted mid-tool resume to the last completed turn boundary; incomplete invocations are discarded with an audit entry.
- Journals are **working state, not durable memory**. Staging/promotion remains the only path into `notes/`.
- Covered by ADR 0049.

### M3 — Daily cost cap enforcement

`limits.daily_usd_cap` becomes a hard gate at turn start. Over-cap turns refuse with an operator-facing message and a `/cost --override <reason>` escape hatch for a single turn.

- CostHook reads today's aggregate from `costs.jsonl` keyed by UTC day (cached 60s).
- Enforcement is kernel-level; applies uniformly to REPL, channel-originated, and job-originated turns.
- Jobs refuse silently, log `cap-reached` as the job's last outcome, and surface it in `jobs status`.
- Cap of `null` (default until wizard sets it) disables enforcement. The wizard offers a sensible starting cap.
- Covered by ADR 0051.

### M4 — Memory drift detection

`memory status` surfaces a checksum comparison between `manifest.json` and the actual `notes/` + `staging/` trees. A new bundled job template `memory-drift-check` runs the comparison daily and stages a `meta`-kind entry naming any out-of-manifest file.

- No filesystem watcher: cross-platform simplicity, aligned with markdown-ground-truth.
- Exit code on `memory status` is non-zero when drift is detected, for scripting.
- The drift-check job is enabled by default per M5.

### M5 — Sensible defaults for bundled jobs

`memory-compile` and `memory-drift-check` flip to enabled-by-default on fresh setup. `trace-triage` and `system-health-check` remain disabled. Upgrade profiles are unchanged: existing disabled state is preserved.

- Wizard text updated to reflect new defaults and explain the opt-out.
- `daily-brief` and `weekly-review` stay disabled: they require operator-confirmed scheduling preferences.

### M6 — End-to-end regression test

Add an integration test under `crates/allbert-daemon/tests/` that covers the daemon + kernel + memory path together. Steps:

1. Boot a fresh profile in a test home.
2. REPL turn writes a staged memory entry.
3. Promote the entry.
4. Restart the daemon.
5. REPL turn retrieves the promoted entry.
6. Assert retrieval succeeded and session state rehydrated from M2 journal.

This is the first test exercising daemon restart + memory + session durability together. It locks the v0.6 behaviour in before v0.7 reshapes the channel surface.

### M7 — Doc alignment pass

- **`docs/vision.md`**: add "Natural interface" as a first-class principle in the Principles section (ADR 0038 codified the decision; vision.md still omits it from the headline list). Add one line on multimodal I/O under Channel Direction as forward reference to v0.7.
- **`README.md`**: update "What v0.5 includes" and "Current limitations" to reflect M1–M5. Document `/cost --override`, `daemon resume`, and `memory-drift-check`.
- **`docs/onboarding-and-operations.md`**: operator walkthrough for daemon resume, turn-end notice toggle, cost-cap override, drift check.

## Expected ADRs

- **ADR 0049** — Session durability is a markdown journal.
- **ADR 0050** — Turn-end staged-memory notice is kernel-rendered, togglable.
- **ADR 0051** — Daily cost cap is a hard gate at turn boundary.
- **ADR 0052** — Memory drift detection uses on-demand checksum (not fs watcher). *May remain as an in-plan decision rather than a standalone ADR; escalate only if the detection strategy is contested during implementation.*

## Dependencies

- v0.5 curated memory — M1 and M4 extend its surface; M3 wraps its staging pipeline in cost accountability.
- v0.2 daemon lifecycle — M2 extends the restart story; M6 tests it end-to-end.
- v0.2 job manager — M4 ships a new bundled job; M5 changes the default-enabled set.

## Out of scope for v0.6

- New channels (v0.7).
- New memory tiers or retrieval strategies (v0.5 shape stays).
- Multi-device memory sync (unscheduled; flag for post-v0.7 ADR).
- Sub-agent depth changes (v0.7).
- Tool-registry refactor to absorb memory/skill kernel-native ops (v0.7).
- A filesystem-event watcher for memory drift (M4 is checksum-based on purpose).

## Remaining planning questions

- Should session journals redact tool outputs by default or preserve them for replay fidelity? Default 4 KiB truncation is a compromise; operators who want full fidelity can raise the limit.
- When a session exceeds retention, should promotion-relevant context be auto-staged before archive, or silently dropped? Current default: silently dropped; staging would require a "salience scorer" that v0.6 explicitly does not introduce.
- Is `/cost --override` REPL-only, or should it extend to channel-originated refusals? v0.6 ships REPL-only; channels negotiate their own override UX in v0.7.
