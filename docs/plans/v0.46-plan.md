# Allbert v0.46 Voice Modality

Request flow: `docs/plans/v0.46-request-flow.md`

ADR: `docs/adr/0042-audio-image-and-media-resource-classes.md`

Status: planned.

Source: `docs/archives/version-1.0-planning-03.md`.

## Purpose

Add experimental voice input and output as media resources and registered
actions that compose with existing channels.

## Expected Inputs

- v0.39 provider doctor path.
- v0.43 channel pack for Discord voice context.
- Existing Resource Access, Settings Central secrets, and trace redaction.
- Proposed ADR 0042 accepted before implementation closeout.

## Operator-Visible Win

Operators can talk to Allbert from CLI or workspace, ingest Telegram voice
notes, and use Discord voice read support where configured.

## Planned Interfaces

- `mix allbert.ask --voice`
- Workspace microphone affordance.
- STT provider profiles.
- TTS provider profiles.
- Audio resources such as `mic://capture/<id>`.

## Security Posture

Microphone capture requires explicit per-session confirmation. Captured audio is
bounded, redacted from traces by default, and retained only by explicit policy.
TTS is an effectful registered action with provider/cost visibility.

## Non-Goals

- No always-on wake word in 1.0.
- No hidden cloud upload of audio.
- No durable audio retention by default.
- No channel permission bypass through voice.

## Milestones

- M1: ADR 0042 audio portion and provider profile checks.
- M2: CLI/workspace voice input.
- M3: TTS output action and provider options.
- M4: Telegram voice-note ingestion and Discord voice support.
- M5: Security evals, docs, and experimental closeout.

## Documentation Deliverables

- Operator voice setup guide.
- Security-hardening notes for audio retention and redaction.
- Provider doctor docs for STT/TTS profiles.

## Definition Of Done

v0.46 is done when voice input/output works through configured providers,
captured audio obeys retention/redaction policy, and channel/runtime authority
remains unchanged by modality.
