## Purpose

Ship a second native consumer of the v0.24 delegate-agent substrate — a
plugin-contributed **research/summarize specialist** — so the
`AllbertAssist.Objectives.AgentRegistry` + `:delegate_agent` contract is
proven against two distinct domains before the v1.0 Tier 1 freeze locks
Objective Runtime.

Through v0.45, StockSage native financial specialist agents (ADR 0022)
are the **only** registered `AgentRegistry` consumer
(`AgentRegistry.register/4` is called in exactly one non-test location:
`plugins/stocksage/lib/stocksage/agents/runtime.ex`). Freezing the
delegation contract on one-consumer evidence risks locking a shape driven
by one domain's needs. v0.46 closes that risk four ways:

1. **Proves the contract.** A research specialist whose commands
   orchestrate the already-shipped v0.43 browser research actions gives
   ADR 0021 its second delegate-agent consumer, so the contract can be
   reviewed against finance + research before the freeze.
2. **Documents the extension point.** A developer guide
   (`docs/developer/delegate-agents.md`) makes the plugin-author path for
   registering a delegate agent discoverable instead of implicit.
3. **Exercises v0.44 inline subagent delegation against a non-StockSage
   agent.** Plan/Build's inline child-event rendering and the
   `kind: delegate_agent` workflow step get a second, operator-relevant
   demo: "plan: research X, delegate the deep-read to the research
   specialist, summarize."
4. **Hardens delegate command dispatch.** v0.44 workflow YAML already
   requires a `command` string on `kind: delegate_agent` steps and the
   v0.44 expander carries it at `action_params.command`, but two gaps
   remain: (a) the `AllbertAssist.Objectives.Commands.ExecuteStep`
   delegate clause hard-codes `command: "execute"`, so the step's real
   command never reaches the action through the objective or Plan/Build path; and
   (b) the `delegate_agent` action accepts `execute` plus any pre-atomized
   atom **unchecked**, rejecting only unknown command *strings* (via the
   already-shipped `:invalid_delegate_command`). v0.46 closes both: it
   threads the step command into the action's `command` arg, then validates
   it against the resolved agent's `allowed_commands` metadata at the
   existing `delegate_agent` action boundary.

v0.46 adds **no new authority**: the research specialist orchestrates
existing registered actions through `Actions.Runner.run/3`. It introduces
no new permission class, operation class, URI scheme, or registered
action surface. Every effectful step keeps its own Security Central floor
and confirmation.

## M1 Locked Decisions

The first milestone preserves these six decisions. Later milestones may
revisit them only with an explicit re-decision recorded here and in the
request-flow document.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Delivery shape | **Plugin-contributed delegate agent** at `./plugins/allbert.research/`, NOT a core module | A plugin proves the *third-party* extension path and keeps core free of a research dependency, mirroring how StockSage proves the app/agent contract. A core module would prove less and grow the kernel. |
| 2 | Authority surface | **Zero new authority.** The agent orchestrates shipped v0.43 browser actions plus pinned model-backed summarization or deterministic extractive fallback through `Actions.Runner.run/3`; no new registered action, permission class, operation class, or URI scheme. Only a small `research.*` settings fragment (enable toggle + source cap). | Adding research-specific actions/permissions would duplicate browser authority and create a second place to audit. Reusing the browser actions means every navigation still confirms and every per-domain grant (v0.43) still applies — delegation never widens authority. |
| 3 | Research scope | **Read-only research**: navigate + bounded extract + advisory summary; a bounded source count; inherits v0.43 deny-by-default for form-fill, download, and authenticated browsing | Broader scope (crawling, authenticated research, multi-tab) widens the v0.43 trust posture, which is parked in `future-features.md` §"Authenticated Browser Operation And Persistent Profiles" and §"Broad Office, Archive, And Unknown-Binary Extraction". v0.46 stays inside the shipped browser floor. |
| 4 | Contract goal | **Second consumer to harden, not abstract.** v0.46 satisfies ADR 0021 A21 (two delegate-agent consumers before freeze) and documents the extension point. It does **not** extract a shared delegate-agent behaviour abstraction, refactor StockSage, or add operator no-code agent authoring. | Extracting an abstraction now would repeat the mistake A20/A21 warn against — generalizing from too few consumers. Operator no-code authoring is parked (Phase 3) behind the v0.36/v0.37/v0.38 supervised path. |
| 5 | Delegate command contract | **Allowlisted commands at the existing boundary.** `delegate_agent` resolves the agent via `AgentRegistry.lookup`, then accepts `execute` plus that agent's `allowed_commands` (such as `research` and `summarize_url`), normalizing string-or-atom commands without dynamic atom creation and rejecting unknown commands on the existing `:invalid_delegate_command` path. | Today the action accepts `execute` plus any pre-atomized atom **unchecked** and rejects only unknown *strings*; that lets an atomized command bypass the metadata allowlist. Metadata-checked normalization closes that gap without turning delegation into an unsafe private command surface. |
| 6 | Command threading | **The step command reaches the action `command` arg.** `Objectives.Commands.ExecuteStep` threads the delegate step's command (carried at `action_params.command` by the v0.44 expander; default `execute`) into the action's `command` arg and passes `action_params.params` as the action `params`, replacing the prior hard-coded `command: "execute"`. No Step-schema migration (ADR 0021 A3 minimal step kind). | If the engine kept hard-coding `execute`, the boundary allowlist (decision 5) would be unreachable through the objective and Plan/Build paths — the only paths M3 and the operator walkthrough use — and a direct-call eval would pass over a boundary the real path never exercises (false confidence). |

Rationales:

- **Delivery shape.** StockSage proves the *app-owned native agent* path;
  the research plugin proves the *focused single-purpose delegate plugin*
  path. Two different plugin shapes exercising the same `AgentRegistry`
  contract is exactly the diversity that hardens it. The plugin also
  becomes the worked example in `docs/developer/delegate-agents.md`.
- **Authority surface.** The research agent is an *orchestrator*, not a
  capability. When it runs `browser_navigate`, Security Central evaluates
  `:browser_navigate` and the operator confirms (or a remembered
  per-domain grant from v0.43 applies) — exactly as a direct browser call
  would. The delegate step adds composition, not permission. This is the
  load-bearing security property and the main thing the hardening release
  proves: **delegation does not bypass or widen action floors.**
- **Research scope.** Bounding to navigate + extract + summary over a
  capped number of sources keeps v0.46 inside the v0.43 floor and avoids
  reopening the parked authenticated-browsing and broad-extraction trust
  posture. Recursive crawling is explicitly out (Non-Goals).
- **Contract goal.** The deliverable is *evidence that the contract works
  for two domains* plus *documentation of the extension point* — not a
  framework. A future release, with more consumers, may extract a shared
  abstraction; v0.46 deliberately stops short.
- **Delegate command contract.** The hardening belongs at the existing
  `delegate_agent` action boundary because that is where Objective Runtime
  already authorizes delegation. The research plugin registers metadata
  such as `allowed_commands: [:research, :summarize_url]`; the action
  resolves the agent, then accepts only `execute` or a
  command present in that metadata, normalizes a string or atom to an
  already-declared value without `String.to_atom/1`, and returns the
  existing `:invalid_delegate_command` error for unknown names. The command
  name selects the delegate's internal advisory flow; it never grants
  browser, memory, or settings authority.
- **Command threading.** Decision 5's boundary only bites if the engine
  actually sends the step's command to the action. Before v0.46
  the `AllbertAssist.Objectives.Commands.ExecuteStep` delegate clause
  hard-coded `command: "execute"` and forwarded the whole `action_params`
  map (so the v0.44 YAML command sat unread at `action_params.command`).
  v0.46 makes the engine read that command and forward
  `action_params.params` as the action params, so the realistic
  objective/Plan-Build path crosses the same validated boundary a direct
  call does. This is a bounded engine change, not a new step column or
  authority surface.

## The Delegate-Agent Contract (Consumed And Hardened)

v0.46 consumes the v0.24 substrate the way StockSage does, with two bounded
changes at existing surfaces — the engine threads the step command, and the
action validates it — and no new authority surface:

```text
AllbertAssist.Objectives.AgentRegistry         (monitored local registry)
  register(id, server, module, metadata)        # research runtime registers here at plugin boot
  dispatch(agent_id, command, params, opts)      # engine reaches the agent through delegate_agent

AllbertAssist.Actions.Objectives.DelegateAgent  (registered action, :objective_write)
  the ONLY boundary the engine crosses to reach a delegate agent;
  routes through Actions.Runner.run/3 + Security Central
```

A `kind: :delegate_agent` objective step carries `delegate_agent_id` and
`action_params` (the v0.44 expander stores the YAML `command` and `params`
inside `action_params`). Before v0.46,
`AllbertAssist.Objectives.Commands.ExecuteStep` hard-coded
`command: "execute"` for that kind; v0.46 changes it to thread
`action_params.command` (default `execute`) into the action's `command`
arg and forward `action_params.params` as the action `params`. The
`delegate_agent` action then looks up the agent in `AgentRegistry`,
validates the command against the agent's `allowed_commands` metadata, and
dispatches the normalized command. The agent returns an advisory report
packet. The research agent is a new *registrant*; the command threading and
metadata-checked normalization are bounded changes to the existing engine
clause and action boundary, not a new authority surface.

## Research Specialist Design

Plugin package `./plugins/allbert.research/`:

- `AllbertResearch.Plugin` — ADR 0017 plugin entry; contributes the
  agent, the settings fragment, and the developer-reference SKILL.
- `AllbertResearch.Supervisor` — plugin child supervisor; starts the
  agent server.
- `AllbertResearch.Runtime` — registers `research.specialist` in
  `AgentRegistry` through `start_link/1` and `register_if_available/2`
  when the agent process is available.
- `AllbertResearch.Agent` — the registered delegate agent. State is
  bounded per-dispatch working context only; it owns no durable
  objective state.
- `AllbertResearch.Settings.Fragment` — `research.*` (see Settings).

Agent command surface (`AllbertResearch.Agent` `dispatch` commands):

| Command | Params | Orchestrates | Returns |
|---|---|---|---|
| `research` | `topic` or `url`, optional `max_sources`, optional `extract_format` | `browser_start_session` → `browser_navigate` → `browser_extract` (per source, bounded) → pinned summarization or extractive fallback → `browser_close_session` | advisory report packet: `{summary, sources: [{url, extract_ref, bytes}], notes}` |
| `summarize_url` | `url`, optional `extract_format` | `browser_start_session` → `browser_navigate` → `browser_extract` → pinned summarization or extractive fallback → `browser_close_session` | advisory report packet for one source |

Each command:

- invokes **only registered actions** through `Actions.Runner.run/3`;
  never the browser driver directly;
- threads the `browser_start_session` `session_id` through subsequent
  steps and always closes the session (or relies on the v0.43
  idle/lifetime timeouts if it crashes);
- pauses at the standard Approval Handoff seam for each
  `browser_navigate` confirmation (or applies a v0.43 remembered
  per-domain grant) — delegation does not pre-approve navigation;
- returns an **advisory** report packet (ADR 0021 §4): the summary and
  extracted evidence are descriptive, never authority, and do not
  auto-promote to memory;
- bounds work by `research.max_sources` and inherits the v0.43 extraction
  byte/page caps.

**Summarization step.** v0.46 does not add a `summarize` registered
action and does not use the v0.11 `summarize_url` operation class as a
summarizer API. M1 must first inspect the shipped runtime and pin a
reusable model-backed summarization entry point if one exists. If none
exists, M1 records that finding and the research agent ships a
deterministic extractive summary (bounded first-N-sentences over bounded
browser extracts) until a model-backed summarizer is wired. Either path
returns advisory metadata only and adds no new authority surface.

## Settings (`research.*`)

Add the `research.*` namespace through
`AllbertResearch.Settings.Fragment.schema/0`; the plugin's
`settings_schema/0` delegates to that named fragment and still uses the
existing `AllbertAssist.Settings.Fragments` plugin composition path.
Declared with `schema_version: 1` per ADR 0046 draft. Research is off by
default.

```yaml
research:
  enabled: false                 # opt-in master switch
  max_sources: 3                 # cap 8; bounds a single research command
  max_extract_bytes_per_source: 524288   # 512 KiB persisted cap; runtime also clamps to browser.extraction.max_bytes
  summary:
    engine: extractive_fallback  # invariant read-only until a reusable model path is pinned
  schema_version: 1              # invariant read-only at v0.46
```

Safe-write keys: `research.enabled`, `research.max_sources`,
`research.max_extract_bytes_per_source`.

Invariants: `research.summary.engine` is read-only at `extractive_fallback`;
`research.schema_version` is read-only at `1`;
`research.max_sources` cannot exceed 8;
`research.max_extract_bytes_per_source` persists values up to 1 MiB; the
M2 runtime effective extraction cap is
`min(research.max_extract_bytes_per_source, browser.extraction.max_bytes)`.

No new permission settings: the research agent uses the existing
`:browser_*` permissions (v0.43) of the actions it orchestrates and the
`:objective_write` permission of the `delegate_agent` action (v0.24).

## Operator-Visible Win

An operator can ask Allbert to research a topic and have a **research
specialist** do the multi-step browser work as a single delegated step,
visible inline, without granting the specialist any authority the
operator did not already grant the browser.

First-time walkthrough:

1. Operator (browser + research enabled): "research the current state of
   WebGPU support across browsers and summarize."
2. Intent routes to a plan with a `kind: delegate_agent` step targeting
   `research.specialist` (or a direct delegated research command).
3. Plan/Build (v0.44) previews the plan; the delegate step shows the
   target agent id, command, and params summary inline.
4. Operator approves the plan-start gate.
5. The research agent runs: it requests a browser session start
   confirmation, then a navigation confirmation per source domain (or
   applies remembered v0.43 grants). These confirmations render **inline
   under the parent delegate step** (v0.44 subagent-delegation
   visibility) — the operator sees exactly which domains the specialist
   is visiting.
6. The agent extracts bounded evidence per source, produces an advisory
   summary, closes the session, and returns a report packet.
7. The RunProgress panel shows the delegated step completed with the
   summary and per-source evidence references. Nothing auto-promotes to
   memory; the operator can manually keep the summary via existing memory
   actions.

A denied navigation fetches nothing for that source; the agent reports
the source as skipped. A denied plan-start gate runs nothing.

## Security Posture

- **Delegation does not widen authority.** The research agent invokes
  registered browser actions through `Actions.Runner.run/3`; each carries
  its own Security Central floor and confirmation. The `:delegate_agent`
  step is `:objective_write` (v0.24) — it authorizes *delegation*, not
  *browsing*. The agent cannot navigate without the operator's
  `:browser_navigate` confirmation or a v0.43 remembered grant.
- **Advisory output, never authority.** Research summaries and extracted
  evidence are descriptive (ADR 0021 §4). They do not alter agent
  behavior and do not auto-promote to memory.
- **Inherits v0.43 browser posture.** Per-domain grants, two-layer
  network policy, extraction byte/page caps, credential-input redaction,
  ephemeral profiles, and trace redaction all apply because the agent
  uses the v0.43 actions unchanged.
- **Bounded fan-out.** `research.max_sources` (default 3, cap 8) bounds a
  single research command. No recursive crawling, no link following
  beyond the operator-named sources.
- **Subagent delegation visibility.** The agent's per-source navigations
  and extractions render inline under the parent delegate step (v0.44),
  so the operator can see what the specialist did and with what scope —
  the "invisible subagents" hardening property, now exercised against a
  second agent.
- **Plugin-owned process.** The agent server lives under
  `AllbertResearch.Supervisor`; `AllbertResearch.Runtime.start_link/1` and
  `register_if_available/2` own registration when the agent process is
  available. Core spawns no research process. A crash takes the agent, not
  core; the v0.43 session idle/lifetime timeouts bound any orphaned browser
  session.
- **No new attack surface.** No new registered action, permission,
  operation class, URI scheme, or memory namespace; the only new settings
  are an enable toggle and bounded caps.

## Non-Goals

- No new permission class, operation class, URI scheme, or registered
  action.
- No shared delegate-agent behaviour abstraction; no StockSage refactor.
- No operator no-code delegate-agent authoring (parked, Phase 3).
- No arbitrary delegate command execution; only `execute` plus commands
  declared in registered-agent metadata are accepted.
- No authenticated research, form fill, download, or login flows
  (inherits v0.43 deny-by-default).
- No recursive crawling, sitemap traversal, or automated link following.
- No multi-tab/window orchestration; one active page per session
  (v0.43).
- No automatic memory promotion from research output.
- No new model dependency in core; summarization uses the existing
  runtime path if present, otherwise deterministic extractive fallback.
- No second advisory-provider role; ADR 0021 A20 stands unchanged.
- No remote/distributed agents; the research agent is a local plugin
  process.

## Definition Of Done

v0.46 is done when:

- a second native `AgentRegistry` consumer (`research.specialist`,
  contributed by `./plugins/allbert.research/`) is registered and
  dispatchable through the `delegate_agent` action, satisfying ADR 0021
  A21;
- the research agent orchestrates only shipped v0.43 browser actions plus
  a pinned model-backed summarization path or deterministic extractive
  fallback, adding no new permission class, operation class, URI scheme,
  or registered action;
- `delegate_agent` accepts only allowlisted command strings for
  registered agents and rejects unknown strings without dynamic atom
  creation;
- a `browser_navigate` invoked inside a research dispatch still confirms
  (or applies a v0.43 remembered grant) — delegation provably does not
  widen authority;
- Plan/Build composes a `kind: delegate_agent` step targeting
  `research.specialist` and renders the specialist's per-source work
  inline under the parent step;
- research output is advisory and never auto-promotes to memory;
  `research.max_sources` bounds fan-out; the browser session is always
  closed;
- `docs/developer/delegate-agents.md` documents the extension point so a
  third-party plugin can register a delegate agent;
- the v0.46 security eval set passes on the stub driver; opt-in external
  smoke passes against a real browser;
- roadmap, vision §v0.46, agent-context-map, ADR 0021 A21,
  future-features, and CHANGELOG reflect the shipped state and the parked
  operator-authoring remainder.
