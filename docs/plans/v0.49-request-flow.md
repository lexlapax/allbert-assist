## Main User/Operator Flows

- Operator pastes or uploads an image.
- Operator captures or imports a screenshot.
- Allbert analyzes image/screenshot content through a vision-capable model.
- Operator requests image generation.
- Workspace renders media with retention/cache controls.

## Data Flow

Image and screenshot inputs become bounded resources, provider calls receive
only approved content, and traces store redacted metadata instead of binary
payloads. Vision analysis resolves `vision_input`; image generation resolves
`image_generation`. Both use the v0.48 capability-aware provider preference
substrate. Image generation is a registered action with provider profile and
display-only cost metadata.

The shared profile schema may include `video_input` or generic media metadata,
but v0.49 does not implement video ingestion, sampled-frame analysis, or video
generation unless this request flow is explicitly amended before code lands.

## Authority Invariants

- Media input does not grant action authority.
- Provider capability metadata and doctor success do not grant permission.
- Binary/media content is redacted from traces by default.
- Generated media calls follow provider and external-action policy.
- Workspace rendering does not execute model-generated UI code.
- `video_input` metadata is not authority to ingest or analyze video.
