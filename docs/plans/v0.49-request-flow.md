## Locked Decisions

Each is recorded in `v0.49-plan.md` §"Locked Decisions".

1. Reuse the v0.48 capability-aware provider/preference framework.
2. `vision_input` is a multimodal `ReqLLM` `ContentPart` on the existing
   text-generation call — no new adapter.
3. `image_generation` is a `generate_image` action wrapping
   `ReqLLM.generate_image/3` — no hand-rolled HTTP.
4. Provider scope: req_llm-backed OpenAI/Gemini remote profiles + fixture
   support for tests. M9 accepts opt-in local Ollama `vision_ollama` and
   `image_ollama` profiles through the existing ReqLLM/OpenAI-compatible local
   endpoint boundary; no ProviderHTTP/ADR 0011 work is added.
5. `image://`/`screen://` are operator-supplied media (paste/upload), not
   autonomous OS capture; vision may also analyze a browser screenshot through
   the explicit `analyze_browser_screenshot` bridge, which resolves an existing
   v0.43 `browser_screenshot` `cache://browser/...` ref into image-input
   metadata.
6. Image normalization = constrain to provider-accepted format/size/dimensions
   + bounded-reject; no resize/transcode dependency.
7. `:image_input` floor `:allowed`; `:image_generate` floor by deployment mode,
   reusing v0.48 `voice_floor` (fake → `:allowed`; remote/unresolved/unknown →
   `:needs_confirmation`, fail-safe-to-confirm, never auto-`:allowed`).
8. Cost is display-only pass-through from `ReqLLM.Response` usage.
9. `release.v049` uses fake providers + fixture images; opt-in `.env` live
   smokes are manual before tag.
10. Capability names are routing predicates; media metadata grants no authority.

## Main User/Operator Flows

- Operator pastes or uploads an image, or points at a screenshot, and asks
  about it; Allbert analyzes it via a `vision_input`-capable model.
- Operator analyzes a browser screenshot via `analyze_browser_screenshot`,
  using a prior v0.43 `browser_screenshot` read op / `screenshot_ref`.
- Operator requests image generation; Allbert runs `generate_image` (remote
  confirmed), returns a bounded generated image output for workspace/runtime
  rendering, and shows display-only provider/cost metadata.
- Workspace renders media with retention/cache controls and no generated UI
  code.
- v0.50 Artifacts Central is the version for the canonical content-addressed
  artifact store. v0.49 image files are bounded input/output resources with
  hashes and redacted metadata, not durable cross-surface artifact records.

## Provider/Capability Resolution

`vision_input` and `image_generation` resolve through the v0.48 `Models.for/2`
resolver (capability validation, ranked candidate list). The one-bounded-pass
call-failure fallback (retryable → next candidate; non-retryable → stop) with
per-profile floor re-authorization is not in the resolver — the consuming action
drives it over `candidates_for/2`, mirroring `transcribe_voice`. Then:

- **vision_input** → the resolved text profile's normal `ReqLLM` chat call with
  the image attached as a `ContentPart` (`image_url` or base64 `image`).
- **image_generation** → `ReqLLM.generate_image("provider:model", prompt, opts)`
  for the resolved image profile; bytes via `Response.image_data/1`.

No new provider HTTP module: `ReqLLM` owns provider endpoints, credentials, and
request shape for both, exactly as for text generation.

M1 must prove the selected model IDs and request paths inside Allbert's
app-started runtime. `mix run --no-start` is not release evidence for ReqLLM
provider/model discovery because provider registration initializes at
application startup. The proof covers `ReqLLM.Providers.list/0`,
`ReqLLM.Images.validate_model/1`, and fixture-backed request paths with no live
network in the deterministic release lane.

## Data Flow

```text
operator paste/upload | operator screenshot upload | analyze_browser_screenshot(cache://browser ref)
  -> bounded image resource (image://capture/<id> | screen://capture/<id>)
  -> :image_input permission (floor :allowed) + Resource Access (size/type/dims)
  -> ImageBounds validates format/bytes/dimensions before provider call
  -> Models.for(:vision_input) -> ReqLLM chat call with ContentPart image
  -> answer as normal runtime text; trace stores redacted media metadata only
  -> optional hash metadata only; no canonical artifact-store write in v0.49

prompt -> generate_image action
  -> :image_generate permission (deployment-mode floor; remote confirms)
  -> Models.for(:image_generation) -> ReqLLM.generate_image/3
  -> ImageBounds validates generated media bounds
  -> bounded generated image file + display-only usage/cost
  -> runtime/workspace receives local image_file plus redacted metadata
  -> trace/action metadata redacts binary and generated resource paths
```

## Multimodal Bridge Boundary

The shared profile schema may describe local or online multimodal models, but
v0.49 adds no catch-all `multimodal` capability and no all-purpose media
router. Every executable bridge remains a named capability:

- `speech_to_text` and `text_to_speech` are v0.48-owned audio paths.
- `vision_input` and `image_generation` are the only new v0.49 executable media
  paths.
- `video_input` stays vocabulary/metadata only.
- Generic audio understanding, video ingestion, sampled-frame analysis, and
  video generation need future capability names or amendments, resource
  classes, permission floors, doctor fields, and release evidence before code
  lands.

## Authority Invariants

- Media input does not grant action authority; provider capability metadata and
  doctor success never grant permission.
- Operator-supplied media only; no autonomous OS screen capture.
- `:image_input` is `:allowed` with redaction; `:image_generate` is
  confirmation-gated for remote providers (deployment-mode floor;
  remote/unresolved/unknown default to `:needs_confirmation`, never
  auto-`:allowed`).
- Binary image/screenshot content is bounded, default-off for retention, and
  redacted from traces/audits/assigns/CLI; only metadata persists.
- Content hashes in v0.49 are integrity/provenance metadata only. They are not a
  content-addressed storage contract and do not create cross-surface artifact
  lookup, retention, or lifecycle authority.
- `image://capture/<id>` and `screen://capture/<id>` are inert identifiers;
  malformed media URIs are rejected, and valid IDs do not grant permission.
- Image generation runs through `ReqLLM` + provider/secret config (as text); no
  bespoke network path, no new credential surface.
- Workspace rendering executes no model-generated UI code.

### Manual Validation Steps

Use the exact numbered shell checklist in
`docs/operator/vision-and-image-generation.md` §"Numbered Validation
Checklist". The expected release-validation order is:

1. Prepare a clean shell with no `MIX_ENV`, `DATABASE_PATH`, or
   `ALLBERT_HOME_DIR`; use
   `docs/samples/media/images/validation-input-small.png` for command-line
   smokes because it is below the 1 MiB metadata read cap.
2. Confirm the commit and clean tracked tree.
3. Validate v0.49 profile registration and Security Central floors/caps in a
   disposable `ALLBERT_HOME`.
4. Run `MIX_ENV=test mix allbert.test release.v049` and capture evidence.
5. Inspect v0.49 deterministic evidence for passed steps, passed secret scan,
   and no SQLite lock signatures.
6. Run `MIX_ENV=test mix allbert.test release` and capture evidence.
7. Inspect full-release phase evidence for all phases passed.
8. Validate image input through the registered action path with `vision_fake`.
9. Validate image generation through the registered action path with
   `image_fake`.
10. Validate Browser doctor readiness.
11. Run OpenAI live vision/image smoke with redacted evidence inspection.
12. Run Gemini live vision/image smoke with generated-output normalization
    evidence inspection.
13. Run local Ollama live vision/image smoke with `qwen3-vl:8b` and
    `x/z-image-turbo`.
14. Run Gemma 4 local vision-candidate smoke with
    `ALLBERT_V049_VISION_MODEL=gemma4:e4b`.
15. Start the disposable workspace server.
16. Validate workspace image upload/paste and image generation with fake media
    preferences.
17. Validate browser screenshot analysis through the operator UI flow.
18. Stop the workspace server, report evidence paths, then remove disposable
    homes only after report-back evidence has been copied.

Report `PASS` only if all required steps complete as expected. If any step
fails, report the step number, command or UI action, error output, and evidence
path.
