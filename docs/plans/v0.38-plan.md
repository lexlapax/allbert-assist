v0.38 is the curated, deterministic creation experience. Where v0.37 is the
dynamic LLM-driven generation/integration engine, v0.38 exposes vetted
templates through Mix tasks, operator workspace flows, and a Canvas Create
destination. Templates produce artifacts to the proven v0.27-v0.35 shapes and
reuse the v0.36 sandbox and v0.37 gate/loader when live integration is selected.

## Purpose

Developers and operators should be able to create common capabilities without
open-ended LLM authoring:

- source-tree plugin;
- workspace app;
- LLM-backed tool/agent;
- scheduled/chron flow;
- objective workflow;
- future reviewed code patterns.

The default output is inert source for human review. Optional operator live
integration reuses the v0.37 trust path exactly.

## Expected Inputs

- v0.36 sandbox/gate runner.
- v0.37 dynamic generation engine and gated live loader.
- v0.27-v0.35 contract shapes: plugin manifest, actions, panels, intent
  descriptors, settings fragments, memory namespace, objectives, destinations,
  and theming/layout docs.
- v0.34 Canvas/destination model and v0.35 layout constraints.

## Design Decisions

### 1. TemplatePattern registry

Templates are vetted, parameterized skeletons registered through a
`TemplatePattern` behaviour. A pattern declares:

- parameter schema;
- reviewed file set;
- output roots;
- target contract shapes;
- optional validation/gate profile;
- whether live integration is supported.

Template rendering is deterministic parameter substitution into reviewed files,
not free LLM authoring. Templates and parameters grant no authority.

### 2. Shipped patterns

- **plugin**: minimal source-tree plugin.
- **app**: full app/surface contract with panels, action stub, settings
  fragment, intent descriptor, memory namespace, objective/canvas hooks, and
  theming/layout docs.
- **LLM tool**: Jido.AI-backed tool/agent using the v0.25 advisory pattern.
- **scheduled/chron flow**: reviewed job/objective template for recurring
  operator workflows, backed by existing jobs/objectives contracts.
- **objective workflow**: reusable multi-step objective scaffold.

### 3. Two output modes

- **Developer scaffold**: writes inert source under `./plugins/<name>/`; no
  compile path change, trust, permission, route, skill enablement, or live
  registration.
- **Operator live integration**: renders the template, validates it, runs the
  v0.36 sandbox gate, and uses the v0.37 operator-confirmed loader. v0.38 adds
  no new sandbox or integration authority.

### 4. Developer Mix tasks

- `mix allbert.gen.plugin NAME [opts]`
- `mix allbert.gen.app NAME [opts]`
- `mix allbert.gen.tool NAME [opts]`
- `mix allbert.gen.flow NAME [opts]`
- `mix allbert.gen.<pattern> NAME [opts]`
- `mix allbert.validate_app NAME`

Generated output should pass validation on first run.

### 5. Operator Create destination

`workspace:create` is a Canvas destination for template gallery, parameter form,
preview, validation status, scaffold output, and optional live integration.
The surface is view/compose only. Effectful work runs through registered actions
and Security Central.

v0.34/v0.35 constraints carry forward: launcher state is view-only,
`active_app` remains handoff-only, Settings/Output are non-hideable,
`app:allbert` is invalid, and layout overrides cannot create routes or actions.

## Milestones

Each milestone closes with focused tests, docs/request-flow notes, and
`mix precommit` unless a narrower gate is recorded with why.

### M1: ADR + template/pattern registry

Accept ADR 0036 for v0.38. Add `TemplatePattern` behaviour, registry, parameter
schema, and target-shape metadata.

Tests: registry/list/validate, parameter schema validation, deterministic
rendering.

### M2: Plugin + app templates and Mix tasks

Ship plugin/app templates and `mix allbert.gen.plugin` / `gen.app` /
`validate_app`.

Tests: generated tree, validate-app first-run pass, inert output,
traversal/size denial.

### M3: LLM-tool, scheduled/chron, and objective templates

Ship LLM-tool template plus scheduled/chron and objective workflow patterns.

Tests: advisory boundary, no auto-enable, schedule/objective scaffolds are
inert until reviewed or live-integrated through v0.37.

### M4: Operator creation flow + Canvas surface

Add `workspace:create`, template gallery, parameter form, preview, validation
status, and developer-scaffold vs operator-live-integration choice.

Tests: workspace flow, view-only surface, defensive DOM handles, no integration
without registered action + confirmation.

### M5: Live templated integration through v0.37

Operator live integration of a templated artifact reuses v0.36 sandbox and
v0.37 loader.

Tests: gated integration, rollback, no bypass of v0.36/v0.37 trust path.

### M6: Security evals and closeout

Add evals for template parameter injection, traversal, authority bypass,
ungated/unconfirmed templated integration, scheduled-flow escalation, and Canvas
surface authority. Update README, roadmap, agent map, future-features, ADR
pointers. Bump version metadata to `0.38.0`.

## Non-Goals

- No new sandbox backend; v0.38 consumes v0.36.
- No new hot-load/integration authority; v0.38 consumes v0.37.
- No remote template marketplace or remote code-bearing templates.
- No autonomous creation from traces.
- No dependency/migration/NIF additions in generated templates.
- No multi-language templates beyond Elixir/OTP.

## Definition Of Done

v0.38 is done when developers can scaffold plugin/app/tool/flow patterns through
Mix tasks with first-run validation, operators can create from templates through
`workspace:create`, generated output is inert by default, optional live
integration reuses the v0.36/v0.37 gated path, and security evals prove
parameter, traversal, authority, scheduling, and Canvas bypass paths fail
closed.
