# Allbert v0.38 Templated Creation: Plugins, Apps, Tools, and Code Patterns

Request flow: `docs/plans/v0.38-request-flow.md`

Status: implemented through M6 closeout on 2026-05-27 and ready for operator
manual verification.

ADRs:

- `docs/adr/0036-templated-creation-and-pattern-registry.md` (this milestone)
- `docs/adr/0035-codegen-agents-and-live-integration-loader.md` (v0.37 engine)
- `docs/adr/0037-elixir-otp-sandbox-backend-and-gate-runner.md` (v0.36 sandbox)
- `docs/adr/0017-allbert-plugin-contract.md`
- `docs/adr/0015-allbert-app-contract-and-surface-dsl.md`

v0.38 is the curated, deterministic creation experience. Where v0.37 is the
dynamic LLM-driven generation/integration engine, v0.38 exposes vetted
templates through Mix tasks, operator workspace flows, and a Canvas Create
destination. Templates produce artifacts to the proven v0.27-v0.35 shapes and
reuse the v0.36 sandbox and v0.37 gate/loader when live integration is selected.

## Purpose

Developers and operators should be able to create common capabilities without
open-ended LLM authoring:

- source-tree plugin;
- workspace app;
- LLM-backed tool/agent;
- scheduled/chron flow;
- objective workflow;
- future reviewed code patterns.

The default output is inert source for human review. Optional operator live
integration reuses the v0.37 trust path exactly.

## Expected Inputs

- v0.36 sandbox/gate runner.
- v0.37.5 dynamic generation engine and gated live loader (released). The v0.37
  loader rejects generated apps, panels, settings fragments, memory namespaces,
  objective wiring, child processes, and route-page artifacts before in-core
  compile; only generated action artifacts can integrate live in this release.
- v0.27-v0.35 contract shapes: plugin manifest, actions, panels, intent
  descriptors, settings fragments, memory namespace, objectives, destinations,
  and theming/layout docs.
- v0.34 Canvas/destination model and v0.35 layout constraints.

## Design Decisions

### 1. TemplatePattern registry

Templates are vetted, parameterized skeletons registered through a
`TemplatePattern` behaviour. A pattern declares:

- parameter schema;
- reviewed file set;
- output roots;
- target contract shapes;
- optional validation/gate profile;
- whether live integration is supported.

The `live_integration?` declaration is consulted by `workspace:create` to hide
or disable the live-integration toggle for patterns whose artifact shape is not
loader-eligible in v0.37.5 (see Design Decision 3 matrix). Pattern declarations
are not authority — operator approval and the v0.37 loader still gate every
integration attempt.

Template rendering is deterministic parameter substitution into reviewed files,
not free LLM authoring. If templates use EEx or an equivalent renderer, only
shipped reviewed templates are evaluated; user parameters are data and are
escaped/normalized before substitution. Templates and parameters grant no
authority.

### 2. Shipped patterns

- **plugin**: minimal source-tree plugin.
- **app**: full app/surface contract with panels, action stub, settings
  fragment, intent descriptor, memory namespace, objective/canvas hooks, and
  theming/layout docs.
- **LLM tool**: Jido.AI-backed tool/agent using the v0.25 advisory pattern. The
  developer-scaffold output may call `Jido.AI` directly (reviewed source). The
  operator-live-integration output must route any provider/network or memory
  effect through `AllbertAssist.DynamicPlugins.Delegate.run/3` to an
  operator-allowlisted reviewed facade per v0.37.3 (`:external_network` →
  `external_network_request`, `:memory_write` → `append_memory`); `:read_only`
  variants need no delegate.
- **scheduled/chron flow**: reviewed job/objective template for recurring
  operator workflows, backed by existing jobs/objectives contracts.
- **objective workflow**: reusable multi-step objective scaffold.

### 3. Two output modes

- **Developer scaffold**: writes inert source under `--target`, defaulting to
  `./plugins/<name>/`; no compile path change, trust, permission, route, skill
  enablement, or live registration. Existing target directories are never
  overwritten without explicit `--force`; preview/diff is shown first.
- **Operator live integration**: renders the template, validates it, runs the
  v0.36 sandbox gate, and uses the v0.37 operator-confirmed loader. v0.38 adds
  no new sandbox or integration authority.

Live integration is bounded by what the v0.37.5 loader accepts. The shipped
patterns split as follows:

| Pattern | Developer scaffold | Operator live integration | Reason |
|---|:-:|:-:|---|
| plugin | yes | no | v0.37 loader rejects generated plugin manifests/apps/settings fragments. |
| app | yes | no | v0.37 loader rejects generated apps, panels, settings fragments, memory namespaces, and objective wiring. |
| LLM tool (action) | yes | yes | Action artifacts are the only v0.37.5 live-loadable shape. |
| scheduled/chron flow | yes | no | Jobs/schedules and objective wiring are deferred v0.37 live targets. |
| objective workflow | yes | no | Objective wiring is a deferred v0.37 live target. |

When an operator selects live integration in `workspace:create` for a pattern
whose `live_integration?` is `false`, the surface disables the toggle and the
`CreateFromTemplate` action fails closed with a bounded diagnostic that points
to developer-scaffold output. A future milestone (after v0.37 loader scope
widens) can flip patterns to `live_integration?: true` without ADR change.

Operator-created artifacts that do support live integration default to Allbert
Home draft roots and use the v0.37 file-backed metadata/gate path. They land
under `<ALLBERT_HOME>/dynamic_plugins/drafts/<slug>/` with `producer:
"template_pattern"` (and a `template_pattern_id` provenance field naming the
shipped pattern, for example `"llm_tool"`) recorded in `metadata.yaml`. They
enter the v0.37 lifecycle at `:draft` and traverse `RunDraftTrial` →
`RunDraftGate` → `IntegrateDraft` → optional `RollbackIntegration` like any
other dynamic draft. The Telegram/email/cross-channel approval-surface
exclusion (`dynamic_codegen.integration_approval_surfaces` default `["cli",
"liveview"]`) applies unchanged; templated integration cannot be approved or
rolled back from excluded channels. Developer scaffolds default to the project
`./plugins/<name>/` path for human review and never touch the dynamic-draft
store.

### 4. Developer Mix tasks

- `mix allbert.gen.plugin NAME [--target PATH] [--force] [opts]`
- `mix allbert.gen.app NAME [--target PATH] [--force] [opts]`
- `mix allbert.gen.tool NAME [--target PATH] [--force] [opts]`
- `mix allbert.gen.flow NAME [--target PATH] [--force] [opts]` — the
  scheduled/chron flow pattern; the objective-workflow pattern also reuses this
  task with an explicit `--pattern objective` flag rather than introducing a
  separate `gen.objective` task.
- `mix allbert.gen.<pattern> NAME [--target PATH] [--force] [opts]` — generic
  pattern dispatcher for future templated patterns.
- `mix allbert.validate_app NAME` — already exists; v0.38 does not introduce
  this task, it only guarantees generated apps pass it on first run.

Generated output should pass validation on first run.

Parameter normalization produces a slug, app id, module namespace, destination
id, schedule id, and output path from reviewed rules. Raw params never create
atoms directly; app ids and destination ids are validated before existing atoms
or registry entries are touched.

### 5. Operator Create destination

`workspace:create` is a Canvas destination for template gallery, parameter form,
preview, validation status, scaffold output, and optional live integration.
The surface owns only view/compose state. Effectful work runs through registered
actions and Security Central.

Templated drafts that enter the v0.37 lifecycle are inspectable through the
existing `mix allbert.dynamic drafts list`, `drafts show <slug>`, and
`drafts discard <slug>` paths exactly like LLM-authored drafts; v0.38 adds no
parallel inspection CLI.
The `producer: "template_pattern"` provenance field distinguishes templated
drafts from `codegen_llm` drafts in `list` output and audit records.

v0.34/v0.35 constraints carry forward: launcher state is view-only,
`active_app` remains handoff-only, Settings/Output are non-hideable,
`app:allbert` is invalid, and layout overrides cannot create routes or actions.
Generated token theme, snippet, and layout stubs are inert and disabled by
default; they document the v0.35 contracts without enabling snippets or layout
overrides.

### 6. Settings Central and observability

Developer Mix-task scaffolds are dev-time and inert and need no runtime settings.
The operator `workspace:create` surface is a runtime surface and is gated through
Settings Central. New settings are default-off and audited:

| Key | Type | Default | Meaning |
|---|---:|---:|---|
| `templates.create.enabled` | boolean | `false` | Master switch for the operator `workspace:create` destination |
| `templates.allowed_patterns` | string list | reviewed defaults | Patterns exposed in the gallery |

Operator **live integration** of a templated artifact adds no new authority: it
additionally requires `dynamic_codegen.enabled`, `dynamic_codegen.live_loader_enabled`,
and `sandbox.elixir.enabled`, and runs exactly the v0.36 sandbox gate plus the
v0.37 operator-confirmed loader. Settings store enablement and selections only;
generated artifacts and metadata remain file-backed per v0.37.

Template render, scaffold, validate, and create-integration are runtime-facing
and observable, so they route through registered actions and emit signals plus
trace/audit records like v0.37; templates and parameters grant no authority.

## Non-Goals

- No new sandbox backend; v0.38 consumes v0.36.
- No new hot-load/integration authority; v0.38 consumes v0.37.
- No live integration of generated apps, panels, settings fragments, memory
  namespaces, objective wiring, jobs/schedules, route pages, or child
  processes — these remain v0.37 deferred live targets and are
  developer-scaffold-only in v0.38.
- No remote template marketplace or remote code-bearing templates.
- No autonomous creation from traces.
- No dependency/migration/NIF additions in generated templates.
- No multi-language templates beyond Elixir/OTP.
- No overwrite of existing user/project files without explicit `--force` and
  preview/diff confirmation.

## Forward Consumers

- v0.46 operator-supervised self-improvement (`docs/plans/v0.46-plan.md`) may
  emit inert trace-derived workflow YAML and template parameter inputs that
  feed into v0.38 patterns; templated creation remains the only sanctioned
  authoring path for those drafts and they still traverse v0.36/v0.37 before
  any live authority.
- v0.45 marketplace-lite (`docs/plans/v0.45-plan.md`) surfaces reviewed
  template catalog metadata in `workspace:create`; v0.38 ships the gallery and
  registry that v0.45 metadata indexes.

## Definition Of Done

v0.38 is done when developers can scaffold plugin/app/tool/flow patterns
through Mix tasks with first-run validation, operators can create from
templates through `workspace:create`, generated output is inert by default and
non-destructive by default (`--target` defaults to `./plugins/<name>/` for
developer scaffolds), the live-integration toggle is available only for
patterns whose artifact shape is loader-eligible in v0.37.5 (LLM-tool/action
in this release), optional live integration reuses the v0.36/v0.37 gated path
and records `producer: "template_pattern"` in the shared draft store, and
security evals prove parameter, traversal, authority, scheduling, unsupported
live target, and Canvas bypass paths fail closed. Developer docs teach the
generator and pattern registry; operator docs teach `workspace:create`,
preview/validation, confirmation, rollback, and non-authority defaults.
