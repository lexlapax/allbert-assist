## Purpose

The downstream build order is **unchanged and correct**: v0.61 overhauls the
presentation layer, v0.62 packages the entry points, v0.63 builds guided
onboarding and personas, v0.64 is the product RC, and v1.0 freezes the public
contracts. What v0.60 changes is **sequencing**: it front-loads the *one* unified
technical-prosumer journey **design** so that presentation, packaging, and
onboarding are each built against a single designed experience instead of being
designed-as-they-go.

Without v0.60, each downstream release would discover the product shape while
implementing it: v0.62 would design entry points that v0.63 onboarding then has
to retrofit, and v0.63 onboarding would retroactively reshape the v0.61
presentation surface it was supposed to land on. That rework is expensive and,
critically, **some of it lands on contracts v1.0 freezes**. v1.0 freezes the
presentation contracts (the Surface DSL, the workspace substrate, the catalog
rendering boundary). The structural redesign that ADR 0074 deferred — the
chat-primary re-layout, the information architecture, the navigation model — must
therefore be **designed now (v0.60) and built in v0.61, before the freeze**. A
redesign discovered after v0.61 ships, or during the v0.64 RC, is a frozen-
contract break.

v0.60 produces six design outcomes plus a navigable skeleton, all on the existing
authority spine:

1. **One product-experience spec.** The install → first-run → onboard →
   first-value → daily-use journey, written once, for the technical-prosumer 1.0
   audience, as the contract every downstream release builds against (M1).
2. **An information architecture & navigation model.** Sitemap, screen inventory,
   navigation model, and workspace/screen composition — the structural redesign
   ADR 0074 deferred — accepted as ADR 0077 (M2).
3. **A First-Model Path decision.** Assisted-local default + BYOK fallback
   (managed-hosted rejected), the QuickStart "first useful chat" contract, and the
   detect + guided-Ollama-install packaging implications, accepted as ADR 0078 (M3).
4. **An onboarding-flow & persona-model design.** The two-track wizard UX and the
   persona/profile model as design specs feeding v0.63 (the ADR 0075 content) (M4).
5. **An entry-point & CLI UX design.** Command taxonomy, first-run detection, and
   wizard launch as a spec feeding v0.62 (the ADR 0076 entry points) (M5).
6. **A walking skeleton + design-system gap analysis.** A navigable nav shell with
   placeholder screens implementing the IA (the gate artifact and the v0.61 build
   scaffold), plus the token/component/pattern gaps the overhaul needs (M6, M7).

The authority boundary is unchanged. No new capability, no new authority surface,
no confirmation-floor change, no egress grant, no contract freeze. The walking
skeleton renders placeholders through the existing catalog; it grants nothing.

## Cross-Cutting Invariants (apply to every milestone)
- **Design + scaffolding only; no new runtime behavior.** v0.60 ships no new
  capability class, no new authority surface, no confirmation-floor change, and no
  egress grant. The only running code it adds is a navigable walking skeleton of
  placeholder screens. Security Central is unchanged and remains the sole
  authority boundary.
- **The catalog stays the rendering boundary.** The walking skeleton renders
  through the unified catalog inside the one app shell (ADR 0030/0073/0074). No
  model-generated or data-generated markup is introduced; placeholder screens are
  static catalog/token-consuming compositions.
- **Placeholder screens carry no live data and grant no authority.** Skeleton
  routes resolve, render a navigable shell, and render placeholder content; they
  do not read business state, do not compose effectful action DTOs as live
  affordances, and grant no authority or egress. They exist to prove the IA is
  navigable and to scaffold v0.61.
- **Settings Central is unchanged.** v0.60 adds no Settings key (see the keys
  table). The persona model, First-Model Path, and onboarding flow are **designs**
  of settings-seeded behavior, not new keys or seeded values.
- **Designs are inputs, not implementations.** M4 designs the ADR 0075 persona
  content, M5 designs the ADR 0076 entry points, M7 feeds ADR 0074 — none of these
  ADRs are implemented in v0.60. v0.60 accepts only ADR 0077 and ADR 0078.
- **One spine.** Nothing in v0.60 reaches `Settings.Store`, `Confirmations`, or
  business logic directly; the skeleton consumes the catalog/shell contract only.

### M1: Product Experience Spec
**Status:** implemented on 2026-06-30.

audience: the **install → first-run → onboard → first-value → daily-use** journey,

### M3: First-Model Path Decision (→ ADR 0078)
**Status:** implemented on 2026-06-30.

Decide how a brand-new operator gets to their **first useful chat** — the
first-value moment M1 defines — and record it as **ADR 0078 (First-Model Path)**.
**Decision (recorded 2026-06-30): assisted-local model as the QuickStart default,
honest BYOK as the Advanced/fallback path, managed-hosted rejected.**

- **assisted-local (default)** — QuickStart runs a local model so the first useful
  chat is fully on-device: no key, nothing leaving the machine — the trust position
  surfaced as the product advantage, and the zero-config bar (LM Studio / Jan) met.
  **Mechanism: detect + guided Ollama install, not a bundled runtime** — then a
  one-click pull of a curated default model selected against criteria (open-weight,
  runs on typical prosumer hardware, modest download weight; no hard licensing or
  family constraint pinned). v0.60 records the selection criteria; v0.62 selects
  and refreshes the concrete curated default.
- **honest-BYOK (Advanced + fallback)** — the Advanced track pastes a provider key
  or points at an existing local model/endpoint, and BYOK is the graceful fallback
  when the machine is under the curated model's hardware floor or Ollama is
  unavailable, so "first useful chat" always has a path — local first, BYOK second.
- **managed-hosted (rejected)** — an Allbert-operated relay is rejected: a perpetual
  abuse/cost-owned service the project does not have, and routing the default first
  chat through it contradicts the local-first trust position.

ADR 0078 records this decision, the **QuickStart "first useful chat"** contract
(local model via guided Ollama; BYOK fallback), and the **packaging implications**
v0.62 must satisfy (detect + guided Ollama install, curated model pull, no runtime
bundling; first-run model-state detection; the OS secret-vault path for BYOK keys).
The decision shapes M4 onboarding (the QuickStart track) and M5 entry points
(first-run detection), so it lands early.

- **Implementation:**
  - **Artifacts:** `docs/adr/0078-first-model-path.md` (Accepted (v0.60));
    `docs/design/first-model-path.md` — the QuickStart "first useful chat" path,
    the option analysis, and the packaging-implications handoff.
  - **Inputs:** M1 first-value definition; ADR 0072 recommended model profiles;
    the v0.59 settings/secret-reference substrate; the packaging constraints
    ADR 0076 anchors.
  - **Outputs:** the QuickStart contract M4 onboarding designs around; the
    packaging implications v0.62 implements; the first-run model-state check M5
    first-run detection designs.
  - **Verify:** the M8 `design_artifacts` presence check (the
    `v060_sweep_eval_test.exs` row that `File.read!`s
    `docs/design/first-model-path.md` and asserts the QuickStart path, BYOK
    fallback, option analysis, and v0.62 packaging-implications handoff) plus
    request-flow **S1** and **S3**.
- **Acceptance:** ADR 0078 is Accepted (v0.60) recording assisted-local-default +
  BYOK-fallback (managed-hosted rejected); the QuickStart "first useful chat"
  contract is defined; the detect + guided-Ollama-install packaging implications

### M6: Walking Skeleton
**Status:** implemented on 2026-06-30.

Build a **navigable walking skeleton** that implements the M2 information
architecture: the navigation shell exposing the IA navigation model, and a
**placeholder screen at every route in the IA sitemap**, rendered through the
unified catalog inside the one app shell. This is the **gate artifact** (the
proof the IA is navigable and a11y-clean) and the **v0.61 build scaffold** (the
route + nav + screen structure v0.61 fills in with the real overhaul).

The skeleton is placeholder-only: each screen renders a static, token/catalog-
consuming placeholder composition with at least one route-specific composition
zone, no live data, no business-state reads, and no effectful affordances. Routes
resolve (every IA route returns a rendered shell), the nav shell exposes the IA's
navigation model (active-route + navigation between screens resolve), and the
existing accessibility axes hold (keyboard focus, high-contrast, reduced-motion)
across every skeleton screen. It introduces no new authority, no new capability,
and no model-/data-generated markup.

  install script, or Home-layout change; the entry-point UX is *designed* (M5) and
  *built* in v0.62.

  repo, no seeded Settings; *designed* (M4), *built* in v0.63.

  key.** Security Central is unchanged.
