## Locked Decisions
Recorded in `v0.55b-plan.md`. Console adds no authority over the v0.55 channel;
each slash-command is a **registered read-only operator inspection action**
resolved through `Actions.Runner.run/3`; inspection actions are `exposure:
:internal`, reachable only through the TUI slash-command allowlist, and **not
intent candidates**; the canonical slash set is fixed
(`/status`, `/confirmations`, `/events`, `/channels`, `/settings get`, `/help`);
slash-commands + `mix allbert.channels status` are **thin views over the same
named read-report interface / existing action reads** their `mix allbert.*` twins
use; output is redacted; the warm `mix allbert.tui` console is the go-forward
standard for interactive operator validation.

## Console Data Flow

The console adds a slash-command branch to the v0.55 TUI normalize step. A leading
`/` token routes to a registered read-only operator inspection action through the
Runner and renders a redacted view; **any other input** follows the v0.55 channel
path unchanged.

```text
terminal input (Owl.IO.input loop, supervised GenServer — the v0.55 adapter)
  -> normalize (trim, drop empty)
  -> leading "/" ?
     |
     |-- YES (slash-command, operator inspection branch):
     |     /help and unknown slash are handled ROUTER-LOCAL (no action, no Runner):
     |       /help -> render the canonical set; unknown -> redacted "unknown command"
     |     the five DATA commands dispatch to the slash-allowlisted internal action:
     |       -> Actions.Runner.run(<inspection_action>, params, operator_context)
     |          (exposure: :internal; permission: :read_only; NOT an intent candidate;
     |           operator_context carries the resolved channel user_id; model never routes here)
     |       -> shared read-report interface / existing action read
     |          (the SAME DTO source the mix allbert.* twin renders, where a twin exists)
     |       -> redacted/truncated render as scrollback lines
     |     NEVER calls Runtime.submit_user_input/1; never a model turn
     |
     |-- NO (normal input): the v0.55 channel path, unchanged:
           Channels.Identity.resolve (channels.tui.identity_map)
           -> Channels.derive_session_id ({:tui_session, prefix: "ch_tui_"})
           -> Runtime.submit_user_input(%{channel: "tui", ...})
           -> Conversations.ChannelThread lookup/link/message-ref
           -> channel_events create/update (dedupe; redacted summary)
           -> response as scrollback lines + one Owl.LiveScreen live block
           -> approval -> Approval.Handoff.render/2 -> :list / :typed_command
```

Argument grammar (the router parses these before dispatch): `/settings get <key>`
takes one required dotted setting key; `/status`, `/confirmations`, `/events`,
`/channels`, and `/help` are bare commands with no arguments in v0.55.1. A missing
required key, invalid setting-key shape, extra argument, or unknown slash is an
inert router-local error (`tui-slash-parse-001`) — never a model turn.

Slash-command → registered action mapping (M1), all resolved through
`Actions.Runner.run/3` over the same read-report DTO source as the named
`mix allbert.*` twin (where a twin exists):

| Slash | Reads | Cold-task twin / facade |
|-------|-------|-------------------------|
| `/status` | warm-session process/channel status (node + `Channels.Supervisor` child health, via an `Operator.Inspection` read — never a direct `Supervisor.which_children`/Repo call) | `/channels` + supervisor status |
| `/confirmations` | pending/resolved confirmations | `mix allbert.confirmations` |
| `/events` | recent `channel_events` rows (redacted) | shared operator event report |
| `/channels` | registered channel release/enabled/supervisor status (same `Channels.list_channels()` substrate as `list_channels`, plus supervisor status) | `mix allbert.channels status` |
| `/settings get` | a settings key (secret fields redacted) | `mix allbert.settings get` |
| `/help` | lists the canonical slash set | (in-TUI; no facade) |

Only `/channels` (↔ `mix allbert.channels status`), `/confirmations`
(↔ `mix allbert.confirmations`), and `/settings get` (↔ `mix allbert.settings get`)
have a cold Mix twin. `/status` and `/events` have **no** cold twin — their single
source of truth is the shared `Operator.Inspection` report itself; the
source-of-truth eval applies the twin-equivalence check only where a twin exists.
`/help` and any unknown slash are router-local (no action, no Runner).

## Authority Invariants
- Slash-commands read **only** through registered internal actions resolved via
  `Actions.Runner.run/3` and reachable through the TUI slash allowlist; **never**
  direct Repo / Security Central / confirmation-store / Settings-authority access
  (ADR 0016/0070).
- Inspection actions are `exposure: :internal`, slash-allowlisted, and **not
  intent candidates** — the intent router cannot select one; a conversation turn
  never routes to one.
- No slash-command mutates: no write, no active-app set, no resource-access grant;
  `/settings` is `get`-only.
- Slash-command and `mix allbert.channels status` output is redacted/truncated; no
  secrets, credentials, or raw payloads.
- Each slash-command and `mix allbert.channels status` reads the **same** named
  read-report interface / existing action read as its `mix allbert.*` twin — one
  source of truth, no forked read path.
- The console adds no authority beyond the v0.55 TUI channel; the channel remains a
  delivery adapter.

## Handles
- In-session TUI slash-commands (the canonical set): `/status`, `/confirmations`,
  `/events`, `/channels`, `/settings get`, `/help`.
- `mix allbert.channels status` (operator-readable channel status report; same
  read-report DTO source as `/channels`; no secrets, no new authority).
- `mix allbert.tui` (the persistent, warm terminal session — DB open,
  `Channels.Supervisor` up, owl input loop live).

## Security / Eval Rows (`:v0551`)
See `v0.55b-plan.md`: `tui-slash-readonly-001` (reads never mutate or leak past the
ADR 0016 boundary; internal + slash-allowlisted; not intent candidates),
`tui-slash-source-of-truth-001` (same report DTO source as the `mix allbert.*` twin,
for the three commands that have one — `/channels`, `/confirmations`, `/settings get`;
`/status` and `/events` are report-backed with no cold twin),
`tui-console-warm-session-001` (full validation turn, no BEAM restart),
`tui-channel-status-redaction-001` (status output is secret-free),
`tui-slash-parse-001` (unknown/malformed slash is inert), and
`tui-inspection-not-agent-candidate-001` (inspection actions are absent from
agent candidates and descriptors).

## Operator Validation
- `mix allbert.test release.v0551` (deterministic; covers the `:v0551`
  readonly/source-of-truth/redaction eval rows with a fake input source — no live
  terminal session required). `mix allbert.test` (CI gates) is otherwise unchanged.
  The lane composition (disposable migration → slash-router/parse tests →
  inspection-action read tests → `:v0551` eval lane → channel-pack secret scan) is
  enumerated once in `v0.55b-plan.md` M4; treat that as the single source of truth.

## Operator-Validation Punchlist (v0.55.1, run entirely IN-SESSION)

Run the whole punchlist inside **one warm `mix allbert.tui` session** — **no cold
mix between checks**. Unlike pre-v0.55.1 operator validation (each step a fresh
cold `mix allbert.ask`/`mix allbert.*` against a shared `ALLBERT_HOME`), this
punchlist holds one supervised node: a single `mix allbert.tui` keeps the node, the
`Channels.Supervisor` tree, and the SQLite store open for the whole run. No step
restarts the BEAM or reopens the DB; every check is typed at the live prompt or
issued through an in-TUI slash-command against that same running node. This warm
in-session run is the **go-forward standard** for interactive operator validation.

These validation **instructions** are migrated in **M3** and finalized alongside
the deterministic eval/docs lane in **M4**. The punchlist is **run as the M5
operator-validation gate**, and a failing run blocks the **M6** release closeout /
version tag.

### Setup (one-time, pre-launch)

Run these commands from the umbrella root before launching the TUI. These setup
commands are intentionally cold Mix commands; the **no cold mix** rule starts once
`mix allbert.tui` is launched in S5.

**S0 — Confirm the upstream tag and prepare local evidence paths.**

```bash
git status -sb
git tag --list v0.55.0

export V0551_MANUAL_HOME="$(mktemp -d /tmp/allbert-v0551-manual.XXXXXX)"
export ALLBERT_HOME="$V0551_MANUAL_HOME"
export V0551_EVIDENCE_DIR="$HOME/.allbert-release-evidence/v0551"
export V0551_TRANSCRIPT="$V0551_EVIDENCE_DIR/manual-v0551-$(date -u +%Y%m%dT%H%M%SZ).log"

mkdir -p "$V0551_EVIDENCE_DIR"
printf 'ALLBERT_HOME=%s\n' "$ALLBERT_HOME"
printf 'V0551_TRANSCRIPT=%s\n' "$V0551_TRANSCRIPT"
test -d "$ALLBERT_HOME" && echo "manual home ok"
test -d "$V0551_EVIDENCE_DIR" && echo "evidence dir ok"
```

Expected: `git status -sb` is clean on the intended release branch, `git tag
--list v0.55.0` prints `v0.55.0`, both paths print, and the two `test` commands
print `manual home ok` and `evidence dir ok`. `V0551_EVIDENCE_DIR` is deliberately
outside the repository; preserve it locally for the operator handoff, but do not
commit transcripts or release JSON evidence.

**S1 — Run the deterministic v0.55.1 gate before manual validation.**

```bash
ALLBERT_TEST_KEEP_TMP=1 MIX_ENV=test mix allbert.test release.v0551
export V0551_RELEASE_EVIDENCE="<paste printed release.v0551 evidence path>"
test -f "$V0551_RELEASE_EVIDENCE" && echo "release.v0551 evidence ok"
```

Expected: `release.v0551` exits 0 and prints an evidence JSON path. The `test`
command prints `release.v0551 evidence ok`. This deterministic evidence proves the
non-operator-observable invariants: slash lines do not create model turns, malformed
slashes are inert, inspection actions are not model-routable, `/channels` and
`mix allbert.channels status` share the same read-report source, and status output
is secret-free.

**S2 — Migrate and configure the validation home.**

```bash
mix allbert.ecto.migrate --quiet

mix allbert.settings set channels.tui.identity_map '[{"external_user_id":"default","user_id":"local","enabled":true}]'
mix allbert.settings get channels.tui.identity_map
mix allbert.settings set channels.tui.enabled true
mix allbert.settings get channels.tui.enabled
```

Expected: migration exits 0. The identity-map `get` output includes one enabled
entry with `external_user_id` `"default"` and `user_id` `"local"`. The enabled
`get` output prints `channels.tui.enabled=true`.

**S3 — Inspect the TUI channel before launch.**

```bash
mix allbert.channels show tui
mix allbert.channels --parity
```

Expected for `show tui`: channel `tui`, provider `terminal`, `Enabled: true`,
`Identities: 1`, and no secrets. Expected for parity: a row for `tui` with
provider `terminal`, primitives `typed_command+list`, threading `rich`, identity
`channels.tui.identity_map`, approval `typed_command+list`, attachments `none`,
streaming `turn_complete`, outbound `none`, release `released`, and live `true`.
These are prelaunch inspection commands; they must not print the TUI banner or
start an input prompt.

**S4 — Preflight the confirmation-required Notes/files route used in B4.**

```bash
mix allbert.plugins show allbert.notes_files
mix allbert.apps show notes_files
mix allbert.settings get apps.notes_files.notes_root
mix allbert.settings get permissions.notes_file_write
```

Expected: the plugin output shows `allbert.notes_files` enabled/trusted with a
`write_note` action contribution; the app output shows app `notes_files` and action
`write_note`; `apps.notes_files.notes_root` resolves inside the current
`ALLBERT_HOME`; and `permissions.notes_file_write` is `needs_confirmation`. If any
of these are missing, stop and fix plugin/app registration or settings before
launching the TUI; B4 depends on this route.

**S5 — Launch the persistent supervised terminal session and capture transcript.**

```bash
script -q "$V0551_TRANSCRIPT" mix allbert.tui
```

Expected on launch: the v0.55 banner (`"tui"` channel, provider `"terminal"`), the
resolved local profile, and a live prompt such as `allbert:default>`. This is the
ONE warm BEAM for B1-B4: DB open, `Channels.Supervisor` up, the Owl input loop
live. Do not exit it until B1-B4 are complete. Every command below is typed inside
this session and is captured in `$V0551_TRANSCRIPT`.

### B. Console checks (all inside the one warm session)

**B1 — `/help` lists the canonical set; slash never routes to the model.**

```text
> /events
> /help
> /events
```

Expected: the canonical list of exactly `/status`, `/confirmations`, `/events`,
`/channels`, `/settings get`, `/help`. Pass criteria: `/help` renders the six
commands, produces no assistant/model response, and the before/after `/events`
views do not show an inbound `/help` row. Deterministic proof that `/help` never
created a `channel_events` inbound row lives in `tui-slash-parse-001` /
`tui-slash-readonly-001`; the manual evidence is the rendered list and the
unchanged event view.

**B2 — Each slash-command returns correct redacted data.**

```text
> /status
> /channels
> /events
> /confirmations
> /settings get channels.tui.identity_map
```

Expected: `/status` reports warm-session process/channel status; `/channels`
reports registered channel release/enabled/supervisor status; `/events` lists
recent `channel_events` rows with redacted/truncated summaries; `/confirmations`
lists pending/resolved confirmations; `/settings get` echoes the key with secret
fields redacted. `/status` must include stable warm-session identity, such as BEAM
OS PID or equivalent VM identity, node name, uptime, and `Channels.Supervisor`
child health. Pass criteria: each rendered view is truthful for the current
session and contains no secrets, credentials, raw payloads, or unbounded text.
Evidence: each rendered view in the transcript. The same-report-DTO guarantee and
the `/channels` ↔ `mix allbert.channels status` equivalence are validated by
`tui-slash-source-of-truth-001`, not by a cold Mix command during the manual
in-session punchlist.

**B3 — Inspection actions are slash-allowlisted internal reads (NOT intent-routable).**

Type a conversation turn whose phrasing resembles an inspection request, and
confirm the **model does not route to** an inspection action — it answers as a
normal turn (or declines), it does not execute `/events`/`/status` semantics:

```text
> what are my recent channel events and confirmations?
> /events
```

Pass criteria: the natural-language turn behaves like a normal conversation turn
(assistant response or refusal), not an operator report. It must not render the
`/events`, `/confirmations`, or `/status` report shape. Only the explicit `/events`
slash-command, on the slash-allowlisted branch, returns redacted event rows.
Evidence: the natural-language turn output next to the explicit `/events` output.
The end-to-end no-route guarantee is locked by `tui-slash-readonly-001` and
`tui-inspection-not-agent-candidate-001`.

**B4 — Full validation turn (send → approve → inspect), no BEAM restart.**

Run one end-to-end validation turn entirely in the warm session: send a turn that
triggers a `needs_confirmation` action, approve it via the typed primitive, then
inspect the result via slash-commands — never leaving the session:

```text
> create a note titled v0551 with body warm console check
> ALLBERT:APPROVE:<CONF_ID>
> /status
> /confirmations
> /events
```

When the approval handoff appears, copy the exact confirmation id into
`ALLBERT:APPROVE:<CONF_ID>`. If no approval handoff appears, stop: the Notes/files
preflight or intent/action route is not valid for this run.

Pass criteria: the turn completes, the approval resolves through
`Actions.Runner.run("approve_confirmation", ...)` (the v0.55 Runner path),
`/confirmations` shows `<CONF_ID>` moved pending → resolved/approved, `/events`
shows the redacted channel-event rows the TUI actually records for the turn
(inbound and callback rows, plus any outbound row only if the channel records one),
and the `/status` warm-session identity still matches B2 with increased uptime —
**all in one persistent session with no node restart and no DB reopen**
(`tui-console-warm-session-001`). Evidence: the approval handoff, the typed
approval command, the resolved confirmation, the `/events` rows present for this
channel, and matching `/status` identity.

### Evidence Retention

The v0.55 manual-validation evidence was left in an ephemeral `/tmp` Allbert Home
that was later emptied, leaving the recorded acceptance artifact unrecoverable at
audit time. To avoid repeating that, the M5 operator **must capture a durable,
redacted record** of this run rather than relying on the `/tmp` session home:

After B4 passes, exit the TUI from inside the same session:

```text
> /quit
```

Then run these post-session checks from the shell that launched S5:

```bash
test -s "$V0551_TRANSCRIPT" && echo "manual transcript ok: $V0551_TRANSCRIPT"
rg -n '/help|/status|/channels|/events|/confirmations|/settings get|ALLBERT:APPROVE' "$V0551_TRANSCRIPT"
printf 'release.v0551 evidence: %s\n' "$V0551_RELEASE_EVIDENCE"
printf 'manual transcript: %s\n' "$V0551_TRANSCRIPT"
```

Expected: the transcript exists, the `rg` command finds the required prompts, and
both evidence paths print. Preserve the local transcript and gate evidence for the
operator handoff, but record release closeout in `CHANGELOG.md` / release logs as
"evidence verified" rather than committing local evidence files. Do not cite the
disposable `/tmp` `ALLBERT_HOME` as the only evidence location.

### Sign-off

B1-B4 pass inside one persistent `mix allbert.tui` session with **no cold mix and
no BEAM restart**; `$V0551_TRANSCRIPT` is retained; the deterministic
`mix allbert.test release.v0551` gate is green; `tui-slash-source-of-truth-001`
proves `/channels` agrees with `mix allbert.channels status`, and neither leaks
secrets. With those, the **M5** operator-validation gate is cleared and the **M6**
release closeout / version tag may proceed.
