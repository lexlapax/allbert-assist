# v0.12 — Self-improvement

Status: Proposed

## Context

After v0.11 shipped the TUI, kernel-owned telemetry, adaptive memory routing, episode/fact recall, and the review-first learning seam, self-improvement is safer to implement. The operator can review patches and skill drafts from a richer UI, with clear cost/token/context visibility, inbox visibility, and better memory recall.

v0.12 carries forward the previous self-improvement design: a Rust rebuild skill, a user-facing skill-authoring skill, and an embedded scripting seam. The sequencing changes; the trust posture does not. All artifacts still route through explicit review. The old v0.11 self-improvement pointer file has been removed; this file is the canonical plan for ADRs 0067-0073, with cross-release framing in [ADR 0080](../adr/0080-self-change-artifacts-share-approval-provenance-and-rollback-envelope.md).

## Goal

Ship three capabilities, all framed as "Allbert does this for the user under review":

1. A `rust-rebuild` skill that reads, modifies, builds, and tests the Allbert codebase inside a sibling worktree, then produces diffs for operator review.
2. A `skill-author` skill that scaffolds AgentSkills-format skills through natural-language conversation and submits them through the standard install quarantine.
3. A `ScriptingEngine` trait with a sandboxed Lua implementation as the first embedded runtime.

## Non-goals

- No autonomous self-modification.
- No automatic binary swap of the running daemon.
- No bypass of skill install preview.
- No Lua filesystem/network access by default.
- No source-checkout requirement for ordinary Allbert usage.
- No local model retraining or distillation pipeline.
- No new inbox kinds beyond `patch-approval`.
- No new status-line items in v0.12 (the existing `inbox` count surfaces pending patch approvals).
- No PROTOCOL_VERSION bump (all changes are additive).
- No automatic installation of the Lua engine; both config and exec-policy gates must be set explicitly.
- No embedded Lua tool-call bridge in v0.12. Lua is a JSON-in/JSON-out transform surface; future host tool access would need its own design pass.

## Design decisions

### Worktree posture

- Self-modification operates in sibling git worktrees rooted at `~/.allbert/worktrees/<branch>/` (configurable via `self_improvement.worktree_root`) with their own isolated `target/`.
- Source checkout resolves in this order: `self_improvement.source_checkout`, `$ALLBERT_SOURCE_CHECKOUT`, then upward walk from the running executable's symlink-resolved path looking for a workspace `Cargo.toml` that contains an Allbert kernel crate member. Detection should accept either a package named `allbert-kernel` in a member manifest or a workspace member path ending in `crates/allbert-kernel`, matching the current repo layout. If none resolves, `rust-rebuild` refuses to activate with an operator-readable error pointing at `allbert-cli self-improvement config`.
- A path allowlist enforces write-only access to the worktree tree. The active source checkout's working tree, any path under `~/.allbert/` other than the worktree root, and the `skills/rust-rebuild/` directory inside the source checkout are off-limits. The kernel exec seam enforces this; the skill cannot grant itself write access by editing its own SKILL.md.
- A `self_improvement.max_worktree_gb` cap (default `10`) bounds total worktree disk use; the daemon refuses to create a new worktree past the cap.
- Worktree garbage collection is operator-driven via `allbert-cli self-improvement gc`. The daemon never runs GC automatically.

### Validation contract

- Tier A (per [ADR 0064](../adr/0064-default-contributor-validation-is-provider-free-temp-home-based-and-network-optional.md)) is the safe-to-merge bar inside the worktree: `cargo fmt --check`, `env -u RUSTC_WRAPPER cargo clippy --workspace --all-targets -- -D warnings`, `env -u RUSTC_WRAPPER cargo test -q`, `env -u RUSTC_WRAPPER cargo run -q -p allbert-cli -- --help`, and a temp-home `daemon status` smoke. Tier B (live-provider smokes) is not required.
- Any Tier A failure marks the patch `needs-review`. Inbox rendering shows the failing context but the install path refuses without explicit override.
- Tier A counts as a single turn-unit for ADR 0051 cost accounting. Wall-clock build time is not metered; only model token usage is. A rebuild turn that exceeds the daily cap mid-run refuses at the next turn boundary, the same way any long-running tool sequence does.
- `cargo` network access for dependency fetches is governed by exec policy (ADR 0004 / ADR 0034) only, not by ADR 0053's web-learning gate.

### Patch approval posture

- A new `patch-approval` inbox kind extends [ADR 0060](../adr/0060-approval-inbox-is-a-derived-cross-session-view.md). File layout, identity scoping, and retention defaults stay in force.
- Diffs are stored as artifacts at `sessions/<sid>/artifacts/<aid>/patch.diff` and referenced by the approval markdown via `artifact_path` — never inlined.
- The renderer surfaces summary, branch, worktree, validation status, file stats, and next-step hints pointing at `self-improvement diff` and `install`.
- `inbox accept` marks the patch approved, records approver identity and reason, and does nothing else. It does not apply the patch and does not swap any binary.
- `inbox reject` marks the patch rejected and (by default) deletes the worktree at `worktree_path`. `self_improvement.keep_rejected_worktree = true` preserves rejected worktrees for forensic review; reclamation then requires `self-improvement gc`.
- Pending and resolved patch approvals follow the same retention as other inbox kinds (`channels.approval_inbox_retention_days`, default `30`).

### Install posture

- The Allbert daemon never automatically swaps its own running binary. Installing a built rebuild artifact is always an explicit operator action.
- `allbert-cli self-improvement install <aid>` applies the patch via `git apply` to the source checkout. Mode is `apply-to-current-branch` by default (`self_improvement.install_mode`); other modes (`merge-into-branch`, `cherry-pick`) are explicit additions in future releases.
- The CLI emits a build-and-install hint pointing at `cargo install --path crates/allbert-cli` and `daemon restart` for the operator to run on their own time.
- An append-only install log lives at `~/.allbert/self-improvement/history.md`. It records approval id, applied SHA, and operator identity (per [ADR 0058](../adr/0058-local-user-identity-record-unifies-channel-senders.md)) at apply time. It does not record whether the binary was subsequently installed.
- `self_improvement.auto_swap_binary` does not exist as a config key. There is no flag that flips this on. A future release may add a `daemon upgrade` command (mediated swap), but it would need its own ADR.

### Skill provenance posture

- ADR 0032 frontmatter gains an optional `provenance` field with values `external | local-path | git | self-authored`. Existing skills without the field load as `external`. Validation is additive — no migration is required for installed content.
- Provenance survives promotion from `incoming/` to `installed/`. Operators cannot accidentally relabel a self-authored skill as external by re-installing.
- `allbert-cli skills list` adds a `Source` column. Install preview surfaces provenance alongside name, description, and allowed-tools. The skill loader logs provenance at activation time.
- Trust posture is unchanged: a self-authored skill goes through preview + confirm just like an external one. Provenance is observability, not policy.

### `create_skill` hardening

- The kernel `create_skill` tool gains a required `skip_quarantine: bool` argument. There is no default value; callers must state intent. The boolean is recorded in hook metadata for audit.
- `skip_quarantine: true` writes to `installed/` and is reserved for first-party kernel seeding (`memory-curator`, `skill-author`). Prompt-originated tool calls cannot use it.
- `skip_quarantine: false` writes to `incoming/`, emits an install-preview event, and routes through the standard confirm flow. This is what `skill-author` uses.
- `allowed-tools` still names tools only; it does not grow parameter-level policy in v0.12. `skill-author` lists `create_skill` in `allowed-tools`, and the kernel `create_skill` handler denies `skip_quarantine: true` for any prompt-originated or active-skill call.
- This hardening lands before `skill-author`; otherwise the authoring skill would have to choose between bypassing quarantine or inventing a temporary draft path.

### Skill-authoring posture

- `skill-author` ships under `skills/skill-author/` in the repo and seeds into `~/.allbert/skills/installed/skill-author/` on first daemon run after v0.12 upgrade — same shape as `memory-curator` per [ADR 0048](../adr/0048-v0-5-ships-a-first-party-memory-curator-skill.md). The seeded skill carries `provenance: external` (it is shipped by Allbert, not authored by the user).
- Intent routing dispatches natural-language requests ("make me a skill that...", "create a skill that...", "build a skill for...") to `skill-author`. Triggering patterns live in the skill's own SKILL.md frontmatter (`intents:` per [ADR 0031](../adr/0031-skills-can-contribute-agents-via-frontmatter.md)), not in the kernel.
- Drafts persist across turns at `~/.allbert/skills/incoming/<draft-name>/` (the standard quarantine path). Iterative drafts survive session exit because `incoming/` is on disk.
- Final submission routes through the standard install preview + confirm flow.
- Drafts written by `skill-author` carry `provenance: self-authored`; the seeded `skill-author` skill itself carries `provenance: external`.
- `allbert-cli skills init` is preserved as the explicit CLI escape hatch. The natural-language path is the default.
- Recommended default scripting interpreter is Python (already on the v0.4 default `security.exec_allow` list). Operators can pick Bash or any other allowlisted interpreter.

### Scripting engine posture

- A `ScriptingEngine` trait lives in `crates/allbert-kernel/src/scripting/`. v0.12 ships exactly one implementation: `LuaEngine` using `mlua` with `lua54` and `vendored` features (no system Lua dep).
- Inputs and outputs are `serde_json::Value` only — there is no shared Rust reference, no pointer into session state, no direct memory service access, and no host tool-call bridge from inside a Lua script in v0.12. Any data a script needs must be passed in by the caller, and any state-changing action after a script returns must go through the existing tool surface outside the Lua VM.
- Lua scripts are declared in skill `SKILL.md` frontmatter under the existing `scripts:` section with interpreter string `lua` (per [ADR 0034](../adr/0034-skill-scripts-run-under-the-same-exec-policy-as-tools.md)). A skill that declares Lua scripts without the engine enabled loads but cannot run those scripts.
- Embedded-script invocations surface as synthetic tool names in `BeforeTool` / `AfterTool` events: `exec.lua:<skill-name>/<script-path>` (e.g. `exec.lua:summarizer/scripts/score.lua`). Future engines follow the same `exec.<engine-name>:<source-ref>` convention.

### Lua sandbox posture

- Two-step opt-in: Lua only runs when `security.exec_allow` includes `"lua"` AND `scripting.engine = "lua"` at the top level of config. Fresh profiles ship with both off.
- Base sandbox uses `Lua::sandbox(true)` plus an explicit stdlib allowlist. Default allow: `string`, `math`, `table`. Default deny: `io`, `os`, `package`, `require`, `debug`, and `coroutine`.
- Three independent caps, all enforced by the engine and reported in `BudgetUsed`:
  - `max_execution_ms` (default `1000`, ceiling `30000`) — `mlua` instruction-count hook plus wall-clock check.
  - `max_memory_kb` (default `65536`, ceiling `262144`) — `mlua::Lua::set_memory_limit`.
  - `max_output_bytes` (default `1048576`, ceiling `16777216`) — engine truncates serialized output past the cap.
- A cap violation terminates the script, returns `ScriptOutcome::CapExceeded { which, budget_used }`, emits an `AfterTool` event with `outcome: cap-exceeded`, and does not panic the host.
- Operator configs above the hard ceilings clamp at load time with a logged warning; per-invocation budgets above the ceiling are denied.
- `deny_stdlib` is the hard floor; entries cannot be re-allowed via `allow_stdlib`. Mirrors `security.exec_deny` precedence over `security.exec_allow`.
- `sandbox_mode = "strict"` is the only supported value in v0.12.

### Status-line items (deferred)

v0.11's handoff section noted that v0.12 *may* add status-line items for patch-approval counts, scripting budgets, and skill-author drafts. v0.12 explicitly defers all three. The existing `inbox` count surfaces pending patch approvals; scripting budgets surface through `AfterTool` hook metadata; skill-author draft counts are not an inbox surface because drafts live under `skills/incoming/`. Operators inspect drafts through the `skill-author` flow and the quarantine directory in v0.12. Additive status-line or CLI draft-list items can land in v0.13 or v0.14 without reshaping existing items.

## Public interfaces

### Config

Add:

```toml
[self_improvement]
source_checkout         = ""                          # auto-resolved when empty
worktree_root           = "~/.allbert/worktrees"
max_worktree_gb         = 10
install_mode            = "apply-to-current-branch"
keep_rejected_worktree  = false

[scripting]
engine                  = "none"                      # "none" or "lua" in v0.12

[security.scripting.lua]
sandbox_mode            = "strict"                    # only supported value
max_execution_ms        = 1000
max_memory_kb           = 65536
max_output_bytes        = 1048576
allow_stdlib            = ["string", "math", "table"]
deny_stdlib             = ["io", "os", "package", "require", "debug", "coroutine"]
```

`scripting.engine = "none"` is the fresh-profile default. Lua requires both `scripting.engine = "lua"` and `"lua"` on `security.exec_allow`.

### CLI commands

Add:

- `allbert-cli self-improvement config show` — print resolved source checkout, worktree root, and per-worktree disk usage.
- `allbert-cli self-improvement config set --source-checkout <path>` — persist the source-checkout pin.
- `allbert-cli self-improvement diff <aid>` — print the full unified diff for a `patch-approval`.
- `allbert-cli self-improvement install <aid>` — apply an accepted patch to the source checkout and emit the build-and-install hint.
- `allbert-cli self-improvement gc [--dry-run]` — list and reclaim stale worktrees.

`allbert-cli inbox` gains `patch-approval` rendering through the existing kind dispatch — no new subcommand. `allbert-cli skills init` (the v0.4 wizard) remains as the explicit CLI escape hatch with unchanged behavior. `allbert-cli skills list` adds a `Source` column.

### Daemon protocol

The new `patch-approval` inbox kind is additive over the existing `Inbox*` messages. PROTOCOL_VERSION does not bump: pre-v0.12 clients render unknown kinds with a generic fallback; v0.12 clients add a kind-specific renderer. No other protocol changes.

### Skill frontmatter

Add to the canonical AgentSkills frontmatter list (per [ADR 0032](../adr/0032-agentskills-folder-format-is-the-canonical-skill-shape.md)):

```yaml
provenance: external | local-path | git | self-authored
```

Default for skills without the field is `external`.

### Hook metadata

Embedded-script invocations record as `exec.<engine-name>:<source-ref>`:

```
exec.lua:summarizer/scripts/score.lua
```

The metadata payload mirrors process-exec metadata (interpreter, skill, path) plus `budget_used` (instructions, peak memory bytes, wall ms, output bytes). The convention is fixed: future engines follow the same shape.

## Implementation prerequisites

The milestones below depend on the following seams. Each is additive over v0.11.

### Kernel seams

- `crates/allbert-kernel/src/self_improvement/`: new module owning worktree creation, GC, source-checkout resolution, path-allowlist enforcement, Tier A invocation, and patch artifact emission.
- `crates/allbert-kernel/src/scripting/`: new module defining the `ScriptingEngine` trait, `ScriptingCapabilities`, `ScriptBudget`, `ScriptOutcome`, and the `LuaEngine` implementation.
- `Skill` model gains `provenance: SkillProvenance` (enum with `External`, `LocalPath`, `Git`, `SelfAuthored`); deserialization defaults to `External` when the field is absent.
- `SkillStore::install_from_incoming` preserves `provenance` on promotion and never silently relabels.
- `SkillStore::seed_first_party_skills` extended to seed `skill-author` alongside `memory-curator` on first daemon run after upgrade.
- `CreateSkillTool` adds a required `skip_quarantine: bool` argument; existing first-party seeding callers update to pass `true` explicitly.
- `Kernel::render_agents_markdown` (or equivalent) regenerates AGENTS.md routing summary on `skill-author` install, matching the existing `skill lifecycle event` regeneration pattern from v0.11.
- Inbox kind dispatch gains `patch-approval`. The renderer reads `validation`, `artifact_path`, `worktree_path`, `branch`, and `overall` from approval frontmatter.
- Hook metadata for embedded scripts mirrors process-exec metadata plus `budget_used`.

### Protocol seams

- `InboxItem` (or equivalent payload) recognizes `patch-approval` as an additional kind value. The protocol message shape is unchanged.
- No PROTOCOL_VERSION bump.

### Config seams

- `SelfImprovementConfig`: `source_checkout: Option<PathBuf>`, `worktree_root: PathBuf`, `max_worktree_gb: u32`, `install_mode: InstallMode` (`apply-to-current-branch` is the only variant in v0.12), `keep_rejected_worktree: bool`.
- `ScriptingConfig`: `engine: ScriptingEngineKind` (`None`, `Lua`).
- `SecurityScriptingLuaConfig`: cap and stdlib fields. Validation rejects unknown sandbox modes and caps below sane minima, clamps operator config above hard ceilings with a warning, denies per-invocation budgets above ceilings, and removes any `deny_stdlib` overlap from the effective allowlist with a warning (`deny_stdlib` wins).
- `Config::validate` refuses `scripting.engine = "lua"` when `"lua"` is missing from `security.exec_allow`.

### CLI seams

- `allbert-cli self-improvement` subcommand tree with `config`, `diff`, `install`, `gc` verbs.
- Inbox kind-aware rendering for `patch-approval`.
- `allbert-cli skills list` adds a `Source` column reading the new `provenance` field.

### Filesystem layout

- `~/.allbert/worktrees/<branch>/` — per-rebuild worktrees (configurable root).
- `~/.allbert/self-improvement/history.md` — append-only install log.
- `~/.allbert/skills/installed/skill-author/` — seeded on first daemon run after v0.12 upgrade.
- `~/.allbert/skills/incoming/<draft-name>/` — same quarantine path as external installs; reused for `skill-author` drafts.
- `sessions/<sid>/artifacts/<aid>/patch.diff` — diff artifact for `patch-approval`.

Everything above is strictly additive: existing config files load unchanged with v0.12 keys taking documented defaults; existing skills load with `provenance = external`; existing inbox kinds render unchanged.

### M8 — Lua sandbox policy

Scope:

- Apply `Lua::sandbox(true)` plus explicit stdlib allowlist; load `string`, `math`, `table` only by default. Deny `io`, `os`, `package`, `require`, `debug`, `coroutine`.
- Enforce `max_execution_ms` via `mlua`'s instruction-count hook plus wall-clock check.
- Enforce `max_memory_kb` via `mlua::Lua::set_memory_limit`.
- Enforce `max_output_bytes` by truncating serialized output past the cap and reporting `cap-exceeded`.
- Apply hard kernel ceilings (30000 ms, 256 MB, 16 MB) — operator configs above the ceiling clamp with a warning at load time; per-invocation requests above the ceiling are denied.
- Two-step opt-in enforced: refuse engine instantiation unless `scripting.engine = "lua"` AND `"lua"` on `security.exec_allow`.
- `deny_stdlib` is a hard floor; entries cannot be re-allowed via `allow_stdlib`.

- Sandbox-escape tests: scripts cannot read files, spawn processes, load modules, introspect host state, or use `coroutine` by default.

- Hard-ceiling clamp test: operator config above ceiling clamps with warning.

## End-user release plan

### Upgrade behavior

- Existing profiles take v0.12 defaults silently: `scripting.engine = "none"`, `lua` not on `security.exec_allow`, `self_improvement.*` keys take documented defaults, `keep_rejected_worktree = false`. No config rewrite is required on upgrade.
- `skill-author` is seeded into `~/.allbert/skills/installed/skill-author/` on first daemon run after upgrade. The seed is recorded in the install log; the operator can uninstall and replace with a custom variant.
- Skills already installed before v0.12 load as `provenance: external`. No migration runs; the field is additive on read.
- `patch-approval` is a new inbox kind. Pre-v0.12 inbox markdown is unaffected; older clients render unknown kinds with a generic fallback. There is no PROTOCOL_VERSION bump.
- No memory or session migration. No bootstrap reload. No status-line item changes.

### Operator messaging

Release notes must explain:

- "Self-modification is opt-in and source-checkout-required. `cargo install` and binary-drop users still get skill authoring and Lua scripting."
- "Allbert never swaps its own binary. `self-improvement install` applies a patch to your source checkout; you run `cargo install` and `daemon restart` on your own time."
- "Self-authored skills are quarantined like any other skill source. Trust posture is unchanged."
- "Lua is off by default. Enabling it requires both `scripting.engine = \"lua\"` and `\"lua\"` on `security.exec_allow`."
- "`skill-author` is the natural-language path for creating new skills. `allbert-cli skills init` remains as the CLI escape hatch."

### Acceptance checklist

- Fresh setup completes without a source checkout; `rust-rebuild` is unavailable but the limitation is documented.
- A source-checkout-pinned setup can produce a fixture patch, run Tier A in a worktree, and route the result through `patch-approval`.
- `inbox accept` on a `patch-approval` records approval but does not install; the renderer points at `self-improvement install`.
- `inbox reject` deletes the worktree by default; `keep_rejected_worktree = true` preserves it.
- `self-improvement install <aid>` applies the patch and emits the build-and-install hint; install log records the event.
- `skill-author` is seeded into installed skills on first run; intent routing dispatches "make me a skill that..." to it; drafts persist in `incoming/` across sessions; final submission routes through install preview + confirm.
- `allbert-cli skills list` shows the `Source` column with `external | local-path | git | self-authored`.
- `allbert-cli skills init` wizard still works.
- Lua engine refuses to instantiate when either opt-in gate is missing; with both enabled, scripts run under sandbox and caps; sandbox-escape tests fail to escape.
- Embedded-script invocations show as `exec.lua:<skill>/<script>` in hook metadata with `budget_used`.
- Lua scripts cannot call host tools directly in v0.12; callers pass JSON input in and handle any follow-up tool calls outside the Lua VM.
- Operator docs (`self-improvement.md`, `skill-authoring.md`, `scripting.md`) and upgrade notes are end-user ready.
- Tier A validation is green.

## Handoff to v0.13 and v0.14

v0.13 and v0.14 consume v0.12 surfaces as public API. Changes after v0.12 ships must treat them as such. These surfaces participate in the cross-release self-change envelope from [ADR 0080](../adr/0080-self-change-artifacts-share-approval-provenance-and-rollback-envelope.md):

- **Sibling-worktree infrastructure.** v0.14's self-diagnose skill produces code-shaped remediation through the same worktree isolation + Tier A validation path as `rust-rebuild`. The worktree management API (creation, cleanup, GC, size caps) must stay stable across v0.12 so v0.14 can reuse it without forking a second isolation mechanism. v0.13 adapters intentionally do *not* reuse the worktree (adapters are not source code; they live under `~/.allbert/adapters/`).
- **`patch-approval` inbox kind.** v0.14's code-shaped remediation routes through the same inbox kind, not a new one. The approval state machine (accept/reject/install) and rendering contract (summary, file stats, validation status) must stay additive-only; v0.14 may add remediation-specific fields but must not reshape existing ones.
- **`skill-author` as a sub-agent-callable authoring surface.** v0.14's self-diagnose may draft diagnostic skills when remediation shape is skill-shaped rather than patch-shaped. The authoring flow (draft → quarantine → install preview) must be invokable as a sub-agent by other skills, not only as a direct operator conversation. Diagnostic skills produced this way carry `provenance: self-diagnosed`.
- **Provenance frontmatter pattern.** v0.12 introduces `provenance: self-authored` for skill-author output. The pattern extends predictably: v0.13 adapter metadata carries `provenance: self-trained`; v0.14 diagnostic skills carry `provenance: self-diagnosed`. The frontmatter slot is a fixed-size enum; new provenance kinds land as additive values, not reshaping of the frontmatter model.
- **`ScriptingEngine` (Lua) vs shell exec.** v0.12's Lua is in-process, sandboxed, JSON-IO. v0.14 introduces a `unix-pipe` tool shape that runs shell composition under exec policy. The two are complementary, not competing: Lua for safe in-process transforms, `unix-pipe` for invoking local utilities that already exist on the operator's system. v0.12 must not close design space that v0.14's shell composition needs (normalised tool surface from ADR 0052, exec policy gates).

v0.13 specifically does *not* depend on v0.12's rust-rebuild or skill-author surfaces; its only v0.12 inheritance is the review-first pattern it mirrors in `adapter-approval`. v0.14 depends on v0.12 substantively.
