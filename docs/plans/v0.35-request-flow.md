## Pre-Flight

Use a fresh checkout or disposable branch. Generated code is source-tree work
and should be reviewed before being compiled into the host.

## Flow 1: Generate A Plugin

Developer runs:

```sh
mix allbert.gen.plugin MyPlugin
```

Expected behavior:

1. A plugin directory is created under `./plugins/my_plugin/`.
2. The scaffold includes manifest, README, plugin module, optional skill root,
   and validation docs.
3. Generated skill material is inert until explicitly trusted/enabled.
4. The generator does not modify compile paths, grant permissions, publish
   anything, add routes, or load runtime code automatically.
5. `mix allbert.validate_plugin MyPlugin` passes.

Failure behavior:

- Existing directories are not overwritten without explicit operator choice.
- Invalid names fail with safe suggestions and no partial trust changes.

## Flow 2: Generate An App Plugin

Developer runs:

```sh
mix allbert.gen.app MyApp
```

Expected behavior:

1. The scaffold includes plugin module, app module, supervision wiring, sample
   `AllbertAssist.Action` action with `capability/0`, sample `SKILL.md`,
   panel SurfaceProvider stub, optional page-surface notes, settings
   `schema_fragment/0` stub, memory namespace stub, optional objective stub,
   Ecto domain stub, canvas wiring stub, theming/layout docs, and dynamic-draft
   review notes.
2. The app validates immediately through `mix allbert.validate_app MyApp`.
3. Generated docs explain the explicit compile-path/project integration step
   needed before code-bearing source-tree plugins can run.
4. The memory namespace stub is declaration-only and writes nothing.
5. The canvas wiring stub uses the v0.30-proven path but emits nothing until
   developer code opts in through registered actions.
6. The panel stub targets a v0.32 known zone and uses unified catalog atoms
   only.

## Flow 3: Inspect Generated Safety Posture

Developer reads generated README/validation docs.

Expected behavior:

1. Docs explain what is inert and what requires explicit operator/developer
   action.
2. No generated action bypasses `AllbertAssist.Actions.Registry`,
   `Actions.Runner.run/3`, Security Central, confirmations, or traces.
3. No generated plugin claims trust, permissions, external publishing,
   core-node dynamic loading, provider credentials, route authority, or theme
   authority.

## Flow 4: Validation Fails Safely

Developer intentionally breaks a generated manifest, app id, panel surface,
zone, settings schema, memory namespace, or canvas stub.

Expected behavior:

1. Validation reports precise diagnostics.
2. The host runtime keeps running.
3. No broken app is registered as active or trusted.
4. No broken panel renders in `/workspace`.

## Smoke Checklist

- Generated plugin validates.
- Generated app validates.
- Generated skill remains inert.
- Generated action routes through registry/runner/security boundaries.
- Generated panel Surface data validates and cannot invent component ids or
  zones.
- Generated namespace, settings, theme, and canvas stubs are declaration-only.
- Milestone closeout runs generator tests, validation-task tests, focused docs
  checks, and `mix precommit`.
