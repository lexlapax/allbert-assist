# Allbert v0.35 Request Flow - Dynamic Plugin/App Generation And Sandboxed Module Loading

Implementation details live in `docs/plans/v0.35-plan.md`.

v0.35 lets Allbert try a generated local plugin/app draft when a request needs
a missing capability. The draft remains untrusted and sandbox-only.

## Operator-Visible Flow

1. Operator asks for work that current actions/apps cannot do.
2. Intent/objective runtime records a capability gap.
3. Allbert proposes an acquisition option: generate a local plugin/app draft.
4. Operator explicitly confirms draft generation.
5. Allbert writes inert source under `<ALLBERT_HOME>/plugins/<slug>`.
6. Operator confirms sandbox compile/trial.
7. Sandbox runtime compiles and runs bounded trial scenarios out of node.
8. Allbert returns redacted diagnostics and usefulness summary.
9. Operator discards the draft or marks it as a promotion candidate.

No generated module is loaded into the core Allbert node at any point.

## Flow 1: Capability Gap Proposal

| Step | Expected behavior |
|---|---|
| Missing capability detected | Objective step records the missing capability and why existing actions cannot satisfy it |
| Acquisition option proposed | Generated local draft is presented as one option, not the default authority |
| Advisory output present | Advisory/model output is explanatory only |
| Operator declines | Objective records decline and no files are written |

Failure behavior:

- If the request asks for unsafe, illegal, destructive, or unsupported
  capability, no generation option is offered.
- If the operator has disabled dynamic drafts, the response explains the
  disabled state and points to Settings Central.

## Flow 2: Draft Generation

| Step | Expected behavior |
|---|---|
| Confirmation approved | Registered action writes inert draft source |
| Draft path | `<ALLBERT_HOME>/plugins/<slug>` only |
| Draft status | `:draft` |
| Metadata | objective id, capability gap, provenance, generated file manifest, and redacted prompt summary |

Failure behavior:

- Path traversal, duplicate slug conflict, oversized output, forbidden file
  type, or forbidden construct fails closed.
- Generation never changes compile paths, trust, permissions, routes, settings
  authority, skills, or app activation.

## Flow 3: Sandbox Compile

| Step | Expected behavior |
|---|---|
| Compile requested | Operator confirmation and Security Central check are required |
| Sandbox started | Disposable out-of-node backend receives only allowed draft files and bounded config |
| Compile succeeds | Draft status becomes `:sandbox_compiled`; diagnostics are stored redacted |
| Compile fails | Draft remains untrusted; diagnostics are redacted and bounded |

Failure behavior:

- Package-manager, new dependency, migration, NIF, port, secret, Repo
  credential, unrestricted network, or core-node load attempt is denied.
- Timeout or sandbox startup failure cleans up temporary state.

## Flow 4: Sandbox Trial

| Step | Expected behavior |
|---|---|
| Trial requested | Operator confirmation and Security Central check are required |
| Scenarios run | Sandbox executes bounded scenarios from the trial manifest |
| Results returned | Redacted structured report is attached to the objective |
| Trial succeeds | Draft status becomes `:sandbox_trialed` |

Failure behavior:

- Output over limit is truncated and redacted.
- Sandbox denial is reported as a policy failure, not a runtime crash.
- Trial success does not promote or enable the draft.

## Flow 5: Discard Or Promotion Candidate

| Operator action | Expected behavior |
|---|---|
| Discard | Draft is deleted or tombstoned according to the implementation safety choice; audit records the decision |
| Mark promotion candidate | Review intent is recorded; no code is loaded, trusted, compiled into project, or enabled |

Failure behavior:

- Promotion-candidate state cannot grant permissions, routes, settings writes,
  child supervision, skills, or action execution.
- Future reviewed integration is out of scope for v0.35 unless explicitly
  added to v0.36.

## Security Eval Catalog

Implementation should add eval rows for:

- core-node dynamic compile/eval/load attempt;
- sandbox secret access attempt;
- package-manager/dependency injection;
- migration injection;
- NIF/port/native-code attempt;
- unrestricted network attempt;
- path traversal outside the draft root;
- promotion without operator approval;
- advisory/model output attempting to authorize generation or trial;
- output exfiltration over size/redaction limits.

## Release Verification

- Use a disposable `ALLBERT_HOME`.
- Generate a harmless local draft, compile it in sandbox, run a bounded trial,
  inspect redacted results, then discard it.
- Verify the core node never loads generated modules and no compile path,
  permission, route, app activation, skill enablement, or settings authority is
  changed.
- Run the full code warning gate, security evals, and `mix precommit`.
