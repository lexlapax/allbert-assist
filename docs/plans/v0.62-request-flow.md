## Main User/Operator Flows

- A technical prosumer installs Allbert from Homebrew or a curl installer on a
  host with no Elixir/OTP toolchain.
- Operator runs `allbert --help` and sees grouped command help.
- Operator runs `allbert serve` and watches runtime, web, and channel health go
  green.
- Operator drives day-to-day work through `allbert ask|chat|tui|admin ...` and
  the mix-free TUI console without raw `mix`.
- Operator stores a provider credential in the OS vault and migrates an existing
  credential through an explicit redacted step.
- Developer/CI continues to use `mix` tasks from a checkout.

## Data Flow

The packaged binary boots the same supervision tree as the checkout. The grouped
command taxonomy implements the v0.60 entry-point / CLI UX design (ADR 0077); it
is not invented here. Each operator command resolves to the same registered
action or read-report interface the equivalent Mix task used, routing through
`Actions.Runner.run/3` and the ADR 0073 spine. If the v0.60 First-Model-Path
decision (ADR 0078) chose the assisted local model path, the packaged binary also
bundles/manages a local model runtime (Ollama); otherwise it stays
provider-key-only. `allbert serve` supervises runtime, Phoenix web workspace, and
channel adapters under one managed process and reports a bounded health snapshot.
Credentials resolve from Settings Central references to OS-vault values injected
at runtime; raw values never enter logs, release evidence, or inspection output.

## Authority Invariants

No new capability is introduced. Packaging changes installation and invocation,
not authority. The binary, dispatcher, daemon, TUI console, and vault remain
behind Security Central, confirmations, action boundaries, and surface policy.
The dispatcher exposes no operator command that cannot be traced to an existing
reviewed Mix task/action path. The TUI convergence adds read-only internal
actions only.

## Handles / CLI Surfaces

## Smoke Checklist

- Install the packaged binary via Homebrew and curl on a host with no Elixir/OTP.
- Run grouped `allbert --help` and a representative command from each operator
  group.
- Confirm developer/CI commands stay `mix`-only.
- Run `allbert serve`, install/uninstall the service, and verify the health
  check.
- Drive admin-inspection reads through the TUI console and confirm they are
  read-only internal actions.
- Store and migrate credentials through the OS vault; scan logs/output/evidence
  for raw secrets.
- Confirm docs and gates are clean.

## Operator Validation

Status: planned. Commands are illustrative until the milestone lands; verify and
replace with observed output at closeout.

```sh
# Install path on a host without Elixir/OTP.
brew install allbert            # or: curl -fsSL https://.../install.sh | sh
allbert --help
allbert serve --health
allbert admin status
mix allbert.test release.v062   # maintainer checkout gate
```

Expected behavior:

- the packaged binary installs and runs with no Elixir/OTP toolchain;
- grouped operator commands route through the spine and add no new authority;
- `allbert serve` starts the managed runtime/web/channels process and health
  goes green;
- credentials live in the OS vault with no raw secret in logs, output, or
  release evidence;
- `release.v062` is deterministic and passes.
