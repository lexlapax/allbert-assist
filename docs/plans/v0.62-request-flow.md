## Main User/Operator Flows

- A technical prosumer installs Allbert from Homebrew or a curl installer on a
  host with no Elixir/OTP toolchain; the installer verifies checksums and
  writes an uninstall manifest.
- First invocation of bare `allbert` runs the v0.60-designed first-run
  detection; the guided Ollama install + curated model pull execute only
  behind explicit per-step confirmations, or degrade to BYOK.
- Operator runs `allbert --help` and sees grouped command help.
- Operator runs `allbert serve` and watches runtime, web, and channel health go
  green; optionally installs the per-user OS service.
- Operator drives day-to-day work through `allbert ask|chat|tui|admin ...` and
  the mix-free TUI console without raw `mix`; when the daemon is running,
  commands attach to it (no second runtime).
- Operator stores a provider credential through the three-tier vault model and
  migrates existing credentials through an explicit redacted step.
- Developer/CI continues to use `mix` tasks from a checkout, under the
  documented coexistence policy (separate `ALLBERT_HOME`/port).

## Data Flow

The packaged binary boots the same supervision tree as the checkout. The grouped
command taxonomy implements the v0.60 entry-point / CLI UX design
(`docs/design/entry-point-cli-ux.md`, ADR 0077); it is not invented here. Each
operator command resolves to the same registered action or read-report
interface the equivalent Mix task wrapped, routing through
`Actions.Runner.run/3` and the ADR 0073 spine — attach-first against a running
daemon, embedded with single-writer discipline otherwise (plan Locked
Decision 5). Per ADR 0078 (Accepted: assisted-local default + BYOK fallback),
first run detects a local Ollama install, guides installation through the
S4-ratified supported upstream path when absent, and manages the curated model
pull — each
  network/execution step behind an explicit operator confirmation with trace
recording; it does not bundle the model runtime; below the hardware floor or on
decline it degrades to BYOK. v0.62 uses `Req` for loopback-only Ollama HTTP and
exact argv for installer execution. `allbert serve` supervises runtime, Phoenix web
workspace, and channel adapters under one managed per-user process and reports
a bounded health snapshot. Credentials resolve from Settings Central references
through the three-tier backend (OS vault → encrypted store → env); raw values
never enter logs, release evidence, or inspection output.

## Authority Invariants

No new capability is introduced. Packaging changes installation and invocation,
not authority. The binary, dispatcher, daemon, TUI console, and vault remain
behind Security Central, confirmations, action boundaries, and surface policy.
The dispatcher exposes no operator command that cannot be traced to a row of
the M3 disposition table (an existing reviewed action/read path). The TUI
convergence adds read-only internal actions only. The release's four new
network touches are enumerated in ADR 0076's Distribution Trust section; the
Ollama installer fetch and model pull are confirmation-gated with trace
records. The binary carries no telemetry, phone-home, or auto-update check.
Release mode must not weaken enforcement (the Mix-probe `production?()` fix is
gate-asserted).

M4 authority is explicit: local Ollama detection is read-only (three-way
probe, localhost only); installer
metadata and model manifests use the reviewed `:external_network` posture
(`Actions.Intent.ExternalNetworkRequest` over `External.RequestSpec`/
`HttpPolicy`, hosts admitted via existing `external_services.*` values);
**all installer execution — Homebrew formula or official script — uses the
existing `:command_execute` posture with exact argv allowlists and resource
refs** (operator decision 2026-07-06; `:package_install` is not applicable —
its `InstallSpec` is npm/pip-only and rejects global installs by design); the
curated model pull is `:external_network` to the local Ollama API
(`/api/pull`, loopback-only `Req` with `stream: false` for a bounded JSON
summary; CLI fallback under the same `:command_execute` allowlist). If the chosen path cannot
fit an existing permission/resource class, S4 records a blocker or invokes the
documented BYOK-primary contingency; implementation does not add a new
permission atom. M5's service-lifecycle commands and M7's migrate-secrets
command are named registered internal actions inside the
`packaging-no-authority-change-001` allowance.

- `mix allbert.test release.v062` (checkout gate) + the artifact smoke harness
  (CI, executes the built binary).
