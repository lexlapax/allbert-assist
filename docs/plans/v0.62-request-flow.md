## Main User/Operator Flows

- A technical prosumer installs Allbert from Homebrew or a curl installer on a
  host with no Elixir/OTP toolchain; the installer verifies checksums and
  writes an uninstall manifest.
- First invocation of bare `allbert` runs the v0.60-designed first-run
  detection; the guided Ollama install + curated model pull execute only
  behind explicit per-step confirmations, or degrade to BYOK.
- Operator runs `allbert --help` and sees grouped command help.
- Operator runs `allbert serve` and watches runtime, web, and channel health go
  green; optionally installs the per-user OS service.
- Operator drives day-to-day work through `allbert ask|chat|tui|admin ...` and
  the mix-free TUI console without raw `mix`; when the daemon is running,
  commands attach to it (no second runtime).
- Operator stores a provider credential through the three-tier vault model and
  migrates existing credentials through an explicit redacted step.
- Developer/CI continues to use `mix` tasks from a checkout, under the
  documented coexistence policy (separate `ALLBERT_HOME`/port).

## Data Flow

The packaged binary boots the same supervision tree as the checkout. The grouped
command taxonomy implements the v0.60 entry-point / CLI UX design
(`docs/design/entry-point-cli-ux.md`, ADR 0077); it is not invented here. Each
operator command resolves to the same registered action or read-report
interface the equivalent Mix task wrapped, routing through
`Actions.Runner.run/3` and the ADR 0073 spine — attach-first against a running
daemon, embedded with single-writer discipline otherwise (plan Locked
Decision 5). Per ADR 0078 (Accepted: assisted-local default + BYOK fallback),
first run detects a local Ollama install, guides installation through the
S4-ratified supported upstream path when absent, and manages the curated model
pull — each
network/execution step behind an explicit operator confirmation with trace
recording; it does not bundle the model runtime; below the hardware floor or on
decline it degrades to BYOK. `allbert serve` supervises runtime, Phoenix web
workspace, and channel adapters under one managed per-user process and reports
a bounded health snapshot. Credentials resolve from Settings Central references
through the three-tier backend (OS vault → encrypted store → env); raw values
never enter logs, release evidence, or inspection output.

## Authority Invariants

No new capability is introduced. Packaging changes installation and invocation,
not authority. The binary, dispatcher, daemon, TUI console, and vault remain
behind Security Central, confirmations, action boundaries, and surface policy.
The dispatcher exposes no operator command that cannot be traced to a row of
the M3 disposition table (an existing reviewed action/read path). The TUI
convergence adds read-only internal actions only. The release's four new
network touches are enumerated in ADR 0076's Distribution Trust section; the
Ollama installer fetch and model pull are confirmation-gated with trace
records. The binary carries no telemetry, phone-home, or auto-update check.
Release mode must not weaken enforcement (the Mix-probe `production?()` fix is
gate-asserted).

M4 authority is explicit: local Ollama detection is read-only; installer
metadata and model manifests use the reviewed `:external_network` posture;
Homebrew execution uses the existing `:package_install` profile; non-Homebrew
installer execution, if selected at S4, uses the existing `:command_execute`
posture with exact argv allowlists and resource refs. If the chosen path cannot
fit an existing permission/resource class, S4 records a blocker or invokes the
documented BYOK-primary contingency; implementation does not add a new
permission atom.

## Handles / CLI Surfaces

- Packaged `allbert` binary (one artifact per target triple).
- Homebrew tap (formula or cask, exact package type chosen at S3) and curl
  installer, checksum-verified, with manifest-driven uninstall.
- Bare `allbert` (first-run/resume dispatcher) and
  `allbert <group> <command>`: `ask | chat | tui | serve | admin <area> | gen`.
- `allbert serve`, per-user service install/uninstall, `/health` +
  `allbert admin status`.
- Mix-free TUI operator console (completed ADR 0070 read set).
- Three-tier secret-vault storage and the redacted migration command.
- `mix allbert.test release.v062` (checkout gate) + the artifact smoke harness
  (CI, executes the built binary).

## Operator / Reviewer Validation

### S0 — Preconditions
- On `main` at or after the v0.62 readiness-pass triad; `v0.61.1` tag present.
- For install S-steps: access to a fresh (or reset) macOS machine/VM and a
  Linux machine/VM with **no Elixir/OTP toolchain and no `~/.allbert`**; for
  the headless vault cell, a Linux host without a desktop keyring.

### S1 — Locked Decisions Sign-off (gates all build work)
- Review the plan's Locked Decisions table: rows 5/8/12/14 already signed
  2026-07-05 (CLI attach-first; Ollama own-milestone Allbert-executes; vault
  three-tier; signing deferred/documented). Ratify or amend the remaining
  rows (spike candidates, artifact matrix proposal, install layout, service
  scope, upgrade/coexistence scope, tag row). Spike-dependent rows 2/3/4 are
  provisionally ratified and finalized at S2.
- Confirm that M0.1 is only the documented v0.61b post-audit web-preflight
  exception, not reopened presentation scope; record whether `macos-x64` is
  freeze-blocking or non-blocking compatibility before the matrix is treated as
  settled.
- **Record:** `S1 sign-off: <verdict>, <date>` in the plan's Locked Decisions
  section.

### S2 — Spike-Report Ratification (gates M1–M8)
- Read the M0 spike report (appended to the plan as the M0 as-built record):
  all eight proofs, the chosen mechanism (Burrito 1.5.x vs hand-wrapped
  release), the artifact matrix, and the ERTS-provenance recommendation.
- The artifact matrix must name the Tier-1 triples, the `macos-x64` decision,
  the `linux-musl` yes/no decision, the glibc floor, and whether ERTS comes
  from pinned packaging-tool builds or project-owned CI builds.
- A failed proof means the mechanism is re-chosen and the spike re-run — M1
  does not start on a failed spike.
- **Record:** mechanism + matrix + provenance in Locked Decisions 2/3/4 with
  the sign-off date.

### S3 — Install-Layout & Distribution-Trust Spec Sign-off (gates M2)
- Review the in-plan spec: file map per install path, Allbert Home layout
  (Locked Decision 10), checksum scheme, the four enumerated network touches,
  the no-telemetry statement, tap/artifact-hosting ownership (the real domain/
  repo replaces the placeholder), formula-vs-cask, uninstall/`--purge`
  semantics, backup-before-migrate behavior.
- Decide the exact Homebrew package shape and commands (formula vs cask),
  whether Homebrew's service block is used or Allbert owns service management
  entirely through `allbert serve`, and the backup path/retention/restore
  expectation before migrations can run from a packaged artifact.
- **Record:** sign-off line in the spec section.

### S4 — CLI Taxonomy Delta + Curated Model Ratification (gates M3/M4)
- Review the M3 disposition table (all 55 tasks: command / mix-only /
  retired) against `docs/design/entry-point-cli-ux.md`; ratify the deltas
  (bare-`allbert` ownership, `allbert chat` fallback yes/no, `home
  export/import` placement, the ~18 previously unmapped tasks).
- Ratify the curated default model + hardware floor (Locked Decision 9), or
  invoke ADR 0078's documented BYOK-primary contingency.
- Ratify the supported upstream Ollama install path per Tier-1 OS (Homebrew
  formula vs Ollama script/tarball), the M4 authority-contract row for each
  effectful step, and the replacement BYOK acceptance rows if the contingency
  is invoked.
- **Record:** sign-off line on the disposition table + Locked Decisions 7/9.

### S5 — Vault UX + Platform Fallback Matrix Sign-off (gates M7)
- Review the in-plan vault section: per-platform backend matrix including the
  headless-Linux cell, tier-resolution behavior (explicit notice, never
  silent), key naming/scoping, migration UX, env-override name, export/import
  seam.
- Confirm the allowed `ALLBERT_VAULT_BACKEND` values, master-key handling for
  tier 2, and the exact headless/service-session behavior before the packaged
  daemon path depends on vault availability.
- **Record:** sign-off line + Settings-Keys ratification.

### S6 — Contained Verification (after M0.1, M3, M4, M5, M6, M7)
- **M0.1:** web suite green; registry diff exactly the one new internal
  action; a11y batch verified in the browser.
- **M3:** grouped `--help` matches the design; a representative command per
  group routes through the spine (trace visible); with the daemon running, a
  second `allbert` command attaches (no second BEAM — verify via process
  list); with no daemon, embedded mode boots and a concurrent second writer
  fails with guidance.
- **M4:** on a fresh machine below/above the floor: detection, the
  confirmation prompts (decline and accept paths), trace records for install
  + pull, BYOK degrade, and authority evidence matching the plan's M4
  permission/resource table with no raw secret or unbounded command output.
- **M5:** `allbert serve` health green from `/health` and `admin status`;
  service install/start/crash-recovery/log-inspection/stop/uninstall on both
  Tier-1 OSes; daemon boots from a login session with no shell env.
- **M6:** each enumerated TUI read from the packaged binary; reads absent
  from intent-router candidates.
- **M7:** vault round-trip per OS (Keychain; secret-tool with a desktop
  keyring; the headless cell resolving to tier 2 with the explicit notice);
  migration rehearsal from seeded plaintext/encrypted credentials; redaction
  scan of logs/traces/evidence afterward.

### S7 — Install/Upgrade/Uninstall Matrices (per Tier-1 OS)
- On macOS and Linux, each × {brew, curl}: fresh install on a no-toolchain
  machine → first-run → `serve` → health green; upgrade from the previous
  artifact (backup taken, migrations logged, SW/assets fresh); uninstall
  (Home preserved) and `--purge` (Home removed on explicit ask). Include
  `macos-x64` only if S2 promoted it to freeze-blocking.
- Coexistence: dev checkout + packaged daemon on one machine under the
  documented separation policy; the single-writer guard fires when misused.
- WSL2 smoke: the Linux artifact installs and serves under WSL2 (Tier-2
  criterion).

### S8 — Final Validation & Release Marking

- ADR 0076 flipped to Accepted (named here) with Distribution Trust; ADR 0070
  marked converged; docs deliverables spot-checked; version reads 0.62.0;
  CHANGELOG present.
- **Record:** approve → apply tag `v0.62.0` (operator-held), or reopen with
