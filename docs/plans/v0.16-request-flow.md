# v0.16 Request Flow: Additional Channels

Status: planned.

This document names the implementation flow for v0.16's first additional
channel: Telegram. It is intentionally concrete so the implementation can be
tested without drifting into a generic provider framework.

## Startup Flow

1. Application supervision starts `AllbertAssist.Channels.Supervisor`.
2. The supervisor reads Settings Central channel settings.
3. If `channels.telegram.enabled` is `false`, the supervisor records disabled
   diagnostics and starts no polling work.
4. If Telegram is enabled, the adapter validates:
   - `bot_token_ref` resolves to an available secret
   - `identity_map` is valid
   - `allowed_chat_ids` and group-chat settings are valid
   - polling intervals are within bounds
5. Invalid configuration leaves the adapter idle with diagnostics. It must not
   crash-loop.
6. Valid configuration starts one Telegram polling process under supervision.

## Settings And Identity Setup Flow

Operator command:

```text
mix allbert.channels telegram set-token TOKEN
```

Expected behavior:

- Writes the token through Settings Secrets at
  `secret://channels/telegram/bot_token`.
- Leaves `channels.telegram.bot_token_ref` pointing at that secret ref.
- Redacts the token in all output, traces, logs, and tests.

Operator command:

```text
mix allbert.channels telegram map --external-user 123 --user alice
```

Expected behavior:

- Adds or updates one enabled identity map entry.
- Requires both values to be non-empty strings.
- Does not create accounts, roles, auth users, or foreign keys.
- Does not allow duplicate `external_user_id` entries.

Unmapping removes or disables the entry. After unmapping, the external identity
cannot submit runtime input.

## Telegram Poll Flow

1. The adapter calls the bounded Telegram client `getUpdates`.
2. Each provider update is parsed into one of:
   - supported text message
   - supported callback query
   - unsupported media/document/voice/sticker/other update
   - malformed update
3. The adapter derives `external_event_id` from the Telegram update id.
4. It attempts to insert a `channel_events` row with status `"received"`.
5. If the unique `{channel, external_event_id}` index rejects the insert, the
   update has already been processed or rejected and is skipped.
6. Supported text messages continue to the text message flow.
7. Supported callback queries continue to the callback flow.
8. Unsupported or malformed updates are marked `"rejected"` with a bounded,
   redacted reason and never call the runtime.
9. After successful handling, the adapter advances provider offset so a restart
   does not replay already recorded updates.

## Text Message Flow

1. Validate text byte size against `channels.telegram.max_text_bytes`.
2. Reject group chats unless:
   - `channels.telegram.allow_group_chats` is true
   - the chat id is listed in `channels.telegram.allowed_chat_ids`
3. Resolve Telegram `external_user_id` through
   `channels.telegram.identity_map`.
4. If unmapped or disabled:
   - mark the event `"rejected"`
   - optionally send a short "not connected" response
   - do not call `AllbertAssist.Runtime.submit_user_input/1`
   - do not create a thread or message
5. Derive a deterministic bounded `session_id` from channel, external user id,
   and chat id.
6. Detect supported channel command prefixes:
   - `/new <prompt>` sets `new_thread: true` and uses `<prompt>` as input
   - plain text uses normal v0.12 default thread selection
7. Build a runtime request:

```elixir
%{
  text: prompt_text,
  channel: "telegram",
  user_id: resolved_user_id,
  operator_id: resolved_user_id,
  session_id: session_id,
  new_thread: new_thread?,
  metadata: %{
    channel: "telegram",
    provider: "telegram_bot_api",
    external_event_id: update_id,
    external_user_id: telegram_user_id,
    external_chat_id: chat_id,
    external_message_id: message_id
  }
}
```

8. Submit through `AllbertAssist.Runtime.submit_user_input/1`.
9. Store returned `thread_id`, `trace_id`, and signal id on the channel event.
10. Render the runtime response back to Telegram.
11. Mark the event `"processed"` or `"failed"`.

## Thread Semantics

- v0.16 does not create channel-specific thread tables.
- If no `thread_id` is passed, runtime uses v0.12 behavior: the user's most
  recently updated general thread, or a new one if none exists.
- `/new` always creates a new general thread for the resolved user.
- Channel event records reference the thread; they are not the conversation
  history source of truth.
- Conversation turns are not promoted to markdown memory.

## Response Rendering Flow

Normal runtime responses:

1. Take the runtime response text and metadata.
2. Redact secrets and sensitive values.
3. Split or truncate to Telegram-safe lengths.
4. Send via the Telegram client `sendMessage`.
5. Record outbound delivery status without raw token or raw provider response
   dumps.

Approval Handoff responses:

1. Detect the v0.11 Approval Handoff result.
2. Render the handoff lines:
   - action title
   - reason/risk
   - resource URI or target summary
   - operation class
   - access mode
   - scope choices
   - confirmation id
3. Render provider buttons only when callback data can safely fit the compact
   callback format.
4. When buttons cannot be safely rendered, include typed commands/instructions
   instead.
5. The rendered message must make it clear that no hidden execution happened.

## Confirmation Callback Flow

Callback data examples:

```text
allbert:v1:approve:<confirmation_id>
allbert:v1:deny:<confirmation_id>
allbert:v1:show:<confirmation_id>
```

Flow:

1. Parse callback data. Reject unknown versions, unknown actions, malformed
   confirmation ids, and overlong payloads.
2. Resolve the callback sender through the identity map.
3. If unmapped, acknowledge with a rejection message and do not mutate the
   confirmation.
4. Insert a `channel_events` row with direction `"callback"`.
5. Execute the matching registered confirmation action through
   `AllbertAssist.Actions.Runner.run/3`.
6. Pass origin/resolver metadata:
   - channel `"telegram"`
   - provider `"telegram_bot_api"`
   - external user id
   - external chat id
   - callback query id
   - mapped `user_id`
   - `session_id`
7. Render the resulting confirmation state back to Telegram.
8. Do not embed resource targets, commands, URLs, credentials, or remembered
   grant scopes directly in callback data.

## Failure Modes

Disabled channel:

- `show telegram` reports disabled.
- Polling does not start.
- `simulate` remains available as a local synthetic update path. It does not
  contact Telegram, does not require a bot token, and does not start polling,
  but it still requires an explicit identity mapping.

Missing token:

- Adapter stays idle with diagnostics.
- CLI output says token is missing without printing any secret ref contents.

Unmapped external user:

- Event is recorded as rejected.
- Runtime is not called.
- No thread/message is created.

Oversized text:

- Event is rejected before runtime.
- The response says the message is too large.

Unsupported media/document:

- Event is rejected.
- No file download or document extraction occurs.

Provider error:

- Adapter marks affected event failed when applicable.
- Poller backs off and remains supervised.
- Token and raw response bodies stay redacted.

Duplicate update:

- Unique insert detects the duplicate.
- Runtime is not called again.
- Existing event is left as the source of truth.

Already-resolved confirmation:

- Callback renders current confirmation state.
- It does not reopen or re-run the action.

## CLI Flows

```text
mix allbert.channels list
```

Shows configured channel ids, enabled/disabled status, diagnostics, and last
event summary.

```text
mix allbert.channels show telegram
```

Shows Telegram settings summary, identity-map count, allowed chat count,
polling status, and diagnostics. Secrets and raw identity details are redacted
or summarized unless the command explicitly names a safe operator-inspection
mode.

```text
mix allbert.channels telegram simulate --external-user 123 --chat 456 "hello"
```

Creates a synthetic Telegram text update and runs the normal channel flow. This
is the primary manual acceptance path and must use a temporary Allbert home in
tests.

```text
mix allbert.channels telegram poll-once
```

Runs one bounded provider poll for manual verification. It does not leave a
background loop running.

## Signals And Trace Metadata

Emit channel lifecycle signals using the existing signal helper:

- `allbert.channel.update_received`
- `allbert.channel.message_rejected`
- `allbert.channel.runtime_submitted`
- `allbert.channel.response_sent`
- `allbert.channel.delivery_failed`
- `allbert.channel.callback_received`

Signal metadata includes:

- `channel`
- `provider`
- `external_event_id`
- `external_user_id`
- `external_chat_id`
- `user_id`
- `session_id`
- `thread_id`
- `trace_id`
- `app_id` when present

Signal metadata excludes:

- bot tokens
- raw provider JSON dumps
- full oversized message bodies
- secrets, passwords, API keys, credentials, and bearer tokens

## Security Boundaries

- Channel adapters call the runtime. They do not call private action modules,
  mutate confirmations directly, bypass Security Central, or write memory.
- Telegram transport can receive updates and send responses only through the
  bounded client.
- Message text is untrusted model/user input and must not become code,
  commands, URLs to fetch, or skill sources by itself.
- External identity mapping is explicit Settings Central configuration.
- Approval callbacks are durable confirmation resolution requests, not
  capability grants.
- Resource Access Security Posture continues to decide every resource access
  operation class independently.
