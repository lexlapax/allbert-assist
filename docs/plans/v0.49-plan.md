## Purpose

v0.49 adds the second non-text modality — image **input** (paste/upload +
screenshot analysis) and image **generation** — on the v0.48 provider
capability/preference substrate, without a separate image-provider framework.

The implementation shortcut for v0.49 is that the underlying LLM library,
`ReqLLM`, exposes APIs that should cover both halves:

1. **Vision input** is a multimodal text call: an image is attached as a
   `ReqLLM.Message.ContentPart` (`ContentPart.image_url/1` or
   `ContentPart.image/2` for base64+mime) on an ordinary chat message to a
   `vision_input`-capable text model. There is **no new provider adapter** —
   v0.49 extends the existing text-generation call path to carry an image part.
2. **Image generation** is a registered action wrapping
   `ReqLLM.generate_image/3` (with the current provider/model IDs proven in
   M1), reading bytes via
   `ReqLLM.Response.image_data/1`. There is **no hand-rolled HTTP adapter** —
   `ReqLLM` owns provider HTTP, credentials, and request shape exactly as text
   generation does today.

M1 must prove those APIs inside Allbert's app-started runtime. A
`mix run --no-start` probe is not sufficient because ReqLLM provider discovery
initializes during application startup; the M1 probe must exercise
`ReqLLM.Providers.list/0`, `ReqLLM.Images.validate_model/1`, and fixture-backed
request paths with no live network in the release lane.

Consequently v0.49 is materially **smaller than v0.48 voice**: it needs no
`Voice.ProviderHTTP`-style substrate, no per-deployment-mode adapter behaviour,
and **no ADR 0011 amendment** (provider network posture is already owned by
`ReqLLM`, as for text). The new work is media resources, image plumbing into
the text path, an image-generation action, permission/operation classes,
workspace paste/upload + rendering, doctor fields, media redaction, catalog
profiles, and evals.

## Readiness Sweep

Status: complete after this second-pass doc patch; implementation-ready for M1
once the patch is committed.

- **Vision fit.** Vision/image is a modality over the runtime, not a separate
  runtime. Media composes with CLI/workspace/channels through Resource Access,
  Security Central, confirmations, traces, and audits. Generated/observed media
  is descriptive evidence, never authority; nothing auto-promotes to memory.
- **Roadmap fit.** v0.49 follows v0.48 voice and precedes v0.50 Channel Pack 1.
  It proves the v0.48 capability/preference substrate against a second modality
  family (image) and completes the v0.42 media-resource vision (audio in v0.48,
  image/screenshot here; generated media added here).
- **Past-plan fit.** Reuses, in place: ADR 0051 capability/preference + the
  `Models.for/2` resolver; v0.48 profile `media` metadata; the v0.48
  display-only `usage`/`cost` pass-through; the v0.48 workspace media-upload +
  server-side bounds pattern (from the mic flow); v0.48 media trace redaction;
  the capability-aware provider doctor (ADR 0047); and the deterministic
  fake-provider release-gate style. `vision_input`/`image_generation` are
  already in the shipped capability vocabulary (ADR 0051; `ProviderCatalog`).
- **Future-plan fit.** `video_input` stays vocabulary/metadata only (parked).
  v0.50 channel plugins may forward image/media references but do not own
  provider authority. Nothing here pre-commits the v1.0 freeze.
- **Multimodal bridge fit.** Provider profiles may describe local or online
  multimodal models, but v0.49 implements only the image/screenshot-to-text
  and text-to-image bridge. v0.48 continues to own speech-to-text and
  text-to-speech. Generic audio understanding, video input, and video
  generation stay future work until a later plan adds capability-specific
  actions, resource classes, doctor fields, permissions, and release evidence.
- **ADR fit.** ADR 0042 gains the image/screenshot amendment (the audio portion
  shipped in v0.48). ADR 0047 gains image-doctor fields. ADR 0051 is consumed
  unchanged. **No ADR 0011 amendment** — `ReqLLM` owns provider HTTP for vision
  input and image generation, as it does for text.
- **req_llm-reuse correction.** Unlike v0.48 voice (no req_llm audio support,
  so hand-built HTTP), v0.49 vision/image-gen ride req_llm's native multimodal
  `ContentPart` input and `generate_image/3`. The plan deliberately avoids
  reproducing the voice provider-HTTP/adapter machinery.

## Locked Decisions

These choices are binding for v0.49. Changing any requires an explicit
re-decision here and in the request-flow document before code lands.

| # | Decision | Chosen value | Trade-off if changed |
|---|---|---|---|
| 1 | Provider substrate | Reuse the v0.48 capability-aware provider/preference framework; vision/image models are just providers. | A parallel image-provider stack would duplicate secrets/doctor/onboarding/settings/traces. |
| 2 | Vision-input path | `vision_input` is a **multimodal `ContentPart`** on the existing `ReqLLM` text-generation call; the resolver picks a `vision_input`-capable text profile and the action attaches the image. No new adapter. | A bespoke vision adapter would duplicate req_llm's multimodal support and the text path. |
| 3 | Image-generation path | A registered `generate_image` action wrapping `ReqLLM.generate_image/3` (provider profile → `"provider:model"`); bytes via `Response.image_data/1` → a bounded image resource. No hand-rolled HTTP. | Hand-rolled HTTP would re-introduce the v0.48 ProviderHTTP/ADR-0011 burden req_llm already covers. |
| 4 | Provider scope | req_llm-backed OpenAI/Gemini remote profiles for vision input + image generation, plus deterministic fixture support for tests. Local vision/image profiles remain future unless M1 proves a req_llm-backed local endpoint without new ProviderHTTP/ADR 0011 work. No non-req_llm image provider in v0.49. | A local image provider would re-introduce ProviderHTTP + an ADR 0011 amendment; deferred. |
| 5 | Media source | `image://capture/<id>` and `screen://capture/<id>` identify **operator-supplied** paste/upload media — NOT autonomous OS screen capture. Vision analysis may also consume an existing v0.43 `browser_screenshot` resource. | Autonomous screen capture is a large new authority (like live mic) and is out of v0.49 scope. |
| 6 | Image normalization | **Constrain to provider-accepted formats/dimensions/size and bounded-reject** otherwise; no resize/convert binary dependency in v0.49. | A resize/convert helper (vix/ImageMagick-class) adds a dependency; deferred until a provider mismatch demands it. |
| 7 | Permission floors | `image_input` (operator-supplied) floor `:allowed` with redaction; `image_generate` floor is derived from the resolved profile's `media.deployment_mode` (remote → `:needs_confirmation`, fake → `:allowed`; unresolved → fail-closed), reusing the v0.48 mechanism. | Treating image generation as `:allowed` would let billable provider calls run without confirmation. |
| 8 | Cost visibility | Display-only pass-through `usage`/`cost` from `ReqLLM.Response` (`source: :unavailable` when absent); no aggregation/enforcement. | A spend subsystem would expand v0.49 beyond the modality. |
| 9 | Test lane | `release.v049` exercises real action/resolver/redaction paths through deterministic req_llm `Req.Test` fixtures + image fixtures; fake provider is fixture support. Opt-in `.env` live smokes (OpenAI/Gemini) are manual before tag. | Live provider tests are credentialed/non-deterministic; fake-only misses the product contract. |
| 10 | Capability semantics | Capability names are routing predicates; media modalities/formats/limits live in profile `media` metadata and never grant permission. | Coupling routing to one vendor's API shape; metadata-as-authority. |

the shipped vocabulary (ADR 0051). Catalog/settings gain vision + image
profiles (mirroring the v0.48 voice profiles); `providers` reuses the existing
`openai`/`gemini` entries (secret refs) — no new provider type:

The resolver (`Models.for/2`, `candidates_for/2`) selects by capability with
the same enable/capability validation, ranked fallback, and one-bounded-pass
call-failure fallback as v0.48 (retryable → next; non-retryable → stop; floor
re-applied per profile). Fixture-only fake profiles must not be operator
default preferences or release-validation targets. `fake_media` is a sample
fixture-provider label; if the catalog already has a different fixture provider
shape, M1 should use that existing test seam instead of adding an
operator-visible provider type.

## Multimodal Bridge Profile Boundary

v0.49 may describe a local or online model as multimodal in profile metadata,
but there is intentionally no catch-all `multimodal` capability and no
all-purpose media router. Each media direction remains a named capability with
its own action/resource/security contract:

| Capability | Direction | v0.49 status | Local/online note |
|---|---|---|---|
| `speech_to_text` | audio -> text | Shipped in v0.48; unchanged here. | Local Allbert voice runtime, OpenAI, and Gemini remain v0.48-owned paths. |
| `text_to_speech` | text -> audio | Shipped in v0.48; unchanged here. | Local Allbert voice runtime, OpenAI, and Gemini remain v0.48-owned paths. |
| `vision_input` | image/screenshot -> text | Implemented in v0.49. | Remote OpenAI/Gemini through ReqLLM unless M1 proves a req_llm-backed local endpoint without new HTTP policy. |
| `image_generation` | text -> image | Implemented in v0.49. | Remote OpenAI/Gemini through ReqLLM; fixture support is test-only. |
| `video_input` | video/sampled frames -> text | Vocabulary/metadata only. | Requires a future plan before any local or online provider can execute it. |
| video generation or generic audio understanding | text/media -> video or audio -> non-transcript reasoning | Not in v0.49. | Requires future capability names, permissions, doctor fields, resource classes, and release evidence. |

## Media Resource And Authority Model

v0.49 accepts the image/screenshot slice of ADR 0042:

- `image://capture/<id>` and `screen://capture/<id>` identify operator-supplied
  paste/upload media (Locked Decision 5). The ids are opaque, Allbert-Home-
  local, and never provider-selected. v0.49 does **not** add autonomous OS
  screen capture.
- Vision analysis may also target an existing v0.43 `browser_screenshot`
  resource (compose the two); the browser-page screenshot path and the
  operator-supplied media path stay distinct resource origins.
- Image input is constrained (Locked Decision 6): format ∈ profile
  `image_formats_supported`, size ≤ `min(global voice/media bound, profile
  max_image_bytes)`, dimensions ≤ profile `max_image_pixels`; oversized/
  unsupported is denied **before** the provider call. No resize/convert.
- Media content is default-off for retention and redacted from traces by
  default. Traces record bounded metadata only: resource URI, byte size,
  dimensions, MIME type, provider profile, content hash, and redaction status.
  Raw image/screenshot bytes never appear in traces, audits, diagnostics,
  LiveView assigns, or CLI output.

New permission classes:

- `:image_input` — floor `:allowed` (operator-supplied content, redacted).
- `:image_generate` — floor derived from the resolved profile's
  `media.deployment_mode` (remote → `:needs_confirmation`; fake → `:allowed`;
  unresolved → fail-closed `:needs_confirmation`), reusing the v0.48
  `voice_floor`-style mechanism.

New operation classes / origins:

- origin kind `:image_input`
- operation `:image_input`
- operation `:image_generate`

## Workspace Media

- Paste/upload crosses the web boundary via a Phoenix LiveView binary upload;
  `:image_input` size/type/dimension bounds are enforced **server-side**
  (client-reported values are untrusted), reusing the v0.48 mic-upload pattern.
- Rendering uses the existing validated workspace surface/catalog (provenance,
  fallback text); **no model-generated UI code** is executed.
- Generated images are written to a bounded Allbert-Home media path; retention
  is default-off and operator-removable.

## Provider Doctor

`doctor` reuses ADR 0047's redacted envelope and adds, for vision/image
profiles, additive fields only: `:image_input_supported` (ADR 0047 reserved),
`:image_generation_supported`, `:image_formats_supported`,
`:provider_usage_metadata_available`, `:fixture_probe_ok`. Dispatched via the
existing `mix allbert.model doctor <profile>`. Read-only; never emits raw
URLs/secrets/media paths.

## Operator-Visible Win

An operator can paste or upload a screenshot/image, ask Allbert about it, and
generate diagrams/images on demand — with visible provider/cost metadata and
default-off retention, and with binary content kept out of traces.

First-time walkthrough:

1. Operator (vision enabled) pastes a screenshot into `/workspace` and asks
   "what's failing in this stack trace?".
2. The upload becomes a bounded `image://capture/<id>`; size/type/dimensions
   validated server-side.
3. The vision flow resolves `vision_input` to a capable text profile and sends
   the image as a `ContentPart` on the text call; the answer renders inline.
4. Operator asks "generate a simple architecture diagram"; `generate_image`
   resolves `image_generation`, confirms (remote floor), runs
   `ReqLLM.generate_image/3`, writes a bounded media resource, and renders it
   with display-only provider/cost metadata.
5. Traces hold only redacted metadata; nothing auto-promotes to memory.

## Security Posture

- Media input is input modality only; it does not change action authority.
- Provider capability metadata and doctor success never grant permission.
- Operator-supplied media only; no autonomous screen capture.
- Image generation is effectful/provider-backed and confirmation-gated for
  remote providers; cost/usage is display-only.
- Binary image/screenshot content is bounded, default-off for retention, and
  absent from traces/audits/assigns/CLI.
- Workspace rendering executes no model-generated UI code.
- No new attack surface beyond the image resource classes + two permission
  classes; provider HTTP/credentials remain req_llm's, as for text.

## Non-Goals

- No video ingestion, sampled-frame analysis, or video generation.
- No autonomous OS screen capture.
- No arbitrary remote media fetch.
- No permanent media retention by default.
- No model-generated UI code.
- No local/non-req_llm image provider (and therefore no ProviderHTTP/ADR 0011
  work) in v0.49.
- No image resize/transcode dependency in v0.49 (constrain + reject instead).

## Definition Of Done

v0.49 is done when:

- vision input works by attaching an image `ContentPart` to a resolved
  `vision_input` text profile via `ReqLLM`, from workspace (paste/upload) and
  over an existing `browser_screenshot`;
- image generation works through a `generate_image` action wrapping
  `ReqLLM.generate_image/3`, confirmation-gated for remote, with display-only
  cost;
- `image://`/`screen://` media are operator-supplied, bounded, default-off for
  retention, and absent from traces;
- the `vision_input`/`image_generation` capabilities resolve through the v0.48
