## Purpose

Expand Allbert reach into team and community chat by adding Discord and Slack
channel plugins that reuse the existing channel substrate. Formalize the
channel approval-primitive contract through an ADR 0016 amendment so all
later channels honor one rendering rule.

The two plugins each ship the provider package shape that v0.17 established
for Telegram and email (`Plugins.<Provider>` + `Channels.<Provider>.
{Adapter,Client,Renderer,Parser}`), plus provider settings fragments and
doctor actions. The amendment graduates approval-rendering from each
renderer's hand-rolled choice (v0.16) to a shared primitive selection module
(`Approval.Handoff.render/2`) that consumes descriptor-declared primitive
sets.

## Research Grounding

The v0.52 design is grounded in:

- **Discord Gateway** (`https://discord.com/developers/docs/topics/gateway`):
  persistent WebSocket connection authenticated with the bot token;
  `READY`, `MESSAGE_CREATE`, `INTERACTION_CREATE` events; heartbeat
  protocol; resume on disconnect; intents (bitmask controlling which
  events the bot receives). Discord REST calls use
  `Authorization: Bot <token>`. The adapter obtains the WSS URL,
  recommended shard count, and session-start-limit state from
  `GET /gateway/bot` at connect time; the response is not cached
  long-term. Bots running on a single shard handle up to ~2,500 guilds;
  v0.52 single-shard. Ed25519 signature verification is
  Interactions-only (not Gateway) and is parked alongside the
  Interactions transport.
- **Slack Socket Mode** (`https://api.slack.com/apis/socket-mode`):
  WebSocket initiated by the bot using an App-Level Token (`xapp-...`);
  `apps.connections.open` is called with the app-level token in the HTTP
  `Authorization` header and returns a temporary WSS URL, so each reconnect
  obtains a fresh URL. Socket Mode carries the same event payloads as
  Events API, wrapped in envelopes that must be acknowledged by
  `envelope_id`. `app_mention`, `message.channels`, and `interactive`
  (Block Kit button taps) envelopes are in scope. Single-workspace bot
  token (`xoxb-...`) for read/write. Multi-workspace OAuth model deferred
  per M0 decision 5.
- **Discord Component System** (Block Kit analog): buttons (action rows),
  select menus, modals. v0.52 renders the `:button` primitive through
  Discord components; select menus and modals deferred.
- **Slack Block Kit** (`https://api.slack.com/block-kit`): blocks +
  actions; button taps return `block_actions` interaction events that
  the bot replies to or updates the original message. v0.52 renders the
  `:button` primitive through Block Kit buttons; rich Block Kit composition
  deferred.
- **The v0.16 channel adapter boundary** (ADR 0016 §Decision lines
  76-103): channels are delivery adapters around
  `AllbertAssist.Runtime.submit_user_input/1` and registered Jido
  actions. They do not own intent selection, action execution, Security
  Central policy, Resource Access grants, confirmation storage,
  conversation history, markdown memory, or app registry semantics.
- **The v0.17 shipped-plugin pattern**: Telegram and email plugins live
  under `./plugins/allbert.<provider>/`; each ships a 5-module quartet
  (`Plugins.<Provider>` + `Channels.<Provider>.
  {Adapter,Client,Renderer,Parser}`) and a plugin manifest
  (`allbert_plugin.json`). v0.52 mirrors the shape for Discord and
  Slack.
- **The v0.42 closeout lessons** (`v0.42-impl-audit-{01,02,final,
  completion,release}.md` archived): release smoke gate (`release.vXXX`),
  enumerated doc-sync discipline, runs-as-written operator validation,
  doctor live-verify, plugin-owned settings, lane annotations per
  ADR 0049.

Implementation consequence: v0.52 adds two plugins, one new module
(`Approval.Handoff`), one descriptor schema field (`primitives:`),
two doctor actions (`discord_doctor`, `slack_doctor`), two settings
fragments (`channels.discord.*`, `channels.slack.*`), and the ADR 0016
amendment Status flip — but no new permission classes, no new
operation classes, no new URI scheme, no new memory namespace, no new
core authority surface. HTTP uses `Req`. WebSocket provider transports are
hidden behind explicit port behaviours; if M1/M2 need a real WebSocket
library, the dependency must be directly declared and reviewed rather than
borrowed through a transitive dependency. Provider stubs remain the release
gate path; real WebSockets are external-smoke only.

## Approval Primitive Selection (`AllbertAssist.Approval.Handoff`)

The ADR 0016 amendment §"Selection Rule" (line 240-244) requires a
shared selection layer with order `:button > :typed_command > :link >
:list`. Today each renderer (`Channels.Telegram.Renderer`,
`Channels.Email.Renderer`) hand-rolls its own choice. v0.52 introduces
the shared module.

### Module surface

`AllbertAssist.Approval.Handoff` lives at
`apps/allbert_assist/lib/allbert_assist/approval/handoff.ex`. It is a
separate module from:

- `AllbertAssist.Intent.ApprovalHandoff` — keeps its existing
  `lines/1` role of producing the text-line representation of an
  approval payload (consumed by all renderers).
- `AllbertAssist.Intent.Handoff` — keeps its existing v0.33 role of
  app-intent handoff and clarification (unrelated despite the similar
  name).

### `render/2` contract

```elixir
@spec render(handoff_payload :: map(), descriptor :: map()) ::
        {:ok, {primitive_kind(), payload :: map()}} | {:error, term()}

@type primitive_kind :: :button | :typed_command | :link | :list
```

The selection rule:

1. Read an effective descriptor. Adapters start from
   `descriptor.primitives` (a non-empty list of `primitive_kind` atoms,
   guaranteed by the registry validator from §"Channel Descriptor
   Primitives Field" to include `:list`) and remove `:button` before calling
   `render/2` when provider settings set `render_approval_buttons: false`.
2. Pick the highest-fidelity primitive the adapter declares, in order
   `:button > :typed_command > :link > :list`. `:link` is eligible only
   when the handoff payload carries a workspace URL in
   `handoff_payload.render_hints.workspace_url` or
   `handoff_payload.workspace_url`; otherwise selection continues to `:list`.
3. Build the payload for that primitive from the `handoff_payload`
   (confirmation id, action label, ObjectiveContext lines, ApprovalHandoff
   lines). The payload shape depends on the primitive:
   - `:button` → `%{text: text, buttons: [%{label, callback_data}]}`
   - `:typed_command` → `%{text: text, commands: ["ALLBERT:APPROVE:<id>", "ALLBERT:DENY:<id>", "ALLBERT:SHOW:<id>"]}`
   - `:link` → `%{text: text, url: workspace_url}`
   - `:list` → `%{text: text, numbered_options: [...]}`
4. Each adapter's `Renderer.render_approval_handoff/1` consumes
   `{primitive_kind, payload}` and formats it into the provider-specific
   wire shape (Telegram inline keyboard, Slack Block Kit blocks, etc.).

The `render/2` signature does not receive settings. Provider renderers own
the settings-to-effective-descriptor step so the module boundary remains
deterministic and testable.

### Renderer retroactive consumption

M0 extracts the primitive selection from
`Channels.Telegram.Renderer.render_approval_handoff/2` and
`Channels.Email.Renderer.render_approval_handoff/2`. Each renderer's
behavior is unchanged at the wire level (Telegram still renders inline
keyboards; email still renders typed commands) — but the *choice* now
flows through `Approval.Handoff.render/2`. M0 acceptance proves the
existing v0.16 Telegram/email approval flows are byte-equivalent before
and after the refactor.

## Channel Descriptor Primitives Field

The amendment requires adapters to declare their supported primitives "in
their plugin descriptor" (line 224). v0.52 makes this concrete.

### Schema extension

The channel descriptor map returned by `<Plugin>.channels/0` (see
`plugins/allbert.telegram/lib/allbert_assist/plugins/telegram.ex:18-36`
for the v0.17 baseline) grows a `primitives:` field:

```elixir
%{
  channel_id: "telegram",
  # ...existing fields...
  primitives: [:button, :typed_command, :list]   # new at v0.52
}
```

### Declared subsets (per amendment table lines 248-260)

| Adapter | `primitives:` |
|---|---|
| CLI | `[:list]` |
| LiveView | `[:button, :list]` |
| Telegram | `[:button, :typed_command, :list]` |
| Email | `[:typed_command, :list]` |
| Discord | `[:button, :typed_command, :list]` |
| Slack | `[:button, :typed_command, :list]` |

CLI and LiveView are not shipped plugin descriptors today, but any local
surface that consumes the shared selector must expose the same effective
primitive subset above. LiveView keeps `:list` as the universal fallback even
when its primary affordance is a workspace modal button.

### Registry validation

`AllbertAssist.Plugin.Registry` rejects channel descriptors that:

- omit the `primitives:` field, OR
- declare an empty list, OR
- declare a primitive not in `[:button, :typed_command, :link, :list]`,
  OR
- omit `:list` (the universal fallback).

Rejected descriptors fail plugin discovery with a structured
diagnostic; the adapter does not start.

### Migration

M0 adds the field to existing Telegram + email descriptors retroactively
and lands the registry validator. M1 (Discord plugin skeleton) and M2
(Slack plugin skeleton) declare the field from day one. M4 wires the
descriptor field through `Approval.Handoff.render/2` for confirmation
callbacks.

## Channel Settings (`channels.discord.*` / `channels.slack.*`)

Per M0 decision 4, settings live in plugin-owned `Settings.Fragment`
modules (`AllbertDiscord.Settings.Fragment`,
`AllbertSlack.Settings.Fragment`), not in core `Settings.Schema`.
Fragments compose into the operator-visible settings namespace at plugin
discovery time per ADR 0031.

The existing settings schema treats `channels` as a reserved namespace. M0
therefore adds an owner-aware fragment bridge that permits trusted,
source-tree channel plugins to contribute only their own
`channels.<provider>.*` subtree (`discord` may not claim `slack`, and
home/imported plugin metadata may not claim any core channel namespace). M1
and M2 use that bridge for Discord and Slack. This is a Settings Central
change, not a bypass: reads, writes, validation, audit, and redaction still
go through Settings Central.

### `channels.discord.*` (Discord plugin)

```yaml
channels:
  discord:
    enabled: false                       # opt-in master switch
    bot_token_ref: null                  # required when enabled: true
    application_id: null                 # bot application id (string)
    allowed_guild_ids: []                # operator-curated allowlist
    allowed_channel_ids: []              # optional sub-allowlist within allowed guilds
    response_style: "mention"            # mention | always | dm_only
    max_text_bytes: 2000                 # Discord message hard limit
    render_approval_buttons: true        # respects primitive declaration
    identity_map:                       # existing v0.16 list shape
      - external_user_id: "11111"
        user_id: "alice"
        enabled: true
    gateway_intents:                     # opt-in subset; default minimal
      - guild_messages
      - direct_messages
      - message_content
    gateway:
      reconnect_max_backoff_ms: 30000
      heartbeat_jitter: true
```

Discord plugin secret refs (stored through Settings Secrets):

- `channels.discord.bot_token_ref` → `secret://channels/discord/bot_token`

Discord plugin does NOT collect: Interactions signing key (Interactions
transport is parked); OAuth client secret (multi-workspace install
parked per M0 decision 5 — Discord doesn't have multi-workspace OAuth
in the Slack sense, but multi-bot per-Allbert-home is parked).

### `channels.slack.*` (Slack plugin)

```yaml
channels:
  slack:
    enabled: false                       # opt-in master switch
    bot_token_ref: null                  # xoxb-... required when enabled: true
    app_token_ref: null                  # xapp-... required for Socket Mode
    signing_secret_ref: null             # reserved for Events API HTTP (parked)
    workspace_team_id: null              # single-workspace per M0 decision 5
    allowed_channel_ids: []              # operator-curated allowlist
    response_style: "mention"            # mention | always | dm_only
    max_text_bytes: 3000                 # Slack message practical limit
    render_approval_buttons: true        # respects primitive declaration
    identity_map:                       # existing v0.16 list shape
      - external_user_id: "U0123ABCDE"
        user_id: "alice"
        enabled: true
    socket_mode:
      enabled: true                      # invariant at v0.52; Events API parked
      reconnect_max_backoff_ms: 30000
```

Slack plugin secret refs:

- `channels.slack.bot_token_ref` → `secret://channels/slack/bot_token`
- `channels.slack.app_token_ref` → `secret://channels/slack/app_token`
- `channels.slack.signing_secret_ref` → reserved; unused at v0.52

### Safe-write keys

Per fragment, the safe-write keys mirror the YAML keys above with the
following invariants enforced at the fragment schema layer:

- `channels.discord.enabled` and `channels.slack.enabled` default `false`.
- `channels.slack.socket_mode.enabled` is read-only `true` at v0.52
  (decision 3).
- `channels.<provider>.identity_map` keeps the existing v0.16
  `channels_identity_map` list-of-maps shape; v0.52 does not migrate it to a
  raw map.
- Provider fragments add cross-field validation for required-when-enabled
  keys. Discord requires `bot_token_ref`, `application_id`, and at least one
  `allowed_guild_ids` entry when enabled. Slack requires `bot_token_ref`,
  `app_token_ref`, `workspace_team_id`, and at least one
  `allowed_channel_ids` entry when enabled.
- Permission settings are core-owned (channels do not invent permission
  classes — security posture preserved from v0.16).

## Permission, Operation, And Identity Additions

**None.** Channels do not invent permission classes, operation classes,
or URI schemes. v0.52 preserves the v0.16 boundary: channels are
delivery adapters; runtime authority lives in Security Central,
Resource Access, `Confirmations`, and registered actions.

This is the explicit non-addition to the substrate. The amendment locks
the primitive contract; the descriptor field formalizes per-adapter
declarations; the `Approval.Handoff` module extracts existing renderer
behavior. No new authority is created.

## Action Schemas And Capability Metadata

The shared `list_channels` / `show_channel` actions (v0.16) remain the
operator-facing handles, but `show_channel` needs a v0.52 generic detail path
for plugin-owned channels so Discord/Slack are not forced through the old
Telegram/email-only detail clauses. v0.52 adds two doctor actions, one per
provider.

| Action | Permission | Execution mode | Exposure | Notes |
|---|---|---|---|---|
| `discord_doctor` | `:read_only` | `:channel_diagnostic` | `:internal` | ADR 0047-style redacted channel-doctor envelope; calls Discord `GET /users/@me` with the configured bot token through `Channels.Discord.Client`; reports gateway connectivity status, bot username, application id, last successful Gateway READY event timestamp, and any auth failures. |
| `slack_doctor` | `:read_only` | `:channel_diagnostic` | `:internal` | ADR 0047-style redacted channel-doctor envelope; calls Slack `auth.test` with the configured bot token; reports workspace name, bot user id, team id, last successful Socket Mode hello-event timestamp, and any auth failures. |

Action schemas:

| Action | Required | Optional |
|---|---|---|
| `discord_doctor` | — | `verbose` |
| `slack_doctor` | — | `verbose` |

Doctors persist their last result envelope under
`<ALLBERT_HOME>/cache/channels/<provider>/doctor/state.json` so the
adapter's startup path can read the last-known status (v0.43 R2 doctor
live-verify pattern). Adapters do not refuse to start solely on a stale
doctor at v0.52 (channels are passive delivery surfaces, not session-
based runtimes like the browser), but the doctor result surfaces in
`mix allbert.channels show <provider>` output.

## Operator-Visible Win

Allbert can be reached from Discord and Slack with safe mention handling,
threaded replies, and confirmation callbacks.

A first-time Discord flow walks through:

1. Operator: `mix allbert.channels discord set-token secret://...`
   (token reference stored; raw token rejected) →
   `mix allbert.channels discord set-application-id ID`
   → adds guild id to `allowed_guild_ids` → `mix allbert.channels
   discord map --external-user 123456 --user alice` → enables in
   settings.
2. `mix allbert.channels show discord` runs `discord_doctor` and shows
   bot identity + gateway status.
3. External Discord user mentions the bot in an allowed guild channel;
   the message reaches `Runtime.submit_user_input/1` with local user
   resolved to `alice`.
4. Bot replies in the same channel/thread.
5. If the operator's prompt triggers an effectful action, the bot
   renders a confirmation as a Discord button (primitive `:button`
   selected by `Approval.Handoff.render/2`); approve/deny resolves
   the pending confirmation through the standard Approval Handoff
   path.

A first-time Slack flow walks through the same shape with Slack-
specific commands (`set-token`, `set-app-token`, `set-team-id`,
allowlist, identity map, doctor, mention, button approval). DM-only mode
is supported via `response_style: dm_only` for both providers.

## Transport And Lifecycle Decisions

### Discord — Gateway WebSocket (M0 decision 2)

- **Why Gateway, not Interactions HTTP?** Gateway requires no public
  inbound URL — Allbert connects out. Mirrors v0.16's "no public
  inbound HTTP" stance for Telegram long-polling. Interactions HTTP
  requires HTTPS exposure + Ed25519 verification per request and is
  parked to v0.52.x.
- **Why single shard?** Discord requires sharding only for bots in
  >2,500 guilds. v0.52 targets per-operator personal-scale and small-
  team scale; single shard suffices. Sharding parked.
- **Connection lifecycle.** `Channels.Discord.Adapter` GenServer owns:
  `GET /gateway/bot` through `Req` to obtain the current WSS URL and
  session-start-limit state; initial `IDENTIFY` with `bot_token_ref`
  resolution + intents bitmask; heartbeat loop with `heartbeat_jitter`;
  `READY` event capture; resume with `RESUME` after disconnect using stored
  session id + seq; reconnect with exponential backoff capped at
  `reconnect_max_backoff_ms`; clean shutdown via a normal Gateway close on
  supervisor terminate. `/gateway/bot` output is a connection-time hint, not
  durable configuration.
- **Interaction acknowledgement.** Gateway-delivered component interactions
  are acknowledged through Discord's interaction callback HTTP endpoint using
  the interaction id and token from the payload. They are not sent back over
  the Gateway. Stub tests assert the callback request shape; real callback
  delivery is external-smoke.
- **Crash isolation.** Adapter supervised under
  `AllbertAssist.Channels.Supervisor` with `:one_for_one`; an adapter
  crash takes its own session, not Slack, not core (v0.16 invariant
  preserved).

### Slack — Socket Mode (M0 decision 3)

- **Why Socket Mode?** Direct analog to Telegram long polling. No public
  URL; bot uses an App-Level Token to open a WebSocket to Slack. Events
  API HTTP requires HTTPS + HMAC signing and is parked.
- **Why single workspace?** Multi-workspace requires OAuth install flow,
  per-workspace token storage, team-id scoping on every API call. Out
  of scope for "Pack 1." Parked to v0.52.x or v1.x per M0 decision 5.
- **Connection lifecycle.** `Channels.Slack.Adapter` GenServer owns:
  `apps.connections.open` call through `Req` with the app-level token in
  the `Authorization` header to get a fresh temporary WS URL; WebSocket
  connect; `hello` event capture; `events_api` envelope handling (sends an
  ack for the exact `envelope_id` before or immediately after durable
  `channel_events` receipt); `interactive` envelope handling for button
  taps; `chat.update` or response-target update for the rendered callback
  result; reconnect with exponential backoff and fresh
  `apps.connections.open` per reconnect; clean shutdown.
- **Crash isolation.** Same `:one_for_one` posture as Discord.

## Why Plugin-Shaped

ADR 0017 made the plugin contract Allbert's contribution boundary. v0.16
established Telegram and email as the proving channel plugins; v0.52
extends the shape to Discord and Slack without growing core surface.
Each plugin contributes:

- one channel descriptor (with the new `primitives:` field);
- one Adapter GenServer for the provider lifecycle;
- one Client for outbound HTTP/WS calls;
- one Renderer that consumes `Approval.Handoff.render/2` output;
- one Parser for inbound payload normalization;
- one `Settings.Fragment` for `channels.<provider>.*`;
- one doctor action (`<provider>_doctor`);
- a `mix allbert.channels <provider> *` CLI subcommand surface.

The four channel plugins (Telegram, Email, Discord, Slack) share the
same v0.16 substrate (`channel_events` for dedupe, `Channels.Identity`
for external→local mapping, session_id derivation, supervised
`:one_for_one`); they diverge only in their provider transport, payload
shape, and primitive subset.

## v0.47 Trace Mining (Post-v0.52 Source)

v0.47 self-improvement ships before team channels and therefore does not
depend on these traces. After v0.52 lands, the same read-only
pattern-discovery action may mine v0.52 team-channel traces as one additional
source for trace-to-skill and trace-to-workflow draft suggestions. v0.52
preserves the v0.16 trace shape:

- `channel_events` carries external identity (redacted), local user
  resolution, dedupe metadata, status; raw provider payloads are
  redacted/truncated, not stored verbatim.
- Trace mining sees the resolved local `user_id`, the channel id, the
  event kind, and the runtime turn — not raw Discord/Slack payloads,
  not bot tokens, not signing secrets.
- The `redact_provider_payload` discipline from v0.16 applies to Discord
  and Slack identically.

v0.47 needs no channel-specific contract. v0.52 supplies the additional traces
later without changing the self-improvement trust tier.

## Security Posture

Channels do not invent permission classes or confirmation shapes.
Group/server authorization, workspace mapping, and callback ownership
must be explicit. Approval Handoff remains provider-neutral and
action-owned.

Specific v0.52 invariants:

- **Allowlist before runtime submission.** Discord guild + channel
  allowlists and Slack channel allowlist gate inbound message handling
  before `Runtime.submit_user_input/1` is called. Messages from
  non-allowlisted surfaces produce a `channel_events` rejected record
  and are not submitted to the runtime.
- **External identity must be mapped.** Inbound events without an
  `identity_map` entry produce a `channel_events` rejected-unknown
  record and no runtime submission. There is no implicit `user_id`
  assignment.
- **Bot tokens are secret refs only.** `bot_token_ref`,
  `app_token_ref`, and `signing_secret_ref` are `secret://` references;
  raw values never appear in `mix allbert.channels show` output,
  in `channel_events` rows, in traces, or in audit records.
- **Provider payload redaction.** `Channels.<Provider>.Parser` redacts
  bot tokens, signing secrets, full Authorization headers, full
  message embed URLs with credentials, and any field marked as
  redacted by the parser's allowlist (v0.16 pattern preserved).
- **No callback-data smuggling.** The callback data string format
  (`allbert:v1:<verb>:<id>`) is preserved from v0.16 Telegram. Discord
  custom_id and Slack action_id fields use the same format. Parser
  rejects malformed callback data; the registered confirmation action
  rejects callback data whose verb is not in
  `{approve, deny, show}` or whose confirmation id does not match an
  open pending request for the resolved local user.
- **Cross-channel confirmation isolation.** A confirmation pending
  resolution from Slack cannot be approved via Discord (and vice
  versa). The pending request records its origin channel/surface; the
  registered confirmation action enforces the channel match.
- **DM vs workspace surface auth.** DM-routed messages are treated as
  authenticated only when the external user has an `identity_map`
  entry. DM-only mode (`response_style: dm_only`) restricts the bot
  to never reply in shared channels.

### Milestone 4: Confirmation Callbacks Through Primitive Selection

Status: planned.

Implementation tasks:

- For each provider, wire the confirmation-rendering path through
  `Approval.Handoff.render/2`:
  - Discord: descriptor declares `[:button, :typed_command, :list]`;
    `render/2` selects `:button`; Renderer emits Discord message
    components with `custom_id: "allbert:v1:<verb>:<id>"`; INTERACTION_CREATE
    events parse the custom_id and resolve the pending through the
    registered confirmation action.
  - Slack: descriptor declares `[:button, :typed_command, :list]`;
    `render/2` selects `:button`; Renderer emits Block Kit `actions`
    block with `action_id: "allbert:v1:<verb>:<id>"`; interactive
    envelope parses action_id and resolves pending.
  - Both: fall back to `:typed_command` when buttons are disabled
    (`render_approval_buttons: false`), then to `:list` for the universal
    fallback path.
- Telegram + email retroactive: confirm M0's primitive-extraction
  refactor still produces the same wire output through the new boundary
  (regression coverage from M0 carries forward).
- Confirmation callback verb invariants: only `approve`, `deny`, `show`
  are accepted; unknown verbs reject; callback id must match a pending
  confirmation owned by the resolved local user; cross-channel
  callbacks reject per M3.
- Callback adapters perform a strict guard before invoking registered
  confirmation actions: parse `allbert:v1:<verb>:<id>`, load the pending
  confirmation, verify pending status, resolved local user, origin channel,
  and provider response target/thread where present, then call
  `Actions.Runner.run/3` with the existing confirmation action. This guard is
  required because the pre-v0.52 confirmation actions are not sufficient by
  themselves to enforce remote callback ownership for approve, deny, and show.

- Discord button taps resolve the right pending confirmation;
  approve/deny propagate through the standard `Confirmations.resolve/4`
  path; `show` returns the pending detail to the originating channel.
- Slack button taps behave equivalently.
- Disabling `render_approval_buttons` for a provider falls back to
  `:typed_command` rendering; operator typing
  `ALLBERT:APPROVE:<id>` in a reply resolves the pending.
- Cross-provider isolation holds: a Discord-origin pending cannot be
  resolved by Slack and vice versa.
- Telegram + email approval flows are byte-equivalent to v0.16 wire
  output (regression).
- §Operator Validation M4 block runs end-to-end without edits.
