## Purpose

v0.19 upgrades Allbert's intent layer from deterministic route predicates into
a registry-aware, app-aware, surface-aware intent engine over the capabilities
that already exist. It keeps the v0.11 `AllbertAssist.Intent.Decision`
contract as the execution gate, but gives the decision richer candidate input:
registered actions, trusted skills, jobs, channels, confirmations, existing
trace and memory metadata, plugin provenance, session scratchpad state,
`active_app`, and v0.18 app/surface metadata.

The result should feel smarter to a user without becoming more dangerous. If a
session is in StockSage, Allbert can prefer StockSage actions, skill paths, and
surfaces. If the same words arrive in neutral Allbert context, StockSage is
only a lower-ranked or suggested candidate unless the request supplies explicit
app evidence. The engine may ask a model for structured classification when
deterministic confidence is low, but model output is treated as untrusted
proposal data and must validate through the same registries and Security
Central posture before any action runs.

## Readiness Decision

v0.19 is ready for implementation when the plan, request-flow doc, roadmap,
development guide, and ADRs agree on these points:

- Runtime turns still enter through `AllbertAssist.Runtime.submit_user_input/1`
  and `AllbertAssist.Agents.IntentAgent.respond/1`.
- `AllbertAssist.Intent.Decision` remains the inert decision contract and the
  only decision shape returned to runtime responses, traces, channels, jobs,
  and conversations.
- Candidate ranking may prioritize by app/session context, but never grants
  permission, trust, or execution authority.
- `active_app` is session context, not authorization. Unknown app ids are
  ignored safely and never atomized from external input.
- Registered app surface metadata can produce a navigation decision, but
  v0.19 does not implement the v0.24 workspace shell, generated UI, dynamic
  routes, or canvas rendering.
- Optional model-assisted classification is bounded by settings, deterministic
  fallback, strict validation, and redacted trace metadata.
- Existing v0.11 behavior for URL/document/package/shell/script/import
  decisions is preserved.
- Existing v0.18 behavior for app/surface validation and `active_app: :allbert`
  defaulting is preserved.
- Reviewed memory retrieval from v0.21 is not a prerequisite. v0.19 consumes
  existing memory and trace metadata only, and exposes a plug-in point for
  v0.21 to add reviewed memory signals later.

## Architecture Readiness Audit

The v0.19 plan is consistent with the current architecture:

- ADR 0001 requires signal-first runtime coordination. v0.19 enriches the
  decision attached to runtime signals and traces; it does not introduce a
  separate intent bus.
- ADR 0006 keeps permission, risk, redaction, trust, and audit decisions in
  Security Central. v0.19 ranking never bypasses those checks.
- ADR 0007 requires runtime-facing, effectful behavior to resolve through
  registered actions. v0.19 may choose an action candidate, but execution still
  happens through `AllbertAssist.Actions.Runner.run/3`.
- ADR 0012 keeps resource access operation classes and remembered grants
  operation-scoped. v0.19 may rank resource workflows, but cannot collapse URL,
  package, skill import, shell, script, or local resource posture into generic
  network or shell intent.
- ADR 0014 defines `user_id`, `thread_id`, `session_id`, and `active_app`
  context. v0.19 reads these fields and records them in decisions without
  adding hosted auth, roles, or new identity tables.
- ADR 0015 defines the app/surface contract. v0.19 consumes app registry and
  `SurfaceProvider` metadata, but does not extend the surface DSL.
- ADR 0017 defines plugins as contribution packages, not authority. v0.19 may
  use plugin ids as explainability metadata only.

No new ADR is required for v0.19 if implementation follows this plan. Add a
new ADR only if implementation changes the authority model, makes model output
authoritative, changes the `Decision` contract incompatibly, or adds a new
execution/resource capability.

## Vision And Roadmap Alignment

The vision says Allbert should listen, remember, route intent, invoke
capabilities safely, and learn from traces while remaining inspectable. v0.19
is the routing upgrade: it makes Allbert better at choosing between existing
skills, actions, channels, jobs, apps, and surfaces without adding new powers.

Roadmap dependency order is intact:

- v0.11 supplied the `Decision`, resource posture, and Approval Handoff shape.
- v0.12 supplied local `user_id` and `thread_id` conversation context.
- v0.13 supplied scheduled jobs as runtime-originated background context.
- v0.14 supplied session scratchpad and `active_app`.
- v0.15 supplied minimal app action/skill registration.
- v0.16 supplied channel adapters and channel metadata.
- v0.17 supplied plugin provenance and plugin-contributed actions, skills,
  settings, channels, and apps.
- v0.18 supplied the full app/surface contract, `SurfaceProvider`, and the
  default `active_app: :allbert` home app.
- v0.19 consumes those inputs and prepares v0.20 StockSage, v0.21 memory
  review, v0.24 workspace UI, and v0.26 security evals.

v0.19 must not pull later work forward. It does not build StockSage domain
records, reviewed memory retrieval, Python bridge execution, native trading
agents, workspace shell rendering, LiveViews, security hardening, canvas
integration, or app generation.

## Current State

- `AllbertAssist.Runtime.submit_user_input/1` normalizes request identity,
  thread, session, and `active_app`, emits input/response signals, persists
  conversation messages, records traces, and calls the intent agent.
- `AllbertAssist.Agents.IntentAgent.respond/1` uses deterministic route
  predicates and builds v0.11 `AllbertAssist.Intent.Decision` structs before
  running actions.
- `AllbertAssist.Intent.Decision` validates selected actions through
  `AllbertAssist.Actions.Registry`, validates selected skills through
  `AllbertAssist.Skills`, applies Security Central authorization, normalizes
  candidate action summaries, records identity/session/app fields, and returns
  trace metadata.
- `AllbertAssist.Actions.Registry` exposes core and plugin-contributed action
  capabilities with permission, execution mode, confirmation, app id, and
  plugin provenance metadata.
- `AllbertAssist.Skills.Registry` exposes built-in, project, user, imported,
  plugin, and app skill roots with trust/source metadata.
- `AllbertAssist.App.Registry` exposes registered apps, registered surfaces,
  `SurfaceProvider` surfaces/catalogs, app action/skill metadata, and safe app
  id normalization.
- `AllbertAssist.Session` and the v0.14 scratchpad store volatile
  `{user_id, session_id}` context including `active_app`.
- v0.21 reviewed-memory retrieval does not exist yet. Existing memory input is
  limited to current markdown memory functions, conversation context, traces,
  and action metadata already available before v0.19.

## Guiding Rules

1. Preserve the decision contract. v0.19 may add optional metadata fields to
   `Decision.trace_metadata` and candidate lists, but should not break callers
   that consume the v0.11 decision map.

2. Ranking is not authority. Candidate scores, app affinity, model
   classifications, channel context, plugin provenance, and surface metadata
   never bypass `Actions.Registry`, Security Central, confirmation workflow,
   skill trust rules, resource posture, or action runner lifecycle.

3. Keep deterministic behavior first. The engine must produce useful decisions
   without model assistance. Model classification is optional, setting-gated,
   timeout-bounded, and used only to rank known candidates.

4. Never atomize external ids. App ids, plugin ids, channel ids, action names,
   skill names, job ids, and surface ids from request text, channels, jobs, or
   model output must normalize through existing registries or remain strings in
   rejected diagnostics.

5. App context must be explicit. Prioritize app actions, app skill paths, and
   app surfaces only when request metadata, scratchpad `active_app`, channel
   context, job context, or an already selected route provides explicit
   evidence. Neutral `:allbert` context should not silently route into
   StockSage or another app.

6. Surface navigation is a decision target, not UI generation. A selected
   surface target returns a safe navigation suggestion or action result through
   existing CLI/web/channel output. v0.24 owns the workspace shell and
   ephemeral UI rendering.

7. Existing resource workflows remain operation-scoped. Package installs,
   shell commands, scripts, URL summaries, document inspection, online skill
   import, local skill import, and unsupported MCP/agent URIs keep the v0.11
   resource access and Approval Handoff posture.

8. Preserve channel and job boundaries. Channels and jobs may supply context
   to the engine, but they do not fork intent logic or own separate policy.

9. Trace the why. Selected and rejected candidates must include bounded,
   redacted reasons so the operator can inspect why Allbert picked or refused
   a route.

## Intent Engine Contract

Add a small engine layer under `AllbertAssist.Intent` that the existing intent
agent can call before building or returning a `Decision`:

- `AllbertAssist.Intent.Engine`
- `AllbertAssist.Intent.Candidate`
- `AllbertAssist.Intent.Ranker`
- `AllbertAssist.Intent.Classifier`
- `AllbertAssist.Intent.Evals`

Recommended public API:

```elixir
@spec decide(map()) ::
        {:ok, AllbertAssist.Intent.Decision.t()} | {:error, term()}
```

The input map is the normalized runtime request plus current deterministic
route hints. It should include:

- `:text`
- `:channel`
- `:user_id`
- `:operator_id`
- `:thread_id`
- `:session_id`
- `:active_app`
- `:thread_context`
- `:metadata`
- `:input_signal_id`

The output is still `AllbertAssist.Intent.Decision.t()`.

Implementation may keep the existing route predicates in
`AllbertAssist.Agents.IntentAgent` at first, but the candidate collection,
ranking, validation, and trace metadata should live in the new intent modules
so later milestones do not add more private predicates directly to the agent.

## Candidate Model

Represent ranked candidates as plain maps or a lightweight struct with fields
equivalent to:

- `:kind`: `:action`, `:skill`, `:surface`, `:job`, `:channel`, `:memory`,
  `:refusal`, or `:direct_answer`
- `:id`: string id or action/skill/surface name
- `:label`: bounded display label
- `:source`: `:deterministic`, `:registry`, `:app`, `:plugin`, `:job`,
  `:channel`, `:memory`, `:trace`, or `:model`
- `:score`: float `0.0..1.0`
- `:selected?`: boolean
- `:status`: `:selected`, `:candidate`, `:rejected`, or `:unavailable`
- `:reason`: bounded operator-facing explanation
- `:rejection_reason`: atom or string when rejected
- `:action_name`, `:skill_name`, `:surface_id`, `:job_id`, `:channel_id`
- `:app_id`: atom when known through registry
- `:plugin_id`: string when known through registry
- `:permission`
- `:execution_mode`
- `:confirmation`
- `:resource_access`: list of v0.11 resource posture entries when applicable
- `:trace_metadata`: redacted map

Candidates are proposal data. Before a candidate can become selected, it must
validate through the relevant registry. Unknown actions, untrusted skills,
invalid app ids, disabled channels, blocked jobs, unavailable surfaces, or
permanently denied permissions remain rejected candidates with diagnostics.

## Candidate Sources

The engine should collect candidates from these sources:

- Existing deterministic route hints from `IntentAgent.route/1` and resource
  workflow detection.
- `Actions.Registry` capabilities, including app id and plugin provenance.
- `Skills.Registry` records, including trust, source scope, app skill paths,
  plugin skill roots, and activation status.
- `App.Registry` app identities, legacy navigation surfaces, and
  `SurfaceProvider` surfaces/catalog entries.
- `Session` scratchpad context, especially `active_app`.
- `Conversations` recent thread context from the runtime request.
- Existing markdown memory metadata and recent memory action summaries where
  available before v0.21. Do not add reviewed-memory retrieval yet.
- Confirmation metadata for pending or recently resolved requests where the
  request appears to be approving, denying, or asking about pending work.
- Jobs metadata for requests that ask to run, pause, resume, list, or inspect
  scheduled work.
- Channel metadata for requests that ask to inspect or configure Telegram,
  email, or future registered channels.
- Plugin metadata for explaining why an action, skill, channel, or app exists.

Candidate collection must be bounded. Use caps such as:

- At most 20 action candidates.
- At most 20 skill candidates.
- At most 20 surface candidates.
- At most 10 job candidates.
- At most 10 channel candidates.
- At most 80 total candidates before final ranking.

Caps should be settings-backed where useful, with conservative defaults.

## Ranking Rules

The first implementation should use deterministic scoring with optional model
assistance as an additive signal.

Recommended scoring inputs:

- exact command/route phrase match
- registry action name or skill name match
- resource operation class match
- active app affinity
- app skill path match
- app action capability match
- app surface path/label match
- channel/job keyword match
- recent confirmation/job/channel context
- recent thread context
- plugin provenance match
- model classification confidence, if enabled and valid

Ranking must apply these safety filters before selection:

- selected actions must be registered and exposed to the agent or appropriate
  runtime surface
- selected skills must be trusted or activation-only according to existing
  skill rules
- selected surfaces must come from a registered `SurfaceProvider` or validated
  legacy app surface
- selected app ids must already exist in the app registry
- selected permissions and execution modes must come from action capability
  metadata, not model output
- selected resource access entries must validate through v0.11 resource
  posture modules
- denied permissions produce refusal or confirmation-blocked decisions rather
  than executing

Tie-breaking should prefer:

1. exact deterministic risky-resource route, so safety posture stays stable
2. active app action or skill when explicit `active_app` is not `:allbert`
3. core Allbert action or skill
4. surface navigation suggestion
5. direct answer

When confidence is below the configured threshold, prefer a clarifying direct
answer or refusal over a side-effecting action.

## Optional Model-Assisted Classification

Add a setting-gated classifier path. Defaults should keep deterministic-only
behavior unless the operator has configured a model profile suitable for intent
classification.

Recommended settings:

- `intent.model_assist_enabled`: boolean, default `false`
- `intent.model_profile`: string, default from existing fast/local profile
  when configured
- `intent.model_timeout_ms`: integer, default `3000`
- `intent.model_min_confidence`: float, default `0.72`
- `intent.max_candidates`: integer, default `80`
- `intent.trace_rejected_candidates`: boolean, default `true`

The classifier prompt must receive a bounded candidate summary, not raw
secrets, full memory files, raw traces, or unbounded conversation history.

Classifier output must be strict JSON or map-like data with only:

- `selected_kind`
- `selected_id`
- `confidence`
- `reason`
- `alternatives`

Validation rules:

- `selected_id` must match a collected candidate.
- `selected_kind` must match the candidate kind.
- `confidence` must be numeric `0.0..1.0`.
- Unknown fields are ignored.
- Invalid, timed-out, low-confidence, or non-JSON output becomes a diagnostic
  and deterministic ranking proceeds.

Model output cannot create a candidate, invent an action, select a hidden
action, declare trust, change permissions, or approve confirmation.

## App And Surface Routing

When `active_app` is a known app other than `:allbert`, the engine should:

- prioritize actions whose capability `app_id` matches `active_app`
- prioritize skills whose source scope or app contribution matches
  `active_app`
- prioritize surfaces whose `app_id` matches `active_app`
- include app and surface labels in candidate reasons
- keep all non-matching app candidates available but lower ranked

When `active_app` is `:allbert` or nil, the engine should:

- prefer core Allbert skills and actions
- avoid selecting app-specific actions unless the prompt explicitly names the
  app or a registered app surface/action/skill
- may suggest an app surface as an alternative when the prompt names an app
  but lacks enough context to run an app action

Surface navigation outcomes should be represented as decisions with:

- `intent: :open_surface` or a more specific surface intent
- no executable action unless a registered navigation action already exists
- selected surface metadata in `trace_metadata`
- safe message text such as "Open StockSage at /stocksage"
- no dynamic route creation and no generated UI payload

## Integration Points

### Runtime

`AllbertAssist.Runtime` should continue normalizing request context and calling
the intent agent. v0.19 may add trace metadata fields, but should avoid changing
the public runtime request/response shape except for additive decision
metadata.

Runtime responses should include:

- selected decision map
- selected and rejected candidate summaries
- active app
- app id / surface id when a surface route is selected
- model-assist diagnostics when used
- unchanged `user_id`, `thread_id`, and `session_id`

### Intent Agent

`AllbertAssist.Agents.IntentAgent` should delegate candidate collection and
ranking to the new intent engine while preserving current deterministic routes
and action execution paths.

The agent should not grow another long list of app-specific predicates. App
specificity comes from app/action/skill/surface registry metadata.

### Actions

No new side-effecting action is required. If implementation needs operator
inspection, add read-only actions only:

- `explain_intent`
- `list_intent_candidates`

These actions must be `permission: :read_only`, `exposure: :internal` or
operator-only unless there is a clear agent-facing use, and they must not run
selected candidates.

### Settings

Add settings schema entries under the static core settings schema:

- `intent.model_assist_enabled`: boolean, default `false`
- `intent.model_profile`: string, default `"fast"` or current local fast alias
- `intent.model_timeout_ms`: integer, default `3000`
- `intent.model_min_confidence`: number, default `0.72`
- `intent.max_candidates`: integer, default `80`
- `intent.trace_rejected_candidates`: boolean, default `true`

Settings values must validate through Settings Central and appear redacted
where appropriate. None of these settings grant execution authority.

### Traces And Signals

Runtime traces should add an `Intent Candidates` section or equivalent
structured metadata with:

- selected candidate
- top rejected candidates with reasons
- deterministic score
- model score and diagnostic when model assist ran
- app/session context used for ranking
- plugin provenance where relevant
- surface target metadata where relevant

Do not emit raw model prompts, raw model completions, full memory contents, raw
thread history, or secret-bearing settings.

Optional lifecycle signals may be emitted if they follow the existing signal
style:

- `allbert.intent.candidates_collected`
- `allbert.intent.candidate_selected`
- `allbert.intent.classifier_failed`

If signals are added, they are observability only.

## Milestones

### M1: Intent Candidate Contract And Engine Skeleton

Goal: introduce the engine modules and candidate contract without changing
runtime behavior.

Implementation tasks:

- Add `AllbertAssist.Intent.Candidate`.
- Add `AllbertAssist.Intent.Engine.decide/1`.
- Add `AllbertAssist.Intent.Ranker` with deterministic scoring helpers.
- Add candidate redaction/bounding helpers.
- Add settings schema entries for candidate caps and optional model assist.
- Add tests that prove candidate maps are bounded, redacted, and do not accept
  unknown app ids or unknown action names as selected candidates.

Acceptance:

- Existing intent tests still pass.
- A direct answer request still produces the same high-level decision.
- Candidate metadata appears in the decision trace metadata without changing
  action execution.

### M2: Registry-Aware Candidate Collection

Goal: collect candidates from real registries rather than hand-coded app
predicates.

Implementation tasks:

- Collect action candidates from `Actions.Registry`.
- Collect trusted/activatable skill candidates from `Skills.Registry`.
- Collect app and surface candidates from `App.Registry`.
- Include plugin provenance from `Plugin.Registry`/action capability metadata
  when available.
- Add collection caps and deterministic ordering.
- Reject but trace unknown, untrusted, hidden, disabled, or unavailable
  candidates.

Acceptance:

- Core Allbert actions and skills appear as candidates in neutral context.
- `:stocksage` placeholder or future registered app surfaces appear only as
  surface/app candidates, not executable action authority.
- Plugin-contributed action/skill candidates carry plugin provenance in trace
  metadata.

### M3: Active App And Surface Ranking

Goal: use session `active_app` and registered surfaces as ranking input.

Implementation tasks:

- Apply app affinity when `active_app` is a known non-`:allbert` app.
- Keep neutral `:allbert` context from prioritizing app-specific actions.
- Add surface navigation candidate ranking.
- Represent selected surface navigation as an inert decision target.
- Add tests for explicit app context, unknown app fallback, and neutral
  context.

Acceptance:

- A request with `active_app: :stocksage` prioritizes StockSage candidates
  when StockSage is registered.
- The same request in `active_app: :allbert` does not silently select
  StockSage app actions.
- "Open StockSage" can return a safe surface navigation suggestion when the
  app/surface is registered, without mounting routes or generating UI.

### M4: Preserve Risky Resource And Approval Handoff Behavior

Goal: route risky resource prompts through the enriched engine without
regressing v0.11-v0.13 safety behavior.

Implementation tasks:

- Keep shell, package, script, URL summary, document inspection, direct skill
  import, local skill import, online skill, and unsupported MCP/agent URI
  decisions mapped to the existing action/resource posture.
- Keep confirmation and Approval Handoff data unchanged except for additive
  candidate metadata.
- Ensure background/job-origin turns preserve `foreground_or_background` and
  job context.
- Ensure channel-origin turns preserve channel metadata and do not fork policy.

Acceptance:

- Existing v0.11 risky prompt tests still pass.
- URL summary and document inspection prompts still require confirmation before
  network fetch.
- MCP/agent URIs still produce unsupported decisions.
- Scheduled job turns and channel turns use the same intent contract.

### M5: Optional Model-Assisted Classifier

Goal: add bounded model assistance for low-confidence cases.

Implementation tasks:

- Add `AllbertAssist.Intent.Classifier`.
- Read `intent.model_assist_enabled` and related settings through Settings
  Central.
- Build a bounded, redacted candidate summary for classification.
- Validate model output against collected candidates.
- Fall back to deterministic ranking on disabled, timeout, invalid, low
  confidence, or unknown candidate output.
- Add tests using a fake classifier, not real network/model calls.

Acceptance:

- Model assistance is disabled by default.
- Invalid model output cannot invent or select unregistered actions.
- Low-confidence model output does not trigger side effects.
- Deterministic behavior remains stable when classifier fails.

### M6: Traces, CLI/User Testing, And Release Readiness

Goal: make the new routing inspectable and manually testable.

Implementation tasks:

- Add trace rendering for selected/rejected candidates and app/surface context.
- Add focused tests for trace redaction and candidate explanations.
- Add user-facing smoke commands to `v0.19-request-flow.md` after
  implementation.
- Update `CHANGELOG.md`, `README.md`, roadmap status, and this plan status
  during closeout.
- Run the full warning gate from `DEVELOPMENT.md`.

Acceptance:

- Operators can inspect why an action, skill, surface, or refusal was selected.
- No raw secrets, raw model prompts, raw completions, or unbounded memory/thread
  text appear in traces or CLI output.
- v0.20 StockSage can rely on app/action/skill/surface metadata for routing
  rather than adding private StockSage predicates to the core intent agent.

## Non-Goals

- No new shell, script, package, network, import, browser, crawler, MCP, or
  agent delegation powers.
- No dynamic route creation.
- No generated UI, ephemeral UI, workspace shell, canvas rendering, AG-UI, or
  A2UI emission.
- No StockSage domain implementation or StockSage-specific core predicates.
- No reviewed-memory retrieval, embeddings, vector search, or automatic memory
  promotion.
- No autonomous skill/app/plugin creation.
- No channel-specific intent fork.
- No model-selected side effect without registry validation, Security Central,
  confirmation where required, and action runner execution.

## Definition Of Done

v0.19 is done when:

- Runtime turns pass through the enriched intent engine.
- Existing v0.11 and v0.18 behavior is preserved.
- Skill, action, job, channel, app, and surface selection is registry-aware,
  security-aware, and traceable.
- `active_app` changes candidate ranking but not authorization.
- Registered surfaces can appear as inert navigation candidates.
