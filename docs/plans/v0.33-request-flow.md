## Preconditions And Settings

- Operator is on `/workspace` (v0.32); `active_app` may be `:allbert` (neutral)
  or a selected app set through the registered `set_active_app` action.
- Descriptor-driven handoff is governed by Settings Central keys (defaults in
  parentheses): `intent.descriptors_enabled` (`true`),
  `intent.handoff_threshold` (`0.6`), `intent.handoff_margin` (`0.15`),
  `intent.clarify_floor` (`0.3`).
- Model-backed direct answers require `intent.direct_answer_model_enabled`
  (`false` by default) and a resolvable profile
  (`intent.direct_answer_model_profile`, default reuses `intent.model_profile`).
- Model-assisted candidate selection is the existing
  `intent.model_assist_enabled` (`false` by default). With every model setting
  off, the system is fully deterministic: descriptor matching produces
  handoff/clarification, and direct answers use the bounded fallback.

## Data Flow: Input → Output

Canonical neutral handoff path for `analyze CIEN` (other paths branch from
step 4). Operator-visible results are in **bold**.

1. **Input.** Operator types `analyze CIEN` in the `/workspace` composer.
   `AllbertAssistWeb.WorkspaceLive` `handle_event("ask", ...)` builds a request
   with `text`, `user_id`, `thread_id`, `session_id`, and the socket's
   `active_app` (here `:allbert`) and calls `Runtime.submit_user_input/1`.
2. **Normalize.** `Runtime.normalize_request/1` resolves identity, thread, and
   `active_app` (neutral), persists the user message, and emits the input
   signal/turn-started trace.
3. **Route.** `Agents.IntentAgent.respond/1` runs deterministic `route/2`; with
   no predicate match it is `:direct_answer`, so the request is passed to
   `Intent.Engine.decide/1` without a forced route decision.
4. **Collect + rank.** `Engine.collect_candidates/2` gathers registered actions,
   surfaces, skills, memory, objectives, and the new **app intent descriptor**
   candidates (from `Extensions.Registry`, validated by `Intent.Descriptor`).
   `Intent.Ranker` scores the StockSage `run_analysis` descriptor from its
   `examples`/`synonyms`; `ticker` slot extractor fills `CIEN`.
5. **Decide.** The engine applies the decision rule: score >=
   `intent.handoff_threshold`, margin over the next candidate >=
   `intent.handoff_margin`, and the `ticker` slot is filled, so it returns a
   `:app_handoff` `Decision` carrying an `Intent.Handoff` payload
   (`app_id: :stocksage`, `action_name: "run_analysis"`,
   `extracted_slots: %{ticker: "CIEN"}`). No action runs. If model assist is on,
   the bounded classifier may only re-rank/select among these candidates.
6. **Respond.** The `IntentAgent` `:app_handoff` branch returns a
   side-effect-free response (`status: :completed`) with the handoff payload and
   no confirmation record. `Runtime` persists the assistant message and the
   decision trace (decision kind, descriptor, slots, threshold/margin,
   classifier diagnostic — no raw model text).
7. **Surface.** The proposal renders as a **v0.26 ephemeral surface** in the
   workspace (handles `#intent-handoff`, `#intent-handoff-accept`,
   `#intent-handoff-decline`), composed from existing catalog primitives:
   **"StockSage can run a financial analysis for CIEN. This requires
   confirmation. Run it?"** with accept/decline controls.
8. **Operator choice → Output.**
   - **Accept** (`#intent-handoff-accept`): `WorkspaceLive` runs the registered
     `set_active_app` action (`:stocksage`), then re-submits `analyze CIEN`
     through `submit_user_input/1` with `extracted_slots`. Now in-context, the
     engine selects `run_analysis`, `Actions.Runner.run/3` enforces app scope,
     Security Central applies `permission: :stocksage_analyze`, and the existing
     `confirmation: :required` flow renders the **normal StockSage confirmation**
     (itself an ephemeral/approval surface). **Only after approval** does
     analysis execute and emit durable canvas tiles (v0.30 path).
   - **Decline** (`#intent-handoff-decline`): `dismiss_workspace_ephemeral`
     clears the surface. **No `active_app` change, no confirmation, no action.**
   - **Clarify** (when `ticker` missing or candidates are close): the engine
     returns `:clarify_intent`; the operator's reply re-enters
     `submit_user_input/1`, fills the slot or selects a `[data-intent-option]`,
     and proceeds to the accept path above.
9. **Direct-answer branch (from step 5).** If no descriptor clears
   `intent.clarify_floor`, the engine returns `:direct_answer`;
   `Actions.Intent.DirectAnswer` produces a **model-backed answer** when enabled,
   else a **deterministic bounded fallback** — never a tool/app/resource call.

Authority invariants across the whole path: descriptors and ranking are
proposals only; `active_app` is set only by an explicit operator accept;
app-scoped execution still requires matching `active_app` at the runner; and
confirmation floors are never lowered (ADR 0019, ADR 0021, ADR 0034, v0.28).

## Flow 1: Plain Direct Answer

Operator asks a general question in `/workspace`.

Expected behavior:

1. Runtime selects the side-effect-free direct-answer path.
2. Direct answer uses the configured answer model boundary when available. The
   model call is gated by Settings Central (the operator decides the default;
   it follows the existing `intent.model_assist_enabled` / `intent.model_profile`
   precedent of being Settings-gated rather than implicitly on), and its prompts
   and completions are redacted in traces and audits per ADR 0019.
3. The model call grants no Resource Access escalation: it is a bounded answer
   call only, never a tool, app action, browser, shell, package, skill, memory
   write, or other network-resource request.
4. No app handoff, action confirmation, memory write, resource request, or
   shell/package/skill execution occurs.
5. If the model is disabled, unavailable, or fails, Allbert returns a
   deterministic bounded fallback that says no action was taken.

Failure behavior:

- Provider failure must not produce hidden tool calls or app handoffs.

## Flow 2: Neutral StockSage Handoff

Operator opens `/workspace` with neutral app context and asks:

`analyze CIEN`

Expected behavior:

1. Intent collection sees the StockSage `run_analysis` descriptor.
2. Allbert renders an app handoff proposal:
   `StockSage can run a financial analysis for CIEN. This requires
   confirmation. Run it?`
3. No StockSage action executes yet.
4. No StockSage confirmation exists until the operator accepts the handoff.

Failure behavior:

- A direct `run_analysis` call without `active_app: :stocksage` is still
  denied by `Actions.Runner`.

## Flow 3: Accept Handoff

Operator accepts the StockSage handoff.

Expected behavior:

1. The handoff sets explicit app context through the existing registered
   session/action boundary.
2. Runtime invokes `run_analysis` with `active_app: :stocksage`.
3. StockSage normal confirmation appears.
4. Only after confirmation approval does StockSage analysis execute.

Failure behavior:

- If app context cannot be set, the action is not attempted.
- If Security Central requires confirmation, the system cannot skip it.

## Flow 4: Decline Handoff

Operator declines the StockSage handoff.

Expected behavior:

1. Active app context remains unchanged.
2. No pending StockSage confirmation is created.
3. The conversation records the decline as side-effect-free assistant state.

## Flow 5: Missing Slot Clarification

Operator asks:

`analyze`

Expected behavior:

1. Allbert recognizes a plausible analysis intent.
2. Required slot `ticker` is missing.
3. Allbert asks a targeted question such as:
   `Which ticker should StockSage analyze?`
4. Slot extraction is descriptor-declared and conservative: Allbert never
   guesses a ticker from free text. If a required slot is not unambiguously
   present, it asks rather than assuming.
5. No app context is set and no action confirmation is created until the
   operator answers and accepts handoff.

## Flow 6: Ambiguous App Intent

Operator asks a request that plausibly matches multiple app capabilities.

Expected behavior:

1. Allbert compares deterministic descriptor scores and top-two margin.
2. If ambiguous, it presents concise choices rather than picking silently.
3. Choosing one option proceeds through the same handoff path.

## Flow 7: Classifier Disabled

`intent.model_assist_enabled=false`.

Expected behavior:

1. Deterministic descriptor matching still works.
2. Clear cases can still produce app handoff.
3. Ambiguous cases still produce clarification.
4. The classifier is not called.

## Flow 8: Classifier Enabled But Invalid

Classifier returns low confidence, timeout, malformed output, or unknown
candidate id.

Expected behavior:

1. The classifier output is rejected as advisory-only.
2. Deterministic descriptor matching or direct answer proceeds.
3. Raw classifier prompt/completion is not rendered in traces, UI, logs, or
   audits.

## Flow 9: In-App Context Still Works

Operator selects StockSage explicitly from the v0.32 app launcher, then asks:

`analyze CIEN`

Expected behavior:

1. The workspace already has `active_app: :stocksage`.
2. The request routes directly to the normal `run_analysis` confirmation.
3. No extra neutral handoff question is required.

## DOM Ids And Events

Stable handles for tests and Chrome manual verification:

| Surface | Handle | Expected use |
|---|---|---|
| Direct answer | workspace chat/thread message node | Side-effect-free answer in the center thread |
| Handoff/clarify surface | `#workspace-node-<surface-id>` (v0.26 ephemeral) | Ephemeral surface wrapper for the proposal |
| Handoff card | `#intent-handoff` | App handoff proposal |
| Accept handoff | `#intent-handoff-accept` | Set app context and re-submit with slots |
| Decline handoff | `#intent-handoff-decline` | Dismiss ephemeral; leave context unchanged |
| Clarification | `#intent-clarification` | Missing-slot or ambiguity prompt |
| Candidate option | `[data-intent-option]` | Ambiguous candidate choice |

## Smoke Checklist

- Plain question produces useful direct answer, not stale static echo.
- Neutral `analyze CIEN` renders handoff, not `run_analysis` execution.
- Accept handoff reaches normal StockSage confirmation.
- Decline handoff creates no confirmation.
- `analyze` asks for ticker.
- Invalid classifier output cannot select or invent an action.
- Direct runner call to StockSage without `active_app: :stocksage` remains
  denied.
