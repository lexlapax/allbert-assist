# Allbert v0.26a UX/UI Substrate Plan

Status: implemented through M35 closeout and merged into the `0.26.1`
release. Planned on 2026-05-21; release closeout recorded on 2026-05-22.

This document is the planning artifact for **v0.26a — Workspace
UX/UI Substrate Pass**, a fast-follow visual + interaction layer
on top of the v0.26 workspace substrate (`docs/plans/v0.26-plan.md`).
v0.26 shipped a structurally-correct workspace (Surface tree,
catalog, persistence, fragments, signals, AGUI bridge, offline
editing) and a passable design system (AppBar, light/dark, tile
cards). What it did NOT ship is an operator-grade interaction
layer: live chat history, working keyboard semantics, approval
ephemerals from confirmations, tile inspection, mobile responsive,
or rich content rendering.

v0.26a closes those gaps so v0.27 (StockSage LiveViews) and
beyond have a real foundation, not a debug-card foundation.

## Substrate Boundary (what v0.26a will NOT touch)

The v0.26 substrate is good enough to keep:

- Catalog (42 components), persistence (3 SQLite tables + YAML),
  FragmentEnvelope (HMAC + 6 validators), 14 settings, 9 signals,
  `:workspace_canvas_write` permission, AGUI bridge mapping.
- Workspace facade (`Canvas`, `Ephemeral`, `Fragment`, `Catalog`,
  `Offline`, `AGUI.Bridge`, `Emitters`).
- M21–M27 remediation closures (thread binding, runtime emitters,
  receiver hardening, authority boundary, ephemeral lifecycle
  signals, settings enforcement, doc truth sync).

v0.26a is a **renderer + LiveView interaction pass** on top of
that substrate. It does NOT change schema, settings keys, signal
topics, or the catalog allow-list. It is purely:

- `apps/allbert_assist_web/lib/allbert_assist_web/live/agent_live.ex`
  + its tests
- `apps/allbert_assist_web/lib/allbert_assist_web/workspace/components/*.ex`
  + their tests
- `apps/allbert_assist_web/assets/css/app.css` (tokens + layout)
- Possibly one or two new helper modules (e.g. a small JS hook for
  copy-to-clipboard, a focus-trap helper for ephemerals)
- One new emitter wiring in `apps/allbert_assist/lib/allbert_assist/workspace/emitters.ex`
  to fire `:approval_card` ephemerals when confirmations are durable
  (closing the M22 gap surfaced in this audit)

## Validation Method

Browser walk-through on 2026-05-21 against a freshly-bootstrapped Allbert with
`ALLBERT_HOME` in `/tmp/`,
`external_services.enabled=true`, `allowed_hosts=[wikipedia.org]`.
Tested at 1440×900 and an attempted 375×800.

End-to-end smokes that actually ran:

1. Initial empty `/agent` render → workspace shell + AppBar + Chat
   + Canvas, all visible, no Phoenix scaffold leaks.
2. Generic chat: `"Hello Allbert, who are you?"` → routes to
   `direct_answer` action, response renders with status + signal id.
3. Setting-shaped prompt: `"Set workspace.theme to dark"` →
   falls through to `direct_answer` (no LLM API key; expected
   behavior).
4. External fetch: `"Fetch https://en.wikipedia.org/..."` → routes
   to `external_network_request`, correctly denied by allow-list
   (`en.wikipedia.org` not in `wikipedia.org` allowlist).
5. Memory write: `"Remember that AAPL is my favorite ticker"` →
   action runs, file persists at
   `<ALLBERT_HOME>/memory/notes/...md`, chat reports path.
6. Shell command: `"Run ls -la in my home directory"` → correctly
   denied at safety floor.
7. Theme toggle in AppBar → instant dark mode, no flash.
8. AppBar overflow `...` → no menu opens; control is decorative.
9. AppBar chips (thread, app, objective count, tile count,
   ephemeral count) → all decorative; no click handlers.
10. Active-app routing: `/agent?app_id=stocksage` → chip updates to
    `stocksage`; routing param honored.
11. StockSage `"analyze AAPL"` → real objective created, confirmation
    durably persisted, chat shows `needs_confirmation`; **NO
    approval ephemeral emitted**; AppBar shows `1 objective | 1
    tile | 0 ephemerals` (the 0 is the bug).
12. CLI approval (`mix allbert.confirmations approve …`) → objective
    transitions to `:running` then `:failed` (no LLM key); tile in
    canvas does NOT update its visual state to reflect lifecycle.
13. Bad thread param: `/agent?thread_id=thr_nonsense_invalid` →
    soft amber banner, URL rewritten to a fresh `thr_*`. ✓
14. Window resize to 375×800 → layout unchanged; mobile breakpoint
    does not engage. ✗

## Findings — Categorized

Severity:
- **CRIT** — blocks confident operator use today
- **HIGH** — blocks v0.27 building on this substrate
- **MED** — operator-facing but workaroundable
- **LOW** — polish / cosmetic

### A. Critical UX Bugs (CRIT)

**A1. Live chat shows only the latest turn.**
After a turn submits, the chat region renders only the current
`You: …` + `Allbert: …` pair. All prior turns disappear until the
operator navigates or reloads, at which point the full history
reappears with timestamps. This is a major break in conversational
state.

Cause: `agent_live.ex` populates `assigns.conversation_messages`
from `Conversations.list_messages(thread, limit: 12)` at mount but
the template (`workspace/components/chat.ex`) renders only
`@prompt` + last assistant response in the live state. After submit,
the message list is not refreshed before the template re-renders.

Fix: re-fetch (or PubSub-driven append) `conversation_messages` on
every `:after_turn` event; render the full timeline in
`workspace-chat-timeline`; remove the special-case "Prompt draft"
rendering for the just-submitted turn.

**A2. Composer does not clear after submit.**
The textarea retains the prompt text after submission. Standard
chat UX clears the composer; operators submit the same prompt twice
by mistake. Verified twice during the audit walk-through.

Fix: reset `assigns.prompt = ""` in the `:after_turn` handler
before re-render; clear the underlying `phx-change` form value.

**A3. Enter does not submit.**
The composer footer reads `Enter submits. Shift+Enter adds a line.`
Pressing Enter actually inserts a newline. Promise vs behavior
divergence.

Fix: add `phx-key="Enter"` capture on the textarea or a JS hook
that intercepts Enter (without Shift) and dispatches the submit;
keep Shift+Enter for newline.

**A4. No `:approval_card` ephemeral emitted for confirmations.**
M22 plan says: "Emit an `:approval_card` ephemeral when a durable
confirmation is created for a workspace-threaded request." Verified
in this audit: `analyze AAPL` created a real confirmation, an
objective tile appeared, the chat showed
`Started objective Analyze AAPL. Confirmation required for step …
conf_…`, and the AppBar updated to `1 objective | 1 tile | 0
ephemerals` — but no ephemeral was emitted. Operators must drop to
CLI to approve. This breaks Flow 2 of `v0.26-request-flow.md` and
makes the workspace unusable for any flow that needs confirmation.

Fix: wire `Confirmations.Store.Persistence.persist_pending/1` to
also call `Workspace.Emitters.emit_approval_card/1` when the
confirmation has a `thread_id` matching a workspace thread. The
ephemeral should carry confirmation id + action target + risk tier
+ reason placeholders, with `phx-click` handlers wired to
`approve_workspace_confirmation` / `deny_workspace_confirmation`
registered actions (M24-class — wire through `Actions.Runner`).

**A5. Whole page scrolls instead of independent panes.**
When chat history grows past the viewport, the entire document
scrolls — AppBar disappears, Canvas pane also scrolls away. Modern
operator surfaces use sticky header + independently scrolling
panes.

Fix: in `app.css`, make the workspace root a 100vh flex column
with `header` fixed-height and the two-pane region using
`overflow: hidden` on the container, `overflow-y: auto` on each
pane individually. Canvas tile list scrolls inside its own pane;
chat timeline scrolls inside its own pane.

### B. Functional Gaps (HIGH)

**B1. AppBar chips are decorative.**
Every chip in the AppBar — `thr_xxx`, `allbert`/`stocksage`,
`N objectives`, `N tiles`, `N ephemerals` — has hover affordance but
no click handler. Operators expect:

- Click thread chip → thread switcher modal/menu (list of recent
  threads in this user's history; option to start a new thread)
- Click active-app chip → active-app picker (registered apps from
  `App.Registry`; sets `session.active_app` via the
  `set_active_app` action so routing follows)
- Click `N objectives` → opens `/objectives` (sibling LiveView)
- Click `N tiles` → focuses canvas pane (or opens a tile list with
  filter UI when tile count >> 1)
- Click `N ephemerals` → focuses first ephemeral, or opens a list
  of pending ephemerals

Fix: implement each chip's click handler as a registered action or
LiveView navigate. Use `phx-click` + `:navigate` for cross-route;
keep effectful changes (active app) through `Actions.Runner`.

**B2. Overflow `...` menu is unwired.**
The kebab in the AppBar's far right opens nothing. Either remove
it or wire it to a useful menu (Profile / Sign out / Workspace
settings / Help / Diagnostics / `mix allbert.workspace inspect`
result viewer).

Fix: implement a minimal overflow menu with at least: Theme cycle
(light/dark/system), Workspace settings shortcut (sibling route),
Diagnostics → opens a debug tile with the resolved Surface tree
(equivalent of `mix allbert.workspace inspect`).

**B3. Tile body click does nothing.**
Clicking the body of an objective tile does not expand, navigate,
or inspect. The only interactivity is the pin button + the kebab
("Unpin tile" / "Remove tile"). Operators can't drill into a tile
to see full content.

Fix: clicking the tile body (or a dedicated "expand" affordance)
opens an inline expanded view OR an ephemeral inspector. For
objectives, the inspector shows objective text, acceptance criteria,
constraints, current step, last observation summary, and links to
`/objectives/:id`. For other tile kinds, the inspector renders the
tile body YAML pretty-printed.

**B4. Tile kebab is sparse.**
Currently: Unpin / Remove. Add: Copy id, Copy fragment payload,
Open trace, Inspect, Export YAML. Each is a 5-line addition + one
registered action where effectful.

**B5. Mobile responsive does not engage.**
Resizing to 375×800 leaves the desktop two-pane layout intact. The
plan claims "two-pane above 768px; single-pane with tab toggle
below." Verified broken.

Fix: confirm the `@media (max-width: 767px)` rule actually fires;
the CSS may be present but the test viewport may not be honoring
it. Add a Wallaby (or LiveView viewport sim) test asserting that
at 375px the chat and canvas are stacked vertically and a tab
toggle is present.

### C. Tile Rendering Shallowness (HIGH for v0.27)

v0.27 will add 22+ StockSage card kinds (`analysis_card`,
`agent_report_card`, `parity_card`, `debate_round_card`,
sub-renderers for fundamentals/news/risk/synthesis). The current
tile renderer is unprepared.

**C1. Objective tile body is truncated metadata.**
The objective tile shows `objective_card tile objective_ob…`
(truncated id) and `emitter AllbertAssist.Objectives 2026-…`. No
objective text. No acceptance criteria. No current step. No status.

Fix: implement
`AllbertAssistWeb.Workspace.Components.ObjectiveCard` with a real
body: title (already there), objective text, status pill,
current step name, progress bar / step count, last observation
summary. Footer keeps emitter + timestamp. Match the catalog atom
contract — no new schema.

**C2. No lifecycle state on tiles.**
Created / running / blocked / completed / failed / abandoned not
visually indicated on any tile. The chat header shows a `blocked`
badge but the objective_card tile doesn't.

Fix: every tile gets a small status pill in its header row, mapped
from `tile.metadata.status` (or, for objective_card tiles, the
linked objective's status). Colors per the v0.26 token palette
(green=completed, amber=blocked, red=failed, neutral=running/open,
gray=abandoned/dismissed).

**C3. No expand/collapse on tiles.**
Long content has no in-place expansion. v0.27's StockSage analysis
card will contain 11 specialist agent reports + a debate transcript
+ parity diff. Today's tile is a single rigid card.

Fix: every tile gets a collapse/expand toggle in the title bar
(LiveComponent local state, no signal needed). Default state is
collapsed showing 3-line summary; expanded shows full body. Tile
height respects `workspace.canvas.tile_body_max_bytes` cap on
content size, not on rendered height.

**C4. Tile body is verbatim text from action result.**
Memory write returned a long path verbatim. No clickable links,
no copy buttons, no markdown rendering, no code highlight, no
diff rendering, no agent-report structure. This breaks every
visual contract v0.27 will need.

Fix: tile body kinds get pluggable renderers:
- `text/plain` (current default) — clean monospace with copy button
- `text/markdown` — render via Earmark / MDEx with syntax highlight
- `text/diff` — `:diff` catalog atom renderer (already exists per
  v0.26 plan; wire it in)
- `application/json` — collapsed tree view with line numbers
- `link` — clickable anchor with target preview tooltip

Render is driven by `tile.metadata.body_kind` (already in schema),
with a sensible default and explicit emitter opt-in.

**C5. No keyboard navigation between tiles.**
Tab/arrow keys don't move focus between tiles. v0.27 WCAG audit
will fail.

Fix: tiles are focusable (`tabindex="0"`), arrow keys move focus
between siblings, Enter activates the primary action (expand /
inspect), Escape collapses.

### D. Composer & Chat Polish (MED)

**D1. "Prompt draft" label is wrong post-submit.**
After submission, the user message still shows the "Prompt draft"
mini-label. It should disappear or change to "You · just now /
timestamp."

**D2. "Status: completed / Signal: 019…" rendered as plain text
in every response.**
Every assistant turn ends with these two lines in plain text.
They're useful for ops but visually noisy.

Fix: render as a small chip row at the bottom of the assistant
bubble (status pill + monospace signal-id chip that's
copy-on-click), collapsed behind a chevron when status is
`completed`.

**D3. No "thinking" stream / progress indicator.**
The Ask button briefly says "Thinking" but the UI freezes for
~5–10s. No token streaming, no step-by-step "calling intent agent
→ selected `direct_answer` → applying redaction → recording trace."

Fix: subscribe to `allbert.runtime.turn.*` and
`allbert.action.requested|completed` signals for the active turn;
render a transient progress row above the assistant bubble showing
the latest sub-step name. Disappears on `turn.completed`.

**D4. No copy buttons anywhere.**
No way to copy trace ids, signal ids, memory paths, objective ids,
fragment ids, tile ids without manual text selection.

Fix: every monospace id token in the workspace gets a copy button
on hover (no JS framework — a tiny phx-hook with the Clipboard API
+ a "Copied!" toast).

**D5. The "Ask Allbert to start a runtime turn…" empty-state
banner persists after the first turn.**
Once there's chat history, the empty-state instructional banner is
unnecessary and visually noisy.

Fix: hide the empty-state banner when
`length(@conversation_messages) > 0`.

### E. AppBar / Header Polish (MED)

**E1. "Allbert Assist / Runtime, canvas, and ephemeral workspace."
header subtitle is verbose.**
Header reads like marketing copy. Operators don't need it on every
view.

Fix: replace the subtitle with the active thread + active app
chip on a single line; relegate the descriptive text to a tooltip
on the wordmark, or only show on the empty workspace.

**E2. Theme toggle is binary (light ↔ dark only).**
No "system" option in the toggle UI, despite `workspace.theme`
schema accepting `["light", "dark", "system"]`. Operators expect a
three-state cycle.

Fix: cycle through light → dark → system on click; show the icon
matching current resolved state with a small tooltip "Theme:
light/dark/system."

**E3. No "active turn" indicator in the AppBar.**
While a turn is mid-flight, the AppBar gives no signal. The "Ask"
button changes locally but the AppBar — which the operator's eye
defaults to — is static.

Fix: small pulsing badge or progress bar in the AppBar when a turn
is in flight; clears on `turn.completed`.

### F. Foundations for v0.27 Extensibility (HIGH)

**F1. No tile categorization / filter / search.**
When 30+ tiles accumulate, the canvas is a vertical wall. v0.27
will add 5–10 tiles per StockSage analysis.

Fix: above the tile list, a small chip-row filter ("All / Pinned /
Objectives / Analyses / Confirmations") + a search input that
filters by title/body substring. Driven by client-side filtering
of `@canvas_tiles`; no new schema.

**F2. No tile grouping by source.**
A StockSage analysis emits objective_card + analysis_card +
agent_report_card × 11 + parity_card + debate_round_card. They
should visually group under the originating objective.

Fix: tiles with the same `metadata.objective_id` render under a
collapsible group header showing objective title + status. Pure UI
grouping, no new emitter contract needed.

**F3. No ephemeral overlay scaffolding.**
The renderer for `:ephemeral_surface` exists but no actual ephemeral
overlay has been rendered in the audit. Confirmations don't emit
them (A4); no other emitter does either.

Fix: implement the ephemeral overlay UI: modal-style (centered,
backdrop) for blocking flows; right-docked panel for non-blocking.
Focus-trap on open, restore on close, Escape to close (M25 wired).
Verify the path works with a test that emits an `:approval_card`
ephemeral, asserts it renders, dismiss + assert restore.

**F4. No `:tabs` / `:tab` / `:tab_panel` interaction.**
The renderers exist but are never rendered in the audit. v0.27
analysis tiles will use tabs for Overview / Steps / Trace.

Fix: implement actual tab UI (underline-active, keyboard nav with
Arrow + Home/End, `aria-controls` / `aria-selected`) and a snapshot
test in the renderer suite.

**F5. No `:trace_link` / `:trace_viewer` interaction.**
v0.27 tiles will deep-link to traces. Today these renderers exist
in the catalog but no flow renders them.

Fix: when a tile carries `metadata.trace_id`, render a trace_link
pill in the footer. Clicking opens the trace_viewer either as an
ephemeral inspector or in a sibling route (decide in M28a-impl).

### G. Polish (LOW)

**G1.** Empty state in canvas ("No canvas tiles yet / Workspace
tiles will appear here as runtime fragments land.") is fine; could
add a 2-line hint about how to trigger a tile (`mix allbert.ask` /
type a prompt + active app).

**G2.** The `1/64 tiles` badge in the canvas header doesn't change
color as it approaches cap. At 56/64 it should turn amber; at
64/64 it should turn red and pulse.

**G3.** Timestamp format `2026-05-21T15:26:32.854952Z` is too
precise for chat. Render as relative ("just now / 2m ago / 14:32")
with full ISO on hover.

**G4.** No way to delete a thread from the UI. Operators must drop
to `mix allbert.threads complete <id>` or DB.

**G5.** No multi-tab sync verified during the audit (couldn't
open two tabs in the controlled Chrome session). Re-verify when
A1/A2 fixes land.

### H. Bugs to Flag Separately (NOT v0.26a scope)

These were not UX/UI substrate issues; they were agent / runtime / intent
issues flagged for separate triage during the v0.26a audit:

Closeout note: these remained outside v0.26a ownership and were resolved by
the sibling `docs/plans/v0.26b-backend-plan.md` backend pass before the
`v0.26.1` release tag.

- **H1 (runtime):** Intent agent falls through to `direct_answer`
  for clear setting-shaped prompts ("Set workspace.theme to dark")
  when no LLM API key is configured. The deterministic fallback
  should at least pattern-match `set <key> to <value>` and route
  to `update_setting` with a confirmation. Currently every prompt
  that isn't URL-shaped becomes `direct_answer`. Resolved in v0.26b.
- **H2 (stocksage):** Native StockSage analysis fails with no
  visible reason in the chat when LLM is unavailable. The objective
  silently transitions to `:failed`. Should emit an ephemeral or
  chat message with the failure reason. Resolved in v0.26b.
- **H3 (runtime):** Default LiveView prompt placeholder reads
  `Hello Allbert. What can you do right now?` — this is a content
  copy, not a prompt placeholder. Move it into the `:empty_state`
  card and leave the textarea placeholder more neutral
  ("Ask Allbert anything…"). Resolved in v0.26b.

## Milestone Structure

v0.26a is structured as 8 flat milestones grouped into 4 phases.
Each milestone has its own acceptance + `mix precommit` gate.
Phase headings are editorial only; the unit of work is the flat
M-number.

```
M28 (chat history live)  ──┐
M29 (composer fixes)       ├──> M32 (approval ephemeral) ──> M35 (operator validation)
M30 (sticky panes)         │                                  │
M31 (mobile responsive)    │   M33 (tile body + inspector)    │
                           │   M34 (AppBar interactions)      │
                           └─────────────────────────────────┘
```

Day-1 startable in parallel: **M28, M29, M30, M31, M34** — each
edits a different part of the LiveView/CSS surface.

### Phase 1: Critical UX Bugs

#### M28 — Live Chat History + Conversation Refresh

Goal: chat region always shows full history; new turns append, do
not replace.

Depends on: nothing.
Parallel-safe with: M29, M30, M31, M34.

Implementation tasks:

- In `agent_live.ex`, refresh `assigns.conversation_messages` on
  every successful turn (`{:after_turn, …}` or PubSub
  `allbert.runtime.turn.completed`).
- In `workspace/components/chat.ex`, render the full timeline
  (`@conversation_messages`) above the composer; drop the
  special-case render of `@prompt` once submitted.
- Auto-scroll timeline to bottom on append (LiveView hook).
- Hide the `Ask Allbert to start a runtime turn…` instructional
  banner when timeline length > 0 (D5).
- Replace "Prompt draft" label with `You · {relative_time}` (D1).
- Render status + signal id as collapsible chips (D2).

Tests:
- LiveView assert: submitting 3 turns leaves all 3 visible without
  navigation.
- LiveView assert: timeline auto-scrolls to last entry.

Acceptance: turn → bottom, history accumulates, no stale "Prompt
draft" label.

#### M29 — Composer Reset + Enter-to-Submit

Goal: composer clears on submit; Enter submits, Shift+Enter
newlines.

Depends on: nothing.
Parallel-safe with: M28, M30, M31, M34.

Implementation tasks:

- Reset `assigns.prompt = ""` in the `:after_turn` handler before
  re-render.
- Add a JS hook (`ComposerHook`) on the textarea that intercepts
  Enter (without Shift) and dispatches the form submit; preserves
  Shift+Enter and IME composition.
- Make the textarea respect the `tile_body_max_bytes` cap and show
  a soft counter in the footer ("142 / 65536 characters").

Tests:
- LiveView test: submit a prompt; assert textarea empty.
- JS hook test (or hand-driven): Enter submits, Shift+Enter
  newlines.

Acceptance: composer behaves like a modern chat composer.

#### M30 — Sticky AppBar + Independent Pane Scroll

Goal: AppBar stays visible; chat and canvas scroll independently.

Depends on: nothing.
Parallel-safe with: M28, M29, M31, M34.

Implementation tasks:

- In `app.css`, make `body > .workspace-shell` a 100vh flex column;
  AppBar fixed height (56px desktop, 48px mobile); main region uses
  `min-height: 0` + `display: flex` so panes can each
  `overflow-y: auto`.
- Test at scroll depths > viewport that AppBar never leaves view.

Tests:
- LiveView snapshot at 1440×900 after appending 30 messages: assert
  AppBar at viewport top.

Acceptance: scrolling chat does not move AppBar; scrolling canvas
does not move AppBar; chat and canvas scroll independently.

#### M31 — Mobile Responsive Actually Engages

Goal: at < 768px viewport, the workspace stacks chat + canvas with
a tab toggle.

Depends on: M30 (sticky shell layout).
Parallel-safe with: M28, M29, M34.

Implementation tasks:

- Audit the existing `@media (max-width: 767px)` CSS in
  `app.css`. Fix what doesn't apply.
- Add a `[Chat] [Canvas] [Ephemerals]` tab toggle in the workspace
  shell below the AppBar, shown only below the breakpoint.
- Each tab swaps which pane is visible (CSS-driven; no JS state
  beyond a selected tab assign).
- Test with Wallaby or LiveView viewport simulation at 375px,
  768px, 1280px.

Tests:
- Wallaby test at 375px: chat is full-width, canvas hidden until
  tab clicked.
- Wallaby test at 1280px: both panes side-by-side.

Acceptance: mobile breakpoint engages and is verifiable in the
test suite.

### Phase 2: Approval Flow End-to-End

#### M32 — Approval Card Ephemeral Wiring

Goal: when a durable confirmation is created with a workspace
`thread_id`, an `:approval_card` ephemeral emits and renders;
operator can approve / deny from the workspace.

Depends on: M28 (live chat refresh — same channel as ephemeral
broadcast).
Parallel-safe with: M33.

Implementation tasks:

- In `apps/allbert_assist/lib/allbert_assist/workspace/emitters.ex`,
  add `emit_approval_card/1` that builds an `:approval_card`
  ephemeral FragmentEnvelope with confirmation id, action target,
  risk tier, reason placeholders.
- Wire `Confirmations.Store.Persistence.persist_pending/1` (or the
  matching durable-write path) to call the emitter when the
  confirmation has a workspace-thread `thread_id`.
- In `workspace/components/structural.ex` (or `allbert_domain.ex`),
  implement the `:approval_card` LiveComponent: full action context,
  redacted parameters in mono, "Approve with reason ___" / "Deny
  with reason ___" twin CTAs.
- Wire the CTAs to registered actions
  (`approve_workspace_confirmation`, `deny_workspace_confirmation`,
  M24-class) so the authority boundary is preserved.
- Implement the ephemeral overlay UI from F3 (modal-style;
  focus-trap; Escape closes-rejects-or-cancels per Flow 2).
- Emit `:closed` lifecycle signal on approve/deny (M25 already
  wires this for `Ephemeral.dismiss/3`).

Tests:
- Integration test: create a durable confirmation with a workspace
  thread_id → assert an `:approval_card` ephemeral exists in DB +
  LiveView renders it.
- LiveView: click Approve with a reason → assert confirmation
  status `:approved`, ephemeral dismissed.
