## Main User/Operator Flows

Discovery:

- Operator enables discovery (`mcp.discovery.enabled`) and optionally resumes the
  background scan job; scans write candidates to the passive Discovery
  Suggestions surface.
- Operator asks "find a tool/server for X"; `find_tools` fans out to
  `find_local_tools` (actions/skills/configured-MCP) and `find_mcp_tools`
  (official registry plus optional keyed PulseMCP when configured) in parallel
  and returns one ranked list.
- Operator reviews a remote candidate, runs `mcp_server_connect`, sees the exact
  command/URL in a consent, and approves; the server is written to
  `mcp.servers.<id>` with a tool-definition baseline hash.

Integration:

- Operator configures an MCP server for calendar/mail/GitHub via Settings Central
  per v0.40 paths (or via the discovery connect gate above).
- Operator opens the calendar/mail/GitHub workspace panel and sees
  MCP-exposed tools/resources.
- Operator asks the workspace for agenda, inbox summary, GitHub overview, or
  notes search; v0.33 intent descriptors route to the right integration.
- Operator approves any effectful integration action through normal Approval
  Handoff.
- Operator (plugin author) reads `./plugins/allbert.notes_files/` as a
  worked-example starter scaffold.

## Data Flow

Discovery requests route through `find_tools`, which runs the local source
(`Actions.Registry.capabilities/0`, `Skills.Registry.list/1`, configured-MCP
`tools/list`) and the MCP-registry source (cascade: `discover_servers →
mcp_fetch_server_manifest → mcp_evaluate_server` across the official registry
and optional keyed PulseMCP behind `AllbertAssist.Mcp.Registry.Provider`) in
parallel, then merges/ranks into `ToolCandidate`s. All registry/server egress
flows through `External.HttpClient`/`HttpPolicy` (SSRF, bounded timeout/body,
redirect denial, redaction); the official registry is a non-durable cache and
discovery degrades to local-only when a source is unreachable. Missing optional
PulseMCP secrets disable only that provider. Remote candidates, evaluation
reports, suggestions, and baseline trust records persist through
`AllbertAssist.Tools.Discovery`. A `:remote_mcp` candidate is inert until
`mcp_server_connect(candidate_id: ...)` runs a pre-config consent
(`Confirmations.create/2`/`resolve/4`) and, on approval, writes
`mcp.servers.<id>` and stores a baseline hash. Background scanning is an
`AllbertAssist.Jobs` job (paused by default) writing persisted candidates to a
passive `AllbertAssist.Surface` panel; it never messages the operator and never
connects.

M2 local discovery flow is implemented. `find_local_tools` invokes
`AllbertAssist.Tools.Source.Local` and returns immediately usable
`ToolCandidate`s for registered actions, trusted/enabled skills, and tools from
already enabled MCP servers. `find_tools` invokes `AllbertAssist.Tools.Finder`,
runs enabled sources in parallel, dedupes by `{source, name}`, ranks local
matches before any future remote source, and currently enables only the local
source. Therefore v0.42 M2 performs no internet MCP-registry I/O. The CLI surface
`mix allbert.tools find "<need>"` calls `find_tools` and prints `source`,
`usable_now`, and `requires` for each candidate. The deterministic IntentAgent
route now maps "what tools..." / "find a tool..." prompts to `find_tools` while
preserving existing configured-MCP "list tools on SERVER" routing.

Integration requests route through intent descriptors, MCP-configured workspace
panels (driven by the v0.40 MCP client) for calendar/mail/GitHub, or the native
`notes_files` plugin for notes operations. All effectful work flows through
`Actions.Runner.run/3`, Security Central, Resource Access, confirmations, traces,
and audits.

v0.42 consumes the v0.40 first-server validation matrix (shipped in v0.40 M5).
Calendar/mail/GitHub panel refreshes may run without repeated prompts only when
the needed data is available through `mcp_read_resource` and an operator-approved
`mcp://` Resource Access grant. Calendar is tool-only per the matrix, so the
agenda panel uses per-call-confirmed tool reads or an empty/"configure a
resource-exposing server" state — it does not promise prompt-free refresh. If a
server exposes required summary data only through `mcp_call_tool`, that
interaction remains operator-triggered and confirmed per call, is parked, or
requires an ADR 0038 amendment before implementation.

## Authority Invariants

Discovery:

- Registry and server metadata (`server.json`, tool descriptions) are descriptive
  only and never grant authority (ADR 0038 §2, ADR 0048 §2).
- A `:remote_mcp` `ToolCandidate` is `usable_now?: false` and connects only
  through the confirmation-gated `mcp_server_connect`. No auto-connect;
  `mcp.discovery.auto_connect` is pinned `false`.
- Every discovery fetch routes through `External.HttpPolicy`; discovery never
  opens a second network path.
- Reconnect re-verifies the tool-definition baseline hash; a change forces
  re-consent.
- Background scanning is opt-in, paused by default, and writes only to a passive
  surface; no unprompted messaging.

Integration:

- Integrations do not live in Allbert core.
- MCP-configured panels inherit the v0.40 MCP client trust tier (ADR 0038).
- `:mcp_resource_read` grants never authorize `mcp_call_tool`, even for
  read-like tools.
- Native `notes_files` plugin owns only its declared surfaces/actions/settings
  fragment/memory namespace and never grants permission by registration.
- The `notes_files` memory namespace cannot auto-promote; promotion is a
  separate confirmed action.
- Intent descriptors propose handoff; they do not authorize execution.

## Smoke Checklist

Discovery:

- Enable discovery; run `mix allbert.mcp discover` and confirm remote candidates
  are `usable_now?: false`.
- Run `find_tools` and confirm local + remote candidates interleave with the
  usable-now vs requires-connect distinction.
- Connect a candidate: deny once (writes nothing), approve once (writes
  `mcp.servers.<id>`); confirm the consent showed the exact command/URL.
- Point a discovery fetch at a private IP and confirm SSRF denial.
- Make the registry unreachable and confirm discovery degrades to local-only.
- Enable PulseMCP without its configured secret refs and confirm only that
  provider is skipped while official-registry discovery still works.
- Run `scan run-once` and confirm suggestions land on the passive surface with no
  proactive message.

Integration:

- Configure at least one MCP server per integration with fake/test credentials.
- Run one read-only flow per integration (agenda, inbox, PR list, notes search),
  and record whether it used `mcp_read_resource` or a per-call-confirmed MCP tool.
- Run one confirmed effectful flow per integration (event, reply, comment, note
  write).
- Confirm workspace panels render without leaking secrets.
- Confirm disabled integration actions fail closed.
- Open `./plugins/allbert.notes_files/` and verify it is documented as the
  reference scaffold (`docs/developer/how-to-create-an-allbert-app.md` cross-
  links to it).

## Operator Validation

Status: planned. Commands are illustrative until the milestones land; verify and
replace with observed output at closeout.

```sh
export ALLBERT_HOME="$(mktemp -d /tmp/allbert-v042.XXXXXX)"
export ALLBERT_TRACE_ENABLED=true
# Discovery
mix allbert.settings set mcp.discovery.enabled true
mix allbert.mcp discover "calendar"
mix allbert.mcp connect "io.example/calendar"   # approve; shows exact command/URL
mix allbert.ask --trace "find me an MCP server for github"
mix allbert.mcp scan run-once
# Integration (configure test MCP servers per integration, then exercise panels)
mix allbert.ask --trace "show me today's agenda"
mix allbert.ask --trace "summarize my inbox"
mix allbert.ask --trace "list my open PRs"
mix allbert.ask --trace "find notes about onboarding"
PORT=4041 mix phx.server   # /workspace: suggestions panel + integration panels
rg -i 'token|secret' "$ALLBERT_HOME/settings" "$ALLBERT_HOME/memory/traces" || true
```

Expected behavior:

- discovery lists remote candidates as not-yet-usable; connect runs a consent
  showing the exact command/URL and writes `mcp.servers.<id>` only on approval.
- a denied connect leaves no server config or baseline trust record.
- the scan writes suggestions to the passive surface with no proactive message.
- each integration renders a workspace summary panel; resource reads use a
  remembered grant where exposed, and calendar uses per-call confirmation or an
  empty state (no prompt-free promise).
- one effectful action per integration routes through Approval Handoff before
  running.
- `./plugins/allbert.notes_files/` is present as the reference scaffold.
- disabled integrations show an empty state; traces leak no secrets and the final
  `rg` prints no matches.
