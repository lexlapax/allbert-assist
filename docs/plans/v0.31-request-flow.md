# Allbert v0.31 Request Flow - Runtime And UI-Substrate Consolidation

Implementation details live in `docs/plans/v0.31-plan.md`.

v0.31 is intentionally mostly invisible to operators. The request-flow purpose
is to keep behavior stable while implementation seams move underneath the
runtime.

## Operator-Visible Contract

| Area | v0.31 behavior |
|---|---|
| Routes | Same as v0.30: `/agent`, `/settings`, and `/stocksage/*` remain until v0.32 |
| Settings | Same keys, defaults, write confirmations, audit events, and secret redaction |
| StockSage | Same analysis, memory, canvas, queue, trends, and reflection flows |
| Workspace | Same durable tiles, ephemeral surfaces, offline edit behavior, and card rendering |
| Security | Same action-boundary checks, confirmation decisions, remembered grants, Resource Access posture, and eval expectations |
| Dynamic code | Still forbidden; v0.34 will define a sandboxed exception |

## Flow 1: Existing Action Execution Through New Action Facade

1. Operator invokes an existing action from CLI, LiveView, job, objective step,
   or StockSage.
2. The caller still resolves through `AllbertAssist.Actions.Registry`.
3. The runner invokes the migrated action through `AllbertAssist.Action`
   metadata.
4. Security Central receives the same permission, app scope, resource context,
   and confirmation metadata as v0.30.
5. The returned response normalizes through `AllbertAssist.Runtime.Response`
   without changing operator-visible status or copy.

Acceptance:

- Existing action ids keep working.
- Confirmation-required, denied, completed, and error outcomes preserve
  current UX.
- Private Jido command modules remain unavailable as action candidates.

## Flow 2: Settings Read And Write Through Fragments

1. Operator lists, searches, reads, or updates a Settings Central key.
2. Settings Central discovers the key through registered schema fragments.
3. The same validation, default, redaction, secret handling, and confirmation
   logic applies.
4. Writes go through the existing registered settings actions and audit path.

Acceptance:

- Every pre-v0.31 setting remains present.
- Unknown keys are rejected as before.
- Secret values are never rendered raw.
- Fragment metadata never grants permission.

## Flow 3: Surface Rendering Through One Catalog

1. Workspace or StockSage produces a `Surface` tree using existing component
   atoms.
2. The unified catalog validates component atoms, primitive status, app-declared
   catalog metadata, and renderer availability.
3. The web renderer dispatches through the single catalog path.
4. Existing cards render with the same DOM test handles expected by v0.30.

Acceptance:

- Existing StockSage cards render in both released route families during
  v0.31.
- Unknown component atoms fail closed.
- App catalog metadata cannot smuggle a renderer or action authority.

## Flow 4: Extension Contribution Discovery

1. Startup discovers shipped source-tree plugins, apps, actions, skills,
   settings fragments, child specs, and diagnostics.
2. The extension registry normalizes contributions into one inspectable index.
3. App and plugin identities remain distinct, but callers no longer need to
   query several registries for the same contribution class.

Acceptance:

- Existing shipped plugins and StockSage register exactly once.
- Home-plugin manifests remain metadata-only unless already trusted/compiled
  by reviewed source.
- Contribution metadata does not grant trust, permission, routes, or execution.

## Flow 5: Audit, Trace, Path, Persistence, And Redaction

1. Existing runtime work writes traces, audit events, persistence records, and
   redacted UI/CLI output.
2. Callers use the shared facade for the relevant substrate.
3. Existing on-disk files and database rows remain readable.
4. Output redaction remains at least as strict as v0.30.

Acceptance:

- Temporary `ALLBERT_HOME` tests do not touch a real user home.
- Trace/audit records keep stable event vocabulary.
- Redaction covers secrets and Resource Access payloads.

- No Chrome route-change verification is required until v0.32 because v0.31 is
  not a UI reset.
