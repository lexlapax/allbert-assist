# v0.2 Closeout Note

Date: 2026-04-19

By the end of M4, the daemon runtime itself existed and the scheduler worked, but the end-user surface was still split across too many implicit assumptions.

The main closeout gaps were:

- the plan already treated `allbert-cli` as the primary user-facing binary, but the shipped surface still pushed jobs into a separate binary and did not expose daemon lifecycle commands there
- the setup flow still reflected v0.1 and did not ask about daemon auto-spawn or recurring jobs
- the operator docs were still mostly written in v0.1 language even though the runtime model had changed to a daemon-backed client
- parts of the plan over-promised operational polish, especially around fully drained daemon shutdown and live job-failure broadcast

Closeout decisions for v0.2:

- `allbert-cli` is the primary supported end-user surface for v0.2 source-based use
- `allbert-jobs` remains as an alias binary, but not as the primary documented path
- setup must expose daemon auto-spawn and optional bundled-job enablement so the daemon/jobs model is discoverable without manual config editing
- readable synchronous surfaces (`daemon status`, `daemon logs`, `jobs status`, failure JSONL, run history) are mandatory
- live failure broadcast into attached interactive clients is also part of closeout, because it is the one daemon-native operator surface that makes recurring work feel integrated rather than bolt-on
- daemon shutdown in v0.2 should be bounded graceful: stop new work, wait briefly for in-flight work, then cancel remainder and record interruptions clearly

That keeps v0.2 honest without dumbing it down: the project remains end-user usable for the intended technical source-based audience, while the known limitations of the lightweight in-process daemon model stay explicit instead of hidden.
