# v0.5 memory architecture deepening

Date: 2026-04-20

## Why this note exists

The first v0.5 memory plan was directionally right but too thin to implement. It named tiered memory, ranked retrieval, and staging, but it did not define:

- what gets created at bootstrap,
- what the daemon/session initializes,
- when memory is assembled in the turn loop,
- when the first root-agent call happens relative to memory retrieval,
- how repeated model rounds change context,
- what sub-agents see,
- what gets written to staging versus durable memory versus ephemeral state,
- how much memory work happens in the hot path versus background maintenance.

That left too much room for accidental implementation choices to harden into product behavior.

## What changed in the v0.5 rewrite

The updated plan turns v0.5 into a full runtime contract rather than a retrieval feature.

### Four-tier memory model

v0.5 now explicitly distinguishes:

- `identity`
- `durable`
- `staging`
- `ephemeral`

This keeps the bootstrap bundle separate from learned durable memory, keeps agent-authored learnings reviewable, and gives the daemon a real working-memory layer without inventing a hidden durable transcript store.

### Stable turn-flow contract

The turn flow is now explicit:

1. input arrives
2. bootstrap and always-on memory are snapshotted
3. intent is resolved
4. bounded memory prefetch may run
5. the first root-agent model call happens
6. the normal tool / skill / sub-agent loop continues
7. one bounded refresh retrieval may run after material external evidence
8. final synthesis happens
9. ephemeral and staged memory updates happen after response generation

That ordering matters because memory should help the turn without turning every turn into a retrieval loop.

### Hot path versus background work

The rewrite also makes a clean split:

- hot path: ephemeral updates and candidate staging
- background or operator workflows: promotion, compaction, pruning, maintenance

This follows the project's daemon-and-jobs direction and keeps curated memory from becoming either too magical or too latent.

## Design influences

The rewrite was informed by outside agent-memory patterns, but adapted to Allbert's local markdown-first architecture:

- Hermes reinforced the value of a small always-visible curated memory surface plus separate deeper search.
- OpenClaw reinforced the split between always-loaded markdown memory and explicit memory search/get flows.
- LangGraph was useful for the hot-path versus background-memory distinction.
- Letta was useful for thinking about ephemeral working memory as a coordination primitive, not just a storage bucket.

Allbert still keeps its own core commitments:

- markdown ground truth
- no hidden transcript memory
- explicit staging before durable promotion
- daemon/session working memory without pretending it is durable

## Outcome

The release plan now has an implementation-ready shape:

- bootstrap/init behavior
- per-turn memory flow
- multi-agent memory rules
- operator surfaces
- M1-M6 milestones
- a matching ADR set for the new memory contract

That should let v0.5 start from a real spec instead of discovering the memory architecture piecemeal during implementation.
