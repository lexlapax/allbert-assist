# v0.14.3 Upgrade Notes

v0.14.3 is an operator reliability patch after v0.14.2. It keeps the public command, protocol, and storage surface steady while repairing live operator-test failures: conversational scheduling with local models, explicit memory capture, OpenAI Responses multi-turn history, and Gemini live-provider response compatibility.

## What Changed

- Default intent routing now uses a bounded schema-validated router before full prompt assembly when `intent_classifier.enabled = true` and `rule_only = false`.
- `intent_classifier.rule_only = true` remains the compatibility path for the legacy rule classifier.
- High-confidence schedule router decisions draft guarded job mutations through the existing `upsert_job`, pause, resume, and remove paths.
- Durable schedule mutations still require the existing structured preview-and-confirm approval surface before persistence.
- If a schedule turn falls through to the full assistant path and the model asks for prose confirmation instead of calling the job tool, Allbert retries once with schedule-specific guidance.
- The tool-call parser accepts the observed flat named local-model shape, such as `{"name":"upsert_job","schedule":"@daily at 07:00",...}`, by moving non-`name` fields into `input` before normal authorization.
- High-confidence explicit memory router decisions stage one `explicit_request` candidate through the existing memory staging pipeline.
- Explicit memory capture remains review-first: router-drafted memory is never promoted directly to durable memory.
- OpenAI Responses requests now serialize user text as `input_text`, assistant history as `output_text`, and user images as `input_image`.
- Assistant-side image attachments for OpenAI Responses fail locally with a provider error instead of sending an unsupported request shape.
- Gemini responses now tolerate non-text live response parts while extracting text-bearing parts.
- The Gemini credentialed release smoke uses a larger output budget so Gemini 2.5 hidden thinking does not truncate exact-response checks.

## Operator Checks

Provider-free release validation remains:

```bash
cargo fmt --check
env -u RUSTC_WRAPPER cargo clippy --workspace --all-targets -- -D warnings
env -u RUSTC_WRAPPER cargo test -q
env -u RUSTC_WRAPPER cargo run -q -p allbert-cli -- --help
tools/check_doc_reality.sh
```

The full release gate also includes Rust docs and the kernel architecture checks from the operator playbook.

For local-model scheduling, run a classic REPL or TUI turn:

```text
schedule a daily review at 07:00
```

Expected: Allbert shows the structured durable scheduling confirmation. After approving with `y`, `allbert-cli jobs status daily-review` shows the job.

For explicit memory:

```text
remember that Allbert operator tests use temporary ALLBERT_HOME profiles
review what's staged
```

Expected: the first turn stages one `explicit_request` candidate, and the second turn lists it. Promotion remains explicit.

With OpenAI credentials:

```text
/model openai <model-id> OPENAI_API_KEY
hello
say one more thing
```

Expected: the second turn succeeds after assistant history exists. An OpenAI 400 about `input[1].content[0]` being `input_text` is a regression.

With Gemini credentials:

```text
/model gemini gemini-2.5-flash-lite GEMINI_API_KEY
hello
```

Expected: the turn succeeds, or any failure clearly reports the provider status, such as quota or temporary availability. A local `error decoding response body` failure is a regression.

## Compatibility

No profile migration is required. Existing profiles that use the default classifier settings become router-first automatically. Operators who need the old rule classifier can set:

```toml
[intent_classifier]
enabled = true
rule_only = true
```

No new daemon protocol version, storage migration, command, or approval kind is introduced.
