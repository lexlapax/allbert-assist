# v0.2 Target Note

Date: 2026-04-18

The next target after shipped v0.1 is still scheduled jobs, but the shape of that target needed to change.

Why the original v0.2 plan was too small:

- it treated jobs as another frontend plus scheduler glue
- it kept runtime ownership in the clients instead of in a daemon-capable kernel host
- it relied on OS cron as the primary scheduling model, which does not line up with the desired micro-kernel shape
- it did not create a shared substrate for future channels, background services, or long-lived sessions

What remained true from the original direction:

- the original note explicitly says "after that, we can have scheduled and cron jobs"
- the vision still puts scheduled work ahead of richer retrieval, scripting, and broader integrations
- v0.1 already shipped the simple assistant foundation that this next step should build on

What the research changed:

- OpenClaw reinforced the value of a long-lived host process, attachable clients, and a clean distinction between exact scheduled work and looser future background awareness
- Hermes reinforced the value of an internal scheduler, fresh isolated job runs, lifecycle commands, and durable local run history
- both pointed toward a daemon/gateway shape even though Allbert should stay more local-first and smaller in scope than either system

What Allbert should intentionally borrow:

- a long-lived local daemon host
- attachable clients and channels
- an internal scheduler owned by the runtime
- fresh isolated job runs
- durable local job definitions, state, and history

What Allbert should intentionally not borrow in v0.2:

- a network-first control plane
- a broad messaging-platform gateway surface
- heavy external service management
- a large distributed-service posture

Planning consequence:

- v0.2 should be reframed around a daemon-capable micro-kernel
- REPL, CLI, and jobs should attach as channels instead of owning runtime state
- the daemon should own a lightweight internal job manager
- OS service install and management can come later, but OS cron should not be the primary runtime scheduler

Additional decisions needed before implementation:

- v0.2 is a real config migration from shipped `setup.version = 1` to `setup.version = 2`
- interval schedules persist their next due time and coalesce downtime into at most one immediate catch-up run
- model selection is session-local for interactive sessions, with daemon config as the default and optional per-job overrides
- scheduling should follow a per-job timezone override first, then the configured jobs default timezone, and only then system local time
