# v0.2 Target Note

Date: 2026-04-18

The next target after shipped v0.1 is scheduled jobs.

Why this, and not RAG or scripting first:

- the original note explicitly says "after that, we can have scheduled and cron jobs"
- the vision explicitly lists scheduled jobs as the first growth path after the early foundation
- v0.1 already shipped the exact simple-assistant foundation that note described

This makes scheduled jobs the cleanest next move:

- it follows the intended sequence
- it expands capability without bloating the kernel
- it creates the runtime surface needed for memory maintenance and later compiled-memory work

The key guardrails for v0.2 planning are:

- keep the kernel central
- make jobs a separate frontend
- stay local-first
- let the OS scheduler drive invocation on macOS, Linux, and Windows
- fail closed on interactive prompts

Platform stance:

- support the scheduler command interface on macOS, Linux, and Windows in v0.2
- keep scheduler setup explicit and documented rather than adding installer helpers yet

Research notes from adjacent systems:

- OpenClaw draws a useful line between exact scheduled jobs and later heartbeat-style ambient awareness
- OpenClaw also treats standing orders plus cron as complementary, which suggests Allbert should let jobs lean on durable procedure rather than giant duplicated prompts
- Hermes shows the value of a fuller job lifecycle, ordered skill attachment, fresh-session execution, local history, and explicit pause/resume/run controls

Planning consequence:

- keep Allbert's OS-scheduler-driven architecture
- add lifecycle and history expectations to the v0.2 plan
- define bundled default job templates as disabled-by-default starter jobs
