# v0.14.2 Size Gate Resolution Note

Status: M5 release-readiness blocker resolved during v0.14.2 review.

The initial M5 size enforcement run was correctly blocked:

```text
allbert-kernel-services src/ 41985 LOC, limit <30000
```

The release did not loosen the gate, skip the check, or mark v0.14.2 ready while this failed.

The fix keeps the <30,000 services limit and reduces counted service implementation code:

- Core-owned contract modules now live in `allbert-kernel-core`; services reuses them through thin module shims where service internals still need `crate::config`, `crate::paths`, `crate::error`, and similar paths during the split.
- The concrete services crate no longer carries duplicate copies of core contracts such as config, settings, paths, errors, identity, command catalog, job-manager traits, and provider DTOs.
- The large runtime unit-test suite moved from `crates/allbert-kernel-services/src/lib.rs` to `crates/allbert-kernel-services/tests_support/kernel_runtime_tests.rs` and is still included by `#[cfg(test)]`, so it remains compiled and run by `cargo test`.
- Production service code remains under `src/` and continues to count against `tools/check_kernel_size.sh --enforce`.

The resulting size check reports:

```text
future allbert-kernel-core src/                   9742 LOC  limit <20000
future allbert-kernel-core lib.rs                   95 LOC  limit <4000
future allbert-kernel-services src/              28447 LOC  limit <30000
retired crate presence: crates/allbert-kernel absent
```

This resolves the size blocker without changing the release's compactness promise. v0.14.2 still must pass the full release validation path before it can be tagged.
