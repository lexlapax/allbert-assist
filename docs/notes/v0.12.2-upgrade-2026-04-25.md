# v0.12.2 upgrade notes

v0.12.2 is the tracing and replay point release after v0.12.1. It keeps v0.12.1 live activity as the current-state surface and adds durable, local trace history for answering what happened after a session or turn.

## Upgrade Behavior

- Protocol v4 daemons accept v2, v3, and v4 clients. Older clients do not receive v4-only trace messages or fields.
- Existing profiles load without a required manual migration.
- When a daemon starts and `config.toml` has no `[trace]` key, Allbert attempts a typed, allowlisted, path-preserving default write.
- If the safe default write succeeds, `config.toml` gets the v0.12.2 trace defaults and `daemon.log` records the informational install line.
- If safe insertion would risk rewriting unsupported TOML shape, Allbert skips the write, loads in-memory defaults, and logs a remediation hint.
- Partial `[trace]` blocks are not rewritten. Missing trace keys load defaults and are visible through `/settings show trace`.

## Default Capture

Tracing is enabled by default:

```toml
[trace]
enabled = true
capture_messages = true
session_disk_cap_mb = 50
total_disk_cap_mb = 2048
retention_days = 30
otel_export_dir = ""
otel_service_name = "allbert"

[trace.redaction]
secrets = "always"
tool_args = "capture"
tool_results = "capture"
provider_payloads = "capture"
```

With `capture_messages = true`, trace artifacts can include full prompt and response text, tool arguments, tool results, provider payload summaries, routing decisions, span events, and durations. Traces stay local unless you explicitly export them.

## Privacy Controls

Secrets are always redacted before persistence and export. `trace.redaction.secrets = "never"` is rejected, and the setting cannot be changed through `/settings` or `allbert-cli settings`.

Lower capture fidelity when you do not want full text on disk:

```text
/settings set trace.capture_messages false
/settings set trace.redaction.tool_args summary
/settings set trace.redaction.tool_results drop
/settings set trace.redaction.provider_payloads summary
```

When `trace.capture_messages = false`, capture policies that would store full tool args, tool results, or provider payloads are coerced to summary for that load without rewriting your partial `[trace]` block.

## Trace Commands

CLI:

```bash
cargo run -p allbert-cli -- trace list
cargo run -p allbert-cli -- trace show
cargo run -p allbert-cli -- trace show-span <span-id> --session <session-id>
cargo run -p allbert-cli -- trace tail
cargo run -p allbert-cli -- trace export <session-id> --format otlp-json
cargo run -p allbert-cli -- trace gc --dry-run
cargo run -p allbert-cli -- settings show trace
```

REPL/TUI:

```text
/trace show
/trace show-span <span-id>
/trace tail
/trace export
/trace settings
```

Telegram:

```text
/trace last
/trace span <span-id>
```

Telegram replies are structural and redacted. Full prompt, response, tool argument, and tool result text stays on local disk.

## Storage And Retention

Session traces live under the session they describe:

```text
sessions/<session-id>/trace.jsonl
sessions/<session-id>/trace-*.jsonl.gz
sessions/<session-id>/current_spans/*.json
```

These are continuity-bearing session artifacts and travel with `sessions/` during profile export/sync. The top-level `traces/` directory remains legacy/debug derived output.

Retention is bounded by:

- `trace.session_disk_cap_mb`
- `trace.total_disk_cap_mb`
- `trace.retention_days`

`allbert-cli trace gc` removes trace artifacts only. It does not remove journals, approvals, metadata, or other non-trace session files.

## OTLP-JSON Export

Export is file-only and stays under `ALLBERT_HOME`:

```bash
cargo run -p allbert-cli -- trace export <session-id> --format otlp-json
```

Allbert's internal trace schema remains the stable replay contract. OTLP-JSON is an export format for operators who want to import traces into their own observability tools.
