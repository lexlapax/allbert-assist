# v0.2 Conversational Closeout Note

Date: 2026-04-19

The daemon-first v0.2 reframe was the right architectural move, and the implementation through M5 delivered the substrate well:

- daemon host
- attachable REPL and CLI channels
- daemon-owned job manager
- durable job definitions, runtime state, run history, and failure logs
- operator-facing jobs lifecycle under `allbert-cli jobs ...`

That still did not finish the user-facing scheduling story.

The missing piece is not the scheduler itself. The missing piece is the prompt-facing bridge:

- a user can manage jobs authoritatively through CLI commands
- a user cannot yet count on normal conversation to create, inspect, pause, resume, or remove jobs through first-class daemon-backed tools
- durable schedule mutation still lacks a conversational preview-and-confirm UX

This matters because the original project direction was not just "have recurring jobs somewhere in the system." It was to build a local assistant that becomes easy to make do things by talking to it. The daemon and jobs substrate are necessary, but not sufficient, if the assistant still cannot author and manage recurring work naturally from the prompt.

So v0.2 should be treated as:

- operator-complete through M5
- conversationally incomplete until prompt-native job management lands

Closeout consequence:

- reopen the v0.2 plan from `shipped` to `in_closeout`
- add explicit M6-M8 work for prompt-native job tools, safe conversational job mutation, and conversational closeout smoke
- keep the current daemon/jobs implementation as the foundation rather than rewriting it

The guiding rule for the remaining work is simple:

- CLI remains the authoritative operator escape hatch
- the assistant prompt must become a first-class path for recurring job management
- durable schedule mutations must remain auditable and explicitly confirmed
