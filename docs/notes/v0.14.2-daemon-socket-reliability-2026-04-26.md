# v0.14.2 Daemon Socket Reliability Note

Status: Baseline fix recorded during v0.14.2 M1.

v0.14.1 release validation used serial workspace tests as a diagnostic workaround after default-parallel daemon integration runs could intermittently fail during daemon boot with local socket `Operation not permitted`.

## Root Cause

The daemon runtime path already removes stale socket files, releases the daemon lock after failed bind, and removes socket/lock artifacts on normal shutdown. The weak spot was the integration test allocator: `TempHome` used `/tmp/abd-{pid}-{counter}`. That made test homes deterministic within a single process, but stale directories from interrupted runs or process-id reuse could collide with later test runs and carry old socket/lock state into default-parallel validation.

## Fix

M1 changes the daemon integration test homes to allocate fresh short `/tmp` roots with process id, counter, attempt number, and UUID entropy. Each root is created with `create_dir`, not `create_dir_all`, so reuse is detected instead of silently accepted. The root is set to `0700` on Unix, and the existing daemon boot path still locks down the `run/` directory and cleans socket/lock files on shutdown.

M1 also adds a concurrent daemon regression test that boots multiple daemon instances on separate temp homes, attaches/status-checks each daemon, verifies same-home second-spawn rejection, shuts each daemon down, verifies socket/lock cleanup, restarts it, and verifies cleanup again. This keeps the release gate on default-parallel daemon integration behavior instead of preserving the serial-test workaround.

## Operator Impact

No operator-visible daemon behavior changed. The default socket path, lock semantics, protocol, and CLI command surface are unchanged.
